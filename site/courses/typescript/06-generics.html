Module: Generics

Generics are how TypeScript keeps a type instead of throwing it away — one definition that stays type-safe across every concrete type it meets. This module orients you on each idea and points to the best material to learn it fully, official documentation first.

What this module covers

Generics parameterize a function, interface, class, or type alias over a type supplied later, so you get reuse without reaching for any. This module walks the three layers interviewers actually probe: writing generic types with constraints and defaults, letting inference fill in the type arguments (including in generic classes), and the variance rules — covariance, contravariance, and the in/out annotations added in TypeScript 4.7. It's one of the most-tested TypeScript topics; the depth lives in the linked resources.

Topics

Generic types, constraints, defaults

A generic is a type parameter — a placeholder like <Type> that a function, interface, class, or type alias fills in at the point of use, so one definition works across many concrete types without losing type information. The identity function is the canonical example: it returns exactly what it received, and the caller's type flows straight through the signature. A constraint (T extends …) restricts which types are allowed so you can safely touch members inside the body, and a default (<T = …>) supplies a fallback when the caller omits the argument.

Why it matters: generics are how you get reuse with full type safety — the relationship "the output type equals the input type" is captured in the signature and checked by the compiler, not left to a comment. Constraints and defaults are the two most-tested refinements: extends keyof for safe property access, and defaults that keep common call sites terse.

function identity<Type>(value: Type): Type {
  return value;                       // caller's type flows through
}
interface Box<Type = string> {         // default type parameter
  value: Type;
}
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b; // .length is safe via the constraint
}

Type inference, generic classes

Most of the time you never write the angle brackets at the call site — TypeScript infers the type argument from the values you pass, so identity("hi") resolves Type to string using a best-common-type of the candidates. The same inference powers generic classes: class Box<Type> parameterizes every instance member over Type, and new Box("hi") is inferred as Box<string> from the constructor argument, exactly like a function call.

Why it matters: knowing when inference is enough versus when to annotate is a practical skill. Pass values and let inference work; specify <…> only when there are no arguments to infer from, or when you want to widen or narrow the result. One gotcha interviewers love: static members cannot reference the class's type parameter, because a single static slot is shared by every instantiation and there is no one Type to bind.

class Box<Type> {
  constructor(public contents: Type) {}
}
const b = new Box("hello");  // inferred Box<string>
const n = new Box(42);       // inferred Box<number>

Variance intuition + explicit annotations (in/out)

Variance answers one question: if Dog is a subtype of Animal, is Box<Dog> a subtype of Box<Animal>? Covariant preserves the direction (a producer/getter of Dog can stand in for a producer of Animal); contravariant flips it (a consumer/setter of Animal can stand in for a consumer of Dog); invariant allows neither. Function types are the everyday case — contravariant in their parameters and covariant in their return type — and strictFunctionTypes decides whether parameter positions are checked strictly or bivariantly.

Why it matters: TypeScript normally infers variance structurally, but TypeScript 4.7 added optional annotations so you can state intent explicitly — out T for covariant (output) positions, in T for contravariant (input) positions, in out for invariant. They are mainly a library-author tool: they document guarantees, raise an error when actual usage contradicts the annotation, and let the checker skip structural variance computation on deeply recursive types — a real compile-time speedup.

type Getter<out T> = () => T;            // covariant  (output only)
type Setter<in T> = (value: T) => void;  // contravariant (input only)
interface State<in out T> {              // invariant (read and write)
  get(): T;
  set(value: T): void;
}

Interview questions

Most-asked generics questions in Indian interviews — from product-company screening rounds to senior deep dives, generics separate confident TypeScript answers from surface-level ones:

Beginner What are generics and what problem do they solve?

Generics let you write a function, class, interface, or type that works over a type supplied later while preserving the exact type. The classic example is identity: without generics you would type it as any (losing the return type) or overload it per type; with <Type> the input type flows to the output. The result is reuse with full type safety.

Beginner What does <T> mean, and are there naming conventions?

<T> declares a type parameter — a placeholder filled in when the code is used. Any name works; conventions are T for a generic type, K for a key, V for a value, E for an element, and descriptive names like <Type> in the official handbook for readability. The name is scoped to that one declaration.

Beginner What is a generic constraint?

A constraint (T extends SomeShape) restricts which types the caller may supply, so inside the body you can safely use the members that shape guarantees. For example <T extends { length: number }> lets you read .length. Without the constraint, TypeScript rejects the member access because an arbitrary T might not have it.

Beginner What are default type parameters?

A default (<T = string>) supplies a fallback used when the caller omits the argument and it cannot be inferred, so callers can write Container instead of Container<string>. Defaults keep common cases terse and must themselves satisfy any constraint on the same parameter.

Intermediate Generics vs any — why prefer generics?

any switches off type checking and forgets the type; a generic remembers it and relates inputs to outputs. wrap<T>(x: T): T[] guarantees the array element type matches the argument, whereas any would let you pass a string and get back something typed any. Generics give reuse without surrendering safety.

Intermediate How does TypeScript infer generic type arguments, and when must you specify them?

At a call, TypeScript infers each type parameter from the argument values (identity("hi") gives Type = string) using a best-common-type across candidates. You annotate explicitly only when there are no arguments to infer from, when inference is wider or narrower than you want, or to disambiguate — otherwise let inference do the work.

Intermediate Explain the getProperty<T, K extends keyof T> pattern.

It is the type-safe property accessor: K is constrained to the keys of T, so getProperty(obj, key) returns T[K] and the compiler rejects any key that does not exist on obj. This keyof-plus-indexed-access combination is one of the most-asked generic patterns in interviews.

Intermediate What is a generic class, and how are its type parameters inferred?

A class declared class Box<Type> parameterizes its instance members over Type; new Box("hi") infers Box<string> from the constructor argument, just like a function call. Generic classes support constraints and defaults exactly as functions and interfaces do.

Intermediate Why can't static members reference the class's type parameter?

Type parameters belong to instances; a static member exists once for the whole class regardless of how many times it is instantiated, so there is no single Type to bind it to — TypeScript forbids it. If you need per-type behavior on a static, put the type parameter on the static method itself.

Pro What is variance? Define covariance, contravariance, invariance.

Variance describes how subtyping of a wrapper relates to subtyping of its parameter. Given Dog a subtype of Animal: covariant keeps the direction (Box<Dog> assignable to Box<Animal>, like a getter/producer), contravariant flips it (a Setter<Animal> assignable to Setter<Dog>, like a consumer), invariant allows neither (a read-write container). Functions are contravariant in parameters and covariant in return type.

Pro What is parameter bivariance, and how does strictFunctionTypes affect it?

Sound typing makes function parameters contravariant, but for ergonomic and historical reasons TypeScript checks method parameters bivariantly. strictFunctionTypes (part of strict) turns on strict contravariant checking for standalone function-type positions — but not for methods — catching unsafe callback assignments. It is why a too-narrow handler can slip through on a method yet be rejected as a function property.

Pro What do the in and out variance annotations do, and when should you add them?

Added in TypeScript 4.7, out T marks a covariant (output) parameter, in T a contravariant (input) parameter, and in out an invariant one. They are optional — TypeScript infers variance structurally — but on public APIs they document intent, raise an error if the real usage contradicts the annotation, and let the checker skip expensive structural variance measurement on deeply recursive generic types, speeding up compilation. Mainly a library-author tool.
