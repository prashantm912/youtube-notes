Module: Basic Types

This is where TypeScript stops being "JavaScript with a .ts extension" and starts catching bugs — the primitive set, the any/unknown/never trio that decides how safe your escape hatches are, tuples, and the literal types that make exhaustive checks possible.

Knowledge Library → Courses → TypeScript → Basic Types

What this module covers

The everyday building blocks of the type system: the primitives (string, number, boolean), arrays, and the special types any, unknown, never and void that behave unlike anything in JavaScript. On top of those sit tuples — fixed-shape arrays with named, optional and rest elements — and literal types, which narrow a value to one exact constant and, with as const, power the union-of-strings pattern that competes with enums. Master these before object types, narrowing or generics; every later module assumes them.

Topics

Primitives, arrays, any/unknown/never/void

The three primitive type names are lowercase — string, number, boolean — not the capitalized String/Number wrapper objects, a distinction interviewers probe. Arrays come in two equivalent spellings, number[] and Array<number>. The interesting part is the special types. any switches type checking off for a value and lets it flow anywhere, which quietly disables the tool you installed TypeScript for. unknown is the safe counterpart: any value is assignable to it, but you can do nothing with it until you narrow it, so it belongs at every untyped boundary (JSON.parse, network responses, catch clauses).

The bottom types complete the picture. never is the type with no values — the return type of a function that always throws or loops forever, and the type you reach in an exhaustive switch once every case is handled. void is the return type of a function that returns nothing meaningful. Knowing when each appears — and why unknown should almost always replace any — is the single highest-signal thing this module teaches.

let ids: number[] = [1, 2, 3];
let raw: unknown = JSON.parse(input);   // safe: must narrow before use
if (typeof raw === "string") raw.trim();

function fail(msg: string): never {     // never returns
  throw new Error(msg);
}

Tuples (named/optional/rest elements)

A tuple is an array whose length and per-position types are both fixed: [string, number] means exactly a string then a number. It is how you type a coordinate pair, a useState-style return, or the arguments of a function you plan to spread. Since TypeScript 4.0 you can label each slot — [start: number, end: number] — which changes nothing at runtime but makes editor tooltips and error messages read like named parameters.

Tuples also flex: a trailing ? marks an optional element ([number, number, number?] for 2D-or-3D points), and a rest element ([string, ...number[]]) captures a variable-length tail — and, since 4.0, can even sit at the front or middle. Add readonly to forbid mutation. The reason tuples matter beyond typing pairs is variadic tuple types, which let generic functions like concat and partial application stay fully typed.

type Point = [x: number, y: number];          // labeled
type Range = [start: number, end?: number];   // optional tail
type Row   = [id: number, ...cells: string[]]; // rest element
const rgb: readonly [number, number, number] = [255, 128, 0];

Literal types, as const, enums vs literals

A literal type is a type whose only value is one constant: the type "left" accepts only the string "left". On their own they are trivial; combined into a union they become one of TypeScript's most-used patterns — "left" | "right" | "center" models a closed set of options without any runtime construct. The catch is inference: TypeScript widens literals by default, so let x = "left" is typed string, and an object literal's properties widen too. as const is the fix — it freezes an expression to its narrowest literal type and makes objects and arrays deeply readonly.

That sets up the module's recurring debate: enums versus literal unions. A numeric or string enum is a real runtime object; a const object with as const, or a bare string-literal union, is erased at compile time and stays closer to plain JavaScript. Modern TypeScript often prefers the union or as const approach for zero bundle cost and native string compatibility — but enums still earn their place for readability and IDE refactoring, so know both sides.

type Align = "left" | "right" | "center"; // literal union — no runtime cost

let a = "left";                 // widened to string
let b = "left" as const;        // narrowed to "left"

const Direction = { Up: 0, Down: 1 } as const; // readonly, literal-valued
type Dir = typeof Direction[keyof typeof Direction]; // 0 | 1

Interview questions

Most-asked basic-types questions in Indian interviews — from service-company screening rounds through product-company deep dives:

Beginner What is the difference between any and unknown?

Both accept any value, but any turns type checking off — you can call, index or assign it anywhere with no check. unknown is the safe top type: you must narrow it (with typeof, instanceof, a check) before doing anything with it. Prefer unknown at untyped boundaries; treat any as a last resort.

Beginner What does the never type mean? Where does it appear?

never is the type with no possible value — the bottom type. It is the return type of a function that never returns normally (always throws or loops forever), the type of a variable in an unreachable branch, and what a value narrows to once every union member has been excluded. Nothing except never is assignable to never.

Beginner Difference between void and never as return types?

void means the function returns, but with no useful value (it returns undefined). never means the function never returns at all — it throws or loops. A logger returns void; a function that always throws returns never.

Beginner What is a tuple and how does it differ from an array?

An array like number[] is any length, all one type. A tuple like [string, number] fixes both the length and the type at each position — first a string, then a number. Use tuples for fixed-shape sequences (a pair, a coordinate, a labeled return); use arrays for homogeneous collections.

Beginner What is a literal type? Give an example.

A type whose only allowed value is one constant — "GET", 42, or true. On its own it is narrow; the power comes from unions: "GET" | "POST" | "PUT" restricts a value to a closed set of options at compile time with no runtime code.

Intermediate What does as const do, and when do you need it?

as const is a const assertion: it stops literal widening so "left" as const is typed "left" not string, and it makes object and array literals deeply readonly with literal-typed properties. You need it to derive a literal union from an object, to freeze a config, or to keep an array as a fixed tuple of literals rather than a widened string[].

Intermediate Why is unknown preferred over any for JSON.parse or catch clauses?

Data from JSON.parse or a thrown error has no compile-time guarantee of shape. Typing it any lets you access non-existent properties and crash at runtime. unknown forces a narrowing check first, so the compiler makes you validate before use. Since TypeScript 4.4 you can even set useUnknownInCatchVariables so catch (e) is typed unknown.

Intermediate How do optional and rest elements work in a tuple, and what are the rules?

A trailing ? makes an element optional ([number, number, number?]); optional elements must follow all required ones. A rest element ([string, ...number[]]) absorbs a variable-length run of one type. From TypeScript 4.0 a rest element can appear at the start or middle, not only the end, which enables variadic tuple types for typing spreads and partial application.

Intermediate Enum vs a union of string literals — what are the trade-offs?

A string-literal union has zero runtime footprint, is directly assignable from plain strings, and needs no import. An enum emits a real runtime object (bundle cost, and numeric enums allow reverse lookup), but gives a named symbol, IDE refactoring, and JSDoc deprecation. Modern guidance leans to unions or as const objects for library and app code; enums remain fine where a named reference reads better.

Intermediate Why does let x = "hello" infer string, but you sometimes want the literal type?

TypeScript widens literals on mutable bindings: let assumes you may reassign, so "hello" becomes string. A const keeps the literal type because it can't be reassigned. To keep the literal inside a mutable structure or an object property, use as const or an explicit literal-type annotation — otherwise property inference widens it too.

Pro What is a const enum, and what is the catch with it?

A const enum is fully inlined at compile time — each member is replaced by its literal value and no enum object is emitted, saving bundle size. The catches: it can't be used where a runtime object is needed, ambient const enums in .d.ts files are fragile across packages, and it is disallowed under isolatedModules/isolatedDeclarations (and single-file transpilers like Babel/esbuild) because inlining needs whole-program type info. Many teams avoid it for that reason.

Pro How do you use never to get an exhaustive switch, and why does it matter?

In a switch over a discriminated union, add a default that assigns the value to a never-typed variable — const _exhaustive: never = value;. While every case is handled, the value narrows to never and it compiles. If someone later adds a new union member and forgets a case, that member is no longer never, the assignment fails to compile, and you catch the missing branch at build time instead of in production.
