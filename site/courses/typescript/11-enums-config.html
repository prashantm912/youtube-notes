Module: Enums, Directives and Config

Three loosely-related corners of TypeScript that interviewers keep coming back to: enums and namespaces — the features that surprise you by emitting runtime code — the modern typing tools satisfies and const type parameters, and the comment directives and tsconfig flags that decide what the compiler even looks at.

Topics

Enums (numeric/string/const enum/ambient), namespaces

An enum defines a set of named constants and is one of the few TypeScript features that emits real runtime JavaScript rather than being purely type-level. A numeric enum compiles to an object that maps name→number and, via a reverse mapping, number→name, auto-incrementing from 0 unless you initialise values. A string enum has no reverse mapping and produces readable values in logs and serialized data. A const enum has no runtime object at all — the compiler inlines each member's literal value at the call site, which is smaller and faster but breaks single-file transpilation. An ambient enum (declare enum) describes an enum that already exists at runtime and emits nothing.

Why it matters: enums are asked about precisely because of that runtime cost and the long-running "should I use them at all?" debate — many teams prefer a union of string literals or an as const object, which stay closer to plain JavaScript and disappear at runtime. Namespaces (the old "internal modules", declared with the namespace keyword) predate ES modules and also emit a runtime object; they still appear in legacy code and in .d.ts files, where declare namespace is a common way to type a global library. Knowing when each construct is appropriate — and that both leave code behind — is standard interview territory.

enum Direction { Up, Down }          // numeric: Up = 0, Down = 1 (+ reverse map)
enum Status { Active = "ACTIVE" }    // string: explicit values, no reverse map
const enum Planet { Mercury, Venus } // inlined at use site, no runtime object
let p = Planet.Venus;                // emits simply: let p = 1;
declare enum Env { Dev, Prod }       // ambient: describes an enum defined elsewhere

satisfies, const type parameters, erasableSyntaxOnly, isolatedDeclarations

The satisfies operator (TypeScript 4.9) checks that an expression conforms to a type without widening it: you keep the narrow, inferred type for downstream use while still catching mistakes, unlike a : Type annotation which replaces the value's type with the wider one you wrote. const type parameters (5.0) let a generic infer the narrowest, as const-like type for an argument, so callers get literal and readonly inference without sprinkling as const at every call site.

The other two are emit-and-tooling flags. erasableSyntaxOnly (5.8) bans TypeScript constructs that generate runtime code — enums, namespaces with runtime members, class parameter properties, and import =/export = — so a file can be run by type-stripping tools such as Node's --experimental-strip-types, where TypeScript syntax must erase cleanly. isolatedDeclarations (5.5) requires explicit type annotations on exported declarations so a .d.ts can be produced from a single file without invoking the full type-checker, enabling faster, parallel declaration emit. Why it matters: these four show up constantly in modern tsconfig discussions and in senior interviews about build performance and the "TypeScript runs natively" direction.

const config = {
  port: 8080,
  host: "localhost",
} satisfies Record<string, string | number>;

config.port.toFixed();     // ok — port is still number, not string | number
config.host.toUpperCase(); // ok — host is still string

@ts-expect-error/@ts-ignore/@ts-check, tsconfig flag reference

These comment directives control the checker per line or per file. // @ts-check opts a plain .js file into type-checking (with types supplied via JSDoc), and // @ts-nocheck opts a file out. // @ts-ignore silently suppresses the error on the following line; // @ts-expect-error also suppresses it but reports an "Unused directive" error if the next line has no error — so it self-cleans the moment the underlying bug is fixed, which is why it is the preferred choice in application code and in tests that assert a type failure.

The tsconfig reference is the map of every compiler option: strictness (strict, noUncheckedIndexedAccess), modules (module, moduleResolution), emit (declaration, outDir), and the flags met above. Why it matters: interviewers probe whether you reach for @ts-expect-error over @ts-ignore, and whether you can navigate tsconfig by intent rather than copy-pasting a config you do not understand — both signal day-to-day fluency with the toolchain.

// @ts-check   <-- first line of a .js file: turn on type checking

// @ts-expect-error  reports "Unused directive" if this line ever stops erroring
const id: number = "not-a-number";

// @ts-ignore        silences the next line forever, even after it is fixed
legacyApi.callWithBadTypes();

Interview questions

Most-asked enums, satisfies and config questions in Indian interviews — from screening rounds on enums to senior probes on build flags and the runtime cost of syntax:

Beginner What is a TypeScript enum, and what does a numeric enum compile to?

A named set of constants, and one of the few TypeScript features that emits runtime code. A numeric enum becomes a JavaScript object that maps name→number and, through a reverse mapping, number→name, auto-incrementing from 0 unless you initialise values. So Direction.Up is 0 and Direction[0] is "Up".

Beginner Numeric vs string enums — what is the difference?

Numeric enums auto-increment and get a reverse mapping, so you can recover the name from the value. String enums require an explicit value per member, have no reverse mapping, and produce human-readable values in logs and serialized payloads — usually the safer, clearer choice.

Beginner What does // @ts-check do?

Placed on the first line of a plain .js file, it turns on TypeScript's type checking for that file (types come from JSDoc), surfacing errors without renaming to .ts. // @ts-nocheck does the opposite for a file, and both work alongside the checkJs tsconfig option.

Beginner What is a const enum, and how is its output different?

A const enum has no runtime object — the compiler inlines each member's literal value directly at the use site. Given const enum E { A }, a reference E.A emits just 0. It is smaller and faster, but it cannot be used safely across single-file / isolated transpilation.

Beginner What does the satisfies operator do, in one sentence?

It checks that a value conforms to a type without widening the value's inferred type — you get the validation of an annotation while keeping the narrow, literal type for everything downstream.

Intermediate Why do many teams avoid enums in favour of union string literals or as const objects?

Enums emit runtime JavaScript (extra bundle code) and carry quirks — plain numeric enums accept any number without complaint, const enums break single-file transpilation, and they are nominal-ish rather than structural. A union like type Status = "active" | "done" or an as const object gives the same safety with zero runtime cost and plain-JavaScript semantics.

Intermediate satisfies vs a colon annotation vs as — when do you use each?

A colon annotation (const x: T = ...) checks the value and widens it to T. as asserts a type you claim to know better than the compiler — an override with no real check. satisfies validates against T but leaves the narrow inferred type intact. Rule of thumb: annotation when you want the wider type, satisfies when you want validation but the exact type, and as only as a last-resort override.

Intermediate What are const type parameters and what do they save you?

Introduced in 5.0, function f<const T>(x: T) makes the generic infer the narrowest, as const-like type for the argument, so callers get literal and readonly inference without writing as const at each call site. Handy for functions that must capture the exact shape of literals — config builders, routers, state machines.

Intermediate @ts-ignore vs @ts-expect-error — which should you prefer and why?

@ts-expect-error. Both silence the error on the next line, but @ts-expect-error itself errors ("Unused directive") when that line no longer has an error — so it self-destructs once the underlying issue is fixed, preventing stale suppressions. @ts-ignore stays silent forever and can hide regressions. Prefer expect-error, especially in tests that assert a type failure.

Intermediate What is a namespace, and should you use one in new code?

A namespace (the old "internal module", namespace X { export ... }) groups declarations under a single object and predates ES modules; it emits a runtime object. In new code prefer ES modules — but namespaces still appear in .d.ts files and legacy code, and declare namespace remains a common way to describe the shape of a global library.

Intermediate What is an ambient enum (declare enum) for?

It describes an enum that already exists at runtime — from a hand-written .d.ts or a JavaScript library — without emitting any code. You get the type-level members for checking, but the compiler generates nothing; the real object is expected to be present when the code runs.

Pro Why can exporting a const enum break consumers, and what catches it?

A const enum is inlined using values the compiler reads from its declaration. Single-file transpilers (Babel, esbuild, anything under isolatedModules) do not cross-read the dependency, so they cannot inline it — the import survives to runtime and fails, or version drift bakes in stale values. isolatedModules flags const-enum exports, and erasableSyntaxOnly bans const enums outright. Fix: don't export const enums — use a regular enum, a union, or an as const object.

Pro What does erasableSyntaxOnly enforce, and why does Node need it?

Added in 5.8, it errors on any TypeScript syntax that emits runtime code — enums, namespaces/modules with runtime members, class parameter properties, and import =/export =. Type-stripping runtimes like Node's --experimental-strip-types only delete type syntax; they do not compile it, so a construct needing codegen would produce broken output. The flag keeps a file safely strippable, and pairs naturally with verbatimModuleSyntax.

Pro What does isolatedDeclarations require, and what does it buy you?

Added in 5.5, it requires explicit type annotations on every exported declaration, so a .d.ts can be generated from each file in isolation — without the full type-checker inferring cross-file return types. That lets external tools emit declaration files much faster and in parallel, at the cost of writing return types on your public API. It is a deliberate bet on build-tool performance for large codebases.
