Module: Type-Level Programming

This is where TypeScript stops merely describing shapes and starts computing them: the type system becomes a small functional language you program with keyof, conditional types, mapped types and template literals.

Knowledge Library → Courses → TypeScript → Type-Level Programming

What this module covers

Type-level programming is TypeScript treating types as values it can query, branch on, transform and recurse over — all erased before any code runs. This module orients you on the four building blocks: the query operators (keyof, type typeof, indexed access), conditional types with infer, mapped types, and template literal types. Together they are how every utility type is built and how library authors express APIs that "just know" the right shape.

Topics

keyof, typeof type operator, indexed access types

These three operators turn an existing type into a new one without you retyping anything. keyof T produces the union of T's keys as literal types; the type typeof (used in a type position, distinct from the runtime operator) captures the static type of a value; and indexed access T["key"] looks up the type stored at a property. Chained together — typeof a config object, keyof its type, index into it — they let a type track a value automatically.

Why it matters: nearly every generic helper you will write or read (a type-safe get(obj, key), a Record derived from real data, an enum-from-object pattern) is built on this trio. Getting them fluent is the entry ticket to reading the rest of the type system.

const config = { host: "localhost", port: 5432 };
type Config = typeof config;      // { host: string; port: number }
type Keys   = keyof Config;       // "host" | "port"
type Port   = Config["port"];     // number

Conditional types, infer, distributivity

A conditional type is a ternary at the type level: T extends U ? X : Y. Inside the extends clause you can place infer R to declare a placeholder TypeScript solves for by pattern-matching the structure — that is how you pull the element type out of an array or the return type out of a function. When the checked type is a bare ("naked") type parameter and you pass a union, the conditional distributes: it runs member by member and re-unions the results.

Why it matters: conditional types plus infer are the engine behind Exclude, Extract, ReturnType, Awaited and most library-grade generics. Distributivity is also the classic trap — the difference between string[] | number[] and (string | number)[] is a standard interview probe.

type ElementType<T> = T extends (infer U)[] ? U : T;
type A = ElementType<string[]>;          // string

type ToArray<T> = T extends unknown ? T[] : never;
type B = ToArray<string | number>;       // string[] | number[]  (distributes)

Mapped types (key remapping, modifiers)

A mapped type builds a new object type by iterating another type's keys: { [K in keyof T]: … }. You can rewrite the value type per key, add or remove the readonly and optional (?) modifiers with +/-, and since 4.1 remap the key itself with an as clause (usually a template literal), returning never to filter a key out entirely.

Why it matters: Partial, Required, Readonly, Record, Pick and friends are all mapped types — being able to write one by hand is what "I understand utility types" actually means in an interview, and key remapping is how you generate getters, event maps and renamed shapes.

type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};
type Optional<T> = { [K in keyof T]?: T[K] };          // add ? modifier
type Mutable<T>  = { -readonly [K in keyof T]: T[K] }; // remove readonly

Template literal types, recursive types

Template literal types apply JavaScript's backtick syntax in type positions: `get${Capitalize<K>}` concatenates literal types, and a union inside the template expands combinatorially into every string it could produce. Paired with infer inside a conditional, they can parse a string type — and because conditional types may reference themselves, that parsing can recurse to walk a whole string, tuple or nested structure.

Why it matters: this is what powers typed route params, typed object paths, string-manipulation utilities and query builders. Recursion is also where you meet the compiler's limits — knowing why a type errors with "excessively deep" and how tail-recursion patterns keep it shallow is a senior-level distinguisher.

type Path  = `/${string}`;                         // any "/..." string
type Event = `${"first" | "last"}NameChanged`;     // "firstNameChanged" | "lastNameChanged"

type Trim<S> = S extends ` ${infer R}` ? Trim<R> : S;  // recursive
type Hi = Trim<"   hi">;                            // "hi"

Interview questions

Type-level programming is the round that separates senior TypeScript engineers in Indian product-company interviews — these are the questions that recur, from screening to design-heavy rounds:

Beginner What does keyof do, and what type does it produce?

It takes an object type and produces the union of its keys as literal types. keyof { x: number; y: number } is "x" | "y". It is how you say "any valid key of this type" in a generic — the foundation for type-safe property access.

Beginner The type typeof vs JavaScript's runtime typeof — what's the difference?

Same keyword, two contexts. In value space it runs at runtime and returns a string like "object". In a type position it never runs — it captures the static type of a variable or property, e.g. type C = typeof config. The type operator is erased at compile time.

Beginner What is an indexed access type?

A type looked up by bracket notation on another type: Person["age"] gives the type of that property. You can index with a union (Person["a" | "b"]), with keyof to get every value type, or with T[number] to get an array or tuple element type.

Beginner What is a mapped type, and name three you have already used?

A type that builds a new object type by iterating another's keys with [K in keyof T]. You have used them without noticing: Partial, Readonly, Record, Pick are all mapped types.

Intermediate What is a conditional type, and what does infer let you do?

A type-level ternary: T extends U ? X : Y. infer R declares a placeholder inside the extends clause that TypeScript solves for by matching structure — pulling the element type out of an array, the return type out of a function, or the resolved value out of a Promise.

Intermediate What are distributive conditional types? Show what happens with a union.

When the checked type is a naked type parameter and the argument is a union, the conditional runs per member and re-unions the results. ToArray<string | number> becomes string[] | number[], not (string | number)[]. This distribution is exactly why Exclude and Extract work.

Intermediate How do you stop a conditional type from distributing?

Wrap both sides in a one-element tuple: [T] extends [U] ? … : …. The tuple removes the naked type parameter, so the union is compared as a whole instead of member by member — the standard trick when you want to test the entire union at once.

Intermediate What are mapping modifiers, and how do you remove readonly or optional?

readonly and ? are modifiers you add or strip inside a mapped type with + (default) or -. -readonly [K in keyof T] makes a mutable clone; [K in keyof T]-? makes every property required — precisely how Required is built.

Intermediate What is key remapping with as, and how do you filter a key out?

The as clause after the key (4.1+) renames the produced key, usually via a template literal such as as `get${Capitalize<string & K>}`. Map a key to never and it is dropped from the result — the idiomatic way to filter properties inside a mapped type.

Intermediate What are template literal types good for?

Building precise string literal types from other types: event names, route paths, CSS units, prefixed keys. With unions they expand combinatorially, and with infer inside a conditional they can parse a string — which is what powers typed routers, object-path helpers and query builders.

Pro keyof on a type with a string index signature — what do you get, and why?

For { [k: string]: T } you get string | number, not just string. JavaScript coerces numeric property access to strings at runtime (obj[0] and obj["0"] hit the same slot), so a string index signature also accepts number keys, and keyof reflects both.

Pro Implement ReturnType from scratch and explain each piece.

type ReturnType<T> = T extends (...args: any[]) => infer R ? R : any;. It checks that T is callable, uses infer R to capture whatever the function returns, and yields R; the any branch only fires when T is not a function. Parameters is the same pattern inferring the argument tuple instead.

Pro Why do recursive conditional types hit "Type instantiation is excessively deep", and how do you avoid it?

A recursive conditional re-instantiates itself; deep or unbounded recursion — long string parsing, large tuples — blows past the compiler's instantiation-depth guard. Keep it shallow with a tail-recursive accumulator (carry results in an extra type parameter so recursion sits in tail position, which the compiler optimises since 4.5), cap the depth, or fall back to a looser type. Tail-recursion elimination raised the practical ceiling but did not remove it.
