Module: Modules and Declarations

Declaration files are how TypeScript describes code it cannot see — the JavaScript libraries, browser globals and untyped modules your app leans on — and module resolution is the machinery that decides where an import actually points.

What this module covers

This module is about the boundary between typed and untyped code. You will meet .d.ts declaration files and the declare keyword that describe shapes without emitting any JavaScript, ambient modules and declare global for the world outside your source, module augmentation and triple-slash directives for reshaping types you do not own, and the DefinitelyTyped ecosystem behind @types. It closes with the settings that govern all of it: moduleResolution, verbatimModuleSyntax and the ever-tricky ESM/CommonJS interop.

Topics

Declaration files: .d.ts, declare, declare global and ambient modules

A .d.ts file contains only type information — no runtime code is emitted from it. The declare keyword introduces an ambient declaration: it tells the compiler "trust me, this value or type exists at runtime" so JavaScript libraries, build-time constants and non-code imports type-check without you rewriting them. An ambient module (declare module "name") types a bare import specifier, and a wildcard form like declare module "*.svg" types the asset imports a bundler injects.

Why it matters: every time you consume an untyped package, add a global your bundler defines, or import a .css or .svg file, you are relying on ambient declarations. declare global — usable only from inside a module — is how you extend built-in globals such as Window without polluting a plain script. Getting this right is the difference between a clean project and a wall of red squiggles.

// assets.d.ts — ambient module so `import logo from "./logo.svg"` type-checks
declare module "*.svg" {
  const url: string;
  export default url;
}

// globals.d.ts — extend the built-in global scope from inside a module
declare global {
  interface Window { __store__: Record<string, unknown>; }
}
export {}; // the empty export makes this file a module, enabling declare global

Module augmentation, triple-slash directives and DefinitelyTyped

Module augmentation reopens an existing module's declarations to add or refine members — the standard way to attach req.user to Express's Request, add a method to a library's interface, or teach a plugin's types to the host it plugs into. You place a declare module "the-package" block (with a matching import so the file is a module) and merge new members via declaration merging. Triple-slash directives (/// <reference ... />) are the older single-file mechanism for pulling in another declaration's path, types or lib; modern code prefers imports and tsconfig settings, but you still see them inside .d.ts files.

Why it matters: when a package ships no types, DefinitelyTyped supplies community-maintained declarations published as @types/* packages on npm — npm i -D @types/node is a daily reflex. Knowing whether to reach for an @types package, write a local .d.ts, or augment an existing module is a very common backend and full-stack interview probe in India.

// augment a third-party module's types from your own code
import "express";

declare module "express-serve-static-core" {
  interface Request {
    user?: { id: string; roles: string[] }; // now req.user is typed everywhere
  }
}

Module resolution: moduleResolution, verbatimModuleSyntax and ESM/CJS interop

How TypeScript turns import x from "y" into a file on disk is governed by moduleResolution. Use node16 or nodenext when your output actually runs on Node.js — they enforce Node's dual-format rules, including mandatory file extensions on relative ESM imports and package.json "exports". Use bundler when a build tool (Vite, webpack, esbuild) resolves the code, which relaxes the extension requirement while still honouring "exports". Node decides whether a file is ESM or CommonJS from its extension and the nearest package.json "type" field, and TypeScript mirrors that with .mts and .cts.

Why it matters: verbatimModuleSyntax (TypeScript 5.0) makes emit predictable — imports and exports without a type modifier are preserved verbatim, anything marked type is erased — which pushes teams toward disciplined import type usage. The famous "it works in the bundler but crashes in Node" bug almost always traces back to a mismatch between these settings and ESM/CommonJS interop, making this the single most interview-relevant configuration topic for senior TypeScript roles.

// tsconfig.json for a modern app that runs on Node.js
{
  "compilerOptions": {
    "module": "nodenext",
    "moduleResolution": "nodenext",
    "verbatimModuleSyntax": true
  }
}

Interview questions

Most-asked declarations and modules questions in Indian interviews — from library-typing screens to senior ESM/CommonJS migration deep dives:

Beginner What is a .d.ts file and why does it emit no JavaScript?

A declaration file holds only type information — signatures, interfaces, ambient declarations — so the compiler can type-check code whose implementation lives elsewhere (a shipped JavaScript library, a global your bundler injects). Because it describes rather than defines, tsc produces no output .js for it.

Beginner What does the declare keyword do?

It creates an ambient declaration: you assert to the compiler that a variable, function, class or module exists at runtime without providing an implementation. declare const VERSION: string; types a global your build defines; no code is generated for the declaration itself.

Beginner What is an ambient module declaration?

declare module "some-lib" { ... } types a bare import specifier that has no types of its own. The wildcard form declare module "*.svg" types non-code imports (assets, styles) that a bundler resolves. It is how you make import thing from "some-lib" compile cleanly.

Beginner What is DefinitelyTyped and what are @types packages?

DefinitelyTyped is a community repository of high-quality declaration files for libraries that ship none. Its contents are auto-published to npm under the @types scope, so npm i -D @types/lodash gives you Lodash's types. TypeScript picks them up automatically from node_modules/@types.

Beginner Difference between import and import type?

import type imports only the type, guaranteed to be erased at compile time — no runtime import is emitted, so no side effects and no circular-dependency surprises. A plain import may carry a real runtime dependency. Under verbatimModuleSyntax the distinction becomes mandatory for type-only symbols.

Intermediate What is module augmentation and give a real use case?

Reopening an existing module's declarations to add members, via declare module "pkg" { ... } plus an import so the file is a module. Classic use: adding req.user to Express by augmenting Request in express-serve-static-core, or adding a custom theme field to a UI library's types. It relies on interface declaration merging.

Intermediate When do you need declare global, and what is the empty export for?

Use declare global to add to built-in globals (e.g. a property on Window, a method on Array) from inside a module. It is only legal inside a module, so a file that otherwise has no import/export needs export {}; to become one — otherwise the declarations leak as plain ambient script globals instead.

Intermediate Are triple-slash directives still needed?

Rarely in application code — imports and tsconfig types/lib options replace most uses. You still meet /// <reference types="..." /> and /// <reference lib="..." /> inside hand-written .d.ts files to pull in dependent declarations or specific library types. Since 5.5 the compiler no longer emits them unless marked preserve="true".

Intermediate What does verbatimModuleSyntax do?

Introduced in 5.0, it makes emit literal: import/export statements without a type modifier are left exactly as written, and anything with type is dropped entirely. This removes the guesswork of the old elision rules, replaces importsNotUsedAsValues and preserveValueImports, and forces explicit import type for type-only symbols.

Intermediate Explain moduleResolution node16/nodenext versus bundler.

node16/nodenext model Node's real dual-format resolution: they enforce file extensions on relative ESM imports and respect package.json "exports"/"imports" and "type". bundler also honours "exports" but never requires extensions, matching how Vite/webpack/esbuild resolve. Pick the one that matches whatever actually loads your output.

Pro How does TypeScript decide a .ts file is ESM or CommonJS, and why does it matter?

Under node16/nodenext it follows Node's rules: .mts is always ESM, .cts always CommonJS, and a plain .ts is ESM only if the nearest package.json has "type": "module", else CommonJS. Format decides the emitted syntax (import vs require), allowed constructs (top-level await), and default-import behaviour — get it wrong and the file crashes at load time.

Pro What is the ESM/CommonJS interop problem, and how do esModuleInterop and default imports relate?

CommonJS has a single module.exports value with no real notion of a default export, so importing it from ESM is ambiguous. esModuleInterop synthesises a default so import express from "express" works and fixes namespace-import call semantics. The danger is code that behaves one way in a bundler and another under Node's native rules, producing x.default is not a function style failures.

Pro Your library's types work locally but consumers get "cannot find declaration file" — how do you debug it?

Check the published package: is there a types/typings field (or "exports" with a "types" condition) pointing at a real emitted .d.ts? Confirm declaration: true ran and the .d.ts is in files/not .npmignore-d. For dual-format packages, verify per-condition types entries under "exports" and validate the whole shape with a tool like @arethetypeswrong/cli before publishing.
