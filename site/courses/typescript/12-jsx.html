Module: JSX and TSX

TypeScript type-checks JSX as a first-class syntax — your tags, attributes and children are validated before a single framework line runs. This module separates the language feature from React: the compiler modes, the JSX namespace, and how element and prop types are resolved.

Knowledge Library → Courses → TypeScript → JSX and TSX

What this module covers

JSX is a syntax that TypeScript both checks and emits on its own: the .tsx extension turns it on, the jsx compiler option decides what it compiles to, and a special JSX namespace decides which tags, attributes and children are legal. This module keeps the focus on JSX as a TypeScript language feature — React is only the most common runtime that plugs into it — covering the compiler modes and the classic-vs-automatic factories first, then how intrinsic elements, components, props and children are typed. Each piece links to the official reference, and the interview section reflects what frontend and full-stack roles in India actually ask.

Topics

jsx compiler options and the JSX namespace

A .tsx file tells the compiler to parse JSX; the jsx option then decides what that JSX becomes. Five modes exist: preserve (keep the JSX untouched, emit a .jsx file for a later tool like Babel), react-native (keep the JSX but emit .js), react (the classic runtime — rewrite tags to React.createElement calls, which requires the factory to be in scope), and react-jsx / react-jsxdev (the automatic runtime, added in TypeScript 4.1 — rewrite tags to _jsx helper calls the compiler auto-imports, so no import React is needed). Three related flags tune this: jsxFactory and jsxFragmentFactory set the classic-runtime functions (for example h and Fragment for Preact), while jsxImportSource sets the module the automatic-runtime helpers come from (default "react", resolving to react/jsx-runtime).

Whichever mode you pick, type-checking is governed by the JSX namespace. JSX.IntrinsicElements maps lowercase tag names to their allowed props — it is how the compiler knows <div> is valid but <dvi> is not — JSX.Element is the type a JSX expression evaluates to, and JSX.ElementChildrenAttribute names the prop that receives children. Crucially this namespace is supplied by the framework's type definitions (or via jsxImportSource), not baked into the language — which is exactly why the same JSX syntax can target React, Preact, Solid or your own runtime.

{
  "compilerOptions": {
    "jsx": "react-jsx",         // automatic runtime — no `import React`
    "jsxImportSource": "react"  // _jsx() comes from "react/jsx-runtime"
  }
}

Typing components, props and children (framework-agnostic)

TypeScript resolves a tag by its case. A lowercase name (<div>) is an intrinsic element and is looked up in JSX.IntrinsicElements; a capitalized name (<Card>) is a value-based element and must resolve to a function or class in scope — this is why components have to be capitalized. For a function component, the type of its first parameter is the attribute (prop) type, so the compiler checks the JSX attributes against that object shape and flags missing or misspelled props. Children are just another prop, named by JSX.ElementChildrenAttribute (conventionally children): type it broadly with the runtime's "anything renderable" type — in React that is React.ReactNode — or narrowly as a single JSX.Element when you truly need one element, or reach for a helper like PropsWithChildren.

Two sharp edges show up in interviews. First, since TypeScript 5.1 the element type is decoupled from the tag's return type via JSX.ElementType, so components that return something other than JSX.Element — an async component returning a Promise, or a component returning a string — now type-check; before 5.1 the return had to be assignable to JSX.Element. Second, you cannot make children "type-safe" per component: TypeScript types every JSX child as JSX.Element regardless of which component produced it, so "only accept <Tab> children" is not expressible through JSX alone. Note too that in .tsx the angle-bracket cast <T>value is ambiguous with a tag and is disallowed — you must write value as T.

type CardProps = {
  title: string;
  children?: React.ReactNode;   // anything the runtime can render
};

// Capitalized name -> value-based (component) element
function Card({ title, children }: CardProps) {
  return (
    <section>
      <h2>{title}</h2>
      {children}
    </section>
  );
}

Interview questions

Most-asked JSX and TSX questions in Indian interviews — from service-company screening rounds to product-company type-system deep dives:

Beginner What is the difference between a .ts and a .tsx file?

.tsx tells TypeScript to parse JSX syntax; .ts does not. A side effect: in a .tsx file the angle-bracket type assertion <T>value is ambiguous with a JSX tag, so it is disallowed and you must use the value as T form instead.

Beginner What does the jsx compiler option do, and what are its values?

It controls how JSX is checked and emitted. The values are preserve, react, react-native, react-jsx and react-jsxdev. They differ in whether the JSX is left untouched or rewritten to factory calls, and in the output file extension.

Beginner Difference between preserve and react mode?

preserve keeps the JSX as-is and emits a .jsx file for a downstream tool (like Babel) to transform. react compiles the JSX to React.createElement calls itself and emits .js. Use preserve when another bundler owns the final transform.

Beginner Do you still need import React to use JSX?

With the classic runtime (jsx: "react"), yes — the factory React.createElement must be in scope. With the automatic runtime (jsx: "react-jsx", TypeScript 4.1+), no — the compiler auto-imports the _jsx helpers from the jsxImportSource's /jsx-runtime, so the import is unnecessary.

Intermediate What is the JSX namespace, and how does TypeScript know <div> is valid?

The JSX namespace is where element typing rules live. JSX.IntrinsicElements is an interface mapping lowercase tag names to their prop types; if a tag is not a key, the compiler errors. The namespace comes from the framework's type definitions (or jsxImportSource) — it is not built into the language, which is what makes JSX runtime-agnostic.

Intermediate How does TypeScript decide whether a tag is an intrinsic element or a component?

By case. A lowercase (or dotted-lowercase) name is an intrinsic element, looked up in JSX.IntrinsicElements. A capitalized identifier is a value-based element, resolved to a function or class value in scope. That is the concrete reason component names must start with a capital letter.

Intermediate What type does a JSX expression have, and how do you type a function component's props?

A JSX expression has type JSX.Element (as defined by the active namespace). For a function component, the type of its first parameter is the props/attribute type — declare it as a plain type or interface, and the compiler checks the JSX attributes against that shape, flagging missing or extra props.

Intermediate How are children typed, and when do you use React.ReactNode vs JSX.Element?

Children are just a prop, named by JSX.ElementChildrenAttribute (conventionally children). React.ReactNode is the broad "anything renderable" type — elements, strings, numbers, arrays, null — and is the default choice. JSX.Element restricts to a single element (no strings, no multiple children). PropsWithChildren<P> is the helper that adds an optional children: ReactNode to your props.

Intermediate What do jsxFactory, jsxFragmentFactory and jsxImportSource control?

For the classic runtime, jsxFactory sets the create-element function (e.g. h for Preact) and jsxFragmentFactory sets the fragment factory. For the automatic runtime, jsxImportSource sets the module the _jsx/_jsxs helpers are imported from (default "react"). Together they let non-React runtimes reuse the same JSX syntax.

Pro Why can't you restrict a component's children to one specific component type?

Because TypeScript types every JSX child as JSX.Element regardless of which component produced it — the produced type carries no information about the source component. So "only accept <Tab> children" is not expressible through JSX typing. Workarounds live outside JSX: a typed array/config prop, render props, or a runtime check.

Pro What changed in TypeScript 5.1 about JSX element and tag types?

5.1 decoupled the element type from the tag's return type via the new JSX.ElementType. A component's return no longer has to be assignable to JSX.Element, so components returning other types — an async component returning a Promise, or one returning a string — now type-check. This unblocked patterns like React Server Components' async components.

Pro How would you type a reusable component so it isn't tied to one JSX runtime?

Lean on the JSX namespace abstractions rather than framework specifics: keep props a plain generic type/interface, return the namespace's element type, and set jsxImportSource so the automatic runtime resolves the helpers — swapping React for Preact then becomes a config change, not a rewrite. For a genuinely runtime-neutral library, avoid importing framework value APIs and rely on 5.1's JSX.ElementType so broader return shapes stay legal across runtimes.
