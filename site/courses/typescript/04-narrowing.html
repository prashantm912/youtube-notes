Module: Unions, Intersections and Narrowing

A union says a value is one of several shapes; narrowing is how you earn the right to use it. This module maps every way TypeScript turns a wide type into a precise one — and the type predicates that let you teach it your own checks.

What this module covers

This is where TypeScript's type system starts to feel powerful: composing types with unions (|) and intersections (&), then letting the compiler narrow a broad type down to a precise one along each code path. You will meet every narrowing technique — typeof, instanceof, in, truthiness, equality, and the discriminated-union switch — plus the user-defined type predicates and assertion functions that extend narrowing to your own runtime checks. Learn this before generics; conditional types and control-flow inference all build on top of it.

Topics

Union and intersection types

A union type A | B says a value is one of the listed types. Until you narrow it, you may only touch members common to every constituent — that shared-member rule is the whole reason narrowing exists. An intersection A & B is the dual: the value must satisfy all the types at once and therefore has all their members. The clean mental model interviewers want: union is OR, intersection is AND.

Why it matters: unions model real-world "this or that" — an id that is number | string, a request that is loading, success or error — and they are the backbone of exhaustive state modelling. Intersections mainly compose object shapes (mixins, merged props). Confusing which one "adds up" the members is one of the most common beginner mistakes, so the OR/AND framing gets tested directly.

type Id = number | string;              // union: one of these
type Draggable = { drag(): void };
type Resizable = { resize(): void };
type UIWidget = Draggable & Resizable;  // intersection: must do both

Narrowing: typeof, instanceof, in, truthiness, discriminated unions

Narrowing is how TypeScript refines a broad static type to a more specific one along a code path, driven by control-flow analysis. The built-in guards are typeof (primitives), instanceof (class instances), the in operator (property presence), truthiness (if (x)), and equality (=== / !== against a literal). Inside a guarded branch, the variable's type is narrowed automatically — no cast needed.

The most important pattern here is the discriminated (tagged) union: give each union member a common literal property (kind: "circle" | "square") and switch on it — each case narrows to exactly one member. Add a default that assigns the value to a never variable and you get compile-time exhaustiveness checking: introduce a new member without handling it and the code stops compiling. This is the idiom that makes illegal states unrepresentable.

type Shape =
  | { kind: "circle"; r: number }
  | { kind: "square"; side: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle": return Math.PI * s.r ** 2;   // s is Circle here
    case "square": return s.side ** 2;          // s is Square here
    default: {
      const _exhaustive: never = s;             // errors if a case is missed
      return _exhaustive;
    }
  }
}

Type predicates (incl. inferred) and assertion functions

When a check is too complex for the built-in guards, write a user-defined type guard: a function whose return type is a type predicate, arg is Type. A true result narrows the argument at every call site. Since TypeScript 5.5 the compiler can infer these predicates for simple functions — so arr.filter(x => x !== undefined) now narrows away undefined on its own, without an explicit annotation.

Assertion functions (TypeScript 3.7) narrow by throwing instead of returning a boolean. A return type of asserts arg is Type (or asserts condition) tells the compiler that if the function returns normally, the assertion held — the model behind Node's assert and runtime validators. Both are far safer than an as cast, which changes the static type with no runtime check and simply silences the checker. Note that an assertion signature requires an explicit return-type annotation.

function isString(x: unknown): x is string {   // type predicate
  return typeof x === "string";
}

function assertDefined<T>(x: T): asserts x is NonNullable<T> {
  if (x == null) throw new Error("value is missing");
}

Interview questions

Most-asked union and narrowing questions in Indian interviews — from service-company screening rounds to product-company type-system deep dives:

Beginner What is a union type, and what can you do with a value before you narrow it?

A union A | B means the value is one of the listed types. Before narrowing you may only access members common to every constituent — shared properties and methods. Anything specific to just one member is a compile error until you narrow to that member with a type guard.

Beginner Union vs intersection — the one-line mental model?

Union (|) is OR: the value is one of the types, and you get only the common members. Intersection (&) is AND: the value satisfies all the types at once and has all their members. Beginners often flip these — a union does not add members, an intersection does.

Beginner What is narrowing, and what are the built-in type guards?

Narrowing is refining a broader type to a more specific one along a code path, using TypeScript's control-flow analysis. The built-in guards: typeof (primitives), instanceof (class instances), the in operator (property presence), truthiness checks (if (x)), equality (===/!==) against literals, and Array.isArray.

Beginner How does a typeof guard work, and what is its classic gotcha?

if (typeof x === "string") narrows x to string inside that branch. Gotchas: typeof null === "object" (so a != null check is often needed too), and typeof only sees JS runtime types — it cannot distinguish two different object/interface types from each other.

Intermediate What is a discriminated (tagged) union and why is it the preferred pattern?

A union whose members share a common literal property — the discriminant, e.g. kind: "circle" | "square". Switching on that property narrows to exactly one member with no casts. Preferred over "bags of optional fields" because it makes illegal states unrepresentable and unlocks exhaustiveness checking.

Intermediate How do you get exhaustiveness checking with never?

In the default/else, assign the value to a never: const _exhaustive: never = x;. When every case is handled, x has narrowed to never and it compiles. Add a new union member and forget its case, and x is no longer never — a compile error tells you exactly where to handle it.

Intermediate When do you reach for the in operator instead of typeof or instanceof?

For plain object shapes with no class (so instanceof can't help) and no shared discriminant property. if ("swim" in animal) narrows to the member that declares swim. It is the go-to guard for structural object unions built from interfaces or type aliases rather than classes.

Intermediate What does a type predicate add over a function that just returns boolean?

A function returning plain boolean narrows nothing at call sites. Declaring the return type as arg is Type makes it a user-defined type guard, so callers get narrowing in the true branch. TS 5.5+ can infer the predicate for a simple single-return function, but an explicit predicate is portable and states intent.

Intermediate any vs unknown when you have to narrow external data?

any switches off checking — no narrowing required, but also no safety. unknown forces you to narrow (via typeof, a predicate, or an assertion) before use, which is exactly what you want for JSON.parse, fetch results and catch clauses. The expected answer is: prefer unknown at the boundary.

Pro What is an assertion function, and how does asserts differ from a boolean type predicate?

An assertion function has a return type of asserts arg is Type (or asserts condition) and narrows by throwing: if it returns normally, TypeScript assumes the assertion held for the rest of the scope. A predicate returns a boolean and narrows inside an if; an assertion narrows the straight-line code after the call, no branch. Assertion signatures must have an explicit return-type annotation.

Pro Why prefer a type guard or assertion over an as type assertion?

as (and <T>) is a compile-time cast: it rewrites the static type with zero runtime check, so a wrong cast just defers the crash. A guard or assertion performs a real runtime check and only then narrows — it is sound and it catches bad data at the boundary. Treat as as a last resort, not a narrowing tool.

Pro How can control-flow narrowing be lost — with let, closures, or after a function call?

Narrowing tracks a specific binding at a specific point. Reassigning the variable, or reading a narrowed let inside a callback/closure that may run later, discards the narrowing because TypeScript can't prove the value is unchanged. A narrowed object property can also widen back after any function call that might mutate it. Fixes: copy to a const before the guard, destructure the discriminant, or don't reassign inside the branch.
