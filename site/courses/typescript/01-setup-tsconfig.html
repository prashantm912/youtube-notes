Module: Why TypeScript and Setup

Before you write a single type annotation you make three decisions: which compiler runs your code, what tsconfig.json puts in force, and how strict the checker is — and in 2026 all three shifted, when TypeScript 6.0 flipped its long-standing defaults and 7.0 went native.

What this module covers

This is the setup layer everything else sits on: why adding a static type checker to JavaScript is worth it, how the tsc compiler turns .ts into .js while catching errors first, and what a tsconfig.json actually controls. It also covers the two changes every 2026 interview now probes — TypeScript 6.0's flipped defaults (strict on, module esnext, a floating es2025 target) and TypeScript 7.0's native Go compiler. Get this right and strict typing, tooling, and CI all fall into place.

Topics

Why TypeScript, type checking, the compiler (tsc, TS 7 native)

TypeScript is JavaScript plus an optional static type layer that a checker verifies before your code runs. Every type you write is erased during compilation, so the output is ordinary JavaScript with no runtime type system. The payoff is catching a whole class of mistakes — typos, wrong argument shapes, null dereferences — at author time instead of in production, plus the type information that powers reliable editor autocomplete and refactoring.

The tool that does both the checking and the translation is tsc, the TypeScript compiler: it type-checks your files and emits JavaScript, down-levelling newer syntax to your chosen target. Note a surprise for beginners — by default tsc still emits JavaScript even when it finds type errors; the errors are advisory to the emit step unless you set noEmitOnError. As of 2026 the compiler itself changed dramatically: TypeScript 7.0 (GA July 2026) is a from-scratch port of tsc and the language service to Go — codenamed Project Corsa — that keeps the same types and semantics as the 6.x JavaScript-based compiler but runs roughly 8–12x faster. Knowing that tsc is now native, not the old JS build, is exactly the kind of currency interviewers test.

npm install -D typescript   # project-local compiler
npx tsc app.ts              # type-check and emit app.js
npx tsc --noEmit           # type-check only (ideal for CI)

tsconfig anatomy, TS 6.0 defaults (strict/module esnext/target es2025)

A tsconfig.json marks a folder as a TypeScript project and holds two kinds of settings: top-level file selection (files, include, exclude, extends, references) and the compilerOptions block where the real behaviour lives — target (which JavaScript version to emit), module (the module format of the output), lib (which built-in type declarations are available, e.g. DOM), plus strictness, module resolution, and output paths. extends lets a project inherit a shared base config so teams do not repeat themselves.

Why the anatomy matters now: the defaults changed. TypeScript 6.0 (March 2026, the last JavaScript-based release before 7.0) flipped several long-standing defaults — strict is now true, module now defaults to esnext, and target now floats to the newest supported ECMAScript version (es2025 at 6.0's release) instead of the ancient old default. It also turned on noUncheckedSideEffectImports, changed types to an empty array (no auto-loading every @types package), defaults rootDir to the config's folder, and deprecated target: es5 (the floor is now ES2015). A green-field 6.0 project therefore starts strict and modern with almost no config — but a project relying on the old implicit defaults must now set them explicitly.

{
  "compilerOptions": {
    "strict": true,          // on by default in TS 6.0+
    "module": "esnext",      // new 6.0 default
    "target": "es2025",      // floating current-year default
    "lib": ["es2025", "dom"] // ambient type declarations
  },
  "include": ["src"]
}

Strictness flags philosophy

strict is not one check but a switch that turns on a family of stricter rules at once — noImplicitAny, strictNullChecks, strictFunctionTypes, strictBindCallApply, strictPropertyInitialization, noImplicitThis, alwaysStrict, and more. The philosophy is to make the type system actively prevent bugs rather than passively describe types. The two that carry the most weight are noImplicitAny (nothing may silently fall back to any — you either annotate or let inference do its job) and strictNullChecks (null and undefined become their own types, so the empty case must be handled instead of dereferenced blindly). strictNullChecks alone accounts for the large majority of what strict mode prevents.

Because strict is now the default in TS 6.0+, the interesting question flipped from "should I turn it on?" to "how do I keep it on in a legacy codebase?" The pragmatic answer is incremental: rather than flipping strict across a large project in a single commit, enable its sub-flags one at a time — noImplicitAny first (easiest, mostly missing annotations), strictNullChecks last (hardest, most valuable) — and clear errors file by file, using @ts-expect-error sparingly as a temporary, searchable marker. Understanding that strict is a bundle of independently-toggleable flags, not a single boolean, is what separates a rehearsed answer from a real one.

{
  "compilerOptions": {
    "strict": true,            // enables the whole family below
    "noImplicitAny": true,     // annotate, or let inference type it
    "strictNullChecks": true,  // the highest-value flag
    "strictFunctionTypes": true
  }
}

Interview questions

Most-asked setup and configuration questions in Indian interviews — from service-company screening rounds to product-company deep dives; TS 6.0 and 7.0 currency is now a common filter:

Beginner What is TypeScript and how does it relate to JavaScript?

A superset of JavaScript that adds an optional static type layer. A checker verifies types before the code runs, then the types are stripped and the output is plain JavaScript — so any valid JS is valid TS, and TypeScript adds no runtime type system of its own.

Beginner Can a browser or Node run a .ts file directly?

No. TypeScript must be compiled to JavaScript first — by tsc, or by a bundler/loader such as esbuild, swc, or ts-node. The types are erased in the process; runtimes only ever see JavaScript.

Beginner What does the tsc compiler do, and does it stop on type errors?

It type-checks your files and emits JavaScript, down-levelling newer syntax to the configured target. By default it still emits output even when type errors exist — set noEmitOnError (or run tsc --noEmit in CI) to make errors actually block the build.

Beginner What is a tsconfig.json?

The file that marks a folder as a TypeScript project. It holds file selection (include/exclude/files/extends) and a compilerOptions block controlling target, module, lib, strictness, module resolution, and output paths.

Beginner What does the strict flag do?

It switches on a whole family of strict checks (noImplicitAny, strictNullChecks, and more) in one line. It is the recommended baseline and, since TypeScript 6.0, the default for new projects.

Intermediate Which defaults did TypeScript 6.0 change?

strict became true, module defaults to esnext, and target now floats to the newest supported ES version (es2025 at release). It also enabled noUncheckedSideEffectImports, set types to [], defaulted rootDir to the config folder, and deprecated target: es5 (floor is ES2015). 6.0 was also the last JavaScript-based release.

Intermediate noImplicitAny vs strictNullChecks — what does each catch?

noImplicitAny flags anything that would silently become any (usually a missing parameter or return annotation). strictNullChecks makes null and undefined distinct types you must handle explicitly — the single highest-value strict flag, since most strict-caught bugs are null/undefined mistakes.

Intermediate What do target, module and lib each control?

target is the ECMAScript version tsc emits and down-levels to; module is the module format of the emitted code (esnext, nodenext, commonjs…); lib is which ambient type declarations are available at author time (e.g. dom, es2025). They are set independently.

Intermediate What is structural typing, and why does it matter here?

TypeScript compares types by shape, not by name — if an object has the required members it is assignable, regardless of the declared class or interface. It is why types from different libraries interoperate and why "it has the right fields" is enough; a frequent conceptual screening question.

Intermediate What is TypeScript 7.0 and why is it a big deal?

A native rewrite of the compiler and language service in Go (Project Corsa), GA in July 2026, delivering roughly 8–12x faster builds and editor responsiveness while preserving 6.x type semantics. It replaces the old JavaScript-based tsc; being aware of it is now common interview currency.

Pro How do you migrate a large codebase to strict without a big-bang change?

Enable the sub-flags incrementally rather than flipping strict at once: noImplicitAny first (mostly missing annotations), strictNullChecks last (hardest, most valuable), fixing errors file by file. Use @ts-expect-error as a temporary, searchable marker and an extends-based base config to roll it out per package.

Pro Types are erased at compile time — what are the runtime implications?

There is zero runtime type enforcement. Data crossing a boundary (API responses, JSON, form input, env vars) is untyped at runtime no matter what your types claim, so you must validate it with runtime guards or a schema library such as Zod. Types guarantee internal consistency, not external shape.

Pro Your build produces JavaScript even though tsc reports type errors — why, and how do you enforce a clean build?

tsc emits by default; type errors are advisory to the emit step unless noEmitOnError is set. Worse, bundlers like esbuild and swc strip types without type-checking at all, so errors slip through entirely. The fix is a dedicated type-check gate — tsc --noEmit in CI (and/or noEmitOnError) — kept separate from the transpile step that produces your bundle.
