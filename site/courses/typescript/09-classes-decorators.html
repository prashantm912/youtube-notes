Module: Classes and Decorators

Classes are where TypeScript's type system meets real object-oriented code, and decorators are the language's newest, most misunderstood feature. This module maps both to the resources that get them right — the modern standard decorators first, the legacy design clearly flagged.

What this module covers

TypeScript classes layer static visibility, readonly, parameter properties, abstract members and interface conformance on top of ES2015 classes. Decorators — reborn in TypeScript 5.0 as the ECMAScript Stage 3 standard, with Symbol.metadata arriving in 5.2 — let you annotate and rewrite those classes declaratively. This module orients you on both and links the official sources for each, keeping the modern standard separate from the older experimentalDecorators design.

Topics

TS class features (parameter properties, abstract, implements, access modifiers)

TypeScript adds a static type layer over the ES2015 class. The access modifiers public, protected and private control visibility (public by default), readonly locks a field after construction, and parameter properties collapse "declare a field, then assign it in the constructor" into a single modifier on the constructor argument. An abstract class cannot be instantiated on its own and can declare abstract members that every concrete subclass must fill in, while implements asserts that a class structurally satisfies one or more interfaces.

Why it matters: this is where object-oriented interview questions concentrate, and where a subtle correctness trap hides. TypeScript's private is erased at compile time — the field is still reachable at runtime — whereas JavaScript's #private is genuinely enforced. Knowing which guarantees are real at runtime versus compile-time only is a common senior screen, and implements versus extends is the other perennial: implements is a type-only contract that inherits no code.

interface Named { readonly name: string; }

abstract class Animal implements Named {
  // parameter properties: declare + assign in one line
  constructor(public readonly name: string, protected legs: number) {}
  abstract speak(): string;          // every subclass must implement
}

class Dog extends Animal {
  speak() { return this.name + " says woof"; }
}

Decorators (5.0 standard) + decorator metadata (Symbol.metadata)

A decorator is a function you attach with @ to a class, method, accessor, field or auto-accessor to observe or replace it. TypeScript 5.0 ships the ECMAScript Stage 3 standard decorators, enabled by default with no experimentalDecorators flag. A standard decorator receives two arguments: the value being decorated and a typed context object — its kind, name, whether it is static or private, an addInitializer hook, and (since 5.2) a metadata bag. Returning a new value replaces the original.

Why it matters: decorator metadata (TypeScript 5.2, from the TC39 proposal) gives every decorator on a class a shared context.metadata object; once the class is defined that object is reachable as SomeClass[Symbol.metadata], the standardized successor to the old reflect-metadata trick used by dependency-injection and ORM frameworks. Note that the legacy Handbook "Decorators" page describes the older experimentalDecorators design — the standard decorators here live in the release notes. The catch that trips people up: standard decorators cannot decorate parameters and do not support emitDecoratorMetadata, so Angular, NestJS and TypeORM still run on the legacy mode.

const FIELDS = Symbol("fields");

function serialize(_value: undefined, ctx: ClassFieldDecoratorContext) {
  (ctx.metadata[FIELDS] ??= []).push(ctx.name);   // shared metadata bag
}

class User {
  @serialize id = 0;
  @serialize name = "";
}

// after the class is defined:
User[Symbol.metadata]?.[FIELDS];   // ["id", "name"]

Interview questions

Most-asked classes-and-decorators questions in Indian interviews — from OOP screening rounds to product-company deep dives on the new decorator standard:

Beginner What access modifiers does TypeScript give a class, and which applies by default?

public, protected and private, with public the default. public is accessible anywhere, protected within the class and its subclasses, private only inside the declaring class. All three are checked by the compiler and erased from the emitted JavaScript.

Beginner What are parameter properties?

Prefixing a constructor parameter with an access modifier or readonly declares the field and assigns it from the argument in one step — constructor(private id: number) replaces a separate field declaration plus this.id = id. It is pure TypeScript sugar that removes constructor boilerplate.

Beginner What is an abstract class and when would you use one?

A class marked abstract cannot be instantiated with new; it exists to be extended. It can hold fully implemented methods alongside abstract members that declare a signature but no body, which every concrete subclass must implement. Use it for a shared base that is deliberately incomplete on its own.

Beginner What is the difference between implements and extends?

extends inherits implementation from a single base class. implements is a compile-time conformance check that a class matches one or more interfaces; it copies no code, so you still write every member. A class can implement many interfaces but extend only one class.

Beginner What is a decorator, and do you need a special flag in TypeScript 5.0?

A decorator is a function applied with @ to a class or its members to observe or replace them. Since TypeScript 5.0 the ECMAScript Stage 3 standard decorators are built in and need no experimentalDecorators flag — earlier versions required it for the older experimental design.

Intermediate TypeScript's private versus JavaScript's #private field — what is the real difference?

private is a compile-time annotation: it is erased on output and the field is still present and reachable at runtime (via obj["x"] or JSON serialization). #name is a runtime-enforced hard private that exists in the emitted JavaScript and throws when accessed outside the class. Reach for # when you need real encapsulation, not just type-checking.

Intermediate How do standard (5.0) decorators differ from legacy experimentalDecorators?

Standard: on by default, no flag; the signature is (value, context); they apply to classes, methods, accessors, fields and auto-accessors; they cannot decorate parameters; and there is no emitDecoratorMetadata. Legacy: needs experimentalDecorators; different signatures such as (target, key, descriptor); supports parameter decorators and reflect-metadata emission. The two are not source-compatible.

Intermediate What does the context object of a standard method decorator contain?

kind (here "method"), name, static and private booleans, an access helper, an addInitializer(cb) hook that runs setup when an instance is created, and (5.2+) a metadata object. The decorator's first argument is the original method; return a new function to wrap or replace it.

Intermediate Is readonly enforced at runtime?

No. readonly is a compile-time check that a property is only assigned in its declaration or the constructor; it is erased from the output, so nothing prevents mutation at runtime. For a real runtime guarantee use Object.freeze or a #private field exposed without a setter.

Intermediate What is Symbol.metadata and how do decorators use it?

Every decorator on a class shares a context.metadata object. Decorators write to it — for example recording which fields are serializable — and once the class finishes evaluating that object is exposed as SomeClass[Symbol.metadata]. It is the standardized successor to reflect-metadata and underpins dependency injection, serialization and validation; Symbol.metadata often still needs a small polyfill today.

Pro When you stack several decorators, in what order do they run?

The decorator expressions are evaluated top-to-bottom to produce the functions, but the functions are then applied bottom-to-top — like function composition, so the decorator closest to the declaration runs innermost. Across a whole class, member decorators run before the class decorator, and addInitializer callbacks fire in a defined order when instances are constructed.

Pro Why can't standard decorators replace experimentalDecorators for Angular, NestJS or TypeORM yet?

Those frameworks depend on two things the standard design dropped: parameter decorators (for constructor injection) and emitDecoratorMetadata plus reflect-metadata to read design:type at runtime. Standard decorators support neither — metadata now flows through Symbol.metadata instead — so such projects keep experimentalDecorators: true until the frameworks migrate.

Pro How would you type a class decorator that returns a subclass adding members (a mixin), and what makes it type-check?

Constrain the decorated target to a constructor type — new (...args: any[]) => object — and return class extends target { ... }. Because the return type widens the class, callers see the added members. In standard decorators a class decorator receives (value, context) and may return a new constructor; the hard part is threading generics so the original instance type is preserved and the new members are added to, rather than erasing, the base type.
