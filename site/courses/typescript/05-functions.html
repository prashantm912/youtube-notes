Module: Functions

Functions are where types earn their keep — every call site is a contract the compiler can enforce. This module orients you on typing parameters and returns, overloads, and generic functions.

Knowledge Library → Courses → TypeScript → Functions

What this module covers

How to describe what a function accepts and returns, from simple parameter annotations to call signatures, overloads and generics. Get function typing right and the compiler guards every place the function is used — wrong argument types, missing arguments and mishandled return values all become compile errors.

Topics

Function type signatures, optional/default/rest params

A function type annotates each parameter and the return: (a: number, b: number) => number. Parameters can be optional (b?: number), have defaults (b = 10), or collect the rest into a typed array (...rest: number[]). Return types are usually inferred, but annotating them documents intent and catches mistakes at the definition rather than at the call site. void return means "ignore any returned value"; never means the function never returns normally (it throws or loops forever).

Why it matters: precise parameter typing is the difference between the compiler catching a bad call and a runtime crash in production.

function greet(name: string, greeting = "Hello", ...extra: string[]): string {
  return [greeting + ", " + name, ...extra].join(" ");
}

Overloads, this typing

When a function's return type depends on how it's called, overload signatures describe each valid shape above a single implementation: multiple declaration signatures, then one implementation signature that must be compatible with all of them (and isn't itself callable). Separately, TypeScript lets you type this as a fake first parameter (function(this: Window, ...)) so methods and callbacks that rely on a specific this are checked.

Why it matters: overloads model real library APIs (like document.createElement) precisely; understanding them explains a lot of the standard library's types.

function parse(x: string): string[];
function parse(x: number): number[];
function parse(x: string | number): unknown[] {
  return typeof x === "string" ? x.split("") : [x];
}

Generic functions, contextual typing

A generic function introduces a type parameter that links inputs to outputs: function first<T>(arr: T[]): T | undefined preserves the element type through the call. TypeScript infers the type argument from how the function is called, so you rarely write it explicitly. Contextual typing works the other way: when a function expression appears where its type is already known (a callback passed to map), TypeScript types the parameters for you from that context.

Interview questions

Most-asked TypeScript-function questions in Indian interviews:

Beginner How do you type a function's parameters and return value?

Annotate each parameter after a colon and the return after the parameter list: function add(a: number, b: number): number. The return type is often inferred and can be omitted, but annotating it catches mistakes at the definition.

Beginner Optional vs default parameters?

Optional (b?: number) may be omitted and is then undefined; its type includes undefined. A default (b = 10) supplies a value when omitted, so inside the function its type is just number. Optional params must come after required ones.

Beginner What does a void return type mean?

The caller should not rely on a returned value. A void-returning function type also accepts functions that do return something (the value is ignored) — which is why arr.forEach(() => arr.push(x)) type-checks.

Beginner How do you type rest parameters?

With an array type: function sum(...nums: number[]): number. All trailing arguments are collected into nums as a typed array.

Intermediate What are function overloads and when do you use them?

Multiple call signatures over one implementation, used when the return type depends on the argument types/count. You write several declaration signatures, then one implementation signature (compatible with all, not directly callable). Modern advice: prefer a single signature with unions/generics when it expresses the same thing more simply.

Intermediate How do you type this inside a function?

Declare a fake first parameter named this: function handler(this: HTMLButtonElement, e: Event). It's erased at compile time but lets the checker verify the function is called with the right this. Arrow functions capture this lexically, so they don't take a this parameter.

Intermediate What is a generic function and why use one?

A function with a type parameter that ties inputs to outputs, e.g. identity<T>(x: T): T. It preserves the caller's type instead of widening to any or a fixed type, giving type-safe reuse across many types.

Intermediate What is contextual typing?

When the type of a function expression is already known from its location (a callback argument whose parameter type is declared), TypeScript infers the parameter types from that context, so you don't annotate them: [1,2].map(n => n * 2) knows n is number.

Intermediate Difference between void and undefined as a return type?

undefined requires the function to actually return undefined (or nothing, in a body that falls off the end). void is looser: it means "return value not observed" and is assignable from functions returning any type — important for callback compatibility.

Pro When would a function return type be never, and how is it useful?

When it never returns normally — it always throws or infinite-loops (function fail(msg: string): never { throw new Error(msg) }). It's the key to exhaustiveness checks: in a switch over a union, a never-typed default branch fails to compile if a case was missed.

Pro Why can a () => void type accept a function that returns a value?

Because void as a return in a function type means the return is discarded, not that it must be absent — this "void return contextual typing" rule makes callbacks like array methods ergonomic. It only applies to the type; a directly-annotated (): void function body still can't return value.

Pro How does generic inference interact with parameter order and defaults?

TypeScript infers type arguments from the arguments passed, left to right, and uses generic defaults (<T = string>) only when inference produces nothing. You can also constrain inference (<T extends object>) and block unwanted widening with NoInfer<T> to make one parameter drive inference while another merely checks against it.
