Module: Feature Timeline and Practice

TypeScript has shipped features steadily for a decade, and in 2026 it changed engines. This module orients you on that evolution and on putting the whole language to work in a real codebase.

Knowledge Library → Courses → TypeScript → Feature Timeline and Practice

What this module covers

A tour of where the language's headline features came from — so you can read any codebase and know which era a pattern belongs to — plus the practical side: typing the DOM, and the strategy for migrating an existing JavaScript project to TypeScript without a rewrite.

Topics

Feature timeline 1.x to 7.x (6.0 default flips, 7.0 native Go compiler, import defer)

Each major line added defining features: 1.x brought generics and union types; 2.x strictNullChecks, never, and control-flow narrowing; 3.x tuples, unknown, and project references; 4.x variadic tuples, template literal types, and short-circuit assignment; and 5.x standard decorators, const type parameters, the satisfies operator, using declarations, and inferred type predicates. Then two landmark releases: TypeScript 6.0 (2026) modernized the defaults — strict, ESM modules and a modern target on by default, legacy options removed — and TypeScript 7.0 replaced the compiler with a native (Go) implementation roughly 10× faster, checking the same language. Recent additions like import defer continue to track the ECMAScript proposals.

Why it matters: knowing the timeline turns "why is this written this way?" into "that's the pre-5.0 idiom" — invaluable when reading older code and choosing modern replacements.

TypeScript in practice (with the DOM), migration strategy

In the browser, TypeScript ships DOM type definitions (lib.dom.d.ts): document.querySelector returns Element | null, event handlers get typed events, and you narrow or assert element types before use. Migrating an existing JavaScript codebase is incremental: add a tsconfig with allowJs, rename files to .ts a few at a time, lean on // @ts-check plus JSDoc for the files not yet converted, add types for third-party libraries from DefinitelyTyped, and turn on strict flags one at a time. The rule is a green build at every commit.

const btn = document.querySelector("button"); // HTMLButtonElement | null
btn?.addEventListener("click", (e) => {       // e is typed as MouseEvent
  console.log((e.target as HTMLButtonElement).textContent);
});

Interview questions

Most-asked TypeScript evolution and practice questions in Indian interviews:

Beginner What type does document.querySelector return, and why does it matter?

Element | null (or a specific element type when given a tag selector). The null forces you to handle the "not found" case — optional chaining or a guard — before using the element, preventing a common runtime crash.

Beginner How do you start adding TypeScript to an existing JavaScript project?

Install TypeScript, add a tsconfig with allowJs (and often checkJs), then rename files to .ts gradually. You don't have to convert everything at once — TS and JS files can coexist during migration.

Beginner Where do types for libraries like lodash come from?

Either bundled with the package, or from DefinitelyTyped as an @types/<name> package you install as a dev dependency. TypeScript resolves these automatically.

Intermediate How do @ts-check and JSDoc help during migration?

Adding // @ts-check at the top of a .js file makes the TypeScript compiler check it, and JSDoc comments (/** @param {string} name */) supply types without converting to .ts. It's type safety on plain JavaScript, easing the transition.

Intermediate Name a few features and which era they arrived in.

Generics/unions (1.x), strictNullChecks and control-flow narrowing (2.x), tuples/unknown/project-references (3.x), template literal types and variadic tuples (4.x), standard decorators, satisfies, const type parameters and using (5.x). Knowing this dates unfamiliar code.

Intermediate How do you safely handle a possibly-null DOM element?

Guard it (if (el) { ... }), use optional chaining (el?.focus()), or a non-null assertion (el!) only when you're certain it exists. Prefer guards/optional chaining; assertions silence the compiler and can crash if you're wrong.

Intermediate What changed in TypeScript 6.0?

The defaults were modernized: strict, ES modules and a current compilation target became the out-of-the-box behavior, and long-obsolete options (like target: es5 and outFile) were removed. New projects get sensible strict defaults without configuration.

Pro What is the TypeScript 7 native compiler and what does it change for teams?

A rewrite of the compiler and language service in Go (project "Corsa"), delivering roughly 10× faster type-checking, builds and editor responsiveness with lower memory — while checking the identical language. For teams it means faster CI and IDE feedback with no code changes; the semantics are unchanged.

Pro How would you plan a large JS-to-TS migration to keep the build green throughout?

Enable allowJs/checkJs with loose settings first; convert leaf modules before their dependents; add @types for dependencies; introduce strict flags one at a time (noImplicitAny, then strictNullChecks, …), fixing each wave; use // @ts-expect-error sparingly as a tracked follow-up item. Every step must compile so the team keeps shipping.

Pro When is a non-null assertion (!) acceptable, and what is the safer alternative?

Only when an invariant guarantees non-null that the compiler can't see (e.g. an element you just created and appended). The safer alternatives are a runtime guard that also narrows the type, or a small assertion helper that throws with a message — turning a silent wrong assumption into a clear error.

Pro Why run tsc separately from your bundler in CI?

Bundlers (esbuild/swc/Babel) transpile by erasing types without full type-checking, for speed. Type errors would slip through. Running tsc --noEmit as its own CI gate performs the actual checking, keeping the fast transpile for builds and the thorough check for correctness.
