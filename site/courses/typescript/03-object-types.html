Module: Object Types and Interfaces

TypeScript describes the shape of your data, not its class hierarchy. This module orients you on the two ways to name a shape, how to fine-tune its members, and the structural rule that decides what fits where — then hands you the best material to master each.

What this module covers

Object types are how you tell TypeScript what an object looks like: which properties it has, which are optional, and which cannot change. You will name shapes with interface and type, refine members with ? and readonly, model dynamic keys with index signatures, compose shapes by extending and merging, and finally understand the structural typing rule that decides assignability — including the excess property check that surprises almost everyone once. These fundamentals sit under every React prop, every API model, and nearly every TypeScript interview.

Topics

Interfaces vs type aliases, optional/readonly

Both interface and type can name an object shape, and for a plain object they are almost interchangeable. The differences that matter: an interface can be reopened and merged (declaration merging) and reads naturally with extends, while a type alias is a single fixed definition that can also name things an interface cannot — unions, tuples, mapped and conditional types. Inside either, ? marks a property optional (its type becomes T | undefined) and readonly forbids reassignment after construction.

Why it matters: "interface or type — which do you use and why?" is one of the most-asked TypeScript screening questions in India, and a good answer names the concrete trade-offs (merging, unions, error-message readability) rather than reciting "they're the same." Knowing that readonly is a compile-time-only, shallow guarantee is a frequent follow-up.

interface User {
  id: number;
  name?: string;          // optional -> string | undefined
  readonly role: string;  // cannot be reassigned after creation
}
type ID = string | number; // only a type alias can name a union

Index signatures, extending, declaration merging

When you do not know a shape's keys ahead of time — a dictionary, a lookup table — an index signature such as [key: string]: number types every property accessed by that key kind. Keys may be string, number, symbol, or template-string patterns, and any explicitly named property must be assignable to the index signature's value type. To compose shapes, an interface uses extends (and may extend several at once); the type equivalent is an intersection with &.

Why it matters: declaration merging is unique to interfaces — two interfaces with the same name in the same scope combine into one. That is the engine behind module augmentation (adding a property to a library's exported interface, or to the global Window), a common real-world task and a favorite "what does this do?" interview trap. It is also the strongest reason some teams default to type: accidental same-name merges are silent.

interface StringScores {
  [key: string]: number;   // any string key maps to a number
}
interface Animal { name: string; }
interface Dog extends Animal { breed: string; } // extends adds members

interface Animal { legs: number; } // merges into the first Animal

Structural typing, assignability, excess property checks

TypeScript uses structural typing: two types are compatible when their shapes match, not when they share a name or an explicit inheritance link. If a value has at least the members the target needs, it is assignable — so an object with extra properties still fits a smaller interface, and unrelated types with identical members are interchangeable. This is why you rarely need to name a type just to pass it around.

Why it matters: the one place TypeScript deliberately breaks pure structural typing is the excess property check. Assigning a fresh object literal directly to a typed target flags unknown properties (often a typo), even though the same object stored in a variable first would be accepted. Explaining that difference — literal vs widened variable — is a classic senior-level question, alongside how function-parameter assignability (bivariance under the default setting) can let unsound assignments through.

interface Point { x: number; y: number; }
function draw(p: Point) { /* ... */ }

const pt = { x: 1, y: 2, z: 9 };
draw(pt);                  // OK: structurally compatible, extra z ignored
draw({ x: 1, y: 2, z: 9 }); // Error: excess property check on the literal

Interview questions

Most-asked object-type questions in Indian interviews — from screening rounds where interface-vs-type is almost guaranteed, up to product-company deep dives on structural typing:

Beginner What is the difference between an interface and a type alias?

For an object shape they are nearly equivalent. Key differences: interfaces support declaration merging (same-name interfaces combine) and read naturally with extends; type aliases are a single definition but can also name unions, tuples, mapped and conditional types, which interfaces cannot. Practical guidance: use interface for public object shapes and inheritance, type when you need a union or a computed type. This is the single most-asked TypeScript interview question.

Beginner What do the ? and readonly modifiers do on a property?

? makes a property optional — it may be absent, and its type widens to include undefined. readonly allows a property to be set at creation but blocks reassignment afterward. Both are compile-time only; they vanish in the emitted JavaScript.

Beginner What is an index signature and when would you use one?

An index signature such as [key: string]: number types objects whose property names are not known in advance — dictionaries, caches, lookup maps. Allowed key types are string, number, symbol, and template-string patterns. Any explicitly named property must be assignable to the index signature's value type.

Beginner How do you extend an interface, and what is the type-alias equivalent?

An interface uses extends (and can extend more than one interface at once). The type-alias equivalent is an intersection: type Dog = Animal & { breed: string }. Both produce a combined shape; extends gives clearer error messages and, on large hierarchies, slightly faster checking.

Beginner What does "TypeScript is structurally typed" mean?

Compatibility is decided by shape, not by name or explicit inheritance. If a value has at least the members a target type requires, it is assignable — even if the two types were declared independently. This contrasts with nominal typing in Java or C#, where types must share a declared name or hierarchy to be compatible.

Intermediate Why does assigning an object literal sometimes error when the same object in a variable does not?

Excess property checking. A fresh object literal assigned directly to a typed target is checked for unknown properties (usually a typo), so extra keys are rejected. Store the literal in a variable first and it is treated as a wider type, so structural typing accepts it — the extra properties are simply ignored. Escape hatches: add an index signature, use a type assertion, or fix the property name.

Intermediate What is declaration merging, and which declarations can merge?

Two declarations with the same name in the same scope combine into one. Interfaces merge (their members union together); namespaces merge; a namespace can also merge with a class, function, or enum. Type aliases do not merge — a duplicate name is an error. Merging powers module augmentation, for example adding a field to a library interface or to the global Window.

Intermediate Is readonly deep or shallow, and does it exist at runtime?

Shallow and compile-time only. readonly stops reassignment of that property, but if the property holds an object or array, that inner object's contents can still be mutated. It is erased during compilation, so nothing enforces it at runtime — use Object.freeze or the ReadonlyArray type when you need stronger guarantees.

Intermediate When would you prefer type over interface, and vice versa?

Prefer type when the shape is a union, tuple, or a mapped/conditional/template-literal computed type — interfaces cannot express those. Prefer interface for object shapes you expect others to extend, and for public API surfaces where declaration merging (intentional augmentation) is a feature. Many teams pick one as a default for consistency; the reasoned answer is what interviewers want.

Intermediate Can both interface and type describe a function type or be implemented by a class?

Yes to both. A call signature like { (x: number): string } works in an interface or a type alias. A class can implements either an interface or an object type alias. The practical split remains: only interfaces merge, and only type aliases can be unions — so a union type cannot be implemented by a class.

Pro Explain assignability for function parameters — why can it be unsound?

Under structural rules a function is assignable when its parameters are compatible. TypeScript checks method and plain function parameters bivariantly by default, which permits some unsound assignments (a handler taking a narrower type assigned where a wider one is expected). Turning on strictFunctionTypes makes standalone function-type parameters contravariant — safer — though methods stay bivariant for ergonomics. Interviewers ask this to see whether you understand why "it compiled" is not the same as "it is sound."

Pro How does declaration merging enable module augmentation, and what are the risks?

Because same-name interfaces merge, you can reopen a library's exported interface from your own file — declare module "lib" { interface Options { extra: boolean } } — or reach the global scope with declare global to add to Window. Risks: it is implicit and global, so an accidental same-name interface silently changes a type elsewhere, and augmentations scattered across files make the true shape hard to see. This silent-merge hazard is a key argument for defaulting to type.

Pro Is there a real performance difference between interface extends and type intersection?

Yes, in large codebases. An interface built with extends is cached by the compiler as a single flattened, named shape, so repeated checks are cheap. An intersection built with & is recomputed and can produce harder-to-read errors, and pathological intersections slow type checking. For object inheritance the guidance is to prefer extends; the difference is negligible for small shapes but measurable at scale.
