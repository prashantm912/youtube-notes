Module: Integration Testing with Testcontainers

An in-memory H2 will happily pass tests your production Postgres would reject. Testcontainers boots the real database, broker or cache in Docker for the length of a test and throws it away afterwards — and interviewers probe exactly why that fidelity is worth the startup cost.

Knowledge Library → Courses → Backend Testing → Integration Testing with Testcontainers

What this module covers

Testcontainers is a library that gives a test a throwaway, Docker-backed instance of whatever it really depends on — PostgreSQL, Kafka, Redis, Elasticsearch, even a browser. This module orients you on the case for testing against real dependencies instead of H2 or mocks, how to create containers (GenericContainer and the typed module containers), how JUnit 5 manages their lifecycle, and the Spring Boot 3.1+ integration (@ServiceConnection) that wires a running container into your application context with almost no code. It builds directly on the previous module's Spring integration testing.

Topics

Why real dependencies, not H2 or mocks

An in-memory database is a different database. H2 in "PostgreSQL mode" still diverges on JSON/array columns, native queries, upserts, sequences, isolation levels and vendor functions — so a green test suite can hide bugs that only surface in production. Testcontainers removes that gap by running the same engine and version you deploy, started fresh per run and discarded when the JVM exits, with no shared local install to drift or pollute.

Why it matters: "why not just use H2?" is the single most common Testcontainers interview question. The answer is fidelity — you test the real SQL dialect, the real driver, the real broker semantics — traded against a few seconds of Docker startup, which reuse and shared containers largely reclaim.

Creating containers: GenericContainer and the module containers

Two ways to declare a dependency. GenericContainer wraps any Docker image and is fully manual — you set exposed ports, environment variables and wait strategy yourself. The module containers (PostgreSQLContainer, KafkaContainer, MongoDBContainer, …) are typed subclasses that pre-configure the image, ports and readiness checks and expose helpers like getJdbcUrl(), getUsername(), getPassword(). Always parse the image with DockerImageName.parse(...) and pin an explicit tag so tests are reproducible.

@Container
static PostgreSQLContainer<?> postgres =
    new PostgreSQLContainer<>(DockerImageName.parse("postgres:16-alpine"));

// GenericContainer for anything without a dedicated module:
static GenericContainer<?> redis =
    new GenericContainer<>(DockerImageName.parse("redis:7-alpine"))
        .withExposedPorts(6379);

Container lifecycle and the JUnit 5 integration

@Testcontainers is the JUnit 5 extension that finds every @Container field and drives its start/stop. The lifecycle turns on the field's modifier: a static @Container starts once and is shared by all test methods in the class; an instance @Container restarts for every method — correct but slow. A container is not "ready" the instant it is running, so Testcontainers applies a wait strategy (port listening, a log-line regex, an HTTP probe or a Docker healthcheck) and only unblocks the test once the readiness check passes.

@Testcontainers
class OrderRepositoryTest {

    @Container
    static PostgreSQLContainer<?> db =
        new PostgreSQLContainer<>("postgres:16-alpine"); // started once, shared

    @Test void persists() {
        String url = db.getJdbcUrl(); // host + mapped port, resolved at runtime
        // ... connect and assert against real Postgres
    }
}

Spring Boot integration: @ServiceConnection and @DynamicPropertySource

Before Spring Boot 3.1 you glued a container to the context by hand: start it, then push its host/port/credentials into the Environment with @DynamicPropertySource. Boot 3.1 added @ServiceConnection — annotate a supported container and Boot detects its type, builds a ConnectionDetails bean, and configures the datasource (or Kafka, Redis, Mongo, RabbitMQ…) automatically. @DynamicPropertySource is still the escape hatch for anything without service-connection support, or for arbitrary property keys.

@SpringBootTest
@Testcontainers
class ApiIntegrationTest {

    @Container
    @ServiceConnection                       // Boot wires the DataSource for you
    static PostgreSQLContainer<?> db =
        new PostgreSQLContainer<>("postgres:16-alpine");

    // Fallback when there is no @ServiceConnection support:
    // @DynamicPropertySource
    // static void props(DynamicPropertyRegistry r) {
    //     r.add("spring.datasource.url", db::getJdbcUrl);
    // }
}

Sharing containers: the singleton pattern and reuse

Startup cost multiplies if every test class spins up its own container. Two remedies: the singleton-container pattern — start one container in a shared base class (or a static initializer) so the whole suite reuses it within a single run — and reusable containers, an experimental opt-in (withReuse(true) plus testcontainers.reuse.enable=true) that leaves the container running between runs so repeated local test cycles skip startup entirely. Reuse is for local speed, not CI, where each pipeline should start clean; Ryuk, the sidecar reaper, otherwise tears containers down when the JVM ends.

Interview questions

Most-asked Testcontainers questions in Indian interviews — a staple for backend and Spring Boot roles at product companies:

Beginner What is Testcontainers and what problem does it solve?

A Java library that provides lightweight, throwaway Docker instances of real dependencies — databases, message brokers, caches, browsers — for the duration of a test. It solves the fidelity vs. convenience trade-off: you test against the actual engine you run in production instead of an in-memory stand-in or a shared, drift-prone local install, and the container is created and destroyed automatically.

Beginner Why use a real database in tests instead of H2?

H2 is a different database. Even in "PostgreSQL compatibility mode" it diverges on JSON/array types, native queries, sequences, upserts, isolation levels and vendor functions — so tests can pass on H2 while the same SQL fails on production Postgres. Testcontainers runs the exact engine and version you deploy, closing that gap. The cost is a few seconds of Docker startup, recovered via shared/reused containers.

Beginner What is the prerequisite for running Testcontainers?

A working Docker (or compatible) runtime the tests can reach — Docker Desktop, Colima, Podman, Testcontainers Cloud, or a Docker-in-Docker service in CI. If no Docker environment is available the containers cannot start; you can annotate tests to be skipped in that case rather than fail.

Beginner GenericContainer vs a module container like PostgreSQLContainer?

GenericContainer wraps any image but you configure ports, env vars and wait strategy manually. Module containers are typed subclasses that pre-set the image, ports and readiness check and add helpers such as getJdbcUrl()/getUsername(). Use the module when one exists; drop to GenericContainer for images without a dedicated module.

Intermediate What do @Testcontainers and @Container do, and how does static vs instance change behaviour?

@Testcontainers is the JUnit 5 extension that manages every @Container field's lifecycle. A static @Container starts once and is shared across all test methods in the class; an instance (non-static) @Container restarts for each method — isolated but much slower. Static is the usual choice for read-heavy or transactionally-rolled-back tests.

Intermediate How do you connect the application to a container's random port?

Never hardcode ports — Testcontainers maps the container port to a random free host port to avoid clashes. Read it at runtime via getMappedPort()/getHost() or module helpers like getJdbcUrl(), then feed those into config. In Spring Boot use @ServiceConnection or @DynamicPropertySource so the values reach the Environment before the context starts.

Intermediate What is @ServiceConnection and how does it differ from @DynamicPropertySource?

@ServiceConnection (Spring Boot 3.1+) is put on a supported container; Boot detects its type, creates a ConnectionDetails bean and auto-configures the datasource/broker — zero property plumbing. @DynamicPropertySource is the older, manual approach: a static method that registers individual property keys (e.g. spring.datasource.url) from the container. Prefer @ServiceConnection; fall back to @DynamicPropertySource for unsupported containers or custom keys.

Intermediate Why are integration tests slow, and how does Spring context caching interact with containers?

Cost comes from container startup plus building the Spring context. Spring caches the context across test classes that share the same configuration, so keep configurations uniform to reuse it; combine that with a shared/static container so the database also starts once. Divergent @MockitoBean/@TestPropertySource setups fragment the cache and re-pay both costs.

Intermediate What is a wait strategy and why is it needed?

A running container is not necessarily ready to serve requests. A wait strategy blocks the test until readiness is proven — the default is a listening port, but you can wait on a log-message regex, an HTTP endpoint returning a status, or a Docker healthcheck. Module containers ship sensible defaults; custom images often need an explicit strategy to avoid flaky "connection refused" failures.

Pro Explain the singleton-container pattern and when you would choose it.

Start one container in a shared abstract base class (or a static block) so every test class that extends it reuses the same instance for the whole run, instead of each class starting its own. You choose it when many test classes hit the same dependency and tests are isolated by transaction rollback or unique data rather than a fresh database — it cuts total startup dramatically while a static @Container alone only shares within one class.

Pro What are reusable containers, and why keep them out of CI?

Reuse (withReuse(true) plus testcontainers.reuse.enable=true in ~/.testcontainers.properties) leaves a container running between test runs so repeated local cycles skip startup — Ryuk does not reap it. It is experimental and local-only: CI wants a clean, reproducible environment each run and ephemeral agents, so reuse there risks stale state and leaked containers. Use it to speed the inner dev loop, not the pipeline.

Pro What is the jdbc:tc: URL scheme and Ryuk?

The special JDBC URL (e.g. jdbc:tc:postgresql:16:///test) tells Testcontainers' JDBC driver proxy to start and manage a database container purely from the connection string — no explicit container code, useful for legacy setups. Ryuk is the sidecar "resource reaper" container Testcontainers launches to guarantee cleanup: it watches the test JVM and removes containers, networks and volumes when the session ends, even if the JVM crashes.
