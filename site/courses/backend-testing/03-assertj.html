Module: AssertJ and Assertion Libraries

JUnit's built-in assertions tell you a test failed; AssertJ tells you the story of why — one chainable assertThat that reads like the requirement it checks, with a failure message that names the exact difference. Knowing where it beats Hamcrest and plain assertEquals is a standard Java-interview probe.

What this module covers

AssertJ is the fluent assertion library most enterprise Java teams reach for on top of JUnit: a single assertThat(...) entry point that chains type-aware checks for objects, collections, exceptions and more, with rich, self-describing failure messages. This module orients you on the fluent API surface — collection, exception and object assertions, recursive comparison, soft assertions, and custom assertions — then places it against Hamcrest and JUnit's own Assertions so you can answer the comparison question interviewers love.

Topics

Fluent assertions and the assertThat entry point

Everything in AssertJ starts from one statically imported method — assertThat(actual) — which returns an assertion object typed to what you passed in. Because the return type knows whether it holds a String, a list, a number or your own class, your IDE autocompletes only the checks that make sense, and each check returns the same assertion so you can chain many conditions in one readable statement. The failure message is generated from the chain, so a broken test explains itself instead of just printing expected: true.

Why it matters: readability and diagnosis. A chained assertThat reads top-to-bottom like the requirement, and when it fails the message pinpoints the offending field or element — the two reasons teams standardise on AssertJ over ad-hoc assertEquals calls.

import static org.assertj.core.api.Assertions.assertThat;

assertThat(greeting)
    .isNotNull()
    .isInstanceOf(String.class)
    .startsWith("Hel")
    .endsWith("lo")
    .isEqualTo("Hello");

Object assertions and recursive comparison

For a single object you get isEqualTo, isSameAs, hasFieldOrPropertyWithValue, returns, and type checks like isInstanceOf. The standout is usingRecursiveComparison(): it compares two objects field by field, walking nested objects and collections, without needing equals() and hashCode() on every class — and you can ignoringFields(...) for volatile values like ids and timestamps. There are also type-specific assertions for Optional, Stream and the date/time types.

Why it matters: comparing DTOs, entities and API responses is the daily work of service and integration tests. Recursive comparison replaces brittle hand-written field-by-field assertEquals blocks with one intention-revealing call.

assertThat(actualUser)
    .usingRecursiveComparison()
    .ignoringFields("id", "createdAt")
    .isEqualTo(expectedUser);

Collection, array and map assertions

This is where AssertJ shines against plain JUnit. Iterable and array assertions include contains, containsExactly (same elements, same order), containsExactlyInAnyOrder, containsOnly, hasSize, allMatch/anyMatch, plus navigation helpers like extracting(...) to assert on one property across every element and filteredOn(...) to narrow the set first. Maps have their own family (containsEntry, containsKeys). Common assertions such as isEqualTo, isNull and isIn are shared across every type.

Why it matters: "assert this list has these three names in any order, ignoring the rest of each object" is a one-liner in AssertJ and a loop-plus-helper in JUnit. Condition objects let you name reusable predicates (is, has, allOf, anyOf) for expressive collection checks.

assertThat(users)
    .hasSize(3)
    .extracting(User::name)
    .containsExactlyInAnyOrder("Asha", "Ravi", "Meera");

Exception assertions

AssertJ turns exception testing into a fluent chain instead of a try/catch/fail dance. assertThatThrownBy(() -> ...) captures the thrown exception and lets you assert on its type, message and cause; assertThatExceptionOfType(X.class).isThrownBy(...) and the shortcuts assertThatIllegalArgumentException(), assertThatNullPointerException() read even better; and catchThrowable(...) hands you the exception object for deeper checks. You assert both that the right exception type fired and that its message and cause are correct.

Why it matters: negative-path testing is half of any solid test suite. Chained exception assertions verify type and message in one place, which is far clearer than JUnit's assertThrows plus a separate message assertion.

assertThatThrownBy(() -> service.load(id))
    .isInstanceOf(EntityNotFoundException.class)
    .hasMessageContaining(id)
    .hasNoCause();

Soft assertions

A normal ("hard") assertion stops the test at the first failure, so you fix one problem, re-run, and discover the next — slow feedback when several related fields are wrong. Soft assertions collect every failure and report them together at the end. Create a SoftAssertions, run all your softly.assertThat(...) checks, then call softly.assertAll() to fail once with the full list; assertSoftly(...) wraps that pattern, and the JUnit 5 SoftAssertionsExtension with @InjectSoftAssertions calls assertAll() for you so you can never forget it.

Why it matters: asserting many properties of one result object is common, and one report showing all mismatches beats five run-fix cycles. The classic bug — forgetting assertAll(), so failures are silently swallowed — is exactly what the JUnit 5 extension prevents.

SoftAssertions softly = new SoftAssertions();
softly.assertThat(order.total()).isEqualTo(250);
softly.assertThat(order.items()).hasSize(2);
softly.assertThat(order.status()).isEqualTo(PAID);
softly.assertAll(); // reports every failure at once

AssertJ vs Hamcrest vs JUnit assertions

Three styles solve the same problem differently. JUnit's own Assertions (assertEquals, assertTrue, assertThrows, assertAll) are always available and fine for simple checks, but argument order is easy to swap and messages are terse. Hamcrest reads as assertThat(actual, matcher) and composes matcher objects (is, hasItem, allOf) — powerful, and still used by MockMvc and REST Assured, but you must import the right matcher and discovery is poor. AssertJ puts the value first — assertThat(actual).matcher() — so autocomplete surfaces every relevant check and you chain without nesting. The JUnit team itself notes that third-party assertion libraries can replace its built-ins.

Why it matters: this exact comparison is one of the most common backend-testing interview questions in India. Be ready to contrast the three on API style, IDE discoverability, error-message quality, and where each still shows up in a real Spring codebase.

Custom assertions and the assertions generator

When you assert the same domain conditions repeatedly (order.isPaid(), invoice.hasNoOutstandingBalance()), promote them to a custom assertion. Extend AbstractAssert<SELF, ACTUAL>, add a static entry point, and write methods that check the actual value, call failWithMessage(...) on mismatch, and return this so your assertions chain like the built-ins. The AssertJ assertions generator can scaffold these classes for your types automatically. This is the natural counterpart to Hamcrest's custom matchers — the same idea in AssertJ's fluent, chainable style.

Why it matters: custom assertions make tests speak the domain language and shrink duplication, and being able to write one — and explain why it beats a helper method — signals senior-level test design in interviews.

public class OrderAssert extends AbstractAssert<OrderAssert, Order> {
  public OrderAssert(Order actual) { super(actual, OrderAssert.class); }
  public static OrderAssert assertThat(Order actual) { return new OrderAssert(actual); }
  public OrderAssert isPaid() {
    isNotNull();
    if (!actual.isPaid()) failWithMessage("Expected order to be paid but was %s", actual.status());
    return this;
  }
}

Interview questions

Most-asked AssertJ and assertion-library questions in Indian backend interviews — the AssertJ-vs-Hamcrest-vs-JUnit comparison is a near-guaranteed screening probe:

Beginner What is AssertJ and why prefer it over JUnit's built-in assertions?

A fluent assertion library layered on top of JUnit. One statically imported assertThat(actual) returns a type-aware assertion object you chain checks on. Wins over plain assertEquals: IDE autocompletes only relevant checks, many conditions read in one statement, and failure messages describe the exact difference instead of just "expected true".

Beginner How do you start an AssertJ assertion, and what do you import?

import static org.assertj.core.api.Assertions.assertThat; then assertThat(actual).isEqualTo(expected). The static Assertions class is the single entry point; a test can alternatively implement WithAssertions to inherit the same methods.

Beginner How do you assert a collection contains specific elements?

assertThat(list).contains(a, b) for "at least these", containsExactly(a, b, c) for same elements in the same order, containsExactlyInAnyOrder(...) for order-independent, containsOnly(...) for the exact set ignoring duplicates/order. Combine with hasSize(n) and extracting(...) to assert on a property of each element.

Beginner How do you assert that code throws an exception in AssertJ?

assertThatThrownBy(() -> code()).isInstanceOf(X.class).hasMessageContaining("..."), or assertThatExceptionOfType(X.class).isThrownBy(() -> code()), or type shortcuts like assertThatIllegalArgumentException(). catchThrowable(...) returns the exception so you can assert on it separately.

Intermediate What are soft assertions and when do you use them?

Hard assertions stop at the first failure; soft assertions collect all failures and report them together. Use SoftAssertions + softly.assertThat(...) then softly.assertAll(), or assertSoftly(...). Ideal when asserting many properties of one result — you see every mismatch at once instead of one per re-run.

Intermediate containsExactly vs containsExactlyInAnyOrder vs contains?

contains: the list has these elements (order-independent, extras allowed). containsExactlyInAnyOrder: exactly these elements, any order, no extras. containsExactly: exactly these elements in this precise order. Picking the wrong one either over-constrains (order flakiness) or under-constrains the test.

Intermediate How do you compare two objects field-by-field without relying on equals()?

assertThat(actual).usingRecursiveComparison().isEqualTo(expected) — it walks fields (and nested objects/collections) recursively regardless of whether equals() is defined. Use ignoringFields("id", "createdAt") for volatile values, or comparingOnlyFields(...). Perfect for DTO and entity comparisons.

Intermediate What does extracting() do?

It projects a property (or several) out of each element of a collection — or out of an object — so you assert on the projection instead of the whole. E.g. assertThat(users).extracting(User::name).containsExactly("Asha", "Ravi"). Keeps tests focused on what matters and avoids needing equals() on the full object.

Intermediate AssertJ vs Hamcrest — the key differences?

Hamcrest: assertThat(actual, matcher) composing matcher objects (is, hasItem, allOf); powerful composition but you must import the right matcher and discoverability is weak. AssertJ: assertThat(actual).check().check() — value first, so autocomplete lists every relevant assertion and you chain without nesting. AssertJ generally gives richer failure messages; Hamcrest still appears in MockMvc and REST Assured matchers.

Intermediate AssertJ vs Hamcrest vs JUnit assertions — when would you pick each?

JUnit Assertions: zero extra dependency, fine for simple equality/throws checks; weaker messages and swap-prone argument order. Hamcrest: expressive matcher composition, and you inherit it where MockMvc/REST Assured expect matchers. AssertJ: the default for readable, discoverable, richly-reported assertions on objects, collections and exceptions. Many teams use AssertJ for assertions while keeping Hamcrest matchers only where a framework requires them.

Pro How do you write a custom AssertJ assertion, and why not just a helper method?

Extend AbstractAssert<SELF, ACTUAL>, add a static entry point, and write methods that check actual, call failWithMessage(...) on mismatch, and return this to stay chainable; the assertions generator can scaffold them. Advantages over a helper: it chains with other assertions, produces AssertJ-style messages, and reads in the domain language (assertThat(order).isPaid().hasItems(2)) rather than as a plain boolean helper.

Pro What are common AssertJ pitfalls in a test suite?

Forgetting assertAll() so soft-assertion failures are silently swallowed (use the JUnit 5 SoftAssertionsExtension with @InjectSoftAssertions to avoid it); using isEqualTo on a collection when order shouldn't matter (use containsExactlyInAnyOrder); over-tight containsExactly causing order flakiness; asserting object identity with isEqualTo when you meant field equality (use usingRecursiveComparison); and leaving an assertion "dangling" — e.g. assertThat(x).isNotNull; without invoking a terminal check — which compiles but verifies nothing.
