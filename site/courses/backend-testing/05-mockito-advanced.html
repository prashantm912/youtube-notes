Module: Mockito Advanced

Basic stubbing gets you a green bar; capturing arguments, spying on real objects and taming static calls is where mocking either sharpens a test suite or quietly rots it — and interviewers know the difference.

Knowledge Library → Courses → Backend Testing → Mockito Advanced

What this module covers

Building on when/thenReturn and verify, this module covers the tools that make mocks expressive: ArgumentCaptor for inspecting what a collaborator was called with, Answer for computed return values, spies for wrapping real objects, and BDDMockito for given/when/then phrasing. It then tackles the hardest interview territory — mocking static and final methods in Mockito 5 with MockedStatic, and the anti-patterns that make over-mocked tests fragile.

Topics

ArgumentCaptor: asserting on what a collaborator received

Sometimes verifying that a method was called is not enough — you need to inspect the object passed to it, especially when the test constructs it deep inside the code under test. ArgumentCaptor registers a special matcher inside a verify(...) call, grabs the actual argument, and lets you assert on its fields afterwards. Use getValue() for the last call and getAllValues() when the method fired several times; @Captor declares one without the boilerplate.

Why it matters: it is the standard way to test "fire-and-forget" collaborators — event publishers, mailers, repositories — whose input never returns to the caller. Interviewers pair it with a rule you should state unprompted: prefer an ArgumentMatcher for stubbing, and reserve the captor for verification.

ArgumentCaptor<Email> captor = ArgumentCaptor.forClass(Email.class);
service.register("asha@example.com");
verify(mailer).send(captor.capture());     // capture at verification time
assertThat(captor.getValue().subject())    // now assert on the real object
    .isEqualTo("Welcome");

Custom return values with Answer

When a fixed thenReturn is too rigid, thenAnswer(Answer) (or doAnswer for void methods) computes the result from the actual invocation. Its single method receives an InvocationOnMock exposing the arguments and the mock itself — so you can echo an argument back, return different values by input, simulate a generated id, or invoke a passed-in callback. Mockito's AdditionalAnswers ships ready-made ones like returnsFirstArg().

Why it matters: it models stateful or input-dependent collaborators without hand-writing a fake. The tasteful boundary — asked in senior rounds — is that heavy logic inside an Answer is a smell; if the stub grows a brain, a hand-written fake or the real object reads better.

when(repo.save(any(User.class))).thenAnswer(inv -> {
    User u = inv.getArgument(0);   // read the real argument
    return u.withId(99L);          // simulate a DB-assigned id
});

Spies and partial mocks

A spy wraps a real instance: unstubbed methods run the genuine implementation while every interaction is still recorded for verification, and chosen methods can be overridden. It is the tool for legacy code you cannot easily inject into, or for stubbing one method of an otherwise-real object. Create one with spy(obj) or @Spy.

Why it matters: the interview trap is stubbing syntax. when(spy.get(0)).thenReturn(x) actually calls the real get(0) first (which may throw); the safe form is doReturn(x).when(spy).get(0), which never invokes the real method. Knowing why the doReturn family exists is a reliable senior signal — and a spy on a huge class usually means the class should be split.

List<String> spy = spy(new ArrayList<>());
doReturn(100).when(spy).size();  // safe: real size() never runs
spy.add("x");                    // real add() DOES run
verify(spy).add("x");

BDD-style stubbing with BDDMockito

BDDMockito renames the same behaviour to match Behaviour-Driven Development phrasing: the arrange step reads given(mock.call()).willReturn(x) instead of when(...).thenReturn(...), and the assert step reads then(mock).should().call() instead of verify(mock).call(). It aligns test code with the given/when/then structure teams already use in specs.

Why it matters: many Spring codebases adopt this style, so recognising given/willReturn/then/should is expected. For void methods the order flips to willThrow(ex).given(mock).voidCall() — a detail interviewers like to probe because it mirrors the doThrow().when() reordering.

given(repo.findById(1L)).willReturn(Optional.of(user));  // arrange
service.activate(1L);                                     // act
then(repo).should().save(user);                          // assert interaction

Mocking static and final methods

Since 5.0.0, plain mockito-core can mock static and final methods (the inline mock maker is the default; before that you added the separate mockito-inline artifact). mockStatic(Utility.class) returns a MockedStatic — a thread-local, scoped mock you stub with mocked.when(Utility::now).thenReturn(...). It must be closed, which is why it lives inside try-with-resources; leaving it open leaks the mock onto the thread and corrupts later tests.

Why it matters: this is the module's headline interview topic. Be ready to write the try-with-resources block, explain the thread-local scope, and — the mature answer — note that needing to mock a static call often signals a design you do not own; wrapping it behind an injectable interface is usually cleaner than reaching for mockStatic.

try (MockedStatic<Instant> mocked = mockStatic(Instant.class)) {
    mocked.when(Instant::now).thenReturn(FIXED);
    assertEquals(FIXED, Instant.now());
}   // static mock released here — scoped to this thread and block

Pitfalls and anti-patterns

Advanced features are also the fastest way to write bad tests. The recurring offenders: over-mocking (mocking value objects or types you do not own instead of using the real thing), leaving a MockedStatic unclosed so it bleeds into the next test, stubbing on a spy with when(...) and triggering the real method, and dead stubs that MockitoExtension's strict mode rejects with UnnecessaryStubbingException. Strict stubbing is a feature, not a nuisance — reach for lenient() only when a shared setup genuinely justifies it.

Why it matters: a senior answer is not "here is another Mockito method" but "here is when not to mock." Tests coupled to interaction detail break on every refactor; the discipline is to mock roles and boundaries, assert on observable behaviour, and let strict stubbing keep the suite honest.

Interview questions

Most-asked advanced Mockito questions in Indian interviews — ArgumentCaptor and static mocking dominate the product-company rounds:

Beginner What is ArgumentCaptor and when do you use it?

A helper that captures the actual argument passed to a mocked method so you can assert on it. Use it when the argument is built inside the code under test and never returned to the caller — e.g. an event or email object handed to a collaborator. Call captor.capture() inside verify(...), then read getValue().

Beginner What is a spy and how does it differ from a mock?

A mock is a bare shell — every method returns a default (null/0/empty) until stubbed. A spy wraps a real object: unstubbed methods run the real code while interactions are still recorded. Use a spy for partial mocking, when you need most of the real behaviour but want to override one or two methods.

Beginner How do you write BDD-style stubs in Mockito?

Use BDDMockito: arrange with given(mock.call()).willReturn(x) and assert with then(mock).should().call() — semantic aliases for when/thenReturn and verify. It aligns the test with the given/when/then structure of BDD specs.

Intermediate ArgumentCaptor vs ArgumentMatcher — when to use which?

Matchers (like any(), eq()) belong in stubbing and verification to select which calls apply. A captor is for inspecting the argument after the fact. Rule: don't drive stubbing with a captor — use a matcher to choose behaviour, and reserve the captor for verifying complex arguments you then assert on.

Intermediate thenReturn vs thenAnswer (and doAnswer) — what's the difference?

thenReturn gives a fixed, pre-computed value. thenAnswer(Answer) computes the result at call time from the InvocationOnMock — access to the real arguments and mock — so you can echo an argument, vary by input, or trigger a callback. doAnswer is the void-method / spy-safe equivalent.

Intermediate Why prefer doReturn().when(spy) over when(spy.method())?

On a spy, when(spy.method()) evaluates spy.method() for real before stubbing — which may run side effects or throw. doReturn(x).when(spy).method() stubs without invoking the real method. Same reason the doThrow/doAnswer/doNothing family exists.

Intermediate How do you mock a static method, and why the try-with-resources?

mockStatic(Utility.class) returns a MockedStatic you stub via mocked.when(Utility::call).thenReturn(...). It is a thread-local scoped mock that must be closed; try-with-resources guarantees close() so the static mock does not leak into subsequent tests on the same thread.

Intermediate mockito-core vs mockito-inline — what changed in Mockito 5?

Static and final mocking need the inline mock maker. Before Mockito 5 you added the separate mockito-inline dependency; from 5.0.0 the inline maker became the default in mockito-core, so plain mockito-core mocks statics and finals out of the box. mockito-inline is now effectively redundant.

Intermediate Can Mockito mock final classes and final methods? How?

Yes, with the inline mock maker (default since Mockito 5; opt-in via mockito-inline before that). It uses bytecode instrumentation rather than a subclass, so final classes/methods and statics become mockable — something classic Mockito could not do and PowerMock was once needed for.

Intermediate getValue() vs getAllValues() on ArgumentCaptor?

getValue() returns the last captured argument — fine when the method was called once. getAllValues() returns a List of every captured argument, used with verify(mock, times(n)) to assert on each call in order.

Pro What is UnnecessaryStubbingException and how do you handle it?

Strict stubbing (default under MockitoExtension) fails the test if a configured stub is never used, catching dead setup and typos in arguments. Fix it by removing the unused stub or correcting the matcher; only when a shared setup legitimately isn't hit by every test do you downgrade that stub with lenient(). Blanket-lenient defeats the purpose.

Pro What are the main anti-patterns of advanced mocking, and how do you avoid them?

Over-mocking (mocking value objects or types you don't own — use the real object or a thin adapter you do own), unclosed MockedStatic leaking across tests, spy-stubbing via when() that fires real methods, and asserting on interactions so tightly that any refactor breaks the test. The remedy is to mock roles at boundaries, prefer state/behaviour assertions over interaction counts, and treat "I must mock a static" as a hint to introduce an injectable seam instead.
