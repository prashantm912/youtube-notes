Module: JUnit 5 Fundamentals

What this module covers

JUnit 5 is the default testing framework for the Java backend — every Spring Boot starter ships it. This module orients you on the foundations examiners actually probe: where unit tests sit in the test pyramid, the three-part JUnit 5 architecture, how a @Test method is written and run, the fixture lifecycle annotations, and the assertion API (including assertThrows and assertAll). Depth for each lives in the linked official docs and guides.

Topics

The test pyramid — where unit tests belong

The test pyramid is a shape for a healthy suite: many fast, isolated unit tests at the base, fewer integration tests in the middle, and a handful of slow end-to-end tests at the top. The lower a test sits, the faster and more deterministic it is, so most of your coverage — and most of JUnit's job — lives at the unit layer.

Why it matters: the inverted shape (an "ice-cream cone" dominated by brittle UI tests) is the anti-pattern interviewers ask you to diagnose. Knowing why fast unit tests should outnumber the rest — feedback speed, isolation, cost of maintenance — is a standard opening question before any JUnit specifics.

JUnit 5 architecture — Platform, Jupiter, Vintage

JUnit 5 is not one library but three sub-projects: the JUnit Platform (the foundation that launches test engines and that IDEs and build tools talk to), JUnit Jupiter (the new programming and extension model — the @Test, assertions and annotations you write against), and JUnit Vintage (an engine that runs legacy JUnit 3 and 4 tests on the same Platform). The one aggregator dependency junit-jupiter pulls in the API, the params support and the engine.

Why it matters: "What are the three parts of JUnit 5?" and "How can old JUnit 4 tests still run?" are near-guaranteed screening questions. The Platform-plus-engines design is also why the same tool discovers Jupiter, Vintage and even third-party engines together.

<dependency>
  <groupId>org.junit.jupiter</groupId>
  <artifactId>junit-jupiter</artifactId>  <!-- API + params + engine -->
  <scope>test</scope>
</dependency>

Your first test — @Test and running it

A JUnit 5 test is a method annotated with @Test from org.junit.jupiter.api (note: not the old org.junit.Test). The class needs no public modifier and no inheritance; the method body arranges inputs, acts, and asserts. Tests run from the IDE, from the build tool (Maven Surefire / Gradle test task, both driven by the Platform), or from the standalone Console Launcher.

Why it matters: candidates who still import org.junit.Test or expect the class to extend TestCase reveal a JUnit 4 mental model. Knowing the Jupiter package and the several ways tests are launched is basic fluency.

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

class CalculatorTest {
    @Test
    void addsTwoNumbers() {
        assertEquals(4, new Calculator().add(2, 2));
    }
}

The test lifecycle — @BeforeEach, @AfterEach, @BeforeAll, @AfterAll

Fixture hooks keep tests independent. @BeforeEach and @AfterEach run around every test method — the place to build and tear down fresh state so tests never leak into each other. @BeforeAll and @AfterAll run once for the whole class (expensive one-time setup like a shared connection) and are static by default, because JUnit creates a new test instance per method before any test runs.

Why it matters: the "each vs all" distinction and the static requirement (relaxed only under @TestInstance(PER_CLASS)) are among the most common lifecycle questions. Misusing @BeforeAll for per-test state is a classic source of flaky, order-dependent tests.

@BeforeAll  static void startDb()   { /* once, before all tests */ }
@BeforeEach void freshFixture()      { /* before every test    */ }
@AfterEach  void rollback()          { /* after every test     */ }
@AfterAll   static void stopDb()     { /* once, after all tests */ }

Assertions — assertEquals, assertThrows, assertAll

Assertions are how a test states its expectation; a failed one throws and fails the test. The staples: assertEquals(expected, actual) (expected argument first — order matters for the failure message, and doubles need a delta), assertThrows(Type.class, executable) which returns the caught exception so you can assert on its message, and assertAll(...) which runs a group of assertions and reports every failure together instead of stopping at the first.

Why it matters: assertThrows replacing JUnit 4's @Test(expected=...), and assertAll versus a run of separate assertions, are frequent follow-ups. Getting the assertEquals argument order and the float-delta right separates careful candidates from the rest.

assertEquals(expected, actual);                    // value equality
assertThrows(IllegalArgumentException.class,
             () -> service.parse("bad"));           // exception expected
assertAll("user",
    () -> assertEquals("Asha", user.name()),
    () -> assertEquals(30, user.age()));            // all run; failures grouped

Interview questions

Java testing is a standard screening topic in Indian services and product interviews — these are the JUnit 5 fundamentals that come up first:

Beginner What is JUnit and why do we write unit tests?

JUnit is the de-facto unit-testing framework for Java: you write small, automated, repeatable tests that verify one unit of behaviour in isolation. They give fast feedback, catch regressions, document intent, and run in the build/CI so bugs surface before release rather than in production.

Beginner What is the test pyramid?

A guideline for suite composition: a large base of fast unit tests, fewer integration tests, and a small number of slow end-to-end tests. Lower tests are faster and more reliable, so they should dominate. The inverted shape — mostly UI/E2E tests — is the "ice-cream cone" anti-pattern that is slow and brittle to maintain.

Beginner What are the three sub-projects that make up JUnit 5?

JUnit Platform (launches engines; the interface for IDEs and build tools), JUnit Jupiter (the new programming and extension model you write tests against), and JUnit Vintage (an engine that runs old JUnit 3 and 4 tests on the Platform). Shorthand: JUnit 5 = Platform + Jupiter + Vintage.

Beginner Which annotation marks a test method, and which package is it in?

@Test from org.junit.jupiter.api. This differs from JUnit 4's org.junit.Test — a common trap. The Jupiter @Test takes no expected or timeout attributes; those became assertThrows and assertTimeout.

Beginner How do you run JUnit 5 tests?

From the IDE's runner, from the build tool (Maven Surefire or the Gradle test task, both talking to the JUnit Platform), or from the standalone Console Launcher. The Platform is what discovers and executes the engines in every case.

Intermediate Difference between @BeforeEach and @BeforeAll?

@BeforeEach runs before every test method — for fresh per-test fixtures. @BeforeAll runs once before all tests in the class — for expensive one-time setup. @BeforeAll/@AfterAll must be static by default; @BeforeEach/@AfterEach are instance methods.

Intermediate Why must @BeforeAll and @AfterAll be static?

By default JUnit creates a new test instance per test method (PER_METHOD lifecycle), so no single instance exists to own a "run once" method — it must be static. Switching the class to @TestInstance(Lifecycle.PER_CLASS) reuses one instance and lets these be non-static.

Intermediate How do you assert that a method throws an exception in JUnit 5?

assertThrows(ExpectedException.class, () -> codeUnderTest()). It fails if the exception is not thrown and, when it is, returns the exception object so you can further assert on its message or fields. It replaces JUnit 4's @Test(expected=...) and scopes the expectation to the exact call.

Intermediate What does assertAll do and when do you use it?

It groups several assertions so all of them execute and every failure is reported together, instead of stopping at the first failed one. Use it when checking multiple related properties of one object — you see all the mismatches in a single run rather than fixing them one at a time.

Intermediate What changed moving from JUnit 4 to JUnit 5?

Annotations renamed (@Before→@BeforeEach, @BeforeClass→@BeforeAll); @RunWith/@Rule replaced by the single @ExtendWith extension model; exception testing moved to assertThrows; and the monolith split into Platform + Jupiter + Vintage. Packages moved from org.junit to org.junit.jupiter.api.

Pro What is the default test instance lifecycle and why does it matter?

PER_METHOD: a fresh instance is created for each test, so instance fields cannot leak state between tests — that isolation is a feature. PER_CLASS reuses one instance (enabling non-static @BeforeAll and stateful @ParameterizedTest setups) but you must reset shared state yourself or risk order-dependent, flaky tests.

Pro In assertEquals(expected, actual), does argument order matter, and how do you compare doubles?

Order does not affect pass/fail but it drives the failure message ("expected X but was Y"), so swapping them produces misleading reports — always put the expected value first. Floating-point values need the three-argument form with a delta, e.g. assertEquals(0.3, x, 1e-9), because binary double arithmetic is not exact. An optional last argument (String or supplier) adds a custom message.

Pro How can a JUnit 4 test and a JUnit 5 test run in the same build?

Through the Platform's multi-engine design: JUnit Jupiter's engine runs the new tests and the JUnit Vintage engine runs the legacy JUnit 3/4 tests, both discovered and launched by the same Platform. Add the junit-vintage-engine dependency and old tests keep passing while you migrate incrementally.
