Module: Spring Boot Integration Testing

A unit test proves a class does its job; an integration test proves the wiring does. This module is about the two dials Spring Boot gives you — how much of the application to start, and what to replace with a mock — and why picking the smallest slice that still exercises the collaboration is the whole skill.

Knowledge Library → Courses → Backend Testing → Spring Boot Integration Testing

What this module covers

Spring Boot's test support lets you load an application context and exercise real collaborations instead of hand-wiring beans. The choices that matter are how much context to start — the whole application with @SpringBootTest or a focused slice like @WebMvcTest and @DataJpaTest — how the framework caches those contexts so the suite stays fast, how to swap a collaborator for a mock with @MockitoBean (the successor to @MockBean), and how to drive the web layer with MockMvc or WebTestClient. This deepens the Spring Boot track: slices versus full context is one of the most-asked interview topics for any Java backend role.

Topics

@SpringBootTest and the full application context

@SpringBootTest bootstraps the whole Spring context for a test — it searches upward for your @SpringBootApplication class and loads every bean, auto-configuration and property source, so collaborations run against real wiring rather than stubs. Its webEnvironment attribute decides how much web infrastructure comes with it: MOCK (the default) builds a mock servlet environment with no listening port, RANDOM_PORT and DEFINED_PORT start a real embedded server, and NONE skips the web layer entirely.

Why it matters: this is the broadest, most realistic — and slowest — test you can write. With a running server Spring Boot auto-configures a TestRestTemplate (or WebTestClient) pointed at the live port, so an end-to-end test can hit an actual HTTP endpoint. Reach for it when the behaviour you are verifying spans several layers; prefer a slice when it does not.

@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
class OrderApiIT {
    @Autowired TestRestTemplate rest;   // auto-pointed at the random port

    @Test
    void createsOrder() {
        ResponseEntity<OrderDto> res =
            rest.postForEntity("/orders", new OrderRequest("book-1", 2), OrderDto.class);
        assertThat(res.getStatusCode()).isEqualTo(HttpStatus.CREATED);
    }
}

Test slices versus the full context: @WebMvcTest and @DataJpaTest

A test slice loads only the beans one layer needs. @WebMvcTest stands up the Spring MVC infrastructure — controllers, @ControllerAdvice, converters, an auto-configured MockMvc — but no services or repositories, so you test request mapping, validation and serialization in isolation and supply the service layer as a mock. @DataJpaTest does the mirror image for persistence: it configures Spring Data JPA, entities and a TestEntityManager, points at an embedded in-memory database by default, and wraps each test in a transaction that rolls back automatically. Other slices (@JsonTest, @RestClientTest, @JdbcTest, @WebFluxTest) follow the same pattern.

Why it matters: a slice starts a fraction of the beans, so it is dramatically faster and its failures point at one layer. "Should I use @SpringBootTest or @WebMvcTest here?" is a genuine design question and a near-guaranteed interview prompt — the answer is the smallest slice that still exercises the collaboration under test, with full context reserved for genuinely cross-layer behaviour.

@WebMvcTest(OrderController.class)   // only the web layer
class OrderControllerTest {
    @Autowired MockMvc mvc;
    @MockitoBean OrderService orderService;   // service layer is a mock
}

@DataJpaTest                          // only the JPA layer, rolled back
class OrderRepositoryTest {
    @Autowired TestEntityManager em;
    @Autowired OrderRepository repository;
}

Context caching: why the suite is fast, or slow

Starting a Spring context is expensive, so the TestContext framework caches each one and reuses it across every test that asks for the same configuration. The cache key is the exact combination of config that loads the context — the @ContextConfiguration classes/locations, active profiles, context customizers (including @DynamicPropertySource and @MockitoBean definitions), and web setup. Tests that share a key share a single context; the cache defaults to holding 32 contexts with least-recently-used eviction, tunable via spring.test.context.cache.maxSize.

Why it matters: this explains the single biggest lever on suite speed. Every distinct combination of test properties, profiles or mocked beans forces a fresh context start, so a suite that varies configuration carelessly — or sprinkles @DirtiesContext, which evicts and rebuilds — can spend most of its time booting Spring instead of testing. Standardizing on a small number of context shapes keeps the cache warm and the build quick.

Replacing collaborators: @MockitoBean and @MockitoSpyBean

@MockitoBean replaces (or creates) a bean in the test's context with a Mockito mock, and @MockitoSpyBean wraps a real bean in a Mockito spy so unstubbed calls still hit the genuine implementation. These are the modern successors introduced in Spring Framework 6.2 and adopted by Spring Boot 3.4, which deprecated the older @MockBean and @SpyBean in favour of them — same idea, built on the framework's general bean-override mechanism. You stub and verify them with the ordinary Mockito API.

Why it matters: this is how a slice test isolates the layer under test — mock the service in a @WebMvcTest, mock the external client in a @SpringBootTest. Two things to remember: mockito bean definitions are part of the context cache key (so over-varying them fragments the cache, per the previous topic), and knowing that @MockBean is now deprecated is a current, frequently-probed detail. Prefer a spy over a mock only when you genuinely need the real behaviour for the un-stubbed methods.

@WebMvcTest(OrderController.class)
class OrderControllerTest {
    @Autowired MockMvc mvc;
    @MockitoBean OrderService orderService;   // successor to @MockBean

    @Test
    void returns404WhenMissing() throws Exception {
        when(orderService.find("x")).thenThrow(new OrderNotFound());
        mvc.perform(get("/orders/x")).andExpect(status().isNotFound());
    }
}

Driving the web layer: MockMvc and WebTestClient

MockMvc performs full Spring MVC request handling through mock request and response objects — no server, no socket — so a test can perform(get(...)) and assert on status, headers and JSON body with andExpect(...) (or the newer AssertJ-based MockMvcTester). WebTestClient is the fluent alternative: bind it to a controller, to the application context, or to a running server for a real end-to-end HTTP test, using one API across mock and live modes. Both let you test controllers alone (with the service mocked) or, under @SpringBootTest, the whole controller → service → repository → database path in a single test.

Why it matters: MockMvc is the default tool for controller and web-slice tests — fast, precise, and it exercises the real MVC machinery (mapping, validation, exception handling) that a plain method call skips. Choosing between MockMvc (mock dispatch), WebTestClient (fluent, works mock or live) and TestRestTemplate (real HTTP against a running server) is a common interview and code-review discussion.

mvc.perform(post("/orders")
        .contentType(MediaType.APPLICATION_JSON)
        .content("{\"sku\":\"book-1\",\"qty\":2}"))
   .andExpect(status().isCreated())
   .andExpect(jsonPath("$.id").exists());

// WebTestClient, same intent, fluent
client.post().uri("/orders").bodyValue(request)
      .exchange().expectStatus().isCreated()
      .expectBody().jsonPath("$.id").exists();

Interview questions

The Spring testing questions Indian interviewers actually ask — the slices-versus-full-context pair is close to guaranteed for any Spring Boot or backend role:

Beginner What does @SpringBootTest do?

It bootstraps the full Spring application context for a test — locating your @SpringBootApplication configuration and loading every bean, auto-configuration and property source — so the test runs against real wiring. It is the entry point for integration tests where several layers collaborate.

Beginner What are the webEnvironment options of @SpringBootTest?

MOCK (default): a mock servlet environment, no listening port — pair with MockMvc. RANDOM_PORT and DEFINED_PORT: start a real embedded server (random avoids port clashes in CI). NONE: load the context with no web environment at all.

Beginner What is a test slice? Name a few.

A slice loads only the beans and auto-configuration one layer needs, not the whole context. Common ones: @WebMvcTest (MVC/web), @DataJpaTest (JPA repositories), @JsonTest (serialization), @RestClientTest, @JdbcTest, @WebFluxTest. They are faster and their failures are localized.

Beginner What is MockMvc and why use it?

A test entry point that performs full Spring MVC request handling through mock request/response objects — no running server or socket. You call perform(get(...)) and assert on status, headers and JSON with andExpect(...). It exercises real MVC machinery (mapping, validation, exception handling) that a direct controller-method call would skip, and it is fast.

Intermediate @WebMvcTest vs @SpringBootTest for testing a controller — the key difference?

@WebMvcTest loads only the web layer (the target controller, advice, converters, an auto-configured MockMvc) and nothing else, so you mock the service with @MockitoBean and test mapping/validation/serialization in isolation — fast and focused. @SpringBootTest starts the entire context and runs the real service and repository — slower but genuinely end-to-end. Rule: pick the smallest slice that still exercises what you are testing.

Intermediate What does @DataJpaTest configure, and which database does it use by default?

It loads the JPA slice — entities, Spring Data repositories, a TestEntityManager — and by default replaces your datasource with an embedded in-memory database (H2). Each test method runs in a transaction that rolls back at the end, so tests stay isolated. Add @AutoConfigureTestDatabase(replace = NONE) (or Testcontainers) to test against the real database engine.

Intermediate @MockBean is deprecated — what replaces it, and why does the change matter?

@MockitoBean (and @MockitoSpyBean for spies), introduced in Spring Framework 6.2 and adopted by Spring Boot 3.4, which deprecated @MockBean/@SpyBean. They do the same job — replace or wrap a context bean — but are built on the framework's general bean-override mechanism rather than Boot-specific plumbing. Knowing the current annotation signals you are on a recent Boot version.

Intermediate @MockitoBean vs @MockitoSpyBean?

@MockitoBean installs a full Mockito mock — every method returns a default until stubbed, and the real bean is replaced. @MockitoSpyBean wraps the real bean in a spy: un-stubbed methods run the genuine implementation, so it preserves lifecycle and dependencies. Use the mock to isolate; use the spy only when you need the real behaviour for the methods you do not stub.

Intermediate MockMvc vs WebTestClient vs TestRestTemplate — when do you use each?

MockMvc: mock MVC dispatch, no server — the default for controller/web-slice tests. WebTestClient: a fluent client that binds to a controller, the context, or a running server, so one API spans mock and live modes (and it is the natural choice for WebFlux). TestRestTemplate: real HTTP against an embedded server started by @SpringBootTest(webEnvironment = RANDOM_PORT) — the most realistic and the slowest.

Intermediate Your integration suite is slow. What is the usual cause and the first thing you check?

Repeated context starts. Spring caches contexts by configuration and reuses them, but every distinct combination of properties, profiles or mocked beans is a new cache key and a fresh boot. Check for needless variety in test config and stray @DirtiesContext; converge on a few standard context shapes and prefer slices so more tests share a context.

Pro How exactly does context caching decide reuse, and how can @MockitoBean or @DirtiesContext hurt it?

The cache key is the full set of parameters that load the context: config classes/locations, active profiles, context customizers (which include @DynamicPropertySource and each @MockitoBean definition), the context loader and web setup. Change any of them and you get a different, separately-cached context (default cap 32, LRU eviction, tunable via spring.test.context.cache.maxSize). So classes that mock different beans each build their own context, and @DirtiesContext evicts and rebuilds — both multiply boots. Keep mock sets and config uniform to maximize reuse.

Pro How do you test the full controller → service → repository → database path in one test, and what are the trade-offs versus slices?

Use @SpringBootTest (with RANDOM_PORT plus TestRestTemplate/WebTestClient, or MockMvc in MOCK mode) so every real bean participates, ideally against a real database via Testcontainers rather than H2. Trade-offs: highest confidence and it catches wiring/serialization/transaction bugs a slice misses, but it is slow, boots a heavy context, and failures are harder to localize. Keep a few such tests for critical end-to-end flows and push the bulk of coverage down to fast slice and unit tests — the test pyramid.

Pro In a @WebMvcTest, a POST returns 403 even though your endpoint has no such rule. Why, and how do you fix it?

If Spring Security is on the classpath, @WebMvcTest auto-applies the security filter chain, and a state-changing request (POST/PUT/DELETE) is rejected for a missing CSRF token — surfacing as 403. Fix it by attaching a token with .with(csrf()) from SecurityMockMvcRequestPostProcessors (and seed a user with @WithMockUser when authentication is required). It is a mock-configuration gap, not a production bug.
