Module: Mockito Fundamentals

Mockito swaps a class's real collaborators for controllable stand-ins so one unit can be tested alone — and the vocabulary of mocks, stubs and spies is the first thing a backend interview checks you actually understand.

Knowledge Library → Courses → Backend Testing → Mockito Fundamentals

What this module covers

Mockito is the standard Java mocking framework: it manufactures fake collaborators for the class under test so a unit test stays fast and isolated from databases, HTTP and other services. This module covers the conceptual half — the taxonomy of test doubles that interviewers love to quiz (mock vs stub vs spy vs fake) — and the mechanical half — creating mocks with @Mock/@InjectMocks under MockitoExtension, stubbing responses with when/thenReturn/thenThrow, and proving behaviour with verify and argument matchers. It pairs directly with the JUnit 5 and AssertJ modules and underpins the later Spring Boot slice tests.

Topics

Test doubles: mock vs stub vs spy vs fake vs dummy

A "test double" is any stand-in that replaces a real collaborator during a test. Gerard Meszaros' taxonomy — popularised by Martin Fowler — names five: a dummy is filler passed to satisfy a signature but never used; a stub returns canned answers and supports state verification (assert on the result); a spy is a stub that also records how it was called (in Mockito, a real object whose methods run unless overridden); a mock is pre-programmed with expectations and supports behaviour verification (assert on the interaction); a fake has a real but shortcut implementation, like an in-memory database.

Why it matters: "what is the difference between a mock and a stub?" is the single most-asked opener on this topic in Indian interviews. The crisp answer is the verification style — a stub feeds data into the unit (state), a mock checks calls that come out of it (behaviour). Mockito blurs the line at the API level (its mock() object can both stub and be verified), so knowing the conceptual distinction is what marks a senior answer.

List<String> mock = mock(List.class);        // every method stubbed; returns defaults
List<String> spy  = spy(new ArrayList<>());   // REAL object; calls run unless overridden

Creating mocks: mock(), @Mock, @InjectMocks and MockitoExtension

There are two ways to make a mock: the static factory Mockito.mock(Type.class), or the @Mock annotation on a field. @InjectMocks goes on the class under test — Mockito builds a real instance and injects the @Mock fields into it, preferring constructor injection, then setters, then direct field assignment. Annotations do nothing on their own; something must initialise them. In JUnit 5 that is @ExtendWith(MockitoExtension.class) from the mockito-junit-jupiter artifact — the modern replacement for the JUnit 4 MockitoJUnitRunner and for a manual MockitoAnnotations.openMocks(this) call.

Why it matters: the @Mock + @InjectMocks + @ExtendWith(MockitoExtension.class) trio is the canonical skeleton of a Spring service unit test, so interviewers ask you to write it from memory and to explain how @InjectMocks chooses its injection strategy. The extension also turns on strict stubbing by default, which fails the test if you declare a stub that is never used — a quality gain worth mentioning.

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    @Mock OrderRepository repo;          // a mock collaborator
    @InjectMocks OrderService service;   // real object; repo injected into it

    @Test void placesOrder() { /* stub repo, call service, verify */ }
}

Stubbing: when/thenReturn, thenThrow and void methods

Stubbing programs a mock's response. when(mock.method(args)).thenReturn(value) defines what a call returns; thenThrow(...) makes it throw; chaining several thenReturn values feeds consecutive calls. An unstubbed method returns Mockito's safe defaults — null for objects, 0/false for primitives, and empty collections — never a real call. Void methods cannot sit inside when(...) because there is no return value to wrap, so you switch to the do* family: doThrow(...).when(mock).voidMethod(), doNothing(), or doAnswer(...).

Why it matters: the when/thenReturn form calls the mocked method eagerly to record the stubbing, which is exactly why it is unsafe on a spy (it would invoke the real method) and impossible for voids. Knowing to reach for doReturn/doThrow in those cases is a frequent senior follow-up, and getting the "what does an unstubbed method return?" question right shows you understand the difference between a mock and null.

when(repo.findById(1L)).thenReturn(Optional.of(user));
when(repo.findById(99L)).thenThrow(new EntityNotFoundException());
doThrow(new IllegalStateException()).when(auditLog).record(any()); // void method

Verifying interactions and argument matchers

Verification is how a mock proves behaviour. verify(mock).method(args) asserts the call happened; cardinality is controlled with times(n), never(), atLeastOnce(), and you can seal a mock with verifyNoMoreInteractions(). Argument matchers relax exact matching so verification and stubbing stay readable: any(), anyString(), eq(x), and custom argThat(predicate). The rule that catches everyone: once you use a matcher for one argument of a call, every argument must be a matcher — mixing a raw value with a matcher throws InvalidUseOfMatchersException, and the fix is to wrap the literal in eq(...).

Why it matters: verify + times is the mechanism behind behaviour verification (the thing that makes an object a mock, not just a stub), and the all-or-nothing matcher rule plus eq() is a near-guaranteed trick question. Being able to explain why verify(repo).update(42L, any()) fails — and how to fix it — is exactly the detail interviewers probe.

verify(repo, times(1)).save(any(Order.class));
verify(notifier, never()).send(anyString());
verify(repo).update(eq(42L), any(Order.class)); // eq() required once a matcher is used

Interview questions

Mockito questions dominate the unit-testing round of Indian backend interviews — and this one almost always opens with "mock vs stub", so the taxonomy is worth over-preparing:

Beginner What is a mock object and why use Mockito?

A mock is a fake stand-in for a real collaborator (repository, client, service) that you fully control. Mockito creates these so a unit test exercises one class in isolation — fast, deterministic, and free of databases or network. You program the mock's answers and assert how the class under test used it.

Beginner mock() vs @Mock — is there a difference?

Functionally the same object; only the style differs. Mockito.mock(Type.class) is an explicit factory call; @Mock on a field is declarative and needs an initialiser (MockitoExtension or openMocks). Annotations reduce boilerplate and give the mock a readable name in failure messages.

Beginner What does @InjectMocks do?

It marks the class under test: Mockito instantiates a real object and injects the @Mock (and @Spy) fields into it — trying constructor injection first, then property setters, then direct field assignment. It wires the collaborators so you do not build the object by hand.

Beginner How do you initialise Mockito annotations in JUnit 5?

Annotate the test class with @ExtendWith(MockitoExtension.class) (from mockito-junit-jupiter). It initialises every @Mock/@InjectMocks field before each test — replacing the JUnit 4 MockitoJUnitRunner and the manual MockitoAnnotations.openMocks(this) in @BeforeEach.

Beginner How do you stub a method to return a value?

when(mock.method(args)).thenReturn(value). Use thenThrow(...) to make it throw, and chain thenReturn(a).thenReturn(b) (or thenReturn(a, b)) to return different values on consecutive calls.

Intermediate What is the difference between a mock and a stub?

The verification style. A stub supplies canned data into the unit and you assert on the resulting state (state verification). A mock has expectations about calls out of the unit and you assert those interactions happened (behaviour verification). Mockito's mock() object can do both, but conceptually: stub = feed input, mock = check output calls.

Intermediate What is a spy and how does it differ from a mock?

A spy wraps a real object: unstubbed methods run the real implementation, and you override only the ones you choose. A mock has no real behaviour — every method is faked and returns defaults. Use a spy for partial mocking of a mostly-real object; prefer a plain mock otherwise, since spies invite calling real code by accident.

Intermediate How do you stub a void method, and why doesn't when() work?

when(mock.method()) needs a return value to wrap, and a void method has none — it also compiles as a statement, not an expression. Use the do* family instead: doThrow(ex).when(mock).save(x), doNothing(), or doAnswer(...), where the mocked call comes after .when(mock).

Intermediate What are argument matchers, and what is the all-or-none rule?

Matchers like any(), anyString(), eq(x) and argThat(...) relax exact argument matching. The rule: if any argument of a call uses a matcher, all must be matchers — mixing a raw literal with a matcher throws InvalidUseOfMatchersException. Wrap the literal in eq(...) to comply.

Intermediate How do you verify a method was called a specific number of times?

verify(mock, times(n)).method(args). Related modes: never() (i.e. times(0)), atLeastOnce(), atLeast(n), atMost(n), and verifyNoMoreInteractions(mock) to assert nothing else was called.

Intermediate What does a mock return for a method you never stubbed?

Safe defaults, never a real call: null for objects, 0/0.0 for numeric primitives, false for boolean, and empty collections for List/Set/Map. (The RETURNS_SMART_NULLS answer can be requested for a more debuggable placeholder.)

Pro When must you use doReturn instead of thenReturn?

when(x).thenReturn(...) evaluates x eagerly, so on a spy it calls the real method (side effects, exceptions). doReturn(v).when(spy).method() avoids that by not invoking the method during stubbing. Also prefer doReturn/doThrow for void methods and when a strict-stubbing setup would otherwise reject the eager call.

Pro What is strict stubbing, and why does MockitoExtension enforce it?

Strict stubbing (default under MockitoExtension, strictness STRICT_STUBS) fails the test if a declared stub is never used (UnnecessaryStubbingException) and flags argument mismatches at the point of use. It catches dead stubs and copy-paste errors early; you can relax it per test with @MockitoSettings(strictness = Strictness.LENIENT) or lenient() on a single stub.

Pro Give the full test-double taxonomy: dummy, stub, spy, mock, fake.

Dummy: passed to fill a parameter, never used. Stub: returns canned answers, enables state verification. Spy: a stub that also records calls (in Mockito, a real object partially overridden). Mock: pre-programmed with expectations, enables behaviour verification. Fake: a working but shortcut implementation (e.g. an in-memory database). Only mocks insist on behaviour verification; the rest typically use state verification. Mockito implements the mock/stub/spy end of this spectrum.
