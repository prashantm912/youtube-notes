Module: API and Contract Testing

A backend is only as reliable as the promises it keeps to the services around it: here you test an API from the outside with REST Assured, fake the dependencies it calls with WireMock, and pin down the contract between services with Pact and Spring Cloud Contract.

Knowledge Library → Courses → Backend Testing → API and Contract Testing

What this module covers

Three tools that test at the HTTP boundary. REST Assured writes black-box API tests in a readable request/response DSL; WireMock stands up a programmable fake of the external services your code calls; and contract testing (Pact, Spring Cloud Contract) captures the agreement between a consumer and a provider as an executable artifact so neither side breaks the other. The contract-testing part is awareness-level — know what problem each tool solves and when to reach for it.

Topics

REST Assured — testing REST APIs from Java

REST Assured is a Java DSL for HTTP API tests written in a readable given / when / then form: build a request, fire it, and assert on the status line, headers and JSON or XML body without hand-rolling an HTTP client or a JSON parser. It leans on Hamcrest matchers and JsonPath, so one fluent chain says "call this endpoint, expect 201, and this field equals that".

Why it matters: it is the default choice for black-box API and integration tests on the JVM, pairs cleanly with JUnit 5 and Spring Boot's random-port server, and its terse syntax is exactly what an interviewer expects you to recognise and read aloud.

given()
    .contentType("application/json")
    .body(newUser)
.when()
    .post("/users")
.then()
    .statusCode(201)
    .body("name", equalTo("Asha"));

WireMock — stubbing external services

Real services your code depends on — a payment gateway, a partner REST API — are slow, flaky and sometimes impossible to reach from a test. WireMock stands up an actual HTTP server that returns the canned responses you program, so an integration test exercises the real HTTP client and the real (de)serialization path against a predictable stub. You register request matchers (URL, method, headers, body) mapped to responses, and can later verify that the expected calls were made.

Why it matters: it isolates the unit under test from the network, makes failure-path testing (timeouts, 500s, malformed bodies) trivial to trigger, and its JUnit 5 extension wires the server lifecycle straight into your tests.

stubFor(get(urlPathEqualTo("/rates/USD"))
    .willReturn(okJson("{ \"rate\": 82.4 }")));

// point your client at wireMock.baseUrl(), call it, then:
verify(getRequestedFor(urlPathEqualTo("/rates/USD")));

Consumer-driven contract testing (Spring Cloud Contract / Pact)

When services talk over HTTP, the danger is that a provider changes its API and silently breaks a consumer. Contract testing captures the agreement between them as an executable artifact and checks both sides against it, without deploying the whole system together. In the consumer-driven style, the consumer's tests declare the exact requests it makes and the responses it needs; that generates a contract the provider must then satisfy.

Pact is code-first and consumer-driven, with a language-agnostic JSON pact plus an optional broker for sharing and deploy-gating. Spring Cloud Contract is Spring-native and usually provider-authored, generating both provider verification tests and WireMock-based consumer stubs from one contract. Know what each solves: this module is about recognising the approach, not mastering the tooling.

Interview questions

Contract and API testing questions surface in Indian backend and SDET interviews the moment you claim microservices experience — from screening to product-company deep dives:

Beginner What is REST Assured and why use it instead of writing your own HTTP client in tests?

A Java DSL for testing REST APIs. It builds requests and asserts on status, headers and JSON/XML body in a readable given/when/then chain using Hamcrest matchers and JsonPath — so you skip boilerplate HTTP-client and JSON-parsing code and keep tests declarative and short.

Beginner Explain the given/when/then structure of a REST Assured test.

given() configures the request (headers, body, params, auth), when() issues the HTTP call (get/post/put/delete), then() asserts on the response (statusCode, body matchers, headers). It reads like a BDD sentence; the legacy given/expect/when form still works for backward compatibility.

Beginner What is WireMock and what problem does it solve?

A programmable HTTP mock server. You register stubs (a request matcher mapped to a canned response) so tests can call an "external service" that is fast, deterministic and works offline. It lets you exercise the real HTTP client and serialization path without depending on the live dependency.

Beginner How is stubbing a service with WireMock different from mocking a Java object with Mockito?

Mockito replaces a Java collaborator in-process — no network, no serialization. WireMock replaces the service over real HTTP, so the client library, URL building, headers and JSON (de)serialization all run for real. Use Mockito to unit-test your logic; use WireMock to integration-test the HTTP boundary.

Intermediate How do you assert on a JSON response body in REST Assured?

Use body() with a JsonPath expression and a Hamcrest matcher: body("user.name", equalTo("Asha")), body("items.size()", is(3)). For richer checks, extract and deserialize into a POJO with .extract().as(User.class), or validate the payload against a JSON schema.

Intermediate How do you set up WireMock in a JUnit 5 test and match requests?

Add the Jupiter extension — @WireMockTest on the class (random port) or @RegisterExtension with WireMockExtension.newInstance() for finer control. Stub with stubFor(get(urlPathEqualTo("/x")).willReturn(...)), match on method/URL/headers/body, and confirm the interaction with verify(getRequestedFor(...)).

Intermediate How would you test how your code handles a slow or failing dependency?

WireMock can return error statuses and inject delays or faults: willReturn(aResponse().withStatus(500)), withFixedDelay(...), withFault(...). That lets you assert your client's timeout, retry and fallback behaviour deterministically — the failure paths you can rarely trigger against a live service.

Intermediate What is consumer-driven contract testing and what does it prevent?

The consumer's tests define the exact interactions (requests plus expected responses) it relies on; that becomes a contract the provider is verified against. It catches provider changes that would break real consumers, without spinning up both services together in a shared end-to-end environment.

Intermediate Walk through how Pact works end to end.

The consumer test runs against a Pact mock provider and, on passing, generates a pact file (JSON of interactions). That pact is shared with the provider — often via a Pact Broker — whose verification replays each recorded request against the real provider and checks the response matches. Only the parts the consumer actually uses get tested.

Pro Spring Cloud Contract vs Pact — how do they differ?

Pact is code-first, consumer-driven and language-agnostic (JSON pacts), with a broker for sharing and can-i-deploy gating — strong for many independent, polyglot consumer teams. Spring Cloud Contract is Spring-native, usually producer-authored (contracts in Groovy/YAML in one shared repo), and generates both provider tests and WireMock-based consumer stubs. Choose Spring Cloud Contract when you live in the Spring ecosystem; Pact for broad consumer-driven, multi-language fleets.

Pro Where do contract tests sit relative to full end-to-end tests, and why prefer them?

Below end-to-end in the test pyramid: each side is tested independently against the shared contract, so there is no need to deploy the whole system together. That makes them fast, stable and able to gate deployments (can-i-deploy), whereas broad end-to-end tests are slow, flaky and pinpoint failures poorly. End-to-end still has a small role for a few critical smoke paths.

Pro How do you keep API and contract tests from breaking on non-deterministic fields like timestamps or generated ids?

Match by type or pattern, not exact value. In REST Assured use Hamcrest matchers (matchesPattern, greaterThan, notNullValue) with JsonPath rather than full-body equality. In Pact use matching rules — type matchers, regex, like()/term() — so the contract asserts shape, not literal values. This avoids brittle tests while still catching real breaking changes.
