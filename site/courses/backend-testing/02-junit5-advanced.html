Module: JUnit 5 Advanced

Anyone can write @Test and a lifecycle hook; feeding one method a hundred cases, organising a suite that reads like a spec, and plugging your own behaviour into the test lifecycle is where Jupiter stops being a runner and starts being a platform — and where interviewers separate users from authors.

Knowledge Library → Courses → Backend Testing → JUnit 5 Advanced

What this module covers

Beyond @Test and assertions, JUnit Jupiter offers a second layer of features that make suites expressive and extensible: parameterized tests that run one method over many inputs, @DisplayName/@Nested for readable structure, dynamic tests generated at runtime, assumptions and conditional/tagged execution for selective runs, the per-method vs per-class instance lifecycle, and — the crown jewel — the extension model that lets libraries like Mockito and Spring hook into the test lifecycle. Parameterized tests and the extension model are the two topics interviewers probe most, so they carry the most weight here.

Topics

Parameterized tests: one method, many cases

A @ParameterizedTest runs the same test method repeatedly, once per set of arguments supplied by a source. It kills copy-paste test methods and turns a table of cases into a table in code. The sources cover most needs: @ValueSource for a single-column literal array, @EnumSource for enum constants, @CsvSource/@CsvFileSource for multi-column rows, @MethodSource/@FieldSource for programmatically built Arguments, and @NullSource/@EmptySource for edge cases. It needs the junit-jupiter-params artifact on the classpath.

Why it matters: it is the single most-asked JUnit 5 topic after the basics. The reliable follow-up is the @MethodSource rule — the factory method must be static unless the class runs in PER_CLASS lifecycle — which links this topic to the instance-lifecycle one below.

@ParameterizedTest
@ValueSource(strings = {"racecar", "noon", "level"})
void detectsPalindromes(String word) {
    assertTrue(isPalindrome(word));
}

@ParameterizedTest
@MethodSource("cityData")                 // static factory unless PER_CLASS
void nameLength(String city, int len) {
    assertEquals(len, city.length());
}
static Stream<Arguments> cityData() {
    return Stream.of(arguments("delhi", 5), arguments("chennai", 7));
}

Readable structure: @DisplayName and @Nested

@DisplayName replaces the terse method name in reports with a full sentence (spaces, emoji, anything), and a @DisplayNameGeneration strategy or custom generator can apply a convention across a whole class. @Nested lets you group related tests in inner classes so the report reads as a hierarchy — "a stack, when new, is empty; when one element is pushed, is not empty". Each nested level can carry its own lifecycle hooks and its own display name.

Why it matters: interviewers use these to check whether you write tests as living documentation. The gotcha worth stating: a @Nested class must be a non-static inner class, so it shares the outer instance and can see its fields.

@DisplayName("A bank account")
class AccountTest {
    @Nested
    @DisplayName("when newly opened")
    class WhenNew {
        @Test
        @DisplayName("has a zero balance")
        void startsAtZero() { assertEquals(0, account.balance()); }
    }
}

Dynamic tests with @TestFactory

Where a parameterized test is declared at compile time, a dynamic test is generated at runtime. A @TestFactory method returns a Stream, Collection, Iterable or Iterator of DynamicNode (usually DynamicTest, or DynamicContainer for grouping), each pairing a display name with an executable. Use it when the number or shape of cases is only known at runtime — reading rows from a file, expanding a grid, or building a test tree from data.

Why it matters: the interview distinction is that dynamic tests are not full lifecycle tests — @BeforeEach and @AfterEach do not run around each generated case. Reach for @ParameterizedTest when a fixed source and full lifecycle suffice; reach for @TestFactory when the cases themselves are computed.

@TestFactory
Stream<DynamicTest> divisibleByThree() {
    return IntStream.of(3, 6, 9)
        .mapToObj(n -> dynamicTest("divides " + n,
            () -> assertEquals(0, n % 3)));
}

Assumptions, conditional and tagged execution

Three different ways to not run a test. Assumptions (assumeTrue, assumeFalse, assumingThat) abort a test mid-flight when a precondition fails — the result is "skipped", not "failed", so an environment that cannot satisfy the setup does not turn the suite red. Conditional annotations decide before the test starts: @EnabledOnOs, @EnabledOnJre/@EnabledForJreRange, @EnabledIfSystemProperty, @EnabledIfEnvironmentVariable, and the custom @EnabledIf. Tags (@Tag("slow")) label tests so the build can include or exclude whole groups.

Why it matters: this is how real suites stay green across machines and split fast unit tests from slow integration tests in CI. Know the assumption-vs-assertion contrast (skip vs fail) and that tag filtering is configured in the build tool (Surefire groups/excludedGroups, Gradle includeTags).

@Test
@Tag("integration")
@EnabledIfEnvironmentVariable(named = "ENV", matches = "ci")
void hitsRealDatabase() {
    assumeTrue(dbReachable());     // abort (skip) rather than fail if not
    // real assertions run only when the assumption holds
}

Test instance lifecycle: PER_METHOD vs PER_CLASS

By default JUnit creates a new instance of the test class for every test method (Lifecycle.PER_METHOD), so instance fields cannot leak state between tests — isolation by construction. Annotating the class @TestInstance(Lifecycle.PER_CLASS) switches to one instance reused across all methods, which unlocks two conveniences: non-static @BeforeAll/@AfterAll, and non-static @MethodSource factories. The trade-off is that you become responsible for resetting shared state.

Why it matters: this is the answer behind two common puzzles — "why must my @MethodSource method be static?" and "why is my instance field zero at the start of every test?". PER_CLASS is also the tidy way to write BDD-style single-instance test classes.

@TestInstance(Lifecycle.PER_CLASS)   // one instance for the whole class
class OrderTest {
    @BeforeAll                       // non-static now permitted
    void loadFixtures() { /* ... */ }
}

The extension model

JUnit 5 replaced JUnit 4's competing runners and rules with a single, composable extension model built on a design principle of "prefer extension points over features". An extension is a plain class implementing one or more callback interfaces — BeforeAllCallback, BeforeEachCallback, AfterEachCallback, ParameterResolver, TestExecutionExceptionHandler, ExecutionCondition, TestWatcher, InvocationInterceptor, and more — each representing a point in the test lifecycle where it can intervene. Register declaratively with @ExtendWith(MyExtension.class), programmatically with @RegisterExtension on a field, or automatically via the ServiceLoader mechanism.

Why it matters: this is how MockitoExtension, SpringExtension and Testcontainers plug in, so understanding it explains the whole ecosystem — and it is a favourite senior-round topic. Be ready to name a few extension points, contrast @ExtendWith with @RegisterExtension, and explain how an extension stores state in the ExtensionContext.Store.

public class TimingExtension
        implements BeforeTestExecutionCallback, AfterTestExecutionCallback {
    public void beforeTestExecution(ExtensionContext ctx) { /* record start */ }
    public void afterTestExecution(ExtensionContext ctx)  { /* log elapsed */ }
}

@ExtendWith(TimingExtension.class)     // declarative registration
class ServiceTest { /* ... */ }

Interview questions

Most-asked advanced JUnit 5 questions in Indian interviews — parameterized tests and the extension model dominate the product-company rounds:

Beginner What is a parameterized test and what problem does it solve?

A test method annotated @ParameterizedTest that JUnit runs once per set of arguments from a source, so one method covers many cases instead of copy-pasted near-duplicate tests. It needs the junit-jupiter-params dependency and at least one argument source such as @ValueSource or @MethodSource.

Beginner How do you supply data to a @ParameterizedTest? Name a few sources.

@ValueSource (a single-column array of literals), @EnumSource (enum constants), @CsvSource and @CsvFileSource (multi-column rows), @MethodSource and @FieldSource (programmatically built Arguments), and @NullSource/@EmptySource/@NullAndEmptySource for edge inputs. For anything custom you implement an ArgumentsProvider and point @ArgumentsSource at it.

Beginner What does @Nested do and why nest tests?

@Nested groups related tests inside a non-static inner class, producing a hierarchical report that reads like a specification ("a stack — when new — is empty"). Each level can have its own @BeforeEach/@DisplayName and, being a non-static inner class, it shares the outer test instance and its fields.

Beginner How do you give a test a readable, sentence-like name?

@DisplayName("...") on a method or class overrides the method name in reports and allows spaces and symbols. For a whole class you can apply a convention with @DisplayNameGeneration (e.g. the built-in ReplaceUnderscores) or a custom DisplayNameGenerator. An explicit @DisplayName always wins over a generator.

Intermediate Why must a @MethodSource factory be static, and when can it be non-static?

Under the default PER_METHOD lifecycle JUnit needs the arguments before it constructs a test instance, so the factory must be static. If you annotate the class @TestInstance(Lifecycle.PER_CLASS), a single instance exists for the class and the @MethodSource method may be non-static. This is the classic link between the parameterized-test and instance-lifecycle topics.

Intermediate Dynamic tests vs parameterized tests — when do you reach for @TestFactory?

Parameterized tests are declared at compile time from a fixed source and run with full per-test lifecycle. Dynamic tests are generated at runtime by a @TestFactory returning a stream/collection of DynamicTest, so the number and shape of cases can be computed — but @BeforeEach/@AfterEach do not wrap each generated case. Use @TestFactory only when the cases themselves must be built dynamically.

Intermediate What are assumptions and how do they differ from assertions and @Disabled?

An assumption (assumeTrue/assumeFalse/assumingThat) aborts a running test when a precondition is not met — the test is reported as skipped, not failed. An assertion failing marks the test failed. @Disabled switches a test off unconditionally at declaration. Assumptions are the tool for "only meaningful in some environments" without turning CI red.

Intermediate How do you run a test only on certain OSes, Java versions, or environments?

Conditional annotations evaluated before the test runs: @EnabledOnOs/@DisabledOnOs, @EnabledOnJre/@EnabledForJreRange, @EnabledIfSystemProperty, @EnabledIfEnvironmentVariable, and the custom @EnabledIf pointing at a boolean method. They are themselves implemented as ExecutionCondition extensions.

Intermediate What are @Tag annotations and how are tags used in a build?

@Tag("slow") labels a class or method. The build then includes or excludes groups: Maven Surefire via groups/excludedGroups, Gradle via includeTags/excludeTags, or the JUnit Platform launcher via tag filters. The standard use is separating fast unit tests from slow integration tests so CI can run them in different phases.

Pro Explain the JUnit 5 extension model and name a few extension points.

It is JUnit 5's single mechanism for adding behaviour, replacing JUnit 4 runners and rules. You implement callback interfaces — BeforeAllCallback, BeforeEachCallback, AfterEachCallback, ParameterResolver, TestExecutionExceptionHandler, ExecutionCondition, TestWatcher, InvocationInterceptor — each firing at a specific lifecycle juncture, and register with @ExtendWith, @RegisterExtension, or the ServiceLoader. It is how MockitoExtension and SpringExtension integrate.

Pro @ExtendWith vs @RegisterExtension — what's the difference?

@ExtendWith(X.class) is declarative — JUnit instantiates the extension and you cannot configure it. @RegisterExtension goes on a field holding an instance you create, so you can pass constructor arguments, configure it programmatically, and reference it from the test (e.g. a WireMock or Testcontainers extension you must call into). Use programmatic registration whenever the extension needs runtime configuration or you need a handle on it.

Pro How does an extension share state across callbacks, and in what order do multiple extensions run?

Extensions are stateless by contract, so per-test state lives in the ExtensionContext.Store keyed within a Namespace (JUnit may create a fresh extension instance and stores are scoped and hierarchical, child contexts inheriting from parents). Ordering is onion-like: for a given test, "before" callbacks run in registration order (outer to inner) and the matching "after" callbacks run in reverse (inner to outer), with user @BeforeEach/@AfterEach methods wrapped by the surrounding callbacks — so cleanup always unwinds in the opposite order to setup.
