Module: Test-Driven Development

TDD is a design technique wearing a testing costume: you write the failing test first to feel the shape of the code before you build it. The interview signal is whether you can tell a test of behaviour from a test of implementation — and defend the difference.

What this module covers

Test-Driven Development flips the usual order: a small failing test comes first, then the minimum code to pass it, then a cleanup pass — the red-green-refactor loop. This module orients you on that discipline (and where it earns its keep), then on the design habits that make the resulting tests worth having: testing observable behaviour rather than internals, structuring tests with Arrange-Act-Assert, naming them so a failure reads like a specification, and the F.I.R.S.T properties that keep a suite fast and trustworthy.

Topics

The red-green-refactor cycle

TDD repeats three tiny steps. Red: write one failing test for the next slice of behaviour — it fails because the code does not exist yet. Green: write the least code that makes it pass, even something crude. Refactor: with a green bar as a safety net, remove duplication and improve names without changing behaviour. Then loop. The step everyone drops is refactor — Fowler calls skipping it the most common way to get TDD wrong.

Why it matters: writing the test first forces you to use the API before you build it, so design pressure lands early and every line of production code is born covered. The loop stays deliberately small — minutes, not hours — so a red bar always points at the last thing you did.

@Test
void add_returnsSum_forTwoPositiveNumbers() {   // RED: Calculator does not exist yet
    assertThat(new Calculator().add(2, 3)).isEqualTo(5);
}
// GREEN: write just enough to pass  ->  int add(int a, int b) { return a + b; }
// REFACTOR: clean names and duplication; the test stays green

Test-first discipline, self-testing code, and when TDD pays off

The point of test-first is not the tests themselves but self-testing code: a suite you trust enough to change any line and let a red bar tell you if you broke something. That trust is what makes refactoring and continuous delivery possible — without it, every change is a gamble. TDD produces a lot of the fast unit tests that sit at the base of the test pyramid, where feedback is cheapest.

Why it matters: TDD is not a religion. It shines for logic with clear inputs and outputs, algorithms, parsers, and anything you will refactor repeatedly; it fights you on throwaway spikes, hard-to-specify UI, and code whose design you genuinely do not understand yet. Knowing where it helps — and saying so in an interview instead of reciting dogma — is the senior answer.

Testing behaviour, not implementation

A good test asserts what the unit does — its observable result or effect — not how it does it. Tests bound to internals (private methods, exact call sequences, over-specified mocks) shatter the moment you refactor, even when behaviour is unchanged; they punish exactly the cleanup TDD is supposed to enable. Fowler's own framing is that a unit test tests the behaviour of a unit, assuming everything else works.

Why it matters: this is the most-argued idea in modern testing and a favourite senior-interview probe. Ian Cooper's talk traces brittle suites back to testing the wrong thing — the fix is to test at the behaviour boundary (a public method, a use case) and treat the internals as free to change. Dan North's point that renaming "test" to "behaviour" clears up most confusion lives here too.

Structure and naming: Arrange-Act-Assert and Given-When-Then

Arrange-Act-Assert (Bill Wake's 3A) gives every test one shape: set up the world, perform a single action, assert one logical outcome. Given-When-Then is the same three phases in behaviour-driven language and reads naturally as a specification. Keeping to one action and one logical assertion per test means a failure names its own cause. A test's name is documentation: Osherove's UnitOfWork_StateUnderTest_ExpectedBehavior or the BDD given/when/then convention lets a red bar read like a sentence without opening the body.

@Test
void withdraw_reducesBalance_whenFundsAreSufficient() {
    Account account = new Account(100);            // Arrange (Given)
    account.withdraw(30);                          // Act    (When)
    assertThat(account.balance()).isEqualTo(70);   // Assert (Then)
}

The F.I.R.S.T properties of a good unit test

F.I.R.S.T names five properties that keep a suite worth running. Fast: milliseconds, so nobody hesitates to run thousands of them. Independent (Isolated): no order dependencies, no shared mutable state — each passes or fails alone. Repeatable: same result on any machine, any time, with no reliance on a live database, network or clock. Self-validating: a pass/fail assertion, never a human eyeballing output. Timely: written right before the code, which is what TDD gives you for free.

Why it matters: these are the yardstick interviewers use to separate real unit tests from slow, flaky integration tests wearing a unit-test label. A test that touches a real database is, by F.I.R.S.T, not a unit test — and that single observation answers a lot of follow-up questions about flakiness and build times.

Interview questions

Most-asked TDD questions in Indian interviews — service companies drill the cycle and F.I.R.S.T; product companies push hardest on behaviour vs implementation:

Beginner What is TDD, and what are the three steps of the cycle?

Test-Driven Development writes a failing test before the production code. The cycle is red-green-refactor: red — write a small failing test; green — write the minimum code to pass it; refactor — clean up duplication and names while the tests stay green. Repeat in small increments.

Beginner Why write the test before the code?

It forces you to design the API from the caller's side before implementing it, keeps every line covered, and gives a fast feedback loop — a fresh red bar points straight at the change you just made. It also stops you from writing code you do not actually need.

Beginner What is the Arrange-Act-Assert pattern?

Three phases in every test: Arrange the objects and inputs, Act by invoking the one method under test, Assert a single logical outcome. Given-When-Then is the same structure in behaviour-driven wording. It keeps tests readable and makes a failure point at one cause.

Beginner What does F.I.R.S.T stand for?

Fast, Independent (Isolated), Repeatable, Self-validating, Timely. Together they describe a unit test that runs in milliseconds, does not depend on other tests, gives the same result everywhere, decides pass/fail on its own, and is written just before the code.

Intermediate What does "test behaviour, not implementation" mean?

Assert the observable result or effect of a public entry point, not private methods or the exact internal call sequence. Behaviour tests survive refactoring; implementation tests break whenever you reshape internals even though nothing the caller sees has changed — which defeats the safety net TDD is meant to provide.

Intermediate How much code should you write to turn a red test green?

The minimum that makes it pass — even a hard-coded return at first. You add generality by writing the next failing test that the crude version cannot satisfy ("triangulation"). This keeps you from building unused code and guarantees each line exists to satisfy a real test.

Intermediate How should you name a unit test?

So the name states the requirement without reading the body. Common conventions: Osherove's method_expectedBehaviour_whenCondition (e.g. withdraw_throwsInsufficientFunds_whenBalanceTooLow) or the BDD given/when/then form. A failing test then reads like a one-line spec of what broke.

Intermediate When does TDD help, and when might it not?

It helps most for logic with clear inputs and outputs — algorithms, parsers, business rules, anything refactored often. It struggles on exploratory spikes where the design is unknown, hard-to-specify UI, and thin glue code. The mature stance is to apply it where feedback is cheap and skip it where the test would cost more than it protects.

Intermediate TDD vs BDD — how are they related?

Same red-green-refactor rhythm; BDD reframes "test" as "behaviour" and uses Given-When-Then so specifications read in business language and can involve non-developers. BDD tends to drive from the outside in at a feature level, TDD often works unit-first, but they are complementary, not competing.

Pro Your suite goes red on every refactor even though behaviour is unchanged. What is wrong and how do you fix it?

The tests are coupled to implementation — asserting on private state, calling internals directly, or over-specifying mock interactions (verifying calls that are incidental). Fix by testing through the public behaviour boundary, asserting outcomes rather than call sequences, and mocking only true external collaborators. Brittle tests usually mean too many mockist tests over-specifying "how".

Pro Solitary vs sociable tests — the classicist and mockist schools?

Solitary (mockist / London school) isolates the unit by replacing every collaborator with a test double. Sociable (classicist / Chicago school) lets the unit use real collaborators unless they are awkward (slow, non-deterministic). Mockist tests localise failures but couple to interactions; classicist tests are more refactor-friendly but a fault can fail several tests. Most codebases mix both deliberately.

Pro Is a test that hits a real database a good unit test, and is 100% coverage a good TDD goal?

No on both. By F.I.R.S.T a database-touching test is not a unit test — it is slower and less repeatable, so it belongs a level up in the pyramid (an integration test). And coverage is a byproduct of TDD, not a target: chasing 100% rewards tests that execute lines without asserting meaningful behaviour. Aim for tests that pin down behaviour; let coverage follow.
