Module: Caching, Messaging and Scheduling Starters

Three cross-cutting concerns Boot turns into a dependency plus an annotation: cache a method's result, hand work to a broker, or run a job on a timer. This module maps what each starter auto-configures — and points clearly at where the real depth lives.

What this module covers

Boot doesn't reinvent caching, messaging or scheduling — those live in the Spring Framework and its sibling projects (Spring for Apache Kafka, Spring AMQP, Spring JMS). What Boot adds is auto-configuration: put the right starter on the classpath, set a few properties, and a CacheManager, a KafkaTemplate, or a task scheduler appears ready to inject. This page is deliberately integration-focused — it explains the wiring Boot gives you and defers the deep protocol-level material to the dedicated tracks and linked resources.

Topics

The cache abstraction and the cache starter

Spring's cache abstraction is a thin, provider-neutral layer: annotate a method with @Cacheable and its return value is stored keyed by the arguments, so the second call with the same inputs skips the method body entirely. @CachePut always runs the method and refreshes the entry; @CacheEvict removes entries. @EnableCaching switches the machinery on. Because it is annotation- and proxy-based, the same code works whether the backing store is an in-memory map or a Redis cluster.

Why it matters: caching is one of the cheapest performance wins in a service, and interviewers probe whether you understand it is a proxy AOP feature (with all the self-invocation caveats that implies) rather than magic. Knowing the three core annotations and what @EnableCaching actually enables is baseline for a Boot role.

@Service
class BookService {
  @Cacheable("books")                 // second call with same isbn is served from cache
  Book findByIsbn(String isbn) { return slowLookup(isbn); }

  @CacheEvict(value = "books", key = "#book.isbn")
  void update(Book book) { repository.save(book); }
}

Cache providers: Simple, Caffeine and Redis

With spring-boot-starter-cache and no provider on the classpath, Boot falls back to the Simple provider — a ConcurrentHashMap-backed CacheManager, fine for a single instance and tests. Add Caffeine and you get a high-performance local cache with size limits and time-based eviction; add spring-boot-starter-data-redis and Boot auto-configures a RedisCacheManager so entries are shared across every instance of a horizontally-scaled service. Boot auto-detects the provider, and spring.cache.type lets you pin it explicitly.

Why it matters: "local vs distributed cache" is a standard design question. Caffeine is fast but per-node (each instance has its own copy, which can go stale); Redis is a network hop but shared and survives a restart. Picking the wrong one is a real production incident, so this trade-off comes up often.

Messaging starters: Kafka and RabbitMQ (AMQP)

Boot ships a starter per broker family. With spring-kafka present, a KafkaTemplate is auto-configured for producing and any @KafkaListener bean gets a default container factory for consuming — all driven by spring.kafka.* properties. The AMQP starter does the parallel job for RabbitMQ: a RabbitTemplate and ConnectionFactory for sending, and a SimpleRabbitListenerContainerFactory so @RabbitListener methods receive. You write the producer and consumer; Boot wires the connection, serializers and listener plumbing.

This is a survey by design: partitions and consumer-group rebalancing, exchange and routing-key topology, delivery guarantees and stream processing are the substance of a dedicated messaging track. Here the goal is to recognise what the starter hands you and how little boilerplate remains.

@Component
class OrderListener {
  @KafkaListener(topics = "orders", groupId = "billing")
  void handle(String message) { process(message); }   // container + factory are auto-configured
}

JMS and choosing a messaging technology

JMS is the Jakarta standard, and Boot auto-configures a JmsTemplate plus a listener container for @JmsListener when an ActiveMQ "Classic" or Artemis broker is on the classpath — the same send/receive shape as Kafka and AMQP, just over a different API. The three feel similar in Boot precisely because the auto-configuration normalises them; the differences that matter are in the brokers, not the annotations. Kafka is a partitioned, replayable log for high-throughput event streaming; RabbitMQ/AMQP excels at flexible routing through exchanges; JMS is the portable enterprise-standard choice, often over ActiveMQ.

Why it matters: senior interviews ask you to justify a broker, not just wire one. The honest answer is that Boot only auto-configures the plumbing — topology, error handling, dead-letter queues and delivery semantics remain your design decisions whichever technology you pick.

Scheduling and async execution: @Scheduled and @Async

Boot auto-configures the executor and scheduler infrastructure, then two Framework annotations do the work. @Scheduled (switched on by @EnableScheduling) runs a method on a timer — fixedRate starts runs at a set interval regardless of duration, fixedDelay waits after each run finishes, and cron takes a six-field expression for calendar-based jobs. @Async (switched on by @EnableAsync) hands a method call to a background thread so the caller returns immediately, optionally yielding a CompletableFuture.

Why it matters: both are proxy-based, so both share the self-invocation trap — calling an annotated method from within the same class bypasses the proxy and the behaviour silently does not apply. And the default scheduler is single-threaded, so one slow job can stall every other scheduled task. These two gotchas are among the most-asked follow-ups on the topic.

@Configuration
@EnableScheduling
class Jobs {
  @Scheduled(fixedDelay = 5000)          // 5000 ms after each run completes
  void poll() { checkQueue(); }

  @Scheduled(cron = "0 0 3 * * *")       // once a day, at hour 3
  void nightlyReport() { buildReport(); }
}

Interview questions

The caching, scheduling and messaging questions that recur in Indian Spring Boot interviews — screening rounds through product-company system-design follow-ups:

Beginner What does spring-boot-starter-cache give you, and how do you turn caching on?

The starter pulls in Spring's cache abstraction and auto-configures a CacheManager. You still enable it explicitly with @EnableCaching on a configuration class; without that annotation the cache annotations are inert. After enabling, @Cacheable/@CachePut/@CacheEvict start taking effect.

Beginner What do @Cacheable, @CachePut and @CacheEvict each do?

@Cacheable checks the cache first and skips the method on a hit. @CachePut always runs the method and stores (updates) the result — use it to refresh an entry. @CacheEvict removes one entry (or all, with allEntries=true) so stale data is dropped.

Beginner If you add no cache provider, which cache does Spring Boot use?

The "Simple" provider — a ConcurrentHashMap-backed ConcurrentMapCacheManager. It is in-memory, per-instance and unbounded, which makes it fine for tests and single-node apps but wrong for a scaled-out service, where you would move to Caffeine (local) or Redis (shared).

Beginner How do you make a method run on a schedule in Spring Boot?

Add @EnableScheduling to a configuration class, then annotate a no-argument method with @Scheduled, giving it a fixedRate, fixedDelay or cron attribute. Boot auto-configures the scheduler; you provide the timing and the method.

Intermediate @Scheduled fixedRate vs fixedDelay vs cron — when do you use each?

fixedRate starts a run every N milliseconds measured from the previous start — steady cadence, runs can overlap if slow. fixedDelay waits N milliseconds after the previous run finishes — no overlap, cadence drifts with duration. cron uses a six-field expression for calendar-based schedules (e.g. every day at a fixed hour).

Intermediate How does Boot choose a cache provider, and how do you force Caffeine or Redis?

Boot auto-detects in a fixed order (Generic, JCache, Hazelcast, Infinispan, Couchbase, Redis, Caffeine, Cache2k, then Simple as the fallback) based on what is on the classpath and configured. To pin it, set spring.cache.type=caffeine or redis — useful when several providers are present or you want to disable caching in a profile with none.

Intermediate What does @Async need to work, and why might it seem to do nothing?

It needs @EnableAsync, and the method should return void, Future or CompletableFuture. The classic "it ran synchronously" bug is self-invocation: calling the @Async method from another method of the same bean bypasses the Spring proxy, so the async advice never fires. Call it through an injected reference to another bean instead.

Intermediate Which messaging starters does Boot auto-configure, and what beans do you get?

Kafka (spring-kafka): a KafkaTemplate plus a default listener container factory for @KafkaListener. AMQP/RabbitMQ: a ConnectionFactory, RabbitTemplate and a container factory for @RabbitListener. JMS: a ConnectionFactory (ActiveMQ/Artemis), JmsTemplate and a factory for @JmsListener. Boot also covers Pulsar and RSocket. You supply properties; it supplies the plumbing.

Intermediate What is the common programming model across @KafkaListener, @RabbitListener and @JmsListener?

All three are message-driven POJOs: annotate a bean method, name the topic/queue, and a listener container (auto-configured by Boot) invokes it for each message, converting the payload to the method's parameter type. The uniform shape is deliberate — swapping brokers is mostly a starter and configuration change at the edges, not a rewrite of consumer logic.

Pro By default @Scheduled tasks share a single thread — what breaks, and how do you fix it?

Boot's default task scheduler is single-threaded, so all @Scheduled methods run on one thread — a long or hung job delays or skips every other scheduled task. Fix it by raising spring.task.scheduling.pool.size, defining your own ThreadPoolTaskScheduler/TaskScheduler bean, or (Boot 4) enabling virtual threads so a SimpleAsyncTaskScheduler is used. Also guard long jobs against overlap.

Pro What are the caveats of Spring's proxy-based caching?

Self-invocation: an internal call to a @Cacheable method in the same class skips the proxy and the cache. Keys: without a key SpEL expression, a SimpleKeyGenerator derives one from all parameters, so overloads or mutable args can collide or surprise you. Null handling: @Cacheable caches null results by default (control with unless/condition), and sync=true avoids a cache-stampede when many threads miss at once.

Pro Kafka vs RabbitMQ vs JMS — how do you choose, and what does "Boot only auto-configures the plumbing" mean in production?

Choose by semantics: Kafka for high-throughput, partitioned, replayable event streams and log-style consumption; RabbitMQ/AMQP for flexible routing (exchanges, routing keys) and per-message workflows; JMS for a portable Jakarta-standard broker in traditional enterprise stacks. Boot's auto-configuration only wires templates, connection factories and listener containers — partition/consumer-group strategy, exchange topology, delivery guarantees, retries and dead-letter handling are still your design, and that ownership is the real answer interviewers want.
