Module: Actuator

Actuator is the difference between an app you can start and one you can operate — it ships the health checks, endpoints, and probes that ops teams, load balancers, and Kubernetes actually query in production.

Knowledge Library → Courses → Spring Boot → Actuator

What this module covers

Add one starter and your application grows a management surface: HTTP (and JMX) endpoints that report health, expose configuration, and let operators manage a running process. This module orients you on enabling and exposing those endpoints, the health endpoint and its indicators and groups, the info endpoint, securing the management surface, writing your own endpoints, and the liveness/readiness probes that make Boot a first-class Kubernetes citizen. Depth lives in the linked official docs; the framing here is what interviews test.

Topics

Enabling Actuator and exposing endpoints

Adding spring-boot-starter-actuator registers a set of built-in endpoints (health, info, metrics, env, beans, mappings, loggers, threaddump, heapdump, shutdown, and more). Two independent controls decide what a caller can reach: exposure — which endpoints are published over a transport (management.endpoints.web.exposure.include/exclude) — and access — whether an exposed endpoint responds and at what level (management.endpoint.<id>.access, or the blanket management.endpoints.access.default). Keeping the two distinct is the point most candidates miss.

The defaults are deliberately conservative: over HTTP only health is exposed, and sensitive endpoints such as shutdown are not openly accessible. You opt in per environment — a little in production, more on an internal ops network.

# expose a curated set over HTTP; never blindly use "*" in prod
management.endpoints.web.exposure.include=health,info,metrics,prometheus
management.endpoints.web.exposure.exclude=env,beans

The health endpoint: indicators, groups, and info

/actuator/health aggregates the worst status of every registered HealthIndicator — Boot auto-configures indicators for the datasource, Redis, disk space, message brokers, and more, and you can add your own by implementing HealthIndicator. By default the response is just an overall status; management.endpoint.health.show-details (never, when-authorized, always) controls whether per-component detail is returned. Health groups bundle a subset of indicators under their own sub-path (for example a readiness group), which is exactly the machinery Kubernetes probes reuse.

The sibling /actuator/info endpoint surfaces static application facts — build version, git commit, arbitrary info.* properties — useful for confirming which artifact is actually deployed.

management.endpoint.health.show-details=when-authorized
management.endpoint.health.group.readiness.include=readinessState,db
info.app.name=orders-service
info.app.version=1.4.2

Securing Actuator and the management server

Because endpoints like env, heapdump, configprops, and shutdown expose or affect internals, the management surface needs the same care as any other. Three layers combine: expose narrowly, restrict access, and put authentication in front. With Spring Security on the classpath, a SecurityFilterChain can require a role for EndpointRequest.toAnyEndpoint() while leaving health open. You can also isolate the surface entirely — a separate management.server.port and a bound management.server.address (or a changed management.endpoints.web.base-path) keep operational endpoints off the public application port.

# run management endpoints on a separate, internal-only port
management.server.port=9001
management.server.address=127.0.0.1

Custom endpoints

When a built-in endpoint does not fit, annotate a bean with @Endpoint(id = "...") and expose operations with @ReadOperation (GET), @WriteOperation (POST), and @DeleteOperation (DELETE); the operations are automatically available over both HTTP and JMX. Transport-specific variants — @WebEndpoint and @JmxEndpoint — let you target one protocol. This is the sanctioned way to publish domain-specific operational actions (feature toggles, cache flushes, on-demand diagnostics) through the same secured, discoverable surface as the built-ins rather than as ad-hoc controllers.

@Component
@Endpoint(id = "features")
public class FeaturesEndpoint {
    @ReadOperation
    public Map<String, Boolean> features() { return toggles.snapshot(); }
}

Kubernetes readiness and liveness probes

Boot models two orthogonal availability states: liveness (is the internal state still valid, or is the process broken beyond recovery and in need of a restart) and readiness (can it accept traffic right now). With Actuator present, these are exposed as health groups at /actuator/health/liveness and /actuator/health/readiness — auto-enabled when Boot detects Kubernetes, or switched on with management.endpoint.health.probes.enabled. Application code can flip the state by publishing an AvailabilityChangeEvent. The interview trap: liveness must depend only on the process itself (a stuck deadlock), never on external dependencies — otherwise a transient database outage triggers a pointless restart storm, when failing readiness to drain traffic is the correct response.

management.endpoint.health.probes.enabled=true
management.endpoints.web.exposure.include=health
# probes then answer at /actuator/health/liveness and /actuator/health/readiness

Interview questions

Actuator recurs in Indian backend and DevOps interviews — health, probes, and securing the management surface are the reliable ones:

Beginner What is Spring Boot Actuator and what does it give you?

A module (spring-boot-starter-actuator) that adds production-ready monitoring and management endpoints — health, info, metrics, env, loggers, threaddump and more — over HTTP and JMX, without you writing them. It is the standard way to make a Boot app observable and operable in production.

Beginner After adding the starter, why can you only reach /actuator/health?

By default only the health endpoint is exposed over HTTP. Others exist but are not published until you list them under management.endpoints.web.exposure.include. The conservative default avoids leaking internals by accident.

Beginner What is the default base path, and how do you expose all endpoints?

Endpoints live under /actuator (so health is /actuator/health). management.endpoints.web.exposure.include=* exposes everything — convenient in dev, but a security risk in production, so prefer an explicit list.

Beginner Difference between exposing an endpoint and its access level?

Exposure decides which endpoints are published over a transport (...web.exposure.include/exclude). Access (management.endpoint.<id>.access or management.endpoints.access.default) decides whether an exposed endpoint actually responds, and at what level (none, read-only, unrestricted). Both must permit a call for it to succeed.

Intermediate How does /actuator/health compute its status, and what are health indicators?

It aggregates every registered HealthIndicator and returns the worst status (DOWN beats OUT_OF_SERVICE beats UP). Boot auto-configures indicators for datasource, disk space, Redis, brokers, etc.; you add custom ones by implementing HealthIndicator and returning a Health object.

Intermediate How do you show full health details, and why is it guarded?

management.endpoint.health.show-details takes never (default), when-authorized, or always. Per-component detail (database name, disk paths, validation queries) is sensitive, so it is hidden unless you opt in — typically when-authorized behind Spring Security.

Intermediate What is a health group and when do you use one?

A named subset of indicators exposed at its own sub-path, e.g. management.endpoint.health.group.readiness.include=... answers at /actuator/health/readiness. Groups let you assemble a purpose-built check — the mechanism the Kubernetes liveness/readiness probes are built on.

Intermediate How do you secure Actuator endpoints in production?

Layer three controls: expose only what you need, restrict access levels, and add authentication. With Spring Security, a SecurityFilterChain can require a role for EndpointRequest.toAnyEndpoint() while leaving health public. Optionally move the whole surface to a separate management.server.port bound to an internal address.

Intermediate Liveness vs readiness probe — what is the difference?

Liveness answers "is the process broken and unrecoverable?" — failing it makes Kubernetes restart the pod. Readiness answers "can it serve traffic now?" — failing it makes Kubernetes stop routing to the pod without restarting it. Boot exposes them at /actuator/health/liveness and /actuator/health/readiness.

Pro How do you build a custom Actuator endpoint?

Annotate a bean with @Endpoint(id = "...") and add @ReadOperation/@WriteOperation/@DeleteOperation methods (GET/POST/DELETE), automatically served over HTTP and JMX. Use @WebEndpoint or @JmxEndpoint to target a single transport. It reuses the same secured, discoverable infrastructure as built-in endpoints — unlike a hand-rolled controller.

Pro Why must a liveness probe not depend on external systems like the database?

Liveness failure triggers a restart, which cannot fix an external outage — so a database blip would restart every pod repeatedly (a crash/restart storm) while making nothing better. Liveness should reflect only internal, unrecoverable state; a downed dependency should fail readiness instead, draining traffic until it recovers.

Pro During graceful shutdown, what should happen to each probe, and why?

Readiness should flip to REFUSING_TRAFFIC first so the platform stops routing new requests while in-flight ones finish; liveness should stay valid so the orchestrator does not kill the pod mid-drain. Boot wires this into its graceful-shutdown and availability events, which is why draining works cleanly behind a load balancer.
