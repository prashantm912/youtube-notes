Module: Packaging and Deployment

Turning a Spring Boot app into something you can ship — a runnable jar, a cache-friendly image, or a native binary — is where build tooling, Docker and GraalVM meet, and where most deployment interview questions live.

Knowledge Library → Courses → Spring Boot → Packaging and Deployment

What this module covers

Packaging is the last mile: how Spring Boot bundles your code and its dependencies into an artifact, and how that artifact becomes a container or a standalone executable. This module covers the executable (fat) jar and the Maven/Gradle plugins that build it, layered jars for cache-friendly Docker images, Cloud Native Buildpacks and hand-written Dockerfiles for OCI images, and GraalVM native images produced through Spring's ahead-of-time processing. Layered jars, buildpacks and native images are the current hot topics — expect all three in interviews.

Topics

The executable (fat) jar and the build plugins

A Spring Boot executable jar (uber/fat jar) is one self-contained archive: your compiled classes under BOOT-INF/classes, every runtime dependency under BOOT-INF/lib, Boot's loader classes at the root, and an embedded server — launched with java -jar app.jar. It is made "executable" by the spring-boot-maven-plugin's repackage goal (or the Gradle plugin's bootJar task), which rewrites the plain jar and sets a launcher Main-Class that drives Boot's nested-jar classloader.

Why it matters: this is the default deployment unit — no external application server to install, and the same artifact runs on a laptop, a CI runner or a container. Interviewers probe the difference between mvn package and spring-boot:repackage, and why you cannot simply drop a fat jar onto the classpath (the nested jars need Boot's own loader to be read).

mvn clean package          # produces target/app.jar (repackaged, executable)
java -jar target/app.jar   # embedded server starts — no external container

Layered jars for efficient images

A plain fat jar is one large blob: change a single line of code and the whole archive — including hundreds of megabytes of unchanged dependencies — becomes a new Docker layer. A layered jar adds a layers.idx metadata file that splits the archive into four default layers — dependencies, spring-boot-loader, snapshot-dependencies and application — ordered least- to most-frequently changed. Docker then caches the stable dependency layers and rebuilds only the small application layer on each code change.

Why it matters: faster builds, smaller pushes and cheaper registry storage. The ordering (stable content at the bottom, volatile on top) is the whole trick, and it is a favourite follow-up once Docker comes up. Layering is on by default since Boot 2.4, and both buildpacks and the recommended Dockerfiles consume the same layer index.

java -Djarmode=tools -jar app.jar list-layers
# dependencies / spring-boot-loader / snapshot-dependencies / application

Container images with Cloud Native Buildpacks

Spring Boot can build an OCI image with no Dockerfile at all: mvn spring-boot:build-image or gradle bootBuildImage hands your jar to Cloud Native Buildpacks (the Paketo implementation), which detect Java, add a right-sized JRE, apply the layer index and output a runnable image straight to your local Docker daemon.

Why it matters: sensible, reproducible, non-root images with no Dockerfile to write, review or patch — the buildpack owns base-image CVEs and JRE selection. Know the command, that Paketo is the default builder, that images run as a non-root user, and that any layers.idx customization is honoured by the buildpack too.

mvn spring-boot:build-image -Dspring-boot.build-image.imageName=demo/app:1.0
# or with Gradle:  gradle bootBuildImage --imageName=demo/app:1.0

Writing your own Dockerfile

When you need full control — a specific base image, extra OS packages, particular JVM flags — write a multi-stage Dockerfile. The recommended pattern runs java -Djarmode=tools -jar app.jar extract --layers in a builder stage, then copies each extracted layer in its own COPY instruction so Docker caches dependencies independently of application code — the manual equivalent of what buildpacks do automatically.

Why it matters: buildpacks cover the common case, but Dockerfiles remain the fallback for bespoke requirements and are still asked about constantly. The jarmode=tools extractor replaced the older manual-unzip approach; it also lays files out to be friendly to Class Data Sharing (CDS) and the newer AOT cache for faster startup.

FROM bellsoft/liberica-openjre-debian:25 AS builder
WORKDIR /builder
COPY target/*.jar application.jar
RUN java -Djarmode=tools -jar application.jar extract --layers --destination extracted

FROM bellsoft/liberica-openjre-debian:25
WORKDIR /application
COPY --from=builder /builder/extracted/dependencies/ ./
COPY --from=builder /builder/extracted/spring-boot-loader/ ./
COPY --from=builder /builder/extracted/snapshot-dependencies/ ./
COPY --from=builder /builder/extracted/application/ ./
ENTRYPOINT ["java", "-jar", "application.jar"]

GraalVM native images (AOT)

A GraalVM native image compiles your application ahead-of-time into a standalone, platform-specific executable with no JVM shipped — startup in tens of milliseconds and a much smaller memory footprint, ideal for serverless and fast-scaling containers. Spring makes this possible through AOT processing: at build time it evaluates your bean definitions, generates equivalent Java configuration code, and emits GraalVM reachability hints (reflection, resources, serialization, proxies) so the closed-world native compiler can see everything it needs.

Why it matters: the trade-offs are the interview. Builds are long and memory-hungry; dynamic features (reflection, dynamic proxies, beans decided at runtime) must be known at build time or registered with hints; and you test against a real native binary. Two build paths: buildpacks (mvn -Pnative spring-boot:build-image / gradle bootBuildImage) for a native container, or the GraalVM Native Build Tools (mvn -Pnative native:compile / gradle nativeCompile) for a local executable.

mvn -Pnative native:compile           # local native executable (Native Build Tools)
mvn -Pnative spring-boot:build-image  # native image inside an OCI container

Interview questions

Packaging and deployment questions asked in Indian interviews — from screening rounds through product-company system-design deep dives:

Beginner What is a Spring Boot fat (executable) jar, and how do you run it?

A single self-contained archive holding your compiled classes (BOOT-INF/classes), all runtime dependencies (BOOT-INF/lib), Boot's loader classes and an embedded server. It is produced by the build plugin and run with java -jar app.jar — no external application server required.

Beginner How is spring-boot:repackage different from a normal mvn package?

mvn package builds an ordinary (thin) jar with just your classes. The plugin's repackage goal — bound to the package phase — takes that jar and rebuilds it as an executable fat jar, nesting dependencies under BOOT-INF/lib and setting a launcher Main-Class. Without repackage the jar is not runnable with java -jar.

Beginner What is inside a Spring Boot executable jar, and why can't you just add it to the classpath?

Your code under BOOT-INF/classes, dependency jars under BOOT-INF/lib, Boot's loader at the root, and a manifest pointing at JarLauncher. The dependencies are nested jars-within-a-jar, which the standard JVM classloader cannot read directly; Boot's own launcher and classloader resolve them at startup.

Beginner jar or war — how do you deploy a Spring Boot app today?

The default and recommended unit is the executable jar with an embedded server (Tomcat/Jetty/Undertow). A war (extending SpringBootServletInitializer, embedded server marked provided) is only for deploying into an existing external servlet container — increasingly rare.

Intermediate What are layered jars and what problem do they solve?

A layered jar adds a layers.idx that partitions the fat jar into dependencies, spring-boot-loader, snapshot-dependencies and application. Docker can then cache the rarely-changing dependency layers and rebuild only the small application layer when your code changes — far faster image builds and smaller pushes than a single-blob fat jar.

Intermediate What are the four default layers, and why does their order matter?

dependencies (released libraries), spring-boot-loader, snapshot-dependencies, and application (your classes and resources). They are ordered least- to most-frequently changed so the volatile application layer sits on top. Docker invalidates a changed layer and everything above it, so keeping stable content at the bottom maximises cache hits.

Intermediate What are the two ways to containerize a Spring Boot app, and their trade-offs?

Cloud Native Buildpacks (spring-boot:build-image / bootBuildImage) — no Dockerfile, sensible non-root image, buildpack maintains the base image and JRE, but less control. A hand-written Dockerfile — full control over base image, packages and JVM flags, but you own patching and layer optimisation. Buildpacks for the common case, Dockerfile when you need customisation.

Intermediate What does mvn spring-boot:build-image do, and what is Paketo?

It builds an OCI image from your jar using Cloud Native Buildpacks without a Dockerfile, publishing it to your local Docker daemon. Paketo is the default buildpack implementation Boot uses: it detects Java, adds a right-sized JRE, applies the layer index and produces a non-root, reproducible image.

Intermediate In a multi-stage Dockerfile, how do you take advantage of layered jars?

In a builder stage run java -Djarmode=tools -jar app.jar extract --layers to unpack the jar into its layer directories, then use a separate COPY for each (dependencies, spring-boot-loader, snapshot-dependencies, application). Each COPY becomes its own Docker layer, so dependencies stay cached across code-only rebuilds.

Pro What is a GraalVM native image, and how does it differ from running on the JVM?

An ahead-of-time-compiled, platform-specific executable containing your app and a minimal runtime but no full JVM. Versus the JVM: near-instant startup and lower memory, but no JIT peak-performance warm-up, a closed-world assumption (no arbitrary runtime class loading), longer and memory-heavy builds, and platform-specific binaries.

Pro What is Spring AOT processing, and why is it required for native images?

At build time Spring AOT evaluates the application context, generates explicit bean-registration code, and produces GraalVM hints for reflection, resources, serialization and proxies. Native compilation is closed-world — anything reached only via runtime reflection or dynamic proxies is invisible unless declared — so AOT (plus RuntimeHints / @RegisterReflectionForBinding for your own cases) makes Spring's dynamic behaviour statically reachable.

Pro What are the practical gotchas when moving a Spring Boot app to a native image?

Reflection, dynamic proxies, resource loading and conditionally-created beans must be known at build time or registered with hints; some libraries lack GraalVM metadata; profiles and conditions are largely fixed at build time rather than runtime; builds are slow and RAM-hungry (size your CI accordingly); and you must run tests against the actual native binary, since JVM-passing code can still fail natively. Mitigate with the native build tools' testing support and by preferring libraries that publish reachability metadata.
