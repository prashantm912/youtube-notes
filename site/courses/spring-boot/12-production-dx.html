Module: Production and Developer Experience

The last mile of a Spring Boot app: the tools that make the inner development loop fast, the switches that make shutdown and startup safe, and the migration story that carries you from Boot 3's Jakarta break to Boot 4's modular rebuild.

Knowledge Library → Courses → Spring Boot → Production and Developer Experience

What this module covers

This closing module is two halves. The developer-experience half is the day-to-day workflow: bootstrapping a project on Spring Initializr, tightening the edit-run loop with DevTools, hooking into the application lifecycle through events and runners, shutting down without dropping in-flight requests, and branding the startup banner. The production half is forward motion: what changed moving to Spring Boot 3 (the Jakarta namespace) and to Spring Boot 4 (module restructuring, Jackson 3, a new Spring Framework baseline), and how to migrate an existing app across both.

Topics

Spring Boot DevTools

The spring-boot-devtools module exists to shorten the feedback loop while you code. It watches the classpath and triggers an automatic restart when your compiled classes change, ships a LiveReload server that refreshes the browser on a resource change, and applies development-friendly property defaults (template and static-resource caching switched off, extra debug logging). Its restart is faster than a cold boot because it uses two classloaders — a base loader for unchanging third-party jars and a throwaway restart loader for your own classes, so only your code is reloaded.

Why it matters: it is a development-only convenience and is engineered to disappear in production — DevTools deactivates itself when the app is launched from a fully packaged jar, and the repackaging plugins exclude it by default. Marking the dependency optional keeps it off the classpath of anything that depends on your module.

<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-devtools</artifactId>
  <optional>true</optional>   <!-- kept out of the production jar -->
</dependency>

Spring Initializr — bootstrapping a project

Spring Initializr (the service behind start.spring.io) is the canonical way to start: you pick the build tool (Maven or Gradle), language, Spring Boot version and dependencies, and it generates a correctly wired project zip — the parent POM or plugin, a BOM-managed dependency set, a main class annotated @SpringBootApplication, and a test skeleton. Most IDEs embed the same service, so "New Spring Boot project" is really Initializr behind a dialog.

Why it matters: it removes the whole class of "wrong version combination" problems by handing you a dependency set that Spring already tested together. Knowing that it is a public REST service — not magic in the IDE — is a common follow-up in interviews, as is naming what a fresh project actually contains.

Graceful shutdown

When a running instance is asked to stop — a rolling deploy, a Kubernetes pod eviction — you do not want it to sever requests mid-flight. Graceful shutdown gives in-flight requests a grace period to finish while the server stops accepting new ones at the network layer, then closes the context. In current Spring Boot it is enabled by default for the embedded servers; you opt out with server.shutdown=immediate, and you size the grace window with spring.lifecycle.timeout-per-shutdown-phase (30 seconds by default). It runs as the earliest phase of stopping the SmartLifecycle beans.

Why it matters: without it, a redeploy shows up as a burst of client-side 5xx errors and half-committed work. Pair it with Actuator's readiness probe so the load balancer stops routing new traffic before the grace period even begins.

server.shutdown=graceful                          # default now; 'immediate' opts out
spring.lifecycle.timeout-per-shutdown-phase=30s   # window for in-flight requests to finish

Application lifecycle: events, listeners and runners

A SpringApplication publishes an ordered sequence of events as it comes up — starting, environment prepared, context prepared, context loaded, started, ready, and (on failure) failed — that you can hook with an ApplicationListener or an @EventListener method. For the common "do this once, after everything is wired" case, Boot supplies two callbacks: CommandLineRunner (raw String[] arguments) and ApplicationRunner (a parsed ApplicationArguments); both run after the context is fully initialized, which is why they are safer than a constructor or @PostConstruct for work that needs the whole application ready. For diagnosing slow starts, ApplicationStartup / BufferingApplicationStartup record startup steps you can inspect through Actuator.

@Bean
CommandLineRunner seed(UserRepository repo) {
    return args -> repo.save(new User("admin"));  // runs once, after the context is ready
}

Customizing the startup banner

The ASCII banner printed at startup is more than decoration — teams use it to make it obvious at a glance which service and which build a log stream belongs to. Drop a banner.txt on the classpath (or point spring.banner.location elsewhere) and Boot renders it, expanding variables such as ${application.version} and ${spring-boot.version}, and honouring ANSI colour codes. From Spring Boot 3 the image-banner formats were removed, leaving text as the supported form; set spring.main.banner-mode=off to silence it entirely.

  ::  My Service  ::   (v${application.version})
  running on Spring Boot ${spring-boot.version}

Spring Boot 3 to 4 migration, Jakarta namespace and what's new

Two upgrades define the current landscape. Spring Boot 3 moved to a Java 17 baseline, Spring Framework 6 and Jakarta EE — the breaking change everyone remembers is the package rename from javax.* to jakarta.* for EE specifications (Servlet, Persistence, Validation, and friends), while core-JDK javax packages like javax.sql and javax.crypto stayed put. Spring Boot 4 then rebuilt on Spring Framework 7 and Spring Security 7, restructured the codebase into fine-grained org.springframework.boot.<module> modules (some features now need an explicit starter), adopted Jackson 3 (new tools.jackson group and packages), dropped Undertow on the Servlet 6.1 baseline, and renamed test annotations such as @MockBean to @MockitoBean. The recommended path is to reach the latest 3.5.x first, then step up.

// Spring Boot 2 (Java EE)          Spring Boot 3+ (Jakarta EE)
import javax.persistence.Entity;    import jakarta.persistence.Entity;
import javax.servlet.Filter;        import jakarta.servlet.Filter;

Interview questions

Production readiness and the Boot 3/4 upgrade are staples of Indian Spring interviews — screening rounds probe the tooling, product-company rounds probe the migration:

Beginner What does Spring Boot DevTools give you, and is it safe in production?

Automatic restart on classpath changes, browser LiveReload, and development-friendly property defaults (caching off, extra debug logging). It is safe because it deactivates when the app runs from a packaged jar and the repackaging plugins exclude it — so it never ships to production. Declaring the dependency optional also keeps it off downstream classpaths.

Beginner What is Spring Initializr and what does a generated project contain?

A web service (start.spring.io, also embedded in IDEs) that generates a ready-to-build project from your choices of build tool, language, Boot version and dependencies. You get the parent POM or Gradle plugin, a BOM-managed dependency set, a main class with @SpringBootApplication, and a test skeleton — a version-consistent starting point.

Beginner How do you customize or disable the startup banner?

Put a banner.txt on the classpath (or set spring.banner.location); Boot renders it and expands variables like ${application.version}. It supports ANSI colours. Turn it off with spring.main.banner-mode=off. Since Boot 3, image-format banners were removed, leaving text.

Beginner Difference between CommandLineRunner and ApplicationRunner?

Both run once after the context is fully initialized. CommandLineRunner.run receives the raw String[] program arguments; ApplicationRunner.run receives an ApplicationArguments that parses option arguments (--name=value) from non-option ones. Order multiple runners with @Order.

Intermediate What is graceful shutdown and how do you control it?

On stop, the server refuses new requests but lets in-flight ones complete before closing the context. Current Boot enables it by default for embedded servers; disable with server.shutdown=immediate, and size the window with spring.lifecycle.timeout-per-shutdown-phase (30s default). Older versions were immediate by default and needed server.shutdown=graceful to opt in. Combine with a readiness probe so traffic drains first.

Intermediate Walk through the application events published during startup.

In order: ApplicationStartingEvent, ApplicationEnvironmentPreparedEvent, ApplicationContextInitializedEvent, ApplicationPreparedEvent, ApplicationStartedEvent, an availability change, then ApplicationReadyEvent when the app is ready to serve. If startup fails, ApplicationFailedEvent fires. Listeners for the earliest events must be registered very early (they run before the context exists).

Intermediate Why prefer a runner over @PostConstruct for startup work?

@PostConstruct runs during that bean's initialization — other beans, and the web server, may not be ready yet. ApplicationRunner/CommandLineRunner (and ApplicationReadyEvent) run after the whole context is refreshed, so work that depends on the fully wired application — seeding data, warming caches, calling other beans — is safe there.

Intermediate What exactly changed in the package namespace for Spring Boot 3, and what did not?

Jakarta EE specification packages were renamed from javax.* to jakarta.* — for example javax.persistence becomes jakarta.persistence, javax.servlet becomes jakarta.servlet. Core-JDK packages that were never part of Java EE — javax.sql, javax.crypto, javax.swing — are unchanged. The trigger was Oracle keeping the javax trademark after donating Java EE to the Eclipse Foundation.

Intermediate What are the Java and Spring baselines for Boot 3 versus Boot 4?

Spring Boot 3 requires Java 17, Spring Framework 6 and Jakarta EE 9/10. Spring Boot 4 keeps a Java 17+ baseline but moves to Spring Framework 7 and Spring Security 7, a Jakarta EE 11 / Servlet 6.1 baseline, and Jackson 3. Knowing that a major Boot bump drags the whole Spring portfolio with it is the point.

Pro How do DevTools' two classloaders make restart fast, and where does that bite you?

Unchanging third-party jars load in a base classloader that is kept across restarts; your own classes load in a disposable restart classloader that is thrown away and rebuilt on each change — so only your code reloads, far faster than a cold JVM boot. The catch: objects created before a restart live in a different classloader than those after, so caching or deserializing instances across a restart can throw ClassCastException. It also is not a hot-swap of method bodies inside a single run.

Pro What are the headline breaking changes when moving from Boot 3 to Boot 4?

The codebase is split into fine-grained org.springframework.boot.<module> modules, so some features now need an explicit starter instead of being auto-detected. Jackson 3 arrives with new tools.jackson group and package names and renamed annotations (@JsonComponent becomes @JacksonComponent). Undertow is dropped on the Servlet 6.1 baseline. Test annotations change (@MockBean/@SpyBean to @MockitoBean/@MockitoSpyBean), and MockMvc/TestRestTemplate are no longer auto-configured by @SpringBootTest. Reach the latest 3.5.x first, then upgrade.

Pro How would you migrate a large javax-based Boot 2 app to Boot 3 with minimal manual editing?

Upgrade to the latest 2.7.x first so deprecations surface, then apply automated tooling: OpenRewrite's Jakarta recipe (org.openrewrite.java.migrate.jakarta.JavaxMigrationToJakarta) and the Spring Boot Migrator rewrite imports and configuration en masse. Let the Boot BOM manage versions, watch for third-party libraries without Jakarta releases, and lean on a strong test suite to catch behavioural changes the recipes cannot see. Migrate incrementally and verify at each step.
