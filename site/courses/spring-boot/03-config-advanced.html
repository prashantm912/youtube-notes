Module: Advanced Configuration

Beyond binding: this is where configuration grows teeth — validated at startup so a bad value stops the app instead of a request, documented so your IDE completes it, and pulled from files, config trees and mounted secrets the code never hard-codes.

Knowledge Library → Courses → Spring Boot → Advanced Configuration

What this module covers

Advanced Boot configuration turns @ConfigurationProperties from a convenience into a contract: how the classes are registered (@EnableConfigurationProperties vs @ConfigurationPropertiesScan), JSR-380 validation that fails the application fast on bad input, a configuration-metadata file that powers IDE auto-completion, and the spring.config.import family — extra property files, config trees and mounted secrets — that keeps sensitive values out of source control. These are the features interviewers probe once you are past "what is @Value".

Topics

Registering property beans: @EnableConfigurationProperties and @ConfigurationPropertiesScan

A @ConfigurationProperties class is inert until Spring registers it as a bean. Two idioms do that: name the types explicitly with @EnableConfigurationProperties(MyProps.class) on a configuration class, or drop @ConfigurationPropertiesScan on the application and let Boot classpath-scan for every @ConfigurationProperties type under the given packages (defaulting to the main class's package, like @ComponentScan). Scan is the low-ceremony choice for application code; explicit enabling is precise and keeps library types free of stereotype annotations.

Why it matters: "how does Boot know about your properties class?" is a routine interview probe. The senior detail is that a @ConfigurationProperties POJO should not also be a @Component — registering it through the stereotype path bypasses the dedicated binding and can double-register the bean. Pick one mechanism.

@SpringBootApplication
@ConfigurationPropertiesScan          // auto-register @ConfigurationProperties types
public class App {
    public static void main(String[] args) { SpringApplication.run(App.class, args); }
}

@Configuration
@EnableConfigurationProperties(MailProperties.class)  // or register specific types
class MailConfig { }

Validating @ConfigurationProperties

Add @Validated to a @ConfigurationProperties class and Boot runs Jakarta Bean Validation (JSR-380) against the bound values while the context starts. Standard constraints — @NotBlank, @Min, @Max, @Email, @Pattern — sit on the fields, and any violation aborts startup with a descriptive error instead of surfacing much later as a mysterious NullPointerException in production. Nested objects need @Valid on the field (Bean Validation does not cascade on its own); a non-inner nested type also wants @NestedConfigurationProperty so its metadata is generated.

Why it matters: fail-fast configuration is a hallmark of production-grade services and a favourite interview theme. You need the constraint API on the classpath (spring-boot-starter-validation). For logic too complex for annotations, declare a static bean named configurationPropertiesValidator implementing Spring's Validator — static so it is created very early, before the beans it guards exist.

@ConfigurationProperties(prefix = "app.mail")
@Validated
public class MailProperties {
    @NotBlank private String host;
    @Min(1) @Max(65535) private int port;
    @Valid  private final Credentials credentials = new Credentials(); // cascade
    // getters and setters
}

Configuration metadata and IDE auto-completion

The spring-boot-configuration-processor is a compile-time annotation processor that reads your @ConfigurationProperties classes and emits META-INF/spring-configuration-metadata.json, describing every key with its type, default and Javadoc. IDEs read that file to offer auto-completion, inline documentation and deprecation strike-through for your properties — the same experience Boot's built-in keys already give you. It runs at build time only; nothing is needed at runtime.

Why it matters: for hand-written keys, values the processor cannot infer (a fixed set of allowed strings, a value provider), or properties not bound to a class, you add META-INF/additional-spring-configuration-metadata.json with hints and the processor merges it into the generated file. @DeprecatedConfigurationProperty documents a retired key with a reason and replacement. Knowing where IDE auto-completion comes from — and that it is opt-in via a build dependency — is a common interview follow-up.

{ "hints": [
    { "name": "app.mail.protocol",
      "values": [ { "value": "smtp" }, { "value": "smtps" } ] }
] }

Importing external config: spring.config.import, config trees and secrets

spring.config.import (Boot 2.4+) pulls additional configuration into the Environment from within application.properties/yml itself — extra property files, extensionless files, whole directories, even non-file sources such as configtree: and vault:. Each import is inserted as a new document just below its declaration, and imported values take precedence over the importing file. The optional: prefix keeps startup from failing when a location is absent, so the same build runs whether or not that file exists.

Why it matters: for secrets, a config tree maps a mounted directory to properties — each file's name becomes the key and its contents the value, exactly how Kubernetes and Docker expose Secrets as files (spring.config.import=optional:configtree:/run/secrets/). That keeps credentials out of application.properties and out of Git; the file-per-secret model also underlies HashiCorp Vault via Spring Cloud Vault. "How do you inject secrets without hard-coding them?" is a standard, system-design-flavoured interview question.

# application.properties
spring.config.import=optional:file:./secret.properties,\
                     optional:configtree:/run/secrets/

Interview questions

Most-asked advanced-configuration questions in Indian Spring Boot interviews — the follow-ups once the screening round has confirmed you can use @ConfigurationProperties:

Beginner How do you tell Spring Boot about a @ConfigurationProperties class?

Two ways: @EnableConfigurationProperties(MyProps.class) on a configuration class to register specific types, or @ConfigurationPropertiesScan on the application to classpath-scan and auto-register every @ConfigurationProperties type under the given packages. The POJO itself should not also be a @Component.

Beginner How do you validate values bound to @ConfigurationProperties?

Put @Validated on the class and Jakarta Bean Validation (JSR-380) constraints — @NotBlank, @Min, @Email, @Pattern — on the fields. It needs a validation provider on the classpath, added via spring-boot-starter-validation.

Beginner What happens when a @ConfigurationProperties validation constraint fails at startup?

The application context fails to start — Boot throws a binding/validation error naming the offending properties and constraints. That fail-fast behaviour is the whole point: a misconfigured app never comes up and never serves traffic with a bad value.

Beginner What is spring.config.import used for?

It imports additional configuration into the Environment from inside application.properties/yml — extra files, directories, config trees, or sources such as Vault — without needing command-line flags. Imported values take precedence over the importing document.

Intermediate @EnableConfigurationProperties vs @ConfigurationPropertiesScan — when do you use which?

@EnableConfigurationProperties registers named types explicitly — precise, good for library code or when you want tight control. @ConfigurationPropertiesScan scans packages (defaulting to the app's package) and auto-registers all @ConfigurationProperties types — less boilerplate for application code. Use one; do not also mark the class @Component.

Intermediate How do you validate a nested configuration object?

Bean Validation does not cascade automatically — annotate the nested field with @Valid so its constraints are checked too. For a non-inner nested type, also add @NestedConfigurationProperty so the annotation processor generates metadata for its keys.

Intermediate What does spring-boot-configuration-processor do, and is it needed at runtime?

It is a compile-time annotation processor that generates META-INF/spring-configuration-metadata.json from your @ConfigurationProperties classes, powering IDE auto-completion and inline docs. It is build/IDE-only — declared optional (Maven) or in the annotationProcessor configuration (Gradle) and not required at runtime.

Intermediate What are config trees (configtree:) and when do you use them?

A config tree maps a mounted directory to properties: each filename becomes a key and the file contents become its value. You use it to consume Kubernetes/Docker Secrets and ConfigMaps mounted as files, e.g. spring.config.import=optional:configtree:/run/secrets/, so secrets never live in the packaged properties.

Intermediate What does the optional: prefix do on a config location?

It marks the location as non-mandatory — if the file or tree is missing, startup continues instead of aborting. It works with spring.config.import, spring.config.location and spring.config.additional-location, letting one artifact run across environments where some sources are absent.

Intermediate spring.config.import vs spring.config.additional-location — what's the difference?

additional-location adds a directory to the set of locations Boot searches for application.properties/yml, supplementing the defaults. import pulls specific documents or sources into the Environment inline, as a new higher-precedence document below the declaration. Different mechanisms, different precedence rules.

Pro Why must a custom configurationPropertiesValidator bean be declared static?

It has to exist very early — before ordinary @Configuration classes and the @ConfigurationProperties beans it validates are instantiated. A static @Bean method can be invoked without creating its enclosing configuration instance, so the validator is available at that early phase and avoids triggering premature bean creation.

Pro A property's valid values aren't an enum, yet you want IDE auto-completion. How?

Provide hints in META-INF/additional-spring-configuration-metadata.json — a hints entry listing the allowed values, or a providers value provider (such as a class reference or logger-name handler). The configuration processor merges it into the generated metadata, and IDEs then offer those completions.

Pro How do you feed secrets into a Boot app without hard-coding them in application.properties or committing them to Git?

Keep them out of the packaged config and import them at runtime via spring.config.import: optional:file: for an un-committed local file, configtree: for secrets mounted by Kubernetes/Docker, or vault:// through Spring Cloud Vault for a secrets manager. Property-source precedence and environment variables then let each deployment override safely, so no credential ever reaches source control.
