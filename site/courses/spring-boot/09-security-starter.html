Module: Security with the Boot Starter

Add one starter and every endpoint is suddenly behind a login you never wrote. Spring Boot's security auto-configuration is equal parts gift and interview trap — this module maps exactly what you get for free and the single bean that takes back control.

Knowledge Library → Courses → Spring Boot → Security with the Boot Starter

What this module covers

Adding spring-boot-starter-security flips a Spring Boot app from wide-open to locked-down with no code: every request needs authentication, a login form and HTTP Basic appear, CSRF and security headers switch on, and a random password is generated for a single built-in user. This module covers those defaults, the generated password and the default filter chain, then the one customization Boot expects you to make — a SecurityFilterChain bean, plus user configuration. Deep Spring Security (JWT, OAuth2, method security, database-backed users) is a separate upcoming track; here we stay at the Boot-integration layer.

Topics

The security starter and its defaults

When spring-boot-starter-security is on the classpath, Boot's SecurityAutoConfiguration and UserDetailsServiceAutoConfiguration apply Spring Security's default chain for you. The result: every endpoint (including Boot's /error) requires authentication, both a generated form-login page and HTTP Basic are offered — the choice keys off the request's Accept header — CSRF protection and a standard set of security response headers are on, and a single in-memory user named user is created.

Why it matters: that user's password is a random UUID generated on every startup and printed once to the console. It exists so an app is never accidentally shipped wide open, but it is explicitly development-only — it changes each restart and lives only in memory. Recognising this as a safety net rather than a configuration is a routine screening check.

The default filter chain

Spring Security runs as a single servlet filter, FilterChainProxy, inserted into the container through a DelegatingFilterProxy that bridges the servlet lifecycle to the Spring application context. FilterChainProxy holds one or more SecurityFilterChain instances; for each request it selects the first whose matcher applies and runs that chain's ordered filters — authentication, CSRF, authorization and the rest.

Why it matters: Boot's auto-configuration publishes exactly one default SecurityFilterChain that matches every request. Once you can picture that chain — and see that declaring your own bean replaces it — the rest of the module follows naturally.

Taking control with a SecurityFilterChain bean

The moment you declare your own SecurityFilterChain @Bean, Boot backs off its default web-security auto-configuration and hands you the HttpSecurity builder. This is where you state which paths are public, which require authentication, and whether to keep form login, HTTP Basic, or both. Since Spring Security 6 / Boot 3 this bean-based style fully replaced the old WebSecurityConfigurerAdapter, which was removed.

@Bean
SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/", "/public/**").permitAll()
            .anyRequest().authenticated())
        .formLogin(withDefaults())
        .httpBasic(withDefaults());
    return http.build();
}

One subtlety catches people out: defining this bean switches off the default request-authorization rules, but it does not by itself remove the built-in in-memory user — that comes from a separate auto-configuration, covered next.

Configuring users

The generated user is meant to be replaced. The lightest override is configuration-only: set spring.security.user.name, spring.security.user.password and spring.security.user.roles, and Boot uses those instead of generating a password. For anything real, provide a bean — a UserDetailsService (in-memory, JDBC, or a custom implementation), or an AuthenticationProvider / AuthenticationManager. Supplying any one of these suppresses UserDetailsServiceAutoConfiguration, so the default user disappears.

spring.security.user.name=admin
spring.security.user.password=s3cr3t
spring.security.user.roles=ADMIN

Interview questions

How the Boot security starter is probed in Indian interviews — a favourite because the "it just works" defaults hide several gotchas:

Beginner What happens the moment you add spring-boot-starter-security to an app?

Auto-configuration secures everything: every endpoint (including /error) needs authentication, a generated form-login page and HTTP Basic are offered, CSRF protection and security headers turn on, and one in-memory user named user is created with a random password.

Beginner Where does the login password come from, and what is the default username?

The username is user. The password is a random UUID generated at each startup and printed once to the console ("Using generated security password: …"). It is explicitly development-only — regenerated every restart, held only in memory.

Beginner How do you set a fixed username and password without writing any code?

Use properties: spring.security.user.name, spring.security.user.password and optionally spring.security.user.roles. Boot then uses those instead of generating a password. Fine for demos; not a real user store.

Beginner Which auto-configurations drive the default behaviour?

SecurityAutoConfiguration applies Spring Security's default web security (effectively the @EnableWebSecurity default chain), and UserDetailsServiceAutoConfiguration creates the in-memory user. Both are conditional on Spring Security being on the classpath and no user-defined equivalents present.

Intermediate What is a SecurityFilterChain bean, and what happens to Boot's defaults when you declare one?

It is the bean that defines the ordered security filters and matching rules for requests. Declaring your own turns off Boot's default web-security auto-configuration and gives you the HttpSecurity builder to set authorization rules and login mechanisms. It does not, on its own, remove the default in-memory user.

Intermediate Explain the default filter chain: FilterChainProxy and DelegatingFilterProxy.

DelegatingFilterProxy is a servlet filter that bridges the container to the Spring context. It delegates to FilterChainProxy, which holds the SecurityFilterChain(s); for each request it picks the first matching chain and runs its ordered filters (authentication, CSRF, authorization, …). Boot publishes one default chain matching every request.

Intermediate How do you replace the built-in user with real users?

Provide a bean — a UserDetailsService (in-memory, JDBC or custom), or an AuthenticationProvider / AuthenticationManager. Any of these suppresses UserDetailsServiceAutoConfiguration, so the default user is no longer created.

Intermediate What security defaults beyond authentication does the starter switch on?

CSRF protection, session-fixation protection, and a standard set of security response headers (HSTS, X-Content-Type-Options, X-Frame-Options, Cache-Control), plus both form login and HTTP Basic. Passwords are stored/compared through a PasswordEncoder (BCrypt by default).

Intermediate How does Boot decide between the form login page and an HTTP Basic 401?

By the request's Accept header. A browser sending Accept: text/html is redirected to the generated login form; an API client that does not ask for HTML gets an HTTP Basic 401 challenge. The same default chain serves both.

Pro You added a SecurityFilterChain that permits all requests, yet the default user still exists. Why, and how do you remove it?

Declaring a SecurityFilterChain only disables the default web-security config; the in-memory user comes from UserDetailsServiceAutoConfiguration, which is still active. Define a UserDetailsService, AuthenticationProvider or AuthenticationManager bean and that auto-configuration backs off, removing the default user.

Pro Since Spring Security 6 / Boot 3, WebSecurityConfigurerAdapter is gone. What replaced it, and why?

Component-based configuration: a SecurityFilterChain bean instead of overriding configure(HttpSecurity), and a UserDetailsService / AuthenticationManager bean instead of configure(AuthenticationManagerBuilder). It favours composition over inheritance, works cleanly with the lambda DSL, and lets you define multiple chains as ordered beans.

Pro Why is the generated password labelled "development use only," and what must change for production?

It is a single user, in memory, with a password that changes on every restart — no persistence, no roles model, no real credential store. Production needs a proper user source (database-backed UserDetailsService or an external identity provider), encoded passwords, and an authorization model — which is where the dedicated Spring Security track picks up.
