Module: Web Applications with Boot

Add one starter and you get a running HTTP server, JSON in and out, and a ready-made error page before writing a line of config — this module names those defaults and the switches that change them.

Knowledge Library → Courses → Spring Boot → Web Applications with Boot

What this module covers

spring-boot-starter-web is the on-ramp to server-side web apps on Boot: it wires Spring MVC, an embedded servlet container and Jackson, then applies opinionated defaults for JSON responses, the whitelabel error page and static-content serving. This module orients you on that stack — choosing and tuning the embedded server, controlling error handling and CORS, shaping JSON, and locating static assets — and points to the official reference and community write-ups that teach each in depth.

Topics

spring-boot-starter-web and the embedded server

spring-boot-starter-web pulls in Spring MVC, Jackson, Bean Validation and — the defining Boot move — an embedded Tomcat, so the application is a self-contained jar that owns its HTTP server rather than a war deployed into an external container. Boot also supports Jetty and Undertow: exclude spring-boot-starter-tomcat and add the alternative starter.

Why it matters: "why does Boot embed a server, and how does a fat jar boot its own Tomcat?" is one of the most-asked Boot interview questions, along with the Tomcat / Jetty / Undertow trade-offs (Undertow is lean and fast; Jetty is lightweight; Tomcat is the default and most familiar).

<!-- pom.xml: swap the default Tomcat for Jetty -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <exclusions><exclusion>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-tomcat</artifactId>
  </exclusion></exclusions>
</dependency>
<!-- then add spring-boot-starter-jetty (or -undertow) -->

Configuring the server: port, context path, threads, compression

Almost everything about the running server is a server.* property: server.port (default 8080; 0 picks a random free port), server.servlet.context-path, thread limits (server.tomcat.threads.max), response compression, SSL (server.ssl.*) and graceful shutdown. For anything the properties don't expose, register a WebServerFactoryCustomizer bean and configure the factory programmatically.

Why it matters: port conflicts, mounting an app under a context path, and tuning the connector's thread pool are day-one operational questions — and knowing that these live in the server.* namespace (not spring.*) is a common relaxed-binding gotcha.

server.port=8081
server.servlet.context-path=/api
server.tomcat.threads.max=200
server.compression.enabled=true

Default error handling: the whitelabel page and BasicErrorController

When an unhandled exception reaches the container, Boot's ErrorMvcAutoConfiguration renders the "whitelabel" error page — a plain HTML page for browsers, and a JSON body ({timestamp, status, error, path}) for API clients — through a BasicErrorController mapped to /error. Content negotiation on the request's Accept header is what makes the same /error path serve HTML or JSON. Turn it off with server.error.whitelabel.enabled=false, override it with an error.html template (or per-status error/404.html), or replace the controller entirely; shape API errors with @ControllerAdvice + @ExceptionHandler, ideally returning an RFC 9457 ProblemDetail.

Why it matters: the whitelabel page is the single most recognizable Boot default, and "what produces it and how do you customize it?" is a staple interview question — followed by "how would you return a consistent JSON error contract across the API?".

@RestControllerAdvice
class ApiErrors {
  @ExceptionHandler(EntityNotFoundException.class)
  ProblemDetail notFound(EntityNotFoundException e) {
    return ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, e.getMessage());
  }
}

CORS: cross-origin requests

Browsers block cross-origin fetch/XHR unless the server returns the right Access-Control-* headers. Spring MVC adds them either fine-grained with @CrossOrigin on a controller or method, or globally via a WebMvcConfigurer that overrides addCorsMappings. CORS is a Spring Framework feature (Boot exposes a few knobs), and the preflight OPTIONS request plus the interaction with Spring Security — CORS must be enabled inside the security filter chain — are where it usually breaks.

Why it matters: every React/Angular front end talking to a Boot API hits CORS, and debugging the preflight failure or the "credentials with wildcard origin" conflict is a routine screening and on-the-job question.

@Configuration
class CorsConfig implements WebMvcConfigurer {
  @Override public void addCorsMappings(CorsRegistry r) {
    r.addMapping("/api/**")
     .allowedOrigins("https://app.example.com")
     .allowedMethods("GET", "POST");
  }
}

JSON with Jackson

With Jackson on the classpath, Boot auto-configures a JsonMapper/ObjectMapper so @RestController return values serialize to JSON and @RequestBody deserializes automatically through the registered HttpMessageConverters. Tune it globally with spring.jackson.* properties (date format, null inclusion, property-naming strategy) or, for custom (de)serializers, a builder-customizer bean or a @JacksonComponent. Boot 4 defaults to Jackson 3.

Why it matters: JSON is the default REST wire format, and controlling ISO date formatting, dropping nulls, and snake_case naming come up constantly — as does knowing that no manual ObjectMapper wiring is needed because Boot already provides one.

spring.jackson.serialization.write-dates-as-timestamps=false
spring.jackson.default-property-inclusion=non_null
spring.jackson.property-naming-strategy=SNAKE_CASE

Serving static content

Boot serves static files from /static, /public, /resources and /META-INF/resources on the classpath, mapped at /**, and treats an index.html in any of them as the welcome page. Change the URL pattern with spring.mvc.static-path-pattern and the source locations with spring.web.resources.static-locations; versioned resource chains give you cache-busting for production assets.

Why it matters: in full-stack setups, knowing where Boot looks for assets — and how the welcome-page index.html supports a bundled single-page app — saves hours of "why is my CSS 404-ing?" debugging.

# default lookup: /static, /public, /resources, /META-INF/resources
spring.mvc.static-path-pattern=/assets/**
spring.web.resources.static-locations=classpath:/assets/

Interview questions

The web module is interview-dense in Indian service- and product-company rounds alike — embedded servers and the whitelabel error page come up again and again:

Beginner What does spring-boot-starter-web give you out of the box?

Spring MVC (dispatcher servlet, @RestController), an embedded Tomcat, Jackson for JSON, and Bean Validation — all auto-configured. The app becomes a runnable jar that starts its own HTTP server on port 8080, with no external container or web.xml.

Beginner What is the default embedded server and port, and how do you change the port?

Embedded Tomcat on port 8080. Set server.port in application.properties/YAML (e.g. server.port=8081), pass it as --server.port= on the command line, or use 0 for a random free port. It lives in the server.* namespace, not spring.*.

Beginner How do you switch the embedded server from Tomcat to Jetty or Undertow?

Exclude spring-boot-starter-tomcat from spring-boot-starter-web and add spring-boot-starter-jetty or spring-boot-starter-undertow. Boot auto-configures whichever container it finds on the classpath.

Beginner What is the whitelabel error page?

Boot's default error view. For an unhandled error it shows a generic HTML page to browsers and a JSON body (timestamp, status, error, path) to API clients, served by BasicErrorController at /error. Disable it with server.error.whitelabel.enabled=false.

Intermediate How does the same /error endpoint return HTML to a browser but JSON to an API client?

Content negotiation. BasicErrorController has two handler methods — one produces text/html, the other JSON — and Spring picks based on the request's Accept header. Browsers send Accept: text/html (whitelabel page); REST clients send application/json (the error map).

Intermediate How do you customize or replace the whitelabel error page?

Several ways: drop an error.html (or per-status error/404.html) template on the classpath; disable whitelabel and let the container show its own page; implement a custom ErrorController; or, for APIs, use @ControllerAdvice/@ExceptionHandler returning a ProblemDetail for a consistent JSON contract.

Intermediate What is CORS and how do you enable it in Spring Boot?

Cross-Origin Resource Sharing lets a browser call an API on a different origin, gated by Access-Control-* response headers. Enable it fine-grained with @CrossOrigin on a controller/method, or globally by overriding addCorsMappings in a WebMvcConfigurer. Preflight OPTIONS requests are handled automatically.

Intermediate How does Boot serialize responses to JSON, and how do you change date format or null handling?

Boot auto-configures a Jackson ObjectMapper/JsonMapper and registers it as an HttpMessageConverter, so controller return values serialize automatically. Change behavior with spring.jackson.* — e.g. write-dates-as-timestamps=false, default-property-inclusion=non_null, property-naming-strategy=SNAKE_CASE — or a builder-customizer bean.

Intermediate Where does Spring Boot serve static content from by default?

From classpath:/static, /public, /resources and /META-INF/resources, mapped at /**. An index.html in any of these becomes the welcome page. Override with spring.mvc.static-path-pattern and spring.web.resources.static-locations.

Intermediate @ControllerAdvice vs @RestControllerAdvice — what's the difference?

Both centralize @ExceptionHandler methods across controllers. @RestControllerAdvice = @ControllerAdvice + @ResponseBody, so handler return values are serialized (to JSON) instead of resolved as view names — the right choice for REST APIs replacing whitelabel behavior.

Pro Fat jar with embedded server vs traditional war — how does it change deployment, and when would you still build a war?

The fat jar bundles the app plus an embedded container, so java -jar is the whole deployment — ideal for containers and cloud/12-factor setups where each app owns its server. Build a war (extend SpringBootServletInitializer, package war) only when you must deploy into a shared external application server mandated by ops or licensing.

Pro Your @CrossOrigin config stops working once Spring Security is added. Why, and how do you fix it?

Security's filter chain runs before Spring MVC, so it can reject the unauthenticated preflight OPTIONS before @CrossOrigin is ever reached. Fix it by enabling CORS in the security filter chain (http.cors(...)) and exposing a CorsConfigurationSource bean, so Security's CorsFilter applies the same policy up front.

Pro Why does allowCredentials(true) with allowedOrigins("*") fail, and what's the correct fix?

The CORS spec forbids returning Access-Control-Allow-Origin: * together with Allow-Credentials: true; Spring throws at startup/handling rather than emit an invalid header. Use allowedOriginPatterns("*") (which reflects the actual request origin) or list explicit origins when credentials are required.
