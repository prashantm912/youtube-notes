Module: Data Access with Boot

Add a driver and three properties and Boot hands you a pooled, migration-ready datasource — this module maps what it auto-configures, how to tune HikariCP, and how Flyway and Liquibase keep your schema under version control.

Knowledge Library → Courses → Spring Boot → Data Access with Boot

What this module covers

Spring Boot turns data access into configuration: put a JDBC driver on the classpath, set a few properties, and you get a HikariCP-pooled DataSource, an auto-configured JdbcTemplate, and — with a starter — Spring Data repositories. This module stays on the Boot integration layer: auto-configured datasources, connection pooling, the JPA/JDBC starters, SQL initialization, and versioned schema migrations.

Deep JPA and Hibernate mapping (entities, relationships, the persistence context) live in a separate track. Here the focus is what Boot wires up for you and the knobs you actually turn — with HikariCP tuning and "Flyway vs Liquibase" flagged because both are perennial interview ground.

Topics

DataSource auto-configuration

When a JDBC driver is on the classpath and connection properties are present, Boot auto-configures a single DataSource bean — no @Bean needed. An in-memory database (H2, HSQLDB, Derby) needs no URL at all: Boot detects the driver and wires an embedded instance. For a real database you supply spring.datasource.url, username and password; the driver class is normally inferred from the URL.

Why it matters: this one DataSource is the backbone every data feature consumes — repositories, JdbcTemplate, and Flyway all draw from it. Interviewers probe the auto-config reasoning: that "driver plus three properties" yields a pooled, production-ready datasource, and that an accidental H2 on the classpath can silently shadow the database you meant to use.

spring.datasource.url=jdbc:postgresql://localhost:5432/orders
spring.datasource.username=app
spring.datasource.password=secret
# driver-class-name is inferred from the URL; set it only to override

HikariCP — the default connection pool

Since Spring Boot 2.0, HikariCP is the default connection pool. It ships transitively with spring-boot-starter-jdbc and spring-boot-starter-data-jpa, so pooling is on with zero extra dependencies. You tune it under spring.datasource.hikari.*: maximum-pool-size, minimum-idle, connection-timeout, idle-timeout and max-lifetime.

Why it matters: pool sizing is a classic performance-and-scaling question. HikariCP's own guidance is counter-intuitive — a small pool usually outperforms a large one, because more connections than the database can serve in parallel just add contention. And max-lifetime should sit a little under the database's own connection timeout, so the pool retires connections before the server drops them. Explaining those two points separates rote answers from real operational understanding.

spring.datasource.hikari.maximum-pool-size=10
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=30000   # ms to wait for a free connection
spring.datasource.hikari.max-lifetime=1800000       # retire a connection before the DB does

Choosing a data starter: JDBC, Spring Data JDBC, or JPA

Boot offers three levels of abstraction over the same DataSource. spring-boot-starter-jdbc brings JdbcTemplate/JdbcClient — you write the SQL and Spring handles the boilerplate. spring-boot-starter-data-jdbc adds Spring Data repositories with a deliberately simple model — no lazy loading, no proxies — that still maps to plain SQL. spring-boot-starter-data-jpa brings Hibernate and the full JPA/ORM model: entity graphs, lazy loading and a persistence context.

Why it matters: "JDBC vs JPA vs Spring Data JPA — when would you pick each?" is one of the most common Spring data questions. JPA is powerful but hides a lot (N+1 queries, lazy-initialization exceptions); Spring Data JDBC is simpler and more predictable. The deep ORM mechanics belong to a dedicated JPA track — here the point is which starter to reach for and what Boot auto-wires for each: repository scanning, a transaction manager, and dialect detection.

SQL initialization scripts: schema.sql and data.sql

Boot can run plain SQL at startup: place schema.sql (DDL) and data.sql (DML) on the classpath and Boot executes them against the DataSource — schema first, then data. It is controlled by spring.sql.init.* (mode, schema-locations, data-locations, platform). By default this basic initialization runs only for embedded databases; set spring.sql.init.mode=always to enable it for a real one.

Why it matters: this is the quick path for demos, tests and seeding an in-memory H2 — but it is not schema management. There is no versioning, no history, no ordering beyond two files. A common gotcha: when Hibernate also creates the schema, data.sql may run before it, so you set spring.jpa.defer-datasource-initialization=true to fix the ordering. And you must not mix these scripts with Flyway or Liquibase — one tool should own the schema.

spring.sql.init.mode=always            # run schema.sql/data.sql for non-embedded DBs too
spring.sql.init.schema-locations=classpath:sql/schema.sql
spring.jpa.defer-datasource-initialization=true   # let Hibernate build the schema before data.sql

Schema migrations: Flyway vs Liquibase

For anything past a demo you want versioned, repeatable migrations. Boot auto-detects Flyway or Liquibase on the classpath and runs pending migrations at startup, before the JPA EntityManagerFactory initializes, so the schema is ready. Flyway uses versioned SQL files (V1__init.sql, V2__add_orders.sql) applied in order and tracked in a flyway_schema_history table. Liquibase uses a changelog (XML, YAML, JSON or SQL) of changesets — database-agnostic and tracked in DATABASECHANGELOG.

Why it matters: "Flyway vs Liquibase" is a staple comparison. Flyway is SQL-first, simple and linear — great when you are comfortable writing database-specific SQL. Liquibase abstracts changes into a database-independent changelog with contexts, labels, preconditions and (paid) rollback — more power, more ceremony. Both make schema changes reviewable, ordered and reproducible across environments, which is the real reason teams adopt them over hand-run scripts. Boot config lives under spring.flyway.* and spring.liquibase.*.

# Flyway: versioned scripts in classpath:db/migration
#   V1__create_orders.sql, V2__add_index.sql
spring.flyway.enabled=true
# Liquibase alternative — a master changelog of changesets:
spring.liquibase.change-log=classpath:db/changelog/db.changelog-master.yaml

Interview questions

Data access is a guaranteed section in Indian Spring Boot interviews — HikariCP tuning and "Flyway vs Liquibase" come up in nearly every backend round:

Beginner What does Spring Boot auto-configure when it finds a JDBC driver and datasource properties?

A pooled DataSource bean (HikariCP by default), an auto-configured JdbcTemplate/JdbcClient, and a transaction manager. For an embedded database it needs no URL at all — it detects the driver and starts an in-memory instance.

Beginner Which connection pool does Spring Boot use by default, and how does it get there?

HikariCP, since Spring Boot 2.0. It is pulled in transitively by spring-boot-starter-jdbc and spring-boot-starter-data-jpa, so no extra dependency is needed. You configure it under spring.datasource.hikari.*.

Beginner Which properties configure a production datasource?

spring.datasource.url, spring.datasource.username and spring.datasource.password; the driver class is inferred from the URL. Pool settings go under spring.datasource.hikari.*.

Beginner Where do schema.sql and data.sql go, and in what order do they run?

On the classpath (usually src/main/resources). Boot runs schema.sql first, then data.sql. By default this happens only for embedded databases unless you set spring.sql.init.mode=always.

Intermediate HikariCP: why can a smaller pool outperform a larger one, and what is max-lifetime for?

More connections than the database can serve in parallel cause contention and context-switching, so a modest pool often yields higher throughput — HikariCP's own guidance. max-lifetime retires each connection a little before the database's own idle/timeout, so the pool never hands out a connection the server has just closed.

Intermediate JdbcTemplate vs Spring Data JDBC vs Spring Data JPA — when would you pick each?

JdbcTemplate/JdbcClient: full SQL control, minimal magic. Spring Data JDBC: repositories over plain SQL, no lazy loading or proxies — simple and predictable. Spring Data JPA: full ORM via Hibernate — entity graphs, lazy loading, a persistence context — powerful but with more to reason about (N+1, lazy-init exceptions).

Intermediate data.sql runs, but the table does not exist yet because Hibernate manages the schema. How do you fix it?

Set spring.jpa.defer-datasource-initialization=true so Hibernate creates the schema before data.sql executes. Better still, hand schema ownership to a migration tool and keep data.sql for embedded/test seeding only.

Intermediate How does Boot run Flyway or Liquibase, and when relative to JPA?

If Flyway or Liquibase is on the classpath, Boot auto-configures it and runs pending migrations at application startup — before the JPA EntityManagerFactory is initialized — so the schema is in place when entities are validated or mapped.

Intermediate Flyway vs Liquibase — the core differences?

Flyway is SQL-first with linear versioned files (V1__, V2__) tracked in flyway_schema_history. Liquibase uses a database-agnostic changelog (XML/YAML/JSON/SQL) of changesets with contexts, labels and preconditions, tracked in DATABASECHANGELOG. Flyway is simpler; Liquibase is more portable and feature-rich.

Intermediate Why prefer a migration tool over spring.jpa.hibernate.ddl-auto=update in production?

ddl-auto=update is non-deterministic, cannot perform data migrations or drops, and leaves no history or review trail. Migrations are versioned, ordered, reviewable and reproducible across environments. The common production combo is ddl-auto=validate plus Flyway or Liquibase owning the schema.

Pro Can you mix schema.sql/data.sql with Flyway, and should you?

Boot can technically run both, but you should not: mixing basic SQL init with a migration tool creates conflicting schema ownership, and Boot is removing support for mixing. Let one tool own the schema and use data.sql only for embedded or test seeding.

Pro A Flyway migration that was already applied gets edited, and startup now fails with a checksum error. What happened, and how do you handle it?

Flyway stores a checksum of each applied migration in flyway_schema_history; editing an already-applied file changes the checksum and validation fails by design — applied migrations are immutable. Fix forward with a new versioned migration. Use flyway repair only to realign the history table for legitimate cases (a failed or edited pending migration), never to paper over changing applied history in shared environments.
