Module: Externalized Configuration

One jar, every environment: Spring Boot reads settings from outside the code and merges a dozen sources into a single Environment. Knowing that merge order — and when to reach for @ConfigurationProperties over @Value — is what interviewers probe here.

Knowledge Library → Courses → Spring Boot → Externalized Configuration

What this module covers

Externalized configuration is how one Spring Boot artifact adapts to dev, test and production without a rebuild. This module covers where properties come from and which source wins, how profiles and relaxed binding shape them, and the two ways to read config into your code: @Value for a single value and type-safe @ConfigurationProperties for grouped, validated config. Property precedence and the @Value-vs-@ConfigurationProperties choice are among the most-asked Spring Boot interview topics.

Topics

Configuration files, property sources and precedence

Spring Boot loads application.properties or application.yaml from a defined set of locations (classpath root, a /config subfolder, the working directory, and external config/ folders) and folds them, together with command-line arguments, JVM system properties, OS environment variables and more, into one merged Environment. The value you actually get is decided by a strict precedence order: later sources override earlier ones.

Why it matters: the order is the single most-tested externalized-config fact. Command-line arguments beat JVM system properties, which beat OS environment variables, which beat the packaged application.properties, which beats framework defaults. That is exactly why an ops team can retune a deployed jar with a --flag or an env var and never touch the build.

# application.properties, packaged in the jar (a low-priority source)
server.port=8080

# an environment variable overrides the packaged file...
export SERVER_PORT=9090
# ...and a command-line argument overrides even that
java -jar app.jar --server.port=9091   # 9091 wins

Profiles: environment-specific configuration

A profile is a named slice of configuration and beans that is only active in some environments. Set spring.profiles.active (via a property, env var or --spring.profiles.active) to switch it on; Spring Boot then automatically layers application-{profile}.properties or application-{profile}.yaml on top of the base file, with the profile-specific values winning. Beans can be gated with @Profile("prod"), and profile groups let one name (say prod) activate several sub-profiles at once.

Why it matters: profiles are the idiomatic answer to "how do you keep dev and prod settings apart in one build?" — a guaranteed interview follow-up to any precedence question. In YAML you can even keep all environments in a single multi-document file, each section guarded by spring.config.activate.on-profile.

# application.yml
spring:
  profiles:
    active: dev
    group:
      prod: "proddb,prodmq"      # one name activates several
---
# a second document, applied only when 'dev' is active
spring:
  config:
    activate:
      on-profile: dev
server:
  port: 8081

@Value and property placeholders

@Value injects a single property into a field or constructor parameter through a ${...} placeholder, resolved against the merged Environment. A colon supplies a fallback (${app.name:MyApp}) so a missing key does not crash startup, and the #{...} syntax evaluates a full SpEL expression when you need a computed value rather than a literal lookup.

Why it matters: @Value is perfect for one-off values, but it is stringly-typed, scattered across the codebase, and easy to typo. That trade-off against @ConfigurationProperties is the comparison interviewers reach for — so know both the mechanics and the boundary.

@Value("${app.name:MyApp}")                            // ':MyApp' is the fallback
private String name;

@Value("#{systemProperties['user.region'] ?: 'IN'}")   // SpEL expression
private String region;

@ConfigurationProperties: type-safe and constructor binding

@ConfigurationProperties(prefix = "...") binds a whole group of properties onto a POJO or record in one shot: type-safe, IDE-navigable, and validatable with @Validated. Relaxed binding means the same field is reached by app.mail.start-tls (kebab-case, the canonical form), app.mail.startTls (camelCase) or the environment variable APP_MAIL_START_TLS — you write the property once and every convention maps to it. Register the bean with @EnableConfigurationProperties, package scanning via @ConfigurationPropertiesScan, or a plain stereotype annotation.

Why it matters: this is the top externalized-config interview topic. @Value pulls one value; @ConfigurationProperties models a structured, prefixed group and is what Spring itself recommends for anything beyond a single setting. Using a record (or @ConstructorBinding) makes the config immutable and available at construction time — the modern, preferred style.

@ConfigurationProperties(prefix = "app.mail")
public record MailProperties(String host, int port, boolean startTls) { }

// Binds app.mail.host, app.mail.port, app.mail.start-tls
// Relaxed binding also accepts APP_MAIL_START_TLS from the environment

Interview questions

Externalized config is a screening staple in Indian Spring Boot interviews — precedence and the @Value-vs-@ConfigurationProperties call come up almost every time:

Beginner What is externalized configuration and why does Spring Boot promote it?

Keeping settings (ports, URLs, credentials, feature flags) outside the compiled code so one artifact runs unchanged across environments. Spring Boot merges files, env vars, system properties and command-line args into a single Environment, letting you retune a deployed jar without a rebuild — the foundation of twelve-factor config.

Beginner Which wins: a command-line argument, an environment variable, or application.properties?

The command-line argument. Broad order (highest first): command-line args > SPRING_APPLICATION_JSON > JVM system properties (-D) > OS environment variables > profile-specific and then plain application.properties > framework defaults. Later sources override earlier ones for the same key.

Beginner What is a Spring profile and how do you activate one?

A named set of config and beans active only in certain environments. Activate with spring.profiles.active=dev in a property, the SPRING_PROFILES_ACTIVE env var, or --spring.profiles.active=dev. Spring Boot then also loads application-dev.properties on top of the base file.

Beginner application.properties or application.yml — what is the difference?

Same capabilities, different format. Properties is flat key=value; YAML is hierarchical and less repetitive, and can hold multiple profile documents separated by ---. YAML cannot be loaded by @PropertySource and is whitespace-sensitive. If both exist, properties is loaded last and so wins on conflicts.

Intermediate @Value vs @ConfigurationProperties — when do you use each?

@Value("${key}") injects one value and supports SpEL and inline defaults; good for a single, one-off setting. @ConfigurationProperties(prefix) binds a group of related keys onto a type-safe object with relaxed binding and validation. Spring recommends @ConfigurationProperties for anything structured; reach for @Value only for isolated values. This is the single most-asked question in this area.

Intermediate What is relaxed binding? Give examples.

Binding that tolerates naming-style differences so one property is reachable several ways. app.remote-address (kebab, canonical), app.remoteAddress (camel) and APP_REMOTEADDRESS (uppercase env var, dots to underscores, dashes stripped) all bind to the same field. It is why SPRING_DATASOURCE_URL maps cleanly to spring.datasource.url. Relaxed binding applies to @ConfigurationProperties, not to @Value.

Intermediate How do profile-specific property files work, and which value wins?

When profile dev is active, application-dev.properties is loaded in addition to the plain application.properties, and the profile-specific value overrides the base one (last-wins). External files still override packaged ones, so a profile file next to the jar beats the same profile file inside it.

Intermediate How do you give @Value a default when the property is absent?

Put a colon and the fallback inside the placeholder: @Value("${app.timeout:5000}"). Without a default, a missing key throws at startup (placeholder cannot be resolved). For SpEL you use the Elvis operator: @Value("#{someBean.value ?: 'fallback'}").

Intermediate What do @EnableConfigurationProperties and @ConfigurationPropertiesScan do?

They register @ConfigurationProperties classes as beans. @EnableConfigurationProperties(MailProperties.class) registers named classes explicitly; @ConfigurationPropertiesScan scans packages so you do not list each one (or you annotate the class as a @Component). Without one of these the binding never happens and injection fails.

Pro What is constructor binding and why prefer an immutable @ConfigurationProperties?

Instead of setters, Spring binds properties through the constructor — automatic for Java records and for classes using @ConstructorBinding. The result is immutable, final-field config that is fully initialised at creation and safe to share. It needs registration via @EnableConfigurationProperties or scanning; it cannot be a setter-bound @Component at the same time.

Pro Map SPRING_DATASOURCE_URL to a property, and explain the rules that make it work.

It binds to spring.datasource.url. Environment-variable relaxed binding: uppercase the canonical name, replace dots with underscores, and drop dashes. So spring.datasource.url becomes SPRING_DATASOURCE_URL, and a kebab key like app.read-timeout becomes APP_READTIMEOUT. This is the standard way to inject config in Docker and Kubernetes without touching files.

Pro Where do profile groups and multi-document YAML fit, and what is a common pitfall?

Profile groups (spring.profiles.group.prod=proddb,prodmq) let activating prod switch on several sub-profiles together. Multi-document YAML keeps per-environment blocks in one file, each guarded by spring.config.activate.on-profile. Pitfall: spring.profiles.active may not be set inside a profile-specific document or one activated by on-profile — Spring Boot rejects it to avoid circular profile activation.
