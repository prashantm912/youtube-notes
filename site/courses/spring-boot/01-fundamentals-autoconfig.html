Module: Fundamentals and Auto-configuration

Auto-configuration is the trick that lets a three-line main method boot a web server, a datasource and a JSON mapper — and explaining how Spring Boot decides what to wire is the single most-asked question this framework produces.

Knowledge Library → Courses → Spring Boot → Fundamentals and Auto-configuration

What this module covers

Two things that together define "Boot" as opposed to plain Spring: the entry-point trio (@SpringBootApplication, the SpringApplication bootstrapper, and starter dependencies with their opinionated defaults), and the auto-configuration engine that turns a classpath into a running application. The second half — how conditions, the AutoConfiguration.imports registry, and the --debug conditions report interact — is the highest-yield interview material in the whole track, so it gets the deeper treatment.

Topics

@SpringBootApplication, SpringApplication, starters and opinionated defaults

A Boot app starts from one annotated class and one line: @SpringBootApplication is a meta-annotation bundling @SpringBootConfiguration (a @Configuration variant), @EnableAutoConfiguration (switches on the engine below), and @ComponentScan (scans this package downward — which is why the main class belongs at the root package). SpringApplication.run(...) then bootstraps the context: it creates the right ApplicationContext for the detected application type, prints the banner, publishes lifecycle events, runs failure analyzers, and hands you a fully wired container.

Why it matters: starters (spring-boot-starter-web, -data-jpa, -test…) are curated dependency descriptors — one coordinate pulls a consistent, version-managed set via the spring-boot-dependencies BOM, so you stop hand-picking versions. Together with auto-configuration this is what "convention over configuration" and "opinionated defaults" actually mean: sensible choices you can always override, not magic you can't see.

@SpringBootApplication   // = @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan
public class StoreApplication {
    public static void main(String[] args) {
        SpringApplication.run(StoreApplication.class, args);
    }
}

Auto-configuration mechanics: @Conditional, AutoConfiguration.imports, --debug

@EnableAutoConfiguration tells Boot to configure beans by guessing from the classpath. The mechanism is concrete, not magic: each starter jar publishes a META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports file listing @AutoConfiguration classes (this replaced the older spring.factories entry). Boot loads every listed class, but each one is guarded by @Conditional annotations — @ConditionalOnClass (is the library present?), @ConditionalOnMissingBean (has the app already defined one?), @ConditionalOnProperty, and friends — so a configuration only takes effect when its conditions pass.

Why it matters: @ConditionalOnMissingBean is what makes auto-configuration non-invasive — define your own DataSource and Boot backs off. When you need to know exactly what was applied and why, start with --debug: it logs a conditions report grouped into positive, negative and unconditional matches (the Actuator conditions endpoint shows the same data at runtime). Being able to read that report — and knowing where the import file lives — is what separates a memorized answer from a real one.

@AutoConfiguration
@ConditionalOnClass(DataSource.class)      // only if the JDBC types are on the classpath
public class MyDataSourceAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean               // back off if the app already declares a DataSource
    DataSource dataSource() { /* sensible default */ }
}
// registered in: META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports

Interview questions

Most-asked fundamentals-and-auto-configuration questions in Indian interviews — auto-configuration internals are a standard product-company deep dive, not just a screening topic:

Beginner What does @SpringBootApplication do?

It is a convenience meta-annotation combining three: @SpringBootConfiguration (a @Configuration class), @EnableAutoConfiguration (turns on auto-configuration), and @ComponentScan (scans the annotated class's package and below). One annotation, three behaviours.

Beginner What is a Spring Boot starter?

A curated dependency descriptor — one Maven/Gradle coordinate (e.g. spring-boot-starter-web) that pulls a consistent, version-managed group of libraries so you don't assemble and version them by hand. It brings the jars; auto-configuration then wires them.

Beginner What does SpringApplication.run() actually do?

It bootstraps the application: detects the application type and creates the matching ApplicationContext, prints the banner, registers/refreshes the context (triggering component scan and auto-configuration), publishes lifecycle events, runs any CommandLineRunner/ApplicationRunner beans, and returns the running context.

Beginner Why must the main class sit in the root package?

Because @ComponentScan (via @SpringBootApplication) scans only that package and its sub-packages. Put the main class at the top so every component below is discovered; a class in a sibling package is silently missed.

Beginner How do you exclude a specific auto-configuration?

@SpringBootApplication(exclude = { DataSourceAutoConfiguration.class }), or excludeName for a string, or the spring.autoconfigure.exclude property. Common when you want to manage a piece (like the datasource) yourself.

Intermediate How does Spring Boot know which auto-configuration classes to load?

Each jar publishes META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports listing its @AutoConfiguration classes, one per line. @EnableAutoConfiguration loads all of them; each is then filtered by its @Conditional guards. (This file replaced the old spring.factories EnableAutoConfiguration key.)

Intermediate What role do @Conditional annotations play?

They make configuration classpath- and context-aware. @ConditionalOnClass applies a config only if a type is present, @ConditionalOnMissingBean only if you haven't defined the bean, @ConditionalOnProperty only if a property is set, plus web/expression variants — so beans switch on exactly when appropriate.

Intermediate Why is @ConditionalOnMissingBean the key to overriding defaults?

Auto-configured beans are declared with it, so they register only when you haven't provided your own. Define your bean of that type and Boot's default backs off automatically — this is what makes auto-configuration non-invasive: your configuration always wins.

Intermediate How do you find out which auto-configurations were applied, and why?

Start the app with --debug: it prints a conditions evaluation report split into positive matches, negative matches (with the condition that failed), and unconditional classes. At runtime the Actuator conditions endpoint exposes the same report as JSON.

Intermediate Difference between @Configuration and @AutoConfiguration?

@AutoConfiguration is meta-annotated with @Configuration(proxyBeanMethods = false) but is specifically for library-supplied config: it is discovered via the AutoConfiguration.imports file (not component scan), supports ordering with @AutoConfigureBefore/After, and always runs after user-defined @Configuration so user beans take precedence.

Pro How is the ordering between auto-configuration classes controlled?

With @AutoConfigureBefore, @AutoConfigureAfter, and @AutoConfigureOrder on the auto-configuration class. Ordering matters because conditions like @ConditionalOnBean depend on whether an earlier config already registered a bean — get the order wrong and the condition evaluates against an incomplete context.

Pro Walk through writing a custom starter with its own auto-configuration.

Create an @AutoConfiguration class guarded by @ConditionalOnClass/@ConditionalOnMissingBean, expose settings via a @ConfigurationProperties type, and list the class in META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. Ship it as an -autoconfigure module plus a thin -starter module that just declares dependencies. Note Spring Boot 4's modularization split many autoconfigurations into their own modules, so upgrades can require pulling in the new module explicitly.
