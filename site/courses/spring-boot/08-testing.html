Module: Testing with Boot

Boot's test support lets you load exactly as much of the application as a test needs — the whole context, or a single slice — and interview panels probe hard on whether a candidate grasps the difference and its cost.

Knowledge Library → Courses → Spring Boot → Testing with Boot

What this module covers

Spring Boot ships a testing toolkit on top of the Spring TestContext framework: the spring-boot-starter-test bundle, @SpringBootTest for full-context integration tests, and a family of test slices that boot only one layer. It also owns the web-layer test clients (MockMvc, WebTestClient) and, since Boot 3.1, first-class Testcontainers integration through @ServiceConnection. Slices and the orElse-style trade-offs between them are among the most-asked Spring Boot interview topics.

Topics

@SpringBootTest and the test starter

Adding spring-boot-starter-test pulls in JUnit Jupiter, Spring Test, AssertJ, Hamcrest, Mockito, JSONassert and JsonPath in one dependency — the assumed baseline for every Boot test. @SpringBootTest then bootstraps a real ApplicationContext from your @SpringBootApplication class, so beans are wired exactly as in production; its webEnvironment attribute decides whether a mock or a real embedded server comes up.

Why it matters: a full-context test is the highest-fidelity and the slowest option, so you reach for it deliberately — for flows that cross several layers — and lean on the context cache and @MockitoBean (the successor to @MockBean) to keep the suite fast. Knowing when not to use @SpringBootTest is what separates a screening answer from a senior one.

@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
class OrderApiTests {
    @Autowired TestRestTemplate rest;      // real HTTP against an embedded server
    @MockitoBean PaymentGateway gateway;   // swap one collaborator in the context
}

Test slices: @WebMvcTest, @DataJpaTest and the family

A test slice boots a narrow, purpose-built context instead of the whole application: @WebMvcTest loads the Spring MVC infrastructure and your controllers (plus MockMvc) but no @Service or @Repository beans — you supply those as mocks; @DataJpaTest loads JPA, your repositories and a test datasource, wraps each test in a transaction and rolls it back afterwards. The appendix lists the full set: @JsonTest, @RestClientTest, @DataJdbcTest, @JooqTest and more.

Why it matters: slices load a fraction of the beans, so they start faster and fail more precisely — and because Spring caches contexts by configuration, consistent slices are reused across the suite. This is the single most-tested Boot testing topic; expect to explain exactly what a given slice does and does not wire up.

@WebMvcTest(OrderController.class)      // only the web layer boots
class OrderControllerTests {
    @Autowired MockMvc mvc;
    @MockitoBean OrderService service;  // the real service is NOT in this context
}

@DataJpaTest                            // only JPA + a test datasource; rolls back per test
class OrderRepositoryTests {
    @Autowired OrderRepository repo;
}

Driving the web layer: MockMvc and WebTestClient

MockMvc performs full Spring MVC request handling against mock request and response objects — no socket, no running server — which is why it pairs with @WebMvcTest and with @SpringBootTest(webEnvironment = MOCK). WebTestClient is the fluent alternative: it drives WebFlux endpoints, can plug MockMvc in as its server for MVC apps, or bind to a live server started with RANDOM_PORT for true end-to-end HTTP — the same test API either way.

Why it matters: choosing the right client is a common follow-up to the slice question. MockMvc for controller-level assertions without a server, WebTestClient (or TestRestTemplate) when a real port is up. Both let you assert status, headers and JSON with JsonPath in one chain.

mvc.perform(get("/orders/42"))
   .andExpect(status().isOk())
   .andExpect(jsonPath("$.id").value(42));

Testcontainers and @ServiceConnection

Testcontainers spins up throwaway Docker containers — Postgres, Kafka, Redis and more — so integration tests run against the real engine instead of an in-memory stand-in. Before Boot 3.1 you wired the container's host and port into the context by hand with @DynamicPropertySource; @ServiceConnection (Boot 3.1+, and the current recommended approach) replaces that boilerplate by generating the right ConnectionDetails bean straight from the container, matching it to the service automatically.

Why it matters: production-like data tests are now a baseline expectation, and Testcontainers plus @ServiceConnection is the idiom interviewers look for. Keep the container static so it is shared across the class — startup is expensive — and remember it needs the spring-boot-testcontainers module and a running Docker environment.

@SpringBootTest
@Testcontainers
class OrderIntegrationTests {
    @Container
    @ServiceConnection                    // JDBC url/user/password wired into the context
    static PostgreSQLContainer<?> db = new PostgreSQLContainer<>("postgres:16");
}

Interview questions

Most-asked Spring Boot testing questions in Indian interviews — test slices are a screening-round staple from service companies to product teams:

Beginner What does @SpringBootTest do, and when do you reach for it?

It bootstraps a full ApplicationContext from your @SpringBootApplication class, wiring beans as in production. Use it for integration tests that cross several layers; for a single layer, prefer a slice — the full context is the slowest, highest-fidelity option.

Beginner What comes bundled in spring-boot-starter-test?

JUnit Jupiter, Spring Test and Spring Boot Test, AssertJ, Hamcrest, Mockito, JSONassert and JsonPath — one dependency that covers most testing needs, which is why it is the assumed baseline.

Beginner What is a test slice, and why use one?

A slice annotation boots only the beans and auto-configuration for one layer (web, JPA, JSON…) rather than the whole app. Result: faster startup, fewer beans to go wrong, and failures scoped to that layer.

Beginner What does @WebMvcTest load — and what does it deliberately leave out?

It loads the Spring MVC infrastructure, your @Controller/@RestController beans, and auto-configures MockMvc. It does NOT load @Service, @Repository or @Component beans — you provide collaborators as @MockitoBean mocks.

Intermediate @WebMvcTest vs @SpringBootTest with @AutoConfigureMockMvc — what is the difference?

Both give you a MockMvc. @WebMvcTest loads only the web slice, so service/repository beans are absent and must be mocked; @SpringBootTest loads the entire context, so real collaborators are present. The slice is faster and more focused; the full context is a genuine integration test.

Intermediate What does @DataJpaTest configure, and what is its default transaction behaviour?

It boots JPA, your Spring Data repositories and a test datasource (by default replacing your DataSource with an embedded one). Each test runs in a transaction that is rolled back at the end, so tests stay isolated and leave no data behind.

Intermediate MockMvc vs WebTestClient vs TestRestTemplate — when do you use each?

MockMvc: mock servlet request handling, no server — controller-level tests. WebTestClient: fluent client for WebFlux, or for MVC (via MockMvc) or a live server. TestRestTemplate: real HTTP against an embedded server started with RANDOM_PORT/DEFINED_PORT.

Intermediate What does @MockitoBean (formerly @MockBean) do, and what is the catch?

It adds or replaces a bean in the ApplicationContext with a Mockito mock. The catch: it changes the context definition, so Spring cannot reuse a cached context — you get a fresh one, which can slow a suite if applied inconsistently across many tests.

Intermediate Why do slice tests run faster than @SpringBootTest, and how does context caching fit in?

Slices instantiate far fewer beans, so each context starts quickly. Spring also caches contexts keyed by their configuration and reuses them across test classes; tests sharing an identical slice configuration share one context. @MockitoBean and @DirtiesContext create new cache entries and forfeit that reuse.

Intermediate What problem does @ServiceConnection solve, and since which version?

Since Boot 3.1. It reads a Testcontainers container and auto-generates the matching ConnectionDetails bean, wiring host/port/credentials into the context automatically — removing the manual @DynamicPropertySource plumbing you needed before.

Pro What are the webEnvironment options of @SpringBootTest and what changes with each?

MOCK (default): a mock servlet environment, no real port — pair with MockMvc. RANDOM_PORT and DEFINED_PORT: a real embedded server on a random or fixed port — use WebTestClient or TestRestTemplate. NONE: context loads with no web environment at all.

Pro @DataJpaTest replaces your datasource by default — how do you test against the real database engine?

Annotate with @AutoConfigureTestDatabase(replace = Replace.NONE) so Boot keeps your configured DataSource, then point it at a Testcontainers instance (ideally via @ServiceConnection). Now the repository tests run against the same engine as production, catching dialect and migration issues an embedded database would hide.

Pro Why keep a Testcontainer static, and how do @Container, @ServiceConnection and @DynamicPropertySource relate?

Container startup is expensive, so a static field is started once and shared across all methods in the class (a JUnit-lifecycle detail), rather than per test. @Container manages the container lifecycle; @ServiceConnection is the modern way to feed its connection details into the context; @DynamicPropertySource is the manual predecessor, still useful for properties no service connection covers.
