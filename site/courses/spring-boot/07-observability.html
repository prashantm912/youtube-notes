Module: Observability

Actuator tells you the app is up; observability tells you why a request was slow. This module maps Boot's three pillars — metrics, traces, logs — to the Micrometer stack that produces them, and points you at the resources that teach each properly.

Knowledge Library → Courses → Spring Boot → Observability

What this module covers

Observability in Spring Boot is delivered through Micrometer: an application metrics facade (think SLF4J, but for telemetry) that Boot auto-configures. This module covers the meter model and common tags, exporting to Prometheus, distributed tracing via Micrometer Tracing over Brave or OpenTelemetry, and structured JSON logging with trace-correlated IDs. It builds directly on the Actuator module — the same /actuator surface exposes /prometheus and drives probes.

Topics

Micrometer: meters, registries and common tags

Micrometer is the metrics facade underneath Spring Boot. You instrument against its API — Counter, Gauge, Timer, DistributionSummary — and Boot auto-configures a composite MeterRegistry that fans each meter out to whatever monitoring backends are on the classpath, without vendor lock-in. Out of the box you already get JVM, system, HTTP-server, datasource and cache meters.

Why it matters: every meter carries dimensions (tags). http.server.requests tagged by uri, method and status is what lets you slice one metric a hundred ways. Common tags (region, instance, stack) are stamped onto every meter for environment-level drill-down — knowing how and where to set them is standard senior-round material.

management:
  metrics:
    tags:                 # common tags applied to every meter
      region: "ap-south-1"
      application: "orders-api"

Exporting to Prometheus

Prometheus is the pull-based default: add the micrometer-registry-prometheus dependency and Boot exposes an /actuator/prometheus endpoint in the text exposition format that a Prometheus server scrapes on an interval. Micrometer translates its dimensional meters into Prometheus time series, applying naming conventions (dots become underscores, base units appended).

Why it matters: Prometheus + Grafana is the de-facto open-source dashboarding stack in Indian product teams, and the endpoint must be deliberately exposed and secured — it is not on by default. Understanding the scrape model (and why high-cardinality tags blow up a Prometheus server) separates people who use a dashboard from people who can build one.

management.endpoints.web.exposure.include=health,info,prometheus
# scraped by Prometheus at GET /actuator/prometheus

Distributed tracing: Micrometer Tracing, OpenTelemetry and the Observation API

A trace follows one request across service boundaries; each hop is a span under a shared trace id. Micrometer Tracing is the facade (the successor to the now-deprecated Spring Cloud Sleuth) that bridges to a real tracer — OpenZipkin Brave or OpenTelemetry — and Boot auto-configures propagation so trace context rides along your RestClient/WebClient calls automatically. Sampling (management.tracing.sampling.probability, default 0.1) controls how many traces are actually recorded.

Why it matters: the unifying idea is the Observation API — you instrument a piece of work once and get a metric, a span, and log context out of the same recording. Interviewers probe whether you understand that one abstraction rather than three parallel libraries, and why you sample instead of tracing every request in production.

Observation.createNotStarted("order.checkout", registry)
    .lowCardinalityKeyValue("channel", "web")   // safe as a metric tag
    .highCardinalityKeyValue("orderId", id)      // trace-only detail
    .observe(() -> processCheckout(id));          // one meter + one span

Structured logging and correlation IDs

The third pillar is logs — but plain text logs are hard to query at scale. Since Boot 3.4 structured logging is built in: set logging.structured.format.console to ecs, gelf or logstash and each line becomes machine-readable JSON that ships straight into Elasticsearch, Graylog or Loki. When tracing is active, Boot injects the trace and span IDs into the logging MDC so every line can be correlated back to its request.

Why it matters: correlated logs are what turn "an error happened somewhere" into "click this trace id, see every log line for that exact request across three services." It is the payoff that ties the three pillars together, and the logging.pattern.correlation / structured-format properties are the concrete knobs interviewers expect you to name.

Interview questions

Most-asked observability questions in Indian interviews — metrics and tracing are a microservices-round staple:

Beginner What is Micrometer and how does Spring Boot use it?

A vendor-neutral metrics (and, with the Observation API, tracing) facade — "SLF4J for observability." You code against its API; Boot auto-configures a composite MeterRegistry that exports to whatever backend (Prometheus, Datadog, OTLP…) is on the classpath, so you can swap monitoring systems without changing instrumentation.

Beginner What are the three pillars of observability?

Metrics (aggregated numbers over time), traces (the path of one request across services), and logs (discrete events). Boot produces all three from the Micrometer stack, and correlation IDs stitch logs and traces together.

Beginner How do you expose Prometheus metrics from a Spring Boot app?

Add spring-boot-starter-actuator and micrometer-registry-prometheus, then expose the endpoint with management.endpoints.web.exposure.include=prometheus. Prometheus scrapes GET /actuator/prometheus. It is not exposed by default — that is deliberate.

Beginner What are tags (dimensions) on a meter, and what are common tags?

Key-value pairs that let one metric be sliced many ways — e.g. http.server.requests by uri/status. Common tags are values stamped onto every meter (region, instance, application) for environment-level drill-down; configured under management.metrics.tags.

Intermediate Counter vs Gauge vs Timer vs DistributionSummary?

Counter: a monotonically increasing count (requests served). Gauge: a value that can go up and down, sampled on scrape (queue depth). Timer: counts events and records their duration (latency). DistributionSummary: like a Timer but for non-time magnitudes (payload size). Timers/summaries can also emit percentiles and histograms.

Intermediate How does distributed tracing identify one request across services?

A trace id is generated at the entry point and propagated over the network (via headers like W3C traceparent); each unit of work is a span with its own span id and a parent pointer. Boot auto-instruments RestClient/WebClient/RestTemplate so the context travels automatically.

Intermediate What replaced Spring Cloud Sleuth in Spring Boot 3?

Micrometer Tracing. Sleuth was deprecated; its tracing functionality moved into Micrometer Tracing, a facade that bridges to Brave or OpenTelemetry. Naming it — and knowing it is a migration, not a rewrite of your instrumentation — is a common Boot 3 question.

Intermediate How do trace and span IDs end up in your logs?

When tracing is on, Boot populates the logging MDC with traceId/spanId and adds them to the default log pattern (customizable via logging.pattern.correlation). That is what lets you jump from a log line to the full trace, and vice versa.

Intermediate Brave vs OpenTelemetry — what do you actually choose between?

Both are tracer implementations behind Micrometer Tracing, selected by dependency: micrometer-tracing-bridge-brave (OpenZipkin Brave, pairs naturally with Zipkin) or micrometer-tracing-bridge-otel (OpenTelemetry, exports via OTLP to Tempo/Jaeger/most vendors). Your instrumentation code is identical either way.

Intermediate What is the Observation API and why "instrument once"?

A single Micrometer abstraction: you wrap work in an Observation and registered handlers derive a timer metric, a trace span, and log context from that one recording — instead of writing three separate instrumentations that can drift out of sync. @Observed is the annotation-driven shortcut.

Intermediate What is trace sampling and why not record every request?

Sampling records only a fraction of traces (management.tracing.sampling.probability, default 0.1 = 10%) to bound storage and overhead. In production you rarely trace 100% — the cost and volume are prohibitive; you raise it in dev or for targeted debugging.

Pro Low-cardinality vs high-cardinality tags — why does the distinction matter?

Every unique tag-value combination is a separate metric time series, so high-cardinality values (user id, order id, request id) as metric tags cause a cardinality explosion that can take down Prometheus. Rule: low-cardinality key-values go on metrics; high-cardinality detail belongs only on spans/traces — the Observation API makes you declare which is which.

Pro How does Micrometer map its meters onto Prometheus, and what naming rules apply?

Dimensional meters become Prometheus time series with tags as labels; Micrometer normalizes names to Prometheus conventions — dots to underscores, base units appended (_seconds, _bytes), and Timers expand into _count/_sum/_max (plus _bucket when histograms are enabled). This is why the metric name you register is not byte-for-byte what you query.

Pro Why use built-in structured logging over a Logstash encoder, and what formats does Boot support?

Since Boot 3.4 structured logging ships out of the box (logging.structured.format.console=ecs|gelf|logstash), so you get queryable JSON without adding a Logback encoder dependency or XML config — and it stays consistent across console and file, integrates with correlation IDs, and supports custom fields plus your own StructuredLogFormatter. The old LogstashEncoder route still works but is more setup and less integrated.
