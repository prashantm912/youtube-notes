Module: Control Flow and Errors

Code runs top to bottom until you tell it otherwise — branches choose a path, loops repeat one, and error handling decides what happens when a path fails. This module maps all three and points you to the definitive material for each.

Knowledge Library → Courses → JavaScript → Control Flow and Errors

What this module covers

Control flow is how a program decides what runs next: conditional branching with if/else and switch, repetition with the loop family (for, for-in, for-of, while, do-while, plus break/continue/labels), and error handling — try/catch/finally, throw, the built-in Error hierarchy, the cause option, and custom error classes. These are the wiring of every function you will ever write, and they turn up in almost every screening round.

Topics

if/else, switch

Conditional statements branch on a boolean test. if/else (and else if chains) handle ranges and arbitrary expressions; the condition is coerced to boolean, so falsy values (false, 0, "", null, undefined, NaN) take the else path. switch compares one value against discrete case labels using strict equality (===), reads cleanly when you have many exact matches, and falls through to the next case unless you break. The ternary ?: is the expression form of a two-way branch.

Why it matters: choosing the right construct is a readability and correctness call interviewers probe directly — switch for a fixed set of known values, if/else for ranges and compound conditions. Fall-through and the strict-equality rule (a string "1" never matches a number 1) are classic gotchas.

const grade = 82;
let letter;
if (grade >= 90) letter = "A";
else if (grade >= 80) letter = "B";
else letter = "C";

switch (letter) {           // matched with ===
  case "A":
  case "B":                 // shared body via fall-through
    console.log("Distinction");
    break;
  default:
    console.log("Pass");
}

Loops (for, for-in, for-of, while, do-while, break/continue/labels)

Loops repeat a block. The C-style for gives explicit index control; while checks its condition before each pass and do-while after (so it always runs at least once). For collections, prefer the iteration-aware forms: for-of walks the values of any iterable (arrays, strings, Map, Set, NodeList), while for-in walks the enumerable keys of an object — including inherited ones, which is why it is wrong for arrays. break exits a loop, continue skips to the next pass, and a leading label (outer:) lets either target an enclosing loop.

Why it matters: picking for-of over for-in for arrays, knowing that break/continue cannot escape an Array.forEach callback, and using a label to exit nested loops in one jump are recurring interview and code-review points. The var-versus-let binding in a loop counter is one of the most-asked JavaScript questions of all.

const nums = [10, 20, 30];
for (let i = 0; i < nums.length; i++) { /* index access */ }
for (const n of nums) { /* values: 10, 20, 30 */ }
for (const key in { a: 1, b: 2 }) { /* keys: "a", "b" */ }

outer: for (const row of grid) {
  for (const cell of row) {
    if (cell === target) break outer;   // exits BOTH loops
  }
}

Error handling (try/catch/finally, throw, Error hierarchy, cause, custom errors)

When something goes wrong you throw a value — by convention an Error (or a subclass like TypeError, RangeError, SyntaxError), which carries a name, message and stack. try/catch/finally contains the failure: the catch block runs only if the try threw, and finally runs either way — the place to release resources. Modern JavaScript adds the cause option (new Error(msg, { cause: original })) so you can rethrow a higher-level error without losing the one underneath, and custom subclasses of Error let callers branch on error type with instanceof.

Why it matters: throw-ing a real Error (not a bare string) preserves the stack trace; orElse-style silent swallowing hides bugs. Interviewers love the fact that a synchronous try/catch does not catch errors thrown later inside a setTimeout or an un-awaited promise — you need await or .catch() for those.

class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = "ValidationError";   // so err.name is meaningful
    this.field = field;
  }
}

try {
  throw new ValidationError("Email required", "email");
} catch (err) {
  if (err instanceof ValidationError) console.log(err.field);
  else throw new Error("Save failed", { cause: err });   // chain it
} finally {
  // always runs: close connections, clear timers
}

Interview questions

Most-asked control-flow and error-handling questions in Indian interviews — from service-company screening rounds to product-company deep dives:

Beginner When would you use switch instead of if/else?

Use switch when you compare one value against many discrete, known constants — it reads more clearly than a long else if chain and signals intent. Use if/else for ranges (x > 10), compound boolean conditions, or when each branch tests a different variable. switch matches with strict equality, so it cannot express ranges directly.

Beginner Which values are falsy in JavaScript?

Exactly six: false, 0 (and -0, 0n), "" (empty string), null, undefined, and NaN. Everything else is truthy — including "0", "false", [], and {}. An if condition coerces its test to boolean using these rules, which is why if (arr.length) works as an is-non-empty check.

Beginner Difference between for...in and for...of?

for-in iterates enumerable keys (property names, as strings), including inherited ones — meant for plain objects. for-of iterates values of an iterable (array, string, Map, Set). For arrays use for-of: for-in gives string indices, skips holes, and can pick up added properties, so it is a common bug.

Beginner What do break and continue do?

break exits the nearest enclosing loop (or switch) immediately. continue stops the current iteration and jumps to the next one — in a for loop it still runs the update step. Both act on the innermost loop unless you give them a label to target an outer loop.

Beginner How does the ternary operator work?

condition ? valueIfTrue : valueIfFalse — the only three-operand operator in JavaScript. It is an expression, so it returns a value you can assign or embed (e.g. in a template literal), unlike if which is a statement. Keep it to a single, readable choice; nested ternaries hurt readability.

Intermediate What is switch fall-through, and why does case matching use ===?

After a matching case, execution continues into the following cases until a break (or return) — deliberate when stacking labels to share a body, a bug when forgotten. Matching uses strict equality, so no type coercion happens: case 1 will not match the string "1". The switch(true) pattern exploits this to test boolean conditions per case.

Intermediate How do you break out of two nested loops at once?

Label the outer loop and target it: outer: for (...) { for (...) { if (done) break outer; } }. Without the label break only exits the inner loop. continue outer similarly skips to the next iteration of the outer loop. Alternatives are a flag variable or extracting the loops into a function and using return.

Intermediate Explain the execution order of try/catch/finally — and what happens if finally has a return?

try runs first; if it throws, control jumps to catch; finally runs last in every case, even after a return or throw in the try/catch. A return (or throw) inside finally overrides whatever the try/catch was going to return or throw — a subtle bug, so avoid control-flow statements in finally.

Intermediate Why throw new Error("msg") instead of throw "msg"?

You can throw any value, but throwing a string loses the stack trace, the name, and instanceof Error checks — debugging and centralized handling break. An Error object captures where it was created and lets callers branch on error type. Rule: always throw an Error or a subclass.

Intermediate What does this log, and how do you fix it: for (var i = 0; i < 3; i++) setTimeout(() => console.log(i))?

It logs 3, 3, 3. var is function-scoped, so all three callbacks close over the same i, which is 3 by the time the timers fire. Fix by using let (block-scoped — a fresh binding per iteration, logging 0, 1, 2), or by capturing the value in an IIFE. This is the classic loop-closure question.

Pro What is the Error cause option and when do you use it?

new Error("High-level message", { cause: originalError }) (ES2022) attaches the underlying error as err.cause. It lets you rethrow a domain-meaningful error at a boundary — e.g. "Failed to load profile" — while preserving the low-level cause (a network or parse error) for logs and debugging, instead of either swallowing it or leaking implementation details.

Pro How do you create a custom error class correctly?

Extend Error, call super(message) first, then set this.name = "YourError" (so it prints correctly and is distinguishable) and any custom fields. Callers then branch with instanceof YourError. In transpiled/old environments you may also restore the prototype (Object.setPrototypeOf(this, YourError.prototype)) so instanceof survives; native class syntax handles this for you.

Pro Does a synchronous try/catch catch errors from setTimeout or a promise? How do you handle those?

No. A try/catch only catches errors thrown while its try block is on the call stack. A callback that runs later (setTimeout, an event handler) or a rejected promise throws in a different tick, so the surrounding try/catch has already exited. Handle timer/callback errors inside the callback; for promises use await inside try/catch, a .catch() handler, and a global unhandledrejection listener as a safety net.
