Module: Security for JS Developers

Front-end code runs in a hostile environment: every string a user controls is a potential script, and every cross-origin call is a policy negotiation. This module maps the browser's security model and the defenses that keep an app from becoming the attacker's tool.

Knowledge Library → Courses → JavaScript → Security for JS Developers

What this module covers

Security is where JavaScript stops being about features and starts being about trust boundaries. This module orients you on the attacks a front-end developer owns — cross-site scripting first and foremost — and the platform mechanisms that contain them: the Sanitizer API and Trusted Types, Content Security Policy, the CORS handshake, cookie attributes, Subresource Integrity, clickjacking and prototype-pollution defenses, the Web Crypto API, and dependency hygiene. Depth lives in the linked specifications and cheat sheets; the goal here is a correct mental model of who can do what, and why.

Topics

XSS classes and defenses, Trusted Types + Sanitizer API

Cross-site scripting (XSS) is the browser attack every JavaScript developer owns: the attacker gets your page to execute their script, which then runs with the victim's origin and session — reading the DOM, calling your APIs with the user's cookies, and exfiltrating data. It comes in three classes: stored (malicious markup persisted server-side and served to every visitor), reflected (echoed straight back from a request parameter), and DOM-based (the dangerous flow never leaves the browser — untrusted input reaches a sink like innerHTML, document.write, eval, or an event-handler attribute).

The durable defense is discipline about sinks. Prefer textContent over innerHTML, use a templating layer that context-encodes by default, and when you truly must insert HTML, run it through a vetted sanitizer (DOMPurify) or the built-in Element.setHTML() from the HTML Sanitizer API, which parses and strips scripts for you. For a systemic guarantee, enforce Trusted Types with require-trusted-types-for 'script': the browser then rejects raw strings at every injection sink, accepting only objects minted by an audited policy — turning "did we miss a sink?" into a compile-and-review problem. Why it matters: XSS sits at the top of nearly every security interview and OWASP list.

el.innerHTML = userInput;      // XSS sink — never with untrusted data
el.textContent = userInput;    // safe: inserted as text, never parsed
el.setHTML(userInput);         // Sanitizer API: parses, then strips scripts

// Enforce platform-wide via a response header:
// Content-Security-Policy: require-trusted-types-for 'script'

CSP, CORS mechanics (preflight/credentials), cookies (SameSite/HttpOnly)

Content Security Policy is a response header that whitelists where scripts, styles, frames and other resources may come from. Its headline use is defense-in-depth against XSS — a strict CSP built on per-response nonces or hashes plus 'strict-dynamic' neutralizes injected inline scripts even when a sink slips through — but the same header also carries frame-ancestors (anti-clickjacking) and upgrade-insecure-requests. Roll it out with Content-Security-Policy-Report-Only so violations are reported without breaking the page.

CORS is often misread as a server protection; it is the opposite. The browser's same-origin policy blocks a page from reading a cross-origin response by default, and CORS is the server's opt-in to relax that, via Access-Control-Allow-Origin. "Simple" requests go straight through; anything with a non-simple method, a custom header, or a non-simple content type triggers a preflight — an OPTIONS request advertising the intended method and headers that the server must approve first. To send cookies cross-origin you need all three: fetch(url, {credentials: 'include'}), a server responding with Access-Control-Allow-Credentials: true, and an explicit (never wildcard) origin.

Cookies carry their own attributes. SameSite controls cross-site sending — Lax (the modern default) blocks cookies on cross-site subrequests but allows top-level navigation, Strict blocks even that, and None (which requires Secure) sends everywhere; this is the primary CSRF mitigation. HttpOnly hides the cookie from document.cookie, so an XSS payload cannot read a session token, and Secure restricts it to HTTPS.

Content-Security-Policy: default-src 'self'; script-src 'nonce-Rk9Pd2'
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Set-Cookie: sid=abc; HttpOnly; Secure; SameSite=Lax

SRI, clickjacking, prototype pollution, Web Crypto basics, supply-chain hygiene

Subresource Integrity closes a supply-chain gap: add an integrity attribute holding a base64 hash to a <script> or <link> and the browser refuses to run a CDN file that has been tampered with, as long as you also set crossorigin. Clickjacking is the framing attack — your real page is loaded invisibly over decoy UI so the victim's clicks land on your buttons; defend it with Content-Security-Policy: frame-ancestors 'none' (or 'self'), the legacy X-Frame-Options: DENY header, and SameSite cookies so framed requests carry no session.

Prototype pollution is JavaScript's own class of injection: an unsafe recursive merge or deep-clone that copies an attacker-controlled __proto__ or constructor.prototype key writes onto Object.prototype, so every object suddenly inherits a malicious property — a gadget that can escalate to denial of service, XSS, or remote code execution. Reject those keys, build lookup maps with Object.create(null) or a real Map, and freeze prototypes you never mutate. The Web Crypto API (crypto.subtle for hashing, signing and AES/RSA; crypto.getRandomValues and crypto.randomUUID() for cryptographically strong randomness) gives you real primitives, but it is low-level and secure-context-only — never hand-roll crypto around it. Finally, supply-chain hygiene is now table stakes: commit a lockfile, run npm audit, pin and minimize dependencies, and review packages' install scripts.

const id = crypto.randomUUID();                    // CSPRNG identifier
const bytes = crypto.getRandomValues(new Uint8Array(16));
const hash = await crypto.subtle.digest("SHA-256", data);

// Prototype-pollution guard when merging untrusted keys:
if (key === "__proto__" || key === "constructor") continue;

Interview questions

The security round is where Indian product-company interviews separate juniors from seniors — these recur from service-company screening to FAANG-adjacent deep dives, hardest last:

Beginner What is XSS and what are its three classes?

Cross-site scripting is getting a target site to execute attacker-supplied JavaScript, which then runs with the victim's origin and session. Three classes: stored (payload persisted server-side, served to everyone), reflected (echoed back from a request parameter), and DOM-based (untrusted input reaches a sink like innerHTML entirely in the browser).

Beginner innerHTML vs textContent for user-supplied data?

innerHTML parses the string as HTML, so <img onerror> or a script-bearing element executes — a classic XSS sink. textContent inserts the value as literal text; nothing is parsed or run. Default to textContent; reach for sanitized HTML only when you genuinely need markup.

Beginner What does the HttpOnly cookie attribute do?

It forbids JavaScript from reading the cookie through document.cookie, so an XSS payload cannot steal a session token. The cookie is still attached to requests the browser makes, so authentication keeps working — it only removes script-level read access.

Beginner Does CORS protect my server? What is it actually for?

No. CORS relaxes a browser rule — the same-origin policy that stops a page from reading a cross-origin response. The server opts in with Access-Control-Allow-Origin. It is not authorization: a non-browser client ignores CORS entirely, so your API must still authenticate and authorize every request.

Beginner What is CSP and how do you deliver it?

Content Security Policy is a browser-enforced allowlist of resource sources, delivered as the Content-Security-Policy response header (a <meta> tag covers a subset). It is defense-in-depth against XSS and can also block framing via frame-ancestors. Use Content-Security-Policy-Report-Only to trial a policy without breaking the page.

Intermediate When does the browser send a CORS preflight, and what is in it?

When a request is not "simple" — a method beyond GET/POST/HEAD, a custom header, or a content type other than the three form-safe ones. The browser first sends an OPTIONS preflight carrying Access-Control-Request-Method and Access-Control-Request-Headers; only if the server approves them does the real request go out.

Intermediate How do you send cookies on a cross-origin fetch?

Three things must all line up: the client sets fetch(url, {credentials: 'include'}), the server returns Access-Control-Allow-Credentials: true with a specific origin (a wildcard * is rejected with credentials), and the cookie itself must be SameSite=None; Secure to be sent cross-site at all.

Intermediate SameSite Lax vs Strict vs None — and why is Lax the default?

Strict never sends the cookie on any cross-site request; Lax withholds it on cross-site subrequests but allows it on top-level navigations (so following a link keeps you logged in); None sends it everywhere but demands Secure. Browsers default absent cookies to Lax because it blocks most CSRF while keeping ordinary navigation working.

Intermediate What are Trusted Types and how do they stop DOM XSS?

With Content-Security-Policy: require-trusted-types-for 'script', dangerous DOM sinks stop accepting plain strings and accept only TrustedHTML/TrustedScript objects produced by a named policy you define. Every unsafe assignment either goes through the audited policy or throws — converting an open-ended "find every sink" problem into a small, reviewable set of policies.

Intermediate How do you prevent clickjacking?

Stop other sites from framing yours: Content-Security-Policy: frame-ancestors 'none' (or 'self') is the modern control, with X-Frame-Options: DENY as legacy backup. Pair it with SameSite cookies so any framed request carries no session, defeating the point of the overlay.

Intermediate How does Subresource Integrity protect a page using a CDN?

You add integrity="sha384-..." (plus crossorigin) to the <script> or <link>. The browser hashes the fetched file and refuses to execute it if the hash does not match — so a compromised or swapped CDN asset simply fails to load instead of running attacker code.

Intermediate How do you reduce npm supply-chain risk?

Commit a lockfile so installs are reproducible, run npm audit (and keep it in CI), pin versions and minimize the dependency count, review or disable untrusted post-install scripts, and use SRI for third-party CDN assets. The threat is transitive dependencies and typosquats, not just your direct ones.

Pro What is prototype pollution, how does it arise, and how do you prevent it?

An unsafe recursive merge or deep-clone copies an attacker-controlled __proto__ or constructor.prototype key onto Object.prototype, so every object inherits the injected property — a gadget that can become DoS, XSS, or RCE. Prevent it by rejecting __proto__/constructor keys, using Object.create(null) or Map for dynamic key stores, freezing prototypes with Object.freeze, and validating input shapes with a schema.

Pro Why is a CSP with script-src 'unsafe-inline' nearly worthless, and what is a strict CSP?

'unsafe-inline' re-permits exactly the inline scripts an injection creates, so the policy no longer blocks XSS. A strict CSP drops it and allowlists inline scripts by a per-response nonce- or a content hash-, then adds 'strict-dynamic' so trusted scripts can load their own dependencies without allowlisting every host. That is Google's recommended, XSS-resistant shape.

Pro Why use crypto.getRandomValues/randomUUID over Math.random(), and name a common Web Crypto misuse?

Math.random() is not cryptographically secure and is predictable, so tokens, IDs and keys built from it are guessable; crypto.getRandomValues and crypto.randomUUID() draw from a CSPRNG. Common misuses: rolling your own scheme instead of crypto.subtle, using AES in ECB mode or reusing an IV, mishandling key storage, and forgetting that subtle only exists in a secure (HTTPS) context.
