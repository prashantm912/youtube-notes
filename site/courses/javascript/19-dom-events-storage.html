Module: Browser — DOM, Events and Storage

This is where JavaScript stops being a language and starts being the web: the tree it reshapes, the events that flow through that tree, and the places it can keep data once the tab closes. Event delegation alone shows up in a large share of Indian front-end interviews.

Knowledge Library → Courses → JavaScript → Browser: DOM, Events, Storage

What this module covers

Four browser pillars that every front-end role assumes you know cold: reading and rewriting the DOM (traversal, node creation, classList, dataset, and the Selection/Range text model); the event system (the three propagation phases, delegation, and custom events); the CSSOM for touching styles from JavaScript (getComputedStyle, constructable stylesheets, matchMedia) plus the Popover API; and client-side storage from localStorage and cookies up to IndexedDB. These are the raw platform APIs that React, Vue and friends wrap — knowing them directly is what separates a framework user from an engineer.

Topics

DOM traversal/manipulation, classList/dataset, Selection/Range

The DOM is the browser's live, in-memory tree of objects built from your parsed HTML — every tag, text run and comment is a node. JavaScript reads and reshapes the page through it: you find nodes with querySelector/querySelectorAll and traversal properties (parentNode, children, nextElementSibling, closest()), then build and place nodes with createElement and the modern insertion methods append, prepend, before, after, replaceWith and remove (which superseded the clunkier appendChild/insertBefore). classList hands you a DOMTokenList with add/remove/toggle/replace/contains so you never do string surgery on className, and dataset maps every data-* attribute to a camel-cased read/write object.

Why it matters: DOM manipulation is where most front-end work — and most performance pitfalls — actually live. Layout thrashing from interleaved reads and writes, XSS holes opened by innerHTML, and O(n) re-renders all originate here, which is exactly why frameworks exist to manage it for you. Selection and Range round out the topic by modelling the user's highlighted text and arbitrary document fragments — the basis of rich-text editors, custom copy behavior and highlight tools. Interviewers lean on this cluster because it reveals whether you understand the platform your framework hides.

const card = document.querySelector("#card");
card.classList.toggle("is-active");        // no string juggling on className
card.dataset.userId = "42";                // sets data-user-id="42"

const badge = document.createElement("span");
badge.textContent = card.dataset.userId;   // textContent, not innerHTML (XSS-safe)
card.append(badge);                        // modern insertion, not appendChild

Events (bubbling/capture/delegation, custom events, common types)

Events are the signals the browser fires — click, input, keydown, submit, load and hundreds more — that you subscribe to with addEventListener(type, handler, options). A dispatched event travels in three phases: capture (from window down to the target), the target itself, then bubble (back up through the ancestors), which is why a click on a button also reaches its parent <div> and the document. The options bag — capture, once, passive, signal — tunes when the listener runs and lets an AbortController remove it cleanly.

Why it matters: event delegation is arguably the single most-asked DOM topic in Indian front-end interviews. It exploits bubbling — attach ONE listener to a container and use event.target with closest()/matches() to discover which descendant was acted on, giving you fewer listeners, automatic coverage of elements added later, and lower memory. Round it out with CustomEvent, which lets a component emit its own named event carrying a detail payload so producers and consumers stay decoupled. Knowing target vs currentTarget, stopPropagation vs preventDefault, and which events do not bubble is what separates a rehearsed answer from real understanding.

// One listener for the whole list, works for rows added later
document.querySelector("#todos").addEventListener("click", (e) => {
  const btn = e.target.closest("button.delete");
  if (!btn) return;                        // click landed elsewhere — ignore
  btn.closest("li").remove();
});

// Emit a custom event with a payload
el.dispatchEvent(new CustomEvent("cart:add", { detail: { id: 42 }, bubbles: true }));

CSSOM (getComputedStyle, constructable stylesheets, matchMedia), Popover API JS

The CSSOM is the DOM's counterpart for styles: it exposes stylesheets, rules and resolved values to JavaScript. getComputedStyle(el) returns the value of every property after the cascade has run — what the element actually renders as, not merely what its inline style attribute says — and is read-only. matchMedia(query) evaluates a media query in JavaScript and returns a MediaQueryList you can both read (.matches) and subscribe to (its change event): the programmatic twin of a CSS @media block, ideal for reacting to dark mode, reduced motion or a breakpoint crossing.

Why it matters: constructable stylesheets (new CSSStyleSheet() with replace/replaceSync, applied through adoptedStyleSheets) let you build a stylesheet once and share it across the document and many shadow roots with no duplicated <style> tags — a Web Components performance staple. The Popover API adds the popover attribute plus showPopover()/hidePopover()/togglePopover() for top-layer, light-dismiss overlays (menus, toasts, teaching UI) without hand-rolled z-index and focus code. Reading computed geometry, driving responsive behavior from JavaScript, and styling Shadow DOM efficiently are recurring senior-level concerns, and the Baseline-2025 Popover API is a fast-moving feature interviewers use to gauge whether you track the platform.

const px = getComputedStyle(box).width;    // resolved value e.g. "240px", read-only

const dark = matchMedia("(prefers-color-scheme: dark)");
dark.addEventListener("change", (e) => setTheme(e.matches ? "dark" : "light"));

const sheet = new CSSStyleSheet();
sheet.replaceSync(":host { color: rebeccapurple; }");
shadowRoot.adoptedStyleSheets = [sheet];   // shared, no duplicated <style>

localStorage/sessionStorage/cookies, IndexedDB

The browser offers a ladder of persistence. localStorage and sessionStorage are synchronous key/value stores of strings, scoped per origin: localStorage persists until explicitly cleared, while sessionStorage lives only for the tab/session. Cookies (document.cookie in JS, or Set-Cookie from the server) are small, are sent with every matching request, and carry the attributes that make authentication safe — HttpOnly (invisible to JavaScript), Secure, SameSite and an expiry. Because storage values are strings, objects go through JSON.stringify/JSON.parse on the way in and out.

Why it matters: for anything large or structured, IndexedDB is the real database — asynchronous, transactional, indexable, and able to store objects and blobs — the right tool for offline apps and caches where localStorage's roughly 5 MB, synchronous, string-only nature would block the main thread. Choosing the correct store is a security and performance decision interviewers probe: never keep session tokens in localStorage, where any XSS can read them; know that the storage event fires in other tabs of the same origin (the classic cross-tab sync trick); and be able to justify localStorage vs cookies vs IndexedDB for a given payload.

localStorage.setItem("prefs", JSON.stringify({ theme: "dark" }));
const prefs = JSON.parse(localStorage.getItem("prefs") ?? "{}");

// Fires in OTHER tabs of this origin — cheap cross-tab sync
addEventListener("storage", (e) => {
  if (e.key === "prefs") applyPrefs(JSON.parse(e.newValue));
});

Interview questions

Most-asked browser-platform questions in Indian front-end interviews — event delegation is the perennial favorite, so several questions circle it from different angles:

Beginner What is the DOM, and how does JavaScript interact with it?

The Document Object Model is the browser's live, in-memory tree of objects built from parsed HTML — each tag, text run and comment is a node. JavaScript reads it (querySelector, traversal properties) and mutates it (create/insert/remove nodes, change attributes and styles); the browser re-renders to reflect changes. It is an API over the page, not the HTML source itself.

Beginner innerHTML vs textContent — what is the difference and the security concern?

textContent sets/reads plain text and is safe. innerHTML parses its string as HTML, so injecting untrusted input opens an XSS hole (a <img onerror> or script-bearing markup can run). Use textContent for text; if you must set markup, sanitize it or build nodes with createElement. (innerText also triggers layout and respects CSS visibility, so it is slower and behaves differently again.)

Beginner How do you add, remove and toggle a class — classList vs className?

classList exposes add(), remove(), toggle(), replace() and contains() on a live DOMTokenList, so you change one class without touching the others. className is the whole class string — reassigning it clobbers every class and forces manual string editing. Prefer classList; toggle("x", condition) even takes a boolean to force the state.

Beginner What are data-* attributes and how do you read and write them from JavaScript?

Custom data-* attributes store app data on elements. In JS they surface through element.dataset as a DOMStringMap, with names converted to camelCase: data-user-id becomes dataset.userId. You read el.dataset.userId, write el.dataset.userId = "42", and delete to remove. Values are always strings.

Beginner localStorage vs sessionStorage — key differences?

Both are synchronous string key/value stores scoped to an origin, accessed via setItem/getItem/removeItem. localStorage persists indefinitely (until cleared) and is shared across all tabs of the origin; sessionStorage is per-tab and is wiped when that tab closes. Neither is sent to the server, and both cap around 5 MB.

Intermediate Explain the three phases of event propagation.

Capture: the event descends from window down to the target's parent. Target: it fires on the element itself. Bubble: it ascends back up through the ancestors. By default listeners run in the bubbling phase; pass { capture: true } (or the legacy true third argument) to listen on the way down. Bubbling is what makes event delegation possible.

Intermediate What is event delegation and why is it useful?

Instead of binding a handler to every child, you bind ONE handler to a common ancestor and, when an event bubbles up, inspect event.target (usually with closest(selector)) to find what was actually acted on. Benefits: far fewer listeners and less memory, no rebinding when elements are added or removed dynamically, and simpler setup. This is the most commonly asked DOM question in Indian front-end rounds.

Intermediate How would you implement event delegation for a list whose rows are added dynamically?

Attach the listener to the stable container, then guard on the target: list.addEventListener("click", e => { const row = e.target.closest("li"); if (!row || !list.contains(row)) return; /* handle row */ });. closest() walks up from the actual click target to the row even if an inner element (icon, span) was clicked, and the contains/null check ignores clicks outside a row. New rows work automatically because the listener lives on the parent.

Intermediate event.target vs event.currentTarget — why does the distinction matter in delegation?

event.target is the element where the event originated (the deepest clicked node); event.currentTarget is the element the handler is bound to (and equals this in a normal function handler). In delegation they differ: currentTarget is your container, while target is the specific child — which is exactly why you resolve the real subject with target.closest(...).

Intermediate stopPropagation, stopImmediatePropagation and preventDefault — what does each do?

preventDefault() cancels the browser's default action (following a link, submitting a form) but the event still propagates. stopPropagation() halts travel to further elements but lets other listeners on the current element run. stopImmediatePropagation() stops propagation and prevents any remaining listeners on the same element from firing. They are independent — stopping propagation does not prevent the default, and vice versa.

Intermediate How do you create, dispatch and pass data with a custom event?

Construct new CustomEvent("name", { detail: payload, bubbles: true }) and call el.dispatchEvent(event); listeners read the payload from event.detail. Set bubbles: true to let it be delegated, and composed: true if it must cross Shadow DOM boundaries. It is the decoupled way for components to signal one another without direct references.

Intermediate localStorage vs cookies — when would you use each?

Cookies are small (~4 KB), are sent automatically with every matching request, and support server-set security attributes (HttpOnly, Secure, SameSite, expiry) — use them for auth/session identifiers the server needs. localStorage is larger (~5 MB), stays on the client, and is never sent to the server — use it for UI preferences and non-sensitive client state. Critically, never store tokens in localStorage: it is readable by any XSS, whereas an HttpOnly cookie is not.

Pro getComputedStyle vs element.style, and how do you react to a media query in JavaScript?

element.style only reflects inline styles set on that element; it is empty for values coming from stylesheets, and it is writable. getComputedStyle(el) returns the resolved value of every property after the full cascade — what the element actually renders as — and is read-only. For media queries, matchMedia("(min-width: 768px)") returns a MediaQueryList: read .matches for the current state and add a change listener to respond when the match flips.

Pro Which events do not bubble, and how do you still delegate them?

focus, blur, mouseenter, mouseleave, and many resource/media events (load, scroll on some targets) do not bubble. To delegate focus, use the bubbling counterparts focusin/focusout; alternatively register the listener in the capture phase ({ capture: true }), which reaches the target on the way down regardless of bubbling. This limitation — and its fix — is a favorite senior probe after the basic delegation question.

Pro localStorage vs IndexedDB, and how do you keep state in sync across tabs?

localStorage is synchronous, string-only, ~5 MB, and blocks the main thread — fine for small preferences, wrong for large or structured data. IndexedDB is asynchronous, transactional, indexable, and stores objects and blobs, so it suits offline caches and datasets. For cross-tab sync, the storage event fires in every other same-origin tab whenever localStorage changes (it never fires in the tab that made the change), letting tabs react to each other; BroadcastChannel is the more general modern alternative for tab-to-tab messaging.
