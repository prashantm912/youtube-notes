Module: Language Basics

Types, coercion, declarations and template strings are the vocabulary every other JavaScript feature is written in — pin these down and the rest of the language stops feeling random.

Knowledge Library → Courses → JavaScript → Language Basics

What this module covers

Language Basics is the ground floor of JavaScript: how values are typed, how the engine silently converts between types, how var, let and const differ in scope and lifetime, and how template literals build strings. Nearly every interview opens here, and every later module — objects, functions, async — assumes you already have these exact.

Topics

Types and values (primitives, objects), typeof

Every JavaScript value is either one of seven primitive types — string, number, boolean, null, undefined, symbol and bigint — or an object (arrays, functions, dates and plain objects are all objects). Primitives are immutable and compared by value; objects are mutable and compared by reference. The typeof operator returns a string naming the type and is the quickest runtime type check.

Why it matters: JavaScript is dynamically typed (a variable can hold any type) and weakly typed (values coerce across types), which is the source of both its flexibility and its famous surprises. Two quirks you must know cold: typeof null returns "object" — a bug preserved for compatibility — and typeof a function returns "function" even though functions are objects.

typeof "hi"          // "string"
typeof 42            // "number"
typeof undefined     // "undefined"
typeof null          // "object"   historical bug, null is its own type
typeof {}            // "object"
typeof function(){}  // "function" functions are callable objects

Type coercion, == vs ===, truthiness

Coercion is JavaScript automatically converting a value from one type to another — to string when concatenating with +, to number in most arithmetic, to boolean in an if condition. === (strict) compares value and type with no conversion; == (loose) coerces both sides to a common type first, which is why 0 == "", 1 == true and null == undefined are all true.

Why it matters: in a boolean context every value is either truthy or falsy. The falsy list is short and fixed — false, 0, -0, 0n, "", null, undefined, NaN — and everything else is truthy, including "0", "false", [] and {}. Interviewers lean on these rules hard: use === by default, memorise the falsy set, and be ready to walk through a coercion trick line by line.

0 == ""            // true  — both coerce to 0
1 == true          // true  — true coerces to 1
null == undefined  // true  — special-cased pair
0 === ""           // false — different types, no coercion
Boolean("0")       // true  — non-empty string is truthy
Boolean(0)         // false — 0 is falsy

var/let/const, hoisting, TDZ, scope

var is function-scoped and hoisted — its declaration is lifted to the top of the function and initialised to undefined, so reading it before its line yields undefined, not an error. let and const are block-scoped ({...}); they are hoisted too, but stay uninitialised in a temporal dead zone (TDZ) from the top of the block until their declaration runs, so touching them early throws a ReferenceError. const forbids reassignment of the binding, though the object it points to can still mutate.

Why it matters: scope and hoisting decide what a name refers to and whether it exists yet — the mechanics behind closures, loop-variable bugs and half of all "why is this undefined?" questions. Modern code uses const by default, let when reassignment is needed, and avoids var; interviewers still probe var vs let precisely because the difference reveals whether you understand the execution model.

console.log(a);  // undefined — var hoisted, initialised undefined
var a = 1;

console.log(b);  // ReferenceError — b is in the TDZ
let b = 2;

const c = 3;
c = 4;           // TypeError — cannot reassign a const binding

Template literals, tagged templates

Template literals use backticks instead of quotes and add three things: interpolation with ${expression}, genuine multi-line strings, and the ability to embed any JavaScript expression without + concatenation. They have largely replaced string concatenation for building HTML, URLs and messages because they read far closer to the final text.

Why it matters: a tagged template puts a function name before the backticks; the function receives the array of literal string chunks as its first argument and the interpolated values as the rest, so it can post-process the result. That is the mechanism behind libraries like styled-components, graphql-tag, i18n formatters, and safe HTML/SQL escaping — and String.raw is a built-in tag that returns the un-escaped raw strings.

const name = "Asha", n = 3;
`Hi ${name}, you have ${n * 2} points`;   // interpolation + expression

function tag(strings, ...values) {         // a tagged template
  return strings[0] + values.map(String).join("|");
}
tag`a ${1} b ${2}`;                        // "a 1|2"

Interview questions

The JavaScript basics most asked in Indian interviews — from service-company screening rounds to product-company deep dives:

Beginner What are the primitive types in JavaScript?

Seven: string, number, boolean, null, undefined, symbol and bigint. Anything that is not a primitive is an object (arrays, functions, dates, plain objects). Primitives are immutable and compared by value; objects are mutable and compared by reference.

Beginner What does typeof null return, and is it correct?

It returns "object". This is a long-standing bug from JavaScript's first implementation, kept for backward compatibility. null is really its own primitive type, so the honest answer would be "null" — but the language can't change it without breaking the web.

Beginner Difference between == and ===?

=== (strict) compares value and type with no conversion. == (loose) converts both operands to a common type first, so 5 == "5" is true but 5 === "5" is false. Default to ===; the one common use for == is x == null, which is true for both null and undefined.

Beginner List the falsy values in JavaScript.

false, 0, -0, 0n, "" (empty string), null, undefined and NaN. Every other value is truthy — including "0", "false", [] and {}, which is what trips people up.

Beginner Difference between var, let and const?

var is function-scoped and hoisted to undefined. let and const are block-scoped and sit in the temporal dead zone until declared. const can't be reassigned, but if it holds an object you can still mutate that object. Modern rule: const by default, let when you must reassign, never var.

Intermediate What is hoisting, and how does it differ for var vs let/const?

Hoisting is the engine registering declarations before running code, so a name exists throughout its scope. var is hoisted and initialised to undefined (readable early as undefined); function declarations are hoisted with their full body. let, const and class are hoisted but left uninitialised — reading them before the declaration throws a ReferenceError.

Intermediate What is the Temporal Dead Zone?

The TDZ is the region from the start of a block until a let/const/class declaration is evaluated. The binding exists but is uninitialised, so any access — even typeof — throws ReferenceError. It exists so that const behaves predictably and to catch use-before-declaration bugs.

Intermediate Predict the output: console.log(x); var x = 5; — and the same with let.

With var it logs undefined (the declaration is hoisted, the assignment is not). With let it throws ReferenceError because x is in the TDZ. This one-liner is a favourite screen because it tests hoisting and the TDZ together.

Intermediate Difference between null and undefined?

undefined means a variable was declared but never assigned (or a missing property/argument) — the engine's default. null is an explicit "no value" you assign yourself. typeof undefined is "undefined", typeof null is "object". null == undefined is true, but null === undefined is false.

Intermediate What do template literals give you over normal quotes?

Backticks add string interpolation with ${expression}, genuine multi-line strings, and the ability to embed any expression without + concatenation. They make building HTML, URLs and messages far more readable, which is why they have largely displaced concatenation in modern code.

Pro const nums = [1, 2, 3]; nums.push(4) works — why, if const means constant?

const freezes the binding, not the value. You can't reassign nums to a different array, but the array it points to is a mutable object, so push, pop and property changes are all allowed. For a shallow-immutable object use Object.freeze; const alone only protects the variable-to-value link.

Pro What is a tagged template, and where is it used in practice?

Putting a function before a template — tag`...` — calls that function with the array of literal string chunks first and the interpolated values after, so it can transform or sanitise the output. Real uses: styled-components (CSS-in-JS), graphql-tag for queries, i18n formatting, and escaping HTML/SQL to block injection. String.raw is a built-in tag that returns the raw, un-escaped string.

Pro Explain why [] == ![] evaluates to true.

![] comes first: [] is truthy, so ![] is false. Now it's [] == false. Loose equality coerces the boolean to a number — false becomes 0 — giving [] == 0. To compare an object with a number, [] is converted to a primitive via toString, yielding "", which then becomes 0. So it reduces to 0 == 0, which is true. It is the classic demonstration of why == and object-to-primitive coercion are dangerous — and exactly why interviewers ask it.
