Module: Regular Expressions

A regular expression is a tiny pattern language embedded inside JavaScript — one line that can validate a phone number, pull every hashtag out of a caption, or rewrite a date format. This module orients you on the pattern syntax, the flags that change the rules, and the methods that run them, then points you at the best official material to master each.

What this module covers

Regular expressions in JavaScript are backed by the RegExp object and a pattern mini-language shared, roughly, across most programming languages. This module splits into three parts: the core pattern syntax — character classes, quantifiers and anchors that describe what to match; the flags and constructs — d/g/i/m/s/u/v/y, capturing and named groups, and lookaround that tune how matching behaves; and the methods on String and RegExp that actually run a pattern, including replacement templates and the newer RegExp.escape. Regex is a perennial interview topic in India because it shows up in every form-validation and log-parsing task.

Topics

Syntax, character classes, quantifiers, anchors

A pattern is built from literal characters plus metacharacters. Character classes describe a set of allowed characters: [abc] matches one of three letters, [a-z] a range, [^0-9] anything but a digit, and the shorthands \d, \w, \s (with their negations \D, \W, \S) cover digits, word characters and whitespace. Quantifiers say how many times the preceding atom repeats — ? (0 or 1), * (0 or more), + (1 or more), and the counted forms {n}, {n,}, {n,m} — and each is greedy by default until you append ? to make it lazy. Anchors match a position rather than a character: ^ and $ pin to the start and end of the input (or of each line under the m flag), and \b marks a word boundary.

Why it matters: this is the vocabulary behind every validation regex you will ever read or write. Being able to look at /^\d{6}$/ and immediately see "exactly six digits, whole string" — a PIN-code check — is the baseline literacy that screening rounds test before anything else.

/^\d{6}$/.test("560001");            // true  — anchored: whole string is 6 digits
"grey or gray".match(/gr[ae]y/g);    // [ "grey", "gray" ]  — character class
"colour color".match(/colou?r/g);    // [ "colour", "color" ] — optional u
/\bword\b/.test("a word here");       // true — \b word-boundary anchor

Flags (d/g/i/m/s/u/v/y), groups (named/capturing), lookaround

Flags are single letters after the closing slash that change matching behavior globally for the pattern: g (find all matches, not just the first), i (case-insensitive), m (multiline — ^/$ match line boundaries), s (dotAll — . also matches newlines), u (Unicode mode, enabling \p{…} property escapes and correct handling of astral characters), v (an upgraded Unicode mode with set notation inside classes), y (sticky — anchor the match at lastIndex), and d (record start/end indices of each match). Groups wrap a subpattern: (…) is a capturing group you can back-reference with \1 or read from the result array, (?<name>…) is a named capturing group readable via .groups.name, and (?:…) groups without capturing. Lookaround asserts context without consuming it: lookahead (?=…)/(?!…) and lookbehind (?<=…)/(?<!…).

Why it matters: the difference between a regex that "works on my machine" and one that behaves correctly on real data usually comes down to a flag (forgetting g, or needing u/v for emoji and non-Latin scripts). Named groups and lookaround are exactly the constructs interviewers use to separate people who copy regexes from people who can compose them — a password rule "at least one digit and one uppercase" is a lookahead question in disguise.

const iso = /(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/;
iso.exec("2026-07-19").groups.year;      // "2026"  — named capture

"1kg 5kg 20kg".match(/\d+(?=kg)/g);      // [ "1", "5", "20" ] — lookahead
/(?<=₹)\d+/.exec("₹499")[0];            // "499" — lookbehind
"hello".replace(/(l)\1/, "[$1$1]");      // "he[ll]o" — backreference \1

String/RegExp methods, replacement patterns, RegExp.escape

A pattern does nothing until a method runs it, and the methods split across two objects. On String: test-style checks aside, match returns matches (or capture groups without g), matchAll returns an iterator of every match with its groups, search gives the first index, split divides on a pattern, and replace/replaceAll substitute. On RegExp: test returns a boolean and exec returns the next match object (advancing lastIndex when the regex is global or sticky). The replacement string has its own mini-syntax — $& (whole match), $1 (group 1), $<name> (named group), $`/$' (text before/after) — or you can pass a function that receives the match and groups and returns the replacement.

Why it matters: choosing matchAll over a manual exec loop, or a replacer function over a static template, is the practical difference between clear and buggy text-processing code. The newest piece, RegExp.escape(str) (Baseline 2025), neutralizes any regex metacharacters in a string so you can safely build a pattern from user input — the correct fix for the classic bug where a search box containing . or ( silently breaks a dynamically built regex.

"a1b2c3".replace(/\d/g, "#");            // "a#b#c#"
[..."a1b2".matchAll(/([a-z])(\d)/g)];    // two match arrays, each with 2 groups
"2026-07-19".replace(/(\d+)-(\d+)-(\d+)/, "$3/$2/$1"); // "19/07/2026"

const needle = RegExp.escape("a.b*c");   // escapes metacharacters -> safe literal
new RegExp(needle).test("a.b*c");        // true — no accidental wildcards

Interview questions

Most-asked regular-expression questions in Indian interviews, from front-end screening rounds to product-company deep dives:

Beginner What are the two ways to create a regex in JavaScript, and when do you use each?

A literal /pattern/flags, compiled once when the script is parsed — use it when the pattern is fixed. Or the constructor new RegExp("pattern", "flags"), which builds the regex from a string at runtime — use it when the pattern is dynamic (built from a variable). The constructor form needs doubled backslashes ("\\d") because the string is parsed first.

Beginner What do ^, $ and \b match?

They are anchors — they match a position, not a character. ^ is the start of the input, $ the end (or the start/end of each line when the m flag is set), and \b is a word boundary between a word character and a non-word character. Anchoring with ^…$ is how you assert the whole string matches, not just a part of it.

Beginner Difference between str.match() and regexp.test()?

test returns a boolean — does the pattern match at all — and is the right tool for validation. match returns the matched data: without the g flag it returns the first match plus its capture groups; with g it returns an array of all full matches (and no groups). Use test for yes/no, match (or matchAll) to extract.

Beginner What is the difference between a greedy and a lazy quantifier?

Greedy quantifiers (*, +, {2,}) match as much as possible, then backtrack if needed. Appending ? makes them lazy (*?, +?), matching as little as possible. Classic example: against "<a><b>", /<.+>/ greedily matches the whole thing, while /<.+?>/ matches just "<a>".

Intermediate What does the g flag do, and why can regex.test() return alternating true/false when reused?

g makes the regex find all matches instead of stopping at the first. But a global (or sticky) regex is stateful: test and exec resume from its lastIndex property and advance it on each call, so calling test repeatedly on the same string walks through matches and eventually returns false, resetting lastIndex to 0. Fix: don't reuse a /g/ regex for boolean checks (drop the flag), or reset lastIndex yourself.

Intermediate Run through the regex flags d/g/i/m/s/u/v/y.

g global (all matches), i case-insensitive, m multiline (^/$ per line), s dotAll (. matches newline), u Unicode mode (enables \p{…}, correct astral handling), v an extended Unicode mode adding set notation and string properties inside character classes, y sticky (match only at lastIndex), d hasIndices (exposes match start/end offsets via .indices). You cannot combine u and v.

Intermediate What are named capturing groups and backreferences?

A named group (?<year>\d{4}) captures like an ordinary group but is readable by name from result.groups.year instead of a numeric index — far more maintainable. A backreference \1 (or \k<year> for named) matches the same text the group already captured, which is how you detect doubled words or matched quotes. Use (?:…) when you want grouping without a capture.

Intermediate Explain lookahead vs lookbehind, positive vs negative, with an example.

Lookaround asserts that something does or does not appear next to the current position without consuming characters. Lookahead checks what follows — (?=…) positive, (?!…) negative; lookbehind checks what precedes — (?<=…) positive, (?<!…) negative. A password "must contain a digit" is (?=.*\d) at the start; extracting the number in "₹499" without the symbol is /(?<=₹)\d+/.

Intermediate When do you need matchAll instead of match?

When you want every match together with each match's capture groups. match with g gives you all full matches but discards groups; match without g gives groups but only for the first match. matchAll(/…/g) returns an iterator of full match objects — each with its groups, index and named groups — which you usually spread into an array. It requires the g flag or it throws.

Pro What breaks without the u/v flag, and what do those flags add?

Without u, the regex engine works on UTF-16 code units, so an astral character like an emoji is two "characters" — . matches half of it and [😀] is a broken range. The u flag switches to code-point semantics and enables Unicode property escapes such as \p{L} (any letter) or \p{Script=Devanagari}. The newer v flag is a strict superset that also allows set operations inside classes — intersection && and subtraction -- — and properties of strings. Use u/v for any input with non-Latin text or emoji.

Pro What is catastrophic backtracking (ReDoS) and how do you avoid it?

When nested or overlapping quantifiers let the engine try an exponential number of ways to match a failing input — the classic /(a+)+$/ against a long run of as followed by a non-match — matching can hang the thread for seconds, a denial-of-service vector on user input. Avoid it by not nesting quantifiers over the same characters, making subpatterns mutually exclusive, anchoring, using atomic-style rewrites, and validating regexes that run on untrusted input (linters like the ESLint redos rule, or a safe engine). Keep genuinely complex parsing out of a single regex.

Pro You build a regex from a user's search box. What can go wrong and how do you fix it?

If the input contains regex metacharacters — ., *, (, [, \ — new RegExp(userInput) either matches the wrong things (. becomes "any character") or throws a SyntaxError on an unbalanced bracket, and a crafted input can trigger ReDoS. Escape the input first: RegExp.escape(userInput) (Baseline 2025) turns it into a literal-matching pattern, or use a well-known manual escape (str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&")) on older runtimes. Better still, if you only need a substring test, skip the regex and use String.prototype.includes.
