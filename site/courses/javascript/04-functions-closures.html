Module: Functions and Closures

Functions are just values in JavaScript — you pass them, return them, and nest them. Closures fall straight out of that, and "explain a closure" is the most-asked JavaScript question in Indian interviews. This module orients you on each idea and links the best material to master it — official docs first.

What this module covers

How functions are defined and behave: declarations vs expressions (and their hoisting difference), arrow functions and the IIFE pattern, the parameter toolkit (default values, rest parameters, the legacy arguments object), and then the payoff — closures, lexical scope, and recursion. Get closures solid here; the next module on this and binding builds directly on how functions capture their surroundings.

Topics

Declarations vs expressions, arrow functions, IIFE

A function declaration (function add() {}) is hoisted in full, so you can call it above its own line. A function expression assigns a function to a variable, so it only exists once execution reaches it. Arrow functions are a compact expression form with deliberate differences — no own this, no arguments, not constructable — which makes them ideal for short callbacks and wrong for object methods that need their own this.

Why it matters: choosing the right form is a daily decision and a standard screening question. The IIFE — a function expression invoked on the spot — was the classic way to carve out a private scope before modules and let/const; recognising it (and knowing what replaced it) still comes up in reviews of older code.

function add(a, b) { return a + b; }            // declaration — hoisted
const sub = function (a, b) { return a - b; };  // expression — not hoisted
const mul = (a, b) => a * b;                    // arrow — concise, lexical this
(function () { /* runs now, in its own scope */ })(); // IIFE

Default/rest params, arguments object

Default parameters (function f(x = 10)) fill in when an argument is missing or explicitly undefined — note that null, 0, and "" do not trigger the default, since only undefined counts. Defaults evaluate at call time, left to right, so a later default can reference an earlier parameter. Rest parameters (...args) gather any extra arguments into a genuine array and must be the last parameter.

Why it matters: rest parameters are the modern, array-method-friendly replacement for the old arguments object — an array-like value available only in non-arrow functions, with a length but no map or filter. Interviewers love asking you to contrast the two, and to note that arrow functions have no arguments of their own at all.

function total(rate = 100, ...items) {
  // rate defaults only when the argument is undefined;
  // items is a REAL array, so reduce works directly
  return items.reduce((sum, n) => sum + n, 0) * rate;
}
total();               // 0   (no items)
total(100, 1, 2, 3);   // 600

Closures, lexical scope, recursion

A closure is a function bundled together with references to the variables of the scope where it was defined — so it keeps reading and updating those variables even after the outer function has returned. This works because JavaScript uses lexical scope: what a function can see is fixed by where it is written in the source, not by where it is called from. In practice every JavaScript function is a closure over its birthplace.

Why it matters: closures are the mechanism behind data privacy, function factories, memoization, and framework state (React's useState included) — and "explain closures" plus the loop-with-var trap are the two most-asked JavaScript interview questions in India. Recursion (a function calling itself down to a base case) rounds out the module: elegant for tree-shaped data, but each call adds a stack frame, so very deep recursion overflows the stack.

function makeCounter() {
  let count = 0;          // private state, kept alive by the closure
  return () => ++count;   // inner arrow closes over count
}
const next = makeCounter();
next(); // 1
next(); // 2  — count survives between calls, invisible to callers

Interview questions

Most-asked functions-and-closures questions in Indian interviews — closures dominate, from screening rounds to product-company deep dives:

Beginner Function declaration vs function expression?

A declaration (function add() {}) is hoisted whole, so it can be called above its own line. An expression assigns a function to a variable (const add = function () {} or an arrow), so only the variable is hoisted — calling it before that line throws (undefined with var, TDZ with let/const). Use declarations for named top-level helpers, expressions when passing a function as a value.

Beginner How does an arrow function differ from a regular function?

An arrow has no own this (it inherits this lexically), no arguments object, no prototype, and cannot be called with new. It is always anonymous. Great for short callbacks; wrong for object methods or constructors that need their own this.

Beginner What is an IIFE and what was it used for?

An Immediately Invoked Function Expression runs the instant it is defined: (function () { ... })(). Before ES modules and let/const, it was the standard way to create a private scope and avoid polluting the global namespace. Modules and block scoping have largely replaced it, though the async form (async () => { ... })() is still handy.

Beginner What are default parameters, and which value triggers the default?

function f(x = 10) uses 10 when the argument is missing or explicitly undefined. Other falsy values — null, 0, "", false — do NOT trigger it. Defaults evaluate at call time, left to right, so a later default may reference an earlier parameter.

Beginner Rest parameters vs the arguments object?

...rest gathers extra arguments into a real Array (so map/filter/reduce work directly) and must be the last parameter. arguments is an array-like object with a length but no array methods, available only in non-arrow functions. Modern code prefers rest parameters; arrows have no arguments at all.

Intermediate What is a closure? (the classic)

A function bundled with references to the variables of the scope where it was defined, so it keeps accessing them even after that outer function returns. In JavaScript every function is a closure. The giveaway example is a factory that returns an inner function still reading the outer function's local variable — the single most-asked JavaScript interview question.

Intermediate What is lexical scope, and how is it related to closures?

Lexical scope means a function's visible variables are fixed by where it is written in the source (its nesting), not by where it is called. An inner function can reach variables of every enclosing function — the scope chain. Closures work precisely because a function captures its birthplace, not its call site.

Intermediate The loop-with-var trap: why do the callbacks all print the last value, and how do you fix it?

With var, every iteration shares one function-scoped i, so callbacks created in the loop (e.g. inside setTimeout) all read the same final value. Fix it with let, which creates a fresh binding per iteration, or by wrapping the body in an IIFE that captures the current value as a parameter. A favourite Indian-interview trap.

Intermediate Give a real-world use of closures.

Data privacy (a counter or module exposing methods while hiding state), function factories and currying, memoization caches, and once-style guards. React hooks like useState also rely on closures. The common thread: remembering state between calls without a global or a class.

Intermediate Why give a name to a function expression (a named function expression)?

The name is visible only inside the function's own body, letting it call itself (recursion) safely even if the outer variable is reassigned, and it produces clearer stack traces while debugging. Example: const fact = function f(n) { return n <= 1 ? 1 : n * f(n - 1); };

Intermediate Does an arrow function have its own arguments? How do you collect all arguments in one?

No — an arrow uses the arguments of its enclosing regular function (or none at the top level). To collect every argument inside an arrow, use a rest parameter: const f = (...args) => args;

Pro Implement a private counter or a once() using closures.

Return an inner function that closes over private state. once(fn) keeps a captured called flag and a result: the first call runs fn and stores its result, every later call returns the cached result. A private counter is the same shape — makeCounter() holds let count = 0 and returns () => ++count. That captured, caller-invisible state is exactly what a class field or module would give you, built from a plain function.

Pro Can closures cause memory leaks?

Yes. A closure keeps its captured variables alive as long as the closure itself is reachable, so a long-lived reference (a global, a cache, an un-removed event listener) can pin large objects in memory. The classic browser leak is a detached DOM node captured by a handler that was never removed. Fixes: remove listeners, null out references, and capture only what you actually need.

Pro What is recursion, its main risk, and how do closures relate?

A function calling itself with a smaller input until a base case. Each call adds a frame to the call stack, so unbounded or very deep recursion throws a stack overflow (engines generally do not optimise tail calls in practice, so rewrite as a loop or use an explicit stack for deep data). Closures often power recursion through named function expressions, and are used to build helpers like a memoized recursive Fibonacci that caches results in a captured variable.
