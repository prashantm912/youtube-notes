Module: this and Binding

The value of this is not written into a function — it is decided the moment the function is called. Learn the small set of rules that decide it, the methods that override it, and why arrow functions opt out entirely.

Knowledge Library → Courses → JavaScript → this and Binding

What this module covers

Almost every confusing this bug comes from one idea: this is bound at call time, by how a function is invoked, not where it was defined. This module orients you on the four binding rules, the call/apply/bind trio that sets this explicitly, and the arrow function's lexical this plus the strict-mode and globalThis details that trip people up. It is one of the most heavily tested topics in Indian frontend and full-stack interviews.

Topics

this binding rules (default / implicit / explicit / new)

Four rules, checked in order of precedence, decide this for a normal function. Default: a plain call (fn()) binds this to undefined in strict mode or the global object in sloppy mode. Implicit: a method call (obj.fn()) binds this to the object left of the dot. Explicit: call/apply/bind set this to whatever you pass. new: constructing with new binds this to the brand-new object.

Why it matters: the same function returns a different this depending only on the call site, so a method pulled off its object and called bare "loses" its this. Reciting these four rules — and their precedence, new > explicit > implicit > default — is a standard interview screen.

function whoAmI() { return this; }
const user = { name: "Asha", whoAmI };

whoAmI();          // default:  undefined (strict) / globalThis (sloppy)
user.whoAmI();     // implicit: the user object
whoAmI.call(user); // explicit: the user object
new whoAmI();      // new:      a fresh empty object

call / apply / bind

These three methods on every function set this explicitly. call(thisArg, a, b) invokes immediately with arguments listed individually; apply(thisArg, [a, b]) is identical but takes the arguments as one array (handy when they are already in an array). bind(thisArg, ...preset) is different: it does not call the function — it returns a new function whose this is permanently locked, optionally with some leading arguments pre-filled (partial application).

Why it matters: bind is the classic fix for a callback that would otherwise lose its this, and "difference between call, apply and bind" plus "write a polyfill for bind" are among the most-repeated JavaScript interview questions in India. Once bound, this cannot be re-pointed by a later call.

function greet(greeting, mark) {
  return greeting + ", " + this.name + mark;
}
const asha = { name: "Asha" };

greet.call(asha, "Hi", "!");    // "Hi, Asha!"   — args individually
greet.apply(asha, ["Hi", "!"]); // "Hi, Asha!"   — args as an array
const bound = greet.bind(asha); // new fn, this locked to asha
bound("Hello", ".");            // "Hello, Asha."

Arrow this, globalThis, strict mode effects

Arrow functions have no this of their own. They capture this lexically from the surrounding scope at definition time, and none of the binding rules — not implicit, not call/apply/bind, not new — can change it. That is exactly why an arrow is the right choice for a callback inside a method (it keeps the outer this) and the wrong choice for an object method itself (there is no outer object to capture).

Two related facts round out the topic. globalThis is the standard, cross-environment way to reach the global object (replacing the old window/self/global patchwork). And strict mode changes the default binding: an unspecified this becomes undefined instead of the global object, and a primitive passed to call is no longer boxed into an object — details interviewers use to separate rote answers from real understanding.

const timer = {
  label: "tick",
  start() {
    // arrow keeps this = timer; a normal function here would not
    setTimeout(() => console.log(this.label), 200);
  },
};
timer.start(); // logs "tick"

Interview questions

Most-asked this and binding questions in Indian interviews — from screening rounds to product-company frontend deep dives, where "call vs apply vs bind" and a bind polyfill are near-guaranteed:

Beginner What does the this keyword refer to in JavaScript?

It refers to the execution context of a function — decided at call time by how the function is invoked, not where it was written. The same function can see a different this from one call site to another.

Beginner What are the rules that decide this?

Four for normal functions: default (plain call → undefined/global), implicit (obj.fn() → the object), explicit (call/apply/bind → what you pass), and new (→ the new instance). Arrow functions are the exception: they take this lexically and ignore all four.

Beginner In obj.method(), what is this?

The object to the left of the dot — obj — via implicit binding. Only the final call expression matters: in a.b.c.fn(), this is a.b.c.

Beginner What is this in a plain function call at the top level?

Default binding: in sloppy mode it is the global object (globalThis/window); in strict mode it is undefined. Reading a property off it in strict mode throws, which is usually what surfaces the bug.

Beginner Difference between call and apply?

Both invoke the function immediately with a chosen this. call takes the arguments listed individually; apply takes them as a single array (or array-like). Mnemonic: apply → array.

Beginner What does bind do, and how is it different from call?

bind does not call the function — it returns a new function with this permanently fixed (and optional leading arguments pre-set). call invokes right away. Use bind when you need to hand a correctly-bound function to something that will call it later.

Intermediate Why does const f = obj.method; f() lose its this?

Because this is set by the call site, not stored in the function. Detaching the method removes the obj. receiver, so f() is a plain call and falls to default binding (undefined/global). Fix it with obj.method.bind(obj) or by wrapping in an arrow.

Intermediate How do arrow functions handle this?

They have no own this; they capture it lexically from the enclosing scope when defined. call/apply/bind and new cannot change it. Great for callbacks inside a method (keeps the outer this), wrong as object methods (no object to capture — this is the outer scope).

Intermediate Why does this go wrong in a setTimeout callback, and how do you fix it?

A normal-function callback is invoked plainly by the timer, so it gets default binding (undefined/global), not the object you expected. Fixes: pass an arrow (captures the surrounding this), or pass fn.bind(this).

Intermediate A class/React method loses this when used as an event handler — how do you keep it?

Bind it in the constructor (this.handle = this.handle.bind(this)), or define it as a class field arrow (handle = () => {…}) so it captures the instance, or wrap the call in an arrow at the point of use. The class-field arrow is the common modern choice.

Intermediate Can you write a polyfill for bind?

Add Function.prototype.myBind that captures the target function and context, returns a new function which merges the preset and later arguments and invokes the original via apply: roughly const fn = this; return function(...later){ return fn.apply(ctx, [...preset, ...later]); };. A full version also handles being called with new. This is a frequent Indian-interview coding task.

Intermediate What is the precedence when the binding rules compete, e.g. a method called with new?

Order is new > explicit (bind/call/apply) > implicit (obj.fn()) > default. So constructing a bound function ignores the bound this and uses the new object (though bound preset arguments still apply); and explicit binding beats the object-on-the-dot.

Pro What happens with f.call(2) in sloppy vs strict mode?

In sloppy mode a primitive this is boxed: this becomes a Number object wrapping 2. In strict mode there is no substitution or boxing — this stays the primitive 2, and passing null/undefined leaves them as-is instead of swapping in the global object.

Pro globalThis vs window vs top-level this in a module?

globalThis is the standardized reference to the global object across browsers, Node and workers; window exists only in browsers. At the top level of an ES module, this is undefined (not globalThis) — a deliberate difference from a classic script, where top-level this is the global object.

Pro Once bound with bind, can this be overridden by a later call or another bind?

No. A bound function's this is fixed: a subsequent boundFn.call(other) or boundFn.bind(other) cannot re-point it — the first bind wins. The only exception is new boundFn(), which ignores the bound this and uses the freshly created object. Arrow functions are even stricter: their lexical this can never be changed by any mechanism.
