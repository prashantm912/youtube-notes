Module: Prototypes and Classes

Under every class keyword sits an object quietly delegating to another object. Learn the prototype chain first and the class syntax stops being magic — which is exactly the leap Indian interviewers keep probing for.

Knowledge Library → Courses → JavaScript → Prototypes and Classes

What this module covers

JavaScript's object model is prototypal: objects delegate to other objects through a hidden link, and class is mostly a nicer spelling of that machinery. This module orients you on the prototype chain (and the perennially confused __proto__ vs prototype), the full class syntax — constructors, fields, private # members, statics, static blocks and accessors — and inheritance via extends, super and mixins. Get the prototype mental model right and everything above it becomes predictable.

Topics

Prototype chain, __proto__ vs prototype

Every JavaScript object carries a hidden internal link, [[Prototype]], to another object. When you read a property the engine checks the object itself, then its prototype, then that object's prototype, and so on until it finds the property or hits null — this is the prototype chain, and it is how methods are shared without copying them onto every instance. The two similarly-named things trip everyone up: prototype is a property that lives on constructor functions and holds the object that instances built with new will delegate to, while __proto__ is an accessor that exposes an object's own [[Prototype]] (its modern, non-deprecated equivalents are Object.getPrototypeOf / Object.setPrototypeOf).

Why it matters: this is the engine beneath classes, inheritance, and a large share of tricky interview questions. Once you see property lookup as delegation, you can reason about method sharing, memory cost, why hasOwnProperty exists, and security pitfalls like prototype pollution — instead of memorising class rules by rote.

function Animal(name) { this.name = name; }
Animal.prototype.speak = function () { return this.name + " makes a sound"; };

const dog = new Animal("Rex");
dog.speak();                                   // found on Animal.prototype
Object.getPrototypeOf(dog) === Animal.prototype; // true
dog.__proto__ === Animal.prototype;            // same thing, legacy accessor
dog.hasOwnProperty("name");                    // true  (own)
dog.hasOwnProperty("speak");                   // false (inherited)

Class syntax (constructor, fields, private #, static, static blocks, accessors)

A class is largely syntactic sugar over a constructor function plus its prototype, but it layers on real semantics you should be able to name: methods land on the prototype (shared and non-enumerable), the whole body runs in strict mode, the constructor must be called with new, and class declarations are not usefully hoisted (they sit in a temporal dead zone). Inside the body you get a constructor, instance fields (declared directly, initialised before the constructor body), get/set accessors, static members that live on the class itself, and static { } initialization blocks for setup that needs statements, try/catch, or access to private state.

Why it matters: private # fields are the one part that is not just sugar — they are genuine, language-enforced encapsulation (reading a # name that isn't there throws a TypeError, and they are invisible to Object.keys, JSON, and the prototype chain). "Is class just syntactic sugar?" is a classic screening question; the sharp answer is "mostly — except private fields, which the old prototype pattern genuinely could not express."

class BankAccount {
  static currency = "INR";        // static field on the class
  #balance = 0;                   // truly private instance field

  constructor(owner) { this.owner = owner; }

  deposit(amount) { this.#balance += amount; return this; }
  get balance() { return this.#balance; }   // accessor, no () at call site

  static { console.log("BankAccount class initialised"); } // static block
}

const a = new BankAccount("Asha").deposit(500);
a.balance;                 // 500  (via getter)
a.hasOwnProperty("deposit"); // false — deposit lives on the prototype

Inheritance, super, mixins

extends wires up the prototype chain between two classes on both sides at once: instances delegate to the parent's prototype for methods, and the child class object itself delegates to the parent class for inherited static members. In a derived constructor, super(...) invokes the parent constructor — and because the parent is what actually produces this, you must call super() before touching this or you get a ReferenceError. super.method() reaches the parent's version of an overridden method; it resolves through the method's [[HomeObject]], which is why it only works inside methods defined in the class (not in a plain function you assigned later).

Why it matters: JavaScript has single inheritance, so shared behaviour across unrelated classes is done with mixins — either copying methods onto a prototype with Object.assign, or a subclass-factory function (Base) => class extends Base { … }. Knowing when to compose with mixins instead of building a deep extends tower is a design-maturity signal senior interviews look for, and super() ordering plus [[HomeObject]] separate memorised answers from understood ones.

class Animal {
  constructor(name) { this.name = name; }
  speak() { return this.name + " makes a sound"; }
}
class Dog extends Animal {
  constructor(name) { super(name); }            // must run before using this
  speak() { return super.speak() + " (a bark)"; } // parent method via super
}
new Dog("Rex").speak();   // "Rex makes a sound (a bark)"

// Mixin: share behaviour without a single-parent chain
const Serializable = (Base) => class extends Base {
  toJSONString() { return JSON.stringify(this); }
};
class Config extends Serializable(Object) {}

Interview questions

Prototypal inheritance is one of the most-asked JavaScript topics in Indian interviews — from frontend screening rounds to product-company deep dives. These are the questions this module powers, easy first, hardest last:

Beginner What is a prototype, and what is the prototype chain?

Every object has a hidden [[Prototype]] link to another object. When you access a property, the engine checks the object, then its prototype, then that object's prototype, and so on until it finds the property or reaches null. That series of links is the prototype chain, and it is how objects share methods without copying them.

Beginner Difference between __proto__ and prototype?

prototype is a property on constructor functions — the object that instances created with new will delegate to. __proto__ is an accessor on any object that exposes its actual [[Prototype]]. So dog.__proto__ === Animal.prototype. Prefer Object.getPrototypeOf/setPrototypeOf; __proto__ is legacy/deprecated.

Beginner Is a JavaScript class a new object model, or syntactic sugar?

Mostly sugar over constructor functions and prototypes — methods still go on the prototype. But it adds real semantics: strict-mode body, must be called with new, non-enumerable methods, no useful hoisting (TDZ), and # private fields, which the old pattern genuinely could not express. So "just sugar" is the wrong answer.

Beginner How do you create inheritance between two classes?

With class Child extends Parent { … }. extends links the prototypes so Child instances inherit Parent's methods, and inside a derived constructor you call super(args) to run the parent constructor. Static members are inherited too, because the child class object itself delegates to the parent class.

Beginner Where do class methods live versus instance fields?

Methods defined in the class body live once on the prototype and are shared by all instances. Instance fields (declared with field syntax or assigned via this.x = in the constructor) are own properties, a fresh copy per instance. That is why obj.hasOwnProperty("method") is false but hasOwnProperty("field") is true.

Intermediate Walk through what the new operator does, step by step.

Four steps: (1) create a fresh empty object; (2) set its [[Prototype]] to the constructor's prototype; (3) call the constructor with this bound to the new object; (4) return that object — unless the constructor explicitly returns a different object, in which case that wins. Understanding this demystifies both function constructors and classes.

Intermediate hasOwnProperty vs the in operator — why does the difference matter?

in returns true for inherited properties too; hasOwnProperty (or the safer Object.hasOwn(obj, key)) only for own ones. It matters when iterating or checking data objects, because you don't want to accidentally match inherited or polluted prototype keys. Object.hasOwn also works when the object has no prototype or overrides hasOwnProperty.

Intermediate How do Object.create, constructor functions, and classes each set up the prototype?

Object.create(proto) makes a new object whose [[Prototype]] is exactly proto — direct delegation, no constructor. Constructor functions set the instance's prototype to Fn.prototype via new. Classes do the same as constructor functions under the hood, plus link the class objects for static inheritance. All three end at the same prototype-chain mechanism.

Intermediate Why must super() be called before using this in a derived constructor?

In a derived class the child does not create this itself — the parent constructor does, via super(). Until super() runs there is no this to use, so any access before it throws a ReferenceError. A base (non-derived) class has no such rule because it creates its own this.

Intermediate What are # private fields, and how do they differ from an _underscore convention or closures?

#name fields are enforced by the language: unreachable outside the class body, absent from Object.keys/JSON/the prototype chain, and reading a missing one throws a TypeError (usable as a brand check with #x in obj). The _ underscore is just a naming hint anyone can ignore; closures give privacy but at the cost of per-instance function allocation and no shared prototype methods.

Intermediate Are static members inherited? And what is a static initialization block for?

Yes — static members live on the class object, and a subclass's class object delegates to its parent's, so Child.staticMethod() resolves to the parent's. A static { } block runs once when the class is defined; it's for setup that needs statements, loops, try/catch, or access to private static state that a single field initializer can't express.

Intermediate Difference between classical (class-based) inheritance and prototypal inheritance?

Classical inheritance copies structure from classes (blueprints) into instances. Prototypal inheritance links live objects that delegate to other live objects — no copying; a lookup that misses just walks up to the prototype. JavaScript is fundamentally delegation-based, and class is a familiar syntax layered on top of it, not a switch to a copy model.

Pro How does super.method() know which parent to call — explain [[HomeObject]]?

Each concise method remembers the object it was defined in as its internal [[HomeObject]]. super.method() looks up method on HomeObject's prototype, not on this's prototype — so it works even when the method is borrowed onto another object. Because a plain function assigned later has no [[HomeObject]], super is unavailable there; that's a common gotcha.

Pro What are mixins, how do you implement them, and why prefer them over deep inheritance?

A mixin adds behaviour to a class without being its single parent. Two idioms: copy methods with Object.assign(Class.prototype, mixin), or a subclass factory const M = Base => class extends Base { … } that you compose (class C extends M(N(Base)) {}). Since JS allows only one extends parent, mixins model cross-cutting capabilities; favouring them over a deep inheritance tower keeps coupling low and behaviour composable.

Pro What is prototype pollution, and why is it a security concern?

If untrusted input is merged into objects without guarding keys like __proto__, constructor, or prototype, an attacker can write onto Object.prototype — and because every object delegates there, the injected property silently appears on all objects, enabling denial of service, logic bypasses, or even RCE in some sinks. Defences: reject those keys, use Object.create(null) or Map for dictionaries, and Object.freeze(Object.prototype) where feasible.

Pro Why can adding a method to a constructor's prototype be more memory-efficient than defining it in the constructor?

Assigning this.method = function(){…} in the constructor creates a new function object for every instance. Putting it on Fn.prototype (or using a class method, which does exactly that) stores one shared function that all instances delegate to — one allocation instead of N. The trade-off is that per-instance closures capture private state, which shared prototype methods cannot.
