Module: Symbols, Iterators and Generators

Three quiet protocols power the visible language: Symbols name behavior without name clashes, the iteration protocols make for...of and spread work on anything, and generators produce values on demand — now with native iterator helpers on top.

Knowledge Library → Courses → JavaScript → Symbols, Iterators and Generators

What this module covers

This module is about the machinery underneath the syntax you already use. Symbols give unique, collision-proof keys and — through the well-known symbols — let objects plug into built-in operations like iteration and instanceof. The iterable and iterator protocols define exactly what makes a value work in for...of, spread and destructuring; generators are the ergonomic way to author those iterators; and the ES2025 iterator helpers add lazy map/filter/take pipelines that never materialize an intermediate array.

Topics

Symbol, well-known symbols, registry

A Symbol is a primitive whose every instance is unique — even two symbols with the same description are unequal. Used as an object key it never collides with string keys and stays out of for...in and Object.keys, which makes it the tool for attaching metadata or library-private state to objects you do not own. The well-known symbols (Symbol.iterator, Symbol.asyncIterator, Symbol.hasInstance, Symbol.toPrimitive, Symbol.toStringTag) are the extension points the language reads to customize built-in behavior.

Why it matters: Symbols are how JavaScript exposes its own protocols to your objects, so understanding them unlocks the rest of this module. The global registry — Symbol.for(key) / Symbol.keyFor(sym) — is a separate, shared table for symbols that must be the same across realms (iframes, workers), and a favourite "is this the same symbol?" interview trap.

const id = Symbol("id");           // unique, even vs another Symbol("id")
const user = { name: "Asha", [id]: 42 };
Object.keys(user);                 // ["name"] — the symbol key is hidden

const shared = Symbol.for("app.id");    // created in / read from the registry
Symbol.for("app.id") === shared;        // true — same key, same symbol

Iteration protocols (iterable/iterator)

Two contracts sit under every loop. An object is iterable if it has a [Symbol.iterator]() method returning an iterator; an iterator is any object with a next() method that returns { value, done } on each call. for...of, spread ([...x]), array/object destructuring, Array.from, Promise.all and new Map(entries) all speak this protocol — which is why implementing it once makes a custom object work everywhere.

Why it matters: this is the single most leveraged abstraction in the language. Knowing that an iterator is "stateful and one-shot" explains why a spread re-runs from the start (it calls Symbol.iterator afresh) while a half-consumed iterator resumes where it stopped — the mental model behind a lot of "why is my array empty?" bugs.

const range = {
  from: 1, to: 3,
  [Symbol.iterator]() {
    let n = this.from;
    return {
      next: () => n <= this.to
        ? { value: n++, done: false }
        : { value: undefined, done: true },
    };
  },
};
[...range];   // [1, 2, 3]

Generators (yield/delegation), async generators

A generator function (function*) returns a generator object that is both an iterator and iterable, so you rarely hand-write the next() boilerplate above again. Execution pauses at each yield and resumes on the next next() call, carrying local state across the gap — ideal for lazy sequences, state machines and tree walks. yield* delegates to another iterable, and communication is two-way: gen.next(v) feeds a value back into the paused yield, while gen.throw()/gen.return() inject an error or an early finish.

Why it matters: async generators (async function*) extend the idea to data that arrives over time — you await inside the generator and consume it with for await...of, the clean pattern for paginated APIs and streams. Interviewers use generators to test whether you understand lazy, pull-based evaluation versus eager arrays.

function* ids() {
  yield 1;
  yield* [2, 3];          // delegate to another iterable
}
[...ids()];               // [1, 2, 3]

async function* pages(url) {
  let next = url;
  while (next) {
    const res = await fetch(next);   // await inside the generator
    const page = await res.json();
    yield page.items;
    next = page.nextUrl;
  }
}

Iterator helpers (map/filter/take/drop/reduce/toArray)

Iterator helpers (ES2025, Baseline since March 2025) add array-style methods directly to iterators: map, filter, take, drop, flatMap, reduce, toArray, plus some/every/find/forEach and the static Iterator.from(). The crucial difference from Array.prototype methods is that they are lazy — each element flows through the whole chain one at a time, so nothing intermediate is allocated and infinite sequences become safe to process.

Why it matters: chaining .map().filter().take() on a generator processes only as many elements as take actually needs, which is impossible with eager array methods that would try to build the full array first. This is the modern answer to "how do you handle a huge or infinite data source efficiently?" — expect it in senior front-end and full-stack rounds now that it ships in every engine.

function* naturals() { let n = 1; while (true) yield n++; }

naturals()
  .map((n) => n * n)
  .filter((n) => n % 2 === 1)
  .take(3)
  .toArray();            // [1, 9, 25] — lazy; the infinite array is never built

Interview questions

Symbols and iteration come up constantly in Indian front-end and full-stack rounds — from "make this object work in for...of" screens to lazy-evaluation deep dives:

Beginner What is a Symbol and why was it added to JavaScript?

A Symbol is a primitive whose every instance is unique. Its main uses: collision-proof object keys (a library can add a Symbol key without clashing with your string keys, and it stays out of for...in/Object.keys), and the well-known symbols that let objects hook into built-in language operations like iteration.

Beginner Are two Symbols with the same description equal?

No. Symbol("id") === Symbol("id") is false — the description is only a debug label, not an identity. The one exception is the global registry: Symbol.for("id") === Symbol.for("id") is true because both look the key up in the same shared table.

Beginner What makes an object iterable, and what does for...of need?

An object is iterable if it has a [Symbol.iterator]() method returning an iterator. for...of calls that method once, then repeatedly calls the iterator's next() until it returns { done: true }. Arrays, strings, Maps and Sets are iterable out of the box; plain objects are not.

Beginner What does an iterator's next() return?

An object { value, done }. While iterating, done is false and value is the current element; when exhausted, done becomes true (and value is usually undefined, or the generator's return value).

Beginner What is a generator function and how do you write one?

A function declared with function* that can pause and resume. Calling it does not run the body — it returns a generator object. Each next() runs until the next yield, returns that yielded value, and suspends. It is a much shorter way to author an iterator than writing next() by hand.

Intermediate Symbol() vs Symbol.for() — what is the global symbol registry?

Symbol() always makes a brand-new unique symbol. Symbol.for(key) looks the key up in a runtime-wide registry and returns the existing symbol or creates one — so the same key yields the same symbol everywhere, including across iframes and workers. Symbol.keyFor(sym) reverses it. Registry symbols are never garbage-collected, so avoid them for per-object private state.

Intermediate What are well-known symbols? Name a few and what they customize.

Static symbols on Symbol that the engine reads to customize built-ins: Symbol.iterator (for...of/spread), Symbol.asyncIterator (for await...of), Symbol.hasInstance (instanceof), Symbol.toPrimitive (type coercion), and Symbol.toStringTag (the tag in Object.prototype.toString). They are the public extension points of the language's own protocols.

Intermediate Difference between an iterable and an iterator — can one object be both?

An iterable has [Symbol.iterator](); an iterator has next(). Yes, an object can be both — a generator object is the classic case: it has next() and its [Symbol.iterator]() returns itself. That self-return is why you can spread a generator or drop it straight into for...of.

Intermediate What does yield* do?

yield* delegates to another iterable/generator, yielding all of its values in place as if inlined, and evaluates to that inner generator's return value. It is how you compose generators — e.g. a tree traversal that yield*s over each child's traversal.

Intermediate What are async generators and when do you reach for them?

async function* can await inside and yield values that arrive over time; you consume it with for await...of. Each next() returns a Promise. It is the natural fit for paginated REST endpoints, streamed responses, or any source where the next value is not ready synchronously.

Intermediate How are iterator helpers different from Array methods like map/filter?

Array methods are eager: arr.map().filter() builds a full intermediate array at each step. Iterator helpers (Iterator.prototype.map/filter/take/...) are lazy: one element flows through the entire chain at a time, allocating no intermediate collections. That makes them memory-efficient on large data and usable on infinite generators, where array methods would never finish.

Pro How do generators and take() let you work with an infinite sequence?

A generator like function* naturals(){ let n=1; while(true) yield n++; } produces values only on demand, so it never overflows. Wrapping it with iterator helpers — naturals().map(...).filter(...).take(3).toArray() — pulls exactly as many elements as take(3) needs and then stops, because laziness propagates back up the chain. The infinite array is never materialized.

Pro Explain two-way communication in generators: next(value), throw() and return().

gen.next(v) resumes the generator and makes the paused yield expression evaluate to v — so a generator can receive input, not just produce output (the basis of coroutine patterns). gen.throw(err) raises err at the paused yield (catchable inside), and gen.return(v) forces an early, clean finish that still runs finally blocks for cleanup.

Pro Are Symbol keys truly private, and how do they compare to #private class fields?

No. Symbol keys are hidden from Object.keys/for...in/JSON.stringify, but they are reachable via Object.getOwnPropertySymbols and Reflect.ownKeys — so they are "soft" privacy, good for de-conflicting, not for secrecy. #private fields are hard private: enforced by the language, invisible to reflection, and accessing one from outside the class is a syntax error.
