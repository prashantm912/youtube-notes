Module: Strings

What this module covers

Two halves that interviewers probe in that order: first the everyday method toolkit — searching, extracting, replacing, padding, trimming and splitting text without mutating the original — then the layer most developers only half-understand, where a JavaScript string is a sequence of UTF-16 code units, not characters. Get the second half right and the classic emoji-length, accent-comparison and Base64 traps stop being surprises.

Topics

String methods: search, slice, replace, pad, trim, split

Strings are immutable primitives, so every method returns a new string and leaves the receiver untouched — a fact worth stating out loud in interviews. The core toolkit clusters by intent: search with includes, indexOf, startsWith/endsWith; extract with slice (negative indices allowed), substring and the deprecated substr; transform with replace/replaceAll, toUpperCase/toLowerCase, trim/trimStart/trimEnd and padStart/padEnd; and break apart or join with split, concat and repeat.

Why it matters: this is the most-used API in day-to-day JavaScript, and the small print is where questions live — replace with a string pattern swaps only the first match, slice vs substring handle negatives differently, and split("") quietly breaks on emoji. Knowing which method allocates a new string and which reads cleanly is table stakes.

const s = "  Hello, World  ";
s.trim();                          // "Hello, World"  (original unchanged)
s.trim().slice(7);                 // "World"
s.includes("World");               // true
"5".padStart(3, "0");              // "005"   — formatting/OTP/currency
"a,b,c".split(",");                // ["a", "b", "c"]
"2024-2025".replaceAll("-", "/");  // "2024/2025"

Unicode, code points, normalization and base64/hex

Internally a JavaScript string is UTF-16: a sequence of 16-bit code units. Characters in the Basic Multilingual Plane are one unit, but astral characters — most emoji, many scripts — are stored as a surrogate pair of two units, which is why "😀".length is 2 and index-based access can split a character in half. Iterating with a for…of loop or the spread operator walks whole code points instead; codePointAt/String.fromCodePoint read and build them, where the older charCodeAt/fromCharCode only see single code units.

Why it matters: the same visible text can have two different byte sequences — é as one precomposed code point or as e plus a combining accent — so === and searching fail until you normalize() both sides (NFC is the usual choice). And because a JS string holds text, not bytes, encoding to Base64 or hex means going through bytes first: the legacy btoa only handles Latin-1 and throws on Unicode, whereas the newer Uint8Array.prototype.toBase64/toHex (with TextEncoder) do it correctly.

"😀".length;                 // 2   — two UTF-16 code units (a surrogate pair)
[..."😀"].length;            // 1   — spread iterates code points
"😀".codePointAt(0);         // 128512
"café".normalize("NFC") === "café".normalize("NFC");  // true — same glyph, different code points, so normalize() first
new Uint8Array([72, 105]).toBase64();                       // "SGk="  ("Hi")

Interview questions

Most-asked string questions in Indian interviews — from front-end screening rounds to product-company deep dives where the Unicode edge cases decide the outcome:

Beginner Are JavaScript strings mutable? What does that imply for methods?

No — strings are immutable primitives. Every method (slice, replace, toUpperCase…) returns a brand-new string and leaves the original untouched; you cannot change a character in place via s[0] = "x". Implication: reassign the result (s = s.trim()) and avoid building large strings in tight loops without care.

Beginner slice vs substring vs substr — differences?

slice(start, end) accepts negative indices counted from the end. substring(start, end) treats negatives as 0 and swaps the arguments if start > end. substr(start, length) takes a length, not an end index, and is deprecated. Prefer slice.

Beginner How do you check whether a string contains a substring?

str.includes(sub) returns a boolean (ES2015). str.indexOf(sub) !== -1 is the older equivalent and also gives the position. startsWith/endsWith check the anchors specifically. For patterns, use a RegExp with test or search.

Beginner replace vs replaceAll — and how did people replace all matches before replaceAll?

replace(str, ...) swaps only the first occurrence; replaceAll(str, ...) (ES2021) swaps every one. Before replaceAll, the idiom was a global regex: str.replace(/-/g, "/"). Note replaceAll throws if you pass a non-global regex.

Beginner What does split() do, and why can split("") be unsafe?

split(separator) breaks a string into an array on each separator (a limit argument caps the count). split("") splits on every UTF-16 code unit, so it corrupts emoji and other astral characters into lone surrogates. To split into real characters use [...str] or, for user-perceived characters, Intl.Segmenter.

Beginner What are padStart and padEnd used for?

They pad a string to a target length with a fill string on the left (padStart) or right (padEnd). Typical uses: zero-padding numbers ("5".padStart(3, "0") → "005"), aligning columns, masking card digits, formatting time or OTP fields.

Intermediate Why is "😀".length equal to 2?

JavaScript strings are UTF-16, and length counts 16-bit code units, not characters. The emoji is an astral (beyond-BMP) character stored as a surrogate pair — two code units — so it reports a length of 2. Any BMP character (most Latin/Devanagari letters) counts as 1.

Intermediate charCodeAt vs codePointAt, and fromCharCode vs fromCodePoint?

charCodeAt returns a single UTF-16 code unit (0–65535); on a surrogate pair it returns only half. codePointAt returns the full Unicode code point, reading a surrogate pair as one value. Symmetrically, String.fromCharCode builds from code units while String.fromCodePoint builds from code points and handles astral characters correctly.

Intermediate How do you correctly count or iterate the characters of a string with emoji?

Iterating with for…of or spreading ([...str]) walks code points, so [..."😀"].length is 1 — right for surrogate pairs. But that still miscounts grapheme clusters like an emoji with a skin-tone modifier or a flag (multiple code points shown as one). For user-perceived characters use Intl.Segmenter(locale, { granularity: "grapheme" }).

Intermediate What is Unicode normalization and when do you need normalize()?

The same text can be encoded multiple ways — é as one precomposed code point or as e + combining accent. normalize() converts to a canonical form: NFC (composed, the common default), NFD (decomposed), and the compatibility forms NFKC/NFKD. Normalize before comparing, searching, deduplicating, or storing user-entered text and data from mixed sources.

Pro Two strings both display "café" but === returns false. Why, and how do you fix it?

One uses the precomposed é (U+00E9); the other uses e followed by the combining acute accent (U+0301). They render identically but are different code-unit sequences, so === is false and indexOf can miss. Fix by normalizing both sides to the same form before comparing: a.normalize("NFC") === b.normalize("NFC").

Pro How do you Base64-encode a Unicode string in the browser, and why does btoa fail?

btoa only accepts Latin-1 (code points 0–255) and throws on anything else, because Base64 encodes bytes and a JS string is text. Encode to bytes first: new TextEncoder().encode(str) gives a Uint8Array, then bytes.toBase64() (or the classic btoa(String.fromCharCode(...bytes))). Decoding reverses it with Uint8Array.fromBase64 + TextDecoder. The toHex/fromHex pair works the same way for hex.

Pro Why does "hello".split("").reverse().join("") fail for emoji, and how do you reverse a string correctly?

split("") cuts on code units, so reversing splits surrogate pairs and detaches combining marks — the emoji or accented letter is destroyed. Reversing code points ([...str].reverse().join("")) fixes surrogate pairs but still reorders combining marks and multi-code-point graphemes wrongly. The robust answer segments into grapheme clusters first: [...new Intl.Segmenter().segment(str)].map(s => s.segment).reverse().join("").
