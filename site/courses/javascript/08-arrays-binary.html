Module: Arrays and Binary Data

JavaScript arrays are resizable, heterogeneous, method-rich objects — and underneath the friendly list sits a second world of raw bytes: buffers, typed views and shared memory. This module maps both and links the material that teaches each properly, official docs first.

What this module covers

Three layers of the same idea — ordered collections of data. First the everyday Array: its iteration, mutation and the newer copying methods that return fresh arrays. Then the objects that look like arrays but are not — array-likes and iterables — and the conversion tools that bridge them. Finally the binary stack: ArrayBuffer, the typed-array views, DataView, and the concurrency pair SharedArrayBuffer and Atomics. Depth lives in the linked resources; this page orients you and hands you the interview questions each layer powers.

Topics

Array methods: iteration, mutation, and the copying family

The array API splits into two camps that interviewers love to probe. Mutating methods change the array in place and often return something other than the array — push, pop, shift, unshift, splice, sort, reverse, fill, copyWithin. Non-mutating methods build a new array or a scalar — map, filter, slice, concat, reduce. ES2023 closed the gap for immutable code with copying twins: toSorted, toReversed, toSpliced and with do the job of their mutating counterparts but leave the original untouched — ideal for React state and any place a shared array must not change under you.

Why it matters: knowing which methods mutate is the difference between a predictable pipeline and a spooky bug where a source array shifts beneath a component. Beyond that, reduce is the folding primitive every senior is expected to wield, flat/flatMap handle nesting, at brings negative indexing (arr.at(-1) for the last element), and Array.fromAsync (ES2024) drains async iterables into an array.

const nums = [3, 1, 2];
const sorted = nums.toSorted((a, b) => a - b); // [1, 2, 3]; a NEW array
nums;                                    // still [3, 1, 2] — untouched
[1, [2, [3]]].flat(Infinity);            // [1, 2, 3]
["a b", "c"].flatMap(s => s.split(" ")); // ["a", "b", "c"]
[10, 20, 30].at(-1);                     // 30
[1, 2, 3, 4].reduce((sum, n) => sum + n, 0); // 10

Array-likes, iterables, and Array.from / Array.of

Plenty of things behave like arrays without being one. An array-like has a length and indexed properties but none of the array methods — the classic examples are the arguments object and a DOM NodeList. An iterable is anything implementing Symbol.iterator, which is what makes for...of and the spread operator work — strings, arrays, Map, Set and typed arrays all qualify. The two categories overlap but are not the same: spread and for...of need iterability, while a plain array-like may not be iterable at all.

Why it matters: Array.from is the universal converter — it accepts both iterables and array-likes, and takes an optional map function, so Array.from({ length: 3 }, (_, i) => i) generates a range in one call. Spread ([...x]) is shorter but works only on iterables. And Array.of exists to fix a genuine footgun: Array(7) makes an empty length-7 array, while Array.of(7) makes [7].

Array.from(document.querySelectorAll("li"), li => li.textContent);
Array.from({ length: 3 }, (_, i) => i); // [0, 1, 2] — array-like + map fn
Array.of(7);   // [7]
Array(7);      // length 7, no elements (a sparse array)
[...new Set([1, 1, 2])]; // [1, 2] — spread needs an iterable

Binary data: ArrayBuffer, TypedArrays, DataView, SharedArrayBuffer, Atomics

When JavaScript touches files, network protocols, canvas pixels, WebGL, WebGPU or WebAssembly, it works on raw bytes. An ArrayBuffer is a fixed-length block of memory you cannot read or write directly — you need a view. The typed-array views (Int8Array, Uint8Array, Uint8ClampedArray, Int32Array, Float32Array, Float64Array, the BigInt64Array pair, and the newer Float16Array) each interpret the same bytes as a fixed numeric type in platform byte order. A DataView is the low-level alternative: manual getInt16/setFloat32-style access with explicit control over endianness, which is what you want for mixed-format binary structures like file headers and wire protocols.

Why it matters: multiple views can alias one buffer, so writing through one view and reading through another reveals byte layout and endianness — a favourite systems question. The concurrency pair goes further: a SharedArrayBuffer can be shared between the main thread and Web Workers (it requires cross-origin isolation via COOP/COEP headers, a Spectre-era safeguard), and Atomics supplies lock-free atomic reads, writes and read-modify-write operations plus a wait/notify mechanism to coordinate threads without data races.

const buf = new ArrayBuffer(8);       // 8 raw bytes
const i32 = new Int32Array(buf);      // view: two 32-bit ints
i32[0] = 256;
const bytes = new Uint8Array(buf);    // SAME memory, byte view
bytes[0];                             // 0 on little-endian (low byte first)

const view = new DataView(buf);
view.setInt16(0, 1, /* littleEndian */ false); // explicit big-endian write
view.getInt16(0, false);              // 1

Interview questions

Array and binary-data questions that recur across Indian interviews — from service-company screening rounds to product-company DSA and systems discussions:

Beginner Which common array methods mutate the array, and which return a new one?

Mutating in place: push, pop, shift, unshift, splice, sort, reverse, fill, copyWithin. Non-mutating (return new array or a value): map, filter, slice, concat, reduce, and the ES2023 copying set toSorted, toReversed, toSpliced, with. Knowing the split prevents accidental shared-state bugs.

Beginner What does at() give you that bracket indexing does not?

Negative indices. arr.at(-1) returns the last element and arr.at(-2) the second-to-last, whereas arr[-1] just reads a non-existent property and yields undefined. It reads the same as positive indexing: arr.at(0) equals arr[0].

Beginner Array(7) vs Array.of(7) — what is the difference?

Array(7) creates a sparse array of length 7 with no actual elements (the single-number constructor sets length). Array.of(7) always treats arguments as elements, so it returns [7]. Array.of exists precisely to remove that inconsistency.

Beginner What is an array-like object, and how is it different from an array?

An array-like has a length property and indexed keys (0, 1, …) but none of the Array.prototype methods — you cannot call .map on it. Examples: the arguments object and a NodeList. Convert with Array.from(x) or, if it is iterable, spread [...x].

Beginner How do you turn a NodeList or arguments into a real array?

Array.from(nodeList) works for both because they are array-like (and both happen to be iterable). Spread [...nodeList] also works since both implement Symbol.iterator. For a purely array-like object that is not iterable, only Array.from works.

Intermediate forEach vs map — when do you reach for each?

map transforms and returns a new array of the same length; use it when you need the results. forEach returns undefined and is for side effects only. Also, neither can break early — use a for...of loop, or some/every, when you need to stop.

Intermediate Explain reduce. What is a common bug with it?

reduce((acc, item) => ..., initial) folds an array into a single value — a sum, an object, a grouped map. The classic bug is omitting the initial value: on an empty array reduce then throws "Reduce of empty array with no initial value", and on a one-element array the callback never runs. Always pass an explicit initial value unless you deliberately want the first element as the seed.

Intermediate Why were toSorted, toReversed and with added when sort and reverse already exist?

Because sort and reverse mutate in place, which is dangerous when the array is shared or held in immutable state (React, Redux). The copying versions return a new array and leave the original alone, so you can write const next = items.toSorted(cmp) without cloning first. Reminder: default sort compares as strings, so numbers need a compare function like (a, b) => a - b.

Intermediate Array.from vs Array.fromAsync?

Array.from is synchronous and works on iterables and array-likes. Array.fromAsync (ES2024) also accepts async iterables (a ReadableStream, an async generator), returns a Promise, and awaits each value sequentially as it pulls it — different from Promise.all, which starts everything concurrently. Use it to drain a stream into an array with await.

Intermediate What is an ArrayBuffer, and why can't you read it directly?

An ArrayBuffer is a fixed-length block of raw bytes with no type information, so there is nothing to read until you attach a view. A typed array (say Uint8Array) or a DataView gives those bytes a numeric interpretation and lets you index or get/set them. One buffer can back several views at once.

Intermediate TypedArray vs DataView — when would you pick DataView?

Typed arrays are simple and fast when all elements share one type and you accept platform (native) byte order. Reach for DataView when a buffer mixes types — a header with an 8-bit tag, a 16-bit length and a 32-bit value — or when you must fix endianness explicitly, as with network protocols and file formats. DataView's get/set methods take a littleEndian flag per call.

Pro What is endianness, and how do TypedArrays and DataView each handle it?

Endianness is the order in which a multi-byte number is laid out in memory — little-endian stores the least-significant byte first, big-endian the most-significant first. Plain typed arrays use the platform's native order (little-endian on most machines), so they are not portable across the wire. DataView takes an explicit littleEndian argument and defaults to big-endian, which is why it is the right tool for byte layouts that must be deterministic regardless of hardware.

Pro What problem do SharedArrayBuffer and Atomics solve, and what does using them require?

A normal ArrayBuffer passed to a worker is copied (or transferred and detached). A SharedArrayBuffer is genuinely shared memory visible to the main thread and workers at once — enabling real parallel data sharing but also data races. Atomics provides lock-free atomic operations (add, compareExchange, load, store) and a wait/notify pair to coordinate threads safely. Using SharedArrayBuffer requires cross-origin isolation — the page must send COOP and COEP headers — a mitigation added after the Spectre timing attacks.

Pro Spread versus Array.from — beyond style, when does the choice actually matter?

Spread [...x] requires an iterable; it throws on a plain array-like that lacks Symbol.iterator. Array.from handles both iterables and array-likes, and it accepts a map function — so Array.from(set, x => x * 2) converts and transforms in one pass without an intermediate array, and Array.from({ length: n }, (_, i) => i) builds a range from a non-iterable. Pick Array.from when the source may be array-like, when you need the map callback, or when you want the index during conversion.
