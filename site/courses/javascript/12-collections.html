Module: Collections

When an object or array stops being the right tool — keys that aren't strings, membership tests, or storage that must not leak memory — the collection types take over. This module maps each one to the job it was built for.

Knowledge Library → Courses → JavaScript → Collections

What this module covers

JavaScript ships four built-in collections beyond arrays and plain objects: Map and Set for ordered, any-typed keys and unique values; the Weak variants (WeakMap, WeakSet) plus WeakRef and FinalizationRegistry for storage that cooperates with the garbage collector; and, newly Baseline, the set-algebra methods on Set together with Map.groupBy and getOrInsert. Knowing which to reach for — and the memory model behind the Weak family — is a common mid-to-senior interview theme.

Topics

Map, Set, when to use vs objects/arrays

A Map is a keyed collection that accepts any value as a key (objects, functions, NaN — not just strings and symbols), preserves insertion order, exposes its count through .size, and iterates directly. A Set stores unique values with the same order guarantee, making membership tests and deduplication a one-liner. Both compare keys/values with the SameValueZero algorithm, so NaN is findable and +0 equals -0.

Why it matters: a plain object coerces every key to a string or symbol, inherits keys from its prototype (a toString key collision is a classic bug), and has no built-in size. Reach for a Map when keys are dynamic, non-string, or added and removed frequently; keep an object for fixed-shape records that serialize to JSON. The "Map vs object" and "how do I dedupe an array" questions are near-universal in Indian screening rounds.

const seen = new Map();              // any value as a key, insertion-ordered
seen.set(userObj, 1).set("id", 2);
seen.size;                           // 2  (Map/Set use .size, not .length)

const unique = [...new Set([1, 1, 2, 3, 3])]; // [1, 2, 3]

WeakMap/WeakSet/WeakRef/FinalizationRegistry

The Weak collections hold their keys (WeakMap) or values (WeakSet) weakly: an entry does not stop its key object from being garbage-collected, and when the key is reclaimed the entry vanishes on its own. Keys must be objects (or non-registered symbols), and — because the timing of collection is not observable — Weak collections are not iterable, have no .size, and cannot be cleared as a whole. This makes them the right tool for associating private data or metadata with objects you don't own, and for caches keyed by object identity, without leaking memory when those objects go away.

Why it matters: WeakRef takes the idea further, holding a weak reference to a single object that you read back with .deref() (which may return undefined if it's gone), while FinalizationRegistry lets you register a cleanup callback that may run after an object is collected. Both are deliberately hard to use well — MDN and the V8 team warn against depending on them for core logic because garbage-collection timing is non-deterministic across engines. Interviewers love probing whether a candidate knows a WeakMap only weakly references its key, not the value, and that it is not a cure-all for memory leaks.

const store = new WeakMap();         // keys must be objects, held weakly
class Account {
  constructor(balance) { store.set(this, { balance }); }
  get balance() { return store.get(this).balance; }
}
// when an Account becomes unreachable, its entry is GC'd — no manual cleanup

Set methods (union/intersection/difference), Map.groupBy/getOrInsert

Since 2024, Set carries the set-algebra methods that previously required manual loops: union, intersection, difference, symmetricDifference, and the boolean predicates isSubsetOf, isSupersetOf, and isDisjointFrom. Each returns a fresh Set (or a boolean) and accepts any set-like object with size, has, and keys. Alongside them, the static Map.groupBy(items, fn) buckets an iterable into a Map keyed by the callback's return value — useful precisely because the keys can be objects, unlike Object.groupBy whose keys are coerced to strings/symbols.

Why it matters: these turn common data-wrangling into declarative one-liners and show up in modern code review. The newest addition, Map.prototype.getOrInsert(key, default) (and getOrInsertComputed for a lazy factory), reads a key or seeds it in a single step — the idiomatic fix for the read-check-then-set dance behind counters and multimaps. Because these features are recent, an interviewer asking about them is really checking whether you keep up with the language.

const a = new Set([1, 2, 3]), b = new Set([2, 3, 4]);
a.union(b);          // Set {1, 2, 3, 4}
a.intersection(b);   // Set {2, 3}
a.difference(b);     // Set {1}

Map.groupBy([1, 2, 3, 4], n => n % 2 ? "odd" : "even");
// Map { "odd" => [1, 3], "even" => [2, 4] }

new Map().getOrInsert("hits", 0); // returns 0 and seeds the key in one step

Interview questions

Most-asked collections questions in Indian interviews, from screening rounds to product-company deep dives:

Beginner What is the difference between a Map and a plain object?

A Map takes keys of any type (objects, functions, numbers), preserves insertion order, reports count via .size, and is directly iterable. A plain object coerces keys to strings/symbols, inherits prototype keys (risk of collisions like toString), and has no built-in size. Use a Map for dynamic or non-string keys; use an object for fixed-shape records that serialize to JSON.

Beginner How do you remove duplicates from an array using a Set?

[...new Set(arr)] or Array.from(new Set(arr)). A Set keeps only unique values (SameValueZero equality), and spreading it back gives a deduped array in original insertion order.

Beginner How do you get the number of entries in a Map or Set?

The .size property — not .length. length is for arrays and strings; Map and Set expose size, and it is read-only.

Beginner Can an object be a key in a Map? What about in a plain object?

Yes in a Map — the object identity is the key, so two different objects with the same shape are distinct keys. In a plain object, a non-string key is coerced via toString(), so every object collapses to the key "[object Object]" and they overwrite each other.

Intermediate What equality does Map/Set use? Can NaN be a key?

SameValueZero. It behaves like === except that NaN equals NaN, so NaN works as a findable key, and +0 and -0 are treated as the same key. This is why new Set([NaN, NaN]).size is 1.

Intermediate When would you choose a Map over an object, and vice versa?

Map: keys that aren't strings, keys not known ahead of time, frequent add/delete, needing size or guaranteed iteration order, or object-identity keys. Object: a fixed set of known properties, JSON serialization, method definitions, or when other code expects a plain record. Rule of thumb — data keyed at runtime leans Map; a record with a known shape leans object.

Intermediate What is a WeakMap and what is it good for?

A Map whose keys must be objects and are held weakly — an entry does not prevent its key from being garbage-collected, and disappears when the key is reclaimed. It is not iterable and has no size. Ideal for attaching private data or metadata to objects, and for caches keyed by object identity, with no memory leak when those objects are discarded.

Intermediate Why can't you iterate a WeakMap or read its size?

Because garbage-collection timing is non-deterministic. If you could enumerate entries or read a count, you would be observing exactly when the collector ran, which the spec deliberately hides. So Weak collections expose only per-key get/set/has/delete, never iteration.

Intermediate What are the new Set methods, and how does difference differ from symmetricDifference?

Baseline-2024 methods: union, intersection, difference, symmetricDifference, plus predicates isSubsetOf, isSupersetOf, isDisjointFrom. a.difference(b) keeps items in a but not b; a.symmetricDifference(b) keeps items in exactly one of the two sets (neither's shared elements). Each returns a new Set and accepts any set-like object.

Intermediate Map.groupBy vs Object.groupBy — what is the difference?

Both bucket an iterable by a callback's return value. Object.groupBy returns a null-prototype object, so keys are coerced to strings/symbols. Map.groupBy returns a Map, so keys can be any value including objects — use it when you want to group by an object key or preserve key types.

Pro Explain WeakRef and FinalizationRegistry, and why the docs warn against relying on them.

WeakRef holds a weak reference to one object; .deref() returns it or undefined if collected. FinalizationRegistry registers a callback that may fire after a watched object is reclaimed. Both depend on garbage-collection timing, which varies by engine and is not guaranteed — finalizers may never run (e.g. on page unload). MDN and the V8 team advise against using them for program correctness; treat them as best-effort cleanup, not core logic.

Pro Does a WeakMap guarantee no memory leaks?

No — the weakness is only on the key. As long as the key object is reachable through any other strong reference, the entry (and its value) stays alive; the value is held strongly for as long as the key lives. And if the value strongly references the key, or you keep the key elsewhere, nothing is collected. A WeakMap avoids leaks caused by the map itself keeping keys alive; it does not fix leaks where the keys are long-lived.

Pro You cache expensive results keyed by objects. WeakMap vs a regular Map with manual eviction — and where does getOrInsert fit?

A WeakMap auto-evicts when a key object becomes unreachable, so there is no unbounded growth and no cleanup code — but you cannot enumerate, size, or bound it, and it only works with object keys. A regular Map is inspectable and lets you implement an LRU/size cap, but you must evict manually or risk a leak. For the read-or-create step in either, getOrInsert(key, default) (or getOrInsertComputed(key, factory) to defer building an expensive default) replaces the has/get/set dance atomically — the idiomatic way to seed counters and multimap buckets.
