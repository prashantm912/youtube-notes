Module: Web Components

Custom elements let you ship a <user-card> that behaves like a built-in tag, and Shadow DOM keeps its markup and styles from leaking in either direction. This module points you at the specs and the practical guides that make the three APIs click.

What this module covers

Web Components are three cooperating browser APIs — custom elements, Shadow DOM, and the <template>/<slot> pair — that package HTML, CSS, and behavior into a reusable tag with no framework required. This module orients you on defining elements and their lifecycle, encapsulating internals behind a shadow root and projecting content through slots, and — the senior-level material — making a custom element a real form control with ElementInternals while exposing styling hooks through ::part. Depth lives in the linked specs and guides.

Topics

customElements.define, lifecycle callbacks

A custom element is a JavaScript class extending HTMLElement, registered with customElements.define("tag-name", Class). The name must contain a hyphen, which reserves the author namespace so it can never collide with a current or future built-in tag. There are two flavors: autonomous elements (extend HTMLElement and render everything themselves) and customized built-ins (extend a concrete class such as HTMLButtonElement and register with { extends: "button" } — not supported in Safari). Once defined, the browser upgrades any matching element already parsed in the DOM as well as any created afterward.

The class plugs into a fixed lifecycle: connectedCallback (inserted into the document — do setup, add listeners, render), disconnectedCallback (removed — tear down listeners and timers so nothing leaks), attributeChangedCallback (a watched attribute changed — but only the names listed in the static observedAttributes array trigger it), and adoptedCallback (moved into a new document). Why it matters: interviewers reliably probe the hyphen rule, why setup belongs in connectedCallback rather than the constructor, and how observedAttributes gates attribute reactions.

class UserCard extends HTMLElement {
  static observedAttributes = ["name"];            // only these fire the callback
  connectedCallback()    { this.render(); }        // inserted into the DOM
  disconnectedCallback() { /* remove listeners */ } // removed — clean up
  attributeChangedCallback(name, oldV, newV) { this.render(); }
  render() { this.textContent = this.getAttribute("name") ?? ""; }
}
customElements.define("user-card", UserCard);      // name MUST contain a hyphen

Shadow DOM API (attachShadow, slots, events)

host.attachShadow({ mode: "open" }) attaches a hidden, encapsulated subtree — the shadow root — to a host element. Styles defined inside it don't leak out, page styles don't leak in (inherited properties and CSS custom properties are the deliberate exceptions), and document.querySelector can't reach across the boundary. Mode "open" exposes the tree through host.shadowRoot; "closed" returns null and hides it. Other options include delegatesFocus and slotAssignment.

Composition comes from <slot>: the light-DOM children an author writes between the tags are projected into slot positions inside the shadow tree, with named slots (slot="header") routing to <slot name="header"> and a default slot catching the rest. Events that cross the boundary are retargeted so event.target reports the host rather than the private internal node, and an event only escapes the shadow root at all when it is dispatched with composed: true — which custom events are not, by default. Why it matters: the open-vs-closed choice, "why is my global CSS not styling the component," and event retargeting are all standard mid-level questions.

class FancyCard extends HTMLElement {
  constructor() {
    super();
    const root = this.attachShadow({ mode: "open" }); // open => page JS can read .shadowRoot
    root.innerHTML =
      "<div class='box'><slot>fallback text</slot></div>"; // slot projects light DOM
  }
}
customElements.define("fancy-card", FancyCard);

Form-associated custom elements, ElementInternals, ::part styling

By default a custom element is invisible to a surrounding <form>. Setting the static field formAssociated = true and calling this.attachInternals() returns an ElementInternals object — the bridge to the element's form and accessibility internals. setFormValue(value) makes the control's value part of the submitted FormData; setValidity(flags, message, anchor) plugs into constraint validation (:invalid, reportValidity()); and the same object carries ARIA role/label defaults plus lifecycle hooks like formResetCallback and formDisabledCallback. This is how you build a genuine <rating-input> or toggle switch that submits, validates, and is labelable — Baseline widely available since 2023.

Encapsulation cuts both ways: outside CSS cannot reach inside a shadow tree. ::part is the sanctioned escape hatch — tag an internal node with part="thumb" and consumers style it with your-element::part(thumb) { … } without breaching the rest of the encapsulation, while exportparts forwards parts up through nested components. Why it matters: form-associated elements with ElementInternals are the current senior-level Web Components topic, and choosing between ::part, exposed CSS custom properties, and ::slotted for theming is a common design-tradeoff question.

class RatingInput extends HTMLElement {
  static formAssociated = true;                 // opt in to form participation
  #internals = this.attachInternals();          // the ElementInternals object
  set value(v) {
    this.#internals.setFormValue(v);            // value now submits with the form
    this.#internals.setValidity(
      v ? {} : { valueMissing: true },          // constraint validation state
      "Please pick a rating", this);
  }
}
customElements.define("rating-input", RatingInput);

Interview questions

Most-asked Web Components questions in Indian interviews, from front-end screening rounds to product-company deep dives:

Beginner What are Web Components, and which browser APIs make them up?

A set of standards for building reusable, encapsulated custom tags with no framework. Three APIs cooperate: custom elements (define a tag and its behavior/lifecycle), Shadow DOM (an encapsulated DOM+CSS subtree), and HTML templates (<template> and <slot> for inert, reusable markup and content projection).

Beginner Why must a custom element's name contain a hyphen?

The hyphen reserves the author namespace: the HTML spec guarantees it will never mint a built-in tag containing a dash, so user-card can't clash with a current or future standard element. It also tells the parser to treat the tag as a custom element. customElements.define throws if the name has no hyphen.

Beginner What are the custom-element lifecycle callbacks?

connectedCallback (added to the DOM), disconnectedCallback (removed), attributeChangedCallback(name, old, new) (a watched attribute changed — fires only for names in the static observedAttributes array), and adoptedCallback (moved to another document). Registration itself is customElements.define.

Beginner Autonomous vs customized built-in elements?

Autonomous elements extend HTMLElement and provide all behavior themselves (<user-card>). Customized built-ins extend a specific class and register with { extends: "button" }, reusing native behavior and rendering as <button is="fancy-button">. Safari has never shipped customized built-ins, so most teams stick to autonomous elements.

Beginner What does attachShadow do, and what is mode open vs closed?

It attaches an encapsulated shadow subtree to a host and returns the ShadowRoot. mode: "open" lets outside JavaScript reach the tree via host.shadowRoot; mode: "closed" makes host.shadowRoot return null. Closed is not a real security boundary — the constructor still holds the reference — so open is the common choice.

Intermediate Why put setup code in connectedCallback instead of the constructor?

The constructor runs at element creation, which may be before it is inserted; at that point you must not inspect attributes or children or add child nodes. connectedCallback runs when the element is in the document, so it is the safe place to read attributes, render, and attach listeners. Note it can fire more than once (on move/re-insert), so make setup idempotent and undo it in disconnectedCallback.

Intermediate How do slots work, and what is a named vs default slot?

Light-DOM children of the host are projected into <slot> positions in the shadow tree. A child with slot="header" lands in <slot name="header">; children with no slot attribute fill the unnamed default slot. Content inside a <slot> tag is fallback shown only when nothing is projected. Slotted nodes stay in the light DOM (styled by the page), and you observe changes with the slotchange event.

Intermediate What is event retargeting across a shadow boundary, and what does composed do?

When an event bubbles out of a shadow root, the browser retargets it so event.target becomes the host element, hiding the internal node from outside listeners (use event.composedPath() to see the real path). An event only crosses the shadow boundary when created with composed: true; built-in UI events mostly are, but CustomEvent defaults to composed: false, so a custom event won't escape unless you opt in.

Intermediate Why isn't your page CSS styling inside a shadow root, and how do you style it on purpose?

Shadow DOM scopes styles: page selectors don't match inside, and inside selectors don't escape. What does cross the boundary is inherited CSS (color, font) and CSS custom properties. The sanctioned styling hooks are exposed custom properties (--thumb-color), ::part() for elements the component marks with part=, and ::slotted() for projected light-DOM nodes.

Intermediate How do you make a custom element participate in a form?

Add static formAssociated = true, call this.attachInternals() in the constructor, and call internals.setFormValue(value) whenever the value changes so it is submitted as FormData. Round it out with internals.setValidity(...) for constraint validation and the formResetCallback/formDisabledCallback/formStateRestoreCallback hooks so reset, disabling, and autofill behave natively.

Pro What does ElementInternals give you beyond a form value?

It is the element's private internals surface. Validity and constraint validation via setValidity (with a validity anchor and message). Default accessibility semantics — role, ariaLabel and other ARIA defaults — set without reflecting attributes onto the element. And custom states through the element's CustomStateSet (internals.states.add("checked")), which style with the :state(checked) pseudo-class. It keeps this internal state off the public attribute surface where consumers could stomp on it.

Pro ::part vs CSS custom properties vs ::slotted — which theming hook, and why?

CSS custom properties expose specific, intended knobs (a color, a radius) and inherit through the boundary — safest and most stable API, but only what you predeclare. ::part() lets consumers apply arbitrary rules to elements you explicitly tag with part= (and forward through nesting with exportparts) — more power, slightly more coupling to your internal names. ::slotted() styles projected light-DOM children only. Prefer custom properties for a small stable surface; add named parts for elements you want deeply themeable; reserve ::slotted for the author's own content.

Pro When would you reach for Web Components instead of a framework like React, and what are the trade-offs?

Choose them for framework-agnostic reuse: design-system widgets shared across teams and stacks, embeddable third-party components, or micro-frontends, where native lifecycle and style encapsulation and no runtime dependency are worth it. Trade-offs: no built-in reactivity or templating (you wire rendering yourself, or use a thin layer like Lit), a weaker server-rendering/hydration story, more boilerplate, and forms/accessibility that must be built by hand through ElementInternals. In practice they are often complementary — a Web Component design system consumed inside a React app — rather than an either/or.
