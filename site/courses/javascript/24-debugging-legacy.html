Module: Debugging, Legacy and Beyond

The last JavaScript module is about the craft around the language: finding bugs with the console and the debugger, knowing what strict mode protects you from and which old features still lurk in real code, and a quick map of the wider platform APIs worth recognizing.

What this module covers

Three loosely related but interview-relevant strands close the track: the debugging toolkit (the whole console API, the debugger statement, and the DevTools breakpoint workflow), the legacy corners of the language (strict versus sloppy mode, why eval and with are shunned, and the deprecated-but-still-shipping features you meet in old code), and a breadth survey of the wider platform — WebRTC, Web Audio, WebGL/WebGPU, WebAuthn, Speech, sensors and Houdini — so you can recognize what each is for even if you never build one from scratch.

Topics

console API (every method), debugger, DevTools workflow

The console object is far more than console.log. It offers leveled output (log, info, debug, warn, error) that DevTools can filter and that Node routes to stdout/stderr; table for arrays or objects rendered as a sortable grid; dir for an object's property tree; group/groupCollapsed/groupEnd for nested, collapsible sections; count, time/timeEnd and trace for counting, timing and stack traces; assert for conditional logging; and %s/%d/%c string substitutions. The debugger statement drops execution into any attached debugger at that exact line — a breakpoint you commit in source.

Why it matters: console.log is fine for a quick peek, but the DevTools Sources panel — line-of-code, conditional and logpoint breakpoints, step over/into/out, the call stack, and the Scope and Watch panes — lets you inspect every variable at a paused instant without editing and re-running code. "How do you debug beyond console.log?" is a near-guaranteed question in Indian screening rounds, and interviewers expect breakpoints, conditional breakpoints, and knowing when each beats logging.

console.table([{ id: 1, city: "Pune" }, { id: 2, city: "Chennai" }]); // grid
console.group("checkout"); console.log("validate cart"); console.groupEnd();
console.assert(total > 0, "total must be positive", { total }); // logs only if false
console.time("fetch"); /* ...work... */ console.timeEnd("fetch");    // fetch: 12.4 ms

Strict vs sloppy mode, eval/with, deprecated-but-real features

The directive "use strict" (ES5) opts a script or a single function into strict mode, which turns silent mistakes into thrown errors: assigning to an undeclared variable, writing to a read-only or non-writable property, duplicate parameter names, octal literals like 0644, and deleting a bare variable all throw. It also changes semantics — a plain function call's this is undefined instead of the global object, and the with statement is banned. ES modules and class bodies are strict automatically, so most modern code already runs this way without the directive.

Why it matters: eval() runs a string as code with the caller's privileges — an injection sink, unoptimizable by engines and minifiers, and slow; the honest alternatives are the Function constructor, JSON.parse for data, or bracket property access. with is deprecated and illegal in strict mode because it makes name resolution ambiguous and blocks optimization. And "deprecated but real" features — HTML-style comments, String.prototype.substr, escape/unescape, the __proto__ accessor, Date.prototype.getYear, the RegExp.$1 statics — still ship for web-compatibility and turn up in legacy codebases you will maintain, so you must recognize them even though you would never write them into new code.

"use strict";
x = 10;                          // ReferenceError — no accidental global
function whoAmI() { return this; }
whoAmI();                        // undefined  (would be the global object in sloppy mode)
// with (obj) { ... }            // SyntaxError in strict mode

Survey: WebRTC, Web Audio, WebGL/WebGPU, WebAuthn, Speech, sensors, Houdini

The platform reaches well past the DOM, and part of being a strong JavaScript developer is knowing what else is there. WebRTC does peer-to-peer audio, video and data with no server in the media path (calls, screen sharing). The Web Audio API builds a routing graph of oscillators, filters and effects for synthesis, playback and analysis. WebGL (OpenGL ES inside a <canvas>) and its successor WebGPU (built on Vulkan/Metal/Direct3D 12) give GPU-accelerated 2D/3D rendering and general-purpose compute. The Web Authentication API (WebAuthn) replaces passwords with public-key passkeys bound to the site origin. The Web Speech API handles text-to-speech and speech recognition, sensor and device APIs (Geolocation, the Generic Sensor family) read the physical world, and CSS Houdini exposes the styling and layout engine through registered custom properties, paint worklets and the Typed CSSOM. You are rarely asked to build these from scratch in an interview, but recognizing each — and roughly when to reach for it — signals platform breadth.

Interview questions

Most-asked debugging and legacy-JavaScript questions in Indian interviews, from screening rounds to senior product-company deep dives:

Beginner What is the difference between console.log, console.info, console.warn and console.error?

All four write to the console, but warn and error add a distinct icon/colour and are separately filterable in DevTools; error (and usually warn) attach a stack trace, and in Node they go to stderr while log/info go to stdout. Using the right level keeps logs filterable instead of dumping everything at one level.

Beginner What does the debugger statement do, and when would you use it over console.log?

It pauses execution at that line and hands control to any attached debugger (DevTools), exactly like a breakpoint set in the Sources panel — but committed in code. Handy for code that is hard to breakpoint by hand (dynamically generated, or only interesting on a specific path). Use it over console.log when you need to inspect the full scope and step through, not just print one value. Remove it before shipping.

Beginner How do you set a breakpoint and step through code in Chrome DevTools?

Open DevTools → Sources, click the line-number gutter to set a line-of-code breakpoint (right-click for a conditional breakpoint or a logpoint), then reload or trigger the code. While paused, use Step over (run the next line), Step into (enter a call), Step out (finish the current function) and Resume, and read the Scope, Watch and Call Stack panes or hover values inline.

Beginner What is strict mode and how do you turn it on?

A restricted variant of JavaScript that converts many silent mistakes into thrown errors and removes error-prone syntax. Enable it with the string literal "use strict"; as the first statement of a script or of a function. ES modules and class bodies are strict automatically. Once on for a scope it cannot be turned off.

Beginner Name a few mistakes that strict mode turns into errors.

Assigning to an undeclared variable (ReferenceError instead of silently creating a global), writing to a read-only or getter-only property (TypeError instead of a silent no-op), duplicate function parameter names, octal literals like 0644, deleting a plain variable, and using future reserved words. Also, this is undefined in an ordinary function call rather than the global object.

Intermediate console.log versus setting breakpoints — when is each the right tool?

console.log is quick, good for high-frequency or production-safe tracing, but you have to guess what to print and re-run to change it. Breakpoints pause execution so you can inspect every variable, the call stack and scope at that instant, change nothing, and step forward — far faster for non-trivial bugs. The senior answer: use logpoints or conditional breakpoints to get logging without editing and redeploying the source.

Intermediate What do console.table, console.group and console.assert do?

table renders an array (or an object of objects) as a sortable grid; group/groupCollapsed plus groupEnd nest related logs under a collapsible header; assert(condition, ...msg) logs only when the condition is falsy — a cheap inline invariant check. Round it out with count, dir, trace and time/timeEnd.

Intermediate Why is eval() discouraged, and what should you use instead?

It executes an arbitrary string with the caller's privileges — a classic XSS/injection sink — and it defeats engine optimization and minifiers while being slow. Alternatives: the Function constructor (runs in global scope, marginally safer), JSON.parse for data, bracket notation obj[key] for dynamic property access, or a real parser. The same caution applies to setTimeout("code") and new Function(...).

Intermediate In a plain function call, how does this differ between strict and sloppy mode?

In sloppy mode, calling a plain function (not as a method, not with new) sets this to the global object (window). In strict mode this stays undefined, so accidental global mutation via this throws loudly instead of silently polluting globals — which is why a method detached from its object fails visibly in strict mode.

Intermediate Are ES modules and class bodies strict by default, and why does that matter?

Yes — the body of every ES module and every class is strict without any directive. It matters because behavior differs from a sloppy <script>: this is undefined at module top level, accidental globals throw, and sloppy-only tricks (with, octal literals) won't parse. Pasting old script code into a module can surface latent bugs that were previously silent.

Pro You log an object, expand it later in the console, and the values differ from the moment you logged it. Why, and how do you snapshot it?

The console often stores a live reference and only renders an object's properties lazily when you expand it, so mutations made after the console.log call show up. To capture the state at log time, log a copy: structuredClone(obj), JSON.parse(JSON.stringify(obj)), a shallow spread { ...obj }, or the individual primitive fields. This lazy-evaluation gotcha is a favourite senior probe.

Pro What is the difference between direct and indirect eval?

Direct eval — eval(str) called by that literal name — runs in the caller's lexical scope and can read and modify local variables. Indirect eval — any call where eval is not referenced directly, e.g. (0, eval)(str) or const e = eval; e(str) — always runs in the global scope and cannot touch locals. In strict mode even direct eval gets its own scope, so it cannot declare variables in the caller. Indirect eval is the safer, more optimizable form.

Pro What is the difference between a "deprecated" and an "obsolete" feature, with an example of each?

Deprecated = still implemented and working, kept for backward compatibility, but not to be used in new code — e.g. String.prototype.substr, escape/unescape, the __proto__ accessor, Date.prototype.getYear, the RegExp.$1 statics, HTML-style comments. Obsolete = removed from the spec and no longer reliably available — e.g. Object.observe, Object.prototype.watch, the old array/string generics. You still meet deprecated features in legacy code, so recognize them; do not add them.

Pro From the survey: how would you choose between WebGL and WebGPU, or between WebSockets and WebRTC?

WebGL for broad-support 2D/3D via OpenGL ES; WebGPU when you need modern-GPU features, compute shaders, or better performance and can accept narrower (not-yet-Baseline) support. WebSockets for a simple bidirectional client-server channel; WebRTC when you need low-latency peer-to-peer media or data (calls, screen share) with the server out of the media path. The interview point is matching the API to the constraint — latency, topology, GPU access — not implementing it from memory.
