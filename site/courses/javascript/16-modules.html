Module: Modules

A module is just a file with its own scope, and after years of competing conventions JavaScript now has one native format for them. This module maps the import/export surface, how a specifier becomes a real URL, and the newest scope-based cleanup syntax — pointing to the canonical docs for each.

What this module covers

ECMAScript Modules (ESM) are the standard way to split JavaScript across files: static import/export with their many forms, lazy import(), the import.meta context object, and import attributes for loading JSON and other resource types. Around that sit the resolution story — import maps in the browser and the reasons bundlers still dominate real builds — and a newer, mostly orthogonal feature that rides on module-era syntax: explicit resource management via using, await using and Symbol.dispose. Modules are a favourite interview area because they expose how the engine actually loads and links code.

Topics

ESM (import/export forms, dynamic import, import.meta, import attributes)

One file is one module, and everything inside it is private until you export it. Exports come in two flavours: any number of named exports (bound by name) and at most one default export (the module's headline value). The matching import forms mirror that — named, default, and import * as ns for the whole namespace object — plus a bare import "./setup.js" that runs a module purely for its side effects. Crucially, imported bindings are live and read-only: you receive a view onto the exporter's variable, not a copy, so a later change in the source module is visible to every importer.

Why it matters: beyond static syntax, the same system offers import() — a promise-returning dynamic form used for code-splitting and conditional loading, valid even in classic scripts — and import.meta, an object of per-module metadata whose import.meta.url is the ESM replacement for __dirname/__filename. The newest addition is import attributes, the with { type: "json" } clause that lets the runtime import non-JavaScript resources safely. Interviewers probe this surface constantly: named vs default, live bindings, and when to reach for dynamic import are staple questions in Indian front-end and full-stack rounds.

// circle.js — named exports plus one default export
export const PI = 3.14159;
export function area(r) { return PI * r * r; }
export default class Circle {}

// consumer.js — the matching import forms
import Circle, { PI, area as areaOf } from "./circle.js"; // default + named (renamed)
import * as shapes from "./circle.js";                     // namespace object
const mod = await import("./circle.js");                   // dynamic — returns a Promise
import config from "./config.json" with { type: "json" };  // import attributes
console.log(import.meta.url);                               // this module's own URL

Import maps, module resolution, bundler reality

A module specifier is the string after from. Relative (./x.js) and absolute-URL specifiers resolve naturally in the browser, but a bare specifier like "lodash" has no meaning to a browser on its own — that convenience came from bundlers. Import maps close the gap: a <script type="importmap"> block declares how bare names and path prefixes map to real URLs, so native ESM can use ergonomic names without a build step. Maps also support scoped overrides (different resolutions under different paths) and integrity metadata, and they are now supported across every modern engine.

Why it matters: even with native modules and import maps, production almost always still ships through a bundler (Vite, esbuild, webpack, Rollup). The reasons are the "bundler reality" an interviewer wants you to articulate — avoiding a network waterfall of hundreds of tiny requests, tree-shaking dead exports, transpiling newer syntax for older targets, hashing filenames for caching, and folding in non-JS assets. Native ESM shines for simple sites, demos and dev servers; bundlers win when performance and compatibility at scale matter. Knowing why both exist — not just how to write an import — is what separates a mid from a senior answer.

<script type="importmap">
{
  "imports": {
    "lodash": "https://esm.sh/lodash-es",
    "@app/": "/src/modules/"
  }
}
</script>

<script type="module">
  import { debounce } from "lodash";    // bare specifier resolved by the map
  import { boot } from "@app/boot.js";  // prefix map — note the trailing slash
</script>

Explicit resource management (using/await using, Symbol.dispose)

Explicit resource management adds deterministic cleanup to JavaScript, the way try-with-resources works in Java or with works in Python. A using declaration binds a resource whose [Symbol.dispose]() method is called automatically when the enclosing block exits — normally or via an exception — so files, locks and streams close without a manual finally. Its async sibling, await using, looks up [Symbol.asyncDispose]() and awaits it. When several resources are declared in one scope, they are disposed in reverse declaration order, matching how nested acquisitions should unwind.

Why it matters: this is the newest topic in the module and still settling in — the TC39 proposal reached Stage 3 and it has shipped in V8 and Chromium and Firefox, but it is not yet Baseline across all browsers, and Safari and older runtimes lack it. The proposal also brings DisposableStack/AsyncDisposableStack for aggregating cleanups and SuppressedError for when a disposal error masks the original. Treat it as survey-level knowledge: understand what using guarantees, that a value must expose Symbol.dispose to qualify, and the reverse-order rule — enough to recognise it in modern code and speak to it if asked.

function firstLine(path) {
  using file = openSync(path);         // file[Symbol.dispose]() runs at block exit
  return file.readLine();
}                                      // disposed here — even if readLine throws

async function query(sql) {
  await using conn = await pool.acquire(); // needs [Symbol.asyncDispose]()
  return conn.run(sql);
}   // multiple resources dispose in reverse declaration order

Interview questions

Most-asked modules questions in Indian interviews, from screening rounds to product-company deep dives:

Beginner What is a module in JavaScript, and how do you turn a file into one?

A module is a single file with its own top-level scope — variables don't leak to the global object, and code runs in strict mode by default. In the browser you opt in with <script type="module">; in Node.js via an .mjs extension or "type": "module" in package.json. A module is evaluated only once and cached, so importing it many times shares one instance.

Beginner What is the difference between a named export and a default export?

A module can have many named exports (each bound to a specific identifier) and at most one default export. Named imports must match the exported name in braces: import { area } from "./m.js" (rename with as). The default is imported with no braces and any name: import Anything from "./m.js". Named exports aid tree-shaking and discoverability; defaults suit a module's single headline value.

Beginner How do ES modules differ from CommonJS (require / module.exports)?

ESM uses static import/export resolved before execution, is asynchronous, gives live read-only bindings, and enables tree-shaking. CommonJS uses require()/module.exports, is synchronous and dynamic, and returns a copied snapshot of the exports object at require time. ESM is the standard everywhere including modern Node.js; CommonJS remains widespread in older Node code, and mixing the two needs care.

Beginner Are import statements hoisted, and how many times does a module run?

Static imports are hoisted — the engine resolves and evaluates dependencies before the importing module's body runs, so an imported binding is available throughout the file regardless of where the import line sits. A given module is evaluated exactly once per realm; every later import returns the same cached module instance (modules are effectively singletons).

Intermediate When would you use dynamic import() instead of a static import?

import() returns a promise for the module namespace and can appear anywhere, so you use it for code-splitting (load a heavy chunk only when needed), conditional or on-demand loading (e.g. a modal opened on click), computed specifiers, and loading modules from a classic non-module script. The trade-off is that it is asynchronous and not statically analysable, so it can't be tree-shaken as precisely as a static import.

Intermediate What are "live bindings"? If an exported variable changes, does the importer see it?

Yes. An import is a live, read-only view of the exporter's binding, not a copied value. If the module mutates an exported let after export, every importer observes the new value on next access. The importer cannot reassign it (that throws), which is why exported counters or state update as expected while staying immutable from the consumer side — a key contrast with CommonJS's snapshot semantics.

Intermediate What is import.meta, and what replaces __dirname in ESM?

import.meta is an object holding metadata about the current module. Its most-used property, import.meta.url, is the module's own URL — the ESM replacement for CommonJS's __dirname/__filename. Combine it with the URL constructor (new URL("./data.txt", import.meta.url)) to resolve sibling files, and import.meta.resolve() turns a specifier into a URL.

Intermediate What are import maps, and what problem do they solve?

An import map is a <script type="importmap"> JSON block that tells the browser how to resolve bare specifiers (like "lodash") and path prefixes to actual URLs. It lets native ESM use clean, bundler-style names without a build step, supports scoped overrides for different resolutions per path, and centralises version/URL control. It is now supported across all modern browser engines.

Intermediate What are import attributes, and how do you import JSON with them?

Import attributes are the with clause that tells the runtime how to interpret a non-JavaScript module, guarding against loading the wrong content type. To import JSON: import data from "./data.json" with { type: "json" }, and the dynamic form takes an options object import("./data.json", { with: { type: "json" } }). They replaced the earlier non-standard assert syntax.

Intermediate If browsers support ESM natively, why do we still use bundlers?

Native ESM is great for dev and simple sites, but production wants more: bundlers avoid a network waterfall of many tiny module requests, tree-shake unused exports, transpile modern syntax for older targets, minify and hash filenames for caching, resolve bare specifiers without shipping an import map, and fold in CSS/assets. Tools like Vite serve unbundled ESM in dev for speed but bundle (via Rollup/esbuild) for the production build.

Pro Explain explicit resource management: using / await using / Symbol.dispose, and the disposal order.

A using declaration binds a resource that exposes [Symbol.dispose](); that method runs automatically when the block exits, normally or on error, giving deterministic cleanup without a manual finally. await using is the async variant that awaits [Symbol.asyncDispose](). Multiple resources in a scope dispose in reverse declaration order. Supporting pieces include DisposableStack/AsyncDisposableStack and SuppressedError. It is a Stage 3 proposal, shipped in V8/Chromium and Firefox but not yet Baseline everywhere.

Pro How do ES modules handle circular dependencies?

Loading has two phases: the module graph is instantiated/linked (all import and export bindings wired up) before any code runs, then modules evaluate depth-first. Because bindings are live, a cycle can work — if by the time you actually read a binding the other module has evaluated it. If you access an imported binding during evaluation before the source has initialised it, you hit the temporal dead zone and it throws (function declarations are hoisted, so function references across a cycle usually resolve). The fix is to defer cross-module access until after both modules have finished evaluating.

Pro Why must ESM import/export be statically analysable, and how does that enable tree-shaking — where does it break down?

Static import/export must sit at the top level with string-literal specifiers, so a tool can build the full dependency graph and the exact set of used bindings without executing code. That lets a bundler mark unreferenced exports as dead and drop them (tree-shaking). It breaks down when analysis can't prove safety: dynamic import() with computed specifiers, namespace access like ns[key], re-exports with side effects, or modules not flagged side-effect-free ("sideEffects": false in package.json). CommonJS's dynamic require resists it entirely — a core reason ESM was designed to be static.
