Module: Async — async/await and Cancellation

Promises made asynchronous code composable; async/await made it readable. This module orients you on writing await-based flows, cancelling work you no longer need, and choosing the right scheduler so the main thread stays responsive.

Knowledge Library → Courses → JavaScript → Async: async/await and Cancellation

What this module covers

This is the second half of JavaScript's asynchrony story. It builds on promises and the event loop (previous module) to cover three practical skills: writing sequential-looking asynchronous code with async/await — including top-level await and try/catch error handling; cancelling in-flight work with AbortController and AbortSignal so a stale request or a slow operation doesn't waste effort; and picking the correct scheduling primitive — timers, queueMicrotask, requestAnimationFrame, or requestIdleCallback — for the job. Together these are among the most heavily tested asynchronous topics in front-end interviews.

Topics

async/await patterns, top-level await, error handling

An async function always returns a promise: a plain return value becomes a fulfilled promise and a thrown error becomes a rejected one. Inside it, await pauses that one function until the awaited promise settles, then resumes with the fulfilment value — or, if the promise rejected, throws that reason at the await point. This lets you write asynchronous steps as ordinary top-to-bottom code and catch failures with a familiar try/catch/finally block, instead of threading .then()/.catch() callbacks by hand. Crucially, awaiting does not block the thread; it yields control back to the event loop and the function continues later as a microtask.

Why it matters: the two classic mistakes here are structural, not syntactic. First, await inside a loop runs each iteration serially — three requests take the sum of their times, not the maximum — so independent work should be started together and combined with Promise.all. Second, forgetting await (or returning a promise without awaiting inside a try) means errors escape your catch and surface as unhandled rejections. Top-level await, usable only at the top level of an ES module, removes the old IIFE wrapper but delays that module's execution — and everything that imports it — until the awaited value resolves, so use it deliberately. Interviewers in India routinely ask candidates to convert a promise chain to async/await, explain error handling, and spot accidental serialization.

async function loadDashboard(userId) {
  try {
    // independent -> run concurrently, not one-after-another
    const [profile, orders] = await Promise.all([
      fetchProfile(userId),
      fetchOrders(userId),
    ]);
    return { profile, orders };
  } catch (err) {
    // any rejection above is thrown right here
    console.error("dashboard failed:", err);
    throw err;               // re-throw so the caller's promise rejects too
  }
}

AbortController, cancellation

A promise, once created, cannot be cancelled — it will settle regardless of whether anyone still cares. AbortController is the platform's standard cancellation mechanism built around that limitation. You create a controller, hand its controller.signal (an AbortSignal) to an operation that understands it — fetch, addEventListener, streams, and a growing list of APIs — and later call controller.abort(reason). The signal then flips signal.aborted to true, fires an abort event, and causes the awaited operation to reject: for fetch, with a DOMException named "AbortError" by default, or with your custom reason if you passed one.

Why it matters: cancellation is what prevents wasted work and stale-response bugs. In a search-as-you-type box, aborting the previous request before firing a new one guarantees a slow older response can't overwrite a newer one. Because the reject looks like any other error, real code must distinguish a deliberate abort from a genuine failure — typically by checking err.name === "AbortError" — and treat the abort as a no-op rather than showing an error to the user. Two ergonomic helpers appear often in interviews and reviews: AbortSignal.timeout(ms) returns a signal that auto-aborts after a delay (rejecting with a TimeoutError), and AbortSignal.any([...signals]) aborts as soon as any input signal does — the idiomatic way to combine a user-initiated cancel with a timeout.

const controller = new AbortController();
// cancel from anywhere:  controller.abort()

try {
  const res = await fetch("/api/search?q=js", { signal: controller.signal });
  const data = await res.json();
  render(data);
} catch (err) {
  if (err.name === "AbortError") return;   // expected: we cancelled, do nothing
  throw err;                                // a real network/parse failure
}

Timers, queueMicrotask, requestAnimationFrame, requestIdleCallback

Not all "run this later" is the same, and the queue you choose decides when the callback runs relative to rendering and to other work. setTimeout/setInterval schedule a task (macrotask): it runs in a future event-loop turn, after the current call stack and the entire microtask queue have drained, and even setTimeout(fn, 0) is subject to a minimum clamp (about 4ms once timers are nested). queueMicrotask schedules a microtask — the same queue promise reactions use — which runs right after the current synchronous code finishes and before the browser gets a chance to render or run the next task. Because the whole microtask queue is drained before each task, microtasks let you defer work while still keeping it in the current turn.

Why it matters: requestAnimationFrame registers a callback to run just before the next repaint, synchronised to the display's refresh rate and paused in background tabs — making it the correct tool for animation, where setTimeout drifts and can tear. requestIdleCallback schedules low-priority background work during the browser's idle periods, handing your callback an IdleDeadline so you can yield before you cause jank (note it is not yet Baseline across all browsers, so guard or provide a fallback). The single most common interview exercise on this topic is predicting the output order of synchronous code, a setTimeout(…, 0), and a promise/microtask — a direct test of whether you understand that microtasks always run ahead of the next timer.

console.log("1 sync");
setTimeout(() => console.log("4 timeout (macrotask)"), 0);
Promise.resolve().then(() => console.log("3 promise (microtask)"));
queueMicrotask(() => console.log("3b microtask"));
console.log("2 sync");
// order: 1 sync, 2 sync, 3 promise, 3b microtask, 4 timeout

Interview questions

Most-asked async/await and cancellation questions in Indian interviews, from screening rounds to product-company deep dives:

Beginner What does an async function return?

Always a promise. A returned value becomes the fulfilment value of that promise; a thrown error becomes its rejection reason. Even return 5 yields a promise that resolves to 5, so callers still use await or .then().

Beginner What exactly does await do?

It pauses the current async function until the awaited promise settles, then evaluates to its fulfilment value — or throws its rejection reason at that line. It does not block the thread: control returns to the event loop and the rest of the function resumes later as a microtask.

Beginner How do you handle errors with async/await?

Wrap awaits in try/catch (with optional finally for cleanup). Since an async function returns a promise, you can also handle failures at the call site with .catch(). A rejected await behaves exactly like a synchronous throw inside the function.

Beginner What is top-level await and where can you use it?

The ability to use await at the top level of a module, without an enclosing async function. It works only in ES modules (not classic scripts or CommonJS). It delays that module's evaluation — and any module importing it — until the awaited promise resolves, so it should be used sparingly for genuine initialization.

Beginner How do you cancel a fetch request?

Create an AbortController, pass controller.signal to fetch(url, { signal }), and call controller.abort() when you want to cancel. The pending fetch rejects with a DOMException named "AbortError", which you check for and treat as an expected cancellation rather than an error.

Intermediate await in a loop is running requests one by one. How do you parallelize them?

await in a for loop is serial — each iteration waits for the previous. Start the independent operations first and await them together: await Promise.all(ids.map(id => fetchOne(id))). Use Promise.allSettled if you need every result regardless of individual failures.

Intermediate Difference between setTimeout(fn, 0) and queueMicrotask(fn)?

setTimeout queues a task (macrotask) that runs in a later event-loop turn, after the microtask queue drains and subject to a minimum clamp. queueMicrotask queues a microtask that runs right after the current synchronous code, before the next render or task. So the microtask always runs first.

Intermediate In a catch block, how do you tell a deliberate abort from a real network error?

Check the error: err.name === "AbortError" (a DOMException) means you cancelled it — usually a no-op. Anything else is a genuine failure to surface. If you aborted with a custom reason, that reason is thrown instead, so branch on it. You can also inspect signal.aborted.

Intermediate Why use requestAnimationFrame instead of setTimeout for animation?

requestAnimationFrame runs your callback just before the next repaint, synchronised to the display refresh rate, and is paused in background tabs to save battery. setTimeout is not aligned to the paint cycle, so it drifts, can drop or double frames, and wastes work when the tab is hidden. rAF also passes a high-resolution timestamp for frame-rate-independent motion.

Intermediate Does an await inside one function block other code or the UI?

No. await suspends only that async function and returns control to the event loop, so timers, events, and other tasks keep running. It is cooperative pausing, not thread blocking — very different from a synchronous busy-wait, which would freeze the page.

Pro What is the difference between AbortSignal.timeout() and AbortSignal.any(), and when would you combine them?

AbortSignal.timeout(ms) returns a signal that auto-aborts after a delay (rejecting with a TimeoutError). AbortSignal.any([...]) returns a signal that aborts as soon as any input signal does. Combine them to enforce a timeout while still letting the user cancel: AbortSignal.any([userController.signal, AbortSignal.timeout(5000)]) — whichever fires first wins.

Pro Predict the output: a synchronous log, setTimeout(…, 0), a Promise.then, and a queueMicrotask.

All synchronous logs run first, in order. Then the event loop drains the entire microtask queue — the Promise.then and the queueMicrotask callbacks, in the order they were enqueued — before running any task. The setTimeout callback (a macrotask) runs last. The rule tested: microtasks always run to completion before the next task.

Pro A "cancelled" fetch's response still seems to arrive and update the UI. What went wrong and how do you fix it?

Almost always the signal was never wired up — the request was started without { signal }, or a stale controller was reused, or the AbortError was swallowed and the code fell through to render. Fix: create a fresh AbortController per request, abort the previous one before starting the next, pass its signal into fetch, and in catch return early on AbortError so aborted responses never touch the UI.

Pro Why can top-level await be dangerous, and what is the safer pattern for expensive startup work?

Top-level await blocks the evaluation of the module and its entire import chain until the promise resolves, so a slow dependency can stall the whole graph and delay first render. It also serializes sibling imports that could have loaded in parallel. Safer: export an async initializer or a promise the app awaits where it actually needs the value, keeping module loading fast and the dependency explicit.
