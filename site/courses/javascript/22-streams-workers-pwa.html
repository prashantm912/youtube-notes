Module: Streams, Workers and PWA

Three power tools for the modern browser: streams process data as it arrives instead of after it all lands, workers move heavy work off the thread that draws your UI, and service workers turn a website into something installable and offline-capable. This module maps each API to the job it was built for.

Knowledge Library → Courses → JavaScript → Streams, Workers and PWA

What this module covers

These three areas share one goal — a responsive app that doesn't freeze or fail when the network does. The Streams API lets you consume and produce data chunk by chunk, with built-in backpressure and transforms like CompressionStream. Web Workers run JavaScript on a separate thread so long computations never block rendering, communicating by message-passing and, when speed matters, by transferring memory. Service workers — the engine behind Progressive Web Apps — sit between the page and the network as a programmable proxy, backing the Cache API, offline support, an installable manifest, and push. All three are recurring mid-to-senior interview themes.

Topics

Streams API (Readable/Writable/Transform, backpressure), CompressionStream

A stream models data as a sequence of chunks that flow over time rather than a single blob you must hold in memory all at once. A ReadableStream is a source you pull from (a fetch response body is one), a WritableStream is a sink you push into, and a TransformStream sits in the middle rewriting chunks — you wire them together with pipeThrough() and pipeTo(). Because chunks arrive incrementally, you can start rendering, parsing, or hashing a multi-megabyte response before the last byte lands, and you never buffer the whole thing.

Why it matters: the piece interviewers probe is backpressure — a fast producer must not overwhelm a slow consumer, so the stream signals "I'm full, slow down" through each queue's desired size, and a well-behaved pipe automatically pauses reading until the sink drains. Built on the same TransformStream shape, CompressionStream and DecompressionStream gzip or deflate data as it passes, letting you compress an upload or decompress a download with a single pipeThrough and no library.

// Consume a response chunk-by-chunk; the loop naturally applies backpressure
const res = await fetch("/big-file");
for await (const chunk of res.body) {
  process(chunk);            // Uint8Array pieces, handled as they arrive
}

// Gzip any readable stream by piping it through a transform
const gzipped = inputStream.pipeThrough(new CompressionStream("gzip"));

Web Workers (module workers, transfer), MessageChannel/BroadcastChannel

JavaScript on a page runs on a single main thread that also handles layout, paint, and input — so a heavy loop there freezes the whole UI. A Worker runs a script on a separate thread with its own global scope, no shared variables, and (crucially) no DOM access. The page and worker talk only through postMessage() and the message event; the payload is deep-copied by the structured clone algorithm. Passing { type: "module" } creates a module worker, which can use static import instead of the old importScripts().

Why it matters: cloning large buffers is expensive, so for big binary data you transfer instead — a transferable object such as an ArrayBuffer is moved (zero-copy) to the worker and detached from the sender, which then can't touch it. For richer wiring, MessageChannel gives you a private two-port pipe you can hand to any context (an iframe, another worker), while BroadcastChannel is a named bus that fans a message out to every same-origin tab, worker, and window at once — the simple way to keep multiple tabs in sync.

// main.js — spawn a module worker and hand off a buffer by transfer
const worker = new Worker("./worker.js", { type: "module" });
worker.postMessage({ buffer }, [buffer]);   // zero-copy: buffer is now detached
worker.onmessage = (e) => render(e.data);

// worker.js
self.onmessage = (e) => {
  const result = heavyWork(e.data.buffer);  // runs off the main thread
  self.postMessage(result);
};

Service Workers + PWA (lifecycle, Cache API, offline, manifest, push)

A service worker is a special worker the browser keeps registered against your origin, running independently of any page and acting as a programmable network proxy: it can intercept every fetch the site makes and decide whether to answer from the network, from a cache, or with a fallback. It has a deliberate lifecycle — register → install → activate — so you precache your app shell during install, clean up old caches during activate, and only then start handling fetch. A freshly deployed worker waits until every tab controlled by the old one is gone, which is exactly why users sometimes see stale content until they fully reload. Service workers require HTTPS (localhost excepted).

Why it matters: paired with the Cache API (a store of Request/Response pairs) this is what makes a site work offline and load instantly on repeat visits. Add a web app manifest — a JSON file naming the app, its icons and display mode — and the browser offers to install the site to the home screen as a Progressive Web App. The same worker also receives server push messages via the Push API and shows notifications while the app is closed. Interviewers use this topic to test whether you understand caching strategies and the update model, not just the API calls.

// page: register the worker
navigator.serviceWorker.register("/sw.js");

// sw.js: precache the shell on install, then serve cache-first on fetch
self.addEventListener("install", (e) =>
  e.waitUntil(caches.open("v1").then((c) => c.addAll(["/", "/app.js"]))));

self.addEventListener("fetch", (e) =>
  e.respondWith(caches.match(e.request).then((hit) => hit ?? fetch(e.request))));

Interview questions

Most-asked streams, workers and PWA questions in Indian interviews, from screening rounds to product-company deep dives:

Beginner Why use a stream instead of reading a whole response with response.json()?

A stream delivers data in chunks as it arrives, so you can start processing (parse, render, hash, save) before the whole payload downloads, and you never hold the entire body in memory. response.json() or text() waits for the complete response and buffers all of it — fine for small payloads, wasteful or impossible for large or endless ones.

Beginner What are the three kinds of streams?

ReadableStream — a source you read chunks from (e.g. a fetch response.body). WritableStream — a sink you write chunks into. TransformStream — a readable/writable pair in the middle that rewrites chunks as they pass. You connect them with pipeThrough() (through a transform) and pipeTo() (into a sink).

Beginner What is a Web Worker and what can't it do?

A worker runs a script on a background thread with its own global scope, so heavy work doesn't block the UI. It cannot touch the DOM, window, or document, and shares no variables with the page — it communicates only by messages. It still has plenty of APIs: fetch, timers, WebSocket, IndexedDB, and more.

Beginner How do a page and a worker communicate?

Through postMessage() to send and the message event (onmessage) to receive, in both directions. The data is deep-copied by the structured clone algorithm — functions and DOM nodes can't be sent, but objects, arrays, Maps, Sets, ArrayBuffers, and more can.

Beginner What is a service worker in one sentence, and what's the HTTPS rule?

A service worker is a background script the browser registers for your origin that acts as a programmable network proxy — intercepting requests to enable caching, offline support, and push. It requires a secure context (HTTPS), with localhost allowed for development, because a script that can rewrite every network response is too powerful to run over plain HTTP.

Intermediate What is backpressure and how do streams handle it?

Backpressure is the mechanism that stops a fast producer from overwhelming a slow consumer. Each stream has an internal queue with a high-water mark; the difference between it and the queued size is the desired size. When a sink falls behind, desired size drops to zero (or negative) and the pipe automatically pauses reading from the source until the sink drains — so memory stays bounded without you writing any throttling code.

Intermediate Copying vs transferring data to a worker — what is a transferable object?

By default postMessage structured-clones (copies) the data, which is costly for large buffers. A transferable object — such as an ArrayBuffer, MessagePort, ImageBitmap, or OffscreenCanvas — can instead be handed over by reference (zero-copy) by listing it in the second argument: postMessage(msg, [buffer]). After transfer the sender's copy is detached and unusable, which is the point: only one side owns the memory at a time.

Intermediate What is a module worker and how does it differ from a classic worker?

Creating a worker with new Worker(url, { type: "module" }) makes its script an ES module, so it can use static import/export and top-level await. A classic worker (the default) has no ES-module semantics and must pull dependencies with the synchronous importScripts(). Module workers are the modern default for anything with dependencies.

Intermediate MessageChannel vs BroadcastChannel — when do you use each?

MessageChannel creates a private point-to-point pipe with two MessagePorts; you transfer one port to another context (iframe, worker) to set up direct one-to-one messaging. BroadcastChannel is a named one-to-many bus: any same-origin tab, worker, or window that opens the same channel name receives every message posted to it (except its own sender) — ideal for syncing state like "user logged out" across all open tabs.

Intermediate Explain the service worker lifecycle and why a new version "waits".

Register → install → activate. On install you precache the app shell; on activate you delete old caches; only an activated worker controls pages and handles fetch. When you deploy a new worker it installs but then enters a waiting state until every tab controlled by the old worker is closed, so two versions never run at once. That wait is why users can see stale content until a full reload — self.skipWaiting() plus clients.claim() can activate immediately when you intend it.

Intermediate What makes a web app installable as a PWA?

Three ingredients: served over HTTPS, a registered service worker (so it can work offline), and a valid web app manifest declaring at least a name, a start URL, a display mode like standalone, and icons of the required sizes. When these are met, the browser offers an install prompt to add the app to the home screen or app launcher, where it runs in its own window without browser chrome.

Pro Compare cache-first, network-first, and stale-while-revalidate. What's the trade-off?

Cache-first: serve from cache, hit the network only on a miss — fastest and offline-friendly, but risks serving stale assets, so it suits fingerprinted/versioned static files. Network-first: try the network, fall back to cache — freshest data, but slow or failing networks add latency; good for API responses that must be current. Stale-while-revalidate: serve the cached copy immediately, then fetch in the background to update the cache for next time — fast and eventually fresh, at the cost of showing slightly old data once. The trade-off is always freshness vs. speed/offline resilience; real apps mix strategies per route.

Pro After a deploy, users still see the old page until a hard refresh. Why, and how do you fix it?

The active service worker is serving your shell cache-first, and the new worker is stuck in waiting because old tabs are still open — so both the cached HTML and the old worker persist. Fixes: version your cache names and delete old ones in activate; call self.skipWaiting() in install and clients.claim() in activate to take control at once (ideally after prompting the user, so you don't swap assets mid-session); and don't cache-first your HTML — use network-first or stale-while-revalidate for navigations so a new build is picked up.

Pro Walk through the web push notification flow end to end.

The page asks permission (Notification.requestPermission()) and subscribes via registration.pushManager.subscribe() with the server's public VAPID key, producing a PushSubscription (an endpoint URL plus keys) that you send to your backend. To notify, your server signs a message and POSTs it to that endpoint at the browser's push service; the browser wakes the service worker with a push event even if no tab is open, and the worker calls registration.showNotification() inside event.waitUntil(). A notificationclick handler then focuses or opens the app. Keys: permission is user-granted and revocable, delivery goes through the browser's push service (not your server directly), and the whole flow needs a service worker and HTTPS.
