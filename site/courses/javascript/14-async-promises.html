Module: Async — the Event Loop and Promises

JavaScript runs your code on a single thread and still never blocks. The trick is an event loop feeding two queues, plus the promise object that rides the microtask queue — get their ordering right and the rest of async JavaScript falls into place.

What this module covers

This is the mental-model layer under everything asynchronous: the call stack and the event loop, the task and microtask queues that decide execution order, the callback pattern that promises replaced, and the Promise object itself — states, chaining, error flow, and the all/allSettled/any/race/withResolvers/try combinators. It is one of the most heavily screened areas in JavaScript interviews, output-order questions above all.

Topics

The event loop: call stack, task and microtask queues

A JavaScript engine has one call stack and runs one thing at a time to completion — it never pauses a function midway to run another. Anything slow (timers, network, user events) is handed to the host environment (browser Web APIs, Node's libuv); when the result is ready the host places a callback on a queue. The event loop is the coordinator: each time the stack empties it takes the next queued job and runs it. There are two priorities — the macrotask (task) queue holds timer, I/O and UI-event callbacks, while the microtask queue holds promise reactions, queueMicrotask and MutationObserver.

Why it matters: after the current script and after every single macrotask, the engine drains the entire microtask queue — including microtasks added while draining — before it renders or picks up the next macrotask. That one rule is why a Promise.then always fires before a setTimeout(fn, 0) queued at the same moment, and predicting this interleaving is a classic interview exercise.

console.log("A");
setTimeout(() => console.log("B"), 0);          // macrotask: runs later
Promise.resolve().then(() => console.log("C")); // microtask: runs first
console.log("D");
// prints:  A  D  C  B

Callbacks and callback hell

Before promises, an async result was delivered by passing a function to be called later — the callback. Node standardised the error-first shape callback(err, data). It works, but chaining dependent async steps nests each callback inside the previous one, marching the code rightward into the "pyramid of doom" (callback hell), with error handling repeated at every level.

Why it matters: the real problems run deeper than looks. There is no unified error channel — a value thrown inside a callback does not reach the surrounding try/catch because that stack frame is long gone — and there is inversion of control: you hand your continuation to someone else's function and simply trust it to call you back exactly once. Promises were designed to fix both, and interviewers want those two reasons named, not just "it looks messy".

getUser(id, (err, user) => {
  if (err) return handle(err);
  getOrders(user, (err, orders) => {
    if (err) return handle(err);
    getInvoice(orders[0], (err, invoice) => {  // marching right...
      if (err) return handle(err);
      render(invoice);
    });
  });
});

Promises: states, chaining, error flow and combinators

A Promise is an object standing in for a value that may arrive later. It has three states — pending, then either fulfilled (with a value) or rejected (with a reason) — and once it settles it is frozen; later resolve/reject calls are ignored. .then registers a fulfillment reaction, .catch a rejection reaction, .finally a settlement callback, and each returns a new promise so calls chain flatly instead of nesting. Returning a value from a handler fulfils the next promise; throwing rejects it — giving one error channel that a single trailing .catch can service.

Why it matters: real code coordinates several promises at once. Promise.all fulfils with an array when every input fulfils but rejects fast on the first failure; Promise.allSettled never short-circuits and reports every outcome; Promise.race settles with the first to settle (fulfil or reject); Promise.any takes the first fulfilment and only rejects (with an AggregateError) if all reject. Two newer statics round it out: Promise.withResolvers hands you the resolve/reject functions to settle from outside, and Promise.try runs a possibly-synchronous function so even a synchronous throw lands in the chain. Chaining, error flow, and the combinators are staple interview ground.

fetchUser(id)
  .then(user => fetchOrders(user.id))  // return a promise: chain stays flat
  .then(orders => render(orders))
  .catch(err => showError(err))         // one channel for every step above
  .finally(() => hideSpinner());

Promise.all([fetchA(), fetchB()])       // run concurrently, wait for both
  .then(([a, b]) => combine(a, b));

Interview questions

The event loop and promises are two of the most-asked JavaScript topics in Indian interviews — from service-company screens to product-company rounds — and the output-order puzzles below appear constantly. Hardest last.

Beginner Is JavaScript single-threaded? How does it do async work then?

Single-threaded — one call stack, one thing at a time. Async work is offloaded to the host environment (browser Web APIs, Node's libuv); when it finishes, a callback is queued and the event loop runs it once the stack is empty. The language isn't running things in parallel — the environment does the waiting and hands work back.

Beginner What is the event loop, in one sentence?

A loop that, whenever the call stack is empty, takes the next queued job and pushes it onto the stack to run — repeated forever. That is the mechanism by which queued callbacks eventually execute.

Beginner What are the three states of a Promise? Can a settled promise change?

pending, fulfilled, rejected. It begins pending and transitions once — to fulfilled (with a value) or rejected (with a reason). Settled is final and immutable: any later resolve/reject is ignored, and handlers still receive the original outcome.

Beginner What is "callback hell", and what actually causes it?

Deeply nested callbacks for dependent async steps — the pyramid of doom. Cause: each step needs the previous step's result, so you nest to keep it in scope, and error handling is duplicated at every level. Promises flatten the nesting; async/await removes it.

Intermediate Microtasks vs macrotasks — which runs first, with examples of each?

After the current synchronous run (and after each macrotask), the engine empties the entire microtask queue before the next macrotask or a render. Microtasks: promise .then/.catch/.finally reactions, queueMicrotask, MutationObserver. Macrotasks: setTimeout/setInterval, I/O, UI events, setImmediate (Node). So a promise callback always beats a setTimeout(fn, 0) queued alongside it.

Intermediate Predict the output: a log of "A", a setTimeout(…,0) logging "B", a Promise.resolve().then logging "C", then a log of "D". 
console.log("A");
setTimeout(() => console.log("B"), 0);
Promise.resolve().then(() => console.log("C"));
console.log("D");

A, D, C, B. The synchronous lines run first (A, D). The stack empties, so the microtask queue drains next (C). Only then does the timer macrotask run (B). setTimeout(…, 0) never means "now" — it means "after the current task and all its microtasks".

Intermediate Predict the output: a setTimeout(…,0) logging "timeout" and a two-step chain Promise.resolve().then(p1).then(p2), with "start" logged first and "end" logged last synchronously. 
console.log("start");
setTimeout(() => console.log("timeout"), 0);
Promise.resolve()
  .then(() => console.log("p1"))
  .then(() => console.log("p2"));
console.log("end");

start, end, p1, p2, timeout. Sync first (start, end). Then microtasks drain: running p1's reaction enqueues p2's reaction, and the loop keeps draining newly-added microtasks to completion — so both p1 and p2 run before the timer macrotask fires.

Intermediate Promise.all vs allSettled vs race vs any?

all — fulfils with a result array when every input fulfils; rejects immediately on the first rejection (fail-fast). allSettled — never short-circuits; fulfils with a {status, value|reason} array once all settle. race — settles as soon as the first input settles, fulfil or reject. any — fulfils with the first fulfilment; rejects with an AggregateError only if every input rejects.

Intermediate How does error handling flow through a promise chain?

A throw or rejection anywhere skips the following .then fulfillment handlers and jumps to the next rejection handler / .catch downstream — so one .catch at the end covers every step above it. If a rejection handler returns normally, the chain resumes on the fulfilled path. .finally runs on either outcome and passes the settlement straight through.

Intermediate What do promises give you that plain callbacks don't?

One error channel (.catch) instead of per-callback err checks; flat chaining instead of nesting; a guaranteed single settlement (a promise can't fulfil twice, unlike a buggy API that calls a callback zero or many times); and they return control to you — you compose the flow rather than surrendering it (fixing inversion of control). Plus combinators like all/race for coordinating many at once.

Pro Predict the output: sync logs 1 and 6, a timer logging 2, a microtask that logs 3 and schedules a timer logging 4, and a second microtask logging 5. 
console.log(1);
setTimeout(() => console.log(2), 0);
Promise.resolve().then(() => {
  console.log(3);
  setTimeout(() => console.log(4), 0);
});
Promise.resolve().then(() => console.log(5));
console.log(6);

1, 6, 3, 5, 2, 4. Sync: 1, 6. Microtasks in order: 3 (which schedules a new timer for 4, appended behind the timer for 2), then 5. The microtask queue is now empty, so macrotasks run FIFO: 2, then 4. The key insight is that the timer created inside the microtask queues behind the timer that was already waiting.

Pro What is Promise.withResolvers and when is it useful?

An ES2024 static that returns { promise, resolve, reject }, letting you settle the promise from outside the executor. It's ideal when the settle trigger arrives elsewhere — an event listener, a socket message, a one-shot you resolve later — and it removes the old "declare let resolve outside, capture it inside new Promise" boilerplate.

Pro What is an unhandled / "floating" promise rejection, and why does it bite?

A rejected promise that nothing catches or awaits — the failure is silently lost. Browsers fire an unhandledrejection event; recent Node versions crash the process by default. "Floating" means you never chained or awaited the returned promise, so its errors and ordering vanish. Fix: always return/await your promises and terminate chains with .catch. Interviewers use this to check you don't treat promises as fire-and-forget.

Pro Why does Promise.resolve().then(cb) run cb asynchronously even though the promise is already resolved — and what does Promise.try add?

The spec guarantees .then callbacks are always delivered as microtasks, never synchronously — so ordering is consistent whether the promise was already settled or settles later, avoiding the "sometimes sync, sometimes async" (Zalgo) bug. Promise.try(fn) (ES2025) runs fn and funnels its result into a promise chain whether it returns a value, returns a promise, or throws synchronously — so a synchronous throw is caught by .catch instead of escaping the chain.
