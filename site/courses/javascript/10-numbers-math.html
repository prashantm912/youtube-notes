Module: Numbers, Math and BigInt

JavaScript has exactly one number type — a 64-bit float — and understanding what it can and cannot represent explains half the "weird" behavior developers hit. This module maps that terrain and links the resources that go deep.

What this module covers

Every JavaScript number is an IEEE 754 double, so this module starts with what that buys you and what it costs: the safe-integer ceiling, the 0.1 + 0.2 rounding reality, and the parse/convert functions. It then tours the Math object end to end — including the new Math.sumPrecise — and finishes with BigInt for exact integers beyond a double's reach.

Topics

Number statics/methods, floating point reality, parseInt/parseFloat

The Number type is a single 64-bit IEEE 754 double that stores every JavaScript number, integer or not. Its statics draw the boundaries — MAX_SAFE_INTEGER, MIN_SAFE_INTEGER, EPSILON, MAX_VALUE — and its guards (isInteger, isSafeInteger, isFinite, isNaN) test values without the coercion the old global functions apply. Instance methods like toFixed, toPrecision and toString(radix) format a number for output, while parseInt and parseFloat pull a number out of the front of a string, stopping at the first character they cannot use.

Why it matters: floating point is where interviews live. Because 0.1 and 0.2 cannot be written exactly in binary, 0.1 + 0.2 is 0.30000000000000004 and the equality check fails — the canonical trap. Knowing the safe-integer ceiling, why parseInt needs an explicit radix, and how Number() differs from parseInt separates a candidate who has been burned in production from one who memorized syntax.

0.1 + 0.2;                 // 0.30000000000000004
0.1 + 0.2 === 0.3;         // false
Number.MAX_SAFE_INTEGER;   // 9007199254740991  (2^53 - 1)
Number.isInteger(4.0);     // true
parseInt("08px", 10);      // 8  (radix removes ambiguity)
Number("08px");            // NaN (strict conversion)

Math object (complete), Math.sumPrecise

Math is a namespace object, not a constructor — every member is static, so there is no new Math(). It holds constants (PI, E, LN2, SQRT2) and roughly three dozen methods: rounding (round, floor, ceil, trunc), powers and roots (pow, sqrt, cbrt, hypot), logarithms and exponentials, the full trigonometric family, and utilities like max, min, sign, random, clz32 and the lower-precision helpers fround and f16round.

Why it matters: the newest addition, Math.sumPrecise, sums an iterable of numbers with a single final rounding instead of accumulating error term by term — so it returns the mathematically correct total where a plain loop or reduce silently loses low-order digits. Interviewers use Math to probe the difference between floor, trunc and round on negative numbers, how to generate a random integer in a range, and why Math methods have nothing to do with BigInt.

Math.trunc(-4.7);   // -4   (drops the fraction toward zero)
Math.floor(-4.7);   // -5   (rounds toward negative infinity)
Math.floor(Math.random() * 6) + 1;          // random 1..6
Math.sumPrecise([1e20, 0.1, -1e20]);        // 0.1 (a naive loop gives 0)

BigInt

BigInt is a distinct primitive for integers of arbitrary size, written with a trailing n (9007199254740993n) or via BigInt("..."). It supports the usual integer operators (add, subtract, multiply, exponentiation, remainder and bitwise) with exact results, reports a typeof of "bigint", and is falsy only at 0n. It deliberately does not interoperate with Number: mixing the two in arithmetic throws a TypeError, and both Math methods and JSON.stringify reject it.

Why it matters: a double can only hold integers exactly up to 2^53 - 1; past that, database IDs, large counters and high-resolution timestamps start colliding. Interview-wise, expect questions on when to reach for BigInt versus a decimal library, why 10n === 10 is false while 10n == 10 is true, and how you serialize a BigInt across JSON given it has no representation there.

const big = 9007199254740991n + 2n;  // 9007199254740993n  (exact)
typeof 10n;    // "bigint"
10n === 10;    // false  (different types)
10n == 10;     // true   (loose equality coerces)
// 10n + 5     -> TypeError: cannot mix BigInt and other types

Interview questions

Most-asked number questions in Indian interviews — from screening rounds to product-company deep dives:

Beginner Why is 0.1 + 0.2 not exactly 0.3 in JavaScript?

JavaScript numbers are IEEE 754 double-precision (64-bit) binary floats. Fractions like 0.1 and 0.2 have no exact finite binary form, so each is stored slightly rounded; the errors add up and the sum prints as 0.30000000000000004. It is not a JavaScript bug — Python, Java and most languages behave the same way.

Beginner What single numeric type does JavaScript use, and what is the largest safe integer?

One type, Number, a 64-bit double — there is no separate int and float. Integers stay exact only up to Number.MAX_SAFE_INTEGER (2^53 - 1 = 9007199254740991); beyond it values collide, so 9007199254740992 + 1 === 9007199254740992. BigInt exists for larger exact integers.

Beginner parseInt vs Number() vs parseFloat?

Number(x) converts the whole string and returns NaN if any part is non-numeric (Number("12px") is NaN). parseInt reads an optional sign and leading integer digits, stopping at the first invalid character (parseInt("12px") is 12), honoring a radix. parseFloat is like parseInt but keeps a decimal part. Use Number for strict conversion, the parse functions for lenient prefix parsing.

Beginner Why should you always pass the radix to parseInt?

The second argument is the base (2 to 36). Without it, parsing depends on the string — a leading "0x" is read as hexadecimal. Passing 10 explicitly (parseInt(str, 10)) removes that ambiguity and is a common lint rule. Historically a leading "0" once meant octal, so being explicit is the safe habit.

Beginner What is NaN and how do you test for it?

NaN ("not a number") is the numeric result of an invalid operation like 0/0 or Number("abc"). It is the only value not equal to itself, so NaN === NaN is false. Test with Number.isNaN(x) (no coercion) or Object.is(x, NaN); avoid the global isNaN, which coerces its argument first.

Intermediate How do you compare floating-point numbers safely?

Do not use ===. Compare within a small tolerance: Math.abs(a - b) < Number.EPSILON, or a domain-appropriate epsilon for larger magnitudes. EPSILON is the gap between 1 and the next representable double. For money, avoid floats entirely — work in integer minor units (paise or cents) or a decimal library.

Intermediate Number.isNaN / isFinite / isInteger vs the global isNaN / isFinite?

The global functions coerce first: isNaN("foo") is true because the string becomes NaN. The Number.* versions do not coerce — Number.isNaN("foo") is false because the string is not the NaN value. Prefer the Number.* forms for predictable checks; Number.isInteger and Number.isSafeInteger have no global equivalents.

Intermediate What is BigInt and why can't you mix it with Number?

BigInt is a primitive for arbitrary-precision integers, written with an n suffix (123n) or BigInt(123). Mixing it with Number in arithmetic throws a TypeError because there is no lossless common type — implicit conversion could silently drop precision. Convert explicitly with BigInt(x) or Number(x) at the boundary. Use it for integer IDs, nanosecond timestamps, and math beyond 2^53.

Intermediate What are the pitfalls of toFixed?

toFixed returns a string, not a number, and rounds the underlying double — so (1.005).toFixed(2) yields "1.00", not "1.01", because 1.005 is actually stored just below 1.005. It also adds no thousands separators. For display use Intl.NumberFormat; for correctness, round on scaled integers rather than the float.

Intermediate Why do Math methods and JSON.stringify not work with BigInt?

Math methods are specified over Number (doubles) and throw on a BigInt. JSON has no BigInt type, so JSON.stringify(10n) throws a TypeError. Serialize BigInts yourself — commonly as strings via a replacer — and convert them back with a reviver when parsing.

Pro What is Math.sumPrecise and what problem does it solve?

Math.sumPrecise(iterable) sums numbers as if with unbounded precision, then rounds once to the nearest double — so [1e20, 0.1, -1e20] gives 0.1, where a naive loop or reduce gives 0 because the 0.1 is lost against 1e20. It removes order-dependent accumulation error. It is recent (Baseline 2026), so feature-detect or polyfill for older runtimes.

Pro How would you handle currency correctly in JavaScript?

Never store money as a float. Keep amounts as integer minor units (paise or cents) or as BigInt, do all arithmetic on those integers, and only format for display with Intl.NumberFormat (for example currency "INR"). This avoids 0.1 + 0.2 style drift and the rounding surprises that show up in totals and tax lines.

Pro Explain +0 vs -0, and Object.is vs ===.

IEEE 754 has two zeros. +0 === -0 is true and they compare equal, but they differ observably: 1/+0 is Infinity while 1/-0 is -Infinity. Object.is(-0, +0) is false and Object.is(NaN, NaN) is true — Object.is is the same-value comparison, whereas === treats the two zeros as equal and treats NaN as unequal to itself.
