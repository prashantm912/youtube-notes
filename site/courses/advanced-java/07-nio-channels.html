Module: NIO Channels and Buffers

Classic streams block a whole thread per connection; NIO's buffers, channels, and selectors let one thread pump thousands of sockets — and the buffer state machine is exactly where most candidates stumble in interviews.

Knowledge Library → Courses → Advanced Java → NIO Channels and Buffers

What this module covers

Java NIO reworks I/O around three primitives: buffers hold the bytes, channels move them to and from files and sockets, and a selector lets a single thread watch many channels at once. This module orients you on the buffer state machine (position/limit/capacity and the flip/clear ritual), the channel families, non-blocking mode with selectors, and memory-mapped files — the foundation under Netty, Kafka, Tomcat's NIO connector, and nearly every high-throughput JVM server. Depth lives in the linked Javadocs and guides; this page maps the terrain and the interview traps.

Topics

Buffers: the ByteBuffer state machine

A ByteBuffer is a fixed-capacity block of memory with three cursors that make it a little state machine: capacity (never changes), position (the next index to read or write), and limit (the first index you may not touch). The invariant 0 <= mark <= position <= limit <= capacity holds at all times. You fill the buffer in "write mode", then call flip() to switch to "read mode" — it sets limit = position and position = 0 so you read back exactly what you wrote. clear() resets to write mode, rewind() re-reads from the start, and compact() keeps the unread bytes and makes room for more.

Why it matters: buffers back every channel operation, so forgetting to flip() (writing nothing, or reading garbage) is the classic NIO bug and a favourite interview probe. Also know the split between allocate() (heap buffer, backed by a Java byte[]) and allocateDirect() (off-heap, native memory) — direct buffers avoid a copy on native I/O but cost more to allocate, so you reserve them for large, long-lived buffers.

ByteBuffer buf = ByteBuffer.allocate(1024);  // write mode: position=0, limit=capacity
channel.read(buf);                            // bytes fill in; position advances
buf.flip();                                   // read mode: limit=position, position=0
while (buf.hasRemaining())
    System.out.print((char) buf.get());
buf.clear();                                  // back to write mode, ready to refill

Channels: FileChannel and socket channels

A channel is a two-way conduit to an I/O source — unlike a stream it is bidirectional, always reads into or writes from a ByteBuffer, and can be put into non-blocking mode. FileChannel connects to files (and gives you transferTo/transferFrom for kernel-level "zero-copy" transfers that skip a user-space buffer entirely — the JVM can map these onto the OS sendfile() call). SocketChannel and ServerSocketChannel are the TCP client and listener; DatagramChannel handles UDP. Scatter/gather reads and writes (an array of buffers in one call) let you split a header and body without concatenating.

Why it matters: transferTo is the standard answer to "how would you copy a large file efficiently in Java", and the socket channels are the building blocks of every selector-based server. Interviewers expect you to name the channel families and explain why a channel needs a buffer rather than reading a byte at a time.

// static imports: java.nio.file.StandardOpenOption.*
try (FileChannel in  = FileChannel.open(src, READ);
     FileChannel out = FileChannel.open(dst, CREATE, WRITE)) {
    in.transferTo(0, in.size(), out);  // OS may perform a zero-copy sendfile()
}

Selectors and the non-blocking I/O model

Blocking I/O dedicates one thread to each connection — fine for a few hundred clients, ruinous at tens of thousands (the C10k problem). A Selector flips the model: you set channels to non-blocking with configureBlocking(false), register() them with an interest set (OP_ACCEPT, OP_CONNECT, OP_READ, OP_WRITE), then loop on select(), which blocks until some registered channel is ready and hands you the ready keys. One thread now multiplexes every connection — the reactor pattern behind Netty and Nginx-style servers.

Why it matters: two details separate people who have used selectors from people who have only read about them. First, a SelectionKey carries an interest set (what you asked to be told about) that is distinct from its ready set (what the OS reports is ready now). Second, the selector adds keys to selectedKeys() but never removes them — you must call iterator.remove() after handling each key, or you will reprocess stale events forever. Both are staple interview questions.

Selector selector = Selector.open();
serverChannel.configureBlocking(false);
serverChannel.register(selector, SelectionKey.OP_ACCEPT);
while (true) {
    selector.select();                       // blocks until a channel is ready
    Iterator<SelectionKey> it = selector.selectedKeys().iterator();
    while (it.hasNext()) {
        SelectionKey key = it.next();
        it.remove();                         // required: the selector won't clear it
        if (key.isAcceptable())      { /* accept a new connection */ }
        else if (key.isReadable())   { /* read into a buffer      */ }
    }
}

Memory-mapped files

FileChannel.map() maps a region of a file straight into the process address space and hands back a MappedByteBuffer. Reads and writes then go through the operating system's page cache — you touch the file as if it were a giant array, with no explicit read()/write() syscalls per access, and the OS lazily pages data in and flushes it out. For large files read or updated repeatedly this is dramatically faster than stream I/O, which is why it underpins databases, message brokers (Kafka's log segments), and off-heap caches.

Why it matters: the trade-offs are the interview content. Mappings are READ_ONLY, READ_WRITE, or PRIVATE (copy-on-write); changes are only guaranteed on disk after force(); the mapping lives (and holds the file) until the buffer is garbage-collected, so unmap timing is non-deterministic; and a mapping is capped at Integer.MAX_VALUE bytes because the buffer is indexed by int. Because the memory is shared with the OS, a mapped file is also a channel for inter-process communication.

try (FileChannel ch = FileChannel.open(path, READ, WRITE)) {
    MappedByteBuffer mbb =
        ch.map(FileChannel.MapMode.READ_WRITE, 0, ch.size());
    mbb.putInt(0, 0x7FFFFFFF);  // writes into the page cache, not via read()/write()
    mbb.force();                // ask the OS to flush dirty pages back to disk
}

Interview questions

NIO is a favourite of product-company and senior backend rounds in India — buffers and selectors especially. The most-asked, roughly in order:

Beginner What are the three core abstractions of Java NIO?

Buffers (fixed-size containers that hold the data), channels (bidirectional conduits to files/sockets that read into and write from buffers), and selectors (let one thread monitor many channels for readiness). Streams are one-directional and blocking; NIO's trio is buffer-oriented and can be non-blocking.

Beginner Explain a buffer's position, limit, and capacity.

Capacity is the fixed size set at allocation. Position is the index of the next byte to read or write. Limit is the first index you may not touch. The invariant 0 <= mark <= position <= limit <= capacity always holds.

Beginner What does flip() do, and why is forgetting it a bug?

flip() switches a buffer from writing to reading: it sets limit = position then position = 0, so you read back exactly the bytes you wrote. Skip it and a channel write sees position already at the end (writes nothing), or a read starts mid-buffer over uninitialised bytes.

Beginner Difference between clear() and compact()?

clear() resets to write mode (position=0, limit=capacity) but does not erase data — it just forgets the old bounds, so old bytes get overwritten. compact() also returns to write mode but first copies the unread bytes (position to limit) to the front, preserving data you had not consumed yet. Use compact when a read left leftovers.

Beginner Direct vs heap (non-direct) ByteBuffer — when do you use direct?

allocate() gives a heap buffer backed by a Java byte[]; allocateDirect() gives an off-heap native buffer. Direct buffers let the JVM do native I/O without copying to an intermediate buffer, but cost more to allocate and free. Reserve them for large, long-lived buffers tied to native I/O; use heap buffers otherwise.

Intermediate How does a channel differ from a classic stream?

A channel is bidirectional (a stream is one-way), always transfers via a ByteBuffer rather than a byte at a time, can be switched to non-blocking mode, and can be registered with a selector. FileChannel also adds transferTo/transferFrom and memory mapping that streams have no equivalent for.

Intermediate What problem does a Selector solve?

It lets one thread manage many connections. Instead of a blocked thread per socket (thread-per-connection, which collapses at the C10k scale), you register non-blocking channels and call select(), which returns only the channels that are ready. This is the reactor pattern behind Netty, Tomcat's NIO connector, and similar servers.

Intermediate Interest set vs ready set on a SelectionKey?

The interest set is what you asked the selector to watch for on that channel (set at register time or via interestOps). The ready set (from readyOps(), or the isReadable()/isWritable()/isAcceptable()/isConnectable() helpers) is the subset the OS reports as actually ready after the last select(). You control the former; the selector fills in the latter.

Intermediate Why must you remove keys from selectedKeys()?

The selector adds ready keys to the selected-key set but never removes them. If you do not call iterator.remove() after handling each key, the same key stays in the set and you reprocess a stale event on the next loop — a very common NIO bug. The key set and cancelled-key set are managed differently; only the selected set is yours to clear.

Intermediate What does FileChannel.transferTo give you over a read/write loop?

A potential zero-copy transfer: the JVM can ask the OS to move bytes from the file's page cache directly to the target channel (e.g. via sendfile()), skipping the user-space buffer and the extra copies a manual read()-then-write() loop makes. It is the idiomatic way to copy files or stream a file to a socket efficiently.

Pro What is a MappedByteBuffer and what are its trade-offs?

It maps a file region into memory via FileChannel.map(), so access goes through the OS page cache with no per-access syscall — great for large, repeatedly accessed files. Trade-offs: durability needs force(); the mapping (and file handle) is released only when the buffer is GC'd, so unmap timing is non-deterministic; a single mapping is limited to Integer.MAX_VALUE bytes; and content can change under you if another process edits the file.

Pro Selector-based non-blocking vs thread-per-connection vs NIO.2 async vs virtual threads — when does each win?

Thread-per-connection (blocking streams) is simplest and fine for modest connection counts. Selector-based NIO scales one thread to many connections but the readiness event loop is complex to write correctly. NIO.2 AsynchronousSocketChannel uses completion callbacks/Futures, moving the loop into the runtime. Java 21 virtual threads (Loom) make blocking code cheap again — a virtual thread per connection now scales to huge counts with simple blocking style, so for most new servers virtual threads or a framework beat hand-rolled selectors; selectors remain where you need fine-grained readiness control or interop with existing reactor frameworks.
