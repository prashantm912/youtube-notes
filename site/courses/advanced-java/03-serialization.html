Module: Serialization

Turning a live object into bytes looks like a checkbox — implement one empty interface and you are done. The senior skill is knowing what that checkbox costs you in versioning, security, and why most teams reach for JSON instead.

Knowledge Library → Courses → Advanced Java → Serialization

What this module covers

Serialization is converting an object graph into a byte stream (and back) so it can be stored or sent across a network. This module covers Java's built-in mechanism — the Serializable marker, serialVersionUID, the object streams, custom hooks and Externalizable — then the deserialization vulnerability that makes native serialization dangerous, and finally the JSON approach with Jackson that production systems actually use for data exchange.

Topics

The Serializable contract and serialVersionUID

Implementing the empty Serializable marker interface makes a class eligible for the object streams: ObjectOutputStream.writeObject walks the whole reference graph and writes every non-transient, non-static field; ObjectInputStream.readObject rebuilds it — without calling any constructor. The transient keyword excludes a field (passwords, caches, derived data); static fields are never written because they belong to the class, not the instance.

Why it matters: the byte format is tied to the class shape, so evolving the class can break old data. serialVersionUID is the explicit version stamp callers check on read — omit it and the compiler generates one from the class structure, so any change (even adding a method) can flip it and throw InvalidClassException. Declaring it by hand is the single most-repeated serialization rule in interviews.

class Account implements Serializable {
    private static final long serialVersionUID = 1L;
    private String owner;
    private transient String sessionKey; // excluded from the byte stream
}
try (var out = new ObjectOutputStream(new FileOutputStream("a.ser"))) {
    out.writeObject(account);            // whole graph, no constructor on read
}

Customizing the format: writeObject/readObject and Externalizable

When default field-by-field writing is not enough, a class can supply private writeObject/readObject methods. They usually call defaultWriteObject()/defaultReadObject() for the standard fields, then read or write extra data — and readObject is the right place to re-validate invariants after a restore. Externalizable goes further: it hands you total control through writeExternal/readExternal, writes no default state, and requires a public no-arg constructor because the object is created the normal way before its state is read back.

Why it matters: Externalizable can be faster and more compact but is fully manual and easy to get wrong on class evolution. Knowing when each is appropriate — and how versioning of the serialized form works — is the difference between a screening answer and a design answer. Java 14+ also lets you tag these members with @Serial so the compiler checks their signatures.

private void writeObject(ObjectOutputStream out) throws IOException {
    out.defaultWriteObject();      // the ordinary fields
    out.writeInt(derivedChecksum()); // plus something extra
}
private void readObject(ObjectInputStream in)
        throws IOException, ClassNotFoundException {
    in.defaultReadObject();
    validateInvariants();          // re-check after reconstruction
}

Security: the deserialization vulnerability

This is the topic that makes native serialization controversial. Because readObject reconstructs arbitrary object graphs from bytes before your code sees them, a crafted stream can drive existing library classes (a "gadget chain") into executing code, exhausting memory with recursive or huge structures, or bypassing validation. It is CWE-502 — deserialization of untrusted data — and has produced remote-code-execution CVEs across the Java ecosystem. The core rule: never deserialize data you do not fully trust.

Why it matters: when you must accept serialized input, defend it. JEP 290 added serialization filters — ObjectInputFilter allow-lists (rather than reject-lists) that screen classes, array sizes, graph depth and stream length before objects are built. Mark sensitive fields transient, validate in readObject, and prefer a data format like JSON that never instantiates attacker-chosen types. Product companies in India increasingly probe this in security-conscious rounds.

// Allow only your own package and java.base; reject everything else
var filter = ObjectInputFilter.Config
    .createFilter("com.myapp.model.*;java.base/*;!*");
ois.setObjectInputFilter(filter);

The modern default: JSON with Jackson

For storing and exchanging data, most teams skip native serialization and use JSON. Jackson's ObjectMapper is the workhorse: writeValueAsString serializes a POJO to JSON and readValue deserializes it back, driven by getters/setters and annotations. Field-level control comes from annotations — @JsonProperty (rename), @JsonIgnore (drop), @JsonInclude (omit nulls/empties), @JsonFormat (dates) — and when a type needs bespoke output you register a custom serializer by extending StdSerializer (or the matching deserializer).

Why it matters: JSON is human-readable, language-agnostic, versions gracefully (unknown fields can be ignored), and does not instantiate attacker-chosen classes the way readObject does. This is the format behind virtually every REST API and Spring Boot service, so Jackson fluency is expected of any enterprise Java developer. Create one ObjectMapper and reuse it — it is thread-safe and relatively expensive to build.

ObjectMapper mapper = new ObjectMapper();       // build once, reuse
String json = mapper.writeValueAsString(user);  // POJO -> JSON
User back   = mapper.readValue(json, User.class); // JSON -> POJO

Interview questions

Serialization questions asked in Indian interviews — a screening staple for core Java, with security and Jackson coming up in product-company and backend rounds:

Beginner What are serialization and deserialization in Java?

Serialization converts an object's state into a byte stream (for storage or network transfer); deserialization rebuilds the object from those bytes. A class opts in by implementing the empty marker interface Serializable, and the work is done by ObjectOutputStream.writeObject and ObjectInputStream.readObject.

Beginner What is serialVersionUID and why declare it explicitly?

It is a version stamp compared during deserialization to confirm the loaded class matches the serialized data. If you do not declare one, the compiler derives it from the class structure, so almost any change alters it and old bytes fail with InvalidClassException. Declaring a fixed private static final long serialVersionUID gives you control over compatibility.

Beginner What does the transient keyword do?

It marks a field to be skipped by serialization — nothing is written for it, and on read it comes back as the type default (null, 0, false). Use it for secrets, caches, and derived values you can recompute. Static fields are also not serialized because they belong to the class, not the instance.

Beginner Serializable vs Externalizable?

Serializable is a marker; the runtime writes fields automatically. Externalizable extends it but you write and read every byte yourself in writeExternal/readExternal, and it requires a public no-arg constructor because the object is constructed first and populated after. Externalizable can be leaner and faster but is fully manual.

Intermediate Does deserialization call the constructor?

Not for the serializable class itself — the object is allocated and fields are restored directly from the stream, bypassing constructors and instance initializers. The exception: fields inherited from a non-serializable superclass are re-initialized by that superclass's accessible no-arg constructor, so a non-serializable parent must provide one.

Intermediate How do writeObject/readObject customize the process?

A serializable class may define private writeObject(ObjectOutputStream) and readObject(ObjectInputStream). They typically call defaultWriteObject()/defaultReadObject() for standard fields and then add custom data; readObject is also where you re-validate invariants after reconstruction, since no constructor ran to enforce them.

Intermediate How are object graphs with shared references or cycles handled?

The stream tracks each object with a handle; repeated references to the same object write a back-reference to that handle instead of duplicating it. This preserves shared identity on read and prevents infinite loops on cyclic graphs. It also means a mutable object serialized twice can be read back as two separate objects if written to different streams.

Intermediate Why is Java's native serialization considered a security risk?

Because readObject builds arbitrary object graphs from untrusted bytes before your code runs. Attackers craft "gadget chains" of library classes that execute code during deserialization, or send recursive/oversized structures to exhaust memory (denial of service). This is CWE-502 and has caused many remote-code-execution CVEs — so untrusted data must never be deserialized directly.

Intermediate Why do most teams prefer Jackson/JSON over native serialization?

JSON is human-readable, language-agnostic, and versions gracefully (unknown fields can be ignored rather than throwing). It never instantiates attacker-chosen types the way readObject does, and it is the format behind REST and Spring Boot. Jackson's ObjectMapper maps POJOs to and from JSON with annotation-driven control.

Pro How do you defend a system that must accept serialized input?

First choice: do not deserialize untrusted data — use JSON. If you must, apply JEP 290 serialization filters (ObjectInputFilter) as an allow-list of permitted classes plus caps on array size, graph depth and stream length; keep libraries patched; mark sensitive fields transient; validate inside readObject; and isolate the code with least privilege. Allow-lists beat reject-lists because new gadget classes appear constantly.

Pro What are some Jackson annotations you have used, and when?

@JsonProperty renames or forces inclusion of a field; @JsonIgnore/@JsonIgnoreProperties drop fields; @JsonInclude(NON_NULL) omits empty values; @JsonFormat controls date/number shape; @JsonCreator/@JsonProperty bind constructor parameters for immutable types. For output a single type can't express with annotations, register a custom serializer by extending StdSerializer.

Pro What do writeReplace and readResolve do, and where would you use them?

They substitute a different object during serialization (writeReplace) and deserialization (readResolve). Classic uses: preserving a singleton's single instance across a round-trip (return the canonical instance from readResolve), and the serialization proxy pattern from Effective Java — serialize a small, well-formed proxy via writeReplace and rebuild the real object through the proxy's readResolve, which sidesteps the constructor-bypass and many deserialization attacks.
