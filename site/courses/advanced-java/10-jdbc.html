Module: JDBC

JDBC is the wire between Java and a relational database — and the one place where a single string concatenation turns a query into a security incident. This module orients you on the API and points to the best material for each part, official docs first.

Knowledge Library → Courses → Advanced Java → JDBC

What this module covers

JDBC (Java Database Connectivity) is the standard java.sql API for talking to relational databases: getting a Connection (via DriverManager or a DataSource), running SQL through Statement and PreparedStatement, reading rows from a ResultSet, and controlling transactions, batching, and pooling. The headline skill — tested in almost every backend interview — is using PreparedStatement to make SQL injection impossible. Everything above JDBC (JPA, Hibernate, Spring Data, JdbcTemplate) is built on these six interfaces.

Topics

JDBC architecture and establishing a connection

JDBC is a set of interfaces in java.sql; a database vendor ships a driver that implements them. Your code asks DriverManager.getConnection(url, user, password) for a Connection, where the URL follows jdbc:subprotocol:subname (for example jdbc:postgresql://host/shop). Since JDBC 4.0 the matching driver on the classpath is discovered automatically through the service-provider mechanism, so the old Class.forName(...) call is no longer needed.

Why it matters: a Connection is a scarce, expensive OS-level resource. Always acquire it in a try-with-resources block so it — and every Statement and ResultSet under it — closes deterministically, even when an exception is thrown. Leaked connections are a classic production outage.

String url = "jdbc:postgresql://db.example.com/shop";   // driver auto-loaded (JDBC 4.0+)
try (Connection conn = DriverManager.getConnection(url, user, password)) {
    // ... use conn; it closes automatically at the end of the block
}

PreparedStatement and SQL injection safety

A Statement sends the exact SQL string you build to the database; if any part of that string came from user input, the input can change the structure of the query — that is SQL injection. A PreparedStatement is a parameterized template: you write ? placeholders and fill them with setString, setInt, setBigDecimal, and so on. The bound values travel to the server separately from the query text, so input is always treated as data, never parsed as SQL. This is the single most important JDBC habit and the most-asked JDBC interview point.

Why it matters: prepared statements are also the recommendation of every security authority (OWASP ranks parameterized queries as the primary defense), and they usually run faster on repeated execution because the database parses and plans the statement once. Treat PreparedStatement as the default and reach for plain Statement only for fixed, input-free SQL such as DDL.

// UNSAFE — concatenation lets input become SQL
String bad = "SELECT * FROM users WHERE email = '" + input + "'";

// SAFE — ? is a bind parameter, never parsed as code
String sql = "SELECT * FROM users WHERE email = ?";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setString(1, input);            // stays a value: ' OR '1'='1 matches nothing
    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) { /* ... */ }
    }
}

Reading results with ResultSet

executeQuery returns a ResultSet: a forward-moving cursor over the rows the database sent back. It starts positioned before the first row, so the idiom is while (rs.next()), pulling columns with typed getters (getLong, getString, getBoolean) by 1-based index or by column label. For non-query SQL, executeUpdate returns an int row count instead.

Why it matters: getters by index are faster but brittle to column reordering; getters by label are clearer. A ResultSet holds database resources and closes when its Statement closes, but closing it explicitly (try-with-resources) frees them sooner. Scrollable and updatable result sets exist, but the default forward-only, read-only cursor is the right choice for most reads.

String sql = "SELECT id, email FROM users WHERE active = ?";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setBoolean(1, true);
    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) {                  // cursor advances one row at a time
            long id = rs.getLong("id");      // by column label
            String email = rs.getString(2);  // or by 1-based index
        }
    }
}

Transactions and batch updates

By default a connection is in auto-commit mode: every statement commits on its own. To make several statements atomic, call setAutoCommit(false), run them, then commit() — and rollback() in the catch block so a failure leaves no partial writes. Isolation levels (READ_UNCOMMITTED through SERIALIZABLE) trade concurrency for protection against dirty, non-repeatable, and phantom reads. Batching is the sibling optimization: addBatch queues many parameter sets and executeBatch ships them in one round trip.

Why it matters: transactions are correctness (money moved out of one account must land in another), while batching is throughput (one network trip instead of thousands). The two combine — run a batch inside an explicit transaction so the whole set commits or rolls back together.

conn.setAutoCommit(false);                        // begin a transaction
try (PreparedStatement ps = conn.prepareStatement(
        "INSERT INTO ledger(acct, delta) VALUES (?, ?)")) {
    for (Entry e : entries) {
        ps.setString(1, e.acct());
        ps.setBigDecimal(2, e.delta());
        ps.addBatch();                            // queue; no round trip yet
    }
    ps.executeBatch();                            // one trip to the database
    conn.commit();                                // all-or-nothing
} catch (SQLException ex) {
    conn.rollback();                              // undo the whole unit
    throw ex;
}

DataSource and connection pooling

In real applications you rarely use DriverManager directly. A javax.sql.DataSource is a factory for connections that hides the URL and, crucially, supports connection pooling. Opening a physical connection (TCP handshake plus authentication) is slow, so a pool keeps a set of live connections and lends them out; calling close() returns the connection to the pool instead of tearing it down. HikariCP is the de-facto default (it is what Spring Boot auto-configures).

Why it matters: pooling bounds how many connections hit the database, cuts per-request latency, and keeps the app stable under load. Pool sizing is its own skill — bigger is not better; a small pool matched to the database's capacity usually beats an oversized one.

HikariConfig cfg = new HikariConfig();
cfg.setJdbcUrl("jdbc:postgresql://db.example.com/shop");
cfg.setMaximumPoolSize(10);
try (HikariDataSource ds = new HikariDataSource(cfg);
     Connection conn = ds.getConnection()) {     // borrowed from the pool
    // conn.close() returns it to the pool, not to the OS
}

Interview questions

Most-asked JDBC questions in Indian interviews — a staple wherever a backend role touches a relational database, from screening rounds to product-company deep dives:

Beginner What are the core steps to run a query with JDBC?

Get a Connection (from DriverManager or a DataSource), create a Statement/PreparedStatement, execute it to get a ResultSet (or an update count), read the rows, and close everything. Use try-with-resources so ResultSet, Statement, and Connection close in reverse order even on exception.

Beginner What does a JDBC connection URL look like, and who chooses the driver?

The form is jdbc:subprotocol:subname, e.g. jdbc:postgresql://host/shop. From JDBC 4.0, the matching driver on the classpath is registered automatically via the service-provider (SPI) mechanism, so Class.forName(...) is no longer needed; DriverManager maps the URL to a registered driver.

Beginner Statement vs PreparedStatement — what is the difference?

Statement sends a raw SQL string every time; PreparedStatement is a precompiled, parameterized template with ? placeholders filled by setXxx. PreparedStatement is safer (bound values can't become SQL), usually faster on repeated execution, and the default choice. Use plain Statement only for static, input-free SQL.

Beginner How do you read rows from a ResultSet?

The cursor starts before the first row, so loop while (rs.next()) and pull each column with a typed getter (getInt, getString, …) by 1-based index or by column label. Close the ResultSet when done; it also closes when its Statement closes.

Intermediate How exactly does a PreparedStatement stop SQL injection?

Bound parameters are sent to the database separately from the query text, so user input is always evaluated as a data value, never parsed as SQL. An input like ' OR '1'='1 becomes a literal string compared against the column, not a new clause. String-concatenated SQL is precisely what lets input alter the query's structure.

Intermediate Why is PreparedStatement often faster for repeated execution?

The database can parse the SQL and build the execution plan once, then reuse it for each execution with new bind values. That removes repeated parsing/planning overhead, and pools plus drivers can cache prepared statements client-side and server-side.

Intermediate What is auto-commit, and how do you make several statements one transaction?

By default each statement commits immediately (auto-commit true). Call setAutoCommit(false), run the statements, then commit(); on error rollback() so nothing partial persists. That gives all-or-nothing atomicity — essential for operations like transfers that must succeed or fail as a unit.

Intermediate Name the JDBC transaction isolation levels and the anomalies they control.

READ_UNCOMMITTED, READ_COMMITTED, REPEATABLE_READ, SERIALIZABLE (plus NONE). Rising levels progressively prevent dirty reads, then non-repeatable reads, then phantom reads, trading concurrency for consistency. Set with Connection.setTransactionIsolation; the exact behavior is enforced by the database.

Intermediate What is JDBC batching and why does it help?

addBatch queues multiple statements or parameter sets, and executeBatch sends them in a single round trip — so you pay network latency once instead of per row, a large win for bulk inserts/updates. executeBatch returns an int[] of per-command update counts.

Intermediate Why use a connection pool instead of DriverManager.getConnection each time?

Opening a physical connection (TCP + authentication handshake) is expensive. A pool (HikariCP and friends, behind a DataSource) keeps live connections ready and lends them out; close() returns the connection to the pool rather than tearing it down. It also caps concurrency against the database and improves latency and stability under load.

Pro Is PreparedStatement a 100% guarantee against SQL injection? Where can it still fail?

No. It parameterizes values, but you cannot bind identifiers — table/column names, an ORDER BY column, or dynamic SQL fragments built by concatenation remain injectable. Validate those against a whitelist of allowed names. Also handle LIKE wildcards and IN-lists carefully (bind one ? per element). PreparedStatement is necessary but not sufficient if you still concatenate query structure.

Pro executeBatch() throws BatchUpdateException partway through — what is the state afterwards, and what is the return array?

Each element of the int[] is one command's update count, or the sentinels SUCCESS_NO_INFO / EXECUTE_FAILED. On failure a driver may stop or keep going; BatchUpdateException.getUpdateCounts() tells you how far it got. Because partial effects could otherwise commit, run batches inside an explicit transaction and rollback on failure so the set stays atomic.
