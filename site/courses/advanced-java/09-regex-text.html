Module: Regular Expressions and Text

Regex and text handling are where correctness quietly meets Unicode: a pattern that flies on your laptop can hang forever on hostile input, and split can drop data you assumed it kept. This module maps the terrain and points you at the definitive resources.

What this module covers

Two clusters. First, java.util.regex: compiling a Pattern, driving a Matcher, capturing and naming groups, lookaround assertions, flags, and the performance traps (catastrophic backtracking / ReDoS) that turn a one-line regex into a denial-of-service. Second, the text toolkit around it: Scanner for tokenized and typed input, BreakIterator for locale-aware word/sentence/line boundaries, and the everyday String operations (split, replaceAll, text blocks) that most parsing actually needs. The theme throughout: reach for the smallest correct tool, and know where each one bites.

Topics

Pattern and Matcher: compiling and matching

A regex in Java is a two-step affair: Pattern.compile(regex) parses the expression once into a reusable, thread-safe Pattern; pattern.matcher(input) produces a stateful, single-threaded Matcher that does the work. The three entry points differ in scope: matches() demands the entire input match, lookingAt() anchors at the start only, and find() scans for the next matching subsequence anywhere — the one you loop over.

Why it matters: String.matches("...") recompiles the pattern on every call, so hot loops should hoist a compiled Pattern to a static final field. Getting matches vs find right is the most common regex misunderstanding in code review.

static final Pattern PHONE = Pattern.compile("\\d{3}-\\d{4}");

Matcher m = PHONE.matcher("call 123-4567 today");
while (m.find()) {          // scans for each occurrence
    System.out.println(m.group());   // 123-4567
}

Groups: capturing, named, and backreferences

Parentheses do two jobs. A capturing group (...) both groups a sub-pattern and remembers what it matched, retrievable by number via group(1) (group 0 is the whole match) or reused inside the pattern as a backreference \1. Named groups (?<name>...) trade numeric indices for readable group("name") — worth it the moment a pattern has more than two groups. When you only need grouping and not the capture, a non-capturing group (?:...) avoids allocating and renumbering.

Pattern p = Pattern.compile("(?<year>\\d{4})-(?<month>\\d{2})");
Matcher m = p.matcher("2026-07");
if (m.matches()) {
    m.group("year");   // "2026"
    m.group("month");  // "07"
}

Lookaround: lookahead and lookbehind

Lookaround assertions test the context around a position without consuming characters — they match the empty string but only where their condition holds. Positive lookahead (?=...) and negative lookahead (?!...) peek forward; lookbehind (?<=...) and (?<!...) peek backward. The classic use is multi-rule validation (a password that contains a digit and an uppercase letter) expressed as independent assertions rather than one tangled alternation. In replacements, lookaround and backreferences are two ways to keep surrounding text intact while rewriting the middle.

// at least one digit and one uppercase, min length 8 — assertions don't consume
Pattern strong = Pattern.compile("^(?=.*\\d)(?=.*[A-Z]).{8,}$");
strong.matcher("Passw0rd").matches();  // true

Flags, modes, and catastrophic backtracking

Flags change how a pattern is interpreted. Pass them as a bitmask to compile (CASE_INSENSITIVE, MULTILINE so ^/$ match per line, DOTALL so . spans newlines, COMMENTS, UNICODE_CASE) or inline as embedded flag expressions such as (?i) and (?m). Beyond correctness lies performance: nested quantifiers over overlapping alternatives (the shape (a+)+) can trigger exponential catastrophic backtracking on non-matching input — a real ReDoS vector when the regex touches user data. Defenses are possessive quantifiers a++, atomic groups (?>...), and anchoring the pattern.

Pattern p = Pattern.compile("error", Pattern.CASE_INSENSITIVE | Pattern.MULTILINE);
Pattern q = Pattern.compile("(?im)error");   // same, embedded flags

Scanner: tokenizing and typed input

Scanner wraps a source (a String, InputStream, or Path) and breaks it into tokens on a delimiter — whitespace by default, any regex via useDelimiter. Its selling point is typed reads: nextInt(), nextDouble(), and friends, each guarded by a matching hasNextInt() so you can validate before consuming. The infamous trap: after nextInt() the newline is still buffered, so a following nextLine() returns empty — you must consume the line ending first. For bulk line-reading Scanner is convenient but slow; BufferedReader wins on large files.

Scanner sc = new Scanner(System.in);
int n = sc.nextInt();
sc.nextLine();               // consume the leftover line terminator
String label = sc.nextLine(); // now reads the real next line

BreakIterator: locale-aware text boundaries

"Split on spaces" and "count sentences by periods" are wrong in almost every language — and even in English, where Dr. is not a sentence end and a URL has no spaces. BreakIterator encodes the Unicode text-segmentation rules for a given Locale, with factory methods for character, word, sentence, and line boundaries. You set the text, then walk boundaries with first()/next() until DONE. It is the correct tool for word counts, sentence extraction, and line-wrap points across scripts.

BreakIterator s = BreakIterator.getSentenceInstance(Locale.US);
s.setText("Dr. Rao arrived. He spoke.");
int count = 0;
for (int e = s.first(); e != BreakIterator.DONE; e = s.next()) count++;
// boundaries yield 2 sentences, not 3 — "Dr." is not treated as a break

Text processing: split, replace, and text blocks

Most parsing does not need a hand-rolled Matcher loop — the String methods already take regexes. split(regex) is the workhorse, with one sharp edge: the zero-argument form discards trailing empty strings, so "a,,b,".split(",") silently loses the final field; pass a negative limit to keep them. replaceAll/replaceFirst apply regex substitutions (mind Matcher.quoteReplacement when the replacement contains $ or \). Text blocks ("""...""") make multi-line regex and expected test strings readable without escape soup.

"a,,b,".split(",");      // ["a", "", "b"]      — trailing empties dropped
"a,,b,".split(",", -1);  // ["a", "", "b", ""]  — limit -1 keeps them

Interview questions

Regex and text handling recur in Indian interviews — from screening puzzles (validate an email, count words) to senior discussions on ReDoS and Unicode correctness:

Beginner What are the main classes in java.util.regex and how do they relate?

Pattern is a compiled regex (immutable, thread-safe); Matcher is the stateful engine you get from pattern.matcher(input) to run match operations; PatternSyntaxException is thrown when the regex itself is malformed. You compile once, match many times.

Beginner Difference between matches() and find()?

matches() succeeds only if the entire input matches the pattern; find() scans for the next matching subsequence anywhere and can be called repeatedly in a loop. lookingAt() sits between them — it anchors at the start but need not consume the whole input.

Beginner How do you extract part of a match?

Wrap the part in a capturing group and read it after a successful find()/matches(): m.group(1) by index (group 0 is the whole match) or m.group("name") for a named group (?<name>...).

Beginner Why prefer a precompiled Pattern over String.matches() in a loop?

String.matches(), replaceAll(), and split() compile the regex on every call. In a hot path that is wasted work — hoist the pattern to a static final Pattern compiled once and reuse its Matcher.

Intermediate Greedy vs reluctant vs possessive quantifiers?

Greedy (a*) matches as much as possible then backtracks; reluctant/lazy (a*?) matches as little as possible then expands; possessive (a*+) matches greedily and never gives characters back. Possessive quantifiers trade some matching power for immunity to backtracking blowups.

Intermediate Capturing vs non-capturing vs named groups — when each?

Capturing (...) when you need the matched text or a backreference; non-capturing (?:...) when you only need to group for a quantifier or alternation and want to avoid renumbering/allocation; named (?<name>...) for readability once there are several groups — the pattern documents itself and refactors safely.

Intermediate What is a lookahead, and how would you validate a password with one?

A lookahead (?=...) asserts what follows without consuming it. Stacking independent lookaheads expresses "must contain X and Y" cleanly: ^(?=.*\d)(?=.*[A-Z]).{8,}$ requires a digit, an uppercase letter, and length 8+ — each rule stays independent instead of one brittle alternation.

Intermediate The Scanner nextInt()-then-nextLine() bug — what happens and why?

nextInt() reads the number but leaves the trailing newline in the buffer, so the next nextLine() returns an empty string. Fix: call an extra nextLine() to consume the line terminator after the typed read, or read whole lines and parse them yourself.

Intermediate How do regex flags work — CASE_INSENSITIVE vs (?i)?

Both do the same thing two ways: pass Pattern.CASE_INSENSITIVE (OR-combined with other flags like MULTILINE, DOTALL) to compile, or embed (?i) at the start of the pattern. For Unicode-correct case folding add UNICODE_CASE alongside CASE_INSENSITIVE.

Intermediate String.split(",") drops your last field. Why, and how do you keep it?

The one-argument split uses limit 0, which removes trailing empty strings — so "a,,b," yields ["a","","b"], losing the final empty field. Pass a negative limit (split(",", -1)) to preserve all trailing empties.

Pro What is catastrophic backtracking (ReDoS), and how do you prevent it?

Nested quantifiers over overlapping matches — the shape (a+)+$ against a long non-matching input — cause the engine to explore exponentially many ways to split the string, freezing the thread. It is exploitable when the regex runs on user input. Prevent it with possessive quantifiers (a++), atomic groups (?>...), anchoring, simplifying the pattern, or bounding input length; never run untrusted regexes on untrusted input.

Pro Why is BreakIterator more correct than split() for counting words or sentences?

split on whitespace or punctuation encodes English-centric assumptions that break on abbreviations (Dr.), decimals, contractions, and non-spaced scripts (Chinese, Japanese, Thai). BreakIterator applies the Unicode text-segmentation rules for a Locale, so word and sentence boundaries are correct across languages — the difference between a demo and a shippable internationalized feature.

Pro Is a Pattern or Matcher safe to share across threads, and how does that shape reuse?

Pattern is immutable and thread-safe — compile it once and share it as a constant. Matcher is stateful and not thread-safe: each thread must obtain its own via pattern.matcher(...), or reuse one with reset() within a single thread. The idiom is a shared static Pattern plus a per-use Matcher.
