Module: Clean Code Practices

Clean Java is not a style preference — it is a set of hard-won defaults from Effective Java that decide whether a class survives the next change, and interviewers probe them because they separate people who ship features from people who ship maintainable systems.

Knowledge Library → Courses → Advanced Java → Clean Code Practices

What this module covers

This is the discipline layer on top of the language: the Effective Java habits that make classes safe to change — static factories and builders for construction, immutability and defensive copies for state, composition over inheritance and minimal accessibility for structure — and how you document and review the result. None of it is new syntax; all of it is judgment. The one genuinely new piece is JDK 23's Markdown documentation comments (JEP 467), which modernize how you write Javadoc.

Topics

Static factory methods over constructors

A static factory is a static method that hands back an instance in place of (or alongside) a public constructor — Optional.of, List.of, Boolean.valueOf, Integer.valueOf. Effective Java opens with it (Item 1) because a factory does three things a constructor cannot: carry a descriptive name, skip allocation by returning a cached instance, and return any subtype of its declared return type.

Why it matters: that last point powers interface-based APIs — callers depend on List, not ArrayList. It is a routine screening opener because the answer reveals whether you have actually internalized the idiom or just memorized the GoF factory pattern (related, but not the same thing).

// named, no forced allocation, can return a subtype
public static Boolean valueOf(boolean b) {
    return b ? Boolean.TRUE : Boolean.FALSE;   // cached, no new object
}
List<String> empty = List.of();               // JDK static factories everywhere

The Builder pattern

When a class has more than a handful of parameters — several of them optional — telescoping constructors become unreadable and JavaBean setters leave the object mutable and temporarily inconsistent. The builder (Effective Java Item 2) gives named, chainable methods and a single build() that validates and returns a finished, usually immutable, object.

Why it matters: it is the standard answer to "how would you construct an object with fifteen fields, ten of them optional?", and it is everywhere in the ecosystem — StringBuilder, Stream.Builder, HttpRequest.newBuilder(), Lombok's @Builder. Know the trade-off too: a builder adds a class and an allocation, so it earns its keep only when parameter count or optionality is genuinely high.

NutritionFacts cola = new NutritionFacts.Builder(240, 8)
    .calories(100)
    .sodium(35)
    .build();          // validated in build(); immutable once built

Immutability first (and defensive copies)

An immutable object's state is fixed at construction: a final class or a record, private final fields, no setters, and defensive copies of any mutable input. Immutables are automatically thread-safe, freely shareable and cacheable, and make excellent map keys — which is exactly why String, the wrapper types, and the whole java.time API are immutable (Effective Java Item 17).

Why it matters: records (Java 16+) make the common case a one-liner but give only shallow immutability — a record with a List component still exposes that list to mutation unless you copy it in the compact constructor. Defensive copying (Item 50) is the discipline that closes the hole on the way in, and again on the way out of any accessor.

public record Range(int lo, int hi) {}          // final fields, no setters

// mutable input? copy it so callers can't mutate our state later:
this.dates = List.copyOf(dates);                // unmodifiable snapshot

Composition over inheritance

Inheritance models "is-a" and ties a subclass to the internal, versioned behavior of its superclass; composition models "has-a" — a class holds another and forwards to it. Effective Java Item 18 prefers composition (with forwarding) because extending a class across package boundaries is fragile: a superclass change to how its methods call one another can silently break an override.

Why it matters: the canonical trap is extending HashSet to count insertions and getting double counts because addAll calls add internally. A wrapper that depends only on the Set interface avoids it. This question separates "inheritance is code reuse" from "inheritance is a subtyping contract" — a distinction interviewers reach for constantly.

// prefer HAS-A: wrap and forward, don't extend
final class InstrumentedSet<E> {
    private final Set<E> delegate;              // depend on the interface
    private int addCount = 0;
    InstrumentedSet(Set<E> s) { this.delegate = s; }
    boolean add(E e) { addCount++; return delegate.add(e); }
}

Minimize accessibility (encapsulation)

Effective Java Item 15 — make each class and member as inaccessible as possible. Fields should be private; behavior is exposed through methods so the class stays in control of its own invariants. The four levels, tightest first: private (class only), package-private (the default, same package), protected (package plus subclasses), public (everywhere).

Why it matters: a public mutable field is an encapsulation leak — anyone can drive the object into an illegal state, and you can never change the representation without breaking clients. The only safe public fields are static final constants of an immutable type. Reproducing the access table (class / package / subclass / world) is a guaranteed screening question.

public class Account {
    private BigDecimal balance;                 // fields private
    public BigDecimal balance() { return balance; }   // expose via method
}

Javadoc and Markdown documentation comments (///)

Good API documentation is part of clean code: every exported class, method, and field gets a doc comment whose first sentence is a self-contained summary, with @param, @return, and @throws spelling out the contract. The javadoc tool turns these into HTML, and DocLint can fail the build on broken comments.

Why it matters: JDK 23's JEP 467 modernizes the format. A comment whose every line starts with /// is written in CommonMark Markdown instead of the traditional /** ... */ mix of HTML and tags — blank lines make paragraphs, backticks replace {@code}, and Markdown links replace {@link}, while the standard @-tags still work. Knowing both forms is what "documentation" means on a modern review checklist.

/// Returns the sum of `a` and `b`.
///
/// Wraps on overflow — see [Math#addExact(int, int)] to fail instead.
/// @param a first addend
/// @param b second addend
/// @return the sum of the two arguments
static int add(int a, int b) { return a + b; }

The code review checklist

A review is not taste-policing; it is a repeatable checklist. Half of it is already the items in this module — minimal accessibility, immutability, defensive copies, sound construction — and the rest is correctness, clarity, and coverage. Consistency beats individual preference, which is why teams delegate formatting to an automated style guide so review time goes to design, not brace placement.

Why it matters: "how do you review code?" is really "do you have a repeatable standard, or do you review by vibes?" A concrete pass looks roughly like this:

  • Correctness: does it do what it claims, and handle the edge and error cases?
  • Design: clear names, minimal accessibility, immutable state where possible, no needless inheritance.
  • Safety: inputs validated, mutable inputs and outputs defensively copied, resources closed.
  • Tests and docs: meaningful tests for the new behavior, Javadoc on the public surface.
  • Style: conforms to the team's automated formatter and linter — flagged by tooling, not by people.

Interview questions

Clean-code questions asked in Indian interviews — Effective Java items are a favourite of product-company rounds because they test judgment, not syntax:

Beginner What is a static factory method, and one advantage over a constructor?

A static method that returns an instance instead of a public constructor — Optional.of, List.of, Boolean.valueOf. Advantages: it has a meaningful name, it need not create a new object each call (it can cache/intern), and it can return any subtype of its declared return type. Trade-off: a class with only private constructors cannot be subclassed.

Beginner What is an immutable class and how do you write one?

A class whose state cannot change after construction. Recipe: make the class final (or use a record), all fields private final, no setters, initialize everything in the constructor, and defensively copy any mutable field on the way in and out. Examples: String, the wrapper types, java.time. Immutables are inherently thread-safe and safe to share and cache.

Beginner What are the four access levels, and which should you default to?

private (class only), package-private / default (same package), protected (package plus subclasses), public (everywhere). Default to the most restrictive that works — private for fields, exposing behavior through methods. The only safe public fields are static final constants of an immutable type.

Beginner What does "prefer composition over inheritance" mean?

Instead of extending a class (is-a), hold a reference to it and forward calls (has-a). Prefer it because inheritance couples you to the superclass's implementation details across versions, while composition depends only on a published interface and lets you wrap any implementation. Use inheritance only for a genuine is-a relationship where the supertype's contract truly holds.

Intermediate When would you choose the Builder pattern over a constructor or setters?

When a class has many parameters, especially optional ones (rule of thumb: four or more, or several optional). Telescoping constructors become unreadable and JavaBean setters leave the object mutable and half-built. The builder gives named, chainable setters, validates in build(), and can hand back an immutable object.

Intermediate Why are defensive copies needed, and where do you make them?

When a constructor or setter stores a caller-supplied mutable object (a Date, List, or array), copy it so later external mutation cannot corrupt your invariants; copy again when returning it from a getter. Validate the copy, not the original, to avoid time-of-check/time-of-use bugs. Immutable types like LocalDate or String need no copy.

Intermediate Records give immutability — so what are their limits?

A record is only shallowly immutable: its component references are final, but if a component is itself mutable (a List, an array, a mutable bean) callers can still mutate what it points to. For deep immutability, defensively copy mutable components in the compact constructor and return unmodifiable views from the accessors.

Intermediate Name three advantages of static factory methods (Effective Java Item 1).

(1) They have names, so several factories with the same parameter types can coexist. (2) They are not required to create a new instance, enabling caching/flyweights (Boolean.valueOf). (3) They can return a subtype of their return type, hiding implementation classes behind an interface. Disadvantage: a class with no public/protected constructor cannot be subclassed.

Intermediate Why does Effective Java say "minimize the accessibility of classes and members"?

Encapsulation decouples modules: the less you expose, the more you can change internals without breaking clients, and the smaller the surface for misuse. Make everything as private as possible and widen only when a client genuinely needs it. Never expose a mutable field or a getter to a mutable internal — it breaks the invariants the class exists to guarantee.

Pro Inheritance can break encapsulation — explain the fragile base class problem and the fix.

A subclass depends on the superclass's implementation, not just its contract. If a later version changes how the superclass's methods call one another (self-use), a subclass that overrode one can break — the classic case is extending HashSet, overriding add and addAll, and getting double counts because addAll calls add. Fix: composition plus forwarding — a wrapper that holds the collection and implements the same interface, depending only on the public API. Otherwise design for inheritance, document self-use, or forbid it with final.

Pro What is JEP 467, and how do /// comments differ from /** ... */?

JEP 467 (shipped in JDK 23) adds Markdown documentation comments: a comment whose every line begins with /// and whose body is CommonMark Markdown instead of the HTML-plus-tags of the traditional /** ... */ form. Blank lines become paragraphs, backticks replace {@code}, and Markdown reference links replace {@link}, while standard @-tags such as @param and @return still work. It makes doc comments far more readable in source form.

Pro A getter returns a reference to a mutable internal List. Why is that a bug, and how do you fix it without killing performance?

The caller now holds a live reference and can mutate your object's state behind its back, silently breaking invariants and thread-safety — the same leak on the input side needs a defensive copy in the constructor. Fixes: return a copy (new ArrayList<>(list)), return an unmodifiable view (List.copyOf or Collections.unmodifiableList), or store the field as an already-immutable type. The unmodifiable-view approach avoids copying on every read while still preventing mutation.
