Module: The Module System (JPMS)

Java 9 broke the monolithic JDK into modules and handed your own code the same tool: a module-info that declares exactly what it needs and what it lets others see. Most projects still run happily on the classpath — knowing why, and when modules earn their keep, is what interviews actually probe.

Knowledge Library → Courses → Advanced Java → The Module System (JPMS)

What this module covers

JPMS (Java 9+) adds a level of grouping above packages: a named, self-describing module that lists its dependencies and its public API. This module orients you on the two guarantees it buys — reliable configuration and strong encapsulation — the module-info.java directives that express them, the services mechanism that decouples modules, the newer module import declarations, and the pragmatic migration story most teams actually live with.

Topics

Why modules exist: reliable configuration and strong encapsulation

A module is a named collection of packages plus a descriptor. At startup the runtime resolves the whole module graph, giving two guarantees the classpath never could. Reliable configuration: every required module must be present exactly once, so the missing-class-at-runtime and duplicate-JAR failures surface immediately as a resolution error. Strong encapsulation: a package is invisible to other modules unless it is explicitly exported, so internal packages stay internal even though their types are public.

Why it matters: this reframes access control from "public means everyone" to "public, and exported, means everyone". The JDK itself was modularised this way (JEP 261 / JSR 376), which is why reflecting into sun.* internals stopped working on Java 9+. Understanding the intent — not just the syntax — is what separates a rehearsed answer from a senior one.

module-info.java: requires, exports, opens

The descriptor lives as module-info.java at the source root and reads like a contract. requires declares a compile-and-runtime dependency; requires transitive re-exports that dependency so your callers read it too (implied readability); requires static is a compile-only optional dependency. exports makes one package's public types visible; exports … to restricts that to named modules (qualified). opens / opens … to grant deep reflective access at runtime — the switch frameworks like Spring, Hibernate and Jackson need to reach non-public members.

Why it matters: the single most common interview trap is confusing exports (compile-time API access) with opens (runtime reflective access). Export lets code call your methods; open lets a framework reflect into your fields. They are orthogonal, and picking the wrong one is the root cause of most "works on the classpath, breaks as a module" bugs.

module com.shop.billing {
    requires com.shop.core;                       // compile + runtime
    requires transitive org.slf4j;                // callers read slf4j too
    requires static org.jetbrains.annotations;    // compile-only, optional

    exports com.shop.billing.api;                 // public API package
    exports com.shop.billing.spi to com.shop.reporting; // qualified export

    opens com.shop.billing.model;                 // deep reflection for frameworks

    uses com.shop.billing.spi.PaymentProvider;    // a service this module consumes
}

Services: ServiceLoader, uses and provides

Modules should depend on abstractions, not implementations. The services mechanism lets a consumer declare uses SomeService and a provider declare provides SomeService with SomeImpl, with no direct module dependency between them. At runtime ServiceLoader.load(SomeService.class) discovers every registered provider and instantiates it (via a no-arg constructor or a static provider() method). It is the same Service Provider Interface pattern that has powered JDBC drivers and java.util.logging for years, now first-class in the module graph; on the classpath the equivalent registration lives in META-INF/services.

Why it matters: services are how the module system does dependency inversion without a container. Being able to sketch a plugin architecture with uses/provides and ServiceLoader is a frequent design-round question, and it lets a provider ship implementation packages without exporting them at all.

// consumer module-info:  uses com.shop.billing.spi.PaymentProvider;
// provider module-info:  provides com.shop.billing.spi.PaymentProvider
//                            with com.shop.upi.UpiPaymentProvider;

ServiceLoader<PaymentProvider> loader = ServiceLoader.load(PaymentProvider.class);
PaymentProvider provider = loader.findFirst()
    .orElseThrow(() -> new IllegalStateException("no payment provider on the module path"));

Module import declarations (JEP 511, Java 25)

Finalised in Java 25 (preview in 23 and 24), a module import declaration — import module java.base; — imports, in one line, every package that the named module exports. It is a source-file convenience that reduces import clutter, and, importantly, it works even in code that is not itself a module. It takes a module name, so it cannot import the unnamed module (the classpath), mirroring the way requires only names modules.

Why it matters: this is the freshest JPMS-adjacent feature and a likely "what's new in recent Java" question. The nuance interviewers look for is that import module affects source-level imports only — it is not a substitute for a requires directive, which still governs the actual module dependency.

import module java.base;      // every package exported by java.base

void main() {
    var today = LocalDate.now();          // no per-package import needed
    var ids = List.of("upi", "card");     // List, LocalDate, etc. all in scope
    IO.println(today + " " + ids);
}

Migration reality: classpath, unnamed and automatic modules

The truth most tutorials bury: you do not have to modularise anything to run on Java 9 or later. Code on the classpath lands in the unnamed module, which reads every other module and exports everything — so existing applications keep working. When you do start, plain JARs placed on the module path become automatic modules: their name comes from the Automatic-Module-Name manifest header (or, as a fragile fallback, the file name), and they bridge modular code to whatever is still on the classpath. That lets you modularise top-down without waiting for every dependency to catch up.

Why it matters: the real pain of upgrading is strong encapsulation of JDK internals — libraries that reflected into java.* throw InaccessibleObjectException, fixed with launch-time --add-opens / --add-exports (or --add-reads). Split packages (the same package in two modules) are forbidden and must be resolved. Knowing these escape hatches, plus jdeps to analyse dependencies and jlink to build a slim runtime, is what "migration experience" means on a resume.

Interview questions

JPMS questions asked in Indian interviews — from screening "what is a module" to senior migration war stories:

Beginner What is the Java Platform Module System and which release introduced it?

JPMS (Project Jigsaw, JSR 376 / JEP 261) shipped in Java 9. It adds a named, self-describing unit above packages — a module that declares its dependencies and the packages it exposes — giving reliable configuration and strong encapsulation across the whole application, including a modularised JDK.

Beginner What is module-info.java and where does it go?

The module descriptor. It sits at the root of the module's source (compiled to module-info.class) and names the module plus its directives: requires (dependencies), exports (public API), opens (reflective access), and uses/provides (services).

Beginner Difference between requires and exports?

requires is about what this module consumes — a dependency on another module. exports is about what this module offers — it makes one package's public types readable by others. One points inward, the other outward.

Beginner Do you have to convert an app to modules to run it on Java 17 or 21?

No. Code on the classpath runs inside the unnamed module, which reads all modules and exports everything, so non-modular apps keep working. Modularising is optional and can be done incrementally.

Beginner What does strong encapsulation mean in JPMS?

A public type is only accessible to other modules if its package is exported. Non-exported packages are hidden even though their classes are public — "public" no longer means "accessible everywhere", which is why access into JDK internals broke on Java 9+.

Intermediate exports vs opens — when do you need opens?

exports grants compile-time and runtime access to a package's public API — callers can invoke it. opens grants deep reflective access at runtime, including to non-public members. Frameworks that reflect (Spring, Hibernate, Jackson) need the package opened, not just exported. Getting these two confused is the classic module bug.

Intermediate requires transitive vs requires static?

requires transitive implies readability: any module that reads yours also reads the transitively required one — used when your API exposes that module's types. requires static is a compile-time-only dependency that may be absent at runtime — used for optional libraries and annotations.

Intermediate What is an automatic module and the Automatic-Module-Name header?

A plain (non-modular) JAR placed on the module path becomes an automatic module: it exports and opens all packages and reads every other module, including the unnamed module, so it bridges modular code to the classpath. Its name comes from the Automatic-Module-Name manifest entry, or is derived from the file name if that is missing — deriving from the file name is fragile, so libraries should set the header before others depend on them.

Intermediate How do uses, provides and ServiceLoader work together?

The consumer declares uses S; each provider declares provides S with Impl. At runtime ServiceLoader.load(S.class) finds and instantiates the providers on the module path (no-arg constructor or a static provider() method). It is dependency inversion without a DI container, and the provider need not export its implementation package.

Intermediate What is the unnamed module?

The catch-all module for everything loaded from the classpath. It has no name (so nothing can requires it), reads all modules, and exports/opens all its packages. It exists to keep pre-module code running unchanged alongside the module system.

Pro After upgrading, a library throws InaccessibleObjectException reflecting on JDK internals. How do you fix it?

Strong encapsulation now blocks that reflection. As a launch-time fix, open or export the specific package: --add-opens java.base/java.lang=ALL-UNNAMED (reflective access) or --add-exports (compile/runtime API access), and --add-reads to add a readability edge. The real fix is upgrading the offending library; the flags are a bridge, and each one is a deliberate, auditable hole in encapsulation.

Pro What is a split package and why does JPMS forbid it?

A split package is the same package name contributed by two different modules. JPMS forbids it because a package must belong to exactly one module for reliable configuration and consistent access rules — otherwise readability and encapsulation would be ambiguous. Split packages are a common migration blocker; the fix is to merge or rename, or during transition use --patch-module.

Pro What do module import declarations (JEP 511, Java 25) do, and how do they differ from a requires directive?

import module M; is a source-level statement that imports every package exported by module M in one line, usable even in non-modular code; it cannot name the unnamed module. It only affects imports in that file — it does not create a module dependency. The actual dependency is still declared by requires in module-info.java; the two operate at different layers (source imports vs the module graph).
