Module: Networking and HttpClient

Java's network stack spans two eras: the 1990s blocking-socket API that still underpins every server, and the fluent java.net.http.HttpClient that speaks HTTP/2 today and HTTP/3 over QUIC as of Java 26. Interviewers probe both — the sockets to test fundamentals, the client to test whether you have shipped modern code.

Knowledge Library → Courses → Advanced Java → Networking and HttpClient

What this module covers

Two layers of the java.net world. First, the classic blocking sockets: Socket and ServerSocket for connection-oriented TCP, and DatagramSocket/DatagramPacket for connectionless UDP. Second, the standardised java.net.http.HttpClient (final since Java 11) — synchronous and asynchronous requests, HTTP/2 by default, HTTP/3 over QUIC newly finalised in Java 26, and a built-in WebSocket. Depth lives in the linked docs; this page maps the terrain and the questions.

Topics

TCP sockets: Socket and ServerSocket

TCP gives you a reliable, ordered, bidirectional byte stream between two endpoints. A client creates a Socket to a host and port; a server binds a ServerSocket to a port and calls accept(), which blocks until a client connects and then returns a fresh Socket for that conversation. From there both sides talk through ordinary InputStream/OutputStream — sockets are just streams with a network on the other end.

Why it matters: this is the substrate under HTTP servers, database drivers, and message brokers. The one-thread-per-connection model here is also where interviewers pivot into concurrency and, later, into the non-blocking NIO selector model that scales past it.

// server: accept one client, echo a line
try (ServerSocket server = new ServerSocket(5000);
     Socket client = server.accept();            // blocks until a client connects
     var in  = new BufferedReader(new InputStreamReader(client.getInputStream()));
     var out = new PrintWriter(client.getOutputStream(), true)) {
    out.println("echo: " + in.readLine());
}

UDP datagrams: DatagramSocket and DatagramPacket

UDP trades reliability for speed and simplicity: there is no connection, no handshake, and no delivery or ordering guarantee. Each message is an independently addressed DatagramPacket that you send and receive through a DatagramSocket. Packets can be lost, duplicated, or arrive out of order — the application must tolerate or handle that. In exchange you avoid TCP's per-connection state and head-of-line blocking, which is why UDP carries DNS, streaming media, game state, and (via QUIC) HTTP/3.

Why it matters: knowing when not to use TCP is a maturity signal. "Fire-and-forget telemetry where a dropped sample is fine" is a textbook UDP answer.

byte[] buf = "ping".getBytes();
try (DatagramSocket socket = new DatagramSocket()) {
    var addr = InetAddress.getByName("localhost");
    socket.send(new DatagramPacket(buf, buf.length, addr, 5001)); // no connection, no ack
}

The modern HttpClient: synchronous and asynchronous

Since Java 11 the platform ships a real HTTP client in java.net.http, retiring the awkward HttpURLConnection. The pieces are immutable and builder-made: one reusable HttpClient, an HttpRequest (URI, method, headers, a BodyPublisher for the request body), and an HttpResponse read via a BodyHandler. Call send() to block for the response, or sendAsync() to get a CompletableFuture<HttpResponse> you can compose without tying up a thread.

Why it matters: this is the default way to call a REST API from plain Java, and the send-vs-sendAsync distinction plus the handler/publisher pattern are standard interview ground. One HttpClient is meant to be created once and shared — it is immutable and thread-safe.

HttpClient client = HttpClient.newHttpClient();          // reuse this instance
HttpRequest req = HttpRequest.newBuilder(URI.create("https://api.example.com/users"))
    .header("Accept", "application/json")
    .GET()
    .build();

HttpResponse<String> res = client.send(req, BodyHandlers.ofString()); // blocking
client.sendAsync(req, BodyHandlers.ofString())                         // non-blocking
      .thenApply(HttpResponse::body)
      .thenAccept(System.out::println);

HTTP/2 by default, HTTP/3 over QUIC (final in Java 26)

The client negotiates HTTP_2 by default and transparently falls back to HTTP/1.1 when a server does not support it — your code does not change. As of Java 26, HttpClient also speaks HTTP/3, delivered as a final feature by JEP 517. HTTP/3 runs over QUIC on UDP (with TLS 1.3 built in) instead of TCP, which removes TCP's connection-level head-of-line blocking. You opt in with .version(HttpClient.Version.HTTP_3); because that is only a preference, use the alt-svc discovery mode so the first request goes over HTTP/2 and later ones upgrade once the server advertises HTTP/3.

HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_3)   // Java 26+; a preference, not a guarantee
    .build();                              // downgrades to HTTP/2 or HTTP/1.1 if needed

WebSocket over HttpClient

The same HttpClient opens WebSocket connections through newWebSocketBuilder(). A WebSocket is a full-duplex, long-lived channel: you send with sendText/sendBinary (each returning a CompletableFuture) and receive through a WebSocket.Listener whose onText/onBinary/onPing/onClose callbacks the client invokes. Flow control is explicit: the listener must call request(n) to signal it is ready for more messages — the same back-pressure idea as the reactive-streams Flow API.

WebSocket ws = HttpClient.newHttpClient()
    .newWebSocketBuilder()
    .buildAsync(URI.create("wss://echo.websocket.org"), new WebSocket.Listener() {
        public CompletionStage<?> onText(WebSocket s, CharSequence data, boolean last) {
            System.out.println(data);
            s.request(1);            // ask for the next message (back-pressure)
            return null;
        }
    }).join();
ws.sendText("hello", true);

Interview questions

Networking questions common in Indian interviews — sockets test fundamentals, HttpClient tests whether you write current code:

Beginner Difference between TCP and UDP, and when would you choose UDP?

TCP is connection-oriented, reliable, and ordered — a byte stream with handshake, acknowledgements, and retransmission. UDP is connectionless and best-effort: no handshake, no delivery or ordering guarantee, lower overhead and latency. Choose UDP when speed matters more than reliability and occasional loss is acceptable — DNS, live audio/video, telemetry, gaming, and the QUIC transport under HTTP/3.

Beginner Socket vs ServerSocket — what are their roles?

Socket is the client endpoint: you construct it with a host and port to connect out. ServerSocket is the server endpoint: it binds to a port and listens. Its accept() returns a new Socket representing the accepted connection, which the server uses to talk to that one client.

Beginner What does ServerSocket.accept() return, and why does it block?

It returns a connected Socket for a single client. It blocks because the server has nothing to do until a client actually arrives — the calling thread parks until the TCP handshake completes. To serve several clients you loop, calling accept() repeatedly and handing each returned socket to a worker thread.

Beginner Which classes send and receive a UDP message?

DatagramSocket is the send/receive endpoint; DatagramPacket wraps the payload bytes plus destination address and port. You call socket.send(packet) and socket.receive(packet). There is no connection object because UDP has no connection.

Intermediate How do you handle many concurrent clients on a TCP server?

Classic model: an accept loop that hands each returned Socket to a thread — but thread-per-connection does not scale to tens of thousands of idle connections (memory and context-switching). Better: a bounded thread pool (ExecutorService), virtual threads (Java 21+) for cheap one-thread-per-connection, or the non-blocking NIO selector model where one thread multiplexes many channels.

Intermediate Why prefer java.net.http.HttpClient over HttpURLConnection?

HttpURLConnection is verbose, mutable, HTTP/1.1-only, and clumsy for anything asynchronous. The modern HttpClient (final in Java 11) offers a fluent immutable builder API, HTTP/2 by default, native async via CompletableFuture, request/response body handlers, WebSocket support, and — from Java 26 — HTTP/3. No third-party library needed for common cases.

Intermediate send() vs sendAsync() — what is the difference?

send() is synchronous: it blocks the calling thread and returns an HttpResponse. sendAsync() returns immediately with a CompletableFuture<HttpResponse> that completes when the response arrives, so you compose with thenApply/thenAccept and never block. Use async for high-throughput or when firing many requests concurrently.

Intermediate What are BodyPublishers and BodyHandlers?

BodyPublisher (from HttpRequest.BodyPublishers) supplies the request body — from a String, file, byte array, or input stream. BodyHandler (from HttpResponse.BodyHandlers) decides how to consume the response — ofString(), ofByteArray(), ofFile(), ofInputStream(). They decouple the request/response mechanics from how bodies are produced and read, using reactive-streams under the hood.

Intermediate What HTTP version does HttpClient use by default, and what if the server can't?

It prefers HTTP_2. If the server does not support HTTP/2 the client transparently downgrades to HTTP/1.1 — the request still succeeds and your code is unchanged. Setting a version (including HTTP_3 in Java 26) is a preference, not a guarantee; the client negotiates and falls back.

Intermediate Is HttpClient reusable and thread-safe? Do you create one per request?

No — create it once and share it. HttpClient is immutable and thread-safe, and it pools connections; instantiating one per request wastes connection setup and defeats HTTP/2 multiplexing. Build a single configured client (timeout, redirect policy, executor) and reuse it across the application.

Pro HTTP/2 vs HTTP/3 — what changed in the transport, and how do you enable HTTP/3 in Java 26?

HTTP/2 multiplexes many streams over one TCP connection, but a single lost TCP segment stalls every stream (connection-level head-of-line blocking). HTTP/3 runs over QUIC on UDP with TLS 1.3 integrated; QUIC's streams are independent, so one lost packet only stalls its own stream, and connection migration survives IP changes. In Java 26 (JEP 517, final) enable it with HttpClient.newBuilder().version(Version.HTTP_3), ideally with alt-svc discovery so the client upgrades after the server advertises HTTP/3.

Pro What is head-of-line blocking, and how does QUIC address it?

Head-of-line blocking is when the first item in a queue stalls everything behind it. Over TCP, all HTTP/2 streams share one ordered byte stream, so a single dropped segment forces every multiplexed stream to wait for its retransmission — even streams whose data already arrived. QUIC gives each stream its own delivery ordering over UDP, so loss on one stream no longer blocks the others. (Application-level HOL blocking within a single stream still exists.)

Pro How do you consume a WebSocket with the JDK HttpClient, and what does request(n) do?

Call client.newWebSocketBuilder().buildAsync(uri, listener), which performs the opening handshake and completes a CompletableFuture<WebSocket>. Send with sendText/sendBinary; receive through the WebSocket.Listener callbacks (onText, onBinary, onClose). request(n) is back-pressure: the client will only deliver up to n more messages until you ask for more, so a slow consumer is not overrun — analogous to a reactive-streams Subscription.
