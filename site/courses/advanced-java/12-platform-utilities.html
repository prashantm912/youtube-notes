Module: Platform Utilities and FFM

The corners of java.base that touch the world outside the JVM — spawning OS processes, speaking every user's language, generating better randomness, and (new in Java 22) calling C without JNI. Breadth over depth: know what each exists for and where to read the rest.

What this module covers

Four platform utilities that appear in senior interviews and real systems: the Process API for launching and controlling native processes, internationalization (Locale, ResourceBundle, MessageFormat) for locale-correct output, the interface-based RandomGenerator API that superseded bare java.util.Random, and a survey of the Foreign Function & Memory (FFM) API — finalized in Java 22 as the modern, safer replacement for JNI. Depth lives in the linked official docs; this page orients you.

Topics

Process API and ProcessHandle (and scripting)

ProcessBuilder is the supported way to spawn an OS process: set the command, working directory, environment, and I/O redirection, then start(). Java 9 added ProcessHandle, which represents any native process — not just ones you spawned — exposing pid(), a snapshot info(), children()/descendants(), an onExit() CompletableFuture, and destroy()/destroyForcibly(). Together they replace the anaemic legacy Runtime.exec.

Why it matters: build tools, test harnesses, and orchestration code all shell out. "Scripting" also lives here — the single-file source launcher (java Hello.java, with a shebang line) lets Java files run as scripts, and ProcessBuilder is how you drive shell scripts from Java. The classic gotcha (a subprocess deadlocking because its output pipe filled) is a favourite interview trap.

Process p = new ProcessBuilder("git", "status")
    .redirectErrorStream(true)      // merge stderr into stdout
    .start();
String out = new String(p.getInputStream().readAllBytes());
int code = p.waitFor();             // 0 = success
ProcessHandle.current().info().command().ifPresent(System.out::println);

Internationalization: Locale and ResourceBundle

Internationalization (i18n) is designing code so locale-specific data is pulled out, not hard-coded; localization (l10n) is supplying that data per region. A Locale (build it with Locale.of("hi", "IN") — the constructor is deprecated since Java 19) names a language and region. ResourceBundle.getBundle("messages", locale) then loads the best-matching .properties file by walking a candidate chain from most specific to the base bundle, falling back through the default locale.

Why it matters: any product shipping to India serves many languages; hard-coded English strings are an immediate red flag. Knowing the fallback order — and that Java 9+ reads property files as UTF-8 — is the level of detail interviewers probe.

Formatting localized messages: MessageFormat

Concatenating translated fragments breaks the moment a language reorders the sentence. MessageFormat solves this with numbered placeholders and format types: {0}, {1,number,integer}, {2,date,long} — so translators move the whole {0} token and numbers/dates render per the active Locale. Pair it with ResourceBundle: keep the pattern in the properties file, feed the arguments at runtime. For plural agreement, ChoiceFormat selects wording by quantity.

String pattern = "{0} sent you {1,number,integer} new messages.";
String msg = new MessageFormat(pattern, Locale.of("en", "IN"))
    .format(new Object[]{ "Asha", 3 });
// -> "Asha sent you 3 new messages."

Modern randomness: RandomGenerator and java.util.random

JEP 356 (Java 17) introduced java.util.random.RandomGenerator, a uniform interface that Random, SecureRandom, SplittableRandom, and ThreadLocalRandom now all implement — plus a family of better algorithms (the LXM and Xoshiro groups) you obtain by name through RandomGeneratorFactory. Sub-interfaces like SplittableGenerator and JumpableGenerator serve parallel and reproducible-stream use cases. Program to the interface, not to new Random().

Why it matters: legacy Random has a 48-bit seed, is not cryptographically secure, and its shared instance is a concurrency bottleneck. Knowing when to reach for ThreadLocalRandom, a splittable generator, or SecureRandom is a common senior discriminator.

RandomGenerator rng = RandomGenerator.of("L64X128MixRandom");
int roll = rng.nextInt(1, 7);            // origin..bound, 1..6
rng.ints(5, 0, 100).forEach(System.out::println);   // stream of 5

Foreign Function & Memory API (FFM)

The FFM API — finalized in Java 22 by JEP 454 — lets Java call native library functions and access off-heap memory without writing C glue or using the fragile JNI. Its pieces: MemorySegment models a bounded region of memory (on- or off-heap); Arena owns a segment's lifetime and deterministically frees it on close(); Linker turns a native symbol into a MethodHandle (a "downcall"), and the reverse ("upcall") lets native code call back into Java. The jextract tool generates bindings straight from C headers.

Why it matters: FFM gives Java spatial and temporal memory safety (bounds checks plus use-after-free detection) that raw JNI never had, and it is the foundation Project Panama built for high-performance native interop. As a survey topic, know the vocabulary and the "why over JNI" story — the linked tutorials carry the depth.

Linker linker = Linker.nativeLinker();
SymbolLookup stdlib = linker.defaultLookup();
MethodHandle strlen = linker.downcallHandle(
    stdlib.find("strlen").orElseThrow(),
    FunctionDescriptor.of(JAVA_LONG, ADDRESS));
try (Arena arena = Arena.ofConfined()) {          // freed at close
    MemorySegment str = arena.allocateFrom("hello");   // UTF-8 C string
    long len = (long) strlen.invoke(str);              // 5
}

Interview questions

What these platform utilities are actually asked about in Indian interviews — from service-company screens to product-company deep dives:

Beginner How do you run an external OS command from Java?

Use ProcessBuilder: set the command list, optionally the directory/environment/redirects, call start() to get a Process, then waitFor() for the exit code. The old Runtime.getRuntime().exec(...) still works but ProcessBuilder is the supported, configurable API.

Beginner What does ProcessHandle add over Process?

Process represents a subprocess you started; ProcessHandle (Java 9) can reference any native process by PID, and exposes pid(), a snapshot info(), children()/descendants(), onExit() as a CompletableFuture, and destroy()/destroyForcibly(). Get one from a Process via toHandle().

Beginner What is a ResourceBundle and how does Java pick the right file?

A ResourceBundle holds locale-specific key/value data, usually in .properties files sharing a base name (messages.properties, messages_hi_IN.properties). getBundle(base, locale) searches from most specific (language+country) down to the base bundle, using the default locale as a fallback — the first match wins.

Beginner Difference between internationalization and localization?

Internationalization (i18n) is engineering the code so locale-dependent data — text, number/date formats, collation — is externalized rather than hard-coded. Localization (l10n) is the act of providing that data for a specific locale (translations, formats). i18n is done once; l10n is repeated per market.

Beginner Why use MessageFormat instead of string concatenation for user messages?

Because word order and number/date formatting differ by language. MessageFormat uses numbered placeholders ({0}, {1,number}), so a translator can reorder tokens and values render per Locale. Concatenation bakes in English grammar and breaks localization.

Intermediate A subprocess hangs when you read its output. Why, and how do you fix it?

The child's stdout/stderr pipe has a limited OS buffer; if you don't drain it (or you read only one stream while the other fills), the child blocks writing and you block waiting — deadlock. Fixes: redirectErrorStream(true) and read fully, use inheritIO() or file redirects, or drain both streams on separate threads before waitFor().

Intermediate Walk through ResourceBundle's fallback resolution.

For getBundle("msg", Locale.of("hi","IN")) it tries msg_hi_IN, then msg_hi, then the default locale's candidates, then the base msg. Missing keys in a chosen bundle fall through to its parent bundle. You can customize the search with a ResourceBundle.Control. Java 9+ reads properties as UTF-8.

Intermediate What does the RandomGenerator API (JEP 356) improve over java.util.Random?

It defines one interface all generators implement, so you program to RandomGenerator and select algorithms by name via RandomGeneratorFactory. It adds modern algorithm families (LXM, Xoshiro), stream and range methods, and specialized Splittable/Jumpable sub-interfaces for parallelism and reproducibility — none of which the single legacy Random class offered.

Intermediate Is java.util.Random suitable for concurrent use? What are the alternatives?

A shared Random is thread-safe but its internal CAS on the seed becomes a contention hotspot. Use ThreadLocalRandom.current() for independent per-thread streams, or a SplittableGenerator (e.g. L64X128MixRandom) when a fork/join computation needs to split off child generators. For security, use SecureRandom.

Intermediate A server formats everything with Locale.getDefault(). Why is that risky?

The JVM default locale reflects the host, not the user — so the same request formats differently across machines and can silently change numbers, dates, and collation. Pass an explicit Locale everywhere; use Locale.ROOT for machine-facing formats (logs, IDs), and remember Locale.Category splits DISPLAY vs FORMAT defaults.

Pro What is the Foreign Function & Memory API and why prefer it over JNI?

FFM (final in Java 22, JEP 454) calls native functions and accesses off-heap memory from pure Java. Over JNI it removes the hand-written C bridge, adds spatial and temporal memory safety (bounds checks plus use-after-free detection via arenas), models signatures with FunctionDescriptor/Linker, and generates bindings from headers with jextract — safer, faster to develop, and maintainable.

Pro How does Arena give native memory temporal safety?

An Arena owns the lifetime of the MemorySegments it allocates and frees them on close(). A confined arena restricts access to one thread; a shared arena allows many; an auto arena defers to the GC. Any access after the arena closes throws instead of corrupting memory — the guarantee raw JNI/Unsafe pointers never provided.

Pro When would you reach for ProcessHandle.onExit() and descendants()?

onExit() returns a CompletableFuture<ProcessHandle> so you can react asynchronously to termination without a blocking waitFor() thread — chain cleanup or restart logic onto it. descendants() (with destroy()) lets you tear down a whole process tree, avoiding orphaned grandchild processes that a plain destroy() on the parent would leave running.

Pro Contrast a downcall and an upcall in FFM, and what jextract is for.

A downcall is Java calling a native function: Linker.downcallHandle binds a native symbol to a MethodHandle you invoke. An upcall is the reverse — you wrap a Java method as a native function pointer (via Linker.upcallStub) so C code can call back, e.g. a comparator for qsort. jextract parses C headers and generates the boilerplate MethodHandle/layout bindings automatically.
