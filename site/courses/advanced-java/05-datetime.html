Module: The Date and Time API

Dates look simple until a daylight-saving jump, a server in the wrong zone, or a shared SimpleDateFormat quietly corrupts output — java.time was designed to make those bugs unrepresentable, and interviewers probe whether you grasp why.

Knowledge Library → Courses → Advanced Java → The Date and Time API

What this module covers

Java 8 replaced the mutable, not-thread-safe java.util.Date/Calendar with the immutable java.time package. This module orients you on the value types (Instant, the Local family, ZonedDateTime, OffsetDateTime), the two ways to measure elapsed time (Duration and Period), zone and DST handling, formatting and parsing with TemporalAdjusters, and interop with legacy code alongside Clock-based testability. Depth lives in the linked official docs; this page maps the terrain and the traps.

Topics

The core value types: Instant, Local, Zoned, Offset

Every temporal type answers a different question. Instant is a machine timestamp — a point on the UTC time-line as epoch-seconds plus nanoseconds. LocalDate, LocalTime and LocalDateTime are wall-clock values with no zone, so a LocalDateTime is not a real instant until you attach a zone. ZonedDateTime pins a local date-time to a ZoneId (with full DST rules), while OffsetDateTime pins it to a fixed offset from UTC only.

Why it matters: choosing the wrong type is the root cause of most date bugs. A booking time the user typed is a LocalDateTime; the moment an event actually occurred is an Instant or OffsetDateTime. Every class here is immutable and thread-safe, so mutating methods return a new object — a favourite interview gotcha.

Instant now      = Instant.now();                     // UTC timestamp, machine time
LocalDate date   = LocalDate.of(2026, 7, 19);         // a date, no zone
LocalTime time   = LocalTime.of(10, 15, 30);          // wall time, no zone
LocalDateTime dt = LocalDateTime.of(date, time);      // date+time, still no zone
ZonedDateTime z  = dt.atZone(ZoneId.of("Asia/Kolkata")); // now a real instant

Measuring elapsed time: Duration and Period

There are two amount-of-time types because there are two kinds of time. Duration is machine time — seconds and nanoseconds, so it is exact and a "day" is always a fixed number of seconds. Period is calendar time — years, months and days — so its real length depends on the calendar (a month is 28–31 days) and on DST. Adding a Period keeps the wall-clock intent; adding a Duration keeps the elapsed-seconds intent.

Why it matters: the "add one month" versus "add 30 days" distinction, and Duration's ISO-8601 text form, are common interview probes. Reach for ChronoUnit.between() when you want a single unit like whole days.

Period   p = Period.of(1, 2, 3);                // 1 year, 2 months, 3 days
Duration d = Duration.ofHours(8).plusMinutes(30);
LocalDate due = LocalDate.now().plus(p);        // calendar arithmetic
long days = ChronoUnit.DAYS.between(start, end);
String iso = d.toString();                      // "PT8H30M" — ISO-8601 duration

Time zones, ZoneId and DST handling

A ZoneId such as Asia/Kolkata or America/New_York is not a fixed offset — it is a named set of rules that includes daylight-saving transitions. ZonedDateTime applies those rules, which is why it handles the two awkward cases the JVM must resolve: the spring gap (a local time that never happens) and the autumn overlap (a local time that happens twice). India is instructive here: it uses a single half-hour-offset zone with no DST, so a bare numeric offset and a zone id are not interchangeable.

Why it matters: "store the instant, display the zone" is the rule that keeps distributed systems correct. Prefer Instant or OffsetDateTime in the database and convert to a ZonedDateTime only for display; explaining Zoned-vs-Offset is a standard senior screen.

ZoneId kolkata = ZoneId.of("Asia/Kolkata");
ZonedDateTime india = Instant.now().atZone(kolkata);
// spring-gap zone: adding a Duration vs a Period differs across DST
ZonedDateTime z = ZonedDateTime.of(2026, 3, 8, 1, 30, 0, 0,
                                   ZoneId.of("America/New_York"));
z.plus(Duration.ofHours(1)); // elapsed time — offset shifts across the gap
z.plus(Period.ofDays(1));    // calendar time — same wall clock next day

Formatting, parsing and TemporalAdjusters

DateTimeFormatter is the modern, immutable, thread-safe replacement for SimpleDateFormat — which is famously not thread-safe and corrupts output when shared as a static field. Build a formatter from a predefined constant (ISO_LOCAL_DATE), a pattern (note uuuu for the proleptic year, not yyyy), or a localized style, then call format and parse. TemporalAdjusters answers calendar questions declaratively — "last day of the month", "next Monday" — via the with() method.

Why it matters: the thread-safety contrast is one of the most-asked Java 8 date questions, and adjusters turn fiddly loop-based date maths into one readable call.

DateTimeFormatter f = DateTimeFormatter.ofPattern("dd MMM uuuu");
String  s    = LocalDate.now().format(f);          // "19 Jul 2026"
LocalDate back = LocalDate.parse("19 Jul 2026", f);
LocalDate lastDay = date.with(TemporalAdjusters.lastDayOfMonth());
LocalDate nextMon = date.with(TemporalAdjusters.next(DayOfWeek.MONDAY));

Legacy interop and Clock for testability

Real codebases straddle both APIs, so know the bridge methods: Date.toInstant() and Date.from(Instant), Calendar.toInstant(), GregorianCalendar.toZonedDateTime(), and TimeZone.toZoneId(). Converting a legacy Date to a local value always needs an explicit zone, because a Date is really an instant. Separately, Clock is the seam that makes time-dependent code testable: every now() has an overload taking a Clock, and Clock.fixed() freezes time so tests are deterministic.

Why it matters: calling LocalDate.now() with no argument silently binds to the JVM's default zone and to the live clock — both are untestable and a frequent source of "works on my machine" date bugs. Injecting a Clock is the senior answer to "how do you unit-test code that reads the current time?"

// legacy interop
Instant i   = new Date().toInstant();
Date legacy = Date.from(Instant.now());
LocalDateTime ldt = new Date().toInstant()
        .atZone(ZoneId.systemDefault()).toLocalDateTime();

// testable time — inject a Clock instead of calling now() bare
Clock fixed = Clock.fixed(Instant.parse("2026-07-19T00:00:00Z"), ZoneId.of("UTC"));
LocalDate today = LocalDate.now(fixed);  // deterministic in tests

Interview questions

Java 8 date/time is a screening-round staple in Indian interviews — SimpleDateFormat thread-safety, Period-vs-Duration and zone traps recur most:

Beginner What was wrong with java.util.Date and Calendar that java.time fixes?

The legacy classes are mutable and not thread-safe, mix responsibilities, use 0-based months (January is 0), have poor time-zone handling, and a confusing API. java.time is immutable, thread-safe, uses clear separate types, and is built on the ISO-8601 calendar.

Beginner Difference between LocalDate, LocalTime and LocalDateTime?

LocalDate is a date only (year-month-day), LocalTime is a wall-clock time only, and LocalDateTime is both together. None of them carries a zone or offset, so none identifies a unique point on the global time-line.

Beginner What is an Instant, and how does it differ from LocalDateTime?

Instant is a single point on the UTC time-line (epoch-seconds plus nanoseconds) — a machine timestamp. LocalDateTime has no zone or offset, so it is a wall-clock reading that could mean different instants in different places. Convert with instant.atZone(zone) or localDateTime.atZone(zone).toInstant().

Beginner Are java.time objects mutable? What does plusDays(1) return?

They are immutable and thread-safe. plusDays(1) returns a new object and leaves the original unchanged — so date.plusDays(1); on its own does nothing useful; you must assign the result. A classic trick question.

Intermediate Period vs Duration — when do they diverge?

Period is date-based (years, months, days) and calendar-aware, so its real length varies — a month can be 28 to 31 days. Duration is time-based (seconds and nanoseconds) and exact. Adding a Period keeps the wall-clock intent across month lengths and DST; adding a Duration keeps a fixed number of elapsed seconds.

Intermediate ZonedDateTime vs OffsetDateTime — which do you store, and why?

ZonedDateTime carries a ZoneId with full DST rules; OffsetDateTime carries only a fixed offset from UTC. Store OffsetDateTime (or Instant) in the database because it always names the same instant unambiguously; convert to ZonedDateTime only when displaying a time in a particular region.

Intermediate How does ZonedDateTime resolve the DST spring gap and autumn overlap?

A gap is a local time that never occurs (clocks jump forward); ZonedDateTime pushes it forward by the gap so it lands on a valid instant. An overlap is a local time that occurs twice (clocks fall back); by default it picks the earlier offset, and withEarlierOffsetAtOverlap() / withLaterOffsetAtOverlap() let you choose.

Intermediate How do you convert a legacy java.util.Date to a LocalDateTime?

A Date is really an instant, so bridge through one and supply a zone: date.toInstant().atZone(ZoneId.systemDefault()).toLocalDateTime(). The reverse is Date.from(zonedDateTime.toInstant()). The explicit zone is mandatory because a local date-time is meaningless without one.

Intermediate Why is DateTimeFormatter preferred over SimpleDateFormat?

SimpleDateFormat is mutable and not thread-safe — sharing one static instance across threads produces garbled or wrong results, a real production bug. DateTimeFormatter is immutable and thread-safe, so it is safe to keep as a shared constant; it also parses and formats the whole java.time type family.

Pro How do you unit-test code that reads the current time?

Do not call LocalDate.now() or Instant.now() bare — inject a Clock and call the now(clock) overload. In tests pass Clock.fixed(instant, zone) to freeze time (or Clock.offset to shift it), making the assertions deterministic. This is dependency injection applied to time.

Pro What is the hidden risk in LocalDate.now() for an app serving users across zones?

The no-arg now() silently uses the JVM's default zone, so "today" depends on where the server runs and can differ from the user's actual date near midnight. Always pass an explicit ZoneId (or a Clock built with one). The same default-zone trap bites parsing and legacy-Date conversion.

Pro Adding a Period vs a Duration across a DST boundary gives different wall-clock results — explain.

Across a spring-forward day, plus(Period.ofDays(1)) keeps the same wall-clock time next day (calendar semantics), while plus(Duration.ofHours(24)) adds exactly 24 elapsed hours, which may land on a different wall-clock time because that local day was only 23 hours long. Period preserves human intent; Duration preserves physical elapsed time — pick per requirement.
