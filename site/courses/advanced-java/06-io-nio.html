Module: IO and NIO.2 Files

Java ships two file APIs side by side — the original stream classes in java.io and the Java 7 NIO.2 Path/Files layer that quietly replaced them. Interviews probe which one to reach for, and why.

Knowledge Library → Courses → Advanced Java → IO and NIO.2 Files

What this module covers

File handling in Java spans byte and character streams, the buffering that makes them fast, the legacy File handle, and the modern NIO.2 world of Path, the Files utility class, recursive tree walking, and the WatchService for change notification. This module orients you across both APIs so you can read the enterprise codebases that mix them and answer the interview questions that compare them — depth lives in the linked official docs.

Topics

The java.io streams model: bytes vs characters

Every classic I/O class descends from one of four abstract types. InputStream/OutputStream move raw 8-bit bytes — images, serialized objects, any binary payload. Reader/Writer move text, decoding and encoding bytes through a charset so multi-byte encodings like UTF-8 round-trip correctly. The bridge classes InputStreamReader and OutputStreamWriter convert between the two worlds.

Why it matters: choosing the wrong family corrupts data. Reading a JPEG through a Reader mangles it; reading UTF-8 text through a raw byte stream loses the character boundaries. The rule interviewers want back is simple — text takes Reader/Writer, everything else takes byte streams.

// Byte stream — raw bytes (an image, a binary file):
try (InputStream in = new FileInputStream("photo.jpg")) { /* read bytes */ }

// Character stream — text decoded through an explicit charset:
try (Reader r = new FileReader("notes.txt", StandardCharsets.UTF_8)) { /* read chars */ }

Buffering, and the legacy File class

An unbuffered stream calls the operating system on every byte or character — thousands of syscalls to read one file. Wrapping it in BufferedReader/BufferedInputStream (or the output equivalents) pulls a chunk into memory and serves from there, cutting native calls dramatically; BufferedReader also adds readLine(). Buffered output only reaches disk when the buffer fills, on flush(), or on close() — forget to close and the last chunk silently vanishes.

The old java.io.File is a thin, system-dependent handle to a pathname. It still appears in older enterprise code, but its failure reporting is weak (methods return false with no reason) and it has no real symlink or metadata support — which is exactly why NIO.2 exists. File.toPath() bridges legacy code into the modern API.

try (BufferedReader br = new BufferedReader(
         new FileReader("app.log", StandardCharsets.UTF_8))) {
    String line;
    while ((line = br.readLine()) != null)
        process(line);
}

NIO.2: Path and the Files utility class

Path is a lightweight, immutable location object — building it does no I/O at all. Path.of("logs", "app.log") just describes where a file would be; the syntactic operations resolve, relativize and normalize are pure path math. All the real filesystem work goes through Files, a class of static methods that read, write, copy, move, delete, and inspect files while throwing descriptive IOExceptions.

Why it matters: this is the API modern Java code actually uses. One-liners like Files.readString, Files.writeString and Files.readAllLines replace pages of stream boilerplate for small files, and Files exposes atomic moves, attribute views and symlink control the old File never could.

Path log = Path.of("logs", "app.log");          // pure location, no I/O
String text = Files.readString(log);            // whole small file at once
Files.writeString(log.resolveSibling("out.txt"),
                  "done\n", StandardCharsets.UTF_8);

Listing and walking the file tree

To read a single directory level, Files.newDirectoryStream returns a closeable DirectoryStream and even supports glob filtering ("*.java"). To descend recursively, Files.walk hands back a lazy Stream<Path> you filter and map like any stream, while Files.walkFileTree drives a FileVisitor with per-entry callbacks — the most control, and the only clean way to do things like a recursive delete.

Extend SimpleFileVisitor to override just the callbacks you need (visitFile, preVisitDirectory, postVisitDirectory, visitFileFailed). Each returns a FileVisitResult — CONTINUE, SKIP_SUBTREE, SKIP_SIBLINGS or TERMINATE — so you can prune or stop the walk. Remember: the streams returned by walk and list hold a handle and must be closed.

try (Stream<Path> tree = Files.walk(Path.of("src"))) {
    tree.filter(Files::isRegularFile)
        .filter(p -> p.toString().endsWith(".java"))
        .forEach(System.out::println);
}

Watching a directory for changes (WatchService)

The WatchService asks the operating system to notify you when files in a registered directory are created, modified or deleted — no polling loop of your own. You register a directory Path with the event kinds you care about and get back a WatchKey; a consumer loop then blocks on take() (or checks with poll()), drains key.pollEvents(), and calls key.reset() to keep the key active.

Two facts interviewers like: it watches a single directory level, not the whole subtree — to watch a tree you walk it and register each directory — and it can report an OVERFLOW event when changes arrive faster than they are drained. It is the backbone of hot-reload and file-drop deployment features in editors and application servers.

WatchService watcher = FileSystems.getDefault().newWatchService();
Path dir = Path.of("inbox");
dir.register(watcher, ENTRY_CREATE, ENTRY_MODIFY);
WatchKey key = watcher.take();                  // blocks until an event
for (WatchEvent<?> ev : key.pollEvents())
    System.out.println(ev.kind() + " -> " + ev.context());
key.reset();                                     // false means the key is dead

Interview questions

Most-asked IO and NIO.2 questions in Indian interviews — file handling is a reliable core-Java screen, and the File-vs-NIO.2 comparison separates rote answers from senior ones:

Beginner Byte streams vs character streams — when do you use each?

Byte streams (InputStream/OutputStream) move raw 8-bit bytes — images, serialized data, any binary. Character streams (Reader/Writer) decode and encode bytes to text through a charset, handling multi-byte encodings like UTF-8 correctly. Rule: text uses Reader/Writer, everything else uses byte streams; InputStreamReader/OutputStreamWriter bridge the two.

Beginner Why wrap a stream in a BufferedReader or BufferedInputStream?

Unbuffered reads and writes hit the OS on every byte or character — a native call each time. A buffered wrapper transfers a block into memory and serves from it, slashing syscalls. BufferedReader also adds readLine(). Always buffer for line-by-line or byte-by-byte work.

Beginner Difference between java.io.File and java.nio.file.Path?

File is the legacy handle: methods report failure by returning false with no cause, and symlink/metadata support is poor. Path (Java 7, NIO.2) is a lightweight location object used with the Files utility class, which throws descriptive IOExceptions and supports symlinks, attributes, atomic moves and directory streams. New code uses Path/Files; File.toPath() bridges the old API.

Beginner Simplest way to read and write a small text file in modern Java?

Files.readString(path) or Files.readAllLines(path) to read; Files.writeString(path, text) or Files.write(path, bytes) to write — one line each. For large files, stream instead of loading everything into memory.

Intermediate Why use try-with-resources for I/O, and what does flush() do?

Streams hold OS file handles; leaking them exhausts descriptors. try-with-resources auto-closes in reverse order, even on exception. close() implicitly flushes buffered output; flush() forces buffered bytes out early. Forgetting to close a buffered writer can silently drop the last un-flushed chunk.

Intermediate DirectoryStream vs Files.list vs Files.walk vs Files.walkFileTree?

newDirectoryStream and Files.list read one directory level (non-recursive). Files.walk returns a lazy Stream<Path> that descends recursively to a depth. Files.walkFileTree drives a FileVisitor with callbacks — the most control, needed for recursive delete or per-entry error handling. All of these hold handles and must be closed.

Intermediate What are FileVisitor / SimpleFileVisitor and the FileVisitResult values?

FileVisitor has four callbacks fired during walkFileTree: preVisitDirectory, visitFile, visitFileFailed, postVisitDirectory. SimpleFileVisitor is an adapter you subclass to override only what you need. Each callback returns a FileVisitResult — CONTINUE, SKIP_SUBTREE, SKIP_SIBLINGS or TERMINATE — to prune or stop the walk. Classic use: recursive delete removes files in visitFile and directories in postVisitDirectory.

Intermediate How does the WatchService work?

Create it from the FileSystem, register a directory Path with event kinds (ENTRY_CREATE/MODIFY/DELETE) and receive a WatchKey. Loop: take() blocks (or poll() returns immediately) for a signalled key, read key.pollEvents(), then call key.reset() to keep watching — reset() returning false means the key is dead. It watches one directory level and reports OVERFLOW if events are lost.

Intermediate resolve vs relativize vs normalize on Path?

resolve(other) joins a base path with a child. relativize(other) computes the relative path from one to another — the inverse of resolve. normalize() strips redundant . and .. segments purely syntactically. None of them touch the filesystem; they are string-level path math on immutable objects.

Pro What is the default-charset pitfall when reading files, and how do you avoid it?

Before Java 18, FileReader/FileWriter and many APIs used the platform default charset, so a UTF-8 file written on Linux could garble on a Windows box defaulting to windows-1252 — a bug that surfaces only in production. Always pass an explicit Charset such as StandardCharsets.UTF_8. Java 18 (JEP 400) made UTF-8 the default, but being explicit remains the safe habit.

Pro Files.lines() and Files.walk() return a Stream — what is the trap?

Those streams are backed by an open file or directory handle, so they must be closed — Stream implements AutoCloseable. Use them in try-with-resources: try (var s = Files.lines(p)) { ... }. A bare Files.lines(p).forEach(...) leaks the descriptor. That is the difference from an ordinary collection stream, and a frequent senior-level gotcha.

Pro Beyond convenience, why did NIO.2 replace the old File API?

File reports failure as a bare false, cannot follow or detect symlinks properly, offers no atomic operations, and has no bulk or attribute APIs. NIO.2 gives descriptive exceptions, symlink control (LinkOption.NOFOLLOW_LINKS), atomic moves (StandardCopyOption.ATOMIC_MOVE), attribute views (PosixFileAttributes), pluggable FileSystemProviders (zip and in-memory filesystems), and the WatchService. For raw throughput it pairs with channels and buffers — the next module.
