Module: Reflection

Reflection lets code inspect and call other code it never imported — the engine under Spring, Hibernate and JUnit — and "how does the framework actually do that?" is a question every serious Java interview eventually reaches.

Knowledge Library → Courses → Advanced Java → Reflection

What this module covers

Reflection is the java.lang.reflect API for examining and manipulating types at runtime without knowing their names at compile time: getting a Class, enumerating and invoking its Method, Field and Constructor members, suppressing access checks with setAccessible under the constraints the module system now imposes, and building runtime stand-ins with dynamic proxies. It closes with java.lang.invoke — MethodHandles and VarHandles — the faster, strongly-typed successor the JDK itself uses.

Topics

Class objects: reflection's entry point

Every reflective operation starts from a java.lang.Class instance — the runtime metadata for a type. You obtain one three ways: obj.getClass() from a live instance, Type.class as a compile-time literal, or Class.forName("com.acme.User") to load by name at runtime (the framework path, where the type is a string in a config file). From a Class you enumerate members with getMethods()/getDeclaredMethods(), getFields()/getDeclaredFields(), getConstructors()/getDeclaredConstructors(), plus metadata like getName, getModifiers, getSuperclass and getInterfaces.

Why it matters: the "declared" versus plain distinction is a constant point of confusion and a favourite probe — getDeclaredMethods() returns every method written in this class regardless of access level but excludes inherited ones, while getMethods() returns only public methods including inherited. Knowing which to reach for is the difference between an NPE and working code.

Class<?> c = Class.forName("com.acme.User");   // load by name at runtime
for (Method m : c.getDeclaredMethods()) {        // this class only, any access
    System.out.println(m.getName());
}

Dynamic invocation and instantiation

Once you hold a Method, Field or Constructor, you can act through it with no compile-time reference to the type. Method.invoke(target, args) calls the method (pass null as the target for a static one); Field.get/set read and write values; Constructor.newInstance(args) builds an object — the modern replacement for the deprecated Class.newInstance(), which could only use a no-arg constructor and swallowed its checked exceptions.

Why it matters: this is literally how DI containers wire beans and how JUnit calls your @Test methods. The interview trap is exceptions — if the invoked code throws, invoke wraps it in an InvocationTargetException, and you recover the real cause with getCause().

Constructor<User> ctor = User.class.getDeclaredConstructor(String.class);
User u = ctor.newInstance("Asha");             // build without 'new'
Method setId = User.class.getMethod("setId", long.class);
setId.invoke(u, 42L);                          // call without a direct reference

setAccessible and the module system

By default reflection honours Java access control, so touching a private member throws IllegalAccessException. AccessibleObject.setAccessible(true) suppresses that check — "deep reflection", the mechanism behind serialization, ORMs and dependency injection. Since Java 9 a second gate applies: deep reflection into a package succeeds only if that package is open to your module (an open module, an opens directive, or an --add-opens flag on the launch line); otherwise you get an InaccessibleObjectException. Java 9 also added the gentler trySetAccessible() (returns false instead of throwing) and canAccess().

Why it matters: "my library ran fine on Java 8 and now throws InaccessibleObjectException on Java 17" is a genuine migration story, and explaining the strong-encapsulation change separates people who read the release notes from those who did not.

Field f = User.class.getDeclaredField("password");
f.setAccessible(true);    // needs the package 'open' on Java 9+
Object value = f.get(u);  // otherwise throws InaccessibleObjectException

Dynamic proxies

A dynamic proxy is a class synthesised at runtime that implements a given set of interfaces and routes every call to one place: InvocationHandler.invoke(proxy, method, args). You create it with Proxy.newProxyInstance(loader, interfaces, handler). This is how Spring AOP, declarative transactions, retry/logging decorators and mocking libraries interpose behaviour without you hand-writing wrapper classes. The defining limitation: JDK proxies can implement interfaces only — the generated class already extends Proxy, and Java's single inheritance leaves no room to also extend a concrete class, so frameworks fall back to subclass-generating tools like CGLIB or ByteBuddy when they must proxy a class.

Foo proxy = (Foo) Proxy.newProxyInstance(
    Foo.class.getClassLoader(),
    new Class<?>[]{ Foo.class },
    (p, method, args) -> { audit(method); return method.invoke(target, args); });

MethodHandles and VarHandles

The java.lang.invoke package — MethodHandle since Java 7, VarHandle since Java 9 — is the modern, faster alternative to core reflection. A MethodHandle is a directly-executable, strongly-typed reference to a method or field, obtained through a MethodHandles.Lookup that captures the caller's access rights once at lookup time, so the per-call permission checks of Method.invoke disappear; invokeExact demands an exact type match. This machinery underpins lambda metafactories, invokedynamic and string concatenation.

A VarHandle gives typed access to a field or array element with explicit memory-ordering modes — plain, volatile, acquire/release, and atomic compare-and-set — and is the supported, safe replacement for the internal sun.misc.Unsafe.

MethodHandles.Lookup lk = MethodHandles.lookup();
MethodHandle mh = lk.findVirtual(String.class, "length",
        MethodType.methodType(int.class));
int n = (int) mh.invokeExact("hello");   // 5 — exact types required

Interview questions

Reflection questions asked in Indian interviews, from service-company screens to product-company framework deep dives — the "how does Spring/Hibernate work under the hood?" line always lands here:

Beginner What is reflection and why does Java provide it?

The java.lang.reflect API for inspecting and manipulating classes, methods, fields and constructors at runtime, even when their names are unknown at compile time. It exists so frameworks — DI containers, ORMs, serializers, test runners — can work generically against types they never imported.

Beginner What are the ways to obtain a Class object?

Three: instance.getClass() at runtime, the Type.class literal at compile time, and Class.forName("fully.qualified.Name") to load by string name. The last is the framework path — the type comes from a config file or annotation, not source code.

Beginner getMethods() vs getDeclaredMethods()?

getMethods() returns public methods including inherited ones. getDeclaredMethods() returns every method declared in this class — public, protected, package-private and private — but nothing inherited. The same declared-vs-plain split applies to fields and constructors.

Beginner How do you create an object with reflection?

Get the constructor and call newInstance(args): clazz.getDeclaredConstructor(String.class).newInstance("x"). The old Class.newInstance() is deprecated — it only handled a no-arg constructor and wrapped its checked exceptions unhelpfully.

Intermediate How do you invoke a method reflectively, and what exception should you expect?

method.invoke(target, args) (target null for static). If the invoked method itself throws, invoke wraps that in an InvocationTargetException; call getCause() to reach the actual exception. Bad arguments or access give IllegalArgumentException/IllegalAccessException.

Intermediate How do you read or modify a private field, and what does setAccessible do?

getDeclaredField(name), then setAccessible(true) to suppress the access check, then get/set. setAccessible turns off Java language access control for that reflected member — this is "deep reflection", used heavily by serialization and ORMs.

Intermediate What changed for reflection with the Java module system?

Since Java 9, setAccessible(true) into another module's package only works if that package is open — via an open module, an opens directive, or an --add-opens launch flag. Otherwise you get InaccessibleObjectException. Java 9 also added trySetAccessible() and canAccess(). This strong encapsulation is a common Java 8-to-17 migration break.

Intermediate Where have you seen reflection used in frameworks?

Spring instantiates beans and injects dependencies reflectively; Hibernate maps rows to entity fields; Jackson and Gson read/write fields for JSON; JUnit discovers and calls @Test methods; validators and mappers read annotations via reflection. Naming a couple concretely is what the interviewer wants.

Intermediate What is a dynamic proxy and how do you build one?

A runtime-generated class implementing chosen interfaces, routing every call to an InvocationHandler. Build it with Proxy.newProxyInstance(classLoader, interfaces, handler). It is the basis of Spring AOP, declarative transactions and mocking — cross-cutting behaviour with no hand-written wrappers.

Pro What are the drawbacks of reflection?

Slower than direct calls (access checks, no inlining in the naive path); loses compile-time type safety, so errors surface at runtime; breaks refactoring and static analysis since usages are strings; can violate encapsulation and raise security concerns; and now runs into module-system restrictions. Use it in frameworks and infrastructure, not hot application code.

Pro MethodHandle vs reflection's Method — what is the difference?

A MethodHandle (java.lang.invoke) is a strongly-typed, directly-executable reference resolved through a MethodHandles.Lookup that captures access rights once at lookup time — so no per-call access check — and it optimizes closer to a direct call; invokeExact requires an exact signature match. It is what the JVM uses for lambdas and invokedynamic, and is generally faster than Method.invoke.

Pro What is a VarHandle and why prefer it over reflection or Unsafe?

A typed handle to a variable — instance field, static field, or array element — supporting explicit memory-ordering access modes: plain, volatile, acquire/release, and atomic compare-and-set. It is the supported, safe replacement for the internal sun.misc.Unsafe, giving low-level concurrency primitives without reflection's overhead or Unsafe's fragility.

Pro Why can a JDK dynamic proxy only proxy interfaces, and what do you use for classes?

The generated proxy class already extends java.lang.reflect.Proxy, and Java allows only single inheritance, so it cannot also extend a target class — it can only implement interfaces. To proxy a concrete class you generate a subclass at runtime with a bytecode library like CGLIB or ByteBuddy (Spring picks the JDK proxy when a bean has an interface and CGLIB otherwise).
