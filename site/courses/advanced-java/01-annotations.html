Module: Annotations

Every framework you will touch — Spring, JUnit, Jackson, Lombok — runs on annotations, yet most developers only ever consume them; this module is about the metadata layer itself: reading it, retaining it, and building your own.

Knowledge Library → Courses → Advanced Java → Annotations

What this module covers

Annotations are typed metadata attached to declarations (and, since Java 8, to type uses). This module covers the built-in annotations the compiler understands, the meta-annotations that configure your own — especially @Retention and @Target — and the two ways annotations get consumed: runtime reflection (how Spring and JUnit work) and compile-time annotation processing (how Lombok and MapStruct work). It is the foundation for the Reflection module that follows.

Topics

Built-in annotations, meta-annotations, retention/targets

The annotations the compiler itself acts on live in java.lang: @Override (fails compilation if the method does not actually override anything — a typo catcher), @Deprecated (marks an API slated for removal; Java 9 added since and forRemoval), @SuppressWarnings (silences named warning categories like "unchecked"), @FunctionalInterface (enforces a single abstract method), and @SafeVarargs. These have no runtime footprint — they exist to help the compiler.

Why it matters: meta-annotations are what let you build annotations that do more. @Retention decides whether an annotation survives to SOURCE, CLASS (the default), or RUNTIME — and only RUNTIME-retained annotations are visible to reflection, which is the single most-tested fact here. @Target restricts which ElementType locations (METHOD, FIELD, TYPE, PARAMETER, TYPE_USE…) accept the annotation, turning misuse into a compile error. @Documented, @Inherited, and @Repeatable round out the set.

@Retention(RetentionPolicy.RUNTIME)   // survives to reflection
@Target(ElementType.METHOD)           // legal only on methods
@interface Timed { }

@Override                             // SOURCE-retained: compiler-only, gone after javac
public String toString() { return "User"; }

Custom annotations, annotation processing (APT) awareness

You declare an annotation with @interface; its elements look like methods, may carry default values and array types, and a single element named value enables the shorthand @Tag("x"). Decorate the declaration with meta-annotations to set its retention and legal targets. To act on a RUNTIME-retained annotation you use reflection — isAnnotationPresent, getAnnotation on Class/Method/Field — which is exactly how Spring wires beans, JUnit finds tests, and Jackson maps JSON.

Why it matters: the second, senior-level path is compile-time annotation processing (APT, JSR 269). A processor extends AbstractProcessor, is invoked by javac across rounds, inspects annotated elements via RoundEnvironment, and generates new source files with Filer or reports errors through Messager — all with zero runtime cost. Lombok, MapStruct, Dagger, and Immutables are built this way. Being able to contrast runtime reflection against compile-time processing is what separates a mid-level answer from a senior one.

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
@interface Entity {
    String table() default "";        // element with a default
}

@Entity(table = "users")
class User { }

boolean tagged = User.class.isAnnotationPresent(Entity.class);
String table  = User.class.getAnnotation(Entity.class).table();

Interview questions

Annotations are a staple of Indian interviews — screening rounds probe the built-ins, and Spring-heavy product rounds push into retention, reflection, and processing:

Beginner What is an annotation in Java?

Typed metadata attached to a declaration (or, since Java 8, a type use) with the @ symbol. It has no direct effect on the code it marks — its value comes from being read by the compiler, by build-time tools, or at runtime via reflection.

Beginner Name the built-in annotations in java.lang and what each does.

@Override (verifies a method really overrides a supertype method), @Deprecated (marks an API as obsolete), @SuppressWarnings (silences named compiler warnings), @FunctionalInterface (asserts a single abstract method), and @SafeVarargs (suppresses unsafe-varargs warnings).

Beginner Is @Override required, and what does it actually buy you?

Not required — code compiles without it. But it makes the compiler verify the method overrides something; a misspelled name or wrong signature then becomes a compile error instead of a silently new, never-called method. It is a cheap correctness guard.

Beginner What are the three retention policies?

SOURCE (discarded by the compiler — e.g. @Override), CLASS (kept in the .class file but not loaded at runtime — the default), and RUNTIME (retained and visible to reflection). Only RUNTIME annotations can be read by a running program.

Intermediate How do you declare a custom annotation?

Use @interface. Its elements are declared like no-arg methods and may have default values and array types; a lone element named value allows the @Name("x") shorthand. Add meta-annotations (@Retention, @Target) to control how and where it applies.

Intermediate Why must framework annotations like @Autowired use @Retention(RUNTIME)?

Because Spring reads them by reflection while the application runs. Reflection only sees RUNTIME-retained annotations; with the default CLASS retention, getAnnotation returns null and the framework would never detect the marker. This is the most common annotation gotcha.

Intermediate What does @Target do, and what happens if you misapply an annotation?

@Target lists the ElementType locations (METHOD, FIELD, TYPE, PARAMETER, TYPE_USE…) where the annotation is legal. Applying it anywhere else is a compile error. With no @Target, an annotation may appear on almost any declaration.

Intermediate How do you read an annotation at runtime?

Via the Reflection API on the reflective element: clazz.isAnnotationPresent(Entity.class) to test, method.getAnnotation(Timed.class) to fetch the instance and read its elements. This requires RUNTIME retention.

Intermediate What does @Inherited change, and what are its limits?

It makes a class-level annotation show up on subclasses when queried by reflection, so a subclass reports its parent's annotation. It applies only to class inheritance — not to annotations on methods, fields, or interfaces implemented by the class.

Intermediate What is @Repeatable and how does it work under the hood?

It lets the same annotation appear more than once on one element (Java 8+). You declare a companion container annotation whose value is an array of the repeatable type and point @Repeatable at it; the compiler bundles repeats into that container, which getAnnotationsByType unwraps transparently.

Pro Runtime reflection vs compile-time annotation processing — when do you reach for each?

Reflection (Spring, JUnit, Jackson) is flexible and dynamic but needs RUNTIME retention and pays a startup/scan and per-access cost. Annotation processing (APT, JSR 269) runs inside javac, generates code or validates at build time, and adds zero runtime overhead — the approach behind Lombok, MapStruct, and Dagger. Choose processing for performance-critical or code-generation cases, reflection for dynamic, configuration-driven behavior.

Pro What is annotation processing (APT) and what are its key API pieces?

A compiler plugin mechanism: a class extends AbstractProcessor, declares @SupportedAnnotationTypes/@SupportedSourceVersion, and its process method receives each round's annotated elements via RoundEnvironment. It emits new source files through Filer and reports errors/warnings through Messager. The processor is registered on the compiler's path (via META-INF/services or @AutoService) and runs across multiple rounds until no new files are produced.
