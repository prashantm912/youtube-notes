Module: Common Pitfalls

These are the bugs that sail through code review and surface in production — Integer identity that flips at 128, money stored in doubles, an equals() with no hashCode() — and they are exactly what a sharp interviewer reaches for first.

Knowledge Library → Courses → Advanced Java → Common Pitfalls

What this module covers

This is a survey of the language's sharp edges: reference-vs-value comparison and the Integer cache, the equals/hashCode contract and the hash-collection failures it prevents, the hidden cost of autoboxing, why money never belongs in a double, quadratic string building in loops, the traps in the legacy date/calendar classes, and why finalizers are being retired. Each one is a favourite screening question in Indian interviews because the wrong answer is subtle and the right answer shows real understanding.

Topics

== vs equals, and the Integer cache

== compares references for objects (are these the same object?) and values for primitives; equals() compares logical value once a class overrides it. Confusing the two is the root of most "why is this false?" bugs, because two distinct String or Integer objects with the same value are equals() but not ==.

Why it matters: the classic trap is the Integer cache. Autoboxing routes through Integer.valueOf, which caches instances for -128 to 127, so Integer a = 127, b = 127 gives a == b true, while 128 yields two objects and == false. The Java Language Specification guarantees the cached range; anything outside it is implementation-defined. Rule: compare wrapper values with equals(), never ==.

Integer a = 127, b = 127;
Integer c = 128, d = 128;
a == b;        // true  -> both from the cache (-128..127)
c == d;        // false -> two distinct objects
c.equals(d);   // true  -> value equality, the correct comparison

equals/hashCode contract traps

Override one and you must override the other. The contract: equal objects must return equal hash codes, and results must be consistent while the object's identifying fields do not change. Break it and hash collections quietly misbehave — a HashMap.get misses a key that is equals() to the stored one because the two land in different buckets.

Why it matters: this is a top interview probe. The follow-ups are always the same — override both together, base both on the same immutable fields, and never mutate a field that participates in hashCode() after the object is used as a map key or set member. Records generate a correct value-based pair for you, removing the hand-written boilerplate that so often gets it wrong.

Autoboxing costs and unboxing NullPointerExceptions

Autoboxing converts primitives to wrappers and back automatically, which reads cleanly but hides two hazards. First, unboxing a null wrapper throws NullPointerException — int n = map.get(key) blows up when the key is absent because null cannot become an int. Second, boxing allocates: a wrapper accumulator inside a tight loop creates a fresh object every iteration.

Why it matters: the NPE is a real production surprise (ternaries with mixed Integer/int branches unbox silently), and the allocation cost is a favourite "spot the bug" question. Rule: use primitives for arithmetic and loop counters; reach for wrappers only when nullability or collections genuinely require them.

Map<String,Integer> counts = ...;
int n = counts.get("missing");     // NPE: null unboxed to int
Long sum = 0L;
for (long x : nums) sum += x;      // unbox + box every iteration -> garbage
long fast = 0L;
for (long x : nums) fast += x;     // primitive: no allocation

BigDecimal vs double for money

Binary floating point cannot represent most decimal fractions exactly, so 0.1 + 0.2 is not 0.3 and rounded totals drift by cents. Never store or compute money in double/float. Use BigDecimal — arbitrary precision with an explicit scale and rounding mode you control.

Why it matters: two follow-ups always appear. Construct from a String (or BigDecimal.valueOf), never new BigDecimal(0.1) — the double constructor captures the full binary error. And compare amounts with compareTo, not equals: equals considers scale, so 1.0 and 1.00 are unequal, while compareTo returns 0 as expected.

new BigDecimal(0.1);            // 0.10000000000000000555... captures binary error
new BigDecimal("0.1");          // exactly 0.1 -> prefer the String constructor
BigDecimal.valueOf(0.1);        // exactly 0.1 -> the canonical double bridge
new BigDecimal("1.0").equals(new BigDecimal("1.00"));      // false (scale differs)
new BigDecimal("1.0").compareTo(new BigDecimal("1.00"));   // 0 (same value)

String concatenation in loops

String is immutable, so s += part inside a loop allocates a brand-new string and copies everything each pass — O(n²) work and heavy garbage. Build with StringBuilder (or String.join/streams for the simple cases), which appends into one growable buffer in linear time.

Why it matters: the compiler rewrites a single a + b + c expression efficiently, but it cannot fuse concatenation spread across loop iterations — that is on you. It is a standard performance question, usually paired with "String vs StringBuilder vs StringBuffer" (StringBuffer is the synchronized, rarely-needed sibling).

String s = "";
for (String p : parts) s += p;             // O(n^2): new String + copy each pass
StringBuilder sb = new StringBuilder();
for (String p : parts) sb.append(p);       // O(n): one reused buffer
String joined = String.join(",", parts);   // clearest for a simple join

Date and time bugs (legacy Date/Calendar)

The legacy classes are a bug factory: java.util.Date is mutable and mostly deprecated, Calendar months are zero-based (January is 0), and SimpleDateFormat is not thread-safe — sharing one across threads corrupts output silently. Prefer the java.time package (LocalDate, LocalDateTime, Instant, ZonedDateTime): immutable, thread-safe, and explicit about time zones.

Why it matters: mutability means a returned Date can be changed by a caller, so old code needs defensive copies; java.time values are immutable, so operations like plusMonths return a new object and clamp sensibly. Knowing why the migration happened — not just the new method names — is what interviewers want.

LocalDate d = LocalDate.of(2026, 1, 31);
LocalDate next = d.plusMonths(1);   // 2026-02-28, immutable -> new value returned
// legacy traps: Calendar.set(2026, 1, 31) is FEBRUARY (months 0-based),
// and one shared SimpleDateFormat is unsafe across threads

Finalizers and resource cleanup

Object.finalize() was meant to reclaim resources before garbage collection, but it is unreliable (no guarantee it ever runs, or when), slow, and a source of resurrection and security bugs — so it is deprecated for removal. Never rely on it to close files, sockets, or connections.

Why it matters: the right patterns are deterministic. Implement AutoCloseable and use try-with-resources so cleanup runs at a defined point; for the rare native-handle case, register a Cleaner as a safety net rather than a primary mechanism. Interviewers ask this to check that you reach for try-with-resources, not finalizers.

Interview questions

The pitfalls Indian interviewers probe most — reference vs value, the Integer cache, and money in doubles headline nearly every core-Java screen:

Beginner Difference between == and equals() in Java?

== tests reference identity for objects (same object in memory) and raw value for primitives. equals() tests logical equality as the class defines it. Two different String or Integer objects holding the same value are equals() but not ==. Always compare object contents with equals().

Beginner Integer a = 127, b = 127 gives a == b true, but 128 gives false. Why?

Autoboxing calls Integer.valueOf, which caches instances for -128 to 127. In range, both references point to the same cached object, so == is true; outside it, two new objects are created and == is false. The values are always equals(). This is the single most-asked pitfall question.

Beginner Why must you override hashCode() whenever you override equals()?

The contract requires equal objects to have equal hash codes. Override equals() alone and two "equal" objects can produce different hashes, so a HashMap/HashSet puts them in different buckets and lookups fail. Override both, from the same fields.

Beginner Why should money never be stored in a double?

Binary floating point cannot represent most decimal fractions exactly, so sums and rounding drift (0.1 + 0.2 is not 0.3). Use BigDecimal with an explicit scale and rounding mode — or an integer count of the smallest currency unit (paise/cents).

Intermediate What is the Integer cache and why does it exist?

A pool of pre-built Integer objects for small, frequently used values (always -128 to 127; the upper bound is tunable via -XX:AutoBoxCacheMax). Integer.valueOf returns cached instances to save allocation. The visible side effect is that == "works" in range and breaks outside it — which is exactly why you never use == on wrappers.

Intermediate new BigDecimal(0.1) vs new BigDecimal("0.1") — what is the difference?

The double constructor takes the already-imprecise binary value, so new BigDecimal(0.1) is 0.1000000000000000055… The String constructor is exact: new BigDecimal("0.1") equals 0.1 precisely. Prefer the String constructor or BigDecimal.valueOf(double), which uses the canonical Double.toString.

Intermediate Why can BigDecimal.equals() surprise you, and what should you use?

equals() compares value and scale, so new BigDecimal("1.0") and new BigDecimal("1.00") are not equal. To compare numeric value, use compareTo(...) == 0. This also means BigDecimal is a risky HashSet/HashMap key unless scales are normalised first.

Intermediate How can autoboxing cause a NullPointerException?

Unboxing a null wrapper fails: int n = map.get(key) throws NPE when the key is absent because null cannot convert to int. Mixed Integer/int ternaries unbox silently and hit the same trap. Keep the type Integer and null-check, or supply a default.

Intermediate Why is building a String in a loop with += a problem, and the fix?

String is immutable, so each s += part allocates a new string and copies all prior characters — O(n²) time and heavy garbage. Use a StringBuilder and append (one growable buffer, linear time), or String.join/a stream collector for simple joins. The compiler optimises a single + expression but not concatenation across iterations.

Intermediate What happens if you mutate an object after using it as a HashMap key?

If you change a field that feeds hashCode(), the object now hashes to a different bucket than where it was stored, so get/containsKey can no longer find it — a silent leak. Use immutable keys, or never mutate the identifying fields once an object is a key or set member.

Pro Show how autoboxing quietly hurts performance in a hot loop.

Declare the accumulator as a wrapper — Long sum = 0L; for (long x : nums) sum += x; — and every iteration unboxes sum, adds, then boxes a fresh Long, generating one garbage object per pass. Declaring long sum removes all of it. The same tax hits Integer/Double keys and accumulators; prefer primitives and primitive streams (IntStream/LongStream) on hot paths.

Pro Why are finalizers discouraged, and what replaces them?

finalize() gives no guarantee it runs, or when; it delays reclamation, can resurrect objects, and has caused security bugs — so it is deprecated for removal (JEP 421). Replace it with deterministic cleanup: implement AutoCloseable and use try-with-resources; for native handles, register a Cleaner only as a backstop, not the primary release mechanism.

Pro Name the classic legacy Date/Calendar bugs and how java.time fixes them.

java.util.Date is mutable (callers can change a returned value — needs defensive copies) and largely deprecated; Calendar months are zero-based, so month 1 is February; SimpleDateFormat is not thread-safe and corrupts output when shared. java.time types are immutable and thread-safe, months are one-based, DateTimeFormatter is thread-safe, and time zones are explicit — removing the whole class of bugs.
