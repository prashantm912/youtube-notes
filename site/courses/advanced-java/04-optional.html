Module: Optional

Optional puts "there might be no value here" into a method's type instead of its Javadoc. This module orients you on each concept and points to the best material for complete study — official documentation first.

What this module covers

Optional (Java 8+) is a container return type for "maybe one result": creation factories, the map/flatMap/filter pipeline, the orElse unwrapping family, and — most tested in interviews — the anti-patterns that defeat its purpose. It pairs naturally with Streams and shows up all over Spring Data (findById returns Optional).

Topics

What Optional is for, and creating one

A method returning Optional<User> makes absence part of the contract — callers must unwrap, the compiler helps, and the null-return convention dies. Three factories: Optional.of (asserts non-null, fails fast), Optional.ofNullable (bridges null-returning APIs), Optional.empty().

Why it matters: the designers' stated intent is return types only — not fields, not parameters. Knowing the intent, not just the API, is what separates answers in interviews.

Optional<String> some  = Optional.of("hello");        // must be non-null
Optional<String> maybe = Optional.ofNullable(lookup); // null -> empty
Optional<String> none  = Optional.empty();

Chaining: map, flatMap, filter, or, stream

The value of Optional is transforming inside the container so the empty case needs no if-statements: map transforms, flatMap flattens Optional-returning mappers (avoiding Optional<Optional<T>>), filter keeps values passing a predicate, or supplies a lazy fallback Optional, and stream() bridges into Stream pipelines.

String city = findUser(id)              // Optional<User>
    .flatMap(User::address)             // Optional-returning accessor
    .map(Address::city)
    .filter(c -> !c.isBlank())
    .orElse("UNKNOWN");

Unwrapping: the orElse family

orElse(x) evaluates its argument always — even when the value is present; orElseGet(supplier) runs only on the empty path. With an expensive default, orElse is a real production bug and the single most-asked Optional interview question. Round out with orElseThrow() (the honest replacement for bare get()), ifPresent, and ifPresentOrElse.

opt.orElse(cheapConstant);            // fine
opt.orElseGet(() -> expensiveCall()); // lazy — only when empty
opt.orElseThrow(NotFound::new);       // empty becomes your exception

Anti-patterns and best practices

The API is easy; discipline is the skill. The big five: isPresent()+get() (null-checking with extra steps — chain instead), Optional fields/parameters (not Serializable, allocation cost, caller ceremony — return types only), returning null from an Optional-returning method (never), ofNullable(x).orElse(y) as a dressed-up ternary, and Optional<List> (return an empty list instead).

Interview questions

Most-asked Optional questions in Indian interviews — Java 8 features remain a screening staple:

What is Optional and what problem does it solve?

A container return type (Java 8) representing a possibly-absent value. It replaces the null-return convention: absence becomes part of the method signature, callers must handle it, reducing NullPointerExceptions.

Optional.of vs Optional.ofNullable?

of throws NullPointerException immediately for null — an assertion the value exists. ofNullable converts null to empty — the bridge from null-returning APIs.

How do you provide a default value when Optional is empty?

orElse(defaultValue), orElseGet(supplier) for computed defaults, orElseThrow() to convert absence into an exception, or or(() -> otherOptional) to fall back to another Optional.

Why is calling get() directly discouraged?

On empty it throws NoSuchElementException — get() without a presence check is null-dereferencing with a new name. orElseThrow() is the semantically honest synonym; chaining avoids the unwrap entirely.

orElse vs orElseGet — the difference that matters?

orElse(x) evaluates x eagerly on every call, present or not; orElseGet invokes its supplier only when empty. With expensive defaults (DB call, construction) orElse causes real production waste. Rule: constant → orElse, computed → orElseGet. The single most-asked Optional question.

map vs flatMap on Optional?

Both transform a present value. map wraps the result in Optional — so an Optional-returning mapper yields Optional<Optional<T>>; flatMap flattens exactly that case. Same distinction as Stream's map/flatMap.

Why should Optional not be used for fields or method parameters?

Not Serializable (breaks serializable classes), allocation per access, and as a parameter it forces every caller to wrap while a null Optional reference is still possible — worst of both worlds. Designers' intent: return types only. Idiom: nullable field internally, Optional-returning getter at the boundary.

Where do you meet Optional in Spring?

JpaRepository.findById returns Optional — the canonical enterprise usage: repo.findById(id).orElseThrow(() -> new EntityNotFoundException(...)). Derived query methods can also declare Optional returns; @Autowired(required=false) alternatives use Optional injection.

How does Optional.stream() help with a list of Optionals?

It turns each Optional into a 0-or-1 element stream, so list.stream().flatMap(Optional::stream) keeps present values and drops empties in one step (Java 9+). Pre-9 idiom was filter(isPresent).map(get).

Is Optional.ofNullable(x).orElse(y) good code?

No — it's a ternary with an extra allocation: x != null ? x : y, or better Objects.requireNonNullElse(x, y). Optional earns its cost as a return-type contract, not as an inline null-check DSL.

Why is Optional<List<T>> considered bad design?

Two empty states (empty Optional vs present-but-empty list) with no clear meaning difference. Collections already model absence as emptiness — return an empty collection. Same logic applies to Optional<Optional<T>>.

What happens if a method declared to return Optional returns null — and how do you prevent it?

Callers get NPE on the first method call — the contract is betrayed at its own game; this is the one unforgivable Optional bug. Prevention: static analysis (@NonNull on Optional returns, SpotBugs/NullAway), code review rule, and tests asserting notNull on Optional-returning paths.
