Module: Bean Definition and Scanning

Spring registers a bean two ways — annotate a class and let the container discover it during a scan, or hand-write a factory method inside a configuration class. Knowing which path fits, and the semantics behind the four stereotype annotations, is what this module maps.

Knowledge Library → Courses → Spring Framework → Bean Definition and Scanning

What this module covers

A bean definition is Spring's recipe for an object the container will create and manage. This module covers the two ways to produce one: the stereotype annotations (@Component and its @Service/@Repository/@Controller specializations) that classpath scanning auto-detects, and the explicit @Bean factory methods declared inside a @Configuration class. It closes with how @ComponentScan is tuned with base packages and filters, and the full-versus-lite proxying that makes @Configuration different from a plain component.

Topics

Stereotype annotations: @Component, @Service, @Repository, @Controller

@Component is the generic marker that makes a class a candidate bean during scanning. @Service, @Repository and @Controller are meta-annotated with @Component, so functionally they are all discovered the same way — but each declares intent for a layer: business logic, persistence, and web handling respectively. Using the specific one documents the class's role and, crucially, unlocks layer-specific behavior.

Why it matters: the choice is rarely just cosmetic. @Repository triggers automatic persistence-exception translation (native/JPA exceptions become Spring's DataAccessException hierarchy), and @Controller makes the class eligible for Spring MVC request mapping. Interviewers probe whether you grasp that they are identical at the scanning level yet differ in semantics and side effects.

@Service                                   // business layer
public class OrderService {
    private final OrderRepository repo;
    public OrderService(OrderRepository repo) { this.repo = repo; }
}

@Repository                                 // persistence layer + exception translation
public class JpaOrderRepository implements OrderRepository { /* ... */ }

@Bean and @Configuration: Java-based container configuration

When you cannot annotate a class — a third-party type, or one needing construction logic — you declare it with a @Bean method inside a @Configuration class. The method body builds and returns the object; Spring registers the return value as a bean whose default name is the method name. This is the programmatic, XML-free way to define beans and wire their dependencies through method parameters.

Why it matters: @Bean versus stereotype scanning is a constant interview fork. Scanning suits your classes; @Bean suits classes you don't own or beans that need explicit setup. Both end up as identical bean definitions in the same container.

@Configuration
public class AppConfig {
    @Bean                                   // third-party class you can't annotate
    public PaymentClient paymentClient() {
        return new PaymentClient("api.example.com");
    }
    @Bean                                   // dependency injected via parameter
    public CheckoutService checkout(PaymentClient client) {
        return new CheckoutService(client);
    }
}

Component scanning: @ComponentScan, base packages and filters

Scanning is what turns a @Component-annotated class into a registered bean. @ComponentScan (implicit in @SpringBootApplication) tells the container which packages to walk; with no basePackages it scans the annotated class's own package and everything beneath it — which is why the layout of your main class quietly decides what gets found. Include and exclude filters (by annotation, assignable type, regex, AspectJ, or a custom TypeFilter) narrow or widen that set.

Why it matters: "my bean isn't being injected" is almost always a scanning-scope problem — the class sits outside the scanned packages or is filtered out. Understanding the default base package and filter types is the practical debugging skill behind that failure.

@Configuration
@ComponentScan(
    basePackages = "com.acme.app",
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.REGEX, pattern = ".*Legacy.*"))
public class ScanConfig { }

Full vs lite mode: @Configuration proxying and @Bean vs @Component

A @Configuration class runs in "full" mode: at startup Spring subclasses it with CGLIB so that one @Bean method calling another returns the shared singleton rather than a fresh object. Declare @Bean methods on a plain @Component (or set proxyBeanMethods = false) and you get "lite" mode — no proxy, faster startup, but inter-bean method calls create new instances, so dependencies must arrive as method parameters instead.

Why it matters: the proxy is why full-mode configuration classes and their @Bean methods cannot be final or private, and why a direct method call inside a @Configuration is safe but the same call in a @Component is a bug. This is a favourite senior-level discriminator.

@Configuration                              // full mode (proxyBeanMethods = true by default)
public class AppConfig {
    @Bean public Repo repo() { return new Repo(); }
    @Bean public Service service() {
        return new Service(repo());             // returns the SAME repo singleton
    }
}
// @Configuration(proxyBeanMethods = false)  -> lite mode: repo() would build a new Repo

Interview questions

Bean definition and scanning is core-Spring territory in Indian interviews — the screening round almost always opens here before touching web or data layers:

Beginner What is a Spring bean, and who creates it?

An object whose lifecycle — instantiation, dependency wiring, configuration and destruction — is managed by the Spring IoC container rather than by your code calling new. A bean definition is the container's recipe for it; the container reads all definitions and produces the beans in the ApplicationContext.

Beginner What are the four stereotype annotations and how do they relate to @Component?

@Component is the generic stereotype; @Service, @Repository and @Controller are specializations meta-annotated with @Component. All four are auto-detected by classpath scanning identically — the specializations add layer intent and, in some cases, extra behavior.

Beginner What is the difference between @Component and @Bean?

@Component is a class-level marker discovered by scanning — use it on classes you own. @Bean is a method-level annotation inside a @Configuration class where you build the object yourself — use it for third-party classes or beans needing construction logic. Both register a bean definition in the same container.

Beginner What does @ComponentScan do, and which package does it scan by default?

It tells the container which packages to search for stereotype-annotated classes. With no basePackages, it scans the package of the annotated (configuration) class and all sub-packages. In Spring Boot, @SpringBootApplication includes it, so the main class's package becomes the scan root.

Beginner @Controller vs @RestController?

@Controller is the stereotype for an MVC controller; its handler methods typically return view names. @RestController is @Controller + @ResponseBody, so every method's return value is serialized straight into the HTTP response body — the default for REST/JSON APIs.

Intermediate What special behavior does @Repository add beyond marking a bean?

It enables automatic persistence-exception translation: a PersistenceExceptionTranslationPostProcessor wraps the bean so native JDBC/JPA/Hibernate exceptions are converted into Spring's consistent, unchecked DataAccessException hierarchy — decoupling callers from the specific persistence technology.

Intermediate How do include and exclude filters on @ComponentScan work?

Each filter has a FilterType: ANNOTATION (has a given annotation), ASSIGNABLE_TYPE (is/implements a type), REGEX (class-name pattern), ASPECTJ (AspectJ expression), or CUSTOM (your own TypeFilter). includeFilters add candidates, excludeFilters remove them; you can disable the defaults with useDefaultFilters = false.

Intermediate How are auto-detected beans named by default?

The AnnotationBeanNameGenerator takes the annotation's value if given; otherwise it decapitalizes the simple class name — OrderService becomes orderService. A @Bean method defaults to the method name. You can override via the annotation value or a custom name generator.

Intermediate @Configuration vs @Component for holding @Bean methods — what changes?

In a @Configuration class (full mode), Spring CGLIB-proxies the class so inter-@Bean method calls route through the container and return shared singletons. In a plain @Component (lite mode), there is no proxy — calling one @Bean method from another runs plain Java and creates a new instance, bypassing container management.

Intermediate Why can one @Bean method call another inside a @Configuration class and still get the singleton?

Because CGLIB subclasses the configuration class at startup and intercepts each @Bean method call: instead of running the body again, the proxy checks the container and returns the already-created singleton. This is "full" configuration mode; it's what makes the direct method call safe there.

Pro What is proxyBeanMethods, and when would you set it to false?

@Configuration(proxyBeanMethods = false) switches to lite mode: no CGLIB proxy, faster startup and lower memory — appropriate when the class's @Bean methods never call each other (dependencies come in as parameters). The trade-off is losing singleton guarantees on direct inter-bean calls. Note full-mode classes and @Bean methods therefore cannot be final or private, since CGLIB must subclass and override them.

Pro A class is annotated @Service but no bean is found and injection fails. What are the likely causes?

Almost always a scanning-scope issue: the class sits outside the scanned base packages (e.g., in a package parallel to, not under, the main application class), no @ComponentScan/@SpringBootApplication covers it, or an excludeFilter removes it. Other causes: the class isn't on the runtime classpath, it's a nested/abstract class scanning skips, or a duplicate bean name collides. Fix by aligning package layout or adding explicit basePackages.
