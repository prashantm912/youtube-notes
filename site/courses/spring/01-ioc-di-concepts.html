Module: IoC and Dependency Injection

Spring's whole programming model starts here: hand object creation and wiring to a container, and everything above it — AOP, transactions, MVC — becomes possible. Get IoC and DI right and the rest of the framework clicks into place.

What this module covers

Inversion of Control is the idea that a container, not your own code, decides when objects are created and how they are wired together; Dependency Injection is the mechanism Spring uses to do it. This module orients you on the container (ApplicationContext vs BeanFactory), how it is bootstrapped, and why injecting a dependency beats calling new. It is the foundation every later Spring topic — beans, AOP, transactions, MVC — stands on.

Topics

IoC/DI concepts, ApplicationContext vs BeanFactory, container bootstrap

Inversion of Control inverts the normal direction of control: instead of your objects constructing and looking up their own collaborators, the Spring IoC container reads configuration metadata (annotations, a Java @Configuration class, or XML), instantiates the objects it manages — called beans — resolves their dependencies, and hands you back a fully wired object graph. Dependency Injection is simply how those dependencies get supplied.

The container comes in two flavours. BeanFactory is the base contract: lazy instantiation and plain wiring. ApplicationContext is a superset you will use in almost every real application — it eagerly pre-instantiates singletons at startup (fail-fast), auto-registers BeanPostProcessor and BeanFactoryPostProcessor beans, and adds event publication, MessageSource internationalisation, and AOP integration. Bootstrapping means creating a context (e.g. AnnotationConfigApplicationContext, ClassPathXmlApplicationContext, or the one Spring Boot builds for you) and letting it wire the graph.

// Bootstrap the container from an annotated @Configuration class
ApplicationContext ctx =
    new AnnotationConfigApplicationContext(AppConfig.class);

// Ask the container for a fully-wired bean — you never call new on it
OrderService orders = ctx.getBean(OrderService.class);

Why DI, inversion of control, comparison with new

Without DI, a class builds its own collaborators: new PaymentService(new StripeGateway()). That single line hard-wires a concrete type — you cannot swap the gateway for a different provider, you cannot substitute a mock in a unit test, and the class now owns the lifecycle of something it should only use. Dependency Injection fixes this by letting the class declare what it needs (an interface, through a constructor argument) and having the container supply the implementation from outside.

The payoff is loose coupling: high-level code depends on abstractions, wiring lives in one place, and swapping or mocking a dependency changes configuration rather than source. Constructor injection is the recommended style because it makes dependencies explicit and lets fields stay final. Note the vocabulary distinction interviewers probe: IoC is the broad principle (the framework controls object creation and flow), while DI is one specific technique for achieving it.

// Tight coupling — the class picks its own concrete collaborator
class PaymentService {
    private final Gateway gateway = new StripeGateway(); // fixed, unmockable
}

// Dependency Injection — the container supplies it, wiring stays external
class PaymentService {
    private final Gateway gateway;
    PaymentService(Gateway gateway) { this.gateway = gateway; } // any impl, mockable
}

Interview questions

The most-asked IoC/DI questions in Indian interviews — this is the opening topic of nearly every Spring screening round, so the fundamentals get pressed hard:

Beginner What is Inversion of Control (IoC)?

A design principle where control over object creation, wiring, and lifecycle is handed to a framework instead of your own code. Rather than an object constructing and looking up its collaborators, the container does it and supplies them — the flow of control is inverted from the application to the framework.

Beginner What is Dependency Injection and how does it relate to IoC?

DI is the technique that implements IoC: a class declares its dependencies (through constructor arguments, setters, or fields) and the container injects them from outside. IoC is the principle; DI is one common way to achieve it. Spring's IoC container performs DI.

Beginner What is the Spring IoC container, and what is a bean?

The container is the object that reads configuration metadata, then instantiates, configures, assembles, and manages application objects. A bean is any object the container instantiates and manages — the backbone objects of the application, created and wired by Spring rather than by new.

Beginner What is the difference between BeanFactory and ApplicationContext?

ApplicationContext extends BeanFactory. BeanFactory gives basic DI with lazy instantiation. ApplicationContext adds eager singleton pre-instantiation, automatic registration of BeanPostProcessor and BeanFactoryPostProcessor, event publication, MessageSource internationalisation, and AOP integration. Use ApplicationContext by default.

Beginner How do you bootstrap a Spring container?

Create an ApplicationContext implementation: AnnotationConfigApplicationContext(AppConfig.class) for Java/annotation config, ClassPathXmlApplicationContext("beans.xml") for XML. In Spring Boot, SpringApplication.run(...) builds and refreshes the context for you. After bootstrap, beans are wired automatically or fetched with getBean.

Intermediate Why use DI instead of creating dependencies with new?

new hard-codes a concrete class, causing tight coupling: you cannot swap implementations, cannot inject a mock in tests, and the object owns a lifecycle it should not. DI depends on an interface supplied by the container, so wiring is external, swappable, and testable, and the container controls scope and lifecycle.

Intermediate Is IoC the same as DI?

No. IoC is the broad principle that the framework controls object creation and program flow. DI is a specific pattern for realising it by injecting dependencies from outside. The Service Locator pattern is another IoC technique. So every DI is IoC, but not every IoC is DI.

Intermediate What types of dependency injection does Spring support, and which is recommended?

Constructor injection, setter injection, and field injection. Constructor injection is recommended: dependencies are explicit and mandatory, the field can be final, the object is fully initialised at construction, and it makes circular dependencies visible. Setter injection suits optional dependencies; field injection is discouraged as it hides dependencies and resists testing.

Intermediate What configuration metadata styles does the container accept?

Three: annotation-based (stereotypes plus component scanning), Java-based (@Configuration classes with @Bean methods), and XML-based (<bean> definitions). All three describe the same thing — which beans exist and how they depend on each other — and can be mixed. Annotation and Java config dominate modern code.

Intermediate Does the container create beans eagerly or lazily?

ApplicationContext pre-instantiates singleton beans at startup (fail-fast: wiring errors surface immediately), unless a bean is marked @Lazy. BeanFactory instantiates on first request. Prototype-scoped beans are always created on demand, regardless of container type.

Pro When would you ever choose BeanFactory over ApplicationContext?

Rarely — only in severely memory-constrained environments where the eager pre-instantiation and extra infrastructure of ApplicationContext are unaffordable, or where you deliberately want full manual control over bean processing. For essentially all applications the guidance is to use GenericApplicationContext or AnnotationConfigApplicationContext.

Pro How does IoC improve testability, and which SOLID principle does it embody?

Because collaborators are injected rather than constructed internally, a test can pass in stubs or mocks without touching the class under test — no framework needed for a plain unit test. It embodies the Dependency Inversion Principle: high-level modules depend on abstractions, and the container injects the concrete implementation, so details depend on abstractions rather than the reverse.
