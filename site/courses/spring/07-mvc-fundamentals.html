Module: Spring MVC Fundamentals

Every HTTP request in a Spring web app passes through one front controller — the DispatcherServlet — and the interview that follows almost always opens by asking you to trace that path from URL to response.

Knowledge Library → Courses → Spring Framework → Spring MVC Fundamentals

What this module covers

Spring MVC is the Servlet-stack web framework at the centre of most Spring applications. This module traces the DispatcherServlet's front-controller algorithm end to end, then shows how annotated controllers declare request mappings and how their method parameters get populated — path variables, query parameters, headers and JSON bodies bound through HttpMessageConverters. It is the foundation the interceptors, REST, validation and exception-handling modules build on.

Topics

The DispatcherServlet and the front-controller pattern

Spring MVC is built around the front controller pattern: a single servlet, the DispatcherServlet, receives every request for its URL space and provides one shared processing algorithm, while the real work is delegated to configurable beans — handler mappings, handler adapters, view resolvers, exception resolvers. You declare and map it per the Servlet spec (Java config, web.xml, or automatically in Spring Boot); it then bootstraps a WebApplicationContext that holds the web beans.

Why it matters: understanding that one servlet owns the whole request lifecycle — and that everything else is a pluggable strategy it calls — is the mental model every other Spring MVC feature slots into. Naming the DispatcherServlet as the front controller is the expected first sentence of the most common Spring MVC interview question.

The request-processing flow, step by step

The flow is the top interview topic in this module, so know the ordered steps: the DispatcherServlet binds the WebApplicationContext, locale and (if configured) a multipart resolver to the request; it asks each HandlerMapping for the handler plus its interceptor chain (a HandlerExecutionChain); it picks a HandlerAdapter to actually invoke that handler; the adapter resolves method arguments, calls your method, and processes the return value; a ViewResolver renders a view when a view name is returned, while a @ResponseBody handler writes the body directly and skips view resolution; any exception is routed to the HandlerExceptionResolver chain.

Why it matters: interviewers use this sequence to test whether you understand the moving parts behind an annotation. Being able to say where interceptors run, where argument binding happens, and where the split between "render a view" and "write a body" occurs is what distinguishes a real answer from reciting @Controller.

Annotated controllers and request mappings

A controller is any bean annotated @Controller (or @RestController, which is @Controller + @ResponseBody) with methods mapped by @RequestMapping and its HTTP shortcuts @GetMapping, @PostMapping, @PutMapping, @DeleteMapping, @PatchMapping. Mappings narrow on URL patterns (including {var} templates and ** wildcards), HTTP method, and the params, headers, consumes and produces conditions. A class-level mapping supplies a shared prefix that method mappings extend.

@RestController
@RequestMapping("/api/users")
class UserController {

    @GetMapping("/{id}")
    User byId(@PathVariable Long id) { /* ... */ }

    @PostMapping(consumes = "application/json")
    @ResponseStatus(HttpStatus.CREATED)
    User create(@RequestBody NewUser body) { /* ... */ }
}

Method arguments: path variables, params and headers

Handler methods take a flexible, resolver-driven parameter list rather than a fixed signature. @PathVariable binds a URI template segment (part of the resource identity), @RequestParam binds query-string or form parameters (optional filters, paging, search) and supports required and defaultValue, and @RequestHeader binds a request header — a missing required value yields a 400 rather than a null. Types are converted automatically, and Optional or required = false models genuinely optional inputs.

@GetMapping("/{tenant}/orders")
List<Order> list(
    @PathVariable String tenant,
    @RequestParam(defaultValue = "0") int page,
    @RequestParam(required = false) String status,
    @RequestHeader("X-Api-Version") String apiVersion) { /* ... */ }

Argument resolvers and HttpMessageConverters

Two extension points sit beneath the annotations. HandlerMethodArgumentResolver is the strategy interface that turns a request into each method parameter — every annotation above has one, and you can register custom resolvers for cross-cutting inputs (a resolved current user, a tenant object). For request and response bodies, HttpMessageConverter does content negotiation and (de)serialization: @RequestBody reads and deserializes the body into an object, @ResponseBody serializes the return value, with the concrete converter (Jackson for JSON, and others) chosen from the request's Content-Type and Accept headers against the mapping's consumes/produces.

Interview questions

The questions Indian interviews ask most on Spring MVC — the DispatcherServlet flow is the single most common, so lead with it:

Beginner What is the DispatcherServlet and what pattern does it implement?

It is Spring MVC's central servlet and the implementation of the front-controller pattern: every request in its URL space enters through it, and it runs one shared algorithm, delegating the actual work to configurable beans (handler mappings, adapters, view resolvers, exception resolvers).

Beginner How do you define a controller and map a URL to a method?

Annotate a bean with @Controller or @RestController, then map methods with @RequestMapping or its shortcuts @GetMapping/@PostMapping/@PutMapping/@DeleteMapping/@PatchMapping. A class-level @RequestMapping adds a shared path prefix.

Beginner Difference between @Controller and @RestController?

@RestController is a meta-annotation combining @Controller and @ResponseBody. So every method's return value is serialized to the response body (via an HttpMessageConverter) instead of being treated as a view name. Use @Controller when you render views; @RestController for REST/JSON APIs.

Beginner @PathVariable vs @RequestParam — when do you use each?

@PathVariable binds a segment of the URI template (/orders/{id}) — required data that identifies the resource. @RequestParam binds query-string or form values (?page=2&status=OPEN) — typically optional filters, sorting or paging, and it supports required and defaultValue.

Intermediate Walk through the request-processing flow of the DispatcherServlet.

Bind WebApplicationContext/locale/multipart to the request → ask HandlerMapping for the handler and its interceptor chain → select a HandlerAdapter → the adapter resolves method arguments, invokes the handler, processes the return value → ViewResolver renders a view if a view name is returned, otherwise @ResponseBody writes the body directly → exceptions go to the HandlerExceptionResolver chain → the response is returned. Interceptors run around the handler invocation.

Intermediate What are the roles of HandlerMapping and HandlerAdapter?

HandlerMapping answers "which handler serves this request?" and returns it wrapped with its interceptors. HandlerAdapter answers "how do I invoke that handler?" — it abstracts the calling convention so different handler styles (annotated methods, functional endpoints) can coexist. The DispatcherServlet iterates configured mappings, then finds an adapter that supports the resolved handler.

Intermediate How does Spring convert a JSON body into an object and back?

Through HttpMessageConverter. @RequestBody triggers a converter (Jackson for JSON) to deserialize the body into the parameter type; @ResponseBody serializes the return value. The converter is chosen by content negotiation — the request's Content-Type and Accept headers matched against the mapping's consumes/produces.

Intermediate How do you make a request parameter optional, and what happens if a required one is missing?

Set @RequestParam(required = false), give a defaultValue, or declare the parameter as Optional<T>. A missing required @RequestParam or @RequestHeader results in a 400 Bad Request (e.g. MissingServletRequestParameterException / MissingRequestHeaderException), not a null value.

Intermediate Can two handler methods map the same path? How does Spring choose?

Yes, if their mapping conditions differ — HTTP method, params, headers, consumes or produces. Spring scores each candidate for how specifically it matches the request and picks the most specific; a genuine tie raises an IllegalStateException about an ambiguous mapping.

Pro What is a HandlerMethodArgumentResolver and why write a custom one?

It is the strategy that produces one controller-method argument from the request; every argument annotation is backed by one. You add a custom resolver to inject cross-cutting values cleanly — a resolved current user, a tenant, a parsed pagination object — so controllers declare a domain parameter instead of repeatedly reading and validating raw request data.

Pro Where do interceptors sit in the flow, and how do they differ from servlet filters?

Interceptors (HandlerInterceptor) run inside the DispatcherServlet, around the handler — preHandle before, postHandle after the handler but before view rendering, afterCompletion at the end — and they know the resolved handler. Filters are a Servlet-container concept that wrap the whole DispatcherServlet and see only raw request/response, with no handler awareness. (Filters vs interceptors is expanded in the next module.)

Pro A request returns 415 Unsupported Media Type even though the controller looks correct. What is happening?

Content negotiation failed against the mapping's consumes — the client's Content-Type did not match, or no HttpMessageConverter can read that media type into the @RequestBody type (e.g. sending form data to a JSON-only endpoint, a missing/incorrect Content-Type header, or no Jackson on the classpath). The mirror case, a mismatched Accept against produces, yields 406 Not Acceptable.

Pro Why can't @RequestBody read HTML form data, and what should you use instead?

The Servlet API consumes the input stream to parse application/x-www-form-urlencoded into request parameters, so by the time a message converter would read the body it is already drained. Bind form fields with @RequestParam (or a @ModelAttribute command object) rather than @RequestBody; reserve @RequestBody for JSON/XML payloads.
