Module: REST API Development

A REST controller is where Spring MVC stops rendering views and starts serializing objects — and the difference between a working endpoint and a well-designed API lives in the status codes, the media types, and the docs you ship alongside it.

Knowledge Library → Courses → Spring Framework → REST API Development

What this module covers

Building HTTP APIs on top of the DispatcherServlet you met earlier: @RestController and the mapping annotations, ResponseEntity for full control of status and headers, content negotiation and the HttpMessageConverter layer that turns your objects into JSON, plus two topics interviewers use to separate levels — HATEOAS awareness and documenting the API with OpenAPI via springdoc. Validation and exception design get their own modules next; here the focus is the response itself.

Topics

@RestController and request mappings

@RestController is a meta-annotation: @Controller plus @ResponseBody, so every handler's return value is written straight to the response body instead of being resolved as a view name. The mapping annotations — @GetMapping, @PostMapping, @PutMapping, @DeleteMapping, @PatchMapping — are composed shortcuts for @RequestMapping with the HTTP method pre-set, and @PathVariable, @RequestParam and @RequestBody bind the URL and payload into method arguments.

Why it matters: this is the surface every Spring API is built on, and the first thing screened. Knowing that @RestController is just two older annotations fused together lets you explain why a plain @Controller returns a view name string while a REST one serializes the same string as a JSON body.

@RestController
@RequestMapping("/api/users")
class UserController {

  @GetMapping("/{id}")                 // GET /api/users/42
  User byId(@PathVariable Long id) {
    return service.find(id);           // serialized to JSON automatically
  }

  @PostMapping                         // body deserialized by Jackson
  User create(@RequestBody NewUser body) {
    return service.save(body);
  }
}

ResponseEntity and HTTP status codes

Returning a bare object always answers 200 OK. ResponseEntity<T> is the escape hatch when you need to control the status line and headers too: ResponseEntity.created(location) for 201 with a Location header, noContent() for 204 on a delete, notFound() for 404. Two alternatives round it out — @ResponseStatus on a method or exception (declarative, fixed code) and ResponseStatusException (programmatic, when the code varies at runtime).

Why it matters: correct status codes are the grammar of REST. Interviewers routinely ask which code a create, an update, a delete, or a missing resource should return, and whether you would hand a client an empty 200 or an honest 404 for something that does not exist.

@PostMapping
ResponseEntity<User> create(@RequestBody NewUser body) {
  User saved = service.save(body);
  URI location = URI.create("/api/users/" + saved.id());
  return ResponseEntity.created(location).body(saved);   // 201 + Location
}

// absence becomes 404, not an empty 200 body
return service.find(id)
    .map(ResponseEntity::ok)
    .orElseGet(() -> ResponseEntity.notFound().build());

Content negotiation and message converters

How does a returned User become JSON? The HttpMessageConverter chain does it — Jackson's converter is registered by default when jackson-databind is on the classpath, so JSON is the out-of-the-box format; adding jackson-dataformat-xml enables XML. Which one a request gets is content negotiation: by default Spring reads the client's Accept header, and you can further constrain a handler with the produces and consumes attributes (produces matches Accept, consumes matches Content-Type).

Why it matters: candidates often know "Spring returns JSON" without knowing the mechanism. Naming the converter layer, explaining that path-extension negotiation is now discouraged in favour of the Accept header, and distinguishing produces from consumes is a clear intermediate signal.

HATEOAS awareness

HATEOAS (Hypermedia as the Engine of Application State) is the top rung of the Richardson Maturity Model: responses carry links telling the client what it can do next, so navigation is not hard-coded into the caller. Spring HATEOAS supplies the building blocks — RepresentationModel, EntityModel<T>, CollectionModel<T>, the Link type, and WebMvcLinkBuilder (linkTo/methodOn) which derives URIs from your controller methods instead of string concatenation — and serializes to the HAL media type by default.

Why it matters: most production Spring APIs stop at level 2 (proper verbs and status codes) and never adopt hypermedia, so this is usually asked at awareness level — what HATEOAS is, where it sits in the maturity model, and the trade-off of added payload and client complexity for discoverability.

EntityModel<User> model = EntityModel.of(user,
    linkTo(methodOn(UserController.class).byId(user.id())).withSelfRel(),
    linkTo(methodOn(UserController.class).all()).withRel("users"));

API documentation with OpenAPI (springdoc)

An API nobody can read is half-built. OpenAPI 3 is the standard machine-readable description of a REST API, and springdoc-openapi generates it for Spring by inspecting your controllers at runtime — add the springdoc-openapi-starter-webmvc-ui dependency and the JSON spec appears at /v3/api-docs with an interactive Swagger UI at /swagger-ui.html. Annotations such as @Operation, @ApiResponse, @Schema and @Tag enrich the generated document.

Why it matters: springdoc is the current, actively maintained choice for Spring Boot 3, having replaced the now-dormant SpringFox. Knowing the modern library, the two default endpoints it exposes, and that documentation is derived from annotations rather than hand-written is a common practical question.

Interview questions

Most-asked REST-in-Spring questions in Indian interviews — a near-guaranteed section once your resume says Spring Boot, from service-company screens to product-company deep dives:

Beginner What is @RestController and how does it differ from @Controller?

@RestController = @Controller + @ResponseBody. With @Controller, a returned String is treated as a view name; with @RestController every return value is serialized to the response body (JSON by default). Introduced in Spring 4.0 to remove the need to put @ResponseBody on every REST handler.

Beginner What do @GetMapping, @PostMapping and friends actually do?

They are composed annotations — shortcuts for @RequestMapping with the HTTP method fixed (@GetMapping = @RequestMapping(method = GET)). They also accept path, produces and consumes, so you rarely write raw @RequestMapping on a handler method.

Beginner How do you read the path, query parameters, and request body?

@PathVariable binds a URI template segment (/users/{id}), @RequestParam binds a query string or form parameter (?page=2), and @RequestBody deserializes the request payload (JSON) into an object via the message converters. Headers bind with @RequestHeader.

Beginner What is ResponseEntity and when do you use it instead of returning the object?

ResponseEntity<T> represents the whole response — status, headers and body — so you use it when you need to set a status other than 200 or add headers (like Location). For the common 200-with-body case, returning the object directly is cleaner; don't overuse ResponseEntity.

Beginner Which status codes fit create, update, delete and not-found?

Create: 201 Created with a Location header. Successful update: 200 (body) or 204 No Content. Delete: 204. Missing resource: 404. Bad input: 400. Conflict (e.g. duplicate): 409. Unhandled server error: 500.

Intermediate How does Spring turn your returned object into JSON?

Through an HttpMessageConverter — specifically the Jackson converter, auto-registered when jackson-databind is on the classpath. The converter chain is chosen by content negotiation; the same mechanism deserializes @RequestBody on the way in. No jackson-databind means no automatic JSON.

Intermediate What is content negotiation and how does Spring decide JSON versus XML?

Spring picks the response media type from the request — by default the Accept header — and selects the matching converter. JSON is default; XML requires jackson-dataformat-xml (or JAXB). Path-extension negotiation exists but is discouraged for security reasons; the Accept header (or an explicit query parameter) is preferred.

Intermediate Difference between the produces and consumes attributes?

consumes narrows a handler by the request's Content-Type (what the endpoint accepts, e.g. application/json on a POST). produces narrows by the request's Accept (what the endpoint can return). They let two handlers on the same URL serve different media types.

Intermediate Three ways to set the HTTP status from a controller?

ResponseEntity (programmatic, per-response), @ResponseStatus on a method or exception class (declarative, fixed code), and ResponseStatusException (programmatic, thrown at runtime when the code depends on logic). ResponseStatusException avoids creating an exception class per status.

Intermediate PUT versus PATCH, and which HTTP methods are idempotent?

PUT replaces the whole resource; PATCH applies a partial update. GET, PUT and DELETE are idempotent (repeating the call leaves the same server state); POST is not. Idempotency matters for safe client retries — a repeated PUT is harmless, a repeated POST may create duplicates.

Pro What is HATEOAS and where does it sit in the Richardson Maturity Model?

HATEOAS is maturity level 3: responses embed hypermedia links describing available next actions, so clients follow links instead of hard-coding URIs. Spring HATEOAS provides EntityModel/CollectionModel, Link and WebMvcLinkBuilder, serializing to HAL. Most APIs stay at level 2 — the trade-off is discoverability and decoupling against heavier payloads and client complexity.

Pro How do you document a Spring REST API, and springdoc versus SpringFox?

Use springdoc-openapi: add springdoc-openapi-starter-webmvc-ui and it inspects controllers to emit an OpenAPI 3 spec at /v3/api-docs with Swagger UI at /swagger-ui.html; @Operation, @ApiResponse and @Schema refine it. springdoc is the maintained choice for Spring Boot 3 — SpringFox is effectively abandoned and does not support the modern stack.
