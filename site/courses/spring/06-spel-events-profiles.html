Module: SpEL, Events and Profiles

Three quieter corners of the core container that senior interviews love: the expression language wired into your annotations, the in-process event bus that decouples beans, and the switch that swaps whole sets of beans per environment.

Knowledge Library → Courses → Spring Framework → SpEL, Events and Profiles

What this module covers

These are the container features you meet through annotations rather than APIs. The Spring Expression Language (SpEL) evaluates the #{…} inside @Value and @EventListener(condition=…); the Environment abstraction resolves the ${…} placeholders that inject configuration; the application event mechanism lets beans publish and listen without referencing each other; and @Profile registers different beans in dev, test and prod. Small features individually — but together they are where "I use Spring" becomes "I understand Spring".

Topics

The Spring Expression Language (SpEL)

SpEL is a runtime expression language for querying and manipulating an object graph — method calls, property access, collection selection/projection, ternary and Elvis operators, bean references (@beanName) and type references (T(java.lang.Math)). You rarely call the parser yourself; you meet SpEL as the #{…} syntax inside annotations such as @Value, @Cacheable(key=…), Spring Security expressions and @EventListener conditions.

Why it matters: the distinction between #{…} (a SpEL expression, evaluated by an ExpressionParser) and ${…} (a property placeholder, resolved from the Environment) is a classic interview trap — and knowing that a StandardEvaluationContext can invoke arbitrary methods is the difference between a convenient feature and an injection vulnerability.

@Value("#{ T(java.lang.Math).random() * 100.0 }")  // SpEL: evaluated at runtime
private double seed;

@Value("${app.retries:3}")                          // placeholder + default, from Environment
private int retries;

// Programmatic use is the same engine:
Expression exp = new SpelExpressionParser().parseExpression("'hello'.toUpperCase()");
String out = (String) exp.getValue();               // "HELLO"

The Environment: property sources and @Value

The Environment abstraction unifies two things: the set of active profiles and a hierarchy of property sources (command-line args, system properties, environment variables, application.properties/.yaml, @PropertySource files). When Spring sees ${db.url} it walks that ordered hierarchy — first source that has the key wins — which is exactly the precedence you must reason about when an environment variable mysteriously overrides your properties file.

Why it matters: @Value("${key}") injects a single resolved value into a field or constructor parameter; for a group of related keys, type-safe @ConfigurationProperties is preferred. Interviewers probe whether you can trace where a value came from, the resolution order, and how to supply a default (${key:fallback}).

@Component
class Mailer {
  @Value("${mail.host}") String host;               // resolved from a PropertySource
  private final Environment env;                     // or read programmatically
  Mailer(Environment env) { this.env = env; }
  void send() { int port = env.getProperty("mail.port", Integer.class, 25); }
}

Application events: publish, @EventListener, async

Spring's ApplicationContext is an event bus. A bean injects ApplicationEventPublisher and calls publishEvent(...); any bean method annotated @EventListener receives it — no direct reference between publisher and listener. Since Spring 4.2 the event need not extend ApplicationEvent (any object works), a listener can filter with a SpEL condition, and the framework also fires built-in lifecycle events (ContextRefreshedEvent, ContextClosedEvent, and others).

Why it matters: events are synchronous by default — the publisher's thread runs every listener in turn, inside the same transaction. Add @Async (with @EnableAsync) to run a listener off-thread, or @TransactionalEventListener to fire only after the transaction commits. These trade-offs — ordering, exception propagation, transaction boundaries — are the heart of the interview follow-ups.

record OrderPlaced(String id) { }                    // plain event (Spring 4.2+)

@Component class OrderService {
  private final ApplicationEventPublisher pub;
  OrderService(ApplicationEventPublisher pub) { this.pub = pub; }
  void place(String id) { pub.publishEvent(new OrderPlaced(id)); }
}

@EventListener void onOrder(OrderPlaced e) { /* sync by default; add @Async to detach */ }

Profiles (@Profile)

A profile is a named grouping of beans. Mark any @Component, @Configuration or @Bean method with @Profile("dev") and it is registered only when that profile is active — the mechanism behind swapping an embedded H2 datasource in development for a pooled production one. Profile strings support expressions: !prod (not prod), dev | qa (or), prod & cloud (and).

Why it matters: profiles are activated declaratively via spring.profiles.active (a property, the SPRING_PROFILES_ACTIVE environment variable, or a --spring.profiles.active= command-line flag), or programmatically. The classic bug — a bean silently missing because its profile was never activated, giving a NoSuchBeanDefinitionException — is a favourite "what went wrong here" question.

@Configuration
@Profile("!prod")                       // everything except prod; also "dev | qa", "prod & cloud"
class EmbeddedDbConfig {
  @Bean DataSource dataSource() { return new EmbeddedDatabaseBuilder().build(); }
}
// activate: spring.profiles.active=prod   (property, env var, or --flag)

Interview questions

Frequently asked in Indian Spring interviews — these three features separate rote annotation use from real container understanding, hardest last:

Beginner What is SpEL and where do you actually use it?

The Spring Expression Language: a runtime expression language for querying/manipulating objects. You rarely call the parser directly — you meet it as #{…} inside annotations: @Value, @Cacheable(key=…), Spring Security @PreAuthorize, and @EventListener(condition=…).

Beginner Difference between #{...} and ${...}?

${…} is a property placeholder resolved from the Environment's property sources (with an optional ${key:default}). #{…} is a SpEL expression evaluated at runtime. They can nest: @Value("#{'${csv.list}'.split(',')}") resolves the property, then runs SpEL on it.

Beginner What are Spring profiles and why use them?

Named groupings of beans. @Profile("dev") registers a bean only when that profile is active, so you bootstrap environment-specific beans — an embedded DB for local, a pooled DataSource for prod — from the same codebase without conditionals.

Beginner How do you activate a profile?

Declaratively via the spring.profiles.active property — in application.properties, the SPRING_PROFILES_ACTIVE environment variable, or a --spring.profiles.active=prod command-line argument. Also programmatically with SpringApplication.setAdditionalProfiles(...) or the Environment. Multiple profiles are comma-separated.

Beginner What does @Value do, and how do you give it a default?

Injects a resolved value (from a property source or a SpEL expression) into a field or constructor/method parameter. Default syntax: @Value("${timeout.ms:2000}") — if the key is absent the value after the colon is used, avoiding a startup failure.

Intermediate How do you publish and consume a custom application event?

Inject ApplicationEventPublisher and call publishEvent(new MyEvent(...)). Consume it with any bean method annotated @EventListener taking the event type as its parameter. Since Spring 4.2 the event need not extend ApplicationEvent — a plain object (or record) works.

Intermediate Are application events synchronous or asynchronous? How do you change that?

Synchronous by default: the publisher's thread invokes every listener in sequence, within the same transaction, and blocks until they finish. To detach one listener, annotate it @Async (with @EnableAsync and a configured executor); to make all events async, register an ApplicationEventMulticaster backed by a task executor.

Intermediate @EventListener vs implementing ApplicationListener?

@EventListener is annotation-based: any public method, multiple event types, SpEL condition, ordering via @Order, no interface needed. ApplicationListener<E> is the typed interface and can be registered very early — it is the way to catch pre-refresh events like ApplicationContextInitializedEvent before annotated listeners exist.

Intermediate What is @TransactionalEventListener and when is it needed?

An @EventListener variant bound to a transaction phase — by default AFTER_COMMIT. It fires only if the publish happened inside a transaction, so side effects (send email, enqueue message) run only when the data actually committed, not on rollback. Set fallbackExecution=true to also run without a transaction.

Intermediate How do @Profile expressions work?

The string is a profile expression: !prod (active when prod is not), dev | qa (either), prod & cloud (both). This lets one annotation cover several environments — e.g. @Profile("!prod") on all non-production configuration.

Pro A bean is null / missing at runtime and you suspect profiles. What happened?

Its @Profile was never activated, so the definition was never registered — injecting it throws NoSuchBeanDefinitionException (or leaves an @Autowired(required=false) field null). Check the active profiles logged at startup, the spring.profiles.active value in each environment, and whether a "default" profile bean should cover the unconfigured case.

Pro If an @Async event listener throws, who sees the exception — and how does that differ from a sync listener?

A synchronous listener's exception propagates back to publishEvent, aborting the publish and any remaining listeners — and rolling back the caller's transaction. An @Async listener runs on another thread, so the exception never reaches the publisher; a void method routes it to the AsyncUncaughtExceptionHandler, and a returned Future would carry it. This is why async listeners need their own error handling.

Pro Why is a StandardEvaluationContext dangerous with untrusted input, and what do you use instead?

StandardEvaluationContext exposes the full language — method and constructor invocation, T(...) type references — so a SpEL string built from user input can execute arbitrary code (the root of several Spring RCE CVEs). For data-binding or user-influenced expressions use SimpleEvaluationContext, which restricts to property access and a curated feature set. Never concatenate untrusted data into a SpEL expression.
