Module: Aspect-Oriented Programming

Spring adds transactions, security, caching and logging to your beans by wrapping them in a proxy — and that same proxy is why a method calling its own annotated method silently skips the advice.

Knowledge Library → Courses → Spring Framework → Aspect-Oriented Programming

What this module covers

AOP is how Spring modularizes cross-cutting concerns — the vocabulary (aspect, join point, advice, pointcut, weaving), the five advice types, and the AspectJ-style pointcut language. Underneath it all sits a runtime proxy, and understanding that proxy explains both how declarative features like @Transactional work and why they sometimes silently don't. The proxy mechanics (JDK vs CGLIB) and the self-invocation trap are the two most-tested AOP topics in Indian interviews.

Topics

What AOP is: cross-cutting concerns and the vocabulary

Some concerns — logging, security checks, transactions, caching, metrics — cut across many unrelated classes. Scattering that code everywhere (tangling) and repeating it (scattering) is exactly what AOP removes: you write the behaviour once as an aspect and declare where it applies, leaving business methods clean. The core terms: an aspect modularizes the concern; a join point is a point in execution where behaviour can be inserted (in Spring AOP, always a method execution); advice is the code that runs there; a pointcut is the expression selecting which join points match; and weaving links aspects into the target — Spring does it at runtime via a proxy.

Why it matters: nearly every declarative feature you already use — @Transactional, @Cacheable, @Async, method security — is AOP under the hood. Knowing the vocabulary is what lets you reason about them instead of treating them as magic.

@Aspect                 // the aspect: a module of cross-cutting behaviour
@Component              // register it as a Spring bean so it gets applied
public class AuditAspect {
  @Before("execution(* com.shop..*Service.*(..))")  // pointcut: which join points
  public void audit() { /* advice: what runs at each matched join point */ }
}

Advice: the five kinds and @Around

Advice is where the behaviour lives, and its type decides when relative to the method it runs. @Before runs first; @AfterReturning only on a normal return (and can read the returned value); @AfterThrowing only when the method throws (and can bind the exception); @After always runs, like a finally block. @Around is the most powerful — it wraps the call, receives a ProceedingJoinPoint, and decides whether, when and with what arguments to invoke proceed(), plus it can transform the return value or the exception.

Why it matters: choosing the wrong advice type produces subtle bugs — timing/retry logic belongs in @Around, cleanup in @After, audit-on-success in @AfterReturning. Forgetting to call proceed() in @Around silently swallows the real method.

@Aspect
@Component
public class TimingAspect {
  @Around("execution(* com.shop.service..*(..))")
  public Object time(ProceedingJoinPoint pjp) throws Throwable {
    long start = System.nanoTime();
    try { return pjp.proceed(); }                 // run the real method
    finally { log.info("{} ns for {}", System.nanoTime() - start, pjp.getSignature()); }
  }
}

Pointcut expressions and designators

A pointcut is a predicate written in the AspectJ pointcut language. It starts with a designator (PCD): execution(...) matches method executions and is by far the most common; within(...) limits to a type or package; @annotation(...) matches methods carrying a given annotation; args, this, target and the Spring-specific bean(...) add more. Wildcards make them flexible — * for a single element, .. for "any number of" packages or parameters. Named pointcuts (@Pointcut) let you define an expression once and combine them with &&, || and !.

@Pointcut("execution(* com.shop.service.*.*(..))")   // any method in a service class
public void serviceLayer() {}

@Before("serviceLayer() && @annotation(org.springframework.transaction.annotation.Transactional)")
public void check(JoinPoint jp) { /* runs before transactional service methods */ }

Proxy mechanics: JDK dynamic proxy vs CGLIB

Spring AOP is proxy-based: when a bean needs advice, the container hands out a proxy that wraps the real object and applies the advice around each call. Two proxy strategies exist. If the target implements at least one interface, Spring creates a JDK dynamic proxy of those interfaces. If it implements none, Spring generates a CGLIB proxy — a runtime-generated subclass of the target. You can force CGLIB everywhere with proxyTargetClass=true (which Spring Boot enables by default). The trade-off: a JDK proxy is only assignable to the interface, so injecting the concrete class fails; CGLIB avoids that but cannot subclass final classes or override final/private/static methods.

Why it matters: this is the single most-asked AOP question. The "why do I get a proxy class, not my class?" confusion, the interface-vs-concrete injection failure, and the reason Spring Boot switched to CGLIB by default all trace back to these two mechanisms.

public interface PaymentService { void pay(); }   // implements interface -> JDK dynamic proxy
public class ReportJob { public void run() {} }    // no interface        -> CGLIB subclass

@Configuration
@EnableAspectJAutoProxy(proxyTargetClass = true)   // force CGLIB even for interface-backed beans
class AopConfig {}

The self-invocation trap

Because advice lives on the proxy and not on the target object, a bean method that calls another method of the same bean directly (this.other()) bypasses the proxy entirely — the second call never crosses the proxy boundary, so its @Transactional, @Cacheable, @Async or custom advice is simply ignored. No error is raised; the behaviour just doesn't happen. This is the classic "my transaction/cache isn't working and I can't see why" bug.

Why it matters: it is the second-most-asked AOP topic and a real production hazard. The fixes: move the inner method into a separate bean and inject it (cleanest); self-inject the bean into itself and call through that reference; use AopContext.currentProxy() with exposeProxy=true; or switch to AspectJ load-time/compile-time weaving, which weaves the class itself and has no proxy boundary.

@Service
public class OrderService {
  @Transactional
  public void placeOrder() { audit(); }   // internal call -> goes to the raw target, NOT the proxy

  @Transactional(propagation = Propagation.REQUIRES_NEW)
  public void audit() { /* the REQUIRES_NEW transaction is never started */ }
}

Interview questions

The AOP questions asked most in Indian interviews — proxy mechanics and the self-invocation trap dominate the senior rounds:

Beginner What is AOP and which problem does it solve?

A paradigm for modularizing cross-cutting concerns — logging, security, transactions, caching, metrics — that would otherwise be duplicated and scattered across many classes. You write the behaviour once as an aspect and declare where it applies, keeping business code focused on business logic.

Beginner Explain aspect, join point, advice and pointcut.

Aspect: the module of cross-cutting behaviour. Join point: a point in execution where advice can run (in Spring AOP, always a method execution). Advice: the code that runs at a join point. Pointcut: the expression selecting which join points match.

Beginner What join points does Spring AOP support?

Only method execution on Spring-managed beans. Unlike full AspectJ it cannot advise field access, constructors, or static/final/private methods. For those you need AspectJ weaving instead of proxies.

Beginner Name the five advice types.

@Before, @AfterReturning (normal return only, can bind the value), @AfterThrowing (on exception, can bind it), @After (finally — always runs), and @Around (wraps the call and controls whether it proceeds).

Intermediate Difference between @After and @AfterReturning?

@After runs whether the method returns normally or throws, like a finally block. @AfterReturning runs only on normal completion and can read the returned value; if the method throws it is skipped and @AfterThrowing fires instead.

Intermediate What is @Around advice and why does ProceedingJoinPoint matter?

@Around wraps the target: code runs before and after, and you decide whether to call pjp.proceed(), alter its arguments, transform its return value, or translate exceptions. It is the only advice that receives ProceedingJoinPoint and must return a value — forgetting proceed() silently skips the real method.

Intermediate Break down the pointcut execution(* com.shop.service.*.*(..)).

execution is the designator for method-execution join points; the first * matches any return type; com.shop.service.* any class in that package; .* any method name; (..) any number and type of parameters. Other common designators: within, @annotation, args, bean.

Intermediate How does Spring choose between a JDK dynamic proxy and a CGLIB proxy?

If the target implements at least one interface, Spring creates a JDK dynamic proxy of those interfaces; if it implements none, it generates a CGLIB subclass. You can force CGLIB with proxyTargetClass=true. Spring Boot sets that to true by default, so Boot apps use CGLIB even for interface-backed beans.

Intermediate What are the consequences of a JDK proxy being interface-based?

The proxy is only assignable to the interface, not the concrete class, so injecting or casting to the concrete type fails. Methods not declared on the interface are not advised. Program to the interface, or use CGLIB — which in turn cannot proxy final classes or methods.

Pro What is the self-invocation trap and why does it happen?

Advice lives on the proxy, not the target. When one method of a bean calls another method of the same bean directly (this.other()), the call goes to the raw target and never crosses the proxy — so @Transactional, @Cacheable, @Async on the inner method are ignored, with no error. It is the most common "why isn't my transaction/cache working?" bug.

Pro How do you work around self-invocation?

Move the inner method into a separate bean and inject it (cleanest); self-inject the bean into itself and call through that reference; use AopContext.currentProxy() with exposeProxy=true; or switch to AspectJ load-time/compile-time weaving, which weaves the class itself so there is no proxy boundary.

Pro Spring AOP vs AspectJ — the real differences?

Spring AOP: runtime, proxy-based, method-execution join points only, Spring beans only, no build step, subject to the self-invocation limitation. AspectJ: true bytecode weaving (compile-time or load-time), any join point including fields and constructors, any object, faster at runtime, but needs the weaver/agent. Spring AOP covers the large majority of enterprise needs.

Pro Why can CGLIB not proxy final or private methods, and how does that bite you?

CGLIB proxies by generating a runtime subclass and overriding methods; final classes can't be subclassed and final/private/static methods can't be overridden, so they can't be advised. A @Transactional on a final or private method is silently ignored — no error, just no advice. Keep advised methods public and non-final.
