Module: Testing in Spring

A green suite is only worth as much as the context behind it. This module maps how Spring builds, caches, mocks, and rolls back that context — so your tests stay fast, isolated, and honest.

Knowledge Library → Courses → Spring Framework → Testing in Spring

What this module covers

Spring's testing story is layered: the TestContext framework wires a real ApplicationContext into plain JUnit classes and caches it across the whole suite, Spring Boot adds sliced contexts and auto-configured helpers, and bean-override annotations swap collaborators for Mockito mocks. This module orients you across that stack — from a full @SpringBootTest down to a focused @WebMvcTest, through @MockitoBean (the current successor to the deprecated @MockBean), MockMvc for the web layer, transactional rollback, and a survey of reactive testing with WebTestClient.

Topics

The Spring TestContext framework and context caching

The TestContext framework (package org.springframework.test.context) provides annotation-driven integration-test support that is agnostic of the test runner — JUnit Jupiter, JUnit 4, or TestNG. It bootstraps a Spring container, injects test fixtures, manages transactions, and fires test events, so your test class stays a POJO instead of extending a base class. This is the machinery underneath every @SpringBootTest.

Why it matters: starting a container per test method would make suites unbearably slow, so the framework caches each ApplicationContext keyed by its full configuration (config classes, active profiles, property sources, and every bean override) and reuses it across tests that ask for the same setup. Understanding the cache key — and that @DirtiesContext evicts an entry when a test corrupts state — is the difference between a suite that runs in seconds and one that reloads the context dozens of times.

@SpringBootTest and Spring Boot test slices

@SpringBootTest boots a Boot application context through SpringApplication, giving tests the full auto-configuration, externalized properties, and every bean. Its webEnvironment attribute chooses the shape: MOCK (default, no server), RANDOM_PORT or DEFINED_PORT (a real embedded server for end-to-end tests), or NONE. Full-context tests are the most realistic but the heaviest.

Why it matters: loading everything to test one controller is wasteful, so Boot offers slices — @WebMvcTest (web layer, MVC infrastructure, and MockMvc only), @DataJpaTest (repositories, an embedded DB, transactional rollback), @JsonTest, @RestClientTest, and more. Each narrows the context to one concern, so tests run faster and fail closer to the cause. Choosing the right slice per test is a core skill interviewers probe.

@WebMvcTest(UserController.class)                // controllers + MVC infra + MockMvc
@DataJpaTest                                     // repositories + embedded DB, tx-rolled-back
@SpringBootTest(webEnvironment = RANDOM_PORT)    // full context + real embedded server

Mocking collaborators: @MockitoBean and @MockitoSpyBean

@MockitoBean overrides a bean in the test's context with a Mockito mock; @MockitoSpyBean wraps the real bean in a spy. Introduced in Spring Framework 6.2 and living in the core framework's bean-override support, they are the current, standardized replacement for Spring Boot's @MockBean and @SpyBean — which are deprecated since Spring Boot 3.4 and slated for removal. Prefer @MockitoBean in new code.

Why it matters: a bean override changes the context's configuration, so it participates in the cache key — a mock that varies per class can quietly multiply your cached contexts. Know the difference from a plain Mockito @Mock (which never touches the Spring container) and from @MockitoSpyBean (which requires exactly one existing bean to wrap). These distinctions are among the most current Spring interview topics.

@SpringBootTest
class PaymentServiceTest {
  @Autowired PaymentService service;
  @MockitoBean GatewayClient gateway;         // Boot 3.4+ successor to @MockBean

  @Test void chargesOnce() {
    given(gateway.charge(any())).willReturn(Result.ok());
    service.pay(order);
    then(gateway).should().charge(any());
  }
}

MockMvc: testing the web layer

MockMvc performs full Spring MVC request handling — mappings, argument resolution, message converters, exception handling — against mock request and response objects, with no server started and no socket opened. It is the standard way to assert controller behavior: status codes, headers, and JSON bodies via matchers. Modern Boot also exposes MockMvcTester, an AssertJ-style fluent wrapper over the same engine.

Why it matters: it sits between a pure unit test (no Spring) and a full HTTP integration test (real server), giving realistic web-layer coverage at slice speed. Pair it with @WebMvcTest and @MockitoBean to test one controller in isolation, or with @SpringBootTest plus @AutoConfigureMockMvc for a fuller stack. Writing a MockMvc assertion by hand is a very common interview exercise in India.

mockMvc.perform(get("/api/users/1").accept(APPLICATION_JSON))
       .andExpect(status().isOk())
       .andExpect(jsonPath("$.email").value("asha@corp.in"));

Transactional tests and rollback

Annotate a test class or method with @Transactional and the TransactionalTestExecutionListener wraps each test method in a transaction that is rolled back by default when the method finishes — so tests never leave rows behind and can run in any order. Override per method with @Commit or @Rollback, run code outside the transaction with @BeforeTransaction/@AfterTransaction, or drive it manually via the TestTransaction API. A PlatformTransactionManager bean must be present.

Why it matters: the automatic rollback is what keeps data-layer tests repeatable, but it also hides bugs — a rolled-back transaction never flushes the way a real request commits, and lazy-loading or constraint errors can slip through. Knowing that @DataJpaTest is transactional by default, and how to force a real commit when you must, separates confident answers from cargo-culted ones.

@SpringBootTest
@Transactional                       // each test runs in a tx, rolled back at the end
class OrderRepositoryTest {
  @Autowired OrderRepository repo;

  @Test void savedRowIsRolledBack() {
    repo.save(new Order("A-100"));
    assertThat(repo.count()).isEqualTo(1);
  }                                  // ROLLBACK here — DB untouched for the next test
}

Reactive and WebFlux testing: a survey

WebTestClient is the non-blocking client for testing web endpoints. It can bind directly to a controller, a router function, or an ApplicationContext without a running server, or hit a live server over real HTTP — and it exposes a fluent, chained expectation API (status, headers, typed body lists) plus AssertJ integration. It is the natural choice for Spring WebFlux applications, and it also works against Spring MVC.

Why it matters: reactive endpoints return Mono/Flux, so blocking assertions do not fit; WebTestClient subscribes and verifies for you. Even in a mostly-MVC shop, knowing that the same client tests both stacks — and that @SpringBootTest(webEnvironment = RANDOM_PORT) auto-configures one bound to the running port — rounds out a testing answer and shows breadth beyond the servlet world.

webTestClient.get().uri("/flux/users")
    .exchange()
    .expectStatus().isOk()
    .expectBodyList(User.class).hasSize(3);

Interview questions

Most-asked Spring testing questions in Indian interviews — service and product companies both probe context management, slicing, and the current mocking annotations:

Beginner What is the Spring TestContext framework?

The runner-agnostic engine behind Spring's integration testing. It bootstraps an ApplicationContext, injects test fixtures, manages test transactions, and caches contexts — all while your test class stays a plain POJO. @SpringBootTest, @WebMvcTest, and friends are built on it.

Beginner Why write a Spring-context test at all instead of a plain unit test?

Unit tests (plain JUnit + Mockito) verify one class in isolation and are fastest. A context test exercises wiring, configuration, proxies, transactions, serialization, and auto-configuration — the parts that only exist when Spring assembles the beans. You need both layers; the pyramid stays mostly unit tests.

Beginner What does @WebMvcTest load, and what does it deliberately skip?

It auto-configures the MVC infrastructure and MockMvc and scans web components — @Controller, @ControllerAdvice, converters, filters, interceptors. It does not load @Service, @Repository, or general @Component beans, so you supply their collaborators as mocks. Faster and focused on the web layer.

Beginner What is MockMvc and why use it over a running server?

It runs full Spring MVC request processing against mock request/response objects — no server, no socket. You assert status, headers, and body without HTTP overhead, so web-layer tests stay fast and deterministic while still going through mappings, converters, and exception handlers.

Beginner Are changes made in an @Transactional test committed to the database?

No — by default the test transaction is rolled back when the method finishes, leaving the database clean for the next test. Use @Commit (or @Rollback(false)) to force a commit when you specifically need one.

Intermediate @MockBean is deprecated — what replaces it and why?

@MockitoBean (and @MockitoSpyBean), introduced in Spring Framework 6.2 and part of the core bean-override support. Boot's @MockBean/@SpyBean are deprecated since Spring Boot 3.4 and scheduled for removal so mocking lives in the framework, not Boot. It is not a blind rename — @MockitoBean is not supported on @Configuration or @Component classes.

Intermediate Difference between Mockito's @Mock, Mockito.mock(), and @MockitoBean?

@Mock and Mockito.mock() create a mock object with no knowledge of Spring — you inject it yourself. @MockitoBean creates a mock and replaces (or adds) that bean in the Spring context, so autowired collaborators receive the mock. Use plain Mockito for unit tests, @MockitoBean when the container is involved.

Intermediate @MockitoBean vs @MockitoSpyBean?

@MockitoBean installs a full mock (strategy REPLACE_OR_CREATE — it creates the bean if absent). @MockitoSpyBean wraps the existing real bean in a spy (strategy WRAP), so real methods run unless stubbed, and it requires exactly one candidate bean to wrap.

Intermediate How does context caching work and why does it matter?

The TestContext framework caches each ApplicationContext and reuses it for every test declaring the same configuration, so the container starts once instead of per class. The default cache holds 32 contexts with LRU eviction. Reusing contexts is usually the single biggest lever on suite runtime.

Intermediate What makes up the context cache key, and what forces a fresh context?

The key combines config classes/locations, context initializers and customizers, active profiles, property sources, the loader, any parent context, and bean overrides (@MockitoBean, @TestBean, @DynamicPropertySource). Any difference yields a new context. @DirtiesContext explicitly evicts an entry when a test mutates shared state.

Intermediate What are the @SpringBootTest webEnvironment options?

MOCK (default) — mock servlet environment, no server, pair with @AutoConfigureMockMvc; RANDOM_PORT and DEFINED_PORT — a real embedded server for end-to-end HTTP tests (a random free port avoids clashes in CI); NONE — a context with no web environment.

Intermediate How would you test a WebFlux (reactive) endpoint?

With WebTestClient. Bind it to the controller, a router function, or the context (no server), or to a live server via @SpringBootTest(webEnvironment = RANDOM_PORT). Its fluent API subscribes to the Mono/Flux and asserts status, headers, and typed bodies without you blocking manually.

Pro Why can a suite full of @MockitoBean tests run slower than expected?

Every distinct set of bean overrides produces a distinct context cache key. If mocks vary from class to class, the cache fills with near-duplicate contexts and evicts useful ones (default max 32, LRU), forcing repeated container startups. Fixes: standardize test configuration, group overrides, and reserve @DirtiesContext for genuine state corruption.

Pro Automatic rollback keeps tests clean — when does it mislead you?

A rolled-back transaction may never flush to the database the way a committed request does, so failing constraints, generated IDs, cascade behavior, or lazy-loading issues can hide. The persistence context can also serve entities from its first-level cache rather than reloading. Force a flush/clear or use @Commit (often with Testcontainers) when you must verify real persistence semantics.

Pro MockMvc standaloneSetup vs webAppContextSetup — trade-offs?

standaloneSetup registers just the controllers you name with minimal MVC infrastructure — no Spring context, fastest, but it does not reflect your real configuration (custom converters, advice, interceptors, security). webAppContextSetup builds MockMvc from the loaded WebApplicationContext, so it exercises the actual configuration at the cost of loading a context. Use standalone for narrow unit-style checks, web-app-context for fidelity.
