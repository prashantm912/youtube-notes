Module: Validation

A DTO's constraint annotations do nothing until something triggers them — and knowing exactly what @Valid, @Validated and BindingResult do at the controller boundary is what separates a passing Spring interview from a failing one.

Knowledge Library → Courses → Spring Framework → Validation

What this module covers

Validation in Spring is Jakarta Bean Validation (formerly JSR-380) driven by a provider — Hibernate Validator — that Spring wires up and runs for you. This module orients you on declaring constraints, the two annotations that trigger validation (@Valid vs @Validated — the single most-asked topic here), turning failures into clean HTTP responses, and the advanced tools: cascading into nested graphs, custom constraints, and validation groups.

Topics

Bean Validation and the standard constraint annotations

Jakarta Bean Validation is a specification for declaring rules on object fields with annotations — @NotNull, @Size, @Min/@Max, @Email, @Pattern, @Positive, @Past and more. Spring does not implement the spec; it integrates a provider (Hibernate Validator, the reference implementation) and reads those annotations at runtime to produce a set of constraint violations. The constraints live on your DTO or entity, in one place, instead of being scattered across if checks.

Why it matters: declarative validation moves "is this input acceptable?" into the type itself. Interviewers probe the annotation vocabulary and especially the trio @NotNull / @NotEmpty / @NotBlank — null vs empty vs whitespace-only. One setup gotcha: since Spring Boot 2.3 you must add spring-boot-starter-validation explicitly to pull Hibernate Validator onto the classpath.

public record RegisterRequest(
    @NotBlank String name,          // not null, not blank after trim
    @Email    String email,
    @Min(18)  int age,
    @Size(min = 8, max = 64) String password) {}

Triggering validation: @Valid vs @Validated

Constraints are inert until something asks the validator to run. In an MVC controller, annotate a @RequestBody, @ModelAttribute or @RequestPart parameter with @Valid (Jakarta) or @Validated (Spring) and Spring validates it before the handler body executes. To validate the parameters or return value of an ordinary Spring bean's methods (a @Service, say), you put @Validated on the class so a MethodValidationPostProcessor proxies it.

The distinction interviewers want: @Valid is standard Jakarta and drives cascading into nested objects; @Validated is Spring-only and adds validation groups (choose which constraints apply per use case), but it does not itself cascade — nested fields still need @Valid. Rule of thumb: reach for @Valid for ordinary and nested validation, and @Validated when you need groups or method-level validation on a bean.

@RestController
@Validated                       // enables method-level & group validation on this bean
class UserController {
  @PostMapping("/users")
  ResponseEntity<Void> create(@Valid @RequestBody RegisterRequest req) { ... }
}

Error binding: BindingResult and the error response

When @Valid fails on a @RequestBody, Spring throws MethodArgumentNotValidException; its BindingResult carries the FieldError list (field name, rejected value, message). Alternatively, if you declare a BindingResult parameter immediately after the validated object, Spring hands control back to you instead of throwing — the classic pattern for server-rendered form pages where you re-display the form with errors.

For REST APIs, centralize the mapping in a @RestControllerAdvice with an @ExceptionHandler that turns MethodArgumentNotValidException into a 400 response with a field-to-message map (or a ProblemDetail, covered in the Exception Handling module). Note that constraints placed directly on method parameters fail with a different type — ConstraintViolationException (or HandlerMethodValidationException in current Spring) — a frequent source of "why isn't my handler catching it?" bugs.

@ExceptionHandler(MethodArgumentNotValidException.class)
ResponseEntity<Map<String,String>> onInvalid(MethodArgumentNotValidException ex) {
  Map<String,String> errors = new HashMap<>();
  ex.getBindingResult().getFieldErrors()
    .forEach(fe -> errors.put(fe.getField(), fe.getDefaultMessage()));
  return ResponseEntity.badRequest().body(errors);
}

Cascading and container-element validation

Validation is not recursive by default. To validate a nested object graph you mark the nested field — or a collection's elements — with @Valid, which tells the provider to descend into it. Leave @Valid off, and an @NotBlank declared inside the nested type is never checked, even though the outer object validates cleanly.

Since Bean Validation 2.0 you can also constrain container elements directly: List<@Email String>, Map<@NotNull K, @Valid V>, Optional<@Positive Integer>. Validating a controller method that accepts a raw List of DTOs is a real-world snag — the list needs to be a wrapped/annotated container or a validated bean property for the elements to be checked.

class Order {
  @NotNull Long id;
  @Valid   Customer customer;               // descend into the nested object
  @NotEmpty List<@Valid OrderLine> lines;   // validate every element
}

Custom constraint validators

When the built-in constraints do not express your rule, define your own: an annotation meta-annotated with @Constraint(validatedBy = ...) plus a class implementing ConstraintValidator and its isValid() method. Because Spring instantiates the validator as a bean, you can inject other beans into it — for example a repository, to check that an email is not already registered.

Custom constraints can be field-level or class-level. A class-level validator sees the whole object, which is how cross-field rules like "password equals confirmPassword" are expressed. Prefer composing existing constraints where the logic allows; reach for a hand-written validator only when the rule is genuinely bespoke.

@Constraint(validatedBy = UniqueEmailValidator.class)
@Target(FIELD) @Retention(RUNTIME)
public @interface UniqueEmail {
  String message() default "email already registered";
  Class<?>[] groups()  default {};
  Class<? extends Payload>[] payload() default {};
}

Validation groups and group sequences

A group is a marker interface used to tag constraints so you can validate a subset at a time. @NotNull(groups = OnUpdate.class) means "only enforce this when the OnUpdate group is active." You activate a group with @Validated(OnUpdate.class) — and because plain @Valid cannot select groups, this is the main reason to prefer @Validated. The unclassified default group is jakarta.validation.groups.Default.

@GroupSequence orders groups so that cheap checks run before expensive ones, and later groups are skipped entirely once an earlier group produces a violation — for instance, skipping a database-uniqueness check when the field is already blank. This keeps a create form and an update form validating the same class under different rules without duplicating the DTO.

interface OnCreate {}
class Account {
  @Null(groups = OnCreate.class)   Long id;       // must be absent on create
  @NotNull(groups = OnUpdate.class) Long version;  // required on update
}
// controller:  create(@Validated(OnCreate.class) @RequestBody Account a)

Interview questions

The questions Indian interviewers actually ask about validation — @Valid vs @Validated leads almost every Spring screening round:

Beginner What is Bean Validation and how does Spring use it?

Jakarta Bean Validation (formerly JSR-380) is a specification for declaring constraints on fields via annotations. Spring does not implement it — it integrates a provider (Hibernate Validator, the reference implementation), auto-configures a validator, and runs it when you ask (for example on an @Valid controller argument).

Beginner Which dependency enables validation in Spring Boot, and why is it separate?

spring-boot-starter-validation, which brings in Hibernate Validator. Since Spring Boot 2.3 it is no longer transitively included by the web starter, so a missing dependency is the usual reason @Valid silently does nothing.

Beginner Difference between @NotNull, @NotEmpty and @NotBlank?

@NotNull: value must not be null (empty string passes). @NotEmpty: not null and size/length > 0 (works on String, Collection, Map, array; a single space passes). @NotBlank: String only — not null and must contain at least one non-whitespace character.

Beginner How do you validate a JSON request body in a REST controller?

Put constraints on the DTO and annotate the parameter @Valid @RequestBody Dto dto. Spring runs the validator before the handler body; on failure it throws MethodArgumentNotValidException, which you translate to a 400 response.

Intermediate @Valid vs @Validated — the key differences?

@Valid is standard Jakarta: it triggers validation and cascades into nested objects, but cannot select groups. @Validated is Spring-specific: it supports validation groups and, placed on a class, enables method-level validation via a proxy — but it does not cascade by itself. Use @Valid for ordinary/nested validation, @Validated when you need groups or method validation.

Intermediate What exception is thrown when @Valid on a @RequestBody fails, and how do you return a clean 400?

MethodArgumentNotValidException. Handle it in a @RestControllerAdvice / @ExceptionHandler, read ex.getBindingResult().getFieldErrors(), and build a field-to-message map (or a ProblemDetail) returned with status 400.

Intermediate How do you validate nested objects and the elements of a collection?

Validation is not recursive: mark the nested field with @Valid to descend into it, and use container-element syntax such as List<@Valid OrderLine> or List<@Email String> (Bean Validation 2.0+) to validate each element. Without @Valid on the field, inner constraints are skipped.

Intermediate How do validation groups work and when do you need them?

Groups are marker interfaces that tag constraints, e.g. @NotNull(groups = OnUpdate.class). Activate a group with @Validated(OnUpdate.class) so only that group's constraints fire. They let one DTO carry different rules for create vs update flows without duplicating the class. Plain @Valid cannot select groups.

Intermediate How do you write a custom constraint that also checks the database?

Define an annotation meta-annotated with @Constraint(validatedBy = ...) and a ConstraintValidator implementing isValid(). Since Spring creates the validator as a bean, you can @Autowired a repository into it to check, say, email uniqueness. Class-level validators handle cross-field rules like matching passwords.

Intermediate What is BindingResult and when would you inject it instead of letting Spring throw?

BindingResult holds the outcome of binding and validation — field errors, global errors, rejected values. Declared as the parameter immediately after the validated object, it suppresses the thrown exception and lets the handler inspect errors itself — the standard approach for re-rendering a server-side form rather than returning JSON.

Pro Why does @NotNull on a @Service method parameter not fire unless the class is @Validated, and which exception differs?

Method-level (parameter/return) validation is applied by a MethodValidationPostProcessor that only proxies beans annotated @Validated at class level. Without it, parameter constraints are ignored. When it does fire, a violation surfaces as ConstraintViolationException (or HandlerMethodValidationException for controller method parameters in current Spring) — not MethodArgumentNotValidException — so your @RequestBody handler won't catch it.

Pro Can @Validated cascade into nested objects, and how do groups plus cascading interact?

No — @Validated selects groups but does not cascade; nested fields still need @Valid. Note that @Valid on a nested field cascades using the currently active group(s); if the nested constraints are tagged to a specific group, they only fire when that group is active. Mixing the two on the same graph is a common source of "my nested validation didn't run" tickets.

Pro How do you internationalize or customize validation messages?

Give each constraint a message that is a resource key in braces, e.g. @Size(message = "{user.name.size}"), and provide values in ValidationMessages.properties (per-locale). Wire a LocalValidatorFactoryBean to Spring's MessageSource so messages resolve through your i18n bundles, and use EL placeholders like {min}/{max} and ${validatedValue} for interpolation.
