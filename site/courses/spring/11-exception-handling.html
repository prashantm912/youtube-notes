Module: Exception Handling

An unhandled exception is the difference between a clean 404 carrying a JSON body and a raw stack trace leaking to your caller — this module maps Spring's handlers, the @ControllerAdvice that centralizes them, and the RFC 9457 error body they should return.

Knowledge Library → Courses → Spring Framework → Exception Handling

What this module covers

Spring gives you a layered toolkit for turning thrown exceptions into well-formed HTTP responses: @ExceptionHandler methods for controller-local rules, @ControllerAdvice to hoist them across every controller, and — since Spring 6 — ProblemDetail, the built-in RFC 9457 error body. The topics below orient you on each piece and the design decisions (status codes, error shape) that separate a leaky stack trace from a clean, machine-readable error contract.

Topics

@ExceptionHandler: catching exceptions inside a controller

@ExceptionHandler marks a method that handles specific exception types thrown by the request-mapping methods around it. Declared inside a @Controller it handles only that controller's exceptions; the type it matches comes from the annotation value or the method's own parameter, and the method can return a view name, a ResponseEntity, or a ProblemDetail.

Why it matters: this is the atom of Spring exception handling — everything "global" is the same annotation hoisted into a @ControllerAdvice. Knowing that Spring dispatches to the most specific matching handler (the nearest supertype wins, not declaration order) is a routine interview probe.

@ExceptionHandler(OrderNotFoundException.class)
public ResponseEntity<String> handle(OrderNotFoundException ex) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND).body(ex.getMessage());
}

@ControllerAdvice and @RestControllerAdvice: global handling

A @ControllerAdvice class collects @ExceptionHandler (plus @InitBinder and @ModelAttribute) methods and applies them across all controllers, so one place defines how the whole API reports errors. @RestControllerAdvice is that annotation meta-annotated with @ResponseBody — the variant you want for REST, since handler return values are serialized straight to the response body instead of resolved as view names.

Why it matters: this is the single most-asked exception-handling topic in Spring interviews. It deletes duplicated per-controller try/catch, and you can scope one advice (by annotation, package, or type) to run different error contracts for different slices of the API — public versus admin, say.

@RestControllerAdvice
public class ApiExceptionHandler {
    @ExceptionHandler(OrderNotFoundException.class)
    public ProblemDetail handle(OrderNotFoundException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    }
}

ProblemDetail and RFC 9457: a standard error body

Before Spring 6, every team invented its own error JSON. ProblemDetail (in org.springframework.http) is Spring's built-in container for an RFC 9457 problem-details response: the spec fields type, title, status, detail and instance, plus a properties map for anything extra. It serializes as application/problem+json, giving clients one predictable, machine-readable error shape across services.

Why it matters: RFC 9457 obsoletes RFC 7807 — the media type and core fields are unchanged, so Spring's ProblemDetail (introduced against 7807 in Spring 6.0) already conforms; 9457 mainly adds an extension registry and clarifications. Quote 9457 in interviews, and know that ResponseEntityExceptionHandler already renders the framework's own exceptions in this shape.

ProblemDetail pd = ProblemDetail.forStatusAndDetail(
        HttpStatus.NOT_FOUND, "Order 42 not found");
pd.setType(URI.create("https://api.shop.com/errors/order-not-found"));
pd.setProperty("orderId", 42);   // non-standard extension field

Status codes and error-response design

Getting the status right is half of error handling. @ResponseStatus stamps a fixed status on an exception class or handler; ResponseEntity lets you set status, headers and body programmatically; and ResponseStatusException throws a status-plus-reason inline without a dedicated exception class — handy for one-offs. ResponseEntityExceptionHandler is the base @ControllerAdvice that already maps Spring MVC's built-in exceptions to sensible statuses and ProblemDetail bodies.

Why it matters: the design questions behind these — never leak a stack trace, map each domain failure to the correct 4xx or 5xx, keep one consistent error shape, and thread a correlation id for support — are exactly what senior interviews push on.

@ResponseStatus(HttpStatus.NOT_FOUND)             // fixed status on the exception
class OrderNotFoundException extends RuntimeException { }

throw new ResponseStatusException(                // or inline, no custom class
        HttpStatus.NOT_FOUND, "Order not found");

Interview questions

Exception handling is a Spring MVC staple in Indian product and services interviews — @ControllerAdvice especially. Ordered easiest to hardest:

Beginner What is @ExceptionHandler and where can it be used?

A method-level annotation marking a method that handles specific exception types thrown by request-mapping methods. Inside a @Controller it handles that controller's exceptions; inside a @ControllerAdvice it handles them globally. The exception type comes from the annotation value or the method's parameter.

Beginner What does @ControllerAdvice do?

It marks a class whose @ExceptionHandler (also @InitBinder and @ModelAttribute) methods apply across all controllers — centralized, cross-cutting error handling instead of duplicating handlers in each controller. It is the single most-asked exception-handling question in Spring interviews.

Beginner @ControllerAdvice vs @RestControllerAdvice?

@RestControllerAdvice = @ControllerAdvice + @ResponseBody. With plain @ControllerAdvice a returned String is treated as a view name; @RestControllerAdvice serializes handler return values straight to the response body (JSON) — the right choice for REST APIs.

Beginner How do you set the HTTP status code of an error response?

Several ways: annotate the handler or exception class with @ResponseStatus(HttpStatus.NOT_FOUND); return a ResponseEntity with an explicit status; throw ResponseStatusException(status, reason); or return a ProblemDetail/ErrorResponse carrying the status.

Intermediate If two @ExceptionHandler methods could handle a thrown exception, which one wins?

Spring picks the handler whose declared exception type is closest in the class hierarchy to the thrown exception — the most specific match, not declaration order. A handler for the exact type beats one declared for a superclass.

Intermediate What is ProblemDetail and what fields does it carry?

Spring's container (since 6.0) for an RFC 9457 problem-details response: standard fields type, title, status, detail, instance, plus a properties map for extensions. It renders as application/problem+json, giving APIs a uniform, machine-readable error shape.

Intermediate RFC 7807 vs RFC 9457 — which does Spring use?

9457 is the current standard and obsoletes 7807; the media type and core fields are unchanged, so Spring's ProblemDetail (introduced against 7807 in Spring 6.0) already conforms. 9457 adds an extension registry and clarifications. Say "9457" in interviews — 7807 is superseded.

Intermediate ResponseStatusException vs @ExceptionHandler — when each?

ResponseStatusException is thrown programmatically to map a status and reason to a response without a custom exception class or advice — fast and local. @ControllerAdvice/@ExceptionHandler centralizes handling and enforces app-wide conventions. Common pattern: ResponseStatusException for one-offs, advice for the global contract.

Intermediate What is ResponseEntityExceptionHandler and why extend it?

A convenient base class for a @ControllerAdvice that already handles all the standard Spring MVC exceptions (validation, unreadable body, missing params, and more) and renders them as ProblemDetail. You extend it and override protected handle* methods to customize, getting consistent responses for framework exceptions for free.

Intermediate How do you add custom fields to a ProblemDetail?

Two ways: call setProperty("field", value) to put extras in the properties map (Jackson flattens them to top-level JSON), or subclass ProblemDetail / ErrorResponseException with your own getters. Useful for a trace/correlation id, an internal error code, or a timestamp.

Pro How do you limit a @ControllerAdvice to a subset of controllers?

Use its selectors: annotations = RestController.class (by annotation), basePackages = "com.app.api" (by package), or assignableTypes = {FooController.class} (by type). With no selector it applies to every controller — this lets you run different error contracts for, say, public versus admin APIs.

Pro Will @ControllerAdvice catch an exception thrown in a Servlet Filter?

No. @ExceptionHandler/@ControllerAdvice only handle exceptions that surface through the DispatcherServlet (controllers and the handler pipeline). Filters run before and outside the DispatcherServlet, so their exceptions bypass it — handle those with the servlet error page, an ErrorController in Boot, or a filter-level try/catch. A classic senior gotcha, especially with Spring Security filter-chain errors.

Pro How do you turn Spring's built-in MVC exceptions into ProblemDetail responses automatically?

Two levers. In Spring Boot set spring.mvc.problemdetails.enabled=true so framework exceptions render as application/problem+json out of the box. For full control, register a @ControllerAdvice extending ResponseEntityExceptionHandler and override the relevant handle* methods. Plain Spring MVC (no Boot) uses the ResponseEntityExceptionHandler route.
