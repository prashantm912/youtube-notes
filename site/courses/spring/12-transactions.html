Module: Transactions

One annotation opens a database transaction, joins or suspends another, and decides whether your data survives an exception — and the reasons it silently does nothing are exactly what Indian interviewers probe. This module orients you on each setting and links the material that teaches it in full.

Knowledge Library → Courses → Spring Framework → Transactions

What this module covers

Spring's transaction abstraction lets you keep JDBC/JPA commit-and-rollback logic out of your code and declare it instead. This module walks the @Transactional settings that matter in production and in interviews: the seven propagation behaviours, the isolation levels and the concurrency anomalies they block, the rollback rules and the readOnly hint, and finally the pitfalls — self-invocation, private/final methods, checked exceptions that quietly commit — plus when to drop to programmatic transactions.

Topics

Declarative transactions: @Transactional and its settings

Annotating a bean's method with @Transactional makes Spring wrap the call in a proxy that starts a transaction before the method and commits it after — or rolls back on failure. You never write begin/commit/rollback; you declare intent and the container binds a connection (or a JPA EntityManager) to the thread for the method's duration. Activation needs @EnableTransactionManagement (Spring Boot switches it on automatically) plus a PlatformTransactionManager bean.

Why it matters: the annotation carries six-plus tunables — propagation, isolation, timeout, readOnly, rollbackFor, noRollbackFor — and its defaults (REQUIRED, ISOLATION_DEFAULT, read-write, rollback on unchecked only) are the baseline every later topic tweaks. Knowing what a bare @Transactional actually means is the foundation of every transaction interview.

@Service
public class OrderService {
  @Transactional                     // propagation=REQUIRED, rollback on RuntimeException/Error
  public void placeOrder(Order o) {
    inventory.reserve(o);            // same physical transaction
    payments.charge(o);             // either both commit or both roll back
  }
}

Propagation — all seven behaviours

Propagation answers "what happens when a transactional method is called while a transaction is already running?" Spring defines seven behaviours. REQUIRED (default) joins the current transaction or starts one. REQUIRES_NEW always suspends any running transaction and runs an independent physical one — it commits or rolls back on its own. NESTED runs inside a JDBC savepoint of the outer transaction, so the inner part can roll back without killing the outer. SUPPORTS joins if a transaction exists, else runs non-transactionally. NOT_SUPPORTED suspends any transaction and runs without one. MANDATORY demands an existing transaction (else throws). NEVER forbids one (else throws).

Why it matters: this is the single most-asked Spring transaction topic in Indian interviews, and the favourite trap is REQUIRES_NEW vs NESTED. REQUIRES_NEW is a separate transaction on a separate connection (survives an outer rollback); NESTED is a savepoint in the same transaction (an outer rollback still discards it). Naming all seven and placing REQUIRED/REQUIRES_NEW/NESTED confidently is what senior answers do.

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void writeAuditLog(AuditEvent e) {
  // suspends the caller's transaction; this log commits even if the
  // outer business transaction later rolls back
  auditRepo.save(e);
}

Isolation levels and the read anomalies

Isolation controls how much of one in-flight transaction another can see, trading consistency against concurrency. Spring's Isolation enum maps to standard JDBC levels: DEFAULT (the database's own default), READ_UNCOMMITTED (allows dirty reads), READ_COMMITTED (blocks dirty reads — the common default for PostgreSQL and SQL Server), REPEATABLE_READ (also blocks non-repeatable reads — MySQL/InnoDB's default), and SERIALIZABLE (blocks phantom reads too, at the highest locking cost). The three anomalies to name are dirty read (reading uncommitted data), non-repeatable read (a row changes between two reads), and phantom read (a range query gains/loses rows).

Why it matters: interviewers pair each level with the anomalies it prevents, and expect you to know that isolation on @Transactional is honoured only when a new physical transaction is created (REQUIRED starting fresh, or REQUIRES_NEW) — a level set on a method that merely joins an existing transaction is ignored.

Rollback rules and readOnly

By default Spring rolls back only on unchecked exceptions (RuntimeException and Error) and commits on checked exceptions — the behaviour that surprises most developers the first time a checked exception leaves half-written data committed. Override it with rollbackFor (e.g. rollbackFor = Exception.class to roll back on checked exceptions too) and noRollbackFor to exempt specific types; the strongest matching rule wins. The readOnly = true flag is a hint — it lets JPA skip dirty-checking/flush and lets the driver or a routing DataSource send the work to a read replica, but it does not by itself forbid writes.

Why it matters: "which exceptions roll back by default, and how do you change it?" is a screening staple, and readOnly is a favourite performance follow-up. A related trap is UnexpectedRollbackException — see the pitfalls topic — thrown when a transaction you expected to commit was already marked rollback-only by an inner participant.

@Transactional(rollbackFor = Exception.class)      // roll back on checked exceptions too
public void importFile(Path p) throws IOException { ... }

@Transactional(readOnly = true)                     // skips JPA flush; can route to a replica
public List<Order> recentOrders() { return repo.findRecent(); }

Self-invocation, programmatic transactions and other pitfalls

Because declarative transactions are proxy-based, they only apply to calls that enter the bean through the proxy. The classic bug — self-invocation — is a method calling another @Transactional method of the same class via this: the second annotation is silently ignored because no proxy sits on that internal call. The same proxy limit means @Transactional does nothing on private methods (the CGLIB proxy cannot override them) and on final methods/classes. Fixes: move the inner method to a separate bean, self-inject the proxy, or switch to AspectJ weaving mode. When declarative control is not enough — conditional commits, fine-grained boundaries inside a loop — drop to programmatic transactions with TransactionTemplate or a PlatformTransactionManager.

Why it matters: self-invocation is, alongside propagation, the most-asked Spring transaction question in India, precisely because the code compiles, runs, and quietly does the wrong thing. Being able to explain why the proxy is bypassed and list the three fixes marks a candidate who has debugged it for real.

@Service
public class ReportService {
  public void run() {
    generate();          // internal call via 'this' -> proxy bypassed, NO transaction
  }
  @Transactional
  public void generate() { ... }   // annotation ignored when reached from run()
}

// Programmatic alternative when you need explicit control:
transactionTemplate.execute(status -> {
  repo.save(entity);
  if (shouldAbort) status.setRollbackOnly();
  return null;
});

Interview questions

The most-asked Spring transaction questions in Indian interviews — propagation and self-invocation dominate, so both carry extra weight here:

Beginner What does @Transactional do, and how does Spring implement it?

It makes a bean method run inside a database transaction: Spring wraps the bean in a proxy that starts a transaction before the method, commits after it returns, and rolls back on failure. You declare intent instead of writing begin/commit/rollback. It needs @EnableTransactionManagement (automatic in Spring Boot) and a PlatformTransactionManager bean.

Beginner What are the default propagation and isolation of @Transactional?

Propagation REQUIRED (join the current transaction or start a new one) and isolation DEFAULT (whatever the underlying database uses). It is also read-write and rolls back only on unchecked exceptions by default.

Beginner Which exceptions trigger a rollback by default, and how do you change it?

By default only unchecked exceptions — RuntimeException and Error — roll back; checked exceptions commit. To roll back on a checked exception use @Transactional(rollbackFor = Exception.class); to keep a runtime exception from rolling back use noRollbackFor. When rules overlap, the most specific matching rule wins.

Beginner What does readOnly = true do?

It is an optimisation hint. It lets JPA/Hibernate skip dirty-checking and the flush at commit, and lets a driver or routing DataSource send the work to a read replica. It does not by itself forbid writes — it signals "no changes expected", it is not a hard guard.

Intermediate Explain all seven propagation behaviours.

REQUIRED — join current or create one (default). REQUIRES_NEW — suspend any current transaction and run an independent one. NESTED — run in a savepoint of the current transaction (inner can roll back alone). SUPPORTS — join if one exists, else run non-transactionally. NOT_SUPPORTED — suspend any current transaction and run without one. MANDATORY — must have a current transaction, else throw. NEVER — must not have one, else throw.

Intermediate REQUIRED vs REQUIRES_NEW — difference and a use case?

REQUIRED reuses the caller's transaction, so everything commits or rolls back together. REQUIRES_NEW suspends the caller's transaction and runs a separate physical one on its own connection, committing independently. Use REQUIRES_NEW for work that must persist regardless of the outer outcome — audit logs, failure records. Caveat: it holds a second connection, so size the pool accordingly.

Intermediate NESTED vs REQUIRES_NEW — what is the real difference?

NESTED is a savepoint inside the same physical transaction: the inner part can roll back to the savepoint without aborting the outer, but if the outer rolls back the inner is discarded too, and they commit together. REQUIRES_NEW is a genuinely separate transaction that commits on its own and survives an outer rollback. NESTED needs JDBC savepoint support (e.g. DataSourceTransactionManager).

Intermediate Name the isolation levels and the anomalies each prevents.

READ_UNCOMMITTED prevents nothing (dirty reads possible). READ_COMMITTED prevents dirty reads. REPEATABLE_READ also prevents non-repeatable reads. SERIALIZABLE also prevents phantom reads. The three anomalies: dirty read (reading uncommitted data), non-repeatable read (a row changes between two reads), phantom read (a range query gains or loses rows).

Intermediate Why does @Transactional silently do nothing when a method is called from within the same class?

Self-invocation. Declarative transactions work through a proxy, and only calls that enter the bean through the proxy are intercepted. An internal call via this goes straight to the target method, bypassing the proxy, so the transaction advice never runs. The code compiles and executes — it just has no transaction, which is why it is such a common production bug.

Intermediate Does @Transactional work on private methods? Why or why not?

No. Spring's CGLIB proxy works by subclassing and overriding methods, and private methods cannot be overridden — the same reason final methods and classes are not advised. Only public (and, since Spring 6, protected/package-visible with class-based proxies) methods reached through the proxy get a transaction. Annotating a private method is silently ignored.

Intermediate When would you use programmatic transactions instead of @Transactional?

When you need finer control than a method boundary: committing part of a loop, deciding at runtime whether to roll back, or multiple short transactions inside one method. Use TransactionTemplate (call status.setRollbackOnly() to abort) or a PlatformTransactionManager directly. Declarative stays the default; programmatic is the escape hatch.

Pro How do you fix a self-invocation problem — list the options?

Three practical fixes: (1) move the transactional method into a separate bean and inject it, so the call crosses a proxy boundary; (2) self-inject the bean's own proxy (via @Autowired or @Lazy) and call through that reference; (3) switch to AspectJ weaving mode (@EnableTransactionManagement(mode = AdviceMode.ASPECTJ)), which weaves the advice into the bytecode so self-calls and even private methods are covered. Refactoring to a separate bean is usually the cleanest.

Pro What causes UnexpectedRollbackException?

An inner REQUIRED method threw, marking the shared transaction rollback-only, but the caller caught the exception and tried to commit anyway. At commit time Spring sees the rollback-only flag, refuses to commit, and throws UnexpectedRollbackException. The lesson: with REQUIRED propagation, a failure anywhere in the logical transaction dooms the whole physical transaction — catching the inner exception does not undo the rollback marker. REQUIRES_NEW or NESTED isolates the failure if you truly need the outer to survive.

Pro Is the isolation level on a joining @Transactional method always applied?

No. Isolation (and timeout) apply only when that method actually starts a physical transaction. If it uses REQUIRED and simply joins an existing transaction, its declared isolation is silently ignored — the outer transaction's level stands. To guarantee a level, the method must create the transaction, e.g. REQUIRES_NEW, or be the outermost REQUIRED boundary. Spring can be configured to fail loudly on such conflicting declarations, but by default it ignores them.
