Module: Logging, Caching and Scheduling

Three cross-cutting concerns Spring collapses into a single annotation each — a logger call, @Cacheable, @Scheduled — and each hides a proxy, a thread pool, or a self-invocation trap that interviewers reliably probe.

Knowledge Library → Courses → Spring Framework → Logging, Caching and Scheduling

What this module covers

This module groups the framework's "auxiliary" features — the ones almost every service turns on but few study in depth. Logging through the SLF4J facade with Logback and MDC; the declarative caching abstraction (@Cacheable, @CachePut, @CacheEvict); and background execution via @Scheduled and @Async. Caching, scheduling and async all run through Spring AOP proxies, so the self-invocation trap you met in transactions returns here — which is exactly why interviewers pair them.

Topics

Logging: SLF4J, Logback and MDC

SLF4J is a facade — your code compiles against one API (LoggerFactory.getLogger, log.info("...{}...", arg)) while the actual engine is chosen at deploy time; Spring Boot wires Logback as the default binding. Prefer the parameterized {} form over string concatenation: it skips building the message string when the level is disabled. Levels (TRACE/DEBUG/INFO/WARN/ERROR) are set per logger, in Boot usually via logging.level.* properties, and the layout, file rotation and appenders live in logback-spring.xml.

Why it matters: MDC (Mapped Diagnostic Context) is the feature that separates a screening answer from a production one. It is a per-thread key/value map — stamp a traceId or userId once and every log line on that thread carries it, making one request traceable across a noisy multi-threaded log. Always clear it in a finally block so a pooled thread does not leak context into the next request.

private static final Logger log = LoggerFactory.getLogger(OrderService.class);

MDC.put("traceId", traceId);            // stamps every log line on this thread
try {
    log.info("Placed order {} for customer {}", orderId, customerId);
} finally {
    MDC.clear();                        // pooled thread must not leak context
}

The caching abstraction: @Cacheable and friends

Turn caching on with @EnableCaching, then annotate methods. @Cacheable checks the store first and returns the cached value without running the method on a hit; @CachePut always runs the method and refreshes the entry; @CacheEvict removes entries (one key, or allEntries = true) to prevent staleness. The abstraction is storage-agnostic: the same annotations sit over an in-memory ConcurrentMapCache, Caffeine, or Redis — you swap the CacheManager bean, not the code.

Why it matters: the interview lives in the edges. Keys default to the method arguments but are customizable with SpEL (key = "#isbn"); because caching is proxy-based, a cached method called from within the same bean bypasses the cache entirely (self-invocation); and the default map cache has no TTL or size bound, so production almost always means Caffeine or Redis with an eviction policy.

@Cacheable(value = "books", key = "#isbn")
public Book findByIsbn(String isbn) {
    return slowRepositoryLookup(isbn);   // skipped on a cache hit
}

@CacheEvict(value = "books", key = "#book.isbn")
public void update(Book book) { repo.save(book); }

Scheduling: @Scheduled and cron

Enable it with @EnableScheduling, then mark a no-argument, void method @Scheduled. Three timing modes: fixedRate starts runs at a constant period regardless of how long each takes; fixedDelay waits a gap after each run finishes; and cron pins runs to wall-clock times. Spring's cron has six fields — second, minute, hour, day-of-month, month, day-of-week — one more than classic Unix cron, and accepts a zone so a nightly job fires in the right timezone.

Why it matters: by default all scheduled methods share a single thread, so one slow task delays every other — configuring a ThreadPoolTaskScheduler is the standard fix. And in a multi-instance deployment every node fires the same job, causing duplicate work; the accepted answers are a distributed lock (ShedLock) or a leader-only design.

@Scheduled(fixedDelay = 5000)                        // 5s gap after each finish
public void pollQueue() { drain(); }

@Scheduled(cron = "0 0 2 * * *", zone = "Asia/Kolkata")   // 2 AM IST, daily
public void nightlyReport() { buildAndEmail(); }

Asynchronous execution: @Async and executors

Switch on @EnableAsync and any @Async method call returns immediately, running on a background thread from a TaskExecutor. A void method fires and forgets; return CompletableFuture<T> when the caller needs the result or must join later. The catch is the default executor: Spring falls back to SimpleAsyncTaskExecutor, which spawns a new thread per task and pools nothing — for production you define a ThreadPoolTaskExecutor bean (core size, max size, queue capacity) and name it in @Async("taskExecutor").

Why it matters: like caching and scheduling, @Async is proxy-based, so self-invocation runs synchronously and only public methods are advised. Exceptions from a void async method vanish unless you register an AsyncUncaughtExceptionHandler; with a CompletableFuture they surface when you resolve it. Combining @Async with @Transactional is a classic trap — the new thread starts a fresh transaction, it does not inherit the caller's.

@Async("taskExecutor")
public CompletableFuture<Report> build(Long id) {
    Report r = generate(id);                 // runs off the request thread
    return CompletableFuture.completedFuture(r);
}

Interview questions

These auxiliary features are staple Indian-interview territory — asked to check whether you have run Spring in production, not just built a demo:

Beginner What is SLF4J and why log through it instead of Logback directly?

SLF4J is a facade — a common logging API your code compiles against, with the real implementation (Logback, Log4j2, java.util.logging) bound at deploy time. It keeps application code independent of the logging engine, so you can switch backends without touching business code. Spring Boot ships SLF4J over Logback by default.

Beginner Why prefer log.info("id={}", id) over log.info("id=" + id)?

The parameterized {} form defers building the message string until SLF4J confirms the level is enabled. With string concatenation the string is built every time, even when DEBUG is off — wasted work on a hot path. It also avoids manual isDebugEnabled() guards.

Beginner How do you enable caching and mark a method's result as cached?

Put @EnableCaching on a configuration class, then annotate the method with @Cacheable("cacheName"). On the first call the method runs and its return value is stored under a key derived from the arguments; later calls with the same key return the cached value without executing the method.

Beginner Difference between fixedRate and fixedDelay in @Scheduled?

fixedRate triggers runs at a constant interval measured from each start, regardless of how long the task takes (runs can overlap or queue). fixedDelay measures the gap after the previous run finishes, so executions never overlap. Use fixedDelay when a run must fully complete before the next begins.

Intermediate What is MDC and why is it essential in a web service?

Mapped Diagnostic Context is a per-thread key/value map the logging framework merges into each log line. Stamp a request/trace id (often in a servlet filter or interceptor) and every log statement while handling that request carries it, so you can grep one request out of interleaved multi-threaded logs. Clear it in a finally block because thread-pool threads are reused.

Intermediate @Cacheable vs @CachePut vs @CacheEvict?

@Cacheable may skip the method — returns the cached value on a hit. @CachePut always executes the method and stores the fresh result (use it for updates that must run). @CacheEvict removes entries — one key, or allEntries = true — to invalidate stale data. A common pattern: @Cacheable on reads, @CacheEvict/@CachePut on writes.

Intermediate How is a cache key computed, and how do you customize it?

By default a KeyGenerator builds the key from the method arguments (a single arg becomes the key directly; multiple args combine). Customize with SpEL: key = "#isbn" or key = "#user.id". For no-arg or overloaded methods, an explicit key avoids collisions. You can also gate caching with condition and unless.

Intermediate How do you run an @Async method on a custom thread pool?

Define a ThreadPoolTaskExecutor bean with core size, max size and queue capacity, then reference it by name: @Async("taskExecutor"). Without a defined executor Spring uses SimpleAsyncTaskExecutor, which creates a new thread per task and does not pool — unbounded thread creation under load. Configuring a bounded pool is the production requirement.

Intermediate Why does @Cacheable / @Async / @Scheduled sometimes "not work" when the method is called from the same class?

All three are implemented with Spring AOP proxies. A self-invocation — one method of a bean calling another method of the same bean — goes through this, not the proxy, so the caching/async/scheduling advice never fires. Fixes: move the annotated method to a separate bean, inject a proxy reference to self, or restructure the call so it crosses the proxy boundary. This is the same trap as @Transactional self-invocation.

Intermediate Describe the Spring cron format. How many fields, and what does "0 0 2 * * *" mean?

Spring's cron has six fields: second, minute, hour, day-of-month, month, day-of-week — one more (seconds) than standard Unix cron. 0 0 2 * * * means second 0, minute 0, hour 2, every day of month, every month, every day of week — i.e. daily at 2 AM. Add zone = "Asia/Kolkata" to pin it to IST rather than the server default.

Pro A @Scheduled job runs on every instance in a 3-node cluster and does duplicate work. How do you fix it?

@Scheduled is node-local — each JVM fires independently, so a horizontally scaled deployment runs the job N times. Solutions: a distributed lock so only one node executes per tick (ShedLock is the common library), a leader-election / single-designated-runner design, or externalizing the schedule to a cluster-aware scheduler (Quartz with a shared JDBC store). Naive fixes like a DB flag race unless the check-and-set is atomic.

Pro What happens to exceptions and return values from an @Async method?

A void async method that throws sends the exception nowhere by default — register an AsyncUncaughtExceptionHandler (via AsyncConfigurer) to observe it. A method returning CompletableFuture (or Future) captures the exception; it surfaces when the caller resolves the future (get()/join() or completion callbacks). Rule of thumb: return a future whenever the result or failure matters.

Pro Can @Transactional and @Async be combined, and what breaks if you assume they compose?

They can, but the transaction does not cross the thread boundary. When a caller invokes an @Async method, the work runs on a new thread that opens its own transaction (transactional state is thread-bound) — it does not join the caller's. So an async method cannot roll back the caller's transaction, and lazy-loaded entities from the caller's persistence context are detached on the new thread. Design each async unit to own its transaction, and never expect atomicity across the async call.
