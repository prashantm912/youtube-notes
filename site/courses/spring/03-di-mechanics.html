Module: Dependency Injection Mechanics

Once the container knows your beans, the next question is how it hands each one its collaborators — and the choice between constructor, setter and field injection is the single most common Spring design question in Indian interviews.

Knowledge Library → Courses → Spring Framework → Dependency Injection Mechanics

What this module covers

This module is the wiring layer: the three ways Spring injects dependencies (constructor, setter, field) and why constructor injection is the modern default, how @Autowired resolves a bean by type and then by name, how @Primary and @Qualifier break ties when several beans match, what circular dependencies are and how @Lazy defuses them, and how @Value pulls external configuration into your beans.

Topics

The three ways to inject: constructor, setter, field

Spring can supply a bean's collaborators three ways. Constructor injection passes every mandatory dependency at creation time, so the object is never half-built and its fields can be final (immutable). Setter injection wires through setter methods and suits genuinely optional or reconfigurable dependencies. Field injection — @Autowired straight onto a private field — is the shortest to type and the one to avoid: it needs reflection, cannot produce final fields, and hides how many dependencies a class really carries.

Why it matters: "which injection style do you use, and why?" is a near-guaranteed opener. Since Spring 4.3 a bean with a single constructor does not even need @Autowired, which is why idiomatic modern code favours a final-field constructor — often generated by Lombok's @RequiredArgsConstructor.

@Service
class OrderService {
    private final PaymentGateway gateway;   // final: set once, never reassigned
    OrderService(PaymentGateway gateway) {  // single constructor, no @Autowired needed
        this.gateway = gateway;
    }
}

@Autowired and dependency resolution

@Autowired tells the container to satisfy a dependency by type. Spring gathers every bean assignable to the required type: exactly one match is injected, no match fails fast (unless required = false, Optional<T>, or @Nullable make it optional), and several matches are narrowed by @Primary/@Qualifier/name before Spring gives up with NoUniqueBeanDefinitionException. The same annotation works on constructors, setters, fields, and arbitrary methods.

Why it matters: knowing the by-type-then-by-name resolution order is what lets you predict — and fix — the two failures every Spring developer eventually hits, "no qualifying bean of type" and "no unique bean". It also explains why renaming a field can quietly change which bean gets injected.

@Autowired(required = false)
private AuditLogger auditLogger;      // left null when no such bean exists

@Autowired
public void configure(Optional<RateProvider> provider) {
    provider.ifPresent(this::applyRates);   // optional collaborator
}

Disambiguation: @Primary and @Qualifier

When two beans satisfy the same type, you tell Spring which to prefer. @Primary marks one bean as the default winner for every unqualified injection point — right when there is an obvious "normal" implementation. @Qualifier is the opposite tool: it names a specific bean at the injection point, and it overrides @Primary when both are present. Qualifiers can even be custom annotations, giving type-safe selection instead of magic strings.

Why it matters: "@Qualifier vs @Primary" is a classic follow-up. The crisp answer interviewers want: @Primary is a producer-side default declared on the bean, @Qualifier is a consumer-side override declared at the injection site, and the consumer's @Qualifier wins.

@Bean @Primary            PaymentGateway razorpay() { ... }   // default choice
@Bean @Qualifier("stripe") PaymentGateway stripe()  { ... }

@Autowired @Qualifier("stripe")   // overrides @Primary at this one site
private PaymentGateway gateway;

Circular dependencies and @Lazy

A circular dependency is A needing B while B needs A. With constructor injection Spring cannot construct either first and throws BeanCurrentlyInCreationException at startup — and since Spring Boot 2.6 such cycles are rejected by default. Setter or field injection can sometimes resolve a cycle, because Spring creates both instances and wires them afterward. @Lazy breaks it a third way: Spring injects a lazy proxy and only creates the real bean on first use, by which point its counterpart already exists.

Why it matters: circular-dependency questions test whether you treat the error as a design smell — the right instinct is to extract a third collaborator or use application events — rather than papering over it. Know @Lazy and setter injection as the mechanical escape hatches, and know that Boot now fails fast on cycles.

@Service
class A {
    A(@Lazy B b) {          // a proxy is injected here...
        this.b = b;         // ...and the real B is built on first call through it
    }
}

@Value and property injection

@Value injects externalised configuration — a value from application.properties, an environment variable, or a computed expression — into a field or constructor parameter. A ${...} placeholder resolves a property (with the :default fallback syntax after the colon), while #{...} evaluates a SpEL expression. The properties themselves come from PropertySources registered with @PropertySource or, in Spring Boot, auto-loaded from application.properties/application.yml.

Why it matters: @Value is the everyday way to keep secrets and environment knobs out of source code, and interviewers probe the ${}-vs-#{} distinction and the default syntax. When many related keys belong together, the expected follow-up answer is to bind a typed object with @ConfigurationProperties instead.

@Value("${app.timeout:5000}")   // property placeholder, defaults to 5000 if absent
private int timeoutMs;

@Value("#{systemProperties['user.region'] ?: 'IN'}")   // SpEL expression
private String region;

Interview questions

The most-asked DI-mechanics questions in Indian interviews — constructor-vs-field injection is the near-universal opener, so the early questions build toward it:

Beginner What are the three types of dependency injection in Spring?

Constructor injection (dependencies passed to the constructor), setter injection (through setter methods), and field injection (@Autowired directly on a field). Constructor injection is the recommended default; field injection is discouraged.

Beginner Is @Autowired always mandatory?

No. On a bean with a single constructor Spring injects its parameters automatically without @Autowired (since 4.3). You still need it to mark one of several constructors, or to request setter/field/method injection.

Beginner Difference between constructor and setter injection?

Constructor injection supplies all dependencies at creation, enabling final fields and guaranteeing a fully-initialised object — use it for mandatory dependencies. Setter injection lets a dependency be omitted or reset later — use it for optional ones. Setter injection can also resolve some circular dependencies that constructor injection cannot.

Beginner What happens if no bean, or multiple beans, match an @Autowired type?

No match throws NoSuchBeanDefinitionException at startup (unless the point is required = false, Optional, or @Nullable). Multiple matches throw NoUniqueBeanDefinitionException, unless disambiguated with @Primary or @Qualifier.

Intermediate Why is constructor injection preferred over field injection?

It allows final/immutable fields, guarantees the object is never used half-initialised, makes dependencies explicit (a long parameter list openly signals too many responsibilities), needs no reflection so the class is unit-testable with a plain new, and surfaces circular dependencies at startup. Field injection cannot do final fields, hides the dependency count, needs reflection or a running container to test, and lets cycles slip through. Spring's own reference no longer lists field injection as a recommended option.

Intermediate @Qualifier vs @Primary — when do you use each?

@Primary marks one bean as the default for all unqualified injection points (producer-side default). @Qualifier names a specific bean at a particular injection point (consumer-side selection). When both apply, @Qualifier wins. Use @Primary for the common case and @Qualifier where a specific site needs the non-default bean.

Intermediate How does Spring decide which bean to autowire?

By type first — it collects every candidate assignable to the target type. One candidate is injected directly. With several, it applies @Primary, then matches a @Qualifier value, then falls back to matching the bean name against the field or parameter name. If it is still ambiguous, or nothing matches, it throws.

Intermediate What is a circular dependency and how do you resolve it?

Bean A depends on B and B on A. With constructor injection Spring cannot build either first, giving BeanCurrentlyInCreationException (and Spring Boot 2.6+ forbids cycles by default). Fixes, best first: redesign to remove the cycle (extract a third bean, or use application events); use setter/field injection on one side; or annotate one dependency @Lazy so a proxy is injected.

Intermediate What does @Value do, and how do ${} and #{} differ?

@Value injects an external value into a field or parameter. ${property.name:default} resolves a property placeholder from the Environment/PropertySources, with an optional default after the colon. #{expression} evaluates a SpEL expression. For many related keys, prefer binding a typed object with @ConfigurationProperties.

Pro How exactly does @Lazy break a circular dependency?

@Lazy at the injection point makes Spring inject a lazy proxy rather than the real bean. The proxy is created immediately so the constructor completes; the target bean is instantiated only when a method is first invoked through the proxy — by which time the other bean already exists, so the cycle never has to be resolved at construction time.

Pro Why does constructor injection catch circular dependencies at startup while field injection can hide them?

Constructor injection needs both beans fully built before either constructor can run, so a cycle is unresolvable and fails immediately. Field injection sets the field after both objects are instantiated, so Spring can wire a cycle without complaint — the design flaw stays hidden until it causes a subtler runtime issue. That fail-fast behaviour is itself a reason to prefer constructor injection.

Pro Can you autowire several beans of the same type at once?

Yes. Declaring the dependency as a List, Set, or array of the type injects all matching beans; a Map<String, T> injects them keyed by bean name. Iteration order follows @Order/Ordered (or @Priority). This is how plugin-style designs — a list of validators or handlers — get wired without hard-coding each one.

Pro Spring Boot 2.6 prohibits circular references by default — what changed, and how do you handle a legacy app that relies on one?

Boot 2.6 flipped spring.main.allow-circular-references to false, so a cycle now fails at startup instead of being silently tolerated. The correct fix is to break the cycle (refactor, @Lazy, or events). The temporary escape is to set spring.main.allow-circular-references=true — but treat that as tech debt, since the cycle signals coupling that will bite later.
