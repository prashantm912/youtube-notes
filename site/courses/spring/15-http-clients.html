Module: HTTP Clients and Server Push

A Spring app is rarely alone — it calls other services and streams events back to browsers. This closing module maps the client that Spring now recommends, the reactive and declarative alternatives, the two push protocols, and the Framework 7 platform features that ship underneath them all.

Knowledge Library → Courses → Spring Framework → HTTP Clients and Server Push

What this module covers

Two directions of traffic. Outbound: how a Spring service consumes other HTTP APIs — RestClient (the modern synchronous default), the legacy RestTemplate, the reactive WebClient, and declarative @HttpExchange interfaces. Inbound push: streaming to clients over Server-Sent Events (SseEmitter) and full-duplex WebSocket/STOMP. It closes with the Framework 7 built-ins that shape all of the above — JSpecify null-safety and ahead-of-time (AOT) processing for GraalVM native images.

Topics

RestClient and the move away from RestTemplate

RestClient (Spring Framework 6.1+) is the client Spring now points you to for synchronous, blocking HTTP calls. It offers WebClient's fluent, chainable API — get().uri(...).retrieve().body(Type.class) — but on the classic servlet infrastructure, so it works in a plain Spring MVC app with no reactive dependency. RestTemplate, the workhorse of the last decade, is now in maintenance mode: still supported, no longer the recommendation, and bloated with dozens of overloaded methods.

Why it matters: "which HTTP client should new code use?" is a live interview question because the answer changed. Knowing that RestClient is the default, that RestTemplate is legacy-but-not-removed, and that both are synchronous (unlike WebClient) is exactly the distinction interviewers probe.

User user = RestClient.create()
    .get()
    .uri("https://api.example.com/users/{id}", id)
    .retrieve()
    .body(User.class);   // synchronous, blocking

WebClient — the reactive, non-blocking client

WebClient is the fully non-blocking, reactive HTTP client from Spring WebFlux. Instead of returning a value and blocking the thread, it returns a Mono<T> or Flux<T> that the caller composes into an asynchronous pipeline — a handful of event-loop threads can hold thousands of in-flight requests. Reach for it when the calling application is itself reactive, when you need streaming or high-concurrency fan-out, or when backpressure matters.

Why it matters: the trap is assuming WebClient is always "better." In an ordinary blocking MVC service, RestClient gives the same fluent ergonomics without dragging in Reactor. The senior answer names the thread model — blocking vs event-loop — as the deciding factor, not the API style.

Mono<User> user = webClient.get()
    .uri("/users/{id}", id)
    .retrieve()
    .bodyToMono(User.class);   // nothing runs until subscribed

Declarative HTTP clients with @HttpExchange

Instead of writing call code, you declare a Java interface: annotate it with @HttpExchange and its methods with @GetExchange, @PostExchange and friends, and Spring generates a proxy that performs the exchanges — the same idea as Spring Data repositories or OpenFeign, but built into the framework with no external library. The proxy is backed by a RestClient or WebClient, so you keep the client's transport while writing zero boilerplate.

Why it matters: Framework 7 / Spring Boot 4 push this further with a registry — @ImportHttpServices and custom registrars replace hand-wired HttpServiceProxyFactory configuration, making declarative clients the ergonomic default for service-to-service calls. Interviewers like the contrast: declarative interface vs imperative call, and where the runtime proxy actually comes from.

@HttpExchange("/users")
public interface UserClient {
    @GetExchange("/{id}")
    User byId(@PathVariable Long id);
}
// Spring builds a proxy backed by a RestClient/WebClient

Server-Sent Events (SSE) with SseEmitter

Server-Sent Events is a one-directional streaming standard: the client opens a single long-lived HTTP connection and the server pushes a sequence of named, id-tagged text events over it. In Spring MVC you return an SseEmitter from a controller (content type text/event-stream), hold onto it, and call send() from another thread as data arrives, then complete(). It fits notifications, progress updates, live feeds and log tails — anything server-to-client.

Why it matters: SSE is the lightweight answer to "how do I push updates to a browser?" — it rides ordinary HTTP, auto-reconnects, and needs no extra protocol. The exam question is knowing its boundary: unidirectional and text-only, so when the client must also send, you graduate to WebSocket.

@GetMapping(path = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter stream() {
    SseEmitter emitter = new SseEmitter();
    // from another thread: emitter.send("tick"); ... emitter.complete();
    return emitter;
}

WebSocket and STOMP messaging

WebSocket (RFC 6455) upgrades a single HTTP connection into a persistent full-duplex channel — both sides send frames at any time, making it the transport for chat, collaborative editing, live dashboards and multiplayer state. Raw WebSocket only moves bytes, so Spring layers STOMP (Simple Text Oriented Messaging Protocol) on top: a frame format with destinations, so you get a lightweight message broker, @MessageMapping controller methods, subscriptions, and pub/sub semantics instead of a bare socket.

Why it matters: interviewers want the decision tree, not just an API. WebSocket earns its cost when communication is genuinely bidirectional and low-latency; STOMP earns its cost when you want routing and broker features rather than parsing raw frames by hand. Both add operational weight SSE avoids.

Null-safety with JSpecify (Framework 7)

Spring Framework 7 annotates its entire codebase with JSpecify annotations — @Nullable, @NonNull, @NullMarked, @NullUnmarked — replacing the older org.springframework.lang ones. A package marked @NullMarked is non-null by default, so only genuinely nullable spots are flagged, and JSpecify now expresses nullness of array/varargs elements and generic type arguments too. Tools such as NullAway (and IDEs, and Kotlin) read these to catch potential NullPointerException at build time instead of in production.

Why it matters: this is a headline Framework 7 feature and a natural "what's new" question. The point to make is that null-safety is a build-time consistency check driven by a vendor-neutral standard — not a runtime cost — and that consuming Spring's null-safe APIs from a JSpecify-aware IDE surfaces mistakes as you type.

@NullMarked                       // whole package non-null by default
package com.example.app;

public @Nullable User findById(Long id) { ... }   // absence is explicit

AOT processing and GraalVM native images

Ahead-of-Time (AOT) processing inspects the ApplicationContext at build time and pre-computes work that normally happens on startup — bean definitions, proxy setup, configuration parsing — emitting generated code and metadata. Its main purpose is to let a Spring app compile into a GraalVM native image: a self-contained executable that starts in milliseconds with a small memory footprint, at the price of build-time complexity. Because native images do no runtime classpath scanning, anything reflective, dynamic-proxy, or resource-loaded must be declared through RuntimeHints.

Why it matters: AOT and native images are the serverless/scale-to-zero story for Spring, and "why does my reflection break in a native image, and how do runtime hints fix it?" is a real production question. Even where you never ship native, AOT on the JVM (spring.aot.enabled) trims startup work.

Interview questions

Most-asked HTTP-client and push questions in Indian interviews — the client recommendation changed recently, so screening rounds test whether you have kept up:

Beginner What is RestClient and why did Spring introduce it?

A synchronous, blocking HTTP client (Spring Framework 6.1+) with a fluent API — get().uri(...).retrieve().body(Type.class). It gives WebClient's ergonomics on the classic servlet stack, without a reactive dependency, and is now the recommended default for making HTTP calls from a Spring MVC application.

Beginner RestTemplate or RestClient for new code?

RestClient. RestTemplate still works and is not removed, but it is in maintenance mode — no new features, a bloated overloaded API — and Spring documents RestClient as its replacement. Existing RestTemplate code is fine to keep; new code should default to RestClient.

Beginner What is WebClient and when would you pick it over RestClient?

WebClient is the non-blocking, reactive client from Spring WebFlux; it returns Mono/Flux and runs on an event loop. Pick it when the app is reactive, when you need high-concurrency non-blocking I/O, streaming, or backpressure. For ordinary blocking MVC code, RestClient is the simpler choice.

Beginner What is an @HttpExchange HTTP interface?

A declarative client: you define a Java interface, annotate it with @HttpExchange / @GetExchange / @PostExchange, and Spring generates a proxy that performs the calls — no implementation code. Same idea as Spring Data repositories or OpenFeign, but built into the framework.

Intermediate How does Spring turn an @HttpExchange interface into a working client?

At runtime a proxy is created (classically via HttpServiceProxyFactory) that is backed by a RestClient or WebClient; the proxy reads the annotations to build each request. In Framework 7 / Boot 4 a registry with @ImportHttpServices and registrars replaces the manual factory wiring.

Intermediate RestClient vs WebClient — the real difference?

Thread model, not API style. RestClient blocks the calling thread and returns the value directly; WebClient never blocks and returns a reactive publisher composed on an event loop. Their fluent APIs look almost identical, which is deliberate — but only one holds a thread per in-flight request.

Intermediate What are Server-Sent Events, and how does SseEmitter work?

SSE is a unidirectional server-to-client stream over one long-lived HTTP connection using text/event-stream. In Spring MVC you return an SseEmitter from a controller, keep the reference, and call send() from another thread as data arrives, then complete(). The browser's EventSource auto-reconnects.

Intermediate WebSocket vs STOMP — what does STOMP add?

WebSocket is the transport — a full-duplex byte channel. STOMP is a text sub-protocol layered on it that adds destinations, subscriptions and a frame format, giving you a lightweight message broker, @MessageMapping controllers and pub/sub instead of hand-parsing raw frames.

Intermediate How do you handle error status codes with RestClient?

retrieve() throws on 4xx/5xx by default; register onStatus(predicate, handler) to translate specific statuses into your own exceptions, or use exchange(...) for full control over the response. This is cleaner than RestTemplate's ResponseErrorHandler wiring.

Pro SSE vs WebSocket vs long-polling — how do you choose?

Long-polling: broadest compatibility, highest overhead — a fallback. SSE: server-to-client only, plain HTTP, auto-reconnect, text — ideal for notifications and feeds. WebSocket (usually with STOMP): full-duplex and low-latency for chat/collaboration/games, at the cost of a persistent connection and more infrastructure. Choose the lightest tool the interaction pattern allows.

Pro What is JSpecify null-safety in Spring Framework 7 and what does it buy you?

Spring 7 annotates its APIs with the vendor-neutral JSpecify annotations (@Nullable, @NonNull, @NullMarked). A @NullMarked package is non-null by default, and nullness now covers generics and array/varargs elements. Tools like NullAway, IDEs and Kotlin read it to flag potential NPEs at build time — a static check, not a runtime cost.

Pro What is AOT processing, and why is it required for GraalVM native images?

AOT inspects the ApplicationContext at build time and pre-generates bean-registration code and metadata, moving startup work to compile time. Native images do no runtime classpath scanning or dynamic loading, so AOT is mandatory there — and every use of reflection, dynamic proxies or bundled resources must be declared via RuntimeHints or GraalVM cannot see it. AOT also runs on the JVM (spring.aot.enabled) to speed startup.
