Module: Spring MVC — Interceptors and Web

Once the DispatcherServlet has routed a request, the web layer still has real work left: cross-cutting interceptors, the servlet filters running beneath them, static assets, view rendering and file uploads — and telling a filter from an interceptor is one of Spring MVC's most-asked interview questions.

What this module covers

This module rounds out Spring MVC beyond routing and controllers. It covers the HandlerInterceptor and its three-phase lifecycle, how interceptors differ from plain servlet Filters (the single most common web-layer interview question), how the framework serves and fingerprints static resources, how view names become rendered pages through the ViewResolver chain, and how multipart requests turn into MultipartFile uploads. Everything here sits inside — or just outside — the DispatcherServlet you met in the previous module.

Topics

HandlerInterceptor and its lifecycle

A HandlerInterceptor wraps controller execution with three callbacks: preHandle runs before the handler and returns a boolean (return false to short-circuit the request), postHandle runs after the handler but before the view renders and can still tweak the ModelAndView, and afterCompletion runs after rendering finishes — even when an exception was thrown — which makes it the right place for cleanup and timing. Interceptors are Spring-managed, so they see the resolved handler and can be scoped to URL patterns.

Why it matters: interceptors are where you put cross-cutting concerns that are specific to your application and need handler context — request timing, auditing, populating a per-request locale or tenant, or logging with an MDC. You register them declaratively through WebMvcConfigurer.addInterceptors, with include/exclude path patterns, rather than wiring them into every controller.

@Configuration
class WebConfig implements WebMvcConfigurer {
  @Override
  public void addInterceptors(InterceptorRegistry registry) {
    registry.addInterceptor(new TimingInterceptor())
            .addPathPatterns("/api/**")        // only these
            .excludePathPatterns("/api/health"); // but not this
  }
}

Filters vs interceptors

Both intercept requests, but they live in different worlds. A servlet Filter belongs to the Servlet container: it runs on the raw HttpServletRequest/Response before the DispatcherServlet even sees the request, knows nothing about which controller will handle it, and can wrap or replace the request/response. A HandlerInterceptor lives inside the DispatcherServlet: it runs after handler mapping, so it knows the target handler, participates in the model/view phase, and is scoped by Spring MVC path patterns. Filters can do things interceptors cannot — modify the response body via wrapping, apply to every servlet request including static resources — while interceptors get handler context filters lack.

Why it matters: this comparison is a staple of Indian backend interviews, and the practical rule follows from the architecture. Use a filter for concerns that are truly container-level and handler-agnostic (security, CORS, request-ID/MDC setup, compression, ETag); this is exactly why Spring Security is built on the filter chain and works even outside Spring MVC. Use an interceptor when you need the resolved handler or the ModelAndView. Extend OncePerRequestFilter so a filter is not re-invoked across async dispatches and forwards.

@Component
class RequestIdFilter extends OncePerRequestFilter {
  @Override
  protected void doFilterInternal(HttpServletRequest req,
      HttpServletResponse res, FilterChain chain)
      throws ServletException, IOException {
    MDC.put("requestId", UUID.randomUUID().toString());
    chain.doFilter(req, res);   // hand control down the chain
  }
}

Serving static resources

Spring MVC serves CSS, JavaScript, images and other static files through a dedicated ResourceHttpRequestHandler, configured with addResourceHandlers: you map a URL pattern such as /assets/** to one or more locations (a classpath folder, the web root, or WebJars) and set cache headers. On top of that, a resource chain of ResourceResolvers and ResourceTransformers adds production features — most importantly content-based versioning (fingerprinting) via VersionResourceResolver, so app.js is served as app-8e0f...js and can be cached for a year without stale-asset problems.

Why it matters: static handling is where caching, cache-busting and CDN-friendly URLs are decided. In Spring Boot the defaults already serve classpath:/static, /public, /resources and /META-INF/resources, but interviewers expect you to know how to override locations, set CacheControl, and enable fingerprinting for the frontend build output.

@Override
public void addResourceHandlers(ResourceHandlerRegistry registry) {
  registry.addResourceHandler("/assets/**")
          .addResourceLocations("classpath:/static/")
          .setCacheControl(CacheControl.maxAge(Duration.ofDays(365)))
          .resourceChain(true)
          .addResolver(new VersionResourceResolver()
              .addContentVersionStrategy("/**")); // fingerprinting
}

View resolution

When a controller returns a logical view name like "orders", a ViewResolver turns that string into an actual View that renders the response. InternalResourceViewResolver maps names to JSPs with a prefix/suffix (/WEB-INF/views/ + .jsp); dedicated resolvers exist for Thymeleaf, FreeMarker and others; and ContentNegotiatingViewResolver picks a view by the requested media type. Two prefixes bypass the mapping entirely: redirect: issues an HTTP redirect and forward: forwards the request server-side. Multiple resolvers can be chained by order, with the JSP resolver placed last because it cannot report that a view is missing.

Why it matters: view resolution is the server-rendered half of Spring MVC — still common for admin panels, emails and classic web apps even in an API-first world. Knowing prefix/suffix mapping, the redirect:/forward: prefixes, and why resolver ordering matters heads off the classic "circular view path" error and explains how the same controller can serve HTML or a downloadable view by content negotiation.

@Bean
InternalResourceViewResolver viewResolver() {
  var r = new InternalResourceViewResolver();
  r.setPrefix("/WEB-INF/views/");
  r.setSuffix(".jsp");
  return r;                       // "orders" -> /WEB-INF/views/orders.jsp
}
// return "redirect:/orders/42"  -> 302 to the client
// return "forward:/legacy"      -> server-side forward, URL unchanged

Multipart and file uploads

Multipart requests (multipart/form-data) carry file uploads. Spring MVC resolves them through a MultipartResolver strategy — the modern default, StandardServletMultipartResolver, delegates to the Servlet container's own parsing configured by MultipartConfigElement (temp location, max file size, max request size). Once resolved, each uploaded part is exposed to your controller as a MultipartFile: bind a simple field with @RequestParam("file") MultipartFile, accept several with List<MultipartFile>, and use @RequestPart when a part is itself structured data (for example JSON metadata alongside the binary file). MultipartFile gives you the original filename, content type, size, bytes/stream and a convenient transferTo.

Why it matters: upload endpoints are everywhere — avatars, documents, bulk imports — and the interview follow-ups are predictable: how size limits are enforced, the difference between @RequestParam and @RequestPart, and where the temp file lives. Getting the resolver and container limits right is also a security and reliability concern, not just a convenience.

@PostMapping("/upload")
public String upload(@RequestParam("file") MultipartFile file)
    throws IOException {
  if (!file.isEmpty()) {
    file.transferTo(Path.of("/data", file.getOriginalFilename()));
  }
  return "redirect:/files";
}

Interview questions

The web-layer questions that recur across Indian Spring/Java interviews — from screening rounds to product-company deep dives; filters vs interceptors is almost guaranteed:

Beginner What is a HandlerInterceptor and where does it sit in the request flow?

A Spring MVC component that wraps controller execution. It runs inside the DispatcherServlet, after handler mapping has chosen the controller, so it has access to the resolved handler and to the model/view. It is the place for application-specific cross-cutting logic like timing, auditing or per-request logging.

Beginner What are the three HandlerInterceptor methods and when does each run?

preHandle — before the controller; returns a boolean, and false stops the request. postHandle — after the controller but before the view renders; can modify the ModelAndView. afterCompletion — after the view has rendered (or an exception occurred); used for cleanup. All three are default methods, so you override only what you need.

Beginner How do you register an interceptor and limit it to certain URLs?

Implement WebMvcConfigurer and override addInterceptors(InterceptorRegistry), calling registry.addInterceptor(...) and chaining addPathPatterns / excludePathPatterns. This is declarative — you never wire interceptors into individual controllers.

Intermediate Filter vs interceptor — what are the real differences and when do you use each?

A servlet Filter is container-level: it runs before the DispatcherServlet, on the raw request/response, knows nothing about the target handler, and applies to every servlet request (including static resources). A HandlerInterceptor is Spring MVC-level: it runs after handler mapping, knows the handler, can touch the ModelAndView, and is scoped by MVC path patterns. Use a filter for handler-agnostic, container-wide concerns (security, CORS, request-ID/MDC, compression, ETag); use an interceptor when you need the handler or view context. Only a filter can wrap/replace the request or response.

Intermediate If preHandle returns false, what happens to the rest of the chain?

The request is short-circuited: the controller and the remaining interceptors' preHandle methods are not invoked, and the view is not rendered. You are expected to have written the response yourself (for example a 401). afterCompletion is called only for interceptors whose preHandle already returned true.

Intermediate postHandle vs afterCompletion for cleanup — which and why?

afterCompletion. It always runs once processing is done — including when the handler or view threw — and it receives the exception, so resources are released reliably. postHandle runs only on the success path before rendering and is skipped when an exception occurs, so it is unsafe for cleanup. Note both fire in reverse registration order.

Intermediate How does Spring serve static resources, and how do you add a custom location?

Through ResourceHttpRequestHandler, configured via addResourceHandlers: map a pattern like /assets/** to locations such as classpath:/static/ and set CacheControl. Spring Boot serves /static, /public, /resources and /META-INF/resources by default; you override the pattern with spring.mvc.static-path-pattern and locations with spring.web.resources.static-locations.

Intermediate What is a ViewResolver, and how does InternalResourceViewResolver map a name to a page?

A ViewResolver converts a logical view name returned by a controller into a View that renders the response. InternalResourceViewResolver applies a prefix and suffix, so "orders" becomes /WEB-INF/views/orders.jsp. The redirect: and forward: prefixes bypass this to trigger a client redirect or a server-side forward.

Intermediate How does file upload work in Spring MVC, and what does MultipartFile give you?

A MultipartResolver (the default StandardServletMultipartResolver delegates to the Servlet container) parses multipart/form-data and exposes each part. In the controller you receive a MultipartFile via @RequestParam("file"), giving you the original filename, content type, size, byte/stream access and transferTo to persist it. Size limits are set through MultipartConfigElement / properties.

Intermediate @RequestParam vs @RequestPart for multipart uploads?

@RequestParam treats the part as a simple value — ideal for a MultipartFile or a plain form field. @RequestPart is for a structured part: it runs the matching HttpMessageConverter (for example Jackson) so you can bind a JSON metadata part to an object while a separate part carries the binary file, and it supports @Valid on that object.

Pro Why is Spring Security implemented as a servlet filter chain rather than as interceptors?

Because authentication and authorization must apply to every request before it reaches Spring MVC — including static resources, error dispatches and non-MVC servlets — and may need to wrap or halt the request/response entirely. Filters run at the container boundary and are independent of handler mapping, so security is enforced even when no controller would match. Interceptors run too late (after mapping) and are scoped to MVC paths, so they cannot provide the same guarantee.

Pro Why extend OncePerRequestFilter, and how do interceptors behave across async and forward dispatches?

A filter can be invoked multiple times for one logical request — on async re-dispatch, error dispatch or internal forwards. OncePerRequestFilter guarantees the logic runs exactly once per request, avoiding duplicated MDC entries, double logging or repeated header writes. For interceptors, an async handler returns from preHandle and MVC pauses; when the async result is ready the interceptor chain resumes for postHandle/afterCompletion. Use AsyncHandlerInterceptor.afterConcurrentHandlingStarted if you need a hook at the point processing goes async.

Pro How does static-resource fingerprinting work, and why chain resolvers for it?

Enable resourceChain(true) and add a VersionResourceResolver with a content version strategy: each file is served at a URL containing a hash of its content (app-8e0f...js), so the URL changes only when the file changes. That lets you set a one-year CacheControl without ever serving stale assets — the new deploy produces new URLs. Templates resolve the versioned URL through ResourceUrlProvider / ResourceUrlEncodingFilter, so you still reference the logical path. The chain also caches lookups so this costs little per request.
