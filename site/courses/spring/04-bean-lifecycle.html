Module: Bean Lifecycle and Scopes

The container decides how many of each bean exist and when they are born and destroyed — and the mismatches between those decisions (a request bean inside a singleton, a prototype that never runs its destroy method) are among the most reliable filters in an Indian Spring interview.

Knowledge Library → Courses → Spring Framework → Bean Lifecycle and Scopes

What this module covers

Once the container knows a bean's definition, two questions remain: how many instances should exist and how long each should live (scope), and what code should run as each instance is initialized and torn down (lifecycle callbacks). This module orients you on both, plus the extension points — BeanPostProcessor, BeanFactoryPostProcessor and FactoryBean — through which Spring itself, and you, hook into that machinery. Depth lives in the linked official docs; the interview section shows why this is such heavily tested ground.

Topics

Bean scopes: singleton, prototype, request, session, application, websocket

A scope governs how many instances the container creates and how long each lives. Spring ships six. singleton (the default) caches one shared instance per container; prototype returns a brand-new instance on every lookup. The remaining four are web-aware: request, session, application and websocket bind an instance to an HTTP request, an HTTP session, the ServletContext, or a WebSocket session respectively.

Why it matters: the questions that trip candidates are scope mismatches. A singleton is wired exactly once at startup, so if you inject a shorter-lived bean (a request/session bean, or a prototype) straight into it, the singleton captures one instance forever. The fix is to interpose a scoped proxy (proxyMode) or a provider (ObjectProvider, JSR-330 Provider) that resolves the correct instance per call. Note too that Spring's singleton is one-per-container, not the GoF one-per-JVM, and that the container never runs destroy callbacks on a prototype.

@Component
@Scope("prototype")                    // new instance on every lookup
class Engine { }

@Component
@Scope(value = "request",
       proxyMode = ScopedProxyMode.TARGET_CLASS)  // proxy lets a singleton hold it safely
class RequestContext { }

Lifecycle callbacks: @PostConstruct/@PreDestroy, InitializingBean/DisposableBean

Every managed bean travels create → inject dependencies → initialize → (in use) → destroy. You attach code to the two ends three ways: the JSR-250 annotations @PostConstruct and @PreDestroy (recommended — framework-agnostic), the Spring interfaces InitializingBean.afterPropertiesSet() and DisposableBean.destroy() (which couple your class to Spring), and the initMethod/destroyMethod attributes on @Bean (or init-method/destroy-method in XML). Alongside these, the Aware interfaces (BeanNameAware, ApplicationContextAware, and others) let a bean receive container infrastructure during the same initialization phase.

Why it matters: when a bean uses several of these at once, the order is fixed and gets asked directly — constructor, then @PostConstruct, then afterPropertiesSet(), then the custom init method; destruction mirrors it in reverse intent. Destroy callbacks fire only for beans the container manages to the end: singletons on context close (in a non-web app you must call registerShutdownHook()), and never prototypes. A neat detail that links this module together: @PostConstruct is itself invoked by a BeanPostProcessor (CommonAnnotationBeanPostProcessor).

@Component
class DataCache implements InitializingBean {
    @PostConstruct
    void warmUp() { }                 // 1. runs first
    public void afterPropertiesSet() { } // 2. then this

    @PreDestroy
    void flush() { }                  // on singleton shutdown
}

Container extension points: BeanPostProcessor, BeanFactoryPostProcessor, FactoryBean

Three interfaces let you extend the container itself. BeanFactoryPostProcessor runs after all bean definitions are loaded but before any bean is instantiated, so it edits the blueprint — the canonical example is PropertySourcesPlaceholderConfigurer resolving ${...} placeholders. BeanPostProcessor runs on each already-instantiated bean, wrapping postProcessBeforeInitialization/postProcessAfterInitialization around the init callbacks; this is the hook through which @Autowired, @PostConstruct and AOP proxies are actually applied.

Why it matters: FactoryBean is a different idea — a bean that is itself a factory for some other object. Callers of getBean("name") receive its getObject() result, and you reach the factory instance itself with the & prefix (getBean("&name")). Because post-processors must exist before the beans they touch, a @Bean method that returns one should be declared static, so the container can build it without prematurely instantiating the rest of its @Configuration class — a favourite senior-round follow-up.

@Component
class TimingBeanPostProcessor implements BeanPostProcessor {
    public Object postProcessAfterInitialization(Object bean, String name) {
        // inspect, wrap, or proxy every bean here, after its init callbacks
        return bean;
    }
}

Interview questions

Bean scopes and lifecycle are top-tier Indian interview territory — asked from screening rounds up to senior product-company panels:

Beginner What is a bean scope, and which scope is the default?

A scope defines how many instances of a bean the container creates and how long each lives. The default is singleton — one shared instance per Spring container, created eagerly at startup and cached for every injection point.

Beginner Singleton vs prototype scope?

Singleton: one instance per container, reused everywhere. Prototype: a fresh instance every time the bean is requested from the container. Singleton is managed through its whole lifecycle including destruction; for prototype the container creates and configures it, then hands it off and forgets it.

Beginner Is Spring's singleton the same as the Gang of Four singleton pattern?

No. The GoF singleton enforces one instance per class loader / JVM via a private constructor. Spring's singleton is one instance per container keyed by bean name — the same class can have multiple beans, and two containers each get their own instance.

Beginner What do @PostConstruct and @PreDestroy do?

They mark methods to run at the ends of a bean's life: @PostConstruct after construction and dependency injection are complete (initialization work — open a connection, warm a cache), @PreDestroy just before the container removes the bean (cleanup). They are JSR-250 annotations from jakarta.annotation, preferred over Spring's own interfaces because they do not couple your code to the framework.

Beginner Name the web-aware scopes and what each is tied to.

request — one instance per HTTP request; session — one per HTTP session; application — one per ServletContext (the whole web app); websocket — one per WebSocket session. They need a web-aware ApplicationContext and something like RequestContextListener/RequestContextFilter to bind the context.

Intermediate You inject a prototype bean into a singleton and expect a new instance each time — what actually happens, and how do you fix it?

Injection happens once, when the singleton is created, so it captures a single prototype instance forever. To get a fresh one per use, don't inject the bean directly — inject an ObjectProvider/JSR-330 Provider and call it each time, use method injection (@Lookup), or apply a scoped proxy. Same trap and fix for request/session beans in a singleton.

Intermediate A bean has @PostConstruct, implements InitializingBean, and declares an init-method. What is the execution order?

Constructor → @PostConstruct method → InitializingBean.afterPropertiesSet() → the custom init-method. Destruction runs the mirror sequence: @PreDestroy → DisposableBean.destroy() → custom destroy-method.

Intermediate Three ways to hook init/destroy exist — which do you prefer and why?

@PostConstruct/@PreDestroy are preferred: concise and framework-agnostic, so the bean stays a plain object. InitializingBean/DisposableBean couple the class to Spring interfaces. init-method/destroy-method keep the bean clean and are useful for third-party classes you cannot annotate. Avoid mixing all three on one bean.

Intermediate Does Spring call destroy callbacks on prototype beans?

No. The container instantiates, populates and initializes a prototype (so @PostConstruct runs), then stops tracking it — it never calls @PreDestroy/destroy(). The client owns cleanup. If you must run destruction logic, do it yourself or use a custom BeanPostProcessor/destruction-aware mechanism.

Intermediate How does a scoped proxy let a request/session bean live inside a singleton?

With proxyMode = TARGET_CLASS (or INTERFACES), Spring injects a CGLIB/JDK proxy instead of the real bean. The singleton holds the proxy permanently; on each method call the proxy looks up the actual instance for the current request/session from the scope and delegates to it. One stable reference, correct per-scope target.

Intermediate What is a BeanPostProcessor, and name real ones Spring uses.

An interface with postProcessBeforeInitialization and postProcessAfterInitialization callbacks that fire around every bean's init phase, letting you modify or wrap instances. Spring's own: AutowiredAnnotationBeanPostProcessor (@Autowired), CommonAnnotationBeanPostProcessor (@PostConstruct/@PreDestroy/@Resource), and the AOP AnnotationAwareAspectJAutoProxyCreator that builds proxies.

Pro BeanPostProcessor vs BeanFactoryPostProcessor — difference and ordering?

BeanFactoryPostProcessor operates on bean definitions (metadata) after they are loaded but before any bean is instantiated — e.g. resolving property placeholders. BeanPostProcessor operates on instantiated bean instances around initialization. BeanFactoryPostProcessors run first (they shape the definitions), then beans are created and BeanPostProcessors act on them. Multiple of each order via Ordered/PriorityOrdered.

Pro What is a FactoryBean, and how do you get the factory itself instead of its product?

A FactoryBean<T> is a bean that produces another object; its getObject() result (not the FactoryBean) is what normal lookups and injection receive — useful for complex construction. To get the FactoryBean instance itself, prefix the name with an ampersand: getBean("&myBean"). It exposes getObject(), getObjectType() and isSingleton().

Pro Are Spring singleton beans thread-safe?

Not automatically — a singleton is shared across all threads, and Spring adds no synchronization. If it holds mutable instance state, concurrent requests race. Keep singletons stateless (the usual service/controller/repository style), push per-request state into locals, method parameters or a request-scoped bean, or synchronize explicitly. A very common India interview gotcha.

Pro Why should a @Bean method returning a BeanPostProcessor or BeanFactoryPostProcessor be declared static?

Post-processors must be instantiated very early, before the beans they act on. A non-static @Bean method forces Spring to instantiate its enclosing @Configuration class first — which can trigger premature creation of other beans before all post-processors are registered, so some beans skip processing (you often see a container log warning). Declaring the method static lets Spring create the processor without fully initializing the configuration class.
