Module: Indexing

An index is the difference between a query that scans a million rows and one that touches a handful — and picking the right index, in the right column order, is one of the most heavily tested database skills in Indian interviews.

Knowledge Library → Courses → Databases → Indexing

What this module covers

An index is a separate, sorted data structure the database keeps beside a table so it can find rows without reading every one. This module orients you on the B-tree that backs almost every index, the rules that decide whether a query can actually use one (column order, sargability), the specialised variants — covering, partial, expression — and the non-B-tree access methods (GIN, GiST, BRIN). It closes on the cost side: every index you add slows writes, so knowing when not to index is as important as knowing when to.

Topics

What an index is, and B-tree mechanics

A B-tree index is a balanced search tree whose leaf nodes hold the indexed values in sorted order, chained together as a doubly linked list, each pointing back to its table row. Because the tree stays balanced, any lookup costs a few page reads regardless of table size — the search descends from root to leaf in logarithmic steps instead of scanning the whole heap. This ordering is also why a B-tree serves equality (=), ranges (<, >, BETWEEN), IN, prefix LIKE 'abc%', and ORDER BY on the same column, all from one structure.

Why it matters: B-tree is the default in PostgreSQL and the mental model behind every follow-up question. Understanding that the leaves are sorted explains column order, covering indexes, and why a leading wildcard (LIKE '%abc') cannot use the index at all.

CREATE INDEX idx_users_email ON users (email);
-- one B-tree serves all of these:
--   WHERE email = 'a@b.com'
--   WHERE email > 'm' AND email < 'n'
--   ORDER BY email
--   WHERE email LIKE 'prefix%'   (but NOT '%suffix')

Composite indexes and column order

A composite (multicolumn) index sorts rows by its first column, then by the second only where the first ties, and so on — exactly like a phone book ordered by surname then first name. This gives the leftmost-prefix rule: an index on (a, b, c) can drive queries filtering on a, on a, b, or on a, b, c, but not on b or c alone. The second design rule is equality before range: once a column is used with a range predicate, columns after it in the index can no longer narrow the scan, so put the equality columns first.

Why it matters: "here is a table and three queries, design the index" is a staple interview task, and the wrong column order produces an index the planner quietly ignores. One well-ordered composite index often replaces several single-column ones.

CREATE INDEX idx_orders_cust_date ON orders (customer_id, created_at);
-- uses the index:  WHERE customer_id = 42
--                  WHERE customer_id = 42 AND created_at >= '2026-01-01'
-- CANNOT use it:   WHERE created_at >= '2026-01-01'   (skips leading column)

Covering indexes and index-only scans

An index-only scan is an execution plan where every column the query needs is already in the index, so PostgreSQL answers it without touching the heap at all — no random row fetches. A covering index is one deliberately built to enable this, using the INCLUDE clause to carry extra payload columns that are stored in the leaves but are not part of the sort key (so they add no uniqueness or ordering constraints). It is a read-latency optimisation, best for hot queries that select a few columns over many rows.

Why it matters: it is the answer to "how would you speed up this specific SELECT without changing the query," and it exposes a subtlety interviewers love — an index-only scan can still visit the heap because row visibility (MVCC) lives in the table, not the index, and is only skipped for pages the visibility map marks all-visible. Keep autovacuum healthy or the benefit fades.

CREATE INDEX idx_orders_cust ON orders (customer_id) INCLUDE (status, total);
-- SELECT status, total FROM orders WHERE customer_id = 42;
--   -> index-only scan: no heap fetch when pages are all-visible

Partial and expression indexes

A partial index adds a WHERE predicate so it indexes only the rows you actually query — the classic case being a small "active" subset of a mostly-inactive table. The index is smaller, faster to scan, and cheaper to maintain, and it can even enforce uniqueness on just that subset. An expression (functional) index indexes the result of a computation rather than a raw column, so a query filtering on that same expression becomes a simple indexed lookup — the canonical example being lower(email) for case-insensitive search. Both share one strict rule: the query's condition must match the index's predicate or expression closely, because the planner does not reason about equivalences.

-- partial: index only the rows that matter
CREATE INDEX idx_orders_open ON orders (created_at) WHERE status = 'OPEN';

-- expression: index a computed value for case-insensitive lookup
CREATE INDEX idx_users_lower_email ON users (lower(email));
-- must be queried as: WHERE lower(email) = lower('A@B.com')

Index types beyond B-tree: Hash, GIN, GiST, SP-GiST, BRIN

B-tree covers ordered, scalar data, but some workloads need a different structure. Hash handles equality only. GIN (inverted index) is the workhorse for multi-value columns — JSONB containment, arrays, and full-text search — storing each contained element as a key pointing at the rows that hold it. GiST and SP-GiST serve geometric, range, and nearest-neighbour queries (overlaps, "closest points"). BRIN stores tiny per-block-range summaries and shines on very large tables whose values track physical order, such as append-only time-series keyed by timestamp — a fraction of the size of a B-tree.

Why it matters: interviewers probe whether you would blindly B-tree a JSONB column (slow, sometimes unusable) or reach for GIN, and whether you recognise that BRIN exists for warehouse-scale, correlated data. A quick note on vocabulary: PostgreSQL has no "clustered index" in the SQL Server / MySQL-InnoDB sense — its indexes are always secondary structures separate from the heap.

CREATE INDEX idx_docs_body   ON documents USING gin  (body jsonb_path_ops);
CREATE INDEX idx_events_range ON events    USING brin (created_at);

When indexes hurt

An index is pure redundancy the database must keep in sync: every INSERT, UPDATE of an indexed column, and DELETE pays to maintain every affected index, so a table with six indexes makes writes several times more expensive and consumes real disk. Indexes also stop helping when a query is not selective — if it returns a large fraction of the table, a sequential scan is genuinely cheaper, and the planner will (correctly) skip your index. Other traps: unused or duplicate indexes that only cost maintenance, and non-sargable predicates (a function wrapped around a plain column, or a type mismatch) that silently disable the index. The discipline is to index for real query patterns, then drop what the statistics show nobody uses.

Interview questions

Indexing is a top-tier topic in Indian interviews, from service-company screens to product-company deep dives — expect column-order design tasks and "why isn't my index being used" scenarios:

Beginner What is a database index and why does it speed up reads?

A separate, sorted data structure kept beside the table that lets the database locate rows without scanning every one — like a book's index. It turns a full table scan into a short tree descent, cutting reads from O(n) to roughly O(log n) for the matching rows.

Beginner What is the default index type in PostgreSQL, and what can a B-tree serve?

B-tree. Because its leaves are sorted, one B-tree handles equality, range (<, >, BETWEEN), IN, IS NULL, prefix LIKE 'abc%', and supplies rows in ORDER BY order. It cannot help a leading-wildcard LIKE '%abc'.

Beginner Does adding an index always improve performance? What's the cost?

No. Reads get faster but every write (INSERT, UPDATE of the indexed column, DELETE) must also update the index, so writes slow down and disk usage grows. Index for the queries you actually run and remove unused ones.

Beginner What is a composite (multicolumn) index?

One index spanning several columns, sorted by the first column, then the second within ties, and so on. It lets a single structure serve queries that filter on the leading columns together.

Intermediate Explain the leftmost-prefix rule. Given an index on (a, b, c), which queries use it?

An index helps only when the query constrains a leading prefix. (a, b, c) serves filters on a, on a and b, or on all three; it does not serve b alone, c alone, or b and c without a — because the leaves aren't sorted by those columns first.

Intermediate Why does column order matter — what is "equality before range"?

Once a column is matched with a range predicate, columns after it can no longer narrow the index scan (they become filters, not access predicates). So put equality columns first and the range column last. Index (status, created_at) beats (created_at, status) for WHERE status = 'X' AND created_at > ?.

Intermediate What is a covering index / index-only scan, and what is INCLUDE for?

When every column a query needs lives in the index, PostgreSQL answers it from the index alone — an index-only scan, skipping heap fetches. INCLUDE adds non-key payload columns to the leaves (no effect on sort or uniqueness) so more queries can be covered without widening the key.

Intermediate What is a partial index and when would you use one?

An index with a WHERE predicate that indexes only matching rows — e.g. only status = 'OPEN' orders. Smaller and cheaper to maintain, ideal when queries always target a small subset; it can also enforce uniqueness on just that subset. The query's condition must match the predicate.

Intermediate What is an expression (functional) index? Give the classic example.

An index on a computed value rather than a raw column. The canonical case is lower(email) for case-insensitive lookup: CREATE INDEX ... (lower(email)), queried as WHERE lower(email) = lower(?). Without it, wrapping the column in a function makes the predicate non-sargable and disables any plain-column index.

Intermediate Does PostgreSQL have clustered indexes like SQL Server or MySQL?

Not in the same sense. In SQL Server / InnoDB a clustered index stores the table rows in index order (one per table). PostgreSQL indexes are always secondary structures separate from the heap; the CLUSTER command physically reorders a table once but does not keep it ordered.

Pro When would the planner ignore your index and choose a sequential scan?

When the query isn't selective — returning a large fraction of rows makes random index+heap access costlier than a straight sequential read — or on tiny tables, or with stale statistics after a bulk load (run ANALYZE). Also when the predicate is non-sargable (function on the column, type mismatch) so no index applies.

Pro How do you choose between GIN, GiST and BRIN?

GIN for multi-value containment — JSONB, arrays, full-text tsvector — where each row has many searchable elements (fast reads, heavier writes). GiST/SP-GiST for geometric, range and nearest-neighbour / overlap queries. BRIN for very large tables whose column correlates with physical row order (append-only time-series), giving a tiny index that skips block ranges.

Pro Why can an index-only scan still access the heap?

Row visibility (MVCC) is stored in the heap, not the index, so PostgreSQL must confirm a tuple is visible to the current transaction. It skips that check only for pages the visibility map marks all-visible. If autovacuum lags and few pages are all-visible, the "index-only" scan degrades into heap fetches.

Pro How do you add an index to a large, busy production table without blocking writes?

CREATE INDEX CONCURRENTLY — it builds without taking an exclusive lock, so reads and writes continue. Trade-offs: it does two table scans and takes longer, cannot run inside a transaction block, and on failure can leave an INVALID index that you must drop and rebuild.
