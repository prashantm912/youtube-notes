Topics

Views

A view is a named SELECT stored in the database. It holds no data of its own — querying it re-runs the underlying query, so it always reflects live rows while hiding join and filter complexity behind a stable interface. Making liberal use of views is, in the documentation's own words, a key aspect of good SQL design.

Why it matters: views give you encapsulation and a security boundary. You can expose a curated, permission-limited view instead of base tables, and reshape the tables underneath without breaking callers. Simple single-table views are automatically updatable (no aggregates, DISTINCT, GROUP BY or set operations); anything more complex takes an INSTEAD OF trigger to accept writes.

CREATE VIEW active_customers AS
  SELECT id, name, email
  FROM customers
  WHERE status = 'active';

Materialized views

A materialized view stores its query result physically, like a table you can index, instead of recomputing on every read. The trade is freshness: the data is a snapshot from the last refresh and goes stale until you run REFRESH MATERIALIZED VIEW. That makes them ideal for expensive aggregations and reporting rollups that tolerate slight lag.

Why it matters: the refresh strategy is the whole design. A plain refresh takes an exclusive lock; REFRESH ... CONCURRENTLY lets readers keep querying during the rebuild but requires a unique index on the view and runs slower. Choosing between live views and materialized ones — and how to keep the latter current — is a common design question.

CREATE MATERIALIZED VIEW sales_by_day AS
  SELECT day, sum(amount) AS total
  FROM orders
  GROUP BY day;

CREATE UNIQUE INDEX ON sales_by_day (day);
REFRESH MATERIALIZED VIEW CONCURRENTLY sales_by_day;

Stored procedures, functions and triggers

Server-side logic in PostgreSQL comes in three shapes. A function returns a value and runs inside the surrounding transaction (it cannot commit on its own). A procedure (PostgreSQL 11+) is invoked with CALL and can manage transactions with COMMIT/ROLLBACK, which suits batch and maintenance jobs. Both are usually written in PL/pgSQL. A trigger binds a function to INSERT/UPDATE/DELETE/TRUNCATE, firing BEFORE, AFTER, or INSTEAD OF, per row or per statement.

Why it matters: triggers keep invariants next to the data — audit trails, derived columns, validation — but heavy trigger logic hides behaviour and complicates debugging, so interviewers probe both the mechanism and the judgement of when not to use it.

CREATE FUNCTION log_change() RETURNS trigger AS $$
BEGIN
  INSERT INTO audit(table_name, changed_at)
  VALUES (TG_TABLE_NAME, now());
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER accounts_audit
  AFTER UPDATE ON accounts
  FOR EACH ROW EXECUTE FUNCTION log_change();

Partitioning

Partitioning splits one logical table into physical child partitions by range, list, or hash of a key. The planner then prunes partitions that cannot match a query, indexes on each partition stay small, and you can retire old data by detaching a partition instead of running a mass DELETE. Declarative partitioning (PARTITION BY) is the modern approach; the older inheritance-plus-CHECK method still exists for edge cases.

Why it matters: it is the standard answer for very large, growing tables — time-series, events, logs. Note the interview trap: partitioning is not sharding. All partitions live in one server, so it helps maintenance and range queries but does not spread load across machines.

CREATE TABLE events (id bigint, created_at date)
  PARTITION BY RANGE (created_at);

CREATE TABLE events_2026 PARTITION OF events
  FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');

Connection pooling

PostgreSQL forks a dedicated backend process for every connection, so thousands of short-lived application connections — typical of web and serverless apps — burn memory and scheduler time even while idle. A connection pooler such as PgBouncer sits between clients and the server and multiplexes many client connections onto a small, reused set of server connections.

Why it matters: pooling is often the cheapest fix for a database that falls over under connection load. The key concept is pool mode: session pooling assigns a server connection for a whole client session, transaction pooling returns it after each transaction (highest reuse, but you lose session-level state and must be careful with prepared statements and SET), and statement pooling is stricter still.

Roles, GRANT and privileges

Since version 8.1 PostgreSQL has a single access entity — the role — that subsumes the old "users" and "groups". A role that carries the LOGIN attribute is what we casually call a user; a role can own objects, hold privileges, and be granted membership in other roles. GRANT/REVOKE assign object privileges (SELECT, INSERT, UPDATE, DELETE, EXECUTE, USAGE, and more), and role membership bundles them into reusable groups.

Why it matters: least privilege. Applications should connect as a limited role rather than a superuser, and you should know that some privileges default to PUBLIC and that WITH GRANT OPTION lets a grantee pass privileges on. Getting this wrong is a common security-review finding.

CREATE ROLE app_rw LOGIN PASSWORD 'change-me';
GRANT USAGE ON SCHEMA public TO app_rw;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_rw;

Row-Level Security (RLS)

Row-level security lets the engine decide, per row, which rows a role may see or modify — an extra filter layered on top of table-level GRANT. You turn it on with ALTER TABLE ... ENABLE ROW LEVEL SECURITY and then define policies: a USING expression controls visible rows and a WITH CHECK expression controls which rows may be written. It is the in-database answer to multi-tenant data isolation.

Why it matters: the gotchas are the exam. Enabling RLS with no policy is default-deny (nothing is visible). Table owners bypass RLS unless you add FORCE ROW LEVEL SECURITY, and superusers or roles with BYPASSRLS always bypass it. Policies are frequently driven by a session variable set per request, read with current_setting().

ALTER TABLE invoices ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON invoices
  USING (tenant_id = current_setting('app.tenant_id')::int);

VACUUM, autovacuum and bloat

PostgreSQL's MVCC keeps old row versions, so every UPDATE and DELETE leaves dead tuples behind. VACUUM reclaims that space for reuse, updates the visibility map (which enables index-only scans), refreshes planner statistics via ANALYZE, and freezes old rows to prevent transaction-ID wraparound. The autovacuum daemon runs all of this automatically once a table crosses configurable thresholds.

Why it matters: neglected vacuuming shows up as bloat and slow queries. Know that plain VACUUM reuses space in place while VACUUM FULL rewrites the table and returns space to the OS but takes an exclusive lock, and that tuning autovacuum scale factors on large, hot tables is a genuine operations skill.

VACUUM (VERBOSE, ANALYZE) orders;

Backup and restore (pg_dump, PITR)

Backups come in two families. Logical backups (pg_dump, pg_dumpall) emit a portable script or archive of a database that restores across versions and architectures — great for migrations and per-database snapshots, but only as of the moment the dump ran. Physical base backups combined with WAL archiving (continuous archiving) enable point-in-time recovery, letting you restore to any instant after the base backup.

Why it matters: interviews want you to match the tool to the goal. Use pg_dump for single databases and moving data around; use base backup plus PITR for large clusters and low-data-loss disaster recovery. The custom format (-Fc) with pg_restore adds selective and parallel restore.

pg_dump -Fc -d mydb -f mydb.dump      # custom-format logical backup
pg_restore -d mydb --jobs=4 mydb.dump # parallel restore

Full-text search

Full-text search normalizes text into a tsvector — a sorted list of lexemes with optional positions and weights — and matches it against a tsquery using the @@ operator. Unlike LIKE/ILIKE, it brings stemming, stop words, ranking, and language configurations, and a GIN index on the tsvector makes it fast. For many applications this means search without bolting on a separate engine.

Why it matters: the building blocks are to_tsvector() and to_tsquery(), a matching operator, and an index; ranking with ts_rank orders results by relevance. Knowing why full-text search beats a leading-wildcard LIKE is a frequent question once a resume mentions search.

SELECT id, title
FROM articles
WHERE to_tsvector('english', body) @@ to_tsquery('english', 'index & scan');

JSONB operators

The jsonb type stores JSON in a decomposed binary form — slightly slower to insert than plain json, but much faster to query and, crucially, indexable. Its operator set is the day-to-day vocabulary: -> and ->> extract a field or element as jsonb or as text, #> and #>> follow a path, @> and <@ test containment, and ?, ?|, ?& test key existence.

Why it matters: jsonb lets semi-structured data live beside relational columns, and a GIN index accelerates @> containment and key-existence checks. Default to jsonb unless you specifically need to preserve exact text, whitespace, or key order — in which case json keeps the input verbatim.

SELECT data->>'email' AS email
FROM users
WHERE data @> '{"active": true}';

Interview questions

The operations-and-administration questions that separate a developer who can write SQL from one who can run a database — a frequent second-round theme in Indian product and services interviews:

Beginner What is a view and how does it differ from a table?

A view is a named stored query. It holds no data of its own (a materialized view is the exception) and re-runs against its base tables on each access, so it always shows current rows. A table stores the rows; a view presents a query over them, mainly for abstraction and access control.

Beginner View vs materialized view — when do you pick each?

A plain view always reflects live data but pays the query cost on every read. A materialized view stores the result for fast, indexable reads but is stale until you REFRESH it. Use plain views for abstraction and security; use materialized views for expensive aggregations and reporting that can tolerate slight staleness.

Beginner What is a trigger, and name a legitimate use?

A function that runs automatically on INSERT/UPDATE/DELETE/TRUNCATE, firing BEFORE/AFTER (or INSTEAD OF on a view), per row or per statement. Legitimate uses: audit logging, maintaining a derived/denormalized column, or enforcing a cross-row invariant. Overusing triggers hides business logic and makes debugging harder.

Beginner Difference between a function and a procedure in PostgreSQL?

A function returns a value and runs inside the surrounding transaction — it cannot commit. A procedure (11+), called with CALL, can manage transactions with COMMIT/ROLLBACK, which suits batch jobs. Both are commonly written in PL/pgSQL.

Intermediate Why partition a table, and is it the same as sharding?

Partitioning splits one large table into child partitions by range/list/hash so the planner prunes irrelevant partitions, per-partition indexes stay small, and old data drops by detaching a partition instead of a mass DELETE. It is not sharding: all partitions live on one server. Sharding spreads data across multiple servers for horizontal scale.

Intermediate Why does PostgreSQL need connection pooling, and what are pool modes?

Each connection is a separate backend process, so many short-lived or idle connections waste memory and scheduling. A pooler such as PgBouncer multiplexes clients onto a few reused server connections. Modes: session (one server connection per client session), transaction (returned after each transaction — highest reuse, but no session-level state and careful with prepared statements), and statement.

Intermediate How do roles, users and GRANT relate?

Since 8.1 there is one entity, the role; a "user" is just a role with LOGIN. GRANT/REVOKE assign object privileges (SELECT, INSERT, EXECUTE, USAGE, and so on) and role membership bundles privileges into groups. Best practice: the application connects as a least-privilege role, never a superuser, and you watch what defaults to PUBLIC.

Intermediate What does REFRESH MATERIALIZED VIEW CONCURRENTLY require, and why use it?

It rebuilds a materialized view without an exclusive lock, so readers keep querying during the refresh. It requires a UNIQUE index covering all rows, runs slower than a plain refresh, and cannot run on a not-yet-populated view.

Intermediate What is VACUUM and why is it necessary?

MVCC keeps old row versions, so UPDATE/DELETE leave dead tuples. VACUUM reclaims that space for reuse, updates the visibility map (enabling index-only scans) and planner statistics, and freezes old rows to prevent transaction-ID wraparound. Autovacuum does it automatically; without it, tables bloat and queries slow down.

Intermediate Full-text search vs LIKE — why bother with tsvector/tsquery?

LIKE/ILIKE do raw substring matching with no stemming, ranking, or stop words, and rarely use an index with a leading wildcard. Full-text search normalizes text to a tsvector of lexemes, matches a tsquery with @@, supports ranking and language configurations, and is GIN-indexable for speed.

Intermediate json vs jsonb — which do you default to?

json stores the exact input text (preserving whitespace, key order, and duplicate keys) and reparses on each access. jsonb stores a decomposed binary form — slightly slower to insert, much faster to query, and indexable with GIN. Default to jsonb unless you specifically need verbatim text or key-order preservation.

Pro How is row-level security enforced, and what bypasses it?

After ALTER TABLE ... ENABLE ROW LEVEL SECURITY, the engine applies each policy's USING/WITH CHECK expression per row, on top of GRANT. Enabling it with no policy is default-deny. Table owners bypass RLS unless you add FORCE ROW LEVEL SECURITY; superusers and roles with BYPASSRLS always bypass. It is typically driven by a per-request session variable read with current_setting().

Pro pg_dump vs continuous archiving/PITR — how do you choose?

pg_dump/pg_dumpall are logical backups: portable across versions and architectures, ideal for single databases and migrations, but a single point-in-time snapshot. A physical base backup plus WAL archiving enables point-in-time recovery to any moment and suits large clusters and low-data-loss disaster recovery. Custom format (-Fc) with pg_restore adds selective and parallel restore.

Pro What is transaction-ID wraparound, and how does VACUUM prevent it?

Transaction IDs are 32-bit and compared circularly; if the oldest unfrozen ID falls too far behind, committed rows could suddenly look like they came from the future and disappear. VACUUM freezes old tuples (marks them permanently visible), advancing the frozen-XID horizon, and autovacuum forces an anti-wraparound vacuum near autovacuum_freeze_max_age. Ignored long enough, the database stops accepting writes to protect itself.
