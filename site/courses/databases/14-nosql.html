Module: NoSQL

The relational table is one answer to storing data, not the only one. This module maps the NoSQL landscape — four data models, MongoDB up close, and the consistency tradeoffs (CAP, PACELC) that decide which store fits which job.

Knowledge Library → Courses → Databases → NoSQL

What this module covers

"NoSQL" is not one database but a family of non-relational stores that trade the fixed schema and strong joins of SQL for flexibility, horizontal scale, and specialized data models. You will place the four main families, work hands-on with the document model through MongoDB, and reason about the distributed-systems tradeoffs — CAP, PACELC, eventual consistency — that make these stores behave the way they do. The closing idea, polyglot persistence, ties it together: pick the store per workload rather than forcing everything into one engine.

Topics

The NoSQL taxonomy: document, key-value, wide-column, graph

Four families cover most NoSQL systems. Document stores (MongoDB, Couchbase) keep self-describing JSON-like records and suit evolving, hierarchical data. Key-value stores (Redis, DynamoDB) are a distributed hash map — blistering lookups by key, ideal for caching and sessions. Wide-column stores (Cassandra, HBase) group columns into families for write-heavy, massive-scale time-series and event data. Graph databases (Neo4j, Neptune) make relationships first-class, for social graphs, recommendations, and fraud detection.

Why it matters: the interview question is rarely "define NoSQL" — it is "which store, and why?" Choosing well means matching the family to the access pattern (lookup by key, traverse relationships, scan a wide row) rather than defaulting to whatever tool is most familiar. Each family also gives up something SQL provides, usually multi-row transactions or ad-hoc joins.

MongoDB: a working introduction

MongoDB is the reference document database: data lives in collections of documents, each a BSON (binary JSON) record up to 16 MB with a mandatory _id primary key. There is no upfront schema — documents in one collection can differ in shape. You interact through the four CRUD verbs (insertOne/find/updateOne/deleteOne and their bulk siblings) and, for analytics, the aggregation pipeline — a chain of stages ($match, $group, $sort) where each stage's output feeds the next.

Why it matters: MongoDB is the NoSQL store you are most likely to be quizzed on and to reach for first. Two facts anchor most questions — writes are atomic at the single-document level (no default multi-document transaction, though they exist since 4.0), and querying is by document fields with rich operators, not SQL.

// a document in the "users" collection
{ _id: ObjectId("64f0a1..."), name: "Asha", city: "Pune",
  roles: ["editor", "admin"] }

db.users.find({ city: "Pune", roles: "admin" })   // read by fields
db.users.updateOne({ name: "Asha" }, { $set: { city: "Mumbai" } })

Modeling in MongoDB: embedding vs referencing

The central schema decision has no SQL equivalent: do you embed related data inside the parent document, or reference it in another collection and join with $lookup? Embed when data is read together, updated together, and bounded — a "has-a"/"contains" relationship (an order and its line items) that you fetch in one read and write atomically. Reference when the child has high cardinality, is written independently, is queried on its own, or would grow the parent past the 16 MB limit. The guiding principle: data that is accessed together should be stored together.

// EMBEDDED — one read returns the order and its items
{ _id: 501, customer: "Asha",
  items: [ { sku: "A1", qty: 2 }, { sku: "B7", qty: 1 } ] }

// REFERENCED — item docs live separately, joined when needed
{ _id: 501, customer: "Asha", itemIds: [ 9001, 9002 ] }

CAP and PACELC: the consistency tradeoffs

The CAP theorem says that when a network partition splits a distributed system, you must choose between consistency (every read sees the latest write) and availability (every request gets a response). Partition tolerance is not optional on real networks, so the live choice is CP or AP — MongoDB leans CP (a replica set pauses writes during a partition to stay consistent), while Cassandra and DynamoDB lean AP (they stay up and reconcile later). This is where BASE (Basically Available, Soft state, Eventual consistency) contrasts with the relational world's ACID.

PACELC extends CAP with the case CAP ignores: else, when there is no partition, you still trade latency against consistency. A system is described as, e.g., PA/EL (available under partition, low-latency otherwise) or PC/EC (consistent in both). Since partitions are rare and normal operation is constant, PACELC captures the tradeoff engineers actually tune day to day.

Polyglot persistence: the right store per job

Martin Fowler's term polyglot persistence names the modern default: a single application uses several data stores, each chosen for a specific need — PostgreSQL for transactional orders, Redis for the session cache, Elasticsearch for search, a graph database for the recommendation engine. Microservices push this further with the database-per-service pattern, where each service privately owns the store best suited to it. The cost is real: no cross-store joins or transactions, data duplication, synchronization work, and more operational surface — so you adopt it deliberately, not reflexively.

Interview questions

Most-asked NoSQL questions in Indian interviews — a staple wherever MongoDB, Cassandra, or system design comes up:

Beginner What is a NoSQL database and how does it differ from a relational database?

A non-relational store that drops the fixed tabular schema and (usually) SQL joins in favour of flexible data models and horizontal scale. Relational databases enforce a schema, normalize data across tables, and give strong ACID transactions and joins; NoSQL trades some of that for schema flexibility, scale-out, and a data model matched to a specific access pattern.

Beginner What are the four main types of NoSQL databases, with an example of each?

Document (MongoDB) for hierarchical records, key-value (Redis, DynamoDB) for fast lookups by key, wide-column (Cassandra, HBase) for write-heavy large-scale data, and graph (Neo4j) for relationship-centric data. The follow-up is always "when would you pick each?"

Beginner What is a document in MongoDB, and how is it stored?

A document is a set of field-value pairs stored as BSON (binary JSON), living in a collection, with a required _id primary key and a 16 MB size limit. Documents in one collection need not share the same fields — the schema is flexible.

Beginner What are the basic CRUD operations in MongoDB?

Create (insertOne/insertMany), Read (find/findOne), Update (updateOne/updateMany/replaceOne), Delete (deleteOne/deleteMany). Reads and updates share the same filter syntax, and every single-document write is atomic.

Intermediate When would you choose NoSQL over a relational database, and when not?

Favour NoSQL for flexible/evolving schemas, very high write throughput, horizontal scale-out, and a model that matches your access pattern (key lookups, graph traversal, huge wide rows). Stay relational when you need multi-row ACID transactions, ad-hoc joins and reporting, strong integrity constraints, or when the data is genuinely relational and moderate in size. The honest answer names both sides.

Intermediate Embedding vs referencing in MongoDB — how do you decide?

Embed when related data is read and written together, bounded in size, and has a "contains" relationship — one read, one atomic write. Reference when the child has high cardinality, is queried or updated independently, is shared, or would push the document toward the 16 MB limit; then join with $lookup. Rule of thumb: data accessed together is stored together.

Intermediate Explain the CAP theorem. Why can't you have all three?

Consistency, Availability, Partition tolerance — during a network partition a node either refuses to answer without the latest data (choosing consistency, sacrificing availability) or answers with possibly-stale data (choosing availability, sacrificing consistency). Since partitions are unavoidable on real networks, partition tolerance is mandatory, so the real choice is CP vs AP.

Intermediate What does BASE mean, and how does it contrast with ACID?

BASE — Basically Available, Soft state, Eventual consistency — is the availability-first philosophy of many NoSQL systems: accept temporary inconsistency so the system stays up and converges later. ACID (Atomicity, Consistency, Isolation, Durability) is the relational guarantee of correct, isolated, durable transactions. BASE trades immediate consistency for availability and scale.

Intermediate Is MongoDB a CP or an AP system?

By default CP. A replica set has one primary; if it is partitioned from the majority it steps down and writes pause until a new primary is elected, preferring consistency over availability. Read/write concerns and reading from secondaries can tune where it sits, but the default posture is consistency-first.

Intermediate What is eventual consistency?

A guarantee that, absent new writes, all replicas will converge to the same value given enough time — but a read right after a write may return a stale value. It is the model behind AP stores like Cassandra and DynamoDB; the application must tolerate temporary staleness (or use tunable/quorum reads to narrow the window).

Pro How does MongoDB scale horizontally — replica sets vs sharding?

Two different axes. A replica set copies the same data to multiple nodes for high availability and read scaling (one primary, secondaries, automatic failover). Sharding partitions a collection across shards by a shard key for write and storage scale-out; a good shard key spreads writes evenly and matches query patterns, while a poor one creates hotspots. Production clusters combine both — each shard is itself a replica set.

Pro What is PACELC and what does it add to CAP?

PACELC says: if Partition (P), trade Availability vs Consistency (the CAP case); Else (E), when healthy, trade Latency vs Consistency. It captures the everyday tradeoff CAP ignores — replicating synchronously for consistency costs latency even with no partition. Systems are classified like PA/EL (Cassandra, DynamoDB) or PC/EC (fully consistent), giving a more complete picture than CAP alone.

Pro What is polyglot persistence, and what are its tradeoffs?

Using multiple purpose-built data stores in one system — relational for transactions, key-value for caching, search engine for text, graph for relationships — chosen per workload rather than forcing one database everywhere; the microservices version is database-per-service. Benefits: each job gets the optimal store. Costs: no cross-store joins or transactions, data duplication and synchronization, eventual consistency across stores, and greater operational complexity — so adopt it where the payoff is clear.
