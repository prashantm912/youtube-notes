Module: Query Optimization

A query is only as fast as the plan the planner picks for it — this module teaches you to read that plan, keep the statistics behind it honest, and rewrite the handful of patterns that quietly force a full-table scan.

Knowledge Library → Courses → Databases → Query Optimization

What this module covers

PostgreSQL uses a cost-based optimizer: for every query it estimates how many rows each step will produce and chooses the cheapest plan it can find. Optimization here means four skills — reading the plan with EXPLAIN, understanding the statistics that drive it, rewriting queries into shapes the planner can index, and replacing OFFSET pagination with keyset seeks. This builds directly on Indexing: an index only helps if the plan actually uses it.

Topics

Reading a query plan: EXPLAIN and EXPLAIN ANALYZE

EXPLAIN prints the plan tree the planner chose without running the query — a set of nodes (Seq Scan, Index Scan, Index Only Scan, Bitmap Heap Scan, Nested Loop, Hash Join, Sort, Limit) read inside-out, each annotated cost=startup..total rows=estimate width=bytes. EXPLAIN ANALYZE actually executes it and adds the actual time, row count, and loop count next to every estimate, so the gap between estimate and reality jumps out.

Why it matters: this is the one tool that turns "the query is slow" into "this node is slow, and here is why." Adding BUFFERS shows how many pages came from cache (shared hit) versus disk (read) — the difference between a warm and a cold query. Reading a plan out loud is a standard interview task.

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders
WHERE customer_id = 42
ORDER BY created_at DESC
LIMIT 20;
-- Limit  (cost=0.42..8.11 rows=20 width=72)
--   ->  Index Scan using orders_cust_created_idx on orders
--         (actual time=0.018..0.041 rows=20 loops=1)

The planner and its statistics

Every plan choice — sequential versus index scan, which join algorithm, which table to join first — is driven by row-count estimates the planner reads from column statistics in pg_statistic (viewable through pg_stats). Those statistics — most-common values, histograms, null fraction, distinct counts — are gathered by ANALYZE and kept current by autovacuum. When they are stale or missing, estimates go wrong and the planner picks a bad plan for the right reasons.

Why it matters: most "why did it choose a seq scan?" mysteries are estimation errors, not planner stupidity. Knowing to run ANALYZE, raise default_statistics_target on a skewed column, or add extended statistics (CREATE STATISTICS) for correlated columns is what separates a guess from a diagnosis.

ANALYZE orders;                       -- refresh planner statistics now
ALTER TABLE orders
  ALTER COLUMN status SET STATISTICS 500;  -- finer histogram on a skewed column
CREATE STATISTICS addr_stats (dependencies)
  ON city, state FROM addresses;      -- teach the planner these correlate

Rewrite patterns that let the planner win

Often the fastest fix is not a new index but the same result expressed so the planner can use one. The recurring patterns: keep predicates sargable (never wrap the indexed column in a function — compare a range instead); prefer EXISTS to IN for correlated existence checks; avoid SELECT * when a covering index could give an index-only scan; and for large multi-table queries, constrain join order with explicit JOIN syntax and join_collapse_limit when the planner's search space explodes.

Why it matters: PostgreSQL has no query hints in the Oracle sense — you steer it by writing queries it can optimize and by feeding it good statistics. Recognising a non-sargable predicate on sight is a frequent screening question.

-- non-sargable: the function on created_at hides it from the index
WHERE date(created_at) = '2026-07-01'

-- sargable rewrite: a half-open range an index can seek
WHERE created_at >= '2026-07-01'
  AND created_at <  '2026-07-02'

Pagination at scale: keyset over OFFSET

LIMIT 20 OFFSET 100000 looks cheap but the database still computes and throws away the first hundred thousand rows — cost grows linearly with page depth, and rows shift under the reader when data changes between requests. Keyset (seek) pagination instead carries the sort key of the last row seen and filters with a WHERE, so an index seeks straight to the next page in constant time. It requires a deterministic ORDER BY with a unique tie-breaker and a row-value comparison for multi-column keys.

Why it matters: this is the standard cure for the slow "jump to page 5000" and for infinite-scroll feeds, and it comes up constantly in both API design and SQL interviews.

-- page after the last row seen: (created_at, id) = ($1, $2)
SELECT id, created_at, total
FROM orders
WHERE (created_at, id) < ($1, $2)
ORDER BY created_at DESC, id DESC
LIMIT 20;

Interview questions

Most-asked query optimization questions in Indian interviews — a favourite of product-company and backend rounds, where reading a plan on the whiteboard is expected:

Beginner What does EXPLAIN do, and how is EXPLAIN ANALYZE different?

EXPLAIN shows the execution plan the planner chose, with estimated cost and row counts, without running the query. EXPLAIN ANALYZE actually runs it and adds the real execution time and actual row counts per node — so you can compare estimate against reality. Because it runs the query, avoid EXPLAIN ANALYZE on writes unless inside a transaction you roll back.

Beginner In a plan node, what do cost, rows, and width mean?

cost=startup..total is the planner's estimated effort in arbitrary units (roughly page fetches) — startup is the cost before the first row, total is for all rows. rows is the estimated number of rows the node emits; width is the estimated average row size in bytes. The planner compares total costs of alternative plans and picks the cheapest.

Beginner Sequential Scan vs Index Scan — when does the planner choose each?

A Seq Scan reads the whole table; an Index Scan walks the index then fetches matching rows. For a small, selective result an index wins. But when a query returns a large fraction of the table, a seq scan is actually cheaper (sequential I/O, no random heap fetches), so the planner deliberately chooses it — that is correct behaviour, not a bug.

Beginner Why can estimated rows differ from actual rows, and why does it matter?

Estimates come from sampled statistics; reality comes from the data. A large gap (say estimate 10, actual 100000) means the planner optimized for the wrong size and likely chose a bad join or scan. The estimate-vs-actual mismatch in EXPLAIN ANALYZE is the first thing to inspect when a plan looks wrong.

Intermediate What does EXPLAIN (ANALYZE, BUFFERS) add, and what do shared hit and read tell you?

BUFFERS reports page access per node: shared hit is pages served from PostgreSQL's cache, shared read is pages fetched from the operating system or disk. High reads mean a cold cache or a plan touching far more data than needed; comparing a warm and cold run tells you whether slowness is I/O or the plan itself.

Intermediate How are planner statistics kept fresh, and what breaks when they go stale?

ANALYZE (run manually, or automatically by autovacuum) samples each table and stores most-common values, histograms and distinct counts in pg_statistic. Stale stats — after a bulk load, or when autovacuum falls behind — give the planner wrong selectivity, so it misjudges row counts and picks poor plans. Fix: run ANALYZE, and raise the statistics target on skewed columns.

Intermediate What makes a WHERE predicate sargable? Give a non-sargable example and its rewrite.

Sargable = "Search ARGument ABLE": the predicate can drive an index seek. Wrapping the indexed column in a function defeats that — WHERE date(created_at) = '2026-07-01' forces a scan. Rewrite as a half-open range on the bare column: created_at >= '2026-07-01' AND created_at < '2026-07-02', which an index on created_at can seek. (An expression index on date(created_at) is the alternative.)

Intermediate Why is OFFSET-based pagination slow on deep pages?

OFFSET n does not skip work — the database still produces the first n rows in order and discards them, so cost grows with page depth. It is also unstable: inserts or deletes between requests shift the window, duplicating or skipping rows. Deep pages get progressively slower; keyset pagination avoids both problems.

Intermediate Nested Loop vs Hash Join vs Merge Join — which does the planner pick and why?

Nested Loop suits a small outer set with an indexed inner side (few lookups). Hash Join suits large unsorted inputs with an equality condition — it builds a hash of one side and probes it. Merge Join suits inputs already sorted (or cheaply sortable) on the join key. The planner estimates each from row counts and available indexes and takes the cheapest; wrong row estimates are what push it to the wrong one.

Pro Explain keyset (seek) pagination, including the tie-breaker requirement.

Instead of an offset, carry the sort key of the last row seen and filter beyond it: WHERE (created_at, id) < ($1, $2) ORDER BY created_at DESC, id DESC LIMIT 20. A matching index seeks straight to the next page, so cost is constant regardless of depth. The unique tie-breaker (id) is mandatory — without it, rows sharing a created_at value are ordered non-deterministically and pages overlap or drop rows. The trade-off: you can only go next/previous, not jump to an arbitrary page number.

Pro The planner uses a Seq Scan even though a relevant index exists. What are the likely causes and fixes?

Common causes: the query is not selective enough (returns a big fraction of the table, so a scan is genuinely cheaper); stale statistics overestimate the result; a non-sargable predicate hides the column from the index; a type mismatch prevents index use; or the table is tiny. Fixes: run ANALYZE, make the predicate sargable, match the parameter type, or accept that the seq scan is correct. Setting enable_seqscan = off is only a diagnostic, never a production fix.

Pro What are extended statistics (CREATE STATISTICS) and which estimation error do they fix?

By default the planner assumes columns are independent, so it multiplies selectivities. When columns are correlated — city and state, or a status that implies a type — that assumption undercounts, e.g. estimating 1 row where 10000 exist. CREATE STATISTICS ... (dependencies, ndistinct, mcv) records the correlation so multi-column predicates and GROUP BY estimates become accurate, steering the planner to the right join and scan choices.
