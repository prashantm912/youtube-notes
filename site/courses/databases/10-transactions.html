Module: Transactions and Concurrency

A transaction is a promise: all of it happens, or none of it does — and concurrency control is how that promise survives thousands of sessions hitting the same rows at once. This module names each guarantee and points you to the definitive reference for it.

Knowledge Library → Courses → Databases → Transactions and Concurrency

What this module covers

Two connected ideas: the ACID contract a single transaction gives you, and the concurrency machinery that upholds isolation when many transactions overlap. We cover the SQL isolation levels and the read anomalies they forbid, PostgreSQL's MVCC snapshot model, explicit row and table locking with SELECT FOR UPDATE, and the two failure modes every backend engineer meets — deadlocks and lost updates. ACID and isolation levels are among the most heavily interviewed database topics in India, so treat this as a core module, not a footnote.

Topics

Transactions and the ACID guarantees

A transaction groups statements into one unit of work that either commits in full or rolls back entirely. In PostgreSQL you open one with BEGIN and close it with COMMIT (or ROLLBACK); outside an explicit block every statement autocommits on its own. SAVEPOINT lets you undo part of a transaction without abandoning the whole thing.

ACID names the four guarantees. Atomicity is all-or-nothing. Consistency means every constraint, trigger and check still holds at commit. Isolation means concurrent transactions do not see each other's uncommitted work. Durability means once COMMIT returns, the change survives a crash — PostgreSQL guarantees this by flushing the write-ahead log (WAL) first. "Expand ACID" opens the transactions round in almost every Indian interview, so be able to give a one-line example of each.

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;   -- both rows change, or ROLLBACK leaves neither changed

Isolation levels and read anomalies

The SQL standard defines four isolation levels — Read Uncommitted, Read Committed, Repeatable Read, Serializable — and defines them by which read anomalies they forbid. A dirty read sees another transaction's uncommitted row. A non-repeatable read is a row whose value changes when you read it a second time. A phantom read is a range query that returns new or missing rows on re-execution. A serialization anomaly is a set of transactions that commits to a state no serial ordering could have produced.

PostgreSQL defaults to Read Committed and never permits dirty reads at any level. It maps Read Uncommitted onto Read Committed, and — thanks to MVCC — its Repeatable Read is stronger than the standard, blocking phantoms as well. You pick a level with BEGIN ISOLATION LEVEL … or SET TRANSACTION. Reproducing the grid of "which level stops which anomaly" is the single most-drilled concurrency question in interviews.

BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT count(*) FROM orders WHERE status = 'NEW';
-- concurrent INSERTs stay invisible; this transaction's snapshot is fixed
COMMIT;

MVCC: isolation without read locks

Instead of blocking readers, PostgreSQL hands each transaction a snapshot. Every row carries hidden xmin/xmax system columns — the transaction that created it and the one that expired it — so a read sees only the versions valid for its snapshot. An UPDATE writes a new row version and marks the old one expired; a DELETE just marks it. The headline consequence: readers never block writers and writers never block readers.

The cost of keeping old versions is dead tuples. They linger until VACUUM (normally autovacuum) reclaims the space, so a heavily updated table bloats without maintenance. VACUUM also freezes old transaction IDs to prevent 32-bit XID wraparound. Understanding "where do the old rows go?" separates engineers who have operated PostgreSQL from those who have only queried it.

UPDATE products SET price = 199 WHERE id = 42;
-- the old version (xmax set) and the new version (xmin set) coexist;
-- concurrent snapshots keep reading the old one until VACUUM cleans up

Explicit locking and SELECT FOR UPDATE

When application logic must serialize access to specific rows — a read, then a decision, then a write — a plain MVCC snapshot is not enough, because two sessions can each read the same starting value. Take an explicit row lock with SELECT … FOR UPDATE: it blocks other writers and lockers of those rows until you commit, while ordinary SELECTs still read them. FOR SHARE is the shared variant; FOR NO KEY UPDATE and FOR KEY SHARE are weaker forms that conflict less. Table-level locks, from ACCESS SHARE up to ACCESS EXCLUSIVE, are mostly acquired automatically by DDL and DML.

FOR UPDATE is frequently stronger than you need — the standard advice is to prefer FOR NO KEY UPDATE unless you change a key column or delete the row. NOWAIT fails fast rather than waiting on a locked row; SKIP LOCKED skips already-locked rows, the idiomatic trick for building a concurrent job queue where each worker claims a different row.

BEGIN;
SELECT * FROM jobs WHERE state = 'ready'
  ORDER BY id
  FOR UPDATE SKIP LOCKED
  LIMIT 1;                 -- claim one row; other workers skip it
UPDATE jobs SET state = 'running' WHERE id = /* claimed id */ 7;
COMMIT;

Deadlocks and the lost update anomaly

A deadlock is a cycle: transaction A holds a lock B wants, while B holds a lock A wants, so neither can move. PostgreSQL runs a deadlock detector that spots the cycle and aborts one transaction with an error — it gives no guarantee which one. The prevention is discipline: acquire locks in a consistent order across your code paths, keep transactions short, and wrap them in retry logic so an aborted transaction re-runs cleanly.

The lost update anomaly is the classic race — two sessions read the same row, both compute a new value, and the second write silently overwrites the first. Prevent it with a pessimistic lock (SELECT FOR UPDATE), with optimistic locking (a version column tested in the WHERE clause, so a conflicting update affects zero rows), or by running at Repeatable Read, which aborts the losing writer. This anomaly is behind most "why did my counter lose increments?" bugs.

-- optimistic locking: only succeeds if nobody else changed the row
UPDATE accounts
   SET balance = 500, version = version + 1
 WHERE id = 1 AND version = 7;
-- 0 rows affected => someone else won the race; reload and retry

Interview questions

Transactions and concurrency is one of the most heavily interviewed database areas in India — ACID and the isolation-level grid come up in nearly every backend and SDE screen:

Beginner What does ACID stand for?

Atomicity (a transaction is all-or-nothing), Consistency (every constraint, trigger and check holds across the transaction), Isolation (concurrent transactions do not see each other's uncommitted work), and Durability (once committed, data survives a crash — PostgreSQL uses the write-ahead log). Give a one-line example of each; interviewers usually ask you to.

Beginner How do you start and end a transaction in PostgreSQL?

BEGIN (or START TRANSACTION) opens one, COMMIT makes its changes permanent, and ROLLBACK discards them. Without an explicit block, each statement autocommits on its own. SAVEPOINT plus ROLLBACK TO lets you undo part of a transaction while keeping the rest.

Beginner Name the four SQL isolation levels from weakest to strongest.

Read Uncommitted, Read Committed, Repeatable Read, Serializable. Each stronger level forbids more read anomalies at the cost of concurrency.

Beginner What is a dirty read?

Reading a row another transaction has written but not yet committed. If that transaction rolls back, you acted on data that never truly existed. Only Read Uncommitted permits it in the standard — and PostgreSQL never allows it, even at that level.

Beginner What is PostgreSQL's default isolation level?

Read Committed. Each statement sees only rows committed before that statement began, so it forbids dirty reads but still allows non-repeatable reads and phantom reads within a multi-statement transaction.

Intermediate Dirty read vs non-repeatable read vs phantom read?

Dirty read: you see uncommitted data. Non-repeatable read: the same row returns a different value on a second read because a concurrent transaction committed an UPDATE in between. Phantom read: the same range query returns a different set of rows because a concurrent transaction committed an INSERT or DELETE. They form a progression of severity.

Intermediate Which isolation level prevents which anomaly?

Read Committed stops dirty reads. Repeatable Read additionally stops non-repeatable reads — and in PostgreSQL also stops phantoms. Serializable stops everything, including the serialization anomaly. Being able to draw this grid on a whiteboard is a staple ask.

Intermediate What is MVCC and what problem does it solve?

Multiversion Concurrency Control keeps multiple versions of each row so every transaction reads a consistent snapshot without taking read locks. The result is that readers never block writers and writers never block readers, giving high concurrency. PostgreSQL, Oracle and InnoDB all use MVCC.

Intermediate What does SELECT … FOR UPDATE do, and when do you use it?

It locks the selected rows for writing, so no other transaction can update, delete, or re-lock them until you commit — while plain SELECTs can still read them. Use it for read-modify-write correctness, where a snapshot alone would let two sessions overwrite each other. FOR SHARE is the shared-lock variant.

Intermediate What is a deadlock and how does PostgreSQL resolve it?

Two transactions each wait on a lock the other holds, forming a cycle so neither can proceed. PostgreSQL's deadlock detector breaks it by aborting one transaction (no guarantee which) with a deadlock error; the survivor continues. Avoid deadlocks by locking rows in a consistent order, keeping transactions short, and retrying on failure.

Intermediate What is the lost update problem and how do you prevent it?

Two transactions read the same value, both write a new one, and the second write silently discards the first. Prevent it with pessimistic locking (SELECT FOR UPDATE), optimistic locking (a version column checked in the WHERE clause so a stale update affects zero rows), or Repeatable Read, which aborts the losing writer with a serialization error.

Intermediate How does PostgreSQL's Repeatable Read differ from the SQL standard?

The standard allows phantom reads at Repeatable Read; PostgreSQL's snapshot-based implementation forbids them too, so it behaves like snapshot isolation. A write conflict raises "could not serialize access," which the application is expected to catch and retry.

Pro NOWAIT vs SKIP LOCKED on a locking SELECT — when do you use each?

NOWAIT raises an error immediately if a target row is already locked — fail fast and surface contention. SKIP LOCKED silently ignores locked rows so each worker claims a different unlocked one — the idiomatic pattern for a concurrent queue or job dispatcher built on a table. Plain FOR UPDATE instead waits for the lock to be released.

Pro How does PostgreSQL implement Serializable, and what must the application do?

Serializable Snapshot Isolation (SSI) adds predicate locks that track read/write dependencies among concurrent transactions; when a dangerous dependency cycle is detected, one transaction is aborted with a serialization failure. Unlike two-phase locking it does not block, but the application must wrap transactions in retry logic. Declaring transactions READ ONLY (and DEFERRABLE where possible) reduces false aborts.

Pro Why does MVCC require VACUUM, and what is transaction ID wraparound?

UPDATE and DELETE leave dead row versions behind; VACUUM (normally autovacuum) reclaims that space and refreshes the visibility map, otherwise tables bloat and index-only scans slow down. VACUUM also freezes old rows: transaction IDs are 32-bit, so without freezing the counter would wrap after roughly four billion transactions and previously-committed rows would appear to be in the future and vanish. This is why disabled autovacuum and very long-lived idle transactions are dangerous in production.
