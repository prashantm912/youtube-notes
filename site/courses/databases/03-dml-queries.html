Module: SQL DML and Queries

DDL builds the table; DML is where the data actually moves — and the SELECT that reads it back is the single most-tested topic in any database interview. This module maps the write and read verbs and points you at the pages that teach each properly.

Knowledge Library → Courses → Databases → SQL DML and Queries

What this module covers

Two halves of everyday SQL. First the write side — INSERT, UPDATE, DELETE, the ON CONFLICT upsert, and RETURNING to read back what a write changed. Then the read side — the anatomy of a SELECT, filtering with WHERE, and shaping the result set with ORDER BY, LIMIT/OFFSET and DISTINCT. The recurring theme interviewers probe is the gap between how you write a query and the order in which the engine actually runs it.

Topics

INSERT and upserts with ON CONFLICT

INSERT adds rows — single-row, multi-row VALUES, or INSERT ... SELECT to copy from a query. The interview-relevant part is the upsert: INSERT ... ON CONFLICT makes a duplicate-key collision a normal outcome rather than an error. DO NOTHING silently skips the conflicting row; DO UPDATE modifies the existing row, reading the rejected values through the special EXCLUDED pseudo-table.

Why it matters: upsert is the idiomatic way to make writes idempotent — safe to retry — which is exactly what queue consumers, sync jobs and "insert if not present" caches need. The conflict target must match a real unique or exclusion constraint, so this ties directly back to the DDL module.

INSERT INTO orders (id, customer, total)
VALUES (1, 'Asha', 4999)
ON CONFLICT (id) DO UPDATE
  SET total = EXCLUDED.total;   -- insert, or update the existing row

UPDATE, DELETE and the RETURNING clause

UPDATE ... SET ... WHERE changes existing rows; DELETE FROM ... WHERE removes them. The WHERE is doing the real work — omit it and every row in the table is rewritten or deleted, the classic production incident. Both are transactional in PostgreSQL, so wrapping them in a transaction (and running the WHERE as a SELECT first) is the standard safety habit.

RETURNING is PostgreSQL's answer to "what did that write actually do?" — append it to an INSERT, UPDATE or DELETE and it hands back columns from the affected rows in the same statement, most often a generated id, saving a second round-trip. UPDATE ... FROM and DELETE ... USING let a write join another table.

UPDATE accounts SET balance = balance - 500
WHERE id = 7
RETURNING id, balance;          -- read the new balance back

DELETE FROM sessions
WHERE last_seen < now() - interval '30 days';

SELECT anatomy and the logical query order

A SELECT is written SELECT … FROM … WHERE … GROUP BY … HAVING … ORDER BY … LIMIT, but the engine does not run it in that order. Logically it processes FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. That single fact explains a whole class of "why doesn't this work?" bugs.

Why it matters: because SELECT runs after WHERE, a column alias you define in the select list is invisible to WHERE — but visible to ORDER BY, which runs later. Understanding the pipeline, not memorising syntax, is what separates a confident answer here.

SELECT   city, count(*) AS n   -- 5. project columns & name them
FROM     users                 -- 1. read rows
WHERE    active                -- 2. filter rows
GROUP BY city                  -- 3. collapse into groups
HAVING   count(*) > 10         -- 4. filter groups
ORDER BY n DESC                -- 6. sort (alias visible here)
LIMIT 5;                       -- 7. take the top slice

Filtering with WHERE

The WHERE clause keeps only rows for which its condition is true — a subtlety, because SQL uses three-valued logic where a comparison against NULL yields unknown, not false. Beyond the basic operators (=, <>, <, >), the everyday toolkit is BETWEEN (inclusive range), IN (membership), LIKE/ILIKE (pattern match), and IS NULL / IS NOT NULL — the only correct way to test for a null.

Why it matters: null handling is a favourite trap. col = NULL never matches, and NOT IN against a list containing a null silently returns nothing. Knowing that the condition must be strictly true is the mental model that prevents both bugs.

SELECT * FROM products
WHERE price BETWEEN 100 AND 500
  AND category IN ('books', 'toys')
  AND name ILIKE '%pro%'
  AND discontinued IS NOT NULL;

Sorting, paging and DISTINCT

ORDER BY sorts the result (ascending by default; DESC, plus NULLS FIRST/NULLS LAST to place nulls). LIMIT caps the row count and OFFSET skips rows first — the standard top-N and pagination pattern. Crucially, LIMIT is only meaningful with an ORDER BY on a unique key: without a defined order the rows returned are arbitrary and can change between runs.

DISTINCT removes duplicate rows; PostgreSQL's DISTINCT ON (expr) extends this to keep the first row per group under an ORDER BY — a compact "latest per customer" idiom. Note the scaling caveat: OFFSET still reads and throws away every skipped row, so deep pagination gets slower the further you page — the reason keyset (seek) pagination exists.

SELECT DISTINCT ON (customer) customer, id, created
FROM orders
ORDER BY customer, created DESC;   -- newest order per customer

SELECT id FROM orders
ORDER BY id
LIMIT 20 OFFSET 40;                -- rows 41-60 (page 3)

Interview questions

The most-asked DML and query questions in Indian interviews — from service-company screening rounds to product-company SQL deep dives:

Beginner Difference between DELETE, TRUNCATE and DROP?

DELETE removes selected rows (honours WHERE), is fully transactional, fires row triggers and can be rolled back. TRUNCATE empties the whole table fast without scanning rows and can reset identity, but can't target specific rows. DROP removes the table definition itself. Rule of thumb: rows → DELETE, all rows fast → TRUNCATE, the table → DROP.

Beginner What happens if you run UPDATE or DELETE without a WHERE clause?

Every row in the table is affected — a full-table update or delete. It's a classic outage cause. Safeguards: run the WHERE as a SELECT first to see the row count, wrap the statement in an explicit transaction so you can ROLLBACK, and never paste a write without its filter.

Beginner Why can't you write col = NULL, and what do you use instead?

NULL means "unknown", so any comparison with it — including = NULL — evaluates to unknown, never true, and the row is filtered out. Use IS NULL and IS NOT NULL, which are the dedicated predicates for testing null presence.

Beginner What is the RETURNING clause used for?

Appended to INSERT/UPDATE/DELETE, it returns columns from the rows that were just written, in the same statement — most commonly the auto-generated primary key after an insert, avoiding a second SELECT round-trip. It's a PostgreSQL feature (the SQL-standard equivalent is more limited).

Beginner What do LIMIT and OFFSET do?

LIMIT caps how many rows the query returns; OFFSET skips that many rows before it starts returning. Together they implement top-N results and pagination. They only make sense alongside an ORDER BY, otherwise which rows you get is undefined.

Intermediate Explain UPSERT in PostgreSQL, including EXCLUDED.

INSERT ... ON CONFLICT (target) DO NOTHING | DO UPDATE. The target names a unique or exclusion constraint. DO NOTHING skips the conflicting row; DO UPDATE modifies the existing row and refers to the values that failed to insert through the EXCLUDED pseudo-table (e.g. SET total = EXCLUDED.total). It's how you make a write idempotent and safe to retry.

Intermediate What is the logical order of execution of a SELECT, and why does it matter for aliases?

FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. A column alias is created at the SELECT step, which runs after WHERE — so WHERE can't reference the alias, but ORDER BY (which runs after SELECT) can. This ordering also explains why WHERE filters rows and HAVING filters groups.

Intermediate DISTINCT vs DISTINCT ON vs GROUP BY?

DISTINCT removes duplicate whole rows from the output. DISTINCT ON (cols) is PostgreSQL-specific — it keeps the first row of each group defined by those columns, decided by the ORDER BY (handy for "latest per key"). GROUP BY collapses rows into groups so you can apply aggregate functions. DISTINCT dedupes; GROUP BY aggregates.

Intermediate IN vs BETWEEN vs LIKE — when do you use each?

IN tests membership in a list or subquery (status IN ('new','paid')). BETWEEN is an inclusive range: x BETWEEN a AND b equals x >= a AND x <= b. LIKE/ILIKE do pattern matching with % (any run) and _ (one char); ILIKE is case-insensitive. Caveat: a NULL inside an IN/NOT IN list changes the result.

Intermediate Difference between WHERE and HAVING?

WHERE filters individual rows before grouping and cannot use aggregate functions. HAVING filters groups after GROUP BY and aggregation, so it can reference aggregates like count(*) > 10. If there's no GROUP BY, HAVING treats the whole result as one group. Filter rows in WHERE where possible — it's cheaper.

Intermediate Why must LIMIT be paired with ORDER BY for reliable results?

Without ORDER BY, SQL makes no promise about row order, so LIMIT returns an arbitrary subset that can differ between executions (and across plan changes). Order by a unique/tie-broken key so "page 2" is stable and reproducible.

Pro Why is OFFSET pagination slow at scale, and what replaces it?

OFFSET still reads and discards every skipped row, so cost grows linearly with page depth, and rows inserted/deleted between requests make pages drift. Keyset (seek) pagination replaces it: remember the last row's sort key and query WHERE key > :last ORDER BY key LIMIT n. That uses an index range scan — constant cost per page and stable under concurrent writes.

Pro What's the NOT IN with NULL trap?

If the list or subquery behind NOT IN contains a NULL, the predicate evaluates to unknown for every row (because x <> NULL is unknown), so the query returns zero rows — silently, no error. Fixes: use NOT EXISTS (correlated), or filter NULLs out of the subquery. It's a direct consequence of three-valued logic.

Pro How do you update one table using values from another?

PostgreSQL's UPDATE t SET col = s.val FROM src s WHERE t.id = s.id joins another table into the update (DELETE has the analogous USING clause). The danger: if the join matches multiple source rows for one target row, which value wins is undefined — ensure the join key is unique on the source side, or aggregate first. RETURNING confirms what changed.
