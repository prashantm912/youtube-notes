Module: Window Functions

A window function computes across a set of related rows yet keeps every row in the output — the one SQL skill that most often separates "can write a query" from "can clear the analytics round."

Knowledge Library → Courses → Databases → Window Functions

What this module covers

Window functions run a calculation over a "window" of rows related to the current row without collapsing them the way GROUP BY does. This module orients you on the mechanics — the OVER clause, PARTITION BY, ORDER BY and the frame that bounds each calculation — and then on the three families of functions built on top: ranking, offset/value, and aggregates used as windows. They power running totals, moving averages, rankings and top-N-per-group, and they are among the most-tested SQL topics in interviews.

Topics

Window functions vs. GROUP BY: the OVER clause

An aggregate with GROUP BY folds many rows into one summary row. A window function computes the same kind of aggregate but leaves every input row in place, attaching the result alongside it. The trigger is the OVER clause after the function call: avg(salary) OVER (PARTITION BY depname) gives each employee their department average without losing the individual rows. An empty OVER () treats the whole result set as one window — a grand total on every row.

Why it matters: this "keep the detail, add the summary" behavior is exactly what reporting and analytics need, and it removes the self-join or correlated-subquery gymnastics people reach for before they learn windows. One catch tested constantly: window functions are evaluated after WHERE, GROUP BY and HAVING, so you cannot reference one in a WHERE clause — you wrap the query in a CTE or subquery to filter on the result.

SELECT depname, empno, salary,
       avg(salary) OVER (PARTITION BY depname) AS dept_avg
FROM empsalary;

PARTITION BY, ORDER BY, and window frames

The window definition inside OVER (...) has three optional parts. PARTITION BY splits rows into independent groups the function restarts on (like GROUP BY, but the rows survive). ORDER BY sequences rows within each partition — it is what makes running totals, rankings and offset lookups meaningful. The frame then bounds which rows around the current one the function actually sees: ROWS counts physical rows, RANGE works in terms of ORDER BY value peers, and GROUPS counts peer groups, each with boundaries like UNBOUNDED PRECEDING, CURRENT ROW and n FOLLOWING.

Why it matters: the default frame surprises people. With ORDER BY present the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (a cumulative window); with no ORDER BY the frame is the entire partition. Getting the frame wrong is the difference between a correct running total and a subtly off one, so naming the frame explicitly is a mark of seniority.

SELECT product, category, price,
       sum(price) OVER (
         PARTITION BY category
         ORDER BY price
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total
FROM products;

Ranking functions: ROW_NUMBER, RANK, DENSE_RANK, NTILE

Ranking functions number rows within a partition according to the ORDER BY. row_number() gives a strict 1, 2, 3 with no ties. rank() gives tied rows the same rank and then skips the next values (1, 1, 3), while dense_rank() gives ties the same rank with no gap (1, 1, 2). ntile(n) distributes rows into n roughly equal buckets — the basis of quartiles and percentiles — and percent_rank() / cume_dist() express relative standing as a fraction.

Why it matters: "second-highest salary per department" and "top 3 per category" are staple interview questions, and the right answer hinges on choosing dense_rank() vs row_number() vs rank() correctly. Confusing them (especially how each handles ties) is the single most common ranking mistake.

SELECT name, department, salary,
       row_number() OVER w AS rn,
       rank()       OVER w AS rnk,
       dense_rank() OVER w AS drnk
FROM employees
WINDOW w AS (PARTITION BY department ORDER BY salary DESC);

Offset and value functions: LAG, LEAD, FIRST_VALUE, NTH_VALUE

Offset functions read other rows relative to the current one within the ordered partition. lag(x, n, default) looks back n rows, lead(x, n, default) looks forward — the direct way to compute month-over-month change, gaps between events, or "compared to previous". Value functions read from positions in the frame: first_value(), last_value() and nth_value() return an anchor value that every row in the window can compare against.

Why it matters: lag/lead replace painful self-joins for time-series deltas. And last_value() hides a classic trap — because the default frame ends at the current row, it returns the current row, not the partition's last; you must widen the frame to ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING to get the true final value.

SELECT month, revenue,
       lag(revenue) OVER (ORDER BY month)          AS prev_month,
       revenue - lag(revenue) OVER (ORDER BY month) AS mom_change
FROM monthly_sales;

Aggregate window functions and use cases

Any aggregate — sum, avg, count, min, max — becomes a window function simply by adding OVER. With an ordered frame this yields the everyday analytics workhorses: a running total (sum OVER (ORDER BY ...)), a moving average over the last seven rows (avg OVER (ORDER BY day ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)), a percent-of-total (value / sum(value) OVER ()), and top-N-per-group by wrapping a row_number() query and filtering the rank.

Why it matters: these are the reports businesses actually ask for, and expressing them with one window pass instead of correlated subqueries is both faster and clearer. Interviewers probe the frame details here — a running total on a column with duplicate order-by values behaves differently under RANGE (all peers included) versus ROWS (strict row-by-row).

-- 7-day moving average
SELECT day, amount,
       avg(amount) OVER (
         ORDER BY day
         ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS rolling_7day_avg
FROM daily_sales;

-- top 3 products per category
SELECT * FROM (
  SELECT *, row_number() OVER (PARTITION BY category ORDER BY sales DESC) AS rn
  FROM products
) t
WHERE rn <= 3;

Interview questions

Window functions are a favourite of Indian SQL, data-engineering and backend interviews — expect at least one, often the round's hardest query:

Beginner What is a window function and how does it differ from GROUP BY?

A window function computes across a set of rows related to the current row (its "window") but returns a value for every row — the rows are not collapsed. GROUP BY folds each group into a single summary row; a window keeps the detail rows and attaches the aggregate alongside each one.

Beginner What does the OVER clause do, and what does an empty OVER () mean?

OVER is what turns a function into a window function; it defines the window via optional PARTITION BY, ORDER BY and frame. An empty OVER () uses the entire result set as the window, so an aggregate like sum(x) OVER () returns the grand total on every row.

Beginner ROW_NUMBER vs RANK vs DENSE_RANK — what is the difference?

All number rows within a partition by the ORDER BY. row_number() is always unique (1, 2, 3), ignoring ties. rank() gives ties the same number then skips (1, 1, 3). dense_rank() gives ties the same number with no gap (1, 1, 2). Choose by whether you want gaps after ties.

Beginner What does PARTITION BY do, and how is it different from GROUP BY?

PARTITION BY divides rows into groups the window function restarts on, resetting counters, rankings and running totals per group. Unlike GROUP BY it does not reduce the number of output rows — the detail rows remain, each carrying its within-partition result.

Intermediate Why can't you put a window function in a WHERE clause, and how do you filter on its result?

Window functions are evaluated after FROM/WHERE/GROUP BY/HAVING, so their result does not exist yet when WHERE runs. Compute it in a subquery or CTE, then filter in the outer query — e.g. rank in an inner SELECT and keep rows whose rank is 3 or less outside.

Intermediate How do you find the second-highest salary per department?

Rank within each department and filter for rank 2: dense_rank() OVER (PARTITION BY dept ORDER BY salary DESC) in a CTE, then WHERE rnk = 2. Use dense_rank (not row_number) so tied top salaries do not push the true second value out, and to treat ties at second as a single rank.

Intermediate What is a window frame, and what is the default frame?

The frame is the subset of the partition a function actually reads for each row, set with ROWS/RANGE/GROUPS and boundaries. With an ORDER BY the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (cumulative). With no ORDER BY, the frame is the whole partition.

Intermediate LAG vs LEAD — give a real use case.

lag(x) reads a preceding row, lead(x) reads a following row, within the ordered partition. Month-over-month growth is the classic use: revenue - lag(revenue) OVER (ORDER BY month). Both take an optional offset and a default value for when no such row exists (avoiding NULLs at the edges).

Intermediate How would you write a running total and a 7-day moving average?

Running total: sum(amount) OVER (ORDER BY day) — the default cumulative frame. Moving average: avg(amount) OVER (ORDER BY day ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), which bounds the window to the current row plus the six before it.

Pro ROWS vs RANGE in a frame — when does the choice change the answer?

ROWS counts physical rows; RANGE works on ORDER BY value peers. When the ORDER BY column has duplicates, a running total under the default RANGE includes all tied peers at once (every duplicate row shows the same summed value), whereas ROWS adds strictly one row at a time. For a true per-row cumulative sum on duplicated keys, specify ROWS.

Pro Why does LAST_VALUE often return the current row instead of the partition's last, and how do you fix it?

Because with ORDER BY the default frame ends at CURRENT ROW, so the "last" row visible is the current one. Fix by extending the frame to ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING (or use first_value() with a descending order) so the window spans the whole partition.

Pro How do you fetch the top-N rows per group efficiently, and what are the tradeoffs?

Assign row_number() OVER (PARTITION BY group ORDER BY metric DESC) in a subquery/CTE and keep rows with number ≤ N. Use rank/dense_rank instead if ties should all qualify. Tradeoff: the window still scans and sorts each partition; for very large tables a LATERAL join with a per-group LIMIT backed by an index can beat it. On Postgres, DISTINCT ON is a concise top-1-per-group alternative.
