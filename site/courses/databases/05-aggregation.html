Module: Aggregation and Grouping

Aggregate functions collapse many rows into one number; GROUP BY decides how many groups you get, and HAVING, FILTER and the grouping-set family decide how much you can ask for in a single pass over the data.

Knowledge Library → Courses → Databases → Aggregation and Grouping

What this module covers

Aggregation turns row-level data into answers — totals, averages, and counts per category. This module covers the aggregate functions themselves, the GROUP BY and HAVING machinery that partitions rows and filters the resulting groups, the FILTER clause for conditional aggregates, and the GROUPING SETS / ROLLUP / CUBE extensions that produce subtotals and cross-tabs in one query. It is the backbone of every reporting query and one of the most heavily tested areas in SQL interview rounds.

Topics

Aggregate functions

An aggregate function reduces a set of input rows to a single value: count(*) tallies rows, sum() and avg() reduce a numeric column, min() and max() find the extremes, and string_agg() or array_agg() collect values into one field. Used without a GROUP BY, an aggregate treats the whole result set as a single group and returns exactly one row.

Why it matters: aggregates are how detail data becomes a report. The detail that trips people up is NULL handling — every aggregate except count(*) skips NULL inputs, so avg() divides by the count of non-null values, not the row count. PostgreSQL also offers ordered-set aggregates such as percentile_cont() and mode() through the WITHIN GROUP syntax.

SELECT count(*)        AS orders,
       sum(amount)     AS revenue,
       avg(amount)     AS avg_ticket,
       max(created_at) AS last_order
FROM orders;

GROUP BY and HAVING

GROUP BY partitions rows into groups that share the listed column values and produces one output row per group; every column in the SELECT list must then be either a grouping column or wrapped in an aggregate. HAVING filters those groups by an aggregate condition — it is to groups what WHERE is to rows.

Why it matters: the WHERE-versus-HAVING distinction is the single most common aggregation interview question. WHERE runs before grouping and cannot reference an aggregate; HAVING runs after and is the only place an aggregate condition like sum(amount) > 10000 belongs. Knowing the logical evaluation order — FROM, WHERE, GROUP BY, HAVING, SELECT, ORDER BY — explains both rules at once.

SELECT customer_id, count(*) AS orders, sum(amount) AS spend
FROM orders
WHERE status = 'paid'            -- filters rows before grouping
GROUP BY customer_id
HAVING sum(amount) > 10000       -- filters groups after aggregation
ORDER BY spend DESC;

The FILTER clause

FILTER attaches a WHERE condition to a single aggregate, so several conditional totals can be computed in one pass over the data: count(*) FILTER (WHERE status = 'paid') counts only paid rows while a neighbouring count(*) still sees everything. It is the SQL-standard, more readable replacement for the sum(CASE WHEN … THEN 1 ELSE 0 END) idiom.

Why it matters: interviewers often ask you to build a pivot-style summary — paid, refunded and pending counts per region — and FILTER answers it cleanly inside one GROUP BY. It reads better than CASE and gives the planner a clearer instruction about intent.

SELECT region,
       count(*)                                   AS total,
       count(*) FILTER (WHERE status = 'paid')     AS paid,
       count(*) FILTER (WHERE status = 'refunded') AS refunded
FROM orders
GROUP BY region;

GROUPING SETS, ROLLUP and CUBE

These extensions let one query return several levels of aggregation at once. GROUPING SETS lists the exact groupings you want; ROLLUP (a, b) produces the hierarchical prefixes — (a, b), (a), and the grand total () — for running subtotals; CUBE (a, b) produces every combination, the full power set, for cross-tabs. The alternative is a UNION ALL of many separate GROUP BY queries, which scans the table repeatedly.

Why it matters: reporting and OLAP work lives here. The subtle part interviewers probe is telling a real NULL apart from a subtotal row's NULL — that is what the GROUPING() function is for, returning a bitmask that marks which columns were rolled up in each row. PostgreSQL computes all the sets in a single pass, so it beats hand-written unions.

SELECT region, product, sum(amount) AS revenue
FROM sales
GROUP BY ROLLUP (region, product);
-- yields: per region+product, per region subtotal, and grand total

Interview questions

Aggregation is a staple of Indian SQL rounds — from service-company screens to product-company data and backend interviews. These are the ones that recur:

Beginner What is an aggregate function? Give examples.

A function that reduces many input rows to one value. Common ones: count, sum, avg, min, max, plus string_agg/array_agg for collecting values. Without GROUP BY they aggregate the entire result set into a single row.

Beginner What does GROUP BY do?

It partitions rows into groups sharing the same values in the listed columns and returns one row per group. Any column in SELECT that is not in GROUP BY must be inside an aggregate, otherwise PostgreSQL raises an error.

Beginner Difference between WHERE and HAVING?

WHERE filters individual rows before grouping and cannot use aggregates. HAVING filters whole groups after aggregation and is where aggregate conditions like sum(amount) > 10000 go. Use WHERE to cut rows early (cheaper), HAVING only for post-aggregate tests.

Beginner count(*) vs count(column) vs count(DISTINCT column)?

count(*) counts all rows including those with NULLs. count(column) counts rows where that column is non-null. count(DISTINCT column) counts distinct non-null values. The NULL-skipping of the last two is a frequent trap.

Beginner How do aggregates treat NULL values?

Every aggregate except count(*) ignores NULL inputs. So avg(x) is sum(x) over the count of non-null x — NULLs neither add to the sum nor to the divisor, which is why AVG can differ from what a manual "sum divided by row count" would give.

Intermediate Which columns are legal in a SELECT that has GROUP BY?

Only the grouping columns and aggregate expressions. Selecting a non-grouped, non-aggregated column is an error in PostgreSQL. The one relaxation: if you GROUP BY a table's primary key, the other columns of that table are functionally dependent and may be selected.

Intermediate What is the FILTER clause and why prefer it to CASE?

FILTER applies a WHERE to one aggregate only: count(*) FILTER (WHERE status = 'paid'). It replaces the sum(CASE WHEN cond THEN 1 ELSE 0 END) idiom with clearer, standard syntax, lets you compute many conditional aggregates in a single pass, and signals intent more directly to the planner.

Intermediate Why can you reference a SELECT alias in ORDER BY but not in WHERE or HAVING?

Logical evaluation order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. WHERE and HAVING run before the SELECT list is projected, so its aliases do not exist yet; ORDER BY runs after, so they do. (PostgreSQL does allow an output-column name in GROUP BY as a convenience.)

Intermediate Over an empty table, what does COUNT return with and without GROUP BY?

Without GROUP BY, an aggregate over zero rows still returns one row — count(*) gives 0, while sum/avg/max give NULL. With a GROUP BY there are no groups to form, so the query returns zero rows. This asymmetry surprises people.

Intermediate How would you compute a per-group percentage, e.g. share of paid orders per region?

Use FILTER (or CASE) inside the aggregate: count(*) FILTER (WHERE status = 'paid') * 100.0 / count(*), grouped by region. Multiply by a numeric literal (100.0) so integer division does not truncate the result to zero.

Pro GROUPING SETS vs ROLLUP vs CUBE — what is the difference?

All produce multiple grouping levels in one query. ROLLUP (a, b) gives the hierarchical prefixes — (a,b), (a), () — for subtotals plus a grand total. CUBE (a, b) gives every subset — the full power set (2^n sets) — for cross-tabs. GROUPING SETS lets you list exactly the combinations you want; ROLLUP and CUBE are shorthands for common patterns of it.

Pro What is the GROUPING() function for?

In a ROLLUP/CUBE/GROUPING SETS result, subtotal rows carry NULL in the columns that were rolled up — indistinguishable from a genuine NULL data value. GROUPING(col) returns 1 when that column was aggregated away for the row and 0 otherwise (a bitmask across several columns), so you can label subtotal rows and separate them from real NULLs.

Pro Why choose grouping sets over UNION ALL of several GROUP BY queries?

Each separate GROUP BY re-scans the table, so a UNION ALL of N groupings does N passes; PostgreSQL computes all the requested sets in a single scan with one sort/hash stage. Fewer passes, less I/O, and one coherent result — the same argument that favours a single FILTER-based query over multiple correlated subqueries.
