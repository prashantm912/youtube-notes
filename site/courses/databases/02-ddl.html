Module: SQL DDL and Constraints

A schema is born in DDL — and the constraints you attach at CREATE time are the cheapest, most reliable data-integrity guarantee you will ever get, enforced by the database no matter which application writes to it.

Knowledge Library → Courses → Databases → SQL DDL and Constraints

What this module covers

Data Definition Language is the half of SQL that shapes the container rather than the data: CREATE, ALTER and DROP TABLE for defining and evolving structure, and the six constraint types (NOT NULL, DEFAULT, UNIQUE, PRIMARY KEY, FOREIGN KEY, CHECK) that push correctness down into the engine. It closes with auto-generated keys — sequences, the legacy SERIAL, and the SQL-standard GENERATED AS IDENTITY. Get the schema right here and every query in the modules that follow gets easier.

Topics

Creating tables: CREATE TABLE

CREATE TABLE names a relation and lists its columns, each with a data type and optional inline constraints. IF NOT EXISTS makes the statement idempotent for setup scripts; column definitions can carry NOT NULL, DEFAULT, UNIQUE, PRIMARY KEY, REFERENCES and CHECK right where the column is declared, or you can list them as separate table-level constraints at the end.

Why it matters: the shape you commit to now decides what the database can guarantee later. Choosing the right type and the right key strategy at creation is far cheaper than migrating a live table, and interviewers routinely ask candidates to write a small schema from a word problem.

CREATE TABLE IF NOT EXISTS author (
    author_id  bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    full_name  text NOT NULL,
    email      text UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);

Evolving and removing tables: ALTER TABLE and DROP TABLE

ALTER TABLE is how a schema changes over its life: ADD COLUMN, DROP COLUMN, RENAME COLUMN, ALTER COLUMN … SET DATA TYPE, SET/DROP DEFAULT, SET/DROP NOT NULL, and ADD/DROP CONSTRAINT. DROP TABLE removes it entirely; RESTRICT (the default) refuses if other objects depend on it, while CASCADE also removes dependent views and foreign-key constraints.

Why it matters: these run on production tables, so lock behaviour is the real subject. Knowing which alterations take a brief metadata-only lock and which rewrite every row — and how to avoid the rewrite — is the difference between a safe migration and an outage.

ALTER TABLE author ADD COLUMN country char(2);
ALTER TABLE author ALTER COLUMN full_name SET NOT NULL;
ALTER TABLE author RENAME COLUMN country TO country_code;
DROP TABLE IF EXISTS author CASCADE;   -- also drops dependent views and FKs

Constraints: the rules that guard your data

Constraints move invariants out of application code and into the schema, where the engine enforces them for every writer. NOT NULL forbids missing values; DEFAULT supplies one when none is given; UNIQUE forbids duplicates; PRIMARY KEY is UNIQUE plus NOT NULL and marks the row identifier; FOREIGN KEY enforces referential integrity against another table, with ON DELETE/ON UPDATE actions (CASCADE, SET NULL, SET DEFAULT, RESTRICT, NO ACTION); CHECK validates a boolean expression per row.

Why it matters: a constraint is documentation the database enforces. Naming constraints explicitly (CONSTRAINT uq_book_isbn UNIQUE (isbn)) gives you stable error identifiers and clean migrations, and the PK-vs-UNIQUE and foreign-key-action questions are staples of Indian SQL screening rounds.

CREATE TABLE book (
    book_id   bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    author_id bigint NOT NULL REFERENCES author (author_id) ON DELETE CASCADE,
    title     text NOT NULL,
    price     numeric(8,2) CHECK (price >= 0),
    isbn      text,
    CONSTRAINT uq_book_isbn UNIQUE (isbn)
);

Auto-generated keys: sequences, SERIAL and identity columns

Most tables want a machine-generated surrogate key. A sequence is a standalone counter you read with nextval(). SERIAL is legacy shorthand that quietly creates a sequence and wires it as the column default — convenient, but the sequence is invisible in the table definition and needs its own privileges. GENERATED ALWAYS | BY DEFAULT AS IDENTITY (SQL-standard, PostgreSQL 10+) does the same job self-documented in the schema, and is the modern recommendation for new tables.

Why it matters: "SERIAL vs IDENTITY" is one of the most-asked Postgres design questions, and the ALWAYS-vs-BY DEFAULT distinction has real consequences for inserts, backups and restores. Knowing why identity won is a senior-level signal.

-- SERIAL: legacy shorthand, the sequence is hidden from the table definition
CREATE TABLE t1 (id serial PRIMARY KEY);

-- Identity: SQL-standard, self-documenting, preferred for new tables
CREATE TABLE t2 (id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY);

-- A standalone sequence for shared or custom numbering
CREATE SEQUENCE invoice_no START 1000 INCREMENT 1;
SELECT nextval('invoice_no');

Interview questions

The DDL and constraint questions that recur in Indian SQL interviews — from service-company screening to product-company schema-design rounds:

Beginner What is DDL and how does it differ from DML?

DDL (Data Definition Language) defines and changes structure — CREATE, ALTER, DROP, TRUNCATE. DML (Data Manipulation Language) works on the rows inside that structure — INSERT, UPDATE, DELETE, SELECT. In PostgreSQL DDL is transactional (you can wrap it in a transaction and roll it back), which is not true of every database.

Beginner Difference between a PRIMARY KEY and a UNIQUE constraint?

A primary key is UNIQUE plus NOT NULL and there is exactly one per table — it is the row's identity. A table can have many UNIQUE constraints, and a unique column may allow one NULL (NULLs are treated as distinct in standard SQL). Both are backed by a unique index.

Beginner What does NOT NULL do, and can a column be NOT NULL with a DEFAULT?

NOT NULL rejects any row that leaves the column empty. It combines happily with DEFAULT: the default fills the value when the insert omits the column, so the NOT NULL check always passes — a common pattern is created_at timestamptz NOT NULL DEFAULT now().

Beginner What is a foreign key and what does it enforce?

A foreign key says a column's values must exist in a referenced table's primary key or unique column — referential integrity. It blocks inserting an order for a customer who does not exist and, by default, blocks deleting a parent row that still has children, unless you specify an ON DELETE action.

Beginner DROP TABLE vs TRUNCATE vs DELETE?

DELETE (DML) removes rows by condition and can be rolled back row by row. TRUNCATE (DDL) empties the whole table quickly by deallocating storage, resetting nothing you filter. DROP TABLE removes the table definition itself, along with its indexes, constraints and triggers.

Intermediate Column-level vs table-level constraints — when must you use table-level?

A column-level constraint is written inline and applies to that one column. You must switch to a table-level constraint when it spans multiple columns — a composite primary key, a multi-column UNIQUE, a composite foreign key, or a CHECK that compares two columns. Naming it (CONSTRAINT …) also gives predictable error identifiers.

Intermediate Explain ON DELETE CASCADE vs SET NULL vs RESTRICT/NO ACTION.

They decide what happens to child rows when a referenced parent is deleted. CASCADE deletes the children too; SET NULL nulls the child's foreign-key column; SET DEFAULT sets it to the column default; RESTRICT and NO ACTION both block the delete, differing only in when the check fires (NO ACTION is deferrable to end of statement/transaction).

Intermediate SERIAL vs GENERATED AS IDENTITY — which do you choose and why?

Prefer identity for new tables. It is SQL-standard, visible in the table definition, and needs no separate sequence privileges. SERIAL hides its sequence, complicates ownership and grants, and lets callers overwrite the value silently. Identity also offers ALWAYS to lock out manual values. Use SERIAL only when you must support pre-10 PostgreSQL.

Intermediate What is a CHECK constraint, and can it reference other columns?

A CHECK enforces a boolean expression per row, e.g. CHECK (price >= 0) or CHECK (end_date > start_date). A column-level check sees only its own column; a table-level check can reference several columns of the same row. It cannot run subqueries or reference other tables — use a foreign key or a trigger for that.

Intermediate When you drop a SERIAL column vs an identity column, what happens to its sequence?

An identity column owns its sequence, so dropping the column (or the table) drops the sequence automatically. A SERIAL column's sequence is owned via ALTER SEQUENCE … OWNED BY, so it is also dropped with the column — but if that ownership link was ever broken, the sequence can be orphaned and left behind. Identity removes that footgun.

Pro Does adding a NOT NULL column with a DEFAULT rewrite the whole table?

Not since PostgreSQL 11 for a non-volatile default: the value is stored once in the catalog and materialised on read, so ADD COLUMN … DEFAULT is a fast metadata change. A volatile default (a per-row function) still rewrites every row and takes an ACCESS EXCLUSIVE lock. On older versions, any defaulted add rewrote the table — a classic large-table outage.

Pro How do you add a foreign key or NOT NULL to a huge table without a long lock?

Add the constraint as NOT VALID first — it is enforced for new and modified rows but skips the full-table validation scan, taking only a brief lock. Then run VALIDATE CONSTRAINT in a separate transaction, which takes a weaker SHARE UPDATE EXCLUSIVE lock and does not block reads or writes. For NOT NULL on big tables, add a CHECK (col IS NOT NULL) NOT VALID, validate it, then SET NOT NULL (modern PostgreSQL uses the valid check to skip the rescan).

Pro GENERATED ALWAYS vs BY DEFAULT AS IDENTITY — behaviour on explicit inserts and restores?

BY DEFAULT accepts a user-supplied value like a normal default; ALWAYS rejects one unless the insert says OVERRIDING SYSTEM VALUE. ALWAYS is safer for surrogate keys, and restore still works because pg_dump reloads with OVERRIDING SYSTEM VALUE. The catch: inserting explicit ids does not advance the identity's sequence, so after a bulk load you may need setval (or ALTER TABLE … ALTER COLUMN … RESTART) to avoid duplicate-key errors later.
