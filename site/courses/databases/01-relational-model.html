Module: The Relational Model and PostgreSQL

What this module covers

Everything that follows in this track sits on two foundations laid here: the relational model — data as tables of typed rows and columns, reached through the psql terminal — and PostgreSQL's rich type system. You will meet the everyday types (numeric, text, date/time) and the three that make Postgres stand out for interviews and real systems: jsonb, arrays, and uuid. Depth on each lives in the linked official docs; this page orients you and points the way.

Topics

The relational model, tables and columns, and getting into PostgreSQL

A relational database stores data in relations — tables — where each row is a record and each column is a named, typed field shared by every row. This is the whole idea: the data's shape is declared once as a schema, the engine enforces it, and you retrieve data by describing what you want (SQL) rather than how to fetch it. Row order is not guaranteed; if you need an order you ask for it with ORDER BY.

Why it matters: PostgreSQL is a full RDBMS you talk to over a client/server connection, and psql is the terminal front-end every backend engineer is expected to drive — connecting, listing objects with meta-commands like \dt and \d, and running statements. Installing a server, creating a database, and reaching it with psql is the baseline setup this whole track assumes.

CREATE TABLE country (
  code    char(2) PRIMARY KEY,   -- one row per country
  name    text NOT NULL,         -- every column has a type
  region  text
);
-- in psql:
--   \dt            list tables
--   \d country     describe the columns
SELECT name FROM country ORDER BY name;   -- order is asked for, not assumed

Data types: numbers, text, dates, JSONB, arrays and UUID

A column's type is a contract PostgreSQL enforces on every value. The core families are numeric (integer/bigint, exact numeric, floating real/double precision), character (text and varchar — prefer them over fixed char), and date/time (date, timestamptz, interval). Choosing the right one is genuine engineering: money is numeric for exact arithmetic, never a float; identifiers are bigint or uuid, not text.

Why Postgres stands out: jsonb stores JSON in a decomposed binary form that is fast to query and indexable (unlike plain json), arrays let a column hold an ordered list of values, and uuid gives a 128-bit key (Postgres now generates both UUIDv4 and the time-ordered UUIDv7). These three are the most-asked "why Postgres" topics in interviews — know when each earns its place and when it becomes a schemaless escape hatch.

CREATE TABLE product (
  id        uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  price     numeric(10,2) NOT NULL,   -- exact money, never float
  tags      text[],                   -- an array column
  specs     jsonb,                    -- semi-structured attributes
  added_on  date DEFAULT current_date
);
SELECT id FROM product WHERE specs @> '{"color":"red"}';  -- jsonb containment

Interview questions

The relational-model and data-type questions Indian interviews open with — screening rounds through product-company deep dives:

Beginner What is the relational model — define table, row, column, and relation?

A relational database stores data in relations, which we call tables. A table is a named collection of rows; every row has the same set of named, typed columns. A row (tuple) is one record; a column (attribute) is one typed field. The schema declares this structure once and the engine enforces it on every write.

Beginner How do you connect to a database and inspect it in psql?

Run psql -d dbname -U user (or psql "postgresql://..."). Inside, meta-commands do the work: \l lists databases, \c dbname connects, \dt lists tables, \d tablename describes a table's columns and indexes, and \q quits. SQL statements end with a semicolon.

Beginner VARCHAR vs CHAR vs TEXT — which should you default to?

In PostgreSQL there is no performance penalty for text; varchar(n) only adds a length check and char(n) blank-pads to a fixed width (rarely wanted). Default to text, and use varchar(n) only when a real business limit exists. Unlike some databases, none of the three is faster than the others here.

Beginner What is NULL and how does it affect comparisons?

NULL means "unknown / no value" — not zero and not an empty string. SQL uses three-valued logic, so x = NULL is never true; you must test with IS NULL / IS NOT NULL. NULLs also propagate: 1 + NULL is NULL, and most aggregates skip NULLs.

Intermediate JSON vs JSONB — what is the difference and when do you use each?

json stores the exact input text (preserving whitespace, key order, duplicate keys) and reparses on every access. jsonb stores a decomposed binary form: no whitespace/order preservation, last duplicate key wins, slightly slower to insert but far faster to query and — crucially — indexable with GIN. Use jsonb for almost everything; reach for json only when you must reproduce the original document byte-for-byte.

Intermediate Why store money as numeric rather than real or double precision?

Floating-point types are binary approximations, so values like 0.1 cannot be represented exactly and rounding errors accumulate — unacceptable for currency. numeric(precision, scale) stores exact decimal values, e.g. numeric(10,2) for two-decimal money. The trade-off is slower arithmetic, which is fine for financial data.

Intermediate serial vs GENERATED AS IDENTITY — which is preferred?

serial is legacy sugar that creates a sequence and a column default. GENERATED ALWAYS AS IDENTITY (SQL-standard, Postgres 10+) is preferred: it ties the sequence to the column, prevents accidental manual inserts into the key, and survives table operations more cleanly. Both are backed by a sequence under the hood.

Intermediate UUID vs bigint as a primary key — trade-offs?

bigint/identity keys are small (8 bytes), sequential, and index-friendly. UUIDs (16 bytes) are globally unique, can be generated on the client without a round-trip, and don't leak row counts — but random UUIDv4 hurts B-tree locality and index cache. UUIDv7 (time-ordered, now generatable in Postgres via uuidv7()) restores insert locality, making it the modern choice when you need UUIDs.

Intermediate How do you declare, insert into, and query an array column?

Declare with text[] (or any base type). Insert with an array literal '{a,b,c}' or ARRAY['a','b']. Query with element access tags[1] (1-based), membership 'a' = ANY(tags), or overlap/contains operators && / @>. Note arrays can defeat normalization — often a join table is the better model.

Pro How do you index a JSONB column, and how does a GIN index with @> work?

A GIN index on a jsonb column indexes its keys and values, so containment queries (specs @> '{"color":"red"}') and key-existence (?) use the index instead of scanning. The default jsonb_ops indexes keys and values; jsonb_path_ops is smaller and faster but only supports @>. For a single hot key, a B-tree on an expression like (specs->>'status') can be cheaper than a whole-document GIN index.

Pro Is the physical order of rows in a table guaranteed?

No. Rows live in a heap and their order can change with updates, deletes, and VACUUM; a query without ORDER BY may return rows in any order, and that order can differ between runs or plans. Never rely on insertion order — always sort explicitly. This also explains why "give me the latest row" needs an ordering column (timestamp or identity), not just LIMIT 1.

Pro When does reaching for JSONB become an anti-pattern?

When it turns into schemaless creep: dumping structured, queried, related data into one jsonb blob to avoid designing tables. You lose column constraints, foreign keys, type checking, and clean indexing; queries grow gnarly and integrity moves into application code. Use jsonb for genuinely variable or sparse attributes and external payloads — keep first-class, relational, frequently-queried fields as real typed columns.
