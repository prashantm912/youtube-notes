Module: Subqueries and CTEs

A subquery is a query nested inside another; a CTE is that same idea given a name and lifted to the top of the statement. This module maps every form — scalar, correlated, EXISTS/IN, recursive WITH — and points to the resources that teach each fully, official PostgreSQL docs first.

What this module covers

Subqueries let one query feed another: a scalar value in a SELECT list, a set to test membership against, or a derived table in the FROM clause. Common Table Expressions (the WITH clause) name those subqueries so a complex statement reads top-to-bottom — and, with RECURSIVE, walk hierarchies and graphs that plain SQL cannot. The module closes with set operations (UNION, INTERSECT, EXCEPT), which stack result sets vertically rather than joining them side by side.

Topics

Scalar and correlated subqueries

A subquery is a SELECT wrapped in parentheses inside another statement. A scalar subquery returns exactly one row and one column, so it can stand anywhere a single value is expected — the SELECT list, a WHERE comparison, an UPDATE ... SET. A correlated subquery references a column from the outer query, which means it cannot run on its own and is conceptually re-evaluated once per outer row.

Why it matters: placement changes meaning and cost. An uncorrelated subquery runs once; a correlated one is the classic "executes per row" performance conversation that interviewers reach for. Two runtime rules trip people up — a scalar subquery that returns more than one row is an error, and one that returns no rows evaluates to NULL rather than failing.

-- Employees paid above their own department's average (correlated)
SELECT e.name, e.salary
FROM   employees e
WHERE  e.salary > (
    SELECT avg(d.salary)
    FROM   employees d
    WHERE  d.dept_id = e.dept_id   -- correlation to the outer row
);

Subquery predicates: EXISTS, IN, ANY/ALL — and when a JOIN is better

Three predicate forms test a value against a subquery's rows. EXISTS asks only whether the subquery returns any row (a semi-join — it can stop at the first match); IN checks membership in the returned list; ANY/SOME and ALL pair a comparison operator with the whole set. The infamous trap is NOT IN against a subquery that can produce a NULL: the whole predicate collapses to unknown and the query returns nothing — which is why NOT EXISTS is the safer default.

Why it matters: these are interchangeable with a JOIN surprisingly often, and choosing well is both a correctness and a performance question. A correlated EXISTS usually expresses "does a related row exist" more clearly than a join that risks duplicating rows; the planner may even rewrite one into the other, but the interviewer wants to hear that you can explain the difference.

-- Customers with at least one order — EXISTS short-circuits per customer
SELECT c.id, c.name
FROM   customers c
WHERE  EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id);

-- Same intent with IN; watch NOT IN if customer_id can be NULL
SELECT c.id, c.name
FROM   customers c
WHERE  c.id IN (SELECT o.customer_id FROM orders o);

Common Table Expressions: WITH and recursive CTEs

The WITH clause defines a named, temporary result set that exists for the duration of one statement. It turns a nest of subqueries into a readable pipeline: define regional, then query it — and a single WITH can hold several CTEs, each able to reference the ones before it. In PostgreSQL a CTE may be inlined into the outer plan or materialized; a plain CTE used more than once, or one marked MATERIALIZED, is computed once (before version 12 every CTE was an optimization fence).

Why it matters: WITH RECURSIVE is the only standard-SQL way to walk a hierarchy or graph — org charts, category trees, bill-of-materials, shortest paths. A recursive CTE has an anchor member (the starting rows) and a recursive member that references the CTE itself, combined by UNION or UNION ALL and iterated until it produces no new rows. Forgetting the termination condition, or using UNION ALL on cyclic data, is how you write an infinite loop.

-- Named pipeline: one CTE feeds the final query
WITH regional AS (
    SELECT region, sum(amount) AS total
    FROM   sales
    GROUP  BY region
)
SELECT region, total FROM regional WHERE total > 1000000;

-- Recursive: walk an org chart down from the top
WITH RECURSIVE chart AS (
    SELECT id, name, manager_id, 1 AS depth          -- anchor
    FROM   employees WHERE manager_id IS NULL
  UNION ALL
    SELECT e.id, e.name, e.manager_id, c.depth + 1   -- recursive member
    FROM   employees e JOIN chart c ON e.manager_id = c.id
)
SELECT depth, name FROM chart ORDER BY depth;

Set operations: UNION, INTERSECT, EXCEPT

Set operations stack the results of two queries vertically instead of joining them side by side. The inputs must be union-compatible — the same number of columns with compatible types. UNION concatenates and removes duplicate rows; UNION ALL keeps every row and is faster because it skips the de-duplication sort. INTERSECT returns rows present in both queries, and EXCEPT returns rows in the first query that are absent from the second.

Why it matters: these express "combine", "in both", and "in one but not the other" directly, which is cleaner than emulating them with outer joins and NULL checks. A common interview note: an ORDER BY applies to the combined result, not the individual arms, and every operator has an ALL variant that preserves duplicates.

-- Emails active in 2024 but not in 2023 (churn-in)
SELECT email FROM customers_2024
EXCEPT
SELECT email FROM customers_2023;

-- Combine two sources, keeping duplicates, then sort the whole result
SELECT city FROM warehouses
UNION ALL
SELECT city FROM stores
ORDER BY city;

Interview questions

Most-asked subquery and CTE questions in Indian interviews — SQL is a screening staple for backend, data-engineering and analyst roles alike:

Beginner What is a subquery and where can it appear?

A query nested inside another statement. It can appear in the SELECT list (as a scalar value), in FROM (as a derived table), and in WHERE/HAVING (as a value or a set to test against). It runs first — or per outer row if correlated — and feeds the enclosing query.

Beginner What is a scalar subquery, and what if it returns more than one row?

A subquery that returns exactly one row and one column, usable wherever a single value fits. If it returns more than one row it is a runtime error; if it returns no rows the result is NULL (not an error). That NULL behavior is a frequent gotcha.

Beginner What is a correlated subquery?

A subquery that references a column from the outer query, so it cannot be run independently and is logically re-evaluated for each outer row. Example: comparing each employee's salary to the average of their department. Uncorrelated subqueries, by contrast, run once.

Beginner UNION vs UNION ALL?

UNION removes duplicate rows (it sorts/hashes to de-duplicate); UNION ALL keeps every row and skips that work, so it is faster. Use UNION ALL when the arms are known to be disjoint or duplicates are acceptable.

Intermediate EXISTS vs IN — how do they differ and when do you pick each?

EXISTS is a semi-join: it returns true at the first matching row and pairs naturally with correlation. IN compares against the full list of returned values. For large or correlated checks EXISTS often reads clearer and lets the planner stop early; for a small static list IN is fine. Modern planners frequently transform between them.

Intermediate Why can NOT IN with a subquery return no rows unexpectedly?

If the subquery yields any NULL, NOT IN evaluates to unknown for every outer row (because "x not in {…, NULL}" can never be proven true), so the query returns nothing. Fix by filtering the NULLs out or, better, using NOT EXISTS, which has the correct semantics.

Intermediate What is a CTE and why use one instead of a subquery?

A Common Table Expression is a named temporary result set defined with WITH, scoped to a single statement. It improves readability by naming steps, lets you reference the same result more than once, chains multiple named steps, and is the entry point for recursion — advantages a deeply nested subquery lacks.

Intermediate Correlated subquery vs JOIN — which performs better?

It depends on the planner. Naively a correlated subquery runs per outer row, while a join processes sets in bulk — so joins often win at scale. But PostgreSQL can rewrite a correlated EXISTS into a semi-join, making them equivalent. Read EXPLAIN rather than assuming; pick the form that expresses intent, then verify the plan.

Intermediate What do INTERSECT and EXCEPT return, and how would you emulate them with joins?

INTERSECT returns rows in both queries; EXCEPT returns rows in the first but not the second (both de-duplicate unless you add ALL). Emulate INTERSECT with an inner join on all columns, and EXCEPT with a LEFT JOIN ... WHERE right IS NULL anti-join (or NOT EXISTS).

Pro Walk through how a recursive CTE executes.

PostgreSQL evaluates the anchor member once to seed a working table. It then repeatedly runs the recursive member against the rows produced in the previous step, appending results, until an iteration returns no new rows. UNION de-duplicates across iterations (helping terminate on cyclic data); UNION ALL does not, so cycles need an explicit path/visited guard or a CYCLE clause.

Pro Is a CTE an optimization fence in PostgreSQL?

Historically yes — before version 12 every CTE was materialized and never folded into the outer query. Since 12, a non-recursive CTE referenced once is inlined by default (NOT MATERIALIZED); you can force the old behavior with MATERIALIZED. Recursive and multiply-referenced CTEs are still materialized. Knowing this explains sudden plan changes after an upgrade.

Pro A scalar subquery in a SELECT list is slow. How do you diagnose and rewrite it?

A correlated scalar subquery in the SELECT list runs per output row. Confirm with EXPLAIN (ANALYZE) — look for a repeated subplan or high loop count. Typical rewrites: convert to a LEFT JOIN against a pre-aggregated derived table or CTE, or a LATERAL join, so the aggregate is computed once per group instead of once per row. A covering index on the correlation key helps either way.
