Module: Normalization and Schema Design

Normalization is theory with a daily payoff — the reasoning that turns a spreadsheet dumped into one table into a schema that cannot drift out of sync. Walking 1NF to BCNF on a whiteboard is close to guaranteed in Indian interviews, so this module orients you on each form and points to the resources that drill them.

What this module covers

This module covers relational schema design end to end: the functional dependencies and keys that normalization is defined against, the walk from 1NF through BCNF with the anomalies each form removes, when deliberate denormalization earns its keep, and the everyday patterns — soft delete, audit trails, multi-tenancy — that production tables grow into. The forms are theory; the patterns are what you actually ship. Both are interview staples for backend and data roles.

Topics

Functional dependencies, keys, and the anomalies that motivate normalization

A functional dependency X → Y means each value of X determines exactly one value of Y (an employee_id determines one name). Keys are the special dependencies: a candidate key is a minimal set of attributes that determines every other attribute; a prime attribute belongs to some candidate key, a non-prime attribute belongs to none. Every normal form above 1NF is stated in terms of how non-prime attributes depend on keys, so this vocabulary is the whole foundation.

Why it matters: normalization exists to kill three redundancy-born bugs. The insertion anomaly (you cannot record a fact without inventing unrelated data), the update anomaly (a value duplicated across rows must be changed everywhere at once or the data diverges), and the deletion anomaly (removing one row silently loses an unrelated fact). Spotting which anomaly a badly-shaped table causes is the framing interviewers want.

First, Second and Third Normal Forms

The classic progression, each form assuming the one before it. 1NF: every column holds a single atomic value — no arrays, comma-lists, or repeating Class1/Class2/Class3 columns. 2NF: no partial dependency — no non-prime attribute depends on only part of a composite candidate key (only an issue when the key spans several columns). 3NF: no transitive dependency — no non-prime attribute depends on another non-prime attribute. If student → zip and zip → city, then city reaches the key only through zip, so it belongs in its own table.

Why it matters: 3NF is the practical target for almost every transactional schema — it removes the common anomalies while keeping writes to a single place. Interviewers routinely hand you an unnormalized table and ask you to decompose it form by form, naming the dependency each split removes.

-- Transitive dependency in one table: student -> advisor -> advisor_room
-- 3NF split moves the advisor -> room fact out of the student row
CREATE TABLE advisor (id INT PRIMARY KEY, name TEXT, room TEXT);
CREATE TABLE student (id INT PRIMARY KEY, name TEXT,
                      advisor_id INT REFERENCES advisor(id));

Boyce-Codd Normal Form and higher forms

BCNF tightens 3NF to a single rule: for every non-trivial dependency X → Y, X must be a superkey. 3NF is more forgiving — it lets a determinant that is not a superkey stand as long as the attribute it determines is itself prime. The textbook failing case has overlapping candidate keys: a relation is in 3NF yet a non-key determinant still exists, so BCNF fails. Decomposing into BCNF can cost dependency preservation, which is exactly why some designs deliberately stop at 3NF.

Why it matters: the "give me a table that is in 3NF but not BCNF" question separates people who memorized the forms from people who understand them. Beyond BCNF, 4NF (multivalued dependencies) and 5NF (join dependencies) exist but are mostly academic — worth naming, rarely worth reaching for.

-- R(student, subject, teacher)
--   {student, subject} -> teacher    each student-subject pair has one teacher
--   teacher            -> subject     each teacher teaches exactly one subject
-- Candidate keys: {student, subject} and {student, teacher}
-- teacher -> subject has a determinant that is NOT a superkey
--   => 3NF holds (subject is prime) but BCNF is violated

Denormalization and its trade-offs

Denormalization deliberately reintroduces redundancy — duplicated columns, precomputed aggregates, materialized views — to cut joins and aggregations on hot read paths. It is not the opposite of normalization but an optimization applied after a clean design, and only once indexing, caching, and query tuning have proven insufficient. It shines for read-heavy and reporting workloads with predictable access patterns, which is why analytics warehouses use denormalized star schemas.

Why it matters: the trade-off is the answer interviewers grade. Normalization gives simple, single-place writes and slower multi-join reads; denormalization gives fast reads and write amplification with a real risk of stale or inconsistent copies. Any denormalized value needs a keep-it-correct mechanism — a trigger, a scheduled job, or change-data-capture. The maxim: normalize until it hurts, denormalize until it works.

-- Cache an aggregate that would otherwise need a JOIN + GROUP BY per read
ALTER TABLE author ADD COLUMN book_count INT NOT NULL DEFAULT 0;
-- The redundant column is only safe if a trigger or job keeps it in sync;
-- a stale book_count is the classic denormalization bug.

Schema design patterns: soft delete, audit trails, multi-tenancy

Real tables grow three recurring shapes. Soft delete keeps a deleted_at timestamp (NULL means live) instead of removing rows — every query must then filter it, and unique constraints must become partial indexes so a deleted row does not block reuse of its key. Audit trails record who changed what and when, most robustly with an AFTER trigger that writes old and new row values into a history table (PostgreSQL's trigger + hstore recipe is the canonical example). Multi-tenancy chooses among database-per-tenant (silo), schema-per-tenant (bridge), and shared tables keyed by tenant_id (pool).

Why it matters: the pool model is the cheapest at scale but leaks across tenants the moment an application query forgets its tenant_id filter. PostgreSQL row-level security moves that isolation into the database, so a policy — not application discipline — decides which rows a tenant can see. These patterns show up in system-design and backend rounds far more than the pure theory does.

-- Soft delete: NULL = live; uniqueness must ignore deleted rows
ALTER TABLE document ADD COLUMN deleted_at TIMESTAMPTZ;
CREATE UNIQUE INDEX ON document (tenant_id, slug) WHERE deleted_at IS NULL;

-- Multi-tenancy: enforce isolation in the DB, not just the app
ALTER TABLE document ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON document
  USING (tenant_id = current_setting('app.tenant_id')::int);

Interview questions

Normalization is one of the most reliably asked database topics in Indian interviews — expect to define the forms, spot the dependency that breaks one, and defend a denormalization decision:

Beginner What is normalization and what problem does it solve?

Structuring tables — guided by functional dependencies — so each fact lives in exactly one place. It removes redundancy and the insertion, update, and deletion anomalies redundancy causes. The price is more tables and more joins, which is a deliberate trade for correctness.

Beginner Name the three anomalies normalization prevents.

Insertion anomaly — you cannot record a fact without inventing unrelated data (no way to list a course with no enrolled student). Update anomaly — a value duplicated across rows must be changed everywhere at once or the data diverges. Deletion anomaly — removing one row silently loses an unrelated fact. All three come from storing a fact in more than one place.

Beginner What is First Normal Form?

Every column holds a single atomic value and there are no repeating groups — no comma-separated phones column, no Class1/Class2/Class3 columns faking a list. Move the repeating data into its own table with a one-to-many relationship.

Beginner What is a functional dependency, and what are prime vs non-prime attributes?

X → Y means each X value determines exactly one Y value (id → name). A prime attribute belongs to some candidate key; a non-prime attribute belongs to none. 2NF and 3NF are both defined by how non-prime attributes depend on the keys.

Beginner Primary key vs candidate key vs super key?

A super key is any attribute set that uniquely identifies a row (may carry extra columns). A candidate key is a minimal super key — no removable column. The primary key is the one candidate key you choose to enforce and reference; the rest remain alternate keys.

Intermediate What is 2NF and what is a partial dependency?

2NF is 1NF plus no partial dependency: no non-prime attribute depends on only part of a composite candidate key. It only arises when the key spans multiple columns. In (order_id, product_id) → …, a product_name that depends on product_id alone is partial — split product into its own table.

Intermediate What is 3NF and what is a transitive dependency?

3NF is 2NF plus no transitive dependency: no non-prime attribute depends on another non-prime attribute. If student → zip and zip → city, then city reaches the key only through zip; move zip/city to their own table. 3NF is the everyday target for OLTP schemas.

Intermediate Difference between a partial and a transitive dependency?

Partial: a non-prime attribute depends on part of a composite key — a 2NF violation, and it needs a multi-column key to even exist. Transitive: a non-prime attribute depends on another non-prime attribute — a 3NF violation that can occur even with a single-column key.

Intermediate In practice, which normal form do you design to, and why?

3NF (or BCNF) for transactional/OLTP schemas — it removes the common anomalies while keeping writes to a single place. Reporting and analytics (OLAP, warehouses) deliberately denormalize into star schemas for read speed. The rule of thumb: normalize until it hurts, denormalize until it works.

Intermediate How do you implement a soft delete, and what does it break?

Add a deleted_at (or is_deleted) column; NULL means live. Then every query must filter it, unique constraints must become partial indexes (WHERE deleted_at IS NULL) so a deleted key can be reused, and foreign keys and reports must account for hidden rows. You gain recoverability and history; you pay with bigger tables and easy-to-forget filters.

Intermediate How do you isolate tenants in a multi-tenant database?

Three models: silo (database per tenant), bridge (schema per tenant), and pool (shared tables with a tenant_id column). Pool is cheapest but leaks the moment a query forgets its tenant filter — PostgreSQL row-level security fixes that by enforcing tenant_id = current tenant in the database, so an application bug cannot cross tenants.

Pro What is BCNF, and when is a 3NF table not in BCNF?

BCNF requires that for every non-trivial dependency X → Y, X is a superkey. It is stricter than 3NF, which still allows a non-superkey determinant as long as the attribute it determines is prime. The classic failing case has overlapping candidate keys — R(student, subject, teacher) with {student,subject} → teacher and teacher → subject: it is 3NF (subject is prime) but teacher is not a superkey, so BCNF fails. Decomposing to BCNF can lose dependency preservation, which is why 3NF is sometimes kept on purpose.

Pro What is denormalization and when is it the right call?

Deliberately reintroducing redundancy — duplicated columns, precomputed aggregates, materialized views — to cut joins on hot read paths. Justified after indexing, caching, and query tuning fall short, for read-heavy or reporting workloads with predictable access. The cost is write amplification and stale-copy risk, so every denormalized value needs a trigger, scheduled job, or change-data-capture to stay correct.

Pro Does more normalization always mean better performance?

No. Normalization optimizes for correctness and cheap writes, not read latency — more tables means more joins. A heavily normalized schema can be slower for read-dominated queries, which is exactly why warehouses denormalize. The right level balances anomaly-free writes against join cost for the actual workload, not an abstract ideal.

Pro What are 4NF and 5NF about, briefly?

4NF removes multivalued dependencies: two independent multi-valued facts (a person's skills and their spoken languages) in one table create a spurious cross-product — split them into separate tables. 5NF (project-join normal form) removes join dependencies not implied by candidate keys. Both are rare in practice; most real designs stop at 3NF or BCNF.
