Module: Joins

A normalized schema deliberately scatters related facts across tables; joins are how you put them back together — and no single SQL topic is asked more often in Indian interviews than this one.

Knowledge Library → Courses → Databases → Joins

What this module covers

Joins combine rows from two or more tables on a related key. This module walks the everyday four (INNER, LEFT, RIGHT, FULL OUTER), the special shapes (CROSS, SELF, LATERAL), the three ways to write a join condition (ON, USING, NATURAL), and — the part that separates a screening pass from a senior answer — the physical algorithms PostgreSQL's planner chooses to actually execute them. It builds directly on the SELECT anatomy from the previous module and underpins aggregation, subqueries and every real reporting query.

Topics

INNER, LEFT, RIGHT, and FULL OUTER joins

Because a normalized design splits related data apart (orders in one table, customers in another), a join re-assembles it on a matching key. INNER JOIN keeps only rows that match on both sides. The three OUTER joins preserve unmatched rows and fill the missing side with NULLs: LEFT JOIN keeps every row from the left table, RIGHT JOIN keeps every row from the right, and FULL OUTER JOIN keeps every row from both.

Why it matters: picking the wrong join type silently drops or duplicates rows — the most common data-correctness bug in reports, and the single most-asked SQL interview topic in Indian screening rounds. LEFT JOIN is the one you reach for most: "give me all customers, and their orders if any."

SELECT c.name, o.id AS order_id
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id;
-- every customer appears; o.id is NULL for customers with no orders

CROSS, SELF, and LATERAL joins

A CROSS JOIN is the Cartesian product — every left row paired with every right row, so m rows and n rows produce m×n. It is rarely intentional (generating grids or a numbers series is the honest use); an accidental cross join from a forgotten ON is a classic runaway query. A self join is a table joined to itself under two aliases, used to relate rows within one table — employee to manager, a node to its parent, or comparing a row against others. A LATERAL join lets a subquery (or set-returning function) in the FROM clause reference columns from the tables listed before it — a correlated join, and the clean way to express "the latest N rows per group."

Why it matters: LATERAL turns awkward correlated-subquery-in-SELECT patterns into readable, index-friendly queries; the top-order-per-customer question below is a product-round favorite that LATERAL answers directly.

SELECT c.name, latest.total
FROM customers c
CROSS JOIN LATERAL (
  SELECT total FROM orders o
  WHERE o.customer_id = c.id
  ORDER BY o.created_at DESC LIMIT 1
) AS latest;                       -- one most-recent order per customer

Join conditions: ON, USING, and NATURAL

ON is the general form: any boolean predicate, columns may have different names, and non-equality conditions are allowed. USING (col) is shorthand for the common case where both sides share the join column name — it also collapses the two matched columns into a single output column. NATURAL JOIN goes further and joins automatically on all identically-named columns, writing no condition at all.

Why it matters: USING and NATURAL read cleanly but are fragile — later adding a column with a coincidentally shared name (an id, a created_at) silently changes what the query joins on. Most teams standardize on explicit ON, and interviewers ask about the trade-off to see whether you understand the risk rather than just the syntax.

SELECT * FROM orders o JOIN customers c ON o.customer_id = c.id; -- general
SELECT * FROM orders JOIN customers USING (customer_id);         -- one shared name
SELECT * FROM orders NATURAL JOIN customers;                     -- all shared names (risky)

How PostgreSQL executes joins: nested loop, hash, merge

The join keyword says what to combine; the cost-based planner decides how. There are three physical algorithms. A nested loop join scans the inner relation once for every outer row — excellent when the inner side is small or reachable by an index, catastrophic across two large unindexed tables. A hash join builds a hash table on one side and probes it with the other — the workhorse for large equi-joins whose hash table fits in work_mem. A merge join sorts both sides on the join key and zips them together in one pass — the winner when the inputs already arrive sorted (for example, straight off a B-tree index).

Why it matters: writing USING or NATURAL instead of ON does not change any of this — the planner still picks by estimated cost. Reading an EXPLAIN plan, recognizing a nested loop over two big tables as the reason a query got slow after data growth, and knowing that an index or fresher statistics fixes it, is exactly what product-company rounds probe.

Interview questions

Joins are the single most-tested SQL area in Indian interviews — service-company screening through product-company deep dives. These run from definitions to the execution and gotcha questions that separate answers:

Beginner What is a JOIN and why do we need one?

A JOIN combines rows from two or more tables based on a related column. Normalization deliberately spreads related data across tables to avoid duplication; joins re-assemble that data at query time, so you can, for instance, list each order alongside the customer who placed it.

Beginner INNER JOIN vs LEFT JOIN — what is the difference?

INNER JOIN returns only rows that have a match on both sides. LEFT JOIN returns every row from the left table plus matching right-table rows, filling the right-side columns with NULL where there is no match. Use LEFT when you must keep unmatched left rows (all customers, even those with zero orders).

Beginner What does a LEFT JOIN put in the right-side columns when there is no match?

NULL. That is the whole point of an outer join — the left row is preserved and every column that would have come from the unmatched right row is NULL. Filtering with WHERE right.col IS NULL is how you isolate the unmatched left rows.

Beginner What is a CROSS JOIN and how big is its result?

A CROSS JOIN is the Cartesian product: every row of the left table paired with every row of the right. With m and n rows the result has m×n rows. It has no join condition; an accidental one (a missing ON) is a common cause of a query exploding in size.

Beginner Is RIGHT JOIN necessary, given LEFT JOIN exists?

No — any RIGHT JOIN can be rewritten as a LEFT JOIN by swapping the two tables. RIGHT JOIN is rare in practice; most developers keep every join LEFT for readability so the "table we must keep all of" is always on the same side.

Intermediate What is a FULL OUTER JOIN, and does MySQL support it?

FULL OUTER JOIN keeps every row from both tables, matching where possible and NULL-padding the missing side otherwise — effectively LEFT and RIGHT combined. PostgreSQL supports it directly. MySQL historically does not, so there you emulate it with LEFT JOIN ... UNION ... RIGHT JOIN — a frequent interview follow-up since many Indian shops use MySQL.

Intermediate What is a SELF JOIN and give a real use case?

A self join joins a table to itself using two different aliases. Classic uses: an employees table where each row's manager_id points to another row (list each employee with their manager's name), traversing one level of a hierarchy, or comparing rows within the same table.

Intermediate ON vs USING vs NATURAL — the differences, and why is NATURAL risky?

ON takes any predicate and allows different column names and non-equality conditions. USING (col) is shorthand when both sides share the column name and collapses it to one output column. NATURAL JOIN auto-joins on all identically-named columns. NATURAL is risky because adding a coincidentally-named column later (like id or created_at) silently changes the join — prefer explicit ON.

Intermediate A LEFT JOIN is returning the same rows as an INNER JOIN. Why?

Almost always because a condition on the right table sits in WHERE instead of ON. The LEFT JOIN produces NULLs for unmatched rows, then WHERE right.col = x discards every NULL row, collapsing the result to an inner join. Move the filter into the ON clause to keep the outer behavior.

Intermediate How do you find rows in table A that have no matching row in table B?

An anti-join: SELECT a.* FROM a LEFT JOIN b ON b.a_id = a.id WHERE b.a_id IS NULL, or equivalently WHERE NOT EXISTS (SELECT 1 FROM b WHERE b.a_id = a.id). Example: customers who have never placed an order. NOT EXISTS is often the cleaner and NULL-safe choice.

Intermediate Does the order in which you list tables in an INNER JOIN change the result?

Logically no — INNER JOIN is commutative, so the row set is identical either way. Physically it can matter: the planner decides which relation is the outer/driving side and which algorithm to use based on cost, so join order is a planner concern, not a correctness one. For OUTER joins the order does change meaning (which side is preserved).

Pro What is a LATERAL join and when do you actually need it?

LATERAL lets a subquery or set-returning function in the FROM clause reference columns from the FROM items listed before it — a correlated join. You need it for per-row correlated computations that a plain join cannot express, most famously top-N-per-group: for each customer, run a small ORDER BY ... LIMIT n subquery over their orders. It also drives functions like jsonb_array_elements per parent row.

Pro What are the three join algorithms, and when does the planner pick each?

Nested loop: scan the inner side once per outer row — chosen when one side is small or the inner side has a usable index. Hash join: build a hash table on one input and probe with the other — chosen for large equi-joins that fit in work_mem with no useful sort order. Merge join: sort both sides on the key and merge in one pass — chosen when inputs already arrive sorted (e.g. via an index). The cost-based planner enumerates these and picks the cheapest; EXPLAIN shows which it used.

Pro A join query was fast in testing but slow in production after the data grew. What happened and how do you fix it?

Typically the planner chose a nested loop that was fine on small data but scans the inner side once per outer row as both tables grow — quadratic work. Diagnose with EXPLAIN (ANALYZE) to compare estimated vs actual rows. Fixes: add an index on the inner join key so the loop uses an index scan (or so a hash/merge becomes cheaper), run ANALYZE to refresh planner statistics, and ensure work_mem is large enough for a hash join to be preferred. Stale statistics and a missing index are the usual culprits.
