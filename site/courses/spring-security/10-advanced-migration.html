What this module covers

Two threads meet here. The first is extending the framework: adding custom servlet filters at the right point in the chain, the awareness-level topics of ACL domain-object security and run-as replacement, and the pitfalls that bite real deployments. The second — and the one interviewers press hardest on right now — is the migration from Spring Security 5 to 6 and 7: the removal of WebSecurityConfigurerAdapter, the mandatory lambda DSL, and authorizeRequests giving way to authorizeHttpRequests.

Topics

Custom filters in the security filter chain

Spring Security is itself a chain of servlet filters, so extending it usually means inserting your own Filter relative to a built-in one. HttpSecurity exposes three placement methods: addFilterBefore, addFilterAfter, and addFilterAt (which sits alongside, not replacing, a given filter). Extend OncePerRequestFilter rather than raw Filter so the logic runs exactly once even across forwards and async dispatches.

Why it matters: multi-tenant resolution, custom token headers, audit logging, and API-key schemes are all implemented as a filter placed just before or after authentication. Getting the position wrong — running before the SecurityContext is populated, or after the endpoint has already executed — is the usual bug.

@Bean
SecurityFilterChain chain(HttpSecurity http) throws Exception {
    http.addFilterBefore(new TenantFilter(),
            UsernamePasswordAuthenticationFilter.class);
    return http.build();
}

Domain object security: ACLs

Request-URL and role checks answer "may this user reach this endpoint?" ACLs (Access Control Lists) answer the finer question "may this user act on this specific object?" — for example, only the owner of document 42 may edit it. Spring Security ships a full ACL module backed by four tables (ACL_SID, ACL_CLASS, ACL_OBJECT_IDENTITY, ACL_ENTRY) and wires into method security through hasPermission(...) in @PreAuthorize/@PostFilter.

Why it matters: it is the framework's answer to per-instance authorization, and interviewers ask about it to test whether a candidate can say where role-based checks stop and object-level checks begin. Most teams treat it as awareness-level — heavyweight to run — and reach for a simpler owner-check in service code, which is itself a defensible answer.

Run-as authentication replacement

Run-as temporarily swaps the current Authentication for one with extra authorities during a secured method call, restoring the original afterwards. RunAsManagerImpl builds a RunAsUserToken whenever a security attribute starts with RUN_AS_, granting a matching ROLE_RUN_AS_* for the duration of the invocation. It exists so a lightly-privileged web caller can invoke a backend that demands stronger roles — classically, calling a remote service.

Why it matters: it is a niche, awareness-level feature, but naming it (and knowing the token is signed with a shared key so it cannot be forged) signals depth beyond the everyday configuration surface.

Common pitfalls

Recurring production mistakes: disabling CSRF because "it was in the way" on a browser session app; ordering rules in authorizeHttpRequests from general to specific so anyRequest() shadows later matchers; registering a filter as a Spring @Bean and then adding it to the chain too, so the servlet container also runs it and it fires twice; and forgetting that a filter placed before authentication has no SecurityContext to read yet. Knowing how to list the actual, resolved filter order is the fastest way to debug all of these.

Migrating Security 5 to 6 and 7: the SecurityFilterChain bean

The single biggest change: WebSecurityConfigurerAdapter was deprecated in 5.7 and removed in 6. Configuration is now component-based — you publish a SecurityFilterChain bean (and, where needed, a WebSecurityCustomizer or UserDetailsService bean) instead of extending and overriding an adapter. The lambda DSL, optional in 5, is the required style from 7 onward: the old .and() chaining is gone.

Why it matters: nearly every legacy Spring app carries an adapter-based config, so "how would you migrate this?" is a top current interview prompt. Show the before-and-after and name the deprecation timeline (5.7 deprecated, 6 removed, 7 lambda-only).

@Configuration
class SecurityConfig extends WebSecurityConfigurerAdapter {
    @Override protected void configure(HttpSecurity http) throws Exception {
        http.authorizeRequests().anyRequest().authenticated()
            .and().formLogin();
    }
}
@Bean
SecurityFilterChain chain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
        .formLogin(Customizer.withDefaults())
        .build();
}

authorizeRequests to authorizeHttpRequests, and the matcher changes

The authorization DSL moved from authorizeRequests (backed by FilterSecurityInterceptor) to authorizeHttpRequests (backed by the newer AuthorizationFilter and the AuthorizationManager API). Two behaviour shifts catch migrators: the new model authorizes servlet dispatcher types too, so an unauthenticated /error forward can suddenly return 401/403; and in 7 the request matchers consolidate onto PathPatternRequestMatcher as antMatchers/mvcMatchers retire in favour of the unified requestMatchers.

http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/public/**").permitAll()
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .anyRequest().authenticated());

Interview questions

The questions Indian interviews lean on for this module — migration dominates, because almost every enterprise codebase is mid-upgrade:

Beginner How do you add a custom filter to the Spring Security chain?

Register it with HttpSecurity.addFilterBefore, addFilterAfter, or addFilterAt, positioning it relative to a built-in filter such as UsernamePasswordAuthenticationFilter. Extend OncePerRequestFilter so it runs a single time per request.

Beginner What replaced WebSecurityConfigurerAdapter in Spring Security 6?

A component-based model: you declare a SecurityFilterChain bean (returning http.build()) instead of extending and overriding the adapter. Web-level tweaks move to a WebSecurityCustomizer bean, and users to a UserDetailsService bean. The adapter was deprecated in 5.7 and removed in 6.

Beginner authorizeRequests vs authorizeHttpRequests — what is the difference?

Same intent, different engine. authorizeRequests used FilterSecurityInterceptor; authorizeHttpRequests uses AuthorizationFilter and the AuthorizationManager API. The former is removed in 7, so the latter is the only supported request-authorization DSL.

Beginner What is the lambda DSL and why does it matter for migration?

Configuring each aspect through a lambda — http.authorizeHttpRequests(auth -> ...) — rather than method chaining joined by .and(). It was optional in 5 and is required from 7; migrating means rewriting chained calls as nested lambdas and dropping .and().

Intermediate Why extend OncePerRequestFilter instead of implementing Filter directly?

OncePerRequestFilter guarantees a single execution per request even when the container re-dispatches (forward, include, async). A plain Filter or GenericFilterBean can run multiple times across dispatches, causing duplicated auth work or double-logging.

Intermediate After migrating to authorizeHttpRequests, requests to /error start returning 401 or 403. Why?

The new model authorizes dispatcher types, so FORWARD and ERROR dispatches (like the internal forward to /error) are now access-checked. Fix by permitting the error dispatch — for example allow the ERROR dispatcher type or the /error path — rather than disabling the new behaviour wholesale.

Intermediate How do you run two SecurityFilterChain beans in one application?

Declare multiple SecurityFilterChain beans, give each a securityMatcher(...) scoping which requests it handles, and order them with @Order — the most specific matcher first. A common split is one chain for a stateless /api and another for the browser-facing UI.

Intermediate What are ACLs in Spring Security, and when would you use them?

Access Control Lists provide per-instance authorization — permissions granted on an individual domain object rather than on a URL or a role. You use them when "who can edit which specific record" cannot be expressed as a role, wiring hasPermission(...) into method security. Many teams judge the ACL module heavyweight and use a simpler owner check instead.

Intermediate What is run-as authentication and give a use case?

It temporarily replaces the current Authentication with a RunAsUserToken carrying extra ROLE_RUN_AS_* authorities for the length of a secured call, then restores the original. Use case: a low-privilege web request must call a backend or remote service that demands stronger roles.

Pro Walk through migrating an adapter-based Security 5 config to 6/7.

Get onto the latest 5.8 patch first, then: replace extends WebSecurityConfigurerAdapter with a @Bean SecurityFilterChain returning http.build(); convert chained calls to the lambda DSL and remove .and(); swap authorizeRequests for authorizeHttpRequests; move configure(WebSecurity) ignores to a WebSecurityCustomizer bean and user setup to a UserDetailsService/AuthenticationManager bean; re-check that /error and other dispatches are still reachable; then move matchers to requestMatchers. Do it incrementally on the latest patch of each line.

Pro Why did request matchers change to PathPatternRequestMatcher in Security 7?

Historically you had to choose between antMatchers, mvcMatchers, and regex matchers, and the wrong choice caused subtle bypasses — for instance an ant matcher not matching the same paths Spring MVC did, letting a request slip past a rule. Consolidating onto PathPatternRequestMatcher (the same path-matching Spring MVC uses) removes that divergence and closes the mismatch as a security hole.

Pro A custom filter registered as a bean fires twice per request. Diagnose it.

A Filter exposed as a Spring @Bean is auto-registered with the servlet container by Boot and added to the security chain, so it runs in both places. Fixes: don't make it a bean (construct it inline when adding to the chain), or if it must be a bean, register a FilterRegistrationBean with setEnabled(false) to suppress the container registration. Extending OncePerRequestFilter also collapses per-dispatch re-runs.
