Module: JWT and Resource Server

A resource server trusts a signed token instead of a session — validate the JWT, turn its claims into authorities, and the API scales without server-side state. Designing that flow is one of the most-asked backend interview topics.

Knowledge Library → Courses → Spring Security → JWT and Resource Server

What this module covers

An OAuth2 Resource Server is a backend that guards its endpoints with bearer tokens issued elsewhere. This module orients you on the whole stateless path: the resource-server role and the Authorization: Bearer flow, the anatomy of a JWT, how JwtDecoder verifies a token against a JWK set, how scopes and custom claims become Spring authorities, and why the session and CSRF machinery is switched off. Depth lives in the linked official docs and standards.

Topics

The resource server role and bearer tokens

Split OAuth2 into two responsibilities: an authorization server authenticates the user and issues a token; a resource server only protects APIs by verifying that token — it never sees a password. In Spring Security, enabling oauth2ResourceServer installs a BearerTokenAuthenticationFilter that pulls the token out of the Authorization: Bearer <token> header; a missing or bad token yields a 401 with a WWW-Authenticate challenge, and a valid one populates the SecurityContext.

Why it matters: this is the standard shape of a modern microservice. Interviewers probe whether you understand that the resource server delegates identity to the issuer and does purely local checking on each call — no login form, no session.

http
  .authorizeHttpRequests(a -> a.anyRequest().authenticated())
  .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()));
// request carries: Authorization: Bearer eyJhbGciOiJSUzI1NiI...

Anatomy of a JWT: header, payload, signature

A JSON Web Token is three Base64URL segments joined by dots: a header naming the signing algorithm, a payload of claims (iss, sub, aud, exp, and app-specific claims like scope), and a signature over the first two. The payload is encoded, not encrypted — anyone can read it — so the security comes entirely from the signature. Because the issuer signs it with a private key, any party holding the matching public key can confirm the token is authentic and untampered without a callback.

Why it matters: candidates who think a JWT is "encrypted" get caught out. The right mental model is a signed, self-describing credential — never put secrets in the payload.

header.payload.signature
{ "alg": "RS256", "typ": "JWT" }                       header
{ "sub": "u-1042", "scope": "orders.read", "exp": 1731000000 }  payload
<RSA signature over base64url(header) + "." + base64url(payload)>

Decoding and validating: JwtDecoder, JWK set, validators

The core component is JwtDecoder (Spring's default is NimbusJwtDecoder). It parses the token, then checks the signature against the issuer's public keys. Point it at the issuer with issuer-uri and Spring auto-discovers the JWK Set endpoint (a URL serving the public keys) via the provider's well-known metadata; or set jwk-set-uri directly. Beyond the signature, validation runs a chain of OAuth2TokenValidator<Jwt>: the timestamp validator rejects expired or not-yet-valid tokens (with a default 60 second clock skew), the issuer validator checks iss, and you should add an audience validator so a token minted for another service is refused.

Why it matters: "how does the API trust the token without asking the auth server every time?" is the crux question — the answer is cached public keys plus local claim validation. Audience checking is the detail seniors are expected to add.

NimbusJwtDecoder decoder =
    NimbusJwtDecoder.withIssuerLocation("https://idp.example.com").build();
decoder.setJwtValidator(new DelegatingOAuth2TokenValidator<>(
    JwtValidators.createDefaultWithIssuer("https://idp.example.com"),
    new AudienceValidator("orders-api")));   // reject tokens for other APIs

Mapping scopes and claims to authorities, and stateless design

Once the token is valid, its claims must become Spring GrantedAuthority objects. By default the JwtGrantedAuthoritiesConverter reads the scope/scp claim and prefixes each value with SCOPE_, so orders.read becomes the authority SCOPE_orders.read — check it with hasAuthority("SCOPE_orders.read"). To use a custom claim (say a roles array) or a ROLE_ prefix, plug a custom converter into a JwtAuthenticationConverter. This is the whole point of stateless auth: because every request carries its own self-contained, signed identity, you set SessionCreationPolicy.STATELESS (no HttpSession) and disable CSRF (no session cookie to forge), letting any instance serve any request.

Why it matters: the trade-off is the interview payoff. Stateless scales horizontally but you cannot instantly revoke a live token — mitigate with short token lifetimes plus refresh tokens, a denylist, or opaque tokens with server-side introspection when immediate revocation is mandatory.

JwtGrantedAuthoritiesConverter scopes = new JwtGrantedAuthoritiesConverter();
scopes.setAuthorityPrefix("SCOPE_");            // the default prefix
JwtAuthenticationConverter conv = new JwtAuthenticationConverter();
conv.setJwtGrantedAuthoritiesConverter(scopes);

http.sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
    .csrf(csrf -> csrf.disable());              // safe: no session cookie

Interview questions

Stateless JWT authentication is a top backend interview topic in Indian product and services companies — from screening rounds to system-design deep dives:

Beginner What is an OAuth2 Resource Server?

A backend application that protects its APIs using OAuth2 access tokens issued by an authorization server. It only verifies tokens on incoming requests — it never authenticates users or issues tokens itself.

Beginner What are the three parts of a JWT?

Header (algorithm and type), payload (the claims), and signature — three Base64URL strings joined by dots. Header and payload are encoded, not encrypted; the signature is what makes the token trustworthy.

Beginner How does a client send a JWT to a resource server?

In the HTTP Authorization header using the Bearer scheme: Authorization: Bearer <token>, per RFC 6750. Spring's BearerTokenAuthenticationFilter extracts it automatically.

Beginner Difference between an authorization server and a resource server?

The authorization server authenticates the user and mints tokens (login, consent, token endpoint). The resource server holds the protected data/APIs and only validates the presented token. They are often separate applications, and one auth server can front many resource servers.

Intermediate How does a resource server validate a JWT without calling the auth server on every request?

It verifies the signature locally using the issuer's public keys, fetched once from the JWK Set endpoint and cached. Nothing else is needed for a signed JWT — validation is offline. This is exactly why JWTs scale, and why revocation is hard.

Intermediate issuer-uri vs jwk-set-uri — what's the difference?

issuer-uri points at the provider; Spring reads its well-known metadata to discover the JWK Set URL and auto-configures an issuer validator. jwk-set-uri supplies the keys endpoint directly (no discovery). Prefer issuer-uri when the provider exposes metadata.

Intermediate How are OAuth2 scopes mapped to Spring authorities by default?

The JwtGrantedAuthoritiesConverter reads the scope (or scp) claim, splits it, and prefixes each value with SCOPE_. So scope orders.read is checked as hasAuthority("SCOPE_orders.read"), not hasRole.

Intermediate How do you map a custom claim (like roles) to authorities?

Register a custom Converter<Jwt, Collection<GrantedAuthority>> inside a JwtAuthenticationConverter and expose it as a bean or wire it via jwt().jwtAuthenticationConverter(...). You can change the claim name, the prefix (e.g. ROLE_), or combine scopes and roles.

Intermediate Beyond the signature, what does Spring validate on a JWT?

A chain of OAuth2TokenValidator: timestamps (exp/nbf with a default 60 second clock skew), the issuer (iss), and — if you add it — the audience (aud). Audience validation stops a token issued for another service from being accepted.

Pro What does "stateless" mean here, and why disable sessions and CSRF?

Each request self-authenticates via its bearer token, so the server keeps no session — set SessionCreationPolicy.STATELESS. With no session cookie, the CSRF attack vector (the browser auto-attaching a cookie) disappears, so CSRF protection is turned off for a pure token API. Any instance can serve any request, enabling horizontal scaling.

Pro Biggest drawback of stateless JWTs, and how do you handle revocation?

A signed JWT stays valid until it expires — you cannot instantly revoke it because nothing is checked server-side. Mitigations: short-lived access tokens (a few minutes) plus longer refresh tokens, a token/JTI denylist checked on sensitive routes, key rotation, or switching to opaque tokens with introspection when immediate revocation is a hard requirement.

Pro JWT vs opaque token on a resource server — the trade-offs?

A JWT is self-contained: validated locally, zero network calls, but larger, readable, and hard to revoke. An opaque token is a random string validated by calling the auth server's introspection endpoint every time (or with caching) — easy to revoke and reveals nothing, at the cost of a dependency and latency on each request. Choose JWT for scale, opaque for tight revocation control.
