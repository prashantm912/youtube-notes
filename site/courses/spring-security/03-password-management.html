Module: Password Management

A stored password is only as safe as the function that hashed it. This module is about choosing that function, tagging the hash so you can change your mind later, and re-encoding quietly on the next login — never storing what the user actually typed.

Knowledge Library → Courses → Spring Security → Password Management

What this module covers

Spring Security never stores or compares raw passwords — it stores the output of a deliberately slow, one-way hash and compares hashes. This module covers the PasswordEncoder abstraction and its adaptive algorithms (BCrypt, Argon2, PBKDF2, scrypt), the DelegatingPasswordEncoder that tags each hash with an {id} prefix so several algorithms can coexist, the storage practices OWASP recommends, and how Spring transparently upgrades an old hash to the current best encoding when a user next signs in.

Topics

PasswordEncoder and the adaptive one-way algorithms

PasswordEncoder is a two-method interface — encode(rawPassword) produces a hash to store, and matches(rawPassword, storedHash) re-hashes the input and compares. The transformation is one-way by design: there is no decode. General-purpose digests like MD5 or SHA-256 are the wrong tool here because they are fast, which is exactly what an attacker with a leaked table wants. Password hashing instead uses adaptive functions — BCrypt, Argon2, PBKDF2, scrypt — that are deliberately expensive and can be made slower as hardware improves.

Why it matters: BCrypt is the everyday default in Spring (cost/strength parameter, built-in per-hash salt), while Argon2id is OWASP's first recommendation for new systems and PBKDF2 is the pick where FIPS compliance is required. Knowing why a slow, salted, tunable function beats a fast digest is the conceptual answer interviewers are really testing.

@Bean
PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // strength (cost) = 12 rounds
}
// encoder.encode("raw") -> a 60-char BCrypt hash, salt embedded

DelegatingPasswordEncoder and the {id} prefix

Since Spring Security 5, the default encoder is a DelegatingPasswordEncoder. It stores each hash prefixed with the algorithm's logical name in braces — {bcrypt}..., {argon2}..., {pbkdf2}..., {noop}... — and reads that prefix at match time to route to the correct delegate. This solves the real-world problem that a password database outlives any single algorithm choice: new passwords are written with today's best encoder while old hashes still verify against whatever wrote them.

Why it matters: this is why a hardcoded {noop}password works in tutorials and why a raw, unprefixed hash throws "There is no PasswordEncoder mapped for the id null". Build the default via PasswordEncoderFactories.createDelegatingPasswordEncoder() rather than newing one algorithm directly — it is one of the most-asked Spring Security design questions.

PasswordEncoder encoder =
    PasswordEncoderFactories.createDelegatingPasswordEncoder();

String stored = encoder.encode("s3cr3t");
// stored -> {bcrypt}$2a$10$... : the {id} names the algorithm
boolean ok = encoder.matches("s3cr3t", stored);

Password storage best practices

Where the hash lives matters as much as how it is computed. Spring's storage mechanisms — in-memory, JDBC, and custom UserDetailsService — all expect an already-encoded, algorithm-tagged password, never plaintext. OWASP's guidance is the baseline every backend engineer should be able to recite: use a memory-hard adaptive function (Argon2id first, then scrypt, then BCrypt, PBKDF2 for FIPS), rely on the built-in per-user salt, tune the work factor to roughly one second per hash, and consider a separately-stored pepper for defense in depth.

Why it matters: practical mistakes sink real systems — sizing a database column below the 60 characters a BCrypt hash needs, logging the raw password, or hand-rolling a salt column when the algorithm already embeds one. These show up as production incidents and as pointed interview follow-ups.

// Store the algorithm-tagged hash, never the plaintext:
UserDetails user = User.withUsername("neha")
    .password("{bcrypt}$2a$10$Dow1o...")  // encoded, not raw
    .roles("USER")
    .build();

Upgrading encoded passwords and migration

Because the delegating encoder tags every hash, Spring can migrate silently. When a user authenticates, DaoAuthenticationProvider validates the raw password against the stored hash and then asks whether that hash is still the current best encoding; if not, and a UserDetailsPasswordService is present, it re-encodes the just-verified password and calls updatePassword to persist the new hash — no reset email, no forced logout. Raising a BCrypt cost factor or switching the default algorithm therefore rolls out gradually, one login at a time.

Why it matters: this is the answer to "how do you migrate an old password database without forcing everyone to reset?" For legacy hashes with no prefix (say plain SHA-256), you first re-tag them with the matching {id} so the delegating encoder can read them, and the automatic upgrade path carries them forward from there.

// DaoAuthenticationProvider invokes this after a successful login
// when the stored hash is no longer the preferred encoding:
public interface UserDetailsPasswordService {
    UserDetails updatePassword(UserDetails user, String newEncodedPassword);
}

Interview questions

Most-asked password questions in Indian Spring interviews — BCrypt and the delegating encoder are near-guaranteed in any backend security round:

Beginner Why hash passwords instead of encrypting them?

Encryption is reversible — anyone with the key recovers the plaintext, so a breach of the key breaks every account. Password hashing is a one-way transformation: you store the hash, and on login you re-hash the input and compare. There is deliberately no way back to the original, so a leaked database does not directly reveal passwords.

Beginner What is PasswordEncoder and what are its main methods?

The interface Spring Security uses for all password handling. encode(raw) returns the hash to store; matches(raw, storedHash) re-hashes the input and returns whether it matches. An optional upgradeEncoding(storedHash) tells the framework whether the hash should be re-encoded with a stronger configuration.

Beginner Why is BCrypt preferred over MD5 or SHA-256 for passwords?

MD5/SHA-256 are general-purpose digests built to be fast — an attacker can try billions of guesses per second against a leak. BCrypt is an adaptive function: it is deliberately slow, has a tunable cost factor you raise as hardware improves, and generates a per-password salt automatically. Speed is a virtue for checksums and a liability for passwords.

Beginner What is a salt, and does BCrypt need a separate salt column?

A salt is random data mixed into each hash so identical passwords produce different hashes, defeating precomputed rainbow tables. No separate column is needed for BCrypt — the salt is generated per hash and embedded inside the 60-character output, so storing the hash stores the salt.

Beginner What is the default PasswordEncoder in current Spring Security?

A DelegatingPasswordEncoder, created via PasswordEncoderFactories.createDelegatingPasswordEncoder(). Its default encoding algorithm is BCrypt, but it can match any prefixed hash. This replaced the older single fixed encoder in Spring Security 5.

Intermediate What is DelegatingPasswordEncoder and what does the {id} prefix do?

It delegates to one of several encoders based on a prefix stored with the hash, e.g. {bcrypt}, {argon2}, {pbkdf2}, {noop}. On encode it writes new passwords with the default algorithm plus its {id}; on match it reads the prefix and routes to the right delegate. This lets multiple algorithms coexist and lets you change the default without breaking existing hashes.

Intermediate What is the BCrypt strength/cost factor and its default?

A logarithmic work factor: cost n means 2^n key-expansion rounds, so each increment roughly doubles the time. Spring's BCryptPasswordEncoder defaults to 10; production systems often use 12, tuned so a single hash takes on the order of a second. Higher is safer but costs CPU on every login.

Intermediate How does Spring Security upgrade an encoded password automatically?

After a successful login, DaoAuthenticationProvider checks upgradeEncoding on the stored hash. If it is stale (weaker algorithm or lower cost) and a UserDetailsPasswordService bean exists, Spring re-encodes the just-verified raw password and calls updatePassword to persist the new hash — transparently, without any user action.

Intermediate BCrypt vs Argon2 vs PBKDF2 vs scrypt — how do you choose?

All are adaptive. Argon2id is OWASP's first choice for new systems (memory-hard, resists GPU/ASIC attacks); scrypt is the memory-hard alternative; BCrypt is the battle-tested default and fine for most apps; PBKDF2 is the choice when FIPS-140 compliance is mandated. Match the answer to constraints — compliance, existing hashes, available CPU/memory.

Intermediate Why does the same password produce a different BCrypt hash each time, yet matches() still succeeds?

Because a fresh random salt is generated on every encode, so the outputs differ. matches does not hash-and-string-compare blindly: it extracts the salt and cost embedded in the stored hash, re-derives the hash of the input with those same parameters, and compares. So verification is deterministic even though encoding is not.

Pro How do you migrate a database of legacy unprefixed hashes (e.g. plain SHA-256) to the delegating format without forcing resets?

The delegating encoder cannot read an unprefixed hash — it fails with "no PasswordEncoder mapped for id null". First re-tag existing rows with the matching {id} (or set a default-for-matches encoder) so they verify. Then supply a UserDetailsPasswordService: as each user logs in, the verified password is re-encoded with the current best algorithm and persisted. The old scheme drains away one login at a time, no reset emails required.

Pro What is a pepper, and how does it differ from a salt?

A salt is unique per password and stored alongside the hash — it defeats rainbow tables. A pepper is a single secret shared across all passwords and stored separately from the database (in a vault, KMS, or app config), so a database-only breach still leaves attackers a missing secret. In Spring you apply a pepper by wrapping or configuring the encoder to combine it with the input before hashing; it complements, never replaces, the per-user salt.

Pro What are BCrypt's known limitations and how do you handle them?

BCrypt only considers the first 72 bytes of input — longer passwords are truncated, so extra characters add no security — and some legacy implementations mishandle embedded null bytes. Mitigations: cap or pre-hash very long inputs (e.g. SHA-256 then BCrypt, applied consistently), or choose Argon2id/scrypt which have no such 72-byte ceiling. Interviewers raise this to see whether you understand an algorithm's edges, not just its API.
