Module: CSRF, CORS and Security Headers

Three defenses that live at the web layer: the CSRF token that blocks forged state-changing requests, the CORS handshake that decides who may call your API, and the response headers that harden every reply — plus the one trap that sinks careless candidates, knowing exactly when it is safe to turn CSRF off.

What this module covers

This module is the browser-facing edge of Spring Security: Cross-Site Request Forgery (CSRF) protection and the synchronizer token behind it, CORS integration for cross-origin front ends, and the suite of secure HTTP response headers (HSTS, CSP, X-Frame-Options and friends) Spring writes for you. All three are configured on the same SecurityFilterChain and all three carry interview traps — above all, when disabling CSRF is correct versus a security hole.

Topics

CSRF protection: how the token works, and when (not) to disable it

A CSRF attack tricks a logged-in user's browser into submitting a state-changing request the user never intended — because the browser automatically attaches the session cookie to every request to your domain, the server sees a fully authenticated call. Spring Security's default defense is the synchronizer token pattern: a random, per-session token the server issues and a same-origin attacker cannot read or guess. The CsrfFilter requires it on every unsafe method (POST, PUT, PATCH, DELETE) and skips safe ones (GET, HEAD, OPTIONS), rejecting a missing or wrong token with 403. It is enabled by default, and the default handler adds BREACH protection by XOR-encoding fresh randomness into each emitted token.

Why it matters: "just disable CSRF to fix the 403" is the single most common wrong answer in interviews. Disabling is correct only when the app does not authenticate via cookies or sessions — a stateless API whose clients send a bearer token in the Authorization header, which the browser does not attach on its own, has nothing for an attacker to ride on. If you use form login, HTTP Basic, or any session cookie, CSRF must stay on. For JavaScript SPAs, keep it on but switch to CookieCsrfTokenRepository (an XSRF-TOKEN cookie the client echoes in an X-XSRF-TOKEN header).

http
  .csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())); // XSRF-TOKEN cookie for SPAs

// Disable ONLY for a stateless, token-authenticated (e.g. JWT) API:
// http.csrf(csrf -> csrf.disable());

CORS integration

The browser's same-origin policy stops a page on one origin from reading responses from another. Cross-Origin Resource Sharing (CORS) is the server's controlled opt-in: through Access-Control-* response headers it tells the browser which cross-origin callers are allowed to read the response. For anything beyond a "simple" request the browser first sends a preflight OPTIONS asking permission. In Spring you declare a CorsConfigurationSource (allowed origins, methods, headers) and switch it on inside the security filter chain with http.cors(...).

Why it matters: the classic gotcha is ordering. Preflight OPTIONS requests carry no cookies or credentials, so if Spring Security's authentication runs before CORS it rejects them as unauthenticated — the browser then reports a misleading "CORS error" that hides the real 401. Enabling CORS inside Spring Security (not just @CrossOrigin on a controller) registers the CorsFilter early enough to answer preflights first. Remember too that CORS is enforced by the browser to protect the user; it is not server-side authorization and does nothing against non-browser clients, and a permissive config is an attack surface, not a fix.

@Bean
CorsConfigurationSource corsConfigurationSource() {
  CorsConfiguration cfg = new CorsConfiguration();
  cfg.setAllowedOrigins(List.of("https://app.example.com")); // explicit; never "*" with credentials
  cfg.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
  UrlBasedCorsConfigurationSource src = new UrlBasedCorsConfigurationSource();
  src.registerCorsConfiguration("/**", cfg);
  return src;
}
// Turn it on inside Spring Security so the CorsFilter runs before authentication:
// http.cors(Customizer.withDefaults());

Security headers (HSTS, CSP, X-Frame-Options and more)

Spring Security writes a set of secure-by-default HTTP response headers on every reply: Cache-Control no-store for authenticated pages, X-Content-Type-Options: nosniff against MIME sniffing, X-Frame-Options: DENY against clickjacking, and Strict-Transport-Security when the request arrives over HTTPS. Two of the strongest — Content-Security-Policy and Referrer-Policy — are opt-in because they need policy decisions from you. All of them are configured under http.headers(...).

Why it matters: these are cheap, defense-in-depth wins that security reviews expect to see. HSTS forces the browser to use HTTPS for a max-age window (with includeSubDomains and preload for the browser preload list), defeating SSL-strip downgrades. CSP is the strongest in-browser mitigation for XSS and, via frame-ancestors, clickjacking — an allowlist of where scripts, styles and frames may come from, though it only supplements (never replaces) proper output encoding. X-Frame-Options (DENY/SAMEORIGIN) is the older clickjacking control that CSP frame-ancestors now supersedes.

http.headers(headers -> headers
  .httpStrictTransportSecurity(hsts -> hsts
    .includeSubDomains(true).maxAgeInSeconds(31536000))            // HSTS: one year
  .frameOptions(frame -> frame.sameOrigin())                       // clickjacking (X-Frame-Options)
  .contentSecurityPolicy(csp -> csp
    .policyDirectives("default-src 'self'; frame-ancestors 'none'")));

Interview questions

Most-asked CSRF, CORS and headers questions in Indian interviews — a staple of product-company and services-company Spring Security rounds alike:

Beginner What is a CSRF attack?

An attacker gets a logged-in user's browser to send a state-changing request the user didn't intend — a hidden form auto-submit, an image tag hitting your endpoint. It works because the browser automatically attaches the session cookie to any request to your domain, so the server treats the forged request as authenticated. The attacker never sees the response; they only need the side effect (transfer money, change email).

Beginner Is CSRF protection on by default in Spring Security, and which requests does it cover?

Yes, enabled by default. It applies to state-changing (unsafe) methods — POST, PUT, PATCH, DELETE — and deliberately exempts the safe, idempotent methods GET, HEAD, OPTIONS and TRACE, which should never change state. A missing or invalid token yields 403.

Beginner What is CORS and what problem does it solve?

The browser's same-origin policy blocks a page from reading responses from a different origin. CORS is the server's opt-in mechanism: via Access-Control-* headers it tells the browser which cross-origin callers are allowed to read the response — for example letting app.example.com call api.example.com.

Beginner What is a CORS preflight request?

For non-simple requests (custom headers, methods like PUT/DELETE, certain content types) the browser first sends an OPTIONS request carrying Access-Control-Request-Method and Access-Control-Request-Headers. The server must approve with matching Access-Control-Allow-* headers before the browser sends the real request.

Intermediate When is it safe to disable CSRF protection, and when is doing so a bug?

The classic trap. Safe only when the app does not authenticate via cookies or sessions — a stateless API whose clients pass a bearer token in the Authorization header, which the browser does not attach automatically, so there is nothing for a forged request to ride on. If you use form login, HTTP Basic, or any session cookie, disabling CSRF opens a real hole. "Disable it to make the 403 go away" is the wrong answer; send the token instead.

Intermediate How does the CSRF token actually work (the synchronizer token pattern)?

The server generates a random token tied to the session and hands it to the client (hidden form field, cookie, or meta tag). The client must echo it on every unsafe request as a parameter (_csrf) or header (X-CSRF-TOKEN/X-XSRF-TOKEN). The CsrfFilter compares submitted vs expected; a mismatch is 403. A cross-origin attacker cannot read or guess the token (same-origin policy), so they cannot forge a valid request.

Intermediate How do you make CSRF protection work with a JavaScript SPA or Angular front end?

Keep CSRF on but switch to CookieCsrfTokenRepository.withHttpOnlyFalse(), which writes an XSRF-TOKEN cookie the JavaScript can read and send back in the X-XSRF-TOKEN header. Angular's HttpClient does this automatically; for others you read the cookie and set the header yourself. From Spring Security 6 you also wire a CsrfTokenRequestAttributeHandler so the token is rendered correctly with the SPA/BREACH handling.

Intermediate Why must CORS be configured before Spring Security, and why do preflights otherwise fail?

Preflight OPTIONS requests carry no cookies or credentials. If Spring Security's authentication runs first it rejects them as unauthenticated (401), which the browser surfaces as a generic "CORS error" that masks the real cause. Enabling CORS inside the filter chain with http.cors(...) registers the CorsFilter early enough to answer the preflight before authentication. A bare @CrossOrigin on the controller runs too late once Security is present.

Intermediate What is the difference between CSRF and CORS? They are often confused.

CSRF is an attack; CORS is a controlled relaxation of the same-origin policy. CSRF protection stops forged state-changing requests from other sites; CORS decides which origins are allowed to read your responses. CORS is not a CSRF defense — in fact a careless "allow all origins" CORS policy widens your exposure rather than protecting you.

Intermediate What does HSTS do, and what are its directives?

Strict-Transport-Security tells the browser to talk to your host over HTTPS only for max-age seconds, upgrading any http:// attempt and refusing to let users click through certificate errors. includeSubDomains extends it to subdomains; preload (with max-age ≥ 31536000 and includeSubDomains) qualifies for the browser preload list. It is only honored when delivered over HTTPS. Spring adds it by default on secure requests.

Intermediate How does X-Frame-Options prevent clickjacking, and what supersedes it?

DENY stops your page being embedded in any frame/iframe; SAMEORIGIN allows only same-origin framing — blocking the invisible-overlay trick where a user thinks they click your button but hit an attacker's frame. Spring sets X-Frame-Options: DENY by default. The modern, more flexible replacement is the CSP frame-ancestors directive, which can list multiple allowed embedders.

Pro What is Content-Security-Policy, which attacks does it mitigate, and what is its main limitation?

CSP is an allowlist, delivered by header, of where scripts, styles, images and frames may load from. It is the strongest in-browser mitigation for XSS (block inline and third-party script) and, through frame-ancestors, clickjacking. Limitations: it is only enforced by the browser, it cannot be your sole XSS defense (proper output encoding still matters), strict nonce/hash policies are hard to retrofit onto existing apps, and a lazy unsafe-inline effectively neuters it.

Pro A team sets a permissive CORS policy (allow all origins) to make errors disappear. What's wrong with that?

Several things. You cannot combine "*" origins with allowCredentials(true) — the spec forbids it, so credentialed calls break or you end up reflecting arbitrary origins, letting any site make authenticated cross-origin reads of a logged-in user's data. Allowlist explicit origins and methods; treat the CORS config as an attack surface. And note CORS protects the browser user, not your server — non-browser clients ignore it entirely, so it is never an authorization mechanism.

Pro A form POST starts returning 403 right after you enable Spring Security. How do you diagnose whether CSRF is the cause, and how do you fix it correctly?

Check whether the failing request is a state-changing method arriving without the CSRF token — no _csrf hidden field, no X-XSRF-TOKEN header — and confirm the client is reading the token from the right place (session-backed default vs CookieCsrfTokenRepository). The fix is to actually send the token (Thymeleaf adds it automatically; SPAs echo the cookie in the header), not to blanket-disable CSRF — disabling is only appropriate for a genuinely stateless, token-authenticated API.
