Module: Testing and Exception Handling

Security config that no test exercises is a guess, and a REST endpoint that answers a missing token with an HTML login page is a bug. This module is about proving your rules hold — and controlling exactly what the framework says when they don't.

Knowledge Library → Courses → Spring Security → Testing and Exception Handling

What this module covers

Spring Security ships a dedicated test module and a set of MockMvc helpers that let you assert who can reach what without standing up real logins. The other half is the ExceptionTranslationFilter — the piece that decides whether a blocked request becomes a 401 challenge or a 403 refusal — and the AuthenticationEntryPoint / AccessDeniedHandler you plug in to shape those responses. Both are interview staples, the 401-vs-403 distinction most of all.

Topics

The spring-security-test module and mock authentication

The spring-security-test artifact adds the test-only support Spring Security needs: annotations that seed a SecurityContext before a test method runs, plus MockMvc and WebTestClient integration. @WithMockUser fabricates a user straight from its attributes (default username user, role USER); @WithUserDetails instead runs your real UserDetailsService so the loaded authorities and account flags match production; @WithAnonymousUser asserts the unauthenticated path; and @WithSecurityContext is the escape hatch for any custom Authentication.

Why it matters: these annotations let you test authorization — the hard, high-value part — without the ceremony of a real login flow. Choosing @WithMockUser versus @WithUserDetails is a judgement call interviewers probe: the first is fast and isolated, the second exercises your actual user-loading and authority mapping.

@Test
@WithMockUser(username = "raj", roles = {"ADMIN"})
void adminReachesDashboard() throws Exception {
    mockMvc.perform(get("/admin/dashboard"))
           .andExpect(status().isOk());
}

MockMvc security: request post-processors and result matchers

For web-layer tests, SecurityMockMvcRequestPostProcessors attaches security state to a request through the fluent .with(...) API: user() and authentication() set the principal, csrf() supplies the token that stateful POST/PUT/DELETE requests require, formLogin() and httpBasic() drive those mechanisms, and jwt(), oauth2Login() and opaqueToken() stand in for a resource server's bearer tokens with no real authorization server. On the assertion side, SecurityMockMvcResultMatchers.authenticated() and unauthenticated() verify the outcome.

Why it matters: the single most common testing surprise is a POST that returns 403 even though the user is mocked — CSRF protection rejects the tokenless request, and adding .with(csrf()) fixes it. For resource-server work, jwt() lets you assert scope-to-authority mapping in complete isolation.

mockMvc.perform(post("/orders")
        .with(csrf())
        .with(jwt().authorities(new SimpleGrantedAuthority("SCOPE_orders.write"))))
       .andExpect(status().isCreated());

How security exceptions become HTTP responses: ExceptionTranslationFilter

Authorization and authentication failures raised deep in the filter chain surface as exceptions, not responses. The ExceptionTranslationFilter sits near the end of the chain and wraps everything after it in a try/catch: an AuthenticationException means the caller is not (properly) authenticated, so it invokes the AuthenticationEntryPoint to start authentication; an AccessDeniedException from an already-authenticated caller means they lack rights, so it invokes the AccessDeniedHandler. An AccessDeniedException from an anonymous caller is treated as "authenticate first" and also routed to the entry point.

Why it matters: the component that decides 401-versus-403 is a filter, not your controller — the decision happens before a request ever reaches a @Controller. This flow also explains why a plain @ControllerAdvice never sees these filter-thrown exceptions.

try {
    filterChain.doFilter(request, response);   // run the rest of the chain
} catch (AuthenticationException ex) {
    authenticationEntryPoint.commence(...);     // no / invalid credentials
} catch (AccessDeniedException ex) {
    accessDeniedHandler.handle(...);            // authenticated but forbidden
}

AuthenticationEntryPoint vs AccessDeniedHandler (401 vs 403)

AuthenticationEntryPoint.commence() answers "we don't know who you are" — for a browser app it redirects to the login page, for an API it should return 401 Unauthorized (often with a WWW-Authenticate header). AccessDeniedHandler.handle() answers "we know who you are, and you may not do this" — a 403 Forbidden. Getting the pairing right is the crux of the most-asked exception question: 401 means not authenticated, 403 means authenticated but not authorized.

Why it matters: the default responses are HTML (the Whitelabel or login page), which is wrong for a JSON API. You register custom handlers via http.exceptionHandling(...); both interfaces write directly to the HttpServletResponse — a JSON body or a ProblemDetail. A frequent production bug is an unauthenticated API call returning 403 instead of 401 because the configured entry point (or anonymous handling) sends the wrong status; for REST, set an entry point that returns 401.

http.exceptionHandling(ex -> ex
    .authenticationEntryPoint((req, res, e) ->
        res.sendError(SC_UNAUTHORIZED))   // 401 — not authenticated
    .accessDeniedHandler((req, res, e) ->
        res.sendError(SC_FORBIDDEN)));      // 403 — authenticated, no rights

Interview questions

The questions Indian interviewers actually ask on this module — the 401-vs-403 pair is close to guaranteed for any backend or REST-API role:

Beginner Which dependency do you add to test Spring Security, and what does it give you?

spring-security-test (test scope). It provides the @WithMockUser family of annotations that seed a SecurityContext before each test, the SecurityMockMvcRequestPostProcessors/ResultMatchers for MockMvc, and equivalent support for WebTestClient.

Beginner What does @WithMockUser do?

It populates the SecurityContext with a mock authenticated user built from its attributes before the test runs — default username user, password password, authority ROLE_USER. That lets you exercise secured endpoints or methods without performing a real login.

Beginner @WithMockUser vs @WithUserDetails — when do you reach for each?

@WithMockUser fabricates a principal from annotation values — fast and self-contained. @WithUserDetails loads a real user through your UserDetailsService, so it exercises your actual user lookup, password/account flags and authority mapping. Use the mock for pure authorization tests, the real load when the loading logic itself matters.

Beginner How do you make MockMvc apply the security filter chain?

Build it with MockMvcBuilders.webAppContextSetup(context).apply(springSecurity()).build(). With @WebMvcTest the security setup is auto-applied, so you can use the post-processors and annotations directly.

Intermediate Your POST test returns 403 even with @WithMockUser. What is wrong?

CSRF protection. State-changing requests (POST/PUT/DELETE) need a valid CSRF token, and MockMvc doesn't add one. Attach .with(csrf()) from SecurityMockMvcRequestPostProcessors. This is the classic MockMvc security trap.

Intermediate How do you test a resource-server endpoint without a real token or JwtDecoder?

Use the jwt() post-processor: .with(jwt().authorities(...).jwt(j -> j.claim("scope", "orders.write"))). It injects an already-authenticated JWT principal, so you test scope/claim-to-authority mapping in isolation. opaqueToken() does the same for introspection-based tokens.

Intermediate What is the ExceptionTranslationFilter's responsibility?

It sits near the end of the filter chain and catches the two security exceptions thrown downstream. An AuthenticationException is routed to the AuthenticationEntryPoint to begin authentication; an AccessDeniedException from an authenticated user is routed to the AccessDeniedHandler. It is what turns those exceptions into HTTP responses.

Intermediate AuthenticationEntryPoint vs AccessDeniedHandler — 401 vs 403?

The entry point handles authentication failures — missing or invalid credentials — and typically yields 401 Unauthorized (or a redirect to login). The access-denied handler handles an authenticated user who lacks the required authority — 403 Forbidden. Shorthand: entry point = "who are you?", access-denied handler = "you can't do this."

Intermediate How do you assert the authenticated user or an unauthorized outcome in a MockMvc test?

SecurityMockMvcResultMatchers: authenticated().withUsername("raj").withRoles("ADMIN") or unauthenticated(), combined with status matchers such as status().isForbidden() or status().isUnauthorized().

Pro An unauthenticated API call returns 403 instead of 401. Why, and how do you fix it?

When no credentials are present the request should reach the entry point and return 401, but a default form-login config redirects (302) and some setups answer 403. For a stateless REST/JWT API, configure a custom AuthenticationEntryPoint that sends 401 (for example a bearer-token entry point) via http.exceptionHandling(), and don't rely on anonymous access masking the distinction.

Pro How do you return a JSON error body instead of the default HTML page for auth failures?

Implement AuthenticationEntryPoint.commence() and/or AccessDeniedHandler.handle() to write JSON (or a ProblemDetail) directly to the response, then register them with http.exceptionHandling(e -> e.authenticationEntryPoint(...).accessDeniedHandler(...)). The default is the Whitelabel/login HTML, which is wrong for an API.

Pro Why doesn't a @ControllerAdvice @ExceptionHandler catch access-denied errors from URL authorization?

URL-level authorization is enforced in the filter chain, so the AccessDeniedException is thrown and handled by the ExceptionTranslationFilter around the DispatcherServlet — never inside a controller dispatch, so @ControllerAdvice can't see it. In contrast, an AccessDeniedException from @PreAuthorize on a controller method is thrown within the dispatch and can be caught by @ExceptionHandler. Knowing which path throws where is the point.

Pro How do you unit-test method security (@PreAuthorize) as opposed to URL rules?

With method security enabled in the test context, seed the context via @WithMockUser/@WithUserDetails and call the secured service method directly in a @SpringBootTest: assert success for an allowed role and that an AccessDeniedException is thrown for a disallowed one. This targets the security interceptor rather than the web layer, so no MockMvc is needed.
