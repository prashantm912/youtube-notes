Module: Authentication

Authentication answers one question — who is making this request — and Spring Security routes that question through an AuthenticationManager, an AuthenticationProvider and a UserDetailsService that interviewers ask you to trace by heart.

Knowledge Library → Courses → Spring Security → Authentication

What this module covers

Authentication is the "prove who you are" half of Spring Security. This module traces the engine that verifies a credential — the AuthenticationManager, its ProviderManager implementation, and the AuthenticationProvider that does the real check — then the UserDetailsService/UserDetails pair that loads the account, and finally the mechanisms (form login, HTTP Basic) that read credentials off the request. All of it is wired by a SecurityFilterChain bean using the current 6.x/7.x lambda DSL. UserDetailsService and a custom AuthenticationProvider are two of the most-asked topics in Indian Spring interviews, so this page weights them heavily.

Topics

The authentication engine: AuthenticationManager, ProviderManager, AuthenticationProvider

An authentication filter builds an unauthenticated Authentication token (username + raw password) and hands it to the AuthenticationManager — the single API that says "verify this". Its standard implementation is ProviderManager, which holds an ordered list of AuthenticationProvider instances and delegates to the first one whose supports() accepts the token type. A provider either returns a fully populated, authenticated token (principal, authorities, credentials erased) or throws an AuthenticationException; returning null means "not my job, try the next provider".

Why it matters: this is the extension seam. For anything beyond username/password — an API key, an OTP, a legacy SSO — you write a custom AuthenticationProvider, and the built-in DaoAuthenticationProvider (which combines a UserDetailsService with a PasswordEncoder) is the concrete example every interviewer wants you to walk through step by step.

public class TokenAuthProvider implements AuthenticationProvider {
    public Authentication authenticate(Authentication a) { /* do the check */ }
    public boolean supports(Class<?> type) {                 // which token can I handle?
        return UsernamePasswordAuthenticationToken.class.isAssignableFrom(type);
    }
}

Loading the user: UserDetailsService and UserDetails

UserDetailsService is the one-method interface that abstracts "where do users live": loadUserByUsername(String) returns a UserDetails or throws UsernameNotFoundException. UserDetails is the account snapshot the framework needs — username, the encoded password, granted authorities, and the enabled/expired/locked flags. Spring ships implementations (in-memory, JDBC, caching); a real application supplies its own that reads from a database.

Why it matters: DaoAuthenticationProvider calls your UserDetailsService, then compares the submitted password against the stored one with the PasswordEncoder — so "connect Spring Security to my users table" almost always means "implement a UserDetailsService", the single most common practical task and interview question on this topic.

@Bean
UserDetailsService users(PasswordEncoder encoder) {
    UserDetails user = User.withUsername("asha")
        .password(encoder.encode("secret"))  // never store raw passwords
        .roles("USER")                        // stored as authority ROLE_USER
        .build();
    return new InMemoryUserDetailsManager(user);
}

Authentication mechanisms: form login and HTTP Basic

A mechanism is how credentials are read off the request. Form login redirects an unauthenticated browser to a login page, then UsernamePasswordAuthenticationFilter reads the posted username/password and starts a session — the default for server-rendered web apps. HTTP Basic sends a WWW-Authenticate challenge and reads credentials from the Authorization header on every request via BasicAuthenticationFilter — stateless and convenient for scripts, tools and simple APIs.

Why it matters: choosing the mechanism is a design decision interviewers probe. Basic is trivial but resends credentials each call (only safe over TLS); form login gives a real UX and session but needs CSRF handling. Both funnel into the same AuthenticationManager — only the filter that reads the credential differs.

Wiring it together: the SecurityFilterChain bean and the lambda DSL

In Spring Security 6 and 7 there is no WebSecurityConfigurerAdapter. You configure security by declaring a SecurityFilterChain bean: take the injected HttpSecurity, configure it with the lambda DSL, and return http.build(). Each configurer (authorizeHttpRequests, formLogin, httpBasic) takes a Customizer lambda instead of the old .and() chaining, so the shape of the config reads top to bottom.

Why it matters: this is the exact code every modern Spring Boot app writes, and migrating from the removed adapter is a guaranteed question. Expose a UserDetailsService and a PasswordEncoder bean and Spring Boot auto-builds a DaoAuthenticationProvider behind an AuthenticationManager for you; declare the SecurityFilterChain to control which requests need authentication and which mechanisms are active.

@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
        .formLogin(Customizer.withDefaults())
        .httpBasic(Customizer.withDefaults());
    return http.build();
}

Interview questions

The authentication questions Indian interviewers lean on — UserDetailsService and a custom AuthenticationProvider come up in almost every Spring backend round:

Beginner What is authentication in Spring Security, and which component performs it?

Authentication verifies identity — "who are you". The AuthenticationManager is the API that performs it: a filter passes it an unauthenticated Authentication token, and it returns an authenticated one (or throws). The result is stored in the SecurityContextHolder.

Beginner What is the AuthenticationManager and its most common implementation?

An interface with one method, authenticate(Authentication). The common implementation is ProviderManager, which delegates to a list of AuthenticationProvider instances rather than authenticating itself.

Beginner What is UserDetailsService and what method must you implement?

The abstraction for loading a user by name. It has one method: loadUserByUsername(String), returning a UserDetails or throwing UsernameNotFoundException. It does not check the password — it only loads the account.

Beginner UserDetails vs UserDetailsService — what is the difference?

UserDetails is the data object — username, encoded password, authorities, and the enabled/locked/expired flags. UserDetailsService is the service that fetches a UserDetails by username. One is the noun, the other is the loader.

Beginner How do you enable HTTP Basic and form login in Spring Security 6/7?

Inside a SecurityFilterChain bean, call http.httpBasic(Customizer.withDefaults()) and http.formLogin(Customizer.withDefaults()) on the injected HttpSecurity, then return http.build().

Intermediate Walk through how DaoAuthenticationProvider authenticates a username and password.

It receives the UsernamePasswordAuthenticationToken from the ProviderManager, calls the UserDetailsService to load the UserDetails, uses the PasswordEncoder to match the submitted password against the stored encoded one, and on success returns a new authenticated token holding the UserDetails as principal and its authorities. Failure throws BadCredentialsException (or the account-status equivalent).

Intermediate What is an AuthenticationProvider and when would you write a custom one?

It performs one specific style of authentication and declares which token types it supports(). You write a custom one when the credential is not plain username/password — API keys, one-time passwords, LDAP, or an external identity system — by implementing authenticate() and supports() and registering it as a bean or provider.

Intermediate Form login vs HTTP Basic — how do they differ and when do you pick each?

Form login redirects to an HTML page, reads a posted username/password, and typically creates a session — good for browser apps. HTTP Basic sends a WWW-Authenticate challenge and reads credentials from the Authorization header on every request — stateless, good for scripts and simple APIs, but the credential travels each call so TLS is mandatory.

Intermediate How do you plug a custom UserDetailsService into the config in Spring Security 6/7?

Expose it as a @Bean alongside a PasswordEncoder bean. Spring Boot then auto-configures a DaoAuthenticationProvider that uses them. If you build the provider yourself, call setUserDetailsService(...) and register the provider — there is no WebSecurityConfigurerAdapter.configure(AuthenticationManagerBuilder) anymore.

Intermediate What does the Authentication object hold, and how does it change during authentication?

It holds the principal, credentials, and authorities, plus an authenticated flag. Before authentication it carries the raw username/password with authenticated=false; after success the principal becomes the UserDetails, authorities are populated, credentials are usually erased, and authenticated=true.

Pro How does ProviderManager select a provider, and what are its parent and credential-erasure behaviours?

It iterates its providers, skipping any whose supports() rejects the token type, and uses the first that returns a non-null result; if none succeed it can delegate to an optional parent AuthenticationManager. On success it clears the credentials from the returned token (eraseCredentialsAfterAuthentication, on by default) so raw secrets do not linger in the SecurityContext.

Pro In Spring Security 6/7, how do you define an AuthenticationManager explicitly?

Either obtain the framework-built one — inject AuthenticationConfiguration and return authenticationConfiguration.getAuthenticationManager() as a bean — or construct your own ProviderManager from a list of providers (for example a configured DaoAuthenticationProvider) and expose that as the AuthenticationManager bean. You need an explicit one when a custom filter has to call authenticate() itself.

Pro Why must UserDetails never expose a raw password, and how does credential erasure help?

The stored password must be a hash (BCrypt/Argon2 etc.), never plaintext, so a leaked user table cannot be replayed; the PasswordEncoder matches the submitted secret against that hash. After authentication, credential erasure (via ProviderManager, and UserDetails implementing CredentialsContainer.eraseCredentials()) nulls the password so it is not held in memory on the authenticated principal.
