Module: Authorization

Authentication asks who you are; authorization decides what you may do. The gap between hasRole and hasAuthority, or a @PreAuthorize silently skipped on a self-call, is exactly where Spring interviews probe hardest.

Knowledge Library → Courses → Spring Security → Authorization

What this module covers

Authorization runs after authentication has established the current Authentication: Spring then guards two surfaces — HTTP requests via authorizeHttpRequests, and Java methods via method security — both routed through a single AuthorizationManager since Spring Security 6. This module orients you on the request-authorization DSL and its matchers, the roles-versus-authorities distinction that trips up most candidates, the AuthorizationManager model that replaced the old voter chain, and the @PreAuthorize/@PostAuthorize family. All examples target current 6.x/7.x with the lambda DSL and @EnableMethodSecurity.

Topics

Request-level authorization: authorizeHttpRequests and request matchers

The authorizeHttpRequests DSL attaches an authorization rule to each incoming request. You pair a request matcher (a URL pattern, optionally an HTTP method) with a rule such as permitAll(), authenticated(), hasRole(...), hasAuthority(...) or denyAll(). Rules are evaluated top-to-bottom and the first match wins, so specific matchers go first and anyRequest() goes last as the catch-all.

Why it matters: this is the everyday surface of Spring Security and a guaranteed interview area. In Security 6 the old authorizeRequests(), antMatchers()/mvcMatchers() and WebSecurityConfigurerAdapter are gone — requestMatchers() replaces them and auto-selects MVC or Ant matching. Rule ordering bugs (a broad pattern shadowing a specific one) are a classic "why is this endpoint open?" incident.

@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/public/**").permitAll()
        .requestMatchers(HttpMethod.POST, "/api/orders").hasRole("USER")
        .requestMatchers("/admin/**").hasRole("ADMIN")
        .anyRequest().authenticated());   // catch-all, must be last
    return http.build();
}

The authorization architecture: AuthorizationManager

Underneath both request and method authorization sits one strategy interface, AuthorizationManager, which takes a Supplier<Authentication> and the object being secured and returns a grant/deny decision. Every DSL rule (hasRole, authenticated, and so on) is really a pre-built AuthorizationManager — for example AuthorityAuthorizationManager and AuthenticatedAuthorizationManager — and .access(...) lets you plug in your own for logic the shortcut rules cannot express.

Why it matters: since Spring Security 6, AuthorizationManager supersedes the older AccessDecisionManager plus AccessDecisionVoter chain (which in Security 7 is moved out to a separate spring-security-access module). Interviewers who have upgraded a project love asking what changed and why the new model is simpler.

http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/reports/**")
        .access(AuthorityAuthorizationManager.hasRole("ANALYST"))
    .anyRequest().authenticated());

Roles vs authorities and the ROLE_ prefix

A GrantedAuthority is a single string permission held by the principal. A role is nothing more than an authority conventionally prefixed with ROLE_. That convention is the whole trick: hasRole("ADMIN") is a shortcut that checks for the authority ROLE_ADMIN, while hasAuthority("...") matches the string exactly with no prefix added. So hasRole("ADMIN") and hasAuthority("ROLE_ADMIN") are equivalent — and mixing the two conventions is where access silently breaks.

Why it matters: this is the single most-asked authorization question in Indian interviews. The practical guidance — roles for coarse, business-facing buckets; authorities for fine-grained permissions, often with roles acting as containers of privileges — signals real design experience beyond reciting the API.

// hasRole("ADMIN") is a shortcut for hasAuthority("ROLE_ADMIN")
.requestMatchers("/admin/**").hasRole("ADMIN")            // needs authority ROLE_ADMIN
.requestMatchers("/reports/**").hasAuthority("REPORT_READ") // exact string, no prefix

Method security: @EnableMethodSecurity and @PreAuthorize / @PostAuthorize / @Secured

Method security moves authorization off the URL and onto the service layer, so the same rule protects a method however it is reached. Annotate a @Configuration class with @EnableMethodSecurity and the @PreAuthorize/@PostAuthorize/@PreFilter/@PostFilter interceptors switch on by default. @PreAuthorize evaluates a SpEL expression before the call; @PostAuthorize evaluates after and can inspect returnObject; a failing check throws AccessDeniedException. The older @Secured (plain authority list, no SpEL) and JSR-250 @RolesAllowed are opt-in via securedEnabled and jsr250Enabled.

Why it matters: @PreAuthorize is a top interview topic, and the gotcha behind it — method security is Spring AOP proxy-based, so an internal self-invocation bypasses the check entirely — separates people who have shipped it from people who have only read about it. Note that @EnableMethodSecurity replaced the deprecated @EnableGlobalMethodSecurity.

@Configuration
@EnableMethodSecurity          // @PreAuthorize/@PostAuthorize active by default
class MethodSecurityConfig { }

@PreAuthorize("hasRole('ADMIN')")
void deleteUser(long id) { ... }

@PostAuthorize("returnObject.owner == authentication.name")
Document findDocument(long id) { ... }   // owner-only: checked on the result

Interview questions

The authorization questions Indian interviews return to again and again — roles vs authorities and @PreAuthorize are near-certain in any Spring Security round:

Beginner What is the difference between authentication and authorization?

Authentication establishes who the caller is (identity + credentials, producing an Authentication); authorization decides what that identity is allowed to do. Authorization always runs after authentication and reads the granted authorities to make its decision.

Beginner How do you secure URLs in Spring Security 6?

Define a SecurityFilterChain bean and configure http.authorizeHttpRequests(auth -> auth.requestMatchers(...)...). The old authorizeRequests(), antMatchers()/mvcMatchers() and WebSecurityConfigurerAdapter were removed in 6; requestMatchers() replaces the matcher methods and auto-picks MVC or Ant matching.

Beginner What does hasRole("ADMIN") actually check for?

The presence of the GrantedAuthority named ROLE_ADMIN. hasRole is a shortcut that prepends the default ROLE_ prefix, so hasRole("ADMIN") equals hasAuthority("ROLE_ADMIN").

Beginner How do you turn on method-level security?

Annotate a @Configuration class with @EnableMethodSecurity (which replaced the deprecated @EnableGlobalMethodSecurity). That activates @PreAuthorize/@PostAuthorize/@PreFilter/@PostFilter by default; add securedEnabled=true or jsr250Enabled=true for @Secured or @RolesAllowed.

Beginner What happens when an authorization check fails?

An AccessDeniedException is thrown. The ExceptionTranslationFilter catches it: an already-authenticated user gets the AccessDeniedHandler (HTTP 403); an anonymous user is sent to the AuthenticationEntryPoint (for example, redirected to the login page) so they can authenticate first.

Intermediate hasRole vs hasAuthority — when do you use each?

Both check for a GrantedAuthority; hasRole(x) auto-adds the ROLE_ prefix, hasAuthority(x) matches the exact string. Use roles for coarse, business-facing buckets (ADMIN, MANAGER) and authorities for fine-grained permissions (REPORT_READ). The framework treats them identically — the difference is convention. The bug to avoid is granting ROLE_ADMIN but guarding with hasAuthority("ADMIN").

Intermediate In what order are request matchers evaluated, and what is the classic bug?

Top to bottom, and the first matcher that matches wins — later rules for the same request never run. The classic bug is placing a broad pattern (or anyRequest()) above a more specific one, so the specific rule is shadowed and its endpoint ends up more open or more closed than intended. Order specific-first, anyRequest() last.

Intermediate @PreAuthorize vs @PostAuthorize — and when is @PostAuthorize the right choice?

@PreAuthorize runs before the method, so it cannot see the return value; @PostAuthorize runs after and its SpEL can reference returnObject. Use @PostAuthorize for decisions that depend on the fetched object — the classic owner-only check returnObject.owner == authentication.name. Caveat: the method still executes, so don't use it on side-effecting operations.

Intermediate @Secured vs @PreAuthorize vs @RolesAllowed?

@PreAuthorize is SpEL-based and the most expressive (on by default with @EnableMethodSecurity). @Secured takes a plain list of authority strings, no SpEL, and is opt-in via securedEnabled=true. @RolesAllowed is the JSR-250 standard equivalent, opt-in via jsr250Enabled=true. New code generally standardises on @PreAuthorize.

Intermediate What is AuthorizationManager and what did it replace?

A single strategy interface — check/authorize(Supplier<Authentication>, T) — used by request, method and message authorization since Security 6. It supersedes the older AccessDecisionManager + AccessDecisionVoter voting chain (which in Security 7 moves to the separate spring-security-access module). Every DSL rule is a pre-built AuthorizationManager, and .access(...) plugs in a custom one.

Pro A @PreAuthorize method called from another method in the same class is not enforced. Why, and how do you fix it?

Method security is applied by a Spring AOP proxy that wraps the bean; a self-invocation (this.method()) calls the target directly and never passes through the proxy, so the interceptor is skipped. Fixes: call through an injected reference to the proxied bean, move the secured method into a separate bean, or use AspectJ compile/load-time weaving instead of proxies.

Pro How do you implement owner-only or attribute-based access — a user may edit only their own record?

Several options: @PostAuthorize("returnObject.owner == authentication.name") on a read; hasPermission(...) in SpEL backed by a custom PermissionEvaluator; or a custom AuthorizationManager registered with .access(...) for request-level checks. For full domain-object security across many objects, Spring Security ACLs are the heavyweight option.

Pro What do @PreFilter and @PostFilter do, and what is the performance trap?

@PreFilter trims a collection/array argument before the method runs; @PostFilter trims the returned collection element by element, each tested against a SpEL predicate via filterObject — handy for returning only the entries a caller may see. The trap: post-filtering a large result set in memory wastes the work of fetching rows the user can't see; push the predicate into the query when the data volume is real.
