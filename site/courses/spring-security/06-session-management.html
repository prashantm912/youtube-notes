Module: Session Management

Between two requests, a logged-in user is just a session — and that seam is where a surprising share of security bugs live. This module maps Spring Security's session controls, from fixation defence to going fully stateless, and points you at the reference for each.

Knowledge Library → Courses → Spring Security → Session Management

What this module covers

Once a user authenticates, Spring Security has to decide whether — and how — to remember them. This module covers the five levers: session fixation protection (on by default), the SessionCreationPolicy that ranges from ALWAYS to STATELESS, concurrent session control that caps simultaneous logins, logout handling, and remember-me for persistent login. Two of them, fixation protection and the stateless policy, are among the most-asked Spring Security interview topics — so they get extra attention here.

Topics

Session fixation protection

In a session fixation attack, the attacker plants a known session id in a victim's browser before login and rides that same session once the victim authenticates. Spring Security defeats this by changing the session id at the moment of authentication, so any pre-login id an attacker knows is discarded. This is on by default — since Servlet 3.1 the strategy is changeSessionId — with migrateSession, newSession, and none as the alternatives.

Why it matters: it is a real, classic web attack that the framework neutralises for free, and interviewers love it precisely because the protection is invisible until you learn to name it. Knowing that a new session id is issued on login — and what each strategy does with the old session's attributes — is the depth they are checking for.

http.sessionManagement(session -> session
    .sessionFixation(fixation -> fixation.changeSessionId())); // default since Servlet 3.1
// alternatives: migrateSession (copy attributes), newSession (fresh, drop attributes), none

Session creation policy: stateless and beyond

SessionCreationPolicy tells Spring Security how eager to be about creating an HttpSession. IF_REQUIRED (the default) makes one only when needed; ALWAYS forces one eagerly; NEVER won't create one but will use an existing session; and STATELESS neither creates a session nor uses one to hold the SecurityContext — every request must carry its own credentials.

Why it matters: STATELESS is the backbone of REST/JWT security and one of the single most-asked Spring Security interview questions. The subtlety to internalise is the NEVER vs STATELESS distinction (using an existing session vs ignoring it entirely) and the fact that going stateless means the SecurityContext is not persisted between requests — so token authentication has to re-establish it every time.

http.sessionManagement(session -> session
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS));
// no HttpSession is created or read for the SecurityContext — pair with token/Basic auth

Concurrent session control

Concurrent session control caps how many active sessions one principal may hold at once — useful for enforcing "one login per account" or licensing rules. maximumSessions(1) sets the cap; maxSessionsPreventsLogin(true) rejects the new login when the cap is reached, while the default (false) expires the oldest session instead. It relies on a SessionRegistry, and that registry only stays accurate if an HttpSessionEventPublisher bean forwards the container's session-destroyed events to Spring.

Why it matters: it is a common product requirement and a favourite follow-up to "how does Spring track sessions?". The gotcha almost everyone hits is forgetting the HttpSessionEventPublisher, after which the count never decrements and legitimate logins start getting blocked.

http.sessionManagement(session -> session
    .maximumSessions(1)
    .maxSessionsPreventsLogin(true));      // reject the new login instead of expiring the old

@Bean HttpSessionEventPublisher httpSessionEventPublisher() {
    return new HttpSessionEventPublisher();  // keeps the SessionRegistry in sync
}

Logout handling

By default a POST to /logout runs a chain of LogoutHandlers: it invalidates the HttpSession, clears the SecurityContext, deletes the remember-me and CSRF tokens, and then a LogoutSuccessHandler redirects to /login?logout. The logout() DSL lets you change the URL, add handlers, delete extra cookies, or return a status code instead of a redirect (the REST-friendly path). Logout is a POST on purpose — a GET could be triggered by a malicious link or image tag, so CSRF protection guards it.

http.logout(logout -> logout
    .logoutUrl("/logout")
    .logoutSuccessUrl("/login?logout")
    .deleteCookies("JSESSIONID")
    .invalidateHttpSession(true));

Remember-me authentication

Remember-me keeps a user logged in across browser restarts, after the HttpSession itself has expired, by issuing a separate remember-me cookie. Spring Security ships two implementations. The hash-based TokenBasedRememberMeServices signs a cookie of username:expiry:password:key and holds no server state — simple, but it can't be revoked and a password change invalidates it. The PersistentTokenBasedRememberMeServices stores a series/token pair in a persistent_logins table, which enables revocation and token-theft detection at the cost of a database.

Why it matters: the hash-vs-persistent trade-off is the standard interview follow-up, and the security implications (a stolen remember-me cookie is a durable credential) make the choice a real design decision, not a checkbox.

http.rememberMe(remember -> remember
    .key("uniqueAndSecretKey")
    .tokenValiditySeconds(1_209_600));     // 14 days; add a token repository for the persistent variant

Interview questions

Most-asked session-management questions in Indian Spring interviews — session fixation and the stateless policy come up in almost every product-company round:

Beginner What is a session fixation attack, and how does Spring Security prevent it?

The attacker plants a known session id in the victim's browser before login, then reuses that session after the victim authenticates. Spring Security prevents it by changing the session id on successful authentication — default strategy changeSessionId (Servlet 3.1+) — so the attacker's pre-login id is worthless. It is enabled out of the box.

Beginner What are the four SessionCreationPolicy values?

ALWAYS (eagerly create a session), IF_REQUIRED (the default — create only when needed), NEVER (don't create one but use an existing session), and STATELESS (never create and never use a session for the SecurityContext).

Beginner What does the default logout do?

On a POST to /logout it invalidates the HttpSession, clears the SecurityContext, removes the remember-me and CSRF tokens, and redirects to /login?logout. The individual steps are LogoutHandlers; the final redirect is the LogoutSuccessHandler.

Beginner What is remember-me authentication?

A persistent-login feature: alongside the session, Spring Security sets a remember-me cookie so the user is re-authenticated automatically across browser restarts, even after the HttpSession has expired — until the remember-me token itself expires.

Intermediate What does SessionCreationPolicy.STATELESS actually do, and when do you use it?

Spring Security neither creates an HttpSession nor reads one to obtain the SecurityContext — so the context isn't remembered between requests and every request must present its own credentials. It's the standard choice for REST APIs secured with JWT or HTTP Basic, where each call is independently authenticated. Note it only stops Spring Security from using sessions; you must ensure nothing else creates one.

Intermediate NEVER vs STATELESS — what's the difference?

NEVER tells Spring Security not to create a session, but if one already exists (created elsewhere) it will still use it to store/read the SecurityContext. STATELESS won't create one and won't use an existing one for the context. A common trap: with NEVER you can still end up session-backed.

Intermediate How do you limit a user to one active session, and what happens at the cap?

maximumSessions(1). With maxSessionsPreventsLogin(true) the new login is rejected; with the default false the oldest session is expired so the new login wins. It requires a SessionRegistry, kept accurate by registering an HttpSessionEventPublisher bean — forgetting it is the classic bug (counts never decrement).

Intermediate What are the two remember-me implementations and how do they differ?

TokenBasedRememberMeServices uses a signed hash cookie (username, expiry, password, key) with no server state — simple, but not revocable and broken by a password change. PersistentTokenBasedRememberMeServices stores a series/token in a persistent_logins table, enabling revocation and token-theft detection, at the cost of a database.

Intermediate Why is logout a POST by default, and how do you make it REST-friendly?

A GET logout could be fired by a malicious link or <img> tag, so logout is a CSRF-protected POST. For an API, use the logout() DSL with a logoutSuccessHandler such as HttpStatusReturningLogoutSuccessHandler to return a status code instead of redirecting.

Intermediate What is the SessionRegistry and why does concurrency control need HttpSessionEventPublisher?

The SessionRegistry maps each principal to its live sessions — the source of truth for concurrent-session limits and for administratively expiring sessions. HttpSessionEventPublisher bridges the servlet container's session-created/destroyed events into Spring's ApplicationContext, so the registry learns when a session ends; without it the active count drifts.

Pro How does concurrent session control expire the extra session, and what does that user see?

Expiry is detected at request time, not pushed. When a new login pushes a principal over the cap (with prevent-login false), the strategy marks the oldest SessionInformation as expired. On that session's next request, the ConcurrentSessionFilter sees the expired flag, logs the user out and redirects to the configured expired-URL. So the evicted user stays "in" until their next request.

Pro In a stateless JWT API, is session-fixation protection still relevant? What replaces it as a concern?

No — with STATELESS there is no server session to fixate, so fixation protection and concurrent session control simply don't apply. The concerns shift to the token: theft and replay, short expiry with refresh-token rotation, signature/audience validation, and making sure no filter (e.g. the request cache) quietly creates a session behind your back. CSRF handling also changes because there's no session cookie to abuse.

Pro A load-balanced app logs users out on failover. How do session policy and session storage explain it, and what are the fixes?

With the default policy the SecurityContext lives in an in-memory HttpSession on one node; when the load balancer routes to another node after failover, that session is gone and the user must re-authenticate. Fixes: sticky sessions (fragile), externalise sessions with Spring Session backed by Redis/JDBC so any node can read them, or go STATELESS with JWTs so there is no server session at all. Each trades operational cost against scalability.
