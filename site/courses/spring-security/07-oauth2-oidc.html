Module: OAuth2 and OpenID Connect

Delegated access without handing over your password is OAuth 2.0; OpenID Connect adds the missing answer to "who is this user". Spring Security ships a client for both and — separately — a server that issues the tokens, and interviewers love the boundary between them.

Knowledge Library → Courses → Spring Security → OAuth2 and OpenID Connect

What this module covers

This module orients you on the protocol before the code: the four OAuth 2.0 roles, the grant types (with Authorization Code + PKCE as the modern default), and how OpenID Connect layers authentication on top with an ID token. Then it maps those ideas onto Spring Security's building blocks — oauth2Login(), ClientRegistration, the OAuth2 client for calling downstream APIs — and closes with awareness of Spring Authorization Server, the project you reach for when your application needs to issue tokens rather than consume them.

Topics

OAuth 2.0 and OpenID Connect: the model and the difference

OAuth 2.0 is a delegated authorization framework: a resource owner (the user) lets a client (your app) obtain a scoped, revocable access token from an authorization server so it can call a resource server's APIs — no password sharing. OpenID Connect (OIDC) is a thin authentication layer on top: add the openid scope and the provider also returns a signed ID token (a JWT of identity claims) plus a UserInfo endpoint and discovery metadata.

Why it matters: "OAuth2 vs OIDC" is one of the most-asked security questions in Indian interviews, and the wrong answer — treating a raw access token as proof of identity — is a real vulnerability. OAuth2 answers "what may this app do"; OIDC answers "who is signed in". Know which token is meant for the API and which is meant for your app.

Grant types and authorization flows

A grant type is the recipe by which a client obtains tokens. Authorization Code is the default for user-facing apps: the browser is redirected to the provider, the app gets back a short-lived code, then swaps it server-to-server for tokens so tokens never sit in the URL. PKCE hardens that swap for clients that cannot keep a secret (SPAs, mobile) and is recommended for all clients in OAuth 2.1. Client Credentials is machine-to-machine with no user; Refresh Token renews an expired access token silently. The old Implicit and Password grants are discouraged and removed in OAuth 2.1.

Why it matters: interviewers routinely ask you to name the grants and pick the right one for a scenario (browser app vs. cron job vs. service call). Getting "Authorization Code + PKCE for anything with a browser, Client Credentials for service-to-service" right signals you understand the model, not just a library.

oauth2Login(): signing users in with an external provider

oauth2Login() wires the Authorization Code login flow so users authenticate through Google, GitHub or any OIDC provider ("Login with…"). Each provider is described by a ClientRegistration (client id/secret, grant type, scopes, endpoints), held in a ClientRegistrationRepository (in-memory by default). Spring Boot builds these from spring.security.oauth2.client.registration.* properties, and CommonOAuth2Provider pre-fills the well-known endpoints for popular providers so you supply only credentials and scopes.

spring:
  security:
    oauth2:
      client:
        registration:
          google:                         # CommonOAuth2Provider fills the endpoints
            client-id: ${GOOGLE_CLIENT_ID}
            client-secret: ${GOOGLE_CLIENT_SECRET}
            scope: openid, profile, email  # openid -> OIDC, returns an ID token
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .oauth2Login(Customizer.withDefaults());   // redirect-based login
    return http.build();
}

The OAuth2 client: calling protected resources

Beyond logging users in, Spring Security can make your app an OAuth2 client that calls other protected resource servers. Here you often use grants without a login — typically Client Credentials for service-to-service calls. The OAuth2AuthorizedClientManager obtains, stores and refreshes tokens on your behalf (delegating to per-grant OAuth2AuthorizedClientProviders), and integrations for RestClient/WebClient attach the bearer token to outbound requests automatically.

spring:
  security:
    oauth2:
      client:
        registration:
          inventory:                              # app-to-API, no user present
            authorization-grant-type: client_credentials
            client-id: ${INV_CLIENT_ID}
            client-secret: ${INV_CLIENT_SECRET}
            scope: inventory.read

Spring Authorization Server: issuing the tokens

Everything above makes your app a consumer of tokens. When you instead need to issue them — to run your own identity provider — that is Spring Authorization Server, a framework (not a hosted product) implementing OAuth 2.1 and OpenID Connect on top of Spring Security. It exposes the authorize, token, UserInfo, JWKS and discovery endpoints and manages registered clients, issuing JWT or opaque tokens. You reach for it when you want full control and no per-user licensing; you reach for Keycloak/Okta/Auth0 when you'd rather buy user management and an admin console.

<!-- Boot starter that brings up the authorization server -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-oauth2-authorization-server</artifactId>
</dependency>

Interview questions

OAuth2 and OIDC are among the most-probed backend security topics in Indian product-company interviews — grant types and the OAuth2-vs-OIDC distinction come up in almost every round:

Beginner What problem does OAuth 2.0 actually solve?

Delegated authorization: it lets a user grant a third-party app limited, revocable access to their resources on another service without sharing their password. The app receives a scoped access token instead of credentials, and the user can revoke it at any time.

Beginner What are the four roles in OAuth 2.0?

Resource owner (the user), client (the app wanting access), authorization server (authenticates the user, gets consent, issues tokens), and resource server (hosts the protected APIs and validates access tokens). In "Login with Google", you are the resource owner, your app is the client, and Google is the authorization server.

Beginner OAuth 2.0 vs OpenID Connect — the difference?

OAuth 2.0 is authorization (what an app may do); OIDC is an authentication layer on top of it (who the user is). OIDC adds the openid scope, a signed ID token (JWT) with identity claims, a UserInfo endpoint, and discovery. Use OAuth2 to access APIs, OIDC to log a user in. Treating an access token as proof of identity is the classic mistake.

Beginner Access token vs ID token vs refresh token?

Access token — presented to the resource server to call APIs (opaque or JWT, short-lived). ID token — an OIDC JWT about the authentication event, meant for the client and never sent to APIs. Refresh token — a longer-lived credential used to obtain new access tokens without re-prompting the user.

Intermediate Walk through the Authorization Code grant.

The client redirects the browser to the authorization server's authorize endpoint with client_id, scope, redirect_uri and state. The user authenticates and consents; the server redirects back with a short-lived authorization code. The client then exchanges that code (plus its secret) at the token endpoint, server-to-server, for the tokens. The code-for-token swap keeps tokens out of the browser URL and history.

Intermediate What is PKCE and why is it now the default?

Proof Key for Code Exchange: the client sends a hashed code_challenge on the authorize request and the matching code_verifier on the token request, so a stolen authorization code is useless to an attacker. It was created for public clients (SPAs, mobile) that cannot keep a secret; OAuth 2.1 recommends it for all clients, and Spring Security enables it automatically for public clients.

Intermediate When do you use the Client Credentials grant?

Machine-to-machine, with no user present — a backend service calling another service under its own identity (a scheduled job, a microservice call). The client authenticates with its id/secret and receives an access token directly; there is no authorization code, redirect, ID token, or refresh token.

Intermediate In Spring Security, what does oauth2Login() set up, and what are ClientRegistration and ClientRegistrationRepository?

oauth2Login() wires the Authorization Code login flow so users sign in via an external provider. A ClientRegistration holds one provider's details (client id/secret, grant type, scopes, endpoints); the ClientRegistrationRepository stores them (InMemory by default). Spring Boot builds these from spring.security.oauth2.client.registration.* properties, and CommonOAuth2Provider pre-fills Google/GitHub/Facebook/Okta endpoints.

Intermediate Difference between oauth2Login() and oauth2Client() in Spring Security?

oauth2Login() authenticates the end user into your app (produces an Authentication, usually OIDC). The OAuth2 client support (oauth2Client()) is about your app acting as a client to call other protected resource servers — obtaining and refreshing access tokens via OAuth2AuthorizedClientManager — without necessarily logging anyone in.

Intermediate How does OIDC discovery cut configuration?

Given only the issuer URI, the client fetches the provider's /.well-known/openid-configuration document, which lists the authorize, token, UserInfo and JWKS endpoints plus supported scopes. In Spring you set provider.<id>.issuer-uri and Spring Security resolves the rest at startup instead of you hard-coding every endpoint.

Pro Why is the Implicit grant discouraged and dropped in OAuth 2.1?

It returned the access token directly in the redirect URL fragment — exposed in browser history, referer headers and logs, with no client authentication and no PKCE protection. Modern guidance replaces it with Authorization Code + PKCE for browser-based apps; OAuth 2.1 removes both the Implicit and the Resource Owner Password grants entirely.

Pro What does OAuth2AuthorizedClientManager do, and how does token refresh happen?

It coordinates obtaining, storing and refreshing authorized clients on the app's behalf, delegating to OAuth2AuthorizedClientProviders per grant type. When an access token has expired and a refresh token exists, the refresh_token provider silently obtains a new one — which is what lets a WebClient/RestClient keep calling downstream APIs without manual token juggling.

Pro When would you stand up Spring Authorization Server instead of using Keycloak, Auth0 or Okta?

Spring Authorization Server is a framework (not a product) implementing OAuth 2.1 and OIDC as an identity provider you embed and fully control — right when you need deep customization, no per-user licensing, or to keep token issuance inside your own Spring stack. A hosted product like Keycloak/Okta/Auth0 wins when you'd rather get user management, an admin UI, social federation and support out of the box than own the authorization server yourself.
