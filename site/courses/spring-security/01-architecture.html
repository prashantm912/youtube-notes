Module: Architecture and the Filter Chain

Spring Security is, at heart, one servlet filter that fans out into an ordered chain of smaller ones — and being able to draw that chain on a whiteboard is the single most reliable way to pass a Spring Security interview.

Knowledge Library → Courses → Spring Security → Architecture and the Filter Chain

What this module covers

Before a single credential is checked, Spring Security establishes an architecture: it hooks into the standard servlet filter chain through a DelegatingFilterProxy, delegates to a FilterChainProxy, and runs an ordered list of security filters described by a SecurityFilterChain bean. Alongside sits the SecurityContextHolder, which holds the current Authentication for the duration of a request. This module maps that machinery on both the servlet and reactive (WebFlux) stacks — the mental model every later module builds on.

Topics

DelegatingFilterProxy, FilterChainProxy, SecurityFilterChain and the servlet filter chain

A Servlet container owns its own filter chain, but its filters are container-managed and cannot be Spring beans. Spring Security bridges that gap with DelegatingFilterProxy: a container filter (registered by Boot under the name springSecurityFilterChain) that simply forwards each request to a Spring-managed bean. That bean is FilterChainProxy, the true entry point. It inspects the request, selects the first matching SecurityFilterChain, and invokes that chain's ordered security filters — authentication, CSRF, header writing, authorization and the rest.

Why it matters: a SecurityFilterChain is just a RequestMatcher plus a list of filters, and you now declare it as a @Bean using the lambda DSL — the WebSecurityConfigurerAdapter base class was removed in Spring Security 6. An app can publish several ordered chains (public API vs admin console), and only the first one whose matcher accepts a request runs. Knowing that "declaring a bean replaces the default chain" and that matching is first-match-wins explains most real configuration behaviour.

@Bean
SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/public/**").permitAll()
            .anyRequest().authenticated())
        .formLogin(withDefaults())
        .httpBasic(withDefaults());
    return http.build();   // returns a SecurityFilterChain
}

SecurityContext, SecurityContextHolder, Authentication and the servlet vs reactive stack

Once a filter authenticates a request, the result lives in the SecurityContext, reachable through the SecurityContextHolder. The context wraps a single Authentication, whose three parts are the principal (who — often a UserDetails), the credentials (usually cleared after login), and the authorities (a collection of GrantedAuthority — roles and scopes). On the servlet stack the holder's default strategy is a ThreadLocal, so any code on the request thread can call SecurityContextHolder.getContext().getAuthentication() without passing anything around. In Spring Security 6 the context is loaded per request by SecurityContextHolderFilter and saved explicitly through a SecurityContextRepository, replacing the older auto-saving SecurityContextPersistenceFilter.

Why it matters: the reactive (WebFlux) stack cannot use a ThreadLocal, because one request hops across many threads. There the analog of SecurityFilterChain is SecurityWebFilterChain (built from ServerHttpSecurity, enabled by @EnableWebFluxSecurity), and the context is read from the Reactor subscriber context via ReactiveSecurityContextHolder. Interviewers love to probe whether you understand that the same concepts wear different types on the two stacks.

// Servlet stack: current user from the thread-bound context
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();
boolean isAdmin = auth.getAuthorities().stream()
        .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));

Interview questions

The filter chain is a favourite opener in Indian backend and product-company interviews — it separates people who configured Spring Security by copying snippets from those who understand it:

Beginner At a high level, how does Spring Security plug into a web request?

It registers a single servlet filter that runs before your controllers. That filter delegates to an ordered chain of security filters, each with one job (CSRF, authentication, authorization, etc.). A request that survives the whole chain reaches your application code; one that fails is rejected or redirected inside the chain.

Beginner What is DelegatingFilterProxy and why is it needed?

Servlet filters are managed by the container, not by Spring, so they cannot be injected with beans. DelegatingFilterProxy is a thin container filter that simply forwards each request to a Spring-managed filter bean (here, FilterChainProxy). It bridges the servlet lifecycle and the Spring application context.

Beginner What does the SecurityContextHolder hold, and how do you get the current user?

It holds the SecurityContext, which contains the current Authentication. Retrieve it with SecurityContextHolder.getContext().getAuthentication(); from there getName(), getPrincipal() and getAuthorities() give you the user, principal and roles. In a controller, @AuthenticationPrincipal or a Principal parameter is cleaner.

Beginner What are the three key parts of an Authentication object?

principal (the identity — often a UserDetails), credentials (the proof, such as a password, usually erased after successful login), and authorities (a collection of GrantedAuthority representing roles and scopes). An isAuthenticated() flag rounds it out.

Intermediate How do FilterChainProxy and SecurityFilterChain relate?

FilterChainProxy is the one bean DelegatingFilterProxy delegates to. It owns a list of SecurityFilterChain instances; for each request it walks that list, picks the first chain whose RequestMatcher matches, and runs only that chain's filters. A SecurityFilterChain is therefore just "a matcher plus an ordered list of filters".

Intermediate With multiple SecurityFilterChain beans, how does Spring pick one for a request?

First match wins. FilterChainProxy tries the chains in order (control the order with @Order or bean ordering) and uses the first whose matcher accepts the request — the others are skipped entirely. So a specific chain such as securityMatcher("/api/**") must be ordered before a catch-all anyRequest chain, or it will never run.

Intermediate Walk through what happens to a request as it passes the security filters.

Roughly: the context filter loads any existing SecurityContext, header/CSRF filters run, an authentication filter (e.g. UsernamePasswordAuthenticationFilter) may authenticate and populate the context, then near the end AuthorizationFilter checks access. If authentication or authorization fails, ExceptionTranslationFilter catches the exception and triggers the entry point or access-denied handler. Order matters: authentication filters run before authorization.

Intermediate What is the default SecurityContextHolder storage strategy, and when would you change it?

Default is MODE_THREADLOCAL — the context is bound to the current thread. Switch to MODE_INHERITABLETHREADLOCAL when child threads spawned from the request thread must see the same user, or MODE_GLOBAL for a single context shared across the whole app (rare, mostly desktop/standalone). The default is filled and cleared per request so contexts never leak between requests on a pooled thread.

Intermediate In Spring Security 6, how is the SecurityContext loaded and saved per request?

SecurityContextHolderFilter loads the context (from the SecurityContextRepository, e.g. the HTTP session) at the start and clears the holder at the end, but it does not auto-save. Saving is now explicit — filters call the repository when the context changes (the "require explicit save" model) — which replaced the older SecurityContextPersistenceFilter that saved on every request.

Pro How does the architecture differ on the reactive (WebFlux) stack?

Same ideas, different types. DelegatingFilterProxy/FilterChainProxy become a WebFilter; SecurityFilterChain becomes SecurityWebFilterChain, built from ServerHttpSecurity and enabled by @EnableWebFluxSecurity. Crucially the context is not stored in a ThreadLocal (a reactive request crosses many threads) but in the Reactor subscriber context, read via ReactiveSecurityContextHolder and returning a Mono.

Pro Why can't you rely on SecurityContextHolder in an @Async method or reactive pipeline, and how do you fix it?

Because the default context lives in a ThreadLocal bound to the request thread; work handed to another thread (or a reactive operator) starts without it, so the user appears absent. On the servlet side propagate it with DelegatingSecurityContextExecutor/...Runnable or the inheritable-threadlocal strategy; on the reactive side use ReactiveSecurityContextHolder, which reads from the Reactor context that flows with the pipeline.

Pro How do you add a custom filter at the right place in the chain, and what goes wrong if you get it wrong?

Use http.addFilterBefore(...), addFilterAfter(...) or addFilterAt(...) relative to a known filter (for example a JWT filter before UsernamePasswordAuthenticationFilter). Get the position wrong and you either run before the context/authentication is set (so no user is available) or after AuthorizationFilter (so your work happens too late). Also extend OncePerRequestFilter to avoid double execution on forwards/async dispatches.
