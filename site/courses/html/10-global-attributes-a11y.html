Module: Global Attributes and Accessibility

These attributes and roles apply to every element on the page — and getting accessibility right has quietly become a hiring filter rather than a finishing touch.

Knowledge Library → Courses → HTML → Global Attributes and Accessibility

What this module covers

Global attributes are the ones valid on any element — identity (id, class), presentation (style, title), internationalization (lang, dir), and interaction control (hidden, inert, contenteditable). On top of them sit custom data-* attributes, microdata, shadow-DOM hooks like part, and the ARIA roles, states and properties that expose your markup to assistive technology. This module closes the HTML track by orienting you on each, and by treating accessibility as a first-class concern rather than an afterthought.

Topics

Core global attributes: id, class, style, title, lang, dir, hidden, inert

Every element accepts this common set. id is a document-unique handle (fragment links, <label for>, scripting); class is a reusable, space-separated hook for CSS and JavaScript. lang and dir declare language and text direction so screen readers pronounce correctly and bidirectional text lays out right. title gives advisory tooltip text, and style carries inline CSS (best avoided in favor of stylesheets).

Why it matters: the two most interview-worthy members here are the state attributes. hidden removes an element visually and from the accessibility tree, while inert keeps an element visible but blocks focus, clicks and assistive-tech interaction for it and its whole subtree — the correct way to freeze the background behind a modal. Knowing hidden-vs-inert-vs-display:none is a common screen.

<article id="post-42" class="card featured" lang="en" dir="ltr">
  <p hidden>Draft — not rendered, not announced.</p>
</article>
<div inert>…background UI, still visible but unfocusable…</div>

Editing and input hints: contenteditable, draggable, spellcheck, translate, autocapitalize

This family tells the browser how the user may interact with content. contenteditable turns any element into an editable region (true, false, or plaintext-only to strip rich formatting); draggable marks an element as a drag source that drives the Drag and Drop API. spellcheck, translate and autocapitalize are hints — the browser is free to honor or ignore them — that improve text-entry ergonomics.

Why it matters: these are hints with real gotchas. spellcheck="true" can send content to third-party services, so set it false on sensitive fields; contenteditable produces unpredictable DOM and needs an explicit role and label to be accessible. Interviewers probe whether candidates treat these as advisory, not as guarantees.

<div contenteditable="plaintext-only">Type here — no rich formatting.</div>
<li draggable="true" spellcheck="false">Reorder me</li>

Custom data and component hooks: data-*, microdata, part/exportparts, nonce

data-* attributes let you attach arbitrary private data to an element and read it back through the dataset property (kebab-case in HTML becomes camelCase in script). Microdata (itemscope, itemprop, itemtype) layers machine-readable meaning onto content for search engines using vocabularies like Schema.org. For web components, part and exportparts expose named pieces of a shadow tree so outside CSS can style them via ::part(), and nonce pairs an inline script or style with a Content-Security-Policy token.

Why it matters: data-* is the sanctioned way to stash state without inventing non-standard attributes — but it is not for large payloads, sensitive values, or content that should be visible or exposed to assistive technology. Knowing what not to put in a data attribute is as tested as the syntax.

<button data-user-id="42" data-role="admin">Edit</button>
<script>/* btn.dataset.userId === "42"; data-user-id → userId */</script>

ARIA: roles, states, and properties

ARIA (Accessible Rich Internet Applications) is a set of roles (what a thing is — role="dialog", role="tab"), states (dynamic conditions — aria-expanded, aria-checked), and properties (relationships and labels — aria-labelledby, aria-describedby). They patch the accessibility tree for interactive widgets that plain HTML cannot express, so screen readers announce a correct name, role and value.

Why it matters: the governing principle — the "first rule of ARIA" — is to not use ARIA when a native element already carries the semantics and behavior you need. A real <button> gives focusability, Enter/Space activation and a role for free; a <div role="button"> forces you to re-add all of it and is easy to get wrong. Bad ARIA is worse than none, and interviewers ask this precisely to hear that answer.

<button aria-label="Close" aria-expanded="false">×</button>
<div role="alert">Your session will expire soon.</div>

Accessibility fundamentals

Beyond ARIA attributes sits the wider discipline: semantic structure, keyboard operability, sufficient color contrast, meaningful text alternatives, focus management, and testing with real assistive technology. WCAG (Web Content Accessibility Guidelines) is the standard these resources build on, organized around content being perceivable, operable, understandable and robust.

Why it matters: accessibility is increasingly a legal and hiring requirement, and it overlaps heavily with everything else in this track — landmarks, labelled forms, alt text, and heading order. Product companies in India now routinely include an accessibility question in front-end rounds, so a working mental model of "how does a screen-reader user experience this page?" pays off.

Deprecated elements: know them, do not use them

A handful of old presentational elements — center, font, marquee, big, strike, tt, frame/frameset and more — still render in browsers for backwards compatibility but are removed from the standard. They mixed styling into markup, the exact thing CSS exists to separate out.

Why it matters: you will meet these in legacy code and occasionally in a trick interview question. The expected answer is recognition plus the modern replacement — CSS for font/center, CSS animation for marquee, semantic <strong>/<del> for <b>/<strike> where meaning matters — never "use it anyway".

<center><font color="red">Old</font></center>   <!-- deprecated -->
<p class="notice">New</p>                           <!-- CSS does the styling -->

Interview questions

Global-attribute and accessibility questions that recur in Indian front-end interviews, from screening rounds to product-company deep dives:

Beginner What is a global attribute, and can you name a few?

An attribute valid on any HTML element regardless of its type. Common ones: id, class, style, title, lang, dir, hidden, tabindex, and the whole data-* family. Event-handler attributes (onclick…) and ARIA attributes are global too.

Beginner Difference between id and class?

id must be unique in the document and identifies one element — used by fragment links, <label for>, and getElementById. class is reusable: many elements can share it and one element can carry several, which is why styling and JS hooks lean on classes.

Beginner What does the hidden attribute do, and how does it differ from CSS display:none?

hidden removes the element from rendering and from the accessibility tree — semantically "not currently relevant". By default it behaves like display:none, but it is a content-layer state a stylesheet can override, whereas display:none is purely presentational. (hidden="until-found" even reveals content when found via find-in-page.)

Beginner Why set the lang attribute, and where?

Put lang on <html> to declare the page language; override on sub-elements for quoted foreign text. It drives screen-reader pronunciation, correct hyphenation, translation offers, spell-checking, and :lang() styling. dir similarly declares left-to-right vs right-to-left text.

Beginner How do you attach custom data to an element and read it in JavaScript?

Use data-* attributes, e.g. data-user-id="42", and read them through the dataset property: el.dataset.userId. HTML kebab-case maps to JS camelCase. You can also target them in CSS with attribute selectors like [data-role="admin"].

Intermediate inert vs disabled vs hidden — when would you reach for inert?

disabled applies only to form controls and blocks their interaction/submission. hidden removes an element entirely (visual + a11y). inert keeps a subtree visible but non-interactive and out of the accessibility tree — the correct tool for freezing the background page while a modal dialog is open, so focus and screen readers cannot wander behind it.

Intermediate What is the "first rule of ARIA"?

If a native HTML element or attribute already provides the semantics and behavior you need, use it instead of re-creating it with an ARIA role. Native elements give focus, keyboard handling and correct roles for free; hand-rolled ARIA widgets are easy to break, and incorrect ARIA is worse for users than no ARIA at all.

Intermediate What is the difference between a <button> and a <div role="button">?

The native <button> is focusable, activates on Enter and Space, submits forms, and exposes a button role automatically. The div gives you none of that: you must add tabindex="0", keydown handlers for Enter/Space, and role="button" — and you will still miss edge cases. Prefer the native element.

Intermediate aria-label vs aria-labelledby vs aria-describedby?

All feed the accessibility tree. aria-label supplies an accessible name as a literal string. aria-labelledby sets the name by referencing the id(s) of visible text. aria-describedby adds a longer supplementary description (hints, error text) on top of the name. Labelledby/describedby take id references; label takes a string.

Intermediate What should you NOT store in data-* attributes?

Large payloads, secrets or security-sensitive values (they sit in the DOM in plain text), and anything that is actual visible content or needs to be in the accessibility tree — data attributes are ignored by assistive technology. They are for small bits of state that scripts and CSS consume, not a data store or a content substitute.

Pro What is the accessibility tree, and how do ARIA roles and states affect it?

The browser derives an accessibility tree from the DOM, computing each node's name, role, value and states, which assistive technologies then read. ARIA overrides the semantics exposed on that tree — role, name, states like aria-expanded — but it changes nothing about actual behavior (focus, keyboard, events). That gap is why ARIA-only widgets so often feel broken: the tree says "button" but nothing makes it act like one.

Pro What is the nonce attribute and how does it relate to Content-Security-Policy?

nonce carries a one-time cryptographic token that must match the nonce-… source in the page's Content-Security-Policy header, allowing a specific inline <script> or <style> to run while everything else inline is blocked. It must be unguessable and freshly generated per request; the DOM property is deliberately hidden after load to reduce the chance of it leaking to an attacker.
