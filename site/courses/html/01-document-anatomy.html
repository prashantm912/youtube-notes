Module: Document Anatomy and Metadata

Before a single word is visible, the browser has already read your doctype, character encoding, and head. This module orients you on everything above the fold of the source — and points you to the official references that define each piece.

Knowledge Library → Courses → HTML → Document Anatomy

What this module covers

Every HTML document has a fixed shape: a doctype, an <html> root split into <head> and <body>, and a head full of metadata that the user never sees but browsers, search engines and social platforms depend on. This module walks the skeleton and each metadata element — meta, title, base, link, script, style, noscript — as orientation, then hands depth to the linked docs. Get this right and the rest of HTML sits on a solid foundation.

Topics

doctype, html/head/body structure

The <!doctype html> preamble is not a tag and carries no content — its only job is to keep the browser in standards mode instead of the legacy "quirks mode" that emulates 1990s bugs. Below it, <html> is the single root element, and it holds exactly two children in order: <head> (metadata for the machine) and <body> (content for the human). The lang attribute on <html> tells screen readers and translators which language to assume.

Why it matters: this skeleton is the one structure every page shares, and "what does the doctype do?" is a guaranteed screening question. Missing or malformed, it silently changes how the box model, tables and legacy CSS render.

<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <title>My page</title>
  </head>
  <body>
    <!-- visible content -->
  </body>
</html>

meta (charset, viewport, description, robots, og)

The <meta> element is the head's workhorse: one void element expressing many different pieces of document metadata depending on its attributes. <meta charset="utf-8"> declares the character encoding and must appear within the first 1024 bytes of the document. name="viewport" hands the browser responsive-layout hints, name="description" feeds the search snippet, and name="robots" controls indexing. Social previews use the Open Graph vocabulary, which uses property (not name) — og:title, og:image and friends.

Why it matters: encoding, viewport and description are the difference between a page that renders as intended and one that shows garbled text on a zoomed-out mobile screen with no useful search listing. Interviewers probe charset placement and the viewport line constantly.

<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<meta name="description" content="One-sentence summary of the page.">
<meta name="robots" content="index, follow">
<meta property="og:title" content="Shareable title">

title, base

The <title> is the one required element inside every head. It names the browser tab, the bookmark, the search result heading and what a screen reader announces on load — so it is a real UX and SEO surface, not a formality, and it is distinct from the page's visible <h1>. The <base> element sets a single default URL and/or default target for every relative link and resource in the document.

Why it matters: a good title is concise and unique per page. <base> is powerful but sharp-edged — only the first one counts, it affects in-page fragment links too, and it silently breaks relative URLs when misplaced, which makes it a favourite "gotcha" question.

<title>Checkout — Acme Store</title>
<base href="https://acme.example/app/" target="_blank">

link (stylesheets, preload/prefetch, icons, canonical)

The <link> element expresses a relationship between the current document and an external resource; the rel attribute is what gives it meaning. rel="stylesheet" pulls in CSS, rel="icon" sets the favicon, and rel="canonical" tells search engines which URL is authoritative when the same content is reachable several ways. The resource-hint family — preload (fetch a critical asset for this page now), prefetch (fetch something likely needed next), preconnect, dns-prefetch — lets you steer the browser's loading priorities.

Why it matters: <link> is where performance and SEO live in the head. Knowing the difference between preload and prefetch, and why a canonical tag matters, marks the line between junior and mid-level front-end answers.

<link rel="stylesheet" href="/site.css">
<link rel="icon" href="/favicon.ico" sizes="any">
<link rel="canonical" href="https://acme.example/products/42">
<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>

script (defer, async, type=module, nomodule)

A classic <script> is render-blocking: the parser stops, downloads, and runs it before continuing. The two loading attributes fix that. defer downloads in parallel and runs after parsing, in document order — the sensible default for app code that touches the DOM. async downloads in parallel and runs the instant it arrives, order not guaranteed — right for independent scripts like analytics. type="module" opts into ES modules (with import/export, strict mode, and its own scope) and is deferred by default; the paired nomodule attribute lets you ship a fallback bundle that module-aware browsers skip.

Why it matters: "async vs defer" is one of the most-asked front-end interview questions in India, and misplacing a blocking script is a common real-world performance bug. Understanding module scripts is now table stakes.

<script src="app.js" defer></script>         <!-- after parse, in order -->
<script src="analytics.js" async></script>    <!-- whenever it arrives -->
<script type="module" src="main.js"></script> <!-- deferred by default -->

style, noscript

The <style> element embeds CSS directly in the document rather than linking it. In production most styles live in an external stylesheet for caching, but inlined <style> has a real niche: the "critical CSS" needed to paint the first screen without waiting on a network round-trip. The <noscript> element defines fallback content shown only when scripting is unavailable or disabled — a message, a plain-HTML alternative, or a stylesheet swap.

Why it matters: knowing when inline <style> is justified (and when it is a maintenance smell) is a judgement question, and <noscript> is the honest answer to "what does a user with JavaScript off see?" — an accessibility and progressive-enhancement concern.

<style>.hero { color: rebeccapurple; }</style>
<noscript>This app needs JavaScript. Please enable it to continue.</noscript>

Interview questions

The document-anatomy questions that open most front-end interviews in India — from service-company screening rounds to product-company deep dives:

Beginner What is the doctype and what happens if you omit it?

<!doctype html> is a preamble (not a tag) that puts the browser in standards mode. Omit it and the browser falls back to quirks mode, emulating old non-standard behaviour — the box model, table sizing and some legacy CSS then render differently and inconsistently.

Beginner What are the two children of the html element, and what goes in each?

<head> and <body>, in that order. The head holds metadata for machines — title, meta, link, script, style — that is not directly rendered; the body holds the visible content. The <html> element itself is the single document root.

Beginner Why do we set the charset, and why must it come first?

<meta charset="utf-8"> tells the browser how to decode the bytes into characters; without it you get mojibake (garbled accents and symbols). It must appear within the first 1024 bytes of the document so the browser knows the encoding before it has parsed much text — hence it is the first thing in the head.

Beginner What does the viewport meta tag do?

<meta name="viewport" content="width=device-width, initial-scale=1"> tells mobile browsers to use the device width as the layout width and not zoom out. Without it, phones assume a ~980px desktop canvas and shrink the page, breaking responsive design.

Beginner Difference between the title element and an h1?

<title> lives in the head; it labels the browser tab, bookmarks and the search-result heading, and is read out when the page loads. <h1> is visible on-page content and the top of the heading outline. They often differ — the title may add site/brand context the on-page heading does not.

Intermediate async vs defer — explain the difference and when to use each.

Both download in parallel without blocking parsing. defer runs scripts after the HTML is parsed, in document order — use it for app code that depends on the DOM or on other scripts. async runs each script as soon as it finishes downloading, order not guaranteed — use it for independent scripts like analytics. Default to defer.

Intermediate How does type="module" change a script, and what is nomodule for?

A module script runs in strict mode, has its own top-level scope (no leaking globals), supports import/export, is fetched with CORS, and is deferred by default. nomodule marks a fallback classic script that module-supporting browsers ignore — the old pattern for shipping a legacy bundle only to older browsers.

Intermediate preload vs prefetch — what is the difference?

rel="preload" is a high-priority hint to fetch a resource needed for the current page early (a font, hero image, critical script) — it needs an as value. rel="prefetch" is a low-priority hint to fetch something likely needed on a future navigation, stored for later. Preload is "need it now"; prefetch is "might need it next".

Intermediate What is a canonical link and which problem does it solve?

<link rel="canonical" href="..."> names the authoritative URL when the same content is reachable via several URLs (tracking params, http/https, trailing slashes, print views). It consolidates ranking signals onto one URL and prevents search engines treating the variants as duplicate content.

Intermediate When is an inline style element the right choice over an external stylesheet?

Mostly the external <link rel="stylesheet"> wins because it is cached across pages. The justified case for inline <style> is critical CSS — the minimal styles to paint the first screen inlined in the head so the initial render does not wait on a separate CSS request. Beyond that, inline styles hurt caching and maintainability.

Pro What is the noscript element for, and what are its limits?

<noscript> shows fallback content when scripting is unavailable or disabled — a message or a plain-HTML alternative — and in the head it may swap in a stylesheet. Limits: it only covers the "JS disabled" case, not "JS loaded but failed", and it is no substitute for real progressive enhancement where the core experience works without JavaScript in the first place.

Pro What are the gotchas of the base element?

Only the first <base> in the document is honoured, and it must precede any element with a URL to affect it. It rewrites all relative URLs, including in-page fragment links like #section (which then resolve against the base's URL, not the current page) — a classic source of broken anchors. A stray <base target="_blank"> silently makes every link open in a new tab.

Pro A render-blocking script sits in the head with no attributes. Walk through the performance impact and the fixes.

The parser halts at the tag, downloads the script over the network, executes it, and only then resumes building the DOM — so first paint is delayed by the full fetch-plus-run time, worst case on a slow connection. Fixes, best first: add defer (or use type="module") so parsing continues and the script runs after; use async if the script is truly independent; move non-critical work out of the critical path; and preload genuinely critical resources. Historically people moved scripts to the end of <body>, but defer is the cleaner modern answer.
