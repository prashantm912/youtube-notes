Module: Forms: Inputs

A form is how a page collects data from a human and hands it to a server. This module walks the container element and every control that goes inside it, then points you to the official reference for each — depth lives in the links.

Knowledge Library → Courses → HTML → Forms: Inputs

What this module covers

Forms are the primary way the web gathers input: logins, checkouts, search bars, filters. This module covers the <form> element and how it submits, then the full range of <input> controls — the text family, numeric and range sliders, native date/time and color pickers, the checkbox/radio/file/hidden group, and the buttons that trigger submission. The next module (Forms: Controls and Validation) adds <select>, <textarea> and constraint validation on top of this foundation.

Topics

The form element: action, method, enctype, and labels

A <form> groups named controls and submits them together as one request. action is the URL that receives the data; method chooses how it travels — GET appends the fields to the URL as a query string (bookmarkable, cacheable, size-limited, wrong for secrets), POST carries them in the request body. enctype sets how that body is encoded, and it matters exactly once: file uploads require multipart/form-data.

Every control needs an accessible name, and the honest way to give it one is a <label> — either wrapping the control or linked by for pointing at the control's id. This is not decoration: clicking the label focuses (or toggles) the control, enlarges the tap target on mobile, and gives screen readers something to announce. A control's name attribute, meanwhile, becomes the key in the submitted data — a control without a name is simply not sent.

<form action="/subscribe" method="post">
  <label for="email">Email address</label>
  <input id="email" name="email" type="email" required>
  <button>Subscribe</button>
</form>

Text-type inputs: text, password, email, url, tel, search, number, range

Most controls are the same element — <input> — differing only by type. The text family is one line of editable text where the type is a hint that changes behavior: password masks characters, email and url add free format validation, search gets a clear button, and on mobile each type summons the right on-screen keyboard. number renders a spinner and accepts min, max and step; range renders the same numeric constraints as a slider for approximate values where the exact figure does not matter.

A common trap: type="number" is for genuine quantities only. For phone numbers, OTP codes, PINs and postal codes it misbehaves — it strips leading zeros and non-digit formatting and shows spinner arrows nobody wants. Reach for type="tel" (plus inputmode and a pattern) instead. Knowing when not to use number is a frequent interview probe.

<input type="password" name="pw" minlength="8">
<input type="email" name="email" placeholder="you@example.com">
<input type="tel" name="phone" inputmode="numeric" pattern="[0-9]{10}">
<input type="range" name="volume" min="0" max="100" step="5">

Date, time, and color inputs

The date/time family — date, time, datetime-local, month, week — gives the browser's native picker so users never type a malformed date. The crucial detail for interviews and back-end work: the submitted value is always a fixed machine format regardless of how the picker is displayed in the user's locale. A date submits as yyyy-mm-dd; a month as yyyy-mm. You can bound them with min, max and step just like numbers.

type="color" opens the platform's color swatch picker and returns a lowercase seven-character hex value such as #1a2b3c. Rendering of all these controls varies by browser and platform — the picker UI is the browser's, not yours — so treat the visible widget as a convenience and rely on the stable submitted value.

<input type="date"  name="dob" max="2026-07-19">
<input type="month" name="expiry">
<input type="color" name="brand" value="#d2521f">

Choice and special inputs: checkbox, radio, file, hidden

A checkbox is an independent on/off toggle: it is submitted only when checked, and its submitted value defaults to the string on unless you set value. A radio is one option in a mutually exclusive set — controls sharing the same name form the group, the browser enforces a single selection, and the chosen option's value is what gets sent. Give each group a real set of values; relying on the default is a bug.

type="file" is the file picker: it needs the form's enctype set to multipart/form-data, and accept filters the offered file types while multiple allows several at once. type="hidden" carries data the user does not see but that still submits — CSRF tokens, record ids, wizard step state. Because a hidden field sits in plain view in the page source and the DOM, never treat its value as secret or trusted; the server must always re-check.

<input type="checkbox" name="terms" value="accepted">
<input type="radio" name="plan" value="pro">
<input type="file"  name="avatar" accept="image/*" multiple>
<input type="hidden" name="csrf" value="a1b2c3">

Button inputs: submit, reset, button, image

Three input buttons plus the richer <button> element trigger form actions. type="submit" sends the form to its action; type="reset" restores every control to its initial value (rarely what users want, and easy to hit by mistake); type="button" has no built-in behavior and exists purely to hang JavaScript on. The dedicated <button> element does the same jobs but can contain markup — an icon plus text — which the self-closing input buttons cannot.

The gotcha worth memorizing: a <button> placed inside a form defaults to type="submit", so a button you meant as a harmless click handler will submit and reload the page unless you write type="button" explicitly. Finally, type="image" is a graphical submit button that also sends the click's x/y coordinates as extra fields — a niche control, but interviewers like to ask what it does.

<button type="submit">Save</button>
<button type="button" onclick="preview()">Preview</button>
<input type="reset" value="Clear">
<input type="image" src="/go.png" alt="Search">

Interview questions

The form-input questions Indian interviews reach for first — from service-company front-end screens to product-company deep dives:

Beginner What do the action and method attributes on a form do?

action is the URL that receives the submitted data. method is how it is sent: GET appends the fields to the URL as a query string (bookmarkable, cacheable, length-limited, visible — never for passwords), POST puts them in the request body (larger payloads, not cached, used for anything that changes state).

Beginner How do you associate a label with an input, and why bother?

Two ways: wrap the control inside the <label>, or give the input an id and point <label for="that-id"> at it. It matters because clicking the label focuses or toggles the control, enlarges the tap target, and gives screen readers an accessible name to announce. A control without a label is inaccessible.

Beginner What does the name attribute do, and what happens without it?

name becomes the key under which a control's value is submitted (name=value pairs). A control with no name is simply omitted from the submission — a very common "my field isn't reaching the server" bug.

Beginner How do radio buttons know they belong to one group?

They share the same name. The browser then enforces that only one in that group can be selected, and the submitted value is the value of the chosen radio. Checkboxes with the same name, by contrast, all submit independently.

Beginner When is a checkbox included in the submitted data?

Only when it is checked. An unchecked checkbox sends nothing at all. When checked, it submits its value, defaulting to the string on if you did not set one — which is why you usually give checkboxes an explicit value.

Intermediate Why is type="number" a bad choice for phone numbers, OTPs or PINs?

number is for true quantities. It strips leading zeros, rejects formatting characters, and shows spinner arrows that make no sense for an identifier. Use type="tel" with inputmode="numeric" and a pattern — you get the numeric mobile keyboard without the numeric-value semantics.

Intermediate What is enctype, and when must you change it?

enctype sets how the form body is encoded. The default application/x-www-form-urlencoded is fine for text. The moment you include an <input type="file">, you must switch to multipart/form-data, or the file's bytes will not be transmitted. (A third value, text/plain, exists but is only for debugging.)

Intermediate What is a hidden input for, and what is the security caveat?

It carries data the user should not edit but that must submit — a record id, a wizard's current step, a CSRF token. The caveat: "hidden" means not rendered, not secret. Its value is right there in the page source and DOM and can be changed by anyone, so the server must never trust it blindly.

Intermediate Difference between the button element and input type="submit"?

Both submit a form. <input type="submit"> is a self-closing control whose label is its value — text only. <button> has an open/close tag so it can contain markup (an icon plus text). The trap: a <button> inside a form defaults to type="submit", so omit that and a "just a click handler" button will submit and reload the page.

Intermediate What format does an input type="date" submit, and why does that matter?

Always yyyy-mm-dd, regardless of how the picker displays the date in the user's locale (dd/mm vs mm/dd). That separation of display from wire format is the point of the native control — the server receives one predictable string and never has to parse ambiguous localized text.

Pro Difference between the disabled and readonly attributes on an input?

readonly keeps the field focusable and, crucially, still submits its value — the user just cannot edit it. disabled greys the control out, removes it from tab order, blocks all interaction, and excludes it from the submission entirely. Pick readonly when the value must still reach the server; disabled when it must not.

Pro The email/required/pattern attributes validate in the browser — is that enough?

No. Client-side constraint validation is a UX convenience: it is trivially bypassed by editing the DOM, disabling scripts, or sending a crafted request straight to the endpoint. Treat it as a fast first check for honest users and always re-validate on the server, which is the only trust boundary. Never rely on HTML validation for security or data integrity.
