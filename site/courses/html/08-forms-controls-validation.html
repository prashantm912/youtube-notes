Module: Forms — Controls and Validation

Inputs collect data; this module is about the controls that shape it and the browser's own gate that rejects bad data before it ever leaves the page — dropdowns, multi-line text, progress and meter readouts, the validation attributes, and the Constraint Validation API behind :valid and :invalid.

Knowledge Library → Courses → HTML → Forms: Controls and Validation

What this module covers

Module 07 covered the <input> family; this one covers the rest of the form toolkit and the layer that checks it. You will orient on <select> and its option grouping (plus the new customizable-select mechanics), the non-input controls (datalist, textarea, output, progress, meter, fieldset), the HTML attributes that declare rules, the Constraint Validation API and its pseudo-classes, and what actually happens on submit — how FormData is built and why GET and POST encode it differently.

Topics

select, option, optgroup, and customizable select

A <select> is a dropdown whose choices are <option> elements; <optgroup> visually groups related options under a non-selectable label. The value that submits is the selected option's value attribute (falling back to its text), so a hidden default option and explicit values matter. Add multiple for a multi-select list and size to show several rows at once.

Why it matters: select was for years the one control CSS could barely touch. The customizable-select feature changes that — placing a <button> with a <selectedcontent> child as the first child of the <select>, then opting in with appearance: base-select, lets you fully style the closed control and its drop-down while keeping native keyboard and accessibility behavior. Knowing both the classic model and where this is heading is increasingly a senior-level talking point.

<select name="size" required>
  <option value="">Choose a size</option>
  <optgroup label="Adult">
    <option value="m">Medium</option>
    <option value="l" selected>Large</option>
  </optgroup>
</select>

Other controls: datalist, textarea, output, progress, meter, fieldset

Beyond inputs and selects, HTML ships a set of purpose-built controls. <datalist> attaches a list of suggestions to an input via list/id — autocomplete hints, not a closed choice like select. <textarea> is multi-line free text (its value is the text between the tags, never innerHTML). <output> is a live region for showing a calculated result. <fieldset> with a <legend> groups related controls and, when given disabled, disables every control inside it at once. A <button> is the flexible click control covered under submission below.

Why it matters: <progress> and <meter> look similar but mean different things and get confused in interviews. progress shows task completion moving toward done (a download, an upload); meter is a static gauge of a value within a known range (disk usage, a score) with low/high/optimum thresholds. Reaching for a real semantic element instead of a styled div gives you accessibility and meaning for free.

<label for="bio">About you</label>
<textarea id="bio" name="bio" rows="4" maxlength="280"></textarea>

<progress max="100" value="70">70%</progress>
<meter min="0" max="100" low="20" high="80" optimum="90" value="64">64%</meter>

Form and control attributes: required, pattern, min/max, autocomplete

Most validation rules are declared, not scripted. required makes a field mandatory; minlength/maxlength bound text length; min/max/step bound numeric and date ranges; pattern supplies a regular expression the value must match in full. The in-field hint attribute (shown in the example below) displays greyed sample text inside an empty control and must never stand in for a real label, and autocomplete tells the browser what a field means (email, tel, postal-code, one-time-code) so it can autofill correctly — a real usability and accessibility win.

Why it matters: these attributes are the cheapest correctness you can buy — they run before any JavaScript loads. A common Indian-interview follow-up is the PIN-code or phone field: express the rule with pattern and inputmode rather than hand-rolling a keydown handler. Pair every pattern with a title so the browser can explain the expected format.

<input
  name="pin"
  type="text"
  inputmode="numeric"
  pattern="\d{6}"
  title="6-digit area PIN code"
  autocomplete="postal-code"
  placeholder="e.g. 560001"
  required>

Constraint validation: the API, :valid / :invalid, custom messages

The declared rules feed the Constraint Validation API. Every form control exposes a validity object (a ValidityState) with booleans like valueMissing, typeMismatch, patternMismatch, rangeOverflow and tooLong explaining exactly why it fails. checkValidity() returns true/false silently; reportValidity() also shows the native bubble. CSS mirrors this state through :valid, :invalid, :required and :optional, and the user-interaction variants :user-valid/:user-invalid that only match after the user has engaged.

Why it matters: to override a native message, listen for the control's invalid event and call setCustomValidity("...") — but you must reset it to the empty string once the value becomes valid, or the field stays permanently in error. This clear-on-fix detail is the single most-missed point in validation interview questions, right alongside "styling :invalid reddens empty required fields on first paint."

const email = form.elements.email;
email.addEventListener("input", () => {
  email.setCustomValidity(
    email.value.endsWith("@example.com") ? "Use your work email" : ""
  );
});
if (!form.checkValidity()) form.reportValidity();

Submission mechanics: FormData, GET vs POST, enctype

On submit, the browser gathers every successful control's name/value pair. With method="get" those pairs are appended to the action URL as a query string — visible, bookmarkable, cacheable, length-limited, and right for searches and filters. With method="post" they travel in the request body — the choice for anything that changes state or carries secrets or files. The enctype attribute controls the body format: the default application/x-www-form-urlencoded, or multipart/form-data which is mandatory for <input type="file"> uploads.

Why it matters: in scripted submissions the FormData object mirrors exactly what a native submit would send — new FormData(form) then hand it to fetch, and the browser sets the correct multipart headers and boundary for you. A <button> without type="button" defaults to submit, so an unmarked button inside a form will submit it — a classic bug worth naming in interviews.

form.addEventListener("submit", async (e) => {
  e.preventDefault();
  const data = new FormData(form);       // name/value pairs + files
  await fetch(form.action, { method: "POST", body: data });
});

Interview questions

Most-asked forms-and-validation questions in Indian interviews — from service-company screening rounds through product-company deep dives, with the browser-native validation API a recurring favourite:

Beginner What do select, option, and optgroup do, and which value gets submitted?

<select> is a dropdown; each choice is an <option>; <optgroup label="..."> groups options under a non-selectable heading. The submitted value is the chosen option's value attribute, or its visible text if value is omitted. Add multiple for multi-select and size to show more than one row.

Beginner datalist vs select — what is the difference?

<select> is a closed set — the user must pick one of the given options. <datalist> is a suggestion list bound to a free-text input via list/id; the user can accept a suggestion or type anything else. Use select when the value must be from the list, datalist for autocomplete hints.

Beginner What does the required attribute do, and where does it apply?

It marks a control as mandatory — an empty required field sets validity.valueMissing and blocks native submission. It applies to most inputs, plus <select> (a required select needs an empty-value prompt option, such as a "Choose one" entry, to force a real choice) and <textarea>. It does not apply to hidden, buttons, or ranges.

Beginner progress vs meter — when do you use each?

<progress> shows completion of a task advancing toward done — a file upload, a form wizard; omit value for an indeterminate spinner. <meter> is a static gauge of a measurement within a known range — disk usage, a rating — with low/high/optimum to colour the good and bad zones. Progress moves; meter measures.

Beginner What are fieldset and legend for, and what does disabling a fieldset do?

<fieldset> groups related controls; its <legend> is the group's caption, announced by screen readers with each control (useful for radio-button groups). Setting disabled on the fieldset disables every control inside it in one move — handy for switching off a whole section of a form.

Intermediate GET vs POST for form submission — differences and when to use which?

GET puts the data in the URL query string: visible, bookmarkable, cacheable, length-limited — right for idempotent reads like search and filters. POST puts it in the request body: not in the URL, not cached, no practical length limit, and required for file uploads — right for anything that changes state or carries sensitive data. GET must not mutate; POST should.

Intermediate What is enctype, and which value do file uploads need?

enctype sets how POST body data is encoded. Default application/x-www-form-urlencoded percent-encodes name=value pairs. File uploads require multipart/form-data, which sends each field as a separate part with its own headers and can carry binary file content; <input type="file"> is only transmitted under this encoding. (text/plain exists but is for debugging only.)

Intermediate How does the pattern attribute work, and what are its gotchas?

pattern holds a regex the value must match in full — it is implicitly anchored, so no ^/$ and no surrounding slashes. It applies only to text-like inputs (text, tel, email, url, password, search) and does nothing on an empty field unless required is also set. Always pair it with title to describe the format, and remember it is client-side only — never trust it as your sole validation.

Intermediate checkValidity() vs reportValidity()?

Both run constraint validation and return a boolean for whether the element (or form) is valid. checkValidity() is silent — use it to gate logic. reportValidity() additionally focuses the first invalid control and shows the browser's native error bubble — use it when you want to surface the failure to the user. Both fire the invalid event on failing controls.

Intermediate How do you build and send form data with fetch?

const data = new FormData(formEl) collects every named, successful control including files, then fetch(url, { method: "POST", body: data }) sends it; the browser sets the correct multipart/form-data content type and boundary automatically — do not set it yourself. For a URL-encoded body instead, wrap it as new URLSearchParams(data). You can also append()/set() extra fields on the FormData before sending.

Pro How do you show a custom validation message, and what is the trap?

Call element.setCustomValidity("your message") — a non-empty string marks the control invalid (validity.customError becomes true) and supplies that text. The trap: it is sticky. You must call setCustomValidity("") once the value becomes acceptable (typically inside an input or invalid handler), otherwise the field stays invalid forever even after the user fixes it. Clearing on fix is the most-missed step.

Pro Why does styling :invalid redden empty required fields before the user types, and how do you fix it?

:invalid matches from first paint — an empty required field is already invalid, so error styles show immediately, which is hostile UX. Fixes: prefer :user-invalid (matches only after interaction or submit), or add a .was-submitted class via JS on the first submit and scope the invalid styles under it. Same reasoning applies in reverse to premature green :valid.

Pro What do novalidate and formnovalidate do, and why is server-side validation still mandatory?

novalidate on the <form> (or formnovalidate on a specific submit button, e.g. a "Save draft" button) skips native constraint checking for that submission — useful when you validate yourself or want to save incomplete data. The deeper point: all client-side validation is a UX convenience that anyone can bypass (disable JS, edit the DOM, hit the endpoint directly with curl). The server must independently validate and sanitise every input — HTML validation improves the experience, it never secures the data.
