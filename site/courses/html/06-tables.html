Module: Tables

A data table is a grid with meaning — rows, cells, and the header relationships that tell a screen reader what each number actually is. This module maps every table element and links the authoritative reference for each.

Knowledge Library → Courses → HTML → Tables

What this module covers

Tables present genuine two-dimensional data: the structural elements that build the grid, the column-level tools (colgroup/col) and cell-level spanning (colspan/rowspan), and the accessibility markup — scope, headers, and caption — that connects each cell to the header describing it. The one rule underneath all of it: use tables for tabular data, never for page layout.

Topics

table, caption, thead, tbody, tfoot, tr, th, td

A table is a grid of rows (tr) containing cells — th for a header cell, td for a data cell. Around the rows sit optional structural wrappers that add meaning beyond the visual layout: caption is the table's title and accessible name (it must be the first child of table), while thead, tbody, and tfoot group the rows into header, body, and summary sections.

Why it matters: semantic grouping is what lets assistive technology, print stylesheets, and CSS treat the data correctly — a real th is announced as a header rather than read as plain text, thead can repeat across printed pages, and tbody is the natural scroll or sort target. Both MDN and the WHATWG standard state plainly that tables must not be used as layout aids.

<table>
  <caption>Quarterly units by region</caption>
  <thead>
    <tr><th scope="col">Region</th><th scope="col">Units</th></tr>
  </thead>
  <tbody>
    <tr><th scope="row">North</th><td>128</td></tr>
    <tr><th scope="row">South</th><td>94</td></tr>
  </tbody>
  <tfoot>
    <tr><th scope="row">Total</th><td>222</td></tr>
  </tfoot>
</table>

colgroup, col, spanning (colspan/rowspan)

colgroup and its col children describe the table's columns before any rows are written, so you can style or size whole columns at once; a col takes a span attribute to cover several columns. Spanning works at the cell level in the other direction: colspan stretches a td or th across columns and rowspan across rows, merging several grid positions into one logical cell.

Why it matters: col only responds to a small set of CSS properties (background, border, width, visibility) — it is not a general per-column styling hook, which surprises people. Spanning numbers must add up to a clean rectangle or the table renders unpredictably, and rowspan="0" is the one special value that extends a cell to the end of its row group. These are staple whiteboard questions.

<table>
  <colgroup>
    <col />
    <col span="2" class="highlight" />
  </colgroup>
  <tr>
    <th scope="row">Q1</th>
    <td colspan="2">Launch window</td>   <!-- one cell, two columns -->
  </tr>
  <tr>
    <th rowspan="2" scope="row">Q2</th>   <!-- one cell, two rows -->
    <td>April</td><td>May</td>
  </tr>
</table>

scope, headers, accessible tables

A sighted reader infers which header governs a cell from its position; a screen reader cannot. The scope attribute on a th (col, row, colgroup, or rowgroup) declares that association explicitly, and for irregular multi-level headers the headers/id pairing lets each cell name the exact header cells that describe it. A caption gives the whole table an accessible name.

Why it matters: accessible tables are a compliance and interview staple, and in India product and service companies both test WCAG awareness directly. The rules are small but strict — use th for headers, add scope, keep one logical header per axis where you can, provide a caption, and never fake a table with divs or reach for a layout table. Getting this right is frequently the line between a passing and a failing accessibility audit.

<table>
  <caption>Team availability</caption>
  <tr>
    <td></td>
    <th id="mon" scope="col">Mon</th>
    <th id="tue" scope="col">Tue</th>
  </tr>
  <tr>
    <th id="asha" scope="row">Asha</th>
    <td headers="asha mon">Yes</td>   <!-- explicit association -->
    <td headers="asha tue">No</td>
  </tr>
</table>

Interview questions

Most-asked table questions in Indian interviews — from HTML screening rounds to accessibility deep-dives at product companies:

Beginner What are the core elements that make up an HTML table?

table wraps everything; tr is a row; th is a header cell and td a data cell. Add caption for the title, and thead/tbody/tfoot to group header, body, and summary rows.

Beginner Difference between th and td?

th is a header cell — bold and centered by default, and, crucially, announced as a header by screen readers, which then associate it with the data cells it labels. td is an ordinary data cell. A bolded td looks like a header but carries none of the semantics.

Beginner What is the caption element and where must it go?

caption is the table's visible title and its accessible name. It must be the very first child of table (before any colgroup or rows). It helps every user, but especially screen-reader users deciding whether a table is the one they want.

Beginner What do thead, tbody, and tfoot do?

They group rows semantically: header rows, body data, and a footer/summary. Benefits include header rows repeating when a long table prints, tbody being the target for scrolling or sorting, and clearer structure for assistive tech. They do not change the visual grid by themselves.

Beginner How do you make a cell span multiple columns or rows?

colspan="n" on a td/th stretches it across n columns; rowspan="n" stretches it down n rows. The merged cell occupies those grid positions, so the remaining cells in the affected rows shift over — the counts must still form a clean rectangle.

Intermediate What is the difference between colgroup/col and the row-group elements, and what can col actually style?

colgroup/col describe columns (the vertical axis) before the rows; thead/tbody/tfoot group rows (horizontal). col only honours a few CSS properties — background, border, width, and visibility: collapse — so it is not a general per-column style hook.

Intermediate How does the scope attribute improve accessibility?

scope on a th declares which cells it heads: col or row for a single line, colgroup/rowgroup for a spanned header. Screen readers use it to announce the correct header before each data cell, so a value like "94" is read as "South, Units, 94".

Intermediate When would you use headers/id instead of scope?

For complex tables where a cell sits under more than one header or the header layout is irregular and scope would be ambiguous. Each header th gets an id; each data cell lists the relevant ids in its headers attribute, pinning the association explicitly. For simple tables, scope is enough and simpler.

Intermediate Why shouldn't tables be used for page layout?

Screen readers announce a table as tabular data, so a layout table produces confusing "row/column" narration for what is really just page structure. It also resists responsive design and is far harder to maintain. Use CSS grid or flexbox for layout; reserve table for real data.

Intermediate How do you make a wide data table responsive without breaking its semantics?

The safe default is to wrap the table in a container with overflow-x: auto so it scrolls horizontally while the markup stays intact. Restructuring layouts (using CSS to relabel cells at small widths) can work, but changing a table's display strips its implicit ARIA table roles — you must restore them with roles or keep header associations working, or accessibility regresses.

Pro What does rowspan="0" do, and is there a colspan="0"?

rowspan="0" tells the cell to span every remaining row in its current row group (thead/tbody/tfoot) — the only spanning value that is not a fixed count. colspan="0" existed in HTML4 to span all columns but was dropped from the HTML5/living standard and is not supported, so never rely on it.

Pro How does a screen reader navigate a table, and what actually makes one "accessible"?

In table mode the user moves cell by cell while the reader announces the associated header(s) for each cell. That association comes from real th elements plus scope (or headers/id for complex grids), with caption supplying the table's name. "Accessible" therefore means: th for headers, correct scope/headers, a caption, no layout tables, and no div grids impersonating a table without table roles.
