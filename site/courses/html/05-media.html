Module: Images and Media

Getting a picture on the page is one tag; getting the right picture, at the right size, with a caption a screen reader can use and a video anyone can follow — that is this module, and it is where markup starts to touch performance and accessibility.

Knowledge Library → Courses → HTML → Images and Media

What this module covers

Media is where HTML stops being pure text. You will learn the <img> element and its accessibility and performance attributes, the <picture>/<source> pairing for responsive and art-directed images, <figure>/<figcaption> for captioned content, and the <audio>, <video> and <track> elements for time-based media. A closing survey introduces <canvas>, inline SVG and image maps so you recognise them when they appear.

Topics

The img element: alt, srcset, sizes, loading, decoding

The <img> element embeds a raster or vector image. Two attributes are non-negotiable: src (the URL) and alt (a text equivalent). Good alt describes the image's purpose, is empty (alt="") for purely decorative images so screen readers skip them, and is never a filename. Setting width and height lets the browser reserve space before the image arrives, which prevents layout shift.

Why it matters: the modern <img> is a performance surface, not just a slot to drop a file into. srcset plus sizes let the browser download the smallest file that still looks sharp on the user's screen; loading="lazy" defers off-screen images; and decoding="async" keeps image decoding off the main render path. On the mid-range Android phones and metered data that dominate the Indian web, these attributes are the difference between a fast page and a bounced visitor.

<img
  src="photo-800.jpg"
  srcset="photo-400.jpg 400w, photo-800.jpg 800w, photo-1600.jpg 1600w"
  sizes="(max-width: 600px) 100vw, 50vw"
  alt="Sunrise over the Western Ghats"
  width="800" height="500"
  loading="lazy" decoding="async">

picture and source: responsive images and art direction

<picture> wraps one or more <source> elements and a mandatory fallback <img>. The browser walks the sources in order and uses the first whose media condition and type it can honour, otherwise it falls back to the <img>. This unlocks two things srcset alone cannot: art direction — serving a differently cropped image on narrow screens (a tight portrait crop on mobile, a wide banner on desktop) — and format switching — offering AVIF or WebP to browsers that support them while older browsers still get JPEG.

Why it matters: know the split. srcset/sizes on a plain <img> is resolution switching (same image, different sizes, browser decides). <picture> is for when you must decide — different crops or different file formats. Interviewers probe exactly this boundary because candidates routinely reach for <picture> when a one-line srcset would do.

<picture>
  <source media="(max-width: 600px)" srcset="hero-portrait.avif" type="image/avif">
  <source srcset="hero-wide.avif" type="image/avif">
  <source srcset="hero-wide.webp" type="image/webp">
  <img src="hero-wide.jpg" alt="Team on stage at a product launch" width="1200" height="600">
</picture>

figure and figcaption

<figure> marks up self-contained content — an image, chart, code listing, quotation or diagram — that is referenced from the main flow but could be moved elsewhere (an appendix, a sidebar) without breaking the sentence around it. An optional <figcaption>, the first or last child, gives that content a visible caption and, importantly, an accessible name.

Why it matters: figcaption is a caption, not a replacement for alt. They do different jobs — alt is the image's text equivalent for people who cannot see it; the caption is supplementary context everyone reads. A common mistake (and a favourite interview trap) is emptying alt because "the caption already describes it." Use both, and let them say different things.

<figure>
  <img src="metro-map.png" alt="Bengaluru Metro network with Purple and Green lines" width="900" height="600">
  <figcaption>Fig 3. Namma Metro operational lines as of 2026.</figcaption>
</figure>

audio, video, and track (captions)

<video> and <audio> embed a media player. Add controls for the native UI; use nested <source> elements to offer several formats (WebM, MP4) so the browser plays the first it supports, with fallback content between the tags for browsers that support neither. preload, poster, muted, autoplay and loop tune loading and playback — note that most browsers only allow autoplay when the video is also muted.

Why it matters: media without captions excludes deaf and hard-of-hearing users and anyone watching without sound. The <track> element points at a WebVTT (.vtt) file and a kind — captions (dialogue plus non-speech sound, for a viewer who cannot hear) versus subtitles (translation of dialogue, assuming sound is audible). Knowing that distinction, and that captions are a WCAG requirement, is a standard accessibility interview beat.

<video controls width="640" poster="cover.jpg" preload="metadata">
  <source src="talk.webm" type="video/webm">
  <source src="talk.mp4" type="video/mp4">
  <track default kind="captions" srclang="en" label="English" src="talk-en.vtt">
  <p>Your browser can't play this video. <a href="talk.mp4">Download it</a>.</p>
</video>

Survey: canvas, inline SVG, and image maps

Three related tools you should recognise even before you use them daily. <canvas> is a scriptable bitmap surface — you draw pixels with JavaScript (2D context or WebGL), which suits games, charts and image manipulation but produces nothing accessible on its own, so always provide fallback content or an ARIA description. Inline SVG embeds vector graphics directly in the markup: resolution-independent, styleable with CSS, scriptable, and ideal for icons, logos and diagrams that must stay crisp at any zoom.

Image maps (<map> with child <area> elements, linked from an <img usemap>) make different regions of a single image clickable. They are a niche, largely legacy technique today — CSS-positioned links or inline SVG usually serve better — but they still appear in exam-style questions, and each <area> needs its own alt text. This is survey-level: aim to recognise and reason about these, not to master them here.

Interview questions

Images and media questions asked in Indian front-end interviews, from HTML screening rounds to product-company performance and accessibility deep dives:

Beginner What does the alt attribute do, and when should it be empty?

It provides a text equivalent read by screen readers and shown when the image fails to load. Write it to convey the image's purpose. Set alt="" (present but empty) for purely decorative images so assistive tech skips them; omitting alt entirely is different and worse — some readers then announce the filename.

Beginner Difference between the src and srcset attributes on an image?

src is a single fallback URL. srcset is a list of candidate files with descriptors (widths like 800w or densities like 2x); the browser picks the best one for the screen and viewport. Older browsers ignore srcset and use src.

Beginner What is loading="lazy" and when should you NOT use it?

It defers loading an image until it nears the viewport, saving bandwidth on long pages. Do not lazy-load above-the-fold images — especially the hero / Largest Contentful Paint image — because deferring it delays the very content that measures your load speed.

Beginner What are figure and figcaption for?

<figure> groups self-contained referenced content (image, chart, code, quote) that could move without breaking the surrounding text; <figcaption> is its visible caption and accessible name, placed as the first or last child.

Beginner How do you embed a video that plays across browsers?

Use <video controls> with multiple <source> children in different formats (e.g. WebM and MP4); the browser plays the first it supports. Put fallback text or a download link between the tags for browsers that support none.

Intermediate With width descriptors in srcset, how does the browser choose, and what does sizes do?

sizes tells the browser how wide the image will render at various breakpoints (e.g. (max-width: 600px) 100vw, 50vw). The browser combines that slot width with the device pixel ratio to compute a target width, then picks the smallest w candidate that meets or exceeds it. Omitting sizes with w descriptors makes the browser assume 100vw, usually over-fetching.

Intermediate picture/source versus img srcset/sizes — when do you use each?

srcset/sizes on a plain <img> is resolution switching: same image, different sizes, browser decides. Use <picture> when you must decide — art direction (different crops per breakpoint via media) or format switching (AVIF/WebP with a JPEG fallback via type). Do not reach for <picture> when a one-line srcset suffices.

Intermediate How do you serve AVIF or WebP with a fallback to JPEG?

Wrap sources in <picture>, ordering most-modern first, each with a type: <source type="image/avif">, then <source type="image/webp">, then a plain <img src="…jpg">. The browser uses the first format it can decode and falls back to the <img>.

Intermediate What does the decoding attribute do?

decoding hints how to decode the image relative to rendering: async decodes off the main flow to avoid blocking other content, sync decodes immediately (useful to avoid a flash for critical images), auto lets the browser decide. It is a hint, not a guarantee.

Intermediate Why set width and height on an img, and how does it relate to CLS?

They let the browser compute the aspect ratio and reserve the correct box before the image downloads, so surrounding content does not jump when it arrives. That prevents Cumulative Layout Shift (CLS), a Core Web Vital. Modern browsers derive an aspect-ratio from the two attributes even when CSS scales the width fluidly.

Intermediate How do you add captions to a video, and how do captions differ from subtitles?

Nest a <track> pointing to a WebVTT file with a kind and srclang; mark one default. kind="captions" includes dialogue plus non-speech audio (sound effects, speaker changes) for viewers who cannot hear; kind="subtitles" is a translation of dialogue that assumes the viewer can hear. Captions satisfy the WCAG requirement for prerecorded audio.

Pro canvas versus inline SVG — how do you choose?

SVG is retained-mode vector: DOM nodes you can style with CSS, script, and hit-test; it scales without loss and is accessible via ARIA — ideal for icons, logos, charts with moderate element counts. Canvas is immediate-mode bitmap: you redraw pixels each frame with JavaScript — better for thousands of objects, pixel manipulation, and games, but it exposes no DOM, so accessibility and interaction are fully your responsibility. Rule of thumb: few, meaningful shapes and crispness → SVG; many objects or per-pixel work at speed → canvas.

Pro Your hero image is the LCP element and it loads late on 4G. What HTML-level fixes apply?

Do not lazy-load it — keep it eager. Hint its priority with fetchpriority="high" on the <img> (and/or a <link rel="preload" as="image"> that matches its srcset/sizes). Serve a modern format via <picture>, size it correctly with srcset/sizes so mobile does not fetch a desktop file, and set width/height to avoid a shift once it lands.

Pro How do you make image maps and complex images (like an infographic) accessible?

For an image map, every <area> needs its own descriptive alt naming where the region leads — the parent <img>'s alt alone is not enough. For a complex infographic, a short alt plus a longer text equivalent nearby (a caption, an adjacent description, or an aria-describedby reference) beats cramming everything into alt; the old longdesc attribute is obsolete. Prefer inline SVG with proper titles/roles when the graphic is genuinely structured.
