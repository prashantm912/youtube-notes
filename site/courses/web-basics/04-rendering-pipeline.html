Module: The Rendering Pipeline

Type a URL, hit enter, and within a few hundred milliseconds raw bytes become pixels. This module walks the machine that does it — parse, style, build the render tree, lay out, paint, composite — and points you at the resources that go deep on each stage.

Knowledge Library → Courses → How the Web Works → Rendering Pipeline

What this module covers

Everything between the HTTP response and the first painted frame: how the browser tokenizes HTML into the DOM, parses CSS into the CSSOM, combines them into a render tree, computes geometry in layout, rasterizes pixels in paint, and assembles layers in compositing. It closes with the resources on that critical path that block rendering — the knowledge behind almost every front-end performance answer.

Topics

HTML parsing into the DOM (tokenizing, tree construction)

When the first bytes of the HTML response arrive, the parser converts them to characters using the declared encoding, then a tokenizer performs lexical analysis — emitting tokens for start tags, end tags, attribute names, attribute values and text. A tree-construction stage consumes that token stream and assembles the Document Object Model: a live, in-memory tree of nodes that mirrors the nesting of your markup and is the structure every script and stylesheet ultimately targets.

Why it matters: parsing is incremental, so the browser can build and even display part of the DOM before the whole document has arrived — which is why streaming HTML early pays off. It is also the point where a plain <script> stops the parser dead: the browser must pause, fetch and run the script (it might call document.write) before continuing. Half of all page-speed advice makes sense only once you picture this step.

CSS parsing into the CSSOM

Stylesheets follow the same bytes → characters → tokens → nodes path, but the output is the CSS Object Model (CSSOM): a tree of style rules in which every property has been resolved through inheritance and the cascade, so each node carries its final computed styles. Unlike the DOM, a partly-built CSSOM is not safe to use — a rule near the end of a stylesheet can override everything declared above it.

Why it matters: because painting with an incomplete CSSOM would show the wrong styles and force a re-do, the browser treats CSS as render-blocking — it will not build the render tree until the CSSOM is complete. That single fact (CSS blocks rendering; a script can block parsing) is the root of most critical-path work, and the reason teams inline the critical CSS and load the rest without blocking.

Render tree, layout/reflow

The browser walks the DOM and, for each node, pulls its computed styles from the CSSOM to build the render tree — a tree of only the boxes that will actually be painted. Nodes set to display: none and non-visual nodes like <head> are excluded, while pseudo-elements such as ::before are included. Layout (also called reflow) then walks that tree and computes exact geometry: the size and position of every box, resolving percentages and viewport units into concrete pixels.

Why it matters: layout is expensive and cascading — changing one box's geometry can force the browser to recompute its ancestors, siblings and descendants. Reading a geometric property such as offsetHeight or getBoundingClientRect() immediately after writing a style forces a synchronous reflow — "layout thrashing" — a classic trap that turns a smooth loop into jank. The fix is to batch all reads before all writes.

// Thrash: each read forces the pending write to reflow, every iteration
for (const el of boxes) {
  el.style.width = container.offsetWidth + 'px';
}

// Better: read once, then write in a batch
const w = container.offsetWidth;
for (const el of boxes) el.style.width = w + 'px';

Paint and compositing (layers, GPU)

Paint — rasterization — fills in the actual pixels for each render-tree box: text, colors, borders, shadows, backgrounds and images, usually onto one or more layers rather than a single flat bitmap. Compositing is the final step: the browser, often handing work to the GPU, assembles those layers in the correct order, applies transforms and opacity, and sends the result to the screen. Properties like transform and opacity can be handled by the compositor alone — it moves a layer that is already painted.

Why it matters: this is why animating transform and opacity stays smooth while animating top, left or width stutters — the former only re-composites, the latter re-runs layout and paint on every frame inside the browser's roughly 16.6 ms budget. You can hint the browser to promote an element to its own GPU layer with will-change, but layers cost memory, so promote sparingly and drop the hint once the animation ends.

/* Cheap: the compositor moves an already-painted layer, no layout or paint */
.card { transition: transform 200ms ease, opacity 200ms ease; }
.card:hover { transform: translateY(-4px); opacity: .95; }

/* Promote to its own layer only while it will actually animate */
.card.is-animating { will-change: transform; }

Render-blocking resources and the critical rendering path

The critical rendering path is the minimum sequence the browser must finish before it can paint meaningful content: parse HTML into the DOM, build the CSSOM, construct the render tree, run layout, then paint. Anything on that path that must download and process before the first paint is a critical resource. CSS in the head is render-blocking by default; a plain <script> is parser-blocking (it halts DOM construction), which makes it render-blocking too.

Why it matters: optimizing the path means shrinking three numbers — the count of critical resources, the critical bytes, and the network round trips. The main levers are loading scripts with defer (fetched in parallel, executed in order after parsing, never blocking) or async (executed as soon as it arrives, order not guaranteed), inlining or preloading critical CSS, and deferring everything non-essential. Knowing exactly when each script runs relative to DOMContentLoaded is a staple of the performance round.

<!-- Blocks the parser until fetched AND executed -->
<script src="analytics.js"></script>

<!-- Fetched in parallel, runs after parsing, keeps document order -->
<script defer src="app.js"></script>

<!-- Fetched in parallel, runs as soon as ready, order not guaranteed -->
<script async src="ads.js"></script>

Interview questions

Rendering-pipeline questions dominate the performance round in Indian front-end interviews — from a screening "what happens after the HTML arrives?" to product-company deep dives on jank:

Beginner What is the critical rendering path?

The sequence of steps a browser runs before the first meaningful paint: parse HTML into the DOM, parse CSS into the CSSOM, combine them into the render tree, run layout to compute geometry, then paint pixels (and composite layers). Optimizing it is about reaching that first paint sooner.

Beginner What is the DOM and how is it built?

The Document Object Model is an in-memory tree the browser builds while parsing HTML: bytes become characters, a tokenizer emits tags/attributes/text as tokens, and tree construction turns those into nodes. It is incremental — the DOM grows as bytes arrive — and it is what scripts and styles operate on.

Beginner What is the CSSOM and why do we need it?

The CSS Object Model is the tree of style rules produced by parsing CSS, with each property resolved through the cascade and inheritance into computed values. The browser needs both the DOM (structure) and the CSSOM (appearance) to know what to render, so neither alone is enough.

Beginner How does the render tree differ from the DOM?

The render tree is DOM plus CSSOM, containing only the boxes that get painted. display: none nodes and non-visual nodes like <head> are excluded; pseudo-elements are included. Note that visibility: hidden elements stay in the tree — they occupy space, they are just not drawn.

Beginner What is the difference between reflow and repaint?

Reflow (layout) recalculates the position and geometry of elements; repaint redraws pixels when only visual properties like color or background change but geometry does not. Reflow is the expensive one and always forces a repaint of the affected area; a repaint alone does not trigger a reflow.

Intermediate Why is CSS render-blocking?

A partially-built CSSOM could paint the page with styles a later rule overrides, then repaint — flicker and wasted work. So the browser blocks render-tree construction until the CSSOM is complete. Mitigations: keep CSS small, inline the critical above-the-fold CSS, and scope non-essential stylesheets with media queries so they load without blocking.

Intermediate Why can a <script> block parsing, and how do defer and async change it?

A plain script is parser-blocking: the browser must stop DOM construction, fetch and execute it (it may call document.write) before continuing. defer fetches in parallel and runs after parsing finishes, in document order, before DOMContentLoaded. async also fetches in parallel but runs the moment it is ready, with no ordering guarantee.

Intermediate defer vs async — which do you pick, and when?

Use defer for your application scripts that need the DOM or depend on each other, since order is preserved and execution waits for parsing. Use async for independent, order-insensitive scripts like analytics or ads. Both download without blocking the parser; the difference is execution timing and ordering.

Intermediate What is layout thrashing and how do you avoid it?

Interleaving DOM writes with reads of geometric properties (offsetHeight, getBoundingClientRect()) in a loop forces the browser to flush a synchronous reflow on every read. Avoid it by batching: read all measurements first, then apply all writes, and schedule visual updates inside requestAnimationFrame.

Intermediate Which properties are cheapest to animate, and why?

transform and opacity — they are compositor-only, so the browser skips layout and paint and just re-composites an existing layer, often on the GPU. Animating width, top or left triggers layout plus paint every frame, which is why those animations feel janky under load.

Pro What does compositing do, and why does it survive a busy main thread?

After paint splits the page into layers, the compositor assembles them in order — applying transforms and opacity — and pushes the frame to the screen, typically via the GPU. Because a transform/opacity change only re-composites an already-painted layer, it can run on the compositor thread and keep animating even while the main thread is blocked. You can promote a layer with will-change or a 3D transform, but each layer costs memory, so promote deliberately.

Pro A page has a slow first paint. Using the pipeline, how do you diagnose and fix it?

Reason stage by stage. Measure the critical path — number of critical resources, critical bytes, round trips — with Lighthouse and the DevTools Performance panel. Cut render-blocking CSS by inlining the critical portion and deferring the rest; make scripts defer/async so they stop blocking the parser; preload key resources. Then check whether the remaining cost is network (download), parse (large DOM/CSSOM), or layout (expensive reflows) and target that stage specifically.
