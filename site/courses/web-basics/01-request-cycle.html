Module: Browsers and the Request Cycle

Type a URL, press Enter, and a dozen systems spring into motion before a single pixel appears. This module maps that journey — URL to DNS to TCP to HTTP to paint — and links the canonical reference for each leg.

What this module covers

This is the ground floor of the whole track: what a URL actually names, how a domain name becomes an IP address through DNS, the ordered sequence that turns "press Enter" into a rendered page, the client-server model that underpins it, and the components inside the browser that do the work. Get this story straight and every later module — HTTP, security, performance — slots onto it. It also happens to be one of the most-asked interview areas, so we treat it deeply.

Topics

URL anatomy: scheme, host, port, path, query, fragment

A URL is the address that names one resource and tells the browser how to reach it. Read left to right: the scheme (https) picks the protocol; the host (a domain name or IP) names the server; an optional port selects the process on that host, defaulting to 443 for https and 80 for http; the path locates the resource on the server; the query string (?key=value) passes parameters; and the fragment (#id) points at a spot inside the returned document and is never sent to the server.

Why it matters: reading a URL fluently is the first thing an interviewer probes, and it underpins routing, caching, and security. Knowing that the fragment stays client-side, or that the port is implied by the scheme, is exactly the detail that separates a memorised answer from an understood one.

https://www.example.com:443/blog/post?id=42#comments
scheme    -> https
host      -> www.example.com
port      -> 443   (implied by the scheme; 80 for http)
path      -> /blog/post
query     -> ?id=42        (sent to the server)
fragment  -> #comments     (stays in the browser)

DNS resolution: recursive vs authoritative, records, caching and TTL

Humans use names like example.com; machines route packets to IP addresses. DNS is the distributed directory that translates one into the other. A lookup walks a hierarchy: your device asks a recursive resolver (usually your ISP's or a public one); if it has nothing cached, the resolver queries a root server, then the TLD server for .com, then the domain's authoritative nameserver, which returns the answer.

Records carry the data — A for IPv4, AAAA for IPv6, CNAME for aliases, MX for mail, NS for delegation, TXT for verification — and every answer carries a TTL that controls how long each layer may cache it. Because DNS sits in front of every request, its caching and TTL behaviour is a favourite interview area, and it is why a DNS change can take time to propagate.

$ dig example.com A +noall +answer
example.com.   3600   IN   A   93.184.216.34
#              ^TTL   ^class ^type ^value (the IPv4 address)

The request lifecycle: from typing a URL to seeing the page

This is the question behind the famous interview prompt. Pressing Enter kicks off an ordered sequence: the browser normalises the URL and checks caches (including HSTS), resolves the host to an IP via DNS, opens a TCP connection with the three-way handshake, negotiates TLS for https, then sends an HTTP request. The server returns a status line, headers, and the HTML body; the browser parses it, discovers and fetches sub-resources (CSS, JS, images), and renders.

Why it matters: interviewers love this because it forces you to connect every layer — DNS, TCP and TLS, HTTP, and rendering — into one coherent story. A strong answer names the steps in order and can zoom into any one of them on request.

1. Parse the URL, check caches (HSTS, browser/OS DNS cache)
2. DNS resolve host  ->  IP address
3. TCP three-way handshake to that IP on the port
4. TLS handshake (only for https)
5. Send HTTP request:  GET /path
6. Server responds: status line + headers + HTML
7. Parse HTML, fetch sub-resources (CSS, JS, images)
8. Render; run JavaScript; page becomes interactive

Client–server model, IP addresses, and ports

The Web runs on the client-server model: a client (your browser) sends requests, a server answers them, and the two never swap roles within a request. Every machine on the network has an IP address that uniquely identifies it, and a port number selects which program on that machine handles the traffic — 443 for HTTPS, 80 for HTTP, 53 for DNS. An IP address plus a port is the socket a TCP connection actually targets.

Why it matters: this is where the common "Internet versus Web" trick question lives. The Internet is the physical network of interconnected machines and cables; the Web is one service — HTTP over that network — built on top of it. Email and video calls are other services riding the same Internet, which is the distinction a good candidate draws without hesitation.

Browser components: rendering engine, networking, JavaScript engine

A browser is several cooperating engines, not one program. The user interface wraps everything you click; the browser engine coordinates; the rendering engine (Blink in Chrome, WebKit in Safari, Gecko in Firefox) turns HTML and CSS into pixels through the DOM, CSSOM, render tree, layout, paint, and compositing; the networking layer fetches resources; the JavaScript engine (V8, JavaScriptCore, SpiderMonkey) executes scripts; and storage handles cookies, cache, and local databases.

Why it matters: at this stage you only need the map, not the internals — but knowing which component owns which job explains why a blocking script stalls rendering, or why layout and paint are counted as separate costs when a page feels slow.

Interview questions

"What happens when you type a URL and press Enter?" is a fixture of Indian SDE and full-stack interviews — from campus screening rounds to product-company deep dives. These build from the one-line answer to the layer-by-layer detail; the hardest sit last.

Beginner Walk me through what happens when you type a URL and press Enter.

The browser parses the URL and checks caches, resolves the domain to an IP via DNS, opens a TCP connection (plus a TLS handshake for https), sends an HTTP request, receives the response (status line, headers, HTML), then parses and renders the page while fetching sub-resources. Name the steps in order — interviewers then drill into whichever one they choose.

Beginner What are the parts of a URL?

Scheme (https), host (domain or IP), optional port, path, query string (?key=value), and fragment (#section). The scheme picks the protocol, host and port locate the server process, the path locates the resource, the query passes parameters, and the fragment points inside the document and is not sent to the server.

Beginner What is DNS and why do we need it?

DNS is the distributed system that maps human-readable names like example.com to the IP addresses machines route to. We need it because people remember names, not numbers, and because the IP behind a name can change without the name changing.

Beginner IP address vs domain name vs URL — how do they relate?

An IP address is the numeric location of a machine; a domain name is a human-friendly alias for one or more IPs, resolved by DNS; a URL is the full address of a resource, wrapping a host (a domain or IP) with a scheme, path, and more. Domain to IP is DNS; a URL adds everything needed to fetch one specific resource.

Beginner What are the default ports for HTTP and HTTPS, and what if none is given?

HTTP defaults to 80, HTTPS to 443. When the URL omits the port, the browser uses the scheme's default, which is why you rarely type it. DNS itself runs on port 53.

Intermediate Describe DNS resolution step by step.

The device asks a recursive resolver. If the answer is not cached, the resolver queries a root nameserver (which returns the TLD server), then the .com TLD server (which returns the authoritative nameserver), then the authoritative server, which returns the record. The resolver caches it for the record's TTL and hands the IP back. Caches at the browser, OS, resolver, and ISP short-circuit these steps.

Intermediate Name the common DNS record types and what each does.

A maps a name to an IPv4 address, AAAA to IPv6, CNAME aliases one name to another, MX lists mail servers, NS delegates a zone to nameservers, TXT holds arbitrary text (SPF, domain verification), and SOA carries zone metadata. A and CNAME are the ones you meet most.

Intermediate What is TTL, and where does DNS caching happen?

TTL (time to live) is how many seconds a record may be cached before it must be fetched again. Caching happens at the browser, the OS stub resolver, the recursive resolver, and intermediate networks. A low TTL propagates changes faster but generates more queries; a high TTL means less traffic but slower propagation — the trade-off you plan around before a migration.

Intermediate What is the client–server model, and how is the Internet different from the Web?

In the client-server model a client requests and a server responds; roles do not swap mid-request. The Internet is the underlying network of interconnected machines; the Web is one service — HTTP documents and APIs — running on top of it. Email and video calls are other services on the same Internet.

Intermediate After DNS resolves, what happens before the first byte of HTML arrives?

The browser opens a TCP connection to the server's IP and port with the three-way handshake (SYN, SYN-ACK, ACK). For https it then runs a TLS handshake to agree on keys and verify the certificate. Only then does it send the HTTP request; the delay until the first response byte is the TTFB (time to first byte).

Pro Recursive vs iterative DNS queries — what is the difference?

In a recursive query the client expects the resolver to do all the work and hand back a final answer. In iterative queries the resolver asks each server in turn and receives referrals (root, then TLD, then authoritative), following them itself. Typically the stub-to-resolver step is recursive, while the resolver-to-nameservers steps are iterative.

Pro After the HTML arrives, walk through the browser's rendering pipeline.

The parser builds the DOM from HTML and the CSSOM from CSS; these combine into the render tree, which is laid out (geometry and positions), painted into layers, and composited to the screen. CSS and synchronous scripts are render-blocking; defer and async change when scripts run so parsing is not stalled. This ordered path is the critical rendering path.

Pro Name the major browser components and how they divide the work.

User interface (the chrome around the page), browser engine (the coordinator), rendering engine (HTML and CSS into pixels: DOM, CSSOM, layout, paint), networking (requests), JavaScript engine (V8, JavaScriptCore, or SpiderMonkey), and data storage (cookies, cache, IndexedDB). Splitting these explains why a blocking script freezes rendering while a network fetch keeps going on its own thread.
