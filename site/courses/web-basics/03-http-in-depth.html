Module: HTTP in Depth

You already send requests and read status codes — this module goes under the hood: how the wire protocol got faster three times over, why a cache returns 304, and what a single cookie attribute has to do with CSRF.

Knowledge Library → Courses → How the Web Works → HTTP in Depth

What this module covers

HTTP is the request/response protocol every web page rides on, and its details decide real performance and security. This module orients you on six areas interviewers actually probe: the HTTP/1.1 → HTTP/2 → HTTP/3 evolution, how connections are managed and stalled, the caching machinery (Cache-Control, ETag, conditional requests), cookie attributes and sessions, content negotiation, and compression. Depth lives in the linked official docs — this page gets you the map.

Topics

HTTP/1.1 vs HTTP/2 vs HTTP/3 + QUIC

Each HTTP version removed the previous one's bottleneck. HTTP/1.1 (1997) added persistent connections, the Host header for virtual hosting, and chunked transfer — but it still processes one request/response per connection at a time, so browsers open roughly six parallel TCP connections per origin to fake concurrency. HTTP/2 (2015) replaced the text protocol with a binary framing layer: many independent streams are multiplexed over one connection, headers are compressed with HPACK, and the six-connection hack becomes unnecessary. HTTP/3 (2022) keeps HTTP/2's semantics but swaps the transport from TCP to QUIC (UDP plus built-in TLS 1.3), eliminating transport-level head-of-line blocking and cutting connection-setup latency.

Why it matters: interviewers routinely ask you to place the three versions on a timeline and state precisely what each fixed. The honest one-liner — "HTTP/2 fixed application-layer HOL blocking, HTTP/3 fixed the transport-layer HOL blocking that HTTP/2 left behind" — is what separates a memorized answer from an understood one. Browsers discover HTTP/3 support via the Alt-Svc header.

HTTP/1.1 200 OK
Alt-Svc: h3=":443"; ma=86400     ; server advertises HTTP/3 (QUIC) on 443
                                 ; the client may switch on its next visit

Connection management: keep-alive, pipelining, multiplexing, HOL blocking

HTTP/1.0 opened a fresh TCP connection per request; HTTP/1.1 made persistent connections (keep-alive) the default so the handshake and TCP slow-start cost is paid once and reused. Pipelining — sending several requests without waiting for each response — was specified but is disabled in practice because responses must return in order (so one slow response blocks the rest) and buggy proxies mishandled it. That ordering constraint is head-of-line (HOL) blocking. HTTP/2's multiplexing solves the application-layer version by numbering streams so responses interleave in any order, which is also why the old workaround of domain sharding became an anti-pattern.

Why it matters: the subtle, most-tested point is that HTTP/2 still runs on TCP, so a single lost packet stalls every stream until it is retransmitted — TCP-level HOL blocking. QUIC gives each stream independent loss recovery, which is the whole reason HTTP/3 exists. Note that Connection and Keep-Alive are hop-by-hop headers and are forbidden in HTTP/2 and HTTP/3.

Connection: keep-alive
Keep-Alive: timeout=5, max=1000   ; reuse this socket for up to 1000 requests
                                  ; or until it sits idle for 5 seconds

Caching: Cache-Control, ETag, Vary, conditional requests

HTTP caching has two mechanisms. Freshness lets a stored response be reused with no network trip until it ages out: Cache-Control: max-age=N (seconds), plus directives like no-store (never cache), no-cache (store but revalidate before every reuse), private vs public (browser-only vs shared/CDN caches), and immutable. Validation handles what happens after a response goes stale: the server stamps a response with an ETag (a version id) or Last-Modified date, and the client revalidates with a conditional request — If-None-Match or If-Modified-Since. If nothing changed the server returns a tiny 304 Not Modified with no body.

Why it matters: the standard production pattern is fingerprinted, long-lived static assets (max-age=31536000, immutable) plus no-cache HTML that always revalidates — a favourite whiteboard question. The Vary header is the trap: a shared cache must key a negotiated response on the request headers used to build it (e.g. Vary: Accept-Encoding), or it will serve a gzip body to a client that cannot read it.

; response the first time
Cache-Control: public, max-age=31536000, immutable
ETag: "33a64df5"

; the browser revalidates later
If-None-Match: "33a64df5"   ->  HTTP/1.1 304 Not Modified   ; no body re-sent

Cookie mechanics: Set-Cookie attributes, SameSite, Secure/HttpOnly, sessions

A cookie is a small key/value the server sets with Set-Cookie and the browser returns on every matching request — the standard way stateless HTTP carries a session. The attributes are the whole exam. Max-Age/Expires separate persistent cookies from session cookies; Domain and Path scope where it is sent; Secure restricts it to HTTPS; HttpOnly hides it from JavaScript (an XSS defence); and SameSite controls cross-site sending — Strict, Lax (the modern default), or None, which must be paired with Secure. The __Host- prefix enforces a hardened combination.

Why it matters: for a login session cookie the expected answer is a short recipe — an opaque server-side session id, marked HttpOnly (XSS), Secure (HTTPS only), and SameSite=Lax or Strict (CSRF). Explaining why each attribute maps to a specific attack is the difference between a junior and senior answer, and SameSite is asked constantly since browsers changed its default.

Set-Cookie: __Host-sid=Ab3xZ9; Path=/; Secure; HttpOnly; SameSite=Lax; Max-Age=2592000
; __Host- forces Secure + Path=/ + no Domain — cannot be overwritten by a subdomain

Content negotiation: Accept, Accept-Language, Accept-Encoding, quality values

Content negotiation lets one URL serve different representations to different clients. In the common proactive (server-driven) form, the client advertises preferences with Accept (media types), Accept-Language (natural language), and Accept-Encoding (compression), and the server picks the best match. Preferences are ranked with quality values — a ;q= weight from 0 to 1, defaulting to 1 — so en-IN,en;q=0.9,hi;q=0.8 means "Indian English first, then English, then Hindi." When no representation is acceptable the server may return 406 Not Acceptable.

Why it matters: negotiation and caching are joined at the hip. Whatever request header drove the server's choice must be echoed in Vary, or a shared cache will hand the wrong language or encoding to the next user. Being able to read a real Accept string with q-values, and explain the Vary consequence, is the crux of the question.

; request
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Accept-Language: en-IN,en;q=0.9,hi;q=0.8

; response — tell caches this body depends on the language asked for
Content-Language: en-IN
Vary: Accept-Language

Compression: gzip, brotli, zstd

End-to-end compression shrinks text payloads — HTML, CSS, JS, JSON — by 70% or more before they cross the network. It is a negotiation: the client sends Accept-Encoding: br, gzip, zstd listing the algorithms it understands, and the server responds with Content-Encoding naming the one it used. gzip is the universal baseline; Brotli (br) generally beats gzip on text and is now widely deployed; zstd (Zstandard) is the newer entrant offering strong ratios with fast decode. Because the same URL now has multiple byte representations, the response must carry Vary: Accept-Encoding.

Why it matters: the expected talking points are "Brotli over gzip for text where the browser supports it," never compress already-compressed media (JPEG, PNG, MP4 — you waste CPU for near-zero gain), and remember the Vary: Accept-Encoding so a proxy does not send a Brotli body to a gzip-only client. It is a fast, high-impact performance win interviewers expect you to know cold.

; request
Accept-Encoding: br, gzip, zstd

; response
Content-Encoding: br
Vary: Accept-Encoding

Interview questions

Most-asked HTTP internals questions in Indian interviews — HTTP/2 multiplexing, caching headers, SameSite cookies, and HTTP/3/QUIC are increasingly standard, from service-company screens to product-company deep dives:

Beginner What are the main differences between HTTP/1.1 and HTTP/2?

HTTP/1.1 is a text protocol handling one request/response per connection at a time, forcing browsers to open several parallel TCP connections. HTTP/2 is binary, multiplexes many independent streams over a single connection, and compresses headers with HPACK — so the same connection carries everything concurrently without the parallel-connection hack.

Beginner What is an HTTP keep-alive (persistent) connection and why does it help?

It keeps the TCP connection open across successive requests instead of reopening one each time. That avoids repeating the TCP handshake and TCP slow-start, cutting latency. It is the default in HTTP/1.1, controlled by the Connection and Keep-Alive headers.

Beginner Cache-Control: no-cache vs no-store — what is the difference?

no-store means never write this to any cache. no-cache means you may store it but must revalidate with the origin (via a conditional request) before every reuse — so it can still return a fast 304. "no-cache" does not mean "do not cache"; that is the common trap.

Beginner What is an ETag and how is it used?

An ETag is a version identifier the server attaches to a response. On the next request the client sends it back in If-None-Match; if the resource is unchanged the server replies 304 Not Modified with no body, saving bandwidth. Weak tags (W/"...") assert semantic, not byte-for-byte, equivalence.

Beginner What do the Secure and HttpOnly cookie attributes do?

Secure sends the cookie only over HTTPS. HttpOnly hides it from JavaScript (document.cookie), which blocks an XSS script from stealing a session cookie. They defend different things and are usually set together on session cookies.

Intermediate Explain HTTP/2 multiplexing. Which HTTP/1.1 problem does it solve?

HTTP/2 splits each message into frames tagged with a stream id, so many requests and responses interleave over one connection and can complete out of order. This removes HTTP/1.1's application-layer head-of-line blocking, where a slow response held up everything queued behind it, and makes domain sharding and six-connection tricks obsolete.

Intermediate Walk through a conditional request. What drives a 304 Not Modified?

When a cached response goes stale, the client revalidates: it sends If-None-Match with the stored ETag (or If-Modified-Since with the stored Last-Modified). If the resource is unchanged the server returns 304 with headers only and no body, and the client reuses its cached copy; otherwise it returns 200 with fresh content.

Intermediate Explain SameSite=Strict/Lax/None and the current default.

Strict sends the cookie only on same-site requests. Lax also sends it on top-level cross-site navigations (following a link) but not on cross-site subresource or POST requests — the modern browser default when SameSite is unset. None sends it on all cross-site requests and must be combined with Secure. SameSite is a primary CSRF mitigation.

Intermediate How does content negotiation work, and why does Vary matter to caches?

The client advertises preferences via Accept, Accept-Language, and Accept-Encoding, ranked by ;q= quality values, and the server picks a representation. Whatever request headers influenced that choice must be listed in the response's Vary header, so a shared cache stores a separate entry per variant and never serves the wrong language or encoding.

Intermediate gzip vs Brotli vs zstd — how do client and server agree, and when do you skip compression?

The client lists what it supports in Accept-Encoding; the server confirms its pick in Content-Encoding and adds Vary: Accept-Encoding. gzip is the universal baseline, Brotli usually wins on text, zstd is the newer fast option. Skip compression on already-compressed media (JPEG, PNG, MP4) — it burns CPU for almost no size reduction.

Pro HTTP/2 multiplexes over one connection but can still stall — why, and how does HTTP/3/QUIC fix it?

HTTP/2 runs on TCP, which guarantees in-order delivery, so one lost packet blocks every multiplexed stream until it is retransmitted — transport-layer head-of-line blocking. HTTP/3 runs on QUIC (over UDP) with per-stream loss recovery, so a lost packet only stalls its own stream. QUIC also folds in TLS 1.3 for a faster handshake and supports connection migration across network changes.

Pro Design a secure session cookie. Why does each attribute map to a specific attack, and what does __Host- add?

Store an opaque server-side session id, not user data. Mark it HttpOnly (an XSS script cannot read it), Secure (never leaks over plain HTTP), and SameSite=Lax or Strict (blocks CSRF via cross-site requests). SameSite=None would require Secure. The __Host- prefix forces Secure, Path=/, and no Domain, so a compromised subdomain cannot overwrite it — defence in depth for the most sensitive cookie you own.
