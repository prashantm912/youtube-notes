Module: HTTP Essentials

Every page load is a short conversation in HTTP — a method, a status code, a stack of headers, and, since the web went encrypted, a TLS handshake underneath. This module maps that vocabulary and points you to the references worth bookmarking for each part.

Knowledge Library → Courses → How the Web Works → HTTP Essentials

What this module covers

HTTP is the request-response protocol the whole web runs on. This module orients you on the five things every developer is expected to know cold: the request methods and their safe/idempotent contract, the status-code families and the specific codes you meet daily, the headers that carry the metadata, the redirect codes and when each applies, and how HTTPS and TLS 1.3 wrap all of it in encryption. Depth on the wire format and versions belongs to the next module; this one gives you the working vocabulary.

Topics

HTTP methods and their semantics

A method (or "verb") states the intent of a request. GET retrieves a resource, POST submits data to create or trigger processing, PUT replaces a resource wholesale, PATCH applies a partial update, DELETE removes it, HEAD is GET without a body (headers only), and OPTIONS asks what a resource or server supports — the method behind CORS preflight.

Why it matters: two properties drive most interview questions and real design decisions. A method is safe if it only reads and changes nothing on the server (GET, HEAD, OPTIONS). It is idempotent if calling it once or many times leaves the same end state — all safe methods plus PUT and DELETE. POST and PATCH are neither guaranteed safe nor idempotent, which is exactly why a retried POST can create duplicates. Idempotency is what lets browsers, proxies, and clients safely retry after a timeout.

GET    /users/42   HTTP/1.1   # safe, idempotent, cacheable
DELETE /users/42   HTTP/1.1   # idempotent: repeat leaves "gone"
PUT    /users/42   HTTP/1.1   # idempotent: same body, same result
POST   /users      HTTP/1.1   # neither safe nor idempotent

Status codes

The response status code is a three-digit number whose first digit sets its class: 1xx informational (rare — 100 Continue), 2xx success (200 OK, 201 Created, 204 No Content), 3xx redirection (301, 302, 304 Not Modified), 4xx client errors (400, 401, 403, 404, 409, 422, 429), and 5xx server errors (500, 502, 503). Reading the family first tells you who is responsible before you look up the exact code.

Why it matters: choosing the right code is how an API communicates. 201 with a Location header signals a create; 204 means "done, nothing to return"; 400 is malformed input while 422 is well-formed but semantically invalid; 401 is "not authenticated" versus 403 "authenticated but not allowed"; 409 flags a conflict; 429 is rate limiting; 502/503 distinguish a bad upstream from a temporarily unavailable service. Interviewers love scenario questions here.

HTTP/1.1 200 OK                     # success, body follows
HTTP/1.1 201 Created                # Location: /users/42
HTTP/1.1 304 Not Modified           # use your cached copy
HTTP/1.1 404 Not Found              # resource does not exist
HTTP/1.1 500 Internal Server Error  # the server broke

HTTP headers overview

Headers are key-value metadata attached to every request and response. Request headers describe the client and what it wants — Host, Accept, Authorization, User-Agent, Cookie. Response headers describe what is being returned — Content-Type, Content-Length, Cache-Control, Set-Cookie, Location. Some, like Content-Type, appear on both sides.

Why it matters: headers drive content negotiation (Accept vs Content-Type), authentication (Authorization), caching (Cache-Control, ETag), sessions (Cookie/Set-Cookie), and security (CORS, HSTS, CSP). Most "why is my API returning the wrong thing?" bugs are a mismatched or missing header, so knowing the common ones by heart pays off immediately.

GET /api/users HTTP/1.1          # request
Host: example.com
Accept: application/json         # what the client can handle
Authorization: Bearer <token>

HTTP/1.1 200 OK                  # response
Content-Type: application/json   # what the server is sending
Cache-Control: max-age=3600

Redirects

A 3xx response with a Location header tells the client to fetch the resource elsewhere. The four that matter: 301 Moved Permanently and 302 Found are the classic pair, but both historically let clients rewrite a POST into a GET. 308 Permanent Redirect and 307 Temporary Redirect were added to guarantee the method and body are preserved. So 301≈308 (permanent) and 302≈307 (temporary), with the "7/8" variants never changing the method. (303 See Other is the deliberate opposite — it forces a switch to GET, ideal after a POST.)

Why it matters: permanence has SEO consequences. A 301/308 tells search engines the new URL is canonical and passes link equity — use it for domain moves, HTTP-to-HTTPS upgrades, and restructures. A 302/307 keeps the original URL indexed — use it for temporary maintenance pages, A/B tests, and login flows. Mixing these up, or building long redirect chains that add a round trip each hop, is a common performance and ranking bug.

HTTP/1.1 301 Moved Permanently
Location: https://example.com/new-path   # permanent; SEO equity moves

HTTP/1.1 302 Found
Location: https://example.com/login       # temporary; original stays canonical

HTTPS and TLS 1.3

HTTPS is plain HTTP carried over a TLS-encrypted connection. TLS gives three guarantees at once: confidentiality (the traffic is encrypted so no one on the network can read it), integrity (tampering is detected), and authentication (a certificate signed by a trusted Certificate Authority proves you are really talking to that domain). The current version is TLS 1.3, and browsers now flag plain http:// as "not secure".

Why it matters: the handshake is a favourite senior-round question. In TLS 1.3 the client's ClientHello already carries its key-share (its ephemeral Diffie-Hellman parameters) and supported cipher suites; the server responds with its own key-share, its certificate, and a signature proving it holds the matching private key. Both sides derive the same session key, the client validates the certificate chain and hostname, and encrypted application data flows — all in a single round trip (with 0-RTT resumption possible), which is why TLS 1.3 is both faster and safer than earlier versions.

Interview questions

The HTTP questions Indian interviews reach for first — from support-desk and service-company screens to product-company system-design rounds:

Beginner What is the difference between GET and POST?

GET retrieves a resource and carries parameters in the URL/query string; it is safe (read-only), idempotent, cacheable, and its data is visible in browser history and server logs, with a practical length limit. POST submits data in the request body to create or trigger processing; it is neither safe nor idempotent, is not cached by default, and suits large or sensitive payloads. Rule of thumb: reads and bookmarkable navigation use GET, state-changing submissions use POST.

Beginner What do the five status-code families mean?

The first digit sets the class: 1xx informational (rare, e.g. 100 Continue), 2xx success (200, 201, 204), 3xx redirection (301, 302, 304), 4xx client error (400, 401, 403, 404, 429), 5xx server error (500, 502, 503). Knowing the family tells you who is at fault before you read the exact code.

Beginner How is 404 different from 500?

404 is a 4xx client error — the server is healthy but the requested resource does not exist (mistyped or dead URL, deleted page). 500 is a 5xx server error — the request was reasonable but the server failed while handling it (unhandled exception, bad config). In short: 404 means "your address is wrong", 500 means "my code broke".

Beginner What is an HTTP header? Give request vs response examples.

Headers are key-value metadata attached to a request or response. Request headers describe the client and its wants — Host, Accept, Authorization, User-Agent, Cookie. Response headers describe the returned resource — Content-Type, Content-Length, Cache-Control, Set-Cookie, Location. A few, like Content-Type, appear on both sides.

Intermediate What does it mean for a method to be idempotent, and which methods are?

An idempotent method leaves the same server state whether it is called once or many times. GET, HEAD, OPTIONS (all also safe), plus PUT and DELETE are idempotent; POST and PATCH are not guaranteed to be. Idempotency is what lets a client or proxy safely retry a request after a timeout without risking duplicate side effects.

Intermediate Safe vs idempotent — what is the distinction?

Safe means read-only, no server state change at all (GET, HEAD, OPTIONS). Idempotent means repeating the call reaches the same end state, even if it did change something the first time (PUT writes the same value; DELETE ends "gone" either way). All safe methods are idempotent, but the reverse is false — PUT and DELETE are idempotent yet not safe.

Intermediate PUT vs PATCH?

PUT replaces the entire resource — you send the whole representation, and it is idempotent. PATCH applies a partial modification — you send only the changed fields, and it is not guaranteed idempotent (an "increment by one" patch differs on repeat). Use PUT for full replacement, PATCH for partial updates.

Intermediate 301 vs 302 — the difference and when to use each?

301 Moved Permanently says the resource has a new canonical URL; browsers cache it and search engines pass link equity to the target — use it for permanent moves (domain change, HTTP-to-HTTPS, restructure). 302 Found is temporary; the original URL stays canonical and indexed — use it for maintenance pages, A/B tests, or login redirects. Swapping them is a classic SEO bug.

Intermediate Which status code for: a create, an empty response, invalid input, not logged in, logged in but forbidden, and rate limited?

201 Created (with a Location header) for a successful create; 204 No Content when there is nothing to return (e.g. after a delete or save); 400 Bad Request for malformed input, or 422 Unprocessable Content for well-formed but semantically invalid data; 401 Unauthorized for missing/invalid authentication; 403 Forbidden when authenticated but not permitted; 429 Too Many Requests when rate-limited.

Pro A user double-clicks Submit and two identical orders are created. How do you fix it at the HTTP layer?

POST is not idempotent, so the second click genuinely creates a second order. The standard fix is an idempotency key: the client sends a unique key (an Idempotency-Key header) with the request; the server records it and returns the original result for any retry carrying the same key instead of creating a duplicate. Complement it with Post/Redirect/Get to stop refresh resubmits and a unique constraint in the database as a backstop.

Pro 401 vs 403, and 400 vs 422 — how do you decide?

401 Unauthorized means "not authenticated" — missing or bad credentials; the response should carry WWW-Authenticate and the client may retry with credentials. 403 Forbidden means "authenticated but not allowed" — re-authenticating will not help. 400 Bad Request is for requests the server cannot parse (malformed JSON, missing required field); 422 Unprocessable Content is for well-formed requests that fail business rules (e.g. an email already in use). Note 401's name says "authorization" but it is really about authentication.

Pro Why were 307 and 308 introduced when 301 and 302 already existed?

Historically, many clients receiving a 301 or 302 for a POST silently changed the method to GET and dropped the body — sometimes convenient, often a bug. 307 Temporary Redirect and 308 Permanent Redirect guarantee the method and body are preserved across the redirect. So 302≈307 (temporary) and 301≈308 (permanent), with the "7/8" variants never rewriting the method — use them when redirecting non-GET requests. (303 See Other is the deliberate opposite: it forces a switch to GET, useful after a POST.)

Pro Walk through what happens, security-wise, when you open an https:// site (the TLS 1.3 handshake).

After DNS resolution and the TCP connect, the client and server run a TLS handshake. In TLS 1.3 the client's ClientHello already lists its supported TLS version and cipher suites and includes a key-share — its ephemeral Diffie-Hellman public parameters — guessing the server's preferred group to save a round trip. The server replies with a ServerHello (chosen cipher and its own key-share), its certificate, and a CertificateVerify signature proving it holds the matching private key; both sides now derive the same session key. The client validates the certificate chain up to a trusted CA and checks the hostname, sends Finished, and encrypted application data flows — all in one round trip, with 0-RTT resumption possible. The net result is confidentiality, integrity, and authentication, which is why HTTPS is the default and plain HTTP is marked "not secure".
