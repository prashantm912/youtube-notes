Module: Modern Selectors and Features

The newest layer of CSS: selectors that finally look inside and around an element, at-rules that gate features and animate whole page changes, and a maturing toolbox of native functions — most of it Baseline, a little still landing.

Knowledge Library → Courses → CSS → Modern Selectors and Features

What this module covers

This module covers the current wave of CSS: native nesting, relational and scoped selection (:has(), @scope), feature detection with @supports, the CSS surface of the View Transitions API (@view-transition, @starting-style), and the emerging function layer (@property, @function, if()) alongside a census of the wider at-rule family. Most of these ship in every major browser today; a few are still experimental, so knowing what is Baseline versus what is not is itself an interview signal.

Topics

CSS nesting

Native nesting lets you write a rule inside another rule, keeping a component's styles co-located instead of repeating the parent selector on every line. A descendant nests directly; the nesting selector & refers back to the parent so you can attach pseudo-classes, build compound selectors, or reverse the context. Media and support queries nest too. It is Baseline across browsers, so no build step or preprocessor is required.

Why it matters: it removes the single biggest reason teams reached for Sass, but it is not a drop-in copy of Sass. Nested type selectors behave as relative descendants, string concatenation of class names is impossible, and each nested selector's specificity is computed as if wrapped in :is() — which takes the highest specificity in the list. Interviewers probe exactly these gaps.

.card {
  color: #111;
  & .title { font-weight: 700; }   /* & is the nesting (parent) selector */
  &:hover { color: #06c; }
  @media (width >= 40rem) { padding: 2rem; }
}

:has() patterns, @scope

:has() is the relational pseudo-class — it styles an element based on what it contains, what follows it, or the state of something inside it. It is the long-awaited "parent selector" and much more: highlight a form field whose input is invalid, lay out a card differently when it holds an image, or build quantity queries that react to the number of children. It has been Baseline since late 2023.

@scope answers the other half of selection — narrowing reach. It scopes a block of rules to a subtree, optionally with a lower boundary (donut scoping), and resolves conflicts by scoping proximity (nearest scope root wins) rather than by piling on specificity. That keeps component styles from leaking without !important or long selector chains. It is a newer arrival (Baseline 2025), so pair it with a fallback where support matters.

/* :has() — style a parent from its contents */
.field:has(input:invalid) { border-color: crimson; }
figure:has(figcaption) { gap: .5rem; }

/* @scope — limit rules to a subtree, with a lower boundary */
@scope (.card) to (.card__footer) {
  a { color: rebeccapurple; }
}

@supports, feature detection

@supports (feature queries) tests whether the browser understands a declaration before applying rules that depend on it. The idiom is progressive enhancement: write a safe baseline, then upgrade inside an @supports block. You can test property/value pairs, selectors via selector(), and font capabilities, and combine conditions with and, or, and not. The same logic is available to scripts through CSS.supports().

Why it matters: it is the correct answer to "how do you use a cutting-edge feature without breaking older browsers?" Note the common trap — @supports can test whether a selector like :has() is supported, but it cannot detect every rendering bug, so it is a support gate, not a bug shield.

@supports (display: grid) {
  .layout { display: grid; }
}
@supports selector(:has(*)) {   /* only where :has() is supported */
  .card:has(img) { padding: 0; }
}
@supports not (aspect-ratio: 1) { /* fallback path here */ }

View transitions (CSS side, @view-transition, @starting-style)

The View Transitions API animates between two DOM states — within a single-page app via document.startViewTransition(), or across same-origin documents in a multi-page app. The browser snapshots the before and after states and cross-fades them, and CSS drives the choreography: view-transition-name tags an element as persistent across the change, and the ::view-transition-* pseudo-elements style the generated animation. The @view-transition at-rule opts a page into cross-document transitions with navigation: auto.

@starting-style is the partner feature for entry animations: it defines the values an element should animate from the first time it renders or when it moves from display: none to visible. Without it there is no "before" frame to interpolate, so popovers, dialogs and toasts would simply pop in. Together these turn transitions that once needed JavaScript timers into declarative CSS.

/* opt in to cross-document (MPA) transitions */
@view-transition { navigation: auto; }

/* mark an element as persistent across the transition */
.hero { view-transition-name: hero; }

/* animate in from a defined start state */
.toast {
  opacity: 1;
  @starting-style { opacity: 0; }
}

@property, custom @function, if(), at-rule census

@property registers a custom property with a type (syntax), an inheritance flag, and an initial value. Typing is what makes a variable animatable: the browser only interpolates a custom property it knows is, say, an <angle> or a <color>, which is how you animate a gradient's angle or a single channel. Beyond it sit two experimental arrivals: @function defines reusable custom functions that take parameters and return a value, and if() supplies inline conditional values driven by style, media, or feature queries. Both have limited browser support today, so treat them as forward-looking.

The module also asks you to place these inside the broader at-rule family. Worth a mental census: @import pulls in other stylesheets and can carry layer() and supports() conditions; @page styles the page box for print; @font-feature-values gives friendly names to OpenType features used by font-variant-alternates. MDN's at-rules index is the map of the whole set — statement at-rules that end in a semicolon versus block at-rules that carry a body.

@property --angle {
  syntax: "<angle>";
  inherits: false;
  initial-value: 0deg;
}
.spinner { background: conic-gradient(from var(--angle), #06c, #0cf); }
/* because --angle is typed, it can now be transitioned or animated */

Interview questions

Modern CSS is a fast-moving favourite in Indian front-end interviews, from service-company screens to product-company deep dives — because it separates "learned it in a bootcamp" from "keeps up with the platform":

Beginner What is native CSS nesting and what does the & selector do?

Nesting lets you write a rule inside another so component styles stay together without repeating the parent selector. The & nesting selector references the parent — needed for compound selectors and pseudo-classes like &:hover, and usable to reverse context. It is Baseline, so no preprocessor is required.

Beginner What does :has() do? Give one example.

It selects an element based on its descendants, following siblings, or their state — the relational or "parent" selector. Example: .card:has(img) targets cards that contain an image; label:has(input:checked) styles a label whose checkbox is checked. Baseline since late 2023.

Beginner What is @supports for?

It is a feature query: apply CSS only if the browser supports a given declaration or selector. The pattern is progressive enhancement — a safe baseline, then enhancements inside @supports (…). Conditions combine with and, or, not; CSS.supports() exposes the same check to JavaScript.

Beginner What does @property add over a plain custom property?

It registers the variable with a syntax type, an inherits flag, and an initial-value. That gives type checking, a guaranteed fallback, and — most importantly — the ability to animate or transition the value, which un-typed --vars cannot do.

Beginner What is the View Transitions API at a high level?

It animates between two DOM states by snapshotting the before and after and cross-fading them. It works inside a single page via document.startViewTransition() and across same-origin pages for multi-page apps. CSS controls the animation through view-transition-name and the ::view-transition-* pseudo-elements.

Intermediate How does native nesting differ from Sass nesting?

No build step, and it runs at the cascade level, not as text expansion. Key gotchas: you cannot concatenate class name strings (no &__title BEM trick), a bare nested type selector reads as a relative descendant, and nested selectors take :is() specificity — the highest in the list — which can surprise you. Sass does none of this at runtime.

Intermediate How is specificity calculated for :has() and for nested selectors?

:has() itself adds no specificity, but its most specific argument does — like :is() and :not(). Nested rules are likewise evaluated as though wrapped in :is(), so a nesting group takes the specificity of its heaviest selector. This is why an unexpected id or heavy selector in a list can raise the whole rule's weight.

Intermediate What problem does @scope solve, and how is scoping proximity different from specificity?

@scope limits rules to a subtree, optionally between an upper and lower boundary (donut scoping), so component styles do not leak. When two scoped rules of equal specificity conflict, the one whose scope root is nearest the element in the DOM wins — proximity, not specificity, breaks the tie. That avoids the !important and long-selector escalation that plagued component CSS.

Intermediate Same-document vs cross-document view transitions — and what does @view-transition do?

Same-document (SPA) transitions wrap a DOM update in document.startViewTransition(). Cross-document (MPA) transitions fire on same-origin navigations and are opted in purely with CSS: @view-transition { navigation: auto; } on both pages. In both cases elements sharing a view-transition-name are matched and animated between states.

Intermediate What is @starting-style and why is it needed for entry animations?

It declares the values an element animates from on first render or when changing from display: none to visible. A transition needs a start and an end frame; a freshly shown element has no prior state, so without @starting-style there is nothing to interpolate and it just appears. It pairs with transition-behavior: allow-discrete to animate discrete properties like display.

Pro Why does @property make a custom property animatable — give a concrete use?

The engine can only interpolate a value whose type it knows. Registering --angle as syntax: "<angle>" tells it the value is an angle, so a transition or keyframe can smoothly sweep it — for example animating conic-gradient(from var(--angle) …) to spin a gradient, or animating one channel of a color. An un-registered variable is treated as a plain string and snaps instead of tweening.

Pro How would you ship a modern feature with a safe fallback using @supports?

Write the baseline first, then gate the upgrade: @supports selector(:has(*)) { … } or @supports (display: grid) { … }, optionally with @supports not (…) for the fallback branch. In JavaScript, mirror it with CSS.supports('selector(:has(*))'). Caveat: feature queries confirm the browser parses a feature, not that it is bug-free — so pair critical features with real testing.

Pro What are @function and if(), and how do they differ from @property and var()?

@function defines a reusable custom function that takes typed parameters and returns a computed value; if() chooses a value inline based on style(), media(), or supports() conditions. Both are experimental with limited support today. They differ from @property/var(), which store and type a single value: functions compute and branch, whereas custom properties hold state. In interviews, flag the maturity gap — @property is Baseline, these two are not yet.
