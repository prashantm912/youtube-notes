Module: UI and Interaction Details

These are the small properties that decide how a page feels in the hand — smooth snapping, native-looking form controls, and pointer behaviour that does exactly what the user expects. None of them lay out a page, but together they are the difference between "works" and "polished".

What this module covers

This module gathers the interaction-layer properties that sit on top of layout: how scrolling snaps and chains, how the browser's built-in form controls take your brand colour and dark-mode preferences, and how the page responds to the mouse, touch and text selection. Each is a one- or two-line addition that most tutorials skip, yet they surface constantly in real UI work and in interviews about "how would you do this without JavaScript?".

Topics

Scroll control: scroll-behavior, scroll-snap, overscroll-behavior and CSS carousels

Four features shape how a scroll container behaves. scroll-behavior: smooth animates programmatic and anchor-link scrolls instead of jumping. CSS scroll snapping (scroll-snap-type on the container plus scroll-snap-align on children, tuned with scroll-padding and scroll-margin) makes a gallery or slider land cleanly on each item with zero script. overscroll-behavior controls what happens at the edge — contain stops a nested scroller from chaining its momentum to the page, and it is the standard fix for accidental pull-to-refresh and back-navigation on mobile.

The newest piece is CSS carousels: the ::scroll-marker and ::scroll-button() pseudo-elements generate accessible dot indicators and prev/next buttons for a scroll-snap container without any JavaScript. They are powerful but still experimental — at the time of writing MDN marks both as "limited availability" (not yet Baseline), so treat them as progressive enhancement and always verify the compatibility table before shipping.

.gallery {
  overflow-x: auto;
  scroll-snap-type: x mandatory;   /* snap horizontally, always */
  scroll-behavior: smooth;         /* animate programmatic scrolls */
  overscroll-behavior-x: contain;  /* don't chain to page/history */
}
.gallery > .slide { scroll-snap-align: center; }

Form control styling: accent-color, color-scheme, appearance, field-sizing and caret-color

Native form controls used to be almost unstyleable; these properties change that. accent-color tints checkboxes, radio buttons, range sliders and progress bars with a single declaration while keeping the browser's built-in accessibility. color-scheme tells the browser which themes an element supports (light dark), so scrollbars, form controls and other UA chrome flip to their dark variants automatically. appearance: none strips a control's native look entirely when you want to build it from scratch.

Two more fill common gaps. field-sizing: content lets an <input> or <textarea> grow and shrink to fit what the user types — the auto-resizing textarea that previously needed JavaScript (Baseline as of 2026). caret-color sets the colour of the blinking text-insertion cursor, a small touch that matters on dark or brand-coloured inputs.

:root { color-scheme: light dark; }        /* opt into native dark UI */
input[type="checkbox"],
input[type="range"] { accent-color: rebeccapurple; }
textarea { field-sizing: content; }        /* grow to fit typed text */
input    { caret-color: hotpink; }

Interaction affordances: cursor, user-select, pointer-events, resize and touch-action

This group tunes how input reaches your elements. cursor signals what an element does (pointer, grab, not-allowed, or a custom image with a required keyword fallback). user-select: none stops text from being selected — useful on buttons and UI chrome, but never on real content. pointer-events: none makes an element transparent to clicks, hover and other pointer input, so events pass straight through to whatever is beneath; setting it back to auto on a child re-enables that child.

resize gives users a drag handle to resize an element (it needs overflow other than visible to work). touch-action declares which touch gestures the browser may handle on an element — manipulation allows pan and pinch-zoom while removing the legacy double-tap delay, and none hands all gesture handling to your own code (as custom drag or drawing surfaces need). Both pointer-events and touch-action carry accessibility caveats worth stating precisely in interviews.

.map-pin      { cursor: url(pin.svg) 4 4, pointer; } /* image + fallback */
.overlay      { pointer-events: none; }   /* clicks pass through */
.copy-guard   { user-select: none; }
.panel        { resize: vertical; overflow: auto; }
button        { touch-action: manipulation; } /* drop double-tap delay */

Interview questions

UI-detail questions that come up in Indian front-end and product-company interviews — often phrased as "how would you do this without JavaScript?":

Beginner What does scroll-behavior: smooth do, and where do you set it?

It animates scrolls triggered by navigation (anchor links) or scripting APIs (scrollIntoView, scrollTo) instead of jumping instantly. Set it on the scroll container — usually html for page-level anchor smoothing. It does not affect the user's own wheel/touch scrolling. Respect prefers-reduced-motion when using it.

Beginner How do you build a snapping image gallery with pure CSS?

Put scroll-snap-type: x mandatory on a horizontally scrollable container and scroll-snap-align: start (or center) on each child. The browser then snaps each item into place as the user scrolls or swipes. Use scroll-padding on the container to offset for a sticky header.

Beginner What is accent-color and which elements does it affect?

A single property that tints the browser's built-in form controls — checkboxes, radio buttons, range sliders and progress bars — to your brand colour while keeping their native behaviour and accessibility. It saves rebuilding those controls from scratch just to change one colour.

Beginner How do you stop users from selecting text on an element?

user-select: none. Reasonable on buttons, icons and UI chrome where selection is noise; avoid it on real readable content, since it also blocks copy for legitimate users. It is a convenience, not a security measure — the text is still in the DOM.

Beginner What does pointer-events: none do?

The element stops being a target for pointer input (click, hover, etc.) and events pass through to whatever sits beneath it. Common for decorative overlays and disabled-looking controls. A child can opt back in with pointer-events: auto.

Intermediate What problem does overscroll-behavior solve?

It controls scroll chaining and browser overscroll effects at a boundary. overscroll-behavior: contain keeps scrolling inside a nested area (a modal, chat pane or map) from bubbling to the page once it hits its edge, and it prevents accidental pull-to-refresh or back-swipe. none also suppresses the bounce/glow effect.

Intermediate color-scheme vs the prefers-color-scheme media query — what is the difference?

prefers-color-scheme is an input: a media query that reads the user's OS preference so you can write your own dark styles. color-scheme is an output/declaration: it tells the browser which schemes your element supports so UA-rendered chrome — form controls, scrollbars, default backgrounds — switches to its dark variant automatically. You typically use both together.

Intermediate When would you use appearance: none, and what must you remember afterwards?

When you want to remove a control's native OS styling to build a fully custom look (a custom select, checkbox or search field). The catch: stripping the native appearance also removes native affordances and focus styling, so you must re-add visible focus states, states like checked/disabled, and keyboard behaviour yourself — otherwise you break accessibility.

Intermediate What is touch-action: manipulation and why use it?

It tells the browser the element supports panning and pinch-zoom but nothing else — which removes the legacy ~300ms double-tap-to-zoom delay on tap targets, making buttons feel instant on touch devices. Use touch-action: none instead when you handle all gestures yourself, e.g. a custom drag, slider or canvas.

Intermediate How do you make a textarea grow to fit its content without JavaScript?

field-sizing: content on the <textarea> (or <input>) lets it size to what the user types instead of a fixed default, replacing the old scrollHeight-measuring JS. It reached Baseline in 2026, so provide a sensible fixed fallback height for older browsers.

Intermediate Two smaller ones: what does caret-color do, and what does resize need to work?

caret-color sets the colour of the blinking text-insertion cursor in inputs and contenteditable elements — useful on dark or coloured fields. resize gives users a drag handle (horizontal, vertical or both), but it only takes effect when overflow is something other than visible.

Pro pointer-events: none vs the disabled attribute — and the accessibility trap?

pointer-events: none only blocks pointer input; the element can still receive keyboard focus via Tab and be activated by Enter/Space, and it is not announced as disabled. The disabled attribute removes the control from the tab order and the accessibility tree and blocks all activation. For a genuinely disabled control use disabled (or aria-disabled plus handling); use pointer-events: none for visual/overlay effects, not to disable interactive controls.

Pro What are CSS carousels (::scroll-marker / ::scroll-button) and what is the catch with using them today?

They are pseudo-elements that generate accessible carousel UI directly from a scroll-snap container: ::scroll-marker creates the dot indicators (grouped in ::scroll-marker-group) and ::scroll-button() creates prev/next buttons — no JavaScript, with built-in focus and active-state handling. The catch: they are experimental and not yet Baseline (MDN lists limited availability), so they only work in some browsers. Ship them as progressive enhancement over a working scroll-snap base and check the compatibility table first.
