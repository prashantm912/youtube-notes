Module: CSS Selectors — Basics and Combinators

A stylesheet is only as good as its aim. This module maps the everyday selectors — element, class, id, attribute — and the combinators that let you target elements by their place in the tree, then points you to the best material to master each.

Knowledge Library → Courses → CSS → Selectors: Basics and Combinators

What this module covers

Selectors are the "which elements?" half of every CSS rule. This module covers the four simple selectors (type, class, id, universal), attribute selectors with their full set of matching operators, the four combinators that select by relationship in the document tree, and selector lists that group several selectors under one declaration block. Master these before pseudo-classes and pseudo-elements — they are the vocabulary everything else extends.

Topics

Type, class, id and universal selectors

These are the four simple selectors you reach for constantly. A type selector matches by tag name (p, h2); a class selector (.btn) matches any element carrying that class and is the reusable workhorse of real stylesheets; an id selector (#hero) targets the single element with that id; and the universal selector (*) matches every element.

Why it matters: the choice encodes intent and specificity. Classes are reused and carry moderate weight; ids are unique and heavy (avoid them for styling in large codebases precisely because they are hard to override); the universal selector powers the near-universal box-sizing reset. Explaining when to prefer a class over an id is a standard early screen.

h2           { color: navy; }            /* type: every h2 element */
.btn         { padding: 8px 12px; }      /* class: reuse across elements */
#site-header { position: sticky; }       /* id: one unique element */
*            { box-sizing: border-box; } /* universal: every element */

Attribute selectors (all operators)

Attribute selectors match on an element's attributes and their values, using square-bracket syntax. [disabled] matches presence; [type="text"] matches an exact value; and a family of substring operators matches partial values — ^= (starts with), $= (ends with), *= (contains), plus ~= (a whitespace-separated word) and |= (an exact value or a hyphen-prefixed one, handy for language subtags).

Why it matters: they let you style by data rather than by adding classes — flagging external links (a[href^="https"]), file types (a[href$=".pdf"]), or form-control state. An appended i makes matching case-insensitive. They carry class-level specificity, and knowing the operator set is a frequent intermediate question.

a[target]            { color: teal; }      /* has the attribute */
a[href^="https"]     { font-weight: 600; }  /* value starts with https */
a[href$=".pdf"]      { color: crimson; }    /* value ends with .pdf */
a[href*="youtube"]   { color: red; }        /* value contains youtube */
input[type="text" i] { border: 1px solid; } /* i = case-insensitive */

Combinators (descendant, child, adjacent, general sibling)

Combinators join two selectors so the match depends on the elements' relationship in the tree. The descendant combinator (a space) matches an element nested anywhere inside another; the child combinator (>) matches only direct children; the next-sibling / adjacent combinator (+) matches an element immediately following its sibling; and the subsequent-sibling / general combinator (~) matches every following sibling.

Why it matters: combinators express structure without extra classes — spacing the paragraph after a heading (h2 + p), styling only top-level menu items (nav > ul > li). A common pitfall is over-using the descendant combinator, which is expensive and fragile; interviewers often ask you to distinguish + from ~ or a space from >.

nav a   { text-decoration: none; } /* descendant: any a inside nav */
ul > li { list-style: square; }    /* child: only direct li children */
h2 + p  { margin-top: 0; }         /* next-sibling: p right after an h2 */
h2 ~ p  { color: gray; }           /* subsequent-sibling: every later p */

Selector lists and grouping

A selector list is several selectors separated by commas that share one declaration block, so h1, h2, h3 { … } applies the same rule to all three. It keeps stylesheets DRY and mirrors how you would say the rule aloud: "for headings one through three."

Why it matters: there is a sharp gotcha — in a plain selector list, a single invalid or unsupported selector invalidates the entire rule, so one unknown selector silently drops all the styles. The forgiving pseudo-classes :is() and :where() (covered in the next module) exist precisely to escape that all-or-nothing behavior. This trade-off is a favorite senior-level question.

h1, h2, h3           { font-family: system-ui; }  /* one rule, three types */
.card, .panel, #hero { border-radius: 8px; }      /* mix selector types freely */

Interview questions

Most-asked selector questions in Indian interviews, from front-end screening rounds to product-company deep dives:

Beginner What is the difference between a class selector and an id selector?

A class (.name) can be applied to many elements and reused across the page; an id (#name) must be unique to one element. In specificity an id far outweighs a class, so for styling most teams use classes and reserve ids for anchors/JS hooks — heavy id selectors are hard to override later.

Beginner What does the universal selector do, and one common use?

* matches every element. Its classic use is the global reset *, *::before, *::after { box-sizing: border-box; }, or zeroing default margins. It has zero specificity, so it is trivially overridden by any other selector.

Beginner How do you apply the same rule to several selectors at once?

Use a selector list — comma-separate them: h1, h2, .lead { margin: 0; }. It is equivalent to writing the block three times but keeps the stylesheet DRY.

Beginner What is a type (element) selector?

A selector that matches by tag name — p selects every paragraph, a every link. It matches all elements of that type regardless of class or position, and has the lowest non-zero specificity.

Beginner Which wins: a class selector or a type selector, and why?

The class. Specificity is compared as (id, class/attribute/pseudo-class, type/pseudo-element); a class scores in a higher column than a type, so .title (0-1-0) beats h1 (0-0-1) even though the type selector may appear later.

Intermediate Descendant combinator versus child combinator?

ul li (space) matches any li nested at any depth inside a ul; ul > li matches only li elements that are direct children. The child combinator is stricter and does not reach into nested lists — useful for menus where you want only the top level.

Intermediate Difference between the + and ~ combinators?

a + b (next-sibling) matches b only if it immediately follows a under the same parent. a ~ b (subsequent-sibling) matches every b that comes after a among the siblings, not just the first. Both require a shared parent and never look backwards.

Intermediate Write a selector for paragraphs that are direct children of an article, and one for the paragraph right after every h2.

article > p for direct-child paragraphs; h2 + p for the paragraph immediately following each h2. If you wanted all paragraphs after the heading instead, use h2 ~ p.

Intermediate Explain the attribute operators ^=, $= and *=.

[attr^="v"] matches values that start with v, [attr$="v"] values that end with v, and [attr*="v"] values that contain v anywhere. Example: a[href^="https"] flags secure links, a[href$=".pdf"] flags PDFs. Add a trailing i for case-insensitive matching.

Intermediate In a comma-separated selector list, what happens if one selector is invalid?

The whole rule is discarded — one unsupported or malformed selector drops the styles for all of them. This is why a single unknown pseudo-class can silently break a rule. :is() and :where() are "forgiving" and ignore only the bad selectors inside them, which is the modern workaround.

Pro Do combinators themselves add specificity? Compare div p with .intro.

The combinator symbols (space, >, +, ~) add nothing; only the simple selectors on either side count. So div p is two type selectors (0-0-2) while .intro is one class (0-1-0). The single class wins, because a class outranks any number of type selectors — a common surprise when a long descendant chain still loses to one class.

Pro Using only selectors, how would you style external links, download links, and required text inputs — without adding any classes?

Attribute selectors carry the data already: a[href^="http"]:not([href*="yourdomain.com"]) for external links, a[href$=".pdf"], a[download] for downloads, and input[type="text"][required] (chained attribute selectors, which combine like an AND) for required text fields. This "style by data, not by class" approach keeps markup clean, though it can hurt readability if overdone.
