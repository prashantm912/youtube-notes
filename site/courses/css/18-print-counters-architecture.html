Module: Print, Counters and Architecture

The finishing chapter of CSS: numbering content without touching HTML, making a page look right on paper, and naming things so a stylesheet survives a growing team. Small topics individually, but each carries a classic interview trap.

Knowledge Library → Courses → CSS → Print, Counters and Architecture

What this module covers

Three loosely related but frequently tested corners of CSS: counters and generated content (automatic numbering and injected ::before/::after text), print styles (making the browser's print/PDF output usable via @media print and @page), and CSS architecture (BEM naming, cascade layers, and the conventions that keep large stylesheets predictable). None needs deep theory — but each has a nuance interviewers reliably probe.

Topics

Counters (counter/counters/@counter-style), generated content, list-style

CSS counters are variables the browser maintains and renders: counter-reset initialises one, counter-increment steps it, and the counter()/counters() functions display it inside the content property. That content property is the engine of generated content — the text, quotes, or images CSS injects into pseudo-elements like ::before, ::after and ::marker. Alongside them, list-style (a shorthand for list-style-type, -image and -position) styles ordinary list markers, and @counter-style lets you define entirely custom numbering systems when the built-ins aren't enough.

Why it matters: this is how you number section headings, figures, legal clauses, or deeply nested outlines without hard-coding a single digit in the markup — and the numbering stays correct as content is added or reordered. The catch interviewers love: generated content is decorative. It lives outside the DOM, is generally not selectable, and its screen-reader support is inconsistent — so it must never hold information the user actually needs.

section { counter-reset: figure; }              /* start/reset a counter */
figure  { counter-increment: figure; }          /* +1 per figure */
figcaption::before {
  content: "Fig. " counter(figure) ". ";        /* render it */
}
ol.legal { counter-reset: clause; list-style: none; }
ol.legal li::marker { content: counters(clause, ".") " "; } /* 1  1.1  1.2 */

Print styles (@page, page breaks, print media)

Every browser can print a page or save it as PDF, and print styles control what that output looks like. Scope rules to print with @media print (or a stylesheet marked media="print"): the usual moves are hiding navigation, ads and buttons with display: none, revealing link destinations, and switching to ink-friendly colours. The @page at-rule sets the printed sheet itself — its size, orientation, and margins, with pseudo-classes such as :first, :left and :right. Where content splits across sheets is governed by the fragmentation properties break-before, break-after and break-inside, which supersede the legacy page-break-* aliases.

Why it matters: invoices, tickets, boarding passes, recipes and reports are still printed or archived to PDF, and an unstyled print is frequently unusable — a full-bleed nav bar and a black background that drains a cartridge. Print is also the one medium you can't test by resizing the window; you emulate print media in DevTools instead. And because support for paged-media and fragmentation features is uneven across engines, knowing which parts are reliable is itself the senior signal.

@media print {
  nav, .sidebar, .no-print { display: none; }
  a[href]::after { content: " (" attr(href) ")"; } /* show the URL */
  .invoice { break-after: page; }                   /* one invoice per sheet */
  table, figure { break-inside: avoid; }            /* don't split these */
}
@page { size: A4; margin: 2cm; }

CSS architecture (BEM, layers strategy, naming conventions)

CSS architecture is the set of conventions that keep a growing stylesheet from turning into a specificity war. BEM — Block, Element, Modifier — is a naming convention where a component is a block, its parts are block__element (double underscore), and its variants are block--modifier (double dash). Because every rule is a single class, specificity stays flat and equal, so components decouple and any rule can override another by source order alone. Cascade layers (@layer) add the other half: an explicit ordering for whole groups of rules — reset, base, components, utilities — so precedence stops depending on accidental source order or ever-climbing selectors. Broader methodologies (OOCSS, SMACSS, ITCSS) and plain file/section organisation round out the picture.

Why it matters: at scale the hard part of CSS isn't any single property — it's changing one thing without breaking three others. Flat specificity plus intentional layering is exactly how large codebases avoid the !important escalation that eventually makes a stylesheet unmaintainable. Interviewers ask about BEM as a baseline and, increasingly, about cascade-layer strategy, to gauge whether you've actually shipped CSS on a team.

/* BEM: single-class selectors keep specificity flat */
.card { }            /* block */
.card__title { }     /* element */
.card--featured { }  /* modifier */

/* Layer order decides precedence, not specificity or file order */
@layer reset, base, components, utilities;
@layer components { .card { padding: 1rem; } }

Interview questions

How counters, print styles and CSS architecture surface in Indian interviews — from UI-developer screening rounds to product-company system-design discussions:

Beginner What do the parts of a BEM class like card__title--featured mean?

Three pieces: card is the block (a standalone component), __title is an element belonging to that block (double underscore), and --featured is a modifier — a state or variant (double dash). Every selector is a single class, so specificity stays flat.

Beginner How do you number elements that aren't an ordered list?

Use CSS counters: counter-reset to initialise, counter-increment on each element to step it, then display it with content: counter(name) in a ::before. Any element — headings, figures, sections — can be auto-numbered this way.

Beginner How do you apply CSS only when the page is printed?

Wrap rules in an @media print { … } block, or link a separate stylesheet with media="print". You typically hide navigation and controls, expand hidden content, and switch to a light background with dark text.

Beginner How do you remove the default bullets from a list?

list-style: none (or list-style-type: none). One caveat worth mentioning: in Safari, removing list markers can drop the list's semantics for VoiceOver, so add role="list" back on the <ul> when the list still needs to read as a list.

Beginner How do you stop a table row or figure from splitting across two printed pages?

break-inside: avoid on that element (legacy alias: page-break-inside: avoid). It asks the engine not to insert a page break inside the box — support is best for simple block boxes and less reliable inside complex layouts.

Intermediate Difference between counter() and counters()?

counter(name) outputs a single counter's value. counters(name, ".") outputs every counter of that name currently in scope, joined by the separator — so nested lists render as 1, 1.1, 1.2.3 automatically. The plural form is the one used for outline-style numbering.

Intermediate Is text added via content: "…" selectable and accessible?

Generally no. Generated content isn't inserted into the DOM, is usually not selectable or copyable, and screen-reader exposure is inconsistent across browsers. Rule of thumb: it's for decoration and visual affordances only — never put information the user must read or copy in there.

Intermediate What does @page control, with an example?

It styles the printed sheet itself — size, orientation, and page margins — not the content flow. Example: @page { size: A4 landscape; margin: 2cm; }, with pseudo-classes like @page :first, :left and :right to target specific pages. Content styling still happens with normal selectors inside @media print.

Intermediate What problem do cascade layers (@layer) solve?

They give whole groups of rules an explicit precedence order that beats specificity and source order. Declaring @layer reset, base, components, utilities means a later layer always wins over an earlier one regardless of selector strength, and unlayered styles outrank all layers. It's the modern way to tame specificity wars and reduce reliance on !important.

Intermediate What is @counter-style for?

It defines a custom marker/numbering system beyond the built-ins — via descriptors like system (cyclic, numeric, additive…), symbols, suffix, range, pad and fallback. You then reference it from list-style-type or a counter(). Common uses: custom bullets, emoji markers, and non-Latin or localized numbering.

Pro Why does BEM deliberately avoid nested selectors and IDs?

To keep every rule at the same low specificity (a single class, 0-1-0). When all selectors are equal, override order is just source order — no rule can accidentally out-specify another, components stay portable, and you avoid the escalating #id .a .b / !important arms race. The discipline trades slightly longer class names for predictable cascade behaviour.

Pro How would you produce multi-level numbering like 1, 1.1, 1.2, 2 for a nested list?

Reset a same-named counter on every <ol>, increment it on each <li>, and render the marker with counters(item, ".") — the plural function walks up through all counters of that name in scope and joins them. Suppress the native marker (list-style: none) and inject via ::before/::marker. For custom formats layer a @counter-style on top.

Pro You must print an invoice list: hide the app shell, reveal collapsed sections, show link URLs, and force each invoice onto its own page. Outline the print-CSS approach.

Everything inside @media print: display: none on nav, sidebar, buttons and anything .no-print; override collapse styles (max-height: none; overflow: visible) so hidden content renders; expose destinations with a[href]::after { content: " (" attr(href) ")"; }; wrap each invoice and apply break-after: page (and break-inside: avoid on totals/tables); set the sheet with @page { margin: 2cm; }. Verify with DevTools print emulation, and remember fragmentation support is uneven, so keep the layout simple.
