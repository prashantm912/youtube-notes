Module: Transitions and Animations

A transition reacts to a state change; an animation runs a script of its own on a keyframe timeline. This module maps both, then the modern edge interviewers now probe — driving motion by scroll position and animating all the way to height: auto.

Knowledge Library → Courses → CSS → Transitions and Animations

What this module covers

Motion in CSS comes in two shapes: transitions interpolate a property between its old and new value when something changes (a hover, a class toggle), while animations play a named @keyframes sequence independently of any trigger. This module orients you on the transition properties and timing functions, the @keyframes plus animation-* family, the newer scroll-driven timelines that link progress to the scrollbar, and the interpolate-size mechanism that finally lets you animate to intrinsic sizes like auto.

Topics

transition-* properties, timing functions, transition-behavior allow-discrete

A transition tells the browser to interpolate a property over time instead of jumping to its new value. The transition shorthand bundles four longhands — transition-property (what to animate), transition-duration, transition-timing-function (the easing curve — ease, linear, cubic-bezier(), or steps()), and transition-delay. The transition fires whenever the property's computed value changes and the element is currently rendered, which is why hover and class-toggle effects "just work" with a single declaration.

Why it matters: transitions are the cheapest, most common animation on real sites, and the classic gotcha — you cannot transition to or from display: none — has a modern answer. transition-behavior: allow-discrete lets discrete properties like display and content-visibility flip at the right moment so an element can fade out and then be removed, instead of vanishing instantly.

.panel {
  opacity: 1;
  transition: opacity 300ms ease, display 300ms allow-discrete;
}
.panel[hidden] {        /* toggled by a class or attribute */
  opacity: 0;
  display: none;        /* now waits for the fade before disappearing */
}

@keyframes, animation-* properties, animation-composition

Where a transition only knows a start and an end, an animation walks a timeline you define with @keyframes — named waypoints at from/to or percentage stops. You attach it with the animation shorthand, which combines animation-name, -duration, -timing-function, -delay, -iteration-count (including infinite), -direction (e.g. alternate), -fill-mode (what styles hold before the first and after the last frame), and -play-state.

Why it matters: animations power loaders, attention cues and looped effects that no state change would trigger. Interviewers commonly test animation-fill-mode: forwards (why the element snaps back without it) and the compositor-friendly rule of animating transform/opacity over layout properties. The newer animation-composition (replace, add, accumulate) decides how multiple animations touching the same property combine rather than overwrite.

@keyframes pulse {
  from { transform: scale(1);   opacity: 1;   }
  to   { transform: scale(1.15); opacity: .6; }
}
.badge {
  animation: pulse 600ms ease-in-out infinite alternate;
}

Scroll-driven animations (animation-timeline/scroll()/view())

Scroll-driven animations retarget a normal @keyframes animation so its progress is driven by scroll position instead of elapsed time. You keep the keyframes and set animation-timeline to an anonymous timeline: scroll() creates a scroll progress timeline tied to how far a scroll container has scrolled, while view() creates a view progress timeline tied to where an element sits within its scrollport as it passes through. Because the browser runs these off the main thread, they stay smooth without a scroll-event listener.

Why it matters: reading-progress bars, reveal-on-scroll and parallax used to need JavaScript and often janked; the CSS version is declarative and performant. This is a modern-CSS topic that senior front-end interviews increasingly ask about, and the scroll-vs-view timeline distinction is the crux.

@keyframes reveal {
  from { opacity: 0; transform: translateY(2rem); }
  to   { opacity: 1; transform: translateY(0);     }
}
.card {
  animation: reveal linear both;
  animation-timeline: view();   /* progresses as the card scrolls into view */
}

Animating to intrinsic sizes (interpolate-size, calc-size)

For years you could not transition height from a fixed value to auto, or to fit-content/max-content, because the browser had no numeric endpoint to interpolate toward — hence the max-height and JavaScript-measuring hacks. Opting in with interpolate-size: allow-keywords makes those intrinsic keywords animatable, and the calc-size() function lets you compute with an intrinsic size (for example calc-size(auto, size + 1rem)) where plain calc() cannot.

Why it matters: expand/collapse accordions and disclosure panels are everywhere, and "how do you animate to height auto?" is a frequent interview question whose honest modern answer is exactly this pair. It is a newer, survey-level feature, so knowing the opt-in and its browser-support caveat is what counts.

:root { interpolate-size: allow-keywords; }

.disclosure {
  height: 0;
  overflow: clip;
  transition: height 300ms ease;
}
.disclosure.open { height: auto; }   /* now smoothly animatable */

Interview questions

Most-asked transitions and animations questions in Indian front-end interviews — from screening rounds to product-company deep dives:

Beginner What is the difference between a CSS transition and a CSS animation?

A transition interpolates a property between two states and needs a trigger — a hover, focus, or class change — to run once each way. An animation plays a named @keyframes sequence on its own timeline with no trigger, can have many intermediate stops, and can loop, reverse, and pause.

Beginner What does transition: all 0.3s ease expand to, and why avoid all?

It sets transition-property: all, duration 0.3s, timing-function ease, delay 0. all watches every animatable property, which can trigger unexpected transitions and extra work; naming the specific properties (opacity, transform) is clearer and cheaper.

Beginner How do from/to (or 0% and 100%) work in @keyframes?

They are waypoints along the animation's duration. from equals 0%, to equals 100%, and you can add any percentage stops in between; the browser interpolates between adjacent stops. A property missing from a keyframe just uses the element's current value.

Beginner What do the common timing functions ease, linear, and steps() do?

linear moves at constant speed; ease (the default) starts and ends slow with a fast middle; ease-in/ease-out/ease-in-out vary the ends; cubic-bezier() defines a custom curve; and steps(n) jumps in discrete increments, used for sprite-sheet and typewriter effects.

Beginner What does animation-iteration-count: infinite plus alternate achieve?

infinite repeats forever; animation-direction: alternate makes every other run play backwards, so a from→to animation ping-pongs smoothly instead of snapping back to the start on each loop — the standard pulse/breathing effect.

Intermediate Why does an element snap back after an animation, and how does animation-fill-mode fix it?

By default an animation only applies styles while running; when it ends the element reverts to its underlying CSS. animation-fill-mode: forwards keeps the final keyframe's styles after it finishes, backwards applies the first keyframe during the delay, and both does both.

Intermediate Why can't you transition to display: none, and what solves it now?

display is a discrete property, so it flips instantly and used to cancel any accompanying fade. Setting transition-behavior: allow-discrete (and listing display in the transition) defers the flip to the end, so the element can fade out first; pair it with @starting-style for the fade-in on appear.

Intermediate Which properties are cheap to animate, and why prefer transform/opacity?

transform and opacity can be handled by the compositor without triggering layout or paint, so they stay at 60fps. Animating width, height, top/left, or margin forces reflow/repaint each frame and janks. will-change can hint the browser to promote a layer, used sparingly.

Intermediate How do you make motion respect user accessibility settings?

Wrap non-essential motion in @media (prefers-reduced-motion: reduce) and reduce or remove it — for example set animation: none or a very short duration. Users who set "reduce motion" in their OS get a calmer experience; this is a standard accessibility check in reviews.

Intermediate What does animation-composition control?

When multiple animations (or an animation and the element's base value) affect the same property, it decides how they combine: replace overwrites (default), add appends the effect (e.g. stacking transforms), and accumulate combines values additively. It matters most for composable transform-based effects.

Pro How do scroll-driven animations work without JavaScript, and how do scroll() and view() differ?

You keep a normal @keyframes animation but set animation-timeline to a scroll-based timeline, so progress maps to scroll position instead of time. scroll() is a scroll progress timeline tracking a scroll container's own scroll offset (good for reading-progress bars); view() is a view progress timeline tracking an element's passage through the scrollport (good for reveal-on-scroll). They run off the main thread, so no scroll listener is needed.

Pro How do you animate an element to height: auto in modern CSS?

Opt in with interpolate-size: allow-keywords (usually on :root), which makes intrinsic keywords like auto, fit-content, and max-content animatable, so a plain transition: height to auto works; use calc-size() when you need arithmetic on an intrinsic size. It is progressively enhanced — where unsupported, the panel simply jumps — so it degrades gracefully versus the old max-height or JS-measuring hacks.
