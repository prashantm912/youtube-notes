Module: Cascade, Specificity and Inheritance

Before a single selector matches, the browser has already decided which rule wins. This module maps the machinery behind that decision — origin, order, specificity, inheritance and cascade layers — and links the canonical reference for each.

What this module covers

This is the module that explains why your CSS sometimes "doesn't work" even though the rule is right there. You will learn how styles get attached to a page, how the cascade ranks competing declarations by origin and order, how specificity is scored, how values flow down the tree through inheritance, and how modern cascade layers and !important reshape all of it. Master this first — every later module assumes it.

Topics

Applying CSS (inline/internal/external), at-rules overview

CSS reaches a page three ways: an external stylesheet linked with <link rel="stylesheet"> (the default for real projects — cacheable, reusable, one source of truth), an internal stylesheet in a <style> block in the <head> (fine for a single page), and an inline style attribute on one element (avoid it — no reuse, and it outranks almost everything, so it is painful to override later). Which method you choose already affects the cascade before any selector runs.

Layered on top of ordinary rules are at-rules: statements that start with @ and instruct the engine rather than style one element. You will meet @import, @media, @supports, @font-face, @keyframes, @container, @layer and more across this course; knowing the two shapes (statement at-rules ending in ; and block at-rules ending in { }) makes the rest of CSS legible.

<!-- external: the standard for whole sites -->
<link rel="stylesheet" href="main.css">

<!-- internal: scoped to one page -->
<style>
  @import "reset.css";              /* statement at-rule */
  @media (min-width: 40rem) {       /* block at-rule    */
    body { font-size: 1.125rem; }
  }
</style>

The cascade and origin/order

The cascade is the algorithm that picks one value when several declarations target the same property on the same element. It resolves in bands: first origin and importance (user-agent vs user vs author stylesheets, plus the !important flag), then specificity, then order of appearance as the final tiebreaker — the last matching declaration of equal weight wins. Normal author styles beat normal user styles beat the browser's defaults; that ordering is the reason a plain reset works at all.

Why it matters: most "my style is being ignored" bugs are not broken CSS but a losing position in this ranking. Reading the cascade correctly — is this an origin problem, a specificity problem, or just a source-order problem? — is the single most useful debugging skill in day-to-day CSS work and a staple of screening interviews.

Specificity calculation

When two rules survive the origin/importance band, specificity breaks the tie. Score each selector in three groups — id selectors, then class / attribute / pseudo-class selectors, then type / pseudo-element selectors — commonly written as a triple like (1, 0, 2). Compare left to right: a single hit in a higher column beats any number of hits in the columns to its right, so one id outweighs ten classes. The universal selector * and combinators add nothing, and the inline style attribute ranks above all id selectors.

Why it matters: specificity wars — patching a stubborn rule by piling on selectors or reaching for !important — are how stylesheets rot. Interviewers love a concrete "which of these two selectors wins, and why" question because it proves you can read a real conflict rather than guess.

a            { }   /* type only            -> (0, 0, 1) */
.link        { }   /* one class            -> (0, 1, 0) */
nav a.link   { }   /* type + class + type  -> (0, 1, 2) */
#nav a       { }   /* id + type            -> (1, 0, 1) wins over all above */

Inheritance, inherit/initial/unset/revert

Inheritance is the second half of value resolution: when no declaration sets a property, an element either inherits its parent's computed value or falls back to the property's initial value. Text-related properties (color, font-*, line-height, visibility) inherit by default; box and layout properties (margin, padding, border, width, background) do not. That split is why setting color on body tints the whole page but setting a border does not cascade downward.

Four CSS-wide keywords let you steer inheritance explicitly: inherit forces the parent's value, initial resets to the spec default, unset acts as inherit for inherited properties and initial for the rest, and revert rolls back to whatever the user-agent (or user) stylesheet would apply — genuinely different from initial. Knowing the four apart, especially unset vs revert, is a common intermediate interview probe.

.child {
  color:  inherit;   /* take the parent's computed color        */
  margin: initial;   /* the property's own spec default (0)      */
  border: unset;     /* inherited prop -> inherit; else -> initial */
  padding: revert;   /* roll back to the user-agent value        */
}

@layer (cascade layers), !important

Cascade layers, declared with @layer, add an explicit tier to the cascade that is checked before specificity. You name your layers in priority order once (say reset, theme, components); then for normal declarations a later layer beats an earlier one, and any unlayered style beats every layered style — so a low-specificity unlayered rule can defeat a high-specificity layered one. Specificity only decides ties within a single layer. Layers let large codebases retire specificity hacks by design.

!important is the escape hatch that lifts a declaration into a separate, higher band evaluated ahead of all normal rules, and it inverts the usual ordering: user-agent important outranks user important, which outranks author important, and — critically — the layer order flips too, so a layered important rule can beat an unlayered one. Treat !important as a last resort or a deliberate architectural tool (utilities, state classes), never as a routine fix.

@layer reset, theme, components;   /* later layer wins for normal rules */

@layer theme {
  a { color: navy; }
}
a { color: teal; }   /* unlayered -> beats the layered rule above */

Interview questions

The most-asked CSS fundamentals in Indian front-end interviews — from service-company screening rounds to product-company deep dives:

Beginner What are the three ways to apply CSS, and which is preferred?

External stylesheet via <link> (preferred — cached, reusable, separates content from presentation), internal via a <style> block in the head (single page), and the inline style attribute on one element (avoid — no reuse and it outranks almost everything, so it is hard to override).

Beginner What is the cascade in one sentence?

The algorithm the browser uses to choose a single value when multiple declarations set the same property on the same element — deciding by origin and importance first, then specificity, then order of appearance.

Beginner Two author rules match one element with equal specificity — which wins?

The one written later in source order. Order of appearance is the final tiebreaker once origin/importance and specificity are equal — "the last one wins".

Beginner How do you calculate specificity?

Count three groups and write them as a triple: id selectors, then class/attribute/pseudo-class selectors, then type/pseudo-element selectors, e.g. (1, 2, 1). Compare left to right — a higher-left column beats any amount to its right. The universal selector and combinators add nothing.

Beginner Between an inline style attribute and an id selector in the stylesheet, which wins (no !important)?

The inline style attribute — it ranks above id selectors (effectively an extra, highest column). Only an !important declaration, or a later inline declaration, can beat it.

Intermediate Which wins: #nav .item a or .menu .item a.active?

#nav .item a is (1, 1, 1); .menu .item a.active is (0, 3, 1). The first column decides — 1 beats 0 — so #nav .item a wins even though the other selector has more classes. A classic "don't count, compare columns" question.

Intermediate Which properties inherit by default, and which do not?

Mostly text-level properties inherit — color, font-*, line-height, letter-spacing, visibility. Box and layout properties do not — margin, padding, border, width, background — they take their initial value on each element unless you say otherwise.

Intermediate Explain inherit vs initial vs unset vs revert.

inherit = use the parent's computed value. initial = the property's spec default. unset = inherit for inherited properties, initial for the rest. revert = roll back to the user-agent (or user) stylesheet value — not the spec initial, which is what makes it distinct from initial and unset.

Intermediate Do :is(), :where(), :not() and :has() affect specificity?

:where() always contributes zero specificity. :is(), :not() and :has() add nothing on their own but take the specificity of their most specific argument — so :is(#id, p) counts as an id. This is why :where() is the tool for writing easily-overridable base styles.

Intermediate How does !important change the cascade?

It moves the declaration into a separate, higher band evaluated before all normal rules, and it reverses origin order: user-agent important outranks user important, which outranks author important — all of them above every normal rule and inline style. Within the same origin and importance, specificity and source order still decide.

Pro How do cascade layers rank against unlayered styles and against specificity?

Layer order is checked before specificity. For normal rules, a later-declared layer beats an earlier one, and unlayered styles beat all layered styles — so a low-specificity unlayered rule can defeat a high-specificity layered rule, and specificity only breaks ties inside the same layer. For !important declarations the layer order inverts, so a layered important rule can outrank an unlayered one.

Pro A third-party stylesheet has .btn { color: red !important } and you cannot edit it. How do you override it cleanly?

You have to win inside the important band. Direct fix: a later author !important declaration with equal-or-greater specificity, e.g. #app .btn { color: … !important }. Cleaner and specificity-free: put the third-party CSS in a cascade layer — because for !important the layer logic inverts, your layered important override then beats their unlayered important rule. Reserve raw !important escalation as the last resort; taming exactly this arms race is what layers were designed for.
