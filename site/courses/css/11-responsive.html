Module: Responsive Design

One document, every screen — from a narrow phone to a widescreen monitor, plus dark mode and reduced-motion. This module maps the three tools that get you there and sends you to the best material to master each.

What this module covers

Responsive design is making one layout work everywhere without shipping separate sites. Three layers do the job: media queries adapt to the viewport and to user preferences, container queries adapt a component to the space it is dropped into, and fluid/intrinsic patterns let content flex on its own so you need fewer breakpoints at all. Master the trio and "mobile version" stops being a separate build.

Topics

Media queries (all features incl. prefers-*/forced-colors)

A media query applies a block of CSS only when the browsing environment matches a condition. Beyond the familiar width and orientation, the feature set includes interaction features (hover, pointer, any-pointer), resolution, and — increasingly the point — user-preference features: prefers-color-scheme for dark mode, prefers-reduced-motion and prefers-contrast for accessibility, and forced-colors for Windows High Contrast mode. Modern range syntax lets you write (width >= 40em) instead of (min-width: 40em).

Why it matters: this is the original and still most-used responsive mechanism, so its breakpoints and quirks are interview staples. Two habits mark a senior answer — think mobile-first with min-width so styles layer upward, and treat preference queries as accessibility work, not decoration. Using em/rem breakpoints (they respect the user's zoom) over hard-coded pixels is another tell.

/* Mobile-first base, then layer complexity upward */
.card { padding: 1rem; }

@media (min-width: 40em) {        /* ~640px; em respects zoom */
  .card { padding: 2rem; }
}

@media (width >= 40em) { /* modern range syntax, same result */ }

@media (prefers-reduced-motion: reduce) {
  * { animation-duration: 0.01ms; }
}

Container queries (@container size/style/scroll-state, cq units)

Container queries flip the question from "how big is the window?" to "how big is the box this component sits in?" You opt an element in as a query container with container-type: inline-size (optionally naming it via container-name), then style its descendants with @container based on that container's dimensions. Alongside size queries there are style queries (@container style(--theme: dark)) and newer scroll-state queries, plus container query units — cqi, cqw, cqb, cqmin, cqmax — that are relative to the container instead of the viewport.

Why it matters: today's UIs are component-driven, and the same card might sit in a wide main column or a narrow sidebar. Container queries let that card adapt to wherever it lands, so it becomes genuinely reusable rather than tied to global breakpoints. This is Baseline (widely available across engines since 2023), so expect interviewers to probe the difference between it and media queries.

.card-wrapper {
  container-type: inline-size;
  container-name: card;
}

@container card (min-width: 30rem) {
  .card { display: grid; grid-template-columns: 8rem 1fr; }
}

/* size the title against the CONTAINER, not the viewport */
.card h2 { font-size: clamp(1rem, 5cqi, 1.5rem); }

Fluid patterns, mobile-first, intrinsic responsive design

The most resilient responsive code often has the fewest breakpoints. Fluid patterns lean on flexible primitives: percentages and fr units, max-width: 100% on media, and the math functions min(), max() and clamp() for fluid type and space that scale smoothly between a floor and a ceiling. Mobile-first means writing the small-screen layout as the baseline and adding complexity with min-width queries. Intrinsic web design (Jen Simmons' term) goes further — let the content and available space decide, using flex wrapping and grid auto-fit/minmax() so items reflow with no explicit breakpoint at all.

Why it matters: hard-coded breakpoints are brittle and multiply with every new device. Fluid and intrinsic techniques degrade gracefully across screens you never tested, and interviewers increasingly ask candidates to solve a layout with clamp() or an auto-fitting grid instead of a stack of media queries. Media queries do not disappear; they become the exception you reach for, not the default.

/* Fluid type: scales smoothly between two bounds, no breakpoint */
h1 { font-size: clamp(1.75rem, 1rem + 4vw, 3rem); }

/* Intrinsically responsive grid — wraps on its own */
.grid {
  display: grid;
  gap: 1rem;
  grid-template-columns: repeat(auto-fit, minmax(min(100%, 16rem), 1fr));
}

Interview questions

Most-asked responsive-design questions in Indian interviews, from front-end screening rounds to product-company deep dives:

Beginner What is a media query and what is the basic syntax?

A rule that applies CSS only when the environment matches a condition. Syntax is @media <type> and (<feature>), e.g. @media screen and (min-width: 600px) { … }. Type is usually omitted (defaults to all); the feature test is the part that matters.

Beginner What does the viewport meta tag do and why is it needed?

<meta name="viewport" content="width=device-width, initial-scale=1"> tells mobile browsers to use the device's real width instead of pretending to be a ~980px desktop and zooming out. Without it, media queries and fluid layouts effectively do not respond on phones.

Beginner min-width vs max-width breakpoints — which fits mobile-first?

min-width applies from that width upward, so you write the small-screen base first and add rules as the screen grows — that is mobile-first. max-width applies downward (desktop-first). Mobile-first is preferred: base styles are simplest and enhancements layer on cleanly.

Beginner How do you make images responsive?

Baseline: img { max-width: 100%; height: auto; } so an image never overflows its container. For serving different files by screen, use srcset/sizes on <img> or the <picture> element for art direction and modern formats.

Intermediate What is mobile-first CSS and why is it recommended?

Write the single-column, small-screen layout as the default, then use min-width queries to add columns and spacing for larger screens. It keeps the base CSS minimal, matches how most users arrive (mobile), and avoids overriding heavy desktop styles back down for phones.

Intermediate Media queries vs container queries — when do you use each?

Media queries respond to the viewport (or user preferences) — right for page-level layout and things like dark mode. Container queries respond to a component's own container size, so a reusable card adapts wherever it is placed. Rule of thumb: page shell uses media queries, self-contained components use container queries.

Intermediate How do you implement dark mode with CSS, and what does prefers-color-scheme do?

@media (prefers-color-scheme: dark) matches when the OS/browser is set to a dark theme. Define colors as custom properties and override them inside that query (or use the light-dark() function with color-scheme). Pair it with a manual toggle for users who want to override the system setting.

Intermediate What is prefers-reduced-motion and why does it matter?

A media feature that matches when the user has asked the OS to minimize animation. Wrap non-essential transitions and animations so they are reduced or removed under @media (prefers-reduced-motion: reduce). It is an accessibility requirement — large motion can trigger nausea or vestibular disorders.

Intermediate How do min(), max() and clamp() help build fluid layouts?

They compute a value from mixed units without a query. clamp(min, preferred, max) gives fluid typography — clamp(1rem, 1rem + 2vw, 2rem) grows with the viewport but never below 1rem or above 2rem. min()/max() cap or floor a single side, e.g. a width that is the smaller of 100% and a fixed max.

Intermediate What are container query units like cqi and cqw, and how do they differ from vw?

They are lengths relative to the query container instead of the viewport: cqw is 1% of its width, cqi 1% of its inline size, plus cqb, cqmin, cqmax. Unlike vw, which always tracks the window, they let a component scale to its actual box — so the same card sizes correctly in a wide or narrow slot.

Pro Why prefer em/rem over px for media query breakpoints?

Breakpoints in em scale with the user's browser font-size/zoom, so someone who increases their default text triggers the layout shift at a proportionally larger window — keeping line lengths readable. Pixel breakpoints ignore that preference. Note media-query em is always relative to the browser default, not the element's font-size.

Pro What is intrinsic/component-driven responsive design, and how does it reduce breakpoints?

Instead of dictating layout at fixed viewport widths, you let content and available space drive it: flex-wrap lets items break to new rows when they run out of room, and grid-template-columns: repeat(auto-fit, minmax(min(100%, 16rem), 1fr)) fits as many columns as fit and reflows otherwise. Add container queries and style queries for component-level decisions. The result adapts on screens you never tested, so media queries become the exception rather than the backbone.
