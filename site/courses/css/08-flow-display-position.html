Module: Flow, Display and Positioning

Before flexbox and grid, the browser already has a layout engine: normal flow, and the switches that bend it — display, float, position and z-index. This module maps each switch and where it belongs.

Knowledge Library → Courses → CSS → Flow, Display and Positioning

What this module covers

This is the layout module that comes before flexbox and grid. It orients you on how boxes flow by default, the display switch that changes an element's role, floats and the block formatting context, the five values of position, how z-index really works through stacking contexts, and overflow with the new reading-order controls. Master these and the modern layout modules make sense; skip them and every positioning bug feels like magic.

Topics

display — block, inline, inline-block, none, flow-root, contents

The display property sets the box an element generates: its outer role in the parent's flow and the inner formatting context for its own children. A block box stacks vertically and fills the available inline width; an inline box sits inside the text line and ignores width, height and vertical margins; inline-block is inline-level yet honours width, height and padding. none removes the box entirely — unlike visibility: hidden it leaves no space and is dropped from the accessibility tree. flow-root makes a block that establishes a new block formatting context (the modern clearfix), and contents makes the element's own box vanish so its children take part in the parent's layout directly.

Why it matters: display is the first decision in any layout, and it is exactly what flex and grid turn on (display: flex / grid). Knowing that inline-block was the pre-flexbox horizontal-layout tool, that display: none has accessibility consequences, and that display: contents has sharp edges (it can remove the element from the accessibility tree in some engines) is standard interview ground.

.menu    { display: flex; }         /* inner: flex formatting context */
.tag     { display: inline-block; } /* inline-level, but a sized box */
.wrapper { display: contents; }     /* box disappears, children promote */

float, clear and the block formatting context (BFC)

float pulls an element to the left or right edge of its container and lets following inline content — classically text — wrap around it; it was the workhorse of pre-flexbox page layout. clear forces an element down past preceding floats. Because a float is lifted out of normal flow, a parent whose only children are floated collapses to zero height — the classic bug. The clean fix is a block formatting context (BFC): a self-contained layout region that contains its own floats, refuses to overlap outside floats, and suppresses margin collapse. display: flow-root creates one on purpose; an overflow value other than visible creates one as a side effect.

Why it matters: floats are retired for page layout — flex and grid own that now — but they remain the right tool for wrapping text around an image or a pull-quote, and BFC reasoning still explains margin collapse and float containment. Expect to explain the clearfix hack, why overflow: hidden "fixes" a collapsed parent, and what a BFC actually is.

.figure    { float: left; margin-inline-end: 1rem; } /* text wraps around it */
.container { display: flow-root; }                   /* new BFC contains the float */

position — static, relative, absolute, fixed, sticky

position decides what the inset properties (top/right/bottom/left) act on. static is the default — in normal flow, insets ignored. relative keeps the box in flow but paints it offset from where it would sit and — crucially — makes it a containing block for absolutely positioned descendants. absolute removes the box from flow and positions it against the nearest positioned ancestor, falling back to the initial containing block. fixed pins the box to the viewport. sticky is a hybrid: in flow until a scroll threshold is crossed, then pinned within its scroll container.

Why it matters: the containing-block rule is the heart of most positioning bugs. "Relative on the parent, absolute on the child" is muscle memory for badges, overlays and dropdowns. Interviewers probe the relative-versus-absolute relationship, why sticky silently does nothing (an ancestor with overflow: hidden/auto, or no threshold set), and how a transform or filter on an ancestor turns it into the containing block for fixed children.

.card   { position: relative; }                 /* becomes a positioning context */
.badge  { position: absolute; top: 0; right: 0; }
.header { position: sticky; top: 0; }           /* pins once it reaches the top */

z-index and stacking contexts

z-index sets paint order on the z-axis, but only for elements that establish a stack level — positioned elements and flex/grid children — and only against siblings inside the same stacking context. A stacking context is a self-contained layer group: once one forms, its children's z-index values carry no meaning outside it. Many properties quietly create one — a positioned element with a numeric z-index, but also opacity below 1, transform, filter, will-change and isolation: isolate.

Why it matters: this is why "I set z-index: 9999 and it still hides behind that other thing" happens — the two elements live in different stacking contexts, so their numbers never compete. The senior move is isolation: isolate, which forms a context with no other visual side effects, so a component's internal layering cannot leak out or be leaked into. Stacking context is one of the most reliably asked "explain this bug" CSS questions.

.modal { position: fixed; z-index: 100; }
.card  { isolation: isolate; } /* new stacking context, no other side effects */

overflow, reading-flow and reading-order

overflow controls what happens when content is larger than its box. visible (the default) lets it spill out; hidden clips it; scroll always shows scrollbars; auto shows them only when needed; clip is like hidden but forbids all scrolling, even programmatic, and is cheaper. Any non-visible value also establishes a BFC and turns the element into a scroll container. overflow-x and overflow-y set the axes independently, with the caveat that pairing visible with a non-visible value computes the visible one to auto.

Why it matters, and what is new: when flex or grid order, or flex-direction: row-reverse, make the visual order differ from the DOM order, tab and screen-reader order still follow the DOM — a real accessibility trap. reading-flow (on the container) and reading-order (on items) let you realign the reading and focus order to the visual one. Both are still experimental with limited browser support, so name them as the modern, spec-blessed answer to a visual-versus-DOM order mismatch — the older answer being "fix the source order instead."

.panel { overflow: auto; max-height: 20rem; }        /* scrolls only when it must */
.row   { display: flex; reading-flow: flex-visual; } /* focus follows visual order */

Interview questions

Flow and positioning run through every Indian front-end interview — from service-company screening rounds to product-company layout-and-DOM deep dives:

Beginner Difference between block, inline and inline-block?

block stacks vertically and takes the full inline width; inline sits in the text line and ignores width, height and vertical margins; inline-block is inline-level (flows in the line) but still honours width, height, padding and vertical margins. inline-block was the pre-flexbox way to lay boxes side by side.

Beginner display: none vs visibility: hidden vs opacity: 0?

display: none removes the box entirely — no space, not painted, gone from the accessibility tree. visibility: hidden keeps the box's space but hides it and takes it out of tab/AT order. opacity: 0 keeps the box, its space, and its interactivity — it is still clickable and focusable, just invisible.

Beginner What are the five values of position?

static (default, insets ignored), relative (in flow but offset from itself; a containing block for absolute children), absolute (out of flow, positioned to the nearest positioned ancestor), fixed (out of flow, pinned to the viewport), and sticky (in flow until a scroll threshold, then pinned within its scroll container).

Beginner An absolutely positioned element — what is it positioned relative to?

Its containing block: the nearest ancestor with a non-static position (relative/absolute/fixed/sticky). If none exists, it falls back to the initial containing block (roughly the viewport). This is why you set position: relative on a parent before absolutely positioning a child inside it.

Intermediate A parent with only floated children has zero height. Why, and how do you fix it?

Floats are taken out of normal flow, so the parent has nothing in flow to give it height and it collapses. Fixes: establish a BFC on the parent — display: flow-root (cleanest), or overflow: auto/hidden as a side effect — or the classic clearfix (.parent::after { content: ""; display: block; clear: both; }).

Intermediate What is a Block Formatting Context, and name a few ways to create one?

A self-contained block layout region that contains its own floats, does not overlap outside floats, and suppresses margin collapse across its boundary. Created by: the root element, floats, absolutely positioned boxes, display: inline-block/flow-root/table-cell, flex and grid items, and any overflow other than visible/clip.

Intermediate How do position: relative and position: absolute relate to each other?

relative stays in flow and, without insets, looks unchanged — but it becomes a positioning context (containing block) for absolute descendants. absolute leaves flow and anchors to that nearest positioned ancestor. The everyday pairing is relative parent plus absolute child for badges, dropdowns and overlays.

Intermediate position: sticky is not sticking. What are the usual reasons?

No threshold set (sticky needs at least one of top/right/bottom/left); an ancestor between it and the scroll container has overflow: hidden/auto/scroll, which becomes the scroll context and clips the effect; the element's own container is not tall enough for it to travel; or a height/flex constraint stops it from moving.

Intermediate overflow: hidden vs overflow: clip?

Both cut off overflowing content. hidden makes the element a scroll container — content can still be scrolled programmatically or by focus — and establishes a BFC. clip forbids all scrolling (no programmatic scroll), can clip on a single axis, respects overflow-clip-margin, and is lighter weight.

Pro You set z-index: 9999 and the element still sits behind another. Why, and how do you fix it?

The two elements are in different stacking contexts, so their z-index values never compare directly — the parent context's order wins. Debug by finding which ancestor created a stacking context (opacity below 1, transform, filter, will-change, a positioned ancestor with z-index). Fix by raising the right ancestor, restructuring the layers, or using isolation: isolate to control layering deliberately.

Pro Besides z-index on a positioned element, what creates a stacking context?

The root element; opacity below 1; transform, filter, backdrop-filter, perspective, clip-path, mask; will-change on any of those; isolation: isolate; position: fixed/sticky; and flex/grid children with a non-auto z-index. The takeaway: many common visual properties silently create one.

Pro Flex order changed the visual order but tab order still follows the DOM. What happens, and what is the modern CSS fix?

Reordering with order or row-reverse is visual only — keyboard tabbing and screen readers keep following source order, so focus jumps around unpredictably. The accessible baseline fix is to reorder the DOM instead. The modern spec answer is reading-flow on the container (e.g. flex-visual) with reading-order on items to realign focus/reading order to the visual order — still experimental, with limited browser support.
