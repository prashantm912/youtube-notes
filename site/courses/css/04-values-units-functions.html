Module: Values, Units and Functions

A CSS value is rarely just a number — it is a length, an angle, a named variable, or a calculation the browser resolves at render time. This module maps that whole vocabulary and links the canonical reference for each piece.

Knowledge Library → Courses → CSS → Values, Units and Functions

What this module covers

Everything to the right of the colon in a CSS declaration is a value, and values come with a grammar: units that give numbers meaning, custom properties that name and reuse them, and functions that compute them live in the browser. This module covers absolute and relative lengths (including the small, large and dynamic viewport units), the lesser-used angle, time, resolution and frequency units, custom properties consumed with var() and env(), and the math family from calc() through clamp() to trigonometry. Get fluent here and responsive, themeable layouts stop being guesswork.

Topics

Absolute/relative lengths, percentages, viewport units (incl. sv/lv/dv)

Lengths are the workhorse of CSS sizing. Absolute units (px, pt, cm, in) resolve against one fixed base, so they render the same everywhere and are best for hairline borders. Relative units are contextual and adaptive: em is relative to the element's own font size, rem to the root font size, ch to the width of the "0" glyph, and percentages resolve against the containing block. Preferring rem for type and spacing keeps a layout responsive to the user's browser font-size setting — an accessibility win that fixed pixels forfeit.

Viewport units size against the viewport itself: vw and vh are 1% of its width and height. The catch is mobile — the classic 100vh resolves against the largest viewport, so a full-height section sits taller than the visible area while the URL bar is showing. CSS Values and Units 4 added the small (svh), large (lvh) and dynamic (dvh) variants so you can pick which state you mean; dvh resizes as the browser chrome appears and disappears.

.box  { width: 60vw; }          /* 60% of viewport width */
.hero { min-height: 100dvh; }   /* dynamic viewport height — mobile-safe */
p     { font-size: 1.125rem; text-indent: 2ch; } /* rem, ch */

Angle, time, resolution, frequency units

Beyond lengths, CSS defines dimensional values for other quantities, and knowing them prevents unit-mismatch bugs. Angles (deg, grad, rad, turn) feed transforms, gradients and hue channels — 0.25turn and 90deg are the same rotation. Time (s, ms) drives transitions and animations, where a bare number is invalid: it must be 200ms, not 200. Resolution units (dpi, dpcm, dppx and its alias x) appear in media queries and image-set() to target high-density displays.

Frequency units (Hz, kHz) are specified for aural properties but are not yet used by any shipping CSS property — worth recognising, not worth memorising. The unifying rule across all of them: a dimension is a number glued directly to its unit with no space, and the wrong unit type silently invalidates the declaration.

.icon { transform: rotate(0.25turn); }      /* angle */
.fade { transition: opacity 200ms ease; }   /* time in ms */
@media (min-resolution: 2dppx) { /* HiDPI / retina rules */ }

Custom properties (--*, var()), env()

Custom properties are author-defined values whose names begin with two dashes (the --* syntax), declared like any property and read back with var(--name, fallback). Unlike a Sass variable, they are live at runtime: they inherit through the DOM, respond to the cascade and media queries, and can be rewritten from JavaScript. Declaring them on :root makes a global palette; declaring them on a component scopes them locally. That single mechanism powers design tokens, light/dark theming and per-component overrides without a build step.

env() looks similar but reads values supplied by the user agent and environment rather than the author — most usefully the safe-area-inset-* values that keep content clear of notches and rounded screen corners on mobile. Both take a fallback; the key difference is who owns the value. A common gotcha: if a custom property holds a value that is invalid for the property using it, the fallback is not applied — the declaration becomes invalid at computed-value time instead.

:root { --brand: #0a7; --space: 1rem; }
.btn  { background: var(--brand); padding: var(--space); }
.app  { padding-top: env(safe-area-inset-top, 0px); }

Math functions (calc/min/max/clamp/round/mod/trig/calc-size)

Math functions let a property value be an expression the browser evaluates. calc() mixes units in one calculation (calc(100% - 2rem)); its two rules trip everyone once — you must put whitespace around + and -, and for division the divisor must be a plain number. min() and max() pick the smaller or larger of several values, and clamp(min, preferred, max) composes them into a bounded value, which is exactly the recipe for fluid typography that scales with the viewport yet never gets too small or too large.

Newer additions widen the toolkit: stepped-value functions round(), mod() and rem(); the trigonometric set sin(), cos(), tan() and their inverses for geometry and motion; and calc-size(), which finally makes animation to intrinsic sizes such as auto possible. All of them nest and combine with custom properties, so a single declaration can encode logic that once needed JavaScript.

.col  { width: calc((100% - 2rem) / 3); }        /* whitespace around - required */
h1    { font-size: clamp(1.5rem, 1rem + 3vw, 3rem); } /* fluid, bounded */
.card { inline-size: min(100%, 40rem); }         /* never wider than 40rem */

Interview questions

Values and units are a favourite warm-up in Indian frontend interviews — from service-company screening rounds through product-company deep dives:

Beginner What is the difference between em and rem?

em is relative to the element's own font size and compounds through nesting, so a child inside a child can grow unexpectedly. rem is always relative to the root (html) font size, so it stays predictable no matter how deep the element sits. Most teams default to rem for type and spacing and reserve em for values that should track their local font size.

Beginner When would you use absolute units like px versus relative units?

Absolute units resolve against one fixed base, so they render identically everywhere — good for hairline borders or a 1px rule. Relative units (rem, em, %, viewport units) scale with the root font size or context, so they respect the user's browser font-size preference and adapt across screens. Prefer relative units for anything a user might want to zoom: font sizes, spacing, and layout widths.

Beginner Why is 100vh a problem on mobile, and what fixes it?

vh resolves against the largest viewport, so 100vh is taller than the visible area while the address bar is on screen, causing content to be cut off or a stray scroll. Use the newer units: svh (smallest, UI expanded), lvh (largest, UI collapsed), or dvh (dynamic, resizes as the chrome shows and hides). 100dvh is the usual fix for a full-height hero.

Beginner What is a CSS custom property and how do you read it?

A user-defined value whose name starts with two dashes, e.g. --brand: #0a7;. You read it with the var() function: color: var(--brand), optionally with a fallback: var(--brand, black). Custom properties inherit and participate in the cascade, so a value set on :root is available everywhere unless overridden lower down.

Beginner A percentage value — what does it resolve against?

It depends on the property. width/height percentages resolve against the containing block's width/height. Padding and margin percentages — on every side, including top and bottom — resolve against the containing block's inline size (its width). A font-size percentage is relative to the parent's font size, and a unitless line-height multiplies the element's own font size.

Intermediate What are the syntax rules of calc()?

Two catch you out. First, you must put whitespace around + and - (because - can be part of an identifier), so calc(100%-2rem) is invalid — write calc(100% - 2rem). Second, for multiplication at least one operand must be a number, and for division the divisor must be a number. calc() happily mixes units (px with %, rem with vw) and can nest.

Intermediate How do min(), max() and clamp() work, and how do they give fluid typography?

min() returns the smallest of its arguments, max() the largest. clamp(MIN, PREFERRED, MAX) is shorthand for max(MIN, min(PREFERRED, MAX)) — it keeps the preferred value between two bounds. Fluid type is a one-liner: font-size: clamp(1.5rem, 1rem + 3vw, 3rem) scales the heading with the viewport but never drops below 1.5rem or exceeds 3rem, replacing a stack of media queries.

Intermediate How do CSS custom properties differ from Sass variables?

Sass variables are resolved at compile time and vanish from the output — they cannot change at runtime. CSS custom properties are live: they exist in the cascade, inherit through the DOM, can be redefined inside a media query or a component, and can be read and written from JavaScript. That runtime nature is why theming and per-component overrides use custom properties, not preprocessor variables.

Intermediate How would you implement light and dark themes with custom properties?

Define the palette as custom properties on :root, then override them inside a prefers-color-scheme media query or under a [data-theme="dark"] selector. Components reference only var(--bg), var(--text) and so on, so flipping the theme re-cascades every value at once with no component changes. This is the standard, build-free theming pattern.

Intermediate What do the ch and ex units measure, and where are they useful?

ch is the advance width of the "0" (zero) glyph in the element's font; ex is the font's x-height. ch is the practical one — setting max-width: 65ch gives a comfortable line length for readable body text, since measure is best expressed in characters rather than pixels.

Pro If a custom property holds an invalid value, does var()'s fallback kick in?

No — that is the trap. The var() fallback is used only when the property is not set. If it is set but its value is invalid for the property consuming it, the declaration becomes "invalid at computed-value time" (IACVT): the property falls back to its inherited value, or its initial value if it does not inherit — not to the fallback in var(). Registering the property with @property and a syntax lets the browser reject bad values to the registered initial value instead.

Pro Why does animating a custom property often need @property, and why can't var() be used in a media query condition?

An unregistered custom property is just a token stream, so the browser has no type to interpolate and it animates discretely (jumps). @property declares its syntax (e.g. <color> or <length>), inheritance and initial value, which makes smooth transitions possible. And var() cannot appear in a media query feature test such as @media (min-width: var(--bp)) because media conditions are evaluated outside the element cascade where custom properties live — this is where env(), which the user agent resolves independently, differs from var().
