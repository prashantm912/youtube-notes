Module: Typography

Type is most of what a user actually reads, and CSS gives you a deep toolkit for it — from a font shorthand with tricky reset rules to logical properties that flip a whole layout for right-to-left scripts. This module maps that toolkit and points you to the canonical reference for each part.

Knowledge Library → Courses → CSS → Typography

What this module covers

Web typography splits into five areas: choosing and sizing type with the font-* properties, loading custom and variable fonts with @font-face, styling runs of text with the text-* properties, tuning readability and wrapping with the spacing group, and controlling text flow direction with writing-mode and logical properties. These are orientation notes — read the linked references for the full detail, official documentation first.

Topics

font-* properties and the font shorthand

Text styling starts with the type itself: font-family lists faces left to right until the browser finds one it has (a generic like sans-serif must come last), font-size accepts absolute keywords, lengths and the relative rem/em units that let text scale, and font-weight, font-style and font-variant cover boldness, italics and small-caps. The font shorthand sets all of these plus line-height in one declaration.

Why it matters: the shorthand has strict rules — font-size and font-family are mandatory and the family comes last, and any longhand you leave out is reset to its initial value, which silently wipes an earlier font-weight. That reset behaviour, plus the case for rem over px so the browser's font-size setting is respected, are routine interview and code-review talking points.

/* style variant weight size/line-height family */
h1 { font: italic small-caps 700 2rem/1.2 "Fira Sans", sans-serif; }
p  { font-family: system-ui, sans-serif; font-size: 1rem; line-height: 1.6; }

@font-face, variable fonts and font loading

@font-face registers a custom font under a family name and points src at one or more files (prefer WOFF2), after which font-family uses it like any installed face. Variable fonts pack many weights, widths and optical sizes into a single file exposed through axes such as wght, wdth and opsz — one download that produces any value between the named instances, driven by font-weight, font-stretch or font-variation-settings.

Why it matters: fonts block text rendering, so loading strategy is a Core Web Vitals concern. font-display: swap shows fallback text immediately (FOUT) instead of invisible text (FOIT), preload and preconnect fetch the file earlier, and unicode-range subsetting trims bytes. The FOIT-versus-FOUT trade-off and the font-display values are frequent performance-round questions.

@font-face {
  font-family: "Inter";
  src: url("/fonts/inter.woff2") format("woff2");
  font-weight: 100 900;   /* variable weight range */
  font-display: swap;     /* show fallback, then swap in */
}

text-* properties

The text-* family arranges and decorates a run of text: text-align handles horizontal alignment with flow-relative start/end as well as center and justify; text-decoration is the shorthand for the underline/overline/line-through line plus its color, style and thickness; text-transform changes case (uppercase, lowercase, capitalize) at render time; and text-indent indents the first line, including a hanging option.

Why it matters: text-transform only changes presentation — the DOM text and screen-reader output keep the original case — which is an accessibility point interviewers probe. Newer properties text-box-trim and text-box-edge strip the inconsistent space fonts leave above and below glyphs for pixel-accurate vertical rhythm, though browser support is still limited and a fallback is wise.

a       { text-decoration: underline dotted 2px; }
.button { text-transform: uppercase; }          /* visual only */
.lead   { text-indent: 2em; text-align: justify; }

Spacing: line-height, letter/word spacing, white-space, wrapping and hyphens

Readability lives in the spacing group. line-height sets leading and is best kept unitless — a multiplier each descendant recomputes against its own size, whereas a fixed length is computed once and inherited flat, breaking on larger text. letter-spacing and word-spacing adjust tracking, and white-space decides whether runs of spaces and newlines collapse and whether lines wrap (normal, pre, nowrap, pre-wrap).

Why it matters: overflow control is a cluster interviewers love to untangle. overflow-wrap: break-word breaks a long unbreakable string only as a last resort, word-break: break-all breaks eagerly between any characters, and hyphens: auto inserts language-aware hyphens (it needs a lang attribute on the element). Confusing these three is behind the classic "a long URL blows out my layout" bug.

body    { line-height: 1.6; }
.code   { white-space: pre-wrap; }               /* keep spaces, still wrap */
.cell   { overflow-wrap: break-word; hyphens: auto; }

writing-mode, direction and logical text

writing-mode lays lines horizontally or vertically (horizontal-tb, vertical-rl, vertical-lr), while direction — better set through the HTML dir attribute — chooses left-to-right or right-to-left inline flow for scripts like Arabic and Hebrew. Together they define the block and inline axes of the element. Logical properties (margin-inline-start, padding-block, inline-size and friends) then reference those axes instead of the physical top/right/bottom/left.

Why it matters: writing one stylesheet that flips correctly for RTL and vertical scripts is exactly why logical properties exist — margin-inline-end stays on the reading-end side whether the language runs left or right, where margin-right cannot. For Indian and global products that ship in multiple scripts, this shows up increasingly in senior front-end rounds.

/* flips automatically for RTL — no separate stylesheet */
.card { margin-inline-start: 1rem; padding-block: 0.5rem; }
.vertical-label { writing-mode: vertical-rl; }

Interview questions

Most-asked typography questions in Indian interviews, from front-end screening rounds through senior product-company deep dives:

Beginner What does the font shorthand set, and which parts are required?

It can set font-style, font-variant, font-weight, font-stretch, font-size, line-height and font-family in one declaration. Only font-size and font-family are required, and font-family must come last; line-height attaches to font-size with a slash. Any longhand you omit is reset to its initial value.

Beginner Difference between px, em and rem for font-size?

px is a fixed length. em is relative to the parent's font-size, so nested elements compound. rem is relative to the root font-size, avoiding compounding. Both em and rem honour the user's browser font-size preference, which is why they beat px for accessible, scalable text.

Beginner Why is a unitless line-height preferred?

A number is inherited as a factor and each element recomputes it against its own font-size. A length or percentage is computed once on the ancestor and children inherit that fixed pixel value — so a larger heading inside can get a line box too short for its text. Unitless (for example 1.5) scales correctly everywhere.

Beginner Does text-transform: uppercase change the actual text?

No — it is purely visual. The DOM text, copy-paste result and screen-reader announcement all keep the original case. It is a presentation layer over the content, which is the accessibility point to make.

Intermediate overflow-wrap vs word-break vs hyphens — when do you use each?

overflow-wrap: break-word breaks a long unbreakable string only when it would otherwise overflow — the right tool for stray long URLs. word-break: break-all breaks between any two characters aggressively (CJK-oriented). hyphens: auto inserts real, language-aware hyphens and needs a lang attribute plus browser dictionary support.

Intermediate How do @font-face and font-display avoid FOIT?

FOIT is invisible text while a web font downloads. font-display: swap renders fallback text immediately and swaps to the web font when ready (accepting a flash, FOUT); fallback and optional shorten or drop that swap for performance. Pairing swap with preload on the WOFF2 file shortens the flash.

Intermediate What are variable fonts and why use one?

A single font file that exposes continuous axes — weight (wght), width (wdth), optical size (opsz) and more — so you get every intermediate value, not just named weights. One request replaces many static files, cutting payload and round-trips. You drive axes with font-weight, font-stretch or font-variation-settings.

Intermediate How does a font-family fallback list work, and what is a system font stack?

The comma-separated list is tried in order until a face is available, and a generic family (serif/sans-serif) should be last as a guaranteed fallback. A system font stack starts with system-ui (or the native OS UI fonts) so the interface uses the platform font with zero download — fast and familiar.

Intermediate Why do logical properties matter for right-to-left layouts?

Properties like margin-inline-start, padding-inline-end and inline-size map to the flow axes (inline/block) rather than physical sides. Set dir="rtl" or a vertical writing-mode and the same rule flips automatically, so you maintain one stylesheet instead of mirroring every left/right by hand.

Pro Adding a font shorthand wiped my earlier font-weight — why?

The font shorthand resets every one of its longhands, so any component you do not name (font-weight, font-style, font-variant, line-height) reverts to its initial value. If a later, equally specific rule applies the shorthand, it overrides the standalone font-weight. Fix: include the weight in the shorthand, or apply the longhand after it.

Pro white-space: normal vs pre vs pre-wrap, with a real use for each?

normal collapses whitespace and wraps — ordinary body text. pre preserves whitespace and newlines and does not wrap — verbatim output where horizontal scroll is acceptable. pre-wrap preserves whitespace and newlines but still wraps — ideal for rendering user-entered text or code snippets that must stay responsive.

Pro What do text-box-trim and text-box-edge solve, and what is the catch?

Fonts ship built-in space above the cap height and below the baseline (half-leading), so a heading rarely aligns to its visual box. text-box-trim with text-box-edge trims that space to cap/baseline for exact vertical rhythm and consistent gaps between text elements. The catch: browser support is still limited, so it needs feature detection or a negative-margin fallback for older engines.
