Module: Entity Mapping Basics

An entity is a plain Java class that a JPA provider agrees to keep in sync with a database row — this module is about the annotations that draw that class-to-table contract, and the one detail (field vs property access) that quietly decides how Hibernate reads every attribute you own.

Knowledge Library → Courses → Hibernate and JPA → Entity Mapping Basics

What this module covers

Object-relational mapping starts here: turning a POJO into a managed entity. You will map the class to a table (@Entity, @Table, @Column), understand how Hibernate reaches an attribute (field vs property access), mark data in and out of persistence (@Basic, @Transient), let Hibernate name tables and columns for you (naming strategies), generate or validate the schema at boot (ddl-auto), and store dates the modern way with java.time. Everything after this — identifiers, relationships, inheritance — is layered on top of these basics.

Topics

Mapping a class to a table: @Entity, @Table, @Column

@Entity is the one annotation that makes a class persistent — it must be a non-final class with a no-arg constructor and an @Id. Without further hints Hibernate applies sensible defaults: the table is named after the class, each non-transient attribute becomes a column named after the field. @Table overrides the table name (and its schema, catalog, unique constraints); @Column overrides a single column's name and declares metadata like nullable, length, unique and insertable/updatable.

Why it matters: this is the vocabulary of every entity you will ever write, and interviewers lean on one subtlety — @Entity(name=...) renames the entity for JPQL, while @Table(name=...) renames the physical table. They are not the same knob.

@Entity
@Table(name = "app_user")
public class User {
    @Id @GeneratedValue
    private Long id;

    @Column(name = "email", nullable = false, unique = true, length = 320)
    private String email;
}

Field access vs property access

Hibernate reads and writes an attribute in one of two ways: field access sets the instance variable directly by reflection, or property access calls the getter and setter. The default is decided by where you put the mapping annotations — specifically @Id: on the field means AccessType.FIELD for the whole class, on the getter means AccessType.PROPERTY. @Access overrides the default, per class or per attribute, and is the only way to mix the two safely.

Why it matters: this is one of the most common JPA interview questions because the choice has real consequences. Property access runs your getter/setter logic on every load and flush (a derived getter can corrupt the mapping); field access ignores methods entirely, so a getter can safely return an Optional or a computed view. Field access is the community default recommendation.

@Entity
@Access(AccessType.FIELD)          // explicit; annotations live on fields
public class Product {
    @Id private Long id;
    private String name;

    @Access(AccessType.PROPERTY)   // override just this attribute
    @Column(name = "price_cents")
    public long getPriceCents() { return Math.round(price * 100); }
}

@Basic and @Transient

Every non-static, non-transient attribute is persistent by default — @Basic is implicit and rarely written. When it is written it carries two useful knobs: optional (whether the column may be null) and fetch = FetchType.LAZY (defer loading a large value). To keep an attribute out of the database, use JPA's @Transient annotation; the Java transient keyword also works but couples persistence to serialization, so @Transient is the clearer intent.

Why it matters: candidates routinely confuse the two "transients." @Transient affects persistence only; the transient keyword affects Java serialization only — and each incidentally influences the other. Knowing which tool expresses which intent is a quick signal of real JPA experience.

Naming strategies: implicit and physical

When you do not name a table or column explicitly, Hibernate derives it in two steps. The implicit naming strategy produces a logical name from the attribute or class name; the physical naming strategy then maps that logical name to the actual database identifier. Splitting the two lets you apply a house style — say, snake_case — in one place instead of hard-coding @Column(name=...) on every field.

Why it matters: Spring Boot silently installs its own CamelCaseToUnderscoresNamingStrategy, so firstName becomes first_name even though plain Hibernate would keep it firstname. Not knowing this is behind a classic "why can't it find my column?" bug when moving between a Spring Boot app and vanilla Hibernate.

Schema generation: ddl-auto

Hibernate can create, update, validate or drop the schema at startup from your entity mappings. The raw setting is hibernate.hbm2ddl.auto; Spring Boot exposes it as spring.jpa.hibernate.ddl-auto with values none, validate, update, create and create-drop. Boot defaults to create-drop for an embedded database and none otherwise — convenient for demos and tests.

Why it matters: update is the trap. It never drops columns, cannot express real migrations, and quietly diverges from your intended schema; the standard interview answer is validate or none in production with a migration tool (Flyway or Liquibase) owning the actual DDL.

# production
spring.jpa.hibernate.ddl-auto=validate
# local dev / integration tests
spring.jpa.hibernate.ddl-auto=create-drop

@Temporal and modern java.time mapping

The java.time types map cleanly with no annotation: LocalDate to DATE, and LocalDateTime/Instant/OffsetDateTime to TIMESTAMP. @Temporal exists only for the legacy java.util.Date and Calendar types, to say whether they become a DATE, TIME or TIMESTAMP — and it is deprecated in Jakarta Persistence 3.2. Applied to a java.time field it is unnecessary and, in recent Hibernate, harmful.

Why it matters: the right answer to "how do I persist a LocalDateTime?" is "just declare it" — no @Temporal. Interviewers use this to separate people who learned JPA on Java 8+ from those still copying @Temporal onto everything out of habit. Time zones then become the follow-up: ZonedDateTime storage is controlled with Hibernate's @TimeZoneStorage.

@Column(name = "created_at")
private Instant createdAt;          // -> TIMESTAMP, no @Temporal

@Temporal(TemporalType.DATE)        // ONLY for legacy java.util.Date
private java.util.Date legacyBirthday;

Interview questions

The entity-mapping questions that recur in Indian JPA and Spring interviews — screening rounds through product-company deep dives, with field-vs-property access almost guaranteed:

Beginner What makes a class a JPA entity?

The @Entity annotation, plus a few rules: a non-final class, a public or protected no-argument constructor, and at least one @Id attribute. Hibernate then maps it to a table (named after the class unless @Table says otherwise) and manages its persistent state.

Beginner Difference between @Entity(name) and @Table(name)?

@Entity(name=...) sets the entity name used in JPQL/HQL queries; @Table(name=...) sets the physical table name in the database. They default to the class name and are independent — you can rename one without the other.

Beginner What does @Column configure, and name some attributes?

It customizes the column a field maps to: name, nullable, unique, length (for strings), precision/scale (for decimals), and insertable/updatable to exclude a column from generated INSERT/UPDATE statements. It is optional — defaults apply when absent.

Beginner How do you stop a field from being persisted?

Annotate it with @Transient (JPA), or use the Java transient keyword. @Transient affects persistence only; the keyword affects serialization only but also stops persistence. Prefer @Transient for clarity of intent.

Beginner Is @Basic required on a persistent field?

No. Every non-static, non-transient attribute is a basic mapping by default, so @Basic is implicit. You write it only to set optional=false or fetch=FetchType.LAZY (deferred loading of, say, a large text/blob column).

Intermediate Field access vs property access — how does Hibernate choose the default?

By where the mapping annotations, chiefly @Id, are placed. On the field → AccessType.FIELD (Hibernate sets the variable directly by reflection). On the getter → AccessType.PROPERTY (Hibernate calls getters/setters). @Access overrides this at class or attribute level.

Intermediate Why do most teams prefer field access?

Field access ignores your methods, so a getter is free to return an Optional, a defensive copy, or a computed value without breaking the mapping; it also avoids side effects on every load/flush. Property access runs getter/setter logic each time, so a "clever" accessor can silently corrupt persisted state.

Intermediate What are implicit and physical naming strategies, and what does Spring Boot default to?

The implicit strategy derives a logical name for anything you did not name explicitly; the physical strategy converts that logical name into the actual DB identifier. Spring Boot defaults the physical strategy to CamelCaseToUnderscoresNamingStrategy, so firstName maps to first_name — different from vanilla Hibernate, which would use firstname.

Intermediate What does spring.jpa.hibernate.ddl-auto do, and which value in production?

It controls Hibernate schema management at startup: none, validate, update, create, create-drop. In production use validate (or none) and let Flyway/Liquibase own migrations. Boot defaults to create-drop for embedded databases, none otherwise.

Intermediate How do you persist a LocalDate or LocalDateTime — do you need @Temporal?

No. java.time types map automatically — LocalDate to DATE, LocalDateTime/Instant to TIMESTAMP. @Temporal is only for legacy java.util.Date/Calendar and is deprecated in Jakarta Persistence 3.2; putting it on a java.time field is unnecessary and can misbehave in recent Hibernate.

Pro Can you mix field and property access in one entity, and what's the risk?

Yes — set a class default with @Access and override individual attributes with @Access(AccessType.PROPERTY/FIELD). The risk: mixing implicitly (annotations scattered across fields and getters) without explicit @Access is undefined by the JPA spec, so behaviour becomes provider-dependent. Always make mixed access explicit.

Pro Why is ddl-auto=update dangerous in production?

It only makes additive changes — it never drops or alters columns, cannot rename, and cannot run data migrations — so the live schema silently drifts from what your mappings imply, and there is no reviewable, versioned record of changes. Real systems freeze the schema (validate) and drive DDL through Flyway/Liquibase.

Pro How do you correctly store a timestamp with a time zone?

Prefer Instant or OffsetDateTime (mapped to TIMESTAMP) so the value is unambiguous. For ZonedDateTime/OffsetDateTime, Hibernate's @TimeZoneStorage chooses whether to normalize to UTC or keep an explicit offset/zone column; storing local LocalDateTime for cross-zone data loses the offset and is the common bug.
