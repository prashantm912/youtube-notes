Module: Entity Relationships

Relationships are where JPA stops being CRUD and becomes an object graph — and the owning side, mappedBy and keeping both ends in sync are where most of the bugs, and most of the interview questions, actually live.

Knowledge Library → Courses → Hibernate and JPA → Entity Relationships

What this module covers

An entity rarely stands alone: orders have items, users have a profile, posts have tags. This module maps those links with the four association annotations, then focuses on the concept that separates a working mapping from a broken one — which side owns the foreign key. Owning versus inverse, mappedBy, and the helper methods that keep a bidirectional graph consistent are the core, because they are simultaneously the commonest production bug and the commonest interview question in this space.

Topics

The four cardinalities: @ManyToOne, @OneToMany, @OneToOne, @ManyToMany

JPA models associations with four annotations that mirror relational multiplicities. @ManyToOne maps the side that carries the foreign key — many rows referencing one parent — and is the single most important of the four. @OneToMany is its collection-valued inverse. @OneToOne links two rows one for one, and @ManyToMany maps a pair of entities through a join table. Cardinality follows the data model, not coding convenience.

Why it matters: the workhorse pairing is @ManyToOne with @OneToMany. The established best practice is to always map the @ManyToOne child (it owns the foreign-key column) and, if you need the collection, add a bidirectional @OneToMany(mappedBy = ...). A unidirectional @OneToMany is discouraged because Hibernate then manages the link through an extra join table or a batch of redundant UPDATE statements. Knowing which side holds the foreign key is a standard screening question.

@Entity class OrderItem {
    @ManyToOne(fetch = FetchType.LAZY)   // owns the FK column
    @JoinColumn(name = "order_id")
    private Order order;
}
@Entity class Order {
    @OneToMany(mappedBy = "order")       // inverse side, no FK here
    private List<OrderItem> items = new ArrayList<>();
}

@OneToOne: foreign key vs shared primary key

A @OneToOne can be mapped two ways. The naive way adds a foreign-key column with @JoinColumn (usually marked unique). The better way shares the primary key: the child reuses the parent's identifier via @MapsId, so there is no separate foreign-key column, the child is fetchable directly by the parent id, and the one-to-one guarantee is enforced by the primary key itself.

Why it matters: the shared-primary-key mapping is the interview-preferred answer, and it comes with a famous gotcha. The parent side of a bidirectional @OneToOne cannot be truly lazy-loaded by default — even with fetch = LAZY — because Hibernate must issue a query to know whether to store a proxy or null; bytecode enhancement is what fixes it. Expect the "why is my LAZY one-to-one still eager?" follow-up.

@Entity class PostDetails {
    @Id private Long id;                 // same id as the Post
    @OneToOne(fetch = FetchType.LAZY)
    @MapsId                              // share the parent's primary key
    @JoinColumn(name = "id")
    private Post post;
}

@ManyToMany and the join table

A @ManyToMany links two entities through a join table holding two foreign-key columns, configured with @JoinTable, joinColumns and inverseJoinColumns. One side owns the mapping; the other declares mappedBy and only mirrors it.

Why it matters: the defaults bite. Use Set, not List — with a List, removing one element makes Hibernate delete every join-table row and reinsert the survivors, while a Set deletes just the one row. Never use CascadeType.REMOVE on a many-to-many, since a chain delete can wipe both sides of shared data. And the moment the association needs its own attributes — a quantity, a price, an added-on date — drop @ManyToMany entirely and model the join table as an explicit entity with two @ManyToOne sides. The "add an extra column" follow-up is a favourite.

@Entity class Post {
    @ManyToMany(cascade = { CascadeType.PERSIST, CascadeType.MERGE })
    @JoinTable(name = "post_tag",
        joinColumns = @JoinColumn(name = "post_id"),
        inverseJoinColumns = @JoinColumn(name = "tag_id"))
    private Set<Tag> tags = new HashSet<>();
}

Owning side vs inverse side, @JoinColumn and mappedBy

Every bidirectional association has exactly one owning side and one inverse (non-owning) side. The owning side is the one Hibernate reads to decide the foreign-key or join-table state, and it declares @JoinColumn or @JoinTable. The inverse side declares mappedBy, naming the owning field, and is purely a read-through view of the same link. The rule is fixed for the common case: in a @ManyToOne/@OneToMany pair the many side (@ManyToOne) is always the owner; only in a @ManyToMany do you get to choose.

Why it matters: this is the single most-tested relationship concept, because getting it wrong fails quietly. If you set only the inverse side, Hibernate generates no SQL for the link and the foreign key stays null — the update simply vanishes. Put @JoinColumn and mappedBy on the wrong sides and you get a duplicate foreign key or a phantom column. Reason about it as: the side without mappedBy is the source of truth.

// Owning side — declares the FK, drives the SQL
@ManyToOne @JoinColumn(name = "author_id") private Author author;

// Inverse side — mirrors it, generates no SQL of its own
@OneToMany(mappedBy = "author") private List<Book> books;

Keeping both sides in sync: bidirectional helper methods

Because only the owning side drives SQL, the in-memory object graph can silently drift from the database when you touch just one end. Add a value to the child's reference but forget the parent's collection and, until the persistence context is cleared, code that reads the collection sees stale data. The fix is a pair of helper methods on the parent — addChild / removeChild — that update both ends together, so the object graph always matches what Hibernate will persist.

Why it matters: consistent bidirectional state is what makes orphanRemoval, cascading and dirty checking behave predictably; without helpers, "my child does not get saved" and "my collection is stale" are recurring, hard-to-explain bugs. Interviewers ask you to write these methods on the spot.

@Entity class Post {
    @OneToMany(mappedBy = "post", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Comment> comments = new ArrayList<>();

    public void addComment(Comment c) { comments.add(c); c.setPost(this); }
    public void removeComment(Comment c) { comments.remove(c); c.setPost(null); }
}

Interview questions

Relationship mapping is the deepest-tested JPA area in Indian interviews — owning side, mappedBy and bidirectional sync recur from screening rounds to senior product-company deep dives:

Beginner What are the four relationship annotations in JPA and what does each represent?

@ManyToOne (many rows point to one parent — carries the foreign key), @OneToMany (its collection-valued inverse), @OneToOne (one row to one row), and @ManyToMany (two entities linked through a join table). They map directly to relational multiplicities.

Beginner In a @ManyToOne/@OneToMany relationship, which side holds the foreign key?

The many side — the @ManyToOne — holds the foreign-key column in its table. The one side (@OneToMany) has no foreign key of its own; it is the inverse view mapped with mappedBy.

Beginner What does the mappedBy attribute do?

It marks a side as the inverse (non-owning) side and names the field on the owning side that maps the association. It tells Hibernate "the other side manages this link — do not generate a second foreign key or join table for it."

Beginner What is a join table and which annotation configures it?

An intermediate table holding two foreign-key columns that connect a many-to-many association. @JoinTable configures it, with joinColumns for the owning side and inverseJoinColumns for the other side.

Intermediate What is the difference between the owning side and the inverse side of a bidirectional association?

The owning side is the one Hibernate reads to generate the SQL for the foreign key or join table; it declares @JoinColumn/@JoinTable. The inverse side declares mappedBy and is a read-only mirror — changes to it alone produce no SQL. The side without mappedBy is the source of truth. This is the most-asked relationship question.

Intermediate You set only the inverse side of a bidirectional relationship, but the foreign key is null in the database. Why?

Hibernate generates the foreign-key SQL only from the owning side. Adding a child to the parent's mappedBy collection without also setting the child's @ManyToOne back-reference leaves the owning side unset, so nothing is written. Set the owning side (or use a helper method that sets both).

Intermediate Why are unidirectional @OneToMany associations discouraged?

Without a @ManyToOne owning side, Hibernate manages the link either through an unnecessary join table or, with @JoinColumn, through extra UPDATE statements after the inserts. Mapping the @ManyToOne side (optionally bidirectional with mappedBy) produces clean, minimal SQL.

Intermediate Why do bidirectional associations need add/remove helper methods?

Only the owning side drives persistence, but application code reads both sides. Helpers like addComment/removeComment set both ends at once, keeping the in-memory graph consistent with what gets persisted — which is what makes orphanRemoval, cascading and dirty checking behave correctly and avoids stale-collection bugs.

Intermediate When do you use @JoinColumn versus mappedBy?

@JoinColumn goes on the owning side to name the foreign-key column; mappedBy goes on the inverse side to point back at the owning field. They are mutually exclusive per side — putting both on the same field, or @JoinColumn on both sides, creates a duplicate foreign key.

Intermediate How do you map a @OneToOne with a shared primary key, and why prefer it?

Use @MapsId so the child entity reuses the parent's identifier as its own primary key and foreign key. It removes a separate unique foreign-key column, guarantees the one-to-one at the primary-key level, and lets you fetch the child directly by the parent id.

Pro For @ManyToMany, why prefer Set over List, and why avoid CascadeType.REMOVE?

With a List, removing one element makes Hibernate delete all join-table rows and reinsert the survivors; a Set deletes only the single affected row. CascadeType.REMOVE is dangerous because deleting one entity can chain-delete shared associated rows, wiping data the other side still needs. Keep cascade to PERSIST and MERGE.

Pro Your many-to-many needs an extra column, such as a quantity. How do you model it?

Replace @ManyToMany with an explicit join entity (for example OrderProduct) that has its own key and two @ManyToOne associations to the endpoints, plus the extra columns. This turns the many-to-many into two one-to-many relationships and gives the link its own identity and attributes.

Pro Why can't the parent side of a bidirectional @OneToOne be lazy-loaded by default, and what fixes it?

The parent has no foreign key, so Hibernate must run a query to know whether the child exists — it cannot decide between a proxy and null without hitting the database, so it loads eagerly even with fetch = LAZY. Bytecode enhancement (lazy loading with a field-level interceptor), or restructuring to a shared primary key / @MapsId, is what makes it genuinely lazy.
