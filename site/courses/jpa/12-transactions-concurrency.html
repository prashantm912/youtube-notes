Module: Transactions and Concurrency

A managed persistence context hides the SQL, but two users editing the same row is a correctness problem no framework quietly solves for you — this is the module where JPA stops being merely convenient and starts being careful.

Knowledge Library → Courses → Hibernate and JPA → Transactions and Concurrency

What this module covers

Two threads sit under every persistence layer: the transaction that groups your writes into an all-or-nothing unit, and the concurrency control that decides what happens when two of those units touch the same data. This module covers how @Transactional draws boundaries and when Hibernate flushes, then the lost-update anomaly and its two cures — optimistic locking with @Version and pessimistic locking with LockModeType. Optimistic-versus-pessimistic locking and the lost-update problem are among the most-asked JPA interview topics, so they get the most room here.

Topics

@Transactional and transaction boundaries

@Transactional marks a method as a unit of work: Spring's proxy opens a physical database transaction before the method runs and commits (or rolls back on a runtime exception) after it returns. Everything inside shares one connection and one persistence context, so the entities you load stay managed and dirty-checking works across all the repository calls you make. Put it on the service layer, where a business operation spans several repositories — not on individual repository methods, which already get their own tiny transaction by default.

Why it matters: the boundary is where correctness lives. Too narrow and a multi-step operation can half-commit; too wide and you hold a connection while doing slow work. Two facts trip people up in interviews — a runtime (unchecked) exception rolls back but a checked exception does not unless you configure rollbackFor, and self-invocation (one method in a class calling another) bypasses the proxy, so the inner @Transactional is silently ignored.

@Service
class OrderService {
  @Transactional                       // one physical transaction for the whole method
  public void placeOrder(Order o) {
    orderRepo.save(o);
    inventoryRepo.decrement(o.sku());  // both succeed, or both roll back together
  }                                    // commit + flush happen here, on return
}

Flush timing: flush-on-commit and FlushMode

persist, merge and dirty state do not hit the database immediately — Hibernate buffers them in the persistence context and flushes (runs the actual INSERT/UPDATE/DELETE SQL) only when it has to. With the default FlushModeType.AUTO that happens at commit and again before a query whose results could be affected by pending changes, so a JPQL query never reads stale in-memory state. This is the "flush-on-commit" behaviour: your SQL is emitted at the transaction boundary, then the COMMIT follows.

Why it matters: flush is not commit. Calling entityManager.flush() pushes SQL early (useful to surface a constraint violation or read a generated id) but the transaction can still roll back. Switching to FlushModeType.COMMIT skips the pre-query flush for speed but risks a query reading pre-change data — a subtle bug worth being able to explain.

Product p = em.find(Product.class, id);
p.setStock(0);                    // change is buffered, no SQL yet
em.createQuery("select p from Product p ...").getResultList();
                                  // AUTO flushes first, so the query sees stock = 0
// tx.commit();                   // flush-on-commit emits any remaining SQL, then COMMIT

The lost-update problem

The lost update is the anomaly all of this exists to prevent. Two transactions read the same row, each computes a new value from what it read, and both write back — the second write silently overwrites the first, and a change simply vanishes with no error. Under READ COMMITTED (the common default) the database does nothing to stop this, because neither transaction sees the other's uncommitted work. It gets worse in "long conversations" where a user reads a record in one HTTP request and saves it several requests later: the physical database transaction is long gone, so no row lock could have helped.

Why it matters: this is the concurrency scenario interviewers reach for first, usually as a story ("two admins edit the same product price — what happens?"). The correct answer names the anomaly, notes that isolation level alone rarely fixes it cheaply, and reaches for a locking strategy — optimistic by default, pessimistic when contention is high.

T1 reads balance = 100        T2 reads balance = 100
T1: 100 - 20  -> UPDATE 80
                              T2: 100 + 50 -> UPDATE 150
final balance = 150           (T1's debit of 20 is lost)

Optimistic locking with @Version

Optimistic locking assumes conflicts are rare and detects them at write time instead of blocking readers. You add one @Version attribute (an int/long or a timestamp) and Hibernate takes over: every UPDATE carries the version in its WHERE clause and increments it. If another transaction already bumped the version, zero rows match, and Hibernate raises an OptimisticLockException — the conflicting write is refused rather than silently lost. No database lock is held, so it scales well and even works across detached entities in a long conversation.

Why it matters: it is the default recommendation for read-heavy, low-contention systems, and "how does @Version prevent lost updates?" is a near-guaranteed follow-up. Know the rules: exactly one version field per entity, never set it yourself, and on the recovery side catch the exception and retry the whole unit of work — the state you read is stale, so re-read before recomputing.

@Entity
class Product {
  @Id Long id;
  @Version int version;   // managed by Hibernate — never assign it yourself
  int stock;
}
// UPDATE product SET stock = ?, version = ? WHERE id = ? AND version = ?
//   -> 0 rows updated means someone else changed it first -> OptimisticLockException

Pessimistic locking with LockModeType

Pessimistic locking assumes conflicts are likely and takes a database lock up front so no one else can interfere. You request it with a LockModeType: PESSIMISTIC_WRITE acquires an exclusive lock (a SELECT ... FOR UPDATE) that blocks other readers and writers of that row, while PESSIMISTIC_READ takes a shared lock that permits concurrent readers but blocks writers. The lock is held until the transaction commits or rolls back. In Spring Data you attach it declaratively with @Lock on a repository method; with the EntityManager you pass the mode to find, lock, or a query.

Why it matters: pessimistic locking is the right tool when contention is high and retrying would be expensive — inventory decrements, seat or ticket booking, wallet debits. The trade-off is throughput and the risk of deadlocks, so bound it with a lock timeout (jakarta.persistence.lock.timeout) and acquire locks in a consistent order. Being able to contrast it with optimistic locking — and say which you'd pick for a booking system — is a classic senior-round question.

public interface SeatRepository extends JpaRepository<Seat, Long> {
  @Lock(LockModeType.PESSIMISTIC_WRITE)
  @Query("select s from Seat s where s.id = :id")
  Optional<Seat> findByIdForUpdate(@Param("id") Long id);
}
// emits SELECT ... FOR UPDATE — concurrent writers block until this tx commits

Interview questions

Transactions and locking are among the most-asked JPA topics in Indian backend interviews — optimistic vs pessimistic locking and the lost-update problem surface in nearly every product-company round:

Beginner What does @Transactional do, and where should you put it?

It marks a method as one unit of work: a Spring proxy opens a database transaction before the method, commits after it returns, and rolls back on a runtime exception. All the work inside shares one connection and one persistence context. Put it on the service layer where a business operation spans multiple repositories — not on individual repository methods, which are already transactional on their own.

Beginner What are the transaction boundaries of a Spring Data JPA repository method?

Each CRUD/query method runs in its own transaction by default (SimpleJpaRepository is annotated), with reads marked readOnly = true. That is fine for a single call, but two repository calls that must be atomic need a surrounding @Transactional service method so they share one transaction instead of two.

Beginner When does Hibernate actually write your changes to the database?

On flush, not on every setter. With the default FlushModeType.AUTO, Hibernate flushes before the transaction commits and before a query whose results could be affected by pending changes — so queries never read stale in-memory state. You can force an early flush with entityManager.flush(), but that still is not a commit.

Beginner What is the lost-update problem?

Two transactions read the same row, each computes a new value from what it read, and both write back — the second overwrites the first and a change disappears with no error. Under READ COMMITTED the database does not prevent it, which is why you add optimistic or pessimistic locking on top.

Intermediate How does optimistic locking with @Version prevent lost updates?

Hibernate adds the version to every UPDATE's WHERE clause and increments it: UPDATE ... SET version = 4 WHERE id = ? AND version = 3. If another transaction already moved the version to 4, zero rows match, and Hibernate throws OptimisticLockException instead of silently overwriting. You recover by re-reading the fresh state and retrying.

Intermediate Optimistic vs pessimistic locking — how do you choose?

Optimistic (a @Version check at write time, no lock held) suits read-heavy, low-contention data and scales well; the cost is retrying on the occasional conflict. Pessimistic (a real DB lock taken up front) suits high-contention or high-cost-of-retry operations like inventory or seat booking; the cost is throughput and deadlock risk. Default to optimistic; escalate to pessimistic when conflicts are frequent.

Intermediate PESSIMISTIC_READ vs PESSIMISTIC_WRITE — what is the difference?

PESSIMISTIC_READ takes a shared lock: other transactions can still read the row but not modify it. PESSIMISTIC_WRITE takes an exclusive lock (typically SELECT ... FOR UPDATE): other transactions can neither modify nor acquire a conflicting lock, so writes serialize. Use WRITE when you intend to update the row you just read.

Intermediate What does readOnly = true on @Transactional buy you?

It is a hint, not a guard. Hibernate sets its flush mode so it skips dirty-checking and auto-flush for the transaction, saving work on pure reads, and it can propagate a read-only flag to the JDBC driver. It does not physically forbid an INSERT/UPDATE, though some databases or replicas may reject writes on a read-only connection.

Intermediate Why does @Transactional sometimes do nothing when one method calls another in the same class?

Spring's transaction advice lives in a proxy that wraps the bean. A self-invocation calls the target instance directly, bypassing the proxy, so the inner method's @Transactional (and any propagation setting like REQUIRES_NEW) is ignored. Fixes: move the method to another bean, inject a self-reference, or use AspectJ weaving.

Intermediate Which exception signals an optimistic-lock conflict, and how do you recover?

JPA throws OptimisticLockException; Spring translates it to ObjectOptimisticLockingFailureException. Because the entity you hold is stale, the recovery is to re-read the current row, reapply the user's intent, and retry the transaction — often wrapped in a bounded retry (for example Spring Retry). Blindly retrying with the old snapshot just fails again.

Pro How do you apply a pessimistic lock through a Spring Data repository, and what SQL results?

Annotate the query method with @Lock(LockModeType.PESSIMISTIC_WRITE). Spring issues the select with a FOR UPDATE clause (dialect-specific), so the row is exclusively locked for the rest of the transaction and concurrent writers block until commit. Pair it with a lock timeout so a stuck holder cannot stall the whole queue indefinitely.

Pro A user edits a record across several HTTP requests. How do you prevent a lost update in that long conversation?

Row locks cannot help — the physical transaction ended when the first request returned. Use application-level optimistic locking: send the @Version to the client with the loaded data and send it back on save; the update's WHERE version = ? fails if anyone changed the row meanwhile. For detached entities, merge re-attaches and the version check still applies.

Pro LockModeType.OPTIMISTIC vs OPTIMISTIC_FORCE_INCREMENT — when would you force a version bump?

OPTIMISTIC just verifies at flush time that the version you read is unchanged. OPTIMISTIC_FORCE_INCREMENT additionally increments the version even if the entity itself was not modified — used to signal that a change to a child or association logically modifies the aggregate root, so concurrent edits elsewhere in the aggregate are detected as conflicts.

Pro What concurrency risk does pessimistic locking introduce, and how do you contain it?

Deadlock: two transactions each hold a lock the other needs and both wait forever until the database aborts one. Contain it by acquiring locks in a consistent global order, keeping locked transactions short, setting jakarta.persistence.lock.timeout so a wait fails fast instead of hanging, and handling the resulting PessimisticLockException / LockTimeoutException with a retry.
