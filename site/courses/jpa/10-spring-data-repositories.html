Module: Spring Data JPA Repositories

Declare an interface, get a working data-access layer: Spring Data JPA generates the implementation from your method names or an @Query. Knowing where that generation stops — and what it costs at runtime — is what interviewers actually probe.

Knowledge Library → Courses → Hibernate and JPA → Spring Data JPA Repositories

What this module covers

Spring Data JPA sits on top of a JPA provider (usually Hibernate) and removes the hand-written DAO. This module walks the repository interface hierarchy, the two ways to declare queries — derived from the method name and explicit with @Query (JPQL or native SQL) — and the read-side features that dominate real code and interviews: pagination and sorting through Pageable, Page vs Slice, and interface- and DTO-based projections that fetch only the columns you need.

Topics

The repository interface hierarchy

You never implement a repository — you extend one of Spring Data's interfaces and the container supplies a proxy at startup. The chain runs Repository (empty marker) → CrudRepository (save, findById, findAll, count, delete) → ListCrudRepository (same but returning List instead of Iterable) and PagingAndSortingRepository (findAll with a Sort or Pageable) → JpaRepository, which adds JPA-specific methods like flush, saveAndFlush, getReferenceById and deleteAllInBatch.

Why it matters: choosing the interface is a design decision. Extend CrudRepository to stay persistence-technology-agnostic; extend JpaRepository — what most applications do — when you want the full JPA surface and List return types. Every method here comes for free without a line of implementation.

interface UserRepository extends JpaRepository<User, Long> { }
// inherited, no code written: save, findById (returns Optional),
// findAll, count, delete, existsById — plus JPA extras:
// flush, saveAndFlush, getReferenceById, deleteAllInBatch.

Derived query methods

Spring Data parses the method name into a query: an introducer (find, read, get, query, count, exists, delete), an optional subject, the word By, then property expressions joined with And/Or and enriched by keywords such as GreaterThan, Between, Like, StartingWith, In, IgnoreCase and OrderBy. findByEmailIgnoreCase writes the JPQL for you. Result limiting comes from First/Top in the name or a Limit/Pageable parameter, and a single result maps cleanly to Optional.

Why it matters: derived queries are the fastest way to a correct query and one of the two most-asked Spring Data topics. The discipline interviewers look for is knowing the ceiling — keep derived methods to two or three simple And/Or conditions; past that the name becomes unreadable and you switch to @Query.

List<User> findByLastNameAndActiveTrue(String lastName);
Optional<User> findByEmailIgnoreCase(String email);
List<User> findTop3ByOrderByCreatedAtDesc();
long countByActiveFalse();

@Query: JPQL and native SQL

When the method name can no longer express the query, @Query puts the statement in your hands. By default the value is JPQL; set nativeQuery = true for raw SQL. Bind parameters positionally (?1) or, preferably, by name with :name and @Param. @Query takes precedence over a @NamedQuery of the same name, and it is the natural home for joins, constructor-expression DTOs and anything a derived name would mangle.

Why it matters: this is the other headline Spring Data interview topic. Know the JPQL-vs-native switch, the parameter-binding styles, and the sharp edge — a native query returns raw SQL results, so an injected Sort is not translated and ordering must live inside the SQL itself.

@Query("select u from User u where u.email = :email")
Optional<User> findByEmail(@Param("email") String email);

@Query(value = "SELECT * FROM users WHERE active = true",
       nativeQuery = true)
List<User> findActiveNative();

Pagination and sorting

Add a Pageable parameter (built with PageRequest.of(page, size, sort)) and Spring applies the offset, limit and ordering. The return type decides the cost: Page knows the total element and page counts, which requires an extra COUNT query; Slice only knows whether a next slice exists, so it skips the count — ideal for infinite scroll; a plain List returns the window with no metadata at all. Sorting is either static (OrderBy in the name) or dynamic through a Sort argument.

Why it matters: the Page-vs-Slice count-query trade-off is a favourite interview question because it maps directly to a production performance decision. Reaching for Page on a huge table when the UI only needs "load more" pays for a count query nobody reads.

Page<User> findByActiveTrue(Pageable pageable);            // + COUNT query
Slice<User> findByLastName(String name, Pageable pageable); // no COUNT

Pageable p = PageRequest.of(0, 20, Sort.by("createdAt").descending());
Page<User> page = repo.findByActiveTrue(p);

Projections: interface and DTO

Fetching whole entities to render a two-column list is wasteful; projections select only the properties you need. An interface projection declares getters and Spring returns a proxy — closed projections (getters that match entity properties) generate a narrow SELECT, while open projections using @Value SpEL compute values from the full entity. A class/DTO projection maps chosen columns into a POJO or Java record, typically via a JPQL constructor expression. A dynamic projection takes a Class<T> argument so one method can return either the entity or a DTO.

Why it matters: projections are the idiomatic answer to "how do you avoid over-fetching in Spring Data?" The catch worth stating: closed and constructor-expression projections narrow the SQL, but an open interface projection loads the entire entity first, so the performance win is lost — and DTO results are read-only, never managed.

interface NameOnly { String getFirstName(); String getLastName(); }
List<NameOnly> findByActiveTrue();            // interface projection (proxy)

record UserDto(String firstName, String email) { }
@Query("select new com.app.UserDto(u.firstName, u.email) from User u")
List<UserDto> findAllDtos();                   // DTO / constructor projection

Interview questions

The Spring Data questions that recur in Indian backend interviews — derived queries and @Query are near-guaranteed at every level:

Beginner What is Spring Data JPA and how does it relate to JPA and Hibernate?

A Spring module that eliminates boilerplate DAO code. You declare a repository interface; at startup Spring generates an implementation backed by a JPA provider — Hibernate by default. JPA is the specification, Hibernate the implementation, Spring Data JPA the convenience layer that writes the repository for you.

Beginner Walk through the Spring Data repository interface hierarchy.

Repository (empty marker) → CrudRepository (save, findById, findAll, count, delete) → ListCrudRepository (same, returning List not Iterable) and PagingAndSortingRepository (findAll with Sort or Pageable) → JpaRepository, which adds JPA-specific methods (flush, saveAndFlush, getReferenceById, deleteAllInBatch). You extend one; the proxy is generated.

Beginner CrudRepository vs JpaRepository — when do you use which?

JpaRepository extends PagingAndSortingRepository, which extends CrudRepository, and adds JPA-specific operations plus List return types. Extend CrudRepository to stay technology-agnostic; extend JpaRepository — the common choice — for the full JPA surface. There is no performance penalty either way; it is about the API you expose.

Beginner What is a derived query method and how is the name parsed?

A method whose name Spring turns into a query with no implementation. Structure: introducer (find/read/get/query/count/exists/delete) + optional subject + By + property expressions joined by And/Or and keywords like GreaterThan, Between, Like, In, IgnoreCase, OrderBy. Example: findByLastNameAndActiveTrue.

Beginner How do you return a single optional result or limit the rows of a derived query?

Declare the return type as Optional<T> for a single result — findById already does this. Limit rows with findFirst/findTop (optionally findTop10By…) in the name, or by passing a Limit or Pageable argument.

Intermediate When do you stop using a derived query and switch to @Query?

Rule of thumb: derived queries stay readable up to about two or three simple And/Or conditions. Beyond that — multiple joins, complex predicates, DTO construction, or a name that would become a sentence — switch to @Query, which gives full control over the statement and frees the method name from any convention.

Intermediate How does @Query work, and how do you write a native query?

@Query("…") holds JPQL by default. For raw SQL set nativeQuery = true and put SQL in the value. @Query takes precedence over a @NamedQuery with the same name. A key caveat: with a native query an injected Sort is not translated to SQL, so ordering must be written into the SQL itself.

Intermediate Positional versus named parameters in @Query?

Positional binds by index — ?1, ?2 — to the arguments in order. Named binds :email to a method argument annotated @Param("email"). Named parameters are preferred: they survive argument reordering and read better. For an IN clause, pass a Collection.

Intermediate Page vs Slice vs List as a pagination return type?

Page exposes total element and page counts, which costs an extra COUNT query. Slice only knows whether a next slice exists, skipping the count — good for infinite scroll or "load more". A plain List returns just the window with no metadata. Build the request with PageRequest.of(page, size, sort).

Intermediate How do you sort, statically and dynamically?

Statically with OrderBy in the method name (findByActiveTrueOrderByCreatedAtDesc). Dynamically by passing a Sort (or a Pageable carrying one): Sort.by("createdAt").descending(). Native queries are the exception — the ordering must be embedded in the SQL because Sort is not applied to them.

Intermediate Interface projection vs class/DTO projection?

An interface projection declares getters and Spring returns a proxy; closed projections (getters matching entity properties) produce a narrow SELECT, open ones use @Value SpEL. A class/DTO projection maps selected columns into a POJO or record, usually via a JPQL constructor expression. A dynamic projection takes a Class<T> parameter so one method returns either shape.

Pro Why do projections improve performance, and what is the trap with open interface projections?

Selecting only the needed columns cuts I/O and avoids managing entities (no dirty checking on read-only DTOs). The trap: an open interface projection using @Value SpEL must load the full entity to compute the expression, so the narrow-select benefit disappears. Closed interface projections and constructor-expression DTOs are the ones that actually reduce the columns fetched.

Pro A @Query UPDATE fails or silently does nothing. What is missing, and what breaks the persistence context?

Modifying JPQL (update/delete) needs @Modifying and must run inside a transaction. Bulk modifying queries bypass the persistence context, so already-managed entities go stale — use clearAutomatically/flushAutomatically or clear the context afterwards. This is expanded in the Spring Data JPA Advanced module.

Pro You paginate a query that JOIN FETCHes a one-to-many collection and Hibernate warns it is paginating in memory. Why, and how do you fix it?

Joining a collection multiplies parent rows, so the database offset/limit no longer maps to whole parents; Hibernate (warning HHH000104) applies pagination in memory after fetching everything — a scalability bug. Fix with a two-step approach: page the parent IDs first, then fetch the collection for those IDs (or use @EntityGraph/batch fetching). For very large sets, keyset pagination avoids the deep-offset cost entirely.
