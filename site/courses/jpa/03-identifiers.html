Module: Primary Keys and Identifiers

The identifier is where an entity meets its row — and the two most common ways to get it wrong, choosing IDENTITY over SEQUENCE and writing a naive equals/hashCode, are also the two questions interviewers reach for first.

Knowledge Library → Courses → Hibernate and JPA → Primary Keys and Identifiers

What this module covers

Every entity needs a primary key, and JPA gives you several ways to produce one: the @GeneratedValue strategies (AUTO, IDENTITY, SEQUENCE, TABLE, and UUID), the sequence tuning that decides how many database round-trips your inserts cost, composite keys via @IdClass and @EmbeddedId, and the identity contract itself — a correct equals and hashCode. The choices here quietly govern insert performance and whether entities behave in sets and maps, which is exactly why they dominate JPA interviews.

Topics

@Id and the @GeneratedValue strategies

A single-attribute key is marked with @Id; add @GeneratedValue to have the provider populate it. Five strategies exist: IDENTITY (an auto-increment column), SEQUENCE (a database sequence object), TABLE (a table simulating a sequence — portable but slow), AUTO (the provider chooses per dialect), and UUID (a generated 128-bit value, standardized in Jakarta Persistence 3.1+). Hibernate 6's AUTO resolves to SEQUENCE for numeric keys and to the UUID generator for java.util.UUID fields.

Why it matters: the strategy decides when the id becomes available and what SQL is emitted, so it is a design decision, not a default to accept blindly. Knowing all five and Hibernate's AUTO resolution is standard screening material.

@Entity
public class Author {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE)
    private Long id;
}

IDENTITY vs SEQUENCE: batching and sequence tuning

This is the most-tested identifier topic. With IDENTITY, the id is assigned by the database only after the INSERT executes, so Hibernate must run each insert immediately and cannot use JDBC batching — a real throughput ceiling for bulk writes. SEQUENCE fetches the next value before the insert, so Hibernate can defer and batch statements and reuse one sequence across tables. Tuning comes from allocationSize on @SequenceGenerator (default 50): raising it makes Hibernate pull a block of ids per database call using the pooled optimizer, cutting round-trips — but the JPA allocationSize must match the database sequence's increment, or ids collide.

Why it matters: recommend SEQUENCE by default, explain the batching consequence of IDENTITY, and describe the pooled optimizer, and you have answered the single most common JPA performance question convincingly.

@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "author_seq")
@SequenceGenerator(name = "author_seq", sequenceName = "author_seq",
                   allocationSize = 50)   // one DB call per 50 ids (pooled)
private Long id;

Composite keys: @IdClass vs @EmbeddedId

When the key spans more than one column, JPA offers two mappings. @EmbeddedId groups the columns into an @Embeddable class used as one field — cleaner when you treat the key as a whole object. @IdClass keeps the fields flat on the entity (each marked @Id) with a separate matching id class — handier when you query the parts individually. Either way the key class must be public, have a no-arg constructor, implement Serializable, and override equals and hashCode. Modern practice favors a surrogate key on such join entities and reserves composite keys for genuine natural or many-to-many association keys.

@Embeddable
public class OrderItemId implements Serializable {
    private Long orderId;
    private Long productId;   // + equals() and hashCode()
}
@Entity
public class OrderItem {
    @EmbeddedId private OrderItemId id;
}

Natural keys and equals()/hashCode() for entities

The other headline interview topic. A JPA entity's equals/hashCode must stay consistent across every state transition — transient, managed, detached — or the entity vanishes from a Set after being persisted. The trap: a database-generated id is null before the flush, so building the hash from it changes the hash mid-life. Two safe recipes: use a stable natural/business key (an email, an ISBN — known at construction) in both methods, or, for surrogate keys, compare on the id with a null guard while returning a constant hashCode (often getClass().hashCode()). Never let a generated field into hashCode directly.

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Author a)) return false;
    return id != null && id.equals(a.id);
}
@Override
public int hashCode() { return getClass().hashCode(); }

Interview questions

The identifier questions Indian interviewers ask most — generation strategy and entity equality lead almost every Hibernate round:

Beginner What does @GeneratedValue do, and what strategies can it use?

It tells the provider to generate the primary key value for a field marked @Id. Strategies: IDENTITY (auto-increment column), SEQUENCE (database sequence), TABLE (a table simulating a sequence), AUTO (provider picks per dialect), and UUID (Jakarta Persistence 3.1+).

Beginner What is the difference between @Id and @GeneratedValue?

@Id declares which attribute is the primary key. @GeneratedValue is optional and says the value is generated rather than assigned by your code. An @Id without @GeneratedValue is an application-assigned (often natural) key.

Beginner What does GenerationType.AUTO resolve to?

The provider chooses based on the dialect and field type. In Hibernate 6 a numeric id resolves to SEQUENCE (or TABLE if sequences are unsupported), and a java.util.UUID field resolves to the UUID generator — a change from older versions that defaulted to a shared hi/lo table.

Beginner When is the id value available for IDENTITY versus SEQUENCE?

With IDENTITY the id exists only after the INSERT runs and the database assigns it. With SEQUENCE Hibernate fetches the next value before the insert, so the id is known while the entity is still being prepared.

Intermediate IDENTITY vs SEQUENCE — which do you prefer and why?

SEQUENCE, in most cases. Because IDENTITY needs the database to assign the id per row, Hibernate must execute each insert immediately and cannot batch them — a throughput ceiling under bulk writes. SEQUENCE gets the id up front, so inserts can be deferred and JDBC-batched, and one sequence can serve several tables. IDENTITY's only edge is simplicity on MySQL, which lacks true sequences before 8.0.

Intermediate Why can't Hibernate batch inserts when using IDENTITY?

JDBC batching means holding a set of inserts and sending them together. But an IDENTITY id is only known after its own insert executes, and Hibernate must have the id to register the entity in the persistence context. It therefore flushes each insert individually, defeating batching.

Intermediate What is allocationSize, and how does the pooled optimizer help?

allocationSize on @SequenceGenerator (default 50) is how many ids Hibernate reserves per database call. With it above 1, Hibernate uses a pooled optimizer — one sequence round-trip yields a block of in-memory ids, cutting network chatter on bulk inserts. Critical rule: the JPA allocationSize must equal the sequence's actual increment in the database, or generated ids can clash.

Intermediate How do you map a composite primary key, and what are @IdClass and @EmbeddedId?

Both express a multi-column key. @EmbeddedId puts the columns in one @Embeddable class held as a single field — best when you use the key as a whole. @IdClass keeps the fields flat on the entity (each @Id) with a parallel id class — handier when you access parts individually or can't modify the key class.

Intermediate What are the requirements for a composite-key class?

It must be a public class, have a public no-arg constructor, implement Serializable, and override equals and hashCode based on all key fields. Hibernate uses those methods to identify the entity in the persistence context, so omitting them breaks lookups.

Intermediate Would you use a UUID as a primary key? Trade-offs?

UUIDs are globally unique and can be generated in the app with no database round-trip — useful for distributed systems and merging data. Costs: 16 bytes vs 8 for a bigint (larger indexes), and random UUIDv4 fragments B-tree indexes badly. Prefer a time-ordered form (UUIDv7) or store as binary to keep inserts index-friendly.

Pro How do you correctly implement equals() and hashCode() for a JPA entity?

Keep them consistent across transient, managed and detached states. If a stable natural/business key exists (email, ISBN), use it in both methods. For surrogate keys, compare on the id with a null guard but return a constant hashCode (commonly getClass().hashCode()) so the hash never changes when the id is assigned at flush. Use instanceof, not getClass() equality, so Hibernate proxies still match.

Pro Why is putting an auto-generated id into hashCode() dangerous?

Before persistence the id is null; after flush it has a value. If hashCode derives from the id, the entity's hash changes mid-life. Add it to a HashSet while transient, persist it, and it lands in the wrong bucket — you can no longer find or remove it. That is the concrete bug behind the constant-hashCode recipe.

Pro Natural key vs surrogate key as the primary key — how do you choose?

A surrogate key (generated numeric/UUID, no business meaning) is stable, compact, and never needs to change — the default for most tables and the safe basis for foreign keys. A natural key (a real attribute) removes a column and is meaningful, but business values mutate and cascade painfully through referencing rows. Common compromise: a surrogate primary key plus a unique constraint (or Hibernate @NaturalId) on the business key for lookups.
