Module: Spring Data JPA Advanced

The senior half of Spring Data JPA: Criteria-backed dynamic Specifications, repositories you extend by hand, bulk writes that skip the persistence context, and auditing that fills who-and-when for free — the features that separate a CRUD developer from someone who owns the data layer.

What this module covers

The previous module covered the repositories you get for free; this one covers what you reach for when derived queries run out. Specifications and Querydsl build type-checked dynamic queries over the JPA Criteria API, custom fragments let you drop to a raw EntityManager without leaving the repository abstraction, @Modifying runs bulk updates and deletes, @EntityGraph defeats N+1 declaratively, and Spring Data auditing stamps creation and modification metadata automatically. All of it is heavily interviewed at the mid-and-senior level in Indian product and services companies.

Topics

Specifications: dynamic queries over the Criteria API

A Specification<T> wraps a single JPA Criteria predicate through toPredicate(root, query, criteriaBuilder), and specifications compose with and(), or(), not() and where(...). A repository gains them by extending JpaSpecificationExecutor<T>, which adds findAll(Specification), findOne, count and exists — all overloaded to accept Pageable and Sort.

Why it matters: the moment a screen has several optional filters, deriving findByAAndBAndC permutations explodes. Specifications let you assemble exactly the predicates a request supplied, at runtime, from small reusable pieces — the standard answer to "how do you build a dynamic query in Spring Data?".

static Specification<Book> byAuthor(String a) {
    return (root, query, cb) -> cb.equal(root.get("author"), a);
}
// compose at runtime, then page:
repo.findAll(byAuthor("Vlad").and(inStock()), pageable);

Custom repository fragments

When derived methods and @Query are not enough — you need the raw EntityManager, hand-built Criteria, or another library — you write a fragment: a custom interface plus an implementation class whose name is the interface name with the Impl postfix. The repository interface extends both the base repository and the fragment, and Spring Data composes the two into one proxy. Multiple fragments can be layered, and declaration order resolves any method-name ambiguity.

Beyond per-repository fragments you can also swap the base class — extend SimpleJpaRepository and register it via repositoryBaseClass to give every repository a shared behaviour (a findByIdOrThrow, soft-delete support, and so on). The point is to keep bespoke JPQL or Criteria code out of the interface while keeping Spring Data's wiring.

interface BookRepositoryCustom { List<Book> search(Filter f); }

class BookRepositoryCustomImpl implements BookRepositoryCustom {
    @PersistenceContext EntityManager em;
    public List<Book> search(Filter f) { /* Criteria here */ }
}

interface BookRepository extends JpaRepository<Book, Long>, BookRepositoryCustom { }

@Modifying queries: bulk updates and deletes

Derived and @Query methods are SELECT by default. Adding @Modifying turns a @Query into an UPDATE, DELETE or bulk INSERT, returning the affected-row count (int) or void. It must run inside a transaction. Crucially, a bulk statement goes straight to the database — it bypasses the persistence context, so dirty checking, cascading and the first-level cache are all skipped, and already-loaded managed entities can now hold stale values.

That staleness is the trap. @Modifying(clearAutomatically = true) clears the persistence context after the statement, and flushAutomatically = true flushes pending changes before it, keeping things consistent. It is also worth contrasting a bulk @Query delete (one JPQL statement, no lifecycle callbacks or cascade or orphan removal) with a derived deleteByStatus(...), which loads entities then removes them one by one so callbacks fire.

@Modifying(clearAutomatically = true)
@Query("update Book b set b.price = b.price * :f where b.genre = :g")
int repriceGenre(@Param("g") String g, @Param("f") BigDecimal f);

Auditing: @CreatedDate, @LastModifiedDate and who changed it

Spring Data auditing populates who-and-when metadata automatically: @CreatedDate and @LastModifiedDate for timestamps, @CreatedBy and @LastModifiedBy for the acting principal. You switch it on with @EnableJpaAuditing on a configuration class, annotate the entity — usually a shared @MappedSuperclass base — with @EntityListeners(AuditingEntityListener.class), and the listener fills the fields on persist and update.

The @CreatedBy/@LastModifiedBy fields need an AuditorAware<T> bean that returns the current user, typically from the Spring Security context; timestamps work without it. Keep this distinct from Hibernate's provider-level @CreationTimestamp/@UpdateTimestamp and from a full change-history trail like Hibernate Envers — a common follow-up in interviews.

@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
abstract class Auditable {
    @CreatedDate       Instant createdDate;
    @LastModifiedDate  Instant lastModifiedDate;
    @CreatedBy         String  createdBy;
}

@EntityGraph in repositories

@EntityGraph on a repository method declares which associations to fetch in the same query — the Spring Data way to kill an N+1 pattern or a LazyInitializationException without hand-writing a fetch-join JPQL. It takes two forms: reference a @NamedEntityGraph declared on the entity by its value, or define one inline with attributePaths = {"author", "reviews"}. Because the graph is attached to the method, the same derived or @Query finder can be reused with or without it.

The type attribute matters: FETCH treats the listed attributes as eager and forces everything else lazy, while LOAD treats the listed attributes as eager and leaves unlisted ones at their mapped default. Keep a graph to one or two associations — pulling several collections in one query produces cartesian-product rows and can be slower than the problem you set out to fix.

@EntityGraph(attributePaths = {"author", "reviews"})
List<Book> findByGenre(String genre);   // author + reviews in one query

Querydsl awareness

Querydsl is a third-party type-safe query API. An annotation processor generates a query type (QBook) from each entity at build time, and you assemble predicates fluently — QBook.book.title.startsWithIgnoreCase("a"). Spring Data integrates it through QuerydslPredicateExecutor<T>, which you extend exactly like JpaSpecificationExecutor to get findAll(Predicate), findOne, count and exists.

Awareness is usually enough for interviews: know it as the fourth dynamic-query option after derived methods, @Query, and Criteria/Specifications; know that its generated Q-classes make queries refactor-safe and far less verbose than the raw Criteria API; and know the cost is the build-time code generation step. The frequent question is simply "Specifications vs Querydsl vs Criteria — when would you pick each?".

interface BookRepository extends JpaRepository<Book, Long>,
                                 QuerydslPredicateExecutor<Book> { }

repo.findAll(QBook.book.price.lt(500).and(QBook.book.inStock.isTrue()));

Interview questions

The advanced Spring Data JPA questions that recur in Indian interviews, from screening rounds to product-company deep dives:

Beginner What is a Spring Data JPA Specification, and what must the repository extend to use it?

A Specification<T> encapsulates a single JPA Criteria predicate via toPredicate(root, query, cb). The repository extends JpaSpecificationExecutor<T>, which adds findAll(Specification), findOne, count and exists — with paged and sorted overloads.

Beginner When would you use Specifications instead of derived query methods?

For dynamic queries with several optional filters — search/report screens — where deriving one method per filter combination would explode into dozens of findByXAndY variants. Specifications compose the predicates a request actually supplied, at runtime.

Beginner How do you add a fully custom method (raw EntityManager or Criteria) to a repository?

Write a custom fragment: a custom interface plus an implementation class named with the Impl postfix. The repository extends both the base repository and the fragment interface, and Spring Data composes them into one proxy.

Beginner What does @Modifying do, and what does such a method return?

It marks a @Query as an updating statement (UPDATE/DELETE/bulk INSERT) rather than a SELECT. It returns the affected-row count as int, or void, and must execute inside a transaction.

Intermediate After a bulk @Modifying update, a managed entity still shows the old value. Why, and how do you fix it?

Bulk statements run straight against the database and bypass the persistence context, so the first-level cache is not refreshed — already-loaded entities keep the old state. Fix with @Modifying(clearAutomatically = true) (and flushAutomatically = true to flush pending changes first), or clear/refresh the context manually.

Intermediate Difference between a derived deleteByStatus(...) and a @Modifying @Query delete?

The derived deleteBy loads matching entities and removes them one by one, so lifecycle callbacks, cascades and orphan removal fire. The bulk @Modifying delete is a single JPQL statement — much faster, but it skips callbacks, cascade and the persistence context entirely.

Intermediate How do you make Spring auto-populate created/modified timestamps and users?

Add @EnableJpaAuditing to a config class, put @EntityListeners(AuditingEntityListener.class) on the entity or a @MappedSuperclass base, and annotate fields with @CreatedDate, @LastModifiedDate, @CreatedBy, @LastModifiedBy. For the user fields, expose an AuditorAware bean.

Intermediate @CreatedDate fills correctly but @CreatedBy is always null. What is missing?

There is no AuditorAware<T> bean supplying the current principal (or Spring Security is not wired to it). Timestamps come from the system clock and need nothing extra; the "by" fields depend entirely on the auditor.

Intermediate What is @EntityGraph on a repository method for, and what are its two forms?

It declares associations to fetch in the same query, defeating N+1 and LazyInitializationException without a hand-written fetch join. Two forms: reference a @NamedEntityGraph by its value, or define one inline with attributePaths.

Pro @EntityGraph type FETCH vs LOAD — what is the difference?

FETCH: attributes named in the graph are treated as eager and every other attribute is forced lazy. LOAD: named attributes are eager, but unlisted attributes keep their mapped/default fetch type. FETCH gives you the tightest control over exactly what loads.

Pro Specifications vs Querydsl vs the Criteria API — when would you pick each?

All three build dynamic, type-checked queries. The raw Criteria API is the verbose JPA standard with no extra dependency; Specifications are Spring's composable, reusable wrapper over Criteria; Querydsl is a third-party fluent DSL with generated Q-classes — the most readable and refactor-safe, at the cost of a build-time code-generation step. Pick Specifications for Spring-native composition, Querydsl when query readability and refactoring safety dominate.

Pro A nightly job must reprice a million rows. Bulk @Modifying update vs loading entities and dirty-checking — trade-offs?

The bulk update is one SQL statement with no memory blow-up, but it bypasses optimistic locking (the @Version column is not bumped unless you set it explicitly), cascades, entity callbacks, and it leaves the first- and second-level caches stale. Entity-by-entity respects locking, events and cascade but is slow and memory-heavy. In practice: prefer the bulk statement, but handle @Version in the SQL, evict/clear affected caches, and chunk the work if it runs in one transaction.
