Module: JPQL, Criteria and Native Queries

JPA gives you three ways to ask the database a question — a portable object-oriented query language, a type-safe builder for queries you assemble at runtime, and an escape hatch to raw SQL — and interviews test whether you can tell which one to reach for and how to map the result back.

Knowledge Library → Courses → Hibernate and JPA → JPQL, Criteria and Native Queries

What this module covers

Once your entities are mapped, you need to query them. This module covers the Jakarta Persistence Query Language (JPQL) and Hibernate's HQL superset, named queries, parameter binding, projections and pagination; then the programmatic Criteria API with its type-safe metamodel; and finally native SQL with @SqlResultSetMapping for when you must drop below the abstraction. Choosing the right tool — and mapping results to DTOs instead of loading full entities — is where most real-world query performance is won or lost.

Topics

JPQL and HQL: querying the object model

JPQL is a SQL-like language that operates on your entity classes and their fields rather than database tables and columns — you write FROM Book b JOIN b.author a, navigating a mapped association by object reference, and the provider translates it to vendor SQL at run time. Because it is entity-oriented, the same query runs unchanged on PostgreSQL, MySQL or Oracle. HQL is Hibernate's dialect: a strict superset that adds features JPQL lacks (extra functions, set operations, common table expressions), so every valid JPQL statement is also valid HQL, but HQL-only syntax ties you to Hibernate.

Why it matters: JPQL is the default query tool in every Spring Boot data layer, and interviewers routinely probe the JPQL-versus-SQL and JPQL-versus-HQL distinctions. Binding parameters (never concatenating strings) is what keeps these queries portable, injection-safe and plan-cacheable.

List<Book> books = em.createQuery(
        "SELECT b FROM Book b JOIN b.author a " +
        "WHERE a.name = :name ORDER BY b.title", Book.class)
    .setParameter("name", authorName)
    .getResultList();

Named queries

A named query is a static JPQL (or native SQL) statement declared once and referenced by name — annotate an entity with @NamedQuery (or @NamedNativeQuery) and call em.createNamedQuery("Book.findByTitle"). Because the persistence provider validates and compiles named queries at start-up, a syntax error fails the application at boot rather than at request time, and the parsed form is cached for reuse. The trade-off is that they are static and live away from the calling code, so genuinely dynamic queries still need JPQL string building or the Criteria API.

Binding parameters: named vs positional

JPQL supports named parameters (:name, bound with setParameter("name", value)) and positional parameters (?1). Named parameters are the recommended default — self-documenting and safe to reorder — and you must not mix the two styles within one query. The rule that underpins everything else: bind values, never concatenate user input into the query text. Binding is both the SQL-injection defense and what lets the provider reuse a cached execution plan across calls.

Projections: entities, scalars and DTOs

A query can return whole managed entities, individual scalar columns, or a purpose-built DTO. Fetching full entities only earns its cost when you intend to modify them; for read-only screens, projecting just the columns the caller needs is far cheaper and avoids dragging in lazy associations. A JPQL constructor expression (SELECT new com.acme.BookSummary(...)) builds a DTO directly in the query, while Object[] or Tuple carry several scalars.

Why it matters: "entity versus DTO projection" and "how do you return a DTO from a query" are frequent intermediate-to-senior questions, precisely because reaching for entities when a projection would do is one of the most common JPA performance mistakes.

List<BookSummary> rows = em.createQuery(
        "SELECT new com.acme.BookSummary(b.title, a.name) " +
        "FROM Book b JOIN b.author a", BookSummary.class)
    .getResultList();

Pagination

setFirstResult(offset) and setMaxResults(limit) turn any query into a page; the provider appends the vendor's LIMIT/OFFSET, FETCH FIRST or ROWNUM syntax for you. Two classic traps show up in interviews and production alike: combining a JOIN FETCH of a to-many collection with pagination forces Hibernate to read every row into memory and paginate in the application (the HHH000104 warning), because a joined row count no longer equals an entity count; and deep OFFSET pagination degrades on large tables, where keyset (seek) pagination scales far better. A real page also usually needs a separate count query for the total.

List<Book> page = em.createQuery(
        "SELECT b FROM Book b ORDER BY b.id", Book.class)
    .setFirstResult(pageNumber * pageSize)  // offset
    .setMaxResults(pageSize)                // limit
    .getResultList();

The Criteria API and the metamodel

The Criteria API builds the same queries programmatically through a CriteriaBuilder, assembling the query from Java objects instead of a string. Its reason to exist is dynamic queries: when the set of filters depends on which search fields a user actually filled in, you conditionally add Predicate objects to the query rather than concatenating SQL fragments and worrying about stray ANDs. The static metamodel — generated Book_ classes — makes it type-safe: b.get(Book_.title) instead of the stringly-typed b.get("title"), so renaming a field becomes a compile error rather than a run-time surprise.

Why it matters: Criteria is verbose compared with JPQL, so the senior-level judgement is knowing to reserve it for genuinely dynamic queries and to keep static ones in JPQL or named queries.

CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Book> cq = cb.createQuery(Book.class);
Root<Book> b = cq.from(Book.class);
cq.select(b).where(cb.equal(b.get(Book_.title), title)); // type-safe metamodel path
List<Book> books = em.createQuery(cq).getResultList();

Native SQL queries and @SqlResultSetMapping

When you need database-specific features JPQL cannot express — window functions, vendor hints, recursive CTEs, or a hand-tuned statement — em.createNativeQuery(...) drops to raw SQL. It is a deliberate leak in the abstraction: you gain full SQL power and lose portability, and you become responsible for mapping the flat result set back. The mapping options are the interesting part: pass an entity class to return managed entities, or declare a @SqlResultSetMapping using @EntityResult, @ColumnResult and @ConstructorResult to shape entities, scalar values or DTOs. @SqlResultSetMapping is the most flexible route and the standard answer to "how do you return a DTO from a native query."

@SqlResultSetMapping(name = "BookSummaryMapping",
    classes = @ConstructorResult(targetClass = BookSummary.class,
        columns = { @ColumnResult(name = "title"),
                    @ColumnResult(name = "author_name") }))
// ...
List<BookSummary> rows = em.createNativeQuery(
        "SELECT b.title, a.name AS author_name " +
        "FROM book b JOIN author a ON a.id = b.author_id",
        "BookSummaryMapping")
    .getResultList();

Interview questions

The query layer is one of the most-probed JPA areas in Indian interviews — expect JPQL versus native, DTO projections and the Criteria API in almost every Spring Boot round:

Beginner What is JPQL and how does it differ from SQL?

JPQL queries the entity object model — classes and mapped fields — while SQL queries database tables and columns. You navigate associations by object reference (b.author), and the provider translates the JPQL into vendor-specific SQL at run time. The payoff is portability: one query works across databases without change.

Beginner JPQL vs HQL — what is the relationship?

HQL is Hibernate's query language and a strict superset of JPQL: every valid JPQL statement is valid HQL, but HQL adds features (extra functions, set operations, CTEs) that JPQL does not define. Sticking to JPQL keeps your code portable to any JPA provider; using HQL-only syntax ties you to Hibernate.

Beginner Named vs positional parameters — and can you mix them?

Named parameters use :name and are bound by name; positional use ?1 and are bound by index. Named is preferred for readability and reorder-safety. You cannot mix the two styles in a single query. Always bind parameters — never concatenate user input — to prevent SQL injection and to reuse the cached execution plan.

Beginner What is a named query and one advantage of using one?

A static JPQL or native query declared with @NamedQuery / @NamedNativeQuery and referenced by name. Advantage: the provider validates and parses it at application start-up, so syntax errors fail fast at boot instead of at request time, and the parsed form is cached.

Intermediate How do you return a DTO instead of an entity from a JPQL query, and why would you?

Use a constructor expression — SELECT new com.acme.BookSummary(b.title, a.name) ... — or a Tuple / interface projection. Why: read-only data does not need managed entities; selecting only the required columns and skipping associations is faster, uses less memory, and avoids accidental lazy loading and the N+1 problem.

Intermediate When would you choose the Criteria API over JPQL?

For dynamic queries where the filters, joins or ordering vary at run time — for example an advanced-search screen where each field is optional. You add Predicate objects conditionally instead of concatenating query strings. For static queries JPQL or named queries are shorter and clearer, so Criteria is not the default.

Intermediate What does the JPA static metamodel give you?

Generated Entity_ classes (e.g. Book_) that let Criteria queries reference attributes type-safely: b.get(Book_.title) instead of the string b.get("title"). Renaming or removing a field then becomes a compile error, making refactoring safe and eliminating a class of run-time bugs from mistyped attribute names.

Intermediate How do you paginate in JPA, and what is the pitfall with JOIN FETCH?

setFirstResult(offset) and setMaxResults(limit). The pitfall: paginating a query that JOIN FETCHes a to-many collection makes Hibernate fetch the entire result into memory and paginate in the application (the HHH000104 warning), because one entity spans many joined rows. Fix by paginating the root entities or their ids first and fetching the collection separately, or by using @EntityGraph / batch fetching.

Intermediate When do you use a native query, and what do you give up?

When you need database-specific SQL that JPQL cannot express — window functions, optimizer hints, recursive CTEs — or a hand-optimized statement. You give up portability, and you take on result mapping yourself; you may also need to declare the affected tables so Hibernate synchronizes (flushes) pending changes and its dirty-check correctly before the query runs.

Pro How do you map a native query result to a DTO?

Declare a @SqlResultSetMapping with @ConstructorResult plus @ColumnResult entries, then reference it by name from createNativeQuery(sql, "MappingName") or @NamedNativeQuery. Use @EntityResult to return managed entities, or @ColumnResult alone for scalars. Note that a native query can specify only one of resultClass or resultSetMapping, not both.

Pro Why can OFFSET pagination be a performance problem, and what is the alternative?

OFFSET makes the database scan and discard every row before the requested page, so deep pages get progressively slower, and results can shift or duplicate if data changes between page requests. Keyset (seek) pagination — WHERE id > :lastSeenId ORDER BY id with a limit — uses the index to jump straight to the next slice and stays roughly constant-time per page.

Pro Are bulk UPDATE/DELETE JPQL statements safe with the persistence context?

They execute directly in the database and bypass the persistence context: managed entities and the first-level cache are not updated, and entity lifecycle callbacks and cascades do not fire. Flush any pending changes before running a bulk operation, and clear or refresh the context afterwards, otherwise it holds stale entity state that can overwrite your bulk change on the next flush.
