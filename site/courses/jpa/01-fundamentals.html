What this module covers

This first module sets the vocabulary the rest of the track depends on: object-relational mapping as an idea, the three-layer stack of Jakarta Persistence (the spec), Hibernate (the provider), and Spring Data JPA (the repository abstraction), and the runtime objects that bootstrap it all — EntityManagerFactory, EntityManager, the persistence unit, and Hibernate's native SessionFactory/Session. Get these distinctions right and every later topic — mapping, fetching, transactions — has a place to hang.

Topics

The ORM concept: JPA spec, Hibernate, and Spring Data JPA — and why ORM

Object-relational mapping bridges two models that do not naturally fit: graphs of Java objects on one side, normalized rows and foreign keys on the other. An ORM maps a class to a table and lets you read, save and query using entities instead of hand-writing SQL and unpacking ResultSets. The stack has three distinct layers. Jakarta Persistence (JPA) is a specification — a jar of interfaces and annotations (@Entity, EntityManager) with no behaviour of its own; the current version is Jakarta Persistence 3.2. Hibernate ORM (7.x) is the most widely used implementation of that spec, plus a superset of native features. Spring Data JPA is neither a spec nor a provider — it is a higher-level abstraction that generates repository implementations on top of a JPA provider (Hibernate by default).

Why it matters: ORM removes boilerplate and adds dirty checking, caching, lazy loading and database portability through dialects — but its cost is a leaky abstraction you must understand to use well. In interviews the single most common confusion is collapsing these three layers into one. Being able to say cleanly "JPA is the standard, Hibernate implements it, Spring Data JPA wraps a provider" is a screening-round staple in Indian product and services interviews alike.

@Entity                                   // JPA annotation (the spec)
class Book { @Id Long id; String title; }

// Spring Data JPA: you write the interface, it writes the query methods
interface BookRepo extends JpaRepository<Book, Long> { }

// Hibernate is the provider that turns all of this into SQL at runtime

EntityManagerFactory / EntityManager, the persistence unit, SessionFactory / Session, and bootstrapping

These four objects are the machinery behind every JPA call. An EntityManagerFactory is the heavyweight, thread-safe, application-scoped object built once per persistence unit; creating it parses mappings and opens the connection pool, so you never build it per request. It hands out EntityManager instances — lightweight, not thread-safe, short-lived handles onto a persistence context that do the real work: persist, find, merge, remove, and queries. A persistence unit is the named bundle of configuration (which entities, which datasource, which provider settings), declared in META-INF/persistence.xml for plain Java SE or assembled automatically by Spring Boot. Hibernate's native pair mirrors the JPA pair exactly: SessionFactory corresponds to EntityManagerFactory and Session to EntityManager — and in Hibernate 7 SessionFactory actually extends EntityManagerFactory, so the native API is a strict superset you can unwrap to when you need it.

Bootstrapping is how the factory comes to life. Standard JPA does it with Persistence.createEntityManagerFactory("unit-name"); Hibernate's native path builds a ServiceRegistry, then Metadata, then a SessionFactory. In Spring Boot you write almost none of this — auto-configuration constructs a container-managed factory and injects a shared, transaction-aware EntityManager proxy via @PersistenceContext. Why interviewers press here: the thread-safety contract (factory shared, manager per unit of work), the one-per-application versus one-per-request lifecycle, and the factory-equals-SessionFactory equivalence are fundamentals that separate memorised answers from understood ones.

EntityManagerFactory emf =
    Persistence.createEntityManagerFactory("my-unit"); // once per application

EntityManager em = emf.createEntityManager();          // once per unit of work
em.getTransaction().begin();
em.persist(new Book("Effective Java"));
em.getTransaction().commit();
em.close();                                            // factory stays open

Interview questions

The fundamentals questions asked first in Indian JPA and Hibernate interviews — the layer distinctions and the factory lifecycle come up in almost every screening round:

Beginner What is ORM and what problem does it solve?

Object-relational mapping maps Java classes to database tables so you work with entities instead of raw SQL and ResultSets. It resolves the object-relational impedance mismatch (object graphs vs normalized rows), removes JDBC boilerplate, and adds services like dirty checking, caching and lazy loading.

Beginner What is the difference between JPA and Hibernate?

JPA (Jakarta Persistence) is a specification — a set of interfaces and annotations with no runtime behaviour. Hibernate is a concrete implementation (a JPA provider) that supplies that behaviour, plus extra native features beyond the spec. You code against JPA for portability and Hibernate runs it.

Beginner Is Spring Data JPA a JPA provider?

No. Spring Data JPA is an abstraction layer on top of a JPA provider — it generates repository implementations from interface method names and cuts boilerplate. It still needs a provider underneath (Hibernate by default) and a JPA EntityManager doing the actual persistence.

Beginner What is an entity in JPA?

A plain Java class annotated with @Entity and given an identifier with @Id, mapped to a database table. Once associated with a persistence context it is a managed object whose state changes are tracked and synchronized to the database.

Beginner What is a persistence unit and where is it configured?

A persistence unit is a named configuration grouping the managed entities, the datasource, and provider properties. In Java SE it is declared in META-INF/persistence.xml; in Spring Boot it is auto-configured from application properties. Its name is what you pass to createEntityManagerFactory.

Intermediate EntityManagerFactory vs EntityManager — lifecycle and thread-safety?

The factory is heavyweight, thread-safe and built once per application (it holds mappings and the connection pool). An EntityManager is cheap, not thread-safe, and created per unit of work (per request/transaction) then closed. Sharing one EntityManager across threads is a classic bug.

Intermediate How do SessionFactory and Session relate to their JPA counterparts?

SessionFactory is Hibernate's native equivalent of EntityManagerFactory, and Session of EntityManager. In Hibernate the native interfaces extend the JPA ones, so you can unwrap() to reach native features (stateless sessions, filters, native batching) while otherwise coding to the standard API.

Intermediate How do you bootstrap JPA in a plain Java SE app versus in Spring Boot?

Java SE: call Persistence.createEntityManagerFactory("unit-name"), then createEntityManager() and manage transactions yourself. Spring Boot: auto-configuration builds a LocalContainerEntityManagerFactoryBean from your datasource and injects a shared EntityManager proxy via @PersistenceContext — no manual factory or transaction code.

Intermediate Name other JPA providers, and why does coding to the spec matter?

EclipseLink (the reference implementation) and Apache OpenJPA are the main alternatives to Hibernate. Coding to the JPA interfaces means the provider is swappable at runtime without changing application code — the portability the specification exists to guarantee, as long as you avoid vendor-specific extensions.

Intermediate What does the EntityManager actually manage?

The persistence context — a first-level cache of managed entities keyed by identity. Within it each entity id maps to one instance; the manager tracks changes (dirty checking) and flushes them as SQL at commit or on demand. The EntityManager is your handle onto that context.

Pro If Hibernate implements JPA, why ever use the native SessionFactory/Session API?

For capabilities the spec does not cover: stateless sessions for bulk work, multi-tenancy, entity filters, native second-level cache control, and finer batching or flush tuning. The trade-off is portability — native calls tie you to Hibernate — so the idiom is to code JPA by default and unwrap() only where a native feature earns its cost.

Pro A developer calls createEntityManagerFactory on every request. What breaks, and what is correct?

The factory is expensive to build (it reparses mappings and rebuilds the connection pool), so per-request creation destroys throughput and leaks resources. Correct pattern: build one EntityManagerFactory for the application lifetime and create a short-lived EntityManager per request/transaction — exactly what a Spring/container-managed setup does for you.

Pro How does an EntityManager injected as a singleton bean stay thread-safe under @PersistenceContext?

Spring does not inject a real EntityManager; it injects a shared thread-safe proxy. On each call the proxy resolves the actual EntityManager bound to the current transaction (via thread-local transaction synchronization) and delegates to it. So a singleton-scoped field is safe because every thread transparently gets its own transaction-scoped manager underneath.
