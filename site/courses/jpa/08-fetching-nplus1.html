Module: Fetching and the N+1 Problem

Fetching is where JPA quietly turns one intended query into hundreds. The N+1 problem, LazyInitializationException and fetch joins are the persistence questions interviewers reach for first — this module orients you on each and links the material that teaches it in depth.

Knowledge Library → Courses → Hibernate and JPA → Fetching and the N+1 Problem

What this module covers

How Hibernate decides when to load associated data: the LAZY/EAGER fetch types and their JPA defaults, the lazy proxies that cause the infamous LazyInitializationException, and the N+1 select problem that sinks so many real systems. Then the toolkit for fixing it per query — JOIN FETCH, @EntityGraph and batch fetching — with the trade-offs (duplicate rows, cartesian products, the pagination trap) that separate a screening answer from a senior one.

Topics

EAGER vs LAZY: the default fetch plan

Every association carries a FetchType. LAZY defers loading until the association is first accessed; EAGER loads it immediately with the owner. The JPA defaults trip people up: @ManyToOne and @OneToOne default to EAGER, while @OneToMany and @ManyToMany default to LAZY. Note that EAGER is a hard requirement the provider must honour, whereas LAZY is only a hint.

Why it matters: the widely-taught best practice is to map everything LAZY and fetch eagerly per query, because a global EAGER plan pulls data you rarely need, defeats query-level tuning, and silently multiplies SQL. Knowing the defaults and why you override the to-one ones is a standard opening question.

@ManyToOne(fetch = FetchType.LAZY)   // default is EAGER — override it
private Author author;

@OneToMany(mappedBy = "author")       // already LAZY by default — keep it
private List<Book> books = new ArrayList<>();

Lazy proxies and the LazyInitializationException

For a LAZY association Hibernate hands back a proxy (or an uninitialised collection wrapper) instead of the real data. The proxy fetches on first access — but only while its persistence context is still open. Access it after the transaction/session has closed and you get a LazyInitializationException: "could not initialize proxy — no Session". It is the single most common Hibernate runtime error, and it usually surfaces in a controller or view after the service-layer transaction ended.

Why it matters: the wrong fixes (switching to EAGER, enable_lazy_load_no_trans, Open Session in View) are traps interviewers probe. The right answer is to fetch what the caller needs inside the transaction — via a fetch join, an entity graph, or a DTO projection — so the data is already initialised when the session closes.

The N+1 select problem

The N+1 problem is when fetching a list of N entities triggers N extra SELECTs — one per entity — to load an associated relationship, instead of one combined query. It arises two ways: iterating a LAZY collection over each parent (a query fires on every access), or an EAGER association that a JPQL/Criteria query cannot satisfy in the primary select, so the provider issues secondary selects. With N in the thousands it turns a fast page into a slow one.

Why it matters: this is the canonical JPA performance interview question. You are expected to define it, show how to reproduce it, explain how to detect it (log show_sql, count statements, tools like datasource-proxy), and then reach for a fix — which the next three topics cover.

Fetch joins: JOIN FETCH, DISTINCT and the bag trap

A JPQL JOIN FETCH initialises an association in the same query as its owner, collapsing N+1 into one round-trip. Unlike a plain JOIN (which only affects the WHERE/filtering), JOIN FETCH actually loads the joined entities into the persistence context. It is the most direct, per-query fix and the one to demonstrate first.

Two gotchas make great follow-ups. Fetch-joining a collection produces duplicate parent references (one row per child) — historically fixed with DISTINCT (Hibernate 6 deduplicates automatically, so passDistinctThrough is gone). And fetch-joining two List collections throws MultipleBagFetchException to avoid a cartesian product — the fix is to fetch one collection per query, not to switch blindly to Set.

// One query instead of N+1 — books initialised while fetching authors
List<Author> authors = em.createQuery(
    "SELECT DISTINCT a FROM Author a JOIN FETCH a.books", Author.class)
  .getResultList();

Entity graphs: declarative fetch plans

An entity graph is a reusable, query-independent fetch plan: you declare which attributes to load and attach it to a query or repository method, so the association is fetched with a LEFT JOIN in one statement. Define one declaratively with @NamedEntityGraph, or build it dynamically via the EntityManager. In Spring Data JPA the @EntityGraph annotation on a repository method wires it up. A FETCH-type graph treats listed attributes as EAGER and everything else as LAZY.

Why it matters: entity graphs are the idiomatic way to keep mappings LAZY yet fetch eagerly per use case — the "override the global plan without JPQL" answer. They also avoid the duplicate-row and bag limitations of collection fetch joins in some cases, making the JOIN FETCH vs @EntityGraph comparison a common senior-level question.

@NamedEntityGraph(name = "Author.books",
    attributeNodes = @NamedAttributeNode("books"))
@Entity class Author { /* ... */ }

// Spring Data JPA repository
@EntityGraph("Author.books")
List<Author> findAll();

Batch fetching: @BatchSize and fetch modes

When a JOIN FETCH would create too large a cartesian product — or when you must paginate — batch fetching is the alternative. Hibernate's @BatchSize (or the global hibernate.default_batch_fetch_size) initialises LAZY associations for many parents at once using a single IN (...) query, turning N+1 into roughly N/batch. Related knobs: FetchMode.SUBSELECT loads a collection for all owners via one subselect, and FetchMode.SELECT is the default per-owner behaviour that causes N+1.

Why it matters: batch fetching is the pragmatic answer to "you can't JOIN FETCH everything." Interviewers like to hear when you would pick batch size or subselect over a fetch join — a signal you have tuned real workloads, not just read the annotation list.

@OneToMany(mappedBy = "author")
@BatchSize(size = 25)     // load lazy books 25 parents per IN-clause query
private List<Book> books = new ArrayList<>();
// or globally: hibernate.default_batch_fetch_size = 25

Interview questions

The most-asked fetching questions in Indian interviews — N+1, LazyInitializationException and fetch joins recur in almost every Java/Spring backend round, screening to system-design:

Beginner What is the difference between EAGER and LAZY fetching?

EAGER loads the association immediately with its owner; LAZY defers loading until the association is first accessed, returning a proxy meanwhile. EAGER is a requirement the provider must honour; LAZY is a hint it may ignore for to-one mappings.

Beginner What are the default fetch types for JPA associations?

@ManyToOne and @OneToOne default to EAGER; @OneToMany and @ManyToMany default to LAZY. The common recommendation is to override the to-one ones to LAZY and fetch eagerly per query instead.

Beginner What is a lazy proxy in Hibernate?

A stand-in object (a runtime subclass, or a wrapper for collections) that Hibernate returns for a LAZY association. It holds the identifier and loads the real data on first method access — provided the persistence context is still open.

Beginner What is the N+1 select problem?

Fetching N parent entities and then triggering N additional queries — one per parent — to load an association, instead of retrieving everything in one (or a few) queries. It happens by iterating a LAZY collection per parent, or when an EAGER association forces secondary selects after a JPQL query.

Intermediate What causes LazyInitializationException and how do you fix it correctly?

Accessing a LAZY association after its persistence context/transaction has closed — typically in the view or controller after the service transaction ended. Correct fixes initialise the data inside the transaction: a JOIN FETCH query, an @EntityGraph, or a DTO projection. Avoid the anti-fixes: making it EAGER, enable_lazy_load_no_trans, or Open Session in View.

Intermediate How does JOIN FETCH solve N+1, and how does it differ from a plain JOIN?

JOIN FETCH loads the associated entities in the same SQL query as the owner, so no secondary selects fire. A plain JOIN only participates in filtering/WHERE and does not initialise the association — the collection stays lazy and N+1 remains.

Intermediate Why is FetchType.EAGER considered an anti-pattern?

It applies globally to every query whether or not the data is needed, so it wastes I/O, prevents per-query tuning, and — with JPQL/Criteria — the provider often issues extra secondary selects (its own N+1). Best practice: map associations LAZY and fetch eagerly per query with JOIN FETCH or an entity graph.

Intermediate What is @EntityGraph and how does it differ from JOIN FETCH?

An entity graph is a declarative, reusable fetch plan (@NamedEntityGraph or built dynamically; in Spring Data via @EntityGraph on a method) that fetches listed attributes with a LEFT JOIN. JOIN FETCH is inline in a specific JPQL query. Graphs decouple the fetch plan from the query and reuse across methods; JOIN FETCH gives full query control but is tied to that one query string.

Intermediate Why does JOIN FETCH on a collection return duplicate parents, and how do you fix it?

The SQL join produces one row per child, so the same parent reference repeats. Historically you added DISTINCT to the JPQL (deduplicated in memory, not pushed to SQL for entity queries). Since Hibernate 6 parent references are deduplicated automatically, so DISTINCT is usually unnecessary.

Intermediate How do @BatchSize and default_batch_fetch_size help?

They batch the initialisation of LAZY associations: instead of one query per parent, Hibernate loads them for many parents at once using an IN (:ids) clause, cutting N+1 down to about N/batchSize queries. Useful when a JOIN FETCH would build too large a cartesian product or when you need pagination.

Pro Why can't you combine JOIN FETCH of a collection with pagination, and what happens if you try?

The joined result has multiple rows per parent, so a SQL LIMIT would truncate mid-parent and return wrong data. Hibernate refuses to paginate in SQL and instead fetches the whole result set and paginates in memory, logging the HHH000104 warning — dangerous on large tables. Use batch fetching or a two-step query (page the IDs, then fetch the collection) instead.

Pro What is MultipleBagFetchException and how do you fetch two collections correctly?

Hibernate throws it when you JOIN FETCH two List (bag) associations at once, because the cartesian product would be huge and unindexed. The right fix is to fetch at most one collection per query — the persistence context merges the second into the already-loaded parents. Switching both to Set silences the exception but still generates the cartesian product, so it is not a real fix.

Pro Is Open Session in View a good solution to LazyInitializationException?

No. OSIV keeps the persistence context open across the view render so lazy access "just works," but it leaks fetching into the presentation layer, holds the DB connection longer, encourages hidden N+1, and can cause data-integrity surprises. Prefer fetching what the caller needs inside the service transaction (JOIN FETCH / entity graph / DTO). Spring Boot enables OSIV by default and warns about it — disabling it forces explicit fetch plans.

Pro When would you choose batch fetching or SUBSELECT over a JOIN FETCH?

When a single fetch join would explode into a large cartesian product (multiple or deep collections), when you need pagination on the parent, or when many parents each need the same collection. @BatchSize/default_batch_fetch_size loads collections in IN-clause batches, and FetchMode.SUBSELECT loads all owners' collections via one subselect — both trade one big joined query for a few small, bounded ones.
