Module: Inheritance Mapping

Class hierarchies have no native form in SQL, so JPA offers three ways to project one onto tables — and choosing the wrong one is a performance and data-integrity decision you will keep re-explaining in every code review that follows.

Knowledge Library → Courses → Hibernate and JPA → Inheritance Mapping

What this module covers

Object models use inheritance; relational schemas do not. This module covers the three @Inheritance strategies that bridge that gap — SINGLE_TABLE, JOINED and TABLE_PER_CLASS — plus the discriminator column that tags rows by type, the classic SINGLE_TABLE-versus-JOINED trade-off, and @MappedSuperclass for sharing state without modelling inheritance in the database. It closes on polymorphic queries: the reason you map a hierarchy in the first place.

Topics

Mapping a hierarchy: @Inheritance and the strategy choice

A superclass annotated @Inheritance tells the provider to persist a whole Java hierarchy, and the strategy attribute picks how: SINGLE_TABLE (one table for everything), JOINED (a table per class, joined by shared id), or TABLE_PER_CLASS (a full table per concrete class). The default is SINGLE_TABLE. The mapping is inherited automatically — subclasses just add their own fields — and the abstract or concrete base class becomes queryable in its own right.

Why it matters: this single annotation determines table count, the shape of every generated SQL statement, and whether the database can enforce your constraints. It is a schema decision that is expensive to reverse once data lands, so interviewers probe whether you understand the consequences, not just the syntax.

@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
public abstract class Payment {
    @Id @GeneratedValue Long id;
    BigDecimal amount;
}

@Entity
public class CardPayment extends Payment { String last4; }   // adds a column

SINGLE_TABLE and the discriminator column

SINGLE_TABLE stores the entire hierarchy in one wide table, so a subclass's fields become nullable columns that only some rows populate. A discriminator column (Hibernate defaults to DTYPE, a VARCHAR of the entity name) records which subclass each row is, and @DiscriminatorColumn / @DiscriminatorValue let you rename it, change its type, or set custom values. It is the fastest strategy — no joins, polymorphic queries hit a single table — which is exactly why it is the default.

Why it matters: the cost is data integrity. Because subclass columns must be nullable for the other subclasses' rows, you cannot declare a genuinely-required subclass field as NOT NULL; you push that check into the application or a database CHECK constraint. A tip worth knowing: an INTEGER discriminator is far more compact than the default string at scale.

@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "payment_type", discriminatorType = DiscriminatorType.STRING)
public abstract class Payment { @Id Long id; }

@Entity @DiscriminatorValue("CARD")
public class CardPayment extends Payment { String last4; }  // last4 must stay nullable

JOINED and TABLE_PER_CLASS: the normalized strategies

JOINED gives each class its own table holding only that class's fields; the subclass primary key doubles as a foreign key back to the parent, and loading a subclass row joins across the levels of the hierarchy. It is the fully-normalized option: every column can be NOT NULL, no space is wasted, and the schema mirrors the class diagram. The price is a join per inheritance level on reads and an extra insert per level on writes.

TABLE_PER_CLASS gives each concrete class a standalone table that repeats the inherited columns, so single-type reads need no join at all. But a polymorphic query over the base type must UNION ALL every concrete table, which scales badly and interacts poorly with identity generation — which is why it is the least recommended of the three.

@Entity
@Inheritance(strategy = InheritanceType.JOINED)   // Payment table + one child table each
public abstract class Payment { @Id Long id; BigDecimal amount; }

@Entity
public class BankTransfer extends Payment { String iban; }  // stored in its own joined table

Choosing a strategy: SINGLE_TABLE vs JOINED trade-offs

When polymorphic queries and associations are needed, the real choice narrows to SINGLE_TABLE versus JOINED — and this comparison is one of the most common JPA interview questions. SINGLE_TABLE wins on query speed (no joins, one table) but forfeits NOT NULL on subclass columns and grows a sparse, wide table. JOINED wins on normalization and integrity but pays joins on every polymorphic read and multiple inserts per save. TABLE_PER_CLASS is usually a distant third; if you do not actually need polymorphism, @MappedSuperclass is the cleaner tool.

Why it matters: the honest answer is "it depends on the read pattern and how strict the constraints must be" — shallow hierarchies with mostly-shared fields and read-heavy workloads favour SINGLE_TABLE; deep hierarchies with many type-specific required fields favour JOINED. Being able to state that trade-off, with the NOT-NULL limitation as the deciding pivot, is what senior interviewers listen for.

@MappedSuperclass and polymorphic queries

@MappedSuperclass models inheritance only in Java: its mappings are copied into each subclass's table, but it is not an entity, has no table, and cannot be the target of a query or an association. It is the right tool for shared technical columns — id, @Version, audit timestamps — that every entity needs but that never define a queryable "base type". Compare it with @Embeddable, which composes rather than inherits; reach for @MappedSuperclass when you must share the id or version attribute.

Polymorphic queries are the payoff of a real inheritance mapping: select p from Payment p returns every subclass instance, and Hibernate assembles the correct types (via discriminator or joins) for you. The TYPE() operator narrows a query to specific subclasses. Crucially, this only works over an @Inheritance hierarchy — you cannot write a polymorphic query against a @MappedSuperclass, because the superclass is not an entity.

@MappedSuperclass
public abstract class BaseEntity {
    @Id @GeneratedValue Long id;
    @Version int version;      // shared columns, copied into every entity table
}

// Polymorphic JPQL over an @Inheritance hierarchy:
// select p from Payment p where type(p) = CardPayment

Interview questions

Most-asked inheritance-mapping questions in Indian interviews — the SINGLE_TABLE-vs-JOINED trade-off is a recurring product-company favourite:

Beginner What inheritance strategies does JPA support?

Three, via @Inheritance(strategy = ...): SINGLE_TABLE (whole hierarchy in one table), JOINED (a table per class linked by the shared id), and TABLE_PER_CLASS (a table per concrete class). SINGLE_TABLE is the default. A fourth option, @MappedSuperclass, shares mappings but is not a database inheritance mapping.

Beginner What is the discriminator column and what is its default?

In SINGLE_TABLE (and optionally JOINED), a column that records which subclass each row represents so the provider can instantiate the right type. Hibernate defaults to DTYPE, a string holding the entity name; @DiscriminatorColumn and @DiscriminatorValue customize the name, type and values.

Beginner How is @MappedSuperclass different from a mapped @Entity superclass?

@MappedSuperclass is not an entity: it has no table, cannot be queried or referenced by an association, and its columns are simply copied into each subclass's own table. An @Inheritance entity superclass is a real, queryable type with a database representation.

Intermediate SINGLE_TABLE vs JOINED — compare the trade-offs.

SINGLE_TABLE: one table, no joins, fastest reads and polymorphic queries, but subclass columns must be nullable (no NOT NULL) and the table grows wide and sparse. JOINED: normalized, every column can be constrained, no wasted space, but each polymorphic read joins across levels and each insert writes multiple tables. Read-heavy with mostly-shared fields → SINGLE_TABLE; strict constraints and type-specific required fields → JOINED.

Intermediate Why can't SINGLE_TABLE enforce NOT NULL on a subclass field?

All subclasses share the one table, so a column that belongs to one subclass is null for every row of the others. Declaring it NOT NULL would break inserts of sibling types. You enforce the rule in application code or with a database CHECK constraint keyed on the discriminator value.

Intermediate How does TABLE_PER_CLASS work and why is it least recommended?

Each concrete class gets its own table repeating the inherited columns, so single-type reads need no join. But a polymorphic query over the base type must UNION ALL every concrete table, which scales poorly, and it conflicts with IDENTITY generation (ids must be unique across the union). If you don't need polymorphism, prefer @MappedSuperclass.

Intermediate What is a polymorphic query and how do the strategies affect its SQL?

select p from Payment p returns all subclass instances of the base type. SINGLE_TABLE resolves it with one table scan filtered by discriminator; JOINED joins the parent to each child table; TABLE_PER_CLASS unions all concrete tables. Same JPQL, very different generated SQL and cost.

Intermediate When would you choose @MappedSuperclass over @Embeddable?

@MappedSuperclass uses inheritance and can carry the @Id, @Version and lifecycle callbacks — ideal for a shared technical base like BaseEntity. @Embeddable uses composition and can be reused as a value object but can't own the primary key or version the same way. Rule of thumb: try @Embeddable for plain value groups; use @MappedSuperclass when the shared attributes include the id or version.

Intermediate Can you make the discriminator column more compact than the default?

Yes. The default string discriminator repeats the entity name on every row. Setting @DiscriminatorColumn(discriminatorType = INTEGER) with numeric @DiscriminatorValues (or a TINYINT columnDefinition) shrinks it dramatically at scale; a small lookup table can map the codes back to readable names.

Pro A base type has many subclasses, is queried polymorphically, and reads must be fast. Which strategy?

SINGLE_TABLE — one table, no joins, discriminator-filtered scans make polymorphic reads cheapest. Accept the trade-off explicitly: subclass columns lose NOT NULL, so move those checks to CHECK constraints or the application layer, and watch the table width. Reach for JOINED only if the integrity constraints are non-negotiable.

Pro Can you run a polymorphic query against a @MappedSuperclass? Why or why not?

No. The mapped superclass is not an entity and has no table, so select b from BaseEntity b is invalid — there is no queryable base type and JPA won't return its subclasses polymorphically. If you need "query all subtypes as one", you must use an @Inheritance mapping instead. This limitation is the usual reason to prefer an entity superclass over @MappedSuperclass.

Pro How do you restrict a polymorphic JPQL query to specific subclasses?

Use the TYPE() operator on the entity's runtime type: select p from Payment p where type(p) = CardPayment, or type(p) in (CardPayment, BankTransfer). It filters the polymorphic result to chosen concrete types — on SINGLE_TABLE it becomes a discriminator predicate, on JOINED it prunes which child tables are joined.

Pro What are the write- and read-time costs of a deep JOINED hierarchy?

Reads: fetching a leaf entity joins every table from the concrete class up to the root, and a polymorphic query outer-joins all child tables — cost grows with hierarchy depth. Writes: persisting one entity issues an insert per level (root table plus each ancestor), and deletes cascade the same way. Keep hierarchies shallow, or switch to SINGLE_TABLE when the join count starts hurting.
