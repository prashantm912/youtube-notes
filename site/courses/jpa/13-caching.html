Module: Caching

Caching is where JPA performance is won or quietly corrupted — the first-level cache you always have, the second-level cache you have to earn one entity at a time, and the query cache you should reach for last. This module orients you on each and points to the resources that get the details right.

Knowledge Library → Courses → Hibernate and JPA → Caching

What this module covers

Two caches sit under every Hibernate application. The first-level cache (the persistence context) is mandatory, per-transaction and free; the second-level cache is SessionFactory-wide, opt-in per entity, and the one you actually configure. On top of those sits the query cache for repeated queries. This module covers how each works, the cache concurrency strategies and providers that keep the second-level cache consistent, the @Cacheable/@Cache/shared-cache-mode switches that turn caching on, and the pitfalls that make a badly-placed cache worse than none. First-level vs second-level cache is one of the most-asked JPA interview questions.

Topics

The first-level cache: the persistence context

Every JPA application already has a cache it never switched on: the persistence context. Each EntityManager (Hibernate Session) keeps a map of the entities it has loaded or persisted, keyed by identifier, for the life of that context — normally a single transaction. Ask for the same row twice in one context and the second call returns the very same object reference from memory instead of firing a second SELECT. This is the first-level cache; it is mandatory and cannot be disabled.

Its role is wider than skipping queries. Because there is exactly one instance per row per context, it gives application-level repeatable reads, it is where dirty checking compares live state against the loaded snapshot at flush time, and it makes identity (==) meaningful inside a transaction. It is never shared across contexts or threads and it dies when the context closes — so it does nothing for the next request. That gap is precisely what the second-level cache fills.

Country a = em.find(Country.class, 1L); // SELECT executes, entity cached
Country b = em.find(Country.class, 1L); // no SQL — served from the context
assert a == b;                          // same reference: first-level identity

The second-level cache: what it is and how to enable it

The second-level cache (2LC) lives on the SessionFactory/EntityManagerFactory and is shared by every context and thread in the JVM — or across a cluster, depending on the provider. When a context asks for an entity by id and misses its first-level cache, Hibernate consults the 2LC before touching the database; a hit rebuilds a managed entity from cached state with no SQL. Unlike the first-level cache, it is off by default and opt-in per entity — Hibernate never caches an entity you did not explicitly mark.

Enabling it takes two coordinated switches: turn the feature on (hibernate.cache.use_second_level_cache=true plus a provider on the classpath) and mark each cacheable entity with JPA's @Cacheable and/or Hibernate's @Cache. The cache stores a disassembled, dehydrated form of entity state — identifiers and column values, not your live object graph — so it is provider-serializable and detached from any session. Good candidates are read-mostly reference data (countries, categories, currencies), not write-heavy tables.

@Entity
@Cacheable                                          // JPA: opt this entity in
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE) // Hibernate: pick a strategy
public class Country { /* ... */ }

@Cacheable, @Cache and shared-cache-mode

Two annotations, two layers. JPA's @Cacheable(true|false) is the portable, spec-level opt-in; Hibernate's @Cache adds what JPA leaves unspecified — which concurrency strategy (usage) and which named region. In a Hibernate application you usually want @Cache to choose the strategy explicitly; @Cacheable on its own falls back to provider defaults.

Whether @Cacheable is even consulted depends on shared-cache-mode (the jakarta.persistence.sharedCache.mode property or the persistence.xml element). ENABLE_SELECTIVE caches only entities annotated @Cacheable(true) — the sane production default. DISABLE_SELECTIVE caches everything except @Cacheable(false). ALL caches every entity and NONE disables the 2LC entirely — both ignore your annotations, and UNSPECIFIED leaves it to the provider. Knowing that ALL/NONE override the annotations is a common follow-up.

jakarta.persistence.sharedCache.mode=ENABLE_SELECTIVE  # cache only @Cacheable(true)

Cache concurrency strategies

A concurrency strategy tells Hibernate how to keep a shared, concurrent cache consistent with the database. READ_ONLY is for immutable reference data — fastest, no locking, and it refuses updates. NONSTRICT_READ_WRITE suits rarely-updated data: it simply invalidates (removes) the entry around a change and tolerates a brief stale window. READ_WRITE, the usual choice for mutable cached entities, uses soft locks — on update it replaces the entry with a lock marker and writes the new value only after the database transaction commits, so a concurrent reader falls through to the database rather than reading stale data. TRANSACTIONAL needs a fully transactional (JTA/XA) provider that participates in two-phase commit.

The practical rule: choose READ_ONLY whenever the data never changes, READ_WRITE when it does, and reach for TRANSACTIONAL only when a provider and infrastructure genuinely support it. Remember that READ_WRITE is an asynchronous, write-through cache — there is a short window after commit before the cache holds the new value, during which readers just hit the database.

Cache providers: JCache, Ehcache, Infinispan

Hibernate does not implement a cache itself; it defines an SPI (RegionFactory) and delegates to a provider. Since Hibernate 5.3/6 the standard integration is JCache (JSR-107), so any JCache-compliant provider plugs in through a single adapter — Ehcache 3 and Infinispan are the common choices, with Hazelcast and others available. Infinispan is the go-to when you need a clustered or distributed cache; Ehcache is the simple in-process default.

The provider owns eviction and expiration (time-to-live, time-to-idle, maximum entries), local-versus-distributed topology, and statistics — Hibernate only decides what is eligible and which strategy governs consistency. Match the topology to your deployment: a distributed cache running READ_WRITE across many nodes behaves very differently from an in-process Ehcache, and a mismatch is a classic production surprise.

The query cache

The 2LC keys entities by id, so it never helps a JPQL/HQL query such as "all active countries" — that still runs SQL. The query cache is a separate region that remembers, for an exact query-plus-parameter combination, the list of identifiers (and any scalar results) it returned. Re-running the identical query returns those ids and then resolves each entity from the 2LC. It is enabled globally with hibernate.cache.use_query_cache=true and, crucially, per query with the org.hibernate.cacheable hint (or setCacheable(true)) — nothing is query-cached unless you opt each query in.

Because it stores ids rather than entity state, the query cache is close to useless without second-level caching of the returned entities — otherwise every cache hit fires one SELECT per id to rehydrate them, re-introducing an N+1. Hibernate invalidates cached results through a timestamps region whenever any table the query touches is modified, which makes the query cache a poor fit for frequently-written tables.

hibernate.cache.use_query_cache=true                 // enable the region
List<Country> list = em.createQuery(jpql, Country.class)
    .setHint("org.hibernate.cacheable", true)        // opt THIS query in
    .getResultList();

Cache pitfalls and best practices

A cache that quietly serves wrong data is worse than no cache. The big traps: caching write-heavy entities, where constant invalidation costs more than it saves; external changes made through native SQL, another application, or a bulk JPQL update/delete that bypass the cache and leave it stale — you must evict the affected regions yourself through the Cache API; and enabling the query cache on volatile tables, where timestamp-based invalidation thrashes.

Best practices: cache read-mostly reference data with READ_ONLY or READ_WRITE; keep the query cache for genuinely repeated, stable queries and always pair it with entity 2LC; size and monitor each region by its hit/miss ratio instead of guessing; and never treat the 2LC as a correctness layer — it is an optimization that must degrade safely to a database read. Programmatic eviction (evictEntityData, evictQueryRegions) is the escape hatch after out-of-band writes.

Interview questions

The caching questions Indian interviewers return to — first-vs-second-level is a staple from screening rounds up to product-company deep dives:

Beginner What is the difference between first-level and second-level cache?

First-level cache is the persistence context — bound to a single EntityManager/Session, always on, cleared when the context closes, not shared. Second-level cache is on the SessionFactory, shared by all contexts (and optionally across a cluster), off by default, and opt-in per entity. On a miss in the first, Hibernate checks the second, then the database.

Beginner Is the first-level cache enabled by default? Can you turn it off?

Yes, it is always on and cannot be disabled — it is the persistence context itself. You can only influence it indirectly: clear() detaches everything, detach(entity) evicts one, and closing the context discards it.

Beginner How do you enable the second-level cache for an entity?

Turn the feature on (hibernate.cache.use_second_level_cache=true and a provider such as Ehcache/Infinispan on the classpath), set shared-cache-mode to ENABLE_SELECTIVE, and annotate the entity with @Cacheable plus Hibernate's @Cache(usage = ...) to choose a concurrency strategy.

Beginner What scope does each cache have?

First-level: transaction/context scope, one per EntityManager, per thread. Second-level: SessionFactory scope, shared across all sessions in the JVM, and cluster-wide with a distributed provider. That difference is why the first-level cache never helps a second request but the second-level cache can.

Intermediate What does the query cache store, and why must it be paired with the 2LC?

It stores the identifiers (and scalar values) a specific query-plus-parameters returned, not the entities themselves. On a hit it resolves each id — from the second-level cache if the entity is cached there, otherwise with a SELECT per id. Without entity 2LC that is an N+1, so the query cache alone can hurt more than it helps.

Intermediate Explain shared-cache-mode: ENABLE_SELECTIVE vs DISABLE_SELECTIVE, and ALL vs NONE.

ENABLE_SELECTIVE caches only entities marked @Cacheable(true); DISABLE_SELECTIVE caches everything except @Cacheable(false). ALL caches every entity and NONE disables the 2LC — both ignore the @Cacheable annotations entirely. ENABLE_SELECTIVE is the safe production default.

Intermediate What are the cache concurrency strategies, and when do you use each?

READ_ONLY for immutable data (no locking, rejects updates); NONSTRICT_READ_WRITE for rarely-changed data (invalidates around updates, brief stale window); READ_WRITE for mutable cached entities (soft locks, updates after commit); TRANSACTIONAL only with a JTA/XA-capable provider. Rule of thumb: READ_ONLY if you can, READ_WRITE if it changes.

Intermediate READ_WRITE vs NONSTRICT_READ_WRITE — what actually differs?

READ_WRITE uses soft locks: during an update the entry becomes a lock marker so concurrent readers skip the cache and hit the DB until the new value is written after commit — no stale reads. NONSTRICT_READ_WRITE does no locking; it just evicts the entry before and after the update and accepts a small window where a stale value could be read. Use nonstrict only for data that rarely changes and can tolerate staleness.

Intermediate Does the second-level cache store your entity objects?

No. It stores a dehydrated, disassembled form — the identifier and column values, associations as foreign-key ids — not the managed object graph. On a hit Hibernate rehydrates a fresh managed entity in the current context. That is why cached entities need no shared mutable state and why the cache is provider-serializable.

Pro Why can the second-level cache serve stale data, and how do you deal with it?

The 2LC is unaware of changes made outside Hibernate — native SQL, another application, or bulk JPQL update/delete statements that operate straight on the tables. Those leave cached entries stale. Mitigations: evict affected regions programmatically via the Cache API (evictEntityData, evictQueryRegions), route bulk changes through Hibernate where possible, and only cache data whose out-of-band mutation you control.

Pro When is the query cache an anti-pattern?

On write-heavy or frequently-queried-with-varying-parameters tables. Every modification to a queried table bumps the timestamps region and invalidates results, so a volatile table thrashes the cache; distinct parameter combinations each create their own entry, so the region can grow unbounded; and without entity 2LC it degrades into an N+1. It pays off only for genuinely repeated, stable queries over rarely-changing data.

Pro How do you cache a @OneToMany collection, and what is the trap?

Collection caching is separate from entity caching: you must annotate the association itself with @Cache, and the collection cache stores only the identifiers of the associated entities — so those child entities must also be cacheable, or each traversal re-queries them (another N+1). The trap is assuming caching the parent entity also caches its collections; it does not. Cache the collection deliberately and only for stable associations.
