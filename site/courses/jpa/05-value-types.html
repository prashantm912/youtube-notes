Module: Value Types, Embeddables and Converters

A customer's address is not an entity — it has no identity of its own and dies with the customer that owns it. This module maps that "part-of" data, and the ORDINAL-versus-STRING enum trap it contains is one of the most repeated JPA interview questions there is.

What this module covers

Not every class in your domain model deserves a primary key. Value types — embeddables, element collections, enums, and converter-mapped attributes — model data that belongs inside an entity rather than existing independently. This module covers @Embeddable/@Embedded composition, collections of value types via @ElementCollection, the @Enumerated ORDINAL-versus-STRING decision, and the AttributeConverter and @Lob machinery for whatever the built-in basic types cannot handle.

Topics

Value types versus entities

A value type has no identity of its own and no independent lifecycle: it exists only as state inside an owning entity, is loaded and saved with that owner, and is never shared by reference between two entities. Hibernate recognises three kinds of type — basic (String, int, LocalDate), embeddable (a composite of basics), and entity (identity plus lifecycle) — and value types are the first two.

Why it matters: value-vs-entity is the first modelling question for any group of fields. Get it wrong and you either bolt a meaningless surrogate key onto an address, or you demote something with real identity (an Order) into a value type and lose the ability to reference it. The distinction also drives equality — value types compare by their contents, entities by their id.

@Embeddable and @Embedded

@Embeddable marks a class whose fields map to columns of whatever entity embeds it; @Embedded (often optional) places one such value object into an entity. There is no separate table and no foreign key — the embeddable's columns sit directly in the owner's table, which keeps a naturally cohesive group (street/city/zip) together in code while staying flat in SQL.

When you embed the same type twice — a billing and a shipping Address — the two copies collide on column names. @AttributeOverride (and @AttributeOverrides) remaps the clashing columns per embedding. Embeddables can nest, and modern JPA/Hibernate let you model them as immutable Java records.

@Embeddable
public class Address {
    private String street;
    private String city;
    private String zipCode;
}

@Entity
public class Customer {
    @Id private Long id;
    @Embedded private Address address;

    @Embedded
    @AttributeOverride(name = "city", column = @Column(name = "billing_city"))
    private Address billing;   // same type, remapped columns
}

@ElementCollection: collections of value types

@ElementCollection persists a Collection of basic types or embeddables — a set of tags, a list of phone numbers — into a separate collection table linked back to the owner, without promoting each element to an entity. @CollectionTable names that table and its join column; @Column (or @AttributeOverride for embeddables) names the value columns.

Why it matters and the caveat: because the elements have no identity, Hibernate usually cannot update one row in place — most changes delete and re-insert the entire collection. That is fine for small, rarely-changing sets but a performance trap for large or frequently-edited ones, where a real @OneToMany is the better mapping. An element collection is also LAZY by default, so touching it outside a session throws LazyInitializationException.

@ElementCollection
@CollectionTable(name = "customer_phone",
        joinColumns = @JoinColumn(name = "customer_id"))
@Column(name = "phone")
private Set<String> phoneNumbers = new HashSet<>();

@Enumerated: ORDINAL versus STRING

An enum field maps to a column two ways. @Enumerated(EnumType.ORDINAL) — the default when you write no annotation at all — stores the constant's zero-based position (0, 1, 2). @Enumerated(EnumType.STRING) stores the constant's name ("PUBLISHED"). Ordinal is compact; string is readable and self-documenting in the database.

The trap is that the default is ORDINAL. Reorder the enum, or insert a new constant anywhere but the end, and every already-stored ordinal now points at a different constant — silent data corruption with no error at write time. The standard advice: use STRING (refactor-safe) unless storage is genuinely critical, and never lean on the ordinal default. This is one of the most frequently asked JPA interview questions.

enum Status { DRAFT, PUBLISHED, ARCHIVED }

@Enumerated(EnumType.STRING)   // stores "PUBLISHED" — safe to reorder
private Status status;
// no annotation, or ORDINAL, would store 0/1/2 — fragile

AttributeConverter and @Convert

An AttributeConverter<X, Y> defines two methods — convertToDatabaseColumn and convertToEntityAttribute — that translate a Java attribute type X to a column type Y and back. Annotate the converter @Converter (with autoApply = true to cover every attribute of that type) or attach it to a single field with @Convert. It is the standard, portable JPA hook (since JPA 2.1) for a type the provider does not handle natively — a Boolean stored as 'Y'/'N', a money type, an encrypted string.

Converters are also the clean way to map enums to explicit, stable database codes — decoupling the stored value from both the name and the ordinal — which sidesteps the ORDINAL fragility entirely; JPA 3.2's @EnumeratedValue is a newer, narrower alternative. Note the limits: a converter cannot be applied to an @Id, a @Version, a relationship, or an attribute already marked @Enumerated or @Temporal.

@Converter(autoApply = true)
public class BooleanToYNConverter
        implements AttributeConverter<Boolean, String> {

    public String convertToDatabaseColumn(Boolean value) {
        return Boolean.TRUE.equals(value) ? "Y" : "N";
    }
    public Boolean convertToEntityAttribute(String db) {
        return "Y".equals(db);
    }
}

@Lob and large objects; custom types

@Lob tells the provider to map a field to a large-object column: a String/char[] becomes a CLOB (character data), a byte[] becomes a BLOB (binary). Use it for article bodies, images, or documents that would overflow an ordinary VARCHAR. Watch the database-specific behaviour — on PostgreSQL a @Lob String maps to a large-object oid rather than TEXT, a classic surprise, so many teams prefer @Column(columnDefinition = "text").

When neither the basic types, an embeddable, nor an AttributeConverter is enough — you need control over the JDBC binding itself, or a JSON/array column — you reach for a custom Hibernate type. In practice most such needs (JSON, arrays) are already solved by the Hypersistence Utils library rather than a hand-written type.

@Lob
private String article;   // CLOB / TEXT

@Lob
private byte[] image;     // BLOB

Interview questions

The value-type questions Indian interviewers return to — mapping fundamentals in screening rounds, and the enum trap in almost every product-company deep dive:

Beginner What is a value type, and how does it differ from an entity?

A value type has no identity of its own and no independent lifecycle — it exists only as part of an owning entity's state, is loaded/saved with that owner, and is not shared by reference. An entity has a primary key and its own lifecycle. Basic types and embeddables are value types.

Beginner What do @Embeddable and @Embedded do?

@Embeddable marks a class as a reusable group of columns; @Embedded places one instance into an entity. The embeddable's columns live in the owning entity's table — no separate table, no foreign key.

Beginner Where are an embeddable's columns physically stored?

In the owning entity's own table, inline alongside its other columns. An embeddable is composition in Java that stays flat in SQL — it does not create a new table or a join.

Beginner What is @ElementCollection for?

Persisting a collection of basic values or embeddables (tags, phone numbers) that are not entities. The elements go into a separate collection table (@CollectionTable) keyed back to the owner, but each element has no identity of its own.

Beginner Difference between @Enumerated(ORDINAL) and @Enumerated(STRING)?

ORDINAL stores the constant's zero-based position (0, 1, 2); STRING stores its name ("PUBLISHED"). Ordinal is compact but positional; string is readable and survives reordering. If no annotation is given, the default is ORDINAL.

Intermediate Why is @Enumerated(ORDINAL) considered dangerous?

Because it is the silent default and it stores position. Reorder the enum or insert a constant anywhere but the end, and every existing ordinal now decodes to a different constant — data corruption with no write-time error. Prefer STRING, or a converter with explicit codes, unless storage size is genuinely critical.

Intermediate How do you embed the same embeddable twice without a column-name clash?

@AttributeOverride (or @AttributeOverrides) on the second @Embedded field remaps each colliding column — e.g. Address as both billing and shipping with billing_city/shipping_city.

Intermediate How does @ElementCollection differ from @OneToMany, and what is its performance caveat?

@OneToMany relates two entities, each with identity; @ElementCollection stores non-entity values owned by one entity. Because element-collection rows have no identity, Hibernate typically deletes and re-inserts the whole collection on any change — fine for small, stable sets, but a trap for large or frequently-updated ones, where @OneToMany wins.

Intermediate What is an AttributeConverter and when do you use one?

A class implementing convertToDatabaseColumn/convertToEntityAttribute to map a Java type to a column type and back — for types JPA does not persist natively (a Boolean as 'Y'/'N', a money type, an encrypted string). Apply it globally with @Converter(autoApply = true) or per field with @Convert. Standard since JPA 2.1 and provider-portable.

Intermediate When would you use an AttributeConverter instead of @Enumerated for an enum?

When you want an explicit, stable database code decoupled from both the enum's name and its ordinal — so reordering or renaming constants never breaks stored data, and the DB value stays meaningful (e.g. status codes 100/200). JPA 3.2's @EnumeratedValue is a newer, narrower option for the same goal.

Intermediate What does @Lob do, and what is the common PostgreSQL pitfall?

@Lob maps a field to a large-object column — CLOB for String/char[], BLOB for byte[]. On PostgreSQL a @Lob String is mapped to a large-object oid (stored out-of-line, accessed by handle) rather than TEXT, which surprises people; @Column(columnDefinition = "text") is the usual fix for large text.

Pro What attributes can an AttributeConverter NOT be applied to?

It only applies to basic attributes. It cannot convert an @Id, a @Version, a relationship (@ManyToOne etc.), or an attribute already annotated @Enumerated or @Temporal. For those you use the appropriate mapping or a Hibernate-specific custom type.

Pro Why do equals and hashCode matter for embeddables, especially inside an @ElementCollection?

Value types compare by content, so an embeddable should implement equals/hashCode over all its fields. Inside an element collection Hibernate relies on that value equality to detect which elements changed and which rows to delete/insert — a missing or wrong implementation causes phantom updates or lost changes.

Pro A production column uses @Enumerated(ORDINAL); a teammate inserts a new constant in the middle of the enum. What breaks, and how do you recover safely?

Every row stored before the change now decodes to the wrong constant, because the ordinals of all later constants shifted by one — a silent, data-wide corruption. Recovery: freeze writes, add the new constant only at the end as a stopgap, then migrate the column to STRING (or a converter with fixed codes) and backfill/translate existing values under a data migration. Long term, never rely on ORDINAL for a persisted enum.
