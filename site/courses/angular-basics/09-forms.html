Module: Forms

Forms are where Angular's binding, validation and reactivity all meet a real user typing — and the template-driven-vs-reactive fork, plus typed forms, is one of the most reliable interview questions in the framework.

Knowledge Library → Courses → Angular Basics → Forms

What this module covers

Angular ships two form-building systems — template-driven (directives in the HTML) and reactive (the model built in TypeScript) — plus the reactive building blocks FormControl, FormGroup, FormArray and FormBuilder. On top of those sit strictly typed forms (default since Angular 14), the validator system (built-in and custom), and the control-state model that drives error display. Reactive forms are the default choice for anything non-trivial, and are what most interview questions target.

Topics

Two ways to build forms: template-driven vs reactive

Template-driven forms live in the template: you add ngModel, required and a #form="ngForm" reference, and Angular builds the form model implicitly and asynchronously behind the scenes. Reactive forms invert this — you construct an explicit FormGroup tree in the component class and bind it to the template with [formGroup], so the model is synchronous, immutable to read, and testable without the DOM.

Why it matters: the official guidance is that reactive forms scale better (validation as plain functions, an observable API, easy unit testing) and template-driven forms suit simple, small forms migrated from legacy code. "Which would you pick and why" is a near-guaranteed screening question — the honest answer names data-flow direction, testability and scalability, not personal preference.

<!-- template-driven: the model is inferred from the template -->
<form #f="ngForm" (ngSubmit)="save(f.value)">
  <input name="email" ngModel required email>
  <button [disabled]="f.invalid">Save</button>
</form>

Reactive building blocks: FormControl, FormGroup, FormArray, FormBuilder

A FormControl tracks the value and validation state of a single field; a FormGroup aggregates named controls (and nests) into one object; a FormArray holds an ordered, dynamic list of unnamed controls — the tool for "add another phone number" style UIs. FormBuilder (injected via inject(FormBuilder) or the constructor) is syntactic sugar with group(), control() and array() factory methods that cut the boilerplate of writing new FormControl(...) everywhere.

Why it matters: interviewers expect you to reach for FormArray the moment the number of fields is unknown at design time, and to know that reading form.value is synchronous while form.valueChanges is an Observable. Confusing setValue (requires every control) with patchValue (partial update) is a classic trip-up.

private fb = inject(FormBuilder);

form = this.fb.group({
  email: ['', [Validators.required, Validators.email]],
  address: this.fb.group({ city: [''] }),   // nested group
  phones:  this.fb.array([ this.fb.control('') ]),  // dynamic list
});

Strictly typed reactive forms

Since Angular 14, reactive forms are strictly typed by default: the compiler infers the shape of a form from the initial values you pass, so form.value.email is known to be string | null and referencing a field that doesn't exist fails at compile time. The recommended style is to let FormBuilder infer everything — do not annotate the property as a bare FormGroup, because that collapses back to FormGroup<any> and silently throws the type safety away.

Why it matters: this is a top current interview topic. The two facts interviewers probe are (1) controls are nullable by default because reset() sets them to null, and (2) you opt out with nonNullable: true (or the NonNullableFormBuilder) so a reset returns to the initial value and the type drops the | null.

// inferred as FormGroup<{ name: FormControl<string | null>;
//                          age:  FormControl<number | null> }>
form = this.fb.group({ name: 'Asha', age: 30 });

// non-nullable: reset() restores 'Asha', type is FormControl<string>
name = new FormControl('Asha', { nonNullable: true });

Validation: built-in and custom validators

Validators are just functions. The Validators class provides the built-ins (required, minLength, email, pattern…) as both template attributes and reactive functions; a custom validator is a ValidatorFn that returns null when valid or an errors object like { forbiddenName: true } when not. Cross-field rules (password-confirm, date ranges) are validators attached to the parent FormGroup so they can read multiple controls, and asynchronous checks (is-this-username-taken) return a Promise or Observable via AsyncValidatorFn.

Why it matters: "write a custom validator" and "how do you validate that two fields match" are among the most common Angular coding-round tasks. Knowing that sync and async validators are separate constructor arguments — and that async validators only run once the sync ones pass — is the detail that separates rehearsed answers.

export function forbiddenName(re: RegExp): ValidatorFn {
  return (control: AbstractControl): ValidationErrors | null =>
    re.test(control.value) ? { forbiddenName: { value: control.value } } : null;
}
// usage: this.fb.control('', [Validators.required, forbiddenName(/admin/i)])

Form state, status and displaying errors

Every control and group exposes a state model: valid/invalid, pristine/dirty (has the value changed?), touched/untouched (has the field been blurred?), plus a status that can also be PENDING during async validation. Angular mirrors these onto the DOM as ng-valid, ng-dirty, ng-touched CSS classes, and exposes valueChanges and statusChanges observables for reacting to edits. The idiomatic error-display rule is to show a message only when the control is invalid and the user has interacted with it — dirty || touched — so a pristine form doesn't shout on load.

Why it matters: the practical "why isn't my error showing / why is it showing too early" question comes straight from this state model. Interviewers also like the distinction between dirty (value changed) and touched (field blurred) — they are not the same event.

<input [formControl]="email">
@if (email.invalid && (email.dirty || email.touched)) {
  @if (email.hasError('required')) { <p class="error">Email is required.</p> }
  @if (email.hasError('email'))    { <p class="error">Enter a valid email.</p> }
}

Interview questions

Most-asked Angular forms questions in Indian interviews — forms sit in almost every frontend screening and machine round:

Beginner What are the two types of forms in Angular?

Template-driven forms (built with directives like ngModel in the HTML, model inferred implicitly and asynchronously) and reactive forms (the FormGroup/FormControl model built explicitly in the component class and bound with [formGroup]). Reactive is the default choice for anything beyond trivial forms.

Beginner What is FormControl, FormGroup and FormArray?

FormControl tracks the value and validity of one field. FormGroup aggregates named controls into a single object (and can nest). FormArray is an ordered, dynamic list of unnamed controls — used when the number of fields isn't known upfront, e.g. a variable list of phone numbers.

Beginner What does FormBuilder do?

It's an injectable helper that reduces boilerplate: fb.group({...}), fb.control(...) and fb.array([...]) replace repeated new FormControl()/new FormGroup() calls. Functionally identical output — purely convenience and readability.

Beginner How do you apply a built-in validator like required or minLength?

In reactive forms pass functions from the Validators class as the second argument: fb.control('', [Validators.required, Validators.minLength(3)]). In template-driven forms use the equivalent HTML attributes (required, minlength) on the input.

Beginner setValue vs patchValue — what's the difference?

setValue requires a value for every control in the group and throws if one is missing — strict, good for full replacement. patchValue updates only the keys you provide and ignores the rest — convenient for partial updates like pre-filling some fields.

Intermediate Template-driven vs reactive — when would you choose each, and why?

Reactive for most real apps: synchronous immutable model, validators as plain functions, an observable API, and easy unit testing without the DOM — it scales. Template-driven for small, simple forms or quick migrations: less TypeScript, but the model is async and harder to test. Interviewers want the reasons (data-flow direction, testability, scalability), not a preference.

Intermediate What are typed forms and what changed in Angular 14?

As of Angular 14, reactive forms are strictly typed by default. The compiler infers a control's type from its initial value, so form.value, setValue, patchValue and field access are all type-checked and auto-completed — accessing a non-existent field is now a compile error instead of a runtime surprise.

Intermediate Are typed form controls nullable by default? How do you change that?

Yes — because reset() sets a control back to null, the inferred type is T | null. Pass { nonNullable: true } to the control (or use NonNullableFormBuilder) so reset() restores the initial value and the type becomes just T.

Intermediate How do you write a custom validator?

A validator is a function taking an AbstractControl and returning null when valid or an errors object (e.g. { forbiddenName: true }) when invalid. For parameters, wrap it in a factory that returns the ValidatorFn. Attach it in the control's validator array; the error key is what you check with hasError(...) in the template.

Intermediate How do you validate that two fields match (e.g. password and confirm)?

Cross-field validation: attach the validator to the parent FormGroup, not a single control, so it can read both children and compare them, returning e.g. { mismatch: true }. Display the error off the group. Field-level validators can't see sibling controls.

Intermediate Difference between pristine/dirty and touched/untouched?

dirty means the value has changed (opposite pristine); touched means the field has been blurred/visited (opposite untouched). They track different events — a user can touch a field without changing it. Error messages usually gate on dirty || touched so they don't appear on a fresh form.

Intermediate How does async validation work, and how does the PENDING status fit in?

Async validators (AsyncValidatorFn) return a Promise or Observable of errors-or-null — used for server checks like username availability. They run only after all sync validators pass; while one is in flight the control's status is PENDING, which you can use to show a spinner or disable submit.

Pro Why can declaring `form: FormGroup` explicitly break type safety, and what's the fix?

A bare FormGroup annotation is shorthand for FormGroup<any>, which discards the inferred control shape and returns you to untyped forms. The recommended approach is to let FormBuilder (or explicit FormControl initialisers) infer the type — don't annotate — or, if you must, write out the full generic FormGroup<{ ... }>.

Pro How do valueChanges/statusChanges work, and what's a common pitfall with them?

Both are RxJS Observables on any AbstractControl: valueChanges emits on every value edit, statusChanges on every validity recalculation — useful for dependent fields or live filtering. The pitfall is subscribing manually and never unsubscribing, leaking on component destroy; prefer the async pipe, takeUntilDestroyed(), or toSignal().

Pro What is updateOn and why would you change it?

updateOn controls when a control syncs its value and revalidates: 'change' (default, every keystroke), 'blur' (when the field loses focus), or 'submit' (only on form submit). Switching to 'blur' or 'submit' reduces validation churn — valuable for expensive async validators or large forms — at the cost of less immediate feedback.
