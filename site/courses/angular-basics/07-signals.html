Module: Signals and Reactivity

Signals are Angular's reactivity primitive — a value that tells the framework exactly what changed. Stable since v17, they are the single most-asked topic in current Angular interviews; this module maps every signal API to where it fits and sends the depth to the official guides.

Knowledge Library → Courses → Angular Basics → Signals and Reactivity

What this module covers

A signal is a wrapper around a value that notifies consumers when the value changes, letting Angular track dependencies and update only what actually depends on that state. This module walks the whole family — the writable signal(), derived computed(), side-effecting effect(), the signal-based input()/model()/output() and view queries, the newer linkedSignal() and resource(), and how all of it interoperates with (and when it replaces) RxJS.

Topics

Signals: the reactive primitive

You create writable state with signal(initialValue), read it by calling it as a function — count() — and change it with set() (replace) or update() (derive from the previous value). The call-to-read matters: it is how Angular records that a template, computed, or effect depends on this signal, so a later write can notify precisely those consumers. This granular tracking is what lets Angular move toward local, zoneless change detection instead of re-checking the whole component tree.

Why it matters: signals turn "when did this value change and who cares?" from a Zone.js guess into an exact, framework-tracked fact. Understanding read-tracking is the foundation every other signal API builds on.

const count = signal(0);
count();                       // read -> 0 (registers a dependency)
count.set(5);                  // replace the value
count.update(n => n + 1);      // derive from the previous value -> 6
const readOnly = count.asReadonly(); // expose without a setter

computed() and effect(): derived state and side effects

computed(() => …) creates a read-only signal whose value is derived from other signals. It is lazy (recomputed only when read after a dependency changed), memoized (cached until then), and tracks dependencies dynamically — only the signals actually read on the last run count. effect(() => …) runs a side effect whenever any signal it reads changes: logging, DOM work, syncing to localStorage. Reach for computed to produce a value and effect only to do something; effects should not be used to push one signal's value into another (prefer computed or linkedSignal for that).

const price = signal(100);
const qty   = signal(2);
const total = computed(() => price() * qty());  // memoized, recomputes on change

effect(() => console.log('total is now', total())); // runs on every change

Signal inputs, model() and output()

The signal-based component APIs replace the classic decorators: input() returns a read-only InputSignal you read like any signal (so it can feed a computed directly), input.required<T>() makes the binding mandatory at compile time, and a transform coerces incoming values (booleanAttribute, numberAttribute). model() is a writable input that wires up two-way binding — [(value)] — by pairing an input with an implicit valueChange output. output() returns an OutputEmitterRef whose emit() raises custom events, the function-based successor to @Output() and EventEmitter.

firstName = input.required<string>();               // must be bound
pageSize  = input(10, { transform: numberAttribute }); // coerced
value     = model(0);                               // two-way: [(value)]
selected  = output<string>();                        // selected.emit('x')

Signal queries: viewChild and contentChild

The query functions viewChild(), viewChildren(), contentChild() and contentChildren() return signals that always reflect the current query result, so you read a child reference inside a computed or effect rather than waiting for an ngAfterViewInit lifecycle moment. Use viewChild.required(...) when the element must exist. The Angular team recommends these signal queries over the @ViewChild/@ContentChild decorators for new code.

firstBox = viewChild.required<ElementRef>('box'); // #box in the template
// read reactively — no AfterViewInit ceremony:
boxWidth = computed(() => this.firstBox().nativeElement.offsetWidth);

linkedSignal() and resource(): writable-derived and async state

Sometimes derived state also needs to be writable — a selected option that recomputes when the source list changes but can still be overridden by the user. linkedSignal() is exactly that: a signal computed from a source that you can also set(). resource() (and its RxJS-flavoured rxResource()) bridges asynchronous data into the signal world: give it a reactive params and a loader, and it exposes value, status and error signals, re-running the loader and cancelling in-flight requests when the params change. These are newer APIs (introduced around v19 and still stabilising) — know what they are for even where they are not yet your default.

const options = signal(['std', 'express']);
const choice  = linkedSignal(() => options()[0]); // derived, but still settable
choice.set('express');

const user = resource({
  params: () => ({ id: this.userId() }),
  loader: ({ params }) => fetchUser(params.id),   // re-runs when userId changes
});

Signals vs RxJS: interop and choosing between them

Signals and RxJS are complementary, not rivals. Signals model synchronous, always-has-a-current-value UI state; RxJS models streams of events over time with operators for combining, debouncing and cancelling. The bridge is the interop layer: toSignal(observable$) turns a stream into a signal (like the async pipe, but usable anywhere and auto-unsubscribing), and toObservable(sig) turns a signal into a stream to plug into an operator pipeline. The interview-ready rule of thumb: use signals for state you render, and reach for RxJS when you are coordinating asynchronous events — then convert back to a signal before the template.

// stream -> signal (auto-unsubscribes; needs an initial or requireSync)
data = toSignal(this.http.get<Item[]>('/api/items'), { initialValue: [] });

// signal -> stream (to use debounceTime, switchMap, etc.)
const query$ = toObservable(this.query);

Interview questions

Signals dominate current Indian Angular interviews — from screening rounds at service companies to product-company deep dives on reactivity and change detection:

Beginner What is a signal and what problem does it solve?

A signal is a wrapper around a value that notifies consumers when it changes. Reading it (calling mySignal()) registers a dependency, so Angular knows exactly which templates, computeds and effects depend on that value and can update only those — replacing coarse, whole-tree change detection with precise, tracked updates.

Beginner How do you create, read and update a signal?

const s = signal(0) creates it; s() reads it (a getter call, not a property); s.set(5) replaces the value; s.update(v => v + 1) derives the new value from the old. s.asReadonly() exposes it without a setter.

Beginner set() vs update() — when do you use each?

set() when the new value is independent of the current one; update() when it is computed from the current value (increment, toggle, push into a copied array). Both trigger the same change notification.

Beginner What is a computed signal?

A read-only signal whose value is derived from other signals via computed(() => …). It is lazy and memoized: it recomputes only when read after one of its dependencies changed, and caches the result otherwise. It has no set/update.

Beginner Why do you read a signal by calling it as a function?

The getter call is the tracking mechanism. Calling the signal inside a template, computed or effect records "this consumer depends on this signal," which is how Angular later notifies exactly the right consumers on a write. A plain property couldn't register that dependency.

Intermediate computed() vs effect() — how do you choose?

Use computed to produce a derived value (pure, returns something, memoized). Use effect only to perform a side effect in response to change (logging, DOM, storage). If you find yourself setting one signal from inside an effect, you almost always want a computed or a linkedSignal instead.

Intermediate How do signal inputs (input()) differ from @Input()?

input() returns a read-only signal, so the value is reactive and can feed a computed or effect directly — no ngOnChanges needed to react. input.required<T>() enforces the binding at compile time, and a transform coerces values. @Input() is a plain property set imperatively by Angular.

Intermediate What is model() and how does it enable two-way binding?

model() is a writable signal input. Declaring value = model(0) lets a parent write [(value)]="x": Angular pairs the input with an implicit valueChange output, and calling value.set(...) in the child emits back to the parent. It is the signal-era replacement for the input + Change-output banana-in-a-box pattern.

Intermediate How do signal queries (viewChild()) improve on @ViewChild?

viewChild() returns a signal that always reflects the current result, so you read it reactively in a computed or effect rather than waiting for ngAfterViewInit. viewChild.required() asserts presence. The team recommends signal queries for new code.

Intermediate What do toSignal() and toObservable() do?

toSignal(obs$) converts an Observable into a signal — like the async pipe but usable anywhere in code, and it auto-unsubscribes when the injection context is destroyed. toObservable(sig) goes the other way, emitting the signal's value into a stream (via an internal effect) so you can apply RxJS operators.

Intermediate How do signals relate to change detection and OnPush?

A signal read in a template marks that view for check on the next change; signal-driven updates are granular, so they work perfectly with OnPush and are the foundation for zoneless change detection (dropping Zone.js entirely). The framework updates only views whose signals changed rather than dirty-checking the whole tree.

Pro Signals vs RxJS — when do you reach for each?

Signals for synchronous UI state that always has a current value and is rendered. RxJS for streams of events over time — debouncing input, cancelling with switchMap, combining sources, retry/backoff. They interoperate: coordinate the async work in RxJS, then toSignal() it before the template. They are complementary, not a replacement of one by the other.

Pro Why is setting a signal inside an effect discouraged, and what are the pitfalls?

Effects are for side effects, not state propagation. Writing signals in an effect risks feedback loops, makes the data flow implicit and ordering-sensitive, and bypasses the glitch-free guarantees you get from derived signals. The idiomatic fix is computed() for pure derivation or linkedSignal() when the derived value must also be writable.

Pro When would you use linkedSignal() over computed() or a plain signal?

When you need derived state that is also user-writable. A plain computed can't be set(); a plain signal won't recompute when its source changes. linkedSignal() recomputes from a source yet still accepts overrides — e.g. a default-selected item that resets when the list changes but can be manually changed.

Pro How does computed() stay efficient and correct — lazy, memoized, glitch-free?

It doesn't recompute on every dependency write; it marks itself stale and recomputes only when next read (lazy), caching that result until a dependency changes again (memoized). Dependencies are tracked dynamically — only signals read on the latest run count, so branches that weren't taken don't create dependencies. Reading a graph of computeds yields consistent values without intermediate "glitch" states.
