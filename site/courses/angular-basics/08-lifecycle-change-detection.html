Module: Lifecycle and Change Detection

Change detection is the single most-asked Angular interview topic — Zone.js versus zoneless, and OnPush. Pair it with the lifecycle hooks that fire around each check, and you can explain not just what a component does but when Angular runs it.

Knowledge Library → Courses → Angular Basics → Lifecycle and Change Detection

What this module covers

Two intertwined subjects. First, the lifecycle: the ordered hooks Angular calls from a component's construction to its destruction (ngOnChanges, ngOnInit, ngAfterViewInit, ngOnDestroy and friends), plus modern cleanup and render-phase APIs. Second, change detection: how Angular decides when to re-read your template expressions and repaint the DOM — the classic Zone.js model, the OnPush optimization, and the current zoneless/signal-based direction. The two meet because most hooks fire during a change-detection pass.

Topics

Lifecycle hooks and their order

A component moves through a fixed sequence: the constructor runs first (dependencies injected, but inputs not yet set), then during change detection Angular calls ngOnChanges (whenever a data-bound input changes, with a SimpleChanges map), ngOnInit (once, after the first inputs are set — the right place for initialization and data fetching), ngDoCheck, the content hooks ngAfterContentInit/ngAfterContentChecked, and the view hooks ngAfterViewInit/ngAfterViewChecked. The reason "put it in ngOnInit, not the constructor" is a stock interview line: at constructor time the @Input bindings have not been resolved yet.

Why it matters: knowing the order explains real bugs — a @ViewChild reference is undefined in ngOnInit because the view is not built until ngAfterViewInit. With signal-based components, an effect() or computed() increasingly replaces the old ngOnChanges plumbing, but the hooks remain examinable and still power template-facing components.

export class UserCard implements OnInit, OnChanges {
  @Input() userId!: string;
  ngOnChanges(changes: SimpleChanges) { /* input changed */ }
  ngOnInit() { /* run once: fetch, subscribe, set up */ }
}

Cleanup and render-phase hooks

ngOnDestroy runs just before Angular tears a component down — the place to unsubscribe from RxJS streams, clear timers and detach listeners so you do not leak memory. Modern Angular offers two cleaner alternatives: inject DestroyRef and register an onDestroy callback anywhere in the injection context, or pipe a stream through takeUntilDestroyed() so it completes automatically when the component dies. For work that must touch the real DOM after Angular has rendered, afterNextRender (once) and afterEveryRender (on every render) run outside change detection, replacing the old habit of a setTimeout in ngAfterViewInit.

Why it matters: "how do you avoid subscription memory leaks?" is a near-guaranteed question, and the expected senior answer is now takeUntilDestroyed or the async pipe rather than a manual Subject + takeUntil boilerplate.

constructor() {
  // completes when this component is destroyed — no ngOnDestroy needed
  this.data$.pipe(takeUntilDestroyed()).subscribe(v => this.value = v);
}

How change detection works: Zone.js and the component tree

Angular keeps the DOM in sync with your data by re-evaluating every template expression and updating the bindings that changed. Each component owns a change detector, and a check walks the component tree from the root downward. In the traditional model, Zone.js monkey-patches async browser APIs (addEventListener, setTimeout, Promise, XHR) so that whenever any of them fires, Angular is notified that "something might have changed" and runs a full check pass. That is convenient but does more work than necessary. In development mode Angular runs a second verification pass and throws ExpressionChangedAfterItHasBeenCheckedError if a value mutated mid-check — a guard against templates whose values depend on their own detection.

Why it matters: interviewers probe whether you understand that the default strategy checks the whole tree on every event, why that can hurt performance in large apps, and what Zone.js actually is under the hood.

The OnPush change-detection strategy

Setting changeDetection: ChangeDetectionStrategy.OnPush tells Angular to skip a component's subtree unless one of a few things happens: an @Input changes by reference, an event fires inside the component, an async-piped Observable emits, or a bound signal updates. The reference-equality rule is the whole point — mutating an object in place (user.name = 'x') will not trigger an OnPush check, so OnPush pushes you toward immutable updates (spread into a new object/array). When you genuinely need to force a check, inject ChangeDetectorRef and call markForCheck().

Why it matters: OnPush is the classic performance-tuning answer and a favourite "explain the trade-off" question. The trap they set: "you set OnPush and the view stopped updating — why?" (Answer: you mutated instead of replacing.)

@Component({ changeDetection: ChangeDetectionStrategy.OnPush, /* ... */ })
export class TodoList {
  @Input() todos: Todo[] = [];
  add(t: Todo) { this.todos = [...this.todos, t]; } // new reference -> checked
}

Zoneless and signal-based change detection

The current direction removes Zone.js entirely. With provideZonelessChangeDetection(), Angular no longer relies on patched browser APIs to guess that state changed; instead it schedules a check only when it gets an explicit notification — a signal being set, an async pipe emitting, an OnPush input changing, or a direct markForCheck(). Signals are the ideal partner: because Angular knows exactly which components read a signal, it can do targeted, local checks instead of walking the whole tree. Zoneless has matured from experimental through developer preview to stable in recent versions (v20+), you can drop the zone.js dependency, and new projects are increasingly scaffolded this way. Practically, "zoneless-ready" code means OnPush plus signals or the async pipe — the same discipline OnPush already teaches.

Why it matters: this is the hottest 2025–2026 Angular topic. Being able to contrast "Zone.js tells Angular something might have changed" with "signals tell Angular exactly what changed" is what separates a current answer from a dated one.

bootstrapApplication(App, {
  providers: [provideZonelessChangeDetection()],
});
// a signal write is itself the change-detection notification
count = signal(0);
inc() { this.count.update(n => n + 1); }

Interview questions

The most-asked lifecycle and change-detection questions in Indian Angular interviews — change detection is a top screening and product-company topic, so expect the OnPush and zoneless questions to go several layers deep:

Beginner What are Angular lifecycle hooks, and can you name the main ones in order?

Methods Angular calls at defined moments of a component's life. Order: constructor → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked, and ngOnDestroy at teardown.

Beginner Difference between the constructor and ngOnInit? Why put init logic in ngOnInit?

The constructor runs at instantiation for dependency injection only; at that point @Input bindings are not set. ngOnInit runs once after the first inputs are resolved, so data fetching, subscriptions and setup that depend on inputs belong there.

Beginner What is ngOnDestroy used for?

It runs just before the component is destroyed — the hook to unsubscribe from Observables, clear intervals/timeouts, and remove manual event listeners so the component does not leak memory.

Beginner In one line, what is change detection in Angular?

The process by which Angular checks whether component state (template expressions) changed and updates the DOM to match. It walks the component tree from root to leaves.

Intermediate When does ngOnChanges fire, and what does SimpleChanges give you?

It fires whenever a data-bound @Input changes, before ngOnInit on the first pass. SimpleChanges maps each changed input to its previousValue, currentValue and firstChange flag. With signal inputs, an effect() or computed() often replaces it.

Intermediate Why is a @ViewChild reference undefined in ngOnInit? Which hook should you use?

The component's view (and its child views) is not constructed until after ngOnInit, so view queries are not populated yet. Access them in ngAfterViewInit. (Signal-based viewChild() queries are read reactively instead.)

Intermediate What triggers change detection in a default (Zone.js) Angular app?

Zone.js patches async APIs — DOM events, setTimeout/setInterval, Promise resolution, XHR/fetch. When any completes, Zone notifies Angular, which runs a full check pass over the tree. You can also trigger it manually via ApplicationRef.tick() or ChangeDetectorRef.detectChanges().

Intermediate What is the OnPush strategy and what makes Angular check an OnPush component?

OnPush skips the component unless: an @Input changes by reference, an event fires inside it, an async-piped Observable emits, a bound signal updates, or markForCheck() is called. It cuts unnecessary checks but demands immutable input updates.

Intermediate How do you prevent RxJS subscription memory leaks in a component?

Prefer the async pipe (Angular unsubscribes for you), or pipe streams through takeUntilDestroyed(), or register cleanup via injected DestroyRef. The older pattern is a Subject + takeUntil completed in ngOnDestroy.

Intermediate What causes ExpressionChangedAfterItHasBeenCheckedError, and how do you fix it?

A value bound in the template changed after Angular checked it but before the dev-mode verification pass — usually mutating state inside a hook like ngAfterViewInit. Fixes: move the change to ngOnInit, defer with setTimeout/a microtask, or restructure so the value is stable during the check. It only throws in development mode.

Pro You switched a component to OnPush and its view stopped updating. Why, and how do you fix it?

You almost certainly mutated an input object/array in place, so its reference did not change and OnPush skipped the check. Fix by replacing the reference immutably (spread into a new object/array), driving updates through signals or the async pipe, or calling markForCheck() when a manual nudge is unavoidable.

Pro How do signals change the change-detection story, and how do they relate to OnPush and zoneless?

A signal knows exactly which views read it, so writing one lets Angular mark just those views dirty — targeted, local change detection instead of a whole-tree walk. Signals notify OnPush components directly (no markForCheck needed), and in a zoneless app the signal write is the notification that schedules detection.

Pro What is zoneless change detection, how do you enable it, and what's the trade-off versus Zone.js?

Zoneless drops Zone.js and runs change detection only on explicit notifications — signal writes, async-pipe emissions, OnPush input changes, markForCheck(). Enable it with provideZonelessChangeDetection() and remove the zone.js polyfill. Trade-off: smaller bundle, fewer needless checks and simpler reasoning, but code that relied on Zone auto-detecting arbitrary async mutations must now notify Angular itself (signals/async pipe). It is stable in recent versions (v20+) and the recommended direction.

Pro Beyond OnPush, what techniques reduce change-detection cost in a large Angular app?

Adopt signals for fine-grained local updates; move heavy non-UI async work out with NgZone.runOutsideAngular (Zone apps); detach a subtree via ChangeDetectorRef.detach() and re-check on demand; keep ngDoCheck cheap (it runs every cycle); use @for with track (or trackBy) and pure pipes to avoid re-rendering unchanged rows; and ultimately go zoneless.
