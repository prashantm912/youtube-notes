Module: Services and Dependency Injection

Dependency injection is the wiring under every Angular app — and in the standalone era the idioms shifted: inject() over the constructor, providedIn: 'root' over module registration. Knowing why is what interviews probe.

Knowledge Library → Courses → Angular Basics → Services and Dependency Injection

What this module covers

Services move reusable logic and shared state out of components; dependency injection (DI) delivers them without new. This module orients you on creating injectable services, the two ways to inject (inject() vs the constructor), providedIn and tree-shakable singletons, then the deeper machinery — the hierarchical injector tree, injection tokens for non-class values, and the provider scopes that decide how many instances you get. DI is arguably the single most-tested Angular topic in interviews.

Topics

Services, providedIn, and inject() vs constructor injection

A service is just a class marked @Injectable() so Angular's DI system can create it and hand it to whoever asks. providedIn: 'root' registers it with the application's root injector as a lazily-created, app-wide singleton — and because the reference points from the service to the injector (not from a module to the service), unused services are tree-shaken out of the bundle. This is why providedIn: 'root' is the default, and why registering services in an NgModule providers array is now the legacy path.

There are two ways to receive a dependency: list it as a constructor parameter, or call the inject() function. They resolve against the same injector, but inject() is the modern idiom — it drops the constructor boilerplate, gives more precise types, sidesteps the super() dependency-forwarding pain in inheritance, and (crucially) works in places a constructor cannot: route guards, functional interceptors, and provider factory functions. The one gotcha it introduces is timing: inject() only works inside an injection context (field initializers, the constructor, factory functions), so calling it in ngOnInit throws NG0203. Why inject() exists — and when constructor injection is still fine — is a top interview theme.

@Injectable({ providedIn: 'root' })   // app-wide, tree-shakable singleton
export class UserService {
  private http = inject(HttpClient);  // field injection — no constructor needed
  getUser(id: string) { return this.http.get(`/api/users/${id}`); }
}

@Component({ /* ... */ })
export class ProfileComponent {
  private users = inject(UserService);          // modern
  // constructor(private users: UserService) {} // classic — same injector
}

Hierarchical injectors, injection tokens, and provider scopes

Angular does not have one injector — it has a tree of them, in two hierarchies. The EnvironmentInjector chain (platform → root → lazy-loaded routes) exists outside the component tree; the ElementInjector is created at every component/directive that declares providers. When you request a dependency, Angular walks the ElementInjector up to the host component, then hands off to the EnvironmentInjector chain up to root; the first provider found wins, and if none is found you get a NullInjectorError. Where you register a provider therefore decides how many instances exist and who can see them: providedIn: 'root' gives one shared instance, while a component's providers array gives a fresh instance per component, scoped to that component and its descendants.

Not every dependency is a class. Interfaces, primitives, configuration objects and functions have no runtime type for DI to key on, so you create an InjectionToken and provide a value for it. Providers can shape that value four ways: useClass (swap an implementation), useValue (a ready-made constant or config), useFactory (compute it at runtime, optionally from other dependencies), and useExisting (alias one token to another so they share an instance). Provider scopes round this out — 'root', the rarely-needed 'platform', and 'any' (a distinct instance per lazy-loaded injector) — and knowing when to reach for each is a senior-level distinction.

export interface AppConfig { apiUrl: string; }
export const APP_CONFIG = new InjectionToken<AppConfig>('app.config');

bootstrapApplication(App, {
  providers: [{ provide: APP_CONFIG, useValue: { apiUrl: '/api' } }]
});

@Component({ providers: [BasketService] })  // ElementInjector: new instance here
export class BasketComponent {
  private cfg = inject(APP_CONFIG);         // reads the token's value
}

Interview questions

DI is the most-probed Angular topic in Indian interviews — inject() vs constructor and providedIn scopes come up in nearly every round, service-company screen to product-company deep dive:

Beginner What is a service in Angular and why use one?

A plain class (usually @Injectable) that holds reusable logic or shared state — data access, business rules, cross-component state — kept out of components. Components stay focused on the view; the service is injected wherever needed, which improves reuse and testability.

Beginner What is dependency injection and why does Angular use it?

A design pattern where a class receives its dependencies instead of constructing them. Angular's injector creates and supplies them, so classes are decoupled from concrete implementations — easier to test (swap a mock), easier to share a single instance, and easier to reconfigure without touching consumers.

Beginner What does @Injectable() with providedIn: 'root' do?

@Injectable() marks the class as participating in DI (it can be injected and can itself inject). providedIn: 'root' registers it with the root injector as a lazily-created, application-wide singleton, and makes it tree-shakable — if nothing injects it, it's dropped from the bundle.

Beginner inject() vs constructor injection — what's the difference?

Both pull a dependency from the same injector. Constructor injection lists dependencies as parameters; inject() is a function you call in an injection context. inject() means less boilerplate, more precise types, no super() plumbing when extending a class, and it works in functions (guards, interceptors, factories) where there is no constructor. It's the recommended modern style; constructor injection still works.

Intermediate Where can inject() be called, and why does inject() in ngOnInit throw?

Only inside an injection context: field initializers, the constructor, and provider/token factory functions. ngOnInit runs after the instance is already created, so no injection context is active and Angular throws NG0203. Fix: inject into a field or the constructor, or wrap the call in runInInjectionContext().

Intermediate When is a service a singleton, and when is it not?

providedIn: 'root' gives one instance for the whole app. But if you also list the service in a component's (or route's) providers array, that injector creates its own instance — so it's no longer a global singleton, it's one-per-that-injector. Lazy-loaded scopes and providedIn: 'any' can likewise produce multiple instances.

Intermediate Explain hierarchical injectors — how does Angular resolve a dependency?

There are two trees: the EnvironmentInjector chain (platform → root → lazy routes) and the ElementInjector created at components/directives with providers. Angular searches the ElementInjector upward to the host, then the EnvironmentInjector chain up to root. The first matching provider wins; if none is found, it throws NullInjectorError: No provider for X.

Intermediate What is an InjectionToken and when do you need one?

A unique token used to inject values that have no class/runtime type — interfaces, primitives, config objects, functions (interfaces disappear after compilation, so they can't be a DI key). You create new InjectionToken<AppConfig>('desc'), provide a value with useValue/useFactory, and read it with inject(TOKEN). The same token instance must be used to provide and to inject.

Intermediate useClass vs useValue vs useFactory vs useExisting?

useClass: instantiate a (possibly different) class for the token — swap implementations. useValue: provide a ready-made static value or config object. useFactory: run a function to compute the value at runtime, optionally using other injected dependencies. useExisting: alias the token to an already-provided one so both resolve to the same instance.

Intermediate providedIn: 'root' vs providing in a component's providers array — how does scope differ?

'root' is an app-wide, tree-shakable singleton. A component providers entry creates a new instance for every instance of that component, visible only to it and its descendants (via the ElementInjector), and destroyed with the component. Use component-level providers when you need isolated, per-instance state.

Pro What makes a provider tree-shakable, and why is providedIn better than an NgModule providers array?

With providedIn, the dependency points from the service to the injector, so there's no import forcing it into the bundle — if no code injects it, the optimizer removes it. Registering in an NgModule providers array requires importing the service into the module, an eager reference the bundler can't drop, so it's never tree-shaken.

Pro Explain the resolution modifiers: @Optional, @Self, @SkipSelf, @Host (and their inject() options).

@Optional: return null instead of throwing when no provider is found. @Self: only look in the current ElementInjector. @SkipSelf: skip the current injector, start at the parent. @Host: stop searching at the host component's boundary. With inject() they're options: inject(X, { optional: true, self: true, skipSelf: true, host: true }).

Pro Compare provider scopes 'root', 'platform' and 'any' — and what problem does 'any' solve?

'root': one instance across the app. 'platform': shared across multiple Angular apps bootstrapped on the same page (rare). 'any': a distinct instance in each lazy-loaded injector while eagerly-loaded contexts share the root instance — it cleanly gives each lazy module its own instance without the old forRoot()/forChild() pattern.

Pro How do multi providers (multi: true) work, and where are they used?

Several providers register against the same token with multi: true; instead of the last one overriding, inject(TOKEN) returns an array of all provided values. Angular uses this for extensible chains — HTTP_INTERCEPTORS, multiple APP_INITIALIZERs, NG_VALIDATORS — where every registered contributor must run.
