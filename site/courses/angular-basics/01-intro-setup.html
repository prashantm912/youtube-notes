What this module covers

Angular is Google's batteries-included framework for building single-page applications in TypeScript. This module orients you on the two things you need before writing features: the big picture — what Angular is, where it sits against React and Vue, and what the CLI gives you — and the shape of a generated project, from the main.ts that boots the app to the standalone components that are now the default. Depth on each topic lives in the linked official docs.

Topics

What Angular is, SPA/framework overview, Angular vs React/Vue, the CLI

Angular is a component-based, TypeScript-first framework maintained by Google for building single-page applications (SPAs) — apps where the browser loads one HTML shell and Angular swaps views on the client instead of requesting a fresh page each time. Unlike a UI library, it ships opinionated answers for the whole app: routing, forms, HTTP, dependency injection and reactivity all come in the box, which is why it dominates in large enterprise teams that value one consistent way to do things.

Why it matters: the framework-versus-library distinction is the classic opening interview question. React is a view library you assemble with third-party pieces; Vue sits in the middle; Angular is the full platform with conventions enforced by the CLI. That CLI — ng — is how you live in Angular day to day: it scaffolds the project, runs a live-reload dev server, generates components and services, and produces optimized production builds, so you rarely wire tooling by hand.

ng new my-app        # scaffold a new workspace
ng serve             # dev server with live reload at localhost:4200
ng generate component hello   # or: ng g c hello
ng build             # optimized production bundle

Project structure, bootstrapping, standalone components (current default), TypeScript role

A freshly generated workspace has a small, learnable set of files: angular.json configures the build and dev targets, package.json lists dependencies, tsconfig.json tunes TypeScript, and under src/ the main.ts is the entry point that boots the app. Bootstrapping is what starts Angular at runtime: bootstrapApplication(App, appConfig) renders a root standalone component and wires up the application-wide providers declared in app.config.ts — the modern replacement for the old platformBrowserDynamic().bootstrapModule(AppModule) ceremony.

Why it matters: standalone components are now the default (stabilized in v14, made the default in v19), so a component declares its own template dependencies in an imports array and needs no NgModule — less boilerplate and trivial lazy-loading. Underpinning all of it is TypeScript: Angular leans hard on types and decorators (@Component) for tooling, editor support and safer refactors, which is why the framework is TypeScript-first rather than optional.

// main.ts — the application entry point
import { bootstrapApplication } from '@angular/platform-browser';
import { App } from './app/app';
import { appConfig } from './app/app.config';

bootstrapApplication(App, appConfig);
@Component({
  selector: 'app-root',
  imports: [Hello],           // standalone: pulls in its own dependencies
  template: `<h1>My App</h1><app-hello />`,
})
export class App {}

Interview questions

Introduction-round questions from Indian service and product companies — Angular fundamentals every fresher and lateral hire is expected to answer:

Beginner What is Angular, and how is it different from AngularJS?

Angular is a TypeScript-based, component-driven framework by Google for building SPAs. AngularJS (1.x) was the older JavaScript framework built on controllers, scopes and two-way digest cycles; Angular (2+) is a complete rewrite — components instead of controllers, TypeScript, a new CLI, and far better performance. "Angular" today means version 2 and above.

Beginner What is a Single Page Application (SPA)?

An app that loads one HTML page and then updates the view on the client as the user navigates, instead of requesting a full new page from the server each time. Angular's router swaps components in and out, giving a faster, app-like feel after the initial load.

Beginner What does the Angular CLI do? Name a few key commands.

It scaffolds and manages projects from the terminal. Core commands: ng new (create a workspace), ng serve (dev server with live reload), ng generate / ng g (create components, services, etc.), ng build (production bundle), ng test. It enforces consistent structure and wires up tooling for you.

Beginner What is the entry point of an Angular application?

src/main.ts. It calls bootstrapApplication(App, appConfig) (or historically platformBrowserDynamic().bootstrapModule(AppModule)) to start Angular and render the root component into the <app-root> element in index.html.

Beginner What is a standalone component and why does it matter?

A component that manages its own template dependencies through an imports array instead of being declared in an NgModule. It is the default since Angular v19 — less boilerplate, easier lazy-loading, and the recommended way to write all new Angular code.

Beginner What is the role of TypeScript in Angular?

Angular is written in and authored with TypeScript. Types plus decorators (@Component, @Injectable) power the tooling, editor autocomplete, template type-checking and safer refactors. It is not optional — the framework is TypeScript-first by design.

Intermediate Angular vs React — how would you contrast them?

Angular is a full, opinionated framework (routing, forms, HTTP, DI, reactivity built in) and is TypeScript-first; React is a view library you compose with external libraries for routing and state. Angular gives one consistent way to do things across a large team; React gives flexibility and a larger ecosystem of choices. Vue sits between the two.

Intermediate Walk through the key files in a newly generated Angular project.

angular.json — build/serve configuration; package.json — dependencies and scripts; tsconfig.json — TypeScript settings; src/index.html — the shell with <app-root>; src/main.ts — bootstraps the app; src/app/app.config.ts — app-wide providers; src/app/app.component.ts (or app.ts) — the root standalone component.

Intermediate How does a standalone component get its dependencies, versus an NgModule component?

A standalone component lists what its template uses (other components, directives, pipes) directly in its own imports array. An NgModule-based component relied on the surrounding module's declarations and imports to make those available. Standalone keeps the dependency graph local and explicit to each component.

Intermediate What is angular.json and what do you configure there?

The workspace configuration file. It defines projects and their build/serve/test targets: entry points, output paths, assets, styles and scripts, build budgets, and named configurations such as production. The CLI reads it for every ng build / ng serve.

Pro How do you register application-wide providers (HttpClient, Router) in a standalone app with no NgModule?

Through the ApplicationConfig passed to bootstrapApplication — typically app.config.ts — using provider functions: provideHttpClient(), provideRouter(routes), provideAnimations(), and so on in its providers array. These replace what you used to put in AppModule's imports/providers.

Pro How does bootstrapApplication differ from platformBrowserDynamic().bootstrapModule?

bootstrapApplication(App, appConfig) boots a standalone root component directly with a flat provider config — no root module needed. The older platformBrowserDynamic().bootstrapModule(AppModule) boots an NgModule whose bootstrap array names the root component. The standalone path is the current default and produces smaller, more tree-shakable startup code.

Pro Why did Angular move from NgModules toward standalone components?

NgModules added indirection and boilerplate — you had to register every component somewhere and reason about module boundaries. Standalone makes dependencies explicit per component, simplifies mental model and testing, enables straightforward route-level lazy loading via loadComponent, and improves tree-shaking. NgModules still work (set standalone: false) for gradual migration, but new code is standalone.
