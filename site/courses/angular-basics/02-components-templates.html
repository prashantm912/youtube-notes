Module: Components and Templates

A component is a TypeScript class, an HTML template and a CSS selector fused into one unit — and the template's binding syntax is where Angular decides, at compile time, exactly how data flows into the DOM and events flow back out.

Knowledge Library → Courses → Angular Basics → Components and Templates

What this module covers

Components are the building block of every Angular app: this module covers the @Component decorator and the standalone anatomy that is the modern default, how selectors attach a component to the DOM, and how styles are scoped through view encapsulation. It then walks the full binding surface — interpolation, property/attribute/class/style binding, event binding and two-way binding — the syntax you use in every template. Everything here targets current Angular (v20, standalone components).

Topics

The @Component decorator and standalone anatomy

Every component is three things bound together: a TypeScript class holding state and behaviour, an HTML template describing the view, and a CSS selector naming how the component is used in markup. The @Component decorator wires them through metadata — selector, template or templateUrl, styles or styleUrl, and imports. Since v19 components are standalone by default: rather than being declared in an NgModule, a component lists its own template dependencies (other components, directives, pipes) in its imports array.

Why it matters: this is the unit of composition for the entire framework, and the standalone default is the single biggest shift in modern Angular — expect to explain it in any current interview.

import { Component } from '@angular/core';
import { AvatarComponent } from './avatar.component';

@Component({
  selector: 'app-user-card',
  imports: [AvatarComponent],   // standalone deps (standalone: true is the default since v19)
  templateUrl: './user-card.html',
  styleUrl: './user-card.css',
})
export class UserCard { /* state + behaviour live here */ }

Selectors: how a component attaches to the DOM

The selector is a CSS selector that Angular matches statically at compile time to decide where a component renders. Three forms are supported: element selectors (app-user, the norm — custom hyphenated tags), attribute selectors ([appTooltip], used to extend an existing native element), and class selectors (rare). Combinators and most pseudo-classes are not allowed; teams prefix their selectors consistently and avoid the reserved ng prefix.

Why it matters: compile-time matching is why a mistyped or unimported selector silently renders nothing rather than erroring — a frequent "why is my component blank?" debugging story.

@Component({ selector: 'app-user' })      // used as <app-user></app-user>
@Component({ selector: '[appTooltip]' })  // used as <button appTooltip>…</button>

Component styles and view encapsulation

Styles declared via styles/styleUrl apply only to that component's template. How that scoping happens is set by the component's encapsulation property. Emulated (the default) rewrites your selectors and stamps generated attributes (_nghost-*, _ngcontent-*) onto elements so styles cannot leak out; ShadowDom uses the browser's native Shadow DOM for real isolation; None disables scoping entirely, making the styles global. Inside styles you also get :host (the host element), :host-context (react to an ancestor class), and the deprecated ::ng-deep (pierce into children).

Why it matters: view encapsulation is one of the most-asked Angular interview topics — knowing all three modes, how Emulated works, and the trade-offs of switching is expected of any mid-level candidate.

import { Component, ViewEncapsulation } from '@angular/core';

@Component({
  selector: 'app-badge',
  styleUrl: './badge.css',
  encapsulation: ViewEncapsulation.Emulated, // default; ShadowDom or None to change
})
export class Badge {}

Interpolation and template expressions

Interpolation, {{ expression }}, binds dynamic text into the DOM; the expression is evaluated in the component's context and its result is coerced to a string. Angular template expressions look like JavaScript but are deliberately restricted: no assignments (outside event bindings), no new, no increment/decrement or chained/bitwise operators, and they must be side-effect-free and fast — because they are re-evaluated on every change-detection pass.

Why it matters: knowing what is not allowed (and that expressions run constantly) is what separates a candidate who writes performant templates from one who calls expensive functions in interpolation.

<h1>{{ title }}</h1>
<p>Total: {{ price * qty }}</p>
<p>{{ isAdmin ? 'Admin' : 'Guest' }}</p>

Property, attribute, class and style binding

Property binding [prop]="expr" sets a DOM property or a component input — the workhorse one-way flow from class to view; interpolation is really shorthand for binding to the textContent property. Because some HTML attributes have no matching DOM property (colspan, aria-*), attribute binding [attr.colspan]="n" targets the attribute instead. Class binding [class.active]="isActive" toggles a single class, and style binding [style.width.px]="w" sets a single style with an optional unit; ngClass/ngStyle handle the multi-value cases.

Why it matters: the property-versus-attribute distinction (why [src] is a property but [attr.colspan] is an attribute) is a classic screening question.

<img [src]="avatarUrl">
<button [disabled]="isBusy">Save</button>
<td [attr.colspan]="span">…</td>
<div [class.active]="isActive" [style.color]="themeColor">…</div>

Event binding

Event binding (event)="statement" listens for a DOM event (or a component's output) and runs a template statement in response; Angular exposes the event object as $event. Key-event pseudo-events such as (keyup.enter) filter for you without a manual event.key check. Event bindings are the one place where assignment inside a template expression is permitted — which is what makes two-way binding possible.

Why it matters: property binding in and event binding out is the complete one-way data-flow picture; interviewers often ask you to draw it before discussing two-way binding.

<button (click)="save()">Save</button>
<input (keyup.enter)="submit()">
<input (input)="onInput($event)">

Two-way binding

Two-way binding [(x)]="prop" — informally the "banana in a box" — is simply property binding and event binding combined: [x]="prop" plus (xChange)="prop = $event". On form controls, [(ngModel)]="name" keeps a component field in sync with an input (it needs FormsModule imported). Between components, the modern model() input provides two-way binding without an explicit @Output.

Why it matters: the payoff answer is recognising that the syntax desugars into two one-way bindings — Angular has no special two-way channel, just a naming convention (xChange).

<input [(ngModel)]="name">

<!-- desugars to two one-way bindings: -->
<input [ngModel]="name" (ngModelChange)="name = $event">

Interview questions

Components and binding dominate Angular screening rounds in Indian interviews — view encapsulation and the property-versus-attribute distinction are perennial favourites:

Beginner What are the three parts of an Angular component?

A TypeScript class (state and behaviour), an HTML template (the view), and a CSS selector (how it is used in markup). The @Component decorator ties them together through its metadata.

Beginner What does the selector in @Component do?

It is a CSS selector Angular matches at compile time to decide where the component renders. Usually an element selector like app-user (used as <app-user>), but attribute selectors like [appTooltip] and class selectors are also allowed.

Beginner Interpolation vs property binding — when do you use each?

Interpolation {{ }} injects a value as text into the DOM; property binding [prop]="expr" sets a DOM property or component input to the value itself (not stringified). Use interpolation for display text, property binding when the target needs a real value — e.g. [disabled]="isBusy" or [src]="url".

Beginner How do you handle an event, and how do you access the event object?

Event binding: (click)="save()". Angular provides the DOM event as $event, so (input)="onInput($event)" passes it to your handler. Key pseudo-events like (keyup.enter) filter specific keys without a manual check.

Beginner What is two-way binding and what is its syntax?

Data flowing both ways between a field and the view. Syntax is [(x)]="prop" — the "banana in a box". On forms, [(ngModel)]="name" syncs an input with a component property (requires FormsModule).

Intermediate Property binding vs attribute binding — give a case where you must use attribute binding.

Property binding targets DOM properties; attribute binding targets HTML attributes that have no matching property. You must use [attr.colspan]="n" for a table cell, and [attr.aria-label]="label" for ARIA — there are no colspan/aria-label DOM properties to bind. In contrast [src], [disabled], [value] are properties.

Intermediate What is view encapsulation and what are the three modes?

It controls how a component's styles are scoped. Emulated (default) scopes styles to the component with generated attributes so they do not leak; ShadowDom uses the browser's native Shadow DOM for real isolation; None turns scoping off, making the styles global.

Intermediate How does Emulated encapsulation actually work under the hood?

Angular adds a unique attribute to the component's host (_nghost-xxx) and stamps a matching attribute on every element in its template (_ngcontent-xxx), then rewrites the component's CSS selectors to include that attribute. The styles therefore match only elements carrying the attribute — emulating Shadow DOM scoping without using it.

Intermediate What does [(ngModel)] desugar into, and what is required to use it?

It expands to [ngModel]="value" plus (ngModelChange)="value = $event" — a property binding and an event binding. It needs FormsModule imported (in a standalone component, added to imports).

Intermediate What are standalone components and how do they differ from NgModule-based ones?

A standalone component declares its own template dependencies in the @Component imports array instead of being registered in an NgModule's declarations. It is the default since v19, which simplifies bootstrapping and lazy loading. NgModule-based components required a module to declare and export them.

Intermediate Class/style binding vs ngClass/ngStyle — when to use which?

Use [class.active]="cond" or [style.width.px]="w" to toggle one class or set one style. Use ngClass/ngStyle when applying several classes or styles at once from an object or array. Prefer plain CSS pseudo-classes (:hover, :focus) where they suffice.

Pro What restrictions apply to Angular template expressions, and why?

No assignments outside event bindings, no new, no chained/increment/bitwise operators, and no side effects. The reason: expressions are re-evaluated on every change-detection cycle, so they must be fast and idempotent. Heavy logic belongs in the class or a pure pipe, not in interpolation.

Pro Explain :host, :host-context and ::ng-deep, and why avoid ::ng-deep.

:host styles the component's own host element; :host-context(.theme-dark) styles the host based on an ancestor's class (useful for theming). ::ng-deep pierces encapsulation to reach child-component elements — but it is deprecated because it leaks styles and breaks isolation; prefer CSS custom properties or exposing a styling API instead.

Pro When would you switch a component to ShadowDom or None, and what are the risks?

ShadowDom gives true native isolation — worth it for reusable widgets or design-system elements — but global styles no longer reach inside and styling projected ng-content behaves differently. None makes the component's styles global, which is occasionally used for app-wide overrides but risks unpredictable leakage and collisions across the app. Emulated is the safe default for almost everything.
