Module: Component Communication

Components are only useful when they talk to each other — data flows down, events flow up, and a parent occasionally reaches into a child. This module maps every wire between components, in both the classic decorator style and the new signal APIs.

Knowledge Library → Courses → Angular Basics → Component Communication

What this module covers

Angular's component tree communicates through a small, well-defined set of channels: inputs push data from parent to child, outputs raise events from child to parent, two-way binding combines both, and queries let a component grab references to its children. Each channel now has two forms — the original decorators (@Input, @Output, @ViewChild) and the signal-based functions introduced from v17.1 (input(), output(), model(), viewChild(), contentChild()). Know both: interviews and legacy code lean on the decorators, new projects lean on signals.

Topics

Passing data down: @Input and the input() signal API

A parent hands data to a child by binding to the child's input in the template (<user-avatar [size]="64" />). The classic form declares @Input() size: number; the modern form declares size = input(48), which returns a read-only InputSignal you read as a function (this.size()). Both support required inputs, aliases, and transforms — but the signal version is reactive, so computed() and effect() react to input changes without the ngOnChanges hook.

Why it matters: unidirectional data flow (parent → child via inputs) is the backbone of Angular's mental model, and the decorator-to-signal migration is the single most-asked "what's new in Angular" interview theme right now.

// Child — signal input (v17.1+)
avatarUrl = input.required<string>();   // required, no default
size      = input(48);                   // optional with default
// Child — classic decorator
@Input() avatarUrl!: string;

// Parent template
<user-avatar [avatarUrl]="url" [size]="64" />

Emitting events up: @Output, EventEmitter and the output() function

A child reports something happened by emitting a custom event; the parent listens with event binding (<editor (saved)="persist($event)" />). The classic form is @Output() saved = new EventEmitter<Draft>() and the child calls this.saved.emit(draft). The modern output() function returns an OutputEmitterRef with the same emit() API but no EventEmitter (and no RxJS Subject) underneath — lighter and clearer. Name outputs as things that happened (saved, itemSelected), not commands.

Why it matters: inputs down + outputs up is the "smart vs presentational component" pattern every Angular codebase uses, and the events-vs-commands naming discipline is a favourite senior-round probe.

// Child — modern output() (no EventEmitter)
saved = output<Draft>();
onSave() { this.saved.emit(this.draft); }
// Child — classic decorator
@Output() saved = new EventEmitter<Draft>();

// Parent template
<editor (saved)="persist($event)" />

Two-way binding: the banana-in-a-box and model()

Two-way binding is syntactic sugar: [(value)]="x" expands to a property binding [value]="x" plus an event binding (valueChange)="x = $event" — the [()] shape is nicknamed "banana in a box". On form elements you get it via [(ngModel)] (needs FormsModule). On your own components you provide it with model(): a single writable signal that is simultaneously an input and a matching xChange output, so writing this.value.set(...) in the child updates the parent automatically.

Why it matters: candidates who can expand [(x)] into its two halves — and explain that model() collapses an input plus a Change output into one declaration — signal a real grasp of the binding system rather than rote syntax.

// Child — one line gives input + valueChange output
value = model<string>('');
pick(c: string) { this.value.set(c); }  // parent updates automatically

// Parent template
<color-picker [(value)]="selected" />   // = [value]="selected" (valueChange)="selected=$event"

View queries and template reference variables: @ViewChild / viewChild()

Sometimes a parent needs a direct handle on a child — to call a method or read a DOM node. First name it with a template reference variable (<video #player>), then query it. The classic @ViewChild('player') populates the field by ngAfterViewInit; the signal viewChild('player') returns a signal you can read reactively (with computed/effect) and never touch a lifecycle hook. Use viewChildren() / @ViewChildren for multiple matches, and the read option to pick what you extract (element vs component vs directive).

Why it matters: view queries are the escape hatch from pure data-binding, and "why is my @ViewChild undefined in ngOnInit?" is one of the most common real-world Angular bugs — and interview questions.

// Template
<video #player></video>

// Class — signal query (reactive, no lifecycle hook needed)
player = viewChild<ElementRef<HTMLVideoElement>>('player');
// Class — classic decorator (ready after view init)
@ViewChild('player') playerRef!: ElementRef;
ngAfterViewInit() { this.playerRef.nativeElement.play(); }

Content queries and content projection: @ContentChild / contentChild()

A container component receives markup from its parent through <ng-content> — content projection, Angular's version of slots. That projected content is not part of the component's own view, so @ViewChild can't see it; you query it with @ContentChild / contentChild() (and their *Children plurals). The rule to internalise: view queries find what your template declares, content queries find what a parent projected into you — and no query ever pierces a nested component's boundary.

Why it matters: reusable UI libraries (cards, tabs, accordions) are built on projection plus content queries, and the view-vs-content distinction is a classic trick question that separates people who memorised decorators from people who understand where content lives.

// Wrapper template — projects whatever the parent supplies
<div class="panel"><ng-content></ng-content></div>

// Wrapper class — query the PROJECTED content, not the view
header = contentChild(PanelHeaderComponent);
@ContentChild(PanelHeaderComponent) headerDecorator!: PanelHeaderComponent;

Interview questions

Component communication is a guaranteed Angular round in Indian interviews — from service-company screening to product-company deep dives; the decorator-vs-signal shift makes it hotter than ever:

Beginner How do you pass data from a parent component to a child?

Declare an input on the child (@Input() name or the signal name = input()) and bind to it in the parent's template: <child [name]="value" />. Data flows one way, parent to child.

Beginner How does a child send data back to its parent?

Through an output event. Classic: @Output() saved = new EventEmitter<T>(), then saved.emit(value). Modern: saved = output<T>(). The parent listens with event binding: <child (saved)="handle($event)" />.

Beginner What is a template reference variable?

A name declared in the template with #, e.g. <input #box>. It references that element, component, or directive so you can use it elsewhere in the template (box.value) or target it from a @ViewChild/viewChild() query in the class.

Beginner What does the [(x)] two-way binding syntax mean?

"Banana in a box." [(x)]="v" is shorthand for a property binding plus an event binding: [x]="v" (xChange)="v = $event". It needs an input x and a matching output named xChange — which [(ngModel)] and model() provide.

Intermediate How does input() differ from the @Input decorator?

input() returns a read-only InputSignal read as a function (this.size()) and is reactive — computed/effect respond to changes, replacing ngOnChanges for most cases. @Input() is a plain property set imperatively. Both support required, alias, and transform; the signal one is the recommended default for new code (v17.1+).

Intermediate @ViewChild vs @ContentChild — what's the difference?

@ViewChild queries elements in the component's own template (its view). @ContentChild queries content a parent projected into the component via <ng-content>. Same split for viewChild()/contentChild() and their *Children plurals. Projected content is not in your view, so ViewChild can't find it.

Intermediate Why is a @ViewChild reference undefined in ngOnInit?

The view isn't rendered yet during ngOnInit; decorator view queries are populated just before ngAfterViewInit, so read them there. The signal viewChild() sidesteps this — it returns a signal you can read reactively once the child exists, no lifecycle timing to remember.

Intermediate What does model() do?

model() declares a writable signal that is both an input and an automatically-paired Change output, giving a custom component two-way binding in one line. Writing this.value.set(x) emits valueChange, so a parent using [(value)] stays in sync. It's the signal-era replacement for a manual input + EventEmitter pair.

Intermediate How do you make an input required, alias it, or transform its value?

Signal: input.required<T>(), input(x, { alias: 'name' }), input(0, { transform: numberAttribute }). Decorator: @Input({ required: true }), @Input('alias'), @Input({ transform: booleanAttribute }). Transforms (e.g. booleanAttribute) replace the old getter/setter pattern.

Intermediate EventEmitter vs the output() function — what changed?

output() returns an OutputEmitterRef with the same emit() API but no EventEmitter / RxJS Subject underneath — lighter and purpose-built for component events. EventEmitter still works and, because it extends Subject, was sometimes (mis)used as an observable; output() discourages that. For RxJS interop use outputFromObservable/outputToObservable.

Pro How do signal queries differ from decorator queries in timing and reactivity?

Decorator queries fill a field once (view queries by ngAfterViewInit, content by ngAfterContentInit) and expose multiples as a QueryList. Signal queries (viewChild(), contentChildren()) return signals that update reactively as matching children appear or leave, integrate with computed/effect, and drop the need for QueryList or the After*Init hooks entirely.

Pro @Input/@Output gets painful across many levels — what are the alternatives?

Inputs/outputs suit parent-child. For deep or sibling communication, prefer a shared service (state held in signals or a BehaviorSubject) injected via DI, so unrelated components read/write the same source without prop-drilling. Other tools: DI injection tokens, router state, or a store. The rule: local relationship → inputs/outputs; cross-cutting state → service.

Pro Can a query reach into a nested child component's own template?

No — queries never pierce component boundaries. @ViewChild/viewChild() see your template's elements and direct child components, but not the internals of those children's templates. To reach a grandchild you expose it deliberately (an output, a public method, or a shared service); reaching in directly would break encapsulation and is intentionally impossible.
