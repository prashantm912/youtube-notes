Module: HTTP and Observables Intro

Every real Angular app talks to a backend. This module wires up HttpClient the standalone way and introduces just enough RxJS — the Observable and the async pipe — to consume the responses; the deep RxJS work lives in a later advanced track.

Knowledge Library → Courses → Angular Basics → HTTP and Observables Intro

What this module covers

Angular's HttpClient is the service you use to call REST APIs. You enable it once with provideHttpClient(), inject it, and call get/post with a generic type for typed responses. Those methods return an RxJS Observable, so this module also gives you the reactive minimum: what an Observable is, why nothing happens until you subscribe, and how the async pipe subscribes for you. RxJS is deliberately kept shallow here — operators, subjects and higher-order streams are a whole advanced track of their own.

Topics

HttpClient and provideHttpClient

HttpClient lives in @angular/common/http and is the single service for talking to a server. In the standalone, signals-era app it is turned on with provideHttpClient() in your app.config.ts providers — the functional replacement for the older HttpClientModule. Features are composed in as arguments: withFetch() to use the Fetch API, withInterceptors([...]) to register middleware. You then pull it in with inject(HttpClient) inside a service.

Why it matters: this one-line provider is where most "NullInjectorError: No provider for HttpClient" bugs come from, and knowing the provider-function model (not the legacy module) is what marks you as current on modern Angular.

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [provideHttpClient(withFetch())],
};

// user.service.ts
@Injectable({ providedIn: 'root' })
export class UserService {
  private http = inject(HttpClient);
}

Making requests: GET, POST and typed responses

http.get<T>(url) and http.post<T>(url, body) issue the request and hand back an Observable<T>. The generic parameter is the key habit: get<User[]>(...) types the emitted value so the compiler and template know its shape. An options object carries query params, headers, and switches like observe: 'response' (full response instead of just the body) or responseType. The request is lazy — it does not fire until something subscribes.

Why it matters: interviewers probe whether you understand that returning the Observable from a service (rather than subscribing inside it) is the idiom, and that the generic gives you type safety end to end.

getUsers(): Observable<User[]> {
  return this.http.get<User[]>('/api/users');
}

addUser(u: NewUser): Observable<User> {
  return this.http.post<User>('/api/users', u);
}

RxJS Observable basics

An Observable is a lazy stream that pushes zero or more values over time. Subscribing is what starts the work — until then the producer is idle (this is why an HTTP call never leaves the browser until you subscribe or use the async pipe). An observer supplies up to three callbacks: next for each value, error if it fails, complete when the stream ends. HTTP observables are "cold" and single-shot: they emit one response, then complete. Simple transformations go through pipe() with operators such as map and catchError — but keep this light, the operator zoo is a later track.

Why it matters: "why didn't my HTTP call run?" is almost always a missing subscription, and the Observable-vs-Promise contrast is a guaranteed Indian-interview question.

this.userService.getUsers().subscribe({
  next:  users => this.users = users,
  error: err   => console.error(err),
  complete: () => console.log('done'),
});

The async pipe and interceptors intro

The async pipe unwraps an Observable straight in the template — it subscribes when the view renders and, crucially, unsubscribes when the component is destroyed, so you never leak a subscription. Paired with @if (or the legacy *ngIf … as) it lets you keep the raw Observable on the class and render its latest value reactively, no manual subscribe needed. Interceptors are the other piece worth meeting early: functional middleware registered through withInterceptors([...]) that sees every request and response — the standard home for attaching auth tokens, logging, or retrying. Their deep patterns belong to the advanced track; here, just know they exist and where they plug in.

<!-- template: no manual subscribe, no leak -->
@if (users$ | async; as users) {
  <li *ngFor="let u of users">{{ u.name }}</li>
}

Interview questions

Most-asked HttpClient and Observable questions in Indian interviews — HTTP plus "Observable vs Promise" is a near-certain screening pair for Angular roles:

Beginner What is HttpClient and where does it come from?

It is Angular's service for making HTTP requests, from @angular/common/http. You inject it (inject(HttpClient) or constructor injection) into a service and call get, post, put, patch, delete. Each returns an Observable.

Beginner How do you enable HttpClient in a standalone Angular app?

Add provideHttpClient() to the providers array in app.config.ts (passed to bootstrapApplication). This is the modern replacement for importing HttpClientModule. Forgetting it produces "No provider for HttpClient" at runtime.

Beginner What does http.get() return, and when does the request actually fire?

It returns a cold Observable. The HTTP request is lazy — nothing is sent until something subscribes, either an explicit .subscribe() or the async pipe. No subscriber means no network call.

Beginner How do you get a typed response from HttpClient?

Use the generic type parameter: this.http.get<User[]>('/api/users') yields Observable<User[]>. HttpClient parses the JSON body and TypeScript treats it as that type — it is a compile-time contract, not a runtime validation.

Beginner What is the async pipe and why is it preferred over subscribing manually?

{{ data$ | async }} subscribes to an Observable in the template and returns its latest value. It automatically unsubscribes when the component is destroyed, so it prevents the memory leaks that manual subscriptions cause when you forget to clean up.

Intermediate Observable vs Promise — the differences that matter?

A Promise is eager (runs immediately) and resolves once; an Observable is lazy (runs on subscribe) and can emit many values over time. Observables are cancellable (unsubscribe) and composable through operators; Promises are neither. HttpClient uses Observables so requests can be cancelled and piped.

Intermediate provideHttpClient vs the old HttpClientModule?

provideHttpClient() is a functional provider for standalone apps, with tree-shakable, composable features like withFetch() and withInterceptors([...]). HttpClientModule is the legacy NgModule approach — still works but not how new apps are wired, and interceptors were registered very differently there.

Intermediate If you subscribe manually, how do you avoid a memory leak?

Unsubscribe when the component is destroyed — store the Subscription and call unsubscribe() in ngOnDestroy, or use takeUntilDestroyed(). Simplest of all: prefer the async pipe, which handles it for you. Note a one-shot HTTP observable self-completes, but a still-open subscription is the risky case.

Intermediate How do you transform an HTTP response before it reaches the component?

Pipe operators onto the Observable in the service: this.http.get<Dto>(url).pipe(map(dto => toModel(dto))). catchError handles failures, retry re-attempts. Keeping the mapping in the service gives the component a clean, ready-to-use stream.

Intermediate What is an HTTP interceptor and a common use for one?

Middleware that sits in the request/response pipeline and can inspect or modify every call. Registered with withInterceptors([authInterceptor]). Typical uses: attaching an Authorization header, centralised logging, global error handling, and retrying failed requests.

Pro Why does an HttpClient Observable complete after one emission, and what does that imply for cleanup?

An HTTP request has a single response, so the observable emits once and then completes — it is effectively single-shot and cold. On completion it tears down its own subscription, so leaks from HTTP observables are rare; the real leak risk is long-lived streams (subjects, router or form value changes). Unsubscribing early still matters because it cancels an in-flight request.

Pro In modern Angular, when would you reach for signals instead of Observables for server data?

Observables model the asynchronous stream — the request, cancellation, retries; signals model synchronous reactive state the template reads. A common bridge is toSignal(this.http.get<T>(...)), and Angular's newer resource/httpResource APIs lean this way. Rule of thumb: keep the HTTP call as an Observable, expose the settled value as a signal or via the async pipe. The deeper RxJS-vs-signals trade-offs are covered in the advanced track.
