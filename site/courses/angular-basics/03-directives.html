Module: Directives and Control Flow

Angular v17 turned templates inside out: @if/@for/@switch are now the default, and "why did you replace *ngIf?" is the fastest way an interviewer checks whether you have kept up.

Knowledge Library → Courses → Angular Basics → Directives and Control Flow

What this module covers

Directives are how Angular adds behaviour to the DOM: structural directives change layout by adding or removing elements, attribute directives change appearance or behaviour of an existing element, and components are directives with a template. As of v17 the old structural workhorses *ngIf/*ngFor/*ngSwitch have a built-in replacement — the block control flow @if/@for/@switch — which is now the default the CLI scaffolds. This module orients you on both, on the built-in attribute directives ngClass/ngStyle, and on writing your own directive with host binding.

Topics

Built-in control flow: @if, @for, @switch

Since Angular v17 the template engine understands control flow blocks written with an @ prefix — no directive, no CommonModule import, available everywhere by default. @if takes native @else if/@else branches and an as alias; @for requires a track expression (stable identity, so a one-item change no longer re-renders the whole list) and offers an @empty block plus contextual variables ($index, $first, $last, $count); @switch/@case/@default mirrors JavaScript's switch.

Why it matters: it is a compiler feature, so it lowers runtime overhead and bundle size, gives better type narrowing inside branches, and the Angular team has benchmarked the new @for as dramatically faster than *ngFor. Knowing this is the current baseline for "modern Angular".

@if (user(); as u) {
  <p>Welcome, {{ u.name }}</p>
} @else if (loading()) {
  <p>Loading…</p>
} @else {
  <p>Please sign in</p>
}

@for (item of items(); track item.id) {
  <li>{{ item.label }}</li>
} @empty {
  <li>No items yet</li>
}

Legacy *ngIf/*ngFor/*ngSwitch and migrating to control flow

The structural directives are still valid and worth understanding: the * is shorthand that desugars the host element into an <ng-template>, they must be imported (via CommonModule) into standalone components, and *ngFor's performance depends on a hand-written trackBy function. Because only one structural directive may sit on an element, patterns like combining *ngIf and *ngFor forced an extra <ng-container> — friction the block syntax removes.

Why it matters: the migration is a top current interview probe. Angular ships an automatic schematic — ng generate @angular/core:control-flow — that rewrites a whole codebase; being able to explain what it changes (and the subtle track vs trackBy difference) signals you have shipped modern Angular, not just read about it.

<!-- legacy: needs CommonModule; the * desugars to an ng-template -->
<div *ngIf="user as u; else signIn">{{ u.name }}</div>
<ng-template #signIn>Please sign in</ng-template>

<li *ngFor="let item of items; trackBy: trackById">{{ item.label }}</li>

<!-- one command rewrites the whole app -->
ng generate @angular/core:control-flow

Attribute directives: ngClass and ngStyle

Attribute directives change an existing element instead of adding or removing it. The two you meet constantly are ngClass (toggle CSS classes from an object, array, or string) and ngStyle (set inline styles from an object). Both are imported the same way and are driven by component state, letting styling logic live in the template or a component method rather than in imperative DOM code.

Why it matters: interviewers probe when to reach for them. Prefer native [class.active]/[style.color] bindings for a single toggle; use ngClass/ngStyle when several classes or styles are computed together; and keep genuine ngStyle use to truly dynamic values (a colour picker), leaving static presentation to a stylesheet.

<div [ngClass]="{ active: isActive(), 'is-error': hasError() }">…</div>
<div [ngStyle]="{ color: textColor(), 'font-size.px': size() }">…</div>

<!-- single toggle: native binding is lighter -->
<div [class.active]="isActive()">…</div>

Custom attribute directives and host binding

You build your own attribute directive with the @Directive decorator and a bracketed CSS selector ([appHighlight]). It reads inputs (the signal-based input() in modern Angular) and talks to its host element through the host property in the decorator — modern Angular's preferred way to bind events, properties, classes and styles; the older @HostBinding/@HostListener decorators do the same and remain common in existing code. The directive composition API (hostDirectives) then lets a component or directive apply other directives to its own host, composing behaviour without inheritance.

@Directive({
  selector: '[appHighlight]',
  host: {
    '(mouseenter)': 'onEnter()',
    '(mouseleave)': 'onLeave()',
    '[style.background]': 'bg',
  },
})
export class HighlightDirective {
  color = input('yellow');       // used as [appHighlight]="'#ffd'"
  protected bg = '';
  onEnter() { this.bg = this.color(); }
  onLeave() { this.bg = ''; }
}

Interview questions

Directives and the new control flow dominate current Angular screening rounds in India — the *ngIf-to-@if migration is the fastest "are you current?" check an interviewer has:

Beginner What are the three kinds of directives in Angular?

Components (a directive with a template), attribute directives (change the appearance or behaviour of an existing element, e.g. ngClass), and structural directives (change layout by adding/removing elements, e.g. *ngFor). The new @if/@for/@switch are built-in template blocks, not directives.

Beginner What is the new built-in control flow, and why was it introduced?

The @if/@for/@switch block syntax added in v17 (now default). It needs no CommonModule import, reads closer to JavaScript, and — being a compiler feature — reduces runtime overhead and bundle size while improving type narrowing. It replaces the microsyntax of the structural directives.

Beginner How does @if differ from *ngIf?

@if is built into the template (no import), supports @else if/@else natively, and gives better type narrowing. *ngIf is a structural directive that must be imported, desugars to an <ng-template>, and expresses "else" through a template reference. Same effect, cleaner ergonomics.

Beginner What do ngClass and ngStyle do, and when would you use a plain binding instead?

ngClass toggles multiple CSS classes (object/array/string), ngStyle sets multiple inline styles from an object. For a single class or style, the native [class.active]/[style.color] binding is lighter and clearer; reach for the directives when several values are computed together.

Beginner What is the difference between an attribute directive and a structural directive?

An attribute directive modifies an element that stays in the DOM (styling, behaviour). A structural directive adds or removes elements, changing the DOM's shape — it is applied to an <ng-template>, which is what the * shorthand creates.

Intermediate Why is track mandatory in @for, and how does it relate to trackBy?

track tells Angular how to identify each item so that changing one row updates just that DOM node instead of re-rendering the list — the same job *ngFor's optional trackBy did. The team made it mandatory because a missing trackBy was a common, silent performance bug; track item.id is an expression, not a function.

Intermediate What is the @empty block, and how did you show an empty state before?

@empty renders when the @for collection has no items — a first-class empty state. Pre-v17 you added a separate *ngIf="items.length === 0" (or an @else-style template), which was easy to forget and duplicated the emptiness check.

Intermediate How do you write a custom attribute directive?

Annotate a class with @Directive and a bracketed selector ([appHighlight]), declare inputs with input(), and interact with the host element via the host property (events, property/class/style bindings) or by injecting ElementRef/Renderer2. Add it to the consuming component's imports and apply it as an attribute.

Intermediate @HostBinding vs @HostListener — and the modern alternative?

@HostBinding binds a property/attribute/class/style on the host element; @HostListener subscribes to an event on it. Modern Angular prefers declaring both in the decorator's host property ('[style.color]': 'c', '(click)': 'onClick()'); the docs keep the decorators mainly for backwards compatibility.

Intermediate How do you migrate an existing app from *ngIf/*ngFor to the new control flow?

Run the official schematic ng generate @angular/core:control-flow. It rewrites *ngIf/*ngFor/*ngSwitch to blocks across the codebase and can drop now-unused CommonModule imports. Review the diff — nested async-pipe conditionals and trackBy functions are where hand-tidying usually helps.

Pro How does the * of *ngIf actually work under the hood?

It is syntactic sugar: <div *ngIf="cond">…</div> desugars to <ng-template [ngIf]="cond"><div>…</div></ng-template>. The directive receives a TemplateRef and a ViewContainerRef and creates or clears the embedded view. Because only one structural directive can own an element, stacking *ngIf and *ngFor is illegal without an extra <ng-container>.

Pro What problem does the directive composition API (hostDirectives) solve?

It lets a component or directive apply other directives to its own host element, declared statically in hostDirectives, and selectively expose their inputs/outputs. This composes cross-cutting behaviour (a11y roles, tooltips, CDK behaviours) without inheritance or wrapper elements — reuse by composition rather than a base class.

Pro A tracked property changes but the object reference stays the same — how does @for behave, and how did *ngFor differ?

With @for, if track yields the same key but a bound value changed, Angular updates that view's bindings in place rather than destroying and recreating the element. Classic *ngFor with trackBy keyed the identity similarly but the diffing semantics around reuse were less explicit — a subtle behavioural detail the migration guide calls out. Choose a genuinely stable, unique track key (an id, not $index) or you reintroduce the very re-render churn tracking exists to prevent.
