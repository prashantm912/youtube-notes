Module: The JVM

Every .class you run is loaded, verified, and eventually compiled to native code by a machine most Java developers never look inside — and the ones who do tend to clear the harder interview rounds.

Knowledge Library → Courses → Core Java → The JVM

What this module covers

The JVM is the abstract computer that loads, verifies, and executes your bytecode — its class loader subsystem, run-time memory areas, and execution engine (interpreter plus JIT) are the machinery behind "write once, run anywhere". This page orients you on how a class travels from a .class file to optimized native code, where objects and metadata actually live, and the decade of language features that rode on top. Class loading and JIT compilation are two of the most-asked JVM topics in Indian interviews, so both get extra weight below.

Topics

JVM architecture, class loading, bytecode and the Class-File API, StackWalker

A JVM has three moving parts: the class loader subsystem (finds, links, and initializes types), the run-time data areas (where classes, objects, and stacks live), and the execution engine (interpreter, JIT compiler, garbage collector). Class loading itself is three phases — loading (read the bytes, build a Class object), linking (verify, prepare static fields, resolve symbolic references), and initialization (run <clinit>) — driven by loaders arranged bootstrap → platform → application under the parent-delegation model.

Why it matters: class identity is name plus defining loader, which is the root of both app-server isolation and the notorious "same class, ClassCastException" bug. The Class-File API (finalized in Java 24) is now the standard, ASM-free way to read and generate bytecode, and StackWalker is the modern, lazy replacement for scraping stack traces — both show up when interviewers probe past the surface.

// StackWalker: walk only the frames you need, lazily — no full trace materialized
Class<?> caller = StackWalker.getInstance(Option.RETAIN_CLASS_REFERENCE)
                             .getCallerClass();

// The delegation chain: application -> platform -> bootstrap (null)
ClassLoader cl = MyApp.class.getClassLoader(); // application / system loader
cl.getParent();                                // platform loader
cl.getParent().getParent();                    // null == bootstrap loader

JIT and tiered compilation; memory areas: heap, stack, metaspace, code cache

HotSpot starts by interpreting bytecode for fast startup, profiles which methods run hot, and hands those to the JIT for native compilation. Tiered compilation (default since Java 8) blends two compilers across levels 0–4: C1 (the client compiler — quick to compile, lightly optimized, gathers profiling data) then C2 (the server compiler — slower to compile, aggressively optimized) once a method is proven hot. This is the "warm-up" every Java benchmark has to reckon with.

Why it matters: knowing where memory lives explains most production incidents. The heap and method area are shared across threads; each thread gets its own stack, pc register, and native method stack. Metaspace (native memory, replacing Java 7's PermGen) holds class metadata, and the JIT's compiled code lives in the fixed-size code cache — when it fills, compilation stops and throughput quietly falls off a cliff.

# See what the JIT is doing and bound where memory goes
java -XX:+PrintCompilation           Main   # log each method as it is JIT-compiled
java -Xss1m -Xmx512m                  Main   # per-thread stack size; max heap
java -XX:ReservedCodeCacheSize=256m  Main   # cap the JIT code cache
java -XX:MaxMetaspaceSize=256m       Main   # bound class-metadata (native) memory

JEP timeline tour: Java 8 to the current release

The JVM is a stable target, but the language on top of it has moved fast since the six-month cadence began in 2017. This is the "what changed and when" narrative interviewers use to gauge how current you are: Java 8 (lambdas, Streams, Optional), Java 9 (the module system, JShell), Java 10/11 LTS (var, the standard HTTP client), Java 17 LTS (sealed classes, records, text blocks, switch pattern matching), Java 21 LTS (virtual threads, record patterns, sequenced collections), and recent releases adding stream gatherers and the Class-File API.

Why it matters: features arrive as JEPs and usually spend a release or two in preview before finalizing, so "is it GA yet?" is a real question. Being able to trace records and sealed classes through Project Amber, or virtual threads through Project Loom, signals you follow the platform rather than a single frozen version.

// A decade in one screen: data-oriented Java the JVM now runs directly
sealed interface Shape permits Circle, Square {}
record Circle(double r) implements Shape {}
record Square(double s) implements Shape {}

double area = switch (shape) {                 // pattern matching for switch (Java 21)
    case Circle c -> Math.PI * c.r() * c.r();
    case Square s -> s.s() * s.s();
};

Interview questions

Most-asked JVM questions in Indian interviews — class loading and JIT dominate, from screening rounds to product-company deep dives:

Beginner What is the difference between JDK, JRE, and JVM?

The JVM is the abstract runtime that executes bytecode. The JRE is the JVM plus the core class libraries needed to run Java apps. The JDK is the JRE plus development tools (javac, javadoc, jar, jdb). Since Java 11 there is no standalone JRE download — you ship a JDK or a trimmed runtime image built with jlink.

Beginner What are the three main subsystems of the JVM?

The class loader subsystem (loads, links, and initializes types), the run-time data areas (heap, method area, JVM stacks, pc registers, native method stacks), and the execution engine (interpreter, JIT compiler, and garbage collector).

Beginner What are the three phases the class loader performs?

Loading (read the .class bytes and create the Class object in the method area), linking (verification of the bytecode, preparation of static fields to default values, and resolution of symbolic references), and initialization (execute the static initializers and static-field assignments via <clinit>).

Beginner Which JVM memory areas are shared across threads, and which are per-thread?

Shared: the heap (all objects and arrays) and the method area (class metadata and the run-time constant pool). Per-thread: the JVM stack (frames with local variables and operand stacks), the pc register, and the native method stack. That split is exactly why local variables are inherently thread-safe while heap state needs synchronization.

Intermediate Explain the class loader hierarchy and the parent-delegation model.

Three loaders: bootstrap (loads core java.* modules, native, null parent), platform (formerly the extension loader), and application/system (your classpath and app modules). Under parent delegation, a loader first asks its parent to load a class and only loads it itself if the parent cannot. This guarantees core classes are loaded once from the trusted source and cannot be shadowed by user code.

Intermediate Interpreter vs JIT compiler — why does the JVM use both?

The interpreter executes bytecode immediately: no warm-up, fast startup, but slower per instruction. The JIT compiles frequently executed ("hot") methods to native code: expensive to warm up, but fast at steady state. HotSpot interprets first, profiles, then compiles the hot spots — getting startup speed and peak throughput from the same run.

Intermediate What is tiered compilation (C1 vs C2)?

A scheme (default since Java 8) that combines two JIT compilers across levels 0–4. C1, the client compiler, compiles quickly with light optimization and collects profiling data; C2, the server compiler, recompiles proven-hot methods with aggressive optimization. Code flows interpreter → C1-with-profiling → C2, balancing fast startup against maximum long-run throughput.

Intermediate What replaced PermGen in Java 8, and why?

Metaspace. PermGen was a fixed-size region inside the heap that stored class metadata and routinely caused OutOfMemoryError: PermGen space. Metaspace moves class metadata into native memory and grows automatically (capped by -XX:MaxMetaspaceSize), which removed a common tuning headache and simplified garbage collection.

Intermediate What is the JIT code cache, and what happens when it fills up?

A fixed-size region of native memory that holds the machine code the JIT produces (sized by -XX:ReservedCodeCacheSize). If it fills, the JVM stops compiling — it may flush cold methods, but new hot methods fall back to the interpreter, so throughput drops silently. A "CodeCache is full. Compiler has been disabled" warning is the tell.

Intermediate Roughly what landed from Java 8 through the latest LTS?

Java 8: lambdas, the Stream API, Optional, default methods, the new date/time API. Java 9: the module system and JShell. Java 10/11: var and the standard HTTP client. Java 17: sealed classes, records (finalized in 16), text blocks, switch pattern matching. Java 21: virtual threads, record patterns, sequenced collections. Recent releases add stream gatherers and the Class-File API. Cadence is six months, with an LTS every two to three years.

Pro What actually triggers class initialization (when does <clinit> run)?

Lazily, on first active use: creating an instance with new, invoking or accessing a static method or non-constant static field, reflection, initializing a subclass, or launching the class that holds main. Reading a compile-time constant (a static final bound to a constant expression) does not trigger it — the value is inlined by the compiler. The JVM guarantees initialization happens exactly once, thread-safely.

Pro What does bytecode verification check, and why is it security-critical?

The verifier, part of linking, proves a .class is structurally valid and type-safe before any of it executes: the operand stack never overflows or underflows, operand types match each instruction, branches land on valid instructions, and local variables are used consistently. It is what lets the JVM run untrusted bytecode safely; disabling it (the deprecated -Xverify:none) removes that guarantee.

Pro Can two classes with the same fully-qualified name coexist in one JVM?

Yes. A class's run-time identity is its fully-qualified name plus its defining class loader, not the name alone. Two loaders can each define com.acme.Foo, yielding two distinct Class objects that are not type-compatible: assigning an instance of one to a reference typed by the other throws ClassCastException, and mixing them can raise LinkageError. App servers exploit this for application isolation and hot redeploy — and it is the usual cause of the baffling "ClassCastException: Foo cannot be cast to Foo".
