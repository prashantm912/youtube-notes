Module: Collections — List and Set

Pick the wrong List or Set implementation and the code still works — until a profiler shows why it crawls. This module maps the collection hierarchy and points you at the sources that settle the ArrayList-versus-LinkedList question interviewers keep asking.

What this module covers

The Collections Framework is a small set of interfaces (List, Set, Queue, Map) with interchangeable implementations — so you program to the interface and swap the concrete class as needs change. This page orients you on the hierarchy, the two List implementations and their internals, the three common Set implementations, and the fail-fast versus fail-safe iterator distinction. It stops at "what and why"; the linked official docs carry the depth. Map, Queue and Comparator continue in the next module.

Topics

Collection hierarchy incl. SequencedCollection/SequencedSet

Everything descends from Iterable → Collection, which branches into List (ordered, indexed, duplicates allowed), Set (uniqueness, extended by SortedSet → NavigableSet), and Queue/Deque. Map sits in a parallel hierarchy and is deliberately not a Collection. Because concrete classes only implement these interfaces, a variable typed as List<T> can hold an ArrayList today and a LinkedList tomorrow with no caller changes — the whole point of the framework.

Why it matters now: Java 21 inserted two interfaces, SequencedCollection and SequencedSet, to give any ordered collection a uniform vocabulary for its ends — getFirst, getLast, addFirst, addLast, removeFirst, removeLast, and a reversed() view. Before this, List reached its ends by index while Deque used its own methods and a plain Set had no notion of "first"; the sequenced interfaces retrofit one consistent API onto List, Deque, LinkedHashSet and SortedSet.

SequencedCollection<String> c = new ArrayList<>(List.of("a", "b", "c"));
c.getFirst();        // "a"  (replaces c.get(0))
c.getLast();         // "c"
c.addFirst("z");     // prepend without index math
c.reversed();        // [c, b, a, z] — a view, not a copy

List (ArrayList/LinkedList internals)

List is the ordered, index-addressable, duplicate-tolerating collection, and it ships with two implementations whose internals decide performance. ArrayList is a resizable Object[]: get/set by index are O(1), appending is amortized O(1) but occasionally copies the whole array into a larger one (grows by roughly half), and inserting or removing in the middle is O(n) because later elements shift. Contiguous memory also makes it cache-friendly.

LinkedList is a doubly-linked chain of nodes implementing both List and Deque: O(1) insert/remove at either end or at an iterator's cursor, but O(n) random access because get(i) walks from the nearer end. Each node also carries two pointers of overhead. The interview reality — and the single most-asked Indian collections question — is that ArrayList is the right default almost every time; LinkedList earns its place mainly as a queue/deque, not for "lots of inserts".

List<Integer> a = new ArrayList<>();   // Object[] backing
List<Integer> l = new LinkedList<>();  // node chain, also a Deque
a.get(1000);   // constant time — direct index
l.get(1000);   // O(n) — walks ~1000 nodes to reach it

Set (HashSet/LinkedHashSet/TreeSet), fail-fast vs fail-safe

Set models uniqueness — no duplicates, decided by equals/hashCode. HashSet is backed by a HashMap: O(1) add/contains, no iteration order, one null permitted. LinkedHashSet extends it with a linked list threading entries in insertion order — predictable iteration for a small memory cost. TreeSet is a red-black tree (a TreeMap) implementing NavigableSet: elements stay in sorted order with O(log n) operations and range queries (floor, ceiling, headSet), but it forbids null and requires Comparable elements or a Comparator.

The second half of this topic is iterator behaviour. Fail-fast iterators — the ones returned by ArrayList, HashSet and HashMap — track an internal modCount and throw ConcurrentModificationException the instant the structure changes mid-iteration, including the single-threaded classic of removing inside a for-each loop. Fail-safe (more precisely, weakly consistent) iterators on concurrent collections such as CopyOnWriteArraySet and ConcurrentHashMap iterate over a snapshot and never throw, trading the guarantee of freshness and some copy cost for that safety.

Set<String> hs = new HashSet<>();       // no order, O(1)
Set<String> ls = new LinkedHashSet<>();  // insertion order preserved
Set<String> ts = new TreeSet<>();        // sorted, O(log n)

for (String s : hs) hs.remove(s);        // throws ConcurrentModificationException
hs.removeIf(s -> s.isBlank());           // the safe way to remove while iterating

Interview questions

Collections is the most-mined topic in Indian Java interviews — from service-company screens to product-company deep dives. ArrayList vs LinkedList and fail-fast iterators come up almost every time:

Beginner Array vs ArrayList — what is the difference?

An array is fixed-size and can hold primitives or objects; an ArrayList is a growable, object-only wrapper over an internal array with rich API (add, remove, contains). Arrays use [] and length; ArrayList uses methods and size(). Arrays are slightly faster and memory-lean; ArrayList trades that for resizing and convenience.

Beginner List vs Set — the core difference?

List is ordered and index-addressable and allows duplicates; Set forbids duplicates (by equals/hashCode) and, except for LinkedHashSet/TreeSet, makes no ordering promise. Use List when position and repeats matter, Set when membership/uniqueness matters.

Beginner How does an ArrayList grow, and what is its default capacity?

It starts empty and allocates an internal array of capacity 10 on the first add. When full, it creates a larger array (about 1.5x) and copies elements over — which is why appending is amortized O(1), not strictly O(1). ensureCapacity pre-sizes to avoid repeated copies.

Beginner HashSet vs LinkedHashSet vs TreeSet — when do you use each?

HashSet: fastest, no order — the default. LinkedHashSet: keeps insertion order for predictable iteration, slightly more memory. TreeSet: keeps elements sorted with range queries (floor/ceiling), at O(log n) and no nulls. Pick on the ordering you need.

Intermediate ArrayList vs LinkedList — internals, and when is LinkedList actually better?

ArrayList is a resizable array: O(1) indexed get/set, amortized O(1) append, O(n) middle insert/remove (shifting), cache-friendly. LinkedList is a doubly-linked list: O(1) add/remove at the ends or at an iterator cursor, but O(n) get(i) and higher per-element overhead. In practice ArrayList wins almost always; LinkedList is justified mainly when you need a Deque/queue with heavy end operations — rarely for "many inserts", because you still pay O(n) to reach the insert point.

Intermediate How does HashSet guarantee uniqueness?

It is backed by a HashMap — each element is a key mapped to a constant dummy value. Adding computes the element's hashCode() to find a bucket, then uses equals() to check for a match; a duplicate is rejected. This is why correct equals/hashCode on your element type is mandatory for a Set to behave.

Intermediate What is a fail-fast iterator, and what triggers ConcurrentModificationException?

Fail-fast iterators (ArrayList, HashSet, HashMap) record a modCount when created and compare it on each next(). If the collection was structurally modified through anything other than the iterator's own remove() — including a single-threaded remove inside a for-each — the mismatch throws ConcurrentModificationException. It is a best-effort bug detector, not a concurrency guarantee.

Intermediate Fail-fast vs fail-safe iterators?

Fail-fast throws immediately on concurrent structural modification (iterating the live structure). Fail-safe (weakly consistent) iterators on concurrent collections like CopyOnWriteArrayList and ConcurrentHashMap iterate a copy/snapshot and never throw, but may not reflect changes made after the iterator was obtained and cost extra memory. "Fail-safe" is informal; the JDK term is weakly consistent.

Intermediate You must remove elements from a list while iterating — how, safely?

Do not call list.remove() inside a for-each — that throws ConcurrentModificationException. Use Iterator.remove(), or better collection.removeIf(predicate) (Java 8+). Alternatively iterate a copy, or use a concurrent collection when true concurrency is involved.

Intermediate What are the load factor and initial capacity of a HashSet, and why 0.75?

Defaults inherited from HashMap: initial capacity 16, load factor 0.75 — when size exceeds capacity × 0.75 the table doubles and rehashes. 0.75 is the empirical sweet spot balancing space against collision probability; lower wastes memory, higher increases collisions and lookup cost.

Intermediate What did SequencedCollection and SequencedSet add in Java 21?

A uniform way to work with an ordered collection's ends: getFirst, getLast, addFirst, addLast, removeFirst, removeLast, and a reversed() view. They slot above List, Deque, LinkedHashSet and SortedSet so all of them share one first/last/reverse API instead of each having its own idiom.

Pro How does TreeSet order elements, and what if compareTo is inconsistent with equals?

TreeSet orders by natural ordering (Comparable.compareTo) or a supplied Comparator, using a red-black tree. Uniqueness is decided by the comparison returning 0 — not by equals. So if compareTo is inconsistent with equals, two objects that are unequal but compare equal will be treated as duplicates (one is dropped), breaking the general Set contract. Keep the comparison consistent with equals.

Pro Give the time complexity of get, add, remove and contains for ArrayList, LinkedList, HashSet and TreeSet.

ArrayList: get O(1), add(end) amortized O(1), remove/insert(middle) O(n), contains O(n). LinkedList: get O(n), add/remove at ends O(1), contains O(n). HashSet: add/remove/contains O(1) average (O(n) worst case before treeification). TreeSet: add/remove/contains O(log n), and it also supports ordered/range operations the others cannot.

Pro Is a "fail-safe" iterator truly safe? Explain weakly consistent iteration and its trade-offs.

Not fully — "safe" only means it will not throw. CopyOnWriteArrayList iterates an immutable snapshot taken at iterator creation, so later writes are invisible and every mutation copies the whole array (great for read-heavy, tiny-write cases). ConcurrentHashMap's iterators are weakly consistent: they traverse without throwing and reflect some but not necessarily all concurrent updates. The cost is possible staleness and, for copy-on-write, memory churn — you trade a strict view for non-blocking iteration.
