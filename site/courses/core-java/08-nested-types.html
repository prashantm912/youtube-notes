Module: Nested Types and Lambdas

Four ways to nest a class, and one way to skip the ceremony with a lambda — the this-scoping and captured-variable rules here are exactly what Indian interviewers use to tell a junior answer from a senior one.

Knowledge Library → Courses → Core Java → Nested Types and Lambdas

What this module covers

Java lets a class be declared inside another, in four flavours — static nested, inner (non-static), local, and anonymous — each with different access to the enclosing instance. Lambdas (Java 8+) then compress the most common of these, the single-method anonymous class, into one arrow expression. This module orients you on when each form applies and on the differences interviewers probe most: the meaning of this, the effectively-final capture rule, and how a lambda compiles differently from an anonymous class.

Topics

Inner, static nested, local, anonymous classes

Nesting a class exists to group types that only make sense together and to tighten encapsulation. A static nested class is namespaced under the outer type but carries no link to an instance — it behaves like a top-level class (think Map.Entry or a Builder). An inner class (non-static) is bound to an enclosing instance, holds an implicit reference to it (Outer.this), and can read that object's private members; you instantiate it through an outer object with outer.new Inner(). A local class is a named class declared inside a method or block, visible only there. An anonymous class is an unnamed local class defined and instantiated in a single expression, implementing an interface or extending a class on the spot.

Why it matters: the implicit outer reference of inner and anonymous classes is both their power (access to enclosing state) and their trap — it keeps the outer instance alive, which is why the docs discourage serializing them and why "why did this leak?" questions point here. Knowing which nesting needs an enclosing instance, and which does not, is a standard screening distinction.

class Outer {
  static class Nested { }                 // static — independent of an instance
  class Inner { int v() { return x; } }   // non-static — bound to Outer.this
  private int x = 1;
}
Outer.Nested n = new Outer.Nested();      // no outer instance needed
Outer.Inner  i = new Outer().new Inner(); // outer instance required first

Lambdas vs anonymous classes

A lambda is a concise literal for an instance of a functional interface — an interface with exactly one abstract method. Before Java 8 you passed behaviour by writing an anonymous class implementing such an interface, all boilerplate around one method body; a lambda expresses the same intent as one arrow expression, with the compiler inferring the target type from context. They are not interchangeable, though, and the gaps are the interview material: inside an anonymous class this refers to the anonymous instance, but inside a lambda this refers to the enclosing instance; an anonymous class can declare fields and several methods (it can hold state), a lambda cannot; a lambda targets only a functional interface, while an anonymous class can extend a class or implement a multi-method interface.

Why it matters: "lambda vs anonymous class" is the canonical Java 8 question, and the follow-ups — what this binds to, why captured locals must be effectively final, and why a lambda emits an invokedynamic instruction instead of a separate Outer$1.class — are what separate a memorised answer from an understood one.

// same behaviour, two forms
Runnable a = new Runnable() {                   // anonymous class
    public void run() { System.out.println("run"); }
};
Runnable b = () -> System.out.println("run");   // lambda — type inferred from Runnable

Interview questions

Most-asked nested-class and lambda questions in Indian interviews — Java 8 features and the anonymous-vs-lambda distinction remain a screening staple:

Beginner What are the four kinds of nested classes in Java?

Static nested class, inner (non-static) class, local class, and anonymous class. The split is static vs non-static: a static nested class has no link to an enclosing instance; the other three are inner classes tied to an instance, with local and anonymous being inner classes scoped to a method or block.

Beginner Static nested class vs inner (non-static) class — the difference?

A static nested class behaves like a top-level class namespaced under the outer; it has no reference to an outer instance and cannot see the outer's instance fields. An inner class holds an implicit Outer.this reference, can read the outer's private instance members, and requires an outer instance to exist.

Beginner How do you instantiate an inner class from outside the outer class?

Through an outer instance: Outer.Inner i = new Outer().new Inner();. A static nested class needs no instance: Outer.Nested n = new Outer.Nested();. The awkward outer.new Inner() syntax is itself a common trick question.

Beginner What is an anonymous class and when would you use one?

An unnamed class defined and instantiated in a single expression, implementing an interface or extending a class on the spot. Use it for a one-off implementation you need exactly once — an event listener, a comparator, a small callback — where naming a full class would be noise.

Beginner What is a lambda expression and what does it require?

A concise literal that implements the single abstract method of a functional interface. It requires a functional interface as its target type (exactly one abstract method); the compiler infers that type from the assignment or method parameter, so the same lambda means different things in different contexts.

Intermediate Lambda vs anonymous class — the key differences?

Four that matter: (1) this inside a lambda is the enclosing instance, inside an anonymous class it is the anonymous instance; (2) an anonymous class can declare fields and multiple methods (state), a lambda cannot; (3) a lambda targets only a functional interface, an anonymous class can extend a class or implement a multi-method interface; (4) a lambda compiles to invokedynamic with no extra class file, an anonymous class generates a separate Outer$1.class. This is the canonical Java 8 question.

Intermediate What does this refer to inside a lambda versus inside an anonymous class?

In a lambda, this is the enclosing instance — a lambda has no identity of its own. In an anonymous class, this is the anonymous object itself, so reaching the enclosing instance needs Outer.this. This difference regularly breaks code that is mechanically converted between the two forms.

Intermediate Why must local variables captured by a lambda or anonymous class be final or effectively final?

They capture the value, not the variable. Local variables live on the stack and can vanish before the lambda/anonymous instance runs, so Java copies the value in; allowing later mutation would make the two copies diverge and invite data races. "Effectively final" means the compiler sees you never reassign it, so the final keyword is not required.

Intermediate Can a lambda replace any anonymous class? When can it not?

No. A lambda only works when the target is a functional interface — one abstract method. It cannot replace an anonymous class that extends a concrete or abstract class, implements an interface with multiple abstract methods, needs instance fields/state, or relies on this meaning the instance itself. For a single-method interface, the lambda is the cleaner form.

Intermediate What is a local class and how does it differ from an anonymous class?

A local class is a named class declared inside a method or block; an anonymous class is the same idea without a name, defined and instantiated in one expression. Choose a local class when you need more than one instance, a constructor, or a named type to reference; choose anonymous when it is a single throwaway implementation.

Pro How are lambdas compiled differently from anonymous classes at the bytecode level?

An anonymous class produces a separate class file (Outer$1.class) at compile time, loaded like any class. A lambda emits an invokedynamic instruction whose bootstrap (LambdaMetafactory) builds the implementation lazily at first use — no per-lambda class file baked in, and stateless lambdas can be cached. That is why heavy anonymous-class use bloats class count while lambdas do not.

Pro Why can an inner class cause a memory leak, and how does that relate to the serialization warning?

A non-static inner (or anonymous/local) class holds an implicit strong reference to its enclosing instance. If the inner instance outlives the outer's intended lifetime — stored in a long-lived cache, registered as a listener — it keeps the whole outer object reachable, so it is never collected. Fix: make it a static nested class and pass in only what it needs. Serialization is discouraged for the same reason: serializing the inner class drags in the outer instance plus compiler-synthetic fields, which is brittle and non-portable across compilers.
