Module: Concurrency — Threads and Memory Model

Threads are easy to start and hard to reason about — the difficulty lives in the memory model, not the API. This module orients you on how threads run and, more importantly, on the visibility and ordering rules (synchronized, volatile, happens-before) that decide whether one thread ever sees another's writes, then points you to the authoritative sources.

What this module covers

This is the foundation of Java concurrency: creating and controlling threads with Thread, Runnable and Callable; the synchronized keyword for mutual exclusion; the volatile keyword for visibility; the Java Memory Model (JMM) and its happens-before relationship — the rulebook that actually governs what a thread can see; wait/notify for coordination; and ThreadLocal for per-thread state. The memory-model trio — synchronized, volatile, and happens-before — is among the most-asked topics in Indian interviews, so it carries the most weight below. Higher-level tools (executors, futures, locks, atomics) build directly on these primitives and are covered in the next modules.

Topics

Thread lifecycle/API, Runnable/Callable

A Thread is Java's unit of independent execution, each with its own call stack. You almost never subclass Thread; instead you pass the work as a Runnable (a no-argument, no-result, no-checked-exception task) and call start() — which spawns a new thread and invokes run() on it. Calling run() directly is the classic mistake: it just executes on the current thread with no concurrency. Over its life a thread moves through the six states of the Thread.State enum — NEW, RUNNABLE, BLOCKED (waiting for a monitor lock), WAITING, TIMED_WAITING, and TERMINATED — and useful control methods include join() (wait for another thread to finish), interrupt() (the cooperative cancellation signal), and setDaemon(true) (a background thread that does not keep the JVM alive).

Why it matters: Callable<V> is Runnable's result-returning cousin — it returns a value and may throw checked exceptions — but it can only be run through an ExecutorService (next module), not by a raw Thread. Interviewers use "Runnable vs Callable" and "start() vs run()" as reliable screening questions, and knowing the state machine lets you read thread dumps. Modern Java adds lightweight virtual threads (Thread.ofVirtual(), Java 21) for massive I/O concurrency, covered later in the track.

Runnable task = () -> System.out.println("run on " + Thread.currentThread().getName());
Thread t = new Thread(task);
t.start();                 // NEW -> RUNNABLE; runs task on a new call stack (NOT run())
t.join();                  // caller blocks until t reaches TERMINATED

// Callable returns a value and may throw checked exceptions
Callable<Integer> job = () -> 40 + 2;   // submit to an ExecutorService to get a Future

synchronized, volatile, Java Memory Model (happens-before), wait/notify

This is the heart of the module. Without synchronization, the JVM and CPU are free to cache values in registers and reorder instructions, so a write by one thread may never become visible to another — that is a memory consistency error, and it is silent. Two keywords fix it. synchronized acquires an object's intrinsic lock (monitor): it gives both mutual exclusion (one thread in the critical section at a time) and visibility (an unlock happens-before the next lock on the same monitor, flushing changes). volatile is lighter — it guarantees visibility and ordering for a single field (every read sees the latest write, and it forbids reordering around the access) but provides no atomicity, so volatile int count; count++ is still a race because the increment is read-modify-write.

The rules that make all of this precise are the Java Memory Model's happens-before relationship (JLS Chapter 17): if action A happens-before action B, then A's memory effects are visible to B. The guarantees you must know: program order within one thread; monitor unlock → subsequent lock; volatile write → subsequent volatile read; Thread.start() → the thread's first action; a thread's last action → another thread's successful join(); and transitivity. Separately, wait()/notify()/notifyAll() (on Object) coordinate threads: they must be called while holding the object's monitor, and wait() must always sit inside a while loop re-checking the condition (guarding against spurious wakeups and lost signals). This cluster — visibility vs atomicity, what happens-before guarantees, why volatile is not a lock — is the single most-tested concurrency area in Indian interviews.

class Counter {
    private int count = 0;
    synchronized void inc() { count++; }     // mutual exclusion + visibility
    synchronized int get() { return count; } // must read under the same lock
}

private volatile boolean running = true;     // visibility only, no atomicity
void stop() { running = false; }             // other threads see it promptly
while (running) { /* work */ }               // safe stop flag; count++ would NOT be safe

ThreadLocal

ThreadLocal<T> gives every thread its own independently initialized copy of a variable: calls to get()/set() read and write only that thread's copy, so the value is confined to a thread and needs no synchronization. The idiomatic pattern is a private static final field created with ThreadLocal.withInitial(supplier). Classic uses are holding non-thread-safe helpers per thread (the textbook example is a SimpleDateFormat, which is not thread-safe) and carrying implicit context — a user/security principal, a transaction, or a trace/correlation ID — through a call chain without threading it through every method signature.

Why it matters: the danger is memory leaks in thread pools. Pooled threads are reused and never die, so a value you set() lives on that thread indefinitely — leaking memory and, worse, leaking one request's data into the next request that reuses the thread. The rule interviewers want to hear: always remove() in a finally block when the unit of work ends. InheritableThreadLocal passes a value to child threads at creation, though it does not propagate across a thread pool.

private static final ThreadLocal<SimpleDateFormat> FMT =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

try {
    String today = FMT.get().format(new Date());  // each thread its own formatter
    // ... use it ...
} finally {
    FMT.remove();   // essential on pooled threads to avoid leaks / stale data
}

Interview questions

The most-asked threads and memory-model questions in Indian interviews — the volatile, synchronized and happens-before cluster dominates, so those lead the intermediate tier and the hardest cases close the list:

Beginner What are the ways to create a thread, and which is preferred?

Two core ways: extend Thread and override run(), or implement Runnable (or Callable) and pass it to a Thread or executor. Prefer Runnable/Callable: it separates the task from the execution mechanism, leaves your single inheritance slot free, and works with thread pools. Extending Thread couples your logic to a specific thread and is rarely justified.

Beginner Difference between start() and run()?

start() creates a new thread and schedules run() to execute on it — you get real concurrency. Calling run() directly just invokes it as an ordinary method on the current thread, with no new thread at all. Also, you can only call start() once per Thread object; a second call throws IllegalThreadStateException.

Beginner Runnable vs Callable?

Runnable.run() returns void and cannot throw checked exceptions; Callable.call() returns a value of type V and may throw checked exceptions. Runnable can run on a raw Thread or an executor; Callable needs an ExecutorService, which hands back a Future to retrieve the result. Use Callable when the task produces a result or can fail with a checked exception.

Beginner What are the states in a thread's lifecycle?

The Thread.State enum defines six: NEW (created, not started), RUNNABLE (running or ready to run), BLOCKED (waiting to acquire a monitor lock), WAITING (waiting indefinitely, e.g. wait()/join() with no timeout), TIMED_WAITING (waiting with a timeout, e.g. sleep(ms)), and TERMINATED (finished). A thread is in exactly one state at a time.

Beginner What is a daemon thread?

A background thread (set via setDaemon(true) before start()) that does not prevent the JVM from exiting — when only daemon threads remain, the JVM shuts down and abandons them. Used for background housekeeping (GC helpers, timers). User (non-daemon) threads keep the JVM alive until they complete.

Intermediate What does volatile guarantee, and what does it NOT?

It guarantees visibility (a read always sees the most recent write, no thread-local caching) and ordering (no reordering across the access; a volatile write establishes happens-before with a later volatile read of the same field). It does not guarantee atomicity of compound actions: count++ on a volatile int is read-modify-write and still races. Use volatile for a one-writer flag or a safely published reference; use locks or atomics for compound updates.

Intermediate What does synchronized lock, and what two guarantees does it give?

It acquires an object's intrinsic lock (monitor): a synchronized instance method locks this, a static synchronized method locks the Class object, and a synchronized block locks whatever object you name. Two guarantees: mutual exclusion (only one thread holds a given monitor at a time) and visibility (releasing the lock happens-before the next acquire of the same lock, so changes are flushed and re-read). Locking on different monitors gives no mutual exclusion.

Intermediate Explain the happens-before relationship.

It is the Java Memory Model's visibility guarantee: if action A happens-before action B, then A's memory writes are visible to and ordered before B. Key edges: statements within one thread (program order); a monitor unlock before a later lock on the same monitor; a volatile write before a later volatile read of that field; Thread.start() before the new thread's actions; a thread's final actions before a successful join(); and transitivity (A→B and B→C imply A→C). Without a happens-before edge, one thread's write may never be visible to another.

Intermediate volatile vs synchronized — when do you pick each?

volatile is non-blocking and covers visibility/ordering of a single field — ideal for a status/stop flag or publishing an immutable object reference. synchronized blocks and covers both mutual exclusion and visibility — required when you must make several actions atomic (check-then-act, read-modify-write, updating multiple fields consistently). Rule of thumb: single-field visibility → volatile; compound invariant → synchronized (or a lock/atomic).

Intermediate Why must wait()/notify() be called inside synchronized, and why wait() in a loop?

wait(), notify() and notifyAll() operate on an object's monitor, so you must already hold that monitor — otherwise you get IllegalMonitorStateException. wait() atomically releases the lock and suspends, reacquiring it on wake-up. You loop (while (!condition) wait();) because of spurious wakeups and because another thread may change the condition between notify and reacquisition — re-checking on wake is the only safe pattern. Prefer notifyAll() to avoid stranding threads.

Intermediate What is a race condition / memory consistency error?

A race condition is when the result depends on the unpredictable interleaving of threads accessing shared mutable state without proper synchronization (e.g. two threads both read-increment-write a counter and one update is lost). A memory consistency error is the visibility side: one thread's write is never seen by another because there is no happens-before edge, so a thread reads a stale value. Both are fixed by establishing happens-before via locks, volatile, or higher-level concurrency utilities.

Intermediate What is ThreadLocal, a real use case, and its main risk?

It gives each thread its own copy of a variable, accessed via get()/set(), so no synchronization is needed for that state. Use cases: a per-thread SimpleDateFormat (not thread-safe), or carrying request context (user, transaction, trace ID) implicitly. Main risk: memory leaks and data bleed in thread pools — because pooled threads are reused and never terminate, a value set for one task lingers for the next. Always remove() in a finally block.

Pro Why was double-checked locking broken before Java 5, and how does volatile fix it?

In lazy-singleton DCL, the object reference could be published before the constructor finished, because the JMM (pre-Java 5) allowed the write of the reference to be reordered ahead of the writes initializing the object. A second thread could then see a non-null but partially constructed instance. Java 5 strengthened volatile: declaring the field volatile forbids that reordering and creates a happens-before edge, so any thread reading a non-null reference sees a fully initialized object. (An enum or a static holder class is often cleaner.)

Pro Is count++ on a volatile int thread-safe? Explain.

No. volatile guarantees each read sees the latest value and each write is immediately visible, but count++ is three operations — read, add one, write — that are not atomic together. Two threads can both read the same value, both increment, and both write it back, losing an update. Use synchronized, a ReentrantLock, or AtomicInteger.incrementAndGet() (a CAS-based atomic) for a correct concurrent counter.

Pro How do ThreadLocal memory leaks happen in thread pools, and how do you prevent them?

A ThreadLocalMap is keyed by a weak reference to the ThreadLocal but holds the value strongly. On a pooled thread that lives for the whole application, values set during a task are never cleared when the ThreadLocal goes out of scope, so they accumulate — and one request's data can be read by the next task that reuses the thread. Prevention: call remove() in a finally at the end of each unit of work (frameworks like Spring do this around request scope); never rely on the thread dying to clean up.
