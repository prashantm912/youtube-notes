Module: OOP — Interfaces, Records, Enums

Three ways Java pins down a shape: interfaces that a class promises to honour, records that carry immutable data with none of the boilerplate, and enums that fix a type to a finite set of constants — the trio modern-Java interview rounds circle back to again and again.

Knowledge Library → Courses → Core Java → OOP: Interfaces, Records, Enums

What this module covers

These three constructs sit on top of the classes-and-inheritance foundation from the previous module. Interfaces define behaviour contracts and — since Java 8 and 9 — can carry default, static and private methods; records (Java 16) generate the boilerplate of an immutable data carrier for you; sealed types (Java 17) let a class or interface name exactly who may extend it; and enums model a closed set of constants with the full power of a class. Together they power a large share of Java interview questions, from the evergreen interface-vs-abstract-class to modern sealed-plus-pattern-matching.

Topics

Interfaces (default/static/private methods, constants)

An interface is a pure contract: any class that implements it promises to supply the declared behaviour, and code can depend on the interface type without knowing the concrete class. Fields declared in an interface are implicitly public static final constants; method signatures are implicitly public abstract. A class may implement many interfaces, which is how Java gets the flexibility of multiple inheritance without the multiple-inheritance-of-state problem.

Why it matters: since Java 8, interfaces also carry default methods (concrete methods that let a library add behaviour to an interface without breaking every existing implementer) and static methods; Java 9 added private methods so default/static methods can share helper code without exposing it. Knowing which method kinds are allowed — and that a default collision between two interfaces must be resolved explicitly — is standard interview territory.

interface Shape {
    double PI = 3.14159;                         // implicitly public static final
    double area();                               // implicitly public abstract
    default String label() { return name() + ":" + area(); }  // default (Java 8)
    private String name() { return "shape"; }     // private helper (Java 9)
    static Shape unit() { return () -> 1.0; }     // static factory (Java 8)
}

Records (compact constructors, invariants), sealed classes/interfaces

A record is a transparent, shallowly-immutable carrier for a fixed set of values. Writing record Point(int x, int y) {} makes the compiler generate the private final fields, the canonical constructor, an accessor per component, and value-based equals, hashCode and toString — the exact boilerplate a data class needs. Records are implicitly final, cannot extend another class, but can implement interfaces. A compact constructor lets you validate or normalise arguments (enforcing invariants) without re-listing the parameters or assigning the fields yourself.

Sealed classes and interfaces answer the opposite question — who is allowed to extend me? A sealed type lists its subtypes with permits, and each permitted subtype must be final, sealed, or non-sealed. Sealing turns an open hierarchy into a closed, known set, which is what lets the compiler check a switch over the hierarchy for exhaustiveness. Records as the permitted subtypes plus a sealed interface is the idiomatic modern way to model an algebraic "one of these" type.

public sealed interface Shape permits Circle, Square {}

public record Circle(double radius) implements Shape {
    public Circle {                                 // compact constructor
        if (radius <= 0)
            throw new IllegalArgumentException("radius must be > 0");
    }                                               // fields assigned for you
}
public record Square(double side) implements Shape {}   // records are implicitly final

Enums complete (fields/methods, EnumSet/EnumMap)

An enum defines a type whose instances are a fixed, compile-time set of named constants — type-safe where bare int or String constants are not. But an enum is a full class: each constant can carry state through a private constructor, and the enum can declare fields, methods, and even per-constant method bodies. Every enum inherits name(), ordinal(), values() and valueOf(String), and works cleanly with switch.

Why it matters: because enum constants are a known, ordinal-indexed set, the JDK ships two specialised collections — EnumSet (a bit-vector-backed Set) and EnumMap (an array-backed Map). Both are far more compact and faster than HashSet/HashMap for enum keys and should be the default when your keys are an enum. A single-constant enum is also the JDK-recommended way to write a singleton.

enum Planet {
    EARTH(5.97e24), MARS(6.42e23);               // constants with constructor args

    private final double massKg;
    Planet(double massKg) { this.massKg = massKg; }   // private constructor
    double mass() { return massKg; }                   // instance method
}

EnumMap<Planet, String> capital = new EnumMap<>(Planet.class);  // array-backed

Interview questions

Most-asked interface, record, sealed and enum questions in Indian interviews — from service-company screening rounds to product-company deep dives:

Beginner What is an interface and how does it differ from a class?

An interface is a reference type declaring a contract — abstract method signatures (implicitly public abstract) and constants (implicitly public static final). Unlike a class it holds no instance state and cannot be instantiated; a class implements it and supplies the behaviour. A class can implement many interfaces but extend only one class.

Beginner Interface vs abstract class — when do you use each?

Use an interface for a capability that unrelated types can share and when you need multiple inheritance of type; use an abstract class when subtypes share state (instance fields) or non-public helper implementation. An interface can't hold instance fields or constructors; an abstract class can. Since Java 8 the gap narrowed (interfaces have default methods), so the state distinction is the sharpest remaining line — the single most common question in this area.

Beginner What is a record and what does the compiler generate for it?

A record is a concise immutable data carrier. From record Point(int x, int y) {} the compiler generates private final fields, a canonical constructor, one accessor per component (x(), y()), and value-based equals, hashCode and toString. Records are implicitly final and can't extend a class, but can implement interfaces.

Beginner Why are enums preferred over int or String constants?

Type safety: a method taking Day can only receive a real Day, whereas an int or String accepts any value. Enums also give readable toString, work in switch, are singletons per constant (safe == comparison), and can carry behaviour. The old "int enum pattern" had none of this and failed silently on typos.

Beginner Why were default methods added in Java 8?

To let library authors add methods to an existing interface without breaking every class that already implements it. The classic example is Collection.stream() — a default method meant every existing List/Set gained streams for free. They enable interface evolution; they are not meant to turn interfaces into full base classes.

Intermediate Can an interface have static and private methods? What are they for?

Yes. Static methods (Java 8) belong to the interface itself — good for factories and helpers, called as Interface.method(), not inherited by implementers. Private methods (Java 9), static or instance, let default and static methods share helper code without exposing it as part of the public contract. Interfaces still cannot have instance fields or protected members.

Intermediate A class implements two interfaces with the same default method — what happens?

Compile error: the "diamond" conflict is not resolved automatically. The class must override the method, and can delegate to a specific one with InterfaceName.super.method(). Note a class's own (or a superclass's) concrete method always wins over an inherited default — "class wins over interface".

Intermediate What is a compact constructor in a record, and when do you use it?

A constructor written as public Point { ... } — no parameter list, no field assignments. You use it to validate arguments (enforce invariants) or normalise them; the compiler runs your block, then assigns the components to the fields automatically. Throwing from it prevents an invalid record from ever existing.

Intermediate What are sealed classes and the three allowed subtype modifiers?

A sealed class or interface names its permitted subtypes with permits (the clause can be omitted when subtypes are in the same file). Each permitted subtype must be exactly one of: final (closes the branch), sealed (continues restricting), or non-sealed (reopens it to any subclass). This makes the hierarchy a known set — enabling exhaustive switch checks.

Intermediate EnumSet/EnumMap vs HashSet/HashMap — why prefer the enum versions?

When keys/elements are enum constants, EnumSet is stored as a bit vector and EnumMap as a plain array indexed by ordinal() — no hashing, near-constant-time ops, tiny footprint, and iteration in declaration order. They are the recommended default for enum keys; a HashMap there just wastes space and speed.

Intermediate How do you add fields, a constructor and methods to an enum?

List the constants first, each passing constructor arguments (e.g. MONDAY(1)), end that line with a semicolon, then declare private final fields, a (necessarily non-public) constructor, and methods. Each constant is a singleton instance built once at class load. You can also give a constant its own method body with a constant-specific class body.

Pro Why is a single-element enum the best way to implement a singleton?

The JVM guarantees exactly one instance per constant, created lazily at class initialisation and thread-safe by construction, with no boilerplate. It is also safe against serialization (no need for readResolve) and reflection attacks that can break a private-constructor singleton. This is the Effective Java recommendation and a frequent senior-round question.

Pro How do sealed types and records combine with pattern matching in switch, and why does exhaustiveness matter?

A switch over a sealed type can use type patterns (and record deconstruction patterns) for each permitted subtype. Because the compiler knows the complete set of subtypes, it verifies the switch covers them all — no default branch needed, and adding a new permitted subtype turns a missed case into a compile error rather than a silent runtime gap. Sealed interface + record subtypes + pattern-matching switch is Java's idiom for algebraic data types and safe, extensible dispatch.
