Module: OOP: Inheritance and Polymorphism

Inheritance is the easy part; the discipline around it — the equals/hashCode contract, overriding versus overloading, and what actually runs at runtime — is what interviewers probe hardest.

What this module covers

This module is about how Java classes build on one another and how the runtime decides which method to run. It spans inheritance and the super chain, overriding versus overloading (compile-time versus runtime resolution), the modern flexible constructor bodies feature alongside final and abstract, and the methods every class inherits from Object — above all the equals and hashCode contract that hash-based collections depend on.

Topics

Inheritance, polymorphism, overriding vs overloading, and super

A subclass declared with extends acquires the accessible fields and methods of one superclass. Java has single inheritance of state (one superclass) but multiple inheritance of type (many interfaces). Polymorphism follows: a superclass or interface reference can point to any subclass object, and an overridden instance method is chosen by the object's runtime type — dynamic dispatch — not by the reference's declared type. Overriding replaces an inherited method (same name and parameters); overloading declares several methods with the same name but different parameter lists, which the compiler resolves from the argument types.

Why it matters: the overloading-versus-overriding distinction — static (compile-time) binding versus dynamic (runtime) binding — is one of the two or three most-asked Java OOP questions in Indian interviews. super does double duty: super(...) chains to a parent constructor, and super.method() reaches an overridden parent method.

class Animal { String speak() { return "generic sound"; } }
class Dog extends Animal {
    @Override String speak() { return "Woof"; }          // overriding: same signature
    String speak(String mood) { return mood + " woof"; } // overloading: new parameter
}
Animal a = new Dog();   // upcast to the superclass type
a.speak();              // "Woof" — chosen at runtime by the object, not the reference

Flexible constructor bodies, final, and abstract

Historically super(...) or this(...) had to be the very first statement of a constructor. Java 25 finalized flexible constructor bodies (JEP 513): you may now run statements — argument validation, field preparation — before the explicit constructor call, provided they don't read the object being constructed. Alongside this, abstract marks a class that cannot be instantiated and may declare abstract methods for subclasses to implement, while final locks things down: a final variable is assigned once, a final method cannot be overridden, and a final class cannot be extended.

Why it matters: flexible constructor bodies close a real safety hole — you can now validate arguments or initialize subclass fields before the superclass constructor runs, avoiding the classic bug where a superclass constructor calls an overridable method that sees uninitialized subclass state. final and abstract are the vocabulary of API design and show up constantly in "how would you prevent subclassing?" style questions.

class Range {
    final int lo, hi;
    Range(int lo, int hi) {
        if (lo > hi)                                  // validate BEFORE super() — legal since Java 25
            throw new IllegalArgumentException("lo exceeds hi");
        super();                                       // explicit superclass call, no longer forced first
        this.lo = lo; this.hi = hi;
    }
}

Object methods: equals/hashCode, toString, clone, finalize

Every class inherits from Object: equals (default is reference identity), hashCode (default derived from identity), toString, clone, and finalize. Override equals to define logical equality, and you must override hashCode to match — the contract requires that objects which are equal return the same hash code. Use instanceof pattern matching to test the type and Objects.hash/Objects.equals to implement both cleanly.

Why it matters: the equals/hashCode contract is the single most-tested Object topic in interviews — break it and HashMap/HashSet silently lose entries or store duplicates. toString aids debugging; clone is widely avoided (broken by design — prefer copy constructors); and finalize is deprecated for removal (JEP 421) because it runs unpredictably — reach for Cleaner or try-with-resources instead.

final class Point {
    final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }
    @Override public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point p)) return false;     // pattern matching
        return x == p.x && y == p.y;
    }
    @Override public int hashCode() { return Objects.hash(x, y); }
}

Interview questions

Overriding-versus-overloading and the equals/hashCode contract dominate Indian OOP screening rounds — these are weighted accordingly, hardest last:

Beginner What is inheritance in Java, and what does single inheritance mean?

A subclass declared with extends acquires the accessible fields and methods of one superclass, reusing and specializing behavior. Java allows a class to extend exactly one class (single inheritance of state) but implement many interfaces (multiple inheritance of type), which sidesteps the diamond problem.

Beginner Difference between method overloading and method overriding?

Overloading: same method name, different parameter lists, resolved by the compiler from argument types — static (compile-time) binding. Overriding: a subclass redefines an inherited instance method with the same signature, resolved by the object's runtime type — dynamic (runtime) binding. Overloading is polymorphism at compile time; overriding is polymorphism at runtime.

Beginner What does the super keyword do?

Two uses. super(...) invokes a superclass constructor to chain initialization; super.method() calls the superclass version of an overridden method (and super.field reads a hidden field). If you don't write super(...), the compiler inserts a no-arg super().

Beginner What is runtime polymorphism (dynamic method dispatch)?

When a superclass or interface reference points to a subclass object, calling an overridden instance method runs the subclass's version — chosen by the object's actual type at runtime, not the reference's declared type. One reference type drives many behaviors.

Beginner What is an abstract class and how does it differ from a concrete class?

An abstract class (declared abstract) cannot be instantiated and may declare abstract methods with no body that subclasses must implement. It can still have constructors, fields, and concrete methods. Use it to share state and partial implementation across related subclasses.

Intermediate What rules must an overriding method obey?

Same name and parameter list; return type identical or a covariant subtype; access modifier no more restrictive than the parent's; it may not declare broader checked exceptions. It cannot override static, final, or private methods. Annotate it @Override so the compiler catches signature mistakes.

Intermediate Can you override static, private, or final methods?

No. static methods are hidden, not overridden — resolution uses the reference type, so there is no dynamic dispatch. private methods aren't inherited, so a same-named subclass method is unrelated. final methods are explicitly sealed against overriding.

Intermediate State the equals and hashCode contract.

equals must be reflexive, symmetric, transitive, and consistent, and x.equals(null) is false. hashCode must be consistent, and — the key link — equal objects must return the same hash code. Unequal objects may share a hash code (collisions are allowed), but equal objects with different hash codes break the contract.

Intermediate What breaks if you override equals but not hashCode?

Two logically equal objects can get different hash codes, so hash-based collections misbehave: a HashMap key you inserted can't be found, a HashSet stores what should be duplicates, and contains/get return wrong results. Rule: override them together, or neither.

Intermediate Difference between == and equals()?

== compares references (identity) for objects or raw values for primitives. equals() compares logical content when overridden (String, wrapper types, records do this). The default Object.equals() is just == until overridden. Also watch Integer caching, which can make == accidentally true for small boxed values.

Intermediate What does final mean on a variable, a method, and a class?

A final variable is assigned exactly once (a blank final may be set in the constructor); a final method cannot be overridden; a final class cannot be extended. Note that final does not make the referenced object immutable — only the reference is fixed.

Pro Are fields polymorphic? Explain field hiding versus method overriding.

No. Fields and static methods are resolved by the compile-time (reference) type, not the runtime object — accessing them through a superclass reference uses the superclass member even if the subclass declares one with the same name. That is hiding, not overriding. Only instance methods are dynamically dispatched; this asymmetry is a common trick question.

Pro Why is finalize() deprecated, and what should you use instead?

finalize() runs at an unpredictable time or never, can resurrect objects, hurts GC performance, and has security pitfalls — so it is deprecated for removal (JEP 421). Release resources deterministically with try-with-resources and AutoCloseable, and for last-resort cleanup use java.lang.ref.Cleaner or a PhantomReference.

Pro How do you implement equals() and hashCode() correctly, and what are the pitfalls?

Short-circuit on this == o, test the type with instanceof (or getClass() when subclasses must never be equal), compare significant fields with Objects.equals, and return Objects.hash(sameFields). Pitfalls: instanceof can break symmetry across a subclass; getClass() breaks Liskov substitution with proxies and subclasses; include only the fields used in equals, and prefer immutable fields so the hash code never changes while the object is a key.

Pro What do flexible constructor bodies (JEP 513) enable, and why are they safer?

Since Java 25 you may place statements before super(...) or this(...), as long as they don't access the instance under construction. This lets you validate or transform arguments and initialize the subclass's own fields before the superclass constructor runs — closing the long-standing hole where a superclass constructor that invoked an overridable method observed the subclass fields still at their defaults.
