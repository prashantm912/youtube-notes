Module: Performance and Design Patterns

The last Core Java module trades language features for judgment: measure before you optimise, know which Gang-of-Four patterns still earn their keep in a records-and-lambdas world, and recognise the anti-patterns that quietly cost throughput — the exact ground senior interviews probe.

What this module covers

This capstone joins two skills interviewers treat as the mark of a senior engineer: proving where time goes, and shaping code well. The performance half is about measurement — profiling a running system with JDK Flight Recorder, benchmarking small code correctly with JMH, and cutting cold-start time with AOT class loading from Project Leyden. The design half covers the 23 Gang-of-Four patterns as they actually look in modern Java, the SOLID principles behind them, and the common anti-patterns to avoid. Singleton, Factory and Builder dominate Indian design-pattern rounds, so they carry the most weight below.

Topics

Profiling (JFR), JMH microbenchmarks, and AOT class loading (Leyden)

JDK Flight Recorder (JFR) is a built-in, event-based profiler: it records method sampling, allocation, garbage collection, lock contention and I/O with overhead low enough to leave running in production. You start it with jcmd or the -XX:StartFlightRecording flag and analyse the resulting recording in JDK Mission Control. JMH — the OpenJDK Java Microbenchmark Harness — measures small pieces of code correctly: it forks a fresh JVM, warms up the JIT, and defeats the dead-code elimination and constant folding that make a naive System.nanoTime() loop lie. AOT class loading (JEP 483, the first Project Leyden feature to ship, in Java 24) writes loaded-and-linked classes into an ahead-of-time cache so a later run skips that work and starts faster — with no source changes.

Why it matters: the through-line is measure, don't guess. A strong candidate profiles to find the real hot path before touching code, benchmarks a change with JMH rather than a stopwatch, and reaches for structural wins like AOT startup on cold-start-sensitive services. "How would you find and prove a performance problem?" is a standard senior-round question, and JFR plus JMH is the honest answer.

@Benchmark                                  // JMH measures this method in isolation
public void lookup(Blackhole bh) {
    bh.consume(map.get(key));               // Blackhole prevents dead-code elimination
}
// @Fork, @Warmup and @Measurement control JVM forks and warmup iterations;
// build the benchmarks jar, then run it standalone — not inside your app or IDE.

GoF 23 patterns in modern Java (records/lambdas change idioms) + SOLID

The Gang of Four catalogued 23 patterns in three families: creational (how objects are made — Singleton, Factory Method, Abstract Factory, Builder, Prototype), structural (how objects compose — Adapter, Decorator, Facade, Proxy, Composite, Bridge, Flyweight), and behavioural (how objects interact — Strategy, Observer, Command, Template Method, State, Iterator, and more). Three lead every Indian interview: Singleton (one shared instance behind a global access point), Factory Method (defer instantiation so callers depend on an interface, not a concrete type), and Builder (assemble a complex, often immutable object step by step, escaping the telescoping-constructor trap).

Why it matters: modern Java rewrites the idioms, and interviewers reward candidates who know the current form. An enum is the safest Singleton — the JVM guarantees one instance and it resists both reflection and serialization attacks. A lambda or method reference often replaces an entire Strategy or Command class. Records give you immutable value objects, sometimes retiring a hand-written Builder, and sealed types make the closed set of subtypes behind many patterns explicit. SOLID — Single-responsibility, Open-closed, Liskov substitution, Interface-segregation, Dependency-inversion — is the design vocabulary these patterns serve; Dependency Inversion and Single Responsibility are the most quizzed.

public enum Config { INSTANCE;              // the safest Singleton: no null, no double-init
    public String get(String k) { /* ... */ return "..."; }
}
// Strategy without a class hierarchy — a lambda is the whole implementation:
Comparator<Order> byTotal = Comparator.comparingDouble(Order::total);

Common performance pitfalls and anti-patterns

A handful of mistakes account for most avoidable Java slowness: building strings with + inside a loop (every iteration allocates a fresh String — use StringBuilder), autoboxing on a hot path (a List<Integer> boxes every int; prefer primitive arrays or primitive streams), throwing exceptions for ordinary control flow (capturing a stack trace is costly), over-synchronising or holding a lock longer than needed, unbounded or leaking caches, and the N+1 query pattern at the data layer. Overarching all of them is premature optimisation — micro-tuning code that the profiler never flagged as hot.

Why it matters: the anti-pattern behind the anti-patterns is optimising without measuring. Even "obvious" wins — parallel streams, adding a cache — can lose to memory-bandwidth limits, lock contention, or the maintenance cost of added complexity. The senior answer is a loop: profile with JFR, confirm the suspected cost with a JMH benchmark, change exactly one thing, then measure again — and keep the change only if the numbers moved.

// Anti-pattern: quadratic allocation from repeated String concatenation
String out = "";
for (String p : parts) out = out + p;       // a new String every iteration

// Fix: one mutable buffer
var sb = new StringBuilder();
for (String p : parts) sb.append(p);
String out2 = sb.toString();

Interview questions

The most-asked performance and design questions in Indian interviews — Singleton, Factory and Builder lead the design rounds, so they come first, with profiling and the hardest judgement calls last:

Beginner What is the Singleton pattern, and what is the simplest correct way to implement it in Java?

Singleton guarantees a class has exactly one instance and gives a global access point to it. The classic form makes the constructor private and exposes a static getInstance(). The simplest genuinely correct form is a single-element enum (enum Config { INSTANCE; }): the JVM guarantees one instance, and it is immune to the reflection and serialization tricks that break hand-written singletons.

Beginner What problem does the Builder pattern solve?

It replaces telescoping constructors — the explosion of overloads you get when an object has many optional fields — and constructors whose arguments are easy to transpose. A Builder assembles the object through named, chainable steps (new Pizza.Builder().size(L).cheese(true).build()) and can enforce invariants in build(), producing a readable, often immutable result.

Beginner What is the Factory Method pattern, and why use it?

Factory Method defines a method for creating objects but lets the concrete type be chosen by a subclass or by logic inside the method, so callers depend on an interface rather than a new ConcreteClass(). It centralises construction, decouples client code from implementations, and makes adding a new type a local change — the essence of the Open-Closed Principle.

Beginner How many Gang-of-Four patterns are there, and what are the three categories?

23 patterns in three families: creational (object creation — Singleton, Factory Method, Abstract Factory, Builder, Prototype), structural (composition — Adapter, Decorator, Facade, Proxy, Composite, Bridge, Flyweight), and behavioural (interaction — Strategy, Observer, Command, Template Method, State, Iterator, and others).

Beginner What is a microbenchmark, and what is JMH?

A microbenchmark measures a very small piece of code — a method or a few lines. JMH (Java Microbenchmark Harness) is the OpenJDK tool for doing it correctly: it runs warmup iterations so the JIT compiles the code first, forks separate JVMs for isolation, and provides constructs (like Blackhole) that stop the compiler from optimising your measured work away.

Intermediate How do you make a Singleton thread-safe?

Three good options. Eager initialisation (private static final X I = new X();) is thread-safe by class-loading guarantees. The Bill Pugh holder idiom uses a static nested class so the instance is created lazily and safely on first access, with no locking. Double-checked locking works but needs the field marked volatile. Best of all is an enum, which is thread-safe by construction.

Intermediate Factory Method vs Abstract Factory — what is the difference?

Factory Method creates one product through a single overridable method and uses inheritance to vary the result. Abstract Factory creates families of related products through an object with several creation methods (a widget factory making buttons, checkboxes and menus that all match one theme) and uses composition. Rule of thumb: one product varies → Factory Method; a whole related set varies together → Abstract Factory.

Intermediate Why can't you trust a hand-written benchmark loop, and how does JMH fix it?

The JIT distorts naive loops: it can constant-fold a computation with fixed inputs, eliminate results you never use (dead-code elimination), and only compile the hot code after a warmup period your timing already included. JMH forks a fresh JVM, runs warmup iterations before it starts measuring, feeds varying state through @State objects, and consumes results via Blackhole so the work can't be optimised out. That turns a misleading number into a defensible one.

Intermediate What is JDK Flight Recorder, and why is it safe to run in production?

JFR is a profiler built into the JVM that records timestamped events — method samples, allocations, GC pauses, lock contention, I/O — into a compact binary file you open in JDK Mission Control. It is designed for very low overhead (roughly a couple of percent with default settings) because it piggybacks on data the JVM already tracks, so unlike heavier sampling profilers it can stay enabled on live systems to catch problems that only appear under real load.

Intermediate Name the SOLID principles, and explain the one interviewers ask about most.

Single-responsibility, Open-closed, Liskov substitution, Interface-segregation, Dependency-inversion. The most-asked is Dependency Inversion: high-level modules should depend on abstractions, not concrete classes, and details depend on those abstractions. It is why we inject an interface rather than new a concrete service — the principle that dependency-injection frameworks like Spring exist to automate.

Pro How do records, enums, sealed types and lambdas change the classic GoF patterns?

They collapse boilerplate. An enum is the canonical Singleton; a lambda or method reference is a one-line Strategy or Command, removing the need for a class per behaviour; records give immutable value objects that often make a Builder unnecessary and are ideal for Memento or DTO roles; and sealed interfaces plus pattern-matching switch express the closed type hierarchy behind Visitor or State without the double-dispatch ceremony. The intent survives; the scaffolding shrinks.

Pro What are the most common Java performance anti-patterns, and how do you avoid premature optimisation?

String concatenation in loops, autoboxing on hot paths, exceptions used for control flow, over-synchronisation, unbounded caches, and N+1 queries. You avoid premature optimisation by refusing to tune anything the profiler has not flagged: measure with JFR to find the real hot path, reproduce the cost in a JMH benchmark, change one thing, and re-measure. Optimisations you cannot demonstrate with numbers are guesses, and often add complexity for no gain.

Pro Explain double-checked locking, and why the field must be volatile.

Double-checked locking checks the instance, synchronises only if it is null, then checks again inside the lock — so the expensive synchronisation happens once. Without volatile, the Java Memory Model allows the write that publishes the instance to be reordered so another thread sees a non-null but partially constructed object. volatile establishes the happens-before edge that forbids that reordering and makes the fully built object visible. Most code should prefer the holder idiom or an enum and skip this entirely.

Pro What is AOT class loading (JEP 483, Project Leyden), and what does it optimise?

It targets startup and time-to-peak-performance. The JVM can record classes it has already read, parsed, loaded and linked into an ahead-of-time cache during a training run; on later runs it loads them straight from that cache instead of redoing the work, so the application reaches a ready state faster. It needs no source or build changes beyond producing the cache, and it fully preserves Java's dynamic behaviour including reflection. It is the first Project Leyden feature to ship, in Java 24, with more static-image work to follow.
