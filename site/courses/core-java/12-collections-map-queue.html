Module: Collections — Map, Queue, Comparator

Half of a Java collections interview lives inside HashMap. This module maps the whole territory — hashing internals, sorted and sequenced maps, the Queue and Deque family, and the ordering rules of Comparator — then sends you to the source docs that go deep.

What this module covers

This is the second half of the Collections Framework: the Map hierarchy (hash-based, sorted, sequenced), the Queue and Deque family (ArrayDeque, PriorityQueue), and the ordering utilities Comparable and Comparator that every sorted structure depends on. HashMap's internal working is the single most-asked data-structure topic in Indian interviews, so it gets the most weight below. Immutable collection factories round out the picture, since read-only views are now the default advice for returned data.

Topics

Map: HashMap internals, LinkedHashMap/LRU, TreeMap/NavigableMap, SequencedMap

A Map stores key-to-value bindings with unique keys. HashMap is the default: keys land in an array of buckets by their spread hashCode, giving amortised O(1) get/put; on collision a bucket holds a linked list, and since Java 8 a bucket with too many entries whose keys are Comparable converts to a red-black tree so the worst case degrades to O(log n) instead of O(n). It grows by doubling capacity and rehashing when size exceeds capacity times the load factor (default 0.75). LinkedHashMap adds predictable iteration order and, in access-order mode, becomes a ready-made LRU cache via removeEldestEntry. TreeMap keeps keys sorted in a red-black tree and implements NavigableMap (floor/ceiling/higher/lower, submaps). Java 21's SequencedMap gives every ordered map a common first/last and reversed() API.

Why it matters: interviewers use HashMap to test whether you understand equals/hashCode, hashing, and amortised cost — not just the API surface. Getting the contract wrong (mutable keys, a broken hashCode) silently corrupts lookups, which is why this shows up in almost every screening round.

Map<String,Integer> freq = new HashMap<>();
freq.merge(word, 1, Integer::sum);        // idiomatic counting
int n = freq.getOrDefault("x", 0);

// access-order LinkedHashMap = a fixed-size LRU cache
Map<K,V> lru = new LinkedHashMap<>(16, 0.75f, true) {
    protected boolean removeEldestEntry(Map.Entry<K,V> e) { return size() > CAP; }
};

Queue and Deque: ArrayDeque and PriorityQueue

Queue models a FIFO line (offer/poll/peek); Deque is a double-ended queue that works as both a queue and a stack. ArrayDeque is the modern, array-backed implementation and the recommended replacement for both the legacy Stack class and LinkedList-as-queue — it has no capacity limit, forbids nulls, and gives amortised O(1) at both ends with far better cache behaviour. PriorityQueue is a different beast: an unbounded binary min-heap that orders by natural ordering or a supplied Comparator, so poll always returns the smallest element in O(log n). Its iterator is not sorted — only repeated polling yields sorted order.

Why it matters: "which class for a stack?" and "how does PriorityQueue order elements?" are staple questions, and the ArrayDeque-over-Stack advice signals command of current idioms rather than 1998 Java.

Deque<Integer> stack = new ArrayDeque<>();   // prefer over java.util.Stack
stack.push(1); stack.push(2); stack.pop();   // LIFO

Queue<Task> pq = new PriorityQueue<>(Comparator.comparingInt(Task::priority));
pq.offer(t);
Task next = pq.poll();                        // smallest priority first

Comparator and Comparable, and immutable collections

Comparable defines a type's single natural ordering through compareTo — the contract must be antisymmetric, transitive, and (strongly recommended) consistent with equals, or TreeMap/TreeSet misbehave. Comparator is an external, reusable ordering you can build declaratively: Comparator.comparing(keyExtractor), chained with thenComparing for tie-breaks, flipped with reversed(), and made null-safe with nullsFirst/nullsLast. Separately, the factory methods List.of, Set.of and Map.of (Java 9+) produce compact, immutable, null-hostile collections — the preferred way to return data that callers must not mutate.

Why it matters: "Comparable vs Comparator" is a guaranteed question, and interviewers increasingly probe whether you reach for List.of and defensive immutability instead of leaking mutable internals.

people.sort(Comparator.comparing(Person::lastName)
                      .thenComparing(Person::firstName)
                      .reversed());

// null-safe key ordering
Comparator<Person> byCity =
    Comparator.comparing(Person::city, Comparator.nullsFirst(Comparator.naturalOrder()));

List<Integer> ids = List.of(1, 2, 3);   // immutable, rejects null, throws on mutation

Interview questions

The most-asked Map, Queue and Comparator questions in Indian interviews — HashMap internals dominate the screening rounds, so they lead here:

Beginner HashMap vs LinkedHashMap vs TreeMap — when do you pick each?

HashMap: no order, fastest, amortised O(1). LinkedHashMap: preserves insertion order (or access order for LRU), slightly more memory. TreeMap: keys sorted by natural order or a Comparator, O(log n), and it alone offers navigation (floor/ceiling/submaps). Pick HashMap by default, LinkedHashMap when order must be stable, TreeMap when you need sorted keys or range queries.

Beginner Which class should you use for a stack in modern Java, and why not Stack?

ArrayDeque via the Deque interface (push/pop/peek). The legacy Stack extends Vector, so every method is synchronised (needless overhead) and it exposes index-based Vector methods that violate stack semantics. ArrayDeque is faster and cleaner; the Javadoc itself recommends it.

Beginner Comparable vs Comparator?

Comparable is implemented by the class itself (compareTo) and defines one natural ordering. Comparator is a separate object defining any number of external orderings without touching the class — passed to sort, TreeMap, or PriorityQueue. Rule of thumb: one intrinsic order → Comparable; multiple or ad-hoc orders → Comparator.

Intermediate Explain how HashMap stores and retrieves an entry internally.

HashMap holds an array of buckets (default length 16). On put, it takes the key's hashCode(), spreads it (see next question), and maps it to a bucket index with hash & (n-1). The entry (key, value, hash, next) goes into that bucket. On get, it recomputes the index, then walks the bucket comparing by hash then equals to find the matching key. Even distribution keeps this amortised O(1).

Intermediate Why does HashMap spread the hashCode instead of using it directly?

Because the index is hash & (n-1), which only uses the low bits of the hash. Two keys differing only in high bits would collide constantly. HashMap's hash() XORs the high 16 bits into the low 16 (h ^ (h >>> 16)) so high-bit variation influences the bucket — cheap insurance against poorly distributed hashCodes.

Intermediate What is load factor, and when does a HashMap resize?

Load factor (default 0.75) is the fullness threshold. When size > capacity × loadFactor, the table doubles capacity and rehashes every entry into the larger array. 0.75 balances space against collision cost. Resizing is O(n); sizing the map up front (or via HashMap.newHashMap(expected)) avoids repeated rehashes when the element count is known up front.

Intermediate What happens on a hash collision, and what changed in Java 8?

Keys mapping to the same bucket form a chain. Pre-Java 8 that was always a linked list — O(n) in the worst case. Since Java 8, once a bucket holds 8 entries (TREEIFY_THRESHOLD) and the table has at least 64 buckets, and the keys are Comparable, the bucket converts to a red-black tree, bounding worst-case lookup at O(log n). It untreeifies back to a list when it shrinks below 6.

Intermediate Why must the equals/hashCode contract hold, and what breaks if it doesn't?

The contract: equal objects must have equal hashCodes. If you override equals but not hashCode, two equal keys can hash to different buckets, so a get after a put returns null — the entry is "lost". If unequal objects share a hashCode it still works, just with more collisions. Always override both together, using the same fields.

Intermediate Why should HashMap keys be immutable?

The bucket index is fixed at insertion from the key's hashCode. If a mutable key changes a field that affects its hashCode after being stored, its computed bucket no longer matches where it lives, so the map can no longer find it — the entry becomes a leak you can neither get nor remove by key. Prefer immutable keys (String, boxed primitives, records with final fields).

Intermediate How do you build an LRU cache with LinkedHashMap?

Construct new LinkedHashMap<>(capacity, 0.75f, true) — the third argument enables access-order, so every get/put moves the entry to the tail. Override removeEldestEntry to return true once size() exceeds the cap; the map then evicts the least-recently-used head entry automatically. It's the standard "implement an LRU cache" answer without hand-rolling a doubly linked list.

Intermediate How does PriorityQueue order elements, and is its iterator sorted?

It's a binary min-heap ordered by natural ordering or a supplied Comparator; peek/poll return the smallest in O(1)/O(log n) and offer is O(log n). Crucially, iterating (or toString) does not yield sorted order — only the head is guaranteed smallest. To get sorted output you must poll repeatedly. For a max-heap, pass Comparator.reverseOrder().

Intermediate How do Comparator chaining and null handling work?

Build with Comparator.comparing(Type::key), break ties with thenComparing(...), and flip with reversed(). For fields that can be null, wrap with Comparator.nullsFirst(...) or nullsLast(...) so nulls sort predictably instead of throwing NPE. These compose left-to-right, replacing verbose hand-written compare methods.

Pro HashMap vs Hashtable vs ConcurrentHashMap?

Hashtable is legacy: fully synchronised on every method, no nulls, effectively obsolete. HashMap is unsynchronised and allows one null key and null values. ConcurrentHashMap is the concurrent choice: thread-safe with fine-grained locking/CAS (bucket-level, not a single lock), high read concurrency, and no nulls (a null return is ambiguous with "absent"). Use HashMap single-threaded, ConcurrentHashMap under contention — never Hashtable in new code.

Pro Why did Java 8 add treeification, and what problem beyond speed did it address?

Besides bounding worst-case lookup at O(log n), it hardened HashMap against algorithmic-complexity denial-of-service attacks: an attacker feeding many keys with colliding hashCodes (e.g. crafted Strings) could force O(n) chains and O(n²) overall behaviour. Converting overloaded buckets to red-black trees caps the damage. It only triggers with a large enough table and Comparable keys; otherwise the map resizes first.

Pro What breaks in a TreeMap if compareTo (or the Comparator) is inconsistent with equals?

TreeMap locates keys purely by comparison, not equals. If two "unequal" keys compare as 0, the second overwrites the first — the map appears to lose entries and its behaviour silently contradicts the Map contract (which is defined in terms of equals). A non-transitive or asymmetric compareTo corrupts the tree ordering outright. This is why the Comparable Javadoc strongly recommends consistency with equals for use in sorted collections.
