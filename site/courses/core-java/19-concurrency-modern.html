Module: Concurrency — Virtual Threads and Modern

Project Loom rewrote what "a thread" costs in Java. This module covers the modern concurrency toolkit — virtual threads and scoped values, structured concurrency (still baking as a preview), and the liveness failures every interviewer still asks about — then points you at the source docs that go deep.

What this module covers

This is the frontier of Java concurrency: virtual threads (final in Java 21) and the scoped values that replace ThreadLocal for passing context, structured concurrency via StructuredTaskScope — a feature still in preview as of Java 26 — and the classic liveness failures (deadlock, livelock, starvation) plus how to detect them and the everyday concurrency bugs that cause them. Virtual threads are the single most-hyped Java feature of the last few releases and now a screening staple in Indian interviews, so they lead. Where an API is still preview, this page says so plainly.

Topics

Virtual threads (Loom), scoped values

Virtual threads (Project Loom, final in Java 21 via JEP 444) are lightweight threads scheduled by the JVM rather than the operating system. Many virtual threads are multiplexed (M:N) onto a small pool of platform carrier threads by a work-stealing scheduler; when a virtual thread hits a blocking call — I/O, a lock, BlockingQueue.take() — it unmounts from its carrier so another can run, and remounts later to continue. The payoff is that the simple thread-per-request, blocking style now scales to millions of concurrent tasks: you write straightforward synchronous code and get throughput that previously demanded reactive frameworks. Create them with Thread.startVirtualThread(...), Thread.ofVirtual(), or Executors.newVirtualThreadPerTaskExecutor() — and never pool them (they are cheap to create); cap access to scarce resources with a Semaphore instead.

Scoped values (final in Java 25 via JEP 506) are the companion for passing context — a current user, a request id, a transaction — down a call chain without threading it through every method parameter, and without ThreadLocal's baggage. A scoped value is immutable, bound only for the dynamic duration of a run()/call() lambda, and unbound automatically when that lambda returns, so it can never leak; it is also inherited cheaply by child threads under structured concurrency, which matters when you fan out across thousands of virtual threads. Why it matters: "what are virtual threads and how do they differ from platform threads" and "what replaces ThreadLocal now" are near-guaranteed questions.

Thread.startVirtualThread(() -> log.info("hi"));          // one-off virtual thread

try (var pool = Executors.newVirtualThreadPerTaskExecutor()) {
    for (var task : tasks) pool.submit(task);             // a NEW thread per task, not a pool
}                                                         // millions of tasks are fine

static final ScopedValue<User> CURRENT = ScopedValue.newInstance();
ScopedValue.where(CURRENT, user).run(() -> handle());     // bound only for this scope
// ...deep in the call stack, with no parameter passing:
User u = CURRENT.get();                                   // auto-unbound when run() returns

Structured concurrency (still preview as of Java 26)

Structured concurrency treats a group of concurrent subtasks as a single unit of work. Using StructuredTaskScope inside a try-with-resources block, you fork() each subtask (each runs on its own virtual thread) and then join() them together: if one subtask fails, the rest are cancelled; when the scope exits, no subtask can outlive it. That removes the classic hazards of raw ExecutorService and CompletableFuture — leaked threads, orphaned work, swallowed exceptions, cancellation that never propagates — and lets concurrent code read with the same block structure as sequential code, while thread dumps show the real parent-child hierarchy.

The caveat matters for both interviews and production: structured concurrency is still a preview feature as of Java 26 (JEP 525, its sixth preview), so it requires --enable-preview and its API is still moving. Java 25 reworked it substantially — public constructors gave way to the StructuredTaskScope.open() factory, and a separate Joiner now defines the completion policy (composition over inheritance) in place of the old ShutdownOnFailure/ShutdownOnSuccess subclasses. Learn the shape and state the caveat; do not assume the surface is frozen.

// Preview in Java 25/26 — compile and run with --enable-preview
try (var scope = StructuredTaskScope.open()) {           // default policy: all-succeed-or-throw
    Subtask<User>    user  = scope.fork(() -> findUser(id));
    Subtask<Integer> order = scope.fork(() -> fetchOrder(id));
    scope.join();                                        // wait for all; a failure cancels the rest
    return new Response(user.get(), order.get());
}                                                        // subtasks cannot outlive this block

Deadlock/livelock/starvation + detection, common concurrency bugs

These are liveness failures — the program is running but no longer making progress. Deadlock: two or more threads each hold a lock the other needs, forming a circular wait; it requires all four Coffman conditions (mutual exclusion, hold-and-wait, no preemption, circular wait), so breaking any one prevents it. Livelock: threads keep reacting to one another and changing state but never advance — like two people repeatedly stepping the same way to pass in a corridor. Starvation: a thread runs but is perpetually denied a resource it needs, because an unfair lock or a greedy neighbour keeps taking it first.

Detect a deadlock in a running JVM with a thread dump — jstack <pid> or jcmd <pid> Thread.print — which prints "Found one Java-level deadlock" and the exact cycle of threads and locks; VisualVM and JConsole surface it in a GUI, and ThreadMXBean.findDeadlockedThreads() reports it programmatically. Beyond deadlock, the common concurrency bugs interviewers probe are race conditions, visibility bugs (a missing volatile/synchronized so one thread never sees another's write), check-then-act and other non-atomic compound actions, and unsafe publication. Prevention beats cure: acquire multiple locks in a globally consistent order, prefer tryLock with a timeout, keep critical sections small, and reach for java.util.concurrent constructs over hand-rolled locking. Why it matters: "what is a deadlock, and how do you detect and prevent it" is one of the most reliably asked concurrency questions in Indian interviews.

// Deadlock: T1 locks A then B; T2 locks B then A -> circular wait, both stuck.
// Fix: impose a global order so every thread takes the lower-id lock first.
void transfer(Account a, Account b, long amount) {
    Account first  = a.id() < b.id() ? a : b;
    Account second = a.id() < b.id() ? b : a;
    synchronized (first) {
        synchronized (second) { /* move funds */ }
    }
}

Interview questions

Modern-concurrency questions dominate current Indian interview loops — virtual threads especially. Ordered easiest first, hardest last:

Beginner What is a virtual thread, and how does it differ from a platform thread?

A platform thread is a thin wrapper over an OS thread — expensive (a large, mostly fixed stack), so you get at most a few thousand. A virtual thread is a lightweight thread scheduled by the JVM, not the OS: it lives on the heap, costs very little, and you can run millions. Many virtual threads are multiplexed onto a small set of platform "carrier" threads. Both are java.lang.Thread; the difference is who schedules them and how cheap they are. Virtual threads became final in Java 21 (JEP 444).

Beginner How do you create and run a virtual thread?

Thread.startVirtualThread(runnable) for a one-off; Thread.ofVirtual().name("x").start(runnable) via the builder; or, most commonly, Executors.newVirtualThreadPerTaskExecutor(), which starts a new virtual thread for every submitted task. That executor is AutoCloseable, so use it in a try-with-resources block that waits for all tasks to finish.

Beginner Why shouldn't you pool virtual threads?

Pools exist to reuse expensive resources. Virtual threads are cheap to create and effectively unlimited, so pooling them just adds contention and defeats the point. The idiom is one virtual thread per task. To limit concurrent access to a genuinely scarce resource (say a database with 10 connections), cap it with a Semaphore, not a thread pool.

Beginner What is a deadlock?

Two or more threads blocked forever, each holding a lock the other needs — a circular wait. Classic case: thread 1 holds lock A and wants B, while thread 2 holds B and wants A, so neither proceeds. It needs all four Coffman conditions (mutual exclusion, hold-and-wait, no preemption, circular wait); break any one to prevent it.

Beginner Deadlock vs livelock vs starvation?

Deadlock — threads are stuck waiting on each other and do nothing. Livelock — threads keep running and changing state in response to each other but still make no progress (two people repeatedly stepping the same way in a hallway). Starvation — a thread runs but is perpetually denied a resource (CPU time or a lock) because others keep taking it first, often due to unfair scheduling or priorities.

Intermediate How are virtual threads scheduled — what are mounting and unmounting?

The JDK runs a work-stealing ForkJoinPool (in FIFO mode) of platform carrier threads. A runnable virtual thread is mounted on a carrier and executes; when it blocks (blocking I/O, a BlockingQueue, most java.util.concurrent locks) it unmounts, freeing the carrier to run another virtual thread, and its stack lives on the heap until it remounts to continue. This M:N multiplexing is what lets a handful of OS threads serve millions of virtual threads.

Intermediate What is pinning, and what changed in JDK 24?

Pinning is when a mounted virtual thread cannot unmount from its carrier, so a blocking call there also blocks the carrier — losing the scalability benefit. Historically (JDK 21–23) it happened inside synchronized blocks/methods and during native (JNI) calls. JEP 491 (JDK 24) reworked object monitors so synchronized no longer pins; native calls can still pin. Diagnose it with the jdk.VirtualThreadPinned JFR event. The older advice to swap synchronized for ReentrantLock is mostly unnecessary on JDK 24+.

Intermediate What are scoped values, and why prefer them over ThreadLocal?

A scoped value (final in Java 25, JEP 506) shares an immutable value with all code called inside a bounded dynamic scope — ScopedValue.where(KEY, value).run(() -> ...) — and the binding disappears automatically when that lambda returns. Versus ThreadLocal: it is immutable (no surprise mutation), has a clear bounded lifetime (no set()/remove() leaks), and is inherited cheaply by child threads in structured concurrency. That last point matters with virtual threads, where per-thread ThreadLocal copies across millions of threads are wasteful.

Intermediate How do you detect a deadlock in a running JVM?

Take a thread dump — jstack <pid> or jcmd <pid> Thread.print — which explicitly reports "Found one Java-level deadlock" and shows the cycle of threads plus the locks each holds and awaits. VisualVM and JConsole detect and highlight it in a GUI, and ThreadMXBean.findDeadlockedThreads() returns the involved thread ids programmatically. Once you can see the cycle, the fix is usually a consistent lock-acquisition order.

Intermediate What is structured concurrency, and what does StructuredTaskScope give you?

It scopes a group of concurrent subtasks to a block so they behave as one unit. Inside a try-with-resources StructuredTaskScope you fork() each subtask (on its own virtual thread) and join() them; if one fails the scope cancels the others, and none can outlive the block. That yields reliable error propagation, no leaked threads, and thread dumps that show the parent-child hierarchy. Note it is still a preview feature as of Java 26 (JEP 525), so it needs --enable-preview and its API is still evolving.

Pro Why do ThreadLocals become a problem at virtual-thread scale?

ThreadLocals hold per-thread mutable state for the life of the thread. With a few pooled platform threads that is fine; with millions of short-lived virtual threads each value is duplicated per thread, inflating memory, and inheritable ThreadLocals copy the whole map to every child. They also depend on discipline (remove()) to avoid leaks. Scoped values fix all three: immutable, automatically bounded to a scope, and shared by reference (cheap) with child threads — which is exactly why they were introduced alongside Loom.

Pro Structured concurrency vs ExecutorService / CompletableFuture — what does it guarantee that they don't?

With a raw ExecutorService an early failure can leave sibling subtasks running (leaked, orphaned work), exceptions can be swallowed inside Futures, and the parent-child relationship is invisible in a thread dump. Structured concurrency enforces that all subtasks complete or are cancelled before the scope exits, propagates the first failure and cancels the rest, and preserves the call hierarchy for observability. CompletableFuture composes async pipelines but has the same lifetime and cancellation gaps. Caveat: structured concurrency is still preview in Java 26 (JEP 525) and its API changed notably in Java 25 (the open() factory plus a Joiner policy), so treat it as not yet final.

Pro Your code must hold two locks — how do you prevent deadlock?

Impose a global ordering: every thread that needs both locks acquires them in the same canonical order (for example by object id or hash), which eliminates the circular-wait condition. Complement it with tryLock(timeout) so a thread backs off and retries instead of blocking forever; keep critical sections small and never call foreign/callback code while holding a lock; prefer a single coarser lock or a lock-free java.util.concurrent structure; and never hold a lock across a blocking call. Consistent lock ordering is the standard interview answer.
