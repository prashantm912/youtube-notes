Module: Exceptions

Exception handling is where interviewers separate candidates who memorised syntax from those who understand control flow — checked versus unchecked and try-with-resources come up in almost every Indian screening round.

What this module covers

Java signals errors and other abnormal events by throwing objects that descend from Throwable. This module walks the hierarchy and the checked/unchecked split, the full try/catch/finally machinery including try-with-resources, AutoCloseable and multi-catch, and finishes with designing custom exceptions, reading suppressed exceptions, and the handling discipline that keeps failures debuggable. The depth lives in the linked resources — official docs first.

Topics

Exception hierarchy, checked vs unchecked

Everything you can throw or catch descends from Throwable, which splits into two branches: Error (serious JVM-level problems such as OutOfMemoryError that applications should not try to recover from) and Exception (application-level conditions). Inside the Exception branch, RuntimeException and its subclasses are unchecked; every other Exception is checked. Checked exceptions are enforced by the compiler through the catch-or-specify rule — you must either catch them or declare them with throws.

Why it matters: "checked vs unchecked" is the single most-asked exceptions question in Indian interviews. Beyond the mechanics, the design guidance is what senior answers add — make it checked if a caller can reasonably recover, unchecked if it signals a programming bug (a null, a bad argument) the caller cannot fix at runtime.

Throwable
 ├─ Error              // unchecked — OutOfMemoryError, StackOverflowError
 └─ Exception          // checked by default — IOException, SQLException
     └─ RuntimeException // unchecked — NullPointerException, IllegalArgumentException

try/catch/finally, try-with-resources + AutoCloseable, multi-catch

A try block wraps code that might fail; catch blocks handle specific thrown types; finally runs on every exit path — normal, exceptional, or return — which historically made it the place to close resources. Multi-catch (Java 7) lets one catch handle several unrelated types with the | operator, where the caught types must not be sub- or super-classes of each other and the parameter is implicitly final.

Try-with-resources (Java 7) replaces most finally cleanup: any resource declared in the header that implements AutoCloseable is closed automatically at block end, in reverse order of declaration, whether the block completes or throws. It is the correct answer to "how do you avoid resource leaks?" — another topic Indian interviews probe heavily — because a hand-written finally can null-check wrongly or let a close() failure mask the original error.

try (var in  = new FileInputStream(src);
     var out = new FileOutputStream(dst)) {   // out closed first, then in
    in.transferTo(out);
} catch (FileNotFoundException | SecurityException e) {  // multi-catch
    logger.warning("cannot copy: " + e.getMessage());
}

Custom exceptions, suppressed exceptions, best practices

Write a custom exception by extending Exception (checked) or RuntimeException (unchecked) — the same recoverable-vs-bug rule decides which. Always offer the (String message, Throwable cause) constructor so callers can chain: wrapping a low-level failure while passing the original as the cause preserves the root stack trace instead of hiding it. Reach for a custom type only when it carries meaning a standard exception cannot; otherwise reuse IllegalArgumentException, IllegalStateException, and friends.

Suppressed exceptions are the subtle case try-with-resources handles for you: if the body throws and then a resource's close() also throws, the body exception propagates and the close() exception is attached via addSuppressed(), readable later through getSuppressed(). The best-practice throughline: catch the most specific type, never swallow silently, preserve the cause, clean up with try-with-resources, and don't use exceptions for ordinary control flow.

public class OrderNotFoundException extends RuntimeException {
    public OrderNotFoundException(String id, Throwable cause) {
        super("No order with id " + id, cause);   // keep the root cause
    }
}

Interview questions

Exceptions are a screening-round staple in Indian interviews — checked vs unchecked and try-with-resources appear in nearly every round, so the first questions here carry the most weight:

Beginner What is the exception hierarchy in Java?

Everything throwable descends from Throwable, which has two children: Error (serious JVM conditions) and Exception (application conditions). Under Exception, RuntimeException and its subclasses are unchecked; all other Exception subclasses are checked.

Beginner Difference between checked and unchecked exceptions?

Checked exceptions extend Exception (not RuntimeException) and the compiler enforces the catch-or-specify rule — you must catch them or declare throws. Unchecked exceptions (RuntimeException, Error, and subclasses) are not compiler-enforced. Rule of thumb: recoverable condition → checked; programming bug → unchecked. The single most-asked exceptions question.

Beginner Difference between Error and Exception?

Both extend Throwable. Error signals problems outside the program's control that it normally cannot recover from (OutOfMemoryError, StackOverflowError) — you generally do not catch them. Exception represents conditions an application can reasonably anticipate and handle.

Beginner What does the finally block do, and when does it NOT run?

finally runs on every exit path from the try — normal completion, a caught or uncaught exception, even a return. It does not run only if the JVM stops first: System.exit(), a JVM crash, or the thread being killed. Its classic job was closing resources, now largely handed to try-with-resources.

Beginner Difference between throw and throws?

throw is a statement that actually raises an exception instance (throw new IllegalArgumentException()). throws is a clause in a method signature declaring which checked exceptions the method may propagate, satisfying the catch-or-specify rule for its callers.

Intermediate What is try-with-resources and how does it work?

A try that declares one or more resources in its header. Each resource must implement AutoCloseable; the JVM calls close() automatically at the end of the block — in reverse order of declaration — whether the block finishes normally or throws. It removes the null-checking, boilerplate finally cleanup that leaked resources when written wrongly. Introduced in Java 7.

Intermediate try-with-resources vs a finally block for closing resources?

A hand-written finally needs null checks, must itself be wrapped in try/catch because close() can throw, and — worst — a close() failure there can replace and hide the original exception. Try-with-resources closes in reverse order, needs no null checks, and preserves the primary exception while attaching the close failure as suppressed. Prefer it in essentially all cases.

Intermediate What is multi-catch and what are its constraints?

One catch handling several exception types separated by |: catch (IOException | SQLException e). The listed types must be disjoint — none can be a subclass of another (that would be redundant) — and the parameter e is implicitly final, so you cannot reassign it. It removes duplicated handler bodies. Java 7+.

Intermediate What are suppressed exceptions?

When a try-with-resources body throws and a resource's close() then also throws, the body exception propagates as primary and the close() exception is attached to it via addSuppressed(). You retrieve them later with getSuppressed(). This is exactly the information a manual finally loses.

Intermediate How do you create a custom exception, and checked or unchecked?

Extend Exception for checked or RuntimeException for unchecked, decided by the recoverable-vs-bug rule. Provide the standard constructors, especially (String message, Throwable cause), so callers can chain. Inherit from the most specific existing exception that fits, and only add a custom type when it carries meaning a standard one cannot.

Intermediate What is exception chaining and why pass a cause?

Wrapping a lower-level exception in a higher-level one while passing the original as the cause (via the constructor or initCause()). It lets you raise a meaningful abstraction — say a DataAccessException — without discarding the underlying SQLException and its stack trace, so the root cause is still debuggable.

Pro Can a finally block change what a method returns or throws?

Yes, and that is the trap. A return in finally overrides any value or exception from the try/catch; a throw in finally replaces an in-flight exception and silently discards it. Both are anti-patterns — never return or throw from finally; keep it to cleanup only.

Pro Both the try body and a resource's close() throw — which exception does the caller see?

The try-body (primary) exception propagates to the caller; the close() exception is added to it as a suppressed exception, not lost. If only close() throws, that one propagates normally. Multiple resources close in reverse order, and each close failure after the first is suppressed onto the primary.

Pro Why are checked exceptions criticised, and how do modern frameworks respond?

Critics argue checked exceptions force throws declarations to leak up the call stack, add boilerplate, break lambda/stream signatures, and tempt developers into empty catch blocks that swallow errors. Many modern APIs — Spring's DataAccessException, for instance — wrap checked exceptions into unchecked ones. The counter-argument stands: use checked only when a caller can genuinely recover, otherwise prefer unchecked.
