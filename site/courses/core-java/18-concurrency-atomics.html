Module: Concurrency — Atomics and Synchronizers

When two threads race for the same counter or the same map bucket, a lock is one answer and lock-free compare-and-swap is the faster one — this module maps the java.util.concurrent toolkit that production code leans on and interviewers keep probing.

What this module covers

This module covers the high-level building blocks that sit above raw threads and locks: the atomic classes and the compare-and-swap (CAS) instruction they ride on, the thread-safe concurrent collections led by ConcurrentHashMap, and the synchronizers that coordinate groups of threads (CountDownLatch, CyclicBarrier, Semaphore, Phaser, Exchanger) plus the work-stealing ForkJoinPool. These are the tools that let you write correct concurrent code without hand-rolling synchronized blocks — and ConcurrentHashMap's internals are one of the most-asked topics in Indian backend interviews.

Topics

Atomics (CAS), concurrent collections (ConcurrentHashMap, CopyOnWrite, blocking queues)

Atomic classes such as AtomicInteger, AtomicLong and AtomicReference give you lock-free, thread-safe updates to a single variable. Under the hood a method like incrementAndGet() runs a compare-and-swap loop: read the current value, compute the new one, and atomically install it only if nothing changed in between (compareAndSet), retrying if it did. CAS maps directly to a CPU instruction, so it is usually far cheaper than acquiring a monitor. The package also ships LongAdder/LongAccumulator for counters that are hammered by many writers, and AtomicStampedReference for the ABA problem.

The concurrent collections apply the same philosophy to data structures. ConcurrentHashMap allows fully concurrent, lock-free reads and locks only the individual bucket being written (Java 8+ dropped the old segment design in favour of per-bin synchronization plus CAS), which is why it scales where Hashtable and Collections.synchronizedMap — both single-lock — do not. CopyOnWriteArrayList trades write cost for lock-free iteration by copying its backing array on every mutation, ideal for read-mostly listener lists. The BlockingQueue family (ArrayBlockingQueue, LinkedBlockingQueue) is the backbone of producer–consumer designs: put blocks when full, take blocks when empty, so no busy-waiting is needed.

AtomicInteger hits = new AtomicInteger();
hits.incrementAndGet();                      // lock-free CAS loop under the hood

ConcurrentHashMap<String, Integer> m = new ConcurrentHashMap<>();
m.merge("key", 1, Integer::sum);             // atomic per-bin read-modify-write
m.computeIfAbsent(id, k -> load(k));         // mapping computed at most once

Synchronizers (CountDownLatch / CyclicBarrier / Semaphore / Phaser / Exchanger), ForkJoinPool

Synchronizers coordinate when threads are allowed to proceed rather than protecting shared state. CountDownLatch is a one-shot gate: threads await() until a counter reaches zero, and it cannot be reused. CyclicBarrier is its reusable cousin — a fixed number of parties wait for each other at a rendezvous, then the barrier resets for the next round, with an optional barrier action. Semaphore hands out a bounded number of permits, so it caps how many threads touch a resource at once (a permit count of one gives you a simple mutual-exclusion lock). Phaser generalises the barrier idea to a dynamic party count across multiple phases, and Exchanger lets exactly two threads swap objects at a rendezvous point.

ForkJoinPool is a different animal: an executor built for divide-and-conquer work. Each worker thread keeps its own deque and, when idle, steals tasks from the tail of another worker's deque, keeping cores busy on recursive workloads expressed as RecursiveTask/RecursiveAction. Its commonPool() also backs parallel streams — which is exactly why you must never run blocking I/O on it: a blocked common-pool thread starves every other parallel task in the JVM (use a ManagedBlocker or a dedicated pool instead).

CountDownLatch done = new CountDownLatch(n);          // one-shot gate
for (var task : tasks)
    executor.submit(() -> { work(task); done.countDown(); });
done.await();                                         // block until all n finish

Semaphore permits = new Semaphore(3);                // at most 3 concurrent users
permits.acquire(); try { useResource(); } finally { permits.release(); }

Interview questions

Weighted to Indian backend interviews, where ConcurrentHashMap internals and the latch-vs-barrier distinction are staples from the screening round up:

Beginner What is compare-and-swap (CAS) and how do atomic classes use it?

CAS is a single hardware instruction that atomically sets a memory location to a new value only if it currently holds an expected value, returning whether it succeeded. Atomic classes expose it as compareAndSet(expected, new) and build methods like incrementAndGet() as a retry loop: read, compute, CAS, and try again if another thread won the race. It is lock-free — no thread ever blocks — which usually beats a monitor under moderate contention.

Beginner Why use AtomicInteger instead of a synchronized counter?

A plain int++ is read-modify-write and not atomic, so it loses updates under concurrency. You can fix it with synchronized, but that blocks threads and adds monitor overhead. AtomicInteger.incrementAndGet() gives the same correctness with a lock-free CAS loop — less contention and no risk of forgetting to synchronize a read.

Beginner Are reads from a ConcurrentHashMap thread-safe without locking?

Yes. Retrievals such as get never block and do not lock the table; they read volatile fields so they see a consistent, recent value. Writes lock only the affected bucket. This is why reads can freely overlap writes, unlike Hashtable where every operation contends on one lock.

Beginner What is a CountDownLatch and when do you use it?

A one-shot synchronizer initialised with a count; worker threads call countDown() as they finish and a waiter blocks in await() until the count hits zero. Classic uses: make a main thread wait for N startup tasks to be ready, or release many threads at once from a shared start signal. It cannot be reset — for a reusable barrier you need CyclicBarrier.

Intermediate How does ConcurrentHashMap work internally in Java 8 and later?

It is an array of buckets (bins), like HashMap. An insert into an empty bin is done with a lock-free CAS; if the bin is already occupied it synchronizes on that bin's first node only, so writers to different bins never block each other. Reads are lock-free volatile reads. Bins that grow past a threshold are treeified into red-black trees for O(log n) lookup. The old Java 7 design used a fixed set of segment locks; Java 8 replaced it with this finer-grained per-bin locking, and size() is tracked with a striped counter (the LongAdder idea) to avoid a single hot field.

Intermediate ConcurrentHashMap vs Hashtable vs Collections.synchronizedMap?

All three are thread-safe, but Hashtable and synchronizedMap guard every operation with a single lock, so concurrent access serialises. ConcurrentHashMap locks per bin and never locks the whole table for reads, so it scales with cores. Its iterators are weakly consistent and never throw ConcurrentModificationException, whereas the wrapped-map iterators are fail-fast and need external synchronization while iterating. Prefer ConcurrentHashMap for anything contended.

Intermediate Why does ConcurrentHashMap forbid null keys and values?

Because a get returning null would be ambiguous under concurrency: it could mean "no mapping" or "mapped to null", and you cannot disambiguate with containsKey without a race, since another thread may change the map between the two calls. HashMap tolerates nulls only because the caller can check single-threaded. Disallowing nulls keeps the concurrent contract unambiguous.

Intermediate CountDownLatch vs CyclicBarrier — what's the difference?

A latch counts down to zero once and is done — waiters and counters are usually different threads. A barrier makes a fixed set of parties wait for each other at a common point, then resets and can be reused for the next phase, and it can run a barrier action when the last party arrives. Rule of thumb: latch for "wait until N things happen", barrier for "N threads meet here repeatedly".

Intermediate When would you choose CopyOnWriteArrayList, and what does it cost?

Use it for read-mostly, rarely-mutated shared lists — event listeners, subscriber lists, cached config. Every mutation copies the entire backing array, so writes are O(n) and memory-churny, but reads and iteration are lock-free and its iterator is a stable snapshot that never throws ConcurrentModificationException. If writes are frequent, it is the wrong choice.

Intermediate What problem does Semaphore solve, and how is a binary semaphore different from a lock?

A Semaphore caps concurrent access to a resource by handing out a fixed number of permits — acquire() blocks when none are free, release() returns one. It is the tool for connection pools or rate limits. A binary semaphore (one permit) behaves like a lock but has no ownership: any thread can release a permit, not just the acquirer, so it can also signal between threads — a ReentrantLock must be released by the thread that holds it and is reentrant.

Pro volatile vs an Atomic class — when is volatile not enough?

volatile gives visibility and ordering: a write is seen by later reads on other threads. But it does not make compound actions atomic — volatile int i; i++ is still read-modify-write and can lose updates. Atomic classes add the atomic read-modify-write (via CAS) on top of volatile semantics, so use volatile for a simple published flag and an Atomic type when you also need atomic updates like increment or check-then-set.

Pro What is the ABA problem and how do you handle it?

A CAS only checks that the value equals the expected one, not that it never changed. If another thread moves it A → B → A, your CAS still succeeds even though the state churned underneath — dangerous for lock-free structures reusing nodes. The fix is to version the reference: AtomicStampedReference pairs the value with an integer stamp so a CAS succeeds only if both value and stamp match, defeating the A-then-back-to-A illusion.

Pro LongAdder vs AtomicLong under high contention?

AtomicLong concentrates every writer on one CAS variable, so under heavy contention threads spin retrying the same hot field. LongAdder spreads writes across an array of cells, each updated independently, and sums them only when you call sum(). So it wins for high-throughput counters (metrics, hit counts) that are written far more often than read; for low contention or when you need an exact instantaneous value from a single variable, AtomicLong is simpler.

Pro How does ForkJoinPool work-stealing work, and what is the pitfall of blocking in the common pool?

Each worker has its own double-ended queue: it pushes and pops subtasks from its own head (cache-friendly), and idle workers steal from the tail of a busy worker's deque, so load balances without a central queue. It shines on recursive RecursiveTask/RecursiveAction splits. The trap: the shared commonPool() also runs parallel streams and has a small, CPU-bound thread count, so blocking on I/O or locks inside it starves every other parallel task in the JVM. Do blocking work on a dedicated pool, or wrap it in a ForkJoinPool.ManagedBlocker so the pool can compensate with a spare thread.
