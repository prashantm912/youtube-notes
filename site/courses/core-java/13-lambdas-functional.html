Module: Lambdas and Functional Programming

Lambdas turn behavior into a value you can pass, store, and compose — the shift from writing loops to describing transformations, and the foundation every Stream in the next module stands on.

Knowledge Library → Courses → Core Java → Lambdas and Functional Programming

What this module covers

A lambda (Java 8+) implements a single-abstract-method interface inline, turning a block of behavior into a value you can pass around. This module covers lambda syntax and its capture rules, the method- and constructor-reference shorthand, the ready-made interfaces in java.util.function, and the default methods that compose small lambdas into larger ones. Together these are the groundwork for Streams, Optional, and CompletableFuture.

Topics

Lambda syntax, scoping, and method/constructor references

A lambda is an anonymous implementation of a functional interface — an interface with exactly one abstract method. The syntax is parameters -> body; parameter types are inferred, parentheses are optional for a single parameter, and a single-expression body needs no return or braces. A lambda captures effectively-final local variables and, unlike an anonymous class, its this refers to the enclosing instance, not the lambda itself.

When a lambda does nothing but call an existing method, a method reference (::) says the same thing with less noise. There are four kinds — static (Integer::parseInt), bound to a specific object (out::println), unbound on an arbitrary receiver (String::toUpperCase), and constructor (ArrayList::new). Why it matters: this is the vocabulary every Stream and callback API is written in, and interviewers use the four reference kinds to check that you understand what a lambda desugars to.

Runnable r = () -> System.out.println("run");        // no-arg
Comparator<String> byLen = (a, b) -> a.length() - b.length();
Function<String, Integer> len  = String::length;       // unbound instance ref
Supplier<List<String>>    make = ArrayList::new;        // constructor ref

java.util.function: the 40+ functional interfaces

You rarely need to declare your own functional interface — java.util.function ships more than forty general-purpose ones. Four shapes cover most needs: Supplier<T> (takes nothing, returns a value), Consumer<T> (takes a value, returns nothing), Predicate<T> (takes a value, returns a boolean), and Function<T,R> (maps one type to another). On top of these sit arity variants (BiFunction, BiConsumer, BiPredicate), operator specializations where input and output types match (UnaryOperator, BinaryOperator), and primitive specializations (IntPredicate, ToIntFunction, IntUnaryOperator, …) that skip boxing.

Why it matters: these interfaces are the parameter types of the Streams, Optional, and CompletableFuture APIs, so reading a signature like map(Function<? super T, ? extends R>) is everyday fluency. Interviewers expect you to name the four core shapes and know when a primitive specialization avoids an allocation in a hot loop.

Supplier<LocalDate>      today   = LocalDate::now;
Consumer<String>         print   = System.out::println;
Predicate<String>        isEmpty = String::isEmpty;
Function<String, Integer> length = String::length;
IntBinaryOperator        add     = (a, b) -> a + b;   // primitives, no boxing

Composition (andThen/compose) and effectively-final

The functional interfaces carry default methods that combine small lambdas into bigger ones without nesting calls. Function.andThen(g) runs this function then g; Function.compose(g) runs g first then this — so f.andThen(g) is g(f(x)) while f.compose(g) is f(g(x)). Predicates add and, or, and negate; Consumers chain with andThen; Function.identity() returns its input unchanged. Composition keeps pipelines flat, named, and unit-testable.

The other half of the story is effectively-final capture. A lambda may reference a local variable only if that variable is never reassigned after initialization — final in effect, if not in keyword. Capture happens by value, so the rule keeps the captured copy consistent with the source and rules out the data races that mutable capture would invite. Fields are exempt (they are reached through this), which is why the "one-element array" trick people reach for is a smell, not a pattern.

Function<Integer, Integer> plus1 = x -> x + 1;
Function<Integer, Integer> dbl   = x -> x * 2;
plus1.andThen(dbl).apply(3);   // (3 + 1) * 2 = 8
plus1.compose(dbl).apply(3);   // (3 * 2) + 1 = 7

int factor = 10;               // effectively final
Function<Integer, Integer> scale = x -> x * factor;   // captured by value

Interview questions

Most-asked lambda and functional-programming questions in Indian interviews — a screening staple from service companies to product-company deep dives:

Beginner What is a lambda expression and what is a functional interface?

A lambda is a concise, anonymous implementation of a functional interface — an interface with exactly one abstract method (a SAM type). The syntax is parameters -> body. Its target type is the functional interface it is assigned to, which the compiler infers from context.

Beginner Is @FunctionalInterface required to use a lambda?

No. Any interface with a single abstract method can be a lambda target (Runnable, Comparator, etc.). The annotation is optional; it makes the compiler enforce the single-abstract-method rule and documents intent. default and static methods do not count toward the one abstract method.

Beginner Name the four core interfaces in java.util.function and their shapes.

Supplier<T> — takes nothing, returns T. Consumer<T> — takes T, returns nothing. Predicate<T> — takes T, returns boolean. Function<T,R> — takes T, returns R. Everything else is an arity variant (Bi-), an operator specialization (UnaryOperator/BinaryOperator), or a primitive specialization.

Beginner What are the four kinds of method reference?

Static (Integer::parseInt), bound to a specific object (System.out::println), unbound on an arbitrary receiver of the type (String::toUpperCase), and constructor (ArrayList::new). Each is shorthand for a lambda that only calls that method.

Beginner Lambda vs anonymous inner class — key differences?

A lambda only implements a functional interface, holds no state, and its this is the enclosing instance. An anonymous class creates its own instance with its own this, can have fields and implement multiple methods or a class type. Use a lambda for pure behavior; an anonymous class when you need state or several overrides.

Intermediate What does "effectively final" mean and why does a lambda require it?

A local variable is effectively final if it is never reassigned after initialization — you could add final without a compile error. Lambdas capture locals by value; requiring effectively-final keeps the captured copy consistent with the original and prevents the concurrency bugs mutable capture would cause. Instance/static fields are exempt because they are reached through this.

Intermediate andThen vs compose on Function — which runs first?

f.andThen(g) runs f first, then feeds its result to g — i.e. g(f(x)). f.compose(g) runs g first — i.e. f(g(x)). Same values, opposite order; the classic trick question is to trace both on a small input.

Intermediate Why do primitive specializations like IntPredicate and ToIntFunction exist?

Generic interfaces only work over reference types, so Function<Integer,Integer> boxes every int. Primitive specializations operate on int/long/double directly, avoiding autoboxing allocations and unboxing — meaningful in hot Stream pipelines processing millions of elements.

Intermediate What does this refer to inside a lambda?

The enclosing instance — a lambda does not introduce a new scope, so this, method calls, and field access resolve against the class that contains it. In an anonymous class, this refers to the anonymous object itself and shadows the outer instance. This is a frequent source of confusion when migrating anonymous classes to lambdas.

Intermediate How do you combine predicates?

Predicate has default methods and, or, and negate: isPositive.and(isEven), p.or(q), p.negate(). They short-circuit like &&/||. The static Predicate.not(...) (Java 11) negates a method reference cleanly, e.g. filter(Predicate.not(String::isBlank)).

Pro Can a lambda mutate a captured local variable? What do people do instead, and why is it discouraged?

No — the captured local must be effectively final, so you cannot reassign it. Common workarounds hold state in a one-element array (int[] holder) or an AtomicInteger. They are discouraged because they smuggle shared mutable state past the safety the rule provides and break under parallel streams; the correct tools are reduction, collectors, or an accumulator returned from the pipeline.

Pro Are lambdas objects, and how are they implemented differently from anonymous classes at the bytecode level?

At runtime a lambda is an instance of its functional interface, but it is not compiled to a separate .class file. javac emits an invokedynamic instruction whose bootstrap, LambdaMetafactory, lazily builds the implementation class on first use and can reuse one shared instance for a non-capturing lambda. An anonymous class, by contrast, is a distinct class file loaded eagerly with a new instance each time — so lambdas have a lower static footprint and let the JVM pick the translation strategy.
