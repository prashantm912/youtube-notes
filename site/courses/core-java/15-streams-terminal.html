Module: Streams — Terminal Operations and Collectors

A stream does nothing until a terminal operation pulls the data through. This module covers where pipelines land — the collect/reduce/find family, the full Collectors toolbox, Java 24's stream gatherers, and the parallel-execution machinery you should reach for only after measuring.

What this module covers

Intermediate operations build the pipeline lazily; the terminal operation is what finally runs it and produces a result or side effect. This module orients you on the three ways streams end and combine: the terminal-operation family (reduce, collect, count, find, match), the Collectors factory that powers mutable reductions like grouping and partitioning, the new Gatherer API for custom intermediate steps, and primitive and parallel streams built on the Spliterator. Depth lives in the linked official docs — this page maps the territory and the interview ground.

Topics

Terminal ops, Collectors (groupingBy/partitioningBy/teeing/joining)

A terminal operation consumes the stream and yields something concrete: a value (count, reduce, min/max), an optional element (findFirst, findAny), a boolean (anyMatch, allMatch, noneMatch), a side effect (forEach), or a container via collect and toList. Once a terminal op runs the pipeline is spent — a stream is single-use. The general form reduce needs an identity plus an associative accumulator, but for anything richer than a single scalar you reach for collect and a Collector.

Why it matters: Collectors is where most real aggregation happens, and groupingBy with a downstream collector is the single most-asked Streams idiom in Indian interviews — group employees by department, then counting(), summingInt(), or mapping(...).collect(joining()) the groups. Round it out with partitioningBy (a two-key Boolean split), joining (delimiter/prefix/suffix), toMap, and teeing (fork every element to two collectors, then merge — Java 12).

// static import java.util.stream.Collectors.*
Map<Dept, Long> headcount = employees.stream()
    .collect(groupingBy(Employee::dept, counting()));

Map<Boolean, List<Employee>> parts = employees.stream()
    .collect(partitioningBy(e -> e.salary() > 100_000)); // both keys always present

String names = employees.stream()
    .map(Employee::name)
    .collect(joining(", ", "[", "]"));

Stream gatherers (Stream::gather, built-in Gatherers)

A gatherer (java.util.stream.Gatherer) is a user-defined intermediate operation — the intermediate-op counterpart to what Collector is for terminal ops — plugged into a pipeline with Stream::gather. It has up to four parts: an initializer (per-run state), an integrator (called per element, may emit zero or more downstream elements and may short-circuit), a combiner (for parallel evaluation), and a finisher (flush at end of stream). That shape lets gatherers transform elements one-to-one, one-to-many, many-to-one, or many-to-many, remember earlier elements to shape later ones, and even turn an infinite stream finite.

Why it matters: before gatherers, custom intermediate logic — sliding windows, running totals, dedupe-by-key, bounded concurrency — forced clumsy stateful lambdas or an escape from the pipeline entirely. The built-in Gatherers factory ships windowFixed, windowSliding, fold, scan, and mapConcurrent (concurrent mapping on virtual threads that preserves order). Gatherers finalized in JDK 24 via JEP 485 (previewed in 22 and 23), so on any current-LTS interview "what problem do gatherers solve, and how do they differ from collectors?" is fair game.

// windowSliding: many-to-many, remembers prior elements
List<List<Integer>> windows =
    Stream.of(1, 2, 3, 4, 5).gather(Gatherers.windowSliding(2)).toList();
// [[1, 2], [2, 3], [3, 4], [4, 5]]

// scan: a running prefix reduction (each intermediate result is emitted)
List<String> running =
    Stream.of(1, 2, 3, 4).gather(Gatherers.scan(() -> "", (s, n) -> s + n)).toList();
// ["1", "12", "123", "1234"]

Primitive streams, parallel streams + Spliterator, when NOT to parallelize

Primitive streams — IntStream, LongStream, DoubleStream — avoid the cost of boxing every element and add numeric terminal ops: sum, average, min, max, and summaryStatistics (all five in one pass). You cross over with mapToInt/mapToObj/boxed. Parallel streams (parallelStream() or .parallel()) split work across the shared common ForkJoinPool by recursively decomposing the source; the engine that does the splitting is the Spliterator (a "splittable iterator") whose trySplit hands off a chunk and whose characteristics — SIZED, SUBSIZED, ORDERED, IMMUTABLE — tell the framework how evenly it can divide and whether order and size are known up front.

Why it matters: parallel is not free, and "when would you NOT parallelize?" is a senior-round favorite. Splitting, thread hand-off, and merging all cost, so a parallel stream is often slower than a sequential one. Favor it only for large element counts with cheaply, evenly splittable sources (arrays, ArrayList, range-based IntStream) doing CPU-bound, stateless, associative, side-effect-free work — and always measure. Avoid it for small data, I/O-bound tasks, ordered or stateful operations, poorly splitting sources (LinkedList, iterate), or shared mutable state. Remember too that the whole JVM shares one common pool, so a blocking parallel stream can starve every other one.

IntSummaryStatistics stats = employees.stream()
    .mapToInt(Employee::salary)   // primitive stream: no boxing
    .summaryStatistics();         // count, sum, min, average, max together

long hits = LongStream.rangeClosed(2, 1_000_000)
    .parallel()                   // splits across the common ForkJoinPool
    .filter(MyMath::isPrime)
    .count();                     // profile before assuming this beats sequential

Interview questions

Most-asked terminal-ops and Collectors questions in Indian interviews — streams are a screening staple and the collector idioms recur in every product-company round:

Beginner What is a terminal operation, and how does it differ from an intermediate one?

A terminal operation triggers the pipeline and produces a result or side effect (collect, reduce, count, forEach, findFirst, anyMatch). It consumes the stream, so a stream is single-use. Intermediate operations (map, filter) are lazy, return a new Stream, and do nothing until a terminal op runs.

Beginner What does collect() do, and what is a Collector?

collect performs a mutable reduction — it accumulates elements into a container. A Collector is the recipe: a supplier (new container), an accumulator (add one element), a combiner (merge two containers, used in parallel), and an optional finisher. The Collectors factory supplies ready-made ones like toList, groupingBy, and joining.

Beginner Stream.toList() vs Collectors.toList() — any difference?

Yes. Stream.toList() (Java 16) returns an unmodifiable list. collect(Collectors.toList()) returns a mutable list with no guarantee about the exact type or mutability. Prefer Stream.toList() unless you specifically need to mutate the result.

Beginner What is a primitive stream and why use one?

IntStream/LongStream/DoubleStream hold primitives directly, avoiding autobox/unbox overhead, and add numeric terminals — sum, average, min, max, summaryStatistics. Convert with mapToInt going in and boxed/mapToObj coming out.

Beginner What are the three forms of reduce()?

reduce(accumulator) returns an Optional; reduce(identity, accumulator) returns a plain value; reduce(identity, accumulator, combiner) adds a combiner for parallel streams (and when input and result types differ). The accumulator must be associative and the identity must be a true identity for correct parallel results.

Intermediate Show groupingBy with a downstream collector.

The second argument post-processes each group: groupingBy(Employee::dept, counting()) gives headcount per department; groupingBy(Employee::dept, summingInt(Employee::salary)) totals salary; groupingBy(Employee::dept, mapping(Employee::name, joining(", "))) concatenates names. A word-frequency map is just groupingBy(identity(), counting()).

Intermediate partitioningBy vs groupingBy?

partitioningBy(predicate) always returns a Map<Boolean, ...> with both true and false keys present, even if one side is empty. groupingBy(classifier) creates a key per distinct classifier value, and keys with no elements simply don't appear. Use partitioning for a yes/no split, grouping for many buckets.

Intermediate What is the teeing collector?

teeing(downstream1, downstream2, merger) (Java 12) feeds every element to two collectors and combines their results with a BiFunction in a single pass — for example computing an average and a count together, or min and max, without streaming the data twice.

Intermediate How does a parallel stream execute, and which pool runs it?

It recursively decomposes the source via Spliterator.trySplit, runs the leaf chunks on the shared common ForkJoinPool (with work-stealing), and merges results with the combiner. Default parallelism is roughly the number of available processors minus one — every parallel stream in the JVM shares this one pool unless you submit the pipeline inside your own ForkJoinPool.

Intermediate What is a Spliterator and why does it affect parallel performance?

A Spliterator is a splittable iterator: trySplit hands off a portion of the elements for another thread to process. Its characteristics — SIZED, SUBSIZED, ORDERED, IMMUTABLE — tell the framework how well and how evenly it can split. Arrays and ArrayList split cleanly in half; LinkedList and iterate-based streams split poorly, which cripples parallel speedup.

Intermediate What are stream gatherers, and how do they relate to collectors?

Gatherers (Stream::gather with a Gatherer) are custom intermediate operations — the intermediate-op analogue of a Collector, which is terminal. They can be one-to-many or many-to-many, keep state across elements, and short-circuit. Built-ins include windowFixed, windowSliding, fold, scan, and mapConcurrent. Finalized in JDK 24 (JEP 485).

Pro When should you NOT use a parallel stream?

When the dataset is small, the per-element work is cheap, or the task is I/O-bound (threads block, no speedup); when the source splits poorly (LinkedList, iterate); when the pipeline is order-sensitive, stateful, or short-circuits early; and whenever there is shared mutable state (races). Also beware blocking calls inside a parallel stream — they tie up the common pool and starve other parallel work. Default to sequential and switch only after profiling proves a win.

Pro Why can findFirst be more expensive than findAny on a parallel ordered stream?

findFirst must honor encounter order, so it cannot just return the first match any thread happens to find — it has to coordinate to identify the earliest one. findAny drops that constraint and returns whatever a thread finds first, which is cheaper in parallel. The same order-vs-speed trade-off separates forEach (unordered, faster) from forEachOrdered.

Pro What must a reduction satisfy to be correct in parallel?

The identity must be a genuine identity (combine(identity, x) equals x), and the accumulator and combiner must be associative and mutually consistent; the behavioral parameters must be stateless, non-interfering, and side-effect-free. Violate these and parallel results become nondeterministic while sequential ones look fine. For concurrent collection into a map, use groupingByConcurrent with an unordered stream so threads can merge into one shared container safely.
