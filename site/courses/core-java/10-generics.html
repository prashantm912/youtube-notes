Module: Generics

Generics move type checks from a runtime cast to a compile-time promise — and the two things interviewers push hardest on, wildcards and erasure, are exactly the parts the syntax hides from you.

Knowledge Library → Courses → Core Java → Generics

What this module covers

Generics (Java 5+) parameterize classes, interfaces and methods by type, so a List<String> rejects an Integer at compile time instead of throwing ClassCastException later. This module orients you on three layers: declaring and bounding type parameters, the wildcard system and the PECS rule that governs it, and type erasure — the way generics actually compile, which explains most of their surprising limits. Depth lives in the linked official docs; this page is the map and the interview drill.

Topics

Type parameters, bounded types, and generic methods

A type parameter is a placeholder — class Box<T> — filled in at use as Box<String>, letting one class serve every element type with full compile-time checking and no casts. Conventional single letters carry meaning: E element, K/V key/value, T type, N number. A bounded parameter (<T extends Comparable<T>>) restricts the argument and, in return, lets you call the bound's methods on T; multiple bounds use & with the class first.

Why it matters: a generic method declares its own parameters in angle brackets before the return type, so it is independent of any class-level generics and usually infers its types at the call site. Understanding that a generic method's <T> is scoped to the method — not the class — is a common point interviewers probe to see whether you actually understand parameterization or just copy the syntax.

class Box<T> {                      // type parameter on the class
    private T value;
    void set(T v) { this.value = v; }
    T get()       { return value; }
}
static <T extends Comparable<T>> T max(T a, T b) { // generic method + bound
    return a.compareTo(b) >= 0 ? a : b;             // bound enables compareTo
}

Wildcards and PECS

A type parameter T names a type once and reuses it; a wildcard ? stands for an unknown type at a single use site. <? extends T> (upper bounded) is a read-only "producer" — you can take out values as T but cannot add any (the compiler cannot know the real element type), while <? super T> (lower bounded) is a write-friendly "consumer" — you can put in T and its subtypes but reads come back as Object. An unbounded <?> means "some type, don't care".

Why it matters: PECS — Producer Extends, Consumer Super — is the single most-tested generics rule in Indian interviews. It is the reasoning behind the JDK's own signatures (Collections.copy, Collections.addAll) and the reason List<? extends Number> lets you sum a list of Integer but not add to it. Knowing when NOT to use a wildcard (a value that is both read and written) is the senior-level follow-up.

// dest CONSUMES, so super; src PRODUCES, so extends
static <T> void copy(List<? super T> dest, List<? extends T> src) {
    for (T t : src) dest.add(t);
}
List<? extends Number> nums = List.of(1, 2, 3);
Number n = nums.get(0);   // OK — producer, read as Number
// nums.add(4);           // compile error — can't add to a producer

Type erasure, raw types, and bridge methods

Generics are a compile-time feature: after type checking, the compiler erases them — every type parameter is replaced by its leftmost bound (or Object if unbounded) and casts are inserted where needed, so List<String> and List<Integer> are the same class List at runtime. This is why new T[n], T.class, and obj instanceof List<String> are illegal, why a method cannot be overloaded on List<String> vs List<Integer> (identical erased signatures), and why generics carry zero runtime cost. A raw type (List with no parameter) is the pre-generics escape hatch that turns off these checks — legal for backward compatibility, but it invites the very ClassCastException generics exist to prevent.

Why it matters: when a generic class is subclassed, erasure can leave signatures that no longer line up, so the compiler quietly synthesizes bridge methods to preserve overriding and polymorphism — they surface only in stack traces and reflection, which confuses people who have never heard of them. Erasure and its consequences are, alongside PECS, the deepest generics material Indian interviewers reach for.

List<String>  a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
a.getClass() == b.getClass();   // true — both are just ArrayList at runtime
// void p(List<String> s) {}    // these two cannot coexist:
// void p(List<Integer> i) {}   // 'name clash' — same erasure p(List)

Interview questions

The generics questions asked most in Indian interviews — wildcards/PECS and type erasure dominate the intermediate and senior rounds:

Beginner What are generics and what problem do they solve?

A way to parameterize types (Java 5+). They move type checking to compile time, so containers and APIs work with a specific type without casts and without the risk of a runtime ClassCastException. Fewer casts, safer code, reusable algorithms.

Beginner What do the conventional letters T, E, K, V, N mean?

Just naming conventions for type parameters: T type, E element (used in collections), K and V key and value (maps), N number, S/U for additional types. Any identifier is legal; these read better.

Beginner What is a bounded type parameter?

<T extends Number> restricts the argument to Number or its subtypes and, crucially, lets you call Number's methods on T. Multiple bounds use & (<T extends A & B>) and a class bound, if any, must come first. extends covers both classes and interfaces here.

Beginner What is a generic method and how is it different from a method inside a generic class?

A generic method declares its own type parameter list before the return type (static <T> T first(List<T> l)). Its T is scoped to that method and inferred per call, independent of any class-level type parameter — so a non-generic class can still have generic methods, and static methods (which cannot use the class's T) must declare their own.

Intermediate Difference between a type parameter T and a wildcard ?

T names a type you can reuse and relate across the signature (parameters, return type, other bounds). ? is an anonymous unknown for a single use site where you never need to name the type — you cannot use ? as a return type you build on, or tie two ? together. Rule of thumb: need the type more than once → type parameter; need it once and only to read/pass → wildcard.

Intermediate Explain PECS. When do you use <? extends T> vs <? super T>?

Producer Extends, Consumer Super. If a structure only produces values for you (you read from it), use <? extends T>. If it only consumes values (you write into it), use <? super T>. In copy(dest, src), src is ? extends T and dest is ? super T. If it both produces and consumes, use an exact type T — no wildcard.

Intermediate Why can't you add an element to a List<? extends Number>?

The list's real type could be List<Integer> or List<Double> — the compiler only knows it is "some subtype of Number". Adding an Integer might corrupt a List<Double>, so add is disallowed (except literal null). You can still read elements as Number. That read-only nature is exactly what "producer extends" means.

Intermediate What is type erasure?

The compiler checks generic types, then removes them: each type parameter becomes its leftmost bound (or Object), casts are inserted, and bridge methods are generated. So generic type information does not exist at runtime — List<String> and List<Integer> are both plain List. It kept generics backward-compatible with pre-5 bytecode and free of runtime overhead.

Intermediate Because of erasure, which operations on a type parameter are illegal?

No new T(), no new T[n], no T.class, no obj instanceof List<String>, no catch (MyException<T> e), and no static field of type T. All need runtime type info that erasure removed. Workarounds pass a Class<T> token or use reflection.

Intermediate What are raw types and why are they discouraged?

A raw type is a generic class used with no type argument, e.g. List list = new ArrayList();. It exists for compatibility with pre-generics code but disables generic type checking, produces unchecked warnings, and reopens the door to ClassCastException. Use the parameterized type, or <?> when the type is genuinely unknown.

Pro What are bridge methods and why does the compiler generate them?

When you extend a generic type, erasure can leave the override with a different signature than the parent's erased method (e.g. setData(Integer) vs erased setData(Object)). The compiler emits a synthetic bridge method setData(Object) that casts and delegates, so polymorphism and overriding still work. They are marked synthetic/bridge and normally only appear in stack traces or reflection.

Pro Can print(List<String>) and print(List<Integer>) be overloads in the same class? Why not?

No — "name clash". After erasure both become print(List), so they have identical signatures and the class will not compile. This is a direct, classic consequence of erasure; the fix is different method names or a different parameter shape.

Pro What is heap pollution, and how do non-reifiable varargs and @SafeVarargs relate?

Heap pollution is when a variable of a parameterized type points to an object of a different type, made possible because erasure lets unchecked operations slip through — later surfacing as ClassCastException. Generic varargs (T...) create a non-reifiable T[], so the compiler warns about potential pollution; @SafeVarargs on a method that only reads the varargs (never stores or exposes the array) suppresses that warning as an author's guarantee.

Pro If erasure removes generic info, how can frameworks still read type arguments at runtime?

Erasure removes it from instances, but type arguments baked into class metadata survive: a superclass/field/method signature like class StringList extends ArrayList<String> keeps String reachable via reflection (getGenericSuperclass, ParameterizedType). That is the "super type token" trick behind Spring's ParameterizedTypeReference and Jackson's TypeReference. Bare instances still cannot report their own <T>.
