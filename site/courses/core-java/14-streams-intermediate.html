Module: Streams — Pipeline and Intermediate Ops

A stream is not a collection — it is a lazy assembly line that does nothing until a terminal operation pulls data through it. This module builds the pipeline mental model and walks the intermediate toolbox, then points you at the docs that go deep. It is one of the most heavily tested topics in Indian Java interviews.

What this module covers

The Stream API (Java 8+) processes in-memory data with a map/filter/reduce style declarative pipeline instead of hand-written loops. This module covers the two foundations you must own before terminal operations and collectors make sense: the pipeline model — source, intermediate operations, one terminal operation — with the lazy evaluation that makes it efficient, and the intermediate operations themselves (map, filter, flatMap, mapMulti, sorted, distinct, limit, takeWhile and friends), including the crucial stateless-versus-stateful distinction. Streams are a screening-round staple in Indian interviews, so both topics carry interview weight below.

Topics

Stream pipeline model, lazy evaluation

A stream is a view over a source (a collection, array, generator or I/O channel), not a data structure — it stores no elements and computes nothing on its own. A pipeline has three parts: a source, zero or more intermediate operations (each returns a new stream, so they chain), and exactly one terminal operation (returns a value or void and triggers execution). Intermediate operations are lazy: calling filter or map builds up a recipe but processes no data — without a terminal operation the pipeline does nothing at all. When the terminal operation runs, elements are pulled through the whole chain one at a time (vertical fusion), not layer by layer, which is why side-effect ordering behaves as it does and why short-circuiting operations such as limit, findFirst and anyMatch can stop early and even consume infinite streams.

Why it matters: the single most common stream misconception is expecting an intermediate chain to "run" without a terminal operation. Interviewers use laziness, single-use semantics (a stream throws IllegalStateException if reused), and short-circuiting to separate people who memorised the method names from people who understand the execution model.

// Lazy: this line does NOTHING — no terminal operation.
Stream<String> pipe = names.stream().filter(s -> s.length() > 3);

// Now a terminal op pulls elements through, one at a time:
Optional<String> first = names.stream()
    .filter(s -> s.length() > 3)   // evaluated only until...
    .map(String::toUpperCase)
    .findFirst();                  // ...the first match — short-circuits

Intermediate ops (map/filter/flatMap/mapMulti/sorted/distinct/limit/takeWhile)

Intermediate operations are the transforms you compose between source and terminal op. Split them two ways. By shape: map transforms one-to-one, filter keeps elements matching a predicate, flatMap flattens a one-to-many mapping (each element becomes a stream that is spliced in — the fix for a would-be Stream<Stream<T>>), and mapMulti (Java 16) does the same one-to-many job by pushing results into a consumer instead of allocating a stream per element — preferable when most inputs produce zero or one output. By state: stateless operations (map, filter, flatMap, mapMulti, peek) process each element independently, while stateful ones must see other elements — sorted and distinct buffer input (and never terminate on an infinite stream), and limit/skip/takeWhile/dropWhile depend on encounter position. takeWhile and dropWhile (Java 9) act on the ordered prefix: takeWhile stops at the first element failing the predicate (unlike filter, which tests every element).

Why it matters: "map vs flatMap", "stateless vs stateful", and "takeWhile vs filter" are guaranteed questions. Interviewers also probe whether you understand that sorted/distinct are performance-sensitive barriers and that limit short-circuits — the difference between writing a pipeline and writing an efficient one.

List<String> result = orders.stream()
    .filter(o -> o.isPaid())              // stateless: keep some
    .flatMap(o -> o.items().stream())     // one order -> many items
    .map(Item::sku)                       // stateless: transform 1:1
    .distinct()                           // stateful: buffers seen values
    .sorted()                             // stateful: buffers, then orders
    .limit(10)                            // short-circuits the pipeline
    .toList();

// takeWhile stops at the first non-match; filter would test all:
Stream.of(2, 4, 6, 7, 8).takeWhile(n -> n % 2 == 0); // -> 2, 4, 6

Interview questions

The most-asked stream pipeline and intermediate-operation questions in Indian interviews — laziness and map-vs-flatMap dominate the screening rounds, so they lead here:

Beginner What is a Stream, and how is it different from a Collection?

A stream is a sequence of elements supporting aggregate operations — a view over a source, not a data structure. A collection stores elements and is about efficient access; a stream computes over them and stores nothing. Streams are also lazy, single-use (consumed once), can be sequential or parallel, and support a declarative map/filter/reduce style instead of external for loops.

Beginner What are the three parts of a stream pipeline?

A source (collection, array, Stream.of, generator, I/O), zero or more intermediate operations (each returns a new stream, so they chain — filter, map, sorted...), and exactly one terminal operation (collect, forEach, count, reduce...) that produces a result or side effect and triggers execution.

Beginner How do you tell an intermediate operation from a terminal one?

By return type. An intermediate operation returns a Stream (so you can keep chaining) and is lazy. A terminal operation returns something other than a stream — a value, a collection, an Optional, or void — and it is what actually runs the pipeline. A pipeline may have many intermediate ops but only one terminal op.

Beginner What does it mean that intermediate operations are "lazy"?

Calling an intermediate operation builds up the pipeline but processes no elements. Nothing happens until a terminal operation is invoked. So list.stream().filter(...).map(...) with no terminal op does literally nothing — a common bug and a favourite trick question. Laziness lets the runtime fuse operations and short-circuit.

Beginner map vs filter?

map transforms every element one-to-one, producing a stream of the same length but a possibly different type (map(String::length) turns a Stream<String> into a Stream<Integer>). filter takes a predicate and keeps only matching elements, so the type is unchanged and the stream is the same length or shorter.

Intermediate map vs flatMap — the difference that matters?

Both transform elements, but the mapper differs. map's function returns a single value, giving one output per input. flatMap's function returns a stream per element, and flatMap splices (flattens) all those streams into one. Use flatMap for one-to-many — flattening a list of orders into their items, or a stream of lists into a single stream. Without it you would get a Stream<Stream<T>> or Stream<List<T>>.

Intermediate Stateless vs stateful intermediate operations?

Stateless ops (map, filter, flatMap, mapMulti, peek) process each element without knowledge of the others, so they stream through with constant memory. Stateful ops need to see beyond the current element: sorted and distinct must buffer input, and limit/skip/takeWhile/dropWhile depend on position. Stateful ops cost more memory and can hurt parallel performance; sorted/distinct on an infinite stream never terminate.

Intermediate takeWhile vs filter, and takeWhile vs dropWhile?

filter tests every element and keeps all matches. takeWhile (Java 9) keeps the leading run of matching elements and stops at the first that fails — so on 2, 4, 7, 8 with "is even", filter yields 2, 4, 8 but takeWhile yields 2, 4. dropWhile is the complement: it discards the leading matching run and keeps everything from the first failure onward. Both act on the ordered prefix, so they are most meaningful on ordered streams.

Intermediate What is short-circuiting, and which operations short-circuit?

A short-circuiting operation can produce a result without processing the whole stream. The short-circuiting intermediate op is limit (and takeWhile); short-circuiting terminals include findFirst, findAny, anyMatch, allMatch and noneMatch. They allow a pipeline over an infinite stream (e.g. Stream.iterate) to terminate, and they let ordinary pipelines stop early once the answer is known.

Intermediate Can you reuse a stream? What happens if you try?

No — a stream is single-use. Once a terminal (or even certain intermediate) operation has been invoked, operating on it again throws IllegalStateException: stream has already been operated upon or closed. If you need to traverse the same source twice, create a fresh stream each time, typically from a Supplier<Stream<T>> or from the source collection.

Intermediate When would you prefer mapMulti over flatMap?

mapMulti (Java 16) replaces each element with zero or more results by pushing them into a Consumer, instead of allocating and returning a Stream per element like flatMap. Prefer it when most elements produce zero or one output (so a per-element stream is wasteful), or when the results are easier to emit imperatively than to wrap in a stream. For genuine one-to-many where a stream already exists, flatMap is usually clearer.

Intermediate Write a pipeline: from a list of words, the distinct lengths greater than 3, sorted descending.

words.stream().map(String::length).filter(n -> n > 3).distinct().sorted(Comparator.reverseOrder()).toList(); — this exercises a stateless transform, a filter, then two stateful ops. A good follow-up is why distinct before sorted can be cheaper (fewer elements to sort).

Pro How are elements pushed through the pipeline — all of filter, then all of map, or one element at a time?

One element at a time (vertical fusion), not stage by stage. A single element flows through filter then map then the terminal before the next element starts — unless a stateful barrier like sorted or distinct forces buffering of the input first. This element-at-a-time model is why laziness and short-circuiting work, why peek ordering looks interleaved, and why an infinite source with a limit can complete.

Pro What is peek for, and why is relying on it in production risky?

peek is a stateless intermediate op intended for debugging — it runs a Consumer on each element as it passes without changing the stream. It is risky for real logic because it is only executed when elements actually flow: with a short-circuiting or size-optimised terminal (for example count(), which since Java 9 may skip traversal entirely when the size is known), peek may run for fewer elements than you expect, or not at all. Never use peek for side effects your program depends on.

Pro How do you create and safely consume an infinite stream?

Generate one with Stream.iterate(seed, next) or Stream.generate(supplier). To make it terminate you must apply a short-circuiting operation — limit(n), takeWhile(predicate) (Java 9), or a short-circuiting terminal such as findFirst. Crucially, a stateful op that buffers the whole input (sorted, distinct) or a non-short-circuiting terminal (count, forEach) on an infinite stream will run forever — the three-argument Stream.iterate(seed, hasNext, next) added in Java 9 lets you bound it at the source instead.
