Module: Memory and Garbage Collection

Java frees memory for you — but interviews probe the machinery underneath: which objects survive, which collector runs and why, and how a "garbage-collected" language still manages to leak.

Knowledge Library → Courses → Core Java → Memory and Garbage Collection

What this module covers

Automatic memory management is Java's headline feature, and this module maps its three interview-critical layers: the object lifecycle and the reference-strength API (SoftReference, WeakReference, PhantomReference, Cleaner) that lets you weaken the collector's grip on purpose; the collectors themselves (Serial, Parallel, G1, ZGC, Shenandoah) and the throughput-versus-pause trade-off between them; and the practical craft of tuning flags, reading OutOfMemoryError messages, and hunting down a leak. GC algorithms and memory leaks are among the most-asked topics in Indian back-end interviews.

Topics

Object lifecycle and reference strength: strong, soft, weak, phantom, Cleaner

An object lives exactly as long as it is reachable — traceable through a chain of references from a GC root (live thread stacks, static fields, JNI handles). When the last such path disappears the object becomes unreachable and eligible for collection; you never call free. The java.lang.ref package lets you deliberately loosen that grip: a SoftReference is retained until the heap is under memory pressure (memory-sensitive caches), a WeakReference is cleared at the next GC once no strong or soft path remains (canonicalizing maps, WeakHashMap keys), and a PhantomReference never hands the object back but signals precisely when it has been reclaimed.

Why it matters: reference strength is how you cache without leaking, and how you replace finalize(). finalize() is deprecated for removal — unpredictable timing, it can resurrect objects, and it burdens the collector — so Cleaner (built on phantom references and a daemon thread) is the modern way to release native resources, paired with AutoCloseable for the explicit path and acting as a safety net when close() is forgotten. "Which reference type would you pick for a cache, and why?" is a routine screening question.

Reference<byte[]> cache  = new SoftReference<>(payload);  // survives until memory is tight
Reference<Node>   canon  = new WeakReference<>(node);      // cleared at next GC if only weakly reachable

Cleaner cleaner = Cleaner.create();
cleaner.register(this, state);   // state.run() executes once 'this' is phantom-reachable

The collectors: Serial, Parallel, G1, ZGC, Shenandoah

HotSpot ships several collectors that make the same generational bet — most objects die young — but strike a different balance between throughput and pause time. Serial (one thread, stop-the-world) fits tiny heaps and single-CPU containers. Parallel, the throughput collector, throws many threads at GC to minimise total collection time — ideal for batch work that tolerates pauses. G1 (Garbage-First, the default since Java 9) divides the heap into equal-sized regions and collects the ones with the most garbage first, chasing a soft pause-time goal (-XX:MaxGCPauseTimeMillis) while doing much of its work concurrently — the sensible default for typical services. ZGC and Shenandoah are the low-latency specialists: they perform nearly all their work concurrently with the application and keep pauses under a millisecond even on multi-terabyte heaps, trading some throughput and footprint for it.

Why it matters: interviewers rarely want barrier-level internals — they want the trade-off named (throughput versus pause time), the default identified (G1), and a switch justified (ZGC for a latency-sensitive service with a large heap; Parallel for a nightly ETL job). The young/old generation split and the difference between a minor GC and a major/full GC underpin every one of these answers.

java -XX:+UseG1GC       -Xms2g -Xmx2g   App   # G1: balanced pauses, the default
java -XX:+UseZGC        -Xmx16g          App   # ZGC: sub-millisecond pauses, huge heaps
java -XX:+UseParallelGC -Xmx4g           App   # Parallel: maximum throughput, batch jobs
java -XX:+UseSerialGC                    App   # Serial: tiny heaps, single CPU

Tuning flags, OutOfMemoryError types, and leak diagnosis

Most tuning is two decisions: pick a collector and size the heap. -Xms and -Xmx set initial and maximum heap — set them equal in production so leaks surface quickly and the JVM never pauses to resize; -Xss sizes thread stacks; -Xlog:gc* prints GC activity so you can watch the live set. When the collector cannot reclaim enough room you get an OutOfMemoryError, and the message is the diagnosis: Java heap space (heap exhausted — a real leak or an undersized heap), GC overhead limit exceeded (over 98% of time spent in GC recovering under 2% of the heap), Metaspace (too many loaded classes — often a classloader leak), and unable to create new native thread (an OS or native limit, not the heap).

A leak in a managed runtime means objects stay reachable but useless. The classic causes: unbounded caches and collections, static references that never release, listeners or callbacks never unregistered, ThreadLocals left set on pooled threads, and classloader leaks in application servers. Diagnosis is empirical — watch the live set (heap occupancy after a full GC) trend upward over time, capture a heap dump (-XX:+HeapDumpOnOutOfMemoryError, or jcmd/jmap on demand), and follow the dominator tree and "leak suspects" in Eclipse MAT back to the retaining GC root. Walking this end-to-end story convincingly is a favourite senior-round question.

-Xms512m -Xmx512m                 # fix the heap so a leak surfaces fast
-XX:+HeapDumpOnOutOfMemoryError   # write a heap dump at the moment of the crash
-XX:HeapDumpPath=/var/log/app     # where the dump lands
-Xlog:gc*                         # GC log — watch the live set climb after each full GC

Interview questions

Most-asked memory and GC questions in Indian interviews — from screening rounds to product-company deep dives, where GC algorithms and leak diagnosis are staples:

Beginner What is garbage collection, and why doesn't Java need manual free()?

The JVM automatically reclaims heap memory occupied by objects that are no longer reachable, so the programmer never explicitly deallocates. This eliminates dangling pointers and double-frees, and reduces (but does not eliminate) memory leaks.

Beginner When does an object become eligible for garbage collection?

When it is no longer reachable — no chain of references from any GC root leads to it. Nulling the last reference, reassigning it, or the reference going out of scope all make an object unreachable. Eligibility is about reachability, not scope alone.

Beginner What are the young and old generations, and what is a minor vs a full GC?

The heap is split by the weak generational hypothesis: new objects go in the young generation (Eden plus two survivor spaces); objects that survive several collections are promoted to the old generation. A minor GC collects only the young generation (frequent, cheap); a major/full GC touches the old generation too (rarer, costlier).

Beginner Does System.gc() force garbage collection?

No — it is only a request; the JVM may ignore it. It typically triggers a full GC with a stop-the-world pause, so calling it in application code hurts performance. Treat it as a code smell and let the collector schedule itself.

Beginner Difference between strong, soft, weak, and phantom references?

Strong: ordinary reference, never collected while reachable. Soft: kept until the heap is under memory pressure — good for caches. Weak: cleared at the next GC once only weakly reachable — used by WeakHashMap. Phantom: never dereferenceable, only tells you the object has been reclaimed — used for post-mortem cleanup.

Intermediate What are GC roots? Give examples.

The starting points the collector traces reachability from: active thread stack variables, static fields of loaded classes, JNI local/global references, and objects held by the JVM itself (e.g. active monitors). Anything not reachable from a root is collectible.

Intermediate Why is G1 the default collector, and how does it work?

G1 divides the heap into equal-sized regions and collects the regions with the most garbage first, so it can meet a soft pause-time target (-XX:MaxGCPauseTimeMillis) while running mostly concurrently. It balances throughput and predictable pauses without hand-tuning, which suits the majority of server workloads — hence the default since Java 9.

Intermediate If GC is automatic, how can a Java application still leak memory?

A leak here means objects remain reachable but never used again, so the collector cannot free them. Common causes: unbounded caches/collections, static references, unregistered listeners, ThreadLocals on pooled threads, and classloader leaks in app servers. The heap grows until an OutOfMemoryError.

Intermediate When would you choose Parallel over G1 or ZGC?

When raw throughput matters more than pause time — batch and ETL jobs that can tolerate stop-the-world pauses but must finish as fast as possible. Parallel spends the least total time in GC. Choose G1 for balanced services and ZGC/Shenandoah when pauses must stay sub-millisecond.

Intermediate Why is finalize() deprecated, and what replaces it?

finalize() runs at an unpredictable time (or never), can resurrect the object, delays reclamation, and slows GC. Its replacement is Cleaner (or try-with-resources on AutoCloseable): register a cleaning action that runs when the object becomes phantom-reachable, with deterministic explicit cleanup available too.

Intermediate What is the difference between OutOfMemoryError: Java heap space and Metaspace?

"Java heap space" means the object heap is exhausted — a leak or an undersized -Xmx. "Metaspace" means the native area holding class metadata is exhausted — usually too many loaded classes or a classloader leak (common with repeated redeploys), tuned with -XX:MaxMetaspaceSize. Different regions, different root causes.

Pro How do ZGC and Shenandoah achieve sub-millisecond pauses?

They do almost all GC work — marking, relocation and reference updates — concurrently with the running application, so pause times stay roughly constant regardless of heap size. They rely on barriers (ZGC uses load barriers with coloured pointers; Shenandoah uses load-reference barriers with Brooks-style forwarding) to keep application threads consistent while objects move. The cost is extra throughput and footprint overhead versus G1.

Pro Walk me through diagnosing a slow memory leak in production.

Confirm it first: enable GC logging (-Xlog:gc*) and watch the live set — heap occupancy after each full GC — trend upward once the app is at steady state. Capture a heap dump (-XX:+HeapDumpOnOutOfMemoryError, or on demand via jcmd/jmap), open it in Eclipse MAT, and use the dominator tree and leak-suspects report to find the object retaining the most memory and the GC root holding it. Then fix the retention (bound the cache, unregister the listener, clear the ThreadLocal).

Pro What does "GC overhead limit exceeded" tell you, and how do you respond?

The parallel collector throws it when the process spends over 98% of its time in GC while recovering under 2% of the heap — the JVM is thrashing near exhaustion rather than failing outright. It almost always signals a genuine leak or a badly undersized heap; the fix is to capture a heap dump and find the retained objects, not simply to raise -Xmx or disable the check.

Pro Why can't garbage collection be made completely pause-free and free of cost?

Even the most concurrent collectors need brief stop-the-world safepoints (e.g. to establish an initial mark snapshot), and running concurrently forces read/write barriers on the application that cost throughput and CPU. Moving objects to compact the heap requires updating every reference consistently while threads run. So GC is always a trade among pause time, throughput, and footprint — you tune which one to favour, you cannot make all three free.
