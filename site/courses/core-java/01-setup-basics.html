Module: Setup and Language Basics

Before any framework, a Java developer has to own the toolchain and the type system. This module maps the pipeline from source to running JVM and lays out the language's ground rules — primitives, literals, var, and conversions — then points you to the canonical source for each.

Topics

JDK/JRE/JVM, javac/java, compact source + instance main (JEP 512)

The JDK (Java Development Kit) is what you install to build: it bundles the javac compiler, the java launcher, and tools, wrapped around the JRE (runtime libraries plus the JVM). The JVM is the abstract machine that loads and executes bytecode. javac compiles .java source into portable .class bytecode; java starts a JVM, loads the class, verifies it, JIT-compiles hot paths, and invokes main. Because the bytecode is platform-neutral and only the JVM is platform-specific, the same class file runs anywhere — "write once, run anywhere."

Why it matters: modern Java strips the ceremony that once confused beginners. As of JEP 512 (finalized in JDK 25) a compact source file can hold a bare instance main() with no class boilerplate, popular java.base classes are implicitly available, and a new java.lang.IO handles simple console output. The launcher can also run a .java file directly in source-file mode. Explaining this pipeline — and how the entry point is found — is a standard opening interview question.

// Modern compact source file (JEP 512, final in JDK 25) — no class, instance main:
void main() {
    IO.println("Hello, World!");
}
// Classic build + run:            javac Hello.java   then   java Hello
// Run a source file directly:     java Hello.java

Syntax, primitives vs references, literals, var

Java has exactly eight primitive types — byte, short, int, long, float, double, char, boolean — whose variables hold the raw value directly. Everything else is a reference type (classes, arrays, interfaces): the variable holds a reference to a heap object, or null. Literals write fixed values: decimal, hexadecimal (0x) and binary (0b) integers, underscores for grouping digits, the L/f/d suffixes, character and string literals with escape sequences.

Why it matters: the primitive-versus-reference split is the mental model behind == versus equals(), behind null, and behind Java's pass-by-value semantics (a method always receives a copy of the value — for objects, a copy of the reference). var (Java 10) is local-variable type inference: the compiler fixes the static type from the initializer at compile time, so there is no runtime cost and Java stays statically typed. It works only for locals that have an initializer — never fields, parameters, or return types.

int count = 26;                 // primitive: the variable holds 26
long big   = 1_000_000L;        // digit grouping + L suffix
int hex = 0x1A, bin = 0b11010;  // same value, different literals
String name = "Ada";            // reference: holds a reference to a String
var list = new ArrayList<String>();  // inferred as ArrayList<String>

Casting, promotion, wrapper types, autoboxing

Widening conversions (int → long → double) happen automatically and lose no information; narrowing conversions (double → int) require an explicit cast and can silently drop the fraction or overflow. In arithmetic, numeric promotion lifts byte, short, and char operands to int before the operation (and the whole expression to the widest operand if a long/float/double is involved) — which is why 'A' + 1 is the int 66. Each primitive has a wrapper class (Integer, Double, Boolean, Character, …): real objects that let primitives live in generics and collections and that expose parsing methods and constants.

Why it matters: autoboxing and unboxing convert between the two worlds automatically (int ↔ Integer), and the seams are a favourite interview trap. Unboxing a null wrapper throws a NullPointerException; Integer caches instances for the range −128 to 127, so == can be true for small boxed values yet false for large ones (always compare with equals()); and autoboxing inside a hot loop quietly allocates garbage.

long l = 100;          // widening: int literal to long, automatic
int  i = (int) 3.99;   // narrowing cast: i == 3 (fraction dropped)
int  sum = 'A' + 1;    // char promoted to int: sum == 66
Integer boxed = 42;    // autoboxing: Integer.valueOf(42)
int back = boxed;      // unboxing: boxed.intValue()

Interview questions

Setup and language basics are the warm-up round in almost every Indian Java interview — screening calls and service-company panels alike open here before moving deeper:

Beginner What is the difference between JDK, JRE, and JVM?

The JVM is the abstract machine that executes bytecode (its implementation is platform-specific). The JRE is the JVM plus the core class libraries — enough to run a Java application. The JDK is the JRE plus development tools (javac, java, jar, javadoc) — what you need to build one. So JDK contains the JRE, which contains the JVM.

Beginner What makes Java platform-independent?

javac compiles source to platform-neutral bytecode (.class), not to native machine code. Every operating system ships its own JVM that runs that same bytecode, so one compiled program runs everywhere — "write once, run anywhere." The JVM is platform-specific; your program is not.

Beginner Explain the traditional public static void main(String[] args) signature.

public so the JVM can call it from outside the class; static so it runs without constructing an instance; void because it returns nothing to the launcher; String[] args receives command-line arguments. Since JEP 512, a compact source file may instead use a simpler instance main(), even with no parameters.

Beginner How many primitive types does Java have, and what are they?

Eight: byte, short, int, long (integers), float and double (floating point), char (a 16-bit UTF-16 code unit), and boolean. Everything else — strings, arrays, your own classes — is a reference type.

Beginner What is the difference between a primitive and a reference type?

A primitive variable stores the actual value; a reference variable stores a reference to an object on the heap, or null. Primitives can never be null and compare by value with ==; references can be null, and == compares identity while equals() compares contents.

Intermediate Is Java pass-by-value or pass-by-reference?

Always pass-by-value. For an object argument the value copied is the reference, so the method can mutate the object it points to, but reassigning the parameter does not affect the caller's variable. This distinction is a frequent trick question.

Intermediate What does var do, and where can you not use it?

var (Java 10) infers a local variable's static type from its initializer at compile time — no runtime cost, and Java stays statically typed. It is allowed only for local variables that have an initializer (including for-loop indexes and try-with-resources). It cannot be used for fields, method or constructor parameters, return types, or a null-only initializer.

Intermediate What are compact source files and instance main methods (JEP 512)?

Finalized in JDK 25, they let a beginner write a program without a class declaration and with a simple instance main() — even void main() with no arguments. Common java.base classes are imported on demand and a new java.lang.IO class provides IO.println. The compiler synthesizes an unnamed class behind the scenes so the launcher and JVM behave exactly as before.

Intermediate Explain widening versus narrowing conversion.

Widening (smaller to larger, e.g. int → long → double) is automatic and lossless. Narrowing (larger to smaller, e.g. double → int, or int → byte) needs an explicit cast and may lose data — the fractional part is dropped or the value overflows the target range.

Intermediate What is numeric promotion in an arithmetic expression?

Before the operation, operands of type byte, short, or char are promoted to int; if any operand is long, float, or double, the whole expression is promoted to that widest type. So 'A' + 1 evaluates to the int 66, and byte + byte yields an int — assigning it back to a byte requires a cast.

Intermediate What are autoboxing and unboxing?

Autoboxing is the compiler's automatic conversion of a primitive to its wrapper (int → Integer, via Integer.valueOf); unboxing is the reverse (Integer → int, via intValue()). They let primitives be used with generics and collections, but the hidden conversions have cost and edge cases.

Pro Why can == give surprising results when comparing Integer objects?

Autoboxing routes through Integer.valueOf, which caches instances for −128 to 127. So two boxed 100s are the same cached object (== is true), but two boxed 1000s are distinct objects (== is false). Always compare wrapper values with equals(), and compare unboxed primitives with ==.

Pro What happens when you unbox a null wrapper, and how do you guard against it?

Unboxing calls a method (e.g. intValue()) on the wrapper, so a null reference throws NullPointerException — often on an innocent-looking line like int x = map.get(key); where the key is absent. Guard by keeping the wrapper type and null-checking, using Optional or a default (getOrDefault), or avoiding needless boxing so the value stays a primitive throughout.
