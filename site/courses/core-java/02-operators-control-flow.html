Module: Operators and Control Flow

Operators are how Java computes; control flow is how it decides. This module maps both — from bitwise shifts to the pattern-matching switch that retired long chains of instanceof.

Knowledge Library → Courses → Core Java → Operators and Control Flow

What this module covers

This module covers Java's full operator set — arithmetic, relational, logical, bitwise and shift, unary, the ternary conditional, assignment, and the pattern-binding instanceof — then the statements that direct execution: if/else, loops, and both flavours of switch. It closes with two features that show intent: labeled break/continue for nested loops, and the unnamed underscore for bindings you must write but never use. These are the building blocks every later Core Java module assumes.

Topics

All operators (arithmetic, bitwise, shift, instanceof pattern)

Java groups its operators by job: arithmetic (+ - * / %), relational and equality (< <= == !=), the short-circuiting logical pair (&& ||) and their always-evaluating bitwise cousins (& | ^), the bit shifts (<< >> >>>), unary (++ -- ! ~), the ternary conditional (? :), plain and compound assignment (= += ...), and the type-comparison operator instanceof. Precedence plus left-to-right evaluation decide how an expression without parentheses is read.

Why it matters: the sharpest interview traps live here — == (reference identity) versus equals, & versus && (short-circuit), >> versus >>> (sign extension), i++ versus ++i, and integer versus floating-point division. Since Java 16, instanceof also binds a pattern variable, so the redundant cast disappears — the modern idiom that record and switch patterns build on.

Object value = fetch();
if (value instanceof String s) {   // test + cast + bind in one step
    return s.length();             // s is in scope only where proven a String
}
return 0;

Control flow, switch statements and switch expressions with pattern matching

Control flow is the set of statements that decide which code runs and how often: if/else, the while/do-while/for loops (including the enhanced for), and the branching keywords break, continue, return, and yield. switch comes in two forms — the classic statement with colon labels and fall-through, and the Java 14 switch expression with arrow labels that produces a value, has no fall-through, and must be exhaustive.

Why it matters: since Java 21 a switch can also pattern-match — a case tests a type and binds a variable, destructures a record, refines the match with a when guard, and handles null explicitly — collapsing long instanceof-else chains into one exhaustive switch. Knowing when a switch must be exhaustive, and how arrow/yield differ from colon/break, is a standard probe.

String describe(Object o) {
    return switch (o) {                        // exhaustive, produces a value
        case null                 -> "nothing";
        case Integer i when i > 0  -> "positive int " + i;
        case Integer i            -> "int " + i;
        case String s             -> "text of length " + s.length();
        default                   -> "something else";
    };
}

Labeled breaks, unnamed variables and patterns (_)

A labeled statement gives a loop or block a name, so break label; and continue label; can target an outer loop from inside a nested one — the structured alternative to a boolean flag (Java has no goto). Unnamed variables and patterns use a bare underscore _ to say "syntactically required, semantically ignored": an unused catch parameter, a loop counter you never read, a discarded lambda parameter, or a record component you skip while destructuring.

Why it matters: labeled breaks keep nested-loop exits readable instead of scattering flag checks; the underscore — preview in Java 21, standard since Java 22 — removes misleading unused names and lets tools flag the genuinely-unused. Both are increasingly asked as Java 21+ features reach interviews.

int total = 0;
outer:
for (var row : grid) {
    for (var _ : row) {                    // underscore: required but unused
        if (++total > limit) break outer; // exits BOTH loops, not just the inner
    }
}

Interview questions

Operators, switch and the newer pattern-matching features are staple Indian-interview screening ground — from == versus equals in the first round to exhaustive pattern switches in senior ones:

Beginner What is the difference between == and equals() for objects?

== compares references — whether two variables point to the same object; equals() compares logical value as overridden (for example, String compares characters). For primitives, == compares values directly. The Integer autobox cache for small values (-128 to 127) makes == occasionally appear to work on wrappers — a classic trap.

Beginner Difference between & and && (and | vs ||)?

&& and || short-circuit — the right operand is skipped once the left decides the result, so x != null && x.isValid() is safe. & and | always evaluate both sides (and act as bitwise operators on integer types). Use the short-circuit forms for guards; use the single forms when you need both side effects or bit math.

Beginner i++ versus ++i — what is the difference?

Both increment i. Post-increment i++ yields the old value and then increments; pre-increment ++i increments and then yields the new value. So int b = i++; gives b the old i, while int b = ++i; gives it the new one. Standing alone in a loop header they behave identically.

Beginner Why is break needed in a traditional switch, and what is fall-through?

After a colon-label case matches, execution keeps running into the following cases' statements until a break (or the end) — that is fall-through, useful for sharing code across labels but a frequent bug source. The arrow-label switch (case L ->) has no fall-through, so no break is needed.

Beginner What is the difference between >> and >>>?

>> is the signed (arithmetic) right shift: it fills the high bits with the sign bit, preserving negativity. >>> is the unsigned (logical) right shift: it always fills with zero. They differ for negative numbers; for non-negative values they agree. There is no <<< — left shift needs only <<.

Intermediate Switch statement versus switch expression?

A statement runs for side effects, uses colon labels with fall-through, and relies on break. An expression (Java 14) produces a value, uses arrow labels with no fall-through (or yield with colon labels), and must be exhaustive — cover every case, usually via default or the full set of an enum or sealed type. You can assign a switch expression to a variable.

Intermediate What is pattern matching for instanceof, and what does it buy you?

if (o instanceof String s) tests the type and, on success, binds the already-cast s — removing the redundant explicit cast. The binding is flow-scoped: s exists only where the compiler proves o is a String (the if-body, or after a short-circuiting &&). Final since Java 16, and the basis for record and switch patterns.

Intermediate What types can a switch's selector be?

Classic switch: byte/short/char/int and their wrappers, enum constants, and String (since Java 7). With pattern matching (Java 21) the selector may be any reference type, matched by type patterns, record patterns, and when guards, plus an explicit case null.

Intermediate What are labeled break and continue, and when do you use them?

A label names an enclosing loop or block; break label; exits that outer statement and continue label; jumps to its next iteration — the clean way to leave nested loops without a boolean flag. Reserve them for genuine nested-loop exits, since overusing labels hurts readability.

Intermediate Integer division by zero versus floating-point division by zero?

Integer 1 / 0 throws ArithmeticException ("/ by zero"), and % by zero does too. Floating-point 1.0 / 0 yields Infinity, and 0.0 / 0.0 yields NaN — no exception. Mixing the two up is a very common Indian-interview trap.

Pro How does switch pattern matching handle null, exhaustiveness, and guards?

A pattern switch throws NullPointerException on a null selector unless you add case null (optionally case null, default). It must be exhaustive — the compiler demands all types be covered, trivially for an enum or sealed hierarchy, otherwise via default. A when guard refines a case (case Integer i when i > 0); a more specific case must appear before a less specific one (dominance ordering) or it will not compile.

Pro What are unnamed variables and patterns (_), where can you use them, and since which version?

A bare underscore _ marks a binding required by syntax but never read — an ignored catch (Exception _), an unused enhanced-for or lambda parameter, a try-with-resources variable, or a skipped record component in a pattern such as case Point(int x, _). Unlike a normal name, _ can be repeated many times in the same scope. Preview in Java 21, standard since Java 22; it documents intent and lets tools flag the truly-unused.

Pro Evaluate int x = 5; int y = x++ + ++x; — what are x and y, and why is it well-defined?

x++ reads 5 (then x becomes 6); ++x makes x 7 and reads 7; so y = 5 + 7 = 12 and x ends at 7. Java guarantees operands are evaluated fully and strictly left to right before the + applies (JLS §15.7), so — unlike C or C++ — this has one defined result rather than undefined behaviour.
