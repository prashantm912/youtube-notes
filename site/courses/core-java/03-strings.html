Module: Strings

Strings look like the gentlest topic in Java and are among the deepest in interviews — immutability and the string pool sit under questions that catch out even experienced candidates.

Knowledge Library → Courses → Core Java → Strings

What this module covers

Text handling in Java rests on three ideas interviewers mine hard: String is immutable and its literals are pooled and interned; the mutable builders (StringBuilder, StringBuffer) exist for the cases where immutability would be wasteful; and modern literal syntax — text blocks — plus String.format cover multi-line and templated text. This module orients you on each, points you at the official reference first, and flags where the Indian screening rounds concentrate: immutability, the pool, and == versus equals.

Topics

String immutability, pool, compact strings

A String is immutable: once created, its characters never change, and every "modifying" method (concat, replace, toUpperCase) returns a brand-new object. String literals live in the string pool — a shared region of the heap — so identical literals resolve to one interned instance; new String("x") deliberately bypasses the pool and allocates a fresh object, which is why == (identity) and equals (value) can disagree. Immutability is what makes pooling safe to share across threads and callers.

Why it matters: immutability is the reason a String is a safe HashMap key, can cache its own hashCode, is thread-safe by construction, and is trusted for class names, file paths and credentials. Compact strings (Java 9, JEP 254) then change the internal storage from a UTF-16 char[] to a byte[] plus a one-byte coder flag, so Latin-1 text uses half the memory — invisible to your code, but a favourite "what changed in Java 9?" prompt. These three — immutability, pool, compact strings — are the single most-asked String area in India.

String a = "hi";              // interned pool literal
String b = "hi";              // same pooled object   -> a == b  is true
String c = new String("hi");  // forced fresh heap object -> a == c  is false
String d = c.intern();        // returns the pooled instance -> a == d  is true
// a.equals(c) is always true — equals compares value, not identity

String API, CharSequence/Character, StringBuilder/StringBuffer

The String API is broad, but the interview-relevant slices are comparison (equals, equalsIgnoreCase, compareTo), searching (indexOf, contains), slicing (substring, charAt), transformation (replace, split, join, strip, repeat, isBlank, lines) and formatting. String, StringBuilder and StringBuffer all implement CharSequence — the read-only interface that lets an API accept any of them without caring which — while Character is the wrapper and utility class for a single char (isDigit, isLetter, getNumericValue).

When you actually need to build or edit text, reach for StringBuilder, a mutable character sequence. StringBuffer is its older, synchronized (thread-safe) twin; StringBuilder is the unsynchronized, faster drop-in you should default to in single-threaded code. The classic trap: concatenating with + inside a loop allocates a new immutable String every iteration (quadratic work), whereas one StringBuilder appends in place.

StringBuilder sb = new StringBuilder();
for (String part : parts) {
    sb.append(part).append(',');   // in-place, one buffer
}
String csv = sb.toString();        // single final String — not N throwaways

Text blocks, formatting

Text blocks (standard since Java 15, JEP 378) are multi-line string literals delimited by triple quotes — ideal for embedding JSON, SQL, HTML or XML in code without a forest of \n and escaped quotes. The compiler strips incidental (common leading) whitespace while keeping the indentation you intend; a trailing \ joins lines, and \s preserves trailing spaces. Crucially, a text block is an ordinary String at runtime — same type, same pool — just a nicer way to write the literal.

For assembling text from values, String.format and the instance method formatted (both backed by java.util.Formatter) fill a template from format specifiers — %s, %d, %.2f, the platform newline %n, and positional %1$s — the printf family for Java. Text blocks pair naturally with formatted for readable templated output.

String json = """
    {
      "name": "%s",
      "age": %d
    }""".formatted(name, age);   // still just a String

Interview questions

Most-asked String questions in Indian interviews — immutability and the string pool dominate the screening rounds, so they carry the most weight here:

Beginner Why is String immutable in Java?

Once created, a String's contents can never change — the internal array is private and no method mutates it. This buys four things interviewers want named: safe sharing in the string pool, a cacheable hashCode (so it is a reliable HashMap key), inherent thread safety, and security for sensitive values like file paths and class names that must not change after a check.

Beginner What is the string constant pool and how does it save memory?

A special heap region where the JVM keeps one copy of each distinct String literal. When you write a literal, the JVM checks the pool: if an equal String exists it returns that reference, otherwise it adds (interns) the new one. Because Strings are immutable, many variables can share the same pooled object safely, cutting allocation.

Beginner How many objects does String s = new String("hi") create?

Up to two. The literal "hi" goes in the pool (one object, if not already there), and new String(...) forces a second, distinct object on the heap. That is exactly why new String is discouraged: you pay for an extra object and lose pool sharing. Call .intern() to get the pooled reference back.

Beginner Difference between == and equals() for Strings?

== compares references (are these the same object?); equals compares character content. Two pooled literals with the same value are ==-equal because they are one object, but a new String with the same text is not. Always use equals for value comparison — the ==-sometimes-works behaviour is a deliberate trap question.

Beginner When would you use String vs StringBuilder vs StringBuffer?

String for fixed or rarely-changing text. StringBuilder when you build or edit text in a single thread — mutable and fast. StringBuffer only when multiple threads share the same builder and you need its synchronized methods; otherwise its locking is pure overhead.

Intermediate StringBuilder vs StringBuffer — the real difference?

Same API, same mutable-buffer behaviour; the difference is synchronization. StringBuffer's methods are synchronized, making a single instance safe to mutate from multiple threads but slower. StringBuilder drops the locks and is the recommended default for single-threaded code. Note that thread-safety per call does not make a shared sequence of operations atomic.

Intermediate What does intern() do, and when is it useful?

intern() returns the canonical pooled instance for a String's value — adding it to the pool if absent — so afterwards == works against other interned copies. It can shrink memory when a program holds huge numbers of duplicate runtime-built Strings, but overusing it moves pressure onto the pool and costs lookup time, so it is a targeted tool, not a default.

Intermediate Why does String immutability make it a good HashMap key?

A HashMap places a key by its hashCode at insertion. If the key could mutate, its hash would change and the entry would be lost in the wrong bucket. Because a String never changes, its hash is stable — and String even caches the computed hash — so lookups stay correct and fast. This ties immutability directly to collection correctness.

Intermediate What are compact strings (Java 9), and what problem do they solve?

Before Java 9 every String held a UTF-16 char[] — two bytes per character even for pure ASCII. Compact strings (JEP 254) store the characters in a byte[] with a one-byte coder flag: Latin-1 text uses one byte per character, UTF-16 is used only when needed. Typical apps see roughly half the String heap and less GC, with no source changes.

Intermediate What is CharSequence and why do APIs accept it instead of String?

CharSequence is the read-only interface for anything that is a sequence of chars — implemented by String, StringBuilder, StringBuffer and CharBuffer. Accepting CharSequence lets a method take any of them without forcing a toString() copy. It only guarantees reading (length, charAt, subSequence), and it does not redefine equals/hashCode, so never use a CharSequence as a map key expecting value semantics.

Pro Is String really immutable — can you change one via reflection?

At the language level, yes, immutable; via reflection you can set the private backing array accessible and overwrite it, mutating the String in place. This is a genuine bug source: because that String may be shared through the pool, you can silently corrupt every other reference to the "same" literal. It is a demonstration of why immutability is a guarantee of the API, not of the raw memory, and why security-sensitive code must not rely on Strings being unforgeable.

Pro Why is building a String with + in a loop bad, and what does the compiler do for a + b + c?

In a loop, each += creates a fresh immutable String and copies all prior characters, so N appends do quadratic work and churn the heap — use one StringBuilder. A single straight-line expression like a + b + c is different: javac compiles it (since Java 9) to an invokedynamic call bootstrapped by StringConcatFactory, which builds the result efficiently in one shot. So the anti-pattern is repeated concatenation across iterations, not one-line concatenation.

Pro How do text blocks handle indentation, and do they create a different kind of String?

No — a text block compiles to a normal String, poolable and identical in type to any other literal. The compiler removes incidental whitespace: it finds the minimum indentation across all non-blank lines (and the closing delimiter's position), strips that common prefix, and keeps the rest as essential indentation. Trailing spaces are stripped unless escaped with \s, and a line-ending \ suppresses that line's newline. Two text blocks with the same resulting characters are equals — and, as literals, share the pool.
