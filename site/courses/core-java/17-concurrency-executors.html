Concurrency: Executors and Futures

Raw threads do not scale; pools, futures and explicit locks do. This module maps the java.util.concurrent building blocks every backend service runs on — and flags the two, ThreadPoolExecutor tuning and CompletableFuture, that Indian backend interviews probe hardest.

Knowledge Library → Courses → Core Java → Concurrency: Executors and Futures

What this module covers

The executor framework decouples what runs (a task) from how it runs (which thread, from which pool). This module covers ExecutorService and the six ThreadPoolExecutor knobs that decide a pool's behavior under load, the Future and CompletableFuture result types — with CompletableFuture's composition and error-handling pipeline — and the explicit Lock family that supersedes synchronized when you need timeouts, fairness, multiple conditions, or reader/writer separation. Two of these, ThreadPoolExecutor sizing and CompletableFuture, are among the most-asked concurrency topics in Indian product and service-company interviews.

Topics

ExecutorService and thread pools (ThreadPoolExecutor parameters)

The Executor framework separates the task (a Runnable or Callable) from the machinery that runs it. ExecutorService is the workhorse interface: you submit tasks and get a Future back, and you shut the pool down with shutdown/shutdownNow plus awaitTermination. Under the hood most pools are a ThreadPoolExecutor, whose constructor parameters — corePoolSize, maximumPoolSize, keepAliveTime + unit, the workQueue, a ThreadFactory, and a RejectedExecutionHandler — fully determine how it behaves.

Why it matters: the task-arrival algorithm is the single most-tested detail. A new task starts a core thread until corePoolSize is reached, then it queues; the pool grows toward maximumPoolSize only when the queue is full; when queue and max threads are both saturated the RejectedExecutionHandler fires. That is why an unbounded queue — the hidden default of Executors.newFixedThreadPool — means maximumPoolSize is never reached, and one slow consumer becomes an OutOfMemoryError. Reaching for a bounded queue with a back-pressure policy is what separates a screening answer from a production one.

ExecutorService pool = new ThreadPoolExecutor(
    4, 8, 60L, TimeUnit.SECONDS,               // core, max, keep-alive
    new ArrayBlockingQueue<Runnable>(100),      // bounded work queue
    new ThreadPoolExecutor.CallerRunsPolicy()); // back-pressure when saturated
Future<Integer> f = pool.submit(() -> compute());
pool.shutdown();

Future and CompletableFuture (composition and error handling)

A Future is a handle to a result that is not ready yet: get() blocks until it is, cancel() tries to stop the task, isDone() polls. Its limit is that you can only block or poll — you cannot say "when this finishes, do that." CompletableFuture (Java 8) removes that limit by implementing CompletionStage: you attach callbacks that fire on completion and chain transformations without ever parking a thread on get().

Why it matters: thenApply transforms a result (like map), thenCompose flattens a CompletableFuture-returning step (like flatMap), and thenCombine joins two independent futures. Failures propagate down the chain and are recovered with exceptionally, handle, or whenComplete — not a try/catch around get(). The subtlety interviewers push on: non-async stages run on whatever thread completed the previous stage, while async stages (thenApplyAsync) run on the common ForkJoinPool unless you pass your own Executor — so blocking work left on the common pool can starve the whole JVM.

CompletableFuture
    .supplyAsync(() -> fetchUser(id), pool)  // run on YOUR executor
    .thenApply(User::profile)                // transform (map)
    .thenCompose(this::loadOrders)           // flatMap another CF
    .exceptionally(ex -> fallbackOrders())    // recover from failure
    .thenAccept(this::render);               // consume, non-blocking

Explicit locks: ReentrantLock, ReadWriteLock, StampedLock, Condition

The Lock interface is the explicit, flexible alternative to synchronized. ReentrantLock gives the same mutual exclusion plus tryLock (optionally with a timeout), interruptible acquisition, an optional fairness policy, and multiple Condition wait-sets — at the price of a mandatory try/finally so the lock is always released. A Condition replaces wait/notify, giving each lock one or more named wait queues (the classic notFull and notEmpty of a bounded buffer).

Why it matters: when reads vastly outnumber writes, ReentrantReadWriteLock lets many readers run together while writers stay exclusive — and you can downgrade a write lock to a read lock but never upgrade a read lock to a write lock (that deadlocks). StampedLock adds optimistic reads: tryOptimisticRead returns a stamp you validate after reading, skipping lock acquisition entirely on the happy path — but it is not reentrant and has no Condition. Picking the lightest tool that is still correct is the judgment interviewers are testing.

Lock lock = new ReentrantLock();
lock.lock();
try {
    // critical section — mutually exclusive
} finally {
    lock.unlock();   // ALWAYS release in finally
}

Interview questions

The concurrency questions that dominate Indian backend interviews — thread-pool tuning and CompletableFuture come up in almost every product-company round:

Beginner What is the Executor framework and why prefer it over new Thread()?

It decouples task submission from thread management. Instead of creating a thread per task (unbounded, expensive, uncontrolled), you submit tasks to an ExecutorService that reuses a managed pool of worker threads, queues overflow, and gives back a Future. You get resource limits, reuse, graceful shutdown, and result handling for free.

Beginner Runnable vs Callable — and how do you get a task's result?

Runnable.run() returns void and cannot throw checked exceptions; Callable<V>.call() returns a value and may throw. Submitting either to an executor returns a Future; future.get() blocks for the result (and rethrows the task's exception wrapped in ExecutionException).

Beginner Difference between shutdown() and shutdownNow()?

shutdown() is orderly: it stops accepting new tasks but lets queued and running tasks finish. shutdownNow() tries to stop running tasks (via interrupt), drops the queue, and returns the never-started tasks. Either way you typically call awaitTermination(timeout, unit) afterwards to block until the pool actually drains.

Beginner submit() vs execute()?

execute(Runnable) is fire-and-forget from the base Executor — no handle, and an uncaught exception surfaces on the worker thread. submit() returns a Future; a task exception is captured and re-thrown only when you call get(). A silently swallowed exception is usually a submit whose Future is never inspected.

Intermediate Explain every ThreadPoolExecutor constructor parameter.

corePoolSize — threads kept alive even when idle; maximumPoolSize — the ceiling on threads; keepAliveTime + unit — how long above-core idle threads survive; workQueue — the BlockingQueue holding pending tasks; threadFactory — names/daemon-flags new threads; RejectedExecutionHandler — what happens when queue and max are both full. These six values fully define the pool's behavior. The most-asked ThreadPoolExecutor question in India.

Intermediate When a task arrives, how does ThreadPoolExecutor decide to run, queue, or reject it?

Ordering: if threads < corePoolSize, start a new core thread. Else try to enqueue in the workQueue. Only if the queue is full does it create threads up to maximumPoolSize. If the queue is full and max threads are running, the RejectedExecutionHandler fires. The trap: an unbounded queue never fills, so max is never reached and rejection never triggers.

Intermediate Why are newFixedThreadPool and newCachedThreadPool risky in production?

newFixedThreadPool uses an unbounded LinkedBlockingQueue — a faster-than-drained producer piles tasks until OutOfMemoryError. newCachedThreadPool has an unbounded thread count (SynchronousQueue + max = Integer.MAX_VALUE) — a burst spawns thousands of threads. Prefer an explicit ThreadPoolExecutor with a bounded queue and a chosen rejection policy.

Intermediate What are the built-in rejection (saturation) policies?

AbortPolicy (default) throws RejectedExecutionException; CallerRunsPolicy runs the task on the submitting thread, which naturally throttles producers (back-pressure); DiscardPolicy silently drops the new task; DiscardOldestPolicy drops the oldest queued task and retries. You can also implement RejectedExecutionHandler yourself.

Intermediate Future vs CompletableFuture — what does CompletableFuture add?

Future only lets you block (get) or poll (isDone) — no chaining, no completion callback, no easy combination. CompletableFuture implements CompletionStage: non-blocking composition (thenApply/thenCompose/thenCombine), explicit completion (complete), combinators (allOf/anyOf), and functional error handling — the foundation of async pipelines.

Intermediate thenApply vs thenCompose vs thenCombine?

thenApply maps the result with a plain function (T → U). thenCompose flattens a function that itself returns a CompletableFuture, avoiding CompletableFuture<CompletableFuture<U>> — the flatMap of futures. thenCombine waits for two independent futures and merges both results with a BiFunction.

Intermediate How do you handle exceptions in a CompletableFuture chain?

exceptionally(fn) supplies a fallback value only on the failure path; handle(bi) runs on both success and failure (receiving result-or-throwable); whenComplete observes the outcome without changing it. An exception short-circuits the remaining stages until one of these recovers it — you never wrap get() in try/catch inside a pipeline.

Intermediate ReentrantLock vs synchronized — when do you choose the lock?

Both are reentrant and give the same visibility guarantees. Choose ReentrantLock when you need tryLock/timeouts, interruptible acquisition, fairness, or multiple Condition wait-sets. Choose synchronized for simple critical sections — it is JVM-optimized and cannot leak an unreleased lock. The lock's price is the mandatory try/finally unlock.

Pro Which thread runs a CompletableFuture callback, and how do you control it?

A non-async stage (thenApply) runs on whichever thread completed the previous stage — possibly the caller, possibly a pool thread. An ...Async stage runs on ForkJoinPool.commonPool() unless you pass an Executor. Because the common pool is shared JVM-wide and small, blocking or long tasks belong on your own executor via supplyAsync(task, myPool) / thenApplyAsync(fn, myPool).

Pro Explain lock downgrading in ReentrantReadWriteLock — and why upgrading is forbidden.

Downgrading is legal: hold the write lock, acquire the read lock, then release the write lock — you keep reading with the fresh data you just wrote. Upgrading (read → write) is not: two readers each trying to become the writer would each wait for the other to release its read lock, a guaranteed deadlock, so the API disallows it. StampedLock offers tryConvertToWriteLock as a conditional alternative.

Pro How do you size a thread pool for CPU-bound vs IO-bound work?

CPU-bound: about the number of cores (N or N+1) — more threads only add context-switching. IO-bound: threads spend most time waiting, so size up by the wait-to-compute ratio — Brian Goetz's formula is N × targetUtilization × (1 + waitTime/computeTime). Measure under real load; a pool that is too large wastes memory and thrashes, too small underuses the machine.

Pro What is a StampedLock optimistic read, and when is it wrong to use?

tryOptimisticRead returns a stamp without acquiring any lock; you read the fields, then call validate(stamp) — if no writer intervened it returns true and you avoided all locking overhead; if not, you fall back to a real readLock. It shines for short, read-mostly sections. It is wrong when reads are long or side-effecting (you may read a torn state before validating), or when you need reentrancy or a Condition — StampedLock supports neither.
