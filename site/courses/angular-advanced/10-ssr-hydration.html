Module: Server-Side Rendering and Hydration

SSR ships HTML the browser can paint before a line of your JavaScript runs; hydration then wakes that same markup up in place instead of throwing it away. This module maps the render-on-the-server path and the v19 incremental-hydration model that keeps the bundle small.

Knowledge Library → Courses → Angular Advanced → Server-Side Rendering and Hydration

What this module covers

Rendering an Angular app on the server (SSR / the former "Universal"), prerendering routes at build time (SSG), and choosing per-route render modes with @angular/ssr. Then hydration: reusing the server DOM without flicker, the incremental @defer hydration added in v19, and the supporting pieces — TransferState, event replay and SEO meta — that make an SSR app fast and indexable rather than merely rendered twice.

Topics

Server-side rendering, prerendering and the server build

SSR runs your components on a Node server for each request and returns fully-formed HTML, so the user sees content (and a crawler sees markup) before the client bundle loads — the win is Largest Contentful Paint and SEO. Prerendering (SSG) does the same work at build time for routes whose output never changes, serving static files with the lowest possible Time to First Byte. Angular's current model is hybrid: ng add @angular/ssr wires up a server build, and a server.routes.ts file assigns each route a render mode.

Why it matters: "add SSR" is no longer one switch — you pick Server, Prerender, or Client per route, and you must know which content can be static, which needs request-time data, and which is better left client-only. That trade-off is the substance of most SSR interview questions.

// app.routes.server.ts — per-route render mode (hybrid rendering)
export const serverRoutes: ServerRoute[] = [
  { path: 'blog/:id', renderMode: RenderMode.Server },    // request-time
  { path: 'about',    renderMode: RenderMode.Prerender }, // build-time (SSG)
  { path: '**',       renderMode: RenderMode.Client },    // CSR fallback
];

Hydration: reusing the server-rendered DOM

Without hydration the client bundle would discard the server HTML and re-render from scratch — the old "destructive" behaviour that caused a visible flicker and wasted the SSR work. Non-destructive hydration (stable since v17, enabled with a single provideClientHydration()) instead walks the existing DOM, reuses those nodes, attaches event listeners and binds state. The constraints follow from that reuse: the server and client must produce the same DOM, so direct DOM manipulation, invalid HTML (a <table> without an explicit <tbody>), and mismatched preserveWhitespaces break it. The escape hatch is the ngSkipHydration attribute on a stubborn component.

bootstrapApplication(App, {
  providers: [
    provideClientHydration(withEventReplay()),
  ],
});

Incremental hydration with @defer (v19+)

Full-app hydration still ships and runs JavaScript for the whole page. Incremental hydration — built on deferrable views and enabled by default under provideClientHydration() from v19 — lets sections stay server-rendered but dehydrated, downloading and hydrating their code only when a trigger fires. You mark a boundary with a @defer block carrying a hydrate trigger: hydrate on viewport, on interaction, on hover, on idle, on immediate, on timer, the conditional hydrate when, or hydrate never for permanently static content. Below the fold means smaller initial bundles with no layout shift, because the server already painted the placeholder's real content.

@defer (hydrate on viewport) {
  <app-comments />              <!-- server-rendered, hydrates when scrolled to -->
} @placeholder {
  <p>Loading comments…</p>
}

TransferState, event replay and SEO

Three details separate a correct SSR app from a naive one. TransferState serialises data the server already fetched into the HTML so the client doesn't repeat the request on load — for HTTP this is automatic via the HttpClient transfer cache, tunable with withHttpTransferCacheOptions. Event replay (withEventReplay()) captures clicks and keystrokes that happen before hydration finishes and replays them afterwards, so an early tap is never lost. And SEO is the whole point of rendering on the server: set per-route document titles and meta/Open-Graph tags with the Title and Meta services so crawlers and social scrapers read real metadata, not a client-side placeholder.

constructor(private title: Title, private meta: Meta) {}
setSeo(post: Post) {
  this.title.setTitle(post.heading);
  this.meta.updateTag({ name: 'description', content: post.summary });
}

Interview questions

Most-asked SSR and hydration questions in Indian interviews — a favourite senior-round topic now that hydration is the default and incremental hydration is the current headline feature:

Beginner What is server-side rendering in Angular and why use it?

The app's components run on a Node server for each request and return fully-formed HTML, so the browser paints content before the JS bundle loads and crawlers see real markup. The payoffs are faster Largest Contentful Paint, better SEO, and correct social-media link previews — things a client-only (CSR) app struggles with because it starts as an empty shell.

Beginner How do you add SSR to an Angular application?

ng new my-app --ssr for a new project, or ng add @angular/ssr for an existing one. The schematic adds a server entry point, provideServerRendering(), a server build target, and hydration on the client. "Angular Universal" is the old name for this same capability.

Beginner SSR vs prerendering (SSG) vs CSR — what's the difference?

CSR renders in the browser at runtime. SSR renders on the server at request time — good for pages with per-request or frequently-changing data. Prerendering (SSG) renders at build time and serves static files — lowest TTFB, ideal for content that rarely changes (marketing pages, docs, blog posts). Hybrid rendering lets you pick one per route.

Beginner What is hydration?

The process where the client reuses the server-rendered DOM instead of re-creating it: Angular walks the existing nodes, attaches event listeners and binds state to make the page interactive. Enabled with provideClientHydration().

Intermediate What problem did non-destructive hydration solve?

Before it, the bootstrapping client threw away the server HTML and re-rendered from scratch (destructive hydration), causing a visible flicker and wasting the SSR work — LCP often got worse. Non-destructive hydration (stable v17) reuses the existing DOM, removing the flicker and improving LCP because nothing is repainted.

Intermediate What breaks hydration, and how do you handle it?

Anything that makes client DOM differ from server DOM: direct DOM manipulation (native appendChild, innerHTML), invalid HTML the browser auto-corrects (a <table> with no explicit <tbody>), and inconsistent preserveWhitespaces between builds. Fix the root cause where you can; as a last resort add ngSkipHydration to the offending component (it forgoes hydration for that subtree).

Intermediate What is TransferState and why does it matter for SSR?

A key-value store serialised into the server HTML and read on the client, so data the server already fetched isn't fetched again on load — avoiding a duplicate request and a flash of re-loading. For HttpClient this happens automatically through the transfer cache, which you can configure with withHttpTransferCacheOptions() (e.g. to include POSTs).

Intermediate How do you set SEO titles and meta tags in an SSR app?

Inject the Title and Meta services from @angular/platform-browser and set them per route (in a resolver or the component), e.g. title.setTitle(...) and meta.updateTag({ name: 'description', ... }). Because the page is rendered on the server, those tags are present in the initial HTML for crawlers and Open-Graph scrapers.

Intermediate What is event replay and why is it needed?

There's a gap between the server HTML appearing and hydration finishing; a user can click during it. withEventReplay() records those events and replays them once the matching listeners are attached, so an early interaction isn't silently dropped. It's the default in newer versions and pairs naturally with incremental hydration.

Pro What is incremental hydration and how do @defer hydrate triggers work?

An advanced hydration mode (v19, enabled by default under provideClientHydration()) that leaves sections server-rendered but dehydrated, hydrating each only when needed. You wrap a section in a @defer block with a hydrate trigger — hydrate on viewport / interaction / hover / idle / immediate / timer, hydrate when (condition), or hydrate never. Until the trigger fires the block's JS isn't downloaded, shrinking the initial bundle while the user still sees fully-rendered content (no layout shift). It builds on deferrable views and event replay.

Pro Your code uses window/document and crashes under SSR. Why, and how do you fix it?

The server has no browser globals, so referencing window, document, localStorage or navigator during rendering throws. Guard browser-only code with isPlatformBrowser(inject(PLATFORM_ID)), or defer it to afterNextRender/afterRender which only run in the browser. Prefer Angular abstractions (Renderer2, the DOCUMENT token) over touching globals directly — that also keeps hydration valid.

Pro When would you NOT use SSR, or choose prerendering over it?

SSR adds a running Node server, per-request compute cost, and browser-vs-server correctness constraints. If content is public and rarely changes, prerender it (SSG) for static-file speed and no server. If the app is behind a login with no SEO need and heavy interactivity (a dashboard), CSR — or CSR routes within a hybrid app — is often simpler and cheaper. Match render mode to each route's data volatility and SEO need rather than turning SSR on globally.
