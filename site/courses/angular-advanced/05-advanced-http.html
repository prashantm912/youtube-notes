Module: Advanced HTTP

Basic HttpClient gets you a response; production Angular is the layer you wrap around it — functional interceptors, HttpContext, caching and cancellation — the middleware that attaches auth tokens and absorbs flaky networks before your components ever see the request.

Knowledge Library → Courses → Angular Advanced → Advanced HTTP

What this module covers

Everything past get() and post(): functional interceptors — the modern middleware for auth, error handling, retries and logging — plus HttpContext for per-request metadata, immutable HttpParams/HttpHeaders, response caching, request cancellation, and typed clients (including the signal-based httpResource). The through-line is that HttpClient returns cold Observables you compose and cancel, and that cross-cutting concerns belong in one interceptor rather than scattered across every service.

Topics

Functional interceptors: the modern middleware

An interceptor is middleware in the HttpClient pipeline: every outgoing request flows through the chain on the way out, and every response event flows back through it. Modern Angular writes one as a plain function — an HttpInterceptorFn that receives the outgoing HttpRequest and a next handler and returns the Observable of HTTP events that next produces. You register the set once with provideHttpClient(withInterceptors([...])), and they run in the order you list them. Auth headers, logging, global error handling, retries and caching all live here instead of being copied into every service.

Why it matters: requests are immutable, so an interceptor that changes one calls req.clone({...}) and forwards the copy to next. The functional form (Angular 15+) is the current recommendation — tree-shakable, composable, dependencies pulled in with inject(), and with more predictable ordering than the older class-based HttpInterceptor registered through the HTTP_INTERCEPTORS token. Knowing that class interceptors are legacy (not wrong, just superseded) is exactly what marks you as current in an interview.

export const loggingInterceptor: HttpInterceptorFn = (req, next) => {
  console.log(`${req.method} ${req.urlWithParams}`);
  return next(req);            // pass it down the chain
};

// app.config.ts
providers: [provideHttpClient(withInterceptors([loggingInterceptor]))];

The classic jobs: auth tokens, error handling and retry

Three jobs account for most interceptors ever written. Auth: clone the request to add an Authorization: Bearer … header (or let Angular's built-in XSRF interceptor attach X-XSRF-TOKEN for you). Errors: pipe catchError onto next(req) to turn an HttpErrorResponse into a redirect on a 401, a toast on a 500, or a domain error — and you must re-emit with throwError so the caller's own error handler still runs. Retry: RxJS retry re-subscribes to the request observable (which re-issues the call) on failure; add a growing delay for exponential backoff against flaky networks, but never blindly retry non-idempotent writes.

Why it matters: "how do you add a token to every request" and "where do you handle errors globally" are near-certain Angular interview questions in Indian screening rounds — and the subtle trap is forgetting to rethrow, which silently makes failed calls look successful.

export const errorInterceptor: HttpInterceptorFn = (req, next) =>
  next(req).pipe(
    retry({ count: 2, delay: 500 }),
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401) inject(Router).navigate(['/login']);
      return throwError(() => err);   // re-emit so callers still see it
    })
  );

HttpContext: per-request metadata and scoped behavior

Sometimes an interceptor needs a hint that is meant for it, not for the server — "skip auth on this one call", "this response is cacheable", "hide the global spinner". HttpContext carries exactly that: a typed map keyed by HttpContextToken<T> that rides on the request's .context and is never serialised onto the wire. You set it per call with context: new HttpContext().set(TOKEN, value) and read it inside the interceptor with req.context.get(TOKEN).

Why it matters: it is the clean alternative to sniffing URLs or headers to decide behaviour. The canonical example in the docs is a caching interceptor whose token lets each caller opt in; the same pattern drives per-route auth toggles and route-scoped interceptor logic.

export const CACHE_ENABLED = new HttpContextToken<boolean>(() => false);

// caller opts in for this request only:
http.get<Config>('/api/config', { context: new HttpContext().set(CACHE_ENABLED, true) });

// interceptor reads the hint:
if (req.context.get(CACHE_ENABLED)) { /* serve or store the cached response */ }

Building requests: HttpParams, HttpHeaders and typed responses

For the request itself, HttpParams builds the query string and HttpHeaders builds the headers — both are immutable, so every set()/append() returns a new instance you must reassign or chain. In everyday code you pass a plain object to the params/headers option and HttpClient constructs them for you; reach for the explicit classes when you need repeated keys, custom encoding, or to build values up conditionally.

Typed responses are the companion habit: the generic on get<T>()/post<T>() tells the compiler and template the shape of the body. It is a compile-time assertion, not runtime validation — the server can still send something else — so treat it as a contract and validate at the boundary when the data is untrusted. Options such as observe: 'response' (the whole HttpResponse) and responseType change what you receive.

const params = new HttpParams().set('page', '2').set('q', term);
const headers = new HttpHeaders({ 'X-Api-Version': '3' });

this.http.get<Product[]>('/api/products', { params, headers });

Typed clients, request cancellation and the reactive future

An HttpClient call returns a cold Observable, so nothing leaves the browser until you subscribe — and unsubscribing aborts an in-flight request. That is the whole mechanism behind cancellation: switchMap cancels the previous request when a new search term arrives (the standard fix for typeahead race conditions), and the async pipe cancels when the view is destroyed. A "typed client" is simply the disciplined pattern of a service that returns Observable<Model> from get<Model>() and keeps the subscription out of the component.

The reactive direction is httpResource (Angular v19/20): a signal-based wrapper over HttpClient that fires eagerly, exposes value()/status()/error() as signals, re-requests when its input signals change, and cancels the outstanding request automatically. It still runs through your interceptors. Use it for reads that feed the template reactively; keep the plain HttpClient — with switchMap, retries and cancellation you control — for mutations and complex streams.

// classic typed client — cancellation via switchMap
this.search.valueChanges.pipe(
  switchMap(term => this.http.get<Product[]>('/api/products', { params: { q: term } }))
).subscribe(list => this.results.set(list));

// signal-based read — auto-cancels when id() changes
readonly user = httpResource<User>(() => `/api/users/${this.id()}`);

Interview questions

Interceptors, error handling and request patterns are staples of Angular rounds in Indian product and services interviews — the ones below run screening depth to senior deep-dive:

Beginner What is an HTTP interceptor and what is it used for?

Middleware in the HttpClient pipeline that sees every outgoing request and incoming response. It centralises cross-cutting concerns — attaching auth tokens, logging, global error handling, retries, caching — so the logic lives in one place and applies to all requests instead of being repeated in each service.

Beginner How do you register a functional interceptor in a standalone app?

provideHttpClient(withInterceptors([authInterceptor, errorInterceptor])) in app.config.ts. Each is an HttpInterceptorFn, and they execute in the order listed. This is the modern replacement for providing class tokens through HTTP_INTERCEPTORS.

Beginner HttpRequest is immutable — how do you modify a request in an interceptor?

You cannot mutate it. Call req.clone({ setHeaders: {...} }) (or set params, body, url) to get a modified copy and pass that to next(). Cloning leaves the original untouched, which matters because the same request flows through several interceptors.

Beginner Functional vs class-based (DI) interceptors — which is current?

Functional interceptors (HttpInterceptorFn + withInterceptors, Angular 15+) are recommended: tree-shakable, composable, dependencies via inject(), predictable ordering. Class interceptors implementing HttpInterceptor and registered via HTTP_INTERCEPTORS still work but are legacy — new code should use functions.

Intermediate How do you attach a JWT/auth token to every request?

An auth interceptor injects the auth service, reads the token, clones the request adding an Authorization: Bearer … header, and forwards it. Skip public endpoints via an HttpContext flag or URL check. Angular's built-in XSRF interceptor uses the same clone-and-add-header idea for X-XSRF-TOKEN.

Intermediate How do you handle HTTP errors globally, and why must you rethrow?

Pipe catchError onto next(req) in an interceptor and branch on HttpErrorResponse.status — redirect on a 401, toast on a 500. You must return throwError(() => err) (or a replacement error) so the observable stays in an error state and the component's error callback still fires; swallowing it makes a failed call look successful.

Intermediate What is HttpContext and when would you use it?

A typed, per-request metadata map (keys are HttpContextToken<T>) that rides on req.context and is never sent to the server. Use it to pass hints to interceptors — skip auth, mark a request cacheable, suppress the global spinner — without abusing headers or sniffing URLs.

Intermediate HttpParams and HttpHeaders are immutable — what does that imply?

Every set()/append() returns a new instance; the original is unchanged. So params.set('a','1') on its own is a no-op unless you reassign or chain (params = params.set(...)). Passing a plain object to the params option lets HttpClient build them for you.

Intermediate In what order do multiple interceptors execute?

They form a chain in registration order. On the way out (request phase) they run top-to-bottom before the call hits the network; on the way back (response phase) events flow through in reverse. So an outer logging interceptor sees the request first and the response last.

Intermediate How do you cancel an in-flight HTTP request?

Unsubscribe — that aborts the underlying request. In practice switchMap does it for you (cancelling the previous request when a new value arrives, the classic typeahead fix), and takeUntilDestroyed/the async pipe cancel on teardown. Only unsubscribing cancels; those operators cancel because they unsubscribe.

Pro How would you implement a caching interceptor, and what are the pitfalls?

Keep a Map keyed by req.urlWithParams; on a cacheable GET, return the stored HttpResponse if present, otherwise let it through and store the response via tap. Gate it with an HttpContext token so callers opt in. Pitfalls: cache only GETs (never mutations), invalidate on writes, and bound size/TTL so you avoid stale data and unbounded memory growth.

Pro How does Angular's built-in XSRF protection work?

HttpClient's XSRF interceptor reads a token from the XSRF-TOKEN cookie and echoes it back as the X-XSRF-TOKEN header on mutating (POST/PUT/DELETE), same-origin requests — not on GET/HEAD. The backend sets the cookie and verifies the header, proving the request came from your app. Customise the names with withXsrfConfiguration, or turn it off with withNoXsrfProtection.

Pro When would you choose httpResource over a classic Observable HttpClient call?

httpResource (Angular v19/20) is a signal-based read: it fires eagerly, exposes value/status/error as signals, re-requests when its input signals change, and auto-cancels the previous request — ideal for template-driven reads in a signals app, and it still runs through interceptors. Keep plain HttpClient for mutations (POST/PUT) and for streams needing explicit operator control (switchMap, retries, manual cancellation). Bridging the other way, toSignal(http.get<T>(...)) turns a one-shot call into a signal.
