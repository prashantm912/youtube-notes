What this module covers

RxJS models values that arrive over time as Observables — lazy, cancellable streams you shape with pipeable operators instead of nested callbacks. This module covers the Observable contract and the creation functions that produce streams, the everyday operators map/filter/tap and how to read a marble diagram, and the part interviewers press hardest: the four higher-order mapping operators and the reasoning that picks the right one. Angular leans on all of it, so this is the reactive foundation the rest of the advanced track builds on.

Topics

The Observable contract: lazy, cancellable streams

An Observable is a lazy producer of zero or more values pushed to an observer over time, ending in either complete or error — or never ending at all. You consume it by subscribing with up to three callbacks: next for each value, error once on failure, complete once on success. Subscribing is also what starts the work — an Observable is cold by default and does nothing until someone listens. subscribe() returns a Subscription whose unsubscribe() tears the producer down, which is how RxJS cancels an in-flight HTTP request or a running interval.

Why it matters: this lazy, cancellable, multi-value contract is exactly what separates an Observable from a Promise, and "Observable vs Promise" opens most Angular interviews. It is also why a forgotten unsubscribe can leak, and why "my HTTP call never fired" is almost always a missing subscription.

const sub = interval$.subscribe({
  next:     v => console.log(v),    // 0, 1, 2, … each emission
  error:    e => console.error(e),  // at most once, ends the stream
  complete: () => console.log('done'),
});
sub.unsubscribe();                   // cancels the producer

Creating observables: of, from, interval, fromEvent

Most streams come from a creation function rather than the raw Observable constructor. of(a, b, c) emits each argument as-is then completes; from(...) turns an array, iterable, Promise or observable-like into a stream, emitting each element; interval(ms) and timer emit ascending numbers on a clock; fromEvent(target, 'input') bridges DOM (or Node) events into a stream — the usual entry point for typeaheads and drag handlers. HttpClient methods are creation functions too: each returns a cold, single-shot Observable that emits one response and then completes.

Why it matters: choosing the factory that fits the source — a fixed set of values, an existing Promise, a timer, or a UI event — is table-stakes, and a debounced fromEvent pipeline is the exact setup behind the switchMap questions further down.

of(1, 2, 3);                        // 1, 2, 3, then complete
from([10, 20, 30]);                 // each array element, then complete
from(fetch('/api').then(r => r.json())); // a Promise becomes a one-value stream
interval(1000);                     // 0, 1, 2, … on a timer (never completes)
fromEvent(searchBox, 'input');      // every keystroke as an event

Pipeable operators and marble intuition: map, filter, tap

Operators are pure functions that take an Observable and return a new one, composed with pipe() as source.pipe(op1, op2, op3). They never mutate the source — each returns a fresh stream, so one source can feed several independent pipelines. map transforms every emitted value, filter drops values that fail a predicate, and tap runs a side effect (logging, debugging) without touching the values flowing through. A good rule: if tap changes what downstream sees, you reached for the wrong operator.

Why it matters: a marble diagram is the standard way operators are explained and interviewed — a horizontal timeline where each marble is an emission, a vertical bar is complete and an X is error. Reading map/filter as "same timeline, changed or fewer marbles" is the intuition you carry into the harder operators below.

source$.pipe(
  filter(n => n % 2 === 0),   // keep only evens
  map(n => n * 10),           // transform each value
  tap(n => console.log(n)),   // peek for debugging, value unchanged
).subscribe();

Higher-order observables and flattening

When an operator's callback itself returns an Observable — say each keystroke maps to an http.get() — you get an Observable of Observables, a higher-order observable. Subscribing to the outer stream would hand you inner Observables, not their values, so you have to flatten: subscribe to each inner stream and re-emit its values on the outer one. The naive fix — calling subscribe() inside a subscribe() — is the classic anti-pattern: it leaks, it cannot be cancelled cleanly, and it throws away ordering guarantees.

Why it matters: the flattening operators exist precisely to replace nested subscribes. They all map-then-flatten; what differs is how each handles a new outer value while an inner stream is still running — cancel it, run both, queue it, or ignore the new one. That single question is the whole topic, and it is the most-asked RxJS material in Indian interviews.

// anti-pattern: subscribe inside subscribe
this.search$.subscribe(term => {
  this.http.get(url(term)).subscribe(res => this.show(res));
});

// flattened, cancellable, correct
this.search$.pipe(
  switchMap(term => this.http.get(url(term)))
).subscribe(res => this.show(res));

Choosing switchMap vs mergeMap vs concatMap vs exhaustMap

All four map an outer value to an inner Observable and flatten the result; they differ only in concurrency. switchMap unsubscribes from the previous inner stream the moment a new value arrives and switches to the newest — only the latest matters, which is ideal for a typeahead search where an older request is now irrelevant and cancelling it also kills a race. mergeMap (formerly flatMap) subscribes to every inner stream concurrently and merges all their emissions — maximum throughput, no cancellation, no ordering. concatMap queues inner streams and runs them one at a time in order, waiting for each to complete. exhaustMap ignores new outer values while an inner stream is still running — the mirror image of switchMap.

Why it matters: the interview drill is mapping a scenario to an operator — reads that supersede each other use switchMap; independent parallel requests use mergeMap; work whose order must be preserved uses concatMap; an action that should ignore repeats while busy (a save or login button) uses exhaustMap. Two traps to name aloud: switchMap on a POST can cancel an in-flight write, so prefer concatMap/mergeMap/exhaustMap for mutations; and mergeMap on a fast source can open unbounded concurrent requests unless you cap it with its concurrency argument.

switchMap(x => inner(x))   // new x cancels the previous inner   → typeahead search
mergeMap(x => inner(x))    // all inners run at once, unordered   → parallel writes
concatMap(x => inner(x))   // queue: one inner at a time, in order → ordered writes
exhaustMap(x => inner(x))  // ignore new x while inner runs        → submit / login button

Interview questions

Most-asked RxJS questions in Indian interviews — "Observable vs Promise" is a near-certain screen, and the switchMap-vs-mergeMap-vs-concatMap-vs-exhaustMap comparison is the single most repeated RxJS question for Angular roles:

Beginner What is an Observable, and how is it different from a Promise?

An Observable is a lazy stream that can push zero, one or many values over time and is cancellable. A Promise is eager (runs immediately), resolves exactly once, and cannot be cancelled or composed with operators. Observables also start only on subscribe, so the same Observable can be run many times; a Promise runs once regardless.

Beginner What does it mean that an Observable is "cold" or "lazy", and when does work start?

Cold means the producer is created per subscriber and does nothing until subscribed. The work — an HTTP request, a timer, an event listener — starts on .subscribe() (or when the async pipe subscribes). No subscriber means no side effect, which is why a returned-but-never-subscribed HTTP Observable never hits the network.

Beginner Difference between of and from?

of(a, b, c) emits each argument whole, in order, then completes. from(x) expects a single iterable/Promise/observable-like and emits its elements one by one — from([1,2,3]) emits 1, 2, 3, and from(promise) emits the resolved value then completes. So of([1,2,3]) emits one array; from([1,2,3]) emits three numbers.

Beginner What is pipe(), and what is a pipeable operator?

A pipeable operator is a pure function that takes an Observable and returns a new Observable. pipe() is the method that chains them left to right — source.pipe(map(...), filter(...)) — building a transformation pipeline without mutating the source. Because each operator returns a new stream, the original stays reusable.

Beginner What is tap used for, and what should it never do?

tap performs a side effect — logging, debugging, triggering something external — while passing every value through unchanged. It must not transform the value or the stream; if you find yourself relying on tap to change downstream data, you wanted map instead. It is the safe way to inspect a pipeline.

Intermediate map vs filter, and how do operators compose without breaking the source?

map transforms every value one-to-one; filter passes only values that satisfy a predicate and drops the rest. Both are pure and return new Observables, so chaining them in pipe() never mutates the source — the same source Observable can feed multiple pipelines independently, each with its own operators.

Intermediate What is a higher-order observable, and why is subscribing inside a subscribe an anti-pattern?

A higher-order observable is an Observable whose values are themselves Observables — common when a mapping callback returns http.get(...). Nested subscribe() calls "work" but leak subscriptions, cannot be cancelled together, break error propagation, and lose ordering. The fix is a flattening operator (switchMap/mergeMap/concatMap/exhaustMap) that subscribes to the inner stream and manages it for you.

Intermediate How does switchMap behave, and what is its canonical use case?

On each new outer value, switchMap unsubscribes from the previous inner Observable (cancelling it) and switches to the new one, so only the latest inner stream is active. Canonical use: a typeahead search — a new keystroke makes the previous request obsolete, and cancelling it avoids an out-of-order stale response overwriting a newer one.

Intermediate When would you choose mergeMap over switchMap, and what is its risk?

mergeMap keeps every inner Observable running concurrently and merges their emissions — use it when results must not be cancelled and can complete in any order, e.g. firing independent writes or loading many items in parallel. Risk: no ordering, and on a fast source it can open unbounded simultaneous requests; cap it with the concurrency argument (mergeMap(fn, concurrent)).

Intermediate What does concatMap guarantee that mergeMap does not?

Order. concatMap queues inner Observables and subscribes to the next only after the previous completes, so emissions stay in source order and never overlap. Use it for sequential work where order matters — dependent writes, a queue of saves. The cost is throughput: a slow inner stream blocks the queue behind it.

Pro What is exhaustMap for, and how is it the opposite of switchMap?

exhaustMap ignores new outer values while the current inner Observable is still running, then accepts the next after it completes. Where switchMap always favours the newest (cancelling the old), exhaustMap favours the in-progress one (dropping the new). Ideal for a login or save button: rapid repeated clicks should not stack duplicate submissions.

Pro Pick the operator: (a) a search box, (b) saving edits where order matters, (c) many independent GETs, (d) a submit button that must ignore double-clicks. Justify each.

(a) switchMap — cancel the superseded request. (b) concatMap — preserve order, one write at a time. (c) mergeMap — run in parallel, order irrelevant. (d) exhaustMap — ignore new clicks while the request is in flight. The reasoning axis is always: on a new value while busy, do you cancel, queue, run in parallel, or ignore?

Pro Why can switchMap be dangerous on a POST/write, and what do you use instead?

Because switchMap cancels the previous inner stream on a new emission, a still-pending mutation can be aborted mid-flight — the server may or may not have applied it, leaving inconsistent state. For writes prefer concatMap (ordered, no loss), mergeMap (parallel, no loss), or exhaustMap (ignore duplicates). Reserve switchMap for idempotent reads whose older results you genuinely no longer want.
