Module: Advanced Directives

Directives are the layer where Angular meets the raw DOM — and "write a directive that does what *ngIf does" plus "what is hostDirectives?" are two of the fastest ways an interviewer separates framework users from framework understanders.

Knowledge Library → Courses → Angular Advanced → Advanced Directives

What this module covers

Two big skills live here. First, custom structural directives: how the * shorthand desugars to an <ng-template>, and how TemplateRef plus ViewContainerRef let you add and remove views yourself — the machinery behind *ngIf. Second, reusing behaviour on a host element: attribute directives with HostBinding/HostListener and the host property, safe DOM writes through Renderer2, and — the modern headline feature — host directives (the directive composition API, Angular v15+), which lets a component apply directives to its own host from within its TypeScript class.

Topics

Custom structural directives: TemplateRef, ViewContainerRef and the microsyntax

A structural directive changes layout by adding or removing views from the DOM. The *appUnless shorthand is pure sugar: Angular rewrites it into an <ng-template> wrapping the host element, then instantiates your directive on that template. You inject two things — a TemplateRef (a handle to the un-rendered template chunk) and a ViewContainerRef (the slot in the DOM where views get inserted) — and you decide when to call createEmbeddedView() or clear().

Why it matters: this is the classic Angular deep-dive question. Understanding the desugaring explains why only one structural directive is allowed per element, why the microsyntax (let, of, as) exists, and how the built-in directives you use daily actually work under the hood.

@Directive({ selector: '[appUnless]' })
export class UnlessDirective {
  private tpl = inject(TemplateRef<unknown>);
  private vcr = inject(ViewContainerRef);
  private shown = false;

  @Input() set appUnless(condition: boolean) {
    if (!condition && !this.shown) {
      this.vcr.createEmbeddedView(this.tpl);  // insert the view
      this.shown = true;
    } else if (condition && this.shown) {
      this.vcr.clear();                        // remove it
      this.shown = false;
    }
  }
}
<!-- what you write -->
<li *appUnless="isLoggedIn">Please sign in</li>

<!-- what Angular desugars it to -->
<ng-template [appUnless]="isLoggedIn">
  <li>Please sign in</li>
</ng-template>

Attribute directives, HostBinding, HostListener and the host property

An attribute directive changes the appearance or behaviour of an existing element without altering DOM structure. It reads/writes the host through binding: @HostBinding('class.active') ties a directive field to a host class/style/attribute/property, and @HostListener('mouseenter') subscribes to host events. Modern Angular prefers declaring both declaratively in the host metadata object (host: { '[class.active]': 'active', '(mouseenter)': 'onEnter()' }) — the decorators still work but exist mainly for backwards compatibility.

Why it matters: attribute directives are the everyday way teams package reusable UI behaviour (highlights, autofocus, tooltips, permission-based disabling). Interviewers expect you to build one live and to know the decorator-versus-host-property trade-off.

@Directive({ selector: '[appHighlight]' })
export class HighlightDirective {
  private el = inject(ElementRef);
  private renderer = inject(Renderer2);

  @HostBinding('class.active') active = false;

  @HostListener('mouseenter') onEnter() {
    this.renderer.setStyle(this.el.nativeElement, 'background', 'yellow');
    this.active = true;
  }
  @HostListener('mouseleave') onLeave() {
    this.renderer.removeStyle(this.el.nativeElement, 'background');
    this.active = false;
  }
}

Renderer2 and safe DOM manipulation

Reaching for ElementRef.nativeElement and mutating it directly works in the browser but breaks server-side rendering, web workers, and Angular's XSS protections — the platform may not even have a real DOM. Renderer2 is the abstraction that keeps DOM writes platform-independent and safe: setStyle, addClass, setAttribute, appendChild, listen, and so on all go through a renderer the platform provides.

Why it matters: "why not just use nativeElement?" is a standard follow-up once you mention DOM access in a directive. The right answer — SSR compatibility, security, and testability — signals you think about Angular beyond the browser.

Host directives — the directive composition API (v15+)

Before v15, sharing directive behaviour meant asking every consumer to add both your component and a pile of directives in their template. The directive composition API moves that inside the class: a hostDirectives array on @Component/@Directive applies standalone directives to the host element automatically. You can selectively expose a host directive's inputs and outputs — and alias them — so the composed component presents one clean, intentional API.

Why it matters: this is the current "advanced directives" headline and a favourite interview topic. Know the rules: host directives must be standalone; their inputs/outputs are private unless you list them; aliasing renames them for the host; and Angular de-duplicates and orders them deterministically. It is how Angular Material now shares behaviours like focus trapping and ripples.

@Component({
  selector: 'app-menu',
  hostDirectives: [
    CdkTrapFocus,                                  // applied, API hidden
    { directive: HasColor, inputs: ['color'] },    // expose 'color' input
    { directive: Tooltip, inputs: ['tooltip: menuTip'] }, // aliased input
  ],
})
export class MenuComponent {}

Interview questions

Directives are a favourite deep-dive in Indian Angular interviews — product companies especially love "recreate *ngIf" and the newer hostDirectives questions:

Beginner What are the three kinds of directives in Angular?

Components (a directive with a template), structural directives (change DOM layout by adding/removing elements — *ngIf, *ngFor), and attribute directives (change appearance/behaviour of an existing element — ngClass, a custom highlight). A component is technically a directive with a view.

Beginner What does the asterisk (*) in *ngIf actually mean?

It is shorthand. Angular desugars *appUnless="x" into an <ng-template [appUnless]="x"> that wraps the host element. The template holds the un-rendered content; the directive decides whether to stamp it out. That desugaring is why only one structural directive is allowed per element.

Beginner What are TemplateRef and ViewContainerRef?

TemplateRef is a handle to a block of template that has not been rendered yet (the content inside an <ng-template>). ViewContainerRef is the location in the DOM where views can be inserted. A structural directive injects both, then calls viewContainer.createEmbeddedView(templateRef) to render and clear() to remove.

Beginner What is the difference between a structural and an attribute directive?

Structural directives add or remove elements, so they change the DOM tree (and use the * shorthand / an <ng-template>). Attribute directives keep the element in place and only change its look or behaviour via host bindings and listeners. Different jobs, different mechanics.

Intermediate How would you write a custom structural directive that reverses ngIf?

Inject TemplateRef and ViewContainerRef. In an input setter, when the condition is false and nothing is rendered, call createEmbeddedView(tpl); when it becomes true and a view exists, call viewContainer.clear(). Guard with a boolean flag so you do not re-create or re-clear on every change-detection pass.

Intermediate What is @HostBinding and @HostListener — and the modern alternative?

@HostBinding('class.active') binds a directive property to a class/style/attribute/property on the host; @HostListener('click', ['$event']) runs a method on a host event. The current preferred style is the host metadata object (host: { '[class.active]': 'active', '(click)': 'onClick($event)' }); the decorators remain for backwards compatibility.

Intermediate Why use Renderer2 instead of touching nativeElement directly?

Direct nativeElement access assumes a browser DOM — it breaks under SSR, prerendering, and web workers, and sidesteps Angular's sanitization. Renderer2 abstracts the platform, so setStyle/addClass/setAttribute work everywhere and stay safe. It also keeps directives unit-testable without a real DOM.

Intermediate What is the directive composition API (hostDirectives) and what problem does it solve?

Introduced in v15, it lets a component/directive apply other standalone directives to its own host element via a hostDirectives array — no template changes required from consumers. It solves reuse: behaviours like focus trapping, tooltips or a colour API can be composed into a component instead of being re-declared in every template that uses it.

Intermediate How do you expose or rename a host directive's inputs and outputs?

By default a host directive's inputs/outputs are hidden. List them explicitly: { directive: Tooltip, inputs: ['tooltip'], outputs: ['closed'] }. To rename, use the alias syntax inputs: ['tooltip: menuTip'] so consumers bind [menuTip] while the directive still receives tooltip. Only the ones you list become part of the host's public API.

Pro What are the constraints and ordering rules for host directives?

Host directives must be standalone. They are instantiated before the component's own constructor, and their lifecycle hooks run before the host's. Angular de-duplicates a directive applied more than once and resolves the "diamond" case; conflicting input aliases are a compile-time error. Order in the array is deterministic, which matters when directives write the same host binding.

Pro How do you pass and update context in an embedded view (like $implicit and let)?

Pass a context object as the second argument: createEmbeddedView(tpl, { $implicit: value, index: i }). In the template, let-item binds to $implicit and let-i="index" binds a named key. To update, mutate the same context object's fields (the view holds a reference) or recreate the view — this is exactly how *ngFor exposes $index, $first and friends.

Pro How do you make a custom structural directive type-check its template correctly?

Add a static ngTemplateContextGuard(dir, ctx): ctx is SomeType so the template compiler knows the type of $implicit and named context keys. For narrowing an input (like *ngIf narrowing to non-null inside the block), add a static ngTemplateGuard_appUnless or the 'binding' guard convention. Without these, strict template type-checking treats context variables as any.
