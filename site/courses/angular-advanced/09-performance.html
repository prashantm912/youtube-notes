Module: Performance Optimization

Fast Angular is mostly about doing less work: checking fewer components, re-rendering fewer DOM nodes, and shipping less JavaScript up front. OnPush and @defer are the two levers interviewers probe hardest right now.

Knowledge Library → Courses → Angular Advanced → Performance Optimization

What this module covers

Two fronts win or lose Angular performance: runtime (how often change detection runs and how much DOM it touches) and load (how much code the browser downloads before it can render). This module orients you on the runtime tools — OnPush, @for tracking, pure pipes, and the signals/zoneless direction — and the load tools — deferrable views, preloading strategies, and bundle budgets — then points you at the resources that teach each in depth. Official angular.dev guidance comes first throughout.

Topics

Change detection and the OnPush strategy

By default Angular uses the Default (CheckAlways) strategy: after any event, timer, or HTTP callback, it walks the entire component tree checking every binding. ChangeDetectionStrategy.OnPush tells Angular to skip a component's subtree unless something meaningful happened to it — a new input reference arrived, an event fired inside it, an async pipe emitted, or you called markForCheck(). On a large tree this turns thousands of checks per cycle into a handful.

Why it matters: OnPush is the single most-tested Angular performance topic, and it only pays off with immutable data — mutating an input object in place leaves the reference unchanged, so OnPush never notices. Knowing exactly what marks an OnPush component "dirty" is the difference between a screening answer and a senior one, and it is also why Angular is steering the whole framework toward signals and zoneless, where this dirty-tracking becomes automatic.

@Component({
  selector: 'app-row',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `{{ user().name }}`,
})
export class RowComponent {
  user = input.required<User>(); // new reference in => re-check
}

Efficient lists: @for track and trackBy

When a list re-renders, Angular must decide which DOM nodes to keep, move, or destroy. Without identity tracking it may tear down and rebuild every row on each change — expensive and it drops focus and animation state. In the new control flow, @for makes track mandatory: you name a stable key (usually item.id) so Angular reuses the matching node instead of recreating it. The legacy equivalent was *ngFor with an optional trackBy function.

Why it matters: a missing or wrong track key is one of the most common real-world perf bugs, and "why is track required in @for?" is a current favourite interview question. Use $index only for genuinely static, index-identified collections.

@for (item of items(); track item.id) {
  <li>{{ item.name }}</li>
} @empty {
  <li>No items</li>
}

Pure pipes and reducing change-detection work

Pipes are pure by default: Angular only re-runs a pure pipe's transform when its input changes by reference (or a primitive by value), so results are effectively memoized across change-detection cycles. An impure pipe (pure: false) runs on every cycle — occasionally necessary, but a performance trap if used carelessly. Preferring pure pipes over method calls in templates keeps repeated work out of the hot path.

The bigger shift is architectural: signals let Angular track precisely which template expressions depend on which state, updating only what actually changed, and zoneless change detection removes Zone.js so detection is driven by signals and events rather than blanket monkey-patching. Together they make much of the manual OnPush discipline automatic — increasingly the default direction of modern Angular.

Deferrable views: @defer

Introduced in Angular 17, @defer is the headline load-time optimization. Everything inside a @defer block — components, directives, pipes — is split into its own JavaScript chunk and downloaded only when a trigger fires, shrinking the initial bundle and improving Core Web Vitals like LCP. Triggers include on idle, on viewport, on interaction, on hover, on timer, on immediate, and conditional when. A separate prefetch trigger lets you fetch the chunk early while still deciding when to render, and @placeholder, @loading, and @error blocks manage the states in between.

Why it matters: this is the top current Angular interview topic. Be ready to contrast it with route-level lazy loading (see below) and to explain that dependencies inside a @defer block must be standalone and not referenced elsewhere in the same file, or they cannot actually be deferred.

@defer (on viewport; prefetch on idle) {
  <app-comments />
} @placeholder {
  <p>Scroll to load comments</p>
} @loading (minimum 200ms) {
  <app-spinner />
}

Preloading strategies and bundle optimization

Lazy loading routes with loadComponent/loadChildren keeps the initial bundle small, but the first navigation to a lazy route then pays a download delay. Preloading closes that gap by fetching lazy chunks in the background after the app boots. PreloadAllModules grabs everything as soon as it can (fine for small/medium apps); a custom PreloadingStrategy — or a Quicklink/predictive approach — lets you preload only routes the user is likely to visit. You wire it up with withPreloading in provideRouter.

On the build side, the modern esbuild-based application builder handles minification, tree-shaking, and code splitting for you. Your job is mostly to set bundle budgets in angular.json so a size regression fails the build instead of shipping — a standard production-hygiene question.

provideRouter(routes, withPreloading(PreloadAllModules))

Interview questions

Most-asked Angular performance questions in Indian interviews — OnPush and @defer dominate current product-company rounds:

Beginner What is change detection and what triggers it?

The process by which Angular checks component state against the DOM and updates bindings. With Zone.js it is triggered by async events — DOM events, timers (setTimeout/setInterval), and HTTP/Promise callbacks — each of which schedules a check of the component tree.

Beginner What does ChangeDetectionStrategy.OnPush do?

It tells Angular to skip checking a component (and its subtree) unless one of a few things happens: an @Input gets a new reference, an event fires inside the component, an async pipe in its template emits, or markForCheck() is called. It reduces the number of components checked per cycle.

Beginner Why is track mandatory in @for, and what was trackBy?

track gives each item a stable identity so Angular reuses existing DOM nodes on updates instead of destroying and recreating rows — faster, and it preserves focus/animation state. @for makes it required for safe diffing; the old *ngFor equivalent was an optional trackBy function returning a unique key.

Beginner Pure vs impure pipe — what is the performance difference?

A pure pipe re-runs only when its input changes by reference (or a primitive by value), so its result is cached across cycles. An impure pipe (pure: false) runs on every change-detection cycle. Prefer pure pipes; reach for impure only when you truly must react to internal mutation, and know the cost.

Intermediate With OnPush, what marks a component dirty?

Four things: a bound @Input receives a new reference; an event handler fires within the component's template; an async pipe used in the template emits a new value; or code calls ChangeDetectorRef.markForCheck(). Absent these, Angular skips the component even if its data mutated internally.

Intermediate Why does mutating an @Input object break OnPush, and how do you fix it?

OnPush compares inputs by reference. Mutating the object in place (user.name = 'x') keeps the same reference, so Angular sees no change and skips the view. Fix it with immutable updates — create a new object/array ({ ...user, name: 'x' }) — or drive the value through a signal or an observable + async pipe.

Intermediate How does the async pipe cooperate with OnPush?

The async pipe subscribes for you and calls markForCheck() on every emission, so an OnPush component updates when the stream emits — no manual subscription or detectChanges. It also unsubscribes on destroy, avoiding leaks. This is the idiomatic way to feed OnPush components with observable data.

Intermediate What is @defer and what problem does it solve? Name some triggers.

Introduced in Angular 17, @defer splits its block's components/directives/pipes into a separate chunk loaded only when a trigger fires, shrinking the initial bundle and improving LCP. Triggers: on idle (default), on viewport, on interaction, on hover, on timer, on immediate, and conditional when.

Intermediate Difference between the main trigger and a prefetch trigger in @defer?

They are independent. The main trigger controls when the block renders (swaps placeholder for real content); the prefetch trigger controls when the JavaScript downloads. A common pairing is on interaction; prefetch on idle — fetch the chunk during idle time so it is ready the instant the user interacts.

Intermediate What preloading strategies does Angular offer for lazy routes?

None by default (lazy chunks load on navigation). PreloadAllModules background-loads every lazy route after boot — good for small/medium apps. For large apps you implement a custom PreloadingStrategy (or a Quicklink/predictive approach) to preload only likely routes. Configure via withPreloading(...) in provideRouter.

Pro markForCheck vs detectChanges vs detach/reattach?

markForCheck() flags this view and its ancestors dirty so they are checked on the next cycle — the OnPush workhorse. detectChanges() runs change detection on this view synchronously, now. detach()/reattach() remove a subtree from the check tree entirely, letting you hand-drive detection for a hot component (e.g. a high-frequency ticker). Reach for the lightest tool that works.

Pro How do signals and zoneless reduce change-detection work versus Zone.js + OnPush?

Zone.js triggers a tree-wide check after any async event; OnPush prunes that tree by reference checks you maintain by hand. Signals instead record exactly which template expressions read which state, so only affected views update — dependency tracking is automatic and finer-grained. Zoneless removes Zone.js entirely, driving detection from signals/events, cutting overhead and easing debugging. It is the direction Angular is defaulting toward.

Pro Why can calling a method in a template hurt performance, and what are the fixes?

A method or getter bound in the template ({{ compute() }}) runs on every change-detection cycle, regardless of whether its inputs changed — easy to call thousands of times a second. Fixes: precompute in the component and bind a field, use a pure pipe (memoized by input), use a computed() signal, or move the value behind an observable + async pipe.

Pro How does @defer differ from route-level lazy loading, and what are bundle budgets?

Route lazy loading splits code at the route boundary — a chunk loads when you navigate there. @defer splits within a single view, deferring part of a page you are already on based on idle/viewport/interaction. They compose. Bundle budgets are thresholds in angular.json that warn or fail the build when a bundle exceeds a set size, catching regressions before release.
