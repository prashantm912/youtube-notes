Module: Routing Advanced

Lazy loading and route guards are where routing stops being configuration and starts being architecture — and the modern, functional versions of both are exactly the topics an interviewer uses to tell a junior Angular developer from a senior one.

Knowledge Library → Courses → Angular Advanced → Routing Advanced

What this module covers

This module takes routing past a flat list of paths. First, structure: child (nested) routes that give a feature its own sub-navigation through a nested <router-outlet>. Then performance: lazy loading with loadComponent and loadChildren, which split a feature into its own JavaScript chunk that downloads only on demand. Finally, control: functional route guards (CanActivateFn, CanMatchFn, CanDeactivateFn) that decide whether navigation may proceed, and resolvers (ResolveFn) that fetch data before a route activates. The class-based versions of guards and resolvers are deprecated — everything here is the current, function-based API.

Topics

Child and nested routes

A route can carry its own children array, and the parent component renders them in its own <router-outlet>. That is how a feature like a product page keeps a stable shell (tabs, header) while an inner outlet swaps between info and reviews as the URL deepens. The parent path prefixes every child path, so products/42/reviews resolves the parent, then the child, then paints the child into the nested outlet. An empty-path child (path: '') is the idiomatic default view for the section.

Why it matters: nested routes are the backbone of master-detail and dashboard layouts, and interviewers use "how do you show a sub-view without reloading the parent?" to check that you understand the outlet-per-level model rather than stacking everything in one flat config.

const routes: Routes = [
  { path: 'products', component: ProductsComponent, children: [
      { path: '',        component: ProductListComponent },   // default child
      { path: ':id',     component: ProductDetailComponent }, // products/42
  ]},
];
// ProductsComponent template renders its own nested <router-outlet />

Lazy loading: loadComponent and loadChildren

Instead of component, a route can declare a loader that returns a dynamic import(). loadComponent lazy-loads a single standalone component into its own chunk; loadChildren lazy-loads a whole set of child routes (typically exported from a *.routes.ts file) into one chunk. Angular downloads that chunk only when the user first navigates to the matching path, so large feature areas — admin, reports, checkout — stay out of the initial bundle. In the standalone era neither one needs an NgModule; loadChildren simply resolves to an array of routes.

Why it matters: lazy loading is the single highest-leverage change to initial load time, and knowing loadComponent (component-level, Angular 14+) versus loadChildren (route-tree-level) — plus the fact that a default export lets you drop the .then() — is a standard senior-screening question in Indian product interviews.

export const routes: Routes = [
  // one standalone component, its own chunk
  { path: 'profile', loadComponent: () =>
      import('./profile/profile.component').then(m => m.ProfileComponent) },

  // an entire feature area (its child routes) in one chunk
  { path: 'admin', loadChildren: () =>
      import('./admin/admin.routes').then(m => m.ADMIN_ROUTES) },
];

Functional route guards: CanActivate, CanMatch, CanDeactivate

A guard is a function you attach to a route to approve or block navigation. All four kinds are now plain functions typed by their signature — CanActivateFn (may the user enter this route?), CanActivateChildFn (its children?), CanMatchFn (may this route even be matched?), and CanDeactivateFn (may the user leave — the unsaved-changes prompt). Each returns a boolean, a UrlTree (redirect), or an async wrapper of either, and calls inject() to reach services. The old class-based, interface-implementing guards (CanActivate as a class, CanLoad) are deprecated; functional guards are terser, testable, and tree-shakable.

Why it matters: CanMatch versus CanActivate is the sharpest question here. CanMatch runs during path matching — a rejection prevents the lazy chunk from ever downloading and lets the router fall through to try the next route, which is why it replaced CanLoad and is the right guard for auth-gating a lazy feature. CanActivate runs after the route is matched and only blocks the navigation.

export const authGuard: CanActivateFn = (route, state) => {
  const auth   = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn()
    ? true
    : router.createUrlTree(['/login'], { queryParams: { redirect: state.url } });
};
// { path: 'admin', canMatch: [authGuard], loadChildren: () => import('./admin/admin.routes')... }

Resolvers: data before a route activates

A resolver is a ResolveFn<T> attached under a route's resolve key. It runs after all guards succeed and before the component is created, and the router waits for it to complete, so the component renders with its data already present instead of flashing an empty state and fetching in ngOnInit. The resolved value lands in ActivatedRoute.data under the key you chose; with withComponentInputBinding() it can even arrive as a component @Input. A resolver may also return a RedirectCommand to send navigation elsewhere on failure.

Why it matters: resolvers are the "no loading spinner on the detail page" pattern, but the senior nuance is knowing the cost — navigation is blocked until the data settles, so a slow resolver makes the whole app feel frozen. Being able to say when not to use one (prefer rendering a shell and streaming data in for slow or optional calls) is what distinguishes a considered answer.

export const userResolver: ResolveFn<User> = (route) => {
  const id = route.paramMap.get('id')!;
  return inject(UserService).getUser(id);   // router waits for this
};
// { path: 'user/:id', resolve: { user: userResolver }, component: UserComponent }
// in the component: this.route.snapshot.data['user'] as User

Interview questions

Most-asked advanced-routing questions in Indian interviews — lazy loading and the modern functional guards are a near-certain senior-round pair for Angular roles:

Beginner What are child (nested) routes and how do you display them?

Routes listed under a parent's children array. The parent component renders them in its own nested <router-outlet>, so the parent's shell stays put while the inner outlet swaps sub-views as the URL deepens (e.g. products/42/reviews).

Beginner What is lazy loading in Angular routing and why use it?

Loading a route's code only when the user navigates to it, via loadComponent/loadChildren returning a dynamic import(). Angular puts that code in a separate chunk kept out of the initial bundle, so the app starts faster and downloads feature code on demand.

Beginner loadComponent vs loadChildren — what's the difference?

loadComponent lazy-loads a single standalone component. loadChildren lazy-loads a whole set of child routes (a feature area) that resolves to an array of Routes. Use loadComponent for one screen, loadChildren for a section with its own sub-routes.

Beginner What is a route guard, and what is a resolver?

A guard is a function that decides whether navigation into or out of a route may proceed (returns boolean or a UrlTree redirect). A resolver is a function that fetches data before the route activates, so the component renders with data already available.

Intermediate Class-based guards vs functional guards — why functional now?

Class-based guards (implementing CanActivate, CanLoad, etc.) are deprecated. Functional guards are plain functions typed as CanActivateFn/CanMatchFn/etc. that call inject() for dependencies — less boilerplate, easily composed and reused, and tree-shakable. New apps should write functional guards only.

Intermediate CanActivate vs CanMatch — the difference that matters?

CanMatch runs during path matching: rejecting it stops the route from matching, prevents a lazy chunk from downloading, and lets the router fall through to the next candidate route. CanActivate runs after the route is matched and simply blocks the navigation. CanMatch replaced the old CanLoad.

Intermediate What does CanDeactivate do, and a typical use?

It decides whether the user may leave the current route. The function receives the component instance, so you can inspect state — the classic use is "you have unsaved changes, are you sure?" by returning a confirmation result (boolean or a Promise/Observable of one).

Intermediate How do you inject a service inside a functional guard or resolver?

Call inject(MyService) directly in the function body. It works because guards/resolvers run in an injection context — but the inject() call must be synchronous (before any await or inside the pipe), not deferred into an async callback.

Intermediate How do you read a resolver's data in the component?

From ActivatedRoute: route.snapshot.data['key'] for a one-time read, or route.data (an Observable) when the same component instance is reused across navigations. With withComponentInputBinding() the resolved value can also be bound straight to a matching @Input.

Pro A guard returns false vs a UrlTree — what's the difference?

Returning false cancels the navigation and the user stays where they are (URL unchanged). Returning a UrlTree (via router.createUrlTree([...])) cancels the current navigation and triggers a new one to that tree — the correct way to redirect, e.g. bouncing an unauthenticated user to /login.

Pro Why prefer CanMatch over CanActivate to auth-gate a lazy-loaded feature?

CanActivate runs after the route matches, so the feature's chunk is already downloaded before access is denied. CanMatch runs during matching, so an unauthorized user never downloads the code at all, and the router can fall through to a fallback route (e.g. a 404 or login). It saves bandwidth and hides the feature's existence.

Pro What's the downside of resolvers, and when would you avoid one?

The router blocks the navigation until the resolver completes, so a slow call makes the app feel frozen with no feedback between the click and the new view. Avoid resolvers for slow, optional, or independently-loading data — render the route shell immediately and stream the data in (async pipe, signals, or a deferred load) instead. Reserve resolvers for fast, mandatory data the page can't render without, and handle failure with a RedirectCommand or by completing empty.
