Module: Security

Angular is secure by default — it treats every value as untrusted and escapes it before it touches the DOM. The risk starts the moment you reach for bypassSecurityTrust or wire XSRF up wrong; this module maps exactly where the framework's guarantees hold and where you take ownership of the danger.

Knowledge Library → Courses → Angular Advanced → Security

What this module covers

Front-end security in Angular is mostly about one attack — cross-site scripting (XSS) — and the defences the framework gives you against it. You will see how Angular's contextual auto-sanitization escapes untrusted values automatically, why the DomSanitizer bypass methods are the one place you can undo that protection, and how the browser-level layers — Trusted Types and Content Security Policy (CSP) — back it up. The module closes on HttpClient's XSRF (CSRF) protection, the cookie-to-header token scheme that needs your backend's cooperation to work.

Topics

XSS and Angular's contextual auto-sanitization

Cross-site scripting is when attacker-controlled data is interpreted as code by the browser — stored (persisted in a database), reflected (echoed from the URL), or DOM-based (a client-side sink like innerHTML). Angular's core defence is that it treats every value as untrusted: interpolation with {{ }} always escapes, so a bio containing <script> renders as visible text, never as an element. When you bind to a sink like [innerHTML], Angular does not escape but sanitizes — it parses the HTML and strips dangerous parts (script tags, on* handlers, javascript: URLs) while keeping safe markup.

Why it matters: the sanitizer chooses its behaviour by context — HTML, style, URL, or resource-URL — because the same string is dangerous in different ways in each. Understanding that "Angular is safe by default, and I only lose that safety when I opt out" is the mental model interviewers are testing, not the API surface.

// interpolation escapes — this renders the angle brackets as text
@Component({ template: `<p>{{ userBio }}</p>` })
// [innerHTML] sanitizes — <script> and onclick are stripped, <b> survives
@Component({ template: `<div [innerHTML]="commentHtml"></div>` })

DomSanitizer and the bypassSecurityTrust escape hatch

Sometimes you genuinely need to render markup Angular would strip — a rich-text field you control, or a trusted embed URL. DomSanitizer is the service that both sanitizes values and, through its bypassSecurityTrust* family (Html, Style, Script, Url, ResourceUrl), lets you mark a value as trusted so Angular renders it verbatim. That is the single most dangerous call in an Angular app: passing user-controlled data to any bypass method reopens the exact XSS hole the framework closed.

Why it matters: the rule interviewers want to hear is "bypass only values you fully control, applied as close to the source as possible, and never raw user input." The common safe pattern is a tiny SafeHtml/SafeUrl pipe wrapping the sanitizer — for example building a YouTube <iframe> src from a known video id, not from a value a user typed.

constructor(private sanitizer: DomSanitizer) {}

// safe: the id is validated/known, not free user input
embedUrl(videoId: string): SafeResourceUrl {
  return this.sanitizer.bypassSecurityTrustResourceUrl(
    'https://www.youtube.com/embed/' + videoId);
}

Trusted Types and Content Security Policy (CSP)

Framework sanitization protects the code paths Angular controls, but raw DOM APIs (element.innerHTML =, eval, document.write) sit outside it. Trusted Types is a browser feature that locks those dangerous sinks so they only accept specially-created typed objects, not plain strings — turning a whole class of DOM-based XSS into a hard error you can catch in development. CSP is the complementary HTTP header that tells the browser which scripts, styles and other resources are allowed to load and run; a strict, nonce-based policy stops injected inline scripts from executing even if one slips through.

Why it matters: these are defence-in-depth layers, not replacements for sanitization. Angular supports both — it can enforce Trusted Types policies and emits a nonce onto its scripts (via the ngCspNonce attribute or the CSP_NONCE token) so a strict CSP does not break the app. Interviewers like candidates who know CSP alone does not stop context-specific and DOM-based XSS — you still need the framework's escaping.

Content-Security-Policy:
  require-trusted-types-for 'script';
  script-src 'self' 'nonce-{RANDOM}';
  object-src 'none'; base-uri 'none';

HttpClient XSRF (CSRF) protection and best practices

Cross-site request forgery tricks an authenticated user's browser into sending a state-changing request the user did not intend, riding on cookies the browser attaches automatically. Angular's HttpClient implements the client half of the standard cookie-to-header defence: an interceptor reads a token from a cookie (default XSRF-TOKEN) and copies it into a header (default X-XSRF-TOKEN) on mutating requests to relative URLs. Because a cross-origin site can send the cookie but cannot read it to set the header, the forged request fails.

Why it matters: this only works if your backend sets that cookie and verifies the header — Angular does the client side, not the whole scheme, and it skips GET/HEAD and absolute-URL requests by design. Round out "security best practices" with the layered discipline: keep AOT compilation (never build templates from user strings), avoid ElementRef.nativeElement/Renderer2 DOM writes with untrusted data, keep dependencies patched (npm audit), and prefer the framework's safe defaults everywhere.

// app.config.ts — customise the cookie/header names if your backend differs
provideHttpClient(
  withXsrfConfiguration({
    cookieName: 'XSRF-TOKEN',
    headerName: 'X-XSRF-TOKEN',
  })
)

Interview questions

Most-asked Angular security questions in Indian interviews — XSS, the DomSanitizer bypass risk, and XSRF are the near-certain trio for senior front-end rounds:

Beginner What is XSS and what are its main types?

Cross-site scripting is injecting attacker-controlled data that the browser then executes as code — stealing sessions, cookies, or defacing content. Three types: stored (saved on the server and served to victims), reflected (echoed back from the request, e.g. a URL parameter), and DOM-based (a client-side sink like innerHTML runs the payload without the server ever seeing it).

Beginner How does Angular protect against XSS out of the box?

Angular treats all values as untrusted by default. In templates, interpolation escapes values so markup renders as text, and binding to sinks like [innerHTML] is sanitized — dangerous parts are stripped. You get this automatically as long as you use template bindings and do not opt out with the DomSanitizer bypass methods.

Beginner Is {{ }} interpolation safe from XSS?

Yes. Interpolation always escapes its output, so a value like <img src=x onerror=alert(1)> is displayed as literal text, not parsed into an element. Escaping is unconditional here — there is no context in which interpolation injects live HTML.

Beginner What happens when you bind untrusted HTML to [innerHTML]?

Angular sanitizes it: it parses the HTML and removes unsafe content — <script>, event handler attributes (onclick, onerror), and javascript: URLs — while leaving benign tags like <b> or <p>. The value is rendered as HTML, but the dangerous pieces are gone. This differs from interpolation, which escapes everything.

Intermediate What are Angular's sanitization contexts, and why do they exist?

HTML, style, URL, and resource URL. The same string is dangerous in different ways depending on where it lands — a javascript: URL is harmless as text but executable in an href, and a resource URL (script/iframe src) can load and run remote code. Angular picks the sanitizer that matches the binding context so it strips the right things.

Intermediate What is DomSanitizer and when would you legitimately use it?

It is the service behind Angular's sanitization, exposing both sanitize() and the bypassSecurityTrust* methods. You reach for it when you must render markup or a URL Angular would otherwise strip and you can prove the value is safe — a rich-text field you own, or an embed URL built from a known id. The idiom is a small SafeHtml/SafeResourceUrl pipe so the bypass lives in one reviewed place.

Intermediate Why is bypassSecurityTrustHtml considered dangerous, and how do you use it safely?

It tells Angular "trust this value, do not sanitize it" — so passing user-controlled data straight to it reintroduces the XSS vulnerability Angular removed. Safe use: apply it only to values you fully control, as close to the source as possible, never to raw user input, and ideally after your own validation or a library like DOMPurify. Treat every call as a code-review flag.

Intermediate How does Angular's XSRF/CSRF protection work?

HttpClient ships an interceptor that reads a token from a cookie (default XSRF-TOKEN) and sets it as a header (default X-XSRF-TOKEN) on mutating requests (POST/PUT/DELETE) to relative URLs. A cross-origin attacker's page can cause the cookie to be sent but cannot read it to reproduce the header, so the forged request is rejected by the server. Names are configurable via withXsrfConfiguration.

Intermediate Why is Angular's XSRF support "only half" of the protection?

Angular implements the client side — copying the cookie value into a header. The scheme fails unless the backend issues that token cookie and verifies the matching header on every eligible request. It also deliberately does not send the header on GET/HEAD or on absolute-URL (cross-origin) requests, so those must be handled by other means. Without server cooperation, the client behaviour alone protects nothing.

Pro What are Trusted Types, and how do they relate to CSP and Angular?

Trusted Types is a browser feature that restricts dangerous DOM sinks (innerHTML, eval, script src) so they accept only typed, policy-created objects instead of raw strings — closing DOM-based XSS at the sink. It is enabled through the CSP header (require-trusted-types-for 'script'). Angular integrates with it, enforcing its own policies, so an app built on safe bindings keeps working while ad-hoc string-to-DOM writes become errors. CSP and Trusted Types are defence-in-depth on top of, not instead of, Angular's sanitization.

Pro How do you run Angular under a strict CSP, and does CSP replace sanitization?

Use a nonce-based policy (script-src 'self' 'nonce-{RANDOM}') and let Angular stamp the nonce onto its scripts via the ngCspNonce attribute on the root element or the CSP_NONCE injection token; add object-src 'none' and base-uri 'none' to harden it. CSP does not replace sanitization: it blocks disallowed script execution but does nothing about context-specific encoding or a sanitizer bypass you wrote yourself — you need both layers.

Pro Beyond bypassSecurityTrust, what escape hatches silently reopen XSS in an Angular app?

Any path that writes untrusted data to the DOM outside Angular's bindings: direct ElementRef.nativeElement.innerHTML assignments, Renderer2 writes of raw HTML, building templates or component source from user strings (which also breaks AOT's safety), server-side rendering that templates unescaped values into the HTML shell, and unpatched dependencies with known XSS CVEs. The fix is discipline: stay on template bindings, keep AOT, sanitize deliberately, and run npm audit. Trusted Types helps catch the raw-DOM cases at runtime.
