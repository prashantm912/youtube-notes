Module: Dynamic Components

Sometimes the template can't know what to render until runtime — a modal, a dashboard widget, a component chosen by a server response. This module maps the three modern ways Angular builds components on the fly, and the setInput/bindings discipline that keeps them leak-free.

Knowledge Library → Courses → Angular Advanced → Dynamic Components

What this module covers

Dynamic components are created and inserted programmatically instead of being written into a template — the machinery behind modals, dynamic forms, dashboards, plugin systems and ad slots. You will meet the imperative ViewContainerRef.createComponent, the detached standalone createComponent function, the declarative ngComponentOutlet directive, and the two ways to feed them data: ComponentRef.setInput and the newer inputBinding/outputBinding array. The old ComponentFactoryResolver ceremony is gone — Ivy lets you pass the component class directly.

Topics

ViewContainerRef.createComponent: rendering inside a view

A ViewContainerRef is an anchor in the view where views can be attached. Get one by injecting it, or — more commonly — by reading an <ng-container> anchor with @ViewChild('host', { read: ViewContainerRef }). Its createComponent(ComponentClass) instantiates the component and inserts its host view into the container; Angular places the DOM node as the next sibling of the element that provided the ViewContainerRef — a gotcha worth remembering, which is why a dedicated ng-container anchor is the clean pattern.

Why it matters: this is the workhorse for runtime UI, and because the component class is imported only where it is created, it can be code-split to shrink the initial bundle. Since Angular 13 deprecated (and later removed) ComponentFactoryResolver, you pass the class straight in — no factory resolution step.

@ViewChild('host', { read: ViewContainerRef }) vcr!: ViewContainerRef;

show() {
  const ref = this.vcr.createComponent(AdBannerComponent);
  ref.setInput('campaign', this.campaign); // triggers ngOnChanges
}

ngComponentOutlet: the declarative approach

When the component to render is a value you can compute, ngComponentOutlet keeps everything in the template — no @ViewChild, no imperative wiring. It is a structural directive: *ngComponentOutlet="currentWidget" renders the class held in currentWidget and swaps it whenever the value changes (a falsy value clears and destroys the view). Since Angular 16+ you pass data through the inputs object, supply a custom injector, project content, and read the live instance via its exported componentInstance.

Why it matters: it is the least-code path for "pick a component from a map and show it" — tab bodies, dynamic dashboards, CMS-driven layouts — and Angular handles creation and destruction for you as the bound value changes.

<ng-container
  *ngComponentOutlet="currentWidget; inputs: widgetInputs" />

The standalone createComponent function: detached rendering

The createComponent function from @angular/core creates a ComponentRef without attaching it to any existing view or DOM location — you get explicit control over where it lives. You provide an EnvironmentInjector, optionally a host element, and then take responsibility for the two things ViewContainerRef did automatically: register the host view for change detection (ApplicationRef.attachView) and place the element in the DOM. This is the foundation for overlays, popups and portals — including Angular CDK's ComponentPortal, which packages this pattern.

Why it matters: modals and tooltips must escape their parent's overflow and stacking context, so they cannot simply nest in a view. The detached function (and CDK Portal on top of it) is how production overlay systems render a component anywhere in the DOM while keeping it in Angular's change-detection graph.

Passing inputs and outputs, and cleanup

Two ways to feed a dynamically created component. The imperative way: componentRef.setInput('name', value) — it sets the input by its public (aliased) name, marks the view for check and fires ngOnChanges, unlike a raw componentRef.instance.prop = value assignment which skips all of that. For outputs you subscribe to the emitter on the instance: componentRef.instance.saved.subscribe(...). The modern way (Angular 19/20+): pass a bindings array of inputBinding, outputBinding and twoWayBinding at creation — a signal input is tracked reactively and updates flow automatically.

Cleanup is the part interviews probe: a dynamic component is not garbage-collected on its own. Call componentRef.destroy(), or viewContainerRef.clear()/remove(index) to destroy views in a container, and unsubscribe from any output subscriptions you opened. A component created through a ViewContainerRef or ngComponentOutlet is destroyed automatically when its host is; the bindings API also manages its own subscription cleanup — the manual instance/subscribe style is what leaks if you forget.

const ref = createComponent(RatingComponent, {
  environmentInjector: this.envInjector,
  bindings: [
    inputBinding('value', this.score),                 // signal, tracked
    outputBinding<number>('valueChange', v => this.onChange(v)),
  ],
});

Interview questions

Dynamic components come up in Indian product-company and senior Angular rounds — modals, dynamic forms and dashboards are the usual framing:

Beginner What is a dynamic component and when would you use one?

A component created and inserted at runtime rather than declared in a template. You reach for it when the template can't know ahead of time what to show: modal/dialog systems, dynamic or schema-driven forms, dashboards with user-configurable widgets, CMS-driven pages, ad slots, and plugin architectures.

Beginner What are the ways to render a component programmatically in Angular?

Three: ViewContainerRef.createComponent (imperative, in TypeScript, inserted at an anchor in the view), the standalone createComponent function (detached — you control DOM placement), and ngComponentOutlet (declarative, in the template).

Beginner What is ViewContainerRef and how do you get a reference to one?

It represents a container in the view where host or embedded views can be attached. You obtain it by injecting ViewContainerRef, or by reading an anchor element — usually an <ng-container> — with @ViewChild('anchor', { read: ViewContainerRef }).

Beginner How do you render a component and pass inputs with ngComponentOutlet?

*ngComponentOutlet="componentClass" renders the class; pass data with the inputs option: *ngComponentOutlet="cmp; inputs: myInputs" where myInputs is an object like { title: heading }. Setting the bound class to a falsy value clears and destroys the rendered component.

Intermediate Where does ViewContainerRef.createComponent place the component in the DOM?

As the next sibling of the element/directive that provided the ViewContainerRef — not inside it. That surprises people who inject the host component's own ViewContainerRef; the clean fix is a dedicated <ng-container #anchor> read as the container so the component appears exactly where the anchor sits.

Intermediate setInput() vs assigning componentRef.instance.prop directly — which and why?

Prefer setInput('name', value): it resolves the input by its public/aliased name, marks the component for change detection (so it works under OnPush) and triggers ngOnChanges. A raw instance.prop = value assignment sets the field but fires none of that, so lifecycle hooks and OnPush views can miss the update.

Intermediate How do you listen to a dynamic component's outputs?

The manual way: subscribe to the emitter on the instance — ref.instance.saved.subscribe(handler) — and unsubscribe when you destroy the component. The modern way: pass outputBinding('saved', handler) in the bindings array at creation, and Angular manages the subscription lifecycle for you.

Intermediate How do you clean up dynamic components to avoid memory leaks?

Call componentRef.destroy(), or use viewContainerRef.clear() / remove(index) to destroy views in a container, and unsubscribe from any output subscriptions you opened. Components created via ViewContainerRef or ngComponentOutlet are destroyed automatically when their host view is destroyed; the risk is the detached createComponent function and manual output subscriptions, which you own.

Intermediate What replaced ComponentFactoryResolver, and why did it go away?

Nothing replaces it — you no longer need it. With the Ivy compiler there is no separate factory to resolve, so you pass the component class straight to createComponent. ComponentFactoryResolver and resolveComponentFactory were deprecated in Angular 13 and later removed; legacy tutorials that still use them are out of date.

Pro ViewContainerRef.createComponent vs the standalone createComponent function?

ViewContainerRef.createComponent attaches the host view into the view hierarchy at the anchor and hooks change detection automatically. The standalone createComponent(Type, { environmentInjector }) is detached: it returns a ComponentRef but attaches to nothing — you call ApplicationRef.attachView(ref.hostView) for change detection and append ref.location.nativeElement to the DOM yourself. Use the detached one for overlays/portals that must escape the parent's DOM.

Pro Explain the modern bindings API (inputBinding / outputBinding / twoWayBinding).

Added around Angular 19–20, the bindings array lets you declare inputs, outputs and two-way bindings at creation for both createComponent and ViewContainerRef.createComponent: inputBinding('value', signalOrGetter), outputBinding('changed', fn), twoWayBinding('model', signal). Passing a signal makes the input reactive — changes propagate automatically — and Angular handles subscription cleanup, replacing the imperative setInput/subscribe dance.

Pro How do change detection and OnPush behave for dynamically created components?

A component attached through a ViewContainerRef joins the parent's change-detection tree and is checked normally; setInput marks it dirty so OnPush children update. A detached createComponent view is checked only after ApplicationRef.attachView — until then you must call detectChanges() on its ChangeDetectorRef manually. Signals passed via the bindings API trigger checks reactively regardless.

Pro How do you project content into a dynamic component and give it a custom injector?

Content projection: pass a projectableNodes two-dimensional array (one inner array per <ng-content> slot) to createComponent, or use ngComponentOutletContent with the directive. Custom DI: supply an elementInjector (per-component providers) and/or an EnvironmentInjector for environment-scoped providers — the directive exposes these as ngComponentOutletInjector and ngComponentOutletEnvironmentInjector.
