Module: RxJS — Subjects, Combination and Error Handling

An Observable that only pushes is half the story. Subjects let you push values in, combination operators braid many streams into one, and getting error handling and cleanup right is what keeps a reactive Angular app from silently leaking. This is the RxJS interviewers actually probe.

What this module covers

Topics

Subjects: Subject, BehaviorSubject, ReplaySubject, AsyncSubject

A Subject is both an Observable and an Observer: you can subscribe to it and you can call next()/error()/complete() on it to push values imperatively. That makes it the standard way to build an event bus or a shared state stream in a service. The three variants differ only in what a late subscriber receives. A plain Subject replays nothing — subscribe after a value was emitted and you miss it. BehaviorSubject requires an initial value and always hands new subscribers the current (last) value, which is why it is the idiomatic holder of "current state". ReplaySubject(n) replays the last n values regardless of timing, and AsyncSubject emits only the final value, and only once the stream completes.

Why it matters: the difference between Subject and BehaviorSubject is the single most-asked RxJS interview question in Angular rounds, because it maps directly to "how do I share state so a component that subscribes late still gets a value?"

const subject  = new Subject<number>();           // no current value, no replay
const behavior = new BehaviorSubject<number>(0);  // seeded with 0
const replay   = new ReplaySubject<number>(2);    // replays last 2 to newcomers

subject.subscribe(v => console.log('A', v)); // only values emitted AFTER this
subject.next(1);                             // A sees 1
behavior.subscribe(v => console.log('B', v));// B fires immediately with current value

Hot vs cold observables and multicasting

A cold observable creates its producer fresh for every subscriber — each subscription to an http.get() fires its own HTTP request. A hot observable shares one running producer across all subscribers; they see values from the moment they join. Subjects are inherently hot (one internal producer, many observers), which is the mechanism behind multicasting: taking a cold source and sharing a single execution. In modern RxJS you rarely wire a Subject by hand for this — you pipe share() or shareReplay(1), which internally multicast through a Subject and, in the shareReplay case, hand the last value to late subscribers. This is the standard fix for duplicate HTTP calls when several components consume the same service stream.

Why it matters: "why is my HTTP call firing three times?" is a real bug and a common interview scenario; the answer is that the stream is cold and each async pipe subscribes independently — shareReplay(1) makes it hot.

// cold: every subscriber triggers its OWN request
const users$ = this.http.get<User[]>('/api/users');

// hot + cached: one request, multicast to all, last value replayed to latecomers
const sharedUsers$ = users$.pipe(shareReplay(1));

Combination operators: combineLatest, forkJoin, merge, zip

These take several source observables and produce one. They differ in when they emit. forkJoin waits for every source to complete, then emits one array/object of each source's last value — the RxJS analogue of Promise.all, ideal for firing several independent HTTP calls and acting once all return. combineLatest emits every time any source emits (once all have emitted at least once), giving the latest value from each — perfect for recomputing a view model from several live inputs like a search box plus a page number. merge interleaves emissions from all sources as they arrive, preserving none-to-one ordering, good for funnelling multiple event streams into one. zip pairs emissions by index — the 1st of each, then the 2nd of each — and is comparatively rare in app code.

Why it matters: combineLatest vs forkJoin and forkJoin vs zip are classic "do you actually understand the timing" questions. Reaching for forkJoin when a source never completes (a Subject, a router stream) is a common bug — it will hang forever.

// forkJoin: all complete, emit last of each ONCE (parallel HTTP)
forkJoin({ user: user$, roles: roles$ })
  .subscribe(({ user, roles }) => render(user, roles));

// combineLatest: re-emit on ANY change, latest of each
combineLatest([filter$, page$]).pipe(
  switchMap(([filter, page]) => this.search(filter, page))
).subscribe(setResults);

Error handling: catchError and retry

An error terminates an observable — it fires the error callback and the stream is done; no further values, no complete. catchError intercepts that error and lets you recover by returning a replacement observable: return of(fallback) to swallow the error and continue with a default, or throwError(() => err) to rethrow after logging (catch-and-rethrow). Its placement in the pipe matters — a catchError inside a switchMap's inner pipeline protects just that inner stream, keeping the outer stream alive; placed at the outer level it kills the whole stream on the first error. retry({ count, delay }) re-subscribes to the source on error, which for a cold HTTP observable means re-issuing the request — useful for transient network failures. (retryWhen is deprecated; use retry's delay option instead.)

Why it matters: candidates routinely put catchError in the wrong place and kill a long-lived stream (a form's valueChanges, a typeahead) after one failed request. Explaining catch-and-replace vs catch-and-rethrow, and where the operator sits, signals real reactive experience.

this.http.get<User[]>('/api/users').pipe(
  retry({ count: 2, delay: 1000 }),   // re-issue the request on transient errors
  catchError(err => {                 // last-resort recovery
    this.logger.error(err);
    return of([]);                    // swallow + provide a safe fallback
  })
);

Subscription management: unsubscribe, takeUntil, takeUntilDestroyed

Every manual subscribe() to a long-lived stream is a potential memory leak: if the component is destroyed while the stream is still alive, the subscription keeps the component (and its closure) in memory and the callback keeps running. HTTP observables self-complete after one emission so they rarely leak, but Subjects, interval, router events, and form valueChanges do not. The strategies, worst to best: store each Subscription and call unsubscribe() in ngOnDestroy (verbose, easy to forget); pipe takeUntil(destroy$) where destroy$ is a Subject you next() in ngOnDestroy; or — the modern default — takeUntilDestroyed(), which reads the injection context's DestroyRef and completes the stream automatically when the enclosing component/service is destroyed. Best of all where possible: skip manual subscription entirely and let the async pipe subscribe and clean up for you.

Why it matters: "how do you avoid memory leaks with subscriptions?" is asked in almost every Angular interview. Naming takeUntilDestroyed and the async pipe — and knowing that outside an injection context you must pass a DestroyRef explicitly — marks you as current.

export class TickerComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    interval(1000).pipe(
      takeUntilDestroyed(this.destroyRef)  // auto-completes on component destroy
    ).subscribe(t => this.tick = t);
  }
}

Interview questions

Most-asked Subjects and RxJS questions in Indian Angular interviews — Subject-vs-BehaviorSubject, hot-vs-cold and unsubscribe strategies are screening-round staples across service companies and product firms alike:

Beginner What is a Subject and how does it differ from an Observable?

A Subject is both an Observable and an Observer — you can subscribe to it and push values into it with next(). A plain Observable only produces values; you cannot feed it from outside. A Subject is also multicast (hot): all its subscribers share one execution, whereas a cold Observable runs separately per subscriber.

Beginner Subject vs BehaviorSubject — the difference that matters?

A BehaviorSubject requires an initial value and always emits its current (last) value to a new subscriber immediately; a plain Subject replays nothing, so a subscriber that joins after a value was emitted misses it. Use BehaviorSubject to hold "current state" (a selected item, a logged-in user) that late subscribers must still receive. It also exposes .value for a synchronous read.

Beginner What are ReplaySubject and AsyncSubject for?

ReplaySubject(n) replays the last n emitted values to any new subscriber regardless of timing (n defaults to infinite) — useful for a small history or a cache without needing an initial value. AsyncSubject emits only the final value and only when the source completes — rare, but fits "one result computed at the end".

Beginner What is the difference between a hot and a cold observable?

Cold: the producer is created per subscription, so each subscriber gets its own independent execution (each http.get subscription fires its own request). Hot: one producer is shared, and subscribers receive values from the moment they join. Subjects are hot; most creation operators and HttpClient are cold until you multicast them.

Intermediate A service exposes an HTTP stream and three components use the async pipe on it — you see three network calls. Why, and how do you fix it?

The HTTP observable is cold, so each async pipe subscribes independently and triggers its own request. Pipe shareReplay(1) (or share()) on the service stream to multicast a single execution and replay the last value to late subscribers — one request, three consumers.

Intermediate combineLatest vs forkJoin — when do you use each?

forkJoin waits for every source to complete, then emits each source's last value once — like Promise.all, for firing parallel HTTP calls and acting when all finish. combineLatest emits on every change from any source (after each has emitted once), giving the latest of each — for live view models built from multiple ongoing streams. Using forkJoin on a source that never completes hangs forever.

Intermediate forkJoin vs zip vs merge?

forkJoin emits once, at completion, the last value of each. zip pairs values by index (1st of each, then 2nd of each) and emits per matched tuple. merge interleaves values from all sources as they arrive, no pairing, no waiting. forkJoin = "all done, final values"; zip = "line them up by position"; merge = "flatten several streams into one".

Intermediate How does catchError work, and why does its position in the pipe matter?

catchError intercepts an error and must return a replacement observable — of(fallback) to recover (catch-and-replace) or throwError(() => err) to rethrow after logging. Position matters because an error terminates the stream: catchError inside a switchMap's inner pipe protects only that inner request and keeps the outer stream alive; at the outer level it terminates the whole stream on the first error. For a typeahead you catch inside the inner call so one failed request doesn't kill the search.

Intermediate How do you avoid memory leaks from subscriptions in Angular?

Prefer the async pipe — it subscribes and unsubscribes with the view automatically. For manual subscriptions to long-lived streams, use takeUntilDestroyed(), or takeUntil(destroy$) with a Subject you complete in ngOnDestroy, or store the Subscription and call unsubscribe() in ngOnDestroy. One-shot HTTP observables self-complete, so the risk is really Subjects, interval, router and form value streams.

Intermediate What does retry do, and when is it appropriate?

retry({ count, delay }) re-subscribes to the source when it errors, up to count times, optionally waiting delay ms between attempts. For a cold HTTP observable that means re-issuing the request — appropriate for transient failures (network blips, 503s), not for deterministic errors like a 404 or 400 where retrying just repeats the failure. retryWhen is deprecated in favour of the delay option.

Pro How does takeUntilDestroyed work, and what is the catch when using it outside a constructor?

It completes the piped observable when the current DestroyRef's host (component/directive/service) is destroyed. Called during construction it reads the injection context automatically; called later — in ngOnInit, a callback, or any non-injection context — there is no ambient context, so you must inject a DestroyRef and pass it explicitly: takeUntilDestroyed(this.destroyRef). Omitting it there throws an "outside injection context" error.

Pro Why is exposing a BehaviorSubject directly from a service risky, and what is the idiom?

If you expose the Subject itself, any consumer can call next() and mutate shared state from anywhere, breaking the single-source-of-truth. The idiom is to keep the Subject private and expose it as a read-only stream via .asObservable() (or a public readonly x$ = this._x.asObservable()), so components can subscribe but only the service can push. In signal-era code the same role is often filled by a writable signal exposed as read-only.

Pro shareReplay can leak or replay stale data — how, and how do you control it?

By default shareReplay(1) keeps its internal replay buffer alive even after all subscribers leave, so the source stays subscribed (a leak for infinite sources) and a later subscriber may get a stale cached value. Pass the config form shareReplay({ bufferSize: 1, refCount: true }) so the subscription is torn down when the refcount hits zero and re-established fresh on the next subscribe. Choose refCount based on whether you want a durable cache or a per-audience live share.
