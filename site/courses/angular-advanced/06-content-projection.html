Module: Content Projection and Templates

A reusable Angular component is only as flexible as the holes it leaves for its caller to fill. This module covers the two mechanisms behind every configurable card, table, tab and dialog you will ever build — projecting markup in with ng-content, and handing whole templates around with ng-template.

Knowledge Library → Courses → Angular Advanced → Content Projection and Templates

What this module covers

Angular gives a component two ways to accept content from the outside. Content projection (ng-content) lets a caller drop real markup into named or unnamed slots inside your template — Angular's take on the Web Components <slot>. Templates (ng-template, TemplateRef, ngTemplateOutlet) go further: they capture a chunk of view as a first-class object you can pass as an input, give a data context, and stamp out where and when you choose. Underneath, the same ng-template machinery is what the * in *ngIf and *ngFor quietly desugars into — so understanding it demystifies structural directives too. These are among the most-asked "senior Angular" interview topics because they separate people who consume components from people who design them.

Topics

Content projection with ng-content, and fallback content

Placing <ng-content> in a component's template marks a slot; whatever the caller nests between the component's opening and closing tags is projected into that spot at render time. The slot is a compile-time placeholder, not a DOM element — nothing extra is added to the output. Any content written inside the ng-content tags acts as fallback, shown only when the caller projects nothing into that slot.

Why it matters: this is the foundation of every wrapper component — cards, panels, buttons, layout shells. Instead of forwarding a dozen inputs and re-emitting events, you leave a hole and let the parent supply arbitrary markup, keeping the component generic and the DOM clean.

<!-- card.component.html -->
<div class="card">
  <ng-content>Nothing supplied yet</ng-content>  <!-- fallback -->
</div>

<!-- caller -->
<app-card><h2>Invoice</h2><p>Projected into the slot.</p></app-card>

Multi-slot projection: the select attribute and ngProjectAs

One component can expose several named slots by giving each ng-content a select attribute — a CSS selector matched against the projected top-level nodes. Selectors can target an element name, an attribute (select="[title]"), or a class; a single ng-content with no select catches everything unmatched. When the content you want to route doesn't naturally carry the selector — or a wrapper would break projection — ngProjectAs lets an element masquerade as a different selector so it lands in the right slot.

Why it matters: named slots are how design-system components ("put your title here, your actions there, your body in the middle") stay both structured and open-ended. Explaining select matching and the single default-slot rule is a very common component-design interview probe.

<!-- panel.component.html -->
<header><ng-content select="[panelTitle]"></ng-content></header>
<section><ng-content></ng-content></section>  <!-- default catch-all -->

<!-- caller -->
<app-panel>
  <h2 panelTitle>Billing</h2>
  <p>Everything without a matching selector falls here.</p>
</app-panel>

ng-container: grouping without an extra DOM node

<ng-container> is a logical wrapper that renders no element of its own. Its classic use is hosting a structural directive so you can, for example, iterate or conditionally show a group of siblings without introducing a meaningless <div> — vital where the parent constrains valid children (a <tr>'s cells, a <ul>'s list items). It is also the standard anchor for ngTemplateOutlet and ngComponentOutlet, and a place to attach a directive that provides a value via DI to a whole region — all while keeping the rendered HTML flat.

Why it matters: candidates constantly reach for a wrapper div that pollutes the DOM and breaks CSS layouts (flex/grid children, table semantics). Knowing ng-container is the "invisible wrapper" answer is table stakes for a mid-level Angular role.

<!-- renders NO element; the <li>s stay direct children of the <ul> -->
<ul>
  <ng-container *ngIf="user">
    <li *ngFor="let role of user.roles">{{ role }}</li>
  </ng-container>
</ul>

ng-template and TemplateRef

<ng-template> defines a fragment of view that is not rendered on its own — it is captured, not displayed, until something instantiates it. Grabbing a reference (a template variable #tpl, a @ViewChild/viewChild query, or injecting it in a directive) yields a TemplateRef: a first-class handle to that template. You render it either declaratively via ngTemplateOutlet or imperatively through a ViewContainerRef's createEmbeddedView, which stamps out an embedded view wherever the container sits.

Why it matters: TemplateRef + ViewContainerRef is the pair every structural directive, dynamic-list, and lazy-render pattern is built on. Being able to say "a template is deferred markup, a TemplateRef is the object, a ViewContainerRef is where it gets stamped" signals real understanding of Angular's rendering model.

<!-- template -->
<ng-template #empty><p>Nothing to show yet</p></ng-template>

// class
private vcr = inject(ViewContainerRef);
emptyTpl = viewChild.required<TemplateRef<unknown>>('empty');
show() { this.vcr.createEmbeddedView(this.emptyTpl()); }

ngTemplateOutlet and passing a data context

ngTemplateOutlet renders a TemplateRef at the outlet's position, and ngTemplateOutletContext feeds it data. Inside the template, let-name binds a named context property and the special $implicit key becomes the default bound by a bare let-x. This is what makes a component caller-customisable: a list or table component accepts a template as an input and, for each row, renders it with that row's data as context — the caller controls the markup, the component controls the iteration.

Why it matters: "how would you let the parent decide how each row of your reusable table looks?" is a classic senior-round question, and ngTemplateOutlet with context is the idiomatic answer (the pattern many component libraries expose as a *appColumnDef-style API).

<ng-template #greeting let-name let-city="city">
  Hi {{ name }} from {{ city }}
</ng-template>

<ng-container
  [ngTemplateOutlet]="greeting"
  [ngTemplateOutletContext]="{ $implicit: 'Asha', city: 'Pune' }">
</ng-container>

Structural directive microsyntax: what the asterisk desugars into

The * in *ngIf and *ngFor is syntactic sugar. Angular rewrites the starred element into an <ng-template> that wraps it and hosts the directive, then translates the microsyntax string into input and context bindings. Microsyntax rules: expressions are split on ;; a let clause declares a context variable; a keyword such as of or trackBy maps to a camel-cased input (ngForOf, ngForTrackBy); and as aliases a value. Because there is exactly one generated template per element, you can apply only one structural directive per element — which is precisely why ng-container exists.

Why it matters: this is the single concept that ties the whole module together, and "what does *ngIf actually compile to?" is a favourite deep-dive question — a correct desugaring answer marks you as someone who understands Angular rather than memorises its syntax.

<!-- shorthand -->
<div *ngFor="let item of items; let i = index">{{ i }} {{ item }}</div>

<!-- desugars to -->
<ng-template ngFor let-item [ngForOf]="items" let-i="index">
  <div>{{ i }} {{ item }}</div>
</ng-template>

Interview questions

Projection and templates are a staple of the "senior Angular" round in Indian interviews — service-company component-library work and product-company deep dives both lean on them heavily:

Beginner What is content projection, and what is ng-content?

Content projection lets a component accept markup from its caller and render it inside its own template. <ng-content> is the placeholder that marks where that projected content goes — Angular's equivalent of the native <slot>. It is a compile-time marker, not a real DOM element.

Beginner How do you project content into a specific slot?

Give the ng-content a select attribute with a CSS selector — element name, [attribute], or .class — and any projected top-level node matching it lands there. One ng-content with no select catches everything unmatched.

Beginner How do you show default content when nothing is projected?

Put the default markup between the <ng-content> tags. It renders only when the caller projects nothing into that slot — the fallback pattern, added in modern Angular.

Beginner What is ng-container and why use it?

A logical grouping element that renders no DOM node of its own. Use it to host a structural directive (or ngTemplateOutlet) without adding a wrapper div that would pollute the DOM or break table/flex/grid layouts.

Beginner What is ng-template? Does it render by itself?

It defines a fragment of view that is captured but not rendered — invisible until something instantiates it via ngTemplateOutlet or ViewContainerRef.createEmbeddedView. It is the deferred-markup building block behind structural directives.

Intermediate Difference between ng-content and ng-template?

ng-content projects content the caller nested inside your tags — that content belongs to the parent and always renders. ng-template captures markup as a reusable, deferred fragment that you decide when and how many times to stamp out, optionally with a data context. Projection is "receive what's passed in"; template is "hold markup to render later".

Intermediate What does the asterisk in *ngIf actually compile to?

Angular desugars *ngIf="cond" into an <ng-template> wrapping the host element, with the directive applied and the microsyntax turned into bindings (e.g. [ngIf]="cond"). The * is pure shorthand for that template-plus-directive form.

Intermediate Why can't you put two structural directives on one element, and how do you fix it?

Each * desugars to one generated ng-template for that element, and there is only one to unwrap onto — so two would be ambiguous. Fix it by moving one directive onto a wrapping <ng-container>, or with modern control flow (@if/@for) which nests naturally.

Intermediate What is ngTemplateOutlet and how do you pass data to the template?

[ngTemplateOutlet]="tpl" renders a TemplateRef at that position; [ngTemplateOutletContext]="{...}" supplies data. In the template, let-x="key" binds a named context property and let-x binds the special $implicit value — the conventional "main" datum for that template.

Intermediate Why can't @ViewChild see content projected through ng-content?

Projected content belongs to the parent's view, not the component's own view. View queries (@ViewChild/viewChild) only see your template; to reach projected nodes you use content queries (@ContentChild/contentChild), which run after content init.

Intermediate What problem does ngProjectAs solve?

When the element you want to route doesn't carry the slot's selector — often because it's wrapped in an ng-container or a different tag — ngProjectAs="[selector]" makes it match a named ng-content select anyway, so it projects into the intended slot.

Pro Which view does projected content belong to, and why does that matter?

Projected content is created and change-detected as part of the parent (the caller), not the component it visually sits inside. Consequences: its bindings run on the parent's change-detection pass, and its injector context is the parent's — so a token provided only by the inner component is not visible to projected content. This trips people who expect DI or lifecycle to follow the visual nesting.

Pro How do you build a reusable list/table where the caller controls how each row renders?

Accept a TemplateRef as an input (rowTemplate = input.required<TemplateRef<RowCtx>>(), or capture the caller's ng-template via a content query). Iterate internally and for each item render [ngTemplateOutlet]="rowTemplate()" with [ngTemplateOutletContext]="{ $implicit: item, index: i }". The component owns iteration and data; the caller owns the markup — the pattern Angular Material's *matCellDef and similar APIs use.

Pro ngTemplateOutlet vs ngComponentOutlet vs createEmbeddedView — when each?

ngTemplateOutlet stamps a TemplateRef (deferred markup you already declared) declaratively. ngComponentOutlet renders a whole component chosen at runtime by class. ViewContainerRef.createEmbeddedView (template) / createComponent (component) are the imperative equivalents — reach for them inside directives or when you need programmatic control over insertion, ordering, and cleanup.
