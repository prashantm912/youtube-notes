What this module covers

This module closes the track with the runtime and architecture patterns that harden an Angular app for the field. Topics

Progressive Web Apps and the Angular service worker

A Progressive Web App is an ordinary web app that, thanks to a service worker plus a web app manifest, becomes installable to the home screen, launches full-screen, and keeps working offline. The service worker is the engine: a script the browser runs in the background as a network proxy, intercepting outgoing requests and deciding whether to answer from cache or the network. Angular ships one as @angular/service-worker — a declarative caching utility you configure rather than hand-code. You add it in one step: ng add @angular/pwa wires in the package, generates ngsw-config.json and a manifest.webmanifest, adds icons, and registers the worker (production builds only).

Why it matters: on flaky Indian networks an instant, offline-tolerant load is a real product advantage, and "how do you turn an Angular app into a PWA?" is a stock advanced-round question. Knowing that Angular's worker is a caching tool — not a general service-worker framework like Workbox — is the nuance interviewers listen for.

ng add @angular/pwa
# adds @angular/service-worker, ngsw-config.json,
# manifest.webmanifest and icons, and registers the
# worker via provideServiceWorker() in production builds

How the service worker caches: ngsw-config.json

The Angular service worker is driven entirely by ngsw-config.json, split into two kinds of rule. Asset groups cover the app's own build output (JS, CSS, index.html, images): installMode: "prefetch" caches them immediately for full offline support, "lazy" caches on first use. Data groups cover runtime data such as API calls, matched by URL and governed by a cacheConfig. The strategy choice is the crux: performance serves from cache first (fast, possibly stale), freshness hits the network first and falls back to cache (current, slower), with maxSize, maxAge, and a network timeout tuning the trade-off.

Why it matters: getting these two strategies backwards is the classic bug — caching an API "for performance" and then serving yesterday's prices. Being able to name performance vs freshness and pick correctly per resource is exactly what a deep-dive interview probes.

{
  "assetGroups": [{
    "name": "app",
    "installMode": "prefetch",
    "resources": { "files": ["/index.html", "/*.css", "/*.js"] }
  }],
  "dataGroups": [{
    "name": "api",
    "urls": ["/api/**"],
    "cacheConfig": {
      "strategy": "freshness",
      "maxSize": 100, "maxAge": "1h", "timeout": "5s"
    }
  }]
}

App shell and handling updates with SwUpdate

Two patterns round out a real PWA. The app shell pre-renders a minimal static skeleton at build time (ng generate app-shell) so the user sees UI immediately while the JS bundle loads — the shell ships inside index.html and is itself cached by the service worker. Updates are the other half: because the worker serves the cached app, a new deployment is not picked up until the worker installs it in the background. The SwUpdate service exposes this through the versionUpdates stream — listen for VERSION_READY, then prompt the user and reload so they move to the fresh version instead of staring at a stale one.

Why it matters: the number-one production complaint about Angular PWAs is "my users don't see my deploy." Explaining why (the worker serves cache) and the fix (SwUpdate + a reload prompt) signals you have actually shipped one.

private sw = inject(SwUpdate);

constructor() {
  this.sw.versionUpdates.pipe(
    filter((e): e is VersionReadyEvent => e.type === 'VERSION_READY')
  ).subscribe(() => {
    if (confirm('New version available. Reload?'))
      document.location.reload();
  });
}

Web workers: offloading the main thread

A web worker runs JavaScript on a separate background thread, so CPU-intensive work — parsing a large file, image processing, heavy math — no longer freezes the UI. It shares no memory with the main thread; you communicate by passing messages with postMessage() and an onmessage / message handler, and the data is copied (structured clone), not shared. Crucially, a worker has no DOM access: no window, no document. Angular scaffolds one with ng generate web-worker <name>, which creates the worker file and configures the build. One caveat: workers don't exist during server-side rendering, so guard for that environment.

Why it matters: when an interviewer asks "the page jank-freezes during a big calculation — what do you do?", the expected answer is a web worker. Bonus points for noting it is a different thing from a service worker, a distinction that trips many candidates up.

// component
const worker = new Worker(new URL('./app.worker', import.meta.url));
worker.onmessage = ({ data }) => console.log(`result: ${data}`);
worker.postMessage(input);

// app.worker.ts  (no DOM here)
addEventListener('message', ({ data }) => {
  postMessage(heavyCompute(data));   // runs off the UI thread
});

Micro-frontends and Module Federation (survey)

Micro-frontends extend the microservice idea to the browser: one product is split into several separately built and deployed frontends, each owned by a different team, composed at runtime. Module Federation (originally a webpack feature) makes this practical — a host app lazy-loads code exposed by remote apps at runtime and shares common libraries (Angular, RxJS) so they aren't downloaded twice. Because modern Angular builds on esbuild/Vite rather than webpack, the community's Native Federation (by Manfred Steyer / ANGULARarchitects) re-implements the same mental model on browser-native ES modules and import maps, so the pattern keeps working with the current build system. Treat this as awareness-level knowledge: know what problem it solves and the vocabulary, not every config key.

Why it matters: micro-frontends show up in system-design and architecture rounds for senior roles. You are expected to weigh the payoff (independent deployment, team autonomy) against the cost (shared-dependency version skew, operational complexity) — not to have memorized a config file.

Interview questions

Most-asked PWA, worker and micro-frontend questions in Indian interviews — these are the closing advanced-round topics for senior Angular roles:

Beginner What is a service worker, and what makes a web app a PWA?

A service worker is a script the browser runs in the background as a network proxy — it intercepts requests and can serve them from a cache, enabling offline support. A Progressive Web App is a web app that combines a service worker with a web app manifest so it becomes installable, launches full-screen, and works offline like a native app.

Beginner How do you turn an Angular app into a PWA?

Run ng add @angular/pwa. It installs @angular/service-worker, generates ngsw-config.json and a manifest.webmanifest, adds app icons, links the manifest in index.html, and registers the service worker (through provideServiceWorker() / ServiceWorkerModule) so it activates in production builds.

Beginner Why doesn't the Angular service worker run during ng serve?

It's registered only for production builds. The dev server serves unhashed files and you want live reloads, not cached responses — a worker there would serve stale assets and mask changes. To test it, build for production and serve the dist output over HTTP (service workers also require HTTPS, with localhost exempted).

Beginner What is the app shell pattern?

Pre-rendering a minimal static skeleton (header, layout, spinner) at build time so the user sees meaningful UI instantly while the JavaScript bundle downloads and boots. In Angular you scaffold it with ng generate app-shell; the shell is baked into index.html and cached by the service worker, giving a fast, offline-capable first paint.

Intermediate Asset groups vs data groups in ngsw-config.json?

Asset groups describe the app's own build artifacts (JS, CSS, index.html, images) with an installMode of prefetch (cache immediately) or lazy (cache on first request). Data groups describe runtime data like API calls, matched by URL patterns and controlled by a cacheConfig (strategy, maxSize, maxAge, timeout). Rule of thumb: assets = your files, data = server responses.

Intermediate Performance vs freshness caching strategy — when do you use each?

performance serves from cache first and only hits the network if the resource is missing or expired — fast, but can be stale; good for rarely-changing reference data. freshness goes to the network first and falls back to cache on failure or timeout — current, but slower; good for data that must be up to date. Choosing performance for volatile API data is the classic stale-data bug.

Intermediate A user doesn't see your latest deploy. Why, and how do you fix it?

The service worker serves the cached app and only installs the new version in the background, activating it on the next full load — so the running tab stays on the old version. Fix it with SwUpdate: subscribe to versionUpdates, and on a VERSION_READY event prompt the user (or call activateUpdate()) and reload the page.

Intermediate What is a web worker, and when would you reach for one in Angular?

A web worker runs code on a separate background thread, so CPU-intensive work (large parsing, image processing, heavy calculations) doesn't block the main thread and freeze the UI. Add one with ng generate web-worker. You communicate via postMessage() and onmessage; the worker has no DOM access, so it's for computation, not UI updates.

Intermediate What can't you do inside a web worker, and how does data cross the boundary?

No DOM: no window, document, or direct Angular component access. It has its own global scope with things like fetch and timers. Data crosses via postMessage() and is copied using the structured clone algorithm (or moved with transferable objects like ArrayBuffers) — nothing is shared by reference, so you can't pass functions or DOM nodes.

Pro Service worker vs web worker — what's the difference?

Both run off the main thread, but they solve different problems. A web worker is a background compute thread tied to a page, used to offload CPU work. A service worker is a network proxy scoped to an origin that lives independently of any page, intercepts fetches, and manages caching for offline/PWA behavior — it can wake up without an open tab (for push/sync). Web worker = computation; service worker = network/caching.

Pro What are micro-frontends, and how does Module Federation enable them?

Micro-frontends split one product into several independently built and deployed frontends, each owned by a team and composed at runtime. Module Federation lets a host application load code exposed by remote apps at runtime and share common libraries (Angular, RxJS) so they aren't shipped twice, with version negotiation for the shared dependencies. The payoff is independent deployment and team autonomy; the cost is shared-dependency skew and operational complexity.

Pro Module Federation vs Native Federation in modern Angular — why does Native Federation exist?

Module Federation is a webpack feature, but modern Angular's build system uses esbuild/Vite, not webpack. Native Federation (ANGULARarchitects) re-implements the same mental model and configuration on browser-native standards — ES modules and import maps — so it's build-tool-agnostic and keeps micro-frontends working with the current esbuild-based Angular builder. Same concepts, no webpack dependency.

Pro What are the sharpest production gotchas of the Angular service worker?

Stale content because updates activate on next load, not instantly (need SwUpdate + reload); a broken deploy getting "stuck" cached (ship the safety-worker.js / unregister path so you can recover); assuming it works in dev (production-only); forgetting HTTPS is required; and mis-scoped or over-eager performance data groups serving outdated API data. It's a caching utility, not a general service-worker framework — for advanced needs (background sync, custom fetch logic) you'd hand-write a worker or use Workbox.
