Module: Routing Fundamentals

A single-page app never reloads the page, so the URL becomes your application's state machine — and the router is what keeps the address bar, the browser history and the rendered view in agreement. This module maps the modern, standalone way to wire that up.

What this module covers

Two halves make up the routing foundation. First, setup: registering routes with provideRouter, marking where views render with <router-outlet>, and linking between them with routerLink. Second, reading state: pulling path parameters and query parameters back out of the URL through ActivatedRoute (or the newer component input binding) and moving between routes programmatically. Everything here uses the current Angular (v20) standalone APIs — no NgModule, no RouterModule.forRoot. Depth on each topic lives in the linked official docs.

Topics

provideRouter, route configuration, router-outlet, routerLink, navigation

A route configuration is just an array of objects mapping a URL path to a component (plus optional title, data, children and lazy loaders). You hand that array to provideRouter(routes) in your application config — the standalone replacement for RouterModule.forRoot — which registers the router services once for the whole app. Wherever you drop a <router-outlet>, the router renders the component matching the current URL; a component can host several outlets and nest them for layouts.

Why it matters: routing is the backbone of every real Angular app, and the standalone provideRouter setup is now the default the CLI scaffolds — so interviewers expect you to explain it, not the legacy module syntax. Navigation is declarative with the routerLink directive (which builds correct URLs, participates in history, and pairs with routerLinkActive for styling the current link) rather than a raw href, which would trigger a full-page reload and throw away application state.

// app.routes.ts
export const routes: Routes = [
  { path: '', component: HomeComponent, title: 'Home' },
  { path: 'products', component: ProductListComponent },
  { path: '**', component: NotFoundComponent },   // wildcard — must be last
];

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [provideRouter(routes)],             // replaces RouterModule.forRoot
};

// template
<a routerLink="/products" routerLinkActive="active">Products</a>

Route params, query params, ActivatedRoute, programmatic navigation

Path parameters (products/:id) name a required part of the URL that identifies the resource; query parameters (?sort=price&page=2) carry optional, cross-cutting state that does not change which route matches. You read both from the injected ActivatedRoute service: the paramMap / queryParamMap observables emit on every change, while route.snapshot gives a one-time static read. The modern shortcut is withComponentInputBinding(), which binds params straight to an input() on the component so you often need no subscription at all.

Why it matters: the single most common routing bug in interviews is a component that fails to update when only the parameter changes — because Angular reuses the component instance and the developer read the value once from snapshot instead of subscribing. Programmatic navigation with Router.navigate([...]) (or navigateByUrl) rounds out the picture, letting you route after a save, a login, or a guard decision, and to attach queryParams and relative paths.

// enable input binding once, in app.config.ts
provideRouter(routes, withComponentInputBinding());

// route: { path: 'products/:id', component: ProductComponent }
export class ProductComponent {
  id = input.required<string>();            // :id bound straight from the URL

  private route  = inject(ActivatedRoute);
  private router = inject(Router);

  ngOnInit() {
    // reacts to /products/1 -> /products/2 without re-creating the component
    this.route.paramMap.subscribe(p => this.load(p.get('id')));
  }

  open(id: string) {
    this.router.navigate(['/products', id], { queryParams: { ref: 'list' } });
  }
}

Interview questions

Routing is a guaranteed topic in Indian Angular interviews — from service-company screening rounds to product-company deep dives. These are the questions it actually powers:

Beginner What does provideRouter do, and how is routing set up in a standalone app?

provideRouter(routes) registers the router services and your route configuration for the whole application. You add it to the providers array in app.config.ts (used by bootstrapApplication). It is the standalone replacement for the older RouterModule.forRoot(routes) — no AppRoutingModule needed. Extra behaviour is opted in through with* features like withComponentInputBinding() or withPreloading().

Beginner What is <router-outlet> and what happens without it?

It is a placeholder directive marking where the router renders the component for the matched route. Without an outlet the route matches but nothing appears on screen — the router has nowhere to put the view. A page can contain several outlets (including nested and named ones) for layouts and sidebars.

Beginner Why use routerLink instead of a normal href for in-app navigation?

href triggers a full browser reload — it re-downloads the app, loses in-memory state and defeats the point of an SPA. routerLink navigates on the client through the History API: no reload, state preserved, and it integrates with routerLinkActive for styling the current link. Use href only for truly external URLs.

Beginner What is the wildcard ** route and why does its position matter?

{ path: '**', component: NotFoundComponent } matches any URL not matched earlier — the 404 catch-all. Routes are evaluated top-to-bottom with a first-match-wins strategy, so the wildcard must be the last entry; place it earlier and it swallows everything below it.

Beginner How do you define a route parameter and read it in the component?

Declare it in the path with a colon: { path: 'products/:id', component: ProductComponent }. Read it either reactively with route.paramMap.subscribe(p => p.get('id')), statically with route.snapshot.paramMap.get('id'), or — modern default — bind it to an input() using component input binding.

Intermediate Route parameters vs query parameters — when do you use each?

Path params identify a resource and are part of the route match (/orders/42) — required, hierarchical. Query params carry optional, cross-cutting state that does not change which route loads (/orders?status=paid&page=2) — filters, sorting, pagination, search terms. Rule of thumb: if removing it should give you a different resource, it is a path param; if it just refines the view, it is a query param.

Intermediate snapshot vs the paramMap observable — what is the classic gotcha?

snapshot reads the value once at creation. When you navigate between two instances of the same route (/user/1 → /user/2), Angular reuses the component, does not re-run the constructor, and the snapshot stays stale. Subscribing to the paramMap observable receives the new value on every change. Use snapshot only when the parameter can never change during the component's life.

Intermediate How do you navigate programmatically? navigate() vs navigateByUrl()?

Inject Router and call it after some logic (save, login, guard result). navigate(['/products', id], { queryParams }) takes a commands array — supports relative navigation via relativeTo and structured query params. navigateByUrl('/products/42?ref=list') takes one absolute URL string and ignores the current route. Both return a Promise<boolean> that resolves to whether navigation succeeded.

Intermediate How do you attach and read query parameters?

Attach declaratively with [queryParams]="{ page: 2 }" on a routerLink, or programmatically via the queryParams option of navigate. Read them from route.queryParamMap (observable) or route.snapshot.queryParamMap. Preserve them across navigations with the queryParamsHandling: 'merge' (or 'preserve') option.

Intermediate What is withComponentInputBinding and why is it now preferred for reading params?

Enabled with provideRouter(routes, withComponentInputBinding()), it binds path params, query params, matrix params and resolved data directly to component input()s whose names match the keys. You get the value with no ActivatedRoute injection and no manual subscription — cleaner, and it composes naturally with signals. It is the modern default for simple parameter reads (Angular 16+).

Pro Navigating /users/1 → /users/2 does not refresh the view. Why, and two ways to fix it?

By default the router reuses the component instance for the same route configuration instead of destroying and recreating it, so lifecycle hooks like ngOnInit do not fire again. Fixes: (1) subscribe to paramMap / queryParamMap (or use input binding with signals) so the component reacts to the new value; (2) force recreation by setting routeReuseStrategy.shouldReuseRoute to false, or use onSameUrlNavigation: 'reload' — heavier, use only when a full reset is genuinely needed.

Pro Explain routerLink array syntax, matrix parameters, and relativeTo for complex navigation.

The array form ['/products', id, 'reviews'] builds a URL segment by segment and is safer than string concatenation for dynamic values. An object inside the array, ['product', { color: 'red', size: 'xl' }], produces matrix parameters (product;color=red;size=xl) — scoped to that segment, unlike query params which are global to the URL. relativeTo: this.route on navigate resolves the commands against the current activated route rather than the app root, which is what lets child components navigate to siblings (['../sibling']) without hard-coding the full path.
