Module: Internationalization and Accessibility

Reaching a global audience and reaching every user are two disciplines that ship together: translating an app across locales, and making it operable by keyboard and screen reader. This module maps both — Angular's compile-time i18n, the runtime alternatives, and the ARIA and CDK tools that make templates accessible.

What this module covers

Two large, interview-relevant areas. First, internationalization (i18n): Angular's first-party @angular/localize is compile-time — you mark text, extract it, translate it, and build one AOT bundle per locale — plus the locale-aware pipes that format dates, currency and numbers, and the runtime libraries (Transloco, ngx-translate) you reach for when users must switch language without a reload. Second, accessibility (a11y): semantic templates and ARIA done the Angular way, and the CDK a11y toolkit for focus traps, focus monitoring and screen-reader announcements. Both are increasingly non-negotiable in production and both come up in senior rounds.

Topics

Compile-time i18n with @angular/localize

Angular's built-in i18n is compile-time. Install it with ng add @angular/localize, then mark static template text with the i18n attribute (and element attributes such as title with i18n-title), or mark strings in TypeScript with the $localize tagged template. Metadata written as meaning|description@@customId gives translators context and — via the custom ID — a stable key that survives copy edits. ICU expressions handle the grammar that differs across languages: plural for counts, select for gender and other categories. Running ng extract-i18n produces a source file (messages.xlf, XLIFF by default) that you copy per target locale for translators to fill in.

Why it matters: because translation happens at build time via AOT, each locale is a separate, fully-baked bundle with zero runtime translation cost — the best performance and SEO story. Interviewers probe whether you understand it is build-time, how ICU plurals work, and why the stable @@id keeps translations from breaking.

<h1 i18n="site header|Home page welcome banner@@homeTitle">Welcome</h1>
<img i18n-alt alt="Company logo" src="logo.svg">
<span i18n>{count, plural, =0 {No results} =1 {One result} other {# results}}</span>

Locales, data formatting, building and deploying

A locale ID is a BCP 47 / Unicode CLDR tag such as en-US, fr-CA or hi-IN — a language plus a region, because "the same language, different country" needs distinct formatting. You set the project's sourceLocale and map each target locale to its translation file in angular.json's i18n block. Locale-aware pipes — DatePipe, CurrencyPipe, DecimalPipe, PercentPipe — format values using the LOCALE_ID token (overridable per pipe). ng build --localize emits one AOT-compiled bundle per configured locale into its own directory; you deploy each under a subpath and redirect users on the Accept-Language header (or let Angular SSR do the redirect automatically).

Why it matters: this is the operational half — how translated bundles actually ship. Knowing that all formatting flows through LOCALE_ID and CLDR data, and that non-default locales need their locale data registered, is a frequent follow-up.

{{ today | date:'fullDate' }}     <!-- formatted for the active LOCALE_ID -->
{{ price | currency:'INR' }}      <!-- ₹ with Indian digit grouping -->
{{ ratio | percent:'1.0-1' }}

Runtime i18n alternatives

@angular/localize is compile-time, so switching language means loading a different build. When users must toggle language in-app without a reload — a language dropdown that swaps content live — reach for a runtime library. Transloco (the actively maintained modern choice, now published as @jsverse/transloco after the org rename from @ngneat) and the older ngx-translate load JSON translation files at runtime, expose a service plus a template directive (Transloco's *transloco), and support lazy-loaded scoped translations to keep bundles small.

Why it matters: a classic design question is compile-time vs runtime i18n — compile-time gives top performance, SEO and one bundle per locale but no live switching; runtime gives dynamic switching and a single bundle at a small runtime cost. Naming the tradeoff, and knowing Transloco superseded the @ngneat package, signals real-world experience.

Accessible templates: semantic HTML and ARIA

Accessibility starts with semantics, not ARIA. Prefer native elements — <button>, <a href>, <input>, <nav>, <main> — because they ship keyboard behavior, focus handling and roles for free; the first rule of ARIA is "don't use ARIA if a native element already does the job." Add ARIA only to fill genuine gaps, and in Angular use attribute binding ([attr.aria-label], [attr.aria-expanded]) so the value stays synchronized with component state — static ARIA such as role="alert" works as a plain attribute. RouterLinkActive's ariaCurrentWhenActive input sets aria-current on the active nav link. For custom widgets you genuinely must build, the WAI-ARIA Authoring Practices Guide is the reference for correct roles, states and keyboard interaction.

Why it matters: keyboard operability, screen-reader support and WCAG conformance are becoming legal and contractual requirements. Interviewers ask why native-first beats "div soup plus ARIA," and specifically how you bind ARIA reactively in Angular.

<button [attr.aria-expanded]="open()" aria-controls="menu" (click)="toggle()">Menu</button>
<a routerLink="/home" routerLinkActive ariaCurrentWhenActive="page">Home</a>

Focus management and the CDK a11y toolkit

Single-page apps break the browser's default focus handling: after a client-side route change focus stays on <body>, so keyboard and screen-reader users lose their place. Manage it explicitly — move focus to the new view's heading on NavigationEnd. Angular's CDK @angular/cdk/a11y package is the idiomatic toolkit: cdkTrapFocus (the FocusTrap) constrains Tab within a modal or dialog and cdkFocusInitial sets the first-focused element; FocusMonitor reports how an element gained focus (mouse, keyboard, touch or programmatic) so you can show focus rings only for keyboard users; and LiveAnnouncer pushes text into an aria-live region so screen readers announce asynchronous changes like "Changes saved."

Why it matters: focus traps, post-navigation focus and live announcements are the accessibility problems unique to SPAs, and the CDK is Angular's built-in answer. This is prime senior-interview territory — expect to be asked how you would make a custom modal accessible end to end.

private live = inject(LiveAnnouncer);

save() {
  this.live.announce('Changes saved', 'polite');
}
// template: <div cdkTrapFocus cdkFocusInitial> … modal content … </div>

Interview questions

Most-asked i18n and accessibility questions in Indian interviews — product companies increasingly gate senior roles on both, and INR/date formatting and RTL come up often:

Beginner What does i18n stand for, and how is internationalization different from localization?

i18n = "internationalization" (18 letters between i and n). Internationalization is designing the app so it can adapt to locales — externalizing text, using locale-aware formatting. Localization (l10n) is the actual adaptation for a specific locale: translating strings and formatting dates, numbers and currency for that region.

Beginner How do you mark static text in an Angular template for translation with the built-in i18n?

Add the i18n attribute to the element: <h1 i18n>Welcome</h1>. For a translatable element attribute like title or alt, use i18n-title / i18n-alt. In TypeScript, wrap the string with the $localize tagged template.

Beginner Which command extracts translatable strings, and what file does it produce?

ng extract-i18n. By default it generates messages.xlf (XLIFF 1.2) in the project root; --format can emit XLIFF 2, XMB, ARB or JSON, and --output-path/--out-file change where and how it is named. You copy that source file per target locale for translators.

Beginner Which pipes format data per locale, and what drives them?

DatePipe, CurrencyPipe, DecimalPipe and PercentPipe. They read the LOCALE_ID injection token and apply CLDR formatting rules for that locale. Each pipe also accepts an explicit locale argument that overrides the global one for that single use.

Intermediate Compile-time (@angular/localize) vs runtime (Transloco/ngx-translate) i18n — when do you choose each?

Compile-time bakes each locale into its own AOT bundle: best performance and SEO, no runtime translation cost, but the user cannot switch language without loading a different build. Runtime libraries load JSON at runtime and switch language live in one bundle, at a small runtime cost. Rule of thumb: marketing/SEO-critical or per-locale-deployed apps → compile-time; apps needing an in-app language toggle → runtime.

Intermediate How do ICU expressions handle plurals? Give an example.

ICU plural maps count categories to messages so each language's pluralization rules apply: {count, plural, =0 {No items} =1 {One item} other {# items}} — # is replaced by the number. There is also select for non-count categories like gender: {gender, select, male {…} female {…} other {…}}. Both keep translation grammatically correct instead of word-for-word.

Intermediate What is the meaning|description@@id metadata for, and why does the custom ID matter?

meaning disambiguates identical source text used in different contexts (so each can be translated differently), description gives translators guidance, and @@id assigns a stable custom translation ID. The ID matters because it decouples the translation key from the source text — editing the English copy no longer invalidates existing translations, since Angular otherwise derives the ID from the text itself.

Intermediate How do you build and deploy multiple locales?

Define locales and their translation files under the i18n option in angular.json, set "localize" (true, an array, or per-configuration), then run ng build --localize — the CLI produces one bundle per locale in its own directory with the correct lang and base href. Deploy each under a subpath and redirect users on the Accept-Language header; with Angular SSR (outputMode: server) that redirect is handled automatically.

Intermediate You set LOCALE_ID to hi-IN (or fr) but dates still render in English format. Why, and how do you fix it?

Only en-US locale data is bundled by default; other locales need their CLDR data registered. Import it and call registerLocaleData (e.g. import localeHi from '@angular/common/locales/hi'; registerLocaleData(localeHi);), and provide { provide: LOCALE_ID, useValue: 'hi-IN' }. With a full --localize build the CLI registers the data for you.

Intermediate Why prefer native HTML elements over a div plus ARIA?

Native elements come with the right role, keyboard behavior and focusability built in — a <button> is focusable, activates on Enter/Space and announces as a button for free. Recreating that on a <div role="button"> means re-adding tabindex, key handlers and states by hand, which is error-prone. The first rule of ARIA: if a native element gives the semantics and behavior you need, use it instead of ARIA.

Intermediate How do you bind ARIA attributes in an Angular template, and why binding rather than static?

Use attribute binding: [attr.aria-expanded]="isOpen", [attr.aria-label]="label". ARIA values are HTML attributes, not DOM properties, so [aria-expanded] would fail — you bind through attr.. You bind (rather than hardcode) whenever the value reflects changing state, so assistive tech always reads the current state. Purely static ARIA like role="alert" can stay a plain attribute.

Pro A screen-reader user can't tell when an async save finishes and gets lost after route changes. How do you fix both?

For async updates, inject the CDK LiveAnnouncer and call announce('Changes saved', 'polite') to push the message into an aria-live region — the reader announces it without moving focus. For routing, subscribe to router NavigationEnd events and programmatically move focus to the new view's main heading (a focusable <h1 tabindex="-1">) or a skip target, so keyboard and screen-reader users land in the fresh content instead of on <body>.

Pro What does cdkTrapFocus solve, and what else makes a modal truly accessible beyond trapping focus?

cdkTrapFocus keeps Tab/Shift+Tab cycling inside the dialog so focus can't wander to the page behind it; cdkFocusInitial sets where focus lands on open. A fully accessible modal also: gives the dialog role="dialog" with aria-modal="true" and an accessible name (aria-labelledby); moves focus in on open and restores it to the trigger on close; closes on Esc; and marks background content inert so screen readers don't reach it. Angular Material's MatDialog handles all of this out of the box.
