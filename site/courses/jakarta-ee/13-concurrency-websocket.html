Module: Concurrency and WebSocket

A Jakarta EE container owns its threads — so background work goes through a managed executor, not new Thread(). This module orients you on managed concurrency, the annotated WebSocket endpoint, and Jakarta Mail, then points to the resources that teach each in full.

Knowledge Library → Courses → Jakarta EE → Concurrency and WebSocket

What this module covers

Three loosely related platform services close out the specification tour. Jakarta Concurrency gives container-managed versions of java.util.concurrent — ManagedExecutorService, ManagedThreadFactory, and the @Asynchronous annotation — so applications run asynchronous work without stealing threads the container needs to manage. Jakarta WebSocket adds full-duplex, low-latency messaging through annotated endpoints. Jakarta Mail rounds things off with a protocol-independent framework for sending and reading email. Spring's @Async, its WebSocket support, and JavaMailSender are the parallels worth holding in mind throughout.

Topics

Concurrency Utilities: managed executors, thread factory, and @Asynchronous

Jakarta Concurrency exposes managed counterparts of the standard executor types: ManagedExecutorService and ManagedScheduledExecutorService for submitting (and scheduling) tasks, ManagedThreadFactory for creating threads, and ContextService for wrapping objects so they carry container context. You obtain them by injection (@Resource) or JNDI lookup rather than constructing them. Since Jakarta EE 10, @Asynchronous on a CDI bean method runs that method on a managed executor and returns void, CompletableFuture, or CompletionStage.

Why it matters: inside a container you must not call new Thread() or spin up a plain ExecutorService. The container owns the thread pool; an unmanaged thread bypasses its lifecycle, misses the propagated naming, security, and CDI context, escapes monitoring, and can leak across a redeploy. A ManagedExecutorService solves all of that — explaining why raw threads are wrong in a managed runtime is the single most common interview point in this module.

@Resource
ManagedExecutorService executor;   // injected, container-owned

public CompletableFuture<Report> build(Query q) {
    return executor.supplyAsync(() -> generate(q)); // runs on a managed thread
}

@Asynchronous                       // Jakarta EE 10 — CDI method on a managed executor
public CompletionStage<Void> notifyUsers(Event e) { /* ... */ }

Jakarta WebSocket: @ServerEndpoint and the endpoint lifecycle

WebSocket (RFC 6455) upgrades a single HTTP connection into a persistent, full-duplex channel — ideal for chat, live dashboards, and push notifications where HTTP polling would be wasteful. In Jakarta EE the annotated model is the easy path: @ServerEndpoint("/chat/{room}") turns a POJO into a server endpoint, lifecycle callbacks @OnOpen, @OnMessage, @OnClose, and @OnError handle the connection, URI templates bind path segments with @PathParam, and Encoder/Decoder pairs convert application objects to and from text or binary frames.

Why it matters: standing an endpoint up is trivial; the subtlety is threading. A Session and its RemoteEndpoint.Basic are not safe for concurrent sends, so broadcasting to many clients means iterating a thread-safe collection of sessions and serialising writes per session. Spring offers its own WebSocket handler and STOMP @MessageMapping as an alternative, but the annotated Jakarta endpoint is the standard's answer.

@ServerEndpoint("/chat/{room}")
public class ChatEndpoint {
    @OnOpen    public void open(Session s, @PathParam("room") String room) { /* register */ }
    @OnMessage public void onMessage(String msg, Session s) { broadcast(msg); }
    @OnClose   public void close(Session s, CloseReason reason) { /* deregister */ }
    @OnError   public void error(Session s, Throwable t) { /* log */ }
}

Jakarta Mail overview

Jakarta Mail (formerly JavaMail; the package moved from javax.mail to jakarta.mail in 2021) is a platform- and protocol-independent framework for composing, sending, and reading email. The core types are Session (configuration plus authentication), Message/MimeMessage (the message itself, with MimeBodyPart and MimeMultipart for HTML bodies and attachments), Transport (send over SMTP via Transport.send), and Store/Folder (read over IMAP or POP3). The API is abstract on its own — you need an implementation such as Eclipse Angus (angus-mail). In a Jakarta EE server a configured Session can be injected with @Resource; standalone you build one from a Properties object.

Why it matters: it remains the standard way to send transactional and notification email from Java. Interviewers probe the roles of Session, MimeMessage, and Transport and the SMTP properties that wire them together (host, port, starttls, auth).

Properties props = new Properties();
props.put("mail.smtp.host", "smtp.example.com");
props.put("mail.smtp.auth", "true");
Session session = Session.getInstance(props, authenticator);

MimeMessage m = new MimeMessage(session);
m.setFrom(new InternetAddress("noreply@example.com"));
m.setRecipients(Message.RecipientType.TO, "user@example.com");
m.setSubject("Welcome");
m.setText("Hello");
Transport.send(m);

Interview questions

The questions Indian interviewers most often ask on managed concurrency, WebSocket, and mail — the managed-executor rationale leads because it is asked in nearly every Jakarta EE round:

Beginner Why shouldn't you start threads with new Thread() inside a Jakarta EE container?

The container owns and pools its threads. A thread you start yourself is invisible to it: no propagated naming, security, or CDI context, no monitoring or lifecycle management, and it can leak across a redeploy or exhaust resources. Submit work to a ManagedExecutorService instead, which runs it on a container-managed thread with context intact.

Beginner What is ManagedExecutorService and how do you obtain one?

It is a container-managed ExecutorService — same submit/execute/invokeAll API as Java SE, but tasks run on managed threads with container context propagated. You do not construct it; you inject it with @Resource or look it up via JNDI (for example java:comp/DefaultManagedExecutorService).

Beginner What does ManagedThreadFactory add over a plain ThreadFactory?

Threads it creates are managed by the container and start with the container's context established, so code running on them can use resources, security identity, and CDI as if on a request thread. A plain ThreadFactory produces context-less threads the container knows nothing about.

Beginner What is @Asynchronous in Jakarta Concurrency, and what may the method return?

Introduced in Jakarta EE 10, it marks a CDI bean method to execute asynchronously on a managed executor. The return type must be void, CompletableFuture, or CompletionStage; for the future types the container completes the stage with the result. It is the standard's counterpart to Spring's @Async.

Intermediate What context does a ManagedExecutorService propagate to worker threads?

Via ContextService it propagates classloader/application, JNDI naming, security identity, and CDI context to the executing thread so the task sees the same environment as its submitter. Transaction context is a deliberate exception — an active JTA transaction is not carried into the task by default.

Intermediate ManagedExecutorService vs ManagedScheduledExecutorService?

The scheduled variant adds delayed and periodic execution — schedule, scheduleAtFixedRate, scheduleWithFixedDelay — plus a Trigger SPI for calendar/cron-like scheduling. The plain managed executor only submits tasks for asynchronous execution.

Intermediate How do you declare a WebSocket server endpoint and handle its lifecycle?

Annotate a POJO with @ServerEndpoint("/path") and add lifecycle handlers: @OnOpen (receives the Session), @OnMessage (message plus Session), @OnClose (with CloseReason), and @OnError (with the Throwable). URI template segments bind with @PathParam.

Intermediate How do you send a message to a client, and why be careful sending concurrently on one Session?

Use session.getBasicRemote() for blocking sends or getAsyncRemote() for non-blocking ones. A Session is not thread-safe for concurrent writes — two threads calling sendText on the same session can corrupt the frame stream. Serialise sends per session (synchronise, or use the async remote which queues).

Intermediate What are encoders and decoders in Jakarta WebSocket for?

They convert between application objects and the text or binary frames on the wire. An Encoder.Text<T> turns an object into a string to send; a Decoder.Text<T> parses an incoming string into an object. You register them on the @ServerEndpoint so @OnMessage can take a typed parameter instead of raw String.

Intermediate Name the core Jakarta Mail types used to send an email.

Session holds configuration and authentication; MimeMessage is the message (with InternetAddress recipients, and MimeBodyPart/MimeMultipart for HTML and attachments); Transport.send delivers it over SMTP. Reading uses Store and Folder over IMAP or POP3. An implementation such as Eclipse Angus provides the protocol code.

Pro How does WebSocket differ from HTTP long-polling and Server-Sent Events, and when do you choose each?

WebSocket is a persistent full-duplex channel over one TCP connection — both sides push at will, with low per-message overhead. SSE is server-to-client only over a long-lived HTTP response, simpler and auto-reconnecting but one-directional. Long-polling repeatedly reopens requests and wastes resources. Choose WebSocket for interactive bidirectional traffic (chat, collaboration), SSE for one-way streams (feeds, notifications), and avoid polling unless the others are unavailable.

Pro Contrast Jakarta EE managed concurrency and WebSocket with their Spring equivalents.

Managed concurrency: Jakarta's ManagedExecutorService/@Asynchronous maps to Spring's TaskExecutor and @Async — both insist you use a managed pool rather than raw threads, differing mainly in how context propagates and how the pool is configured. WebSocket: Jakarta's annotated @ServerEndpoint maps to Spring's WebSocketHandler or the higher-level STOMP @MessageMapping broker model. The standard favours annotations on POJOs; Spring layers a messaging abstraction on top.

Pro If you submit a task from within a transaction, does the task run in that transaction — and how should you handle it?

No. Managed executors do not propagate or join the caller's JTA transaction; the asynchronous task runs outside it, so work done there does not commit or roll back with the originating transaction. Treat the task as its own unit of work — begin an explicit transaction inside it (or call a transactional bean method) — and never assume the submitting transaction's changes are visible or that a failure will roll them back.
