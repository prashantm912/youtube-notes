Module: Jakarta Messaging (JMS)

Messaging is how enterprise systems stay decoupled: one component drops a message and moves on, another picks it up when it is ready. JMS is the vendor-neutral API for that conversation — this module maps its two delivery models, its modern simplified API, and where message-driven beans hand off to Spring's @JmsListener.

What this module covers

Jakarta Messaging (JMS) is the standard API for asynchronous, loosely coupled communication between Jakarta EE components and message brokers. This module covers the two delivery models — point-to-point queues and publish/subscribe topics — the connection factory and producer/consumer objects, the JMS 2.0/3.0 simplified API built around JMSContext, and message-driven beans. It closes with a side-by-side against Spring JMS and @JmsListener, the framework most enterprise teams actually ship.

Topics

Point-to-point queues vs publish/subscribe topics

JMS defines two messaging domains. Point-to-point uses a queue: each message has exactly one consumer, the broker retains it until it is consumed or expires, and delivery is load-balanced across competing consumers — ideal for distributing work so each task runs once. Publish/subscribe uses a topic: a published message is broadcast to every active subscriber (zero to many), like a bulletin board — ideal for fanning an event out to many interested parties.

Why it matters: picking the wrong domain is a classic design error. Queues give you a durable work backlog with competing consumers; topics give you broadcast but, by default, only reach subscribers connected at publish time — durable subscriptions close that gap. Both destinations are administered objects (Queue/Topic) looked up or injected, so application code barely changes between them. Interviewers almost always open a JMS round with "queue vs topic".

ConnectionFactory, producers and consumers

Every JMS client starts from a ConnectionFactory — an administered object (configured on the server, injected with @Resource) that hands out connections to the broker. In the classic (JMS 1.1) API you then open a Session, and from it a MessageProducer to send and a MessageConsumer to receive; the session is single-threaded and owns transaction and acknowledgment behaviour. Messages come in five body types — TextMessage, BytesMessage, MapMessage, ObjectMessage, StreamMessage — plus headers and application properties used for routing and selectors.

Why it matters: the ConnectionFactory and Destination pair are the only two things your code needs to stay portable across WildFly, Payara or Open Liberty — the broker wiring lives in server config, not in source. Understanding the classic Connection → Session → Producer/Consumer chain is what makes the simplified API (next topic) click. Receiving can be synchronous (receive() blocks) or asynchronous (a MessageListener callback).

@Resource(lookup = "jms/ConnectionFactory") ConnectionFactory cf;
@Resource(lookup = "jms/OrderQueue")        Queue queue;
// classic API path: Connection -> Session -> MessageProducer / MessageConsumer

The simplified API: JMSContext (JMS 2.0 / 3.0)

JMS 2.0 — carried into Jakarta Messaging 3.x — added a simplified API that collapses Connection and Session into one object, JMSContext, and replaces the verbose producer/consumer with JMSProducer (method-chaining / builder style) and JMSConsumer. It throws unchecked JMSRuntimeException instead of checked JMSException, and JMSContext is AutoCloseable, so try-with-resources closes the connection and session together.

Why it matters: the result is dramatically less boilerplate — "create context, send, done" is a few lines instead of a four-object dance. Inside a Jakarta EE container an injected JMSContext is even managed for you and scoped to the active transaction. This is the API interviewers expect you to reach for today; being able to contrast it with the classic API signals you have used JMS in its modern form.

try (JMSContext ctx = connectionFactory.createContext()) {
    ctx.createProducer().send(orderQueue, "PLACED:42");        // no checked exceptions
    String body = ctx.createConsumer(orderQueue).receiveBody(String.class);
}

Message-driven beans (MDBs)

A message-driven bean is an enterprise bean that consumes messages asynchronously: the container listens on a destination, and when a message arrives it invokes the bean's onMessage(Message) method. Unlike session beans, an MDB has no business interface and no client-visible identity — clients never call it directly; the message is the trigger. It implements MessageListener and is configured with @MessageDriven plus activation-config properties (destination lookup, destination type, acknowledge mode).

Why it matters: MDBs give you container-managed concurrency, transactions and pooling for message consumption for free — the container runs many bean instances in parallel and can roll back and redeliver a message if onMessage throws (watch for poison messages and redelivery limits). They are the Jakarta EE-native answer to "how do I process a queue," and the direct conceptual ancestor of Spring's @JmsListener.

@MessageDriven(activationConfig = {
    @ActivationConfigProperty(propertyName = "destinationLookup", propertyValue = "jms/OrderQueue"),
    @ActivationConfigProperty(propertyName = "destinationType",   propertyValue = "jakarta.jms.Queue")
})
public class OrderProcessor implements MessageListener {
    public void onMessage(Message message) { /* business logic, transactional */ }
}

Contrast: Spring JMS and @JmsListener

Spring provides the same capabilities without a full Jakarta EE application server. JmsTemplate wraps the send/receive boilerplate (much like JMSContext), and @JmsListener turns any Spring bean method into a listener endpoint — the Spring-managed equivalent of an MDB's onMessage — activated by @EnableJms and a DefaultJmsListenerContainerFactory. It supports concurrency, message selectors, durable and shared subscriptions, and typed payload binding through message converters.

Why it matters: for most teams the choice is architectural, not technical. MDBs need a Jakarta EE container (WildFly, Payara, Open Liberty); @JmsListener runs in a plain Spring Boot app against any broker (ActiveMQ, Artemis, IBM MQ). The programming models mirror each other — onMessage vs an annotated method, activation config vs container-factory plus annotation attributes — so knowing one transfers to the other. Expect interviewers to ask you to compare them directly.

@JmsListener(destination = "OrderQueue")
public void handle(String body) {
    // Spring-managed equivalent of an MDB's onMessage
}

Interview questions

The JMS questions Indian interviewers lean on — from the queue-vs-topic opener through acknowledgment and MDBs to the inevitable "JMS vs Kafka":

Beginner What is Jakarta Messaging (JMS) and what problem does it solve?

A standard API for asynchronous, loosely coupled communication between components through a message broker. Sender and receiver are decoupled in time and location — the producer sends and moves on, the consumer processes when ready — which improves resilience, absorbs load spikes, and lets components scale independently.

Beginner Queue vs Topic — the core difference?

A queue is point-to-point: each message is delivered to exactly one consumer and retained until consumed or expired, with delivery load-balanced across competing consumers. A topic is publish/subscribe: each message is broadcast to every active subscriber. Queue = distribute work once; topic = fan an event out to many.

Beginner What is a ConnectionFactory and why is it an administered object?

It is the object a client uses to create connections (or a JMSContext) to the broker. It is administered — configured on the server and injected with @Resource — so broker addresses and credentials stay out of application code, keeping the app portable across servers and brokers.

Beginner Name the JMS message body types.

TextMessage (string, often JSON/XML), BytesMessage (raw bytes), MapMessage (key/value pairs), ObjectMessage (a serialized Java object), and StreamMessage (a stream of primitives). All carry standard headers plus custom application properties used for routing and message selectors.

Intermediate Classic API vs the simplified API (JMSContext)?

Classic (JMS 1.1): separate Connection, Session, and MessageProducer/MessageConsumer, with checked JMSException. Simplified (JMS 2.0+): JMSContext combines connection and session, JMSProducer/JMSConsumer replace the classic pair, exceptions become unchecked JMSRuntimeException, and JMSContext is AutoCloseable — far less boilerplate for the same functionality.

Intermediate What is a message-driven bean and how does it receive messages?

An enterprise bean with no client interface that the container invokes via onMessage(Message) whenever a message arrives on its configured destination. It implements MessageListener, is declared with @MessageDriven and activation-config properties, and gets container-managed threading and transactions — the EE-native way to consume a queue or topic asynchronously.

Intermediate Durable, non-durable and shared subscriptions on a topic?

Non-durable: the subscriber receives only messages published while it is connected. Durable: the broker retains messages for a named subscriber while it is offline and delivers them on reconnect. Shared (JMS 2.0): multiple consumers share one subscription so topic messages are load-balanced among them — combining broadcast semantics with horizontal scaling.

Intermediate Synchronous vs asynchronous message receipt?

Synchronous: the consumer calls receive() or receiveBody(), which blocks (optionally with a timeout) until a message is available. Asynchronous: you register a MessageListener — or use an MDB / @JmsListener — and the container invokes onMessage when a message arrives, freeing the thread in between.

Intermediate How does Spring's @JmsListener compare to a message-driven bean?

Both are asynchronous listener endpoints invoked once per message, with mirror-image programming models (an annotated method vs onMessage; container-factory attributes vs activation config). The difference is the runtime: @JmsListener needs only Spring and a broker and runs in a plain Spring Boot app, while an MDB needs a Jakarta EE container. Skills transfer directly between them.

Pro What are acknowledgment modes, and how do transactions change them?

AUTO_ACKNOWLEDGE, CLIENT_ACKNOWLEDGE and DUPS_OK_ACKNOWLEDGE control when the broker considers a consumed message removed. In a transacted session or a JTA transaction, acknowledgment is bound to commit — a rollback returns the message for redelivery. MDBs typically use container-managed transactions, so an exception in onMessage rolls back and the message is redelivered.

Pro What is a poison message and how do you handle redelivery?

A message that fails processing and, because each failure rolls back the transaction, is redelivered endlessly. Guard against it with a redelivery limit and the JMSXDeliveryCount header, routing exhausted messages to a dead-letter queue so the broker sidelines them after N attempts. Also make onMessage idempotent, since at-least-once delivery means duplicates are possible.

Pro When would you choose JMS over Apache Kafka?

JMS is a broker-neutral API for classic enterprise messaging: transactional queues, request/reply, per-message acknowledgment, and tight JTA integration at moderate throughput. Kafka is a distributed, partitioned, replayable log built for high-throughput event streaming with consumer-group offsets and long retention. Reach for JMS for reliable task queues and transactional workflows; reach for Kafka for event sourcing, stream processing and very high volume.
