Module: RESTful Web Services (JAX-RS)

A handful of annotations turn a plain Java class into a REST endpoint — the same @GET/@Path vocabulary whether you deploy on WildFly, Payara or Quarkus. This module maps every annotation to the resource that teaches it, and sets each against its Spring MVC equivalent.

Knowledge Library → Courses → Jakarta EE → RESTful Web Services (JAX-RS)

What this module covers

Jakarta RESTful Web Services (JAX-RS) is the platform's standard, annotation-driven API for building REST endpoints. This module walks the whole request path: declaring resource classes with @Path, wiring HTTP verbs, pulling data out of the URI/query/headers/body, negotiating media types with @Produces/@Consumes, serializing with JSON-B, and bootstrapping the application on Jersey or RESTEasy. Because most Indian teams also run Spring, the final topic contrasts JAX-RS with Spring MVC's @RestController — a perennial interview question.

Topics

What JAX-RS is, and resource classes with @Path

JAX-RS is a specification — a set of annotations and interfaces in the jakarta.ws.rs package — not a runtime. The runtime is supplied by an implementation: Jersey (the reference implementation, bundled with GlassFish), RESTEasy (WildFly/JBoss), or Apache CXF. You annotate an ordinary POJO with @Path to make it a root resource class; its methods answer requests under that URI, and templates like @Path("/books/{id}") capture path segments.

Why it matters: portability. The same annotated class runs unchanged on any compliant server — so interviews often open with "is JAX-RS a framework or a spec?" and "name two implementations."

@Path("/books")
public class BookResource {
    @GET
    @Produces(MediaType.APPLICATION_JSON)
    public List<Book> all() { return repo.findAll(); }
}

HTTP method annotations and reading request data

@GET, @POST, @PUT, @DELETE, @HEAD, @OPTIONS and @PATCH designate which HTTP verb a resource method answers — each is itself meta-annotated with @HttpMethod, so custom verbs are possible. Request data is injected through parameter annotations: @PathParam (URI-template variable), @QueryParam (?key=value), @HeaderParam (an HTTP header), @FormParam (a form-encoded body field), plus @CookieParam and @MatrixParam. @BeanParam aggregates several into one object, @DefaultValue supplies a fallback, and @Context injects framework objects like UriInfo, HttpHeaders and SecurityContext.

Why it matters: this is the core of request handling and a guaranteed interview area — the difference between a path param (identifies the resource) and a query param (filters/paginates it) comes up constantly.

@GET
@Path("/{id}")
public Book byId(@PathParam("id") long id,
                 @QueryParam("fields") @DefaultValue("all") String fields,
                 @HeaderParam("X-Trace-Id") String traceId) {
    return repo.find(id);
}

Content negotiation: @Produces / @Consumes and building responses

@Produces lists the media types a method can return (matched against the client's Accept header); @Consumes lists what it accepts in the request body (matched against Content-Type). Together they drive content negotiation and select the right MessageBodyWriter/MessageBodyReader. Return a domain object and let the provider serialize it, or return a jakarta.ws.rs.core.Response to control status code, headers and entity explicitly.

@POST
@Consumes(MediaType.APPLICATION_JSON)
@Produces(MediaType.APPLICATION_JSON)
public Response create(Book incoming) {
    Book saved = repo.save(incoming);
    return Response.status(Response.Status.CREATED)
                   .entity(saved)
                   .build();
}

JSON binding with JSON-B

JSON-B (Jakarta JSON Binding) is the platform's standard object↔JSON mapper; its reference implementation is Eclipse Yasson. When a resource declares @Produces(APPLICATION_JSON) and returns a POJO, the JAX-RS runtime picks the JSON-B MessageBodyWriter and serializes it automatically — no Jackson ObjectMapper wiring. Customize with @JsonbProperty, @JsonbTransient, @JsonbDateFormat or a JsonbConfig. JSON-B sits on top of JSON-P, the lower-level streaming/DOM JSON API.

Worth knowing for the Spring contrast: Jakarta EE defaults to JSON-B, whereas Spring Boot defaults to Jackson — a small but common "which JSON engine?" follow-up.

Configuring the application: the Application class, @ApplicationPath and implementations

A JAX-RS application is bootstrapped by subclassing jakarta.ws.rs.core.Application and annotating it @ApplicationPath("/api") to set the base URI — no web.xml needed on a Servlet 3+ container. Leave getClasses() empty for classpath auto-discovery of @Path/@Provider types, or override it to register resources and providers explicitly. The chosen implementation (Jersey, RESTEasy) supplies the runtime plus its own extra configuration knobs — Jersey's ResourceConfig, RESTEasy's deployment options.

@ApplicationPath("/api")
public class RestApplication extends Application {
    // empty body: resources & providers are auto-discovered
    // effective endpoint: /<context-root>/api/books
}

JAX-RS vs Spring MVC @RestController

Both expose REST endpoints by annotating a class, but JAX-RS is a vendor-neutral Jakarta standard while @RestController is Spring-specific. The mental mapping: @Path ≈ @RequestMapping; @GET/@POST ≈ @GetMapping/@PostMapping; @PathParam ≈ @PathVariable; @QueryParam ≈ @RequestParam; @Produces/@Consumes ≈ the produces/consumes attributes. The request body is an un-annotated entity parameter in JAX-RS versus an explicit @RequestBody in Spring; serialization is JSON-B/JAXB via MessageBodyWriter versus Jackson via HttpMessageConverter; and JAX-RS resources default to per-request instances while Spring controllers are singleton beans.

Why it matters: this exact comparison is one of the most-asked enterprise interview questions in India, especially where a team runs both stacks. Knowing the annotation table cold is the differentiator.

Interview questions

Most-asked JAX-RS questions in Indian enterprise interviews — from screening rounds to product-company deep dives, with the Spring contrast that follows almost every time:

Beginner Is JAX-RS a framework or a specification, and name two implementations.

A specification — a set of annotations and interfaces in the jakarta.ws.rs package, part of Jakarta EE (formerly Java EE). It needs a runtime implementation to actually serve requests: Jersey (the reference implementation, bundled with GlassFish), RESTEasy (WildFly/JBoss), or Apache CXF. Your annotated code is portable across all of them.

Beginner What does @Path do, and how do path templates work?

@Path binds a class or method to a URI path. On a class it declares a root resource; a template in braces (@Path("/users/{id}")) captures a segment, read back with @PathParam("id"). You can constrain a template with a regex, e.g. {id: \d+} to accept digits only.

Beginner Which annotations designate HTTP methods on a resource method?

@GET, @POST, @PUT, @DELETE, @HEAD, @OPTIONS and @PATCH. Each is itself meta-annotated with @HttpMethod, which is how you could define a custom verb. A resource method with no HTTP-method annotation is not directly reachable — it can only be a sub-resource locator.

Beginner Difference between @PathParam and @QueryParam?

@PathParam reads a variable from the URI path template (/books/42 → id=42); @QueryParam reads it from the query string (/books?author=x → author=x). Path params identify the resource; query params filter, sort or paginate it. Pair either with @DefaultValue to handle absence.

Beginner What do @Produces and @Consumes do?

@Produces lists the media types a method can return, matched against the client's Accept header; @Consumes lists the media types it accepts in the request body, matched against Content-Type. They drive content negotiation and select the correct MessageBodyWriter/MessageBodyReader.

Intermediate How do you configure a JAX-RS application without web.xml?

Subclass jakarta.ws.rs.core.Application and annotate it @ApplicationPath("/api") to set the base URI; on a Servlet 3+ container the implementation auto-registers it. Leave getClasses() empty for classpath scanning of @Path/@Provider types, or override it to register them explicitly.

Intermediate How does a returned POJO become JSON?

When a method declares @Produces(APPLICATION_JSON) and returns an object, the runtime selects a MessageBodyWriter — by default JSON-B (Eclipse Yasson) on Jakarta EE — and serializes it. Customize field names and formats with @JsonbProperty, @JsonbDateFormat and friends. No manual Jackson ObjectMapper wiring is required.

Intermediate List the parameter-injection annotations beyond @PathParam and @QueryParam.

@HeaderParam (an HTTP header), @FormParam (a field in an application/x-www-form-urlencoded body), @CookieParam, @MatrixParam; plus @BeanParam to bundle several into one object and @Context to inject framework objects such as UriInfo, HttpHeaders, SecurityContext and Request. @DefaultValue can accompany any of them.

Intermediate How do you return 201 Created with a Location header?

Return a jakarta.ws.rs.core.Response: Response.created(uri).entity(dto).build(), or Response.status(CREATED).header("Location", uri).entity(dto).build(). Returning a bare object yields 200 with defaults; returning a Response gives full control of status code, headers and entity.

Intermediate Sub-resource method vs sub-resource locator?

A sub-resource method carries both an HTTP-method annotation and @Path — it handles the request directly. A sub-resource locator has @Path but no HTTP-method annotation; it returns an object that JAX-RS then continues matching against, letting you build resource trees chosen at runtime.

Pro Walk through how JAX-RS matches a request to a method.

Three filters, in order: (1) match the request URI against root-resource @Path values, then sub-resources, to find candidate methods; (2) filter by HTTP method (@GET etc.); (3) filter by media type — the request Content-Type against @Consumes and the Accept header against @Produces, choosing the most specific match. No candidate at each stage yields 404, then 405 (Method Not Allowed), then 406/415 respectively.

Pro Are JAX-RS resource classes singletons? What about thread-safety and CDI?

By default root resource classes are per-request — a fresh instance per request — so instance fields can safely hold request state. Providers registered through getClasses() are singletons. You can make a resource a singleton (register it via getSingletons() or as an @ApplicationScoped CDI bean), after which shared mutable state needs synchronization. JAX-RS integrates with CDI, so you can @Inject services into resources.

Pro JAX-RS vs Spring MVC @RestController — the differences an interviewer wants.

JAX-RS is a vendor-neutral Jakarta standard, portable across Jersey/RESTEasy/CXF and any compliant server; @RestController is Spring's own (a meta-annotation combining @Controller + @ResponseBody). Annotation mapping: @Path↔@RequestMapping, @GET↔@GetMapping, @PathParam↔@PathVariable, @QueryParam↔@RequestParam, @Produces/@Consumes↔produces/consumes. Body binding: an un-annotated entity parameter in JAX-RS vs explicit @RequestBody in Spring. Serialization: JSON-B/JAXB via MessageBodyWriter vs Jackson via HttpMessageConverter. Lifecycle: JAX-RS resources default per-request; Spring controllers are singleton beans. Same job — portability and the default JSON engine are the headline distinctions.
