Module: MicroProfile and Modern Jakarta EE

Jakarta EE standardises enterprise Java; MicroProfile is the sister umbrella that adds the twelve-factor pieces microservices need — externalised config, health probes, resilience, typed REST clients and token security — and Quarkus, Helidon and Open Liberty are how you ship it. This is the module that lets you answer "so why not just use Spring Boot?"

What this module covers

Jakarta EE gives you CDI, JAX-RS and JPA; MicroProfile layers the cloud-native concerns on top — configuration, observability, fault tolerance and JWT security — as small, CDI-friendly specifications you compose per service. This module orients you on each MicroProfile spec and on the runtimes that implement it (Quarkus, Helidon, Open Liberty), and it keeps one eye on the Spring Boot equivalent throughout, because that comparison is the heart of most interview conversations about "modern" Jakarta EE.

Topics

What MicroProfile is, and how it relates to Jakarta EE

MicroProfile is an Eclipse Foundation umbrella of specifications that optimise enterprise Java for a microservices architecture. It deliberately reuses Jakarta EE building blocks — CDI for wiring, JAX-RS for REST, JSON-B/JSON-P for serialization — and adds the operational specs a container platform expects: Config, Health, Metrics/Telemetry, OpenAPI, Rest Client, Fault Tolerance and JWT. A MicroProfile application is portable: written to the spec, it runs on any compliant runtime.

Why it matters: the two umbrellas are complementary, not competing. Jakarta EE 10 introduced the Core Profile — a slim platform aimed at small runtimes — precisely so MicroProfile can build on a lean, CDI-plus-REST foundation instead of the full application-server stack. Knowing that MicroProfile is "Jakarta EE for microservices, minus the heavyweight profile" is the framing interviewers look for. The Spring parallel is Spring Boot's starters and Actuator, but MicroProfile is a vendor-neutral standard with multiple independent implementations.

MicroProfile Config

MicroProfile Config aggregates configuration from multiple ordered sources — system properties, environment variables, microprofile-config.properties, and any custom ConfigSource you register — into a single merged view, with the highest-priority source winning. You read values by injecting them with @ConfigProperty or by resolving them programmatically through the Config object, and it converts strings into typed values (including Optional and lists).

Why it matters: it is the twelve-factor "config in the environment" principle as a portable API — the direct counterpart to Spring's @Value / @ConfigurationProperties and Environment. Because sources are ordered, a bundled default can be overridden by a container environment variable without a rebuild.

@Inject
@ConfigProperty(name = "payment.gateway.url",
                defaultValue = "https://sandbox.example.com")
String gatewayUrl;                       // injected, env var overrides default

@Inject
@ConfigProperty(name = "payment.retries")
Optional<Integer> retries;              // absence is explicit, not a crash

Observability: Health, Metrics and Telemetry

MicroProfile Health exposes machine-readable status endpoints so an orchestrator can act on them: @Liveness ("is the process wedged — restart it?"), @Readiness ("can it take traffic yet?") and @Startup ("has slow initialisation finished?") map one-to-one onto Kubernetes probes. MicroProfile Metrics adds counters, gauges, timers and histograms on a /metrics endpoint for scraping.

Why it matters — and the modern twist: newer MicroProfile releases position MicroProfile Telemetry (built on OpenTelemetry, covering traces, metrics and logs) as the forward-looking observability story, gradually superseding the standalone Metrics spec. Interviewers reward candidates who know Metrics is the classic answer but Telemetry/OpenTelemetry is where the ecosystem is heading. The Spring parallel is Actuator's health and metrics endpoints plus Micrometer/OpenTelemetry.

@Readiness
public class DbReadyCheck implements HealthCheck {
  @Override public HealthCheckResponse call() {
    return HealthCheckResponse.named("db")
        .status(database.isReachable())   // false -> pod removed from Service
        .build();
  }
}

MicroProfile OpenAPI

MicroProfile OpenAPI generates an OpenAPI v3 document for your JAX-RS endpoints automatically, served at /openapi, and lets you enrich it with annotations (@Operation, @Schema, @APIResponse) or supply a hand-written base document. Most runtimes also wire up a Swagger UI so the contract is browsable and testable.

Why it matters: a live, generated API contract is table stakes for a microservice that other teams consume. This is the standardised equivalent of springdoc-openapi / Swagger in the Spring Boot world — same output, defined by a portable spec rather than a third-party library.

MicroProfile Rest Client

The Rest Client turns service-to-service calls into a type-safe Java interface: you declare an interface annotated with JAX-RS annotations (@Path, @GET, @Produces) and @RegisterRestClient, and the runtime generates the HTTP client, wires the base URL from config, and lets you @Inject @RestClient it. No manual request building, no string URLs scattered through code.

Why it matters: it is the declarative analogue of Spring's @FeignClient / HTTP Interface clients, and it composes with Fault Tolerance and JWT propagation. Defining the remote API once as an interface is the idiom interviewers expect for inter-service calls.

@Path("/orders")
@RegisterRestClient(configKey = "order-api")
public interface OrderClient {
  @GET @Path("/{id}")
  @Produces(MediaType.APPLICATION_JSON)
  Order byId(@PathParam("id") String id);
}
// base URL: order-api/mp-rest/url in config; inject with @Inject @RestClient OrderClient

MicroProfile Fault Tolerance

Fault Tolerance provides resilience patterns as declarative annotations you place on a method: @Retry (reattempt transient failures), @Timeout (cap how long a call may block), @CircuitBreaker (stop calling a failing dependency to prevent cascading failure), @Bulkhead (limit concurrent calls so one slow dependency can't exhaust threads) and @Fallback (a graceful alternative when the operation ultimately fails). They compose on a single method.

Why it matters: these are the Netflix-Hystrix patterns, standardised and interceptor-driven — the counterpart to Spring's Resilience4j integration. Being able to name all five and explain why a circuit breaker differs from a retry is a very common senior-round question in India, especially for product companies building distributed systems.

@Retry(maxRetries = 3)
@Timeout(2000)                     // milliseconds
@CircuitBreaker(requestVolumeThreshold = 8, failureRatio = 0.5)
@Fallback(fallbackMethod = "cachedQuote")
public Quote liveQuote(String symbol) { ... }

public Quote cachedQuote(String symbol) { return cache.get(symbol); }

MicroProfile JWT Authentication

MicroProfile JWT defines how a service validates a bearer JSON Web Token and maps its claims onto the Jakarta EE security model. The token's groups claim becomes caller roles, so standard @RolesAllowed annotations enforce access, and you can inject the verified token (or individual claims) to read the identity. It is designed for stateless, propagated identity across a fleet of microservices.

Why it matters: it is the standards-based route to token security without a session — the Jakarta/MicroProfile counterpart to Spring Security's resource-server / JWT support. Explaining how a role claim drives @RolesAllowed across service boundaries is the usual interview probe.

@Path("/admin")
@RequestScoped
public class AdminResource {
  @Inject JsonWebToken jwt;              // the verified caller token

  @GET @RolesAllowed("admin")           // driven by the token's "groups" claim
  public String dashboard() { return "hello " + jwt.getName(); }
}

Cloud-native runtimes: Quarkus, Helidon and Open Liberty

MicroProfile is a specification; you run it on an implementation. Quarkus ("Supersonic Subatomic Java") does most of its CDI wiring and metadata processing at build time and compiles to a GraalVM native image, giving millisecond startup and small memory footprints ideal for containers and serverless. Helidon offers two flavours — Helidon MP (full MicroProfile, declarative) and Helidon SE (a lightweight functional core on virtual threads). Open Liberty is IBM's flexible runtime where you add exactly the MicroProfile and Jakarta EE features a service needs.

Why it matters: portability is real but not total — you still choose a runtime, and the choice is about startup time, memory, native-image support and ecosystem. Being able to place these three against Spring Boot on those axes is the payoff of the whole module.

Jakarta EE + MicroProfile vs Spring Boot for cloud-native

Both stacks build twelve-factor microservices; they differ in philosophy. Spring Boot resolves much of its dependency injection and auto-configuration at runtime via classpath scanning, which is flexible but costs startup time and heap. Quarkus (and Micronaut) shift that work to build time, resolving CDI ahead of time and enabling native compilation — hence the widely-cited native-image startup of roughly 15 milliseconds and far lower RSS memory versus a comparable Spring Boot native image. Spring's advantage is ecosystem breadth and team familiarity.

Why it matters: the honest answer is "it depends" — Spring Boot for large existing Spring estates and the widest library ecosystem; Jakarta EE + MicroProfile (on Quarkus/Helidon/Open Liberty) for vendor-neutral standards, strict resource limits, and serverless or heavily-scaled workloads. Framing the trade-off as build-time-vs-runtime processing, and standard-vs-framework, is what separates a rehearsed answer from an understood one.

Interview questions

MicroProfile and cloud-native Jakarta EE questions asked in Indian interviews — product companies and services firms both probe the Spring Boot contrast hard:

Beginner What is MicroProfile and how does it relate to Jakarta EE?

MicroProfile is an Eclipse Foundation umbrella of specifications that optimise enterprise Java for microservices. It reuses Jakarta EE building blocks (CDI, JAX-RS, JSON-B) and adds cloud-native specs — Config, Health, Metrics/Telemetry, OpenAPI, Rest Client, Fault Tolerance, JWT. Jakarta EE's slim Core Profile exists partly to give MicroProfile a lean foundation. They are complementary umbrellas, not rivals.

Beginner Name the core MicroProfile specifications and what each does.

Config (externalised, ordered configuration), Health (liveness/readiness/startup checks), Metrics and the newer Telemetry (observability), OpenAPI (generated API docs), Rest Client (type-safe HTTP clients), Fault Tolerance (retry/timeout/circuit-breaker/bulkhead/fallback), and JWT (token-based auth). GraphQL, Reactive Messaging and LRA are additional standalone specs.

Beginner What does MicroProfile Config do and how do you read a property?

It merges configuration from ordered sources (system properties, environment variables, config files, custom sources) into one view, highest priority winning. You inject values with @ConfigProperty(name=...) — supporting defaults, Optional and typed conversion — or resolve them via the Config object. It is the portable equivalent of Spring's @Value / @ConfigurationProperties.

Beginner Difference between liveness, readiness and startup health checks?

Liveness answers "is the process broken — should it be restarted?"; readiness answers "can it accept traffic right now?" (fail it and the orchestrator stops routing to the pod); startup answers "has slow initialisation completed?" so liveness probes don't kill a still-booting instance. They map directly to Kubernetes liveness/readiness/startup probes.

Intermediate How does MicroProfile Rest Client differ from the JAX-RS Client API?

The JAX-RS Client is imperative — you build a target, set path/query params and invoke. The Rest Client is declarative: you define a Java interface with JAX-RS annotations plus @RegisterRestClient, and the runtime generates the client, injects the base URL from config, and lets you @Inject @RestClient it. It is the MicroProfile analogue of Spring's @FeignClient/HTTP Interface.

Intermediate Explain the MicroProfile Fault Tolerance annotations.

@Retry reattempts transient failures; @Timeout caps blocking time; @CircuitBreaker opens after a failure threshold to stop hammering a broken dependency and prevent cascading failures; @Bulkhead limits concurrent invocations so one slow call can't exhaust the pool; @Fallback supplies a graceful alternative result. They compose on one method and are interceptor-driven. Same patterns as Hystrix/Resilience4j.

Intermediate How does MicroProfile JWT enforce authorization across services?

The service validates the incoming bearer JWT and maps its groups claim onto caller roles, so standard @RolesAllowed checks apply. You can @Inject JsonWebToken to read the verified identity and claims. Identity is stateless and propagated between services with the token — the standards counterpart to Spring Security acting as a JWT resource server.

Intermediate MicroProfile Metrics vs MicroProfile Telemetry — what changed?

Metrics is the classic spec exposing counters, gauges, timers and histograms on /metrics. Telemetry is the newer, OpenTelemetry-based spec covering traces, metrics and logs together, and the ecosystem is steadily moving toward it as the unified observability standard. Correct answer: Metrics is the traditional choice, Telemetry/OpenTelemetry is the modern direction.

Intermediate How do Quarkus, Helidon and Open Liberty relate to MicroProfile and Jakarta EE?

They are implementations you deploy on. Quarkus does build-time processing and native compilation for fast startup and low memory. Helidon offers MP (full MicroProfile) and SE (a lightweight functional core on virtual threads). Open Liberty lets you add exactly the MicroProfile and Jakarta EE features a service needs. A MicroProfile app written to the spec is portable across all three.

Pro Why does Quarkus start so much faster and use less memory than Spring Boot?

Spring Boot resolves much of its DI and auto-configuration at runtime through classpath scanning and reflection, which costs startup time and heap. Quarkus moves that work to build time — CDI wiring and metadata are computed ahead of time and dead code is trimmed — and it targets GraalVM native images. The result is native startup around 15 ms and much lower RSS versus a comparable Spring Boot native image. The trade-off is a build-time closed-world model with less runtime dynamism.

Pro How do you choose between Jakarta EE + MicroProfile and Spring Boot for a cloud-native system?

Weigh standards vs framework and build-time vs runtime. Choose Jakarta EE + MicroProfile (on Quarkus/Helidon/Open Liberty) for vendor-neutral portability, strict memory/startup budgets, and serverless or heavily-autoscaled workloads. Choose Spring Boot for large existing Spring estates, the widest library ecosystem, and team familiarity that avoids retraining cost. There is no universal winner — the honest answer names the axes and the context.

Pro How is API documentation generated in MicroProfile, and how does it compare with Spring Boot?

MicroProfile OpenAPI scans JAX-RS endpoints and emits an OpenAPI v3 document at /openapi, enriched by annotations (@Operation, @Schema, @APIResponse) or a supplied base document, with Swagger UI usually wired in. It is a portable specification with implementations in every runtime, whereas Spring Boot typically relies on the third-party springdoc-openapi library. Same OpenAPI output; standard vs library is the distinction.
