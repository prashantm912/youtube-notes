Module: JAX-RS Advanced

Once the resource classes answer requests, the interesting problems begin: custom serializers, one place to turn exceptions into clean responses, cross-cutting filters, a client for calling other services, and streaming with SSE — the layer that turns a REST endpoint into production-grade plumbing.

Knowledge Library → Courses → Jakarta EE → JAX-RS Advanced

What this module covers

JAX-RS Advanced is the extension layer of the specification: providers that plug into serialization and error handling, filters and interceptors for cross-cutting concerns, and the client side — the Client API, asynchronous and reactive invocation, and Server-Sent Events. This is where a working REST resource becomes a real service that serializes custom media types, returns consistent errors, calls other services and streams updates. Because most Indian teams run Spring alongside Jakarta EE, the filter and client topics note the Spring equivalents an interviewer will ask you to compare.

Topics

Entity providers: MessageBodyReader and MessageBodyWriter

JAX-RS turns Java objects into bytes and back through entity providers. A MessageBodyWriter<T> serializes a return value into the response stream for a given media type; a MessageBodyReader<T> deserializes a request body into a method parameter. Built-in providers already handle String, byte[], and JSON-B/JAXB types; you register your own @Provider when you need a custom media type or wire format. Its isWriteable/isReadable and getSize methods let it opt in only for the types and media types it understands.

Why it matters: this is the extension point behind @Produces/@Consumes. Interviewers probe whether you understand that JSON binding is "just a provider" and that the provider participates in content negotiation rather than being hard-wired.

@Provider
@Produces("application/vnd.acme.book+json")
public class BookWriter implements MessageBodyWriter<Book> {
    public boolean isWriteable(Class<?> t, Type g, Annotation[] a, MediaType m) {
        return Book.class.isAssignableFrom(t);
    }
    public void writeTo(Book b, /* ... */ OutputStream out) throws IOException {
        out.write(toJson(b).getBytes(UTF_8));   // custom media-type serialization
    }
}

Mapping exceptions: ExceptionMapper

An ExceptionMapper<E extends Throwable> catches an exception thrown anywhere in the request pipeline — resource method, provider or filter — and converts it into a Response. It is the clean alternative to a try/catch in every method or throwing a raw WebApplicationException. Register it as a @Provider; the runtime selects the mapper whose exception type is the nearest supertype of what was thrown, so specific mappers win over general ones.

Why it matters: a consistent error payload — a JSON body with the right HTTP status — is a design decision, not an accident. Knowing that a mapper centralizes error-to-HTTP translation, and that WebApplicationException already carries its own Response, is standard interview ground.

@Provider
public class NotFoundMapper implements ExceptionMapper<EntityNotFoundException> {
    public Response toResponse(EntityNotFoundException ex) {
        return Response.status(Response.Status.NOT_FOUND)
                       .entity(new ApiError("NOT_FOUND", ex.getMessage()))
                       .type(MediaType.APPLICATION_JSON)
                       .build();
    }
}

Filters and interceptors

Filters run around the request/response and can read or mutate headers, the URI or the status, or abort early: ContainerRequestFilter/ContainerResponseFilter on the server, ClientRequestFilter/ClientResponseFilter on the client. Interceptors — ReaderInterceptor and WriterInterceptor — wrap the entity marshalling step, sitting around the MessageBodyReader/Writer to transform the stream itself (gzip, encryption). Bind them globally, or scope them to specific methods with a custom @NameBinding annotation, and order them with @Priority.

Why it matters: cross-cutting concerns — authentication, logging, CORS, compression — belong here instead of polluting every resource. The filter-versus-interceptor distinction and the request/response execution order are among the most-asked advanced JAX-RS questions, and map onto Spring's HandlerInterceptor/servlet filters in the contrast rounds.

@Provider
@Priority(Priorities.AUTHENTICATION)   // ordering within the request phase
public class AuthFilter implements ContainerRequestFilter {
    public void filter(ContainerRequestContext ctx) {
        if (ctx.getHeaderString("Authorization") == null)
            ctx.abortWith(Response.status(Response.Status.UNAUTHORIZED).build());
    }
}

The JAX-RS Client API

The Client API is the fluent, portable counterpart to the server annotations: build a Client, target a URI with WebTarget, set path and query templates and headers, then invoke request().get(...)/post(...). It reuses the same providers, so your JSON binding, filters and interceptors work on the client side too. A Client is expensive to create — build it once, reuse it, and close it when done.

Why it matters: microservices call each other, and this is how one JAX-RS service consumes another. Interviewers compare it with Spring's RestTemplate (legacy) and WebClient (reactive), and check that you reuse and close clients rather than creating one per call.

Client client = ClientBuilder.newClient();
try {
    Book book = client.target("https://api.example.com")
        .path("books/{id}").resolveTemplate("id", 42)
        .request(MediaType.APPLICATION_JSON)
        .get(Book.class);               // deserialized by a MessageBodyReader
} finally {
    client.close();                     // Client is heavyweight — reuse and close
}

Asynchronous and reactive JAX-RS

By default a resource method holds its request thread until it returns. Suspending with @Suspended AsyncResponse hands control back to the container and resumes later from another thread — raising throughput for slow, I/O-bound work without pinning the limited request pool. On the client, async() returns a Future (or takes an InvocationCallback) and the reactive rx() returns a CompletionStage, so calls compose without blocking the caller.

Why it matters: thread-per-request scaling is a real ceiling, and async is the JAX-RS answer before you reach for a fully reactive stack. The key insight interviewers want: @Suspended does not make the work faster — it frees the thread while the work runs elsewhere.

@GET @Path("report")
public void report(@Suspended AsyncResponse ar) {   // frees the request thread
    executor.submit(() -> ar.resume(buildReport()));
}
// client side, non-blocking:
CompletionStage<Book> cs = target.request().rx().get(Book.class);

Server-Sent Events (SSE)

SSE, standardized in JAX-RS 2.1, is a one-way, long-lived HTTP stream from server to client over text/event-stream — lighter than WebSocket when you only need to push (notifications, live metrics, progress bars). A resource produces SERVER_SENT_EVENTS and writes to an injected SseEventSink; an SseBroadcaster fans a single event out to many subscribers; clients consume with SseEventSource, which reconnects automatically and can resume from the last event id.

Why it matters: SSE answers "how do I push updates without polling or a full WebSocket?" Knowing that it is unidirectional and self-reconnecting — versus WebSocket's bidirectional, protocol-upgraded channel — is the usual discriminator in the streaming question.

@GET @Path("events")
@Produces(MediaType.SERVER_SENT_EVENTS)
public void subscribe(@Context SseEventSink sink, @Context Sse sse) {
    broadcaster.register(sink);                 // fan-out to many clients
    sink.send(sse.newEvent("connected"));
}

Interview questions

The most-asked advanced JAX-RS questions in Indian enterprise and product interviews — providers, the filter/interceptor chain, the client, async and SSE, with the Spring contrast that follows almost every time:

Beginner What is a JAX-RS "provider", and name the main kinds.

A provider is an extension component, annotated @Provider, that the runtime discovers and plugs into the request pipeline. The main kinds: entity providers (MessageBodyReader/MessageBodyWriter), ExceptionMapper, ContextResolver, and the filters and interceptors. They customize serialization, error handling and cross-cutting behavior without touching the resource classes.

Beginner What do MessageBodyReader and MessageBodyWriter do?

They convert between Java objects and the wire. A MessageBodyWriter serializes a return value into the response entity stream for a media type; a MessageBodyReader deserializes a request body into a method parameter. Built-ins cover String, byte[] and JSON; a custom one, registered as a @Provider, handles your own formats. @Produces/@Consumes select which provider runs.

Beginner What is an ExceptionMapper and why use one?

An ExceptionMapper<E> converts a thrown exception into a Response, centralizing error-to-HTTP mapping instead of a try/catch in every method. Register it as a @Provider; the runtime picks the mapper whose type is the closest supertype of the exception. The payoff is consistent status codes and a uniform error body across the API.

Beginner How do you call a remote REST endpoint with the JAX-RS Client API?

ClientBuilder.newClient() → target(uri) → path/queryParam/resolveTemplate → request(mediaType) → get/post(...). The response is read into a type by the same providers used server-side. Because a Client is heavyweight, create it once, reuse it, and close() it.

Intermediate Filters vs interceptors — what is the difference?

Filters (ContainerRequestFilter/ContainerResponseFilter, and the client variants) act on the message metadata — headers, URI, status — and can abort the request. Interceptors (ReaderInterceptor/WriterInterceptor) wrap the entity marshalling, so they transform the body stream itself (gzip, encryption). In short: filters wrap the message, interceptors wrap the entity I/O.

Intermediate What is name binding, and how does it differ from a global filter?

A global filter or interceptor runs for every resource method. Name binding attaches one only to methods or classes marked with a custom annotation that is itself meta-annotated @NameBinding — for example a @Logged filter applied wherever you put @Logged. Use it to scope cross-cutting logic instead of running it everywhere. Dynamic binding via DynamicFeature is the programmatic alternative.

Intermediate How do you control the execution order of filters and interceptors?

Annotate them with @Priority using the Priorities constants (AUTHENTICATION = 1000, AUTHORIZATION = 2000, HEADER_DECORATOR, ENTITY_CODER, USER). Request-side filters run in ascending priority order, response-side in descending; a lower number means earlier on the request path. Assigning explicit priorities beats relying on registration order.

Intermediate What is a @PreMatching filter and when would you use it?

A @PreMatching ContainerRequestFilter runs before the runtime matches the request to a resource method, so it can change the HTTP method, URI or headers and thereby influence which resource is selected — think method override or URI rewriting. Ordinary (post-matching) filters run after matching, when the target resource is already known.

Intermediate How do you make a JAX-RS resource method asynchronous, and what does it buy you?

Inject @Suspended AsyncResponse, return void, and call resume() from another thread (often a CompletableFuture or executor). The container thread is released immediately, so slow I/O-bound requests do not tie up the limited request pool, raising throughput. It does not make the work itself faster — it only frees the thread while the work runs elsewhere.

Intermediate JAX-RS Client vs Spring — how do you contrast them?

JAX-RS is a portable spec (Jersey/RESTEasy implement it); Spring MVC/WebFlux is Spring's own stack. Server-side: @Path/@GET versus @RequestMapping/@GetMapping. Client-side: the JAX-RS Client/WebTarget versus Spring's RestTemplate (legacy) and WebClient (reactive), plus declarative HTTP interfaces. Same REST concepts, different APIs, and JAX-RS resources default to per-request instances while Spring beans are singletons.

Pro Walk through the server pipeline order, including where an ExceptionMapper fits.

Pre-matching filters → resource matching → post-matching request filters → ReaderInterceptor → MessageBodyReader → the resource method → response filters → WriterInterceptor → MessageBodyWriter. If anything throws, control jumps to the matching ExceptionMapper; the response filters and writer interceptor then still run over the mapped Response. Priorities order the filters and interceptors within each phase.

Pro async() vs reactive rx() on the client — what is the difference?

async() returns a Future<T> (or takes an InvocationCallback): you still poll, block, or handle callbacks. rx() returns a CompletionStage<T>, so responses compose with thenApply/thenCompose and combine without blocking. Both avoid blocking the caller, but the reactive extension (rx(), JAX-RS 2.1) composes cleanly, which is why it is preferred for chaining calls.

Pro SSE vs WebSocket — when do you choose SSE in JAX-RS?

SSE is one-way (server → client), rides plain HTTP text/event-stream, and auto-reconnects with event ids — ideal for notifications, live feeds and progress where only the server pushes. WebSocket is bidirectional and a separate protocol upgrade. In JAX-RS you produce SERVER_SENT_EVENTS, write to an SseEventSink, fan out with SseBroadcaster, and consume via SseEventSource. Choose SSE when you do not need client-to-server streaming and want to stay on ordinary HTTP.
