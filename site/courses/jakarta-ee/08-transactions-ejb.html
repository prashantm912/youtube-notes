Module: Transactions (JTA) and Enterprise Beans

A transaction is the enterprise "all or nothing" guarantee; enterprise beans are where the container wires that guarantee in for you, so a business method commits or rolls back cleanly without a line of begin/commit plumbing of its own.

Knowledge Library → Courses → Jakarta EE → Transactions (JTA) and Enterprise Beans

What this module covers

Jakarta Transactions (JTA) is the standard for demarcating and coordinating transactions across the Jakarta EE stack — a single unit of work that either fully commits or fully rolls back, even when it spans two databases or a database and a message queue. Enterprise Beans are the server-side components where the container applies that transaction management (plus concurrency, security and pooling) declaratively, in three flavours: stateless, stateful and singleton. Because most Indian teams work in Spring, the module sets each idea against its Spring equivalent — the two @Transactional annotations, EJB beans versus Spring beans — since that contrast is exactly what interviews probe.

Topics

Jakarta Transactions (JTA): the transaction model

JTA is an API, not an engine: it defines the interfaces a transaction manager exposes so an application can mark transaction boundaries and so multiple XA-aware resources can be coordinated under one global transaction using two-phase commit. The actual manager is supplied by the runtime (WildFly, Payara, Open Liberty) or a standalone library such as Narayana or Atomikos. Its guarantees are the classic ACID properties — atomicity, consistency, isolation, durability — the reason a money transfer that debits one account and credits another can never leave the system half-done.

Why it matters: JTA underpins both enterprise beans and JPA — the persistence context flushes and commits on the JTA transaction boundary. Knowing that JTA is the specification while the manager is the implementation, and that a global (XA) transaction differs from a plain local resource transaction, is foundational for every question in this module.

Container-managed vs bean-managed transactions

With container-managed transactions (CMT — the default), the container opens a transaction as a business method is entered and commits or rolls back as it returns. You tune the scope with @TransactionAttribute, whose six values decide what happens relative to any transaction already in progress: REQUIRED, REQUIRES_NEW, MANDATORY, SUPPORTS, NOT_SUPPORTED, NEVER. To abort without throwing, call setRollbackOnly(). The same declarative style reaches CDI-managed beans through jakarta.transaction.Transactional and its matching TxType values, with rollbackOn / dontRollbackOn to override defaults.

With bean-managed transactions (BMT), you opt out of the container's demarcation via @TransactionManagement(BEAN) and drive UserTransaction yourself — begin, commit, rollback. It buys fine-grained control at the cost of more code and the risk of leaking an open transaction. The single most bug-prone default to remember: a container transaction rolls back automatically on unchecked (system) exceptions, but not on checked (application) exceptions unless you mark them with @ApplicationException(rollback = true) or list them in rollbackOn.

@Stateless
public class OrderService {
  @TransactionAttribute(TransactionAttributeType.REQUIRES_NEW)
  public void audit(Event e) { /* its own transaction, always */ }
}

@TransactionManagement(TransactionManagementType.BEAN)   // opt into BMT
public class BatchJob {
  @Resource UserTransaction utx;
  public void run() throws Exception {
    utx.begin();
    try { doWork(); utx.commit(); }
    catch (Exception ex) { utx.rollback(); throw ex; }
  }
}

Enterprise Beans and the three session bean types

An enterprise bean is a server-side component that holds business logic and runs inside the EJB container, which supplies transactions, security, concurrency, instance pooling and remoting as declarative, cross-cutting services — you annotate, the container implements. Session beans come in three types: stateless, stateful and singleton. (Message-driven beans, the fourth kind, consume JMS and belong to the Messaging module.) A plain annotation such as @Stateless is enough to turn a class into a managed, transactional component.

Why it matters: the container's declarative services are the entire value proposition of enterprise beans, and choosing the right session-bean type for a given job — a request-scoped calculation, a per-user conversation, an application-wide cache — is a classic screening question that quickly separates people who have shipped Jakarta EE from those who have only read about it.

@Stateless           public class TaxCalculator { }  // pooled; no client state kept
@Stateful            public class ShoppingCart { }   // one instance bound to one client
@Singleton @Startup  public class ConfigCache { }    // exactly one per application

Stateless vs stateful vs singleton — choosing and concurrency

A stateless bean keeps no conversational state between calls, so the container pools instances and any instance can serve any client — the highest-throughput choice, and the right default for service facades and JAX-RS endpoints. A stateful bean holds conversational state tied to one specific client across several calls (a shopping cart, a multi-step wizard); the container may passivate and later activate it, and it is removed on an @Remove method or session timeout. A singleton has exactly one instance for the whole application, ideal for caches, configuration and shared read-mostly state, with @Startup for eager initialisation.

Concurrency is where the singleton earns its own question: by default it uses container-managed concurrency, where @Lock(WRITE) takes an exclusive lock and @Lock(READ) allows shared access, and switching to @ConcurrencyManagement(BEAN) hands you responsibility for thread safety. Stateless and stateful beans never see concurrent calls on a single instance, so they need no such locking.

Enterprise Beans vs Spring beans and @Transactional

Both stacks offer declarative transactions, but the mechanism differs. EJB container-managed transactions are bound to JTA and an application-server environment; Spring's @Transactional works on any Spring bean through AOP proxies backed by a PlatformTransactionManager that can be a local JPA or DataSource manager or a JTA one. Spring exposes more knobs at the annotation itself — isolation, read-only, timeout, a NESTED propagation mode and rollbackFor — whereas the EJB model keeps those concerns closer to the container and the resource.

The naming trap worth internalising: there are two @Transactional annotations — jakarta.transaction.Transactional (Jakarta EE, on CDI and EJB beans) and org.springframework.transaction.annotation.Transactional. Spring understands both, but neither works outside its own container. Because both mechanisms rely on proxying, a plain self-invocation — one method in a bean calling another annotated method on this — bypasses the interceptor and silently runs without the expected new transaction. That gotcha, and the EJB-versus-Spring comparison generally, are heavily favoured in Indian interviews where migration between the two is common.

// Jakarta EE — on a CDI or EJB bean
@jakarta.transaction.Transactional(Transactional.TxType.REQUIRED)
public void placeOrder(Order o) { }

// Spring — on a Spring bean, richer tuning
@org.springframework.transaction.annotation.Transactional(
    readOnly = true, timeout = 30, rollbackFor = Exception.class)
public Report build() { return report; }

Interview questions

The questions most likely to come up in Indian interviews — session-bean choice and the Jakarta-versus-Spring transaction contrast dominate, so the hardest of those sit last:

Beginner What is JTA and what problem does it solve?

Jakarta Transactions is the standard API for demarcating and coordinating transactions so a unit of work is atomic — it fully commits or fully rolls back. Its real strength is coordinating multiple XA resources (two databases, or a database and a message queue) under one global transaction via two-phase commit. JTA is the specification; the transaction manager (in the app server, or Narayana/Atomikos) is the implementation.

Beginner What are the two ways to demarcate a transaction in Jakarta EE?

Declarative — annotate with @Transactional or @TransactionAttribute and let the container start and end the transaction (container-managed). Or programmatic — inject UserTransaction and call begin/commit/rollback yourself (bean-managed). Declarative is the default and the norm; programmatic is for fine-grained control.

Beginner What are the three types of session bean?

Stateless (no client state, pooled), stateful (conversational state tied to one client), and singleton (one instance per application). Message-driven beans are a separate kind used for JMS, not a session bean.

Beginner Container-managed vs bean-managed transactions — the difference?

In CMT the container sets the boundaries around each business method and you only tune scope with @TransactionAttribute. In BMT you take over via @TransactionManagement(BEAN) and drive UserTransaction explicitly. CMT is less code and the default; BMT gives precise control at the cost of managing commit/rollback yourself.

Beginner What does @TransactionAttribute(REQUIRES_NEW) do?

It always runs the method in a brand-new transaction. If the caller already has one, the container suspends it, runs the method in a fresh transaction that commits or rolls back independently, then resumes the caller's transaction. The canonical use is an audit or log write that must persist even if the surrounding business transaction later rolls back.

Intermediate Stateless vs stateful session bean — when do you use each?

Stateless keeps no state between calls, so the container pools instances and any instance serves any client — best throughput, ideal for service facades and REST endpoints. Stateful keeps conversational state tied to one client across calls (a cart, a wizard) and may be passivated/activated. Rule of thumb: default to stateless; reach for stateful only when you genuinely need per-client memory across method calls.

Intermediate What is a singleton session bean and how is its concurrency handled?

One instance for the entire application — good for caches, configuration and shared state, with @Startup for eager creation. By default it uses container-managed concurrency: @Lock(WRITE) gives exclusive access, @Lock(READ) allows shared reads. Switch to @ConcurrencyManagement(BEAN) to take responsibility for thread safety yourself.

Intermediate Does a container-managed transaction roll back on a checked exception by default?

No. The container rolls back automatically on unchecked (system/runtime) exceptions, but a checked (application) exception commits by default. To force rollback on a checked exception, annotate it with @ApplicationException(rollback = true), list it in @Transactional(rollbackOn = ...), or call setRollbackOnly(). This default trips up almost everyone at least once.

Intermediate Explain the six transaction attributes.

REQUIRED joins the caller's transaction or starts one; REQUIRES_NEW always starts its own, suspending any caller transaction; MANDATORY demands the caller already has one (else exception); SUPPORTS joins one if present, runs without otherwise; NOT_SUPPORTED suspends any transaction and runs with none; NEVER requires no transaction and throws if one exists.

Intermediate jakarta.transaction.Transactional vs Spring's @Transactional?

Same declarative idea, different packages and reach. jakarta.transaction.Transactional applies to CDI/EJB beans in a Jakarta EE container; org.springframework.transaction.annotation.Transactional applies to Spring beans and adds isolation, read-only, timeout, a NESTED propagation and rollbackFor. Spring can honour both annotations, but neither functions outside its own container. Naming them apart is a frequent trick question.

Pro How does declarative transaction management actually work, and why does self-invocation break it?

Both EJB and Spring wrap the bean in a proxy/interceptor that opens a transaction before the call and commits or rolls back after. A call from outside goes through the proxy, so the advice runs. But when one method calls another annotated method on this, the call never leaves the object, bypasses the proxy, and the second method runs in the caller's context with no new transaction — the fix is to call through an injected reference or split the methods across beans.

Pro An audit record must persist even when the business transaction rolls back. How?

Put the audit write in its own method annotated REQUIRES_NEW (or on a separate bean it calls through). That suspends the caller's transaction, commits the audit independently, then resumes the caller — so a later rollback of the business work leaves the audit row intact. Watch for the self-invocation trap: the audit method must be reached through the proxy, not a direct this call.

Pro Contrast EJB container-managed transactions with Spring for distributed transactions and testability.

EJB CMT is tied to JTA and a full application server, which makes genuine XA distributed transactions native but couples your code to that runtime and complicates lightweight unit testing. Spring abstracts behind a PlatformTransactionManager: a local JPA/DataSource manager for the common single-resource case, or a JTA manager (app-server or standalone like Atomikos/Narayana) when you truly need XA — and it runs in a plain container, so beans are easy to test in isolation. The practical guidance: use JTA when you must span multiple transactional resources; prefer a local manager for the far more common single-database case.
