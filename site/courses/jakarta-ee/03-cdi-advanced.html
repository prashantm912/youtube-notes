Module: CDI: Producers, Events and Interceptors

The loose-coupling half of CDI: how @Produces supplies objects the container cannot build itself, how beans talk through fired events instead of direct calls, and how interceptors and decorators layer behaviour without editing the business code — with the Spring AOP contrast interviewers reach for.

Knowledge Library → Courses → Jakarta EE → CDI: Producers, Events and Interceptors

What this module covers

Module 02 covered beans, injection and scopes; this one covers everything CDI adds on top to keep components decoupled. Producers and disposers hand the container objects it could not instantiate on its own; alternatives swap implementations per deployment; the event model lets beans notify each other with no compile-time link; and interceptors and decorators bolt cross-cutting and business behaviour onto beans from the outside. The @PostConstruct and @PreDestroy callbacks round out the container-managed lifecycle. Where each mechanism has a Spring equivalent — events vs ApplicationEvent, interceptors vs AOP — the contrast is called out, because Indian enterprise and product interviews lean on it.

Topics

Producer methods, fields and disposers

Not everything you inject is a class CDI can construct: third-party types with no bean-defining annotation, objects that need runtime configuration, or JDK resources like an EntityManager. A method annotated @Produces becomes a factory whose return value is injectable, with its own scope and qualifiers; a producer field is the one-line form when the value is a plain constant or resource. When the produced object owns a resource that must be released, a matching @Disposes parameter defines the cleanup, which the container calls when the producing context ends.

Why it matters: producers are how CDI bridges to the non-CDI world, and the producer plus disposer pair is the standard pattern for connection- and manager-style resources. Interviewers use it to check that you understand who owns an object's lifecycle.

@Produces @RequestScoped
EntityManager em(EntityManagerFactory emf) {   // factory for a non-bean type
    return emf.createEntityManager();
}
void close(@Disposes EntityManager em) {        // matched by type + qualifiers
    em.close();                                 // called when the context ends
}

Alternatives: selecting an implementation

When two beans implement the same type, @Alternative marks one as a swappable substitute that is disabled by default. You turn it on for a particular deployment either by listing it in that archive's beans.xml, or — the modern, portable way — by adding @Priority, which enables it application-wide and, when several alternatives match, lets the highest priority win. This is how you drop in a mock or a stub for testing, or a region-specific implementation, without touching any injection point. The related @Specializes goes further: it fully replaces and inherits another bean's types and qualifiers.

Why it matters: alternatives are CDI's answer to "how do I change wiring per environment without changing code" — the same problem Spring solves with profiles and conditional beans.

CDI events: fire and observe

Events are CDI's built-in publish/subscribe: one bean injects Event<T> and calls fire(), and any bean with an @Observes T method is notified — with no compile-time dependency between them. Qualifiers narrow which observers respond. Firing is synchronous by default (observers run in the caller's thread and can take part in the current transaction); fireAsync() with @ObservesAsync runs observers on another thread and returns a CompletionStage collecting any failures. Transactional observers add a phase — for example @Observes(during = AFTER_SUCCESS) defers the callback until the JTA transaction commits, so a confirmation email never goes out on a rollback.

Why it matters: events are the standard way to decouple side effects (auditing, notifications) from core logic, and the sync-vs-async and transaction-phase distinctions are common follow-up questions. The Spring analogue is ApplicationEventPublisher plus @EventListener.

@Inject Event<OrderPlaced> events;        // producer side
events.fire(new OrderPlaced(order));       // synchronous, same thread
events.fireAsync(new OrderPlaced(order));  // handled on another thread

void onOrder(@Observes OrderPlaced e) { /* ... */ }        // synchronous observer
void email(@ObservesAsync OrderPlaced e) { /* ... */ }     // async observer

Interceptors and the Spring AOP contrast

An interceptor factors out a technical, cross-cutting concern — logging, timing, security, transactions — so it lives in one place instead of every method. You define an @InterceptorBinding annotation, write an @Interceptor class with an @AroundInvoke method that receives an InvocationContext and calls ctx.proceed() to continue the chain, then put the binding annotation on the beans or methods to advise. Interceptors are off until enabled, ordered by @Priority (lower runs first, outermost) or listed in beans.xml. Beyond business methods, interceptors can also target lifecycle callbacks with @AroundConstruct, @PostConstruct and @PreDestroy.

Why it matters: this is CDI's AOP, and the comparison to Spring is a favourite interview line. Spring AOP is proxy-based and selects join points with expressive pointcut expressions in a decoupled aspect; CDI uses typesafe binding annotations applied directly on the target — more discoverable and compile-checked, but with less flexible selection.

@InterceptorBinding
@Retention(RUNTIME) @Target({TYPE, METHOD})
public @interface Logged {}

@Logged @Interceptor @Priority(Interceptor.Priority.APPLICATION)
public class LoggingInterceptor {
    @AroundInvoke
    Object log(InvocationContext ctx) throws Exception {
        // pre-processing here
        return ctx.proceed();              // invoke the next interceptor / target
    }
}

Decorators

A decorator is the object-oriented twin of an interceptor: instead of a generic technical concern, it adds or modifies business logic for one specific interface. It implements the bean's interface and holds a @Delegate injection point of that same type, so it can call the original and wrap its result — a discount layer over a price calculator, say. Decorators are disabled by default and enabled with @Priority or in beans.xml. In the invocation order, interceptors always run before decorators.

Why it matters: interviewers ask "interceptor or decorator?" to see whether you separate technical concerns (broad, binding-driven) from business concerns (interface-specific, delegate-driven). Knowing that @Delegate is what makes a decorator, and that interceptors fire first, is the crisp answer.

@Decorator @Priority(Interceptor.Priority.APPLICATION)
abstract class DiscountDecorator implements PriceCalculator {
    @Inject @Delegate @Any PriceCalculator delegate;   // the decorated bean
    public BigDecimal total(Cart c) {
        return delegate.total(c).multiply(new BigDecimal("0.9"));
    }
}

Lifecycle callbacks: @PostConstruct and @PreDestroy

These two annotations mark methods the container runs around a bean's managed lifetime. @PostConstruct runs once after injection has completed but before the bean is put into service — the right place for initialization that depends on injected collaborators, which a constructor cannot see yet. @PreDestroy runs before the bean is removed from its context, to release resources it holds. Both methods take no arguments (except on interceptor classes) and cannot be static. They are defined by the Jakarta Annotations spec and honoured across CDI beans, session beans and other container-managed objects.

Why it matters: candidates routinely put startup logic in constructors and find their injected fields still null. Naming @PostConstruct as the fix — and knowing why the constructor runs too early — is the expected answer.

Interview questions

CDI's advanced features are a staple of Jakarta EE and enterprise-Java rounds in India, especially where teams run WildFly, Payara or Open Liberty and interviewers probe the Spring contrast:

Beginner What is a producer method and when do you need one?

A method annotated @Produces whose return value becomes injectable. You need it when CDI cannot instantiate the type itself — a third-party class with no bean-defining annotation, an object requiring runtime configuration, or a JDK resource such as an EntityManager. The producer can declare its own scope and qualifiers.

Beginner Producer field vs producer method?

Both use @Produces. A producer field is the shorthand when the injectable value is a simple field or resource with no logic; a producer method is used when construction needs parameters or code. The field form is common for declaring resources.

Beginner What is a disposer method and how is it matched to a producer?

A method with a @Disposes parameter whose type and qualifiers match a producer; CDI calls it to clean up the produced object when the producing context ends — for example closing an EntityManager or a JDBC connection. It pairs cleanup with the producer that created the resource.

Beginner What does @Alternative do and how do you enable it?

It marks a bean as a substitute implementation that is disabled by default. Enable it per bean-archive by listing it in that archive's beans.xml, or application-wide with @Priority. It is the standard mechanism for test doubles and environment-specific implementations, swapped without editing injection points.

Intermediate How do CDI events decouple beans?

The producer injects Event<T> and calls fire(); observers declare a method with an @Observes T parameter. Neither references the other at compile time — the container routes the event, and qualifiers narrow which observers are notified. It is CDI's built-in publish/subscribe.

Intermediate Synchronous vs asynchronous events?

fire() with @Observes runs observers synchronously in the caller's thread and can participate in the caller's transaction. fireAsync() with @ObservesAsync runs them on another thread, does not block the producer, and returns a CompletionStage that collects any observer exceptions.

Intermediate What is a transactional observer and why use one?

An observer that specifies a JTA phase, e.g. @Observes(during = AFTER_SUCCESS). The container defers notification to that phase of the current transaction, so a side effect like sending a confirmation email fires only after commit and is skipped on rollback. Other phases include BEFORE_COMPLETION and AFTER_FAILURE.

Intermediate How do you write and enable an interceptor?

Define an @InterceptorBinding annotation, write an @Interceptor class with an @AroundInvoke method that takes an InvocationContext and calls ctx.proceed(), then place the binding on the target beans or methods. Enable it with @Priority (application-wide, ordered) or in beans.xml — interceptors are disabled by default.

Intermediate Interceptor vs decorator — when do you use each?

Interceptors handle technical cross-cutting concerns (logging, security, transactions) applied broadly through binding annotations. Decorators add business logic for one specific interface through a @Delegate injection point that wraps the original bean. When both apply, interceptors run before decorators.

Intermediate What do @PostConstruct and @PreDestroy do, and why not use the constructor?

@PostConstruct runs once after injection completes and before the bean serves requests — so injected fields are available, which they are not in the constructor. @PreDestroy runs before the bean leaves its context, to release resources. Both are no-arg, non-static methods defined by Jakarta Annotations.

Pro Compare CDI interceptors with Spring AOP.

Both implement aspect-oriented cross-cutting behaviour. CDI uses typesafe interceptor binding annotations placed on the target, which is discoverable and compile-checked but offers no expression language for selecting methods. Spring AOP is proxy-based, keeps the aspect decoupled from targets, and selects join points with flexible pointcut expressions (for example execution(...)), optionally weaving via AspectJ. CDI trades selection flexibility for typesafety and being part of the standard.

Pro How is invocation order controlled across interceptors and decorators?

@Priority sets ordering: a lower value runs earlier and therefore outermost in the chain. Interceptors as a group always execute before decorators. Enabling with @Priority is application-wide and ordered, whereas listing in beans.xml is scoped to that bean-archive with no global ordering guarantee against other archives.

Pro @Alternative via @Priority vs via beans.xml — what is the difference?

beans.xml enables an alternative only within its own bean-archive, so different modules can select different implementations, with no cross-archive ordering. @Priority enables it for the whole application and, when several alternatives are eligible, the one with the highest priority is selected. @Priority is the modern, portable choice; @Specializes is the stronger option when you want to inherit and fully replace another bean.
