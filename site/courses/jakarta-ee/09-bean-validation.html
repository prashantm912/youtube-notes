Module: Jakarta Bean Validation

Instead of scattering null-checks and range-checks through every service method, you declare the rule once as an annotation on the field — and the same jakarta.validation model that ships with Jakarta EE is exactly what Spring Boot bootstraps under the hood.

Knowledge Library → Courses → Jakarta EE → Jakarta Bean Validation

What this module covers

Jakarta Bean Validation (the specification lives in the jakarta.validation namespace) is a declarative constraint model: you annotate a class's fields, getters, method parameters or return values with rules such as @NotBlank or @Size, and a Validator — or the container, at a well-defined boundary — enforces them. This module walks the built-in constraints and the @Valid trigger, then cascading and validation groups, then custom constraints written with a ConstraintValidator, and finally method validation and how it plugs into JAX-RS and CDI. The spec is Jakarta-native, but it is the very same API Spring reuses (Spring Boot auto-configures Hibernate Validator, the reference implementation), so the two hardest interview topics here — a custom ConstraintValidator and validation groups — pay off on both stacks.

Topics

Constraint annotations and triggering validation with @Valid

The heart of the spec is a set of built-in constraint annotations in jakarta.validation.constraints — @NotNull, @NotEmpty, @NotBlank, @Size, @Min/@Max, @Positive, @Email, @Pattern, @Past/@Future and more — placed directly on the data they govern. Declaring a constraint does nothing on its own; validation runs when you call Validator.validate() (which returns a Set<ConstraintViolation>, empty when valid) or when a runtime marks a value with @Valid and validates it automatically at a boundary such as a JAX-RS resource method, a Faces form, or a Spring MVC controller.

Why it matters: the rules become a single source of truth carried by the object itself rather than re-implemented in every layer, and the @NotNull / @NotEmpty / @NotBlank distinction — null-only, null-or-size-zero, and null-or-whitespace-only respectively — is one of the most frequent screening questions on this topic.

public class Customer {
  @NotBlank                 // not null AND not blank/whitespace-only
  private String name;
  @NotNull @Email
  private String email;
  @Min(18) @Max(120)
  private int age;
}
// nothing runs until you validate:
Set<ConstraintViolation<Customer>> violations = validator.validate(customer);

Cascading validation and validation groups

Real objects are graphs, so annotating a nested field with @Valid makes validation cascade into the referenced object — and into every element of an annotated collection or map — instead of stopping at the top level. Validation groups solve the other real problem: sometimes only a subset of a bean's constraints should apply at a given moment (a two-step form, create versus update). Each constraint carries a groups attribute naming marker interfaces, and you validate one group at a time; @GroupSequence orders groups so a later, more expensive group is skipped once an earlier one has already failed, and @ConvertGroup translates a group as validation cascades.

Why it matters: groups let one DTO serve several validation scenarios instead of spawning a parallel class per use case, and "explain validation groups and a group sequence" is a classic intermediate interview question — the naive alternative of duplicating beans is exactly the wrong answer it screens for.

public class Order {
  @Valid @NotNull                          // cascade into each Item, and each Item's constraints
  private List<Item> items;
}

public interface PaymentStep {}            // a group is just a marker interface
public class Checkout {
  @NotBlank(groups = PaymentStep.class)
  private String cardNumber;
}
validator.validate(checkout, PaymentStep.class);   // validate only that group

Custom constraints: ConstraintValidator and composition

When no built-in captures a domain rule, you define your own constraint: an annotation meta-annotated with @Constraint(validatedBy = ...) — carrying the mandatory message, groups and payload members — paired with a class implementing ConstraintValidator<A, T> whose isValid method holds the logic. A class-level constraint (validating T as the whole object) is how cross-field rules such as "password equals confirmation" are expressed, since a single field can't see its sibling. Constraint composition is the lighter-weight route: annotate a new constraint with existing ones to bundle, say, @NotNull @Size @Pattern under one reusable @ValidUsername. Because validators are CDI-managed beans, they can @Inject collaborators (a repository for a uniqueness check, for instance).

Why it matters: writing a custom ConstraintValidator from scratch is among the most-asked practical exercises in Indian Java interviews, and knowing that the null case is conventionally treated as valid (leave null-ness to @NotNull) is the detail that separates a rehearsed answer from a real one.

@Constraint(validatedBy = UppercaseValidator.class)
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Uppercase {
  String message() default "must be uppercase";
  Class<?>[] groups() default {};
  Class<? extends Payload>[] payload() default {};
}

public class UppercaseValidator implements ConstraintValidator<Uppercase, String> {
  public boolean isValid(String v, ConstraintValidatorContext ctx) {
    return v == null || v.equals(v.toUpperCase());   // null is left to @NotNull
  }
}

Method validation and integration with JAX-RS/CDI

Beyond validating an object's state, the spec supports method validation — constraints on method or constructor parameters (preconditions) and on the return value (postconditions), the "programming by contract" style. You annotate parameters directly (@NotNull, @Positive, @Valid to cascade a parameter object) and put a constraint before the method to guard its result; a managed-bean interceptor throws ConstraintViolationException when a rule fails. This is what makes JAX-RS validation automatic: in Jersey or RESTEasy, constraint annotations plus @Valid on a resource method's parameters or entity are checked on every request, a failure on an input mapping to 400 Bad Request. CDI is the glue — managed beans get method validation applied for them, and custom validators are injectable beans.

Why it matters: validating the HTTP request body at the REST boundary is the single most common production use of this spec, and interviewers probe the Spring parallel closely — Spring reaches the same result with @Validated on the bean plus a MethodValidationPostProcessor, and (like EJB) relies on proxying, so a self-invocation quietly skips the check.

@Path("/users")
public class UserResource {
  @POST
  public Response create(@Valid @NotNull User user) { ... }   // invalid body -> 400

  @GET @Path("{id}")
  public @NotNull User find(@PathParam("id") @Positive long id) { ... }
}

Interview questions

The questions most likely to come up in Indian interviews — the custom ConstraintValidator, validation groups, and the Spring-reuse contrast dominate, so the hardest of those sit last:

Beginner What is Jakarta Bean Validation and what problem does it solve?

A specification (package jakarta.validation) for declaring data-integrity rules as annotations on fields, getters, parameters or return values, enforced by a Validator. It replaces null- and range-checks scattered through every layer with one declarative source of truth carried by the object, so the same rules serve persistence, the UI and the REST boundary. Hibernate Validator is the reference implementation.

Beginner @NotNull vs @NotEmpty vs @NotBlank?

@NotNull only rejects null — an empty string passes. @NotEmpty rejects null and size/length zero (empty string, empty collection). @NotBlank is for strings only and rejects null, empty, and whitespace-only. A frequent screening trap: @NotNull on a String does not stop "".

Beginner Declaring a constraint doesn't validate anything — so what actually triggers validation?

Either an explicit call to Validator.validate(obj) (returning a Set<ConstraintViolation>, empty when valid), or a runtime validating a value marked with @Valid at a boundary — a JAX-RS resource method, a Faces form, a Spring MVC controller. The annotation is just metadata until something runs the validator.

Beginner Where can constraint annotations be placed?

On fields, on property getters, on the class itself (class-level, for cross-field rules), on method and constructor parameters, on method/constructor return values, and on type arguments of a generic container (e.g. List<@Email String>). That breadth is what enables both bean validation and method validation from one model.

Beginner What does a ConstraintViolation give you?

Each violation in the returned set describes one failed constraint: the interpolated message, the property path that failed, the invalid value, the root bean, and the constraint descriptor. That is what you turn into a field-by-field error response for an API or form.

Intermediate What does @Valid on a nested object do, and how do you validate a list's elements?

@Valid on a member cascades validation into that referenced object so its own constraints are checked too — without it, only the top-level bean is validated. On a collection field (@Valid List<Item>) it cascades into every element. You can also constrain the elements directly with a type-argument constraint, e.g. List<@NotBlank String>.

Intermediate What are validation groups and when do you use them?

Groups are marker interfaces named in a constraint's groups attribute; validating against a specific group runs only the constraints tagged with it (unassigned constraints belong to Default). Use them when the same bean needs different rules at different times — a multi-step form, or stricter rules on update than create — instead of cloning the class per scenario.

Intermediate What does @GroupSequence do?

It defines an ordered sequence of groups so validation proceeds group by group and stops at the first group that produces a violation — later groups are not evaluated. That lets you run cheap checks first (format, non-null) and skip expensive ones (a database or remote check) when the cheap ones already failed.

Intermediate How do you write a custom constraint?

Two parts: an annotation meta-annotated with @Constraint(validatedBy = MyValidator.class) declaring the mandatory message, groups and payload members; and a class implementing ConstraintValidator<MyAnnotation, TargetType> whose isValid returns the verdict. Convention: return true for null and leave null-ness to @NotNull. Because the validator is a CDI bean it can @Inject dependencies.

Intermediate How does Bean Validation integrate with JAX-RS, and what happens on failure?

Put constraint annotations and @Valid on resource-method parameters or on the entity, and the JAX-RS runtime (Jersey/RESTEasy, backed by Hibernate Validator) validates each request automatically. A violation on an input — request body, path/query param — maps to 400 Bad Request; a violation on a return value is a server-side error (500). ValidationConfig customizes the validator used.

Pro How does method validation work, and how is it enforced at runtime?

Constraints on parameters express preconditions, constraints on the return value express postconditions, and @Valid on a parameter cascades into it. Enforcement is via an interceptor on the managed bean: the container validates arguments before the call and the result after, throwing ConstraintViolationException on failure. Because it is interceptor/proxy based (Spring's equivalent needs @Validated plus a MethodValidationPostProcessor), a self-invocation — one method on the bean calling another on this — bypasses the interceptor and is silently unvalidated.

Pro How would you validate that two fields match (e.g. password and confirmation)?

A field-level constraint can't see a sibling field, so use a class-level custom constraint: annotate the type with @Constraint(validatedBy = ...) whose ConstraintValidator<A, TheClass> receives the whole object and compares the two properties in isValid. To attach the error to a specific field rather than the whole bean, disable the default violation and build one with ConstraintValidatorContext against that property node.

Pro Jakarta Bean Validation and Spring — same thing or different?

Same specification and annotations: Spring does not reimplement validation, it bootstraps a Bean Validation provider (Hibernate Validator by default) and exposes it through LocalValidatorFactoryBean, MVC data binding, and method validation. The nuance interviewers want: @Valid is the standard jakarta.validation annotation that triggers (and cascades) validation, whereas @Validated is Spring's own annotation that additionally selects validation groups and enables method-level validation on a bean. Custom ConstraintValidators written for Jakarta EE work unchanged in Spring, and there they too become injectable Spring beans.
