Module: Jakarta Persistence in Jakarta EE

In Jakarta EE the application server owns your EntityManager — it creates it, injects it, binds it to the transaction, and closes it for you. This module is about that container-managed model, and how it differs from the Spring Data JPA most people meet first.

What this module covers

Jakarta Persistence (JPA) is the standard for object/relational mapping in Java, and Hibernate and EclipseLink are the providers that implement it. This page focuses on how JPA works inside a Jakarta EE server: the persistence unit declared in persistence.xml, the container-managed EntityManager injected with @PersistenceContext, and the persistence context that tracks your entities. Entity mapping itself is kept deliberately brief — the Hibernate and JPA track covers it in full — and the module closes by contrasting this model with Spring Data JPA.

Topics

The persistence unit and persistence.xml

A persistence unit is a named group of entity classes that map to one database, together with the configuration to reach it. You declare it in META-INF/persistence.xml: a unit name, a transaction-type (JTA under a server, RESOURCE_LOCAL in plain Java SE), a data source, and optional provider properties. The provider — Hibernate, EclipseLink — is usually inferred from what is on the classpath, so a Jakarta EE persistence.xml is often just a few lines.

Why it matters: the persistence unit is the bootstrap contract. Its name is what @PersistenceContext(unitName=...) refers to, and getting transaction-type and the data source right is the difference between the container managing your transactions and you doing it by hand.

<persistence xmlns="https://jakarta.ee/xml/ns/persistence" version="3.1">
  <persistence-unit name="orders" transaction-type="JTA">
    <jta-data-source>java:app/jdbc/OrdersDS</jta-data-source>
    <properties>
      <property name="jakarta.persistence.schema-generation.database.action" value="none"/>
    </properties>
  </persistence-unit>
</persistence>

The container-managed EntityManager and @PersistenceContext

The EntityManager is the working interface to persistence: persist, find, merge, remove, and createQuery. In Jakarta EE you never call EntityManagerFactory.createEntityManager() yourself — that is the application-managed style. Instead you write @PersistenceContext EntityManager em; and the container injects a container-managed one, already enrolled in the active JTA transaction. You do not open it, you do not close it, and you do not manage its transaction directly.

Why it matters: this is the single defining feature of persistence in a Jakarta EE server. Note the two annotations that look alike: @PersistenceContext injects an EntityManager (what you almost always want), while @PersistenceUnit injects the EntityManagerFactory for the rare application-managed case where you need full lifecycle control.

@Stateless
public class OrderService {
    @PersistenceContext(unitName = "orders")   // container injects & scopes it
    private EntityManager em;

    public Order find(long id)   { return em.find(Order.class, id); }
    public void create(Order o)  { em.persist(o); }   // INSERT flushed at commit
}

The persistence context: first-level cache, entity states, transaction-scoped vs extended

Behind every EntityManager sits a persistence context — the set of entity instances it currently manages, and the first-level cache guaranteeing one object per row within its scope. Managed entities are dirty-checked: change a field on a managed Order and the provider issues an UPDATE at flush time, no explicit save call needed. Every entity is in one of four states — new/transient, managed, detached, removed — and the EntityManager methods move entities between them.

Why it matters: the context's lifetime is an interview favourite. The default is transaction-scoped — created when a JTA transaction begins, flushed and cleared when it commits, so managed entities become detached the moment the method returns. An extended persistence context (PersistenceContextType.EXTENDED) survives across several transactions and may only be injected into a stateful session bean — useful for multi-step conversations, and a classic source of bugs when misused.

Entities, mapping and queries — the short tour

An entity is a plain class marked @Entity with an @Id; it maps to a table, each instance to a row. You reach data three ways: em.find() by primary key, JPQL through createQuery() (object-oriented SQL over entities, not tables), and the type-safe Criteria API for dynamic queries. Named queries let you predefine JPQL and reference it by name.

This module keeps mapping deliberately shallow — relationships, inheritance, fetch strategies, the N+1 problem and JPQL/Criteria in depth all live in the Hibernate and JPA track. Here it is enough to recognise the annotations and know how the container-managed EntityManager runs the queries.

@Entity
@Table(name = "orders")
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String status;
    // no-arg constructor + accessors
}

Contrast: container-managed JPA vs Spring Data JPA

Spring Data JPA is not a competitor to JPA — it sits on top of it. The same entities, the same provider (usually Hibernate), the same persistence context. What differs is the boundary you code against: instead of injecting an EntityManager and writing find/persist, you declare a JpaRepository interface and Spring generates the implementation, deriving queries from method names. Transactions come from Spring's @Transactional rather than the server's JTA, and the EntityManager is a Spring-managed shared proxy rather than one bound by the Jakarta EE container.

Why it matters: most Indian teams reach Spring Boot first and meet the standard only later. Interviews probe whether a candidate grasps that repository.findById() is ultimately em.find(), that both share JPA's mapping and lifecycle, and that the choice is really Jakarta EE server + JTA versus Spring container + @Transactional — not two different persistence technologies.

// Jakarta EE — container-managed EntityManager
@PersistenceContext EntityManager em;
Order o = em.find(Order.class, id);

// Spring Data JPA — no EntityManager in the code you write
interface OrderRepository extends JpaRepository<Order, Long> { }
Order o = repository.findById(id).orElseThrow();

Interview questions

The persistence questions Indian interviews lean on — from screening-round definitions to the container-managed details that separate Spring-only candidates from ones who know the standard:

Beginner What is Jakarta Persistence (JPA), and how does it relate to Hibernate?

JPA is a specification — a standard set of interfaces and annotations for object/relational mapping in Java. Hibernate and EclipseLink are providers that implement it. You code to the JPA API (EntityManager, @Entity) and a provider does the actual SQL, so swapping providers ideally needs no code change.

Beginner What is a persistence unit, and where is it configured?

A named set of entity classes mapped to a single database, plus the config to connect. It is declared in META-INF/persistence.xml with a unit name, transaction type, data source and provider properties. Its name is what @PersistenceContext(unitName=...) and createEntityManagerFactory(name) reference.

Beginner What does @PersistenceContext do?

It injects a container-managed EntityManager into your component. The Jakarta EE server creates the EntityManager, binds it to the current transaction, and closes it for you — you just use it. It is the standard way to obtain an EntityManager inside a server.

Beginner Name the core EntityManager methods and what they do.

persist makes a new entity managed (INSERT at flush); find loads by primary key; merge copies a detached entity's state into a managed one; remove marks a managed entity for DELETE; createQuery runs JPQL or Criteria queries.

Intermediate Container-managed vs application-managed EntityManager?

Container-managed: injected with @PersistenceContext, lifecycle and transaction handled by the Jakarta EE server — the norm inside a server. Application-managed: you obtain an EntityManagerFactory (via @PersistenceUnit or Persistence.createEntityManagerFactory), call createEntityManager(), and must close it and manage transactions yourself — used in Java SE or when you need explicit control.

Intermediate @PersistenceContext vs @PersistenceUnit?

@PersistenceContext injects an EntityManager (container-managed, transaction-bound) — the everyday choice. @PersistenceUnit injects an EntityManagerFactory, from which you create application-managed EntityManagers yourself. Rule of thumb: inject the context unless you specifically need factory-level control.

Intermediate Transaction-scoped vs extended persistence context?

Transaction-scoped (the default) lives for one JTA transaction — created at begin, flushed and cleared at commit, so entities detach when the method returns. Extended (PersistenceContextType.EXTENDED) spans multiple transactions and may only be injected into a stateful session bean, keeping entities managed across calls for multi-step conversations.

Intermediate What is the persistence context, and what are the entity states?

The persistence context is the set of entities an EntityManager currently manages, and its first-level cache — one instance per database row within scope. Four states: new/transient (not yet associated), managed (tracked, dirty-checked, auto-flushed), detached (was managed, context closed), removed (scheduled for DELETE).

Intermediate transaction-type JTA vs RESOURCE_LOCAL in persistence.xml?

JTA (default in Jakarta EE) hands transaction control to the container's JTA transaction manager and uses a jta-data-source — you rely on @Transactional/container demarcation. RESOURCE_LOCAL (typical in Java SE) means the EntityManager manages its own resource-local transactions via em.getTransaction().begin()/commit().

Intermediate If you change a managed entity's field, do you need to call save?

No. A managed entity is dirty-checked: at flush (usually transaction commit) the provider compares its state and issues an UPDATE automatically. There is no save() in JPA. This only applies while the entity is managed — a detached entity's changes are ignored until you merge it back.

Pro persist() vs merge() — semantics and return value?

persist makes a new entity managed, returns void, and throws if it already exists. merge takes a detached (or new) entity, copies its state onto a managed instance, and returns that managed instance — the argument you passed stays detached. Continuing to use the argument after merge instead of the return value is a common bug.

Pro How does container-managed JPA differ from Spring Data JPA, and where does @Transactional fit?

Spring Data JPA layers on JPA — same entities, provider and persistence context. You write JpaRepository interfaces and Spring derives implementations, instead of injecting an EntityManager. Transactions come from Spring's @Transactional (an AOP proxy) rather than the Jakarta EE container's JTA. So it is Spring container + @Transactional versus Jakarta EE server + JTA — the same underlying JPA, wrapped differently.

Pro EntityManager is not thread-safe — why is injecting it into a field of a shared bean still safe?

Because @PersistenceContext injects a proxy, not a raw EntityManager. On each invocation the proxy delegates to the correct transaction-scoped persistence context for the current thread/transaction, so concurrent calls never share one context. That is why a stateless session bean (or a Spring shared EntityManager) can safely keep the injected reference in a field.
