Module: Jakarta Faces (JSF)

JSF is the enterprise world's component-based, server-side answer to web UIs. Largely eclipsed by single-page apps, its request lifecycle is still a favourite interview probe — and it quietly runs a great deal of internal Java software.

Knowledge Library → Courses → Jakarta EE → Jakarta Faces (JSF) Overview

What this module covers

Jakarta Faces (JSF), formerly JavaServer Faces, is a component-based, server-side MVC framework: you compose a page from stateful UI components, bind them to CDI beans through Expression Language, and let the FacesServlet drive a fixed request-processing lifecycle. This is a survey module — enough to recognise JSF in a legacy codebase, explain its lifecycle in an interview, and reason about when a modern SPA plus REST is the better call. Depth lives in the linked Jakarta EE resources.

Topics

What Jakarta Faces is: the component model

JSF is a specification, not merely a library: a component-based, server-side MVC framework in which every page is a tree of UIComponent objects rendered to HTML by a render kit. The FacesServlet is the single front controller — mapped to a URL pattern such as *.xhtml — that orchestrates the whole lifecycle. The current release is Jakarta Faces 4.1 (Jakarta EE 11, requiring Java SE 17), living under the jakarta.faces package after the javax-to-jakarta namespace rename.

Why it matters: the server-held component tree is what separates JSF from request-based frameworks like Servlets or Spring MVC. You work with components and their bound values, not raw request parameters — and recognising that model, plus the namespace change, is the baseline interview expectation for anyone maintaining enterprise Java UIs.

<h:form>
  <h:outputLabel value="Name:" for="name"/>
  <h:inputText id="name" value="#{greeter.name}"/>   <!-- two-way bound to a bean -->
  <h:commandButton value="Go" action="#{greeter.submit}"/>
</h:form>

Facelets: the view declaration language

Facelets is JSF's view declaration language (VDL): XHTML pages that pull in tag libraries (h: for HTML components, f: for core tags, ui: for templating) and use Expression Language to bind values and actions. It replaced JSP as the default view technology in JSF 2.0, and adds templating (ui:composition / ui:define / ui:insert), composite components, and faster compilation and rendering.

Why it matters: templating is how JSF applications share a common layout without a build step, and it is where third-party component suites earn their keep. PrimeFaces — the most widely used suite in the Jakarta ecosystem — layers a rich set of AJAX-enabled widgets on top of Facelets under the p: namespace, which is what most real JSF projects you will meet actually use.

<ui:composition template="/WEB-INF/template.xhtml"
    xmlns:ui="jakarta.faces.facelets">
  <ui:define name="content">
    <h:outputText value="#{bean.message}"/>
  </ui:define>
</ui:composition>

Managed beans (CDI) and navigation

The model-and-controller layer is a set of CDI beans annotated @Named and given a scope — @RequestScoped, @ViewScoped, @SessionScoped, or @ApplicationScoped. The legacy @ManagedBean annotation is deprecated; CDI is now the unified bean model shared with the rest of Jakarta EE. A backing bean holds the view's state and exposes action methods bound from component tags via EL.

Navigation is outcome-driven: an action method returns a String, and JSF resolves it by implicit navigation (an outcome of "result" maps to result.xhtml in the same directory) or by explicit navigation rules declared in faces-config.xml. Returning null or the same view re-renders the current page — the normal path for a validation failure.

@Named
@ViewScoped
public class Greeter implements Serializable {
    private String name;
    public String submit() { return "result"; } // -> result.xhtml (implicit navigation)
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}

The request-processing lifecycle

The lifecycle is the single most-tested JSF concept. A postback (form submission of an already-loaded view) runs all six phases in order: Restore View (rebuild the component tree and re-wire validators/listeners), Apply Request Values (each component decodes its submitted value), Process Validations (conversion and validation), Update Model Values (push valid values into bean properties), Invoke Application (run the action method, decide navigation), and Render Response (produce the HTML).

An initial request — the first time a page is opened — runs only Restore View and Render Response, because there is nothing to process. A conversion or validation failure short-circuits straight to Render Response with messages attached; the immediate attribute pulls a component's processing forward to Apply Request Values (the classic trick for a Cancel button that must skip validation). An <f:ajax> request runs a partial lifecycle over just the component subtree named by execute, then re-renders the subtree named by render.

<h:commandButton value="Save" action="#{form.save}">
  <f:ajax execute="@form" render="messages"/>   <!-- partial process + partial render -->
</h:commandButton>

When to use JSF vs a modern SPA + REST

The architectural fork is where JSF questions get interesting. JSF keeps UI state on the server and ships rendered HTML; a single-page app keeps state in the browser and talks to a stateless REST/JSON API (JAX-RS). JSF shines for internal, form-heavy enterprise applications built by a Java-only team: server-side state plus a component suite like PrimeFaces lets a backend developer ship a full data-entry UI without a separate frontend stack, build pipeline, or JavaScript framework.

A SPA plus JAX-RS wins for public-facing, mobile, or highly interactive UIs, for independent frontend and backend teams, and for horizontal scaling — server-held view state complicates clustering and session affinity. The well-known criticisms of JSF (statefulness and scalability cost, tight coupling of the view to the lifecycle, a steep learning curve) are real, and the mitigations — stateless/transient views, @ViewScoped beans, and client-side state saving — are worth naming in a senior interview.

Interview questions

Most-asked JSF questions in Indian interviews — it still surfaces in enterprise and services-company screens, and the lifecycle question is nearly guaranteed:

Beginner What is Jakarta Faces (JSF) and what kind of framework is it?

A specification for a component-based, server-side MVC framework for web UIs (formerly JavaServer Faces). You build a page as a tree of stateful UI components bound to server-side beans, and the FacesServlet front controller drives a fixed request-processing lifecycle that produces HTML.

Beginner What is Facelets, and how does it relate to JSP?

Facelets is JSF's view declaration language — XHTML pages using JSF tag libraries and EL. It became the default in JSF 2.0, replacing JSP, because JSP's own lifecycle conflicted with JSF's. Facelets adds templating, composite components, and better performance.

Beginner What is the FacesServlet and what does it do?

The single front controller of a JSF app, declared in web.xml (or auto-registered) and mapped to a pattern like *.xhtml. Every JSF request goes through it; it builds/restores the component tree and runs the request-processing lifecycle.

Beginner What is a backing bean, and how is one defined today?

A bean that holds a view's state and its action methods, bound from component tags via EL (#{bean.property}). Today it is a CDI bean annotated @Named with a scope such as @ViewScoped. The old @ManagedBean is deprecated in favour of CDI.

Beginner How does page navigation work in JSF?

An action method returns a String outcome. Implicit navigation maps an outcome of "result" to result.xhtml; explicit rules in faces-config.xml can override the target. Returning null or the same view id re-renders the current page — what happens after a validation failure.

Intermediate Name the six phases of the JSF request-processing lifecycle.

Restore View, Apply Request Values, Process Validations, Update Model Values, Invoke Application, Render Response — in that order. Restore View rebuilds the component tree; the middle phases decode, convert/validate, and push values into beans; Invoke Application runs the action; Render Response emits HTML.

Intermediate Which phases run for an initial request versus a postback?

An initial request (first view of a page) runs only Restore View and Render Response — there is no input to process. A postback (submitting an already-rendered form) runs all six phases. Knowing this distinction is a common follow-up to the phases question.

Intermediate What does the immediate attribute do?

Set immediate="true" on a component to move its conversion, validation, and action handling forward into the Apply Request Values phase, bypassing the later phases (and other components' validation). The classic use is a Cancel or navigation button that must work even when the form has invalid input.

Intermediate What is @ViewScoped, and when do you reach for it?

A CDI scope whose bean lives as long as the user keeps interacting with the same view (across postbacks) and is discarded on navigation. It is ideal for AJAX-heavy pages, wizards, and editable data tables where per-request state is too short and session scope leaks memory across views.

Intermediate How does JSF save component state, and what is the trade-off?

JSF persists the component tree between requests via the jakarta.faces.STATE_SAVING_METHOD setting: server stores it in the session (fast, uses server memory, complicates clustering) or client serialises it into a hidden field in the page (stateless servers, but larger payloads and encryption concerns). This statefulness is the root of most scalability debates about JSF.

Pro Where does AJAX fit into the JSF lifecycle?

<f:ajax> triggers a partial lifecycle: the execute attribute names the component subtree to process (Apply Request Values through Invoke Application), and render names the subtree to re-render in Render Response. The full lifecycle still runs conceptually, but only over the targeted components, and the response is a partial-page XML update rather than a full page.

Pro When would you still choose JSF over a SPA plus REST, and what are its main criticisms?

Choose JSF for internal, form-heavy enterprise apps built by a Java-only team, where server-side state and a suite like PrimeFaces avoid a separate frontend stack. Prefer a SPA plus JAX-RS for public/mobile UIs, decoupled teams, and easy horizontal scaling. Criticisms: server-held state hurts scalability and clustering, the view is tightly coupled to the lifecycle, and the learning curve is steep — mitigated with stateless views, @ViewScoped, and client-side state saving.
