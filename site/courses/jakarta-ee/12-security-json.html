Module: Jakarta Security and JSON

Two specs sit behind almost every Jakarta endpoint: one decides who is allowed to call it, the other shapes what crosses the wire. This module maps both against their Spring equivalents and links the specifications and tutorials that teach them properly.

Knowledge Library → Courses → Jakarta EE → Jakarta Security and JSON

What this module covers

Jakarta Security standardises authentication and authorization as CDI-friendly annotations — HttpAuthenticationMechanism decides how a caller proves identity, IdentityStore validates the credential, @RolesAllowed guards methods, and SecurityContext exposes the caller programmatically — replacing the old container-specific web.xml realm setup. Alongside it, the platform ships two JSON APIs: JSON-P parses and builds JSON at a low level (object-model and streaming), and JSON-B binds Java objects to and from JSON the way Jackson does in Spring. Between them they cover securing an endpoint and serialising its payload, and both are contrasted here with the Spring stack you likely already know.

Topics

Jakarta Security: authentication mechanisms

An HttpAuthenticationMechanism is the CDI-managed component that obtains a caller's credentials over HTTP — there is one per application, discovered as a bean. You rarely write one from scratch: the container ships built-ins activated by annotations — @BasicAuthenticationMechanismDefinition, @FormAuthenticationMechanismDefinition, @CustomFormAuthenticationMechanismDefinition, and (since Security 3.0) @OpenIdAuthenticationMechanismDefinition for OpenID Connect. Placing one annotation on an @ApplicationScoped bean turns login on, no XML deployment descriptor required.

Why it matters: this is the modern, portable replacement for the servlet-container realm configuration every legacy Java EE app carried. The contrast with Spring is worth stating in interviews — Spring wires authentication through a SecurityFilterChain and a servlet filter, while Jakarta Security expresses the same intent declaratively on a CDI bean the container discovers.

@FormAuthenticationMechanismDefinition(
    loginToContinue = @LoginToContinue(
        loginPage = "/login.html", errorPage = "/error.html"))
@ApplicationScoped
public class AppSecurityConfig { }   // one bean switches form login on

Identity stores, roles and SecurityContext

Authentication answers "who are you"; the IdentityStore answers "are these credentials valid, and what groups do they belong to". The container mandates a database-backed store (configured with @DatabaseIdentityStoreDefinition) and offers an LDAP one (@LdapIdentityStoreDefinition); you can also implement the interface directly. When several stores exist, the built-in IdentityStoreHandler validates against them in priority order and aggregates the groups they return. Groups map to roles, and @RolesAllowed("admin") — the declarative guard from Jakarta Annotations — then denies anyone lacking the role at the method or class level.

For decisions the annotations cannot express, inject SecurityContext: it gives you getCallerPrincipal() and isCallerInRole(...) for programmatic checks inside a method. This split — declarative @RolesAllowed for coarse gates, programmatic SecurityContext for fine-grained logic — mirrors Spring Security's @PreAuthorize versus injecting the SecurityContextHolder, a comparison interviewers like to draw out.

@Inject SecurityContext securityContext;

if (securityContext.isCallerInRole("admin")) {        // programmatic gate
    Principal caller = securityContext.getCallerPrincipal();
    audit(caller.getName());
}

Jakarta Security versus Spring Security

Both solve the same problem; the packaging differs. Jakarta Security is a container-provided standard — you declare intent with annotations and the application server (WildFly, Payara, Open Liberty, GlassFish) implements it, so the same code is portable across runtimes. Spring Security is a framework you assemble yourself: a chain of servlet filters configured through a SecurityFilterChain bean, far larger in surface area (OAuth2 client and resource server, SAML, method security, CSRF, remember-me, a deep testing story) and evolving faster, but tied to the Spring ecosystem rather than the Jakarta container.

Rule of thumb for interviews: if the app already runs on a Jakarta EE server, Jakarta Security adds no second runtime or patching cadence and covers Basic, Form, Custom Form and OpenID Connect out of the box; if you need breadth — social login, fine-grained OAuth2 scopes, reactive support — Spring Security is the richer toolkit. Know both mental models and when each is the operationally simpler choice.

JSON Processing (JSON-P)

JSON-P is the low-level API for reading, writing and querying JSON without binding it to a domain class. It offers two models. The object-model API builds an in-memory tree — JsonObject, JsonArray, and the Json.createObjectBuilder() / createArrayBuilder() factories — convenient when you need random access to fields. The streaming API (JsonParser reading events, JsonGenerator writing them) processes one element at a time and is the right choice for very large documents where holding the whole tree in memory is wasteful. JSON-P also implements JSON Pointer, JSON Patch and JSON Merge Patch for addressing and mutating parts of a document.

Why it matters: JSON-B and JAX-RS providers are built on top of JSON-P, so it is the foundation layer; and when you receive arbitrary JSON with no matching Java class, JSON-P is how you parse it precisely. It is the Jakarta counterpart to Jackson's low-level JsonParser/tree model.

JsonObject json = Json.createObjectBuilder()
    .add("name", "Asha")
    .add("roles", Json.createArrayBuilder().add("admin"))
    .build();                                   // object-model tree

JsonObject back = Json.createReader(
        new StringReader(json.toString())).readObject();
String name = back.getString("name");           // random-access read

JSON Binding (JSON-B)

JSON-B is the high-level binding layer: it maps Java objects to and from JSON with a standard default algorithm, so a POJO round-trips with almost no code. Create a Jsonb instance via JsonbBuilder.create(), then toJson(obj) serialises and fromJson(json, Type.class) deserialises. Defaults handle public fields and getters/setters; you override them with compile-time annotations — @JsonbProperty to rename, @JsonbTransient to skip, @JsonbDateFormat and @JsonbNumberFormat for formatting — or at runtime through JsonbConfig (property-naming strategy, null handling, pretty printing). Yasson is the reference implementation.

Why it matters: this is the JSON-B versus Jackson question. In a JAX-RS application JSON-B is the standard, container-supplied body converter; a Spring Boot app uses Jackson by default. Knowing the annotation mapping (@JsonbProperty is the counterpart to Jackson's @JsonProperty) and that JSON-B is built on JSON-P underneath is the level of detail interviewers probe.

Jsonb jsonb = JsonbBuilder.create();
String out  = jsonb.toJson(user);                 // serialize POJO
User   back = jsonb.fromJson(out, User.class);    // deserialize

// customise a field: @JsonbProperty("full_name") on the getter

Interview questions

Most-asked Jakarta Security and JSON questions in Indian interviews — enterprise-Java and product-company rounds probe both the annotations and the Spring contrast:

Beginner What problem does Jakarta Security solve over the old Java EE security model?

It standardises authentication and authorization as portable, CDI-based annotations instead of container-specific web.xml realms and vendor consoles. The same @BasicAuthenticationMechanismDefinition or @FormAuthenticationMechanismDefinition code runs unchanged across WildFly, Payara or Open Liberty.

Beginner What is HttpAuthenticationMechanism and how do you enable Basic or Form login?

It is the CDI bean that obtains a caller's credentials over HTTP — one per application. You enable a built-in by placing an annotation on an @ApplicationScoped bean: @BasicAuthenticationMechanismDefinition, @FormAuthenticationMechanismDefinition, @CustomFormAuthenticationMechanismDefinition, or @OpenIdAuthenticationMechanismDefinition for OpenID Connect. No XML needed.

Beginner What is an IdentityStore and what built-in stores exist?

It validates a credential and returns the caller's groups. The container mandates a database-backed store (@DatabaseIdentityStoreDefinition) and offers LDAP (@LdapIdentityStoreDefinition); you can also implement the IdentityStore interface for a custom source. Authentication mechanism gets the credential, identity store verifies it.

Beginner What is the difference between JSON-P and JSON-B?

JSON-P is a low-level API to parse, build and query raw JSON (JsonObject, JsonParser) with no domain class. JSON-B is a high-level binding layer that maps whole Java objects to and from JSON (jsonb.toJson / fromJson). JSON-B is built on top of JSON-P.

Beginner Object-model API vs streaming API in JSON-P?

The object model builds the whole document as an in-memory tree (JsonObject/JsonArray) giving random access — convenient but memory-heavy. The streaming API (JsonParser/JsonGenerator) reads or writes one event at a time, minimal memory — the right choice for very large payloads.

Intermediate How does @RolesAllowed work, and how does it differ from a SecurityContext check?

@RolesAllowed("admin") is declarative — the container blocks any caller not in the role before the method runs, at class or method level. SecurityContext.isCallerInRole(...) and getCallerPrincipal() are programmatic, used inside a method for logic annotations can't express. Coarse gate vs fine-grained decision.

Intermediate How do the authentication mechanism, identity store and IdentityStoreHandler fit together?

The HttpAuthenticationMechanism collects the credential and hands it to the IdentityStoreHandler, which iterates the available validating IdentityStores in priority order until one validates it, then aggregates groups from group-providing stores. It returns the validated caller and groups, which become roles.

Intermediate How do you customize JSON-B mapping?

Compile-time annotations on the class: @JsonbProperty("full_name") renames, @JsonbTransient skips a field, @JsonbDateFormat/@JsonbNumberFormat format values, @JsonbCreator picks a constructor. Runtime: a JsonbConfig passed to JsonbBuilder.create(config) sets property-naming strategy, null handling and pretty printing.

Intermediate JSON-B vs Jackson — which does a Jakarta EE app use, and why care?

In JAX-RS, JSON-B is the standard container-supplied body converter (Yasson as RI); Spring Boot defaults to Jackson. They are functionally similar — @JsonbProperty mirrors Jackson's @JsonProperty — but the annotations and default naming/null behaviours differ, so mixing them or porting between stacks needs care.

Intermediate How does Jakarta Security compare with Spring Security?

Jakarta Security is a container-implemented standard driven by annotations — portable, no extra runtime, covers Basic/Form/Custom Form/OpenID Connect. Spring Security is a self-assembled filter chain (SecurityFilterChain) with far broader features (OAuth2, SAML, reactive, deep testing) but tied to Spring. Choose by ecosystem and required breadth.

Pro How would you implement a custom HttpAuthenticationMechanism, e.g. for JWT bearer tokens?

Implement HttpAuthenticationMechanism as an @ApplicationScoped bean and override validateRequest(request, response, httpMessageContext): read the Authorization: Bearer header, verify the token, and on success call httpMessageContext.notifyContainerAboutLogin(principal, groups) returning SUCCESS; otherwise return SEND_FAILURE or doNothing() for unprotected paths. Optionally back it with a custom IdentityStore.

Pro Walk through the validateRequest flow and its AuthenticationStatus values.

The container calls validateRequest on every protected request. The mechanism returns an AuthenticationStatus: SUCCESS (caller established via notifyContainerAboutLogin), SEND_CONTINUE (a multi-step flow like a redirect to a login page is in progress), SEND_FAILURE (authentication failed), or NOT_DONE (no authentication attempted, unprotected resource). The container then applies authorization.

Pro When do you choose JSON-P streaming over the object model, and what is the trade-off?

Streaming (JsonParser/JsonGenerator) when the document is large or you process it once forward-only — it uses constant memory instead of materialising the whole tree. The trade-off is no random access and more procedural code: you react to events (START_OBJECT, KEY_NAME, VALUE_STRING) rather than calling getString("field").

Pro What are JSON Pointer, JSON Patch and JSON Merge Patch in JSON-P?

Standards JSON-P implements. JSON Pointer addresses a node by path (/roles/0). JSON Patch applies an ordered list of operations (add, remove, replace, move, copy, test) to a document. JSON Merge Patch applies a partial document as an overlay, where null means "delete this key". Useful for HTTP PATCH semantics and partial updates without deserialising to a POJO.
