Module: Platform Overview

Before you write a single @Inject, you need the map: what "Jakarta EE" actually is, why every enterprise import quietly changed from javax to jakarta, and how one set of specs runs on five different servers. This module draws that map — and marks where Spring sits on it.

What this module covers

Jakarta EE is the vendor-neutral standard for enterprise Java — the platform once called J2EE, then Java EE, now stewarded by the Eclipse Foundation. This overview orients you on the four facts every interview and every project assumes: the Java EE → Jakarta EE rename and its javax→jakarta namespace break, the specifications-plus-implementations model, the Core/Web/Platform profiles up to Jakarta EE 11, and how application servers relate to (and differ from) Spring.

Topics

From Java EE to Jakarta EE: the javax → jakarta namespace change

Jakarta EE is the current name of the platform that was J2EE and then Java EE. Oracle donated Java EE to the Eclipse Foundation in 2017; a 2018 community vote produced the name "Jakarta EE". Because Oracle kept the javax trademark, Eclipse could evolve the APIs only under a new root package — so Jakarta EE 9 (2020) did the "big bang": every enterprise package moved from javax.* to jakarta.* (javax.servlet → jakarta.servlet, javax.persistence → jakarta.persistence). Jakarta EE 8 was Java EE 8 under a new name; from version 9 on, the namespace is jakarta.

Why it matters: this is the single most-asked platform fact in interviews, and a genuine migration hazard well beyond Jakarta EE itself — moving to Spring Boot 3, Tomcat 10 or Hibernate 6 drags you across the same line. Note the Java SE packages (javax.sql, javax.crypto) did not move; only the enterprise APIs did.

// Java EE 8 and earlier
import javax.inject.Inject;
import javax.ws.rs.GET;

// Jakarta EE 9+  — same APIs, new namespace root
import jakarta.inject.Inject;
import jakarta.ws.rs.GET;

The specifications umbrella: specs, APIs and compatible implementations

Jakarta EE is not a downloadable product — it is roughly forty independent specifications (CDI, Servlet, RESTful Web Services/JAX-RS, Persistence, Messaging, Bean Validation, Security and more) released together under one umbrella version. Each specification ships three things: a written specification document, an API JAR you compile against, and a Technology Compatibility Kit (TCK) that a runtime must pass to call itself compatible.

Why it matters: this spec-plus-implementation split is the whole point of the platform. You code against the standard API (Maven scope provided), and any server that passes the TCK supplies the implementation at runtime — so the same WAR can run on WildFly, Payara or Open Liberty. It is enterprise Java's version of coding to an interface rather than a concrete class.

<!-- Compile against the spec API; the server provides the implementation -->
<dependency>
  <groupId>jakarta.platform</groupId>
  <artifactId>jakarta.jakartaee-api</artifactId>
  <version>11.0.0</version>
  <scope>provided</scope>
</dependency>

Profiles: Core, Web and Platform — and Jakarta EE 11

The full specification set is large, so Jakarta EE bundles it into three profiles a product can certify against. The Platform is everything; the Web Profile is the subset aimed at web applications (Servlet, CDI, JPA, JAX-RS — no full Enterprise Beans or JMS); the Core Profile, introduced in Jakarta EE 10, is a lean subset (CDI Lite, JAX-RS, JSON) targeted at smaller cloud-native runtimes and the foundation MicroProfile builds on. A runtime certifies against a profile, not spec by spec.

Why it matters: the current release is Jakarta EE 11 (Platform GA mid-2025), which requires Java 17 as a minimum with first-class Java 21 support (virtual threads) and adds the new Jakarta Data specification. Knowing which profile a runtime targets tells you what you may assume is present — a Core Profile server will not hand you Enterprise Beans or Messaging out of the box, while a Platform server will.

Application servers vs Spring: runtimes and portability

A Jakarta EE application server is the runtime that implements the specifications and hosts your deployment. The open-source mainstays are WildFly (Red Hat, the upstream of JBoss EAP), Payara (a GlassFish fork, now under Azul), Open Liberty (IBM, a composable feature model where you enable only the specs you need), and GlassFish (the reference implementation). Because they all pass the same TCKs, a standards-only application can move between them — that portability is Jakarta EE's structural advantage: no single vendor owns the runtime.

Why it matters: the classic interview contrast is Jakarta EE versus Spring. Spring is a single-vendor framework, usually shipped as a self-contained JAR with an embedded server via Spring Boot; Jakarta EE is a multi-vendor standard you deploy onto a server. The line has blurred — Spring 6 / Spring Boot 3 run on the jakarta.* namespace, and Jakarta EE + MicroProfile on Quarkus or Helidon competes directly with Spring Boot on startup time and memory. But portability is not free: CDI edge cases, logging, JNDI names and deployment behaviour differ per server, so you still test against each target runtime.

Interview questions

Most-asked platform-overview questions in Indian interviews — the Jakarta EE rename and the Spring contrast are staple screening questions for enterprise-Java roles:

Beginner What is Jakarta EE, and how is it related to Java EE?

Jakarta EE is the current name of the enterprise Java platform that was J2EE and then Java EE. Oracle donated Java EE to the Eclipse Foundation in 2017; the community renamed it Jakarta EE in 2018. Jakarta EE 8 equalled Java EE 8; later versions evolve independently under Eclipse's open, vendor-neutral process.

Beginner What changed in the javax → jakarta switch, and in which version?

In Jakarta EE 9 (2020) all enterprise API packages moved from javax.* to jakarta.* (for example javax.servlet → jakarta.servlet). Oracle kept the javax trademark, so Eclipse had to re-root the packages. Java SE packages such as javax.sql and javax.crypto were unaffected.

Beginner What is a Jakarta EE application server? Name a few.

A runtime that implements the Jakarta EE specifications and hosts your deployed application (WAR/EAR), providing services such as CDI, transactions, persistence and security. Examples: WildFly, Payara, Open Liberty and GlassFish (the reference implementation).

Beginner What are the Jakarta EE profiles?

Curated subsets of the specifications. Platform is the full set; Web Profile is the web-application subset (Servlet, CDI, JPA, JAX-RS, and so on); Core Profile is a lean cloud-native subset (CDI Lite, JAX-RS, JSON) introduced in Jakarta EE 10. Products certify against a profile.

Beginner At a high level, how is Jakarta EE different from Spring?

Jakarta EE is a vendor-neutral standard (specifications) implemented by multiple servers; Spring is a single framework and ecosystem from one vendor. You deploy a Jakarta EE app onto a compliant server, whereas Spring Boot usually bundles an embedded server into a self-contained JAR.

Intermediate What do "specification", "API" and "compatible implementation" mean, and how do they fit together?

A specification is the written contract; the API JAR is what you compile against (scope provided); a compatible implementation is a runtime that passes the spec's TCK. You code to the API and the server provides the implementation — which is what enables portability.

Intermediate What is a TCK, and why does it matter for portability?

The Technology Compatibility Kit is the conformance test suite a runtime must pass to be certified for a specification or profile. Because WildFly, Payara and Open Liberty all pass the same TCKs, a standards-only application can move between them — the TCK is what makes "vendor-neutral" real rather than a slogan.

Intermediate Why was the Core Profile introduced?

To serve cloud-native, microservice-style runtimes that do not need the full platform. It is a minimal subset (CDI Lite, JAX-RS, JSON-P/JSON-B) that MicroProfile builds on, so lightweight servers like Quarkus and Helidon can be Jakarta-compliant without shipping Enterprise Beans, JMS or Faces.

Intermediate Spring Boot 3 uses jakarta.* imports — does that make it Jakarta EE?

No. Spring builds on individual Jakarta specifications (Servlet, Persistence, Validation) and follows the namespace, but Spring itself is not a Jakarta EE implementation and is not TCK-certified as a platform or profile. It consumes Jakarta APIs; it does not implement the profile.

Intermediate How do you migrate a Java EE app to Jakarta EE 9 or later?

Move enterprise imports and configuration from javax.* to jakarta.*, bump dependencies and the target server to Jakarta 9+ versions, and update descriptor namespaces (web.xml, persistence.xml). Use tooling — IntelliJ's "Migrate to Jakarta EE" or OpenRewrite — rather than blind find-and-replace, because some javax.* packages stay. Do not mix both namespaces in one application.

Pro "Portability is free" — where does that claim break down in practice?

The APIs port, but behaviour at the edges does not. CDI ambiguous-resolution rules, classloading, deployment and error handling, logging, JNDI names and vendor-specific extensions differ across WildFly, Payara and Open Liberty. Portability is a strong default, but you still test — and occasionally tweak — per target server.

Pro For a new cloud-native microservice, how would you choose between Jakarta EE (+ MicroProfile) and Spring Boot?

Weigh standards-and-portability against ecosystem-and-velocity. Jakarta EE Core Profile with MicroProfile on Quarkus or Helidon gives standard APIs, fast startup, low memory (and native images) and no single-vendor lock-in; Spring Boot gives a larger ecosystem, more integrations and a bigger hiring pool. Factor in team skills, existing infrastructure, whether you need Enterprise Beans/JMS/JTA, and whether vendor-neutrality is a real requirement. There is no universal winner — match it to the constraints, and note both now share the jakarta.* namespace so the API surface overlaps heavily.
