Module: Jakarta Servlet

The servlet is the request-handling primitive every Java web framework is built on — Spring MVC's DispatcherServlet, JAX-RS, and JSF all run inside a servlet container. Learn the raw API and you can read the frameworks that hide it.

Knowledge Library → Courses → Jakarta EE → Jakarta Servlet

What this module covers

Jakarta Servlet is the low-level HTTP contract of the Java enterprise platform: a servlet is a container-managed object that turns an HttpServletRequest into an HttpServletResponse. This module orients you on the servlet and its container, the init/service/destroy lifecycle, the filter and listener extension points, session state over stateless HTTP, and — the interview favourite — how Spring's DispatcherServlet is nothing more than one clever servlet sitting on top of this same API.

Topics

Servlets, the container, and the request/response cycle

A servlet extends HttpServlet and overrides doGet, doPost and friends; the servlet container (Tomcat, Jetty, or the web layer of WildFly / Payara / Open Liberty) does everything else — it listens on the socket, parses HTTP, builds the request and response objects, routes the request to the right servlet by URL pattern, and manages that servlet's whole life. You write request-handling logic; the container owns the plumbing, concurrency and networking.

Why it matters: this is the foundation the rest of Jakarta EE and Spring stand on. Understanding that your code is a callback invoked by a container — not a program with a main — explains threading, scopes, and why frameworks configure so much through the container.

@WebServlet("/hello")
public class HelloServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws IOException {
        resp.setContentType("text/plain");
        resp.getWriter().write("Hello, " + req.getParameter("name"));
    }
}

The servlet lifecycle: init, service, destroy

The container creates one instance of each servlet and reuses it for every request. On first use it calls init() once (initialise resources — DB pools, config); for each incoming request it calls service(), which dispatches to doGet/doPost; on shutdown or undeploy it calls destroy() once. Because a single instance serves many requests on many container threads at once, servlets must be stateless — mutable instance fields are shared across threads and are a classic bug.

Why it matters: "explain the servlet lifecycle" is one of the most-asked Jakarta/Java-web interview questions in India, and the follow-up — "is a servlet thread-safe?" — separates candidates who memorised three method names from those who understand the single-instance, multi-threaded model.

public void init() throws ServletException { /* once, before the first request */ }
public void service(ServletRequest req, ServletResponse res) { /* once per request, on a container thread */ }
public void destroy() { /* once, when the servlet is unloaded */ }

Filters and listeners

A Filter intercepts requests and responses around a servlet — the container builds a chain, and each filter calls chain.doFilter(...) to pass control down and get it back on the way out. Filters are how you do cross-cutting concerns without touching servlet code: logging, authentication, compression, CORS, request timing. Listeners react to container events rather than requests — ServletContextListener fires on application startup/shutdown, HttpSessionListener on session create/destroy — the hooks you use to initialise and tear down application-scoped resources.

Why it matters: filters are the servlet-level ancestor of Spring's filter chain (Spring Security is filters) and of Jakarta's interceptors. Knowing where the chain runs — before any servlet, controller or handler — is essential for reasoning about security and request processing order.

@WebFilter("/*")
public class TimingFilter implements Filter {
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        long start = System.nanoTime();
        chain.doFilter(req, res);              // hand off down the chain
        log(System.nanoTime() - start);        // control returns here afterwards
    }
}

Sessions and state management

HTTP is stateless — each request stands alone — so servlets track a user across requests with HttpSession. On the first request.getSession() the container creates a session, stores it server-side, and sends the client a JSESSIONID cookie; subsequent requests carrying that cookie resolve to the same session (URL rewriting is the cookie-less fallback). You bind objects with setAttribute/getAttribute, and sessions expire on a configurable timeout or explicit invalidate().

Why it matters: sessions are the original answer to "how do I keep a user logged in?" and understanding them clarifies the three servlet scopes — request, session, application — that CDI and Spring later mirror with @RequestScoped, @SessionScoped and singletons.

HttpSession session = req.getSession();          // created if absent
session.setAttribute("cart", cart);              // lives across requests, server-side
Object saved = session.getAttribute("cart");     // next request, same JSESSIONID

Servlet vs Spring MVC DispatcherServlet

Spring MVC does not replace the Servlet API — it sits on top of it. DispatcherServlet is an HttpServlet, registered (usually at /) as the single front controller that receives every request, then delegates to handler mappings, controllers, argument resolvers and view resolvers. Where a plain servlet couples URL, parsing and business logic in one doGet, the DispatcherServlet centralises the boilerplate and lets you write annotated @Controller methods instead.

Why it matters: "how is DispatcherServlet related to a servlet?" and "servlet vs Spring MVC" are staple Indian-interview questions precisely because the honest answer — it is a servlet — proves you understand what the framework abstracts. It also explains why Spring apps still deploy to a servlet container and why servlet filters run before any Spring controller.

// Spring registers DispatcherServlet — itself an HttpServlet — as the front controller
ServletRegistration.Dynamic reg =
    servletContext.addServlet("dispatcher", new DispatcherServlet(appContext));
reg.addMapping("/");                             // every request enters through it
reg.setLoadOnStartup(1);

Interview questions

Most-asked servlet questions in Indian interviews — Jakarta/Java-web rounds lean hard on the lifecycle and on the DispatcherServlet connection:

Beginner What is a servlet, and what does the servlet container do?

A servlet is a Java class (usually extending HttpServlet) that handles HTTP requests and produces responses. The container (Tomcat, Jetty, WildFly's web layer) manages everything around it: opening sockets, parsing HTTP, creating request/response objects, mapping URLs to servlets, managing threads, and running the servlet's lifecycle. Your servlet is a callback the container invokes — not a standalone program.

Beginner Difference between GenericServlet and HttpServlet? doGet vs doPost?

GenericServlet is protocol-independent; HttpServlet extends it for HTTP and provides doGet, doPost, doPut, doDelete etc. doGet handles GET (idempotent, params in the URL); doPost handles POST (body payload, non-idempotent, form submits). You override the ones you need; service() dispatches by HTTP method.

Beginner Name the three servlet lifecycle methods and when each runs.

init() — once, before the first request, to set up resources. service() — once per request, dispatching to doGet/doPost. destroy() — once, when the container unloads the servlet, to release resources. init/service/destroy is the servlet lifecycle.

Beginner How do you map a servlet to a URL?

Two ways: the @WebServlet("/path") annotation on the class (Servlet 3.0+, no XML), or a <servlet> + <servlet-mapping> pair in the web.xml deployment descriptor. Annotation is the modern default; web.xml still wins for externalised or overriding configuration.

Beginner What do HttpServletRequest and HttpServletResponse give you?

The request exposes parameters, headers, cookies, the session, the input stream and path info; the response lets you set status, headers, content type and write the body via getWriter() or getOutputStream(). Both are created per request by the container and are valid only for that request.

Intermediate How many instances of a servlet does the container create, and how does it serve concurrent requests?

One instance per servlet (unless it implements the deprecated SingleThreadModel). The container serves concurrent requests by running service() on multiple threads against that single shared instance — a thread from its pool per request. This single-instance, multi-threaded model is the crux of most lifecycle follow-up questions.

Intermediate Are servlets thread-safe? What is the risk with instance fields?

No — because one instance is shared across request threads, mutable instance fields are shared mutable state and cause race conditions. Keep servlets stateless: use local variables (per-thread) and request/session attributes for state. Never store per-request data in a field.

Intermediate What is a Filter and how does the filter chain work?

A Filter intercepts a request before it reaches the servlet and the response on the way back. The container assembles matching filters into a chain; each calls chain.doFilter(req, res) to pass control onward, and code after that call runs on the return trip. Used for logging, auth, compression, CORS — cross-cutting concerns kept out of servlet code.

Intermediate What are listeners? Give two examples.

Listeners react to container lifecycle events, not requests. ServletContextListener fires on web-app startup/shutdown (initialise/close app-wide resources); HttpSessionListener fires on session creation/destruction (e.g. counting active users). Registered with @WebListener or in web.xml.

Intermediate HTTP is stateless — how does HttpSession maintain state across requests?

On first getSession() the container creates a server-side session and sends a JSESSIONID cookie; later requests carrying that cookie resolve to the same session. If cookies are disabled, URL rewriting (;jsessionid=...) is the fallback. Attributes bound to the session persist until timeout or invalidate().

Intermediate ServletContext vs ServletConfig, and what are the servlet scopes?

ServletConfig is per-servlet init configuration; ServletContext is one shared object per web application — the app scope, used for application-wide attributes and shared by all servlets, filters and listeners. The three scopes are request (one request), session (one user), application (whole app) — mirrored later by CDI/Spring scopes.

Intermediate forward() vs sendRedirect()?

RequestDispatcher.forward() is server-side: the same request/response are handed to another resource, the URL in the browser doesn't change, and request attributes survive. response.sendRedirect() tells the browser to make a new request to a new URL (302) — URL changes, request attributes are lost, but it works across contexts and avoids double-submit on refresh.

Pro What changed between javax.servlet (Java EE) and jakarta.servlet (Jakarta EE)?

Only the package namespace: Oracle transferred Java EE to the Eclipse Foundation, which could not use the javax namespace, so from Jakarta EE 9 every API moved javax.servlet.* → jakarta.servlet.*. It's a mechanical rename with no behavioural change, but it's a hard, all-or-nothing migration — servers/frameworks target one namespace, which is why Servlet 5.0+ (jakarta) and Servlet 4.0 (javax) don't mix.

Pro Filter vs Spring MVC HandlerInterceptor — which runs where?

A servlet Filter is part of the container and wraps the whole request, running before it even reaches the DispatcherServlet and around every servlet. A Spring HandlerInterceptor runs inside the DispatcherServlet, only for Spring-mapped handlers, with access to the resolved handler and ModelAndView. Rule of thumb: container-wide, framework-agnostic concerns → filter; Spring-aware, handler-level concerns → interceptor.

Pro Is Spring's DispatcherServlet a servlet? How does it relate to the Servlet API?

Yes — DispatcherServlet extends FrameworkServlet extends HttpServletBean extends HttpServlet. It is registered as a normal servlet (typically mapped to /) and acts as the front controller: every request enters its service(), and it delegates to handler mappings, controllers, argument resolvers and view resolvers. Spring MVC is therefore a sophisticated single servlet, not a replacement for the Servlet API — which is why Spring apps still need a servlet container.

Pro Contrast building a web app with raw servlets versus Spring MVC's DispatcherServlet — what does the framework actually add?

With raw servlets you write one servlet per URL family and hand-code parsing, dispatch, validation, content negotiation and view selection inside doGet/doPost — total control, heavy boilerplate, no thread-safety help. The DispatcherServlet centralises all of that: URL-to-method routing via handler mappings, automatic binding of params/body to method arguments, pluggable view/JSON rendering, exception handling, and interceptors — so you write small stateless @Controller methods. It's the same Servlet API underneath; Spring trades a little indirection for a lot less repetition and a consistent request-processing pipeline.
