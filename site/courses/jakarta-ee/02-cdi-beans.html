Module: CDI: Beans, Injection and Scopes

CDI is the wiring layer of Jakarta EE — the "glue" that hands beans their dependencies by type and gives each one a lifecycle. Coming from Spring, this is the standardized cousin of @Autowired, and the differences are exactly what interviewers probe.

What this module covers

Jakarta Contexts and Dependency Injection (CDI) provides two complementary services: typesafe dependency injection (wiring components by type and qualifier instead of by string name) and contexts (binding a bean's lifecycle to a well-defined scope). This module orients you on @Inject and how beans are discovered, on qualifiers and @Named, and on the scope model — then sets each concept against its Spring equivalent, because the CDI-vs-Spring comparison is a standing interview theme in the Indian enterprise-Java market.

Topics

@Inject and typesafe dependency injection

An injection point is any field, constructor parameter, or initializer method marked @Inject; at instantiation the container finds a bean whose type and qualifiers satisfy that point and supplies it. Resolution is typesafe — matching is on the Java type, not a string bean id — so a wrong wiring is a compile-visible type problem, not a runtime name typo. Constructor injection is favoured for the same reasons as in Spring: it allows final fields, makes dependencies explicit, and needs no container to unit-test.

Why it matters: CDI turns "who creates my collaborator?" into a declaration. An unsatisfied or ambiguous dependency is a deployment error — the application server refuses to start rather than failing on first use, which is one of CDI's strongest reliability guarantees and a favourite interview point.

@ApplicationScoped
public class OrderService {
    private final PaymentGateway gateway;

    @Inject                                 // constructor injection
    public OrderService(PaymentGateway gateway) {
        this.gateway = gateway;             // resolved by type at startup
    }
}

Bean discovery: bean archives, beans.xml and discovery modes

CDI only manages beans it finds in a bean archive — a jar or module the container scans. A beans.xml descriptor (in META-INF/ or WEB-INF/) marks and configures that archive. Its bean-discovery-mode chooses what gets picked up: annotated (only classes carrying a bean-defining annotation such as a scope or stereotype), all (every eligible class), or none. Since CDI 4.0 an absent or empty beans.xml implies annotated discovery — a change from the older default of all that catches out people upgrading legacy apps.

Why it matters: "my bean isn't being injected" is usually a discovery problem, not an injection one. CDI 4.0 also split the spec into Lite (a build-time-friendly core used by Quarkus and friends) and Full (which Jakarta EE platform servers implement) — worth naming when discussing modern runtimes.

Qualifiers and @Named

When more than one bean implements a type, the injection point is ambiguous and deployment fails. Qualifiers resolve this: every bean and injection point without an explicit one carries @Default, while @Any matches them all. A custom qualifier is just an annotation meta-annotated with @Qualifier, placed on both the bean and the injection point — a typesafe alternative to Spring's string-based @Qualifier("name"). @Named is a special string qualifier whose real job is exposing a bean to Expression Language (Jakarta Faces), not general wiring; reaching for it as a normal disambiguator is a common anti-pattern.

@Qualifier
@Retention(RUNTIME)
@Target({ TYPE, METHOD, FIELD, PARAMETER })
public @interface Reliable {}

@Reliable @ApplicationScoped
public class StripeGateway implements PaymentGateway { /* ... */ }

@Inject @Reliable PaymentGateway gateway;   // no ambiguity now

Scopes and contexts

A scope binds a bean's lifecycle to a context that the container creates and destroys for you. The normal scopes are @RequestScoped (one instance per HTTP request), @SessionScoped (per user session), @ApplicationScoped (one per application), and @ConversationScoped (a developer-demarcated span across several requests). Two pseudo-scopes round it out: @Dependent — the default, where the bean lives and dies with the object it is injected into — and @Singleton. Normal-scoped beans are injected as client proxies, which is what lets a short-lived @RequestScoped bean be safely injected into a long-lived @ApplicationScoped one: the proxy resolves the correct contextual instance on each call.

Why it matters: scope choice is a correctness decision. Holding request state in an application-scoped bean is a classic concurrency bug. Note too that CDI @ApplicationScoped differs from an EJB @Singleton — it is lazily created and proxied but does not bring container-managed locking or timers.

@RequestScoped        // fresh per request
public class CartView { /* ... */ }

@ApplicationScoped    // one shared instance, proxied
public class PriceCatalog {
    @Inject CartView cart;   // proxy injects the *current* request's CartView
}

CDI vs Spring: @Inject against @Autowired

Both frameworks inject by type, but the details are the interview. @Inject (from jakarta.inject, the JSR-330 standard) is portable across CDI runtimes; @Autowired is Spring-proprietary — though Spring also understands @Inject, @Named and @Qualifier. @Autowired has a required attribute; @Inject has none, so you express optionality with Optional or @Nullable. The sharpest gap is defaults: a CDI bean is @Dependent unless annotated (a new instance per injection point), whereas a Spring bean is a singleton by default. Spring falls back to bean-name matching to break ties; CDI insists on typed qualifiers.

Interview questions

Most-asked CDI questions in Indian enterprise-Java interviews — the Spring contrast comes up in almost every round:

Beginner What is CDI and what two core services does it provide?

Jakarta Contexts and Dependency Injection is the standard wiring layer of Jakarta EE. It provides typesafe dependency injection (supplying beans by type and qualifier) and contexts (binding each bean's lifecycle to a well-defined scope managed by the container).

Beginner What does @Inject do and where can it be applied?

It marks an injection point the container must satisfy. It works on fields, on one constructor, and on initializer (setter-style) methods. Resolution is by type plus qualifiers, done when the bean is instantiated.

Beginner What is a qualifier, and what is @Default?

A qualifier is an annotation that distinguishes among multiple beans of the same type. Any bean or injection point without an explicit qualifier implicitly has @Default; @Any matches every bean of the type regardless of its qualifiers.

Beginner What is @Named used for?

It assigns a bean a String name — chiefly so it can be referenced from Expression Language in Jakarta Faces/JSP (for example #{orderService}). It is a string qualifier and is not the recommended way to disambiguate ordinary injection; a typed custom qualifier is preferred.

Beginner What is the default scope of a CDI bean if you annotate none?

@Dependent — a pseudo-scope. The bean is not shared: a new instance is created for, and bound to the lifecycle of, whatever object it is injected into. This differs sharply from Spring, whose default is a shared singleton.

Intermediate Explain bean discovery modes and the role of beans.xml.

beans.xml marks a module as a bean archive and sets bean-discovery-mode: annotated picks up only classes with a bean-defining annotation (a scope or stereotype), all takes every eligible class, none disables discovery. Since CDI 4.0 an absent or empty beans.xml means annotated.

Intermediate How do you resolve an ambiguous dependency when two beans implement the same interface?

Distinguish them with qualifiers: define a custom @Qualifier annotation, put it on the intended bean and on the injection point. Alternatives are marking one bean @Alternative (and enabling it), or selecting via @Named. Left ambiguous, deployment fails.

Intermediate What are the built-in normal scopes and their lifecycles?

@RequestScoped (one instance per request), @SessionScoped (per HTTP session), @ApplicationScoped (one per application), and @ConversationScoped (a developer-demarcated span over multiple requests). All four are normal scopes and are injected via client proxies.

Intermediate Difference between @ApplicationScoped and @Singleton?

@ApplicationScoped is a normal scope: one instance per application, created lazily, injected through a client proxy — so it mixes safely with shorter scopes. @Singleton is a pseudo-scope: a single eager instance with no proxy. Both are one-per-app, but only the normal scope proxies. Neither is the EJB @Singleton, which adds container locking and timers.

Intermediate @Inject vs @Autowired — the differences that matter?

@Inject is the JSR-330 standard (portable across CDI); @Autowired is Spring-only, though Spring also honours @Inject. @Autowired has a required flag; @Inject expresses optionality with Optional. Spring falls back to bean-name matching; CDI relies on typed qualifiers. Default scope differs: CDI @Dependent vs Spring singleton.

Pro What is a client proxy, and why must normal-scoped beans use one?

A client proxy is a generated stand-in injected in place of the real bean. On each method call it looks up the correct contextual instance for the currently active context. This is what allows a short-lived @RequestScoped bean to be injected once into a long-lived @ApplicationScoped bean and still resolve to the right per-request instance. @Dependent beans are not proxied.

Pro Why can't a @RequestScoped bean be safely held as a plain field in an @ApplicationScoped bean, and how does CDI make it work?

The application-scoped bean is created once, so a captured raw request-scoped reference would be stale and would leak across concurrent requests. CDI injects a client proxy instead of the raw instance; every call through the proxy is dispatched to the active request's instance, keeping state correct and thread-safe without you managing it.

Pro What changed about bean discovery and CDI packaging in CDI 4.0?

The default discovery for an archive with no (or an empty) beans.xml became annotated rather than all, so only classes with bean-defining annotations are managed by default. CDI 4.0 also split the specification into Lite (a smaller, build-time-oriented core) and Full (implemented by Jakarta EE platform servers), which underpins fast-boot runtimes like Quarkus.

Pro Constructor injection vs field injection in CDI — which do you prefer and why?

Constructor injection: it permits final fields (immutability), makes required collaborators explicit in the signature, prevents partially-constructed objects, and lets you instantiate the class in a unit test with plain new — no container or reflection. Field injection is terser but hides dependencies and needs the container (or a framework) to populate them in tests.

Pro What is the lifecycle pitfall of @Dependent beans?

A @Dependent bean is bound to the object it is injected into: it is created and destroyed with that owner and is never shared. Inject one into an @ApplicationScoped bean and it lives for the whole application; obtain many via a producer or Instance without destroying them and they accumulate. It is the right default for stateless helpers but wrong when you expected sharing or timely cleanup.
